# 002.项目设计业务需求

好，同学们，那接下来就让我带大家一起来正式进入我们点评推荐搜索项目的一个项目的设计阶段。首先来看一下我们本章的一个目标，我们整个的一个课程都是会围绕着对应的一个点评搜索推荐的应用来做的，那在启动一个项目的过程当中，就和其他所有的一个企业级的应用项目一样，都是会收到 1 对应的一个需求，因此我们首先就是要做需求分析，理解需求所要达成的一个目标。然后我们的架构师、我们的程序员都结合对应的这些需求的理解和目标来进行一些技术分解，最后确定我们对应的一个技术的选型。在这个过程当中也需要在技术上面做模块化的一些设计来完善我们整个的技术到系统的流转的一个流程。


首先来进入需求分析阶段，那在我们一般的业务研发的一个公司内，我们首先会由业务的需求团队，如果说这家公司是与运营为驱动的一个公司，一般来说我们对应业务方就是运营人员，那也有的是以市场销售为驱动的公司，比如说卖软件的公司，那这些对应的业务需求就是出自于我们的市场的一个销售的人员。


那因此首先我们对应的业务需求会输出一个类似于 BRD 这样的一个东西，这个 BRD 的全称是 business requirement document，也就是一个业务需求的一个文档。那我们在互联网公司当中一般都会听到过有PRD，也就是 product requirement document。


之后我们也会讲到，但是其实在真正的产品文档之前，是会有一个对应的业务需求的一个文档，描述它对应业务的一个需求，那之后才是我们对应的产品将这个需求做翻译的一个过程。因此我们做需求分析，首先就要有业务人员给我们输出的一个 business requirement document，也就是我们的BRD。然后我们的一个产品同学就会结合对应的业务所输出的一个需求去做基于用户故事的一个文档。那这个用户故事我们把它叫做 user story，那这个用户故事，也就是说我们结合对应的这个具体的业务需求，让用户来使用的时候，它的整个的一个使用的路径，它的使用的一个闭环的故事是怎么样的一个场景？怎么样的形式？那这些就是用户故事。


最后我们会在用户故事的基础上出一份最终产品交付给我们研发人员的一个 product requirement document，也就是产品的PRD，那这份产品的 PRD 包含了所有的业务需求和用户故事的一个闭环，并且以产品的语言做到了一个开发可落地的产品文档当中，那这个就是需求分析的一个过程。


首先来看一下我们这个项目详细的一个业务需求是一个什么样的内容，那这个业务需求看似非常简单，它就是要做仿大众点评，做一个 H5 的应用。那这边有几个关键字，我们是仿大众点评的一个搜索推荐，做对应的一个基于 H5 的一个应用，也就是说程序是可以跑在移动端，是基于 H5 实现的。


当我们都知道大众点评最突出的一个亮点就是它可以给我们推荐线下的一个门店服务，因此这样的一个业务需求是我们需要做一个具有搜索线下门店服务的功能。那接下来还有一个业务需求，就是它必须得具备推荐线下门店服务的一个功能，类似于猜你喜欢，那这边大家可能看到两个关键词，一个代表的是说它要有搜索线下门店服务的功能，第二个是说要具备有推荐线下门店服务的功能，那这个搜索跟推荐其实经常我们会把它联系在一起，但是它其实是两个完全不同的目的和行为。


对应的一个搜索线下门店服务，也就是说我们对应的用户会有一个明确的目标，我要有一个关键字让我可以找到对应的一些服务，那这种是用户主动触发的一个搜索功能推荐的话，也就是说我们打开点评的首页会有一个类似于猜你喜欢这样的一个频道，结合用户的一些历史行为，或者对一些用户画像做一些喜好，做一些推荐线下门店的服务的功能，在用户没有一个主动的搜索欲望的时候，给用户可以在 APP 内逛起来这样的一个情形，那这种方式我们就叫做推荐。


好，我们可以看一下，这边是我在大众点评的平台上截了三张图片，首先它是有一个地理位置的信息，比如说我现在是在上海。然后我对应会有一个搜索框，那这个搜索框内其实小小的一个搜索框就包含了我们所有的一些搜索的流量入口和对应后台复杂的搜索引擎的功能。那接下来这边下面是一个服务品类的一个推荐，也就是说我们对应会有不同的一个服务类型，虽然都是线下服务，但是其实美食，也就是吃饭、电影门票、酒店门票等等，这些都是属于不同的服务类型。然后这边就是我们所要做的一个推荐，也就是说我们对应的会做结合个人喜好的一些推荐。比如说这边是一个私密的一个会所，那这边是一个我们吃的一龙虾等等，那这种样子的一个首页集成了地理定位搜索入口、服务的引类入口以及推荐的一个瀑布流。那当我们输入了一个搜索关键字，比如说这边我们要搜凯越这样的一个关键字，那我们就将流量引入了对应的一个后台的搜索引擎，会给我们返回出全部的一个搜索页面，那这边会给我们搜出凯悦两个关键字的凯悦酒店、咖啡厅，然后这边有一个五胶厂、凯悦酒店等等这样的一个情况，那我们单纯的一个搜索，如果说仅仅只不过是这么简单的一个关键词匹配，那就肯定是没有办法满足错综复杂的业务需求的。


那其中有一个最重要的一个关键点，就是我们可以通过对应的搜索引擎给我们做的一些筛选条件，更精细化的找到我们所需要的一个服务。比如说我们可以在这边选择不同的商区，去缩小我们想要查找的地理范围，也可以选择不同的一个品类分类去明确我们到底是要找凯悦酒店还是凯悦咖啡厅，还是凯悦饭店？对应的智能排序我们也可以做距离排序、价格优先级排序等等。


那这边的这个筛选的话，是结合我们这边定位的一个服务的一个标签，包含服务的价格以及其他更多的选项，给我们做一个更加精细化的筛选服务。那这个就是我们对应的一个业务诉求，其实也就是一句话，仿大众点评做一个搜索推荐的服务。那有了这些明确的业务需求之后，我们对应的业务方会将刚才所说的这几点的业务需求再加类似的一些 Demo 的效果页面，以业务 BRD 的一个文档的方式输出给我们对应的一个产品人员，表明业务需求是明确的，并且也可以清楚地通过一些文字和外部应用的一些图片的内容来清晰地表达自己的 business 的requirement。


那接下来我们就进入了需求的一个分解，也就是说我们需要以用户故事的维度去把用户在我们这个 H5 内所能做的事情讲清楚。那这个用户故事我们其实在刚才对应的一个文档当中已经给大家详细的介绍过了。那其实首先用户可以打开我们对应 H5 应用的首页，可以查看我们这个应用内所有的服务分类，包含对应的美食、酒店、休闲娱乐等等，这是一个 H5 应用首页查看服务分类的一个功能，那接下来这个用户在打开 H5 应用首页的时候，可以查看到推荐给我，猜你喜欢的一个门店列表。


这个门店列表我们刚才看到有包含对应的私密会所，也有包含龙虾、美食等等这样的一个推荐列表。那最后我们的用户可以通过首页的搜索栏搜索对应的关键字，查找到自己想要的一个门店列表，并且可以智能的筛选或者调整排序规则，以使得其可以更加准确化的查找自己想要获得的服务的一个门店信息。那这个就是我们对应的一个用户故事。总结一下，我们的用户可以打开应用首页查看服务分类，查看推荐的瀑布流，通过关键字搜索自己想要的一个服务门店。那最后产品同学会输出一份产品PRD，也就是我们研发可执行的一个产品文档，供给我们研发做对应的一个业务诉求。那对应的这个产品 PRD 内不光是包含了业务诉求这样的一些简单的一句话描述，也不光是包含了用户的整个的一个使用路径，更多的是它要能够精细化、明确化的讲清楚程序员开发在系统当中所要完成的一些产品的功能。


那首先第一个就是要有一些用户的注册跟登录的流程，并且要有进入应用首页的一个流程。我们其实在刚才的那些业务需求里边并没有看到对应有用户注册跟登录的一些行为。但是做产品方案的时候就必须得要考虑到由于涉及对应的一个推荐应用，也就是我们之后要做的一个千人千面的推荐，每个用户看到的都是不一样的一个推荐的门店服务。那自然而然我们的系统就必须得要知道我们当前登陆的这个用户是谁，以使得其可以根据这个用户的一些历史行为做对应的一个千人千面的推荐。因此我们对应的这个产品需要有用户注册及登录的这样的一个服务，然后它要有进入用户整个首页的完善的一个产品流程。


那其二可以看到在我们的业务需求内有一块就是服务分类这样的一个功能，那这个服务分类并不是平白无故涨上去的，或者我们天生数据库内就有的，它是需要有一个提供给我们运营的人员去维护对应的服务分类的一个运营后台，也就是说我们的运营人员需要根据自己对应的产品的特点，逐步放开维护我们自己这个应用所能够支持的服务。


像点评这种一开始也是以做美食起家，然后再不断地扩充到各种的服务品类当中，因此我们这个产品需要有一个运营后台去做服务分类维护的一个功能。同样我们对应的这个服务会引入一些商家，比如说我们刚才搜索的凯悦酒店，其实就是凯悦集团这个商家下的一个服务的一个门店。因此这个运营后台需要有整套的一个商家入驻的流程，也就是说凯悦集团需要入驻到我们对应这个点评的运营平台上，才能签署协议，提供对应的酒店类的一个服务。


那有了服务分类和商家入驻的系统之后，需要有最终的一个呈现在用户面前门店的一个功能，也就是说商家会在我们的平台上开出自己对应的一个线下服务的门店。一，使得用户可以进行搜索和推荐，因此整个产品我们需要运营维护、服务分类、商家入驻、门店管理等这样的一个运营后台的功能。那最后我们对应的这个产品 PID 内需要包括用户可以通过搜索关键字筛选条件找到自己想要的门店，这种样子的一个服务的过程。用户打开首页系统可以根据用户的历史行为做对应的一个推荐门店服务的过程。那其实我们可以看到真正意义上的产品 PRD 会比我们对应的业务需求和用户故事多很多，因为它必须得是一个闭环的、可落地的一个产品方案。也就是说产品 PRD 不仅需要考虑业务需求和对应的用户故事，更重要的是它要完成一个产品的闭环，使得用户可以通过登录注册查看自己的推荐列表。运营人员可以维护对应的服务分类。


商家入驻门店，使得用户可以去搜索对应的关键字，找到自己想要门店的一个服务，因此整个产品的 PID 其实是比业务需求复杂很多。但是这个产品的 PRD 也是我们的研发人员可以真正意义上去做对应的一个研发的关键。
