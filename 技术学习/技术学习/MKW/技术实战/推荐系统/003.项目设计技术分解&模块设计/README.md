# 003.项目设计技术分解&模块设计

在产品 PRD 完善之后，就需要做到对应的一个技术服务体系了。那首先我们来看一下这一套防点评的一个搜索推荐的应用需要用到哪些技术？首先我们很容易的可以想到后端服务模块肯定是需要的，也就是说我们要有后端业务模块去提供对应的一个业务服务之后，我们还需要有对应的后端的存储系统，我们需要有数据库去存储这些业务功能的数据的累积，然后我们要有一个搜索系统，不同于单纯依靠数据库的一个存储系统，它更多的是要与带有搜索引擎功能的一个搜索系统之后，我们还需要有一个推荐系统，可以结合用户的千人千面的属性去给用户推荐不同的一个服务。那最后当然前端的页面展示层是必不可少的，我们需要有前端的页面提供给用户浏览对应 APP 内所有功能的一个场景的入口。


那接下来看一下我们作为对应的这个项目的一个架构师会去怎么样子做模块设计。首先将模块设计分为两层，第一层是业务模块的架构设计，第二层是系统模块的架构设计。那什么是业务模块？什么是系统模块？我们大家可以简单的理解为业务模块的架构设计是完全对等于产品的 PRD 的设计。也就是说我们的产品 PRD 内有哪些模型，有哪些业务元素？我们对应的这个业务模块的架构设计就需要围绕这些功能点去做。


那系统模块的架构设计更多的是偏向于技术落地细节，也就是说我们对应的系统是要分为刚才说的后端系统、存储系统以及搜索系统、推荐系统等，它也为对应的一个产品 PRD 服务，但是它更多的是偏向于技术层面的一个架构思维的一个框架。那首先我们来看一下项目的一个业务模块的架构会怎么样子去做设计，我们对应的业务的设计必须得是跟产品的一个 PRD 完全对称。那我们首先就要来看一下在我们整个的产品的一个闭环逻辑当中会有哪些业务实体出现，首先我们会需要有一套用户管理的一个业务实体，也就是我们所谓的用户会有一些用户的一些行为，之后我们会有一些商家管理，比如说入驻进来的这个商家，包括商家提供服务的这样的一个能力。因此会有一套商家管理的体系，然后我们会有一个服务类型管理的体系，也就是说我们有美食、酒店、休闲娱乐等等这样的一些服务类型。


最后我们会有门店的一个管理，那这个门店的管理就是我们作为这个项目的一个业务实体的核心，围绕着门店去做对应的一个搜索推荐。因此我们整个的一个根据产品需求翻译下来业务的技术人员要实现用户、商家、服务类型以及门店的四个业务实体。那接下来我们要定义对应的这些业务实体的行为有哪些单纯的用户它是没有任何意义的，它最多就是类似于我们面向对象编程当中的一个类，它必须还需要有类的方法去定义它对应的这些业务实体的行为。那我们来看一下用户的行为，会包括有用户注册、用户登录，那对应的商家这样的行为我们需要在运营后台有商家创建，在前台有对应的商家的一个浏览，那在服务类型这边也是一样，要有后台的一个创建的功能，以及对应的一个查询浏览的功能。那门店这边就是我们最复杂的一块业务实体的行为，也就是门店要有创建，也能被搜索及筛选，最后还要可以被推荐。


那结合这样的一些业务实体，我们用一张简单的 er 图去做对应的一个表示，首先用户的一个业务实体它是相对来说是比较独立的。它的业务实体是用户它有对应的字段，比如说这个用户的编号，也就是我们常常说的唯一的一个ID，然后会有一个用户登录的名称，以及对应的一个登录的密码。当然用户的一些模型远不仅于此，还会有一些性别以及喜好等等，这样子用于做千人千面的一些标签。但是我们这张图上就暂时不表现出来。然后我们对应的会有一个商家入驻的过程，这个商家会有他自己的商家名以及商家评分，以及我们为了控制风险需要做一些是否禁用商家这样的一个功能，防止其上传恶意的一些门店，那对应的这个商家可以开 n 家门店，我们的凯悦集团可以在上海的五角场开门店，也可以在浦东新区开门店，更甚至于它不光可以开酒店，还可以开咖啡厅、美食馆等等。因此我们对应的商家可以开 n 多家门店。


当然，每一家门店都有一个固定的服务类型，也就是说我们对应的这个门店是对应哪一个服务的？是酒店，就是酒店，是美食就是美食，那自然我们对应的商家对门店的关系是一对n。那门店对服务类型的关系就是 n 对一了，那我们回过来看一下，服务类型就有对应的一个服务类型的名称，也就是说它是美食还是酒店？它有对应的一个服务类型的图标，用来做前端展示，以及有自己对应的一个排序，也就是说为什么是美食出现在第一个，为什么不是酒店出现在第一个？等等这种样子的一个排序的规则。


那对应的门店就是我们要做的最复杂的一个业务实体，目前来看会有对应的一个门店的名字，以及我们人均消费的一个价格，对应的一个评分怎么样？对应的地理位置的坐标在哪里？然后这个门店会有哪些标签，以及它的营业时间是在什么样的一个营业时间范围内？那这个就是门店的一个属性。


好，我们将业务模型架构做完之后，就可以做简单的一个系统模块的架构的一个拆分了。那让我们来看一下我们结合产品的一个业务诉求。首先我们自上而下的看前端会有一个 c 端的一个 H5 接入，也就是说用户可以通过一个 H5 的页面直接登录注册到我们对应的眼屏的一个 c 端的一个界面上，做搜索以及推荐浏览的一个功能。那同样我们对应的有一个 admin 的运营后台，要通过一个 PC 的网页去做运营后台的一个接入。那我们将对应的一个业务系统，不管是 H5 的接入还是 PC 的接入，统一打包在一个业务系统内提供对应的鉴权的接入服务。


当然我们对 c 端用户的接入鉴权跟 admin 运营后台的一个接入鉴权是完全不一样的，但是我们可以封装统一的一个鉴权的接入来做对应的管理的功能，首先整个的一个项目还是要有对应的一个业务功能的模型，比如说我们的一个商家门店的服务，包括服务品类的一个管理是如何维护进系统，如何展示给 c 端用户的，那这些业务功能的模型在业务系统当中实现完成之后，最后会落到数据层，也就是我们的 MySQL 的业务数据库中。
我们对应的会有一个搜索服务提供给 c 端的一个 H5 的接入，会有一个搜索服务的一个接入，然后在搜索服务接入之后，我们会有一个搜索语义构建的过程。我们需要将前端的非常简单的一个关键字的一个搜索的bug，通过一些语义构建做成一个搜索引擎可以理解的语言，然后扔到对应的 Elastic search，我们选型的一个搜索引擎当中去做对应的查询。


那当然这个 Elastic search 的一个数据的源都是通过索引的一个构建服务，全量的通过 MySQL 数据库构建到 Elastic search 当中。那当然全量的搜索构建的过程当中，我们一般这个全量构建会以每天或者每隔 8 小时等等比较长的时间去做对应的一个全量索引的构建。


那在这个时间段内，若我们对应的一个门店的数据，或者我们商家服务品类的数据发生了任何的变化的时候，也需要在我的搜索引擎当中能够感知到，这个时候我们就需要借助于我们对应的 MySQL 业务数据库到 Canal 管道的一个 binlog 的同步，将对应的数据库内的一个增量数据同步更新到 Elastic search 当中，完成对应的一个 Elastic search 索引的更新。


那我们在课程当中会以两种方式做搜索的接入，首先我们会使用最简单的通过 MySQL 的业务数据库提供搜索接入服务的一个存储的能力，来给大家做一个非常非常简便甚至于简陋的搜索引擎的一个简单的接入。再之后我们会用 Elastic search 对比的功能去做对应的搜索服务的一个数据层的一个构建，来使得大家可以深刻的理解 Elastic search 作为搜索引擎的优势，以及和传统的 MySQL 的业务数据库相比所能提供的一些高级功能的一个支持。


那最后我们需要完成推荐服务的这样的一个模块，那在推荐服务当中我们也一样会有对应的推荐服务的接入，那这个推荐服务的接入最后会去访问我们对应的 MySQL 的一个推荐数据库。那这个推荐数据库我们在整个项目当中会和对应的这个 MySQL 业务数据库放在同一个存储设备上，但是两张不同的表，大家从模型实体上来说，把它认定为是两个数据层就可以了。然后推荐数据库提供给我们对应的推荐接入服务，做查询的只是一个推荐数据的一个结果集，我们需要有一套推荐算法，通过离线计算的方式，将对应的一个推荐结果的准备离线计算完成放在对应的推荐的一个数据库当中，也就是说我们提供给 c 端的一个猜你喜欢的模块，其实查询的是已经通过离线的推荐算法计算完成了 MySQL 的一个推荐数据。


结合我们刚才整个项目的一个框架构图，我们确定了一下技术选型。首先后端的业务会使用 Java JDK 1.8 做主要开发语言，并且结合 spring boot 框架 2.1.5 的版本做对应的一个业务系统的开发。那接下来我们对应的一个后端的存储仍然使用的是 MySQL 数据库 5.6 这样的一个稳定的版本去做 MySQL 的接入，然后我们会使用 mybadcase 做对应的一个中间数据处理层的接入。


之后我们的核心的一个搜索系统会使用最新的 Elastic search 分布式的一个搜索引擎 7.3.0 的版本，并且会使用 Canal 1.1.3 的一个版本，再加一些改造去适配Elasticsearch。那最后我们的推荐系统将使用 Spark machine learning LIB 机器学习组件最新的 2.4.4 的版本去做对应的一个机器学习的推荐系统。
