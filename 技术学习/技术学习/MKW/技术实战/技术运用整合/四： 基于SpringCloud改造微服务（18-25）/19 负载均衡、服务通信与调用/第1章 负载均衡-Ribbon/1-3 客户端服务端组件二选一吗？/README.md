个集群中的某些服务节点。下面主角要登场了是谁？客户端我们说基于客户端的负载均衡，也就是说这个客户端要知道我如何做选择，从而选出一个 target server 也就是在这个服务集群中选择我需要访问的这个节点。那它必然有一个前提，既然让我做选择，我肯定要知道有哪些选项对不对？也就是说服务集群中的所有的机器的列表要在客户端保存一份 copy 那说到这里，同学们应该就非常明白了。


前面我们学到尤瑞卡的服务注册以及服务发现对不对？那我们在客户端来说，获取到节点的方式就是通过尤瑞卡的服务发现从注册中心那里拉到所有的可用节点的列表，也就是机器列表。好，有了这个机器列表以后，那我们的客户端就可以把这个机器列表作为一个输入条件，结合我们在本地指定的负载均衡策略，它的输出是什么？它的输出是一台机器，也就是通过这种方式我就能获知接下来要访问哪一个具体的物理服务器地址。那我们说这个服务列表它是不是一成不变的，并不是因为服务发现它会从注册中心拉取到最新的服务列表。这样的话，我在客户端保存的这份机器列表就可以反映这些服务在注册中心的状态变化。如果我在注册中心下线了一台机器或者服务，剔除了一个服务，并且或者有新的机器上线，那在客户端也会同步的反应过来。也就是说这个负载均衡策略的输入条件，这个机器列表它是动态变化的。 OK 大家一定要记住这两个前提条件，一个是机器列表要保存在客户端。第二，这个机器列表要能动态的反映当前服务的状态，这样我们才可以有条件把负载均衡策略放在本地来实现。

与客户端相同的是，他们都有一个服务集群，那这个集群下有 service a service B 和 service C 可以看作是集群中的各个节点。那也有一个客户端，并且我们知道客户端发送出来的请求最终肯定是要抵达这个服务集群中某个服务器的。对不对？那他是直接来访问吗？并不是为什么，因为客户端他这里没有一个机器的列表，对不对？他不知道该访问谁，那怎么来做我们看这里凭空冒出了一个特殊的服务端，这个服务端是什么？大家可以把它理解成一个特殊的组件，那它是独立于这个服务集群以及独立于当前这个客户端的一个组件。它这里面包含了具体的负载均衡策略。比如说我们前面已经学过在分布式章节利用 N gx 做负载均衡，对不对？那它就是一种软件形式的负载均衡，同样的也是安装在一个独立的服务组件上。那与此相对的，我们还有硬件的负载均衡策略。我们后面的图文教程里面会提及这个策略叫f5，那它这个费用可是相当的不便宜的。


Ok.那客户端的请求为了抵达服务器中的某个具体的物理地址，他要怎么做？他要通过这个中间的转发层对不对？所以说他的请求首先是要转发到中间这个红框里的服务端负载均衡组件，通过服务端负载均衡的组件，再来把这个请求具体的分发到某台机器。那客户端他知不知道是哪台机器响应了他的请求他并不知道，我只用把具体的请求负责发送给服务端的负载均衡组件。剩下的事我就不管了，由负载均衡组件来替我选择机器，并且把这台机器最终的 response 返回给我客户端。


那跟前一种客户端的负载均衡来比较，这种服务端的负载均衡策略是不是对客户端来说更轻松客户端少了一个负载均衡的任务，但是既然需要负载均衡，那它要么在客户端，要么在服务端，它总归是有一端存在的。所以我们有这么一个提示，对于大型应用来说，通常是怎么样是客户端加服务端负载均衡搭配使用的？我们说没有银弹，也就是没有一种负载均衡策略能满足所有的情况的。同样服务端负载均衡有他自己的 B 端。那客户端负载均衡同样也是，那在各种场景下，我们都需要用多种技术的组合来搭配使用，共同解决问题。我们对客户端和服务端这两种模式做一个比较，那红色的部分是客户端的负载均衡模式，黑灰色的部分是服务端的负载均衡模式。

第一回合，我们从做一个需求变更的难易程度上来做一下比较。这里可以看到红方也就是客户端的模式，它对开发团队来说更加的友好。为什么？因为客户端在哪里，客户端就在我们的代码里，对不对？向另一个服务发送一个 request 那究竟选择哪种负载均衡策略？我们可以是在代码中通过一些配置文件就指定了。如果想对这个负载均衡策略做一些自定制或者修改，那我们是不是通过代码重新创建一个新的负载均衡策略，并且应用到当前服务就可以了，对不对？那这都是我们很熟悉的过程。


而反观服务端的负载均衡模式，一般来说开发团队很难去主动的发起修改。为什么呢？因为服务端的负载均衡往往是放在网关层，像 NGINX 像某些 gateway 包括 f5 的组件网关上。那这些部分通常都是由 DevOps 团队或者 NetOps 团队来把控的，开发团队很难直接修改这里面的内容，因为我们的 security level 也就是安全等级没有达到这个程度去修改网关层的组件。


好，这是一方面。另一方面从开发语言的角度来说，网关层的负载均衡组件所用的语言跟我们日常开发所用的 Java 那真是风马牛不相及。为什么呢？大家之前在分布式章节应该也有过 NGINX 的使用经验，对不对？大家看里面的配置和我们日常用的加吧，包括 practice 甚至是 yml 配置文件，那都是完全不一样的。其实相对来说 ngx 还算对开发团队比较友好的一种方式。那如果我们采用了f5，比如我现在工作的这家公司就是依托于 f5 做负载均衡的，那里面的负载均衡策略你要想动起来简直是不可能的一件事情，你开发团队根本不具备这种特殊的能力。那他需要怎么样？需要专业的网络维护人员来进行修改？ OK 接下来我们从运维的成本上来做一个比较。


那我们说客户端的负载均衡模式运维成本比较低，为什么？因为它不需要一个额外的组件，它的所有的负载均衡策略都是在客户端的程序里就制定好了。而服务端的负载均衡模式往往需要像网关层这样一个额外的组件。如果你使用了f5，那还需要搭配硬件设备，这个成本相对于客户端的负载均衡来说那是相当高了。


接下来我们从依赖的服务项上面做一个分析。客户端的负载均衡组件它强依赖于注册中心，为什么它需要从注册中心那边拉到一个可用服务的列表？并且这份服务列表还需要具备一个动态更新的功能，也就是说我要动态地反映出当前所有可用节点的状态变化。因此我们说基于客户端的负载均衡组件，它需要一种类似 eureka 服务注册中心的机制来替它完成这样一个服务列表的更新以及获取动作。
那对于服务端的负载均衡策略，我这里写到通常不依赖。为什么我写通常来说难道还有反例吗？没错，前面我们看到的都是 NGINX f5 这种形式对不对？那它当然不依赖于注册中心了。那假如我们使用了 spring cloud 的架构，我们知道，这里面也有 zoom 也有 gateway 这种网关层。那在这些网关层上面同样也是可以应用负载均衡策略的，我们依然可以使用注册中心把服务列表推送到 zoom 推送到 gateway 这类网关层组件。所以说服务端的负载均衡依赖项，它根据你具体使用的方式、你使用的负载均衡技术的不同而有所不同。


最后一项，我们从使用范围上做一个简单的比较，客户端的负载均衡组件往往和微服务框架搭配比较多，因为微服务框架大多都提供了一个注册中心，对不对？可以拿来就用。像 ribbon 它就广泛的使用于 spring cloud 的所有组件库当中，比方说 gateway 比方说 hidge streaks 或者是 fin 那后面我们都会使用到的。那对于传统模式的 f5 或者是 NGINX 来说，那它应用在 J boss 或者汤姆 cat 这类传统应用上会非常多，非常常见。但是我们在微服务框架中就不会使用的，其实并不是绝对的。有的时候即使我们使用 spring cloud 架构的应用系统，那依然可以看到 NGINX 的身影。因为 NGINX 它使用广度确实非常非常的广，而且它的性能也经过了长期的考验，是个合格的共产主义战士。


那对于一些老系统来说，它可能网关层已经使用了 NGINX 那当我们给它做微服务改造的时候，把它升级成了 spring cloud 应用，不能完全的把网关层替换过来。因为迁移这些路由规则或者负载均衡策略其实并不是一件非常容易的事情，所以可能依然会继续使用 ngx 做网关层组件，那并不会把它全部替换到 spring cloud 中的 zoom 或者是 gateway 那其实在真实的项目中，往往就是客户端的负载均衡组件，服务端的负载均衡组件，各种网关层搭配使用。同学们在自己的项目当中，就要根据需要根据实际的业务场景选择合适的负载均衡技术以及合适的负载均衡策略应用到项目当中。 OK 前面我们做了很多的铺垫，那么接下来我们就正式进入瑞本的学习。下面请收看图文教程。瑞本体系架构好，同学我们下期再见。
