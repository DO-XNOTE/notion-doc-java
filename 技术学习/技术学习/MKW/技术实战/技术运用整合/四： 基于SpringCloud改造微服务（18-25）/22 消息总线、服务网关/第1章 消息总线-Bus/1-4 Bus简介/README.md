还有七这葫芦娃七兄弟，那这几个弟兄合在一块，不就成了一个小鸡群吗？葫芦娃七兄弟在后面嚷嚷着爷爷什么时候给我们刷新属性，这时候是不是感觉到有点那么吃不消了，你要挨个的给这七个节点，每一个节点发送一个请求，让他来拉取最新的属性别急，好戏还在后头。


又过了这么段时间，你这个业务一发而不可收拾爆发性的增长。那领导给你加了更多的机器。哟了不得，像你这么厉害的机器，还有一万多个吓死了，吓死了。那我问问你，这个服务中心你还能带得动吗？ 1 万台机器挨个刷新，你就是写个程序来帮你做这件事儿，恐怕等你刷新完这 1 万台机器，眼睛一闭一睁一天就过去了。对不对？所以这个配置中心表示实在是带不动。所以如何来解决眼前的这个问题，我们把这个难题就留给了 bus 组件，看他怎么来解决这个问题。



当我们有了 bus 以后，集群还是这个集群？那它面临的任务依然是批量刷新。在这个场景下 bus 是怎么做的？他既然叫消息总线，那他后台一定是用到了消息队列。那这个消息驱动的模式中有一种非常非常经典的模式叫发布订阅。那我们能不能把这种发布订阅的交互模型套到现在这个批量刷新的业务场景上来？当然可以。
假如我们所有的节点都订阅了某个 topic 也就是消息组件中的属性刷新这样一个 topic 那当你的属性发生变动的时候，只要发送一个广播消息，那么所有的节点都会去消费你这个消息，并且触发他们的刷新动作。这种通信模式简直是为批量刷新场景量身定做的。甭管你后台集群装的是七个葫芦娃，还是一万多个很厉害的兄弟啊，一个 bus 到来，大家全部上车，通一波带走，再也不用哼哧哼哧，像老黄牛一样挨个节点进行刷新了。各个服务分别表示求带老司机带我也要上车。那咱上车归上车，上了车以后要怎么样？先上车后补票，这个票是什么？那就是 bus 的标签。

接下来我来带大家过一下 bus 这个组件都有哪些特点？有请主角登场消息总线。那咱这次只挑选四个比较主要的具备特征的标签来跟大家说一下。好，它第一个特点就是轻量级。为什么咱说轻量级实际上巴斯只是对这个消息进行了简单的封装。为什么这么说呢？因为它底层还需要依赖一个组件。这个组件并不是说的 Kafka 或者 rap 的 MQ 这种消息组件，而是 spring cloud 中的一个组件。它的名称叫 streamstream 是在 spring cloud 中专门用来与消息中间件进行通信的组件。那么在后面的章节，大家也会学到 stream 的细致用法。


那 buzz 这里实际上并没有多少料，它底层全都是借助 spring cloud stream 组件来与消息中间件进行通信的。那 bus 在 stream 之上它仅仅是一个壳子，它只是对消息广播这个场景做了一个简单的封装，自己创建了一个消息刷新的 topic 和相应的整个流服务。所以 bus 本身是一个非常轻量级的组件。那么接下来第二个特征叫消息广播，这是巴斯最擅长的一项工作。那它的底层是基于消息的发布订阅模型来做的。我们大家可以这样认为，但凡在 spring cloud 当中你想使用消息广播这个场景，你有两种方式，要么使用 spring class dream 来实现这个消息广播，或者咱也可以直接使用 bus 来对你的消息广播场景进行一番扩展。


OK 咱再看他的第三个标签无缝继承 config 什么意思呢？咱待会儿，在真正的实践环节你就会发现了，在配置上，你只要简单的配置一个 Kafka 或者是 rabbit MQ 的连接，你甚至都不需要针对 config 和 buzz 之间做任何的集成。那么 buzz 就会很自觉地到 Kafka 或者 rabbit MQ 上面发布一个 topic 那么这个 topic 就会被用来监听你的属性变化，也就是说它可以代替咱前面使用到的 activator refresh 方法，对我们的服务节点进行属性的刷新操作。


OK 那最后一个标签是自定义消息，其实它是 bus 的一个扩展点。通常来讲 bus 的使用场景就是和 spring cloud config 一起搭配起来，执行批量的刷新操作。但是我这里提醒大家一点， bus 所能做的工作远远不止有刷新这么简单。


就像前面说到的， bus 构建在 spring cloud stream 以上，但是他对消息广播的这个场景做了一系列的封装，我们可以利用 bus 的数据流结构自定义一个消息体，然后实现自己自定义的消息广播场景。所以说但凡你只要想在自己的业务中实现一个消息广播的场景，那我们可以考虑一下采用 bus 的自定义消息来做。
OK 那咱这里对 bus 的两个应用场景做一个总结。那它的第一个应用场景也是目前应用最广泛的场景，就是配置变更通知。这里为什么叫配置变更通知而不是配置变更推送呢？ bus 其实只是负责通知到你告诉我们的服务你的配置属性可能发生了变化，相当于通知你赶紧去取快递。你的快递现在存放在 spring cloud config 配置中心那里。但是巴斯可不负责送货上门。所以后面的配置变更拉取动作还是由各个服务节点来自己完成的。
