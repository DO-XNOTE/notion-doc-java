商品详情页它承担的其实更多的是一个聚合的接口功能。因为你要在详情页当中显示很多的信息，比方说用户的点赞数对不对？给这个商品点赞，还有你的营销优惠的价格、商品原价，还有商品打折后的价格，以及这个商品所能参加的所有营销服务。


对不对？当然了，还有商品的主图、商品的轮播副图，还有下面的详情页中的副文本。那所以商品详情页接口更多的是起到一个聚合作用，从各个上下游当中把这所有信息聚合在一起返回给前端应用。那接下来的功能是获取 SKU 等等的信息 SKU 是电商领域中非常常见的一个概念，比方说你有一个主商品对不对？这个主商品下面有不同的规格，比如你在双 11 大促买一个充其娃娃，那这个充其娃娃是这个主商品，它下面你选择规格的时候可以选择什么某冰冰款对不对？还有什么 angelababy 款还有仓老师款等等。


那这每一个不同的款式其实就是架设在主商品之下的一个 SKU 有的时候我们也把它叫做规格。那咱真正的库存通常来讲是落在哪里，落在 SKU 这个层级上面。 OK 那我们往后看前面提到了库存，那自然而然的库存模块扣减库存，也把它放到了商品中心这个领域。因为库存跟商品是息息相关的。那这里我再跟同学多提一点，关于库存扣减在超高并发的应用当中是怎么来做的？比如在阿里的淘系应用当中，同学们买一件商品，那这个库存扣减同学们想一想是在哪里完成的？告诉大家，实际上它就是在数据库完成的，也就是说库存扣减最终一定是落在数据库当中，这就是为了保证强一致性。


那同学们可能会想，这么多的用户一起去操作数据库扣减库存，那你每一次扣减的时候岂不是要上一个锁行级别的锁，那么对这个数据库的吞吐性能也有很大的影响，对不对？那我就分两点，以最简要的方式跟大家大致说一下，我们把这个问题一分为二，首先我们看数据库以外的这些操作，那在前面的应用当中，我们肯定会利用一些限流，包括排队等等削峰填谷的这些策略，保证流量相对平滑的导向数据库端。


那当你真正到了数据库以后怎么办呢？同学们可能想不到，阿里里面藏着很多个 MySQL 的原作者的高手，那他们都是 MySQL 的 commiter 他们对马斯克的引擎做了很多的改造。简单来说就是在马斯克的引擎当中引入了一种类似排队机制的策略。但是它并不需要等待业务层发送全 section 提交指令，才认为你这个事物已经完整提交了。他们通过对 MySQL InnoDB 层上面打了一些 patch 补丁。那做到了什么呢？做到了在数据库层面上对你单行记录了这种修改，可以做到一种并发的排队机制。那我们如果对 nod B 的引擎比较熟悉的话，肯定知道 commit on success 或者是 rollback on feel 这些操作。


淘宝的团队对这些步骤打了一些功能补丁，并且怎么样呢？配合我们在 SQL 里面可以加一些 SQL hands 实现了这样一种优化功能。也就是说在我们的数据库事务里，我不需要等待应用层提交这个 commit 请求，而是怎么样呢？在数据执行完你最后一条的 SQL 后，根据你执行过后的 target effect row 也就影响的行数的结果来决定它是否当即提交或者是回滚。那这样可以减少一些等待的时间。当然了，我们在数据库层做的优化还不止如此，如果同学们有兴趣，可以去网上收集了解一下相关的资料。
