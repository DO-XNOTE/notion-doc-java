那除了主搜以外，还有一个在整个电商项目中最重要的一个模块叫什么呢？商品中心 foodie 杠 item 那商品中心为什么是电商项目中最重要的一个模块啊？因为我们这个电商项目所有的上下游链路都是围绕商品中心来展开的。但是我们这里要提醒大家一下，我们在项目当中这个商品中心是一个相对比较粗的领域模型。其实在整个商品中心里面它的领域划分非常的详细。如果是作为像阿里这种级别的电商项目，一个商品中心可能要拆成 10 到 20 个甚至更多的微服务组件。


我打一个比方，商品领域当中你的营销优惠、商品优惠计算对不对？包括你商品优惠券。那这些往往在大型互联网项目当中都是独立的微服务模块。那再打个比方，像用户的评论模块对不对？往往也是单独抽出来的。而且是在历次的双 11 等降级活动当中，像评论模块这种非主链路的模块，首当其冲被降级的，还有我们看商品的详情页、搜索页，这里面都有图片。对不对？那就牵扯到商品领域的图片空间。如果从用户场景的角度来分析，还有某些非常高频的应用场景，比方说商品详情页，往往我们也会把它独立成微服务。那么这样细分下去，整个商品域我们会拆分出非常多的微服务。随着你的项目规模、用户基数越来越大，那么注定你的微服务拆分力度也会相对较细。那我们作为一个教学级别的 demo 项目，这里我们就笼统地把整个商品领域划分成一个独立的微服务模块，不做过细的拆分。 OK 我们接下来往后看。


下一个模块是 foodie card 这个 card 其实就是 shopping card 也就是购物车的一个概念。那购物车是介于商品域和订单域之间的一个环节。通常我们在大型的电商项目当中，也是把购物车单独作为一个微服务模块来对外提供服务的。好，那接下来往后看，我这里还抽了一个 foodie off 这个 off 的是什么意思呢？权限验证校验。那这里我们把权限校验模块给单独抽成一个微服务。待会在后面的章节中，我们尝试把它跟网关层结合起来。使用在真实的项目当中，用户鉴权其实并不全是在网关层完成的，有的很多情况下我们也会把它下放到服务端来做鉴权。


接下来往后一个模块是 foodie user 这个大家很好理解了，是用户管理的模块对不对？包括用户的注册，还有 customer profile 的更新操作以及用户信息的查看等等。我们最后一个模块也是整个电商领域当中非常重要的一个模块。大家应该能猜到订单模块 food order 那这里就包括了订单的创建状态的更新等等。那这些模块之间其实有一个相互依赖的关系，比方说你的订单模块肯定是需要查询商品的，对不对？所以它也依赖于商品模块，我们在下单的时候也需要获取用户信息，对不对？所以说这个订单模块也需要依赖于 foodie user 模块。那看完了咱微服务的外部结构划分。接下来我们看一下每一个独立微服务，它内部的文件夹结构都有什么变化。这里左边咱列出来的是当前的文件夹结构，咱看到有 apiservicemapper podio 和 common 这五个模块。

那在咱微服务章节改造的过程中，我们秉承一个准则是什么呢？就是说对现有的这些文件夹结构尽量少做改动，因为咱同学们已经非常熟悉现有的这套文件夹结构了。为了让这个微服务改造迁移的过程更加顺利，我们保留了大部分的模块。但是对其中的某些小模块有一些微调。那我们来举一个例子，就拿其中的某个微服务咱就叫微服 A 来说，看一看它这里都有哪些变化啊？首先 API 这一层在之前是用作什么功能呢？是放 controller 对不对啊？那在这里也就是微肤改造之后，咱这个 API 层做了一个非常大的改动，把它改成了纯接口层。这个纯接口层顾名思义它就是放的纯接口，那是 Java 的 interface 那为什么把它单独抽出来呢？因为我们后面如果牵扯到不同微服务模块之间的调用，到了后面的章节，当大家学习到服务间的调用 thin 这个章节以后，就会发现我在多个微服务之间互相调用，实际上只用把对方的这个接口层引入进来就行了。那我这里并不需要把这个接口的实现引入进来。所以咱抽出这一个接口层是为了什么？就是为了专门给那些需要调用我的服务来用的。
举一个最简单的例子，我们的 foodie order 订单服务是不是需要调用商品服务的接口？那么在我的订单服务当中，只要把商品服务的接口层引入进来，那借助 spring cloud 当中的份组件，我就可以很方便地发起一个从订单中心到商品中心的无调用。


OK 那接下来我们看其他的模块有没有变化？这个原先的福地 API 它是放 ctrl 了对不对？由于它这里的语义发生了变更，它变成了纯接口层。那么原先的 controller 放到哪里呢？我们这里创建了一个新的模块，叫做 food web 那所有的 controller 也就是跟页面层交互的这些服务，我们统统放到了福利 web 当中。那这里只是改了一个名字，它的内容其实没有发生什么变化。


OK 接下来往后看这个 foodie service ，我们同样的把它平行的迁移过来，也叫 food service 那这里面同样放的东西还跟以前一样，就是它的 service 层实际上在企业项目当中我们可以怎么样把 foodie web 和 food service 二合一，因为本质上它们其实都是接口，只不过一个是跟 UI 层打交道的，另外一个是在服务内部打交道的。我们这里为了保持和之前的项目结构一致，所以把 food API 和 food service 平行的迁移过来，也同样保有这两个子毛九。


OK 那接下来往后看 foodie mapper 有没有变化呢？没有，它这里依然是存放 mapper 类，还有咱的数据层访问文件的地方。 OK 那再往后 food pogio 这里也是没有变化的，这里存放了数据库的 entity 对象，还有 vo bo 等等。


接下来发现独立 common 这个去哪了？咱前面提到过有一个独立的 common 组件把它放在哪里呢？抽成了一个公共的猫臼对不对？因为咱在 common 当中存放的大部分是一些需要在多个微服务组件中都会被用到的工具类，所以这里被单独抽成了一个猫，就大家在新的微服务改造过后的项目结构当中，就见不到它了。


那除此以外，我这里还会要提醒大家一下，在微服务领域当中，我们的数据库是怎么样，是每一个微服务其实有自己独立的 schema 也就是说我当前的商品中心。那我访问的是什么？是商品中心自己的数据库。那订单中心就是订单中心自己的数据库，不像之前单体和分布式应用，咱就一套数据库一个 schema 那所有的服务都去访问同一个数据库。那在微服务改造过程中，同学们需要把这些数据库表都把它拆分到独立的 schema 当中。比方说我的商品，我这里可以给它创建一个新的 mariadb 的 schema 把它起名就叫 item 然后订单就叫 order 通过这种方式，咱就模拟了一套微服务，有对应的一个独立的数据库这种结构。 OK 那么这一节当中，我们介绍了整个项目会被拆分成哪些微服务的模块，也就是微服务的外部划分，以及每个微服务模块当中它对应的子模块都有哪些。那在下一节当中，我将跟大家详细介绍每一个微服务它其中包含的业务内容。好，同学们，我们下一小节再见。
