在一开始我先跟大家介绍一下 spring cloud 和微服务之间的关系。为什么说 spring cloud 是为微服务打造的一款框架？ OK 接下来带大家去看一下 spring cloud 的整体架构，这里带大家去了解一下 spring cloud 都提供了哪些的功能。


紧接着咱得知道这些功能都是由 spring cloud 的哪些组件来支持的，对不对所以这里我要带大家去了解一下 spring cloud 的，这个组件库里的组件可不完完全全是由开源社区一手贡献的，其中大部分的爆款组件都是由 Netflix 这家公司来贡献的。后来阿里巴巴公司也不甘寂寞，他也成为了 spring cloud 的一个核心组件供应商。


除了这两家公司以外，还有第三个地方，那就是等等。这里大部分是由 spring framework 这个开源组织来提供的，当然它还包括一些其他的开源项目。也就是说它由 spring 本尊以及其他的小公司或者小的开源组织共同组成了剩余的部分来补充 Netflix 和阿里巴巴组件库以外的功能。介绍到这里，相信大家对 spring cloud 的全貌都比较了解了。那我们接下来要引入一点思考题，我们知道 spring 


cloud 的版本更新速度是非常非常的快，它的大版本甚至是一年不到就推出一个小版本，更是三天两头就推陈出新。那面对疯狂更新的版本，对你的业务来说你如何做选择？

那我们技术人员都是喜新厌旧的，对不对？有了新，忘了老朋友见到新版，就想用迫不及待的想去体验它的一些新特性。面对这个情况，咱在自己的项目中如何做选择呢？你的选择是一个字跟迎头赶上，你出一版，我更新一版还是说要不起过以自己的业务稳定性为主。不去更新新版本，这一节，咱就从架构的层面对这个问题做一番探讨。 


OK 接下来我们再跟大家去理一下电商系统它的微服务化的构想。我们打算把这个电商系统怎么样的做拆分，分为哪些模块在这里，我们是以教学的目的做微服务拆分，所以并不会将我们的电商系统拆分的过细。比如说咱的商品可能就直接拆分成一个商品的微服务。不过大家要知道，在现在的企业级电商系统中，尤其是像一线的大厂里，它的一个商品微服务下面还有很多很多个十几数十个小的微服务。比如我以前在阿里工作的时候，在我们的事业部是做商品中心的，那我们的一个商品服务从商品发布到商品买卖，包括详情页等等，数下来真的有数十个微服务在背后支撑着。所以为了从教学层面顺利推进以及简化这个服务的复杂度，我们只会做相对简单的拆分。但是我也会跟大家介绍一下在阿里中它的整个电商的链路，比如商品中心，它的后台都拆分成了哪些具体的服务，让大家对当前业界的微服务架构也有一个比较直观的了解。


接下来我们就要去看一下本地的开发环境，我这里会给大家列出我所需要的依赖的中间件，包括我所需要用到的 spring cloud 的版本。那么在最后就是一个简单的课程小结和作业，这个作业其实就是让大家去检查本地安装的中间件以及开发环境等等，确保这一切都安排妥当，我们下一章才好正式进入实战的环节。 OK 同学们，那接下来就跟我一起走进 spring cloud 的世界，我们下一节课程再见。
