拆字当头怎么说。第一步我们要拆分 pom 依赖项。前面看到我们只有五个模块。但是在微服务当中我们拆分出的项目模块非常之多。那首先要面临一个问题，这个 pum 依赖怎么办？那咱说在微服务项目当中有这么一个，什么准则呢？每一个 pum 也就是我们每一个项目的毛九指引入自己用到的依赖，我们这里就不要引入多余的依赖。但是在这微服务改造的过程中，我们把这些依赖都做了很好的隔离。比如说我们微服务阶段，我们的 map 这一层只使用我在 mapper 当中需要用到的一些数据库操作的等等这些依赖项。那相对应的我们的 API 层也就是接口层，我也只引入自己需要的依赖，绝不去引入多余的依赖。因为你在大型的互联网项目当中，你经常要引用别人给你提供的一些价包。


包依赖冲突是非常常见的一件事，你引入的依赖越多，你造成包冲突的概率也就越大。碰到包冲突，我们又不得不去手动的排除这些冲突项，排报是一件体力活，也是一件靠运气的事。所以我们在微服阶段秉承一个原则，我用到哪些类，我就引用哪些依赖，那我没用到的，在当前的 module 里面就不要把它的依赖项加入进来。那接下来我们看看这拆迁通知里面还有哪些条款。


第二个是微服务模块的拆分。那这一个拆分是我们整个微服改造过程中最核心的一部分。变成这样一个原则叫力度适中。什么意思，我们这里会将微服务拆分的力度控制在一个可控范围内，根据项目的规模决定它拆分力度的粗细。其实咱技术同学有这样一个情节叫追求极致对不对？我们去做一个非常小的项目，我也想一口把它吃成一个大胖子，打造成像阿里 BA T 这种级别的互联网项目，我恨不得把这个项目当中的每一个服务都给它拆成独立的微服务。但是这样其实并没有必要。我们的服务拆分还是要秉承一个什么原则呢？力度适中就是它要适应你当前的项目规模。如果你是做了一个 BT 的商城，应用承载这么多的用户量，那你自然要把你的主链路好好规划好，把你的每块服务拆分的力度相对细一些。但是对于我们天天吃货这样一个教学级的项目，我们不要过分追求极致，我们微负拆分的力度，就做了一个相对折中的方案，把几个主要的模块独立拆分成了微服务。


OK 那我们接下来看还有一个拆迁的规则叫公共组件剥离。那咱说公共组件包括哪些呢？打个比方，咱前面在抛们包下用到的 utils 类这些工具类，包括一些公共的泡酒类，它是要在各个微服务模块之中共享使用的，所以这部分就要剥离出来，组成一个公共组件的依赖。


OK 那最后一条叫平台中间件隔离。这里我提到的中间件并不是说 rabbit MQ 或者是 Kafka 或者是 Redis 我指的是什么呢？是注册中心配置中心之类的公共平台组件。这些内容将在后面微服务的具体章节里逐步的引入。这些平台组件我们也会把它抽成一个又一个独立部署的项目应用。


OK 接下来我们一起看一下新的项目结构都有哪些组成部分。这个顶层项目我们把它叫做 foodie cloud 不过这里我们从教学的角度出发，就使用一个 Repo 来放所有的项目。**但是在真实的项目中，我们往往把每一个微服务放到独立的 git repository 当中**。在这个教学项目当中是为了同学们拉取代码方便，所以把它糅合在了一个 git 目录。

接下来我们看一下这个 git 目录下面都有哪些组成部分啊？我把它大致划分为了三个目录。第一个目录，commoncommon当中放什么东西？首先公共的组件，比如前面我们提到的一些工具类、 utils 类，还有一些公共的 portal 类，比比如前面我们做分页，包括返回给 UI 的 response 结构这部分通用的组件，那我们把它放到了 common 当中。除此以外它还负责什么呢？还负责去管理引入一些公共的 Maven 依赖项。


第二个组件就是我们的重头戏部分了叫 domain 从这个名字大家就可以看出，它是倾向于一个领域模型的概念。这个模块下面放什么内容，大家应该可以猜到，它是存放各个微服务模块的文件夹，比如商品模块订单模块还有购物车，这些都是放在 domain 下面的不同的子项目当中。


最后一个模块也是相当重要的。它名字叫什么？ Platform. 这个 platform 下面放的是什么呢？平台组件。平台组件就是咱前面提到的注册中心，还有配置中心，还有我们后面要学到的网关层组件 gateway 以及 hystrix 一些兄弟组件，像 turban 还有 hystrix dashboard 等等。那这些平台组件，我们就统统把它放到 platform 下面。


大家在下面一个章节去做微服务改造的时候，要特别留心一下平台组件，因为你想启动微服务组件，要求你要把某些平台组件先给它启动起来。而且这些平台组件当中有一些启动顺序的要求的，大家可要多多注意，不然可能就会问出很多问题。我这个服务为什么启动不起来，我为什么报这个错，当碰到这类启动问题的时候怎么办呢？不是求助老师也不是喊警察叔叔，是要到自己的 get 目录当中去看一下关于启动项目的说明。这里面会提到应该先启动哪些平台组件，然后再启动哪些微服务。同时这些微服务也有相互依赖的关系。 OK 同学们，这一小节我从比较宏观的角度跟大家介绍了拆分的项目结构和方向。在下一小节当中，我们来着重介绍一下整个项目将被拆分为哪些微服务模块。在这些微服务模块当中，它的文件夹目录结构都有哪些改变？好同学们，我们下一小节再见。
