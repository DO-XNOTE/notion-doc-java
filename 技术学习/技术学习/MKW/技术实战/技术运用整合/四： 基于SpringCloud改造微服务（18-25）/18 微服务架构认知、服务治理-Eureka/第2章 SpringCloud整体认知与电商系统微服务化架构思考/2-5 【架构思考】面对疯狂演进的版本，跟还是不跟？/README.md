spring cloud 的第一个版本，它的名称叫做 angel 天使的意思，它是诞生于 2015 年的 3 月份，这个日期可不是它的立项日期。这是它的正式版发布的日期。那在它这个版本发布之前当然它在内部孵化酝酿了很长时间，然后才在 15 年 3 月份面世的他，那可谓是 spring cloud 的开山利派的 10 组版本。


再往后， spring cloud 发布了一个新的版本，它的开头字母是以 B 开头的，叫 braxton 那他是在 2016 年的 5 月份发布的，距离上一次发布足足有一年零两个月。可能同学们觉得这个时间并不算太长，对不对但是老师要告诉大家，这已经是 spring cloud 中相距时间最长的两个大版本号了。
从 braxton 以后， spring cloud 的发展路线就完全驶进了快车道。它的下一个大版本 C 版，也就是 camden 它是仅仅在四个月以后，2016年的 9 月份就发布了正式版本。


然后 spring cloud 发布了著名的第一版，而这个版本 delston 是在业界广泛使用的一个版本。这个地板是一个蛮经典的版本。这里倒不是说它功能有多么精妙，多么完善多么好。只不过经过前面 AB C 三板的铺垫， spring cloud 已经有了非常广泛的群众技术。所以这个 delstan 版本出来的恰到时机，它也是早期应用非常非常广泛的一个版本。


即便是现在来看依然有相当一大部分比例的应用依然是跑在 dalston 上面。那在 dalstan 后面发布了一个 E 版 elshware 这个版本相对来说就稍显低调一点了，它是在 17 年的 11 月份发布了正式的版本。然后在一版之后是谁？那就是咱大名鼎鼎的 F 版分歧力。这个 2018 年 6 月份发布的 F 版应该是目前来说在近两年内应用最广泛的一个版本了。那转了一圈了，咱的课程用的是哪个版本呢？ we are here the greenwich 2019 年出品的最新版的 G 版 spring cloud 大家放眼望去，这每个版本之间的时间间隔几乎就是半年更新一个版本，这可不是小版本。同学们这可是 abcd E 以字母序开头的大版本号，这个更新迭代的速度太快了，简直就像那脱缰的野马一般。


尽管当前看 green witch 是最新的版本，但是想必过几个月可能 H 班就要面试了。不过同学们请放心，老师这个课程是有售后三包的。当 spring cloud 出了一个新版本之后，我也会同步的把咱的课程升级到新版本，然后再跟大家加一些章节，告诉大家这个升级过程中所遇到的坑。那咱领教了 spring cloud 在大版本上的发力程度。

我们再来看一看小版本，我们就以 F 版，也就是 finish 里这个版本为例，看一看这个大版本之下都有哪些小版本。首先 F 版先是 milestone 版本，这个 milestone 版本可不是正儿八经的 release 版本，它只是一个里程碑版本。这个里程碑版本相对酝酿的时间会比较长。


我们精品窖藏始于2017，直到发展到了 m9 这个版本在日趋成熟之后，它发布了 RC 版。那 RC 的意思就是 release candidate 但是这依然不是正式的 release 版本，那它总共发布了 RC one 和 RC two 那这个时间相对比较集中一些，是在 18 年的四五月份，这是为发布之前做的准备。然后就是我们定稿之后的 release 版，它是在 2018 年的 6 月份发布的。不过咱在项目当中引用的还真的不是这个 release 版是谁呢？是后面的 sr 正式版本。同学们可以看到咱引用项目的时候都会引用一个大版本号，后面跟着 sr one sr two 那这是相对来说稳定的，可以去在项目中做实际开发用的版本。这些版本总共历时到现在为止已经有一年了。同学们可能会奇怪，这里说的历史一年好像看上去现在还在开发对吧？没错是这样的。


尽管咱现在有了 green witch 但是 spring cloud 并没有停止之前的版本的研发维护工作。这每一个大版本都有一个持续的维护计划，并不是说新的大版本出来，上一个大版本就立马停用的，它还会继续维护那么一段时间。那我们这里就简单的跟大家说一下 spring cloud 的维护策略。

咱从表面上看， spring cloud 是从 abc D 按照字母序依次向后发布大版本，而且发布的非常频，大概在一年以内必定发布一版。不过这个发布过程并不是一条道走到黑的。什么意思呢？它会在多个版本线上并行开展。比如说咱现在用到的 G 版 green witch 正在如火如荼的在小版本上开始进行迭代。但是与此同时，下一个版本已经是在规划或者是在开发当中了。


如果大家到 spring 的官方网站上看它的文档，它其中有一个文档叫做 release note 就是它的内部版本的发布记录。你会发现它在正式 release 之前会有很多的里程碑版本。这些里程碑版本很早之前就已经在开发当中了。所以咱说前后的多个大版本其实是交叉迭代往前推进的。
条条大陆通罗马，只要认准 spring cloud 不管你用哪个版本，其实都是可以把你的微服务搭起来的。那这里讲到的是它的发布策略，接下来还有它的维护策略。即使大版本升级到了一个新的版本，那上一个版本它依然还会持续维护很长一段时间，直到它在官方网站上宣布，比如说 A 版或者 B 版在某年某月某日停止维护。


所以从这个角度来说，当大家看到 sprint cloud 发布了一个新的字母序版本，那有的同学会觉得老版本它可能不会再继续支持了，我要赶紧升到新版本。大家不必惊慌， spring cloud 超级专业的，它可不只是干这一锤子买卖。你即使升了新版，它的老版本依然有一个维护计划。就像早些年你的电脑使用的是 Windows XP 的操作系统。那后面微软又陆续推出了什么vista ，还有 win 7 win 8 win 9 win 11 系列后续版本，不过它的 XP 版本依然还是处于维护当中。不过老版本并不会这样永无止境的维护下去，这样就得永生了。对不对？所以 spring cloud 官方也会发布通知告知大家某一个老版本将会在什么时候停止维护。


OK 以上便是它的发布和维护计划，我们接下来往下看。咱说有程序的地方就有 bug 对不对？有些开玩项目并不那么靠谱，它对用户提出的 issue 或者是 bug 视而不见。但是咱 spring cloud 这里可不一样，它的社区是非常非常积极的。甚至说如果你发现了 spring cloud 的 issue 你可以去提交源代码，你去来 fix 这个问题，然后提交给 sprint cloud 的 contributor 让他们来审核。所以同学们甚至还有机会为 spring cloud 贡献代码的。这是一件非常荣耀的事情，它的整个开源的社区文化非常的 open 而且也非常的积极。


OK 那最后我要跟大家讲一讲这几个大版本之间的升级感受。什么意思就是你从一个老的版本升级到一个更新的版本，比如说你从 B 版升级到 C 版或者从 D 版升级到 F 版。那从我个人的感受来说，如果你升级一些早期的版本，比如从 B 版、 C 版移步，把它跨版本升级到 D 版 F 版，那这里面的坑可是要把人给淹死的，里面水非常非常的深。


这里边代码改动就不说了，它还有配置项的改动，还有配置文件里面的属性名称的改动，这些小细枝末节的改动你很容易把它忽视掉，如果想全部摆平，可是要花一番功夫的。不过这个情况现在已经大大改善了。如果咱是从比较新的版本升级上来，比如说我们从 F 版升级到 G 版，这里面的改动其实是相当少的，轻轻松松就可以升级到最新版本了。


好，那我们这四个象限的内容都跟大家看完了，大家心里有什么感受。 sprint cloud 官方对待这个升级维护的态度可谓是尽心尽力，非常的积极。所以我把这个情况概括成四个字，来形容 spring cloud 如今的精神面貌叫什么蓬勃发展。接下来我们就来聊一聊 spring cloud 的升级版本的利弊如何来做权衡升级。新版本有利自然也有弊。所以我们要把利弊两方摆在一个天平上面做一个衡量。那我们先来衡量一下追新版本都有什么显而易见的好处。

第一个，显而易见的好处就是更新的版本它会包含更新的组件，或者它的已有组件当中会提供更多的新功能。那这些新内容可以帮助我们玩转新的业务场景。这个因素是我们做技术升级的主要考量标准，这种情况其实不单只体现在 spring cloud 上面，就连咱们平时使用的中间件包括 JD K 都有这样的好处。比如你从 JDK 7 升级到 JDK 8，那你可以享用 JDK 8 中很多非常酷炫的功能，有的功能可以让你在并发操作 concurrent 当中更游刃有余。那有了操作，可以让你的代码逻辑变得更加优雅，比如莱姆塔表达式。


那咱升级版本的主要动力就是去享受新版本提供的新的业务功能。当然我们也要考量一个标准，你的新功能对你的业务是不是有帮助？因为从老板的角度来说，我不在乎你用什么技术，我只在乎你把业务给搞定。所以我们不光只单单从程序员的角度出发，而新版本有潜在的好处，你也要说服你的老板或者是我们叫你的业务部门。所以为了获取必要的研发资源来让你投入到新版本的升级当中，你可能也需要编造一个美丽的谎言来说服你的领导。


OK 除了这个点以外，咱还有一个显而易见的好处是 bug fix 尤其是对于 spring cloud 的这种开源组件来说，全世界开发者的眼睛都盯着你，即便一个微小的 bug 也很容易被查找出来。相对来说更新的版本意味着更少的 bug 以及更稳定的特性。


那除了 bug fix 以外，当然还有它的安全性补丁。因为我们知道现在的黑产包括黑客真是非常非常的猖狂的。而且有很多傻瓜式的专业找 bug 的工具，你只要输入一个网站，那这个工具后台会分析你的网站漏洞。那从这个角度来说，如果你的安全性不能够得到保证，那么对你的网站来说可能会引发一些灾难性的问题。那除此以外还有一个可圈可点的特性，那就是技术栈的升级所带来的可扩展性的增强。我们过去十多年中有非常非常多的框架诞生什么 struts 1， struts 2 各种框架。那在这滚滚历史的车轮当中，为什么活到现在的是 spring 呢？ spring 从发展之初到今天，它的可扩展性都是做得非常非常好的。那 spring cloud 的各个组件也秉承了这个传统，我们可以看到很多组件它都留有可扩展的接口，那让我们可以非常自由地添加自定义的逻辑。那除了这些显而易见的优点以外，咱来看一看它的弊端。


第一个可以想到的弊端就是你的替换成本，把你的整个项目的技术栈从老技术应用到新技术，这需要人力的投入，它就牵扯到开发资源、测试资源以及你的组件升级是否可以带来额外的运维成本。比如说你曾经是一个单外包的分布式集群，你想把它拆成微服务那一个显而易见所带来的运维成本提升，就是你的机器数量肯定会增加。虽然咱在云端大多使用的都是虚机，不过基于微服务架构的系统，它的运维成本和复杂度相对于传统应用还是要高一些的。


除了这一点以外，我们还要考虑上下游系统的兼容性。比如说某一天你做了一个系统升级，采用了 spring cloud 的 eureka 作为服务发现框架，但是上游的系统它未必买账，它可能还是基于最老式的 web service 形式的调用，甚至有的更老的系统它还采用 SOAP 协议。所以说你对业务模块或者是业务模组的升级并不是一个一拍脑袋就可以上的，你要有一个整个全局统筹的规划来协调上下游的应用。


除了这两个以外，我们就要谈一谈业务了。你的升级对你当前业务都有什么影响？我们希望你的升级对当前业务一定有一个积极的印象。打个比方，大家都用过消息中间键对吗？假如咱用的是非常老的版本，那可能只有非常基本的消费者、生产者的功能。那如果大家升级到新版本，它可能添加了像死信队列、延迟消息这种非常酷炫的功能，那我们自然可以利用这个特性来完成更复杂的业务场景，那这是业务团队比较愿意看到的。不过如果我们哼哧费老鼻子劲把自己的组件升级了，但是对业务却没有什么影响。


你的老板疑问，小张，你让整个团队花了一个星期升级这个系统带来了什么好处呢？你总不能告诉老板，为了技术人员的信仰和追求，我们很多时候做决定，不能单单从技术的角度出发，一定要从业务这个角度来思考一下。因为真正拍板的人大部分不是技术人员，而是业务团队。
那咱看了这个升版的好处和升版可能带来的弊端，如果让同学做个抉择该怎么办呢？站在天平的两端一样的为难。唯一的答案，我们要结合业务需要和技术需要来做这一个决定。所以这个意思就告诉大家， you must know what you are doing there must be a reason for doing this 我们要为这个决定找一个借口，你也要非常清楚为什么做出这个决定，不管他是从业务上的需要，还是从技术上的考量。


那么在这个过程中，我们一定要杜绝一个技术人员非常容易犯的错误叫什么呢？为新而新，为了新版本而升级新版本，你甚至不知道这个新版本能给你带来什么好处，一拍脑袋就上了，这种价值观可不可取。讲了这么多，同学们心里可能也在嘀咕。老师，那这个结论到底是什么呢？这里咱就先呈上我推荐的方式，用当今非常流行的知乎体来问问题，就叫做你如何看待项目技术的升级。

谢邀我这里先上答案，我的观点是要更新，但不要追星。咱面对项目的技术栈升级，我这里再给四个小建议。第一个是我们需要在团队内或者是部门内建立一个 tag stack review 的机制，就是每隔半年度或者每个年度我们要有经验足够丰富，并且非常了解你们业务的架构，是来对你的技术栈进行一个 review 很多团队在项目成型了以后，这个技术栈选型就成了一滩死水，没人会去管，也没人会去有这个动力让它持续更新。那我们建立这种 tag stack review 的机制，就是从部门层面甚至是说公司层面推进你技术的更新换代，逐渐淘汰掉老旧的系统，还有过时的技术栈。


除此之外，面对 sprint cloud ，因为它的版本更迭速度非常快，新的功能也不断的推出，我们要持续关注新的大版本，这里面就包括了它所提供的新功能以及新的组件，结合你的业务场景，看一看这些的新元素能否助力你的业务更好的发展。一旦你决定使用了新版本以后，我这里有一个建议，只采用 sr 的发布版，这里是针对 spring cloud 来说了，我们只采用 sr 绝对不要冒进去使用 milestone 版本或者是 RC 版本只认准 sr 发布版。那在你规划升级过程当中，我们要做一个 POC 先行 POC 的意思就是 proof of concept POC 相对来说是一个轻量级的可行性验证的阶段。在确认方案可行的基础上，我们改造完之后一定要记得进行全链路的测试，并且制定好完善的回滚计划，不怕 1 万，只怕万一生产上没有小事故。


OK 同学们，那到这里本节的内容就结束了，我来带大家稍微回顾一下。咱前面讲到了 spring cloud 疯狂发布的新版本，包括它的大版本更新节奏以及小版本的更新策略。后面又跟同学交代了 spring cloud 的维护策略。就是它在大版本上是由多个版本同时迭代。然后每推出一个大版本之后，上一个版本依然会维持相当长的一段时间的维护。
