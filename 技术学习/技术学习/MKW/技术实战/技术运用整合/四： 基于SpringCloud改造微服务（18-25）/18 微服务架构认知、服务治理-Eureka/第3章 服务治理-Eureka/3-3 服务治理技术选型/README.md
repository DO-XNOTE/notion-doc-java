hello 同学们好，我们在这一节将讨论服务治理的技术选型。在正式进行技术选型的讨论之前，我先来带大家看一个 cap 大定理，cap待会儿选型的时候会用到这里面的理论知识。所谓 cap 其实就是三个单词的首字母拼写而成的，这个 C 的意思是 consistency 也就是一致性。那这个词相对于大家来说应该都非常熟悉了，他是说在分布式系统中所有的数据副本，那这个副本不是让你下副本，它的副本意思是指在分布式环境中分布的所有节点。比方说我的数据库有主从备份，我的 Redis 也是一个集群。那么在这个集群当中，每一个节点它可能都保存着一份数据的副本。那我的一致性在这里的含义就是说在同一时刻，同一个值在每一个副本中它的数值都是完全一样的。也就是说这个一致性实际上指的是数据的一致性。

OK 说到一致性，我再来跟大家聊一点题外话啦。大家知道强一致性对不对？弱一致性，最终一致性很多同学分不清楚这三种一致性的区别。 OK 我们这里来跟大家举一个简单的例子，帮助大家理解一下。我拿强一致性来举例，我们现在对系统中的一个数据做了一次更新操作， update 对不对？那么在这次更新操作以后，所有的后续的访问请求都能取得到这个更新过后的值，也就是说一次更新终生有效。那么我们叫这种情况叫强一致性。那弱一致性又是什么含义呢？如果说在更新过后，后续的访问请求有一部分仍然拿不到，更新过后的数据，或者说索性全部访问请求都拿不到。那这种情况就叫做弱一致性。最终一致性是什么含义呢？它的意思是说经过一段时间以后，在某个未来的时间点，我能保证在这个时间点以后访问的数据它都能取得到更新过后的值。也就是说在这个时间点所有的数据副本中的值都会保持一致。那这种情况叫做最终一致性。


目前来说，大部分的超高并发的应用，尤其像我们这个互联网行业的应用应该都是使用的最终一致性居多。那么我们这个 C 在这里指的是哪一种一致性呢？我们的 C 在这里记住了，指的是强一致性。接下来我们说这个 A1 的含义是 availability 可用性。


可用性是指不管是风吹还是雨打，你始终要保证自身的服务处于一个可用的状态。大家知道一般来说我们用什么指标来衡量系统的可用性吗？宕机时长对不对？宕机时间占全年总服务时长的比例，那么这就是衡量可用性的一个标准。大公司对自身服务的可用性都有一定的要求。比方说像电商业务的核心系统，我们一般要保证 5 个 9 的可用性，99.99，那这个数值相当不容易达到。他要求你全年的档期时间要小于 5 分钟。那即便像阿里集团有一些业务部门一次小的故障，都有可能把全年的档期指标一次性用光。那这里并不是说不允许你宕机，比如阿里集团的主链路的应用，随随便便一个微服务几千台甚至上万台服务器，让所有机器不当机，那是完全不可能的。


常在河边走，还哪有不湿脚对不对啊？但是这里 availability 的意思是说，你即便有部分机器处于宕机状态，你仍然要保证整个服务是处于可用性的。大家明白吗？也就是说除非你所有机器全部团灭了，不然你要想方设法让剩余还在继续工作的机器保持这个服务处于可用状态，并且你要尽最大努力将故障机器从故障状态恢复过来。


那最后我们来说，这个 PP 可能大家之前都没有听说过 here the he stands for partition tolerance 分区容错性我来举一个例子，大家知道现在大型的应用都有异地机房来做容灾对不对？两岸三地多机房，那这些机房之间的数据其实是在互相进行同步和备份的。那假如在某两个分区之间，这个数据同步的通信通道被切断了，那么这两个机房之间的数据停止了同步对不对？这样就会产生数据不一致的现象。


而对于支持分区容错性的系统来说，他就必须做出一个艰难的二选一的抉择。那假设这时候他选择了坚固可用性，那好两个机房继续提供服务，但是数据会又不一致。你请求 A 机房和请求 B 机房得到的数据可能是不相同的。那如果我不选择可用性而选择一致性，那你就要等待阻塞，在这里等数据同步恢复过后才继续提供服务。


所以说作为分区容错性的系统，必须在可用性以及一致性之间做一个抉择，不可能二者兼得。所以说这里我们就引出了一个 capcap 大定，也叫 cap 原则，它指的是分布式系统中一致性、可用性、分区容错性这三者三个要素最多只能同时实现两个不可能三者兼顾三选二。但是实际上对于我们分布式系统来说，也就是在一致性和分区容忍性或者可用性和分区容忍性这两种选择之间二选一。那同学会说了，我既然是三选二对不对？我可不可以选择一致性和可用性。


C 和 A 我不要这个分区容错性实际上不太可能，因为我们面对的是一个分布式的系统，而分布式系统中你的服务节点数量一定是大于或者等于 2 的，你不再是一个单体的应用，那你就不得不去考虑分区容错性。那对于单体应用来说，你倒是有可能达到一致性和可用性。比方说我是一个服务器连接一个数据库，那对于这个数据库的读和写，那我完全可以既保证一致性又保证可用性。但是当你后台的这个数据库从单库变成了多库或者是主从，那你就不能保证这两者可以兼得了，你需要必须考虑分区容错性。

第一回合我们先来看一下一致性在这三种组件中的实现方式。 eureka 和 console 是同样使用的 ap 也就是可用性分去容错性这两种弱一致性的解决方案。那 next 在这一点上做的就比较牛了，它既知识可用性、分区容错性这种组合同时也支持 CP 也就是一致性和分区容错性组合，我们可以在 ap 和 CP 中进行 2 选一自由切换。


那第二回合我们关注一下性能上的比较，在这一回合胜出的是尤瑞卡和耐克斯。那尤瑞卡它快在哪里？我们将在后面的小节中，通过对尤瑞卡服务治理功能的深入学习带大家探究。那这里我们先重点关注一下康色 console 这个 man 它是事出有因的，因为它用的是 Raft 的协议。那我们知道 Raft 是一个基于 leader 角色进入写入操作的协议，那他要求必须是过半数的节点都写入成功，我才认为这个服务注册流程它最终是成功的。那既然有 leader 那我们就有一个选举 leader 的过程，选举 leader 大家可以把它理解成是一个任主人的过程。因为你只有 leader 才可以进入写入操作。但是当 leader 挂掉的时候，那我就要进行一个重新选举，再重新选举，整个期间场所是处于一个不可用的状态。


所以大家可以想象它为什么慢对不对？第三回合，我们从网络协议上做一个比较，那有瑞卡支持的是 HTTP 协议， console 比它牛那么一点点，它多支持了一个 DNS 协议，nicos就是暴罗万象，既支持 HTTP DNS 也支持 UDP 协议。那这几种协议相信大家应该都比较清楚，我这里就不展开来介绍了。
最后一个回合，我们从应用广度上做一个比较，那有瑞卡是当之无愧的王者，目前的主流康宋目前有那么一点非主流，因为它的应用并不广泛，而奈克斯它的应用是一个稳中有升的状态，但是它尚待观察。那我这里为什么说奈克斯尚待观察呢？其中一个原因是它面向市场开始使用的时间并不长，那它还需要更广泛的企业使用它以及开源社区的力量来让它变得更稳定，它需要时间来证明自己。


第二点，我们知道 negos 是阿里巴巴的出品，那我个人对老东家的这款组件，感觉它更像是为了强推 double 做的一番努力，因为它其实也能跟 double 做很好的继承。那所以我希望耐克斯并不仅仅是为了强推集团内部开发的组件，而是能证明自己，它可以更好地服务于 spring cloud 生态系统。所以从这一点上，我希望时间可以让奈克斯证明他自己的企图性。
