第一个问题，假如说我们现在的单中心节点破该了怎么办？那意思就是它挂掉了怎么办？那同学们知道怎么办吗？

说实话，并没有什么好的办法。那如果你硬是要我给出一个解决方案，我只能说是尽快把这个挂掉的节点给她抢救回来，让他继续工作。


因为我们在单中心的情况下，一个注册节点挂掉了以后，所有已经注册的服务，它依然会基于他们已经拿到的这个服务注册表来继续工作。那我们讲这个服务注册表可能跟真实环境是有出入的，就是有不同。比如说有的服务它下线了或者新的服务上线了。那这些变化并不能通过注册中心再同步到所有节点上。


那么新上线的服务我们说你调用不到为什么？因为你不知道他的 IP 地址，你无法通过注册中心拉取新的 IP 那如果说老的服务下线了，对你来说你还会继续发起调用，因为它在你的服务注册表中依然是存在的。那我们想随着时间的往后推移，这种不协同的情况是不是会变得越来越严重？大家在分布式系统里面一直强调这样一个词叫什么高可用。那我问问同学们，你们对高可用有什么理解啊？大家可能都知道说从字面意义上，它就是不管什么情况保证系统的可用性。那对于说这种异常情况，我们来怎么理解它，我想它有两层含义，那第一个维度我认为是故障容错。我们拿阿里集团的淘系应用来举例子。


其实服务降级在淘系应用中就是一种故障容错的手段。比如大家经常在网上购物发现商品详情页都有评论，那如果某一天这个评论模块不可用了，也就是 API 超时了，那我们是让整个商品详情页面等待评论模块恢复服务之后再响应用户还是说我们这里写一个降级流程？那降级流程是什么？就是我忽略评论模块，我继续渲染整个商品详情页面，那当然是降级了。


因为对于我这个下单的流程来说，有评论和没有评论是一种用户体验上的差别。但是如果从主链路的角度来分析，它并不会影响你的下单流程，所以我们这里采用一个降级的容错方案。那除了容错以外，还有一种手段是故障恢复。我们这里再拿陶信用来举例子，因为大部分陶信用都是部署在阿里云的。那比如说当前我们有一台机器， CPU 和内存使用都过高，双高，导致他已经无法继续的响应请求了。那这时候阿里云的运维脚本会怎么样？他会侦听到这个情况，然后尝试着去自动重启这台机器，把这台机器从故障状态恢复成正常状态。所以我们不能 100% 保证没有异常情况，关键是我们的系统如何处理这个异常。但如果你能处理好所有的异常情况，保证你的机器在各种各样的异常面前仍然能继续提供服务。我们就说你这个系统是高可用。

目前它还是一个单体的节点，一个注册中心后面新上线的服务提供者还是连接到这个注册中心进行服务注册。这一步没有变化，服务调用方也就是服务的消费者也是通过这个单中心进行服务，发现获取到服务注册表。往后的事就有那么点变化了。给他加了一个框，这什么他要变身，接下来他就不再是一个人了。他的好兄弟上线了，那这哥俩感情好，要互相说一说悄悄话。



所以他们之间有一个什么数据备份对不对？数据同步的方案，那你的数据我可以同步给另一台机器。从此以后，这些调用方服务提供者就不再是连接到一个节点，而是连接到集群中配置的所有节点。那么我们想，如果一个节点因为故障下线了，那这些所有的服务节点会尝试连接高可用注册中心集群中的其他可用节点，那数据会不会丢失？并不会注册中心之间有一个数据互通的手段，数据备份、数据同步，所以你在另一个节点上依然能获得全量的服务注册表。 OK 那接下来，我们就到代码里去亲手搭建一个高可用的注册中心集群架构。
