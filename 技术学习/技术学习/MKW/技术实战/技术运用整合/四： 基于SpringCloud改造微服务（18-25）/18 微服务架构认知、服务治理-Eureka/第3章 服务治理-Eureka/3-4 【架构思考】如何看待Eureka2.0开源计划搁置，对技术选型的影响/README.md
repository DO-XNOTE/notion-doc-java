我这里举一个例子，我在一年一二年那会儿的时候，当时还在一家传统的老牌的 it 外企做一个项目。那段时间前端技术推陈初心特别快，我想零几年的时候我们会用 G query 了不得了。对不对？但那时候在一二年短短两三年时间， angular gsxggs 还有谷歌的 gwt 前端的新技术如过江之青一样退出市场。当时我们那个项目就是看什么技术新用哪个开发人员很高兴。可以使用新技术了，简历上又多了一个技术可以聊，所以他们对这个研发也非常有兴趣。先是用了 angular GS 再换成了 exdjgs 最后换成了 gwt 以为背靠谷歌好乘凉是不是？那结果现在 extjs 和 gwt 早就凉了零下好几度凉到。但是回过头来，我们使用的这些新技术当时有没有对项目产生实质性的帮助？没有。所以我的观点是，如果我们的新技术不能更好的支持你的业务，你可以做 POC 也就是做调研验证可行性，做技术储备或者是备选方案。但是尽量还是使用目前业界主流的开源的项目，不要为了新而新。


那第二个考量的观点是未来研发支持。那这个是什么意思呢？我们说这个研发支持包括社区的开源力量或者说商业公司的支持，这决定着你这项技术在未来的生命活力。比如说你使用一个开源爱好者自己开发的一个服务治理组件，你是使用它来做微服务，还是选择 I patch 基金会，它顶级项目 spring cloud 而想答案不言而喻，当然是 spring cloud 所以我个人建议活跃的开源组织的项目大于商业公司，他自己维护的项目再大于个人开发者的开源项，我们选择的时候首选开源组织的项目。


那第三点就是团队的技术栈和学习成本，这也是不得不考虑的一个问题，你的团队对你将要使用的这个技术的熟练程度，其实一定程度上也决定着你这整个项目的研发速度和研发周期。那我们说经济基础决定上层建筑，所以你的技术选项一定不能脱离自己的实际团队。假如你让一个团队的 PHP 的程序员去接触 Java 的项目，让他转型做 Java 比如像淘宝当年从 PHP 转型成了 Java 如果这个项目交给你来做，我们从事这么多年 Java 的人都知道，那简直有点自寻死路。这里水深，一不小心可能就要淹死了。


所以当年淘宝它的转向实际上也是从外面吸纳的很多 Java 的人才，甚至是高管才顺利的把PHP。
的架构过渡到了 Java 那我再举一个简单的例子，假设你们现在要做一个前端的项目，前端都是 javascript 那你是用 React 还是用 extgs 那又是你选行李选哪个 React 对不对？市场上 React 学习者和开源社区的活跃程度远远大于 xggs 而且 React 的学习曲线更加平滑。


OK 那最后一个维度，我们来聊一聊这个替换成本运维成本。这什么意思呢？很简单，比如说我们拿网关层组件举例子，我们都知道目前来说功能最完备的网关测组件是什么？f5，我们后面的课程会提到这个f5，假设你现在项目用 ngx 你想要替换成f5，那运维成本上来说， f5 是软硬件结合的网关层，那它的运维成本远大于单纯的这个软件层的 NGINX 而且 f5 的许可证费用非常贵，那你就要做一个权衡，你要替换的这项技术，它给你的项目给你的公司带来的这个利润是不是可以覆盖掉这个替换成本？


我再举一个例子，我在一五年的时候参与了一个非常大规模项目的微服务改造，就是从传统的分布式集群改造成微服务的架构。那我是全程参与，这是一个技术替换过程。我们花了多长时间，整整三个月不接受新需求，全力把项目改成微服务架构，这个成本大不大相当大。但是以当时这个项目的规模来说，如果不替换成微服，那么在未来项目的迭代速度将陷入泥沼。所以尽管说这个成本是非常大的，但是它所能带来的远期的效益是很可观的。那么我说这个替换就是很值得的。所以我们大家来决定是否使用一个新技术，一定是从多个维度考量的。千万不能说这个技术目前来说非常火热，那我就去用它。
