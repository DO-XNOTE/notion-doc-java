**金丝雀测试**

金丝雀测试是一种轻量级的灰度发布策略，在做版本发布的时候，先将新版本部署到一台服务器上，这台服务器我们就称之为“金丝雀”，区别于其他部署了老版本代码的服务器。金丝雀开始运作以后，经过一定时间的孵化期，其各项指标（如HTTP请求错误率，响应时间，CPU使用率等）都在正常情况下，这时候可以逐渐把新版本部署到其他服务器。在实际应用中，我们可以通过配置中心的“功能开关”来代替部署发布流程，先将代码发布上线，功能处于关闭状态，再通过配置中心将开启功能的配置推送到一台服务器上，以此来完成金丝雀测试。一个例子是在商品中心提供的商品搜索服务上，商品搜索分为前端搜索和门店搜索，其中前端搜索是专供给app访问，承载手淘的导流以及独立app端的用户请求，这部分是通过服务淘宝的搜索接口来完成。但对于后端门店搜索来说，因为主要是供产品和运营人员使用，对时效性要求较高（尽量避免读取缓存带来的延迟），初期业务规模和运营团队规模较小的情况下，我们直接通过数据库读写分离，从读库中查询商品。当业务量和运营团队逐渐扩大后，数据库指标逐渐感受到了压力，我计划将业务切换到淘系的Open Search服务。在搜索接口里我们引入了enable_open_search开关，在代码全部上线的时候处于关闭状态，之后使用Switch将其中一台服务器的开关打开，观察接口响应时间、数据同步情况无异常后，逐渐打开所有开关。


如果有同学希望深入了解金丝雀部署方式，可以参考这篇文章https://martinfowler.com/bliki/CanaryRelease.html 其中摘录了《DevOps in Practice》中有关金丝雀测试的章节。


**人工熔断 - 记一次服务雪崩的经历**

开关不光可以用作金丝雀测试，还可以作为一种简易的人工熔断手段。

一个例子是商品中心的类目加载器。商品类目是淘系商品的一个主属性，常用在搜索服务中，用户可以将商品类目作为一个搜索过滤条件。在做系统改造的过程中，我们接入了天猫供应链系统，其中商品类目的加载服务就由这套供应链系统提供。考虑到这套系统是首次上线使用，天猫出品也未必都是精品，尽管测试相当充分，但是在超高访问量的情况下难免出现问题，因此我在商品中心的类目加载器上做了一道隔离防护，除了自动熔断机制以外，又加上了一个可以人工干预的熔断开关disable_homes_category

，当开关开启的时候，通过降级逻辑从本地缓存获取商品类目。一来可以保护类目加载器，二来可以保护依赖类目数据的下游服务（比如商品搜索），避免发生服务雪崩的现象。异常状况在上线后不久发生了，由于初次对接新的系统，需要将所有在售商品全部导入天猫供应链系统，大规模的数据导入拖垮了供应链系统的负载能力，导致所有对接该系统的接口全部超时异常，一次服务雪崩就在极短的时间里发生了。在接到鹰眼系统报警后，我立即开启人工熔断开关，在第一时间建立了隔离地带，阻止了雪崩现象到商品中心的蔓延。有人说为何不依赖自动熔断，自动熔断仍然会时不时发起重试（还记得Hystrix里的Half-Open吗？），重试意味着向远程服务发起调用，尤其对于高频调用场景来说，越是在雪崩的情况下越是要减少调用次数。因此可以通过强制开启熔断器（比如Hystrix 的 hystrix.command.default.circuitBreaker.forceOpen 属性），或者通过上面这种人工熔断开关的方式阻止重试，等待雪崩情况恢复了以后，再逐步打开服务。

**香烟买卖 - 推送JSON格式业务数据**

这是一个比较特殊的场景，大家有没有从网上购买香烟的经历？我想没有吧，因为国家规定任何企业和个人不得通过信息网络销售烟草专卖品。但是新零售业务怎么能少得了那手指淡淡烟草味道？

我们提出了“代购香烟”的途径，即每一家线下渠道都有一个对应的拥有烟草许可证的供应商，用户的下单是通过“代购”的途径从线下进行购买，而非网上售卖。这就对商品详情页提出了一个需求，需要对不同门店的香烟商品，指定一个供应商的名称，在详情页展示出来。比如“该商品由xxxx代理销售”。由于代理商经常变更，所以需要有一个便捷的渠道，让运营和采购团队可以随时做变更，添加新的供应商或者更换供应商描述。我在具体实现的时候使用配置中心完成了这个功能，即配置中心使用一串JSON格式的字符串，以“shopID:供应商名称”的格式描述线下渠道和供应商的关联关系，比如：

{

"100": “静安烟草公司”,

"200": “浦东烟草公司”

...

}

当需要变更某个线下渠道对应的供应商时，运营团队可以直接在配置中心完成变更，并推送到所有服务器（为了防止运营变动其他数据，配置中心只对他们开放了这么一个配置属性），这样的好处就是不用额外做一个专门服务于运营的代理商编辑页面。后台商品详情页接口只要检查到当前商品属于香烟类目，就会根据其所属的门店ID，到这串JSON里查询对应的供应商信息，然后输出到商品详情页。

**小结**

这一节我们讲了几个配置中心的实际应用场景，但在实际项目中远不止这些，还要靠同学们灵活运用配置中心各种功能，让它助力更多的业务。
