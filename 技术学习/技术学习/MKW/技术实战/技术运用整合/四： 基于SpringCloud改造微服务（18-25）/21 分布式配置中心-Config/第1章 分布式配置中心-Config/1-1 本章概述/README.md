config 这个组件是 spring cloud 的生态系统下唯一一个看了名字你知道它会干什么的组件，仅此一家，别无分店。跟前面那些名字起的像牛鬼蛇神一样的组件比 config 这个名字传达出来的气质就是低调有内涵。大家有没有注意到这幅图上有一个很大的变化的？ Netflix 这个单词去哪了？为什么 Netflix 不见了？因为 config 组件是直接挂牌在 spring cloud 项目下面的，它并不是 sprint cloud Netflix 组件包的一部分。


咱前面提到过 config 是一个打辅助的角色，所以它的功能学起来也比较容易上手。好，那接下来我们看看这个辅助怎么打好自己的位置。纵观全局，开启上帝视角，我们来看一下本章的课程目录。那我们开局先是平稳过渡，我来跟大家介绍一下配置中心都有哪些功能，微服务中有哪些应用的场景。接着节奏进入到了前期，我们跟大家介绍一下 config 的核心功能以及相应的应用。紧接着咱要练一练手了。这里跟大家介绍直连式的架构模型怎么来搭建，它分为三个部分。第一部分我们要借助 github 来存储相应的配置属性。所以这里带大家去创建一个 github 的 Repo 然后在 Repo 中定义某些特殊格式特殊文件名的 practice 文件。


紧接着第二步我们要搭建一个 config server 应用，它和 eureka server 非常相似，也是一个中心节点式的服务应用。好。第三部分我们要将应用方连接到上一步中配置的 config server 上。这样一来，我们就可以借助这个中心节点 config server 来帮所有的应用方拉取到对应的属性文件。整个流程就是 github 存放文件， config server 拉取文件。然后应用方从 config server 里面读一条龙服务，再往后我们的节奏进入到了中期，这里就要啃一啃硬骨头了，都有哪些个硬骨头给大家啃当然是源码阅读，每一章的源码阅读都是相对困难一些的章节。在这个部分跟大家介绍一下资源文件是如何加载的。这个资源文件指的就是 git 上面的资源文件。


再往后我们要介绍一个非常常用的功能，那就是参数的动态刷新。因为咱这个参数定义不是一锤子买卖，那它中间可以会发生变化。所以这里各自的业务方可以根据一定的条件，在某个合适的时机从配置中心上拉取到更新的参数。当然这一节不光有图文部分，我们也会通过 demo 跟大家展示如何搭建一个动态刷新的服务中心。然后下一节我们的主题就叫展望未来，看看我们如何对这个 config server 进行高可用的改造。这里主要分为两个部分。第一个部分我们来设想一下，当单中心的 config server 如果挂掉了，那我们的服务节点该向哪里读取数据呢？大家可以想一想，高可用改造有哪些思考的方向？咱是不是可以借助前面学过的某个组件来助力 config server 实现高可用化呢？没错，这就是第二部分， demo 里面要介绍的了，我们借助 ureka 来实现高可用配置中心的架构。其实不光 config 后面还有很多组件都会借助 eurika 实现高可用的架构。 eureka 在整个 spring cloud 的生态系统中，真的称得上是一个万金油。


再往后节奏进入了后期，这边安排了一个架构思考的环节，思考什么呢？思考两个问题。第一个问题叫总线式架构展望。我这里为什么用展望这个词呢？因为这一章我们并不会涉及到总线式架构的落地。这个功能我们放到后面一个章节，也就是 buzz 章节中再跟大家展示好。第二个要思考的问题就是我们的分布式配置中心它可以应用在哪些业务中？这里我也会跟大家分享一下我在淘系业务中是如何使用配置中心完成某些特定的用户场景的。 OK 接下来一个章节，咱的配置文件中是不是有可能有一些隐私的信息啊？那怎么办呢？我们是不是需要对它进行加密啊？我这里挑选了一个简单的方式来实现，用对称密钥对配置项进行加密以及解密。剩下的两个环节大家都非常熟悉了，它们分别是什么电商项目改造，紧接着课程小结和作业。 OK 这就是全部的课程目录了。尽管 config 是一个打辅助的章节，但是并不代表着他是打酱油，同学们还是要提起精神跟我进入接下来的正课学习。好，那我们就在下一个章节再见
