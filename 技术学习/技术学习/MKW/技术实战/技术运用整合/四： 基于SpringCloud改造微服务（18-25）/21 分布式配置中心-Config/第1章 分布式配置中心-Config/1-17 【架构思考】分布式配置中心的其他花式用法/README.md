好了，所谓三碗不过岗，咱先来喝这第一碗酒。好。第一个场景咱叫环境隔离是什么意思呢？


比如大家在做项目的时候，肯定有各种各样的环境，比如咱本地的就叫开发环境，或者我们有一个日常的集成测试环境，我们可以把它叫为日常环境。那在上线以前，所有的应用都会放在预发环境里面执行一把大规模的集成测试。那最终上线了，咱就会叫它生产环境，这每个环境之间肯定多多少少有那么点配置上的区别。


咱就拿阿里系的应用来说好了，比如这里是一个 config server 助力，在此在阿里的部署架构里我们把整个上线流程抽象成了三个环境，

它们分别是日常环境、预发环境和生产环境。就像前面说了，日常环境是一个完全隔离的测试环境，它访问的数据库都是与生产环境隔离的。那预发和生产环境实际上连接的是同一套数据库，也就是说预发环境是在上线之前最后一步做集成测试的地方。那么它连接的数据库也是线上环境的，这三个环境它的配置项目一定是不同的，但是相同的是它们都要通过 config server 来拉取配置文件。


对吗？比如咱就对这个接口超时的调用来说，这三个环境肯定是不一样的。比如日常环境，咱宽进宽出，没有这么高的高标准严要求。所以它的 time out 超时时间，我们可以把它设置成 10 秒钟，再到预发环境它是 5 秒钟。那么到了生产环境，咱对这个接口的调用时间就有一定的要求了，如果你超过一秒钟，我就不能再等你了，否则后续的请求那就处理不了。所以这个超时时间在三个环境中是不同的。那怎么来办到我们知道在配置文件当中肯定不是写死的这个 timeout 所以这里应该是写了一个我们叫 placeholder 的东西，就像美元符标注了一个 timeout 的时间，从外部把这个属性再给配置过来。从我们前面学到的 config server 的知识来说，我们对  每一个环境应该生成一个单独的配置文件，所有这个跟环境相关的配置项都放到这个独立的文件当中。也就是说日常预发和生产这三个环境实际上应该对应三个不同的配置文件。


咱通过什么样的方式来拉取这三个不同的配置文件呢？其实有很多种方式了，比如说他们这三种文件 git 的文件名通常来讲是不同的，对不对？那假设你 git 文件名相同，那么你的 path 还有 branch 可以是设置不同的，以此来区分不同文件的位置。但是通常来讲，我建议把这两个都给它结合起来，也就是说你的文件名上也要体现出不同，怎么体现呢？咱前面学到有一个 profile 对不对？ application 加上 profile 共同配置了一个文件名。那 profile 这个字段其实就可以对应到不同的环境。比如日常环境的 profile 我们可以把它指定成 DEV ， 预发环境可以把它指定成 staging ，生产环境可以把它指定成 production 通过 profile 这个字段来在 git 文件名上做一个区别。


然后如果咱在某一个环境下面都有很多的配置文件，我不想把这三个配置文件全混到一个文件夹下。那这个时候我们就可以使用 git 文件的 path 或者是 git 文件创建不同的 branch 来区分。通过这种方式，咱就把配置项和环境一一对应了起来，并且不同环境的配置项做了隔离。那同学们可能会想了，这个 path 和 branch 不也是在配置属性当中指定的吗？比如咱们配置 YAML 文件的时候， profile 就已经写死了 DEV 或者production 。
那如何实现动态配置呢？大家可能忘了我在之前课程中说的实际上这个 profile 或者是 brunch 还有 path 虽然是在配置文件当中，比如 boots YAML 中指定好的，但是它也可以从外部注入。我们指定的时候可以用怎么样一个 dollar 符这个具体属性可以是从外部注入来的。比如说你在启动 Tomcat 的时候，咱在日常环境的 Tomcat 或者是预发环境、生产环境的 Tomcat 启动脚本中都加入一些系统参数，也可以在这台系统的机器上设置系统变量。如果你是 bootstrap 的应用，也可以在启动脚本中添加参数。


总之在你具体的开发机器上，通过各种方式把这个参数在项目启动的时候注入进来就可以了，这样咱的代码就不用做改动了。对不对？咱代码只用接收一个固定的 property 这个变量是什么值呢？那就通过启动项目的时候从外部传入进来就好了。这样咱的代码呢就可以以不变应万变，对应不同环境，读取不同的配置内容。 OK 这三碗不过岗干了一碗啊，咱接下来喝第二碗啊。

很简单，它就是一个开关的功能。比方说咱现在有这么 12345 台服务器的集群，我上线了某个新功能，但是我不想一次性把这个新功能在所有集群机器中全部给它打开，怎么办呢？这时候咱可以在业务代码中做一个埋伏。什么个意思呢？就是加一个控制开关，当开关打开的时候，那么这个新功能就生效，关闭的时候就不生效，这样就可以借助我们的 config server 怎么办呢？比如我想把这个新功能在某些指定的服务器上开启，那 config server 就可以把这个开关的值推送到对应的服务器这里，是不是就用到咱前面学到的动态属性刷新了？那咱就可以随心所欲地推到你想推的服务器，那咱就挑选两台服务器，就这两台了，你俩吃我一招，其他的服务器保持原样不动，你们俩把这个新功能就给打开了，这其实就是一个非常非常常见的场景。


大家看了这是不是能想到一些案例啊？比方说咱现在非常流行的 AB test 对不对？ AB test 不是指 angelababy test ，它是一种测试方式，让用户作为一个选择题。比如说咱上线了一个新功能，在页面上额外的显示后端计算出的某些数据。那这个功能你可以让一半的用户看到另一半的用户不看到，这样用户就被分成了两个盲测组，每个组咱可以给用户流量做一个打标，然后在最后计算哪一个组对业务的转化率，或者说对业务的贡献率更高。


OK 这就是简单的  咱刚才说到的业务开关、定向推送还有哪些更常见的场景吗？它是不是跟我们的发布灰度更相关呢？没错，我们可以利用业务开关和定向推送，尤其是定向推送的这个功能，做这么几件事，比如说灰度发布，这个大家应该非常非常熟悉了。除了这个以外，蓝绿发布金丝雀测试咱都可以通过定向推送的方式来做。 OK 咱第二杯酒喝完了，咱去干最后一杯。
