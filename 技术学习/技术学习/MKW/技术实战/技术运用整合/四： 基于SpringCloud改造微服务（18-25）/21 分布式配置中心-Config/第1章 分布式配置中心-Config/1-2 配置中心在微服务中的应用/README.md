我这里一拍脑袋就想到了四种方式，我们看一看你是不是曾经在项目中使用过这四种方式。第一种方式就是在程序中写死一段配置。那比方说在我们的 Java 代码里我想到一个地方读取文件，那可以在代码中把这个文件路径直接写死了。对不对？假如某一天你要从这台电脑换到另一台电脑，或者是你用 Windows 开发，但是部署的服务器是 Linux 那你就不得不在上线之前把代码中的文件路径给更正过来。同学们听我描述就知道了这种方式不可取的。那咱还有什么更好的更优雅的方式吗？那接下来咱看第二种方式是配置文件。有的时候老板经常会说把程序中的 property 抽到配置文件中。实际上这两种方式都有异曲同工之妙。什么意思呢？也就是说你每次需要对它更改的时候，你恐怕都需要把程序先给它关停，改完之后再把它重启起来。


那我们的配置文件指的是就指的是这两种了，application.yaml或者 boot strap 的 YAML 通常我们会把什么样的属性放在配置文件当中通常是那些万年不变的尔曹深与名句面不废江河万古流啊万古流放遗臭万年从来不带变的。比如我们的服务器端口 server.pot 是吧，谁没事会改服务器的端口今天10086，明天 0010 咱又不是移动手机号换联通这么方便的。


OK 除了这两种方式还有什么方式吗？咱不在程序中定义可不可以，从外部引入当然可以了，那就是环境变量。它常见的环境变量有哪些？比如说咱最早学 Java 的时候配置过什么 Java home Java home 的实际上就是一种在操作系统层面设定的环境变量对不对？那除了这里还有吗？我们以前用 Tom cat 启动应用的时候，或者是你本地使用 run 方法启动的时候，咱可以指定启动参数。举个例子，我们在启动的时候通过设置 were both 冒号 GC 来设置一些 GC 相关的参数。同学们可能在本地开发测试的时候不常碰到，但是在生产环境上我们启动应用的时候通常会指定参数。


OK 除了这三种，我们还有一种相对来说灵活那么一点的配置方式。大家可能猜到了，那是数据库存储对不对？同理咱也可以把这个属性存到缓存里面，每次程序需要读取这个属性的时候，都到数据库里重新读取一把。但是这个操作可不是万斤油的，这个对数据库的访问压力可是相当大的。那有的公司比较有钱，直接上的 Oracle 性能来说确实会高一些，但是咱互联网行业还是用 MySQL 居多，你真的考虑好把这个访问压力推给弱不禁风的 MySQL 吗？我想还是要三思而后行的。


同学们，我这举出的四种方式你中枪了哪一个？反正这四种方式在老师的项目中是经常用到的。这四种方式各有利弊，简单高效的方式它不够灵活，灵活多变的方式它性能可能也有问题。下面这三种配置方式，多多少少还能说得过去。上面那一种要是被你的队友们看到了，那可直接要开启喷子模式了。 OK 那讲到这里，咱不妨看一看以上四种传统形式的配置管理方式都有哪些缺点？那吐槽大会现在开始，我们今天的主题就叫配置管理。好，现场嘉宾可以发言了。那有的嘉宾说格式不统一。没错，这个观点很犀利，我想想这里面有多少种格式啊？ practice 有没有？那 YAML 有没有？还有 Jason 那这三种格式各自不同，我们在项目中究竟用哪种呢？很好办嘞，你喜欢？用哪种没人管着你，那除了格式不统一，还有其他缺点吗？又有人说了没有？版本控制，这点说的没错，假如你是在 problems 文件中，那可能还有 git 或者 SN 等等版本控制工具来管着。那如果你在数据库中做一个更改，那真的就是放飞自我，想改就改喽。


不管是在数据库还是在配置文件中，假如你改错了一个属性，想要回滚也是蛮痛苦的。那如果你说这两个缺点你都能忍受，那接下来的缺点看你还扛不扛得住。他这说了基于静态配置什么意思呢？这里指的就是基于配置文件或者说是基于 Java code 中的 hard code 形式配置的属性每次对它做修改我们都要怎么样先修改，然后再重新发布整个应用流程是不是非常非常的繁琐上面提出的这三个问题都是在微观层面的问题，我们作为架构师要有什么？大局观，那从大局上考虑怎么样分布零散每一个项目，每一个团队有自己的配置文件管理方式。


你用数据库，我用配置文件，有的团队比较放飞自我，直接写到 Java code 里面了，没有一个统一的地方对这些所有的配置进行管理对不对？比如说我们有五个项目共同访问了同一个数据库，那你这个数据库的连接串是不是要分别配置在五个项目中？假如有一天你数据库升级或者是你数据库维护换了一个网络地址，那怎么办呢？你只好把这个属性通知各个应用方，让他们自己把属性在自己的应用中改变过来。


听到这里是不是脑袋都要炸开了？那我们有没有什么更好的方式来管好这些配置文件呢？别着急，在探究更好的方式以前，我们先对配置项里面的内容做一个分门别类。怎么个分门别类法很简单，我们把配置项区分为静态内容还有动态内容。什么是静态内容呢？很简单，就是我们前面说的万年不变的内容，比如说端口号，再比如说应用的名称等等。那这部分内容我们可以把它分为两个类型。


第一种类型就是环境配置。我们来看一下环境配置中的几个代表属性，比如说数据库的连接串对不对？还有我们前面配置了很多次，在每个应用中都有配置的 eureka 的注册中心对不对？再比如 Kafka 的连接串，还有就是应用的名称、端口号等等。除了环境配置以外，还有一部分很特殊的属性类型，它们统称为安全配置项目，施工安全第一。配置属性中的有些配置项跟安全息息相关，比如说连接的密码对不对？还有公钥私钥或者是 htd P 连接的 sert 等等这部分内容咱可不能把它光明正大的使用明文保存在配置文件里。我们要怎么样对它进行一个加密操作？当使用到它们的时候，再用对应的解密方式把它解密出来使用。


看完静态的内容，我们再来看看都有哪些动态的内容需要控制。动态配置的第一种类型也是我们最常用的就是功能控制。什么叫功能控制呢？比如说我们研发了一个新功能，这个新功能第一次上线，为了安全起见，我是不是要做一个应用开关，当这个开关开启的时候，我的功能也随之开启，关闭的时候我的新功能也就被 disable 掉了。这是其中一个用途。还有没有人工熔断开关，它和前面的功能开关是类似的。比如说当开关开启的时候，我把所有的用户请求都自动的导向到降级流程里面。
OK 除了这两个还有什么？我们发布的时候是不是可以配置一些蓝绿发布的策略啊？除了蓝绿发布以外，大家应该还听过一个名词叫金丝雀测试，那都是可以通过动态配置来控制的。除了这些控制方式以外，还有一种牵引发而动全身的配置数据源切换。 OK 除了功能控制这一个大类以外，还有哪些类型呢？那当然要数业务规则喽，比如某些随着时间会发生变化的数据，就像当日的外汇利率。


再比如说双十一的当天，我想对某些爆款商品生成动态的文案，我的后台系统可以实时分析买这款商品人的性别比例。如果败家老爷们居多，那我这款商品的文案就叫行、男必备潮牌等等。如果败家老娘们居多，那我这款商品的文案就可以写成重返 18 岁或者是女神必备。
除了这些，我们还可以指定规则引擎的参数。对于一些非常复杂的业务，比如说营销优惠的计算或者是下单流程，通常都会用到一些规则引擎，这里面的参数也是需要动态指定的。那接下来还有没有其他类型那当然就是应用参数了，这个应用参数指什么呢？打个打比方网关层的黑名单。白名单对不对？我需要根据一定的规则甄别哪些是恶意访问的用户，把他的 ID 加到黑名单里对不对？所以这个名单也是需要动态变化的。


再比如我的程序在写入缓存时候，通常会指定一个过期时间。那我可以在某个时间点通过修改配置把这个缓存的过期时间拉大或者缩小。除了这以上两种以外，我们还可以对日志级别的 mdc 进行设置，这个功能不知道大家有没有用过或者有没有听说过日志组件中的 mdc 可谓是功能强大。我们有些链路追踪的框架也是利用了 mdc 技术。我曾经接过一个私活，他有这样一个需求，就是我想在运行期动态的指定某个用户他的 log 输出级别。那么这里就可以用动态属性获取的方式，在日志 mdc 组件中配置不同的用户 idok 刚才带同学们把配置项中的内容进行了分门别类。


其实这就是一个需求的梳理过程，我们弄清了配置项中的内容，知道了有哪些类型的配置项需要管理，这样也就清楚的知道配置管理都有哪些需求，就可以对症下药，使用正确的配置管理姿势，管好所有的应用属性。那我们就顺着这些需求，看一看新时代的配置管理中心都有哪些高标准严要求。


首先咱这个配置中心是服务于谁呢？微服务集群对不对？那它必须要顶着高可用的口号，就是说身板要硬要能经得起摔的。除了这个以外，咱还对他的业务素质有一些要求，那便是要具备版本管理的能力。首先要把所有对配置文件的修改记录一一记录在案，这样我就具备了一个版本控制的能力。并且我需要对能访问这些配置属性的人进行一个权限控制，不是说所有人都可以访问的。紧接着咱还有一些业务上的需求，比如说我的属性要能通过动态推送的方式，把变更推送到服务集群中的每一个节点上。对于有些有加密需求的属性来说，我还要提供内容加密以及对应的解密功能。最重要的咱要是一个具备大局观的配置管理中心，因此要有配置分离的中心化管理思想。对于这些配置属性，我希望把它从应用层抽离出来，放到一个中心化的节点或者某些存储介质上，这样方便我进行管理修改。同学们可能在想了，办到这四点好难，其实一点都不难， spring cloud config 已经替我们摆平了一切，这一节中我们已经做足了铺垫。那么接下来就请同学们和我一起走进 spring cloud config 的世界，看一看它的体系架构和核心的功能，紧接着就顺手搭建一个配置中心的小 demo 好同学们，那这一节的内容就到这里结束了，我们下一节再见。
