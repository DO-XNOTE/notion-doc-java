hello 慕课网的各位同学们，大家好，这一节咱就来聊一聊如何对配置属性进行加密操作。在咱 it 行业有这么一个新造出来的四字成语，它叫做山库跑路，它用来形容有些受尽气压、受尽凌辱的程序员，以及数据库管理员，在离职前做了一项伟大的壮举，那就是把整个数据库删掉，然后跑路。这个成语一方面赞扬了程序员们不畏墙报、敢于反抗的精神，另一方面提醒各个企业一定要把控好数据库的访问权限。


讲到这里，数据库的访问权限无非是一个用户名和密码，对不对用户名其实无关紧要了，但是这个密码却是一个非常非常隐私的信息。咱能不能在 git 文件中直接保存明文呢？如果你不怕山库跑路的话，那可以这样做。但是对于大公司来说，这个密码必须要使用密文，也就是加密后的数据才能保存到 git 服务器上。我们这一节就围绕着数据的加密来展开，它主要包含三个部分内容。
第一部分咱要对 GD K 进行一番改造，大家没有听错，真的是 gdk 在 GD K 中有这么一个组件，叫做 gce gc E 其实是用来做一个加密操作的核心组件。那么咱这里要从 Oracle 的官网上下载一个特定的 GC E 组件，把咱 GD K 中的 GC E 替换过去。待会子我会讲一下为什么要做这个替换。


那接下来第二点是什么呢？就是改造 config server 让它可以生成加密的字符串。那一谈到加密，咱知道有对称密钥和非对称密钥对不对？使用对称密钥加密非常简单，使用非对称密钥加密稍微会那么麻烦一点。那咱这一节老师使用哪种方案，当然必须是迎难而上，使用简单的了，怎么又叫迎难而上呢？同学们就不懂老师的良苦用心了，这个迎难而上是让你们迎难而上，老师实现一个简单的对称密钥加密，然后留一个作业让大家实现非对称密钥加密。当然了，这里面的思想都是大同小异了，相信同学们可以举一反三，顺利通关的。


OK 那第二部分完了以后，那接下来第三部分就是修改 github 文件，把我们经过加密后的数据存放在 github 上面，然后启动服务器拉取配置。在这当中还牵扯到一个本地配置的改动，那咱们本地的文件要给它加那么一个小的机关，告诉 config server 请使用解密动作，先把 github 拉取过来的这个配置文件属性进行解密，然后再使用。



待会儿咱在 demo 环节再向大家来展示。乍一看信息量是不是很大？实际上对代码的改动非常非常少。同学们准备好的话跟我轻装上阵， intelligi 走起编程使我快乐。 996 是我的福报，那咱今天先享受第一个福报，更改 gdk 可不是让大家冲到 gdk 里面改源码不是这样玩的，咱要怎么样？咱要替换一个 gdk 中的组件就叫 gce 那大家平时工作中应该也听说过这个组件，咱这里说一下为什么要对它进行替换，是因为咱 spring cloud config 在执行加解密操作的时候实际上会用到 gce 这个组件。而我们平时下载的 gdk 中，这个 gce 通常来讲是一个限定长度的版本，那我们要把它替换成什么？替换成一个不限长度的 gce 版本。

具体怎么做呢？我们来看下面的这一个网址，我把需要下载的网址已经给注释到这里了，我把它放到了 config server 这个项目下的 application.yaml 文件夹。但是大家注意，这个网址的后面是一串数字，所以看起来它也不是个永久的网址，可能会发生变化。
