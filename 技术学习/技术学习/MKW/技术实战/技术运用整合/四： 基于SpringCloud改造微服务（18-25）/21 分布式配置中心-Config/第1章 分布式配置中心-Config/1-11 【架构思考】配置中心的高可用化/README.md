所以作为一个微服务集群的应用，我们每一个环节每一个组件都必须考虑的首要问题是什么？高可用对不对？像这种单节点的情况，一旦你挂掉了，那所有的服务方干瞪眼，全部团灭。其实咱说高可用它不是一个什么高大上的名词，它的道理非常非常简单，就是 try everything you can to keep the business running 你只要确保我的服务不管出现任何情况，它对外总能输出战斗力，也就是保持可用状态。那这就是高可用是不是很简单啊？那一提到高可用，我们最容易想到的一个词是什么？集群对不对？把你的节点从单节点也变成一个集群，道理是这个道理。

可是我们在 spring cloud 架构下该怎么做这件事呢？大家能回想起来之前有哪一个组件可以帮我们完成这项任务呢？相信同学们这里已经猜出来剧情的发展方向了，那大家就配合我继续往下演戏。这里咱们把注册中心首先改造为了集群，那接下来服务还是那么个服务 AB C 都在这，那它怎么向我们的服务注册中心集群拉取文件呢？我们这里要借助 eureka 怎么个做法？首先第一步，咱所有的注册中心都要去 eureka 那里报道一番，把自己的 IP 地址都注册上去。


然后我们的服务 AB C 从 eureka 这里先拿到 config server 的完整注册列表。接下来的事情就顺理成章了。这些服务 AB C 就像调用一个远程方法一样，对我们的配置中心发起一次调用就可以了。那咱这三台 config server 具体调用哪台副驾是不是通过 ribbon 的负载均衡就可以做到非常简单。通过这种方式，咱的服务还有 config server 以及 eureka 就形成了一整套闭环。那我的 config server 去把自己的信息发布到 eurika 上，这样的话也可以利用 eureka 的服务续约，还有服务剔除等等的功能来维护一个正确的 config server 列表。通过这种方式，达到一个高可用的架构。 OK 那有的同学说如果我没有 eureka 怎么办？其实很多互联网公司也有自己的配置中心解决方案，他们也并没有用 eureka 但是实现高可用的方式有很多种，我们把 eureka 完全抛掉。


如果让你来搭建一个这个高可用的方式，其实从最简单的角度出发，咱可以利用网关就可以做到怎么办还是这么三台的 config server 然后服务依然是这些个服务，这之间怎么做联通咱的服务其实只要往一个指定的域名上面发送一个 request 并且这个 request 精油咱是配置的负载均衡网关层。打个比方，它可以是 f5 网关，也可以是 NGINX 网关。
