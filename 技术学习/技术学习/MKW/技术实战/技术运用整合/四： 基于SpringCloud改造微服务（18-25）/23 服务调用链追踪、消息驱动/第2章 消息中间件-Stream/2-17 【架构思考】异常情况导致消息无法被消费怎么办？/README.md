中科网的各位同学们，大家好，在前面几个小节，咱学习了 stream 中重试消息的几种不同的方式，比如 consumer 端原地重试，还有利用 requeue 重新发回到 rabmq 重试。但是咱在实际的应用场景中，总会碰到那么一两个无法靠重试解决的异常，这一节，咱就来探讨一下对这类场景如何对症下药。
那首先咱来看一下常见的几种顽固异常都有哪些。第一种 case 是咱在生产环境当中最常见的，这里面其实包含了两种不同情况，第一种情况是指消息被消费者拒绝掉。那第二种情况是指消费者在进行异常重试的时候，它的重试次数达到了阈值的最高限度，那么就不能继续再重试下去了。


除了这两种情况以外，还有一个常见的 case 是消息过期。就是说咱这个消息在队列里待的时间实在是太长了，占着茅坑不拉屎。因此对于这些设置了 TTL 信息的消息来说，如果达到了预定的时间还没有被消费，那么这时候就要从队列中移除掉了。接下来还有一种情况就是队列长度已满。这种情况通常出现在什么时候呢？比如说咱的消息生产者他在持续输出，这个生产力又猛又持久。而与此同时，消费者处理消息的速度远远赶不上生产者生产消息的速度。那因此这个队列中的消息数量就会一直一直的增加。最终这个队列就被填满了，那后续来的消息就无法添加到这个队列当中。


那以上三类情况都是咱在生产环境里面经常碰到的，也称他们为顽固异常。这些异常消息就好似游离于三界之外，不受咱正常程序的管控，那现在决策权就交给了你。对于这些消息，你打算将它们怎么办？凉拌番茄炒鸡蛋，我把它直接丢弃好了啊，装作什么事都没发生，反正天知地知我知，rabbit MQ 不知。但是这么做的话似乎有那么些不妥吧。我有一言不知讲不当讲，但说无妨。好，那我就讲了咱架构师要具备的一个意识是什么叫一个都不能少什么意思。咱在设计复杂业务场景的时候，尤其是在像支付或者是金融领域，说咱讲究一个字叫一致性。就是说不管怎么样，你这个数据即使它处理失败了，你也不能把这条原始消息直接丢弃。咱一定要考虑好容错、重试等等各种方案，以确保最终一致性。


那这边有几个比较重要的点咱要去考虑到比如说刚才提到的一致性，还有要保留异常现场，你出错的数据是什么？原封不动的把它保留下来。假使我们后台有定时任务重试，或者是有其他的补偿机制，那在这些机制通通不生效的情况下都不能处理这个异常了，怎么办？那就人工介入排查数据。总之除非是那些可有可无的数据，否则我们都应该把它保存下来。即便它是处理异常情况下出错的数据，可是巧妇也难为无米之炊，工欲善其事，还必先利其器。那有什么工具可以帮助我们搞定这种顽固异常吗？这个可以有死信交换机，就是你的不二选择。


这名字听起来蛮吓人的，有点像死亡笔记对不对？实际上这里面的概念并不复杂，它只有两个需要大家了解的地方。第一个是 dlx 叫死信交换器，它的作用非常简单，就是将异常的消息路由到死信队列。也就是说死信交换机只是一个中转方，它的目标地址是一个单独的消息队列，咱叫它死信队列。那死信交换器如何将一个消息路由到死信队列呢？这里还有一个概念叫 dlk 这个 dlk 的意思是一个死信路由 K 有了它咱的死信交换器就知道该把这条信息传递给哪一个死信队列了。所以 dlk 在这里是一个向导的作用。


莱芜导夫先录。 OK 咱看完了这两个概念以后，我们再看一下死信队列它的工作流是什么样子。首先咱的生产者创建了一条消息，把这条消息发送给了 rabmq 接着 rabmq 又把消息存到了相应的消息队列里面去，接下来的事儿就交给 consumer 了。咱的消费者从消息队列里面拉取消息进行消费。但是在这个过程中可能碰到异常，如果咱的异常重试次数超过了预定的阈值，这个消息就是消费失败。那消费者自然就要把这个信息上报上去，让消息队列调用 DL X 就是死信交换器，但是不仅仅这种情况会调用到死信交换器。就像前面咱说的，如果消息队列中的消息过期，或者是消息队列已经被塞完，容不下后续的新消息了，这些情况都可以被传送给 dlk 这些情况都可以调用死信交换机。然后这个交换机将借助 rk rk 实际上就是 routine key 就是咱前面说到的 D L key 那借助 routine key ，咱就知道该把这个消息发送给哪一个死信队列了。


消息到了死信队列，那后面的玩法就多了，咱可以专门创建一系列的 consumer 来监听死信队列。比如说咱创建一个人工介入的流程，专门监听死信队列。如果有一个消息到达死信队列之后，就把这条消息转发给某个团队。当然了，即使没有消费者，早上这条消息躺在死信队列里也没问题。咱在 rabbit MQ 里面有一个功能，也就是可以将死信队列中的消息移动到其他的队列当中，咱可以把它再移回到正常的消息队列里重新消费。


那以上便是死信队列的简单工作流了。那在本小节的最后，我给大家做一下下集预告，带大家看一下死信队列在 rabi MQ 中有哪些特殊的标签。那这幅图就是我们在 rabbit MQ 中创建的死信队列，大家看到这里有一个黑色和一个黄色两个不同的队列。那这个黄色的队列是咱的原始队列，如果在原始队列的消息处理失败了，它会被转发到下面那个队列。下面这个队列是以 dlq 结尾的，这个才是咱的死信队列。


不过在原始队列中有两个非常特殊的标签，我这里给它标注出来了 dlx 和 dlk 它就是死信交换机和死信的路由 key 所具备的功能。也就是说只有你的消息队列具备这两个功能以后，它才可以有能力将处理失败的消息转发到对应的死信队列中。 OK 同学们，那这一节的内容到这里就结束了。同学们在之前的分布式章节里，也细致地学过了死性队列。那咱这里对死性队列进行了一个简单的复习。在接下来的一个小节里，我将和大家动手搭建一个死信队列的样例程序。 OK 同学们，那我们下一节课程再见。
