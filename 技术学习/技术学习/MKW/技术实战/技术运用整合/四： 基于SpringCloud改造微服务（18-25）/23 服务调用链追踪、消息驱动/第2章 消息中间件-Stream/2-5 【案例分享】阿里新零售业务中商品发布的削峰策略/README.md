话音刚落，这个总监就使出了一击必杀的大招我们来看一看它的技能说明。这个大招的名字就叫一键开店，它有什么功能就是瞬间打出 5 万以上的 QPS 秒杀所有商品的服务。这个一键开店的技能是一个需要持续施法的技能，它一旦开始吟唱，大概会在 10 秒钟不到的时间打出小几十万的商品。那对应到商品服务的集群来说，大概就是每秒 5 万加的 QPS 以当时我们的服务集群机器数量是远远承载不够的。

运营总监这一招厉害，估计还没有吟唱完，这个所有商品服务全部都团灭了。咱不光说商品服务要团灭，一个商品发布，它调用的远远不止商品服务这一个服务它还会调用后续的订单服务、库存服务。总之很多下游微服务都会受到这个影响。眼看服务雪崩，就要发生了问题，就是这么个问题。那么接下来我们怎么办呢？那就是必须要引入消息组件来做流量削峰，所谓兵来将挡，水来土淹一物降一物。那这个东西就是完课运营总监技能的法宝，我们看看消息组件的技能说明都有哪些。


第一个是平滑输出。我们采用客户端主动拉取的方式，它并不是主动的往你嘴里硬塞东西，而是客户端根据自己的处理能力主动去拉取有多大本事就干多大的事儿。第二个特点是不怕 timeout 咱如果在短时间内像刚才的商品发布一样，10秒钟内发布 50 万个商品，如果你接口响应超时了，那你还要设置很多的容错策略来补偿，保证最终一致性。那你这个业务场景就被 API 超时给限制住了，而消息组件需要担心这个问题吗？完全不用担心，你的所有请求都被存放在消息组件这里了，什么时候有资源？

那什么时候就会有客户端来主动拉取去做，完全没有基于 API 调用的超时烦恼。那最重要的当然还是依靠消息组件的高吞吐量特性。你 50 万个请求如果全部落在我们的业务服务器上，这个造成的伤害很大。但是如果你落在中间件消息组件的集群上，你这个真的就是毛毛雨了。要不怎么说能担当中间件的大人，咱但凡提到中间件，一般来说吞吐量都是相当优秀的。那我们有了消息组件这件神装的加持，看看如何来抵挡运营总监的营卫啊。啊总监这时候又开始吟唱起了一键开店的技能抵达了商品中心。


接下来我们就寄出了消息组件这件神装，商品中心把一件开店所对应的所有商品一股脑的发送给了消息中间件。那商品发布消息抵达到消息中间件以后，我们每一个下游应用都会去监听这个消息队列。比如我们的商品服务监听到这个消息，开始发布淘系商品，那销售单服务发布一些销售计划等等。库存服务就是去初始化库存，发布每日库存，所有的下游服务全部井然有序地运作着。


运营总监这一个大招砸下来，就被咱的消息组件完美化解了，对我们的服务没有造成任何一点伤害。此刻我心中不由燃起了对从事技术工作的这种优越感。对运营总监、产品经理这些技术小白，只有他们想不到的问题，没有技术人员解决不了的问题，程序员的超能力，他们不懂言归正传，本小节到此结束。在下一小节里，我将带大家看一下 stream 的 Binder  作用机制。那同学，我们下一小节再见。
