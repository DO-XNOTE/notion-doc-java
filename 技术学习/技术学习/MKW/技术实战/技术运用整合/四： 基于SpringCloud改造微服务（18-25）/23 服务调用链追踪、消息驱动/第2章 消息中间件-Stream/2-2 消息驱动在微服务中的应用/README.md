互客网的各位同学们，大家好，这一节我们来聊一聊消息驱动在微服务中有哪些的应用场景。相信大家对消息驱动的业务模型都非常熟悉了，所以这一节咱就采取短平快的打法快速通关。咱先来看一下消息驱动那万年不变的架构模式。首先当然是有一个消息的发布源，那它会将消息发送给中间件，比如说 Kafka 或者是 rabbit MQ 这两个组件，应该可以说是目前业界中应用最广泛的两个消息中间件了。当你的消息发送给中间件以后，它会被分发到对应的消息队列，不同的消息跑到不同的队列当中。比如咱现在有消息队列，a B 和 C 消息已经就绪了，接下来就要看咱的消费者如何去拉取消息了。

咱的消费者实际上就是业务方他会根据预定的配置到对应的消息队列里面拉取消息。这整个流程实际上是异步处理的。也就是说消息的发送端和消息的消费端他们之间并不需要同步等待，我发完消息就可以转头干其他事了。所以生产者和消费者也就是各忙各的。那这整套流程真的是万年不变，在各种消息中间件基本上都是这样的一个模式。


那接下来咱看一看这种消息驱动的场景在咱微服务架构下面都有哪些用途。它其中一个应用场景就是跨系统异步通信。这里的跨系统不仅仅是说跨两个不同的应用，甚至它可以跨不同语言的系统平台来通信。假如说你在集团内部有两个不同的业务，这两个业务方它的技术选型是不一样的。比方说我们比较先进。那咱选了 spring cloud 另一方业务可能是非常非常老的业务，它还甚至用的是 gbox 并没有集成到咱 sprint cloud 的服务。发现上来这两个业务并不能发起欢快的调用。
那怎么办呢？


这里就可以采用消息驱动的应用场景来解决。因为对于消息组件来说，你的生产者和消费者都是使用了同一套的消息规范。那我并不关心你的底层是使用的什么技术，甚至连你的语言平台也不关心。不管你是用世界上最优秀的语言 PHP 还是用 Java 你只要遵循了这同一套消息规范，向消息组件内发送了一条消息。


那么生产者和消费者自然可以借助消息中间件进行畅通无阻的通信。所以在有些大公司里面，我们经常可以看到这样一个情况，新研发的业务系统和老系统之间需要进行对等通信那怎么办？就采用消息中间件的驱动。但是这个应用场景有一定的局限性，因为毕竟消息驱动是一种异步通信的策略，如果大家需要进行同步阻塞的通信，那可能需要想一些其他方式。


OK 除了这个业务场景以外，还有一个最重要的功能跟我们的架构层次相关，它就是应用解耦怎么来理解？比如说我们在电商业务场景中都有一个下单对不对？我们讲电商系统中有一个很重要的概念叫 order lifecycle 就是你订单的生命周期。那当用户点了下单按钮并且付了款之后，大家以为这件事就结束了对吧？但是实际上这个订单的生命周期才刚刚开始，他后续要对接很多很多的业务方进行处理，不仅仅支付，当然还有订单的履约、平台配送以及后续的邮件通知等等非常非常多的子系统。


所以说对于下单这个环节来说，我们知道后面还有很多很多的子系统后续流程需要触发。那如果我们把这个触发动作交给下单接口来做，对于上游应用是个非常非常大的负担。你要挨个调用下游泄物系统的接口，把这些流程触发。那假设以后新加入了一个业务方，这个业务方会触发一个购物返券的功能，每次下单发放一个优惠券。那对于下单接口来说，你不得不去再接入这一家新的业务方，这种场景就可以通过消息驱动的模式来完美解决。

我的下单接口只要简单的推送一条消息到消息中间件中，如果你的下游应用需要在订单下单以后进行一些其他的业务流程，那很简单，你只要订阅这个消息主题就可以，一旦有一个新的消息到达，所有的下游应用都会去消费这个消息。因此对于你的下单接口来说，作为上游应用，你并不需要知晓究竟有哪些下游应用跟你要做对接，你只要简单的发送一个消息就可以了。所有的下游应用对你来讲都是透明的，他们如果要消费这个消息，只用去监听对应的 topic 就可以。通过这种方式我们就做到了应用间的解耦。


那它还有一种应用场景叫流量削峰，咱说削峰填谷是一种平滑流量的手段。


我这里跟大家举个例子，电商场景中经常出现秒杀，对不对？那我上了一款热门的商品，12点钟开始抢购，咱这个商品限量有 1 万个，那抢购的人可能非常多，百万雄狮一起来抢，这 1 万个有可能在一秒钟之内就被秒杀完对于系统来说，你可能需要在 1 秒钟之内处理 1 万个下单请求，但是咱的集群服务器数量不够，可能扛不住。


那这种情况怎么办？[很简单，把用户的下单请求放到消息队列当中，然后让我们的消费者也就是处理下单请求的这些服务器，去主动的拉取这个下单消息。在这种模式下，下单请求并不是强行推送给我的服务器，而是说咱的业务服务器会根据自己的处理能力主动的从队列中领取消息。如果你的机器性能差，可能一分钟领 20 个消息，那如果你性能好，处理的快，可能一分钟处理 100 个或者是 200 个。因此你在短时间内有用户访问量，造成了这种洪峰，流量冲击并不会全部压到你的服务器上，而是暂存在消息队列当中。](/05f59d36344e486c9889853acfc258fe)
