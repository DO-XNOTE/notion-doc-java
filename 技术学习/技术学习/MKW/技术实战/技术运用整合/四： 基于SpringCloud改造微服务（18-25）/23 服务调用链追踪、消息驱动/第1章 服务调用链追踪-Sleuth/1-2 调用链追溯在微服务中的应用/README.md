这里强调一点分布式，因为我们的线上环境肯定都是集群部署，链路追踪技术不光需要在 log 里打入一些标记，它也同样的要在分布式的环境中从不同的机器上收集这个信息。 OK 接下来的一个考核指标叫 timingtiming 是什么意思呢？它就是指出你整个调用链路，比方说用户从点击下单按钮到最终订单生成。这个链路里每一个微服务模块所花费的实践排名信息是咱生成报表数据的一个重要的维度。它不仅可以帮助我们获知在整个链路中哪个服务是性能瓶颈或者哪个服务花费了过长的时间。同时我们也可以设定一些业务警报的门槛。比如说咱可以设置一个预警机制，当有 50% 的服务处理时间大于 5 秒钟的时候，发出一个业务预警，因为可能产生了某些性能瓶颈，导致业务响应过慢。


除了这两点以外，它还有需要一个定位链路的功能。这是什么含义呢？比如还从我们下单接口来说，从下单完成到最终订单创建的这整个链路，我们需要有一个唯一的 trace ID 当我后台的研发人员拿到这个 trace ID 以后，我可以通过 log 查询系统查询到整条调用链路中，从上游到下游，所有的调用情况一目了然。这是线上问题排查了一个不可或缺的功能点。


那最后一个指标是信息的收集，还有展示，咱不能光闷头干活，收集情报，收集 timmy 又是定位链路了，我们要有能力把这些东西展示出来。所以对链路追踪框架咱在各个维度上都有要求，你不仅需要对 log 日志进行打标，还需要从分布式环境下的不同机器收集 log 文件，还要做数据分析的功能。比方说分析 timing 信息，不仅如此，咱还要给研发团队提供一个日志搜索的功能。比方说它可以通过一个 trace ID 或者业务关键字，比如 order ID 搜索上下游所有相关的日志文件。


你看咱有这么多的要求。那么链路追踪框架的搭建恐怕不是一个组件可以完成的，我们往往要搭配多种技术，协调起来，一起作用，共同构成一个链路追踪系统。 OK 那么这一小节的内容就到此结束了。我们在后续的小节依次向大家揭开链路追踪技术的神秘面纱。同学们，我们下一节课程再见。
