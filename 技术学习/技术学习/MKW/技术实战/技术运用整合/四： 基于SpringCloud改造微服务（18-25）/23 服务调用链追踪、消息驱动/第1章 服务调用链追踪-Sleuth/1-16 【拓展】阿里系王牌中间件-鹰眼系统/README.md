把这些请求调用平摊到每一秒里，都有一千多万次的调用，如果把晚上访问量低谷给去掉（剁手族晚上也要睡觉），那么正常的画风每秒都有小几千万的调用。而且集团主链路背后的微服务十分庞大，不光体现在集群的装机数量上，也反映在微服务模块的数量上，单就一个下单来说，后台就要协同大小几十个不同的微服务模块共同完成下单任务。

从上面的情况来看，构建一个实时的链路追踪服务技术对阿里系也是一个不小的挑战，不光需要解决业务埋点以及Trace信息的传递，还要考虑如何在如此庞大的集群规模上做链路信息收集，把信息汇聚在一点，供业务部门使用。

**鹰眼系统**

大部分链路追踪技术都来自于Google的那篇著名论文，阿里的链路追踪中间件-鹰眼系统也不例外。

鹰眼系统（Eagle Eye）-目前集团的王牌中间件，服务于主站各个集群，同时与集团各类自研中间件做了集成，一个调用链路中所有组件的动作全部记录在案。更牛的是，它不像Zipkin一样只面向Timing Data，而是全方位各项附加功能加身，完全制霸电商场景下的链路追索。

好了，牛皮也吹完了，上面那是2016年经过大改造之后的鹰眼系统。在这之前的早期版本也没有那么灵光，各种问题层出不穷，比起Eagle Eye这名字来说我更喜欢叫它Single Eye，或者瞎眼系统。

**鹰眼系统的核心功能**

Eagle Eye几乎集成了一切你能想得到的东西：

业务埋点 和Sleuth一样，鹰眼也会在一次调用的全链路中贯穿一个不变的Trace ID，并且对一个Trace中的单元事件生成RPC ID（相当于Span ID）。在这之上，还根据淘系电商业务的特点在链路中写入了很多主数据信息，比如操作人ID，订单或商品编号等等，这些信息都被加上索引存储，用于日志检索。我们在自己的业务中还可以加入自定义的索引信息，比如说我认为运单编号是个重要的业务信息，那么可以让鹰眼写日志时标记这个字段，将业务ID和Trace ID双绑定，这样就可以实现自定义链路的需求。

聚合统计 鹰眼会为调用链数据打上一系列的Tag，比如应用名称、所属机房、上游入口、用户区域、容器类型等等，后台通过对调用链数据的分析，可以从不同维度对热点瓶颈进行定位，也能对容量预估提供参考信息

性能指标分析 提供各种维度的性能指标，实时/峰值QPS、平均响应时间、耗时比例、是否跨机房调用、调用成功/失败率，鹰眼通过一系列规则引擎的分析，可以自动标记出性能瓶颈和潜在的关键链路（即被依赖程度较高的接口）

流计算引擎 核心数据和重要业务数据由流计算引擎进行增量计算，经统计聚合后落地到HBase中，这一步大大提高了关键链路监控的实时性。流计算引擎基于高可用的中控节点，可以自动发现异常服务进行重启，达到自愈的目的（这和Eureka中的服务剔除还不一样，Eureka是放弃队友，鹰眼会再抢救一下）

数据清洗 强大的分词能力，加上规则引擎的判断，可以从杂乱的日志中清洗出精准的业务数据，同时丢弃无效日志行

超强持久化能力 使用HiStore从物理存储层面对链路数据进行压缩（可以达到惊人的20：1的压缩比），采用基于HBase的OpenTSDB时间序列数据库来实现基于时间顺序的存储和分段查找等需求。不得不感叹鹰眼在每个环节都选取了最专业的组件，对性能的要求达到了极致

智能异常识别 这是一种主动定位异常的机制，它通过识别->关联->定位的路子发现问题。首先通过识别非结构化日志数据中的“离群点”（就是指不正常的数据，比如我们QPS一直平稳在2000，突然有那么几秒变成了10000，那么这短短几秒的数据就是离群点）。再关联同一时刻链路上下中的其他离群点，或者是变更事件（比如大规模发布，属性变更推送等），将同一时刻不同地方的离群点关联起来（比如说某次变更推送后发生了大面积接口超时，那么就说明这两件离群点是由某种关联的）。最后一步就是根据前面的数据分析，找出引发离群点的最上游的可疑环节。

业务报警 可以选择钉钉、短信或者自动电话等多级业务报警机制，报警规则可以自由定制。比如某个异常出现次数不能高于10次/秒，或某个接口失败率不能高于10%，这一点和Hystrix的熔断配置思想比较像。记得曾经有一次发生了严重的线上问题，花了几个小时才修复，我的手机收到了上千条短信报警，第二天我就接到集团的特别通知，意思是让我珍惜集团的短信资源，及时关闭报警

还有很多五花八门的功能就不一一介绍了，而且鹰眼目前还处于发展期，未来会加入越来越多的”智能分析“功能。鹰眼只是一个例子，各大互联网公司有各种各样的看家法宝，所以同学们一定要找机会去技术实力比较扎实的大厂体验一下。

**小结**

这一节我们认识了阿里系的链路追踪系统，同时我们这个大章节也告一段落了，下一章节我们将要去学习Spring Cloud中消息中间件的应用。

学习Tips：每个互联网公司的不同部门都有自己的面试套路，要把握住风格做对口的准备。从社招的角度来说，比如头条非常喜欢考算法，那么刷题在所难免，而阿里更偏向系统架构和高可用、高并发场景的落地经验，所以大家尽量通过自己的关系，在选定目标公司和部门后，尽可能找线人打听打听。
