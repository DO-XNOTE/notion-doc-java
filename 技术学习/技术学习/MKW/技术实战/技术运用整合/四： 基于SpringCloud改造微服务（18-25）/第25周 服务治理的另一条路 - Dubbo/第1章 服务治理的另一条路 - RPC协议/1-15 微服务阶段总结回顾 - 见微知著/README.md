各位同学们，大家好，经过前面两个月的紧张学习，我们终于来到了整个微服务篇章的收尾阶段。既然是收尾，我们总要讲点不一样的。那同学们这一章就跟老师一起来总结过去，展望未来这场大结局。我把它分成了上半场和下半场。在上半场里，我跟大家聊一聊架构层面的一些简单而又深刻的道理。在下半场，我就要跟大家唠一唠这技术以外的事情，结合老师自己的工作经验，跟同学谈一谈如何在职场中脱颖而出。所以我们上半场的标题叫做见微知著，而下半场的标题就叫做放飞自我。


OK 同学们，那先跟老师一起见微知著。我们的故事是从微服务开始的，所以这第一个问题，我就跟大家聊一聊微服务拆分的简单道理。就像前面老师说过的，微服务拆分并没有一个标准的范式，也就是说它没有标准答案，如何去划分服务与服务之间的界限，又如何去界定微服务拆分的力度，这种感觉就像知人如竹水，冷暖自知，老师并没有答案，这个答案就在同学们的心里。所以老师这里要告诉大家一句话叫手中无剑，心中有剑。这手中的剑就是我们掌握的技术技能，我们会的框架，这些都是用来完成任务的工具。而心中的贱是我们内心对自己业务的深刻理解。很多技术同学或者架构师只是盯着技术层面的一亩三分地，缺少了对业务层面的理解。如果不能抓住业务的本质，那么你想出的架构方案会不会就是无源之水了。那微服务拆分虽然有很多的方法论，但是它毕竟要实打实的将项目落地下来，即便我可以口吐莲花，把自己的架构说的是天花乱坠，但是到最后落不了地一样是没有用的。所以为负拆分是一个实实在在需要靠经验积累的过程。在这个过程中，我们说白猫黑猫抓到老鼠就是好猫，意思就是我们不必要拘泥于微服务拆分的条条框框，适合自己的就是最好的。


我们最终的目的是要解决问题。而有的时候我们做的架构并没有去真正的解决问题，反而制造出了更多的问题。比如说把简单的问题复杂化，一副令人眼花缭乱的架构图，服务与服务之间的力度简直细到像芝麻粒这么大小。你如果问他为什么他会搬出一套理论叫单一职责？单一职责其实很无辜的耶，他被当作了一个挡箭牌。我在工作中也经常的碰到这种会把简单问题复杂化的技术人员。对于他们，我只能说这叫 drumbility 而真正的架构所要做的是化繁为简，拨云见日。


如果我们回顾咱之前学过的 spring cloud 组件，包括 spring frame work 框架本身它的治理理念，你就会发现它的核心思想核心理念其实是非常简单的。它的架构结构分层完全是为了实现一个简单单纯的目的。所以我们说一个好的架构，它往往是大道至简，把复杂的业务抽象化，找到他们之间的共性。就如同如今阿里集团的大中台战略一样，在纷繁复杂的各项业务中抽象出了一套大中台服务。正是得益于这种大道至简的架构思路，大大降低了新业务接入到淘系链路中的成本。这套大中台战略架构就像打通了一个人的任督二脉。不管今后我想踏足到哪一个新的业务领域，都可以让自己的 it 系统急速落地。


2 万架构，我们又不得不说在架构层面中一个非常重要的话题叫做技术选型。很多同学喜欢追赶潮流，长江后浪推前浪，一代杏仁换旧仁。不过大家可要小心，万一潮水褪去，那个裸泳的人可能就是你。不知道大家还记不记得当年 javascript 的框架群魔乱舞的景象那段时间 Java scribe 的新框架层出不穷，很多当时流行的当红框架。现在一提起来可能大家都不记得了，比如说 extjs 还有谷歌出品的 gwt 追赶明星。我们并不要一个理由，我们可以是脑残粉，但是追赶新技术，我们必须要给自己一个足够的理由。为什么选择这门新技术？他对自己的业务都有哪些优势？使用新技术替换老技术都需要付出哪些替换成本？他是否有足够的社区力量的支持？未来的发展如何？可扩展性又如何？我们说技术选型其实是一个没有极致选项的答案。


很多时候我们不能光凭自己对新技术的这种狂热的崇拜，必须要跳出技术人员的视角，从业务层面好好审视这门新技术对业务带来的好处。所以技术选型并不是一个非黑即白的选项，而往往是一个 balance 平衡各方利弊最终做出的一个中和。我们回顾过去十几年间，技术的不断发展可谓是日新月异。最早的时候，我们整个业务系统甚至只构建在数据库层面，使用 procedure 可以玩转所有的业务，这在今天看来甚至是不可想象的一件事情。后来我们就有了 serve let 和 GSP 从静态的网页转换到了动态的网页编程。那段时间 MVC model will control 这个概念非常的流行， GSP 作为视图层， server let 作为 controller 层后面再有业务模型。后来在企业级开发领域又出现了 egb 这种非常非常复杂，甚至可以说是反人类的技术规范。


不过没过多久好像就变了天，我们的企业级开发日渐的向轻量级发展，后来就有了 SS H 三板斧打天下的时代，逐渐的 SS H 三兄弟中的 strats 慢慢的落伍了，而后被 spring MVC 所取代，后面又慢慢过渡到 spring boot 全家桶。直到今天微服大行其道的年代是 spring cloud 的舞台，技术总在不断的推陈诛心。


接下来等待我们的是什么呢？ no matter whats next there must be a next 所以新的东西总会出现的。在这个过程中 please stay hungry 保持你的求知欲，保持进取。那正经的聊完了，老师再跟大家聊一些不正经的同学们可知道中国特色的互联网开发模式是什么吗？大家还在用瀑布模型或者是敏捷开发 HO scrum 等形式。No！No，no.这些模式已经被互联网公司甩掉十几条街了。现在所有的互联网公司括弧中国境内的互联网公司，他们都使用一种模式，叫做超快猛，简单而又粗暴。他充分把个人的脑力、体力以及脾气利用到了极致。比如一些老阿里就经常用自己的事迹告诉新人，要能扛压还要皮实。


可是这种开发模式真的效率很高吗？我不得不打一个大大的问号，放眼世界，真的没有任何一个国家在互联网行业像中国这样发展的这么迅速，似乎我们逐渐走入了一个互联网的陷阱。这个陷阱就好像互联网的大佬们已经为我们挖好了坑，互联网的从业者只不过是在坑里挣扎而已，996甚至都可以说是一种比较轻松的工作制了。在这个意识形态中仿佛没有加班搞不定的事情，那如果加一天班不行，那就加两天好了，在这种大跃进的思潮下，什么都不想拍脑袋就上，只是非常常见的一件事情。对于少数聪明而又高效的员工来说，这其实是变相降低了他们的效率。不过我们也提到了这类员工毕竟是少数。所以从群体收益上来讲，对于管理层来说，这种疯狂加班的鸡血文化在群体的大样本中其实还是会增加企业的收益的，这是一个互联网的陷阱。


所以如果你也是一个聪明而又高效的小伙伴，你就会明白有些事情我们可以做，但是我们要知道这样做是不对的。好了，讲了这么多，这其中各中滋味还要同学们自己去体会。人生就是一场漫长的修行。那今天我们就修行到这里了。下一节老师带大家一起放飞自我，同学们下一节再见。
