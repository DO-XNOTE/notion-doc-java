hello 我慕课网的各位同学们，大家好，这一节我们来聊一聊如何做主链路的规划。不知道同学们在自己的项目中有没有认真考虑过什么是主链路？没有。我想有可能是系统太过简单，用户场景并不那么复杂，又或者是用户量访问并不大，我们没有必要保卫主链路。还有可能大家并没有理解什么叫做主链路。那这里在开始之前，我先跟大家说一说什么叫做主链路，拿我们一个人的人生做比较。什么才是人生中的主链路呢？谈恋爱上学，找到好的工作。这些都太浮夸了。真正的主链路就有两个字，活着。那围绕活着。这个主链路它可能有四个业务场景，分别是吃喝拉和撒没有这四样基础，你人生中的所有重要活动都无法开展。所以对我们每个人来说，主链路就是活着整个一条人生的主线，都是围绕着这个主链路来展开的。所以主链路对于一个系统来说，它是最核心的一个业务流程。如果主链路的任何一个环节出了问题，那对你整个系统来说都是一个灾难性的结果。尤其对于互联网项目这种用户基数非常非常大、访问压力非常高的应用来说，识别出正确的主链路，动用一切的资源保障主链路不失守，是重中之重的任务。

那这里就牵扯到一个问题，如何识别出你重要的业务主链路？再回到人生这个例子，有的同学说谈恋爱是自己的主链路，那可能谈恋爱对他非常重要对不对小情侣一分手了，男的鬼哭狼嚎，宝贝儿没有你，我就活不下去。煽情的话说的再多也不管用。男人的嘴骗人的鬼，你把他甩了，他可能过得更潇洒了。


是说谈恋爱对人生来说是不是主链路啊，你没有谈恋爱，人生就过不下去了，不是那么回事儿。因此这个主链路可不是你随便以为我很重要，然后它就是主链路了。No，no，no.我们要基于一定的事实基础来识别出整个系统整个业务流程中最核心最 key 的那个环节。好预热我们做完了，有请下一页。接着前面讲的内容，我们这里来探讨一下主链路保卫战中三个核心要素。就像打一场战争一样，你要知道你的主战场是哪里，海量的用户请求就像不断来犯的这个敌军一样，你要知道在哪里需要布重兵来把守，这样才能在有限的资源内保障系统最大的可用性，达到什么效果正所谓间隔峥嵘而摧尾，一夫当关，万夫莫开。

那怎么来识别这个主链路非常简单。在电商场景，我们有一个原则叫什么，一切向钱看哪一个钱 money 这个钱。所以说在整个主链路的识别过程中，它的主线剧情就围绕着钱这个东西来展开。你的系统中直接导向利润生成的部分往往就是最核心的主线。我们很容易想到订单系统对不对？你直接牵扯到下单。如果这个环节失守了，用户下不了单了，那你的钱从哪里来啊？再比如说订单系统后面的支付引擎，这个扣款动作完成不了，那你钱是不是还拿不到啊？所以大家记住，我们互联网人非常纯粹的一切跟钱过不去的问题，那就是跟我们自己过不去。所以这个钱一定是主线中的主线。


除了钱以外还有一个要素叫什么，我们叫它 traffic 流量，你的钱从哪里来？从用户上来，用户从哪里来？同学们是不是我们的导流端引过来的流量比如说全国各地铺开这么多的莆田系医院，那个医院挂号处的收费系统是不是主链路啊？没错，因为那关系到这个医院的钱从那里来，但是更重要的是他们的流量从哪来啊？那这些莆田系的医院，它的流量往往通过一些搜索引擎过来。


我这里说一些搜索引擎可能地土炮开的有点大，莆田系医院的流量往往指从某个比较大的在线中文搜索引擎中过来。所以说对于莆田系医院来说，那这个搜索引擎也就是流量的入口，它更是需要保卫的主链路了。没有了这个流量入口，那就像无源之水一样，这些假医院还怎么去骗人对不对？所以总结一下，主链路中钱和流量是两个最核心的考虑要素。


当我们正确识别了主链路以后，我们接着要考虑这样一个问题，底线做人得要有底线，做主链路同样也要有底线。这个底线它在哪里？它是由你来定的，你要清楚的知道当前业务的最小可用性在哪里，怎么来理解这个问题呢？我举一个例子，比如在下单结束后的支付链路中，我们网站可能提供了很多种支付方式，比如支付宝支付、微信支付、银联支付等等。假如你某些支付渠道不可用了，那么你的主链路有没有崩坏呢？并没有。


那我作为业务方，我可以给这个支付系统设置一个底线是什么呢？就是说保障至少有一种服务可用，比如你四种支付渠道坏了三种，那么你保住了最后一种，这整个主链路依然是通畅的，对不对？它可能会对你的利润造成一定的影响。但是总的来说，你要保证整个主链路是通畅的，可以从头走到尾。不过一旦你的系统可用性突破了这个底线，比如四种支付渠道团灭了，在这个整个主链路环节中就有一环断掉了，那我们只能宣告主链路失守。所以在这个底线上，为了保证业务的最小可用性，我们可是要花足心思利用各种的熔断降级补偿，总之是八仙过海，各显神通，力保底线不失。


OK 前面的主线剧情和底线我们分别讲了如何能在抵抗住敌人正面进攻的情况下采取积极的策略。那么假如敌人的攻势太猛，我不小心失手了怎么办？这也是主链路保卫战中要着重思考的一个问题，那就是故障恢复这个做法，那就是百花齐放了。比如在支付系统中，我们可以采用异常的补偿机制来实现故障恢复，怎么来理解呢？比如我们的扣款请求已经发送到了银行那里，但是在我们的系统中，因为种种原因导致这个扣款请求并没有被记录下来。这种情况下，用户在你的系统中看到这笔扣款并没有真实发生，但是在银行那边他的钱却被扣了，怎么办呢？这就要求我们在系统中实现一个对账功能，将你系统中的记录和银行端的记录进行一个匹配。通常来讲，我们做银行集成的时候，经常会听到一个词叫 settlement file 包括一些国外银行也是通过这种文件形式的对账。那么我们通过对账发现了异常交易，就可以针对不同的情况做出相应的补偿机制。比如前面的例子，我们发现银行的文件中有这条记录，而本地系统中却没有。那么怎么办呢？很简单，把这条缺失的记录在本地的系统中重建起来就可以了。

这里看到了一个很特殊的概念叫 SKU 经常做电商系统的同学应该非常了解 SKU 它是什么含义呢？比如你到了情人节想给女朋友买花，店家提供了很多种不同的规格，比如玫瑰花，比如康乃馨，比如蓝色妖姬，这每一种类型我们都叫 SKU 再比如我们大家买衣服的时候是不是要选择尺码还有颜色。那这些尺码颜色的组合我们把它叫做 SKU 它是当前商品下设的一个子类目的概念，通过 SKU 之间的组合，最终确定了你所下单的商品它是什么类型的规格。所以我们看它是不是主链路，它一定是很核心的主链路。因为假设你没有 sq 的话，那我买一件衣服，我怎么选择尺码？我怎么选择颜色，难道随便给用户寄吗？不可能的，因此 SKU 在商品详情页中一定是一个非常非常重要的绝对核心主链路的部分。



那我们接着往下看，库存服务是不是？假如你库存服务挂掉了，你怎么知道当前商品还可以继续往后走下单流程呢？比如说你的库存当前已经是 0 了，你在商品详情页点击的加购物车行为，包括后面购物车的下单行为，一定要把商品跟库存强绑定起来。如果你的库存是0，那么你还能继续下单吗？如果可以的话，那就要产生超卖现象喽，卖家可能发不出货。所以说库存不光是详情页，它是整个电商链路中一个非常核心的主链路，它决定了你这个商品是可卖还是不可卖。如果可卖的话，那么可以卖多少？ OK 我们继续往后看。


这里提到了一个 tfs 副本详情，它是什么？那大家在浏览商品详情的时候，是不是看到下面有很多的图片，包括文字介绍，甚至有一些短视频，那这里实际上是一个 tfs 的 link 也就是一个链接，在这个链接中可以获取它的富文本详情。那么这个富文本详情对于用户下单来说是一个非常重要的强参考因素。即使你没有商品详情中的富文本内容，用户也可以下单。但是它正像盲人摸象一样，它连你这个商品详细的任何说明都没有看到，那直接下单了。那这真的就是真爱了或者是老用户。因此我们说尽管在不需要 tfs 富文本详情的情况下，用户依然可以完成下单动作。但是由于这个富文本详情是对用户下单有一个强引导作用，它对转化率的影响是非常非常大的。因此我们把它也放为主链路中的一部分了。但是在主链路中的优先级可能比不上前面的 SKU 和库存。所以在整个主链路的流程中，大家每一个服务功能也是要梁山好汉排座次的。那我们继续往后看。


这里来了一个叫图片空间。图片空间是什么东西呢？我们在详情页中看到的图包括商品的几个滚动图片的主图，它就是存放在淘系图片空间中的内容，这也是一个会强影响转化率与用户下单行为的要素。所以我们这里把它标黄，也就是说没有它依然可以完成主链路，只不过它的重要性，相对于评论服务这类功能来说会更重要那么一些。


OK 接下来我们看一个打酱油的角色，玩兔视频空间什么鬼，大家放到短视频了对不对？其实我购物的时候基本都不会看里面的视频，我觉得太啰嗦了，通常看一看主图，看一看评论，看看详情就可以了。那么这个打酱油的视频空间，也就算是一个锦上添花的功能。 OK 那在这一页 PPT 上我们理了两个方向，一个是导流端流量方向，一个是商品详情页内置功能。这个方向在这两个方向上，我们分别识别了一些主链路，相信大家对什么叫做主链路上的功能有了一定的理解。所以说从业务架构层面识别主链路是需要基于你的业务场景还要基于具体的数字，比如转化率等等，综合各方要素最终定下来的，可不是拍脑袋定下来喽。好，我们先总结到这下一章，我们继续来探讨主链路规划三部曲之二，同学们，我们下期视频再见。
