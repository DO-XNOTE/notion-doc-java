hello 我们课网的各位同学们，大家好，我们继续上一节的内容讲如何来规划主链路。这一节我们将要讲一个让大家很头疼的话题，购物车。男同胞们看到这个字啊脑袋好疼了，一看到购物车呢，就想到了家里的败家老娘们逢节必庆什么五一劳动节、六一儿童节、十一国庆节还什么鬼街，那都要买一番。对不对？所有商品详情页来的商品是不是都要加到购物车里面呢？接下来我们看一看购物车的哪些内容可以被识别为主链路。

我这里只挑选几个比较有代表性的功能。第一个，删除商品这个功能是不是很重要呢？有的同学可能会想如果这个商品我不想买了，但又不能删除，那岂不是整个购物车都不能下单了吗？飞翼我们规划主链路的同时，也要想一下某个功能可不可以被其他功能替代。如果有可替代功能，那么我们说当前这个被替代的功能一定不是主链路中的一部分。比如说删除商品有哪些功能可以替代啊？在购物车进到下单页面之前，是不是每个商品前面通常会有一个小勾号你选中需要结算的商品，假如你不能删除商品，那可不可以把它前面的勾号给它点掉，这样的话你这个商品也不会进入后续的下单页面了。所以我们说删除商品的这个功能是有替代方案的。因此对于这类场景来说，通常不会被考虑成主链路中的一部分。


接下来我们看更新数量。这个场景在购物车中，我们可以通过点击加号或者减号来更改购物车中的商品数量，也可以直接更改这个数字。假如这个功能不可用，我们有什么替代方案吗？很简单，我们可以删除商品，然后再到商品详情页指定好商品数量，一键加入购物车。那如果删除商品的功能也坏掉了，别急，我们还有一键清空购物车的功能，如果这个功能也坏掉了怎么办？你的购物车中的商品永远只是数量。一我想买五个，依然有办法，大家现在买东西都是包邮的，对不对？你连下 5 次单喽。所以说对于更新数量来说，它同样有多种替代方案，那么它是不是主链路中的一部分啊？当然不是啦。 OK 接下来往后看商品列表服务。


这个可就重要喽，你在购物车里是不是要看都买了哪些商品啊？你如果什么都看不到，你还会点后面的下单页面吗？肯定不会的，这个功能一定是主链路中很核心的一环，甚至我可以说它是整个购物车中最重要的两个功能之一。 OK 下一个功能我们看叫 UMP 优惠。它跟前面好像不同了。前面在商品详情页， UMP 优惠服务是个摆框，对不对？那这里为什么变黄了？难道在购物车页面， UMP 优惠的重要性比商品详情页更高吗？没错，你说对了，为什么更高？它会影响转化率？比如说我在前面添加商品的时候，看到这个商品是买一赠一或者是买两件减五折。那么我开开心心的加入到购物车，可是在购物车结算的时候，我看不见这个优惠了。


那对于用户的心理来说是什么样的？我感觉自己被欺骗了对不对？我们不光要研究业务场景，也要研究用户的这个心理的变化。因此我们说在购物车上的优惠计算结果是会强影响用户最后点击下单页面的行为的。 OK 我们从用户行为学的角度再设想一个场景。前面我们提到了商品详情页的优惠计算和购物车以及下单页的优惠计算都是不同的服务接口。那么假如在商品详情页的优惠计算服务挂掉了，你没有看到优惠。那么你点击商品到购物车之后，在最终要结算之前，在购物车中看到优惠生效了。那对用户心理是一种怎么样的影响？他会阻止你下单吗？并不会，因为当你在选择买这件商品的时候没有看到优惠，而在最终决定要下单的时候却看到了优惠，这是一种失而复得的感觉。对不对？它并不会对你的心理形成冲击。反之如果你在商品详情页看到了优惠，而在购物车没看到优惠，这就像我前面说的，对用户的心理来说，这会产生一种很大的落差，用户甚至会认为这是一种欺骗，也许会就此终止下单流程。所以我们说 UMP 服务在购物车的场景下，它的重要性要比商品详情页高很多。
那我们接着往后看，导流购物车里我们也可以看到五花八门的边栏广告或者下边页面底端的广告。那它其实也是用户推荐的一部分。这个重不重要呢？大家想一想也知道了，完全不重要，你都快要下单了，我只不过是锦上添花，想诱导你再多买几个商品喽，对不对？它并不会实质性的影响你这个下单的流程。


OK 我们看最后一个功能，生成订单相当重要，同学们它是有购物车导向最终下单付款页的。这个功能如果他出问题了，可想而知所有的用户行为都在购物车这里被终结了。 OK 那我们从整个购物车中几个比较有代表性的功能中，识别出了几个核心主链路商品列表服务和生成订单，还识别出了一个次重要级的 UMP 优惠对象。看到这里，我想大家应该都是怀着比较轻松的心情来看的。其实我每次回顾这些主链路的时候就一句话往事不堪回首，这每一个主链路后面都是背着无数条的人命。怎么说你稍加不慎就是一个故障单在阿里里面，你要是拿到一个 p2p1 级别的故障单，那真的可要背一口大锅喽。如果因为你的问题导致了主链路降级甚至是熔断，更有甚者服务不可用，那么你的年度绩效可能就会被打上一个 3.25 的烙印。


很多同学可能不知道 3.25 代表什么意思呢？阿里集团评绩效是遵循 361 的标准，前 30% 是优秀，通常来说是 3.75 的绩效分，中间 60% 是 3.5 的绩效分。最后10%，那就是3.25。 3.25 就代表着你这一年白干了，不能晋升，不能转岗，也不会发给你新的股票。而且你当年的年终奖会极惨。所以说一定要谨慎对待主链路，确保主链路万无一失。尤其应对那些用户量非常大的电商系统来说，要提起百分之一百二十的精神来应对。


OK 我们看完购物车以后，下一节来了解一下最后一个环节，订单页。

这个页面就一个非常重要的功能叫提交订单，你 996 挥洒的汗水就要成为公司老板口袋里的一块钱了。我们系数提交订单中的几个关键功能。第一个，商品列表服务重要不重要非常重要。如果你在商品列表中一个商品都没见到，然后结算订单这里让你付 100 块钱，你会付吗？傻子才会付，所以说它是不是主链路中的一部分呢？必然是的。那我们再来看这个 UMP 服务这个 UMP 有意思了，他表演变脸，详情页是白的，后来变成黄的。那再提交订单页变成红的了怎么回事啊？这红脸的曹操唱什么戏呢？我们想在这个关头提交订单的时候， UMP 参与的是什么？计算订单价格的计算？如果你 UMP 服务在订单页面不可用了，那代表着什么呢？代表着你所有的优惠规则都不会被计入订单的总价。我想大部分用户在下单之前如果看到自己的优惠规则没有生效，那他一定是不会点击下单按钮的。那么你的下单行为就在最终提交订单之前终止了，都快煮熟的鸭子这下飞了。另一种情况，假如用户一时走眼，并没有注意到营销规则未生效，而是点击了下单。


那么对用户来说是不是会多付钱呢？作为用户，他怎么看待这件事情呢？心态好的用户可能会联系客服解决。那心态崩掉的用户怎么办呢？ 315 打起来。所以说 UMP 服务在这个提交订单的环节是非常非常重要的，一定要确保它的最后关头不失手。 OK 我们接着往后看支付系统。严格来说，支付系统实际上是在订单系统后面的流程，也就是下游系统这块决定了你客户的付款行为最终是否成功，不用说也知道非常重要了，你多扣钱、少扣钱，甚至这笔钱压根就没扣，都会带来一系列非常麻烦的问题。 OK 再往后看地址模块有的电商系统是在购物车页面选择地址，大部分电商系统可能都是在提交订单的时候选择地址，那这个模块为什么非常重要？比如说我买了一箱杜蕾斯，我本来想寄到我家，但是这个后台地址模块出错了，我寄到了我丈母娘家，他肯定不会觉得你这个女婿很体贴对不对？所以说地址模块也是主链路中的一部分。


但是地址模块通常我们也会给它设置一些降级的措施。比如说当用户模块出故障的时候，我强制地在降级流程中指定用户的收货地址为默认地址。从我个人角度来说，一个公司如果这样做，我会认为这个公司的价值观是有问题的。你在地址模块出问题的情况下，强制使用默认地址。那么这个地址真的是用户想要寄送的地址吗？未必。那么你为了赚到这笔订单的钱而不惜损害用户的利益，从商业的角度来说，电商公司都是为了钱无所不用契机。但是从价值观的角度来说，我真的非常不能认同这种行为。


刚才我说的地址模块降级实际上是一个真实发生的案例，它发生在哪里？我就不必指名道姓地说了，很多同学甚至可能亲身体验过这次地址模块的降级，我认为公司还是应该遵守一定的价值观和道德底线的，而不是把价值观挂在嘴上。往往越强调价值观的公司，可能自己就离价值观越来越远了。
OK 那我们前面提到的这四个模块，它都被划为了核心的主链路。大家有没有发现一个规律，从订单详情页到购物车页面，再到最后的提交订单的页面，主链路模块所占所有模块的比例是不是越来越高的？详情页面的主链路最少，购物车的主链路模块多了一些。然后到了订单页面，基本上每个功能都是主链路了。那我这里就不得不说起一个叫漏斗模型理论的东西。也就是说所有的电商场景实际上都可以把它抽象为一个漏斗模型。那么在漏斗的最上端是导流或者商品详情页这类承接用户流量的应用。接着到了购物车页面，再到下单页面，它的用户流量是会逐渐减少的。也就是说你可能 100 个用户进来它在漏斗的最上方，那么最终下单的用户可能只有两到三个。所以漏斗模型理论一定程度上可以反映一个转化率。


那么如果对应到主链路规划，通常来说越在漏斗窄端也就是在下端的应用，它所提供的功能被划分为主链路领域的比例也就越高。 OK 那这一节的内容就到这里结束了，我来总结一下，通过前面的商品详情页以及购物车页面和订单页面，我们大体了解了在电商场景下规划主链路的几个思考的方向。
最后我们又提到了一个漏斗模型理论，越接近漏斗尾端的应用部分，它所提供的功能在整个链路中相对来说重要性会越高。大家可以顺着这两节中的例子，对照自己的系统思考一下，看看自己的系统如何做主链路的规划。 OK 这一章就讲到这里了。下一章我来带大家看一下阿里系的容灾策略，容灾其实也是主链路保卫战中不可或缺的一部分。那同学，我们下期再见。
