好有请下一页大家可能平时都听说过，要多机房，什么异地双活，什么两岸三地多备份，其实这都是为了保障系统的高可用性，在部署策略上做的一些调整。那么阿里系在整个应用的部署策略中也经历了一番演变的过程。最初的容灾策略非常的简单，他只要求同城双火什么意思呢？比如说我的第一个机房中心设立在杭州，我可能在杭州的西湖区有一个机房对不对？那么我在杭州的余杭区再设立一个机房，那么这个叫同城双户，一个城市中有两个机房互相备份。那么如果有一天一对挖掘机浩浩荡荡的开过来了，一不小心咔嚓一下，把西湖区机房的电缆铲断了，怎么办？我们还有余杭区的机房对？不过大家有没有想过，假如有一天台风肆虐，不小心刮到了杭州，那有可能整个杭州的电信工程是不是都会受到牵连呢？那指不定这两个机房的线路都会被这个自然灾害所影响，夫妻双双把家还这两个机房都挂掉了。


那对应这种情况我们怎么办呢？后来阿里系做了一番调整，他把应用部署策略定为异地短距双火。所谓异地，就是只在两个不同的城市做多机房。那这里的短距是指两个异地城市之间的距离，比方说杭州和上海，这之间只有小几百公里的距离。那我们叫短距双火。比如现在新上线的集团应用，它都要求你必须要部署在多机房。那我们给自己应用申请服务器的时候，就会选中多个机房来进行分配。就拿我刚才说的例子好了，假如上海和杭州我们部署了两个机房，那么在某些极端场景下，比如海啸和台风，它可能会肆虐整个沿海城市。对不对？如果我们的机房距离相对较短，而且又全部分布在沿海。那么有没有一定的可能性被某个非常强大的自然灾害一波带走呢？我们不能说是完全没有这种可能性，即使没有自然灾害，也可能有一些你不可预料的事件发生。


那么我们提出了一个比 ED 短剧双活还具有高可用的架构叫什么异地长距多活。那这个长距多活有多长的距离呢？通常我们要求有一千千米以上的距离，这个范围是一个相当大的范围。你的两台机房机器要在距离一千千米以上的两个城市进行部署。比如现在集团常用的有杭州机房，还有张北机房，那么还有美国的机房，这样常去多活的部署架构，可以在最大范围内保证你免受各种突发情况的干扰。
那你说为什么早些年它不使用常去多活呢？其实有很多原因，一方面是资金的限制，你做异地常据多活，比如你设置一个美国机房，你搭建这个异地海外机房的费用和你在杭州本地搭建一个机房的费用那可是一天一地的差别。海外机房的耗资是相当巨大的，而且常去多活牵扯到数据同步的问题。比如你在杭州机房同步到美国机房，这之间相隔万里，你如何保证数据的时效性？所以总结来说，异地常去多活考验的不仅是你的资金实力，也是你的技术能力，你是否能做到低延迟的情况下，把所有数据同步到各个不同的 ED 机房来。


不光如此，在阿里系里面集团还会时不时的做出一些惊人的举动。叫什么？叫容灾演练，他有事没事就把你的某个机房给他切断。假如你的事业部没有做异地多活，那么你的唯一的机房被切断了，所有的线上业务就停滞了。我们当时刚做新零售业务的时候，只申请了一个机房，然后一不小心就在集团发起了常规容灾演练上中枪了。


所以说后来每次我上线新应用去申请机房的时候，我不管它应用承接的流量再小，它即使这个应用集群只有十几台服务器，我也会分别在杭州机房、张北机房等多个机房里面申请机器分配，就是为了集团这种没事跳大神发起的容灾演练。其实从集团的角度来说，这也是一种倒逼的策略。我通过常规的容灾演练就可以迫使各个事业部一定遵守异地多活机房的部署策略，最大限度的保障系统的高可用性。好，我们进到下一页，这里有跟大家讲一讲阿里系应用的单元隔离策略。

我们在系统部署和应用之间的调用上有两个概念，一个是叫中心服务，另一个叫单元服务。大家可以简单的把这个中心和单元理解成两个不同的服务集群。那么这之间是通过什么样的方式来进行数据同步的呢？我们使用的是 DRC 数据同步策略，它是基于 Binlog 的一种同步机制。 Binlog 大家有没有听说过呢？当你向数据库中进行一些增删改查的 CQ 指令的时候，你的数据库会产生一些 log 文件，我们把它叫做 bin log 现在市面上有很多数据同步的解决方案，他们大多都是基于 bin log 做的同步，也就是分析数据库的 log 文件，然后将数据改动按照顺序的方式发送到另一个机房。


但是这种数据同步策略对大部分的一线开发人员是很少去涉及到的。但是如果让你设计一个高性能低延迟的数据同步方案，那你就不得不去考虑这件事了。 OK 那我们为什么要把应用分为中心和单元呢？其实如果你在阿里系应用的内部，根据你自己的支付宝号码去搜索你的用户，你可以看到你的用户界面上有一个属性，它会区分你是单元用户还是中心用户。比如我的支付宝账号是中心用户。那你作为当前这个用户发起某些指令的时候，比如说付款或者是添加商品到购物车，然后点击下单流程。
这整个链路的所有请求我们都会把它封闭在同一个单元进行。比如说你是一个中心用户，那么你接下来所发生的所有业务数据的请求都会在中心服务这个节点下完成。如果你是一个单元用户，同样的你所有的请求都会被封闭在同一个单元中进行。这么做有这样几种考量。一方面你的所有用户请求被封闭在一个单元，这就相当于是在一个服务器集群内部来进行调用。相比较调用其他单元来说，它的性能会得到很大的提升。


另一个方面，这也是某种负载均衡的策略。比如我有 10 万个用户，那么这 10 万个用户会被我根据单元的负载能力分配到不同的单元进行处理。这个听上去是不是非常像我们在瑞本章节学习的一致性哈希的策略对不对？根据用户的 ID 将用户分配到某一个机房来进行访问。
那么假如某一天我一个单元挂掉了怎么办？那我在应用层面只需要把用户的流量切换到后面的单元或者是中心节点就可以了。因为中间有 DRC 数据同步方案，会保证你在这个单元做出的数据修改会被同步到其他的节点。这种基于用户数据做切分以及封闭单元的操作在大型互联网应用中是非常常见的。

首先是实时监控维度，我们需要获知这么几个信息，一个是 QPS query per second 就是你每秒钟所要承接的访问请求数量，还有 rt response time 你的接口响应请求的时间是多少？从这两个指标我们可以衡量出并发度。理论上你的并发度越高，你接口的响应时间也就越长。


我们要设置一个底线，也就是说我能容忍这个接口在多少秒内返回是可被业务方接受的。比如我在自己的商品详情页的业务中，我设置了营销优惠的 rt 容忍程度是 1 秒钟。也就是说如果淘气的 UMP 服务没有在一秒钟内把我的优惠计算出来，我就直接做丢弃静默处理，不显示优惠内容。这里要根据你调用的接口，它实际的作用场景以及重要程度来确定你能容忍的返回时长是多少。


还有一个维度是从流量的角度，这个流量并不是指广义上的带宽占用量，它也可以指示什么呢？线程数。比如说我的应用有一个线程池，可以承接 1000 个线程访问请求，那我可以给这个线程池设置一个可变长度，也就是说它最高可以达到 2000 左右的线程。当我侦测自身服务的线程压力从 1000 逐渐上升到 2000 或者是某个特定的阈值的时候，我就要采取一个措施，它可以是限流，也可以是将你的流量通过降级流程导入到一个排队的序列中。


总之目的是要尽可能减少自身所承载的访问压力，并且减少施加给其他业务方，比如说下游应用或者是缓存或者是数据库的压力。那从这个角度来考虑，我们也可以对数据库的线程池做一个监控。因为数据库往往是整个链路中承载压力最薄弱的一环，即便你当前的业务接口承载的压力并不高，但是如果多个服务同时访问一个数据库，把这个数据库的线程打得非常高，那么数据库上的瓶颈同样也可以触发你接口的降级流程。


所以在整个阿里系应用中，我们会设置很多的监控来监控你当前的数据库性能，它的连接数并发请求数分析你哪些 SQL 语句跑的时间过长，数据库真的是一个需要重点盯梢的环节。还有一个维度是什么减负。我想 90 年代上小学或者初中的同学们可能都听过这样一个词叫减负。那个时候国家大力推行给学生减少作业量，减轻大家的负担。减负两个都是负数，结果负负得正，压力反而增加了。那在我们熔断和降级当中如何进行减负呢？有这样两个途径，第一个我们叫做 feel fast 这里我既可以通过降级流程来给它返回一个默认值，作为一个静默处理的快速失败通道，也可以通过自动触发的熔断开关将你的请求直接打回到降级流程。所以这是一个节流的过程，减少远端访问来的请求。那我们在 spring cloud 中就可以通过 high streaks 设置的一些指标参数来自动触发它的降级熔断器的开关等等。那么除此之外，经验和教训告诉我们，我们一定要设置一个什么人工开关来进行人工干预。我可以动态的调整在降级和熔断中的指标参数，也就是阈值，或者它直接可以是一个手动的熔断开关。当我开启的时候，整个服务直接打向熔断流程，进而将请求导入到降级业务中。
