快速进入这个 sentence 在生产环境下如何去进行使用？首先我们来说一说哨兵，它的生产环境中有几个比较重要的特点。首先第一个就是我们的这个规则管理。

规则管理什么意思呢？就是说我们在这个集中管理和推送规则的过程中，我们的 sentence call 它提供的 API 和扩展接口，它是来接收信息的。那开发者需要根据自己的这个环境选取一个可靠性的推送的这个规则的方式。那同时这个规则最好在控制台中能够做一个集中的管理。也就是说我们对于之前我们看到规则那些规则都是说，我设置完了之后，它是通过这个 HTTP 的那个请求发送给我们具体的某一个 client 端，然后去进行热更新。当然它这种规则是基于内存的，也就是说当我们的服务挂掉的时候或重启的时候，这些规则都没了。


那接下来其实我们应该做什么呢？我们应该做一种叫做可靠性规则的一个集中管理的这么一件事情。 OK 这是第一点规则管理。然后第二点就是我们对于这个规则推送了，那规则推送也很简单，我们之前在使用的这个控制台跟我们自己的那个 and 去集成的时候，它都是控制台直接去推送给我们的 client 端，就是说它是发 HTTP 去推送的。但是现在的规则推送，就比如说我们要跟这个携程的阿波罗或者是阿里的 nicos 或者是租 paper 或者是 Redis 这些东西去进行集成。这个时候规则的推送可能就不再是控制台去推送了，而是说控制台只是做一个更新持久化的像阿波罗拉 cos 等等，然后由我们的这个持久化的数据源阿波罗动态的去推送给我们自己的这个某一个服务，可能就是需要转换一下角色。

然后接下来就是我们比较关注的一点就是我们的这个流量监控。那流量监控一定是支持这种可靠的快速的实时的监控以及历史的监控数据的一个查询。那 sentence call 它其实是记录秒级的资源的运行的一些情况，并且它提供了一些这个 API 去获取资源的一些运用信息。在这里其实 dashboard 它其实是主动的去拉取去聚合我们这些平暗端记录的那些就是在日志里面所记录的那些就是流控的信息，然后去做一个统计分析。然后接下来一点就是整个规则推送管理。在我们的这个 sentence 中，它到底有几种模式？在这里我们来看一看，这是官方的一个文档。

第一种模式叫做原始模式，第二种模式叫做破模式，第三种模式叫做 push 模式。那原始模式我们先来说一说，原始模式它就是 API 将规则推送至客户端，直接更新到我们具体的某一个 CAM 端的一个内存中。比如说你是流控规则，就是我们的 follow ruler manager 然后这是扩展写数据源叫做 write ball source 那这个有什么优点呢？就是简单，没有任何依赖，直接通过 dashboard 推给我们具体的我们的 client 端就好了。缺点就是说不能保证一致性对吧，数据可能会有一些这个脏读的现象。


然后规则保存在内存中重启它就消失了，这是一个严重的问题，它也是一说严重不建议在生产中去使用。那你规则没了，那这个你还得重新配置规则，这个太麻烦了对吧，这是原始模式。其实原始模式就是用于我们去刚开始你入门的时候，但是生产环境中是不建议去做的。第二种模式叫做破模式，破模式表示拉模式。拉模式它是什么呢？就是客户端主动的向某一个规则管理中心去定期的去轮询拉取这个规则。比如说我说我每一分钟我去持久化的中心，比如说 Redis 或者是 nicos 或者是阿波罗，我去每一分钟我去拉取一下这个规则。那这种的好处也是简单，没有任何依赖。然后规则是可以持久化的。注意这一点是规则可以持久化的，但是它的缺点就是也是保证不了数据的一致性，保证不了实时性。因为可能你拉取的频繁它会有性的问题，拉取的不频繁，可能我修改完了之后，比如说现在我刚修改完了之后，过一分钟之后才拉取，它就会有一个比较高的延迟，这就是拉模式的一个缺点。


然后我们再看这种 push 模式。那 push 模式说白了，我们看到最后面它的这个生产环境一般采用 push 模式推模式去做数据源。也就是说我们可以通过客户端注册监听的方式，然后去修改我们的什么，耐克斯做 keeper 等配置中心，然后通过配置中心特有自己的这种 watcher 或者是 list 的机制，当然是我们的客户端去监听，然后及时的去感知它的好处是什么。好处就是说规则能够持久化，然后能够一致坏处就是有一点就引入了第三方的一个依赖，没办法。目前来讲主流的方式都是使用这种 push 模式。
