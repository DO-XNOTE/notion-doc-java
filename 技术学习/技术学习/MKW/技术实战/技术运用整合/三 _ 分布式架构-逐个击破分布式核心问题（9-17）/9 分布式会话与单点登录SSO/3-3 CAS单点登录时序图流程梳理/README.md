接下来我们就来聊一下咱们的一个单点登录了。单点登录的话其实在面试过程中也是有可能会被问到的，它主要是有两种情况。第一种情况就是咱们的一个顶级域名，如果说顶级域名相同的话，它的一个单点登录  SSO 其实就可以去使用 cookie 加上 Redis 共同的实现。因为在我们的后端，由于是使用的缓存中间件Redis，整个在我们的集群或者分布式系统的这个环境之下，用户的会话在我们的后端是可以获得的。


另外对于咱们的一个copy，由于它的一个顶级域名是相同的，所以它的 cookie 是可以在所有的产品线在所有域名下共享的，所以这个时候我们完全可以使用这种方式来实现咱们的一个单点登录，这个也就是 8 杠 10 这节课里面所阐述的一些内容，它的原理的话，本质上就是分布式绘画，也并不是很难。


如果说这个时候我们的一个顶级域名发生了一个变化，有两个不同的网址，有两个不同的网站，甚至是多个的话，他们能不能够去做到单点登录，其实也是可以的。OK，假设我们现在是有一个MTV，另外还有是一个music，有两个不同的网站，有这样的一个网址，我们是可以通过另外的一种手段去实现它的一个单点登录。


那在这里的话，其实我们就需要去借助 CAS 这样的一个系统了，那么它其实全称就是 center authentication service，它其实是一种中央认证服务，它可以去把我们的一个会话信息，或者说是我们的整个全局的 token 保存在 CAS 里面。随后当其他系统发起调用的时候，完全可以通过它去做一个中转，它其实就是一个中介，或者说是一个中转站。对于我们的一个域名来讲，那么预先我也是配了一下来看一下。那么MTV， music 这两个是两个前端的站点，我们会使用的。另外对于我们的 CAS 来讲的话，我就可以使用CAS，当然你也可以配置为 SSO 这两个取其一就可以了。


好，随后我们来看一下。那么在这里之前我们的文档里面也是贴了这样的一张时序图，我们这一节就根据这样的一张图，我们先来聊一下。首先在我们的客户端，其实这里就是用户在访问浏览器。如果说我们这个时候用户在我们的所有的系统，他从来没有登录过。假设现在是第一次访问的我们的 MTV 网站，然后他要去验证一下，肯定喽他是没有登录的。如果说没有登录，这个时候他会携带相应的参数请求到c，s，然后这个参数其实就是一个 return URL。携带的这个 return URL 参数是什么意思？它主要是用于去做我们的一个请求的回跳，因为我们现在是多个系统，所以说你在这里如果说处理完毕以后，你要去做一个回调，回调是基于这个 return URL 的。


OK，然后在我们的 CAS 这一端的话，肯定也要去验证一下我们这个用户他有没有登录，很明显他也没有登录，所以这个时候他会把一个登录页面展示给我们的用户，让用户在浏览器上去输入用户名和密码去做一个登录。登录的话，其实相应的一些登录的判断，我们是要在 CAS 的后端去做一个判断的，也就是这个部位。


第七步，假设是登录成功了。登录成功之后，那么我们有几步操作是需要去做的，比较重要。第一个是创建用户的会话，那么这个其实就是用户的一个分布式法，我们之前是有操作过，所以这一个步骤是一模一样的，把用户的会话给创建了。最后第九步，第九步的话是创建用户的全局门票，这个是什么意思？这个其实也就是生成了一个 token 之后，它是作为一个用户的全局门票保存在 CAS 这一端的，它会保存在我们的一个 cookie 里面，同时他也会在 Redis 和我们的一个用户会话，也就是用户的 ID 做一个关联的绑定，就是代表我当前这个用户在我们的全局的范围内，在我们的 CAS 系统里面是做了一个登录的，所以这个是第九步所做的一个操作。


随后第十步的话，它要为用户去创建一个临时的票据，这个临时的票据创建完毕之后，是给我们的 MTV 这个站点去使用的，因为这个时候用户要去做登录对吧，所以它会回跳到这个位置，回跳到 mtv 的时候是根据这里的 return rul 回跳过来的。回跳的同时它会携带上这个临时票据，其实也就是我们的 CAS 颁发给他的。然后他要去做一个校验的时候，他是携带上这个临时的票据，再一次的发起请求，发到我们的c、a、s，让他去做一个验证。


如果说校验成功，判断一下它这个临时的票据在我当前这个系统里面是有的，是存在的，也就是校验成功了。所以这个时候的话，它可以去获取在我们  CAS 里面的用户的会话，用户汇报获得以后回传回跳给我们的MTV，那么在我们的 MTV 这个站点上就可以拿到用户的会话了，所以他就可以去做一个用户会话的保存这一步操作。


保存的话，其实它是保存在它的一个前端，也就是在浏览器的 cookie 里面。这样子在它的一个站点上，在它的网页里面就可以展示说欢迎某某登录，你目前已经是成功的登录了，他就可以做到这样一个操作。OK，这个是我们的一个初次从来没有登录过的情况之下，假设这个步骤我们都已经是做过了。接下来我这个用户他要访问到我们的 music 音乐站点，这个时候怎么办？来看一下。首先他也是初次访问到咱们的这个站点，验证一下有没有登录，很明显没有登录。随后携带他的一个站点 return 去请求  CAS 系统做一个校验，这个时候他会发现我们的  CAS 系统其实他已经是登录过了，他会进行一个校验的。这个登录是根据谁来，它其实是根据这个根据用户的一个全局门票。这个全局门票只要是存在的，所以我们的用户会话已经是有了。用户全局的会话有了之后，就代表你的这个用户当前这个发起的请求，我不需要再让你去做二次登录了，他就直接可以去创建一个临时的票据。创建了临时票据之后，再一次的回跳，携带返回给我们的 music 站点。下面的这些步骤。


下面的这些步骤和我们上方的这一些步骤是一模一样的，来看一下它也是一样，它会携带着这样的一个临时的票据，然后让咱们的  CAS 去做一个校验，校验成功回传用户，把用户的会话返回给他，然后在他的一个前端的站点去做保存会话的操作，这样子就 OK 了。那么在这边是需要去注意，就是说校验成功之后，这边由于是一个临时的票据，所以临时票据一旦使用完毕之后，那么你是需要去删除的，因为它是一次性使用的。OK，然后在这个地方，不管是在我们的哪一个站点，在前端在保存用户会话，在这个过程的时候，其实它仅仅只是在   Token 里面去把用户会话的信息做了一个保存。因为用户的会话由于我们在后端使用的是一个Redis，只要我们把用户的 user ID 以及是这个会话里面的一个 token 携带，放到 header 里面传过去，在我们的一个后端是可以获得用户整个信息的，所以这一块内容的话在前端保存就可以了。
