前面几节课我们是基于 Redis 去实现了一个用户的会话，其实也就是一个分布式会话。接下来我们来看一个问题，就是说在我们的一个用户中心，在用户中心和用户相关的一些界面，其实我们在发起一些请求的话，都是会调用咱们的一个后端的接口的。在进行一个接口调用的时候，那么其实现在我们并没有去为每一个用户做到一个相应的一个限制。其实我们就应该要去做一个最基本的一个限制，其实也就是一种基于权限的一个拦截。比方说现在如果说当前是这个 Tom kit 用户登录的，它是可以去修改对应的一个用户信息的，如果说你没有去增加一个拦截的一个判断，没有去判断这个当前用户是不是这个 topic 的，那么也就是说别人可以用其他的某种手段来更改非自己的个人信息，那么其实这个就是涉及到一个用户的一个权限。


这这个的话我们完全可以通过拦截器来解决这样的一个问题，并且这个问题是和我们的会话相关联的，既然有了会话以后，那么我们只需要去判断用户的会话是不是一致就可以了。这个其实也就是我们之前所用的一个门票，也就是token，基于 token 的一个判断就可以了。那么接下来我们就来写一下咱们的一个拦截器，来看一下是如何去实现的。

必须要去实现一个对应的接口，这个接口其实就叫做 handle 了。我们先写一下 handle 了， interceptor 也就是这个，那么在这里面我们是需要去把这里面相应的一些方法要去实现的。 override methods。其中我们来看一下和拦截器相关的，其实总共是有三个方法，我们点一下OK，这样子就会帮我们做到一个相应的生成。这三个方法对应是什么意思？我们可以加一个注释。


首先第一个是一个 pre handle 了，这是一个我们的一个拦截的请求，就是说当我们的一个用户请求请求我们的接口了以后，那么在调用我们的 control 了之前，所需要去进行的一层处理也就是这个 pre handle 了。写一下就是拦截请求再访问 control 了。调用之前好，OK，那么随后的话还会有另外的两个，那么一个是叫做 post handle，那另外一个叫做 after completion，那么这两个是什么意思？我们直接把注释给加上。


这个其实是在我们的一个请求，是在我们的 Ctrl 之后请求访问 control 了之后，渲染视图之前。因为其实我们会有一个渲染视图的过程，渲染视图就是我们的一个数据的一个渲染，渲染到咱们的一个页面上，在渲染之前，那么它会进入到这个 post handle 里面去。此外的这个 after 这个其实也是在请求访问到 Ctrl 了之后，但是在进入这个方法之后的话，就是我们的一个视图是已经被渲染了，所以是在渲染视图之后。OK。他们是这样的一个区别。对于我们来讲，我们要去做拦截的话，其实我们应该是在进入到 Ctrl 了之前，在调用 Ctrl 了之前去进行我们的一个访问。所以在这个 pre handle 了，在这里面去做一个对应的一个处理，去处理我们的一个用户请求就可以了。OK。


另外对于我们一个用户请求的话，其实在这边很明显它是返回了一个不保值，你可以返回一个false，你可以去返回一个true，那么这是什么意思？我们在这里也去加上，那么 force 是代表咱们的一个请求被拦截，被驳回，也就是说验证出现问题，这样子的话我就不会让你去继续的访问了。


那么如果说是 true 的话，如果说是true，就代表我们的一个请求在经过验证校验以后是 OK 的，是可以放行的， OK 吧。那么这样子的话，一旦是返回了处，那么我们的请求就能够到达咱们的 Ctrl 了，可以让 Ctrl 了再去做一些相应的处理，所以它就是这样的一个处理的流程。


OK。那么在这边我们先在这边是直接写一个force，我们就当做是当前的请求是被驳回，然后你可以去做一个测试的话，直接写一个 s 到UT，为了测试我们就可以先写一下，就是进入到拦截器被拦截好。OK，那么写完了这样的一个 intercept 拦截器以后，那么我们是不能够直接去使用的，你要去使用的话，那么我们就必须要去做相应的配置，那么这个配置的话其实也就是一个注册，你是需要把这个相应的intercept，你只要是编写了intercept，那么你就需要把它们都注册到咱们的容器里面去。那么如何去注册，我们来看一下。
