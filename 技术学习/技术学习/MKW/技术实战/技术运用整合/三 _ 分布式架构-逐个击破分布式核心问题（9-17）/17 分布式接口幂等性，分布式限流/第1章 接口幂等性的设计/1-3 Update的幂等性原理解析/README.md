那么我们怎么去更新呢？一般的情况下都是先查出要修改的这条数据，然后把这个要修改的数据返回到页面当中，将这个数据的版本号放在隐藏域，这里边有一个关键词，就是数据的版本号修改操作一般咱们在系统里怎么做，是不是一开始都要进入一个列表页比如说咱们的用户列表页包括订单列表页，它都是先有一个列表页，然后我们在列表页当中每一条记录的后边有一个修改按钮，然后跳到具体的修改页面。比如说我要修改其中的某一个用户，我要修改其中的某一个订单。这个时候这个订单 ID 和这个用户 ID 是不是就是它的唯一业务号？因为我们跳到详情页，要根据这个订单 ID 或者用户 ID 把相应的这一条数据给查询出来。那么这个订单 ID 和用户 ID 就是唯一的业务号。我们在这个详情页把这个信息修改完成以后，我们要根据这个唯一的业务号去更新数据。那么这个时候我们更新的这一条数据是不是就是一条唯一的数据啊？那这个数据版本号是什么意思呢？因为我们这个时候在更新的时候要去使用 update 的乐观锁。


一提到乐观锁，那么咱们肯定要联想到数据的版本，那么这个时候数据的版本号就起到了非常重要的作用。当然了像咱们前面在分布式锁的那一章节也给大家介绍过，这个数据版本号，咱们可以用数据的 update time 更新时间作为它的数据版本号也是没有问题的。然后用户修改数据修改完了以后点击提交，把这个版本号也一同提交到后台。然后后台在更新这条数据的时候，要使用这个版本号作为更新条件。

那么像这种情况下，这个 up day 的语句它也是一个幂等性的操作。那么什么时候这个 up day 的语句变成了不是幂等性呢？比如说咱们这个用户表里边还有另外一个字段，它记录着用户的修改次数。想一想有这么一个字段是不是特别有意思了？同样我还是进入到修改用户的这个页面，我点击第一次提交，它标志着我修改了一次。那么这个时候我的修改次数应该加1，但是页面没有给我响应，我又连着点击了多次这个提交按钮。那你说后边修改次数它到底要不要增加多次呢？还是说只增加一次呢？这个咱们当然是只增加一次，因为按照我用户的意图，我其实只是修改了他一次。虽然我点击了多次的提交，但是你不应该把这个修改次数给我增加多次。


那么这个时候这个接口的幂等性是不是就不具备了？那么这个时候咱们就要使用这种 update 语句的方式，保证接口是幂等性，让他多次操作，他后边的修改次数还是只增加一次。这个咱们就是使用乐观锁，然后还有一个 update 与生计来的行所去保障接口人民导向。然后没有业务单号，咱们就要使用 token 机制。好到这里，这个 PPT 的内容就给大家介绍完了。接下来咱们还是在前面的项目当中，用前面的例子咱们再做一个更新的操作。
