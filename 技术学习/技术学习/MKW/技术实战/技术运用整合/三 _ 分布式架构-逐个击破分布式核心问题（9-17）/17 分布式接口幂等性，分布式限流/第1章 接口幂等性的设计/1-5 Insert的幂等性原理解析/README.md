那么咱们看一下后边的这个例子，比如在这个秒杀的情况下，一个用户只能购买一件秒杀商品，那么这个时候这个商品的 ID 加上用户的 ID 是不是就构成了一个唯一的业务号？那像这种情况的 Excel 的操作咱们怎么去做呢？咱们怎么保证这个接口的幂等性呢？这个咱们可以通过分布式锁来保证接口的幂等性。


然后这个时候还是前面的这个例子，秒杀的情况下，商品 ID 加上用户 ID 当有多次多个请求同时到达你的后台服务的话，咱们可以用分布式锁保证只有一个请求能够生成记录，咱们还可以把这个商品 ID 加上用户 ID 给他创建一个唯一的索引。这个时候其他的请求过来以后，是不是就生成不了新的记录了？保证了咱们的这个接口只能生成一条记录。执行完成以后，其实咱们这个分布式锁也不必释放，让它进行自动的过期，把它过期自动释放。这个时候过来重试了这些请求，因为获得不到所是不是也就失败了？这个就是有唯一业务号的 insert 操作的幂等性。那么大部分这个隐私的操作还是没有唯一业务号，没有唯一业务号怎么办？

包括前面咱们 delete 的操作，还有这个 update 的操作，都有这个没有唯一业务号的情况。那么没有唯一业务号的情况，咱们可以统一使用 token 去解决。


那么这个 token 机制是什么样的呢？咱们往下看。首先在进入到注册页的时候，后台统一生成一个 token 然后把这个 token 统一的返回到前台的隐藏域当中。就是咱们在进入注册页的时候，其实后台这个 token 已经生成好了，把这个 token 返回到注册页的一个隐藏域当中。然后用户填写他的注册信息，最后点击提交按钮，把这个用户的注册信息统一的提移交到了后台。那么这个时候把隐藏域当中的这个 token 也统一的提交到了后台。那么后台再根据这个 token 去获取分布式锁，跟这个每个页面的 token 都是不一样的。


我根据这个 token 去获取分布式锁，它的 key 也是不会重复的获得分布式锁，这个请求他完成 insert 的操作，没有获得分布式锁的，他的这个请求就是不成功的，保证了多次请求的时候只有一个请求能够完成后台的操作实行成功以后，同样也是不释放锁，等待锁自动过期。


那么在过期的期间，你从事机制调用的这个接口同样是获取不到锁的，它也完成不了后续的操作。这个是不是就保证我这个没有唯一业务号的 insert 的操作也是一个幂等性的，我们再看看最后就是混合操作。混合操作什么意思？
