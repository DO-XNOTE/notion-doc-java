最终抑制方案它采用的就是贝斯原理。贝斯原理它强调的就是保证事物的最终一致性。咱们前面两个分布式事务的解决方案，可以说都是事务的强抑制方案，就是她要么全成功，要么全失败，并没有一个中间的状态。但是咱们如果采用了这一次的这个这种原理，它是可以有一个中间状态的，就是有一部分成功有一部分失败。所以咱们在这个一致性方面，允许一段时间内的不一致，但是最终它会全部保持一致的。

就是 base 的原理的一个核心的思想。在咱们这个实际的系统当中，也是要根据具体的情况判断是否要采用这个 base 原理去解决事物的一致性。有一些业务场景对这个数据的一致性要求非常非常的高。那么这种情况下咱们还是，要谨慎。但是大多数的情况下这个系统对这个数据一致性方面要求不是太高的话，咱们就可以采用这个 base 原理。


本节的内容就是 base 原理的一个经典的实现方式，就是本地消息表的这种解决方案。咱们看一下本地消息表在本地消息表的这种解决方案当中，将本事务外的操作都给记录到一张消息表当中，这块大家理解可能比较困难。咱们还是举一个现实当中的例子，比如说咱们在这个电商网站当中，咱们下订单，然后要支付。那么你的系统当中，订单和支付它是不是就是两个事物了？因为你支付要去跳到支付宝或者微信去支付。那么这些支付的这些状态等于是在支付宝和微信的系统。而订单是在你的电商系统当中。比如说你支付了我在微信，在支付宝在微信里边让我支付了，那么我是不是要生成一条支付的记录啊？那么这条支付的记录是不是就存储到了微信或者支付宝的系统当中？然后他要给你的系统去发通知，告诉你这一笔订单已经支付了，支付宝和微信会定期的会查询出这条记录，然后给你的系统发通知对吧？其他的事物要提供这种操作的接口。


这个操作的接口在这个支付的例子当中，是不是就是你的支付宝和微信的回调接口？对，支付宝和微信把支付的消息通过这个接口传到你的系统当中，你的系统要去更改订单的状态，将以前订单未支付的状态改成已支付定时任务轮询本地消息表，将未执行的消息发送给这个操作的接口。
