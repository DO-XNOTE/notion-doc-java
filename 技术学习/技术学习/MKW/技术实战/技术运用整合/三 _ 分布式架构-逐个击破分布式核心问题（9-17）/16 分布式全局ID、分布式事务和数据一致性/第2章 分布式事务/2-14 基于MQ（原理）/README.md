之前，咱们每一个事务的消息都是存到了消息表当中。那么基于 MQ 的这个方案，咱们要把这个消息存到 MQ 等于是你的存储层发生了变化，从数据库当中移到了消息队列。然后第二个不同点，定时任务咱们改为 MQ 的消费者。之前这些消息咱们发送给这个操作接口是通过定时任务轮询本地的消息表，然后取出这些消息发送给操作接口。那么现在咱们通过 MQ 就可以通过消费者直接从队列当中把这些消息取出来，然后调用操作的接口是不是就可以了？咱们就不去依赖定时任务了。这个就是基于 MQ 最终一致方案的一个整体的架构图。

同样也是业务系统操作业务数据，首先要把数据保存到数据库当中，然后把另外一条消息给它存放到消息罗列当中，然后消费者从队列当中取出这条消息，然后执行分布式事务的第二段的这个服务，然后把这一条具体的操作的消息落入到数据库当中，咱们还是沿用上一节的例子支付和订单状态。
那么很显然这个业务系统就是支付系统，支付系统它支付了咱们要做一些数据的处理，在这个用户的账户上去扣减他的余额。这个操作就是操作 DB 1 然后将支付成功这一条消息给它放入到消息队列当中，然后消费者取出这条消息，拿到了支付成功的这个消息，他去更新订单数据库，把这条订单的状态给他更新为移支付，这个就是一个整体的流程。


那么一会儿咱们再给大家演示的时候，消息队列，咱们这个前面的老师已经给大家做了详细的介绍，我在这就是给大家简单的搭建一下。在咱们的实例当中，使用的是 Rocket MQ 用 RocketMQ 咱们简单的搭建一下，给大家做一个示例。基于 MQ 方案它有什么优点呢？首先它不依赖于定时任务，基于 MQ 是更高效更可靠的一种方式。之前基于定时任务，咱们要依赖于定时任务的这个固定的周期，按照这个周期去发送我的消息。那么基于MQ ，如果那个队列当中的消息不是特别多的话，咱们可以说是瞬时的就可以完成这个信息的同步。

比如说我有多个系统，我直接要做一些一致性的这些需求。那么这个时候我们就可以采用 MQ 因为都在一个公司内，系统都部署在同一个机房当中，咱们使用 MQ 是非常的方便的。但是如果不同的公司之间我要保证分布式的事物一致，就无法基于 MQ 了。这种情况下咱们还是基于本地消息表更合适。就比如说咱们所做的电商系统对接支付宝对接微信，咱们都是和支付宝和微信都属于不同的公司，咱们之间的对接是无法基于 MQ 的。那么有 MQ 是不是就更方便了呢？确实是更方便。但是不同的公司之间他们的服务器部署肯定不在同一个机房内，咱们没法去基于 MQ 做一些这个消息的同步。所以这个时候咱们还是使用本地消息表，就像支付宝或者微信把你的支付我给你存储起来，然后我定期的给你的系统发送通知，这个就是本地消息表的这么一个整体的思路。
