其实事物补偿它的核心的思想就是针对每一个操作都要注册一个与群对应的补偿的操作。这个补偿的操作也就是撤销操作。在执行失败的时候，咱们调用补偿操作。撤销之前咱们的这个动作，比如说咱们下单的时候扣减库存，那么订单库和商品库它是分开的，这个时候就存在了分布式的事务问题。那我下单之前要先把库存进行扣减。这个时候我可能调用一个 update 的语句把库存给更新了，然后我再进行创建订单的操作。我要 insert insert 这订单。但是如果在创建订单的时候操作失败了，我是不是就直接抛出异常了？抛出异常。

我之前库存商品库的东西我是没有办法进行回滚的，因为它在两个不同的数据库当中。那么我们怎么保证商品库存数解进行回滚呢？这个时候咱们就需要调用一个补偿的操作，把这个商品的库存再给它加回来，这个就是事物补偿机制的这么一个核心的思想。接下来咱们再看另外一个例子，也就是 A 给 B 转账这么一个例子。那么 A 和 B 它在两家不同的银行说明什么？说明 A 和 B 它的账户是在不同的数据库上，当然就存在了分布式事务的这个问题。 A 给 B 转账 200 元， A 账户要扣减 200 元， B 账户要增加 200 元，这个是转账的一个最基本的操作。两个操作要保证原子性。什么是原子性，就是它是一个不可分割的这么一个操作了，它就是原子性。这个转账的动作要么全成功，要么全失败。你不可能说出现 A 扣减二百元成功了， B 增加二百元不成功，那么 A 账户就要凭空的少了二百元，也不可能说给 B 直接加二百元对吧，那这钱是不是平白无故的就多出来了。所以要保障这两个操作是原子性的，要么全成功，要么全失败。
