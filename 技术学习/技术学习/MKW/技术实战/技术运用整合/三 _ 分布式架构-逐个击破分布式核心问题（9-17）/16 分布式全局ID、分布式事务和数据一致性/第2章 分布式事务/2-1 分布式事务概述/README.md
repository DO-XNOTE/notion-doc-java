从这一章开始，咱们要给大家介绍分布式事物这个词大家都听说过吧，在数据库当中也就是对应它的 aci D 这四个属性。那么什么是分布式事务呢？分布式事务它是怎么形成的呢？这个还要从前面的分库分表那一章节给大家说起。在传统的数据库当中，因为咱们满足了 aci D 的四个属性对吧，数据库的事物得到了保证，这个是没有问题的。如果咱们进行分库分表，把这个一个数据库拆分成了多个，那么它就会引发分布式的事物。分布式事务最主要是什么问题呢？就是你在一个方法里面， D 操作了 A 数据库，又操作了 B 数据库。那么这个时候你保证 A 和 B 数据库的事物是统一的，它是很难保证的，没有这种机制。所以它引出了分布式事务，咱们看看读分布分表都引发出了什么问题。咱们本来一开始使用一个数据库，使得好的，但是由于数据量太大，咱们要进行。
分库分表以后引发的第一个问题就是他的 ID 问题。这个在咱们的上一章的内容当中，给大家讲解解决了分布式 ID 的问题。然后就引出了这一章的内容，就是它的事务问题。多个数据库之间咱们怎么保证它事务的一致是吧，这个就是本章要给大家讲解的内容。咱们先来看一下本章的概述。首先要给大家介绍的是 cep 原理， cep 原理它只能满足其中的两项对吧，CE P 三个是不能够同时满足的。然后给大家介绍 aci D 和 base 原理，这些内容在网上都可以查到，这些咱们也是不会在视频当中给大家讲解，会以图文的方式给大家做具体的介绍。然后就是分布式事务的解决方案。第一个是基于 x1 协议的两阶段提交，这种方式有现成的解决方案，

很多储蓄库厂商也是支持 xa 协议的，而且也有专门的厂商提供了 xa 协议的这种数据源。**但是在咱们的应用当中是不提倡使用 xa 协议的**。因为它的提交的过程比较复杂，如果出现问题会导致数据库的挂起状态是吧，这种情况是咱们不希望出现的。所以咱们在项目当中使用基于 x1 协议的这个两阶段提交还是要谨慎的去采用。

然后就是事务补偿机制。事务补偿机制它比较的简单是吧，理解起来比较简单，但是实现起来的话对于程序员的要求还是比较高的。它无非就是提供一个回滚的接口，你在操作多个数据库的时候，一个数据库成功了，另外一个数据库失败了，这个时候你操作成功的这个数据库要提供补偿机制，回滚的接口要提供出来。我把之前已经执行成功的这些数据再给它通过补偿机制让它再回到原来的状态。这个就是事务补偿机制。


最后两个它的整体的方案都是一致的是一样的，都是让数据的最终一致性。它分为两种实现方式，一种是基于本地消息表的，另外一种是基于 MQ 消息队列的对吧，他们都有不同的使用场景，这些咱们会在视频当中给大家做详细的介绍。好了，本章概述就到这里告一段落了，欢迎大家收看下面的详细的内容。
