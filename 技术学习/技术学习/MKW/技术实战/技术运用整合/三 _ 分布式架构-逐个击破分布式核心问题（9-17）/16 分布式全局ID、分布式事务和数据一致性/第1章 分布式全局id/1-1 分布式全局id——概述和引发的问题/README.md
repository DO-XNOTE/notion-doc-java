接下来咱们看一下本章课程的概要。首先要跟大家介绍的是分库分表系统当中 ID 所引发的这个问题对吧，它是由于什么原因导致的？他会对业务系统引发什么样的问题？然后就是分布式 ID 的解决方案。其中第一个就是用 UUID 作为全局性的唯一主键，对吧，这个比较容易理解，然后会通过统一的 ID 序列表，实现全局的 ID 接下来还有这个 Twitter 提出的这个雪花算法，咱们可以运用这个算法作为全局性的 ID 也可以避免 ID 重复的问题。


最后咱们会把这三种分布式的 ID 进行比较，看看咱们在业务系统当中选用哪一种方案比较合适。接下来咱们看一下这个分库分表引发了这个 ID 的问题，它到底引发了什么样的问题？它产生的原因是什么样对吧？首先咱们看一下每一张表都有一个唯一的标识是吧，通常这个字段叫做 ID 这个是最基础的一个知识了，每一张表都有一个 ID 的字段。当然了你也可以叫其他的。比如说在咱们的系统当中可能叫 order ID 可能叫 user ID 对吧？但是这个字段它是一个全局的唯一标识，你拿到这个 ID 的值，你就能确定一个唯一的订单是吧，确定一个唯一的用户。总之这个 ID 是在你每一个业务表当中，它是唯一的才对吧。然后 ID 咱们通常采用的是自增的方式，咱们使用 MySQL 数据库的时候都会勾上下边有一个自增是吧，咱们都给它勾上这个。咱们在插入数据的时候，这个 ID 的字段咱们就不用管了，它会根据 MySQL 系统里边这个 ID 的值是吧，自动的去增加。


但是在这个分库分表的情况下，每一张表的 ID 都从 0 开始自增对吧？咱们想象一下，比如说有一个 user 这张表是吧，咱们要给他做分库分表，假如它分了两个分片，咱们在插入数据的时候，无论你这一条记录插入到了哪一个分片表当中，这张分片表的 ID 他都会从 0 开始自增。比如说咱们插入到了分片库1，那么这个 user ID 就是一对吧，然后你再插入，如果还插入到了分片库1，那么这个 user ID 会是自增的，变成了 2 是吧，以此类推。但是如果这个时候又有一条 user 进来了，它分配到了分片库 2 当中，那么这个时候分片 2 当中这个 ID 也要从 0 开始自增。


那么这个时候 ucid 又变成 1 了对吧？那也就是说在我的这套系统当中， ucid 等于 1 的有了两条记录，对吧？那如果咱们查询的时候前端传过来一个 ucid 等于1，那你需要去查哪个数据库呢？它代表的又是哪一个用户呢？是吧？这个就是分库分表这个 ID 引发出来的问题对吧？不同的分片上 ID 可能重复，导致 ID 全局不约一，业务上也会出现问题。


接下来看到的是另外的一个例子，咱们举的是 order 对吧。 order 它的 ID 咱们也是称为自增，可以看到这一块 user ID 等于 20 的这个订单分配到了 DB 131 这个数据库当中的 T order 1 这张表对吧？然后下边 user ID 等于 19 的用户，他的 order 分配到了 DB 132 当中，他的表是 T order 2 是吧。如果咱们的 order ID 都首先自增，咱们可以看一下 DB 131 当中的 T order 它的 ID 有 4567 是吧，但是 132 的数据库当中也有 4 和 5 是吧。那么订单 ID 为 4 这一条记录它代表的到底是哪一个订单呢？对不对？到底是 ucid 等于 20 的这个用户的订单，还是 ucid 等于 19 的这个用户的订单呢？这个就是分库分表这个 ID 所引发出来的问题。最后导致业务上的混乱。
