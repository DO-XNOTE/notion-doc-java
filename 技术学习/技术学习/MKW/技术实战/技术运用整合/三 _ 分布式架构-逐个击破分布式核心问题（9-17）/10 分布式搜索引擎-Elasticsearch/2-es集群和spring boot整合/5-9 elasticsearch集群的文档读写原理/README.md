好，然后我们来看一下当我们有文档去操作要进行写的时候，他应该如何去进行一个处理。那么写的话其实是包含了文档的新增修改以及是删除，他们都是归属于写操作的。随后我们就可以来看一下。在我们使用客户端去操作 ES 的时候，其实我们由于现在是一个 ES 集群，那么我们的客户端里面其实我们会写上每一个节点的 IP 和它的一个端口号。当我们现在是有三个节点组成的时候，就会有三个。那么咱们的一个客户端其实会去选择某一个节点去处理，去负责相应的一个对接的工作。来看一下。假设这个时候的客户端他是选择了我们的中间这一个节点中间的这个服务器。那么这个的时候这个服务器的话其实就会被认为是我们 ES 局执行中的一个协调节点，也就是 coordinating node OK 。那么这个协调节点的作用就是用于去负责接收用户的一个请求的，你可以把它理解为是一个 ctrl 了，在这里是接收了一个用户的请求。


OK ，那么接受了一个用户的请求以后，那么这个时候假设现在是我们要去做一个写文档，那么这个协调节点会针对于我们的一个文档去做一个路由，不管你是一个写操作，是新增还是修改或者说是删除的话都会去做一个路由。这个路由的话那么他就会去查找，我们会去做一个计算，他会算一下我们这个如果说是一个新增的话，那么他会去做一个算法，来看一下。


我这个数据应该要写到每一个主分片里面，也就是P0、 P1 和P2，他会去做一个抉择，他会去选择的。这个其实也就是一个哈希的算法。那么当他获得一个具体的对应的主分片的时候，那么他就会去做一个文档的存储。那么假设现在进行一个计算以后，那么他会把这个数据存到咱们的这个 PE 的主分片之上。


OK 吧，那么这个其实就是第二个步骤，文档的路由并且是转发，他会把相应的一个数据转发到 P1 上，再去做一个对应的存储，由他去进行一个请求的处理的。OK ，那么这是我们的 PE 由于我们是在一个集群里面，那么肯定我们还会有一个副本分片，也就是对应的 RE 那么这个时候他还会把数据同步到我们的 RE 上，也就是数据的一个同步的更改。Ok 。那么这样子当我们的 PE 和 RE 这两个地方的内容我们都存储成功之后，那么这个时候他会把请求再一次的跳到我们的这个协调节点，最终由协调节点可以看到。


这个第四步是文档写操作完毕，就是我们的 PE 和 RE 这个操作都要写完成之后，那么他才会把这个请求的最终由我们的这个协调节点再回给他，最终那么再告知客户端响应说我这个写操作已经是成功了。那么其实这个其实就是我们文档写的一个原理。那么如果说我们在下一次的一个请求，那么用户客户端又要去做一个写操作，这个时候它的协调节点就可能是第三个。OK ，然后它的一个数据的存储就有可能会分配到这个 P2 上。那么这个其实都是由它自己本身去做到的一定的协调。然后在这里这个协调节点你可以把它当做看成是一个 control 了，他可以去负责接收用户的请求以及是响应用户的请求。


OK ，然后这是我们的这个写的原理。我们再来看一下我们文档的一个读原理，其实也是差不多的。读原理的话其实是根据我们的一个文档的 ID 去做一个查询，要把它给读取出来。即使在我们这里面，它其实也会有一个协调节点这样的一个概念。假设这个时候我们的这个客户端请求是打在了我们第一个节点上，那么这个时候它就是我们的一个协调节点，那么这个协调节点的话是会负责去接收用户的一个请求。Ok 。那么然后这个时候要去读某一个文档的话肯定会有 ID 随后我们的协调节点会针对于我们的文档的 ID 去做一个路由，也就是一个哈希的算法，会计算出我们这个文档它是保存在哪个位置的。那么这个文档它有可能是保存在这个 P0 上。当然这个 P1 和 P2 都是有可能的，所以它就会到我们的P0，也就是咱们的一个主分片上去读取相应的一个数据。


OK ，那么读取数据了以后这是一种方式。那么由于它还会有一个208，这个 20 是它的一个副本分片，它两个数据其实全部都是一模一样的，所以用户下一次请求就有可能会读取到这个 20 上。其实他们之间他们本身其实是这样子，就是说他们可以从主分片或者副本分片上去轮询的读操作。那么也就是说他们之间是存在了一种负载均衡这样的一种概念。


OK ，那么这个负载均衡的话其实也是起到了我们集群的一个高负载的一个目的的。OK ，如果说我们现在是有 5 个甚至是 100 个节点的话，那么只要我们的一个主分片有对应的很多的负反分片，那么他们都可以去提供一种读的操作。那么当我们的数据不管是从 P0 或者 20 上读取数据以后，那么他们都会返回给我们的这个协调节点。那么这个就是咱们的第三步OK ，那么返回给他了之后，最终他是需要去进行一个响应，那么他就会第四步，响应给我们的客户端。那么这个其实就是我们文档的一个读原理，那么相对来说也是比较简单。那么这个就是我们本节所讲的一个在 ES 集群中文档的一个读写原理。
