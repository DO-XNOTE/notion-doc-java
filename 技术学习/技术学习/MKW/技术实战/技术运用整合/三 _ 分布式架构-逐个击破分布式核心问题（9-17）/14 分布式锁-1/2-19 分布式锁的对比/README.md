咱们先看第一个，数据库，数据库的这种分布式锁实现起来非常的简单，而且开发人员也非常容易理解，因为它是基于数据库的， SQL 咱们开发人员都经常写，但是它的缺点是什么呢？对数据库的压力比较大，比如说你要做一个什么秒杀活动的话，那你这个对数据库压力得有多大。所以咱们建议如果你使用数据库这种分布式锁，建议你的这个锁的数据库和你的业务数据库分开，即使获取锁这个压力把数据库压垮了，但是也不影响你的业务数据，两个数据库是分开的。

然后 Redis 分布式锁这个 Redis 分布式锁是咱们自己实现的，那个，它接触于 Redis 优点也是易于理解，咱们 Redis 在项目当中使用的频率也非常高，也一个理解缺点是什么样呢？就是咱们是自己实现的可能存在一些 bug 还有一个缺点就是它不支持阻塞。咱们刚才也已经回忆了你的这个 Redis 这个分布式锁你获取不到锁的这些线程马上就返回了，并没有在那里等待，这个是它的一个缺点。然后组 keeper 组 keeper 这种分布式锁优点，它支持阻塞是吧，咱们自己实现也是支持阻塞的。咱们在这个阻 keeper 的这个分布式锁里，咱们写了一个为的方法，在那里等待，这个其实就是阻塞，然后等待这个线程被唤起是吧，这是它的一个优点。然后缺点就是主 keeper 咱们使用起来这个频率不是太高，你使用主 keeper 也只不过是基于一些其他的框架去使用，咱们自己去记忆主题的编程，这种情况非常非常少，所以它这个程序也比较复杂，不是太容易理解的。这个是它的缺点。



后两个就是人家已经把这个分布式锁的方法已经提供好了，咱们就直接调用就可以了。一个是 Q 瑞特，他是阻 keeper 的一个升级的版本，提供锁的这个方法也是支持阻塞的。这个 Q 瑞德和 ready 神都是支持阻塞的。缺点，这个 Q 瑞特它是依赖于组 keeper 组 keeper 大家都知道，它是强一致的。如果你组 keeper 是一个集群部署的情况下，你获取这个锁，它要把这个瞬时节点同步到你所有的主题班的节点上。组 keeper 它要保持一个强一致，这个可能会对你组 keeper 的集群这个压力会比较大。这个可以说是 create 一个小小的缺点，但是咱们一般使用还是不太受影响的。然后 Redis 提供了各种的基于 Redis 的这种连接方式，比如说这个单体应用、集群、哨兵模式、主从它都是支持的。


最后推荐使用这个 ready 神和 Q 瑞特实现的这个分布式锁，这个都已经经过大量的验证了是吧，人家官方推出的假包有这个分布式锁的方法，咱们还是使用人家县城提供好的这个分布式锁，不推荐自己编写的分布式锁。自己编写分布式锁其实对你的这个理解能力包括你的动手能力都是有不少的提高。但是在项目当中如果你出现一点点的纰漏，那这个损失可能就无法挽回了，自己实现的这个分布式锁存在什么样的 bug 也不知道。所以还是推荐 ready 审核 creator 这个官方提供的这种分布式锁。那么到这里，分布式锁的所有的内容就都给大家介绍完了。下一章的内容，咱们要给大家介绍分库分表，咱们还是提前做好准备。好，谢谢大家。
