那么这一节我们要通过一个实战的案例，亲身的体验一下如何使用锁来解决电商中的超卖的问题。超卖这个问题在电商的系统当中是比较忌讳的一系列非常难以解决的问题。下面咱们就看一下什么是超卖。什么是超卖呢？咱们先举一个例子，大家看一下，某件商品的库存数量是 10 件，结果卖出了 15 件，这种现象就叫做超卖。如果非要给超卖下一个定义的话，咱们看看它是怎么定义的。商品卖出的数量超出了这个商品库存的数量，这种现象就叫做超卖。超卖会引起很多非常严重的后果，比如说商家在发货的时候，由于你超卖商家没有商品可以发了，导致他必须要尽快的进货。这一段时间就延长了这个发货的时间。


最后还有可能引起买卖双方的一些纠纷，比如说我的秒杀商品只有一件，那结果你可能卖出了两件三件，引起了这个买卖双方的纠纷，这些都是非常严重的后果。那么咱们接下来看一下超卖对系统都有哪些影响。超卖在系统当中可以表现为两种现象，咱们看一下。第一个现象，咱们也举一个例子，通过这个例子，看看这个超卖的现象是怎么产生的。现象 1 系统中库存只有一个，但是却产生了两笔订单。这个就是刚才咱们前面举的那个例子，卖家在商品发货的时候，发现只有一件商品，但是有两笔订单，所以商家只能把这个货物发送给一个买家，另外一个买家就没法给他发货。这种现象是怎么产生的呢？咱们看一下具体的流程，商品只剩下了最后的一件。然后这个时候 A 和 B 两个人同时看见了这件商品，并且同时加入了购物车，而且同时下静态。这个时候 A 和 B 这两个人下单的操作就产生了一个并发，最后导致了超卖的现象。


那么产生这种现象的具体原因是什么呢？大家想一想，看看能不能找到具体的原因。就是在扣减库存的时候出现了问题。扣减库存的动作在程序当中进行，在程序当中计算剩余的库存，最后再把这个计算好的库存数更新到数据库当中，这个逻辑看似没有问题，很正确，但是在并发的场景下就导致了超卖的现象。


接下来咱们通过一张图可以更直观的理解这个超卖现象产生的原因。首先商品的库存都是存在数据库当中，然后 A 和 B 同时进行下单的操作，然后同时读到商品的库存数为 1 经过判断库存是可以满足下单的，然后继续往下走。 A 和 B 同时在程序当中计算库存的，库存的数量计算成为零，因为之前只有一件，然后我买了一件，最后库存是0。最后 A 和 B 同时更新数据库，将这个 0 更新为最新的库存。这个时候数据库当中的库存是 0 了，但是我们却产生了两笔订单这块。大家这个流程再仔细的看一遍，尤其是第二步和第三步，我库存数只有一个，但是 A 和 B 同时读到这一个库存，那 A 和 B 又同时只买了一件商品，都满足他下单的要求，然后流程就继续往下走了。 A 和 B 在程序里边去计算剩余的库存，把这个库存数给它更新为0。最后一个库存产生了两笔订单，这块流程大家要多看几遍。最后产生了超卖的现象。


那么这个问题咱们怎么解决呢？咱们看一下具体的步骤。首先就是把扣减库存的这个动作不在程序当中进行，而是给它下沉到数据库。因为咱们出现问题就是在扣减库存的这个过程当中出现的问题。所以咱们要把这个过程给它优化掉，不在程序当中去进行库存的扣减了，而是直接下沉到了数据库当中。那么在数据库当中怎么去扣减库存呢？就是通过 update 的语句，咱们向数据库传递你库存的增量。那么什么是库存的增量？后边也有一个例子，扣减一个库存，那么增量就是负1，我们通过库存数加上增量数，这么一个方法去更新库存，在数据库当中，通过 update 语句计算这个库存。


那么有同学可能会问了，这样就可以防止病发吗？其实这里边有一个非常重要的概念就是这个 up day 的航锁。 up day 的航锁可能有的同学不太清楚，咱们在这里稍微的给大家说一下行锁就是在更新数据的时候给这个数据加上一把锁。在这个上锁的期间，其他的 update 的语句只要是更新同样的这条记录的都是不可以执行的。只有等当前的上锁的 update 语句执行完以后，把锁给释放了其他的 update 依据才可以更新这条记录。这个就是 update 的航锁，我们正是通过 update 这个航锁去解决了这个并发的问题。好，下面我们通过代码给大家演示1。
