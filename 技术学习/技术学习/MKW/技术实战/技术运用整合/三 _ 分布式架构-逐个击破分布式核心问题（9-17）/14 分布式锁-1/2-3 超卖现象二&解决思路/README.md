最后咱们数据库里边得到的结果是负4，咱们看一下关键的步骤是哪一步是校验库存这一步。这一步由于是并发执行的，所以每一个线程判断这个条件都是满足的，然后就会进行下面的更新库存的操作。咱们再具体看一下。创建订单的方法，首先检索出你要购买的商品，获得商品的当前库存，这一步就是并发执行的这件商品只有一件，这个时候凯瑞的 count 都是一你所有的线程走到这一步都得到了 1 这个数字，然后判断你购买的商品的数量和当前的库存数，你购买的是一件满足，满足它就不会抛出这个异常了，继续往下走。

那么这个时候数据库里这个库存数还是0，咱们从头再捋一遍。比如说 A 和 B 两个人同时下订单，商品只有一个库存并发的去执行校验库存的时候都满足下单的要求，都继续往下走更新库存。由于 update 有行锁， A 和 B 只能顺序的去实行更新的操作。这个时候 A 先更新，将库存从 1 更新成为0，更新完以后继续再次检索这个商品的库存。这个时候库存数为0，不是负数，而是正常的。那么程序继续往下走，创建订单没有问题。然后这个时候 B 又开始更新了。 B 更新是将商品的库存数从 0 更新成为负1，更新成负 1 以后再次检索商品的库存，这个时候发现库存数为负数了。那么这个 B 是不能够购买这件商品的。所以这个时候，抛出异常整个事物回滚，他之前的从 0 更新成负 1 的，这个操作也回滚了，这个时候商品的库存数还是0，然后抛出了异常，给用户提示说你下单失败，这个是一个解决思路。


但是这个解决思路并没有用到锁，咱们这一节的内容主要是使用锁来解决这个超卖的现象。接下来咱们就看一下怎么样用锁去解决这个第二个超慢现象。首先在校验库存、扣减库存这一步统一的给它加上一把锁，统一加上一把锁使之成为一个原子性的操作。就是说校验库存和扣减库存是在一起执行的，要把它视为一个整体，是不可分割的一个整体。这个时候并发的时候只有获得锁的线程才能够进行库存的校验和扣减。如果 A 和 B 同时去下订单。由于咱们校验库存和扣减库存这块统一加了锁，只有获得锁的这个线程才可以校验库存扣减库存。而另外一个线程只能等待咱们。比如说 A 获得了这个锁，校验库存。扣减库存结束以后， A 把这个锁就释放了。释放以后 B 获得了这个锁，然后他去校验库存。这个时候他再去校验库存的时候，他获得的库存数就变成了 0 了。因为之前 A 已经把库存给扣减了， B 在获取的时候它就变成了0。那这个时候再校验，库存只有零件不满足他下单的要求，这个时候就抛出了异常，这个就是使用锁去解决第二个超面现象的这么一个思路。
