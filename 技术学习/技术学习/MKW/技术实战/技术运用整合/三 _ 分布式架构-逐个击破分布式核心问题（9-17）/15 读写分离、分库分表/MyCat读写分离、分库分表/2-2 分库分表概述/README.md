及了这个分布式锁，咱们再快速的回忆一下，分布式锁它主要解决的是什么问题？因为这个 Java 当中已经给咱们提供了这个单体应用锁对吧？但是我们为什么要使用这个分布式的锁呢？就是因为现在互联网时代，我们的这个访问量也比较高，单体应用已经很难承载这么大的访问量了。所以我们要部署集群最常见的我们腾开的要部署集群。部署集群的话我在执行某一种业务的时候，那可能也会用到锁。部署集群以后，你每一个请求落到的这个单体应用可能是不一样的。那么你这个单体应用锁只针对一个 Tom Kite 内，它是有用的，在多个 Tomcat 之间，它怎么去锁呢？这个时候单体应用锁是没有办法满足的。所以我们要借助第三方的组件去实现分布式锁，让它在多个 Tom cat 之间也可以给它锁住。


这个咱们是借助三种组件，一个是数据库，一个是 Redis 一个是组 keeper 它主要是解决这个问题。那么这一章咱们要学习的是分布式课程当中的第二部分就是数据库的读写分离和分库分表。下面咱们看一下本章的概述。首先在当前那个互联网这个时代，数据量都非常的大，而且数据对于一个企业来说也是命根子对吧，因为我所有的客户的一些资料，这些客户的关系包括一些其他的像什么订单都在我的数据库当中，所以这些数据对于一个企业来说是非常非常重要的。但是现在数据这么大，我们传统的一个数据库是存储不了这么多数据的。


那么怎么办？我们如何解决这个数据库的性能瓶颈对吧？我们解决在互联网当中都是读多写少这么一种情况是吧，大家看一下，就是平时咱们登陆网站也都是去浏览比较多，咱们真正的去录入数据，这些操作是比较少的。这个就是互联网的一个特点，就是读多写少。那么读多写少我们怎么去解决数据库的这么一个性能，咱们就可以简单的使用读写分离。读写分离是什么意思？就是你所有读的请求都落到负责读的数据库当中，写的请求负责写的数据库当中。当然了这个读的数据库我可以部署多台对吧，写的数据库只部署一台，就和前面一样是吧，读多写少。那你的数据库也是读多写少，就是读库要多部署对吧，写库要少部署。这样解决了一个读的这么一个问题是吧，大量请求读过来以后，我都去从读库去读，那么每一台读库它所承担的压力就会减少，这个就是读写分离。


另外还有一个数据存储问题，像淘宝、京东这些大的互联网公司，它的这些订单量、商品的数量都是非常非常多的，传统的一个数据库肯定是存储不了的。那我们怎么办？就要采用这个数据切分。数据切分都有几种。第一种方式，垂直切分。垂直切分什么意思就是按照业务去述职的去切订单分隔开，商品分隔开是不同的数据库。第二个。水平切分是同一类数据，我要切分到不同的数据库当中。比如说订单把订单尾号为基数的在一个数据库，订单尾号为偶数的在另外一个数据库，这就叫水平切分。具体的这些垂直切分。水平切分咱们可以看这个图文的教程。那么实现读写分离和数据切分。总体的来说有两种方式，第一种就是中间层代理。中间层代理是什么意思？就是说我在我的数据库前面一层再放置一个代理层，然后我所有的应用程序都连接这个中间的代理，由这个代理层去决定你去操作哪个数据库，这个就像 my cat 这个数据库的中间件，这个就是带底层的这么一个整体的架构。前面是应用了集群对吧，部署了多台，然后后边是数据库，可能是垂直切分的数据库，也有可能是水平切分的数据库。不管它是垂直还是水平，我们所有的请求都会去请求这个中间代理。然后由这个中间代理去决定，我去哪个数据库去操作，包括你的读和写都是由中间这个代理层去决定的。这个就是中间代理层的模式。


另外一种形式就是客户端模式。客户端模式是什么样呢？它没有中间代理层，不用你去安装这个中间代理层的这个插件，它是客户端模式，只需要在咱们的系统当中引入这个客户端的假包就可以了。大家可以看到这个示意图，你部署以后，每一个业务系统里边都有这么一个客户端的假包，你所有的这种操作数据库的请求都是由这个假包去决定的，他去决定你操作哪一个数据库，就像咱们这图一样，这个就像咱们的这个也是一个非常好用的这个分库分表的这么一个工具，就是沙定 gdbc 这个会在咱们的课程当中也会给大家讲解。好了，本章概述就先给大家介绍到这，谢谢大家。
