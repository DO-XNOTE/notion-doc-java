垂直切分很形象是吧，就是怎么切竖着切竖着切是什么意思？咱们的数据库当中有很多表是吧，你竖着切无非就是把这些数据以表的维度去给它切分出去。以前可能我们的订单商品对吧，用户这些都在一个数据库当中。


我垂直切竖着切对吧，我把订单的表单拆出一个数据库，去，用户的表单拆出一个数据库去对吧，然后最后剩下一个商品的表单独一个数据库，这个就是垂直切分，垂直切分每一种业务一个数据库。然后不同的业务之间是禁止去跨库去照应严查的。比如说我这个订单对吧，我这个订单他是谁下的单？我是不是要通过订单里边的 user ID 去查这个用户。那通常情况下，咱们在一个数据库的时候，可能就直接拿订单表和用户表去联合查询，就把这个用户给带出来了。但是现在你把数据库切分出来了，我们就要禁止跨库的这个联查跨库的联查是非常消耗性能的，我们一般都是怎么样去通过服务去查询用户，然后把这个用户再 set 到你的这个订单当中。

第一个，拆分的规则很难抽象，拆分以后带来了很多的麻烦。就像刚才我说的同样是一个订单列表，我用户去查询和我商户去查询对吧，我怎么办？如果你这个订单不拆分，那我都去一张表查，去一个数据库里边去查没有问题。但是你按照用户的维度，你给它拆分了，你用户查询是方便的，我能找到一个确定的数据库。但是我商户查询这些订单我怎么办？我就要把所有的数据库都要查一遍。所以这个拆分规则很难取舍有利，它就有弊。你满足了用户，那商户这边就麻烦你满足了商户，用户这边就麻烦，同样也是这个分片的事物，事物的一致性非常的难以解决。这个跟垂直切分其实是一样的。然后二次扩展时数据迁移维护难度大，这个是什么意思？比如一开始咱们的分间库有两个，咱们还是看一下上面的这个架构图。

比如这个订单库现在分为两个库，一个存用户 ID 为奇数的，一个存用户 ID 为偶数的。但是如果现在你两个数据库还是不能满足这个数据存储，我定的数据量实在是太差了，和两个数据库没有办法满足了。那么这时候怎么办？那么还要再扩展是吧，再扩展出两个变成四个数据库。
那么这个时候我们就要用户 ID 对 4 去取模了，拆分为四个数据库对吧？


那我再加两个数据库，那我原来的这个数据库当中的订单怎么办？我要把这些订单迁移过去是吧，因为我的规则变了，那我迁移的时候是不是就比较麻烦了？我每一次扩展数据迁移的过程是非常麻烦的对吧？这个就是垂直切分的这个缺点当中，二次扩展我的数据迁移难度非常大，现在这个水平切分和垂直切分都给大家介绍完了。



那么在我们的项目当中怎么去选择应用哪种切分呢？其实咱们就是按照**先垂直后水平**，因为一开始你肯定是一个单体数据库，那么我的业务发展我的数据存储量在增高。那这个时候我们要首先选择垂直切分，把你的业务给它拆得碎一点，一个数据库只存储一部分业务，这样如果再解决不了你的数据库的问题，那么就要采用水平减分了。好维度你也可以去无限的这样扩展是吧。比如说你一开始你切分成两个数据库，如果满足不了切分成 4 个，再满足不了切分成 8 个，慢慢的去扩展下去。这个就是如何去选择垂直和水平切分好这一节的内容，就先给大家介绍到这，谢谢大家。
