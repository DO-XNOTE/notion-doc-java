首先幂等性是什么？那我们可以通过借鉴我们数据库乐观所的机制来举一个例子，比如说我们对一条库存表，对一条数据进行更新，写了一个cycle，在这里这个 cycle 就是说 update t 杠repository，然后 set count，等于 count 减1。


那这什么意思？比如说你库存表存了一个商品，是手机哈，里边有 100 件商品，然后卖了一件就是 100 减去1，还剩 99 件，是不是？然后咱们同学考虑一个问题，就比如说我这个 count 最终卖的时候就剩一件了，那就是 count 一一减去 e 等于0，就没有办法再去卖了，是不是？那这个时候如果有一个并发问题呢？如果有两个请求同时过来，如果你再去减的话，那最终更新的结果就可能这个 count 值变成负数-1，那这个肯定是我们具体的业务不允许的，对吧？那我们如何去解决这个问题呢？很简单，我们可以加一个这个 version 版本号，那我们在查询的时候 where 条件我们要带着这个version，并且肯定要带着你那个商品的 ID 了，这会老师就没有去写，那最关键的就是要带着这个沃森更新的时候也要更新这个沃森，并且把这个沃森进行一个加1，那我们怎么去操作？比如说现在我要更新一个，有人去买了一个手机，是不是它就剩一件了？那我应该怎么去操作呢？首先第一步我应该先去查我们的库存表select，去查出来我们的沃森当前的在数据库里的那一瞬间的版本号到底是几。


比如说等于1，那可能并发的时候两条语句查出来的这个 version 都是一，但是我们更新的时候是 where 条件等于 1 的，我先更新了之后，我就把这个 count 库存 1- 71 = 0，然后把这个沃森进行一个加1，那哪怕第二条并行语句过来了，那它会出现什么效果？他再去查 where 等于 1 的时候，这个 version 等于 1 的时候，他就查不到这条数据了。


因为这个时候我们之前那一瞬间已经有 update 操作，执行了它的沃森就是 2 了。所以说我们利用这种加沃森版本号的控制来保障一个乐观锁，来保障一个这个幂等性。那这个其实在我们很多框架都有去使用幂等的机制，比如说我们的 elected search，它也是一个这个严格的幂等性，你每次去更新数据的时候，他那个沃森永远都是进行一个递增的。那关于幂等性，老师用一句话来概括说明一下什么是幂等性？那幂等性其实就是说可能你要对一件事情进行一个操作，这个操作你可能执行 100 次、 1000 次，那么我们最终操作的结果，这 100 次、 1000 次的结果都是相同的，那样就是一个幂等性。比如说我们执行一条SQL，并行的一瞬间可能要执行 100 次，但是无论我执行多少次，这个结果都是唯一的相同的，这个就是一个幂等性的保障。

我们来看一看消费端怎么去保障易等性。首先看一个问题，就是说在我们这个海量订单生产的这个业务高峰期产生的这个业务高峰期如何去避免消息的重复消费？这个其实是一个问题，就是说我们可能在并行的情况下，高并发的情况下可能会有好多消息到达我们的MQ，然后我们的消费端可能要监听这个很大量的消息，在这个时候难免会出现一些什么呢？消息的这个重复投递，或者是说网络的一些原因导致的一些闪断，导致我们不是可重发消息，那这个时候如果你不去做幂等，就会出现重复消费，那我们消费端为了实现幂等，它就意味着我们消息永远不会被消费多次，只能消费一次。


即使是我们收到了好多条同样的消息，但是我们最终只对这个消息消费一次，可能我的代码会跑多次，但是我的数据库只能执行这一步操作，这个就是一个幂等。如果说咱们同学还不是特别明白，我们来看一下，就是说我们目前业界主流的这个大厂也好，还是一些互联网公司也好，还是一些互金行业也好，


一般来讲幂等的要求比较严格的就是互联网金融行业，因为它涉及到钱，是吧？我们来看一下主流的幂等性操作解决方案是有哪些。

坏处就是在我们高并发的时候，如果是你是一个数据库的话，你会有数据库写入的瓶颈，因为你毕竟去操作数据库的 insert 了，就哪怕你都成功的话，那你每一次成功肯定得做一次数据库的这个写操作嘛，然后你还有读操作等等。好了，那这是坏处怎么去解决呢？我们的解决方案就是什么呢？就是利用这个 ID 去进行一个分库分表的策略，然后采用一个自己的一些路由算法去做一个分压分流的这么一个机制。比如说我的 ID 可能是一串数字，最后拼完了之后，然后我们利用一些哈希算法，或者一些其他的自己定义的一些算法，把这个 ID 进行一个路由路由到指定的，我们可能后端有 10 个数据库，就分库分表嘛，可能这个 ID 路由好了的规则可能就落到第5个库了，然后每次这个 ID 再来的时候它也会落到第5个库。这样的话相当于我们还是利用单库的这个 ID 就可以去进行一个唯一的这个幂等性操作了。


那下一个 ID 可能落到第三个库或者第二个库，是不是？这样的话，就相当于把单库的这个幂等利用我们的 ID 分库分表策略变成多库的幂等，这样的话就能降低或者说叫分摊我们的整个数据的这个流量的压力，保障我们的数据库能够扛得住这个流量就可以了。好了，那这是主流的一种方案，就是进行分工分表。接下来我们来看第二种方案，第二种方案就是说我们要利用 Redis 的原子性去实现一个middle，那 Redis 原子性大家肯定都知道，是不是因为我们无论是这个集群 cluster 还是单点，它都可以去做一个这个。比如说我们去 set 一个key，然后 set 完了之后我们第二次如果去再 set 的话，它会把这个值更新成最新的。
