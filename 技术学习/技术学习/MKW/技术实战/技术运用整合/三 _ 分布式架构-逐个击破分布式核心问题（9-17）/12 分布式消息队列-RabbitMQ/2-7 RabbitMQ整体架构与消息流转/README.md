那么这节课呢，我们来看一看 red and q 整体的这个架构是一个什么样子？来，我们来看这张图，那这张图呢？其实是有左边和右边，还有中间三部分。那首先左边这个蓝色的 p 表示的是这个producer，也就是我们的消息的生产者。右边 c 就表示我们的这个 consumer 消息的消费者。中间这一块就是我们的 server 端，也就是我们的 RABMQ broker。那通过这个概念其实就很显然很清晰地表述了我们 YVMQ 它整体的一个架构。

那首先我们关注圣诞者把消息投递到了 server 端，具体投递到哪了？它投递到了这个绿色的这个叉，那它其实就是一个exchange，就是说交换机这么一概念，然后这个交换机又怎么去做流转？他又把消息往下传递到了红色的这个叫做队列，就是 message king，其实它整体就是一个这样的简单的机制，我们的生产者只需要关注把消息投递到指定的 exchange 即可，我们的消费者也只需要监听指定的队列就可以了。


通过这个图清晰的说明了生产者不需要关注我要把消息投递到哪个队列，消费者也不需要关注他这个消息是从哪个 exchange 上来的，也就说这两块完全没有说耦合的这么一个情况，那它们之间又是怎么去流转？无非就是说 exchange 跟我们队列有一个绑定关系，那通过这个上面有一个英文叫做 exchange root and filter message，就是说它可以通过 exchange 把消息进行一个过滤，进行一个路由到指定的一个队列上，然后我们的消费者去监听这个队列，然后接收消息进行消费就好了。那这也是一个非常简单的这么一个 red MQ 的一个架构图。

那这个就是我们的生产者，就是 publish application，就是它生产出来一个消息message，然后投递到我们的 MQ 上，投递到 MQ 的哪个位置就是这个exchange，然后一个 exchange 其实可以绑定多个 message queue 的，也就是说这个 exchange 是可以绑定多个队列的。然后为什么我们看这个图，这里面有三个队列，只有其中一个队列收到了这个消息，这个原理就是因为说我们的 exchange 其实是有一个路由策略的，这个路由就是 routine key，就是我们之前所讲到的那个概念，也就是说我们发消息的时候要指定两个非常关键的点或者说属性。


第一个就是说你这个消息要发到哪个 exchange 上，第二个就是说你发消息的时候要带上这个路由的key，就是 routine key，然后通过 exchange 跟队列建立一个绑定关系，通过你的路由 key 把消息路由到指定的一个队列上，然后我们的 consumer 端就是消费端，然后去直接监听这个队列就好了，然后就可以进行消费了。好了，那么这节课。
