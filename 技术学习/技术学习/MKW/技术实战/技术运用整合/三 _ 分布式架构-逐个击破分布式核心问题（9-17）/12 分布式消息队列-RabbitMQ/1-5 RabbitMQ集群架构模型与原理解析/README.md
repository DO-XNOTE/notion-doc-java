那接下来我们再看一种远程模式，那远程模式这个是 RabbitMQ 早期版本提供的一种多活的存储，主要是做这个数据的，异地的容灾，异地的这个转移的，那它也可以提升我们的性能。比如说当我们单节点就是单个集群处理不过来的时候，可以把消息转发到下游的某一个集群模型中，那这种方式它的架构其实非常简单，但是它的配置说白了就非常的复杂，所以说后面也考虑，一般都会用这种菲德 a 神就是多活模式来替代这种远程模式，那远程模式我们仅仅去了解一下就可以了。


那接下来要介绍这种叫做镜像模式，这种进账模式是业界使用最为广泛的一种 RabbitMQ 集群架构的模型，那这种模型呢？能够保障消息是非常可靠的，后面我们会详细的去介绍。那最后一种就是多活模型了，多模型其实跟远程模型也差不太多，就是做一个异地的容灾或者是双活，或者是数据的一个转储功能，路由转发的功能。

菲德恩森这里边本质上大家可以看到它其实就是一个exchange，那可以看成当 stream 与从 upstream 主动拉取消息，那也就是说我们的下游从上游主动拉取消息，对不对？但并不是拉取所有的消息。看见了，其实这个东西感觉就跟我们的什么呢？之前所说那个远程模式就是 slovo 差不多，但是它解决了这个 slovo 的弊端是什么？ slovo 它需要在配置文件里去配，他没办法动态去做，就你想要去加一个什么呀？加一个exchange，去让他这两个集群多一个 exchange 去做。同步的时候你必须要重启服务，然后在配置文件里先配置好了之后再起来，这就非常麻烦。


那 federation 它的优点是什么呀？它的优点就是说你想去加哪几个 exchange 进行相互之间的通信和转发，可以，你直接可以通过控制台操作就好了，所以说这个是可以可视化，而且是计时生效的，不需要写一些这种复杂的配置，也是支持这种热更新的。


那在这里必须是 Downstream 上已经明确了绑定关系的exchange，也就是说你的 Downstream 一定要 upstream 上面的 exchange 的一个绑定，这样的话它才能够通过当 stream 自己的一个物理的队列去拉取 up stream 给我传过来的消息。


那么这里边也就是说 AMQP 它实现的是这个中间两个集群的一个通信代理，当 stream 会将绑定关系组合在一起，绑定解除命令也可以去做，也就是说你说我配完了之后，我两个集群之间我不想去做通信了，可以直接通过控制台把那个 x 甚至之间的这个连线的一个桥梁给它删掉就可以了，所以说这是非常非常方便的。
