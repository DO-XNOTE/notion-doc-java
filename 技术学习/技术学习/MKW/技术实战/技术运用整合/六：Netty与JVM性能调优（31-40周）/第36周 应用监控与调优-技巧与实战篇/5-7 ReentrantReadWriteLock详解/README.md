实际项目中常常会有这样的场景，比方有一个共享资源叫 some data，多个线程去操作 some data。操作有读操作，也有写操作，并且是读多写少的。在没有写操作的时候，多个线程同时读 some data 是不会有线程安全问题的，因为大家只是访问，并没有去修改，不存在竞争对吧？


所以这种情况下，应该允许多个线程同时读取 some data。但是如果某个瞬间线程 x 正在修改 some data 的时候，就不能允许其他的线程对 some data 做任何操作，否则就会有线程安全问题。比方线程 x 正在修改 some data 还没改完，线程 y 想要去读，读到的可能是原来的数据。而线程 y 如果也要修改 some data，更可怕，就是典型的线程安全问题了对吧？针对这种读多写少的场景， Java util concurrent 提供了安全的 read write lock。

首先，不能有其他线程的写锁。如果其他的线程都已经拿到写锁了，而写锁是排他的，这个线程肯定就没有办法获得读锁了。没有写请求，说明其他的线程都是在做读操作，顶多添加了读锁，而读锁是可以共享的，所以线程也能够获得读锁。


另一种可能是当前有协请求，但是调用线程和持有锁的线程是同一个，这个时候也可以获得读锁。这意味着如果某个线程已经获得了斜锁，它可以再去获得读锁。规律大家可能会觉得有点绕，一会儿看到代码失利，大家就知道是什么意思了。
