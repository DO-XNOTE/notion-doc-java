大家好，我是大木。这一节来探讨如何去调优锁。调优的第一个措施是减少锁的持有时间，很好理解，因为锁释放的越快，对性能的影响就会越小。这个原则要求加锁的代码要执行得非常快。第二是减少锁的力度。

举个例子，如果我们的锁对象是一个很大的数组，那么多个线程在竞争数组的时候很可能会比较尴尬。这个时候应该考虑把数组拆分成若干多个小对象，从而降低锁的竞争，提升并行度。说到减少锁的力度，最经典的案例就是 JDK8 之前，肯凯伦的哈士 map 的实现。在 GDK 8 之前，肯卡人的哈士map，它使用 segment 实现分段，把锁的力度变小，降低锁的竞争，从而提升性能。如果同志们对肯克人的哈士 map 源码感兴趣的话，可以阅读一下这篇文章。 JD K8 之前以及 JD K8 之后原理分析都有探讨到，我们可以从中吸收到将你所粒度的思想和实践。


第三是锁粗化。前面讲过，如果我们需要频繁的加锁以及解锁，可以把锁的边界放大一些，变成这样可以避免频繁加锁以及解锁的开销性能可能反而会更好一些。第四是所分离。有两个层面，一是读写分离。在读多写少的场景下，可以用 range 的 read write lock 或者是 stamped lock 去优化 signized 或者是 range 的lock。另外，我们也可以将业务进行拆分，让不同的功能使用不同的锁。比如前面讲到的 link 的 blocking queue，它就实现了锁分离。它把业务拆分成了入队操作以及出队操作。入队操作统一使用 put lock，也就是一定要 put 方法， offer 方法等等都是使用的 puddlock 这个锁。而出内操作使用的是 take lock，当你调用 take 方法，破方法等等都是使用的 take lock，这样就可以让入队操作和出队操作更有一把独立的锁。


最后是无锁，一般指的是case， case 操作是无锁非阻塞的。使用 case 可以减少锁的竞争以及线程频繁切换带来的系统开销。但是使用 case 的代价也很明显，因为写出来的代码会比较复杂。好，这节课就到这里，谢谢大家。
