大家好，我是大木。这节课来探讨各种连接池如何优化。连接池有很多种，但是总的工作过程都是这样子的。我们的工作线程获取一个连接去执行相关的业务，执行完业务之后再把连接归还到池子里面。所以不管是哪种连接池，在调优的时候还是有一定的共性。连接池的调优比线程池更加复杂，因为决定连接池工作效率的因素太多了。它不仅和你用连接做什么有关系，还和带宽、网络以及你应用当前有多少个线程正在获取连接都有关系。好在业界也有一个经验值估算公式叫 2N 加可用磁盘数。

 n 指的是 CPU 的核心数。比方服务器有 8 个内核，有一个硬盘，连接数就可以配置成17，不过根据公式计算出来的也仅仅是一个参考值，实际项目中还是要结合压测才能估算出一个真正适合项目的值。你可以先为连接值配置成公式估算的值，然后逐步调整连接池的大小，并多次压力测试，在估算池的周边滑动，最终看连接池配置多少表现最好。

此外，对于一些性能要求比较高的项目，也可以设置两个连接池。一个连接池用来运行执行很快的任务，另一个连接池用来运行耗时比较长的任务，从而把两个连接池进行分离，这样连接池的表现就会更加的稳定。比方，你在调优数据库连接池的时候，就可以搞一个连接池，专门用来执行实时的查询，这些实时的查询执行起来都非常的快。而另一个连接池，专门用来执行比较慢的SQL，比方一些报表SQL，因为太复杂了，执行非常的耗时。类似的，你也可以采用这种方式去优化 HTTP 连接池、 Redis 连接池等等。好，这节课就到这里，谢谢大家。
