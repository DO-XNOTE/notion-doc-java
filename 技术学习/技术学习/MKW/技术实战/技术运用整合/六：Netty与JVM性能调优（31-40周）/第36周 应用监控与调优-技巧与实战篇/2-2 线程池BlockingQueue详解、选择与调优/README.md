blocking queue 也叫阻塞队列，它是 Java 并发包里面非常常用的一个接口。说到队列，相信大家都很熟悉了，排队先进，先出阻塞队列。它指的是当队列为空的时候，如果你去获取对象，就会阻塞当前线程。而对象满了的时候，如果你尝试放入对象，也会阻塞当前线程。前面用到的 link 的 blocking queue 是 blocking queue 的一个实现类。 blocking queue 的主要作用是首先，它实现了完整的队列应该具备的基本功能。

好，现在我们有这么多的 blocking queue 可供选择，但是创建线程池的时候该怎么用？比方我们的 thread poor test，这里的 link 的 blocking queue 不写容量，这个时候是一个无界队列。所以现在是 new 出来 submit 任务的时候，起先会创建核心线程，也就是招募正式员工对吧？当线程数达到 core process 的时候，就会进入排队。而由于这里的 link 的 blocking queue 是一个无界队列，所以就会一直排队下去，不会去创建非核心线程。因为当队列满了之后，公司才会认为正式员工顶不住，才会招募外包。所以如果使用 link 的 blocking queue 五界队列，可以把线程池的活跃线程数限制在 core process 以内。


同时使用 link 的 blocking queue 的缺点是系统消耗会比较高，因为这是一个五界队列任务，可以无限排队。如果流量非常大的时候，就会占用很多的内存，严重的时候甚至会导致内存溢出。又比如如果使用 signals q，这个队列根本没有存储能力。所以一旦达到核心线程数之后，假如依然有比较多的任务进来，就会尝试非核心线程。当然，这里的 maxim precise 要设成更大的值，等到某个时候所有任务提交进来的时候，非合金线程也会保留一段时间，对吧？还记得 keep a lifetime，这样就可以实现对这些线程的缓存。同学们要能够分析出来。
