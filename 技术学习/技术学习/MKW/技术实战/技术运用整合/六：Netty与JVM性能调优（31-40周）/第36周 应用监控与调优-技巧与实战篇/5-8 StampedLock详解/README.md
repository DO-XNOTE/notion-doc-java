举个例子，假设现在有超巨多的线程在操作 range 的 reader at lock，现在执行读操作的线程超级多，比方有 500 个。而执行写操作的线程非常少，只有一个现在执行写操作的线程想要拿到写锁，而瑞安全的 read write lock 他的解锁是排他的。要想拿着写锁，这意味着其他的线程不能有读锁，也不能有写锁。所以在读线程超级多，而写线程非常少的情况下，就容易造成写线程的饥饿。

 distance from origin 用来计算这个点到原点的距离，也就是这个距离。 move if at origin 表示当这个点位于原点的时候，用来移动这个点的坐标。


好了解业务之后，我们来分析源码。 move 方法是一个纯粹的写操作，在操作之前添加写锁，操作结束释放写锁就 OK 了。 distance from origin 首先获得了一个乐观锁，然后开始读数据。假设在这一行读到的 x 和 y 分别是十，但是这里是一个乐观读锁，所以 x 和 y 可能会被其他的线程修改为二十。20，我们记一下。所以在这里需要用 validate 的方法去验证一下在获得乐观锁之后，有没有发生过写操作。如果 validate 结果是true，表示没有发生过写操作。如果发生过写操作，就会改用悲观读锁。重读数据，然后计算结果。当然要把锁释放掉，放在 finally 里面执行。最后 move if at origin，方法也比较简单，主要演示了怎么从悲观读锁转换成写锁。


好简单总结一下， stand lock 主要通过乐观读的方式提升性能，同时也解决了显现成的饥饿问题。但是有得必有失，从这个代码不能看出来。 STEMT lock 使用起来要比原选的 readerite lock 复杂不少，所以使用者需要在性能和复杂度之间做一个取舍。好，这节课就到这里，谢谢大家。
