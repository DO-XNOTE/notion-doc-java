首先是 agent 或者探针。它负责采集应用的监控数据，并将监控数据通过 HTTP 或者是g、r、b、 c 上报给 skyworking 的后端。前面讲解的价位 agent 就是一个最流行的agent。


第二部分back， and 它是 Skyworking 的后端，也叫 o a p。它负责对数据进行查询以及分析，对应着 Skyworking 安装包的 o a p labs 目录以及 CONF 目录。其中 o a p Labs 目录是 Skyworking 的一堆架包， Conf 目录则是 Skyworking 后端的一些配置。
第三部分UI。顾名思义，这是 sky working 的界面，对应着 sky working 的 web app 目录。 UI 使用 Graphql 和 sky walking 的后端通信。考虑到有同学对 Graphql 不了解，简单普及一下。
这是一种比较强大的数据查询语言，经常被拿来和我们平时更加熟悉的 Restful API 进行对比。通俗的可以认为， Graphql 是一种比较强大的用来发送与响应 HTTP 消息的语言或者是规范。目前来说，虽然 Restful 还是主流，但是 Graphql 正在变得越来越流行，像 Facebook Github 等等公司都在大规模地使用Graphql。


从这张图我们可以发现 Skywalking 的 UI 和 Bacon 的是独立的。这也是为什么前面我们在启动 sky walking 的时候， sky walking 提示类两行 successful 类的原因。最后，存储，负责存储数据。默认情况下， sky walking 使用 H2 存储数据。 HR 是一种比较流行的内嵌式的数据库。可以发现， Skyworking 的架构其实还是比较清晰的，是一个典型的前后端费力架构。理解这张架构图之后，再回来看 Skywalking 的官方架构图就比较轻松了。


上面的这两块相当于前面图上的 agent 或者探针。 Skywalking 支持很多的 agent 或者探针，比方Zipking，Jagger， Java agent，又比如 service mesh 圈子里面常用的 Istio 以及 Envoy 等等。当然图里面细分成了跟踪以及采集，不过本质都是一样的。


将数据丢给 skyworking 的后端，中间这一部分就是 o a p 了。从这里我们就可以知道 o a p 的全称了，对吧，叫 Skywalking 观察分析平台。听起来比较的高大上，其实就是一个后端。他负责对各种探针以及 agent 上报上来的数据进行格式化，把数据格式化成 sky working 需要的数据格式，存储起来。分析和查询。引擎负责分析和查询数据。另外， Skyworking 还支持多种存储，比方 elastic search、Mysql、泰迪、b、h、 r 等等。这也是 sky walking 的一大亮点。好，回到 p t 理解 sky walking 的架构图之后，再来探讨 sky walking 的核心概念。首先是服务。服务可以认为是对微服务的一种抽象。第二，服务实例比较好理解，就是微服务的一个实例。第三，端点，通俗的说就是我们请求的 API 路径。 sky working 的端点有两种，一是 HTTP 的 URL 路径，第二种是 GRPC 服务的类名加方法签名。第四，追踪。通俗的说可以理解成一次请求完整的经过。这个所谓的一次请求，可以是某个微服务内的请求，也可以是跨微服务的。


好，演示一下。访问 local host 8080，打开 Skyworking 的界面，点击追踪，选择服务，点击搜索，就可以按照微服务的名称筛选指定微服务的监控信息了。微服务的名称是加微agent，读取 agent Conf 里面的配置上报上来的。选择服务之后再选择实例，可以发现默认情况下，实例名称是服务名称。杠 p i d 冒号实例的进程号。 at 微服实力所在机器的主机名。状态可以筛选出请求成功或者失败。比如选择error。


点击搜索可以看到现在所有的请求都是成功的。单点就是我们这里所访问的路径。 copy 一个单点切过来，点击搜索就可以展示富帝 Dev 服务。这一个实例请求成功的端点花费了多久，它的详情是什么了。最后追踪的围标时可以在这里看到，点击可以复制。在追踪 ID 一栏粘贴，点击搜索就可以搜索指定的追踪了。好，现在我们已经知道 sky working 的核心概念，并且能够清晰的知道 sky walking 上面的搜索框每一栏所表达的意思是什么了。好，这节课就到这里。
