大家好，我是大木，欢迎来到性能调优的世界。在这一节，大木必须给你点个赞，因为你已经坚持学完了视频的大半部分。从本章开始，相对前面的内容来说，会稍微难那么一点，希望同学们也能够继续坚持下去，最终成长为一个能够解决复杂问题，综合实力比较均衡的架构师。当然，同学们也不要因为我说难就给自己太大的压力。事实上，我会尽量讲得通俗易懂，复杂的问题简单化，简单的问题，大白话是我的特色。好话不多说，咱们来看看本阶段要完成的任务。


从本章开始，本质上咱们只会探讨一个话题，就是性能调优。

从客户的角度来说，性能提升能够带来更好的用户体验，从企业的角度来，能够节省更多的资源。从个人的角度来说，性能调优也是一个程序员能力的体现，是我们能够拿高薪的必备技能之一，所以非常的重要。性能调优我专门总结了一个四板斧，这个四板斧其实不仅适用于性能调优，还广泛适用于定位各种复杂的问题。一起来看看。第一板斧借助监控预防问题，发现问题。在微服务的世界，如果所有的问题都去遵循传统的方式定位，效率无疑是极低的。流行的做法是做好方方面面的监控，当监控到异常情况的时候，就推送告警，比方发送短信、邮箱了等等。举个例子，某些 API 平时调用可能非常的快，但突然调用变得很慢，监控工具就会推送告警给项目的负责人，项目负责人收到告警之后，就可以去排查一下是数据库出了问题，应用出了问题，还是仅仅因为网络抖动导致的。总之，监控工具能够帮助我们了解系统的状态，并且也可以为定位问题提供大量的线索。


这里我收到的告警短信，它指的是在这个时间点部署在这台服务器上的 content center 微服务里面的 page by params。这个方法调用太慢了，花了 653 毫秒才返回。当我收到这条告警短信之后，就可以点击短信上面的 URL 去看看是怎么回事。虽然这个短信是我编的，不过我们团队实际项目中的告警短信和也大差不差。我几乎每天都要处理很多类似这样的告警短信，尽早的发现问题，这样我们的系统才能够坚不可摧。


第二，板斧借助工具定位问题。这一板斧看着有点像永远正确的废话，不过绝对值得你在心里默念 100 遍，默念到to，直到把这一板斧深深地印在你的脑海里。为什么这么说呢？这是因为道理大家虽然都懂，借助工具肯定比手动在那边瞎忙活高效嘛。但真正遇到问题的时候，如果你所在的项目还没有引入某款地位问题的工具，大部分的程序员也不会尝试去使用，而是宁可在现有的体系下刀根火种手动去定位问题。我希望大家一定要能够跳出怪圈，遇到问题的时候，如果发现得去手动定位，不妨先搜索一下，看是不是有合适的工具能够帮助到我们，如果有，可以评估一下要不要引入到项目中。希望大家能够记住这一板斧，并且合理的投入到项目中。第三，板斧定期复盘问题，防止同类的问题再次出现。这一板斧可谓是重中之重。举个例子，大木上家公司的 wiki 就有一个专门的文章空间，叫做写得教训。听起来有点逗是吧。不过里面记录的全是非常棘手非常严肃的问题，很多都是生产中的故障和总结。 wiki 维护了好多年了，很多业务相关的复杂问题都可以在 wiki 里面找到答案，并且能够规避掉。除此以外，我个人每当遇到比较有趣的问题的时候，也会做相应的复盘。比如这篇手记，我遇到了一个比较奇葩的尔瑞 index out 棒子exception，它是由框架底层抛出来的。于是在定位出问题之后，我就记录了一下，否则以后遇到类似问题，我又得借助搜索引擎到处找答案。有时候找不到答案的话，还得自己定位源码。总之，好记性不如烂笔头，要善于总结。这样一旦以后遇到相同的坑，就是展现自己技术的时候了。资深的程序员之所以能有更好的价钱，从来不是因为他年纪大，而是因为他踩过的坑。

多数语句在学习的时候也是一样，要多记笔记，把实用的内容记录下来，也是一种对技术的复盘。举个例子，我在学习 Maven 的时候，笔记是这样记的，我把学习过程中认为比较重要的内容都记录了下来，这样一来有助于自己构建关于 Maven 完备的知识体系，二来这个笔记也是我复习的好帮手。
