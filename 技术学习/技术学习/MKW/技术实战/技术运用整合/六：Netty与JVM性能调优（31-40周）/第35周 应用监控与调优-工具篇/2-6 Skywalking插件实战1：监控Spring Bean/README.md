但是在定位一些复杂问题时，我们往往还需要更加细致的监控力度。比如一次慢调用一个 controller 会调用一个service，而一个 service 可能又会调用多个其他的service，或者调用多个DAO。如果性能瓶颈是由某一个 service 的业务特别复杂处理的慢所导致的，而不是访问数据库导致的，目前我们 sky working 的配置根本无法从中快速检测出性能瓶颈究竟出在哪儿，所以就催生出了我们想要监控 service 的需求。不光监控service，我们甚至想把所有的 spring 的 bin 都监控起来，这样对于我们刚刚的场景就能够迅速的了解问题发生在哪儿了。

选择十九点五十六分的数据。我们可以看到它会告诉我们 delete API 首先进入了一个切面，也就是我们打印 API 执行了多久的一个切面。再之后它又调用了 service 的方法，可以看到在切面环节花费了 12 毫秒，而在 service 阶段花费了 820 毫秒。这样我们一眼就能看出性能瓶颈是出在哪儿。


目前来看，迪丽特 API 它的性能瓶颈处在了 service 的调用上面。不难发现这个 API 真的非常省心，而且非常强大。它让我们对应用的监控更加细致。一旦项目调用慢或者调用异常，我们可以更加精确的知道是哪个 spring 的病出了问题。这节课就到这里，谢谢大家。
