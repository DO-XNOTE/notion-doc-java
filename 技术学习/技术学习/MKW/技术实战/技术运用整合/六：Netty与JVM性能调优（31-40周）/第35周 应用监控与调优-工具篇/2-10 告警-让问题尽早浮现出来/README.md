在这个文件里面，默认已经定义好了好几个告警规则，当微幅的表现不满足规则的定义时，就会推送相应的告警。我把告警规则里面每一个配置属性所表达的含义也列出来了，在这里同学们花一点时间看一下。其中需要注意的有两点一是规则名称必须为1，并且以下划线弱结尾。第二是 matrix name 的取值，可以在这个文件里面找到。我们打开看一下，只要写这个文件里面的 key 就可以了，比如 or 下划线P99。如果你不知道这个 key 是什么含义，可以打开 sky walking 仪表盘和界面对照一下，就知道是什么意思了。比如刚刚的P99。其实就对应指标表示系统总体情况下 99 百分位的响应时间。至于百分位，这是一个统计学的概念，同学们可以百度一下，也比较的简单，大学的时候应该都学过。



回到文档默认情况下，我们不做任何改动时， sky working 所定义的告警规则如下。可以发现这些规则还是比较贴近生产的，如果对这些规则不满意，可以修改。当然了，也可以添加自己的告警规则。下面我们就来添加一个自己的告警规则。打开 alarm settings y mail，加到最下面规则名称，比如叫 service response time rule 还记得要以下划线弱结尾，并且这个名称得唯一。 Matrix name。


到这里面找一个，比如叫 service response time。 copy 粘贴。 o p 操作符大雨，阈值，比如1，它的单位是毫秒皮尔的，比如 22 分钟 count 一。最后还有 message 可以这样写服务可以用大括号 name 占位符，它就会取出问题的服务名称。最近 2 分钟以内，最近一分钟的响应时间超过了 1 毫秒。这样配置的话，只要最近一分钟的响应时间超过 1 毫秒的话，就会发生告警。而一般来说，以最请求 1 毫秒之内都不会完成，所以必然会发生告警。好规则已经定义好了。

启动 sky working，首先把 Skygun 停掉，用 GPS 查到 skywalking 的进程。 kill 杠9、 bus scalping 先停掉，再切到 SKYWALKING 的 b 目录，并执行 start up s h。启动SKYWALKING。启动完成之后，启动我们的项目，访问API，多访问几次，等待 2 分钟左右，因为这里我们设置的 peer 的是 2 好 2 分钟差不多到了。


打开 sky working，点击告警，选择时间范围最近的 15 分钟，可以发现现在已经能够看到告警了。但实际项目中，我们不可能天天盯着这个菜单看，而是希望告警一旦产生，就推送邮件或者是短信给项目的负责人。这该怎么实现呢？ Skyworking 并没有实现告警消息推送的能力，不过他提供了 Webhook 来看一下。


在配置文件中，还有一段叫做Webhook，它表示当告警发生的时候，就会请求 Webhook 这一段所配置的URL，并且请求的时候会携带这样一段JSON。这一段杰森所表达的含义我已经详细列出来了，所以我们完全可以自己写一个专门的微服务，比如叫做 alarm 微服务。在里面定一个 rest API controller，这样写就可


以了。 API 接收这样的一个 list 的定义，像我这样玩就可以了。恰恰是按照这里的 Jason 所定义的一个类，把 alarm 的具体地址配置到这里的 web hook 里面去。这样 Skywalking 一旦触发告警，就会请求 API 在这个 API 里面，我们可以做告警之后的业务处理，比如发送短信、推送邮件等等，非常的简单。现在告警规则以及 web hook 都是存放在配置文件里面的，每当修改配置文件之后，都得重启 skyworking 才能生效。如何实现配置的动态刷新，也就是配置一旦修改立即生效。我们将在下一节详细探讨这个话题。好，这节课就到这里，谢谢大家。
