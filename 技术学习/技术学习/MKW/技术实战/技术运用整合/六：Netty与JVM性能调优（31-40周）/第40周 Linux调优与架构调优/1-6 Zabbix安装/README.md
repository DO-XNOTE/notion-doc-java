第二， share meaning 一次只能知道一台机器的运行情况，而实际情况是微服构建的系统往往部署在 n 多的机器上。
第三，使用命令。只有当你发送了对应的命令，你才能知道系统的运行情况。而实际项目中，我们需要及早的发现问题并去解决问题。

第四，现在越来越多的公司从安全的角度去考虑，禁止开发人员直接触碰生产环境。最主要是部分命令是非常危险的。

**比方 r m 命令，如果你一不小心发送了个 r m 杠、 f 杠，你就得跑路了，对吧？相信大家都听说过山库跑路的故事，一旦你的企业禁止开发人员直接触碰生产环境的话，哪怕你的命令玩的再好，你也只能干瞪眼牢，对吧**？总而言之，仅仅使用命令行去监控操作系统的话是不够的。**实际项目中我们的诉求应该是这样子的。**

**首先，出现问题，它可以自动告警，从而帮助我们尽早的发现问题。
第二，应该有可视化的图标展示，这样可以直观的知道系统的运行情况。**

**第三，能够了解多台机器的运行情况。市面上有很多的监控系统可以帮助我们实现这几点目标**，

第二是database，用来存储 Zabbix 的配置信息以及采集到的数据。

第三，agent，它会部署在被监控的机器上，用来采集数据，并把这些数据传递给 Zabbix server。
怎么样传递给 Zabbix server 有两种模式第一是主端模式，由 agent 主动建立 TCP 连接，并向 server 端发送请求第二是被动模式，由 server 端建立 TCP 连接，并向 agent 发送请求。也就是主动和被动是站在 agent 的角度去说的。


第四，proxy，它可以代理多台机器，一般适用于监控节点非常多的场景。

大家想假设我们的 Zabbix 需要监控1万台服务器，每一台服务器都需要去安装agent，如果有 agent 直接和 server 通信， server 端的压力是可想而知的。proxy，它可以代理多台机器，把监控数据汇总之后再统一发送给server，这样就可以降低 server 端的压力了。


最后， web UI 是 Zabbix 的界面，也叫 zabbix-frondend。
