由于我们创建了作用在 emp_no 以及 salary 这两个字段上面的组合索引，所以索引里面的数据大概是这样子的比方10001，salary，比方5万美金第二条一万零一，第二个 salary 比方 51000 等等等。之后比方页面 number 是一万零二， salary 是3万，32000。等等等。这里同学们不要忘记，索引是有顺序的。


好，现在索引里面存储的是这样的数据，当我们执行这条 sql 的时候，会怎么做？理论上是这样子的。首先一先扫描 emp_no 等于 10001 的数据，并计算出最小的 salary 是多少。一看，比如是5万，于是就把一万零一5万获得了托利。


扫描 emp_no 是 10002 的数据，计算出最小的 salary 是多少，知道是一万零二三万。以此类推，遍历出每个员工的，最小薪资并返回，对吧？如果不做任何处理，理论上 MySQL 应该是这么玩的。但是既然索引是有顺序的，而且我们现在的索引是 emp_no, salary ，它会先按照 emp_salary 排序，再按照 salary 去排序，而我们又是 group by emp_no 所以，这 3 个步骤可以做一个改进。


改进怎么玩？一、先扫描 emp-no 等于一万零一的数据，取出第一条数据，就是员工工资最小的数据。然后，直接跳过所有的 emp_no 等于 10001 的数据。继续扫描 emp_no 得 10002 的数据。取第一条，就可以拿到 emp_no 得 10002 的员工工资最小的那条数据了。


以此类推，可以发现，改进之后，我们可以节省掉大量的数据扫描，以及大量的计算和对比。


这种优化机制就是所谓的松散索引扫描。记录一下松散，索引扫描，你不需要扫描索引里面的所有键值，就可以拿到你想要的结果。好，下面我们来用 explain 看一下，这节课的执行计划。执行可以发现 type  是 range，在 extra 里面有一个 using index for group by 出现。这一段内容，说明这条 sql 使用了松彩的索引扫描。我们也记一下 explain 的 extra 展示。user index for group by 说明使用了松散重新扫描。

第二， group by 所制定的所有字段要符合最左前缀原则，并且没有其他的字段。比方有一个组合索引转在 C1、C2， C3 字段上面。如果 group by C1，C2，则可以使用仲裁中心扫描。而 group by C2、 C3 不能使用中产索引扫描，因为不符合最左前缀原则。 group by C1、C2、 C4 也不能使用送散索引扫描，因为 C4 不在索引里面，不符合这里没有其他字段的要求。

紧凑索引扫描是相对松散索引扫描来说的紧凑。所以扫描需要扫描满足条件的所有收益件，才能够返回结果。

这节课我们探讨了 MySQL 处理 group by 的 3 种方式。首先是松散索引扫描，它的性能最好，不需要扫描，满足条件的所有索引键就可以返回结果了。但是要想使用松散索引扫描，需要满足一系列的条件。这些条件同学们需要记住，如果 SQL 能够使用素材重新扫描的话， explain 的 extra 选字段里面会展示 user index for group by。


第二是紧凑索引扫描性能。

第二，它需要扫描满足条件的所有索引键才能够返回结果。使用紧凑索引扫描的时候， explain extra 选字段里面没有明显的标识。

第三是临时表性别最差。当使用临时表的时候，会在 explain 的 Excel 字段展示 using memory。此外，课上探讨了 group by 语句以及 distinct 的语句的优化措施。要尽量的避免临时表优化成松散索性扫描或者是紧凑重新扫描。好，这节课就到这里，谢谢大家。
