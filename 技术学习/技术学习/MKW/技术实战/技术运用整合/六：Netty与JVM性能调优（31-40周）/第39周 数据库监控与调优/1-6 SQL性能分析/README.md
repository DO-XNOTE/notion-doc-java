大家好，我是大木。上节课我们已经探讨了如何使用 explain 去分析 circle 的执行计划。一般来说，使用 explain 已经可以满足大多数场景下分析 circle 的需求了。但是如果想要更加细致的分析SQL，用 explain 可能还是不够的。这一节我们就来探讨如何深入到 circle 内部，去了解一条circle，它的执行到底包括了哪些步骤？每个步骤花费了多久？性能瓶颈出在哪个步骤？关于这个话题，我又编写了一篇手机。说实话，大漠为了这套课程，可以是写手机写到手软了，课程配套的手机已经快 20 万字了。其实我也想过不写，或者是课上少拓展一点，但是又感觉这样对不起观众。而且大木也是真心希望大家能够通过这套课程，真正的成长为一个高手，而不只是浮于表面。回到话题，一起来看一下这个手机吧。


在手记里面，我提供了 3 种方法去分析某一条 SQL 的性能。


第一种方式 show profile，它的语法是这样子的。使用步骤在这里一起来玩一下。首先 select@@have_profiling，去查看我们的 MySQL 是不是支持 show profile 功能运行，它说 yes 是支持的。接着 show select at profiling，查看当前有没有开启 super file 功能。没有开启。于是我们使用 set at add profiling，等于 1 开启一下。接着再次使用 select profiling，已经变成 1 了，开启好了。
