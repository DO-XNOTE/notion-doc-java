首先是业务需求，不合理的需求可能会导致很多问题，而且有些问题是非常难解决的。举个弹幕曾经经历过的例子。我曾经手了一个财务 SARS 系统。在财务领域，有一种叫做代账的概念，通俗的说就是账务外包，自己不招募财务，而让外包公司给自己记账。


外包公司也叫代账公司。系统，就是专门给代账会计去使用的一个代账会计，可能会给若干家企业记 n 多的账。当代账会计登录我们的系统之后，会展示会计当前管理的所有的账。这样的一个列表，听起来不是很难，对吧？

让这样的一个页面把程序员弄的是叫苦不迭。**为什么？因为我们的产品经理，他提了很多的需求。比方他要求列表必须展示每套账的各种汇总信息，比如同比增长、环比增长等等。而且这个结果一定要精确展示，导致我们必须 n 张表联合查询，实时计算出结果。同时，这个页面还不允许分页，必须展示当前登录的会计所管理的所有的账。**


针对这样的需求，我们初期使用 Java 代码写了一版结果，对于管理的账非常多的会计，需要 8 秒左右才能返回，实在是太慢了。之后团队的某一位大佬，他用存储过程做了一个改进，实现了相同的业务逻辑，把时间缩短到 5 秒左右。这里提一下为什么存储过程能够提速。


首先，存储过程只需要在创建的时候编译就 OK 了，而 SQL 每次执行都需要被数据库编译。

第二，圈出过程可以降低网络开销。像这样的一个业务，我们的 SQL 非常的复杂，有一两百行，如果使用 Java 代码去玩，你得把 SQL 发给数据库对吧？而存储过程本身就是存储在服务器端的，在调用的时候只需要直接用 call 语句传输参数就 OK 了，因此会节省一些网络。但是即使用存储过程也需要 5 秒，返回依然是很慢的。


之后有个同事找我来寻求帮助，**我先做了一个 SQL 的调优，主要是添加了一些索引，同时改写了部分的SQL，让执行效率更加好一些，但是也只能优化到 2 秒左右。而业务上却认为 2 秒依然是不可接受的。于是我们的工作陷入了僵局**。


这个时候我就直接找老板和他解释，业务上提出这样的需求，技术是比较难保证业务功能的基础上还能高效返回的。也就是我客观的和老板解释了需求本身的不合理性之后，我提出来了我的优化方案。
