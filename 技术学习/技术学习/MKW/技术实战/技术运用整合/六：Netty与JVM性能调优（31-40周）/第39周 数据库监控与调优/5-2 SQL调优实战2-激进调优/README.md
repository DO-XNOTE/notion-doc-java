好，经过上一节课的调优，这条 sql 的速度已经比较快了。这节依然围绕这条 sql 台单一些激进的优化方式。首先是耐克查询的优化。前面说过，全模糊是用不了缩影的，而又模糊则可以使用缩影。因此，如果业务上允许的话，应该尽量使用又模糊，避免全模糊。记一下激进优化方案一、尽量使用又模糊，避免全模糊。同学们可以对比一下使用又模糊以及全模糊时候的 explain 结果，并据执行下这道SQL，看看对比下性能的差异。


顺便说一下，目前在 MIT 里面是这样写的这种写法存在 SQL 注入的风险，我们使用的是刀来打括号，到了大括号使用的是statement，而不是 prepare 的 statement 是把参数直接传入的。因此如果 keywords 是这样子的，就会存在巨大的风险。来举个例子 copy 这一段粘贴传入的参数是这么玩的。比方叉叉叉百分号引号，YY，分号 drop table 比方 users 分号，然后 select sing from items， where item name like。也就是假设 keywords 传的是这么一个玩意儿，买标历史就会让 MySQL 先去select，再去drop，最后又执行了一条select。这样就会导致直接把 user 表给删掉了，风险巨大对吧？怎么样修改？可以这样玩用concat，百分号，第二个参数，卡入我们的参数，并使用井号百分号最后第三个参数百分号。这样的就不会有 SQL 注入的风险了，因为井号百分号是 prepared statement。okay。


回来，继续探讨优化方案。第二种方案，是彻底使用冗余去优化这条circle。彻底使用冗余优化 circle 什么意思？我们来搜索这一段，可以发现在下面里面有多处去查询上面的最低优惠价。 Inter 我们可以把，商品的最低优惠价直接冗余到 items down。这个方案上节课已经讲过了。同理。我们再来搜索。商品主图。 copy 搜索可以发现项目里面有多处使用到了商品主图，因此我们可以把商品主图也融入到 items 表。


但是是要注意，商品主图，是一个URL，字段，是比较大的，因此如果同学们在实际项目中需要针对大字段做冗余，要慎重考虑一下，看是不是确实有必要。在表结构设计优化一节讲过，应该要让表的字段少而精，对吧？一般来说，比较大的字段适合把它独立到一张词表里面去，而不是冗余到主表。但是，如果你的项目确实有必要，那么就大胆的去做冗余吧。我们备注一下，商品主表字段较大，实际项目中如果要冗余较大的字段，应该谨慎考虑，看是否有必要。


OK，按照方案 2 的描述，我们需要在 items 表里面添加两个字段，比方一个叫 price discount，记录最低优惠价，另外一个叫 image URL，表示商品主图。这样，如果想要实现这套 SQL 的业务，只需要编写这样的 SQL 就 OK 了


select id item name， see accounts， Image 21， price discount， from ITEMS，where？
item name 贴过来，然后 out by 接过来，对吧？一个三表查询就变成了一个单表查询。我们注释一下。OK。单表查询不管是编写还是优化，又或者是后期维护都要简单很多，而且性能也要相对更好。但是同学们务必注意，这里把它叫做激进的优化方案，是为了告诉大家，冗余也不是万能药，同学们千万不要滥用，因为一旦滥用就会导致在数据增删改的时候非常的麻烦。 OK 第三种方案，是考虑使用非关系型数据库。


对于商品这样的业务，它不需要非常严格的事务支持，但是性能却是硬需求。因此，我们可以把商品相关的信息存储到非关系型数据库里面，比方 elastic search 或者是 Mongo DB 都是 OK 的，这样的往往也能得到比较大的性能提升。但是同学们要知道，引入非关系型数据库也不是万能钥，它带来了额外的学习成本、使用成本以及运维成本。总而言之，引入任何组件，你都得 hold 住它。


值得一提的是，就目前来说，各种非关系型数据库正在变得越来越流行，并且此前大目经手的电商项目商品中心就是使用 Elssky 设置存储数据的。这也说明在一些场景下，应该根据业务场景选择合适的数据库。 OK 第四种方案，业务土协。一些场景下，如果业务能够妥协，甚至可以把比较难优化的问题变成不是问题。比方，如果你的业务，能够接受不展示最低优惠价，这个子查询直接不要了， SQL 就变成了一个比较简单的量表连查。当然了，对于咱们这个场景，业务上是不能妥协的对吧？不可能在商品列表页不显示商品的价格对吧？这里只是开拓一下同学们的思路。
