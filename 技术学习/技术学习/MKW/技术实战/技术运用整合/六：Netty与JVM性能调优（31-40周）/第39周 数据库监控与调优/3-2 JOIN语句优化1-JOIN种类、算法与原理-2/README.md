一个join查询 formal person 表和 countries 表， join 字段是 persons 表的 country ID 以及 countries 表的 country ID。那么 hard join 的执行过程大致上可以分成两步。首先是 build a phrase 构建阶段。在这个阶段，他拿 countries 表的 country ID 构建了一个哈希表，存放到了内存。在probophrase，也就是探测阶段，会拿 persons 表的 country ID 计算hash，到 hash 表去匹配。另外还有一个非主流的阶段，叫做 spare to disk，翻译成中文是溢出到磁盘，它指的是如果 countries 表非常的大，基于内层的 hash table 放不下，就会存放到磁盘。这是哈斯 join 的执行过程。在这个文章的最下面，有哈士 join 和 BNR 界的性能对比，可以发现哈士 join 的性能要好很多。

也就是你哪怕把 join buffer size 设置得非常大，比方设置了1G，也并不是 MySQL 上来就会为每个 join 分配 1G 的内存。这个内存分配是逐渐递增的，可能一开始只占用了 256 k。当 MySQL 发现 256 k 不够用的时候，会自动的扩容，一直扩到你设置的 join buffer size 的值为止。所以当时由哈希 join 的时候，你可以把 join buffer size 设置的比较大。


但问题是，如果你使用的是 外 连接，而刚刚又说了，在 MySQL 八点零点一八里面， 外连接是没有办法使用 hash join 的。这种情况下，join buffer size 是直接按照你设置的值去分配的，也就是你把join buffer size 设置成1G，买这个会上来就为一个 join 分配 1G 的内存。JOIN_BUFFER_SIZE 还是得谨慎设置，不要把它设置的太大。

比方，在这里他说如果你的 MySQL 版本低于八点零点二零，至少要有个等值的校验条件才可能会用到 hash join，否则的话用不了。 hash join，也就是你的 on 值句。一定要有等值条件才能使用 hash join。在这里他就说了，如果你的 MySQL 版本大于等于八点零点二零，刚刚说的至少要有一个等值的 join 条件已经不是必须的了。


又比如在这里他说 MySQL 八点零点一八里面有一个 optimizer switch 的子参数，叫做hash join，你设置的 or 可以开启 hash join，设置的 off 可以关闭 hash join。而从 MySQL 八点零点一九以及更高版本的参数已经没用了。


总的来说， hash join 有非常多的注意点，不同的版本表现可能不一样。同学们将来在使用 hash join 的时候，可以通读一下这篇文章 [https://dev.mysql.com/doc/refman/8.0/en/hash-joins.html](https://dev.mysql.com/doc/refman/8.0/en/hash-joins.html)，避免踩到不必要的坑。
