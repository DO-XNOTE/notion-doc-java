第二是 pt-index-usage 分析比较大的慢查询日志会比较的慢。而前面又说了，工具占用的 MySQL 资源是比较多的，如果分析比较慢，就会在长时间影响 MySQL 的性能。因此你需要做一定的处理。


比方你可以把超大的慢差聚日志先删掉，新建一个慢查询日志，运行一段时间后，在慢查询日志还没有变的特别大之前，使用 pt-index-usage 去分析


第三个注意点，这一点也是最重要的。由于 pt-index-usage 它只会扫描慢查询日志，而不是所有的查询语句，所以有可能某一个索引在慢查询日志里面并没有使用。但其实它还是被使用的，只是使用索引的SQL，它并不是一个慢查询。


因此，在正式删除索引之前，应当


先 review 一下，确保可以删除，之后再去操作，避免发生问题。


第二是对 MySQL 8. 0 以及更高的版本，你可以使用不可见索引这个特性。不可见索引它是不会被 MySQL 优化器使用的。你可以把要删除的索引标记成不可见索引，这样优化剂就会忽略索引，相当于把索引删除了。


有一天你发现这个索引其实不能删除，再把它设置成可见索引，这样对业务的影响比较小。因为我们知道创建索引以及删除索引是一个非常耗时非常重的操作。而使用不可见索引，相当于对索引做了一个标记，非常的快，等确认可以删除的时候再正式删除。
