先来探讨长字段的索引调优。


**实际项目中，我们可能需要索引很长的字符串字段**。

比方，假设我们这里 first 的 name 字段是一个非常长的字符串字段。比如里面存储的数据奏战的长度普遍在 200 以上。

当然了， employees 表里面的 first name 字段其实并不长，我们这里只是举一个例子。

好吧，当我们做这样的查询的时候，如果字段的长度比较小，直接为 first name 字段创建索引就 OK 了。但是如果字符上的长度很大，就会导致索引占用的空间比较大，而且作用在超长字段上面的索引，查询效率也是不高的。


该怎么样优化？第一种做法是创建一个字段。比方 modify table 创建一个字段，叫 first_name_hash, type int, default 0， not null。还记得上一节我们说过 MySQL 建议把字段设成 not null，对吧？写上注释叫 first_name 字段的 Hash_code ok execute。


在存储数据的时候可以这样玩儿 insert into employees，选择字段，value，emp_no。比如 99999 生日随便写。 first_name 大幕非常长。 last_name 大， gender mail hiredate 随便写。


最后， first_name_hash 我们可以传入 first_name 的值。用 crc32 去做一个hash，换个行。这里 first_name_hash 使用了 MySQL 内置的 CRC32 函数，同学们也可以使用其他的方法去计算 hash 值。存进来比比方，你可以存储字符串的 hashcode 值。需要注意这个字段的值应该具备几点要求。

我们记一下first_name_hash 的值应具备以下要求。

**一是字段的长度应该比较的小，所以像SHA1， MD5 等等的算法计算出来的字符串哈时是不太合适的**。是不合适的。这是因为SHA以及MD5 这两种算法得到的字符串本身也是很大的，达不到我们的要求。


**第二，应当尽量避免哈希冲突**，因为一旦存在大量的哈士冲突，性能就会降低。前面说了，所以选择性越低，性能也会越差。目前来说， MySQL 圈子里面比较流行的 hash 函数有这么两个，流行使用 CRC32 或者 FNV64。这两个哈希函数都是比较好用的，能够尽量的避免哈希冲突。


如果同学们手头上没有更好的哈希算法可以使用，可以直接使用这两个函数。一般认为 FNV64 哈希冲突的概率比 CRC32 要更低一些，但是一般来说，使用 CRC32 也是足够的好。

facello 这样的条件，是为了让 SQL 在哈希冲突的时候也能够正确的返回结果。OK，这是第一种优化的方案，


但是这种优化对于 SQL 是无能为力的。因为我们是对 first_name 的完整值进行 hash 的，所以 like 查询没有办法使用这种方式进行优化。如果我们依然希望索引比较的小，该怎么办？ 


MySQL 支持前缀索引。创建前缀索引的语法是这样子的 alter table，跟下表名， add t， first name，跟上你要的前缀长度比方5，这样就表示取 first name 字段的前 5 个字符做索引。但问题是数字写多少比较合适？
