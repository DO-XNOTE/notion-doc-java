**第二种方案，把数据库引擎换成 MyISAM**。前面说了， MySQL 引擎针对没有条件的 COUNT 语句会直接返回，所以不管数据量是多少，查询都会非常的快。不过这种方案在实际项目中用的还是比较少的，如果不到万不得已的话，尽量不要去修改数据库引擎。**实际项目用的很少，一般不会修改数据库引擎**。OK，下面来看一下。


**第3种方式，叫做汇总表**，怎么玩？可以创建一张表，比如table，里面有两个字段，一个叫 table_name，另外叫count，分别记录表明以及这张表的数量。比方里面有条数据叫 employees 200 万。如果 employees 表里面的数据发生变化，那么就更新一下汇总表里面的这条数据。比方 employees 表里面新增了一条数据，就把 count 加1，删除一条数据就把它的减1。你可以通过触发器去自动的维护汇总表。这种方式的好处是什么？在于结果比较准确，而且也比较的灵活。你可以根据需求去设计你的汇总表。

这是有 count  8 种优化方式。简单总结一下这节课。我们首先对比了不同版本的 MySQL 针对count 语句的返回结果。可以发现，当这个版本大于八点零点一三的时候，使用 innodb 引擎不带条件的 count(*) 语句有大幅提升。之后，探讨了针对 count(*)语句的 8 种优化方案。



第一种方案可以创建一个更小的非主键索引。不过这种方式测试发现提升并不明显，所以并没有去演示。但是同学们还是要了解有这样的思路。

第二种方案修改数据库引擎。这种方案实际项目里面很少去使用，因为一般不会在线上修改数据库引擎。并且即使把数据库引擎改成 MyISAM，也只能提升不带条件的看了count(*) 语句的性能。


第三种方案使用汇总表。这种方案比较准确，也比较灵活，但是缺点在于你要额外维护好汇总表的数据。

第四种方案 sql_culc_found_rows 。这种方案比较适合扉页查询的场景。但是缺点在于它在 MySQL 八点零点一七已经废弃了，将来会被删除。所以实际项目里面个人也不太建议使用。

第五种方案，比如缓存。使用缓存的性能比较好，同时结果一般不是精确值，不过误差是比较小的。

而方案 6  7 8 都是不直接操作原始表的。

但缺点在于这几种方案返回的都是估算值，而不是准确值。它虽然你的项目能够容忍返回的结果不准确。
