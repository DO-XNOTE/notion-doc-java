大家好，我是大木。这节课来探讨索引底层的数据结构。考虑到这节课的内容相对偏理论一些，我们采用看图说话的方式进行讲解。在正式进入主题之前，先来做一点小小的铺垫，不妨先聊一聊。二叉树查找。说到二叉树查找，相信同学们还是比较熟悉的。假设现在有这样的一棵二叉树，对于相同深度的节点，左侧的节点总是比右侧的节点小。在搜索的时候也是一样的。如果要搜索的值是key，假设 key 大于根节点，也就是这里的6，就会在右侧的子数里面进行查找。如果 key 小于根节点，就会在左侧的子树里面进行查找。


假设我们想要查找到这里的2，过程大致上是这样子的先找到根节点6，使用我们想要查找的数值和 6 进行对比。一看我们要查的数值是22，是小于 6 的，于是就到 3 这颗子数里面去查找。接着再用 2 和 3 去对比，命中到 2 这颗子数，最后再用 2 和 2 进行对比，发现 2 = 2，于是就找到这个节点了。


在这个例子里面，我们只要通过三次的查找就能够找到我们想要的节点 2 了。并且对于这样的一棵树，由于它的深度是 3 一 23 对吧，不管是哪一个元素，最多只需要 3 次查找就能够找到想要的节点。

好，我们现在假设有一棵二状树是这样子的，我们现在想要查找的8，那就尴尬了，需要一二三四五次查找才能够定位到 8 这个节点。所以尽管左边的 二叉树 和右边的二叉树 都是 6 个节点，但是查询效率是不一样的。右边的 二叉树 数比左边的 二叉树 数性能要差一些。这主要是右边的 二叉树它不够平衡所导致的。所谓的不够平衡，就是一棵树它的左侧的节点和右侧的节点严重的不一致。于是为了提升二叉树的查询性能，

对于一棵平衡阿叉搜索树，如果有 n 个节点，树的深度就是以 2 为底，求 n 的对数时间复杂度就是 o log r n。这样一算，二叉树的性能还是比较好的。比方，对于一个有 64 个节点的平衡二叉搜索树，最多只需要查询 6 次，就可以找到里面的任意一个节点了。但是假设 n 非常的大，比方有1万，这个数的深度依然会比较高，这意味着查询的次数也会很多。于是有人就想，如果我搞的不是一棵二叉树，而是一个 m 叉树。比方弄个五叉树，一个节点，它可以有 5 个子节点，这样树的深度就可以变小一些。当我们想定位某一个节点的时候，查询的次数也减少了。确实是这样的一个道理。下面我们来看看 b-tree是怎么玩的。

首先，对于 b 加去一个有 n 个子节点的节点，它可以包含 n 个关键字，而 b 去里面 n 个节点只能有 n 减 1 个关键字。还记得刚刚 b 去里面一个节点，它有 3 个子节点，它只有 2 个关键字。而在 b 加去里面，它可以有 3 个关键字。


第二，在 b 加区里面，所有的叶子节点包含了全部关键字的信息，并且叶子节点会按照关键字的大小从小到大顺序排列，并且会构成一个有序列表。而 b 去的叶子节点不包括全部关键字，这一点是 b 去和 b 加去之间的显著差异。在 b 加 t 里面，父节点的所有关键字在子节点里面都会展示出来。比方根节点里面的 52865 会在中间节点里面展示， 52865 中间节点里面的五十、二十又会在叶子节点里面的五十、二十展示。并且叶子节点里面包含了所有关键字的信息。不管哪一个父节点里面的关键字，在叶子节点里面都会记录一份。而在 b 区里面，并不会要求叶子节点包含所有关键字的信息，比如这里的八叶子节点里面就没有包含。

正常情况下，由于哈希索引是基于哈希表的，所以时间复杂度是O1，性能非常的好。至于为什么时间复杂度是O1，如果同学们研究过 Hashmap 或者是哈是 set 的源码就知道了。如果不了解，可以百度一下。这块内容比较的简单，课上就不展开了。假设出现了哈希冲突，查询过程大致上是这样子的。
