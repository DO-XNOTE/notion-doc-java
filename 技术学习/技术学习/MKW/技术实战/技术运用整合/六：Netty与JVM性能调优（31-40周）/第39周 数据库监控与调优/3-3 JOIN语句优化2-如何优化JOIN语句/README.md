如果想要拆分这张 sql 该怎么做？记录一下拆分前。拆分后，可以这么玩儿首先使用 select 新 from employees where MP number，等于10001，查询员工的信息。之后使用 select 新 from d p， d m p where 点评 number 等于 10001 执行。拿到 d e p d number 是什么？ D005 copy。


最后使用 select c from departments where DP 的 number 等于 D 005，这样就可以知道部门的信息是什么了。
如果你使用的是 my 贝利斯，可以分别把这三条 SQL 写成 3 个方法，在 service 里面去调用并去组装数据，从而达到 3 秒 join 的效果。当然了，这里只是举一个例子。


**一般来说， 3 张表的 join 还是可以接受的。使用这种方式，尽管要写一堆的业务代码，而且要和数据库通信三次，但是拆分之后我们执行的都是简单的SQL，并且都是基于主键的。现在非常的好**。


在一些场景下，拆分后的代码可能反而比单条 SQL 搞的一切性能更好。


再一个，拆分之后的SQL，它非常简单，所以优化起来也比较容易。而对于复杂SQL，比方操作了 10 来张表， explain 的执行计划也会很复杂，分析起来也会让人很头痛，这种 SQL 也比较难以优化。总之，希望大家不要以编写复杂 SQL 为荣。会写复杂sql，在现在这个年代已经不再是什么值得夸耀的事情了。



最后还有一个原则，如果被驱动表的 join 字段，它就是用不了索引，而且你的内存比较充足，这个时候可以考虑把 join buffer 设置的大一点。
