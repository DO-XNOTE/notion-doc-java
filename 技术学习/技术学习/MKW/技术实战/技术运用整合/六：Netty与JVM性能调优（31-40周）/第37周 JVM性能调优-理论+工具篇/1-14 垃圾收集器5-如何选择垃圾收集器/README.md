比方某个项目是一个数据分析类的应用系统，我们更加希望它能够尽快的获得执行结果。吞吐量主要矛盾点可以考虑使用 Paris Gavenge。


而假设某个应用它是一个 web 应用， stop the word 可能会直接影响服务质量，甚至可能会导致调用超时、业务执行失败等等严重的后果。低延迟就是我们的主要矛盾点，可以考虑使用 CMS 或者是G1，


将来山林刀或者是z、g、 c 发展成熟之后，也可以使用这两款垃圾收集器。再比如某个应用，它是一个桌面端的应用，目前它最大的问题是启动非常慢，


如果我们只是希望它能够启动的快一点，就可以考虑使用Cereal。另外，前面还提过可以使用 verify none 关闭掉校验的过程，也可以进一步的提升启动速度。

总而言之，同志们应该找到自己应用程序最关注的矛盾点，并去结合前面介绍每款垃圾收集器的时候所总结的适用场景，去合理的选择。


第二，选择哪款垃圾收集器和你的基础设施也有很大的关系，比方你的应用运行在一个单核的嵌入式机器上，使用Paris、Gavench、pair、 old 等等并行收集器，或者是cms、g1 并发收集器可能都不是特别合适。又比如你所用的操作系统是windows，并且使用的是 JDK11，没有办法使用 zgc，因为 jdk14 才能支持 windows 操作系统。
