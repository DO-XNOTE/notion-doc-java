第三款故障排查工具Jazach。这款工具是用来打印当前虚拟机的线程快照的。所谓的线程快照也叫线程dump，或者是 Java core 文件，其实就是当前虚拟机里面每个线程正在执行的方法的堆栈的集合。在线程快照里面可以看到每个线程到底在做什么事情。从功能上来说， jeystack 和前面讲解的 spring boot 的 actuator thread、 dump 端铁作用是差不多的。这里我们也可以把 jstack 和 JMAP 关联起来，一起去学习。这些。 map 可以实现对内存的dump，也就是把对内存的快照给存储起来。而接 stack 可以对线程进行dump，也就是把线程的快照给存储起来。


Jetstack 这个工具也有点奇葩，不同版本的参数是不一样的，功能也不一样。 JDK8 的功能甚至比 JDK 11 还要强大一些。不过 jeystack 也是一个实验性的工具，我们还能说什么，对吧？况且这个工具也挺好用的。 stack 的参数比较简单，只有两个杠 l 和杠e。使用的时候有三种姿势，你可以不带参数，也可以杠l，还可以杠e。不带参数的时候打印的是基本线程栈，杠 l 可以打印所相关的信息，杠 e 可以打印更多的额外信息。好，演示一下。我们先不带参数执行，接 stack 26089，这样可以看到每个线程正在做什么，代码执行到哪里。下面我们再来做一个实验。我们先不带参数执行 j stack，把它输出到 T 1点 t x t，再带上杠 l 参数。把它输出到 T2T x t。最后额外带上杠 e 参数。把它输出到 T3 t x t。我们来对比这三个文件之间的区别。打开文件所在的目录，在 Mac 里面可以用 open 点儿。


先来对比 T1 和T2。可以看到带杠 l 的参数比不带参数的时候，多了一段这个玩意儿锁的相关信息。 j v m 会在每个线程的最后一部分输出线程所持有的锁对象。目前里面都是none，都没有持有锁。如果持有锁，它其实会展示当前线程持有的锁对象，锁的类型以及锁的地址。我们再来对比一下 T2 和 T3 有什么区别？可以发现，相对 T2 来说， T3 额外打印了当前分配了多少内存，定义了多少个类等等这种信息。


简单总结一下jeystack。在实际项目中，如果你的应用出现线程长时间等待的时候，可以考虑使用jeystack。比方线程死锁死循环，远程请求长时间得不到返回，都可能会出现线程长时间的等待，对吧？使用 Jessica 可以看到每个线程的调用信息，这样就可以知道那些没有响应的线程在后台到底在干什么事情，或者正在等待什么样的资源。
人人工分析 jetstack 当不出来的线程还是比较累的。在后面的章节，我们会探讨怎么样去可视化的分析 jstack 当不出来的结果。第四款故障排查工具 j hat，它可以用来分析 j map 生成的堆裆盆文件。但是 j head 这个工具它的能力比较弱，有一大堆的替代品。比方。
