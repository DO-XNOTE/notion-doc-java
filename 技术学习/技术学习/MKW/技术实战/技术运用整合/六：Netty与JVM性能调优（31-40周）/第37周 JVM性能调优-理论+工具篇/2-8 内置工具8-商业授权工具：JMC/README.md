大家好，我是大木。这节课来探讨 JDK mission control，也叫 Java mission control，简称 g m c，GMC，出款工具，可以说是超级强大，从功能上来说，哪怕是比起前面讲的 VCVM 也是有过之而无不及的。它的使用并不难，但是使用的细节非常之多，所以建议同学们如果时间上不是那么紧迫，尽量不要跳过这一节的学习。


好，下来看一下 g m c 是干什么的。 JMC 的主要作用有两点，第一是作为 JMC 控制台，用来监控虚拟 g m b 提供的数据。有关 JMC 相信同学们应该比较熟悉了，前面讲的 gconsole 以及 v c v m 也都是基于 JMC 去玩的对吧？ JMC 也支持 g m x。


第二是 GMC 能够持续性的收集数据的GFR，并且可以作为 GFR 的可视化分析工具。所谓GFR，全称叫 Java flight recorder，翻译成中文叫 Java 飞行记录器。这是一种用来收集应用程序的诊断和性能数据的工具。它的特点就是性能开销非常的小，即使是在负载很大的生产环境也都可以使用，并且是随用随停的渐发，可以用来做性能分析，配合分析以及调试。


好。关于这一点，说的更通俗一点，就是 GMC 能够可视化的分析应用所产生的 GFR 数据。这是 GMC 的两大功能。了解这一点之后，我们还必须强调两个注意点。首先是 g m c 和 V10 v m 类似，不同的版本差异是比较大的。所以建议同学们在学习的时候务必使用和大幕一致的版本，避免踩到不必要的坑。


第二是有关 JMC 的商业授权问题。大家知道在本道课程做工具分类的时候， g m c 是被归类到商业授权工具里面对吧？但其实商业授权这四个字也包含着很多的细节。回来。首先在早期的 JDK 版本里面，比如 JDK8 或者是更早一些的 JDK 7，要想使用JMC，是必须购买商业许可的。在那个年代，我们只能以个人的身份拿 GMC 做一些测试，但是绝对不允许拿 g m c 去商用。不过到了 g d k 11， g m c 以及 g m c 所使用的 g f r 都已经开源了，你可以免费去使用。当然了，使用的前提是要符合这里的 u p r 协议，同学们可以看一下。总的来说， u p r 协议还是比较宽松的。经过分析我们可以看出来价位的发展是越来越开放的，对吧。


再一个，我们也算是发现了升级 g d k 版本的一个好处，就是像 g m c 这样的一个神器，你用 g d k 8 得掏钱，而用 g d k 11 就不需要掏钱了。好，下面我们就来玩一玩 g m c。首先安装 g m c，对于 g d k 10 以及更早的版本， g m c 也是内置在 g d k 里面的，只要使用 g m c 命令就可以启动 g m c 了。使用这种方式启动的 GMC 版本相对比较的低。比如 GDK 8 里面内置的 GMC 版本是 GMC 5. 5，它和现在最新的 GMC 7. 0. 1 差异特别的大，几乎已经看不出是一个软件了。所以我们着重探讨第二种方式。


从 g d k 11 开始， g m c 被独立出来并开源了。我们可以下载独立版本的 g m c，也可以让 g m c 作为 eclipse 的插件去存在客角。我们着重探讨独立版本的 g m c 如何去安装使用。首先点击这个地址，进入到 GMC 的官网，根据自己的操作系统下载对应的文件就 OK 了。当然了，如果网络不好，也可以使用我提供的百度盘地址进行下载。下载完成之后，参考一下这篇文档的说明，就可以启动 GMC 了。各种操作系统怎么玩儿都有写。其实也都非常的简单，比如对 Mac OS 解压，把 APP 打开就 OK 了。对 windows 解压，运行下 g m c e x e，朋友们可以看一下下面的参数，也可以了解一下。回来，这里我已经下载好了，我们来启动一下 g m c 解压，里面有一个APP，双击启动，点击好没有反应。我们需要设置一下应用的安全性编号，找到安全性疑虑、隐私点击人要打开，这样的 GMC 就可以启动了。下了，点击 j v m 浏览器，这样就可以看到所有启动者的本地 Java 进程。如果想要连接远程服务器上的进程也是可以的，右击新建连接，配置 g m x 连接就可以了。远程这个问题我们还是放到后面专门去讲。这里我们使用 12702 本地进程去测试。点击展开可以发现有两个菜单，这个菜单非常清晰。


m b 服务器是 g m c 作为 g m x 控制台的相关能力，也就是。而飞行记录器则是 GMC 作为 GFR 可视化分析工具的相关能力。好，我们先来看一下 MB 服务器里面的东西商机可以看到各种监控数据，各种图表，里面又有若干多的子页面。比方 m a b 浏览器触发器系统的相关信息，内存相关信息，线程相关信息以及诊断命令等等。这些子页面里面大部分信息和前面的微视 VM 都是大同小异的。所以这里弹幕就不带大家一个一个展开去看了。只在两个比较有特色的子页面。


一是触发器。触发器可以让你配置，当条件满足的时候就触发一件事情，比方我们可以配置当 CPU 占用过高的时候，就推送一个告警。可以这么玩首先选中 CPU 过高点击的最大触发值，把它设置成比方 0. 001，点击这里的操作，选中告警。这样一旦 CPU 当前的使用率也就是这里的当前值超过这里设置的阈值，就会推送告警了。勾选一下这条触发规则就可以启用了，可以看到已经触发了告警。当然了，你也可以配置其他的操作，比方可以在 CPU 过高的时候就淡破一下堆发送邮件等等。可以发现触发器这种功能在实际项目里面还是非常实用的，它可以帮助我们尽早的去识别应用不正常的情况。


好。第二个比较有特色的子页面是诊断命令。这些诊断命令非常的多，它就相当于可视化的 jcmd。这个子页面把j、m、 c 的所有子命令都列出来了，并且标记出来了每个子命令的影响程度以及每个子命令需要传递的参数，而且会自动给你提示，你只要填就 OK 了。右边还有比较详细的注释，非常的方便。所以如果同学们觉得手动去操作接 c m d 比较麻烦，不妨可以试试 g m c 为我们提供的界面。


好，这是 g m c 的 m b 服务器菜单。下面我们来探讨飞行记录器这个菜单，也就是 g m c 作为建发控制台的部分。这部分功能的使用和你想监控的应用所使用的 JDK 版本有比较强的关系。如果你的项目使用的是 JDK 11 之前的版本，在启动的时候，你的应用必须添加这两个参数，否则这部分功能根本没有办法正常使用。举个例子，现在我们把负 d d v 项目 g d k 版本设成 g d k 8 project the structure，这里给 8 apply OK 重启项目。


目前我们并没有设置这两个参数，对吧？我们回到控制台，进程号变成了12844，展开使用。 m b 服务器不受任何的影响。我们点击飞行记录器，你会发现它会有错误提示，告诉你必须要添加这两个参数。所以我们可以 copy 这两个参数，再去添加 apply OK，再次启动，才能正常使用。展开飞行记录器没有问题了。好，而对于这11，我们就不需要设置这两个参数了。我们把应用的 JDK 版本还设回来，接了一个 11 apply okay 应用停掉。 edit configurations，把参数去掉， apply okay 再次启动。


好，回来我们展开飞行记录器也可以正常使用，对吧？下面我们就来用 GFR 去监控应用名称。随便填一个比方 my j f r TESTS。这里表示你想用 GFR 监控应用多久，你可以随便指定一个时间。如果选中继续记录，它会一直记录下去，直到 JFA 文件达到你所设置的值，或者是达到了这里的最大存货时间才会停止。这里我们就设置固定时间记录 1 分钟。


在下面还有一个事件设置模板管理器，我们可以点开看一下。它可以用来设置各种事件的开关。目前大家可能还不知道事件是什么，一会儿大家就知道了， GMC 内置了两个模板，一个叫 cleaners 模板，它是针对应用持续性能的事件模板。第二是 profile 模板，它主要是用来诊断应用性能的一个实践模板。这里我们可以创建一个自己的模板，点击新建点编辑，起个名字叫 my template。在这里有很多的选项，有若干多的开关，我们把它全部打开。每个开关还有若干多的子项目，每一个子项目都是一个事件。现在我们已经创建了一个模板，并且勾选了所有的事件，点击确定。在这里事件设置选中我们刚刚创建的模板，点击下一步可以发现 g m c 会让我们针对事件再去做一系列的配置，这个配置和上一步也都是一样的，点击完成，这样就开始录制了。它会进行倒计时，直到我们设置的 1 分钟时间结束。在这 1 分钟以内，我们就来调用一些API，让应用执行一些任务。 send send，多 send 几次send。疯狂的调用。


好。时间到了，它会自动生成一个报告给我们，告诉我们当前应用可能会存在哪些性能问题。比方他认为我们现在的应用 CPU 的负载有点过高，内存上面也有可优化的空间，这样我们在做性能优化的时候就比较容易有的放矢了，对吧？可以点击右上角的表格的按钮，这样可以看到更多的详情。


这里的分数越高，表示这一项问题越严重。像我们现在应用存在的问题有这么多，对吧，还是很可怕的。点击没了的详情，也会跳转到相应的分析详情页面。比方双击对内容，它会跳转到内存页面，你就可以根据这里的信息去分析相关问题的原因了。好，下面我们就来系统的过一下左侧的大纲。整个大纲主要分成了 5 部分。第一部分自动分析结果。刚刚看到的界面，它展示了当前应用可能存在的问题，以及针对应用的每一项打分。加班应用程序。这一栏，展示了 Java 应用的各种执行情况，里面包含了现场的执行情况。在这个里面可以分析线程帧用的情况，内存的使用情况，所得相关信息，文件 IO 的信息。也就是这段时间里面读取或者写入了哪些文件 socket 的相关信息？这段时间打开了多少的 socket 文件 IO 以及 socket IO 对我们分析应用 IO 相关的问题，是比较有用的。


方法。概要信息里面展示了当前应用里面的热点方法，但是目前来说我们的应用还没有热点方法。异常错误展示了在录制过程中应用发生过的异常。线程转储，也就是 thread dump。可以发现在 1 分钟里面做了两次 thread dump。 j v m 内部这个页面会展示 g f m 层面的执行情况，里面的信息相对会底层一点。在这个页面的右边会展示当前应用所有叉叉开头的参数。这个里面又包含了若干多的子页面。垃圾收集展示了当前应用有关垃圾收集器的各种信息。在这里可以发现触发了 3 次垃圾回收。如果你想检查原因，可以按照这里的原因去搜索。比方你可以搜索一下这里的 HEAP inspection initial 的GC，你就会知道 heap inspection initial 的 GC 是在做建 map Histo 的时候导致的，也就是在生成堆内存的直方图的时候，会导致 heap inspection initial 的g、c。同时这个事情也告诉我们，在生产环境，我们要慎重地对应用进行j、m、a、p、 s to 操作，对吧？这些配置展示了有关垃圾收集器相关的各种配置。比方在并发阶段有多少个线程，并进阶段有多少个线程。还记得并发和并行之间的区别，还有一系列各种各样的配置。


信息编译这一栏描述了方法的编译情况，编译花费了多久，编译后的机器码有多大，内联的代码有多大等等。而失败的编译这一栏又会展示哪些代码编译失败了？这一栏也比较有意思。如果你的某些代码执行效率不够理想，并且它恰好在编译失败这一栏，你应该想办法参照消息这一栏的提示，让你的方法能够编译执行。


类加载这一栏描述了当前应用在这段时间的类加载以及卸载的情况。 VM 操作指的是 g m m 底层操作的统计，比方 g c。 keep inspection 是对垃圾收集的一个诊断，它会告诉你这个过程花费了多久？ Redefine classes 则描述了类文件重新定义的相关信息。其他的 VM 操作同学们可以百度一下，也都比较的简单，一搜就知道是什么意思了。


t lab 分配描述了 t lab 这一块内存区域的分配情况。有关 t lab，我们将会在课程的实战章节专门去探讨。环境描述的是操作系统层面的执行情况，里面的信息会相对更加底层一些。里面包含了比如应用层用到的系统内裤有哪些，环境变量有哪些，系统属性有哪些。针对这一次建发录制的一个记录和配置是怎样的，他告诉你记录名是什么时候开始的，什么时候结束的等等等。各种设置的值是怎样的。最后还有一个事件浏览器，它展示了录制期间所发生的所有事件。这里的事件种类就是根据前面我们配置的模板去采集的，它可以统计出更像你关注的事件发生了多少次。在这个下面还会有发生这个事件时候的对战跟踪。好，这就是使用 GMC 去可视化分析GFR。
