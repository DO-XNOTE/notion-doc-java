好，经过前面的讲解，同学们应该可以发现，使用接 Jstatd 实现远程连接虽然相对方便，但是它并没有去考虑网络安全相关的问题。如果你的远程服务器部署在公网环境，直接开放端口还是可能会存在安全风险的，对吧。而 GMX 虽然可以开启用户认证以及 SSL 去增强安全性，但是要做一大堆的配置。


这一节我们要探讨的基于 shh socks 代理的方式，就能够实现比较安全同时也比较方便的远程连接。实现 ssh socks 代理之后，请求过程就不再是本地直接和远程服务器上的进程或者是 Jstatd 通信了，而是本地进程去连接 shh socks 代理，再由 s h 代理把请求转发到 Jstatd 的地，或者是你的远程进程。由于 shh 的安全性是有保证的，所以这种方式也会比较的安全。


下面我们就来以 g m x 为例去演示怎么样让 jmx 连接上 ssh 去代理实现远程连接。首先停掉之前的远程进程是查看进程号， Q 45963，杀死也能进城。接着 copy 这一段粘贴，跟上价包名称，重启远程进程。从这些参数可以发现，其实这里就是用了 GMX 的匿名访问对吧。这里我们使用了 1235 以及 1250 两个端口，但是目前我们并没有去开放这两个端口。新开一个窗口，使用 list ports 查看一下当前开放的端口，可以看到里面并没有包含 1235 以及 250 这两个端口。


好，接着看第二步，使用这个命令去实现 s h 端口转发。这个命令是在本机执行的， copy 粘贴，输入密码。这样执行完成之后，就会分配一个sock，并且监听 9696 端口本地的进程。只要连接上 9696 端口，就可以通过安全通道由 s h 把请求转发到远程服务器。
如果你不想用 s h 命令，也可以借助一些基于 shh 的远程工具去实现端口转发。常见的secureCRT、Xshell、 party 等等都可以支持端口转发。好，做完这两步之后，就可以使用 shh 实现远程连接了。我们还以为是 m 为例， bin 打开WM，拖动 enter 启动。首先点击 Richard venom preferences，点击network，选择 Minerproxy settings 手动配置代理soxproxy，写 12 七点零点幺，端口是 9696 NO proxy hosts，我们可以把它删掉。


点击 apply okay，点击 file add GMX connection，输入3W，点 itmuch.com 1235，勾选 do not require ssl connection okay 连上了，双击可以看到使用没有问题。好。回到手机不难发现，使用shh 的方式比前面要简单不少。我们不但需要为Jmx 配置账号密码，也不需要配置ssl 就可以实现安全的连接。最后总结一下 3 种方式各自的优缺点，同学们可以参考一下这个表格，从而帮助你在实际项目中选择远程方式。好，这节课就到这里，谢谢大家。
