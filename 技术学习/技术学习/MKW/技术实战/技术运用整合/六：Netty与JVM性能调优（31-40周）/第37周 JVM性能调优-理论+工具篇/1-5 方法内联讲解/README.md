大家好，我是大木。上一节我们已经讲过， JVM 会用编译的方式为代码提速，在这个基础上，即时编译器还做了很多的优化措施，从而进一步的提升性能。这一节探讨的方法内联就是即时编译器的优化措施之一。来看一下这段代码。这段方法非常的简单。 add 1 方法里面调用了两次 AT2 方法，大家想 BAT 会怎么样优化这段代码？我们知道调用方法需要经过压栈和出栈的操作，进入方法的时候会向栈里面押入一个元素， return 的时候会从栈里面弹出这个元素。而压站和出站操作都是有开销的对吧？比方压站得往站里面存数据，所以存在内存的开销。同时压站和出站也都是需要时间的，所以还有时间的开销。如果这段代码的调用次数不多，压战和出战的开销倒也无所谓了。但是假设这段代码调用非常的频繁，比方每秒要调用2万次，累计下来的开销还是比较可观的，对吧？是不是有什么办法去优化？当然是有的。


j v m 它会自动识别热点方法，并自动的去进行方法内联。所谓方法内联，就是把目标方法的代码复制到发起调用的方法里面去，这样去避免掉真实的方法调用。比如像这样的一段代码，经过方法内敛之后，就变成了这样子。我们写一下内联后，加X4，也就是它会自动的把 AD1 方法和 AD2 方法合并到一起去，从而减少掉压站和出站的操作。需要注意的是，使用方法内联，它是有条件的。第一是你的方法体要足够小，如果方法体太大， j v m 是不会内敛的。默认情况下。


对于热点方法，如果方法体小于 325 字节，都会去尝试内联。你也可以使用 frequence in line size 去修改方法的大小。有关键 VM 如何找到热点方法，上一节已经详细探讨过了对吧。对于非热点方法，如果你的方法体小于 35 字节，也会去尝试内联。你也可以使用 Max in line size 去修改方法的大小。这是方法内联的第一个条件。

被调用的方法能在运行时，方法的实现可以被唯一确定，这样的方法才可能会被内联。对于 static 或者 private final 修饰的方法都是可以唯一确定实现的。但是对于 public 修饰的实例方法，它指向的实现可能是这个方法自身或者是父类或者子类的方法实现，因为有多态的存在。这种情况下，只有当即时编译器能够唯一确定方法的实现时，才可能完成自联。


根据这两点条件，我们可以总结出使用方法内联的几个注意点。第一是我们在写代码的时候，应当尽量避免在一个方法里面编写大量的代码，让方法体尽量的小一点。第二，我们应当尽量使用final，private， static 等等关键字收饰方法，避免因为多肽要对方法做额外的类型检查，而且很可能检查之后发现还没有办法内敛，因为没有办法唯一确定方法的实现。第三，在某一些场景下，可以通过设置 JVM 参数去减少热点的阈值，或者修改方法体大小的阈值，从而让更多的方法去进行内联。

比如配置 frequency inline size 或者是 Max inline size，对吧下面来探讨一下使用方法内联可能带来的问题。


首先要强调的是，方法内联不是万能药，它也是有缺点的。内联本质上是一种用空间换时间的玩法，也就是即时编译器在编译期间把方法调用连接起来，从而减少进站和出站的开销。但是经过内联之后的代码会变多，而增加的代码量又取决于方法的调用次数以及方法本身的大小。所以在一些极端场景下，内联甚至可能会导致 code cache 的溢出。


code cache 是热点代码的一个缓存区，即是编译器编译后的代码以及本地方法代码都会存放在 code cache 里面。这块空间是比较有限的， JDK8 默认情况下只有 240 兆。这块空间一旦溢出，甚至可能会导致 JVM 放弃编译运行而退化成解释执行模式。目前同学们只要知道有这么一回事就可以了。在后面的章节，关于 code cache 的调优会详细探讨的。
