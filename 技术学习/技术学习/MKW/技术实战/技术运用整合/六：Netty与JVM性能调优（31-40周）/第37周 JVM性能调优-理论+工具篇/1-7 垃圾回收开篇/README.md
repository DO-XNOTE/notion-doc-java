垃圾回收是性能调优的重中之重，也是高端程序员面试必备的领域，绝对值得我们深入研究垃圾回收相关的内容。水非常的深，细节也非常的多，大部分在准备这部分课件的时候也着实花费了非常多的精力。不过同学们不用担心，这块知识点其实单纯从难度上来讲并不是那么难以接受。而且大漠已经把比较抽象、难以理解的部分都用代码演示或者类比的方式给具象化了，所以理论上应该不会感到无所适从。这一点同学们应该对自己有一些信心。


强烈建议同学们在学习这部分知识的时候，关键还是要能够保持耐心，多花时间琴记笔记，如果学完一遍之后没有轻车熟路的感觉，建议二刷甚至是多刷。另外也可以结合市面上的一些文章或者是博客学习这部分内容，使用多方印证，加深印象的方式在学习垃圾回收是很有必要的。


再一个，当遇到问题的时候，千万不要羞涩，同学们可以在课程的问答区提出来，大漠会在第一时间和大家交流，为大家作答。好，下面一起来跟随大漠来啃垃圾回收硬骨头。我们知道，在开发 Java 程序的时候， Java 程序员一般是不需要关注对象的回收的，而是由 Java 的垃圾回收机制帮助我们自动回收掉没用的对象，对吧？但是 GM 提供了多种垃圾回收算法，以及多种垃圾回收策略。不同的回收算法以及策略有不同的适用场景。如果你在项目中使用了不合适的垃圾回收算法或者策略，系统的性能将会很难达到最优。在某些业务场景下，不合适的垃圾回收算法或者策略甚至可能会导致性能的大幅下降。所以，垃圾回收的重要性是毋庸置疑的。这节课我们主要探讨 3 个话题。

第二，垃圾回收发生在哪些区域？第三，对象在什么时候能够被回收。先来探讨一下什么场景下该使用什么样的垃圾回收策略。一般来说，总体原则大致上是这样的如果你的项目对内存要求非常苛刻，这个时候就要去想办法提高对象的回收效率，在回收的时候要尽量多回收掉一些对象，从而腾出更多的内存。如果你的项目 CPU 使用率比较高，这个时候就要去想办法降低高并发时候垃圾回收的频率，从而让 cpu 更多的去执行你的业务，而不是垃圾回收。

在做垃圾回收的时候，首先会判断对象有没有和跟对象的引用链，如果存在引用链就不回收。如果没有引用链会给他判死缓。怎么样给这个对象判死刑？ JVM 首先会判断有没有必要去执行对象的 finalize 方法，如果没有必要执行对象的 finalize 方法，就直接回收。如果你的对象没有重写过 finalize 方法，或者是 finalize 方法已经被调用过了，都会认为没有必要再去执行 finalize 方法了。如果判断出来有必要执行 finalize 方法，就会把这个对象放到一个名为 f q 的队列里面去，会由虚拟机自动建立一个低优先级的 finalizer 线程，去执行这个对象的 finalize 方法。

代码执行到这里。按照刚刚的讲解，首先会调用 finalize 方法，把对象扔到 f q 里面去，因为这里我们重写了 finalize 方法，
