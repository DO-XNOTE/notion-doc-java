新生代又被细分成一等，也就是一电源survivor，也就是存活区。而存活区又由 from survivor 和 to survivor 两个区域组成。在 GDK 8 之前，对内存主要分成了新生代、老年代以及持久代，也叫永久代。而从 JDK8 开始，持久代被废弃了。由原空间代替。原空间并不是堆内存的一部分，而是一块本地内存。

虚拟基站由战争组成，每一次的方法调用都会创建一个战争，然后去压战。当方法返回的时候，则对应着战争的出战操作。也就是你每调用一次方法，就会往里面压入一个元素，当方法返回的时候，则会把元素弹出。


战争。里面又存放着一系列的数据。首先是局部变量表，第二是操作数栈方法里面的代码。在执行的时候会从局部变量表或者是对象实例的字段里面复制变量或者是常量，放到操作数站里面。当计算的时候，会使用一系列的指令往操作数站里面放入数据，或者是取走数据进行计算。可以认为操作数站是用来存放临时数据的地方。此外还有指向运行时常量池的引用方法的返回地址、动态连接等等。
第三是本地方法站。和虚拟机站的功能是类似的，虚拟基站管理 Java 方法，而本地方法站管理的是 native 方法。说到 native 方法，还记得前面讲 range 的 lock 的时候提到的 unsafe 类、 on safe 类调用的就是本地方法对吧？本地方法都是由 c 语言去实现的。


第四是程序计数器。程序计数器用来记录各个线程执行的字节码的地址。像分支循环、跳转异常、线程恢复等等操作都需要依赖程序计数器。大家想为什么要有程序计数器空间？这是因为由于 Java 它是一个多线程的语言，当执行的线程数量超过 CPU 核心的时候，线程之间就会根据时间片去争抢 CPU 资源，对吧？举个例子，某一个县城，它的任务还没有执行完，但是 CPU 资源就被其他的县城给抢了。之后如果又轮到县城去执行，你知道从哪边继续执行？所以为每一个线程都分配一个程序计数器，用来记录它下一条运行的指令是什么等等。

字符串常量池可以理解成运行时常量池里面分出来的一部分，当类加载到内存的时候，字符串会存到字符串产量池里面去。值得一提的是，方法区和堆之间是存在交集的静态变量，以及字符串产量池存储在堆里面。而类的相关信息以及运行时产量池存储在原空间里面。


还记得原空间是一块本地内存对吧？当然这个图指的是 j d k 8 以及之后版本的内存分布。事实上，不同 j d k 发行版，甚至是不同的 j d k 版本，实现上是有差异的。比方我们比较熟悉的 Oracle j d k 加把杠version，它是一个 hotspot 虚拟机。


hot spot 虚拟机。在 JDK 1. 7 之前，它使用永久代来实现方法区。但是其他的虚拟机，比方借 Rocky 的虚拟机，或者是 IBM 的街酒虚拟机，根本就没有永久带这种东西。又比如 hotspot 虚拟机。到了 j d k 1. 7 的时候，它在原先的基础上做了一些改进，把字符串常量时移动到了堆里面去了，而符号引用则被放到了本地内存里面去了。之后到了 j d k 1. 8 hot spot 虚拟机直接把永久代给干掉了，用圆空间取而代之。所以要使用圆空间替代永久代，主要有两方面的原因。一方面是奥瑞克把 hot spot 虚拟机和杰 Rocky 的虚拟机都收购了，而刚刚说了解 Rocky 的虚拟机没有持久代这种东西，于是为了融合 hard sport 虚拟机以及 JIRA k 的虚拟机，所以干脆把永久代给去掉了。


第二个原因是持久代在使用的过程中还是挺容易发生问题的，相信很多同学都见过这种异常。 out of memorial 来一个permission。我们知道 GDK 8 之前使用持久代来实现方法区，而方法区存储的主要是一系列的常量和类以及方法的相关信息，这块区域其实是比较难以准确确定大小的，因为每个项目所加载的类方法都是不太一样的，你的持久带设置的太小，就容易出现这一块区域的内存溢出，而设置的太大又会导致内存上的浪费。于是把持久带给去掉了。用圆空间代替。用圆空间代替有什么好处？圆空间是一块本地内存，理论上取决于操作系统可以分配的内存大小，这样就解决了持久贷空间难以分配的问题。感兴趣的同学也可以阅读一下这篇文章，里面详细探讨了为什么要把持久贷给干掉。
