大家好，我是大木。下面我们花一些时间探讨编译器的优化。先来探讨自解码是如何运行的。众所周知，加法有两种运行模式，第一种叫解释执行。所谓解释执行，它是由解释器一行一行的去翻译执行的。

第二种叫编译执行。编译执行它会把自检码编译成机器码去执行机器码。

解释执行的优势在于没有编译的等待时间，但是它的性能相对会差一些，因为一行一行的去翻译，性能可想而知，不会很高。编译执行的优势在于运行效率会高很多，一般认为它比解释执行会快一个数量级。但是编译执行也不是十全十美的，它带来了额外的开销，比方额外的内存开销， CPU 开销等等。

一会儿来揭晓。为了提高热点代码的执行效率，就会用即时编译器，也就是平时我们说的JIT，去把这些热点代码编译成和本地平台相关的机器码，并且会执行各种层次的优化。这里本地平台有多种层次的含义，比方操作系统的不同， Linux 操作系统和 windows 操作系统，它的平台是不一样的。又比如 CPU 架构的不同， X86 CPU 架构和arm CPU 架构也可以认为平台不同。还有这里他说会执行各种层次的优化。这种各种层次的优化究竟是什么，一会儿也会详细揭晓。

第0个级别叫解释执行。第一个级别叫简单 c 1 编译，这个级别下会使用 c 1 编译器进行一些简单的优化，并且不会开启profiling。所谓的 profiling 也就是jvm 的性能监控。第二个级别叫受限的 c 1 编译，这个级别下仅会执行带有方法调用次数以及循环回编执行次数。 profiling 的 c e 编译方法调用次数以及循环回编执行次数。1、会详细探讨第三个级别完全 c e 编译。

在这个级别下会执行带有所有 prefiling 的 c e 代码。第四个级别也是最高级别 C2 编译。这个级别会使用 C2 编译器进行优化，并且会启用一些编译耗时比较长的优化。某一些情况下甚至会根据性能监控的信息进行一些非常激进的性能优化。


一般来，级别越高，应用的启动也会越慢，优化的开销也会越高，当然了，峰值性也会越高。所以现在大家应该知道这里所谓的各种层次的优化是什么意思了。

方法调用计数器统计的并不是方法被调用的绝对次数，而是一个相对的频率，也就是一段时间之内这个方法被调用的次数。如果超过一定的时间限度，这个方法调用的次数还是不能让它提交这里的编译请求。这个方法的调用计数器值就会被减少一半，这个过程被称为是方法调用计数器热度的衰减。所谓的时间限度就被称为是这个方法的统计。


半衰周期。衰减的动作是在虚拟机垃圾回收的时候顺便执行的。你可以使用这个参数去关闭热度衰减。关闭掉热度衰减之后，方法计数器统计的就是绝对次数了，这样你的系统只要运行的时间够长，总有一天会超过阈值，这样绝大部分代码都可以编译执行。另外你也可以使用参数去设置半筛周期。



再来看一下回编计数器是怎样的流程。当解释器遇到一条回编指令的时候，会先查找一下要执行的代码片段，是不是有编译好的版本，如果有就直接执行编译的代码。如果没有回编技术记值会加1。会判断方法调用计数器和回编计数器的和是不是超过阈值，如果超过阈值，就向编译器提交一个 OSR 编译的请求，同时会把回编计数器的值降低一些。也是继续以解释方式运行代码。当编译器编译结束之后，后面的请求就可以编译执行了。
