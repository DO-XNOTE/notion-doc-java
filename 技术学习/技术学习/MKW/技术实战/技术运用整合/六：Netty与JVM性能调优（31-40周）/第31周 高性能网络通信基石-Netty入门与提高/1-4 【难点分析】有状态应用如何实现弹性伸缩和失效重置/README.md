这一节给大家介绍个小花絮，有状态应用是如何进行弹性伸缩的。大家想一想有状态应用该怎么伸缩，好不好弄，其实非常复杂，所以我们会有几个思路。我们先要看一下什么样的状应用是有状态应用，通常它是什么？它要本地，要写一些数据对不对？它在本。所谓的本地，通常是指在自己的服务器所在什么文件系统里面，以这种形式为主导的。这个时候我们要想一下它是不是能把它的数据用共享磁盘的方式，不管磁盘是以数据库或者其他方式能给分离出去，还是他必须用 share nothing。

我跟你之间是没有任何共享的，我没法用共享数据的方式跟你沟通。我必须是我有一份，你有一份。我们之间要么是什么复制，要么是什么无关的这种关系。所以这种应用就分为两种模式的。我们把有状态应用分为这两种模式。一旦碰到第一种模式，你跟我其实都是有数据的，在本地文件系统，但是我们其实可以把数据共享出来。这种时候我们可以采用什么？把共享磁盘模式的这种应用转成无状态应用？后面一个篇章我会看一下片子，会看一下怎么样转。


另外一个就是情况确实我们是 share nothing，比如芒果d、b、self、 Kafka 这些应用，我跟你之间不需要进行任何共享的。这个时候我们采用一些合适的集群管理和 c i p 目标。c、 a p 什么一致性高，可用分区对吧，这三个不同的目标来进行合理的掌控，逐步实现。即使是 share nothing 这种有状态的时候也能少量的伸缩。首先看一下怎么样从有状态向无状态转移。

如果你是结构化的数据是吧？肯定是可以放在共享的数据库里面的。数据库完全可以给共享给所有节点，是不是这样你的本身的上面的信息就可以少了。所以本来是放本地文件系统的，但它其实是一个有结构化的内容，为什么不放在共享数据库里面？如果它是个非结构化内容，很怪的对不对？我们或者是键值对，或者是一些 session 的信息，我们完全可以放到类似于 Redis 这样的什么共享缓存里面。


如果你是些文件系统信息文件，比如很大是照片对不对，视频，也没必要放在本地文件系统的，完全可以有对象存储。当前可以支持什么 t 比、 p 比大容量的，这种相当于是什么bucket，相当于是存储块。同时在每个文件里面也解决了以前什么只能到 1GB 的这种问题，现在 16GB 这些大文件支持都没有问题，所以不会。


我是一个很大的视频，我是一个什么蓝光视频，所以我不适合对象存储，以及不存在这样的情况。所以非结构化数据完全可以通过对象存储，通过 API 编程的形式把数据对外提供服务。除此以外，还有一些废弃优化，比如是我们要查询一些数据，我们扔到搜索引擎，我们不需要在本地存储，我们可以往外扔对不对？e，l， k 一旦 search 都可以做一些额外的处理好。
除此以外，我们还可以考虑什么？我们尽量减少对文件系统依赖。越来越多的系统会采用一些共享数据库，共享缓存，共享 NO SQL，共享对象存储的方式，把大量的这种对文件系统依赖抛离掉。即使是你对文件系统有很强的依赖时候，你也可以使用。因为共享的分布式文件系统外面已经有很多，比方self，比如像GPFS，一些开源的，或者是一些闭源的。这种某个企业提供的共享文件系统。这些共享文件系统都能实现。


大部分的什么？大部分的这种分布式功能，集群功能，它自身实现。当你把你的应用系统的数据剥离出去以后，你就变成了一个彻底的无状态应用。这里还有一个经典例子，当数据被剥离到对象存储以后，甚至于它都可以真正实现，跟服务器无关了。服务器只要记录一些 Meta data，c，d，n，我们的什么，我们的语音上的内容，缓存系统跟客户非常接近的这样一个 content delivery network。它可以直接从对象程度上拉取数据，来实现快速的用户的数据可达。对这种层层的剥离以后，使你的应用服务器逐渐无状态，逐渐透明化。
