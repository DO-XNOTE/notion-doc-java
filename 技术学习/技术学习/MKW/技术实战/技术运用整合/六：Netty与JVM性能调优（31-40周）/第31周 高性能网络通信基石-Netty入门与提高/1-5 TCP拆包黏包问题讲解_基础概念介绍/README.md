OK 小伙伴们大家好，这节课我们继续来往下来讲解。来说一说 TCP 拆包年包的问题， native 是如何去处理的。首先我们对于拆包年包的概念简单的做一个介绍。 TCP 底层有一个拆包和年包的机制，你可以认为它其实不是具体的，产生的问题是因为我们在实际的处理数据的时候，需要知道哪些多大形成一个数据包。所以机制这是什么概念？就是我们熟悉 t c p 编程的小伙伴可能都知道，你无论是 client 端还是 server 端，当我们去读取或者发送数据的时候，都需要考虑 TCP 拆包年包的这件事情。


TCP 本质上它其实是一个流协议，所谓流其实就是它没有界限的。大家可以想象一下我们的 TCP 协议，你可以理解为好像是河里的水一样，这个水就好比我们自己 TCP 协议上的实际的数据，它们是连成一片的，是没有任何分界线的。所以 t c p 底层它其实并不了解你上层的业务具体是什么含义，多少个字节自己是一个数据包，他并不知道，所以他只能根据 TCP 缓冲区实际的情况对我们的数据包进行划分。也就是在业务上，一个完整的包可能会被 TCP 协议分成多个包进行发送，分批分成多次进行发送，也可能把很多小的数据包打成一个大的数据包发出去，这都是有可能的。这个就是所谓的 t c p 拆包与年包的这么一个问题？举一个例子，比如我们现在想要 5 个字节当成一个数据包发出去，结果他给我发了 7 个，对不对？就是一个年包的问题对吧？比如现在我想把 10 个字节当成一个数据包发过去，结果他发了两三次，对不对？可能发了 2 次，第一次发 6 个字节，第二次发了 4 个字节，对吧？我们自己的业务逻辑上肯定是不完整的，我们期望按照我们自己所规定的长度读取数据，这样才是一个非常非常完整的一些 CP 的通信。

我了解了我们业界主流的方案是怎么去对 TCP 拆包粘包处理的？对应着你也应该知道 native 他是怎么去做的了。其实都差不多。比如第一种方案，我们可以说消息可以去做一个定长的处理。比如我每一个报文的固定大小 200 个字节，如果不够，我可以用什么空格去进行一个补位？比如我发了 179 个字节，对吧，我还差 21 个字节，我就可以打 21 个空格做一个补位，把数据包满足成 200 个，发到另一端去解析 200 个长度，认为是一个数据包就可以了。 OK 咱们继续往下。


还有比如在我的包的尾部，我们发送最后的时候，我加一个特殊的字符，这个当然是我们双方约定好的最简单的。比如我加一个回车，加一个 dollar 符号等等，这都是可以的。除了这两种方式，还有一种方式其实是业界使用的最多的。无论是我们后面的一些什么自定义协议站，可能都去用这种方式。


我把消息分为我们的消息头 header 以及消息体body。在消息头中，我们可以附加一些属性，比如验签的一些数据，比如我们消息的一个总长度，这是比较关键的。当我读完消息头之后，从消息头里边我拿到当前发的这次数据包，它的数据总长度，比如 216 个字节对吧？接下来我就从消息体开始，往后读 216 个字节，直到处理完成就可以了。好了，这就是我们业界对于 TCP 拆包、年包最常用也就是最有效的解决的 3 种方式。接下来我们对于 Nike 而言，它也是给我们提供了这三种方式。当然了，第三种，这是属于自定义协议站的事情，它有一些最基本的这种 base 类。我们下节课跟小伙伴们来看一看实际的代码是怎么去做的。这节课我们就先把一些概念介绍到这儿，感谢小伙伴们收看。
