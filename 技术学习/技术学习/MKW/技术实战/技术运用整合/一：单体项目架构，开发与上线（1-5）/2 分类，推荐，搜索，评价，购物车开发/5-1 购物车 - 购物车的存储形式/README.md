课程到这里，我们已经是完成了首页商品搜索以及是商品详情页，这些页面其实所涉及到的内容大多都是和查询相关的。接下来其实我们就要涉及到商品的支付了，也就是网站电商平台的一个核心的功能。核心功能它主要会涉及到购物车订单以及是支付。首当其冲我们所需要接触的应该是购物车这个功能。来看一下在商品详情页在这里中间位置有一个加入购物车。什么是购物车？其实线上的电商平台里的购物车和线下你去超市购买东西的购物车其实是同一概念。不管是线上还是线下，我们看中的一些商品，都可以把它们一个一个的放到购物车里，其实是一个道理的，在最后结算的时候，他们也是一次性的去结算的。线下所有的商品是放到了收银台，统一的一次性去支付。在线上也会在购物车里面会有一个结算按钮，点击结算会跳转到一个收银台页面，也就是订单结算页面。所有的商品其实都会包含在同一笔订单里面去进行支付的，它的总价格每一样商品累计所叠加的一个价格。OK。


关于购物车在线上，其实我们是要把它们给存起来的，存到哪里？在这里我们是需要去考虑两种不同的情况，第一种情况是用户在登录的情况下，另外一种是用户在未登录的情况下。两种情况其实我们都应该要去考虑的。购物车里面的数据我们应该是保存到哪里？所以我们这节课就带着大家先来看一下购物车的存储形式。

 cookie 我们其实在之前就已经过了， cookie 是类似于在浏览器在客户端上的一种缓存，它是有一定的大小，大小是不能够超过 4 k d 的。所以我们可以把一部分的数据放到客户端，其实也是可以使用cookie，可以看到我们是无需登录，也无需去查询数据库，可以把一些数据保存在浏览器端的。


由于数据是保存客服端的，相应的它就会有一个优点，它的性能是比较好的，访问也比较快。因为用户是直接访问浏览器，浏览器会把 cookie 里面的数据，也就是购物车数据直接拿出来，随后再渲染到页面，里面的用户在进行页面展示的时候，请求是没有发送到后端，也就没有和数据库进行交互。这样子访问的速度很明显会比发送请求到后端要来的快。所以这一点就是它的优点。由于 cookie 是保存在浏览器端的，所以它也会有相应的缺点。


第一个缺点，你换电脑的时候，购物车它的数据是会丢失的。在这里换电脑，我们所说的是你在未登录的情况下，你从一台电脑换到另外一台，你打开同样的一个电商平台，购物车里面的数据很显然会不一样。OK。第二个缺点，由于 cookie 是保存在浏览器端的，我们之前也说了，它会有一个隐私安全性的问题，如果有这个时候，你的电脑被黑客访问了，它可以窃取你的cookie。虽然 cookie 是经过加密的，通过一些技术手段也可以去解密。去看一下你的 cookie 里面的信息是什么，你买了一些什么东西，你把一些什么商品放到了数据库里面，都是可以看得到的。还有一种情况下，也就是你的家人会访问你的电脑，可能比方你买了，你想要去买一些游戏点券，放到了购物车里面，这个时候你的师傅访问了你的电脑，打开了你的电商平台，看到你购物车里面有一些游戏相关的东西，可能就不开心了对吧？这个时候就让你去固键盘了，也会有这种可能。


这个就是你使用 cookie 的一个隐私，安全使用 cookie 是肯定会有这种情况所发生的。OK，随后我们再来看第二种形式。

session 是一个用户的会话。用户的会话是在用户登录以后才会产生的，是通过 request 去 get 一个session。这个时候其实我们是可以把一些购物车的数据放在用户的会话里面的，是绝对没有问题的。当然，使用会话可能会存在一些问题，我们可以来看一下。


先来看一下它的优点session。它其实是基于服务器的内存的，如果一开始用户量比较少，也就是业务比较少。其实在公司初创期，在你的网站初期，性能会比较好，访问速度也比较快，因为你的用户量比较少，这是一个优点。如果你的一个网站发展了很多年，用户量庞大增长，它会出现一些问题，这就是它的缺点。

session 是基于内存的，用户量一旦增长了，会影响服务器的性能。因为所有的一些购物车数据，你都是会存放到服务器里的内存里的。哪怕你升级内存，比方你从 8G 升值到 16G 64G，这个时候虽然你是扩充了你的内存，但是只要你的用户量一直在庞大的增加，显然内存是吃不消的对吧。而且内存越大，也是对于企业来讲也是一种服务器的硬件成本，这个是需要去让老板去了解去考虑的一种因素。


OK。随后是第二个缺点，对于 session 来讲，用户和 session 是相关联的，并且它的关联是只存在于当前的会话。如果你的用户换了一个电脑，或者你当前的一个服务器，你发生了一个集群以及是分布式的扩展，你的 session 只会在同一个服务器上，它是不会同步到其他的集群或者其他的分布式系统里面的。这绝对是一个已使用这种单机 HTTP session 的一个问题，所以这种 session 我们是不会采用的。


OK，当然在我们课程里面，我们目前虽然是单级的，但是我们后面是会逐渐发展成集群和分布式系统，所以 HTTP session 用户这种有状态会话，我们目前是不会使用的。

数据库可以在用户登录以后，只要是添加商品到购物车，相应的数据我们是可以保存到数据库里面的，所以在数据库里面，我们肯定会有一张购物车表，并且这一张表是和用户表要相互关联的。既然涉及到数据库了，所以所有的购物车商品数据是可以做到数据的持久化的。这个时候对于用户来讲，它是可以在任何地点，任何时间去访问的。只要用户登录，不管是你有没有换电脑，它都可以访问到以前的那一台电脑所保存的一些购物车数据。这就是它的一个优点。使用数据库，它其实会有一个致命的缺点，也就是频繁读写操作数据库。假设现在网站做得很大，可能会有同时几千甚至几万个人在线，对于用户来讲，他们是可以频繁的去把一些商品加入到购物车，也可以频繁的去删除一些商品，这个时候频繁的读写购物车也间接的就是频繁的读写操作数据库了。很显然会对数据库造成一定的压力，这种情况我们是要避免的。所以我们不会使用数据库来保存用户的一些购物车数据的。

它是一个分布式的缓存中间件，它也是可以做数据的持久化的。它和绘画是不一样，绘画是把一些内容存到内存里面去的。 Redis 它是可以把数据持久化到浮潭里面。相应的，当用户去访问的时候，如果现在我们的服务器进行重启了会话，数据是会丢失的。 Redis 是不一样，因为它是把数据持久化到磁态里面，而且持久化磁盘用户在请求的时候查询的效率，查询的速度要远远的要比查询数据库要更快。所以 Redis 就可以作为这样的一种媒介或者是载体，为我们去提供一个购物车存储的位置。


来看一下，在用户登录以后，购物车的数据我们是可以存入到 Redis 缓存里面去的，缓存，它的读取的效率是非常非常高，也是非常非常快的。其实在之前我们已经是提到过一个分布式绘画，分布式绘画可以依靠 Redis 去做。对于我们来讲，我们的购物车，其实我们在后续也会整合到 Redis 里面，会使用 Redis 来做，也能够称之为是一种分布式的购物车。


OK，只用Redis，它是会有一些优点的，比方第一个是数据库可以做到持久化，可以在任何地点任何时间去访问。这里是要区别于数据库，它也是数据化，也可以在任何地点任何时间访问。但是使用数据库，它的性能要比 Redis 要更低，因为数据库它往往会成为一个整个网站最终的一个瓶颈。所以我们能够使用缓存，就尽量的使用缓存来替代数据库。OK。


第二个优点，其实我们之前也提了，频繁的读写是基于缓存的缓存，我们是不可能再会让请求到达数据库，所以也不会让数据库有一定的性能瓶颈，缓解数据库的压力。第三一个使用Redis，它是非常适合于在集群和分布式系统里面去使用的。如果我们现在使用的是 HTTP session，每一个服务器，它可以有一个独立的绘画。既然是独立的绘画，在集群和分布式里面，每个服务器之间的绘画其实它都是不一样的。但是一旦使用了 Redis 以后，绘画师可以进行一个统一。所以我们可以在集群和分布式环境里面，我们可以拿到缓存里面的数据。用户同意了以后，用户相应的购物车数据我们也都是可以拿到的。使用Redis，它的可扩展性是非常强的。
