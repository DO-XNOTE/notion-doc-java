在这边会有一个 payment server URL，其实和我们以前所使用的前端的 server URL 是不一样了。我们来看一下。目前在我们的页面里面使用的是一个，在这里 a p 点 g s 在这里有一个 payment server u l 这是一个支付中心的服务地址，这个地址在本地是没有的，大家在使用再去调用的时候也是会使用。在云端有老师所提供的一个支付中心，所以直接注释掉不去使用就可以了。把下方给开放才是我们目前的一个真正的在云端的一个支付中心。


foodie payment 这个是大家所需要去使用的，改完了以后， payment server UL 在这里就可以正常的去使用了。后方这一段是它的一个路由，路由在后方又拼接了两个参数，第一个是 merchant user ID，第二个是 merchant order ID。 user ID 其实当前在我们的一个电商平台上，是哪个用户下了单，哪个用户的一个用户ID，也就是当前登录者的一个用户ID，另外就是一个 match order ID，这是商户的订单ID。商户 match 其实就是商户的意思，这个就是代表我们的一个天天吃货购物平台。 order ID 就是在我们上一个页面订单生成以后所得到的一个订单ID，也就是从上一个页面在这里所传递过来的一个 order ID。这两个参数是需要去提供给接口的，提供给他了以后就可以去获得一个相应的支付的二维码，就是微信的支付二维码。但是一定要注意，我们在之前已经是说了，大家要去调用支付中信，你是需要找到老师为你提供相应的一个权限，也就是为你分配一个账号以及是密码。在这里可以看到是我目前自己做测试的时候使用的两个字段，其实一个就是幕后网的一个用户ID，另外是你的一个密码，这个密码是由老师为你分配的，后续会为你生成一个随机的字符串分配给你。以后在这里只要在调用我们的一个支付中心的时候，在黑德子里面加上这两个参数就可以了。需要注意的是拼写一定要一致，一个是 m 可 user ID，另外一个是password。分配权限以后的账号和密码在这边写一下就可以去调用咱们的支付中心了。


OK，现在我们其实可以是正常的去调用接口了，因为我们的 server UL 支付中心的就已经是改掉了，我们到前端页面，我们直接去刷新一下，在这边我不管刷新多少次，我们的二维码还是显示不了，这是为什么？我们可以来看一下。按一下 F12 放大在这边给大家来看一下。在这里会有一个地址，这个地址来看一下，你会发现它还是调用我们本地的一个支付中心，这个地址没有发生任何改变。所以在我们的当前，浏览器其实是会有缓存的，这一点你一定要注意。当我们去更改了 APP 点JS，这个 JS 其实是我们前端的，这些源码是放在了 Tom kit 里面，你修改了 JS 以后，其实它会在本地是有一定的缓存的，你是需要强制在刷新我们当前页面的。好，现在我来尝试刷新当前页面。好，现在是强制了刷新了。刷新以后会出现一个错误，来看一下，他会说该订单不存在或者已经支付。这个又是什么情况？主要是因为我们现在的一个请求，其实已经是请求到了我们的一个支付中心了，只不过当前这一笔订单在支付中心并不存在。因为我们在之前只要是在本地用户把相应的订单数据提交以后，创建了一个订单以后，这个订单是在我们天天吃货购物平台里面的。


其次，我们是需要把咱们的一个商户的订单相关的一些信息和数据发送到支付中心，因为支付中心才是真正的向第三方支付，也就是向微信和支付宝去发起支付请求的，你必须要有自己的相关的一些订单信息，你要让他去进行保存，你要让支付中心也有这样的一些订单，才可以让他去进行一个支付的请求发起。这一点是需要去做的。


关于支付中心的一个订单，我们来在这里也给大家去看一下它的一个数据库表设计是怎样的。打开数据库，我们来看一下，在我们的数据库里面，名字叫做 IT 自习配，在库里面有一个 orders 收集表，一起来看一下。在这里面首先第一个是ID，是订单的一个组件，第二个和第三个是商户的订单号，以及是商户方用户发起的一个用户组件。我们刚刚已经是说了，这两个是必须要传过来去进行保存的。最后一个amount，这个 amount 是实际用户所要去支付的一个总金额。这个总金额也就是之前用户所有商品的一个最终的优惠价格，乘以它的一个 by COMS 购买的件数的一个总和，再去加上你这笔订单。如果你有运费，你还要把运费的一个金额给加进去，这个才是你真正要去做支付的一个金额。因为这个时候是需要向支付中心，向我们的第三方支付，要让他们来为我们提供一个支付的场景。所以这是一个最终的价格。如果你仅仅只是把一个订单金额提过来了，你可能会有一个邮费，你邮费怎么去付？所以邮费和你的一个订单的总的费用都要累计。放在这个地方。又是它的一个amount，下一个是一个支付方式，因为我们的微信和支付宝它的支付方式，它的一个接口都是不一样的，所以你的支付方式是 1 还是 2 都要往这边传。随后又有一个 pay status，这是一个支付的状态，区别于订单状态。来看一下它有 10 和 2010 和 20 是最主要的， 10 是未支付， 20 是已支付，随后还有是 30 和40，是一个支付失败和已退款。 30 和 40 在我们其实几乎是用不到的，主要是 10 和20。


下一个是 come from，这是我们的一个订单，我们支付中心里面的订单要保存的时候要看一下订单是从哪一端来的。虽然我们现在其实都是从天天吃货那一端来的，因为我们的支付中心其实不仅仅可以提供给我们的天天吃货，如果老师我以后有其他的一些实战课程，我可以直接对接我们当前支付中心。这个支付中心其实它本身就是一个聚合的支付，我可以提供给其他的一些项目去使用都是没有问题的。OK？ come from 也就是用于去做一个标记是从哪一个平台过来的。下面有一个 return your，这个 return your 其实就是一个支付成功之后的一个回调的通知地址，是开发者那一端的，而不是第三方通知的一个回调。我们会在后面讲到我们的一个支付的时序图，讲完时序图以后大家可能就会理解了，在这里大家只需要知道这个是我们自己的支付成功后的一个通知的回调地址。
