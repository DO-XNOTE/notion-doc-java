在支付宝的一个支付上，其实是分为了三端，首先一个是用户，也就是购买人，付款方是一个用户，随后一个是商户系统，这个商户系统其实我们自己我们的一个支付中心，外加天天支付的一个后台接口，两个在一起。我们的一个商户系统下就是一个支付宝，支付宝其实就是第三方支付系统的一个支付网关。我们来看一下。


首先第一步用户要进行一个下单，这个下单的操作其实会在我们的商户系统里面生成一个订单，其实在我们自己的一个天天吃货平台会有一笔订单。另外在支付中心里面也会有一个订单。随后用户要发起一个请求，这个请求是从我们的商户系统发起的，它会调用一个阿里 pay 点 trade 点、 page 点 pay 这一串的字符串，这个字符串就是标识我们当前的一个用户请求，它其实就要去进行一个支付了，发起请求会到达支付宝这一端，在支付宝这一端会生成一个订单，这一个订单其实它就是一个预付款订单，或者是一笔交易流水。它会在这里去存一下的，它也是为了我们的历史数据，一种历史凭证。


这个是和微信支付也是一样的，在微信那一端你要去支付，其实在微信的支付系统里面也会帮你去保存这一笔交易的，肯定是会有的。处理完毕以后，页面会发生一个跳转，会跳转到支付宝，随后这个网页用户是需要去做一个登录的，输入你的用户名，密码等等。登录完毕以后，你就可以去选择自己的一个支付渠道，可以选择银联，也可以去选择信用卡以及是余额宝都可以去支付。第四步输入完支付密码，确认支付之后，你的金额就会从相应的一个支付渠道里面扣掉，就会到达这一步。OK，其实这一步这一步操作其实都是用户的操作，他没有经过我们的商户系统。除了使用用户名和密码登录以外，还有一种方式就是使用你的一个支付宝的客户端，打开支付宝客户端，通过扫一扫也可以去完成支付的，也是没有问题。只不过他官方的文档在这里他没有去加，因为这个时序图是在很早以前就已经是有了。好继续。


当我们的一个支付的款项扣除以后，在这里它其实在支付宝网关，它会向我们的商户系统会发送一些相应的支付结果，它其实会发两次请求，应该是两种不同的请求，并且是并行的，和微信其实是一样的。


首先它会有一个 get 请求，它是一个同步请求，它会返回到我们的一个return。 while return u l 是在我们支付中心里面去进行设置的。它返回过来其实是以一种同步的形式，返回过来会提示你是支付成功了对吧？是一个页面，但是这个页面是你不可以作为一个真正的支付的凭证。真正支付成功一定要以异步为主，和微信支付是一样的，也就是在这里。这里其实 get 和 post 这两个都是请求，都是并行操作的。 post 其实也是一样，它是一个异步请求，会通知到我们的系统，返回异步通知的相关参数。一旦获得到的一个数据，我们确认是支付成功了。这个时候我们会标记支付中心的一个订单，把它改为已支付的一个状态。随后再会通知到天天吃货的一个平台，会把我们订单状态这张表里面对应的订单，它的状态改为20，也就是支付成功。


OK，这样子就完成了一个支付的闭环。一定要注意，它和微信支付唯一的一个区别就是它多了一个同步的return。 while OK，同步的 return your。我们是不可以作为真正支付的一个凭证。我们是要以异步为主。异步为主和微信是一样，这一点是一定要去注意的。 OK 好，下方它其实还会有一个第 8 步。这个第 8 步和我们之前在讲微信的时候也是一样，他会提供给我们一个主动调用的请求的查询，这就是可以去查询到我们某一笔交易，他在支付宝那一端，他有没有真正的去支付，它会提供这样的一个接口，提供给我们去使用和查询的。
