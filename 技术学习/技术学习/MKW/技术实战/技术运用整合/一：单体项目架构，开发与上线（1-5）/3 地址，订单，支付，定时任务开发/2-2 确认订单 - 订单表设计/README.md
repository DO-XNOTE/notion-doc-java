在订单表里面，首先一个订单的组件是需要有的组件是一个ID，并且这个 ID 是一个全局唯一的。随后下一个是用户ID，也就是这一笔订单是哪些用户的，是需要有一个相应的用户 ID 去进行关联的。最后下面几个是收货人，这是收货人的姓名，收货人的手机号以及是收货地址。在这里我写的是一个快照，也就是当我们用户下单把订单提交以后，用户当时所选择的某一个收货地址，我们在这里我们是不会去保存一个收货地址的外键的，我们会把收货地址具体的一些值拿出来存入到订单里面去。为什么要这么做？主要是因为用户的地址他可能会实时的更改，有可能用户他可以去把相应的这地址给删掉。如果你在这里是做的一个外界关联，一旦发生更改或者删除用户的订单数据，尤其是收货地址相关的内容，可能会发生一个错误，有可能是找不到，也有可能信息是会发生变更。


其实订单它是作为一种**历史数据的，你是不可以因为用户修改了自己的地址，而相应的把订单表里面的曾经的一些收货地址去更改。所以我们在这里放的是一个快照，也就是当时用户在下单的时候，相应的收货人相关的地址信息在这里进行一个保存，也就是当时的信息去保存的，所以我们在这里存的是值，而不是外键关联的ID**。OK，好。


最后下一个是订单的总价格。 total amount 购物车里面有很多的商品总价格会在这里进行一个保存。此外还有是一个实际支付的总价格，因为有些订单可能在最终用户会使用一些优惠券。打个比方，所以我们在这里就会有两个价格。在我们的系统里面应该是两个价格应该都是一样的，除非会有一些额外的像使用积分优惠券，这样子实际的总价格可能会低于订单的总价格。


下一个是一个邮费，在我们的平台统一的我们的一个邮费都是免邮的，都是包邮的。再下一个是订单的支付方式，可以是微信，也可以是支付宝。再下一个是买家的留言，这个是在确认订单里面，在这个地方在这里有一个买家留言，用户可以去填的，填了之后相应的内容在这里面都会有好。

下一个是扩展字段，这个是可以用作预留的。再下面几个是涉及到了一个买家是否评价，这个是用于去判断我们的订单有没有评价过。再下一个是逻辑删除，状态 is delete。为什么要有一个逻辑删除？对于订单来讲，其实订单不管是针对于用户还是平台，其实它都是一些比较宝贵的历史出去，如果用户进行了一个删除，我们是不应该真正的去进行删除，而是进行了一个逻辑删除，相当于是用户删掉以后这一笔订单用户看不到了。但是对于咱们的系统来讲，订单的历史数据我们还是有的，OK，因为**历史数据这些东西都可以作为大数据的一些种子，可以用作大数据分析的**。所以往往我们涉及到一些主要的信息，像订单用户他的一些信息我们一般都是不会真实的去删掉，往往我们都是使用一种逻辑删除的状态去让用户删除的好。

这一张表我们又可以称之为是一种子订单，我们一个订单下，其实这个订单它是包含了很多的商品，它可能有两个，也有 3 个，也有可能是一个。所以相应的一些商品，每一个你应该要把它独立成一项，作为某个订单的子项。所以我们会把相应的商品的一些信息数据保存到里面去。


涉及到的商品内容，其实我们应该把它作为是什么，作为是一个商品，以及是规格。这些相应的一些历史数据，其实和我们刚刚所涉及到的快照内容是一样的。在这里我们所保存的也是像商品的快照，规格的快照信息在这边进行一个保存。只不过我们额外的多了两项，一个是商品的ID，一个是规格的ID，因为是涉及到了商品，商品的数据是比较珍贵的，所以我们也要把他当时的一个商品 ID 规格 ID 也要放过来。因为有可能我们在后台让我们的管理员去进行分析的时候，可能会要去看一下，做一个对比。所以曾经的一个商品ID，不管里面的数据发生了怎样的更改，外界的 ID 你也要关联一下。


OK。当然商品图片、商品名称、规格名称、成交的价格以及是购买的数量，相应的你也要在这边去保存一下。图片名称规格，这些其实都是一个快照。另外成交价格也就是最终的一个价格，你需要在这边去保存一下，其实就是在购物车里面每一项的最终的优惠的价格，提交的时候要的价格。另外还有是一个购买数量，它可以是多个。


另外我们订单是归属于这一个主订单的，所以在这里它会有一个归属的订单ID，也就是它的外键关联的ID。
