先说一下我们整体的架构，是一个前后端分离的架构，所以我们会从用户访问一起开始来说。首先我们先来看一下，一开始用户会通过我们的浏览器或者手机端等等来访问到我们具体的内容，这是一块前端的内容，用户请求经过前端会发送到我们的后端，所以对于这样的一个请求来讲，会进入到我们的负载均衡器，也就是Nginx。在这里 Nginx 我们会有两台，为什么会有两台，主要是因为我们的 Nginx 它也是一个服饰，它也会有可能会宕机，会挂掉。所以我们在这里通过 lvs 加 keepalive 做了一个主备，就保证了我们的一个 Nginx 是不会宕机，就算宕机了也会有备用机给替换掉。


随后负载均衡器会把我们的请求经过了一些处理以后，会发送到我们后面的一部分。在我们的后面的一部分可以看到，在这一块其实是我们的一个微服务了。在这里其实我们已经不再是一个单体，这个是我们在大后期的一个整体的架构。在这里面我们就举例了像用户商品以及是订单服务，这些服务每一个服务它都是一个单独的系统，也是一个单独的服务，服务会向外提供一些相应的接口。当然我们服务和服务之间也是可以相互去进行通信的。这一整块就是我们的微服务。


spring cloud 对于我们的微服务来讲，其实它自己本身就是一个系统，一个系统就会对应到一个相应的数据库，所以相应的它会有用户数据库，商品数据库以及是订单数据库，他们每一个数据库在我们的用户请求量非常庞大的时候，各自对自己的库又可以做一个读写分离组成的分离。好学会来看一下。


在我们的一个应用层到我们数据层之间，他们会有一个缓存，缓存我们会通过 Redis 来进行一个构建，主要也是因为我们用户请求，其实有一部分数据访问的时候，我们是完全的可以从我们的 Redis 里面，从我们的缓存里面去获取的。因为这一部分数据在我们的数据库里面，它几乎是不会去动的，所以我们往往把这些数据放到缓存里面，让我们的用户去读取就可以了，这样子也是对我们数据库减轻了一些负担。


好，随后我们再来看，在我们服务和服务之间，其实是可以进行相互通信的。另外我们的服务也可以去调用我们一些相应的公共资源，在这里有一个公共的服务资源，在我们之前也说了，它包含可以是一些像不管是推送还有是短信以及是邮件，他们都可以去作为一些公共的服务，或者当我们去对接了一些第三方的物流等等，这些都可以作为我们一些相应的公共服务资源去使用。要去调用的时候，往往我们都会通过 Rabbit MQ，我们可以通过一些 MQ 机制，消息队列，也就是生产者和消费者模式来进行一个相应的解耦，并且它们之间的调用是一个异步调用。OK。当然在我们目前的一个系统来讲，其实它本身是一个大型的分布式系统。在分布式系统里面，我们所必须要去关注的点。首先一个像分布式锁要保证我们数据是统一的，像我们的商品数据是不可以超卖的，所以我们通过锁的机制来控制我们的数据的一个唯一性，数据的一致性。另外在我们的用户会话里面，其实如果我们在使用单体的时候，我们会有一个session， session 其实它本身是一种有状态的绘画。当我们在集群和分布式系统里面，我们用户的会话它是属于无状态的。所以在我们整个分布式环境里面，我们必须会采用一种分布式会话这种技术来实现我们的一个用户会话。


随后，既然我们的会话是存在于整个分布式系统里面的，我们会实现单点登录。单点登录保证当我们用户在某一端登录了以后，其余所有的系统都会认为我们当前这样的一个用户已经是登录过了。随后对于我们的数据一致性。另外还有是在高并发下的一个限流，以及是分布式日志，我们都会通过相应的技术手段去实现。另外在分布式系统里面，幂等性也是我们所必须要去进行考虑的。


好。随后我们再来看一下，在我们的整个服务集群里面，其实各个服务他们都会涉及到一些相应的文件。比方在我们用户服务里面，它可能会涉及到一些用户的头像的上传，在我们的商品服务里面，会涉及到一些商品图片的上传，这些内容都是和文件有关的。所以我们会有一个分布式文件系统，我们可以通过 fast d f s 来构建。对于这样的一个分布式文件系统来讲，是可以为我们整个系统提供一个非常好的一种服务机制，我们所有的文件都可以保存到我们这样的一个系统里面，并且我们的 fastdfs 它本身也是可以去构建成一个集群的。


随后分布式搜索引擎 elastic search 在我们的课程里面，我们可以通俗的简称为 e s。 e s 是一种搜索引擎，也是因为我们的用户的一些检索请求是多样化的。我们在上一节课其实也提到过，在数据库查询里面，模糊查询在现如今不太符合用户的一种检索需求，所以我们会通过 yes 来实现用户的一个检索。而且它是一种海量数据的检索，不仅可以去检索出数据，同时其实也是相当于为我们的数据库来分担了一些检索查询的压力。


另外卡夫卡录制收集其实在我们整个系统里面，像不管是阿里还是京东，当我们的一个服务器到达上百台，甚至上千上万台的时候，其实我们的每一个服务器，每一个系统，它都会有日志的产生。这个时候我们如何去管理我们的日志？其实我们应该要对日志去进行收集，去监控，去排错。所以这对于我们运维来讲也是一个非常大的挑战。


如果在很早以前，在传统开发的时候，我们要去排查错误的时候，往往只需要去打开我们的服务器，去看一下我们 Tomcat 里面报了哪些错。这个是在单机的情况下，一旦我们的系统无限扩张，有很多服务器，我们就必须要通过日志收集来去监控我们的一个日志了，这样子就可以做到我们整个所有分布式系统里面所有节点的日志的一个监控。


随后我们再来看一下，在这一块，我们其实在最后我们会涉及到一个多可隆计划。其实我们在前期，如果我们节点比较少，我们像很多的中间件，像一些服务，在我们进行部署上线的时候，我们都可以去单独的去进行配置和安装，那么容易。计划其实也是实验如今非常流行的一种手段。我们的微服务，我们的一些中间件也可以通过 Docker 去进行一些部署和安装，都是没有问题的。此外，我们在整个课程里面，我们也会学习 Netty 相关的一些知识点，比方 Netty 的通信，它里面所包含的一些内部趋势。另外我们也会去通过 Netty 来手把手的去实现 RPC 的一个通信也是可以的。


当然，在我们其实整个开发的生命周期，我们肯定会涉及到一些相应的优化技术，不管是 JVM 还是 Tomcat 还是数据库，都会有相应的一些优化手段。这些都是在相应的产生出不同的问题的时候，我们都要去针对性的去分析，去实现，去调优，使得我们的架构更加的稳定与流畅。


另外在这一块，在之前我们是说了主从分离读写分离。在我们的数据库在大后期的时候，其实我们的整个数据是海量级别的，是非常非常大，它可能会达到上千万甚至是上亿，所以在这个时候我们必须要进行分库分表，也要把我们某一张表里面的数据要进行一个切割，把一张表可能要散列到多个数据库，成为一个分布式的数据库或者是一个数据库集群。在这种情况下，我们会通过 mycat 去实现分布分表。并且我们其实在一开始就要应该要有一个非常好的思想，就是针对我们所有数据库的表组件的 ID 设计，就必须要是一种全局唯一的这种设计。我们的 ID 在不管是任何的一个节点下，在任何的服饰下，唯一的它都必须要有唯一的一个组件。这个是我们在分布式环境下所必须要强调的一点。在我们的课程里面也会去讲到如何去实现全局唯一的分布式 ID 的。

第四点，把控团队，忙而不乱。作为架构师，不仅我们技术要担当，而且我们也要会管理，会管人排兵布阵，人多而不乱，会指导作战下一个系统分解与模块拆分。拆分和解耦其实是一个最基本最常见的模式，很好的把业务去分离，才能够把代码更好的解耦，这些也能够在微服务阶段都得到体现。


第6点，指导和培训。指导开发的同时，其实我们也是需要去培训，去培养一些新人，保持我们团队的整体综合水平的一个一致性，这样子你才能够更好的带领团队去做一些事情。

下一个沟通和协调能力架构，它不仅是一个技术能力担当的人，同时它其实也是一个管理者，是一个组织者。你是需要去和老板进行沟通的，同时也需要和下属去沟通。合格的架构师会综合考虑上下级的一些想法，才能够做出更好的决策。下一个抽象举例和画图。我们是需要去把技术高度抽象，因为在你平时的工作过程中，你肯定会和老板去进行交流，会去谈一些事情。作为老板，他是肯定不懂技术的，所以在这个时候你就需要去解释给领导听。所以我们就需要去抽象，多举例，多画图，把一些技术更好的，更简洁的形式去解释给领导听，让领导更好的去理解你的意图。当然我们也是需要去把自己的思想注入给我们的下属，同时我们也应该要让我们的下属来理解我们，来领会到我们的想法和意图。


最后一点就是软技能，软技能其实它有一点会偏向于项目管理。首先我们来说一下针对于我们的下属，其实作为每一个架构师，你都希望你的下属能够理解你，来服帖你，来信任你。这个的时候，在我们平时下了班以后，在业余的时候，其实我们可以去更好的组织一些活动，我们可以多去进行 team building，比方聚餐，KTV，去爬山。这些其实都是为我们以后的工作做好一个更好的铺垫。当我们团队的凝聚力高了，你的下属就会更好的去支持你的工作，这一点是非常有必要的。


其次，针对于我们的上属，就是一些领导以及是老板，其实你应该要去在合适的时候要去适当的说一些好话。可能有一些程序员会说，当你自己的一个本职工作做好了，并且你的技术也非常牛的时候，其实你是不需要去做类似一些拍马屁的事的，其实是不对的。只要我们在职场，不管你是不是程序员，其实拍马屁是少不了的。对领导多说好话，在某一些情况下，你的领导是会支持你的。因为在公司里面，架构师可能不是一个，可能会有多个，架构师，之间也是会有冲突，也是会有一些竞争的。所以这个时候如果你的领导愿意帮你，是最好不过的。OK。


还有一点就是谈判能力，谈判其实也是属于软技能的，我们在平时的日常工作中，我们不管是针对于下属还是领导，还是我们的一些第三方公司竞争对手，其实我们的谈判能力都是会体现的，主要是体现在我们的沟通上。如果你的下属要离职，这个时候你就需要去和你的下属去进行谈判，去进行沟通，你要留住他，这个其实就属于是谈判。如果你针对于你的领导，假设你的工作你需要你领导的支持，你需要去谈判。如果你要求给团队进行涨薪去加薪，其实你也应该要和你的领导去进行谈判，这些过程中你的沟通能力其实是少不了的。


其次，再遇到一些第三方公司，遇到一些竞争对手的时候，谈判肯定也是少不了的。另外，在我们平时工作的过程中，其实我们遇到最多的对手就是产品人员，产品人员会向我们开发人员提出很多的一些需求和功能，当然有一些需求其实是不合理的。所以在这个时候，如果你的谈判能力非常强的时候，就可以去把控一些相应的业务边界了。OK，以上这些内容其实就是我们本节课所讲到架构师所需要去掌握的一些相应的技术栈，以及我们架构师所需要具备的一些基本的能力。
