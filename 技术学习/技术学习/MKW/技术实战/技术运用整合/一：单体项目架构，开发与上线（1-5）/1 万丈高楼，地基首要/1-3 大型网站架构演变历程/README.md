这节课我们一起来讲一下大型网站架构的演进历程。首先我们来先看一下用户是如何去上网的。在很早以前，用户会通过浏览器，在浏览器的地址栏里面去输入一个相应的网址，随后去打开相应的网站。对于这样的一个网站来讲，往往都是一个静态页面，它是一个单页面的一种网站。在这个网站里面，它所包含的一些内容仅仅是一些html、 Javascript 以及是 css 样式这样的一种网站。其实他们都是在 5F 1. 0 里面最为常见的一种模式。所有的网站几乎都是静态网站，它没有和用户进行过一些相应的交互，他们都是单向的。

随后再进阶为 2. 0 的时候，我们就会成为这样的一种情况，用户通过浏览器去进行访问，访问的时候我们的服务器里面也包含了一些相应的内容，但是这个时候会有一个数据库，所以用户和服务器之间他们可以是一种双向的交互的。之前其实服务器和用户是干上的服务器，它仅仅只是提供了一些相应的内容，给用户去进行一个查阅。现在我们的用户和服务器之间有了一定的交互，用户可以去增加、删除，以及是修改一些相应的数据，这些数据都会保存到数据库里面。这个就是从最早期慢慢向后进行的一种演进的过程。

首先还是一样用户会访问我们的一个服务器，在这里我把浏览器先暂时的先拿掉了，我们可以直接来看这样的一个架构图。在服务器里面，我们会开发一个相应的项目，这个项目在打包完毕以后，它是一个外包，我们会把外包直接放到咱们的服务器里面去。在外包里面会包含相应的内容，比方在之前我们所提到的 html、 js 和 css 样式，这些都会有。其次，对于我们一些处理用户请求的一些内容，比方 model view 以及是 controller 也都会有。其实就是早期传统 Java web 开发所包含的一些相应的内容，它本质其实就是一个 server 层。或者我们又可以把它称之为是一种 MVC 的模式。 MVC 模式其实我们在后面其实也会用到像s， shh、 ssm 这样的一种架构。像这样的网站其实最开始是没有太多人的访问的，到了这个时候，我们其实是只需要有一台服务器就 OK 了。


在服务器里面，除此以外，我们还会有相应的文件服务器，以及是数据库。文件服务器是会为用户提供一个文件存储的地方，像用户的一些头像，用户上传的一些文件等等，我们都会放在文件服务器里面。只不过文件服务器和我们当前应用服务器是在一起的。此外，我们还有一个数据库，所有的用户数据也都会保存在数据库里面。这个其实就是一种单体的架构模式。单体的架构模式其实也是早期传统的 Java web 的这种开发模式。随着网站业务的发展，一台服务器它是不能够满足所有用户的访问需求的。这个时候流量可能会越来越多，越来越多会导致我们服务器的性能也会越来越差。最常见的问题其实可能就会导致我们的一个空间不足，不管是图片和数据库都会乏力。所以这个时候我们单体一旦宕机了，我们所有的文件和数据库的数据都将无法访问。所以这个时候我们可以这样子。


用户访问我们的一个服务器的时候，我们服务器里面仅仅只包含了我们的一个网站，网站数据都会在这里面。随后用户所上传的一些头像或者其他的文件数据会有一个额外的文件服饰，这个时候其实这两台服务器或者两个节点，他们已经是分离了。另外我们的数据库也会独立的部署到另外一个单独的服务器里面，现在我们就把它们完全的分离了出来。


分离的好处就是对我们用户的一些请求进行了一定的降压，这个就是最初的一种分离的模式。这样的好处就是我们当我们的服务器挂掉以后，比方我们的网站挂了，但是我们的文件服务器以及是数据库还是可以去工作，还是可以被我们去进行访问的。像这种情况下，我们每一个节点都是需要去分开部署，独立出来。


网站和数据库，还有图片服务器，所有的内容全部进行分离，不同的服务器充当不同的角色，提供各自不同的服务。并且网站的并发能力以及是数据库存储的空间都有了一定的提高，这个时候企业的业务发展就不会被这种单体的架构熬束服了。

这个时候随着一个时间的推移，用户的请求量或者我们的用户会成倍的增加，企业网站架构又会遭遇到另外的一个问题，数据库的压力会随着这个问题导致我们的一个用户访问请求的延迟。这个问题其实就是用户的查询一旦会很多，其实它的所有的请求量都会直接落到我们的一个数据库上。在这个时候我们会引入一个缓存中间件，它其实相当于是一种防护机制，用户要去进行一些数据查询的时候，它是不会直接接触到数据库的。它首先会落在缓存的中间件里面，它会去缓存里面查询一下相关的数据，如果有数据就直接返回了，如果没有数据，它才会去访问我们的一个数据库。如果我们一开始用户的请求变慢，用户体验变差，这样子肯定会导致一定用户的流失。所以我们必须要做出这样的一种调整，来引入这样的一个缓存中间件。这样子我们也说了，用户绝大多数的查询都会落在缓存库里面，而不是直接接触到我们的数据库，为我们数据库做好了一个不错的防范措施。这样子保证我们用户的体验可以有一定的提高，请求响应的速度也会加快。

我们暂时还仅仅只是单库，我们还不会去做一些相应的咨询。虽然我们在前面其实已经是使用到了缓存，但是其实我们还是会有一部分的数据，它的读操作会直接落在数据库上，当然同时还会有所有的一些写操作，也是会直接访问到数据库的。所以在网站用户达到百万甚至千万级别的以后，我们数据库的负载能力其实就会成为我们网站架构的一个瓶颈。


如何来解决这样的问题？大约有 70%- 80% 的一个请求其实都是读请求，写操作其实是有 20%- 30%。其实这也是一个二八原则。这个时候数据库又有读又有写，它的压力是非常巨大的。所以我们可以通过数据库的读写分离，设置两台数据库可以来看一下。


一台我们可以设置为是一个主库，它的数据是可以写入到主库里面去的。同时我们还会有一个从库，这个从库我们是从这里面我们还是会有一个缓存，会有一部分的数据会从我们的从库里面去进行一个读取，这个其实就是一个主从分离的架构，这样的两波不同的流量，一个是读，一个是解，就被我们所分开了，从而可以改善我们数据库的一个负载能力。


当然读写分离还是需要去注意我们数据的一个同步的，也就是我们主库会定时的向从库去同步数据。虽然我们做了毒血分离，有了两个库，但是对于用户来讲，我们对于用户是透明的，这些东西只有我们自己才知道，用户只需要知道他访问的时候它比较快就可以了。


好到这里，虽然我们已经是做了数据库的读写分离，但是大型网站的一个业务，它还是会急剧的增长的，这个时候怎么办？如果我们的数据库扛不住了，我们就需要去针对我们的数据库做分库和分表。

其实可以是非常非常复杂的，俗话合久必分，这个是永恒不变的一个道理。所以当我们的一个业务处于非常非常复杂的时候，这个时候我们就需要去进行拆分了。把我们一个非常大的业务拆分成不同的小的产品线，不同的业务会独立成一个的子项目。我们来看一下，比方我们是在做一个电商，我们是可以把商品相关的内容可以给直接拆分出来，拆分成一个独立的子系统，这个独立的子系统专门是用于提供和商品相关的服务的。与此同时，我们也可以把订单拆出来了，以后它是可以专门为订单相关的内容去做一个相应的服务。当然还会有很多，比方我们可能还会有一些用户服务，库存服务等等。在这里我们就举例商品和订单了。既然拆了一些相应的服务出来以后，一定要注意，一旦去进行了拆分，即使我们在这个时候就处于了一个微服务的阶段，我们的数据库也是必须要跟着我们的业务去拆的。


相应的我们和商品表相关的一些商品数据，比方我们可能会有一些商品表，商品规格表，商品的图片表等等的内容，把这些表单独的拆出来，作为一个独立的商品数据库，相应的订单也是一样。并且我们可以参照我们之前所做的，针对这些商品数据库以及是单数据库做一个主，从分离对他们进行一些分库分表，这些操作也都可以去做，也都是没有问题的。这样其实我们每个系统都可以把他们去交给不同的团队去进行维护了。


当然这样子对于开发、测试、运维来讲也都是一个比较大的挑战，因为我们的业务变得复杂了，这样其实不同的个体整合在一起就是一个庞大的大型系统。当然它也会有一定的优缺点。优点无疑，我们的负载会降低，我们的业务分离，开发人员，开发团队就可以自己负责自己的一些模块了。它的缺点就是它的代码会比较复杂，运维当然也会变得更加的繁琐和复杂。


还有一个是需要去必须考虑的问题，就是分布式事务。因为我们的一个服务拆出来了用户的一个业务，用户的请求他可能会同时到达多个子系统。所以分布式事务也必须是我们要去考虑的一个问题。

在一个系统里面，我们往往会有一些非常通用的接口。我们来看一下。用户其实是可以在访问我们的一个每个子系统的时候，去调用一些相应的公用的服务。比方我们有一些短信、邮件推送，把消息推送到APP，这些我们可以把它们称之为是一种公共的服务资源。对于这些公共的服务资源，其实我们在服务和服务之间进行相互通明的时候，我们都会使用到一些相应的分布式的服务中间线。比方像zookeeper，它就可以去处理一些分布式锁。


当然，在我们分布式系统里面，我们还会有一个分布式会话。与此同时，在我们系统之间调用的时候，我们往往会使用一种异步通信，或者是异步调用的一种方式。在这里我们又会使用异步队列，也就是 MQ 消息队列来处理这一切的调用请求。另外像一些分布式锁，分布式事务，分布式会话在这边我们都列出来。这些都在我们系统里面都是必须要去考虑的一些问题。其实当我们的一个网站架构演变到这里的时候，其实我们可以解决绝大多数的一些技术问题。当然其实我们对于我们的网站，对于我们的架构还是要去进行一些相应的优化的，比方像 jvm 、Tomcat 数据库等等，我们都需要去根据不同的情况，比方发生一些问题的，我们都要去根据相应的日志分析，去做一定的相应的调优。所以对于架构师来讲，调优能力也是应该要有具备的。


其实在整个架构演变的过程中，我们会根据企业当前的一个业务去进行演变的。前面我们所提到的一些演变的顺序，其实都是可以自由调整的，并不是我们必须在某一个时候，必须在某一个节点使用一个特定的技术，这是没有必要的，我们的课程其实也是会与绝大多数的一个公司常见的顺序来进行一个演进，大家在学习的过程中也是需要去知道，我们是需要灵活的去掌握就 OK 了。以上就是我们在这一节所讲的一个架构的演变过程。
