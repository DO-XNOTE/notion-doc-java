其次就是我们的一个订单，订单对于我们来说是比较核心的，这里面所有的一些字段，所有表的一些设计，我们在这一节我们暂时先不去说，因为我们在后续。我们在讲到要做某一个功能模块的时候，我们配合着我们的业务和我们自己的需求，再详细的去讲一下我们的数据库表。如果现在我们在前期把所有的一些表的字段去跟大家说了以后，可能你会忘记，所以我们一开始先不说，等到用到了以后再去说。


在这里我们先缩放，缩，放了以后大家可以看到，其实我们从这一个关系图里面看得出来，每一张表相当于都是一个独立的个体，不存在有一些相应的物理的外界关联。我们来看一下咱们的一个官网的图。这是一个官网的图，在这个官网的图里面其实可以看得出来，它每张表和表之间其实都有一些相应的表关联。可以看到不管是一对多，多对一还是一对一，相应的一些表的关联，其实都可以从这个图里面通过一条一条的线可以看得出来，但是我们在设置的时候并没有这样的一个物理关系，这是为什么？这个是我们这一节所要来聊一聊的。


首先一个性能，如果我们使用了数据库外键，其实对于我们整个项目来讲，它是会有一定的性能影响的。一开始其实在最早我们接触数据库的时候，会有一个数据库 3 范式，其中就有一点要强调，我们要保证咱们的数据的一个完整性，也就是数据库在表和表之间一旦有关系，我们要使用一个外界，并且外界是要有一个物理外界的关系的。也就是刚刚我们在图里面看到的一条线。

外径的好处。一个是数据库的完整性，另外一个其实它就是强调了数据的强力执行。但是在我们的一个分布式项目里面，大型互联网项目里面，在我们去进行一些插入修改以及是数据删除的一些操作的时候，对于我们整体的数据库性能会有一定的影响的。因为我们的数据库它数据的完整性去做一些相应的检查操作，所以只要是在我们有一些大批发的时候，数据库的性能是一定会受到相应的影响的。所以数据库外界物理外界的关系，我们是尽量的要移除。


另外一个第二点，热更新。就是在分布式环境下，其实我们的项目可以绝大部分都是要进行热更新的。什么叫**热更新？我们不停机去进行一个维护，不管是我们的一些互联网产品，还是像一些游戏、手游，大多都会采用热更新的一种方式。如果我们的数据库有外界，可能会导致我们新更新上去的一些代码会无法运行，因为要去匹配到现有的一些外键，所以你可能要去重启一下服饰，这样子就达不到热更新的目的了。因为外界它是强一致性的，这一点是和分布式绝对是冲突的。**


第三一个是降低耦合度，删除了外界，其实并不是我们真正的不要外界的，而是物理外界物理的一层关系，我们不需要了。是这个意思。但是逻辑上这样的一个外径还是存在的，这样子其实也是适当的降低了表和表之间的一个耦合度。什么叫物理和逻辑？逻辑其实就是比方我们现在有一个用户表，还有一个用户地址。照理用户地址其实是有一个外界要关联到我们的一个用户表，外界就是 user ID。但是 user ID 这一个字段这一列我们是不会去删除的，还是会有，只不过它的物理外界我们不需要再去设置了，是这个意思。


最后一个是数据库的分库分表，其实在大型系统里面，我们都是有多个数据库的，我们在之前的架构演变里面也说了，在最后可能会有很多数据库是一个分布式的数据库集群。如果我们有外间关联，会导致我们难以做分库分表，因为耦合度实在是太高了。假设现在是有两张不同的表，一个是用户表，一个是商品表，用户表跟商品表，假设他们是有一个外进关联，当我们现在要做一个分库了，我们要把两张表放在不同的两个数据库里面，但是他们却有一个外界关联关系，这个时候你就很难去实现了。这仅仅只是一个外镜而已。


如果当我们的数据库我们有上百张表，有几十个外镜关联，这个时候分布分表该怎么做？会难以去进行，会难以去实现，所以这一点一定要去注意的。如果一个互联网的项目在前期没有做到很好的一个控制，控制我们的外界关系，当我们的业务上来了以后，物理的外界一定会让大家，尤其是让运维人员会欲歼欲死，这一点一定是要去注意的。所以很多的架构师和DBA，还有是运维工程师都是会弃用数据库外建的，在最一开始进行项目开发的时候，也是会最新的提出来的，这一点大家是需要去明确的。简单的小结一下，当你在设计一个小型项目或者是一个传统的项目的时候，数据库的物理外镜关系是可以存在，也可以不存在，这一点无所谓。如果在你设计一个互联网项目或者是一个大型项目的时候，我们是绝对不能用数据库物理外键的，我们一定要为以后的架构多多考虑。OK。
