对于 SpringMVC 来讲，它其实是一个框架，它是在 spring 里面的，它是其中的一个子集，或者我们也可以称之为它是一个 spring 的 web 模块。


对于 SpringBoot 来讲，其实很多人都称 spring boot，它是一个框架。其实我们从严格意义上来讲，它是一个配置工具，整合工具。它里面包含了很多的东西，就像是机器猫的口袋，你想要什么东西都可以从里面去拿。我们是通过 Pom 去进行依赖的，它其实主要实现了自动配置。我们在使用我们 spring MVC 的时候，其实在 spring MVC 里面我们会有很多的配置项，绝大多数我们都是通过 xml 去进行配置的。使用 spring boot 以后，其实我们会强调一个叫做 0 配置，我们是没有叉妙这样的配置的。我们会把所有的配置精简到 yml 文件里面，也就是雅某文件。我们的一个目的是零配置。


对于 spring MVC 来讲，一旦我们的项目经过周期很长以后一直在迭代，其实我们的项目会越来越复杂，我们的配置也会越来越多。其实我们的配置项很多，对于新人来讲，或者是新入职的人员来讲，会眼花缭乱，你可能会无从下手如何去做一个模块。当我们去使用了 SpringBoot 以后，其实 SpringBoot 可以适当的提高开发效率和生产力。 SpringBoot 它本身就是开箱即用的，非常的方便。像数据源、数据、Redis、mq 等等中间件我们都是可以通过 starter 来进行依赖和引入的，集成起来也是非常的方便，使得我们开发人员开箱即用 starter，其实我们又可以俗称为是懒癌程序员的拯救者，非常的方便，一定程度上简化了每轮的配置。


此外，在我们使用 spring boot 的时候，我们的 template 其实它本身也是集成到了 spring boot 里面去，我们直接可以使用内置的就可以去进行启动了。通过 YAML 文件，我们可以在里面去配置一些template，里面的一些端口号，配置项等等都是可以去做的。如果我们使用SpringMVC，我们的 Tomcat 无疑是需要去额外下载。所以要是需要去依赖外置的 Tomcat 才可以去运行我们的项目的，这一点我们是需要去进行区别的。

struts 它也是一个 MVC 框架，对于一些老手来讲，你肯定会知道。 struts 对于新车来讲可能你并不了解，因为 struts 它其实在 10 年前是非常火，非常流行的。几乎所有的一些项目采用的 NVC 框架都会使用struts。 struts 它有 1 也有2，到现在为止，其实几乎没有人在用了，因为它本身安全性就不太好，但是对于一些非常老的项目，像一些金融类的保险项目，可能还是会继续使用。


首先其实 structs 它本身有一些安全漏洞，在之前发现了安全漏洞以后，官方又一次对 struts 进行了维护，进行了版本的提交。在今年的 4 月份，我们自己公司有一个项目，这个项目其实是在 2009 年就已经是开发了，那个时候使用的是struts，也就是 SSH。因为 struts 的漏洞，导致了我们的一部分代码被恶意的篡改，并且我们的服务市场有一部分黑客所上传的可执行的代码是非常非常危险的。所以我们当时就跟我们的甲方客户商量了这个事情，直接把这样的一个项目给停掉了。


随后我们就使用了 spring boot 来进行重新的开发。我们把所有的功能全部都进行迭代了，只不过我们由原先的网页版变成了一个小程序。这样子我们就不再会因为 struts 会造成一些相应的损失了。所以对于我们技术选型来讲，安全性是必须要考虑的一个问题。


随后第二点，如果我们使用struts，一旦我们的用户量增多，请求变多，对于我们的 action，也就是请求到 action 去处理的时候，相应的一些变量。它有可能会出现共享的情况。这个问题实际是由于它的 action 线程安全的一个问题所造成的，所以这一点是需要去注意的，当然也可以通过一些手段去进行处理。所以几乎我们到现在位置，我们肯定新项目是不会再去使用struts。


第三个是它的一个入口类，像我们在使用 spring 的时候， spring MVC 和 SpringBoot，其实他们本身他们的一个拦截机制是使用的。 AOP 是基于切面的一个拦截，是通过方法去拦截的。 struts 是通过 filter。 Filter 其实是过滤器，过滤器它是类拦截，它们的两种其实是不一样的。


随后之前我们也提到了 ssh, struts 是和 spring 可以去结合的。但是其实 struts 它本身它并不是 spring 官方的，它不是自产自销的。所以当我们使用 Strux 和 spring 去进行结合，可能会略显有一点重。当然如果我们使用 spring MVC， SpringBoot，本身两者都是 spring 官方的亲儿子，整合起来是非常轻松的，毕竟是自家的产品。


这些其实我们当前所聊的关于 struck 的一些小历史，大家可以去了解一下。随后下一个我们再来看一下我们项目的一个前端技术选型。


其实我们虽然是做后端开发的，但是对于前端其实我也是希望能够去了解一点的。虽然我们不要去接触，不要去编写额外多的前端代码，但是如果你想要职位上有更好的发展，你作为架构师或者以后作为技术总监，你对于前端技术你一定要有一定的了解。

首先来看一下前端使用的开发模式是 mvvm，它是在前端领域里面的一种现有非常流行的开发模式，不管你是使用像 feel 还是react，甚至是小程序，它们的一个开发模式其实都是mvvm。它们是通过数据的双向绑定去渲染我们的一些页面的。它也是去 DOM 的，它是不会再去像以前那样去根据我们的 DOM 去操作我们节点。



随后第 3 点和第 4 点无疑就是我们的页面和样式，这个主要就是HTML， CSS 样式，这两个是比较基本。其实对于我们不管是 Java 开发人员，还你是一个架构师，基本的一些 HTML 基本的一些 CSS 样式，其实你都应该要去写。其实如果你能够作为是一个全栈开发人员，全栈架构师那就更好了。

首先，第一个切合业务。我们能否实现现有的功能，能否预估到后续的功能，和现有技术的一个贴合度能否完全的匹配，这些我们都是应该要作为一个架构师所能够考虑和预估到的，必须要切合我们的业务。如果这个技术现在可以完全的贴合，但是在后续所预估到的一些功能模块，一些业务可能无法满足的时候，这个时候我们就需要去仔细的去斟酌了。


第二一个是收集活跃度。其实当有一个系数出来的时候，不管是在国外还是在国内，其实它都会有相应的社区。在我们这里，一些相应的中文社区，如果它的活跃度非常高，就代表使用率也是比较高的，相应的它所体现出来的一些问题，优点、缺点其实都会有。像这样的一种技术，我们是可以考虑去使用的。如果社区活跃度是非常低下，对于这样的一种技术，就代表可能并没有多少人去关注它。


随后一个是我们整体团队的技术水平，所以在我们进行一个技术选择的时候，如果我们团队经验的一个不够，我们是需要去对他们进行培训的。或者我们在开发周期短结合等等综合因素的时候，我们是需要去斟酌我们到底要不要去使用一些团队比较熟悉的技术，还是我们团队比较陌生的技术。这一点作为领导者，你是需要去斟酌的。如果你使用了一个技术比较新，或者整体团队可能都没有去接触过，你也是要去适当的对大家去进行培训，提高整体的技术水平，业务水平，都是要去跟着提升的。


下一个是版本更新迭代的周期。如果一个开源的技术，它本身它迭代是非常快，它的一个周期迭代起来也是比较短。对于这种技术，我们也是可以考虑去选择的。举个例子，在曾经有一个框架Dubbo，大家可能都知道，在之前是版本停更了一段时间，在这段时间里面，其实有一部分公司在使用 Dubbo 的时候，只出现过一些相应的问题，但是得不到解决。所以当一个开源之处停更了以后，你一定要去斟酌后续要不要去使用它。


虽然 Dubbo 在现如今已经是恢复了更新，其实毕竟在中间有一段时间的停更，对很多公司的业务肯定是多多少少会造成一些相应的影响。当然我们国内有一个非常强大的公司叫当当，当当是针对 Dubbo 又做了一个二次开发，所以又会有另外一个版本叫做 Dubbox，这也是另外一个版本的 Dubbo。有兴趣的同学也是可以到我们的幕后网上去学习一下 Dubbo 相关的一些技术。

第五个是试错精神。什么叫试错精神？在这里并不是在我们的实际项目开发中去试错，其实主要还是要我们在平时接触各类技术的时候，我们要去多试多体验。不论对是错，其实对于我们来讲它都是一种经验，我们要去善于总结，多去试验，多去验证，这样子我们就可以会有很多的一些备选方案，备选的技术了。


下一个是安全性。之前我们在讲到 Struts 的时候，最大的一个问题就是一个安全漏洞，如果某些技术一旦有了安全性，或者曾经出现过安全漏洞，你就一定要去仔细的斟酌后续你要不要去使用它了。下一个是成功案例。其实我们在使用某些开源技术，某些框架的时候，那么多多少少其实在我们的行业内，如果口碑都不错的，有成功案例的，大家如果都去推进，我们都是可以去使用的。不管是小公司还是知名公司都在用，我们都是可以采纳去使用。最后就是一个太原精神了，现在其实我们在做开发的时候，我们所使用到的技术其实都是一些开源技术，在某些情况下，其实我们使用到的技术有可能会闭源。当我们再去做技术选型的时候，我们也应该要考虑到，如果这个技术在未来它有可能会闭源的时候，其实我们就需要去仔细斟酌了。虽然我们在前期，其实一般来说我们都不会知道它有没有可能会被闭眼，但是如果一旦你在某个时期听到有这样的风声了，你就要仔细的去斟酌了。


像我们现在自己公司里面的前端某一个框架，其实已经开始收费了，它也闭源了，所以每年你是需要去缴纳一定的费用，也是好几千，一次性买断也要接近上万。当然对于我们前端团队及我们老板来讲，他们是不买账的，虽然对于公司来讲，其实这并不是一笔大钱，但是毕竟我们是秉持要有开源精神的。好。以上这些就是我们本节课所谈到的和技术选型相关的一些内容。
