在上一阶段，咱们是通过学习了nginx，能够通过 nginx 去部署集群和负载均衡，并且也是配合了 LVS 和 keepalived 的，实现了高可用、高性能的集群负载均衡。那么这也是在很多企业里面非常流行的一种负载均衡的解决方案。当一个架构进阶为集群以后，其实对于项目它本身来讲的话，它还是一个个体，是单个应用，是单独的部署在 Tomcat 里面的，它是一个 webapps 。那么如果一个请求它经过的业务比较长，比较的耗时，这个时候怎么办？我们就应该要使用分额制止的手段来拆解系统，引入分布式技术，或者把这个系统拆分为子系统，形成一个分布式系统，这样综合的一个分布式的运算能力要远远的超过单个计算机节点，它所执行的运算能力，那么当然处理用户的请求的话也会更加的得心应手。

首先我们先来看一下什么是分布式架构，对于分布式架构的话，那么其实它每一个单独的一个子系统都是从我们的整个系统可以说是去拆出来的。不同的业务，不同的功能模块的话，是分散的部署在不同的服务器里面去的，那么这样子的话其实是为单个服务进行了一个降压，提升了整体的一个运算能力，那么同时不同的子系统去处理不同的业务，这样子的话就相当于是一个专人做专事。每个子系统它会负责一个或者多个不同的业务模块。那么对于我们的每一个子系统来讲的话，服务之间是可以相互的。交互和通信的方式的话，那么他们是可以使用，像RPC， web service 或者说是 HTTP 通信，其实都是可以的。

当我们的业务量越来越大，非常庞大的时候，我们是需要去应对这些业务量的。每一个单独的计算机的性能，它肯定是没有办法去满足我们当前的一个庞大的业务量，所以我们就需要有多台计算机，有多台服务器，才能够应对大规模的这种使用场景。所以我们就需要去针对我们的业务系统，让它变成一个分布式的一个架构。


所以我们要做到一个业务的解耦，比方说我们可以把像搜索用户上传的那一块，都可以独立出来拆分为一个单独的子系统，这样子的话我们就只需要创建一个新的子系统子项目就可以了。并且我们不同系统的 control 了，都可以依赖同一个service，这样子的话我们的 service 也是可以重用的，它的重用度就是非常的高了。这个其实就是系统的模块化，高度的可重用化，那么我们就会有多个子系统。那么咱们的并发量的话，在使用分布式架构以后，它也是有所提升的。原先单个的并发量肯定是有限的，拆分成分布式系统了以后，用户的流量值是会被打散的，打散了以后整体的并发量就上去了。


OK，下一个是和我们的运维部署相关的，我们在一开始使用单体的时候，就是我们整个包是独立的去打包部署的，它是体积比较小，当我们的业务量上来了以后，那么这单个包其实会非常非常的庞大。所以当我们拆分成分布式了以后，迭代和发布就会变得非常的轻巧。测试团队的话，他也只需要去测试对一些新发布的内容，而不是一个大的项目包，那么这样子的话我们整个整体的团队的协作能力就更高了。

那么首先第一个是一个异步解耦，所以说我们的分布式是把耦合的模块进行了一个解耦，所以相互进行通信的时候，如果说能够使用异步，那就使用异步，那么实在不行的话，那么就使用同步，因为异步的效率远远的要比同步来的更好，除非你不得已才去使用同步。异步的话会需要使用到消息队列，这个的话会在后面的课程里面大家也会接触到的，也就是MQ，那么下一个是一个幂等性。


那么在使用分布式的时候，那么用户的一个请求可能会经过多个子系统，这个时候就需要去保证用户不管是查询还是增删改，那么数据的一致性是需要去得到保证的。那么比如说在增加的时候，不能够因为用户提交了两次或者三次，而导致我们的数据库多了两条三条一模一样的数据，那么这样子肯定是不行的，所以我们是需要保证用户对于同一个操作发起的一次请求或者多次请求，他们的最终的结果是一致的，不会因为多次的点击而产生了副作用。这一点的话，尤其是对于增加和修改，是要去做一个幂等性的操作的。因为像查询那么查询的话，因为用户的话一直可以去查，不管是发起了一次还是多次侵权，查询的结果是没有什么改变的。另外对于用户的一个删除，那么删除的话，那么它只会删除一次，因为这个数据删掉了以后就没有了，你多次的去删除的话，那么你数据库也删不了。所以立等的话，主要是针对于增加和修改来讲的，那么下一个的话就是它的一个拆分原则。


系统拆分的原则其实有多种，比方说可以根据业务去拆，那么我们可以分为像门户子系统、用户中心子系统、搜索子系统、文件处理子系统。根据系统功能去拆的话，又比如我们可以把文件系统进行一个拆分，我们可以拆分成一个文件上传，或者说是文件下载这两个子系统，那么这些都是拆分的一个标准。
