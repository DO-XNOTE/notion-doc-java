那么接下来我们就应该要来聊一下 Redis 的一个线程模型，在讲之前的话，我们就应该要先来聊一聊什么是阻塞和非阻塞，什么是多路复用性。那么这个的话阻塞和非阻塞同步异步，其实他们的一个概念和举例，我们之前在 Nginx 的部分是写过一个图文。那么在这里的话我们再来聊一下阻塞和非阻塞的一个概念。


那么我们先来看一下。首先是有一个服饰用户要和咱们的服务器要去进行一个通信，要发起请求，那么这个时候请求发到服务器了，然后服务器也会去做一些相应的处理。然后这个时候又有第二个或者第三第四个客户想要和我们的服务器通信，建立连接，这个时候的话，由于我们第一个客户，他可能正在处理相应的一些内容，还没有处理完毕，服务器还没有把一些相应的内容响应给我们的一个客户，所以我们后面的所有的一个顾客，所有的客户端都是处于一个排列的状态，都要等待。

所以他们其实都是会被我们第一个顾客给阻塞住的，他要直到我们第一个客户要处理完毕这样的一个请求，把响应返回给我们的客户以后，那么第二个第三个客户，他才能够去和我们的服务去进行交互，直到他们处理完毕。也就是他们之间相当于是一种一对一个一个去处理的一种模式，那么这个的话是属于一种阻塞的一种通信模式，阻塞的话其实它就相当于是一种一对一个去进行处理的，那么它的一个效率其实是非常低下的，如果说我们这个时候用户量比较少，那么其实我们的服饰还能够处理的下来，如果说对于我们的一个用户量很多，并发量很大的话，那么我们的服饰它的开销肯定是吃不消的。当然也可以把我们这样的一个服务器设置为以一个多线程的方式去处理用户的话，那么这样子也是可以的，一个线程去处理一个用户，那么肯定没有问题。


那么如果说我们有上万上百万的一个客户端要进行链接的话，那么我们在服务器里面，那么你就要开上很多很多的线程去处理，这对于我们的服务器的一个开销来讲的话是非常的大的，而且对于服务器的一个成本也是非常的大，所以在后续的一个发展过程中，这一块内容的话，它是可以转变为一种叫做尾翼步的线程模型。


那么其实就是由多个线程组成的一个线程池，针对于我们的客户端去进行的一个处理，但是它们的一个本质上的话，它还是属于一种阻塞的一种模式去处理的。OK，所以我们再来看一下非阻塞。那么对于非阻塞来讲的话，那么假设这个时候还是一样有一个客户，第一个客户和咱们的服务器建立连接，要去通信，要去请求，然后又会有第二个第三个客户要和咱们的这个扶持要去建立连接，去请求。那么这个时候如果说我们第一个客户他没有处理完毕请求，他其实还是处于一种阻塞的状态的话，应该说是他还在等待我们的响应的时候，那么其实他是不会被阻塞的。


那么第二个用户，第三个用户，这个时候他是可以直接把请求发送给我们的服务器，我们的服务器发现，如果说我们当前第一个客户没有处理完毕以后，那么他其实会轮询的去往下面一个一个去找，发现有新的一个客户端进来，那么他就会去做一个相应的处理。其实这种模式的话，就是一种多路复用的一种模式，它是一个同步的非阻塞的模式，那么它的处理效率是非常非常的高的。当然如果说在我们这里第一个第二个客户一个一个的请求处理完毕以后，那么我们后面的话，它其实会轮着去进行的一个相应的响应的，所以说对于这种非主策的一种模型来讲的话，其实它的一个用户的一个处理，它的一个性能是非常的高的，而且它能够支持的一个并发也是非常的高。


那么当然对于我们之前来讲的话，之前其实是阻塞的，那么现在我们是一种非阻塞的，而且非阻塞的话，它对于我们的一个整体服务器的一个开销来讲的话是比较低的，所以它的成本也不会那么高，相应的而且它的一个支持的并发处理的效率要比我们的阻塞更好更高。所以在这种模式下， IO 的一个多路复用就是在我们的 Redis 线程模型里面，它也是使用到这样的一种模式的。


OK，那么对于我们的一个 l 多路复用式的话，那么你是可以把它理解成是一种环状型的一个东西就可以了，这是一个阻塞的，这是一个非阻塞的。

在这里我们把这个多路复用器的话，可以比作是在一个酒店门口的一个接待员，可以这样子去做一个比喻，那么它就是一个多路复用器。那么这个时候会有很多的客户，很多的顾客要来和我们的这个接待员去做一个连接，也就是要去做一些询问，因为他们要去住酒店，那么肯定我们的酒店会有一些不同的价格，有不同的一些房间档次，还有是一些优惠活动等等的都是有不一样的。那么这个时候我们第一个用户要肯定要去询问，询问的时候我们的这个接待员告诉他我们这边的一些活动等等的信息，然后这个时候这个顾客他可能比较犹豫，他会去思考我到底要去住哪一间房间，选择什么样的价格，他可能会有一段时间的考虑的时间，那么对于我们的这个酒店的鸡蛋园来讲的话，他不可能会等你，我不可能等着你再回复我，那么你可能会思考个 10 分钟甚至半个小时。他如果说是等着的话，那么对于我们的一个工作效率来讲的话，肯定是非常的低下的，所以这个接待委员这个时候就说，没关系，你自己去思考一下可以，然后我就去接待我下一个顾客了。


那么下一个顾客那么这个时候他可能他的一个决断性比较的高，他直接会说我就要住这一间，然后这个多路复用是我们的接待员直接就可以把它去丢给后面去处理了。那么当然我们的第三个第四个也是同样的道理，他们会向我们的这个接待员去询问，然后询问了以后，接待员会把相应的活动告诉给他们。


其其实对于我们的接档员来讲的话，其实在这边不管是有多少个顾客，不管是有多少个用户来和他进行交互的话，只要他发生了一个思考，那么他就不会等着他，他会去处理下一个顾客的一个请求。如果说有用户确定要去住酒店了，那么他是不会去负责后续的一些接入的服务的，那么他直接会把这些顾客丢给我们的酒店，让酒店的前台人员去做一个相应的处理，这样子我们的这个多路复用器，它的一个性能就很高了，他只负责接待，他不去负责我们用户的一个真实的处理。所有的处理直接交由我们的酒店，让酒店里面的人去做一个接待。OK，那么它其实就是一个多路复用系，它的一个性能是非常的高的，那么这个其实就是咱们的一个模型，大家可以通过这个图去做一个相应的理解，因为在我们后面一节的一个 Redis 宣城模型里面，也是会涉及到多路复用的。
