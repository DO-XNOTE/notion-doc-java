首先我们先来看一个，在这里会有一个 global defense，在这个里面其实这一段内容的话就是一个全局的配置。那么首先你会看到在这里会有一个 notification email，这个其实就是一个通知的 email，当我们的一个主节点 master 发生故障了以后，那么其实我们的一个虚拟 ip Vip 会进行一个切换，会切换一个绑定，那么它的一个绑定的话，这个时候它会和我们的一个备用机绑定到一起。一旦发生了切换，只要你在这里配置了email，那么它就会向这个 email 里面发送相应的邮件去通知你的管理员。但是你要去做一个配置的话，必须要有自己的一个邮箱，那么你还要去配置一些相应的邮箱的协议，那么在这里的话，其实我们用不到我们就删掉，那么在这边我们只保留一个配置，那么这个配置就叫做 router_id


这个是什么意思？那么它就是一个路由ID，也就是我们当前安装 keepalived 的它的当前这个节点的一个标识符，而且你要保证它是一个全局唯一的。在这里边我们可以去加一下注释，这是一个路由 ID 当前安装 keepalived 的节点主机的标识符，它是全局唯一的，你就可以把它理解成是一个组件也是没有问题的。然后在这个下方还会有一些相应的配置，这个我们都用不到，我们把这些全部都删掉。然后针对于我们的 root ID，我们做一个修改，我们把它改为keep。然后再来一个下划线171，那么 171 就代表我是当前的幺九二点幺六，八点幺七幺这个内网的ip。把这个当做是我们的一个当前的一个全局的唯一的节点标识符。最后下一个，那么下面其实我们就可以看到它会有一个，我们把这个往下面移一些，在这里有这么一段，那么这一段的话它是一个 vrrp instance，那么它其实就是基于 vrrp 协议的一个实例，那么这个实例的话你可以认为它其实就是一个对象，它就是一个扶持节点。然后在这里它会有一个名称，它定义为这个 vi 下划线 e，这个我们可以不用管，保持默认就可以了。


我们在这边在这里面的话，在这个上方加一个注释，这是一个计算机节点。那么首先我们当前的这台172，这一台 linux 其实我们是把它作为是一个 master 节点，所以它的一个状态位在这边，我们就是定义为master，我们写一下故事，这是表示的状态。当前的 171 为  nginx 的主节点，如果说是主节点的话，那么它就是master。如果说是备用机的话，那么它其实就是一个叫做 backup，它是有这两种区分的。

so，下一个。那么下一个是一个 vertical root ID，从字面意思上我们就能够看出来，它是一个虚拟的路由的ID，默认的配置是51，那么在这边其实我们呢，保证我们的每一台的一个主备节点在这里配置是一致就可以了，就相当于是在同一个在相同的路由器组里面就可以了。保证主备节点一致都采用51。


最后下一个是一个priority，那么它是一个优先级的关系，其实它就是一个权重，优先级。权重。那么这个意思的话是指如果说我们有多个节点，一旦我们的 master 挂了以后，那么其实它就会进行一个选举，谁的优先级高，谁就能够成为master，这个我们其实之前也已经是提过了，在这边写一下。谁的优先级高在 master 挂掉以后就能成为 master。
