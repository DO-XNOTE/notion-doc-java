举个例子，在我们点外卖的时候，其实会有一个下单，然后会有一个外卖小哥在接到订单去取餐的这样的一个流程，那么在这个流程的过程中，其实我们的外卖小哥他其实取到了很多的一些餐品之后，会把这些餐品放到一个外卖的盒子里面，然后再一个送过来。那么这个其实就相当于是 TCP no push 这样的一个模式，他会把很多的吃的东西先去取到了以后，再一个一个的去分发，那么这样子他在进行一个运输的过程中，他的工作效率就提升了。那么如果说你不使用这种模式的话，这个外卖小哥他会一个一个去取，比方说我现在我下单了，那么这个外卖小哥会到店里去拿一个吃的，拿了这个吃的以后再送给我，随后再去接下一个订单，然后再去取餐，再去送货。那么这样子的话，对于他自己本身来讲的话，它其实就相当于是一个同步阻塞的一种取材方式，它的效率是极其低下的。所以这个 TCP 的 push 打开之后，就可以起到一个高效传输的一个目的了。

这个又是什么意思？那么这个其实是指我们客户端连接服务器的一个超时的时间，如果说你不使用，默认的话是一个0，在这里的话，它其实是为我们打开了，并且设定的是一个 6565 的话，它的单位是秒，是以秒为单位的。


那么其实我们的 HTTP 的话，它本身是一种无状态的协议。客户端向服务器去发送一些相应的请求的时候，那么我们服务端在响应完毕之后是需要去进行响应的。断开，它是要断开的。如果说客户端向服务器发送了多个请求的话，每个请求它都需要去自己各自独立的去连接，去传输数据，他要去自己独立的去建立连接的。


那其实 HTTP 的话，它会有一个 keep line 这样的这种模式，它会告诉我们的 web 服务器在处理完一个请求之后，可以保持我们这个 TCP 的连接，可以保持它是一个打开的状态，那如果说接收到来自客户端的一些其他的请求的话，这个时候服务端它会利用这个还没有被关闭的这个连接继续去处理，而不需要再一次的再去创建一个新的连接，这样子就大大的节省了我们的资源的开销。


那么这个 keep life 其实在这一段的时间内会保持打开的状态，只要它处于打开的状态，那么它其实也会占用一定的资源，占用的太多它也会影响服务器的性能。其实 engines 的话就通过了这样的一个 keep alive time out 这个指令来指定我们的一个超时的时间。每一个 TCP 的连接最多可以保持多长的时间，都是在这里可以去进行设置的，这个是他在这里所设置的一个默认值，对于这个超时间来讲的话，其实有一些浏览器它自己本身也会有这样的一种设置，像有的它大部分都是一个默认的为 60 秒。那么在这边其实我们也可以和浏览器保持一致，你可以设置为 60 秒也是没有问题的。当你不想使用这个的时候，每一次的链接，每一次的用户请求都要去创建一个新的连接的时候，那么你把这个给注释掉，把这个设定为 0 就可以了。
