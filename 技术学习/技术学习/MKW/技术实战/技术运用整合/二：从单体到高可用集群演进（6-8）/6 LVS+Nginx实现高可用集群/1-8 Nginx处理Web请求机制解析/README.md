那么首先我们在服务器里面会有一个 master 这样的一个主进程，它监听的是一个 80 端口，那么这个时候当我们的一个 Nginx 启动了以后，假设我们是配置了一个相应的worker，这个时候 worker 是默认的只有一个，那么我配置多个，我可以配置为三个，那么这个时候三个 worker 就会创建成功。


那么创建成功的时候，其实我们应该要说的**更加的规范一些，应该叫做Fork，那么 Fork 的话其实是在底层源码里面的一个调用 Nginx，它底层是用 c 语言去开发的，所以我们可以称之为是由一个 master 主进程去 Fork 了 3 个新的一个 worker 进程。**


OK，那么这个时候当我们的客户端有请求进入到我们 Nginx 的时候，那么我们的 worker 如何去和相应的一个请求和相应的客户端去进行一个连接呢？那么现在我们有一个客户端只有仅仅的一个，他发起了请求以后，这个时候我们有三个work，那么他们三个是应该要去抢吧。


那么怎么去抢呢？其实我们会有一个互斥锁，那么这个锁叫做 accept mutex 这个东西的话，它就是一个锁，这个锁的话和我们 client 是对应的，那么我们相应的 work 是需要去争抢这个锁，抢到了以后它才能够去处理相应的一个client，那么这个时候我们的 work 1 抢到了，那么沃克 2 和沃克 3 他没有抢到，所以这个时候的 client 和我们的沃克 1 他们之间是建立了一个相应的一个关系。建立了关系以后，那么我们的沃克 1 就可以为这个 client 去处理请求等等的一些操作事项了。


OK，那么我们的这个 work 必须要把我们当前的这个请求要进行读取、解析、处理、响应。把相应的一系列的操作处理完毕以后，那么我们的数据会彻底的返回到客户端，这个时候的话请求就结束了，请求结束以后，那么我们这个流程就可以说是处理完毕。

如果说我们的 client 2 是正常的，没有问题那么多，可 2 在处理完毕 client 2 以后，那么会处理 client 3。假设我们在 client 2 和 client 1 一样，它也发生了阻塞，那么这个时候 client 3 会一直卡着，等到 client 2 处理完毕以后，那么 worker 2 才会去处理 client 3。OK。所以跟着这个上方一样， master 它会 fork 一个新的 worker 进程，这个进程会处理 client 3。
