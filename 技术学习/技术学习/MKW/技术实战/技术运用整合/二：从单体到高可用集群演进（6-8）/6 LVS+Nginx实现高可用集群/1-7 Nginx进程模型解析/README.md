随后对于我们的一个用户来讲，其实我们的用户就是一个一个的客户端，那么客户端他会发起大量的请求，那么在这里每一个 worker 都可以去连接处理相应的客户的，这是客户端，也就是请求，那么在他们中间会有很多的连接，每一个 worker 都会有很多的链接去处理很多的client，也就是用户的请求。


OK， master 接受操作人发送的一个指令，可以读取并且验证它核心配置的文件的有效性，那么这个是我们刚刚演示过的，也就是一个 -t 。那么接受了指令以后会分发给其他的工作进程，可以不需要中断服务，可以平滑的去热重启或者说是热加载。我们的我可会连接很多的一些客户端请求，并且去响应数据和我们的后端服务去进行通信。接收到 master 的指定以后，那么可以去做重启，关闭等等这些指定的一些操作。假如操作人这个时候要发送一个停止的一个信号，让我们的一个 worker 去执行，那么这个时候假设我们的 worker 1， worker 2 他们没有连接着的客户端，没有任何的请求，此时沃克 1 和沃克2，那么它会关闭。那么我们的沃克 3 正在处理一些相应的客户端的请求，那么它是不会直接去关闭的，它会等到我们的这个连接释放，等到我们的请求处理完成之后，他才会去做一个关闭。


OK，那么对于安吉克斯来讲的话，它的这个进程模型的一个设计，它是有相应的好处的。那么每一个进程其实都是相互独立的，如果说是一个多线程的话，那么虽然多线程的开销要少于进程，但是他们会共用一个内存，而且线程之间它们相互是会有影响的，是需要开发人员自己去维护内存管理，并且也是会增加了一定的风险，会导致出错。


对于我们的一个多进程的这种模式来讲的话，他们是相互独立的，保证他们相互之间是安全的。那么如果说一个沃克异常退出了，那么其他的沃克是会继续的去正常运作的。对于 master 而言的话，那么只需要再 fork 一个新的 worker 这样的一个新的进程就可以了，而且它也是非常的安全的。**假设现在有一个黑客来攻击了我们的服务器，那么这个时候这个黑客他攻击的可能仅仅只是某一个进程，那么对于我们的一个运维监控人员来讲的话，他只需要把这个 work 直接给关闭就可以了。其他的worker，那么由于他们相互之间是独立的，互不影响的，所以这个 work 2， work 3 是绝对没有问题的。OK，被黑客攻击的那个work，我只是要去关闭掉，那么这样子就 OK 了**。那么这节的话就是我们所讲的关于 engines 的一个进程模型。
