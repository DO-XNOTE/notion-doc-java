**动静分离，它的一个本质就是一个分布式，**

**它是把动态的一些请求，动态的一个 API 接口和我们的静态数据，静态资源把这两部分进行了一个分开。那么这样子的话，其实我们就可以减少后台服务器的压力，提高了我们静态资源的一个访问请求的速度。**因为静态资源它其实是可以被缓存的，那么一旦有了缓存的话，用户在访问起来会更快。后端的话它是不需要再去渲染我们的一些页面了。按照我们很早以前的传统式开发，html， jsp 这些内容的话，我们是需要在服务器里面去渲染的。


现如今一旦我们做到了动静分离的话，那么静态数据静态资源我们都是交由客户端去渲染的，随后第二个，那么第二个的话其实就是一个前后端的一个解耦，前后端分开的一个分离的开发模式。这个其实我们在课程的一开始我们就已经是提到了，那么我们由于是重金分离，那么相应的我们的前端人员和后端人员也是需要分开开发，当然我们后续的一些部署环节也是要分开去部署的，各归各，各自归，各自去协调的工作，哪边有问题就可以去处理并且协调发布。那么这样子的话其实也会降低我们运维的一个复杂度，发布的时候会十分的快速，十分的便捷，那么前端人员和后端人员可以并行的去开发。


OK，这个我们前面是提过的，那么随后一个是一个咱们的静态资源，那么静态资源的话，我们不管是HTML，Javascript，css 样式，还有是图片、音频、视频等等。那么这些资源的话，我们可以让他们交由  nginx  去进行管理，也就是用户在访问的时候，其实我们是请求到了  nginx  里面，是从  nginx  里面去加载，去读取一些静态资源的。


OK，那么随后一个是接口的服务化，由于我们是动静分离，前后端也分离，所以我们的接口其实也是单独的去进行部署的，我们是作为一个 Tom k 的集群去部署，这样子的话其实我们是可以统一的封装一些接口，封装一些数据提供出来让我们的前端去进行对接。而不是说像我们以前那样，我们以前在使用 spring MVC 的时候，那么我们是返回出来的，是一个 model and view。那么我们服务化了以后，咱们的好处就是可以同时为多个跨平台的应用去提供我们统一的对接服务，不管是有 iOS 端、安卓端、小程序这一端，那么其实我们只要有一个接口，那么就可以为所有的前端去提供相应的服务了。那么动静分离的话，和前后端分离的开发模式，它们是相辅相成的，那么前后端分离了，咱们的动静也会分离。

随后再来看我们在动静分离中的一个动态数据和静态数据。先来看什么是静态数据，静态数据化，像 CSS 样式， js 脚本， HTML 页面，图片，视频、音频这些其实全部都是作为静态数据。静态数据的话，那么它本身是不会发生改变的，你要去修改，要发生改变就需要人为的去修改。比方说我们去修改了 JS 中某一片脚本，去修改了我们cs 某一部分的样式，那么对于这种修改的话，那么我们其实是修改了静态数据的内容，那么一旦修改它的一个 last modified，一个time，它的最后一次修改的时间会发生一个更改，从而会导致我们的一个缓存的刷新。


那么这是静态数据。那么再来看动态数据的话，往往我们都是以一个接口的形式去进行的一个请求，那么动和进这两种数据其实是可以相互结合到一起的。那么我们的动态数据，我们每次去请求的时候，可能我们第二次得到的数据和第一次数据可能是一样的，有可能是不一样。那么在这里我们可以举一个例子，比方说我们在看电视的时候，那么电视机是一个静态资源，它是不会发生更改的。电视机里面播放的内容，播放的一些影片，那么它是动态的数据。所以当我们在请求访问网站的时候，其实我们访问到的内容是静态数据和动态数据的一个相互结合，本质上和我们在看电视的时候是差不多的一个道理。

那么随后我们来再来看一下动静分离的一个实现方式，第一种方式就是 CDN 的话，它是一个内容分发网络，是 content deliver network，那么首先用户要发起请求，去请求一个网站的时候，肯定是通过域名去打开的，那么这个域名的话会经过 DNS  去解析，解析了以后会获得一个IP，那么这个 IP 是你将要去请求的那个服务器的一个IP。


那么对于我们的服务器来讲的话，一个网站如果说做的业务量很大的时候，那么它其实在我们的全国范围甚至是全球范围内都会有相应的一些机房的，比方说就像淘宝一样，那么淘宝在全球范围其实都会有一些机房的。那么我们的请求其实是会根据你的一个地域去选择你最近的一个地方，比方说如果说我现在这个时候我在南京，我在江苏的话，那么我访问到了一个地方，可能是离我最近在上海，那么如果说你是在黑龙江，在东北，那么你会访问到北京的机房，那么这个是根据地域去区分的。
