你如果在这个集群里面，就是 Claude 这个集群里面要起什么样应用，通常它都会去绑一个叫 auto scale 的这个服务，那这是在生产环境当中，那绑完这个服务以后，生产环境当中的所有应用，它的所有的这种 CPU 内存， HTTP 的这个实际的这个压力带宽以及它的延时，还有它和如果和 rapid MQ 沟通的话，它的那个队列深度都会被实时监控到。


根据这些资源的实际使用情况， Claude 方式也会做出一些提示，而这些提示就会进入一个什么定义一个策略的阶段，通过一个策略的最终来实现容器数量是增加还是减少，是非常一之简洁明了的一个触发机制。而在实际的这个运维过程当中，大部分的这个真正的 Claude foundry 都是以什么，都是以 CPU 和内存作为最主要的这个弹性扩缩容的这个依据的，而且其中 CPU 又是重中之重，因为每一个容器它其实默认都是占一核，当你这一核跑得很满的时候，其实它已经很难承载这个压力了，这个时候就需要去弹出去更多的容器。而内存通常我们每个容器都会适当的配置多一些，比如说我们的这个 Java 堆可能是 512 兆，那很多时候容器可能配了一个 g 的内存，所以相对来说内存还是有富裕的，所以不一定是真正紧张的地方。


而 CPU 通常是各个我们的这个弹性扩收容的第一依据，其次是内存，然后 HTP 延时带宽，因为各种各样的实际的使用场景，还是有些使用的 rapid MQ 用的是最少的。那除此以外，我们还可以做一些稍微复杂一点点的资源触发，比如说普罗米修斯的监控，也就是第三方工具的监控的结果回馈给我们的 Claude foundry 平台，来导致它的一个规则的触发。那除此以外，我们的这个 cloud function 里面还支持一个叫做 metric register 这样一个高级功能，它可以跟我们的 PP auto Scala 配合起来使用。


好，讲完资源触发以后，下面就要讲讲这个简单决策。所谓简单决策就是它的思路非常简单，


没有很复杂的控制条手段，但是也基本能达到我们弹性扩缩容的要求。怎么一个决策方法有一个最主要的决策点，就是伸缩上下线，所谓伸缩上下线就是我整个应用，我可以比如说从 3 个实例扩展到 10 个实例，也就是我可以什么以三核的这样的形式跑三个instance，在我们的这个服务器上运行起来也可以跑到 10 个，但是我们不能更多，也不能更少，这就是伸缩的一个上下限。

而这中间的伸缩可以定义规则，就根据前一页讲到的那些metric，我们可以定义什么时候开始增加一个instance，还是增加两个，什么时候是缩减一个还是缩减两个，通过这个 instance 数量最终决定，是吧？我们的实际的承载能力，那一个 instance 是占一核，同时也占它指定的内存，比如说 512 兆或者1G，同时也具有一定的这种网络能力，磁盘能力好，这是定义伸缩规则。


大家是不是有时候发现就是我们比如说要进行 on sale 了，就是打热卖了促销，或者是说我们要进行那种秒杀系统了，这个时候我们其实知道时间点的，那是不是应该在这个时间点我们要重新调整一下我们原来策略呢？本来比如说我们是允许这个应用跑 3- 10 个节点，但是我今天晚上要开始凌晨秒杀了，我就可以提前设置好，在晚上凌晨之前，比如说 15 分钟开始，我可以允许 3- 100 个节点，这样的话它如果有压力过来，它会快速向上伸缩。那甚至于我们可以提前什么提前变成 10- 100 个节点，就是使它默认就是到晚凌晨 12 点之前就先已经变成 10 个节点准备着了，真正到热卖的时候，到火爆的时候，它会快速增长到 100 个节点，这样的一种时间规则来控制什么弹性升猪的上下限，只是按照不同的时间点，它的上下线不同，能产生一些更灵活的要求和功能。
