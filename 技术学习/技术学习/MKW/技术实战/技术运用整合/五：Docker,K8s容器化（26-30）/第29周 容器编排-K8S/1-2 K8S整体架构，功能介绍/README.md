那除此以外，还有一些像库布里特、proxy Doc 这些基本的节点，那么他们或者说模块他们配合起来，形成了 node 这样一个手脚的这个实际功能。所以了大脑，有手脚中间还有一些耦合的模块就形成了一个什么库普内提斯的整体。下面我们先来看一看库普内提斯大脑。 master 节点 master 节点我们可以类比什么呢？类比成 mess 的 master 加上马拉松的 schedule 的合体。那它具有哪些功能呢？首先它具有类似于 message master 的那个就是 API server 的功能。当我们 kubernetes 的命令发一条命令的时候，不管你是在 kubernetes master 上跑，还是在远端的一台命令行的这个终端上跑，它的命令都会通过网络或者是本机的形式发给我们的 master master API server 来会接收这个命令，判断它是不是符合语法标准，同时根据我们当前的这个实际的国内特斯集群的环境进行相应的处理。这就是 API server 另外一个很大的功能模块就是 schedule 的就类似于马拉松 schedule 的做一个什么，对所有资源按照应用需求进行统一调度，任务发布。

当我们一个应用需要访问什么其他应用的时候，他会去尝试找一个 DNS server 那找完了以后，他知道了 IP 地址，他就会通过 kube-proxy 去访问这个 POD 但是如果这个 kube-proxy 当时拿到的地址是要访问另外一个 POD 我们前面也说过了，就是那个 kube-proxy 具有高可用负载能力，同时只有业务服务的发现能力，就他能知道我们所有的 POD 就同一个 deployment 里面的 POD 到底分布在哪些的不同的节点上，从而进行什么网络的数据包的转发。所以这就是一个 service 的概念。



这整个一套功能就是结合我们的 kube-proxy  结合我们的 DNS 来实现网络服务的功能，我们抽象出来叫一个 service 这个 service 很好地描述了我们应用和应用之间的互相访问。所以有些人把 service 看成是一个网络，就是类似于我们 Doc 里面有网络配置。所以我们库内特斯有 service 配置。
其实不尽然， service 不光配置了网络，比如 IP 地址端口，同时还配置了服务方式，甚至还选择了 DNS 选择了负载均衡，选择了我们是用主机 IP 地址网络访问还是我们是容器之内的内部，通过我们的这个印花布还是通过我们的这个法兰绒进行互相通讯，整个一套的什么用户访问机制才是 service 的核心。


后面我们会在服务发信和网络这里详细的介绍 service 还有一些什么概念呢？除了我们刚刚 service Paul 的 con replication controller 这些 deployment replicate set 以外，我再介绍两个另外的概念，一个叫 label 这个概念其实在后面也会反复出现。

所谓 label 就是什么，我对某些特殊的 POD 我打一些标签。那这个标签打完了以后，我就可以用这个标签来查询和寻找这些 POD 比如我们有一个 POD 上面打个标签叫什么 App 它的名字叫什么叫 my App 打完这个标签以后，我就知道这个 POD 上发布的应用叫 my App 下次当我用 service 来做负载均衡网络管理或者服务。发现的时候，我希望什么对外提供一个特殊的比如 IP 地址或 URL 这个应用的后台，只给 my App 这样一个名字的应用提供这个网络路由，不要把我的 traffic 打到另外一个叫 your App 的应用上，这里面就可以通过什么可以通过一个 selectors 来选择一定的 label 来实现网络的这种负载均衡和指向。所以这个 label 也是很关键的概念。


除此以外还有个概念叫 namespace 这个概念可能就更大一层，就是我们平时的所有的 deployment POD 所有的这些概念都是在一个虚拟集群上的。
虽然我们物理上是一个库伯内特斯 class 的，但是一个物理的库内斯克拉斯默认就会生成两个虚拟集群，一个是kubernetes，也就是库布内特斯自身管理自身的这样一个集群。还有一个叫 default 是我们应用部署，如果你不指定 namespace 会默认发布到一个叫 default 的 namespace 里面，那这两个就是虚拟的集群。


这种虚拟集群的概念我们就叫 namespace 同时一个Kubernetes 可以支持几十个甚至于上百个不同的 namespace 只要你在应用发布，只要你在软件包安装的时候，你指定我希望用一个新的 

namespace 名称来部署这个应用，那它就会在一个新的虚拟化集群当中很方便的进行管理。
所以 namespace POD 这两概念的引入，使得Kubernetes有更大的灵活性，它能够把一个物理集群变得更多的虚拟化，虚拟成更多个，同时又能把很多很小的这种 Doc 容器进行大的拼装。这样的层层的这种封装处理以后，就使得容器上面有 POD, POD 上面有 controller 的 controller 的上面有 namespace, namespace 上面才是 kubernetes , 这样一个层层套层层的套娃结构，使得我们真正的一个大数据中心的大批量成千上万的应用的容器化或者说是小的微服务，可以最后通过一个高可用的层层管理的具有可扩展性的这个kubernates 集群。
