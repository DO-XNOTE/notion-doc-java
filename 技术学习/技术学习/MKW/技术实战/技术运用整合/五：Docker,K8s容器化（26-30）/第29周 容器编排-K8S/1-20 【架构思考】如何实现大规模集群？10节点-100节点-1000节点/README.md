那这些节点的这个之间的沟通是什么？是有大量的网络的这个负载的。那同时我们任何一条发给这些节点的指令什么都会记录在我们 ETCD 里面。所以 API server 是要记录大量不同的指令，同时你要记录什么各种各样我们集群、节点之间的关系、节点之间的服务的通讯的情况等，大量的业务流、信号流什么都会把这些数据写到我们 ETCD 所以 API server 会什么会成为一个命令控制的瓶颈，而 ETCD 会成为一个什么？所有的状态，所有的这个命令所有的之间沟通心跳沟通这些数据保存读取时候的一个瓶颈。


所以很多时候我们用命令超时，其实什么是后台承载不了这些命令就压力比较大了，或者是 API server 有点承载不住了或者更极端的情况，就是 ETCD 来不及往什么往它的后端磁盘去写这些数据了，就这两个产品的播那除此以外还有什么样的问题呢？还会发现各种各方慢。就是当我们节点数增加的时候，虽然我们的带宽吞吐量不管你是命令流的带宽还是什么数据流的能力都提升了。但是有些时候一个这个 DNS 解析会变慢，还有些时候一个容器的启动会变慢，那各种各样的慢，那大家都想想可能来自于什么地方。


好，我们看下一页你就会找到它的答案，我看到底它是什么样的原因来导致这些问题的一个问题是什么？ API server 其实本质上什么是一个无状态的应用服务，所以它大部分的时候，每一个 API server 的这个过程什么都会跟 ETCD 有沟通。所以 API server 的慢很多时候不是 API 思维本身，它的本质还是 ETCD 慢。
那 ETCD 因为什么因为有太多的血导致什么 ETCD 的本身的性能会下降？为了防止这种问题的出现，我们有一个很好的方法就是什么通过我们硬件升级采用比如说 SSD 这种固态盘，或者是采用一些闪存设备或者是提供更多的什么内存技术来实现什么来实现。他写的时候不需要什么，不需要有一个很长的我们的这个延时，他写延时越短， ETCD 什么能够同时处理掉并发写的能力越强，这个时候它的集群的能力就越大。同时那除了我们这种硬件升级以外，我们增加这台 ETCD 节点，有时候也能够解决一些问题。
