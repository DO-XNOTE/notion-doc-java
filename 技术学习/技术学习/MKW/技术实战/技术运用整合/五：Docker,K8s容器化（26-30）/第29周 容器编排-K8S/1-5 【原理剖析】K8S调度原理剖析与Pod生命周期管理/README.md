到底什么是 POD 呢？这里是一个例子就是一个应用里面其实起了两个容器，一个用来做什么文件的拉取，就是 far 的 polo 它从一个 content manager 那里面去拉取文件，然后放到我们的什么 volume 里面，后台的磁盘里面。那另外一个容器是 web server 它起的是一个类似于 NGINX 或者 htb D 或者是阿帕奇的这个应用服务。然后让消费者去访问这个页面来查看里面所有的照片、视频资源。


这样两个容器，因为它会共享同样的什么同样的卷，所以我们完全可以把它作为一个 POD 来进行统一管理。那这里就要再一次强调一下 POD 是什么，是库布内特斯里面最小的一个可部署单元。那它其实可以有很多很多个不同的容器共同形成一个 POD 那这些容器有哪些东西是共享的呢？首先他们在 name space 跟 C group 的隔离上是隔离成了一套环境，所以相当于是通过 Linux 什么 name space 以及 C group 和其他 POD 之间进行隔离。但是在一个 POD 的容器之间确实是什么，是可以互相互通有无的。那同时它也让所有的这些容器共享同样的网络资源，所以你很方便在同一个 POD 里面，各个容器之间用 local house 之间是可以很方便的进行什么网络通信的。同时，所有的卷不管你是容器 1 还是容器2，他们都可以访问相同的物理卷。这个卷其实就来自于 POD 里面，所以 POD 里的卷是在整个 POD 所有容器之间是共享的。


那除此以外，我们的这个 POD 还有一个特点，什么 POD 是在一个物理节点上存在的。也就是说如果我们发现说这个物理节点出错了以后，我们通过比如像那个 deployment 里面选的什么 replicaset 的功能，在另外一个我们的物理节点上重新起一个新的 POD 那这两个 POD 即使所有的内容都一样，他们的 POD ID 、 UID 也会不同。


那这就是什么 POD 的一个重要概念，就是 POD 只存在于某一个物理节点上，或者说某一个我们在 coupon native 的 node 上，一旦它从一个 node 漂移到另外一个 node 或者说是跨 node 之间，它们是不同的 POD 承载相同的功能，所以 POD 必须在一个 node 上，那同时它又可以运行很多个 container 那这样一个有什么样的好处呢？就刚刚说到的有多个容器要进行文件共享，要进行什么信号量共享，要进行环境共享，要进行网络共享的时候，很方便的通过一个 POD 来进行处理。
那同时如果大家对 service mesh 就是网格服务或者说是一种很新的计算的这种框架了。服务框架很熟悉的话，大家可以知道未来越来越多的服务它会带一个什么？带一个边车，就相当于是一个大众的摩托车，旁边带一个小的这个边车斗子上面可以做人可以放东西。


那我们到底通常会在 service mesh 这种网格性的这种服务框架里面， service mesh 的这个边车上放什么呢？一大块就是监控。如果你有一个系统，你需要把所有的监控服务做统一管理，那你要装很多的监控 agent 对不对？那这些 agent 装在你的系统上很麻烦，不如在我们的应用旁边做一个小应用，他专门负责监控。


那还有什么？还有熔断？比如你每个应用都要写一套自己的熔断，很麻烦。但如果你让那个大应用绑定一个小的鞭策车斗，这车斗里面可以放监控，可以放熔断，可以放配置管理可以放服务发现可以放路由可以放那个我们 srouted 是吧，就是网络受限等等非功能性的需求。这样你在这应用只要完成功能性，而你旁边的那个边车是吧，sidecar里面可以完成所有的非功能性，但是它又和你应用什么一一捆绑。那这种捆绑方式很容易就是在 kubernetes 里通过 POD 来实现。


POD 里起两个容器，一个容器作为比如像我们的 web server 这种什么主要的应用服务器。另外一个容器起一个就是专门做 operational 的这种编车服务，它可以把我们的那个熔断我们的配置管理服务发现对吧。 sloutling 甚至于什么纳斯访问、数据库访问等等很多其他的跟非功能相关的这些需求放在那个边车的容器里。然后这两容器共享文件系统，共享信号量，共享我们的什么 Linux 的这个域的隔离共享我们的 IP 地址共享我们的文件系统等等。这样一个 POD 很容易的部署在所有的什么 node 上，不同的 node 上部署不同的 POD 对吧，形成一套多个 replicate 复制的，然后很方便的进行迭代管理的这种 deployment 那这就是如何或者说最常见的那种在新的服务框架底下实现多炮的更多的这种容器化的这种解决方案。
