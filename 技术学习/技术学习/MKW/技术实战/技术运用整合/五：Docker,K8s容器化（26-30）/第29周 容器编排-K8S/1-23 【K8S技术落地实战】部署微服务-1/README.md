那这些无状态的应用怎么管理呢？大家可以想到就是虽然它的应用代码无状态，但是在配置信息的时候是不是也要传进去一些简单的用户配置信息，哪种传法比较好？我们是选用我们在 kubernetes 章节学到 config map 这种方式，还是大量应用的配置信息都用 spring cloud config 用我们 spring cloud 的思路，这就是两个冲突，就是 kubernetes 跟我们的 spring cloud 在使用的时候，大家经常会碰到类似问题，到底是用我们的底层偏系统的集群的方式，还是用在 Java 的应用软件当中的这种 springcloud 框架的方式？那我这里推荐的是 springcloud config 因为 config map 虽然也能实现动态的配置，那它的动态是从 map 到应用软件之间是吧？用一个 volume 的 mount 的方式来回一下是不是可以动态的部署，但它只是一个 mount 一个文件。它不是从 github 直接这里动态的拖出它的变化，然后刷新到我们应用里面，它动态不足以能覆盖我们应用到我们 github 之间的完整链路。所以正常的情况我们即使用 coconities 部署还是应该选后者。

当客户端是没有选择权的对不对？当你对一个 UL 发起访问的时候，你是不能来选择的。那另外一种思路其实就是 spring cloud 里面，像 ribbon 这种，就是让我的客户端成为我的上帝，我来决策，我到底是要红天使还是蓝天使来陪伴我左右到底是哪一个更合适我呢？到底是 server 端的 ingress service 还是像 sprinkle 的 ribbon 这种客户端的负载均衡方案呢？那我这里也要强调一下，其实我们很多的实战当中都是选了 ribbon 因为 ribbon 的客户端的方式更灵活，server端的方式相对来说多了一道。因为当你在用 spring car 的时候，你是默认其实是会尝试走 ribbon 那些库，走 ribbon 那些功能。然后你只是发现一个 IP 地址，然后让后台的 server 再去重新做一次负载均衡。就是你做了两次负载均衡有点浪费。还不如用日本一次性做负载均衡。然后我们底下就不用去进 ingress 也不用去见 service 省了很多事情。所以在无状态应用当中，我们会更推荐 spring card 的功能给大家。
