那到底有哪些 controller 来围绕这个 POD 来进行控制这些 POD 最经典的一个 controller 就是 replica set 所谓 replica set 就像图里面看的一样，这是四台对吧，长得像扑克一样或者贴着扑克粘贴的手机，它的四台手机本质上其实是一样的东西。但是它贴上我们的扑克皮以后，你会发现它有的叫 K 有的叫 Q 有的叫结，有的叫什么 Joker 不同的这个 POD 会有很多个不同的副本，那这些副本合在一起，其实实现相同的功能。


那这就是一个 replica set 的概念，把相同功能的 POD 以及多副本不同的 UID 的形式的发布到了吗？发布到不同的这个 node 节点上，这就是 replica set 而在 replica set 之上再一级那就是 deployment deployment 部署最后小单元还是 port 但是它通过一套或多套的 replica set 来把它的业务应用部署在多个 node 节点上，以多个不同的 POD 的形式运行起来来，做到负载均很多，做到高可用。


那同时这里看到这个图标是什么？这是一个升级图标，是不是也就意味着 deployment 不仅是负责这种多 POD 的发布，同时它也负责所有 POD 里面这些容器的迭代升级。当我的那个新的代码有新的这个容器 image 准备好以后，我可以修改我的 deployment 这个 YAML farm 从而重新的去 apply 这个 deployment 去生成一个新的 deployment 同时它还支持什么很好的一个回滚，所以升级回滚加上 reputation set 的这些复制功能才是真正的 deployment 的核心。


那多数的应用其实我们都是以什么 deployment 方式进行发布的？好除了这种模式以外还有什么？还有一些短作业，就比如说我只要运行一次就可以退出的那些，他不需要作业长时间运行。那这个时候我们可以选一个类型叫什么叫 job 的 control 来进行处理。同时 job 还可以支持其他不同的 type 的这个发布方式，比如像 cron job 定时 job 那本质上它的 control 就是一种就是 job 好，那除此以外我们还有一个什么？还有一种叫 state for the set 那大家如果后面的实战会仔细看到，就是我们 POD 每次发布以后，它的这个名称会变。同时如果这个 POD 在这个 node 死掉以后，如果是 restart 一个是吧或者 replicate 一个出来，那它这个新的 UID 也会变。那怎么样才能让我们整个这个 POD 就算它再怎么变，它整个的这个名称不变就固定一个名称。那我们其实也是支持这样的，就比如这个图里面小明是吧，他永远叫小明，不管他在哪一本语文书里面，他的画的样子可能都不一样，但他主人公永远叫小明。我们要保证我们这个号的不停的进行处理、重启、故障，它的各个副本的名称永远不变。在某些英文当中，因为会有指定 POD 的名称，所以 POD 的名称不变也是一个很重要的概念。这就是 state for set 但是我们 demo 这里就不做了，因为相对来说还是比较少见的。


下面一个也是非常常见的就是 demo set 所谓 demo set 就是这样，相当于什么一个值班人员，一栋大楼可能只需要一个值班人员在网上值班，就可以保护整栋大楼一样。道理，一个 node 节点上只要跑一个 demo set 就可以完成我们整个业务需求。特别是对那些有占用端口的，比如说我们占用的端口都是相同的端口，这样你跑多个容器，如果大家都占用同一个服务器端口，就会出现意外。


很多时候我们一些特殊的功能，这个功能就比如说我们要把这个整台 node 里面的所有的 log 往一个地方输出，同时我们占用这个 which 的，比如说 43 端口或某一个端口来完成用户的这个节点的应用的登录。那这种情况下我们就要求什么这个 POD 不要在同一个节点上起多个不同的 UID 的 POD 这个时候我们就会选 demo set 那 demo set 是可以保证我们单节点上的 POD 支起一个 replica set 可以实现很多个不同的 node 之间进行大量 POD 的发布。在同一个 node 上可能会有多个 POD 而 diploma 会完成什么，这些 POD 的什么迭代周期、升级、回退等等。而 job 是可以跑一两个或者并行的跑多个。但它跑完以后它会自动退出，或者你可以控制定时的像那个我们图标里面那个计时器一样计时的进行跑。 state four 永远跑起来，它的名字就叫小明。
