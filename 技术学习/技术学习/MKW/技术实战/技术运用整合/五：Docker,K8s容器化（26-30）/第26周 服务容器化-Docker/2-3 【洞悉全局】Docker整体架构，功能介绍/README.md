所以说这些东西都是跑在什么 Docker 命令里面的，不管是 Docker wrong 还是其他的 Docker 什么 exact 那除了这个 Docker 命令以外，我们的客户端其实还可以跑另外一种就是 rest F 形式的 API 那通过这些 API 也能实现类似于 Docker 命令达到了要求，比如运行一个容器或者下载一个 image 等等。

当容器把这套静态环境加载起来的时候，容器会在顶层尝试开一层什么可读写的文件的小螺丝系统。那这样的一层结构，就是它既能访问之前那些静态的，比如阿帕奇目录、那些 E max 目录，那些底边的操作系统目录，同时它又能够有一层可读写层。然后用户可以在这个容器里面进行修改。只要容器不销毁，这些修改都是有效的。它容器销毁以后，这些修改的内容却不会影响到镜像。这很像就是我们虚拟机环境当中的我们的一个的这个模板。当你模板启动了以后，你会成为一个虚机，虚机里可以做所有的操作，只要虚机不销毁，虚机的内容都是保留的。当虚机销毁以后，你重新再用这个模板生成新的虚机，你会发现之前那些操作并没有保存，那这两个概念是非常紧致的。


那讲完这些镜像以后，我们其实就讲到了最核心的内容了。容器前面一直在说说 Doc 到底为什么有优势是吧，微小通用化其实都是指它什么它的这种集装箱能力，这种容器能力。这种容器能力的实现，底层其实依赖于好几个核心的功能，一个就是隔离每一个容器既来自于相同的操作系统，但他们同时又在内核态和什么和我们的这个用户态上进行了部分的隔离。他们隔离了一部分的这个我们的进程隔离了一部分网络，隔离了它的这个信息量，隔离它的内存，隔离它的文件系统，隔离了它的用户的主账号的管理，也隔离了它的用户名机器名。

所以这些隔离就实现了什么？实现了大部分的容器都能共享操作系统所有的这个能力，但是他们的每一个这个独有的这些进程运行时状态又不会互相之间得到干扰。这就是 name space PID net ipcmntuts 所实现的 name space 就是它的隔离的核心。


PRD 就是指我们进程的隔离。 net 就是指网络的隔离，IBC是内存和信号量的隔离。 mnt 就是指的是什么？指的是我们文件系统的隔离。 UTS 其实就是指我们包括主机名，甚至于我们还包含一些就是我们用户账号的隔离等等。系统的所有的 CPU 内存都用光的话，其实这些就所谓进程隔离、网络隔离都没有真正的效果，因为所有的资源已经被消耗光了。所以为了防止这种问题出现，我们还可以对所有的这些容器进行 CPU 内存和 IO 的限制。 by default 其实很多时候容器起来的时候是没有限制，所以我们应该在生产环境的容器中额外去尝试增加这些限制来防止我们的什么物理资源被消耗。那除了文件隔离和限制之外，其实刚刚也聊到了这个容器运行起来的时候，它有一层那个底层叫联合文件系统，这种连痕的 union FS 它可以支持不同的这种文件系统驱动，在文件系统驱动底下又可以有很多的不同的文件系统选型。同时这些文件系统和之前刚刚聊到的这个镜像里面那些静态的一层层加载的那个目录结构相关，使它变成一个可动态可修改，然后可以稳定运行，又互相和其他的这个容器的文件系统不干扰了这样一种 union FS 这层层的内容加在一起就实现了什么？实现了我们一个集装箱容器能够很好的从它的仓库里拿到它的镜像，启动起来也能很好在我们 demo 上进行管理和配置。同时很好的和我们的命令行客户端和我们的 API 客户端进行沟通。
