前面聊到了微服落地有诸多需求，那容器化这个解决方案就是成为它的不二之选。那我们首先来看一下容器化是怎么样通过它的三大助力实现微服务的落地的。那第一点就是我们的容器化解决方案它具有一个普适性，它可以适合于各种不同的码头，各种不同的软硬件平台。其次容器化有一个微小性，它的微体现在什么？它的微体现在所有的那个应用，大小尺寸跟微幅非常之接近，小它的灵活体现在它能够做到分钟级的启停和部署。

所谓码头其实我们可以从好几个角度来看，那这张图的下半部分正是所谓的码头。从虚拟化的角度来看，我们的大量的操作系统现在不直接在物理机上，是在虚拟化的平台上。那虚拟化有个什么呢？有像 power 的一种叫 power VM 的虚拟化技术，有 vmware 的那种基本通过 hypervisor 提供的那个虚拟化技术。那也有，比如像一些zen ，像一些微软的 hypov 提供了虚拟化技术，大量的虚拟化技术，上架的这个操作系统平台也各不相同。有这个底边架构的 Linux 由我们 centos fedora 为平台的 Linux 有像 sororis 这些 UNIX 也有 Windows 以及一些其他的第三方的操作系统。那除此以外，我们硬件平台也五花八门，也有传统的项目 IBM hp 、戴尔的那些服务器，也有像我们的 bat 亚马逊、微软、谷歌的云平台的服务器。


那除此以外，我们后台的存储设备也很多，就像 em C 、H DS 这些传统的存储厂商也有更多的基于这种互联网的分布式的存储架构。那这些大量的不同的存储服务器、虚拟化操作系统就决定了我们有各种各样不同的码头，大到海运码头，小到胡运码头、合运码头。

那之前我们的所有的应用服务，比如左图所示，他把一大堆的小的服务成为一个大的应用，布在一台物理机上或者一台虚拟机上。那这个虚拟机需要大量的内存，需要大量的计算单元，还需要一个独立的操作系统承载它整个这一套应用。到了这种刚刚看到了右图中那种小方格以后一个小方格。如果你还占据一个操作系统，占据一套独立的 CPU 内存空间，这样的资源消耗是非常非常巨大的。


以我们微服务改造之前应用为例，之前应用可能只是要一两个虚机就能完成。如果我们把微服务改造完以后，上百个这种微服务全部用虚机来承载，我们需要几百台虚机，这还是不考虑我们什么高可用，不考虑我们数据备份。容灾的情况下，如果有高可用性能要求，容灾考虑进去，我们可能需要上千台虚机。可以想象我们的数据中心的承载压力有多大。


那如果有了容器化会怎么样呢？所有这些微小的内容成为一个把 runtime 和应用打包好的一个微小题。这个题可能原来代码有 20 兆，我们打包完了以后可能只有 30 兆。那原来代码的这个启停在一个操作系统里面，包括操作系统启停、数据库启停应用启停可能是几十分钟，在我们的容器化里面可能就是几十秒到几分钟。通过这样的一个技术，我们仍然可以把它几百个不同的这些微服在一台操作系统上运行起来。那为了实现高可用，我们可能有多个操作系统，但这个多也是 10 这个级别来完善的，而不是以千这个级别来完善。那通过这样一套比较微小化的容器化的部署方式，能够实现所有的这个微服可以很方便在各个地方飘来飘去起来起去。但是不受限于硬件平台的制约，这就是微服和我们的容器化相耦合的一个点。微和小。那来看一下第三点。

那在互联网的发展或者说电商发展的前期，我们通常一个部门叫开发部门，这个部门既负责比如像我们的互联网那个应用里电商平台里面的支付告警，订单的管理、用户的管理、买和卖业务流程的管理。同时我们的运维部门搭建整个数据中心，完成所有的虚拟机的环境，所有的网络的接入。同时有可能还会需要对外我们提供一些底层的数据库，比如底层我们的 API 对外暴露的这些网络的负载均衡、内容的转存，这些功能它们是两个大的耦合性很弱的部门，它们相对独立又互相制约。


为了满足我们互联网的不停的这种新需求的发展，很多的公司在组织结构上做了很大的调整，把我们的开发测试和我们的这个运维团队合体成一个小的事业组。这个事业组负责用户，那他既负责用户的这部分代码的开发，也负责它的质量的检测，以及它的那落地和实施。另一个用户组可能是支付，这个组既包含它的这个所有的应用的架构，包含它的这个部署架构，也包含对客户提供实际的响应服务。如果我们的支付平台坏了，他们是提供 27 乘 24 小时的响应。这样一种就是我们组织结构的变化。按照我们现在比较流行的康威定义来说，就是组织架构决定上层建筑，当一个公司有怎么样的一个数字结构，它就会有怎么样的一个业务形态，会有怎么样的一个 it 系统。那就要求我们任何一个组织的这个 it 系统他们的成员其实大部分还是我们的这个工程师。


如果以我们当前最热的 Java 为例，就是我们的 Java 架构师研发人员和我们的 Java 平台的测试人员以他们为主体。那你要完成基本的这个所有业务系统的 QA 发布、准生产发布，以及我们生产平台弹性的可扩缩容的高稳定性的发布要求。我们每个 Java 的架构是和夹外的开发人员对系统的运维有很深入的了解，其实这有些不现实。那容器技术怎么样解决呢？容器技术是我们的有一个统一的容器平台，搭建在一个开源的或者是第三方支撑好的一个平台上。那我们的所有的研发人员只需要了解这个平台的特征，学会怎么样通过这个平台发布我们的应用。那这个平台自动的完成了，像我们的扩缩容，高可用，快速启停，这就满足了我们的组织结构变化。以后，对我们这个所有的系统所有的架构师，所有的开发人员没有太多的额外的要求，非常之普适于我们来组织结构的变化，甚至这三点实现了我们微服务改造当中前面说的很多的问题。
