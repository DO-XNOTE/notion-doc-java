这个时候就推出了一个概念叫 WPA 这不是个虚拟化的概念，依然采用相同的操作系统。但他把一部分作业任务一部分的功能编排抽象出来，成为一个运行的操作系统里面独立的任务的管理。那这个技术在 10 年之前，我们大型机的发展过程当中，逐步又推广到了我们 IBM 的小型机。同时除了 IBM 以外很多的这种我们操作系统或者是硬件服务厂商都有类似的思路，把大量的应用独立化的包装独立化的一个运行平台抽象出来。这就是我们容器技术的原人阶段到了下一个阶段，智人阶段那是以 Linux 或者说是 Linux container 这样一个组织为主体。他们发现其实我们最广泛使用零子平台，它应该也有类似的这个思路，应该把我们的内核和我们的用户态的核心的内核部分进行隔离，也就是 name space 隔离方法。


同时在隔离完了以后，我们也希望对所有的资源进行一部分限制，不要让隔离的这个部分应用把整个操作系统的所有的 CPU 内存都占光了，不要把所有的文件系统把所有的这个性能都占光了，所以 C group 的资源的控制尺度也展现出来。那在这样的过程当中，我们更方便在 links 上面以各种各样的应用节点为主体进行合理的隔离和控制。


那到了下一个阶段，就是最辉煌的容器的发展阶段。 Lxc. 这个阶段，容器成为一个独立的名词，提出来肯定的，在提出来的过程当中，我们有大量的这个翻译人员把它翻译成集装箱，其实这是一个很对的翻译。因为整个容器物在所谓的这个出现到我们当前这个理解过程当中，它就是一个集装箱，它就是隔离硬件软件的码头和我们实际不同的货物的一个集装箱平台。但是为了更高大上，更符合 it 的需求，我们现在统一称它为容器。那么在这个山顶洞人出现以后，就进入了现代化时代。那现代化时代有两位及现代化时代的达成的，容器的技术的达成者就是 Doc 和 cloud founder 这之后我们会有两个 MD 图文的方式来进行实际的讲解，说什么样的高客技术，那使容器化能达到更加辉煌的境地。那卡尔方尺又有怎么样独特之处？正是当前这种现代化的 Doc 和克朗方罪的广泛使用、庞大的生态圈的支持以及大量的开源化的解决方案，使得我们很多很多的应用，特别是互联网应用和我们电商应用。

就是当前我们现在在这个 201 几年甚至于 202 几年这样的阶段，所谓的这个技术我们都在以一个以容器化为主体的方式运行。那在未来的五年十年我们会有怎么样的变化呢？有一个很可能的猜测就是平台化发展。也就是说我们到未来阶段可能不会特别关注于某一个容器技术，而是更加会立足于我们的抽象的技术。比如说是一个公有云的 pass 平台，一个很开放的编排式平台。我们怎么样通过这样一个平台实现容器的很方便的扩缩容，很方便的网络互通，很方便的数据持久化，很方便的进行整个这个容器的生命周期的管理。那这样一个平台要求我们对整个容器的搭建这个要求的越来越低，但是对整个容器框架对编排框架的理解的要求越来越高。所以这也就是可能我们未来的 Java 的架构师要非常了解我们的这个应用是怎么样部署在容器上，不一定要对我们的容器实际的命令作为很深入的了解架构原因以及可适用性。我们的优劣比较这些知识可能是对于我们未来的加入架构是一个重中之重。



第二点是无服务企划。所谓无服务器化，其实就是另外一种容器主推的，比如像我们亚马逊的兰德技术， AWS 云上面的一个兰德技术，比如像我们微软 zo 上面的 functions 或者像我们谷歌 GCP 上面的 cloud function 这些技术要求就是我们对于底层的容器的这个实现都不了解。那只要把应用代码发布上去所有的这个云平台来完成它的后台的容器化，完成它的操作系统的虚拟化，以及完成所有的对这种业务输入流和输出流的管理。那比较经典的案例就是当前很多物联网系统，很多事件触发的系统已经开始采用这种无服务化的架构，可以理解为就是我们进一步抽象容器，进一步实现了应用的解耦。



那第三个想法就是我们的无服务计划，它很朴素，也许只适用于像物联网这样的场景。对于更多的电商，这种主流的支付平台更多，比如大数据，更多像应用为主体的平台，他们可能会向一个统一的生态圈靠拢。目前来看，最大的可能的生态圈就是 Doc，它可能会逐步扩大。它包括像卡奥斯方瑞、库布内特斯这些平台，全部依托于导塔为主体，形成一个大的生态圈。在这样一个大的生态圈的体系下，各种开放的第三方技术以及开源技术围绕 Doc 形成一个完整的生态系统。这样的生态系统要求我们的 Java 架构师对整个生态的核心有深入的了解，以及对其他分支的优劣有熟悉程度。之后，可以通过我们的公有云服务，方便地部署我们的微服务。
