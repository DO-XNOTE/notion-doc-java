进入我们容器化的实际章节内容之前，我们回顾一下之前的分布式改造，微服改造对我们的落地提出了什么样的要求。首先第一个要求就是我们的业务中间是有很大差别的。比如我们封建老师一开始部署的那个就是 monolithic 这种单一性的应用，它这中间的很多的服务本身对 CPU 和内存的消耗都很大。大家如果在自己的环境虚机跑起来，会发现 CPU 内存消耗是一个主体消耗。那之后我们阿神老师的那些消息中间件是什么？是 CPU 加上 IO 消耗都很大的类型。而后面林波老师分布式数据库明显是 IO 吞吐型压力很大的应用。那这些应用它们的特征有点不同。其实我们在整个的这个电商网站当中忽略了一部分，比如像大数据、人工智能这些应用。
那正常的一个公司，如果你上传一张我们的产品图片，它除了会族哈希比较之外，还会去尝试去用图像查重 open CV 的方式来看你这张图片和原来那张图片是不是一致，然后它会提醒你我们已经有一张更清晰的图片。那这种服务就是通过 GPU 计算来做图像识别的。所以整个真正的电商网站，有 CPU 有内存、有 GPU 有 IO 甚至有网络吞吐型很强的不同的应用，他们的差异非常大。

那如果是让我们的应用管理员或者说我们的夹发的架构师来管理这么多不同的应用摆放，不同的位置是很复杂的，它需要有一个比较智能的自动选型，找到合适匹配资源的这样一个方法来实现我们的应用的落地。除了这个应用需求以外，还有一个比较典型的需求就是敏捷性要求。那之前一个应用可能是一个大应用是吧，是一个大的外包。那拆分成微服务以后，我们光上普通的这种微服务，提供了 spring cover 的一些组件就十来个，而服务本身又拆成了十几个不同的这个 API 的服务，再加上前端的应用，再加上一些其他的配套的，比如管理的应用、图形化界面的应用，加起来已经有二三十个。
而真正的一套电商系统，它除了要保证我们的买方的需求还要能够上架，还要能够做一些卖方的支付。所以还有很多卖方的需求。同时也要让我们的后台管理员、我们的电话中心的服务人员能做后台的 back office 系统的管理。除此以外，还有刚刚提到的人工的智能大数据业务分析团队等等，有很多不同的团队的需求。


其实整个电商系统远远不止几十个服务，通常都是成百上千的服务在那里跑。每个服务要求很快速的启动，几十分钟的启动肯定是不够快速的，通常要在几分钟之内能让一个应用能够以它的依赖关系的形式能够完完整整地启动起来，同时也要需要优雅的停止。比如说我们压力比较大的时候，我们对一些服务做降级，那这些服务降级了以后，某些功能模块停下来，所以某些后台的这个虚机节点或者是容器节点可以上当了就停止。那这个时候要求它的本身业务做完，它的本身的那些数据，要跟别人达到一致的时候再停止。就这种优雅的停止，也可以通过容器的方式来实现，这也是对整个敏捷提出一个要求，既要快速启动，也要快速停止，同时还要比较优雅，同时有成百上千的不同的服务。那这些服务如果要保证高可用，就是成千上万的不同的虚机，如果你不采取用容器化的话，很有可能就是成千上万个蓄积的管理，那非常之庞大复杂的一个任务。


那除此以外，还有一些组织架构的变化，特别是在半新老师的那段课里，大家应该听到这样几个名词， DevOps 团队要微小化一个披萨团队。那这个披萨团队通常就是 6 到 10 来个人，这个团队完成整个一条链从设计、开发、测试甚至于部署上线等等一条链某一个功能模块的完整的团队的合作。


这样一个小小的团队，他就要求他要管理一套系统，这种系统从他的前台到他的后台的中间件的调度以及后台的落地要统一管理起来。不是每一个这样的小团队里都有专门的运维人员的，你没法在每个团队里面都配一个系统管理员很了解 Linux 命令，很熟悉我们的CPU ，内存的调，系统的调优很熟悉我们 JVM 底层或者说操作系统内核里面那些方法的处理调优化。那同时他也对网络很熟悉，对我们的不同其他应用之间的接口很熟悉，那找这样的人其实就很困难。


也正因为这样困难，就希望在底下的 OPS 那一层就是我们的系统层，我们的网络层还有我们的调度层面，做到一个统一的平台化。这样我们的所谓的夹外架构是更偏向于应用架构，更偏向于基础架构。而可以忽略掉底层的那些系统底层的网络以及底层很多运维相关的，比如监控管理这些任务，这些任务放到哪里去呢？那后面我们就会仔细来聊一下怎么样在容器里面通过三大优势来满足这里差异环境差异大，应用启停敏捷以及组织架构变化所带来的压力和挑战。好，谢谢大家。
