一般情况，对于相同策略下数字越大，那么版本是越新，我们看一下 spring 的版本管理说明。首先我们看几个版本的例子，从这里面我们可以看到 spring 的版本可以分为主版本号、次版本号和增量版本号，加上一个版本后缀。我们看几个版本版本后缀的说明。首先 GA 表示稳定版本后缀代表是release，那么 Pre 它是属于发布预览后缀，代表的是RC。我们从这里面看到当前最新的稳定版本就是 5. 2. 9 release，我们看到 Pre 版本是 5. 3. 0 的RCE，它是一个预览版，并没有真正发布。


那么对于 SNIP sold 版本我们就不做过多介绍，对于熟悉 Maven 的同学应该知道， SNIP sold 版本是我们在开发过程进行联调或者说测试过程的一个快照版本，这个快照版本一般是不会被引入第三方的业务系统使用的。这里面还有一种情况，就是以 m 结尾的这种 m 是 Milestone 的缩写，也就是里程碑版本。里程碑版本儿跟我们的预览版有点儿类似，所以说我们可以参考这种方式。


还有我们看一下这里面主版本儿、次版本儿和增量版本儿这三个版本儿的一些关系。首先第一个数字是主版本，它是有可能进行大的架构调整，各大版本之间是不一定兼容的，就像spring，我们由 4 升到 5 这个版本，它中间会存在一些差异。那么第二个版本是属于次版本，在主版本架构不变的前提下，它会增加一些新的特性或者变化，相对来说是比较稳定和兼容的。第三个数字就是增量版本，这个增量版本大多数情况属于 bug 修复，细节的完善，它一般不会有兼容性的问题。

因为 spring Claude 项目是由多个独立项目集合而成，在早期每个项目都是独立的，各自进行自己的迭代和版本发布，所以 spring Claude 它不方便使用版本号来管理，所以说使用了一个版本名，以避免和子项目的版本号冲突。


版本名它最初的来源是以伦敦的地铁以字母排序，比如说最早的 release 版本为 n goal，第二个是 release 版本为Boston，通过字母顺序向后编排，我们可以看到现在通过a，b，c， d e f g h 它现在已经到最新版本是 h 版本。我们看到这个 s 版本，它的后缀有 SR 8，它是什么意思？这个SR，它也就是 service release，它是当一个版本的更新积累到比较多，或者是解决一个严重的 bug 时，会发布一个 service release 版本，也就是简称SR。我们注意到 SR 后面还有一个数字8，它是什么意思？它也就是说在 Husten 这个版本下面已经发布了 8 次更新的意思。


其实对于我们来说，我们看到这些以字母为前缀后缀的这种版本管理，其实我们理解起来还是有点骚恼，好在今年，也就是 2020 年， spring 官方它对于 spring close 的版本管理发布了一些策略变更，我们现在可以看到它有一个 2020 杠点儿零杠 m 3 的一个 Pre 版本儿，也就是说后面 spring Pro 的版本儿发布，它也会采用速制的方式去叠加的方式去发布版本儿。我觉得这种方式的一个版本管理，让我们更明确地知道版本之间的前后顺序的依赖关系，这对于我们开发同学理解起这个版本还是比较有好处的。
