我们可以从 githume 上去获取到 spring boot 官方的它的 stream 补的代码库，我们通过这样命令就可以把代码克隆到本地。这也没有太多要说的，这就是 Git 的基础知识，如果大家对 Git 还有一些技术上的不理解的话，可以先学一下 Git 相关的操作，当然因为在 Git Hub 上它下载内容是相对比较慢的，大家应该都可以理解，因为它毕竟是一个国外的网站，一些不明说的原因我们就不去介绍了。


如果说大家想让它速度更快一些的话，我们可以看一下 get e，它对于比较热门儿的一些开源库都做了一些镜像，这是 spring boot 在 gate e 一样的开源框架。 get e 也号称马云，这个名儿起得挺响，我们通它取克隆代码的时候，速度还是挺快的，一般能达到 2 到 3 兆的样子。如果说下面通过 GitHub 上去克隆代码的话，应该也就是几十k，甚至最多几百k。


我们把代码克隆下来以后，因为 spring boot 它本身是 gradle 去编译部署的，那么我们克隆下来千万去不要进行编译部署，也不要去导入idea，因为导入 idea 它自动就会去进行 build 编译。为什么呢？我们在编译之前，我们先把我们的分支切换一下，我们切换的是最新的 release 版本分支，也就是说 V 二点儿三点儿4，在这里面执行 get checkout 杠 b V 二点儿三点儿四点儿steady。


为什么呢？我们自己创建了一个用于学习的分子，创建完这个分支以后，我们才能进行去一些编译的操作。为什么代码刚克隆下来不去做编译？是因为我们默认克隆下来的是 master 分子。 master 分支其实对于这些开源系统来说，它是不稳定分支，并且它依赖的炸包都是 Snapshot 版本，所以说我们如果说用它去编译的话，会下载大量的其实我们用不着的一些 small channel memory 的扎包，并且有很大概率其实我们可能编译不通过，所以说我们一定要选择一个稳定的版本。在这里面我们会选择最新的稳定版本，也就是 V2. 3. 儿四的 release 版本儿，我们把代码儿 check 完成以后，我们去执行 Grid 的编译操作。在这里面我建议大家用 gradle test class 这样一个命令去执行。如果说我们执行 gradle 的话，它默认是 gradle builder，它会进行代码编译、打包测试等等一系列工作，花的时间非常长。其实对于我们来说，去探究源码的话，我们只需要把源码 check 下来，把它所依赖的用于编译的栅包下载下来就可以足够我们使用。所以说我们就执行 gradle test class，就把我们所有这个项目所依赖的栅包都下载下来了。当然我们也可以用点儿杠， gradle w test class 这两个是有区别的。前面 gradle 它是执行我们默认的环境的Griddle，那么下面这种方式它是用了 Griddle wapper 带这种包装的方式，当然这两种方式执行的结果是一样的，如果说我们的 Grid 版本也一致的话。


好，如果说我们把这些代码执行完成以后，我们就可以去对环境做一个验收的check，当然在这个过程其实并不会太顺利，因为我们在执行编译的过程中会需要从外网下载很多的一些第三方的炸包，这些炸包它可能会下载的速度比较慢，我们花的时间比较多，也或者是我们把 bug 下载完成以后，在本地编译会出一些各种各样的幺蛾子。
