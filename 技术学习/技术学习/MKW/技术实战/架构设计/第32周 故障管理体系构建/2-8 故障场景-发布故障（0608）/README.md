hello，慕课网对外带架构师们欢迎回来。在上下节里，我们为大家介绍的是数据故障。数据故障通常不会自己出现，通常他来说它都是围绕着代码的问题，或者是在整个线上进行数据操作的时候出现了问题。数据问题通常来说，如果可以通过订正去解决，通常可以消除或者取消。但是如果因为业务的原因无法订正，相关的业务方已经接受了相关的报表，产生了相关的契约，数据问题就会形成相关的一些故障，包括自损。这时候我们只能够通过其他的方式来进行相关的一些补偿了。

因为通常来说 60% 的故障都是由发布引。发布是我们将一个新代码发到线上的一个过程，在这个过程里其实是最容易出现故障。发布通常其实都是由需求或者bug，它来引发出一个变更。我在变更之后，我们需要把它上到线上去发布流程，最终完成把所有的代码发布到生产环节。在这个过程里，其实变更本身的问题，以及发布过程中操作问题，都有可能已经发布的故障。比如我们代码有 bug 没有测出来，比如代码和其他模块兼容性有问题，没有。在发布之前的这种回归测试中发现，都有可能在发布之后发现出现问题。

很多时候是代码的问题，比如我代码就写出 bug 了，在测试环境中没有测出来，或者是有些场景下，测试环境是没有完全百分百模拟线上环境的，包括一些场景可能也都没有办法百分百复现。所以在测试环境的时候就出现了疏漏，最终导致线上出现问题。


第二，可能会有配置的问题。比如你发布的时候，你除了发布代码之外，还要发布配置，而配布置往往一小步就会有大影响。比如你推了个配置，你可能所有机器一下子都被变更了，这时候所有用户都被影响到了。第三，顺序的问题。你发布其实是有依赖的，你先发，我再发，他再发。顺序有的时候是不可以颠倒的，比如你发布的内容依赖了，你发布的内容只有你发完之后我才能发布，否则我发上去的代码就会报错，一旦顺序错误，就会导致依赖缺失或者逻辑混乱之类的，从而导致相关的故障。


当然还有一块兼容的问题，新发布的内容往往与历史存在的一些兼容性的适配，而这种如果出现问题，其实往往是最难测出来的，因为大规模的回归成本太高了。并不是每一次发版都会进行大规模的回归，这时候一旦发版，就会出现了我历史版本不兼容，从而导致的相关的问题。

发布故障的发现其实有的时候实际是五花八门，因为它表象其实就是五花八门，各种各样的。有的可能直接就报错了。有的时候环境出现了一些奇怪的现象，比如我的内存开始慢慢变高，我的线程数开始变多，我的网络重传率开始变高等等。这些最开始的并不是直接体现成一个故障，而是在它发展到一定阶段时候才以故障的现实出现。而所以这时候如果能够在业务监控层面上做好，其实大概率是能够发现它的。比如我们发现我们的业务波动和我们的业务失败的情况，这时候就容易去发现相关的问题，业务的不正常变化都有可能是故障。当然如果这时候有发布，很有可能就是发布引入的。


第二就是应用的监控，来看看应用那些基准的一些现象，还有一些基础的指标，还有他是否报出来的一些异常错误，还有应用这些性能的一些指标，这些都可以帮助我们来看一看应用本身是否体现出来的一些问题。而这些问题其实很多时候是反馈出来的，只是我们发布的代码存在的相关的问题。而且应用本身的性能下降是一个常规的预兆。所以我们在发布完之后，不仅仅只关注业务的人，还要关注我们应用本身的这一类指标的变化。如果它的变化开始出现了陡增或者陡降，亦或是出现了随着时间的推移，逐步的变多，慢慢的逼近一个临界值，这时候都是我们需要去关注的。


第三环境的，比如像这种机器性的指标， GVM 那些指标，这些指标也是我们需要去关注的，它其实也意味着上层的程序可能会出现问题。环境的变化。环境的恶化很多时候都是由于带马波恰当的使用导致的。发布故障怎么解决？
