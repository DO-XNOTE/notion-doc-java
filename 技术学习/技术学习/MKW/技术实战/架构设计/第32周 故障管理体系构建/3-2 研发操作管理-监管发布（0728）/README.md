这种状态下就有可能会在这种发布高风险性的这样的背景下产生相关的故障。所以针对这种线上变更我们就要考虑到，因为它可能会影响到其他的线上功能，甚至产生问题后会直接影响用户，所以对其要做好对对于供应的监管。


第二就是代码变更，代码的调整发布上限。这些因为上去之后功能也会直接影响到用户，所以其本身也是具备高风险性，我们也要将其纳入到监管发布的范围里。第三，我们其实要针对这一些做好发布的测试，没有测试的发布是不让他发布的。这也意味着测试本身就是一类监管，并且测试也可能存在盲区或者未测出的部分。所以我们在测试之后，现场发布仍然也要走向灰度发布。这种具备强监管性的发布过程，并且能够将这个过程进行一个细分拆解，避免一次性的影响股东的涌。

比如在谁都不知道的情况下，你发了一个变更，但这个变更其实影响到5，但是我不知道，所以我并没有对做相关的处理。比如我可能要调整个开关，再切换一下链路，但是我并不知道，所以这时候可能我所有功能就不可用了，从而引发线上的崩溃。


第二，跳过测试一个环节，就是测试这一个。最后的兜底方没有了，我们就会将大量的可能存在问题的代码上到线上去。第三，我们尽可能的要避免魔鬼代码。什么是魔鬼代码？它并没有走一个发布流程上到线上去，而是在线上直接做相关的调整。比如我直接网上传了一个 class 文件，我直接在线上做了相关配置的变化，这些就是魔鬼代码，因为我发现我的整体的，我的所有的 Gata 里面，环境里面，我的 IDE 里面跟线上的不一样，整个的问题的发现和问题的修复都会非常的麻烦。


最后还有一些我们现在有些变更，它这些变更并不是持久化的，可能应用一重启它，这些变更就消失了，它就恢复回去了。这时候就会出现线上不同机器所表现出来的症状不一样，它就会出现一些奇奇怪怪的问题，这类都是过于放任自由的线上变更。这些变更带来的危害其实真的是非常非常大，它可能会带来一灾难性的结果，并且在灾难发生之后，我们还很难去找到原因并且修复它。


这些变更又会带来什么样的问题呢？第一，我没有任何发布记录，可能过了几天，谁在什么时候干了什么事，什么都不知道了，再去找到问题就会非常的难。第二，它也没有回滚方案，比如像魔鬼代码，等到后期去发现问题的时候，再怎么回滚，一点一点的在线上改回来吗？其实都是一些很大的问题。第三，没有人员值守，因为我们希望走相关的流程，发布的时候是为了通知到相关的其他方，我们业务的相关方都会有人值守，去看这些系统是否正常。但是你自由的发布的时候，他们是不知道的，所以他们也不会留人去值守。所以一旦出了问题，等大家反应过来的时候，问题都已经产生影响。


最后没有相关触达我们，剩下由我们所有合作方，我们的业务方，我们的测试同学，他们都不知道你发了，他们也不知道该什么时候去帮你去验收，等到真正出现问题的时候，还是用户反馈上来，这些都是很大的问题。这些问题都会导致故障概率大幅上升，不知道哪一天现场就出问题了。所以我们要建立好对应的发布监管流程。

第一，我们要通过系统流程去把所有东西管理起来。系统流程的最大的好处是什么？它是有系统记录的，我们谁在什么时间做了什么事，做了哪些事，影响了那些范围，我们都能够通过系统记录下来。第二，因为系统它流程，它可以挂相关的审批单元，就会有相关的人来审批，比如你的主管来审批，你主管来审批，或者你的测试同学来审批。通过这样一个审批流，大家都知道了你在什么时间做了什么样的事情，从而来保证你这个事情另外一方来帮你做好相关的监管。通过这样的系统流程串联整体发布，它就可以保证记录的存储，同时审批可以保证流程触达。这就是我们为什么希望通过流程来进行发布的监管的。


第二类就是发布的卡点，比如像代码扫描这样的卡点，像自动化测试的卡点，这类的方便。它可以保证这些不会少的环节都会在。比如自动化测试，如果没有测试，可能会存在着大量的问题代码。但是自动化测试最起码是一个最基本的保障，它可以保证代码的质量，本身这些基础质量是可靠的。


第三，我们通过发布窗口来进行相关的管理。第一，可以限定这个时间，这样子大家都知道哪几天会是发布日，对于发布日之外的都不会去发布，降低线上的一个变更的风险。第二，在发布期间会有人员相关值守来保证在发布过程里大家都去关注是否存在问题，发布之后有人去验收，从来保证整体的发布过程，发布的结果都纳入了监管之中，从而避免了因为过于自由散漫的发布而导致故障概率上升，最终让我们线上的概率下降了来完成了故障预防。这样一个效果好。


发布监管本身是一个很小的一个点，但是这个点其实在之前包括安全生产，包括质量管理，因为大家讲过很多次，但是在这里还是要着重的。再强调一下，发布监管是降低故障的一个非常有利的这样的手段，它可以让线上故障至少降低 20 个百分点。 20 个百分点可不是少数。假如每年线上的故障的造成的损失是 100 万， 20 百分点就是 20 万。我们再从其他的方面找，再去找一些概率下降，最终我们能够将故障概率下降到一个非常可观的数字。好到这里关于发布监管就给大家介绍完了，在下一小节里，我们为大家介绍一下发布窗口相关的一些事情。来介绍一下我们怎么样子通过时间限定来确保有人员职守，并且最大程度上降级发布带来的一些问题。好，各位的加工师们，我们下一节课再见不散。
