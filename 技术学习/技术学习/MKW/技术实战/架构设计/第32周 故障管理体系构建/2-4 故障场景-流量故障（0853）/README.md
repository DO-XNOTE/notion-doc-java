尤其是当我们想要去迎接，就像双十一， 6118 电商节或者大促，比如年货节等等，在这种节日下，流量会比日平常要高很多。这也意味着我们要怎么样子应对这些流量变化。超出预期可能会导致我们的系统过载，从而引发系统无法处理过多的流量，进而导致故障。这种超出极限的流量在流量故障中最常见的。


当然还有一种就是流量的陡增。饮料它就突然多了一下，又少下来了，但是这突然多了一下，可能就会把我们的群打爆。我们要客观的去看待这个东西。不管是超出极限还是流量陡增，都是我们本身的一个承载能力不足导致，或者是我们本身的数据库存在的承载不足。服务是代码和机器的原因，在数据库，除了数据库本身，机器还有链接池，意味着我们的 SQL 本身的性能。


导致。基础设施能力不足也是流量故障中很容易出现的一种原因。但流量最终都体现在用户行为，因此我们处理不了的时候，对于用户的体验影响极大，他不会感知到你系统挂了。大家都知道挂的词在用户侧是特别不容易接受的，他们会将这个点进行放大，就会导致进而舆论的增加。


他们会说你挂了你的系统，你们技术不行。但是你要知道任何一个产品一旦被打折了，技术不行，标未来抢 3 升，那就太难。而且流量故障很多时候都存在着雪崩效应，一旦他把你系统打崩了，所有人都用不了了。不是只说只有超出的那部分人用不了，是所有人都用不了，这时候影响就非常的大了。

第一个就是未预期的流量，流量的变化超出预期，我们原来预期只有 5000 人，结果来了2万人，这时候我们本身的设计或者资源是不足以扛住流量的，这种情况下流量故障就非常容易产生了。第二，我们本身的处理能力没有达预期，都以为我们的系统能扛1万，实际上到了 8000 它就崩了。这就是我们对于我们本身的一个认知的负责。首先就是系统瓶颈。就是系统中存在的一个瓶颈点，这个瓶颈点成了木桶的最短板，它先被打爆了，进而导致了其他模块也开始进入到相关的问题。第二就是热点的崩溃。系统中存在的热点资源，从而在短时间大流量下，热点性能出现瓶颈而导致崩溃。比如我们有 man circle、曼萨霍顿，就把我们的数据库给拖垮了。这时候整体的一个系统就进入到了不可用的状态。第三就是有资源的争抢，本身就有大流量在打过来了，而这边也还有其他的功能的争抢资源，比如 c 端正在大抢， b 端又在拼命的去进行库存的调整，进行下单等等。这样的就会导致双方进行资源的争抢，甚至在这个过程中导致了服务的承载能力大幅度下滑。这种情况下都有可能是我们的系统达不到我们的预期的这样一个处理能力，从而产生流量故障。

同时我们最关注我们当前的系统可能性了，因为无论是大抢还是大促，在这种状态下，我们都会去格外的关心我们的系统是否正常，我们的用户是否正常。所以故障往往时间都会被第一时间被发现。当然监控报警会告诉我们存在的一些问题，比如像流量监控，流量水位是否正常，流量成功率是否正常，我们流量水位已经很高了，结果成功率跌破了80%，这时候意味着存在着相关问题了。第二就是资源的监控，我们去看看我们相关的资源指标，比如 g v m，比如这些load，这些是不是开始出现飙高了，是否存在了大量异常偏离。我们看出现大量的 four g c，倒有可能是资源出现不够了。第三就是热点的监控，我们看看这个热点它是否会开始变多，热点的性能开始急剧下降，比如我们开始出现大量的慢circle，并且每一条 circle 的 r t 从原来的几十毫秒变成了五六百毫秒，这时候热点就开始爆发了。第二也是通过用户的反馈，用户会第一时间感知系统故障，所以用户的反馈会第一时间到达。比如你电商系统，它直接都挂了，用户肯定打爆了，为什么下不了单了，你们又挂了？这种反馈会很及时的，也会第一时间的作答。这些都能够帮助我们去发现流量的故障。

比如限流。我们针对流量入口进行相关的限流，使得进入整体系统链路的流量下降到安全水位之下。我们系统当前的旗舰水位是5000，安全水位是4000，我们将入口的限流限到了4000，这时候其实系统还留有那么一丢丢的余力，可以去支撑偶尔进来的这种偏高一点的流量，从而保证整体的系统响应是正常的。当然被限流挡在外面的用户也会经历到相关的失败的效果，但是他可能再重新刷新又可以进入到系统，所以只是体验上会有那么一点点差，但是还是可以用的。


第二，熔断。这种情况下，限流都已经救不了这个时候了，我们赶紧对这个问题进行相关的熔断，避免问题在系统内部进行扩散，导致整体的崩溃。另外一个就可以通过降级将一些存在的性能问题，或者存在着相关的故障的这些模块，但是我们可以通过一些降级方案，使它对用户的影响降到最低，甚至降到没有。通过这种急居保帅的方式保证主功能可用性。比如我们在大抢的时候，我们的库存模块被打爆了，我们是否可以对库存进行降级，返回来一个占位的库存，让用户能够下单。只要我们的库存足够多，本身库存显示不显示都是不妨碍用户下单的，意味着我们的主功能仍然可用。还有我们去重启我们的服务，让系统恢复服务能力。这种其实是一种直显，而且重启是有时间的，最好是在充气之前先进行相关的限流。


还有扩容这种，当我们的机器资源部署的这种资源已经不足以承载流量了，我们赶紧对其进行扩容，增加机器，增加配置，让其能力提升，解决掉性能问题。但是一扩张时效时间。第二，并不是所有问题都能够通过扩容器解决，扩容只能解决一部分，当然也有的时候你的再怎么扩，它也没有办法改变这些了，比如尤其当系统瓶颈的时候。所以更好的还是我们通过上面的这种限流容量降级来完成流量故障解决。
