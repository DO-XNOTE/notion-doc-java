我们面对于其中某些基础设施故障之后产生的影响，我们可以通过一些预案来规避掉，比如通过机房切换，比如通过主备的切换来完成我们在故障线程的逃离，从而保护我们的系统不受到更大的影响。这里解决其实最关键的一点就是故障逃逸的能力。我们如果能逃出来，就能活下来。如果我们逃不出来，我们就会跟着基础设施的塌陷一起埋在里面。


第二，多套的备案能力。因为你一套可能会遇到这个问题。假如你一套系统遇见这不少概率事件的概率是 1/ 0000，我有两套系统，它就是 1/ 1，它的概率下降得非常非常低。所以当我们有两套系统的时候，我们本身在面对这种小概率引发的故障的时候，它可以将它的概率下降非常非常多，从而能够帮助我们最大程度上避免因为其影响。


第三就是更多样化的应急措施，因为有的时候这类故障并不是完全作用于系统，可能会有更多其他综合性的影响。我们就需要针对故障见招拆招，多样化应急。而小概率实际上我这里就不再去讲它的定级，因为定级有的时候是因不拯而异的，有的可能小概率事件我们压根就不定级，而有的我们会根据我们逃逸的情况，我们的备案的情况，去考虑我们应该做好没有做好的这些部分，我们去做相关的定级。


好。介绍到这里，我们整个的一个故障场景就为大家介绍完了。故障场景是我们通过提前针对于一些场景进行预设，从而帮助我们更好的能够在故障来临的时候进行相关的定义，抽象解决，去提前设定好相关的预案，从而保障当月了故障来临的时候，我们能够最快速第一时间的从过程当中恢复回来，降低我们相关的损失。
