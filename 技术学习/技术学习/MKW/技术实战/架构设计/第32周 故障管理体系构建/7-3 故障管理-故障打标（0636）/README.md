它的目的是为了方便后面的管理和后续的回顾。比如现场发生故障，我们后面再去做整理。做回顾的时候，我们有没有办法不用一个过程去看，而是通过这些标记就能够方便我们快速的 get 到之前都有哪些故障，针对于我现在的目标，我可以怎么样子快速的筛选处这些故障，这就是打标了。


首先我们会有类型的标记，这是针对不同的故障类型进行差异化的标记，从而一是方面跟进，二是方便后续的我们进行分类整理。这样的过程里，有这个类型之后，我们很容易的就区分出来到底是哪一类类型，是流量故障还是研发的故障，还是因为基础设施导致的故障，我们快速的就能定位到这些故障，从而帮助我们进行相关的一些后续的管理。


第二就是原因类型的标记，这些其实是针对于故障发生的原因，取出一些标关键词，我们可以方便相关的团队及跟进处理。比如现有出现了一个故障，让故障的一些原因的标签已经打上去了，后面相关团队，比如受到影响的，被波及的这些团队，他们就可以通过这些关键词快速的 get 到这个是个什么原因导致的故障。同时在后续进行故障预防的时候，我们进行代码回归的时候，可以快速的根据标记来找到我们哪些故障场景是我们这次需要去回归的。


第三类就是处理的标记，这也是针对于处理过程的关键信息进行记录，明确出我们处理的关键点。我们在处理过程中，我们打了几个标，比如重启，这是一个标对吧，后人在看我故障的时候，去看我的止血方法的时候，他一眼就能看到了。这是通过知识充气来制写的。接下来一步我怎么样去还原现场，找故障的原因，从而明确出当时为什么去重启。


最后一类就是撤圈的标记，这一类是针对于那些其实本不是故障的故障，我们要对其做撤销，我们就打成一个撤销的标记。这样子别人就很明确的看出这一类其实之前并不是故障。我也可以继续看为什么当时被误报了，是我的报警的问题，是我的策略的问题，还是其实它本身有下跌？报警策略没有问题，但是实际上去查看是一个正常的业务的现象等等，都有可能。


这一些标题都可以帮助我们在故障发车时候进行处理，以及故障之后进行分类整理。同时在后续的故障预防中，也能够帮助我们快速的明确出哪些故障需要去回归做打标管理的时候，其实更多的还是我们要明确处我们打哪些标签。

第一类就是标签以短词的这样的一个标记帮助我们具备有关键行。第二类就是人，一个故障打上了一个处理人的这样的一个标签，就可以很明确的知道我应该由谁来跟进处理。当然这个人也可以没有，因为有时候人相对来说还比较敏感。但是通常来说这一类标签是在那种叫处理过程中的标签，而在真正处理完了之后，可能这类标签就删掉了，这样方便于故障它处理过程中的一个跟进处理。但是在处理以后，我们要避免有人拿着这个事来缺货算账，所以这个东西它是根据我们的管理流程来确定的。


第三个就是事儿记录需要去做的事儿，单纯的标签儿，其实信息过少，我们可以在标签上延展出来标签后续的一个具体适应结果，这样子通过标签可以快速的筛选，再点击标签，我又能延展出来。像当时的一些场景，相关做的人和处理的事以及处理的结果，这样子整个标签的延展出来的一些信息全都被记录下来了。同时光有标签也不够，我们还要根据标记去进行处理，确保故障的一个落地。

是否能够及时发现问题。第二，我们的策略是否需要去调整，通过调整策略来明确出故障和我们当前的这些行为是否能够有比较好的对应。第三就是告警的触达。我们的报警是不是触达到了应该触达的人了，还是他触达了一些不该触达的人，亦或是该触达的人没有触达到，这都是我们需要在后续去跟进的。同时围绕我们的响应是否及时，我们的响应链路是否去优化，也要根据标签进一步的去看。因为你可以在处理过程中打上响应不及时这样的一个标签，这时候就可以快速的去 get 到这些点，方便我们去做跟进处理。同时在处理流程中是否快速有效，而到我们过程，包括我们这些流程是否要优化，这都是我们可以在后续跟进的。同时。最后系统要做理财这方面的升级，同时避免后续的故障。


基于打标的跟进，打标是为了能够有一些短词，帮助我们快速的定位相关的信息，将这些信息传递给后续的人。但是复盘其实跟打标是相对应的，在复盘的时候就可以根据这些标签，针对我们线上面提到的 6 类关键信息做相关的优化和复盘。所以标签其实只是很小的一个功能，但它有的时候可以帮助我们在整个故障处理过程中快速有效地传递信息，并在最终在复盘的时候能够快速的还原当时的信息。好标签就是这样的一个简单的小东西。在下一小节我们就为大家来明确一下共创复盘都要复盘哪些内容好。各位未来教故事，我们下一节课不见不散。
