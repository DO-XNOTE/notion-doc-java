首先我们需要针对预案有一个设定，这样子当线上出现问题的时候，再触发到相关的报警，比如像我们的业务报警、系统报警等等，通过这些报警能够帮助我们感知到相关指标的变化，从而触发到预案的执行。或者是收到报警之后，由报警的救生人进行手动预案的执行，从而完成线上的快速止血，再进一步的做相关的排查和操作，进行现场的恢复。这就是整个的线上故障预案的解决路径。通过自动化预案可以帮助我们提高解决问题的解决效率。


其次，预案可能会根据具体发生的问题进行设定，所以预案其实是切合于一个场景的。我们在之前也讲了，预案其实是需要针对一个故障场景去做好相关的设定的。在报警触发时，自动完成预案，并且快速止血，同时根据恢复的情况选择降级常规路径的修复方案。这里是指如果预案并没有能够达成恢复的这样的一个效果，我们就要降级到常规的问题，修复佣金。所以这里的降级是指从执行预案降级到常规卫的修复路径，而不是指我们上一节课提到的降级。那个降级是一个预案的一种。但是这里要注意几个点。


第一预案不能覆盖全部问题。因为线上的问题可能出现在任何一个细微的角落，但是预案通常是针对核心场景、核心问题、核心链路进行覆盖。所以预案通常来说没有办法覆盖全部问题。第二就是预案结构也需要人的后置确认。通常来预案可能会包括人工的预案，也包括自动的预案，不管是哪一类预案，最终的结果都需要人进行确认。第三，预案它还是有好处的。针对于这种能够预案能够止血的这种问题，可不可以投入的人力较大幅度的下降的，因为它能够帮助我们进行相关的止血。所以我们不需要在发现问题的时候，再需要去跟根据相关信息去定位问题，再去查问题，再继续做相关的执行。这些工作都由于帮助我们来执行了。


当然第四，如果是通过报警相关的指标触发的自动化预案，其实存在一定的风险的。也许线上一个抖动预案就被执行了，执行完了，预案并不一定会自动恢复，这时候线上可能就会一直处于一个预案执行的状态，它可能会带来一些体验上的影响或者链路上的影响，所以要认知到这一个点，就是自动化是存在一定风险的。所以这也是映衬了上面点预案结果也需要人的后置确认。所以不管是自动护理权还是手动预案，我们都需要根据问题的发展逐步的完善，让线上的预案体系能够越来越好。所以在这里我们就需要结合我们的第一点就是预案叫场景化。

其次还有手动的预案。手动预案是指当现场出现问题的时候，我们根据故障的这样的现象以及指标所体现出来的问题，手动的进行相关预案的触发，来完成线上的恢复。这类我们就需要做好相关的面向场景工具，同时在过程中要做好强干预，明确出预案的执行，帮助我们完成了相关的恢复，并且做好人工的验收。


通常来说，自动预案跟手动预案都是相辅相成的，没有一个场景下说所有的事情都能够由自动预案来完成。当然手动预案中也可以随着预案执行的不断完善，逐步的转化为自动预案。将这二者共同起来，就能够帮助我们更好的在线上完成预案的执行。所以它二者之间的关系也是以互备的关系，同时也是一个从手动医院向自动医院不断的发展，又从自动预案中提炼出一些自动预案，没有办法覆盖特别好的沉淀成手动预案，二者之间也是一个互相转化的过程。


当然，无论是自动预案还是手动预案，人都是保障的最后一道防线，人在里面起到的作用是任何机器都没有办法替代的。

这是我们希望的能够在预案的制定与执行过程中达成的一个效果。所以预案维护它是一个在日常过程中不断沉淀的过程，我们需要针对指标做好相关的沉淀，在预案执行到之后能够完成风险的预防，并最终针对于指标沉淀意愿之星风险预防的效果做好一个评估，并且用评估反向的进一步的迭代我们能预案，同时在预案这个过程里，我们也要进一步的完善我们的问题发现体系。


在问题发现之后，能够自动或者手动的去触发相关的预案执行，再去评估预案的止血情况，从你也要评估预案的执行效果。所以这里的整个环就呈现了两个反馈循环。第一个反馈循环是指我们在预案设定，预案执行到风险预防的结果的这样的一个状态上，进行一个正反馈循环。而另一个是对线上出现问题之后，问题的发现、执行、止血效果评估这样进行一个正反馈取关。这两条链路都能够在日常化的这样的一个效果上，帮助我们不断的迭代我们的预案，并且不断的帮助我们更好地做好风险的预防和线上的执行。
