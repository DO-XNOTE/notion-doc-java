所以从系统的表象进行分析定位，从而明确问题的原因。在这个过程里，首先我们肯定是要面对的是整个故障的现场。根据现场中体现出来的现象，我们要去找这些现象，它体现的是个什么样的一个问题。比如下单的成功率下跌，这是一个现象，这个现象可能说明是我们的下单出现问题，这中出现更细节，可能是我们的添加购物车出现了问题，从而导致了下单的成功率下跌。


所以在这样的一个从现场到现想到问题的过程里，我们就能够将整个的问题进行更细的场景的定位，并且根据这些场景，我们去明确相关的指标，用这些指标来去指向那些我们已经预设的可能会存在问题的问题场景，有了这些场景，并且有了这些指标作为指导，我们就更加容易更加快速去找到问题的原因。所以这就是我们在系统诊断中需要去做的，也是它能够帮助我们去找到故障原因的思路。

针对于输入存在问题的，肯定是上游的问题，比如我的参数就不对了，肯定是上游哪里出了问题，所以他传给我的就不对。但是如果输入没有问题，他输出不对，肯定是当前模块的问题。就可以通过输入输出的方式来明确出我们出问题的模块到底是哪里，并且跟结合错误的情况来去看。比如我报了一些系统的错误，有些 now Ponder，我们就可以去定位 null Ponder。


为什么会出null，是因为它本身的逻辑问题，还是因为它传过来的参数不对，进而的一步一步的向上去追踪？其次，是否存在一些逻辑错误，就是这个数据。比如这次请求按理来应该打到 a 分支的，但它却打到了 b 分支，它在进行逻辑判断的时候，进行走到了错误的分支里去，肯定是有哪里。无论是代码错误还是传过来的数据错误，导致其逻辑判断出现了问题。当然还有一些可能是业务错误，比如直接返回了某些业务错误，比如返回了该商品库存为 0 这样的一个错误。这种错误不属于系统的一个错误，它属于一个正常的业务错误，但是它也可能指向这次的故障，可能适用于一个业务状况而产生，而非是一个基础的问题。

而且这里的指标更重点还是在于我们的非错误性指标。比如我们没有报了一些系统错误，也没有报portal，没有报一些参数校验的这些错误。但是我们还可以要去关注其他的一些指标，比如我们的基础指标，我们的 CPU 是不是打满了，我们的内存是不是满了，通过这些指标来看看它的基础性的这样的一个环境是否存在问题。


其次，去关注它的性能指标，比如整个请求的r、t，它当天的情况怎么样，它是不是已经超过了我们约定的这样的一个超时时间，所以导致了大量请求失败。第三，资源上的指标，就看我们本身资源它是否当前是满足的。比如我们的链接池，这是一种资源，比如我们的所有的基金的负载，它是另一张这种机器资源的一个情况。通过这些来明确出我们当前的环境、性能和资源是否存在了相关问题，从而推断出我们出现问题的可能的原因，并由这些原因进一步的去看看。
