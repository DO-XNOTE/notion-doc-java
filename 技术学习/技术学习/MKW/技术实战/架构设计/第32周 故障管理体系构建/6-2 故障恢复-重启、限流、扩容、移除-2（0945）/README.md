我们要去增加我们线上承载我们服务的这种服务器的数量，同时也可以通过增强它的配置来快速的完成记忆的扩容。
比如我们买的是云服务，我们买的都是二合 4G 的这样的一个e、c、s，当线上出现极限的时候，你去做数量的扩充可能还是需要一定的时间，但但是我们比如通过将两核 4G 快速的升至为四核8G，因为本身它都是虚拟机，所以它可以在一瞬间就完成相关的配置的升级。虽然可能会存在的一些闪断，但是相比较我们去增加机械数量来说，它是一个时间更短，效果更好的一种方式。但是如果本身来说配置已经比较高了，我们只有一种方法增加数量。除了机器之外，还有会有中间件相关的扩容。中间件比如我们通过 MySQL 或者是像 Redis 这种数量增加其实是比较少的。通常来说类似于那种分布式的，比如这种分布式存储，这种可以通过我们的数量增加来增加它的计算能力或者是存储能力。但是至于其他的都是在于配置的增强，比如我们 MySQL 配置的增强，通过配置增强来获取更强有吸引能力。


但是不管是机器扩容还是中间机扩容，不管是数量增加还是可以增强，我们的目的都是让我们的线上的能力得以增强，从而让线上能够承载更多的流量，使我们系统的响应的这样的一个能力提升到我们之前的极限水位之上，从而我们可以解除限流，让用户可以更加顺畅使用我们的系统。

这个故障是某些机器出现了问题，但是却不是大规模的出现问题，所以这也意味着我们使用移除的时候有一个前提，它的故障部分是非阻断的。出来故障之后，我所有都不可用了，它只是其中的一部分不可用，或者一部分环节不可用。我们可以采用移除这些环节，或者移除这些资源，使得线上恢复到一个完全正常的百分百响应的这样一个状态。同时它也要求整体的能力是可备份的，比如之后这是个唯一的不可备份的这样一个能力，一般移它就没有了，它也不是可以使用移除这样的一个方式的这样一个手段。


其次，它可以被降级，比如我把它移掉了之后，它可以降级为一个默认的实现，或者降级到第二出口的备份能力上。这时候我们可以把出现故障的这部分移除掉，它也不存在这样的相关的一些副作用。所以如果这 3 个前提不存在，没有办法使用溢出，只能采用之前的限流、扩容这些方式去做相关的一个缓解。并且遗书通常面对的是一个局部故障，同时我的整体没有过载。这也意味着我移除掉问题部分的时候，不会导致整体系统马上进入到一个崩溃的状态。是在于问题部分被移除掉了之后，它就开始恢复了。所以移除就像什么，就像是我们整个系统中存在了某些问题的问题，像一个肿瘤，把它割掉了就好了。而不是在于我整体来说是一个窒息或者心脏停止跳动。你这时候割了什么都没有用。所以一处不扩张部分可以通过在这样的一个背景下，帮助我们进行系统的恢复而移除。

其次，移除具备一个可回滚性，我移除了发现不行，我可以把它挂回去。所以这样的一个回滚也是非常快速的，也可能也是秒级的。但是不管是移除还是挂载回滚，都会要面临的一个节点，就是移除之后，流量其实是会倾斜获取的，清洁完了之后，原来正常的这种节点是否会因为这些多出来的流量而被打爆，需要评估确定没有问题的时候才能够实施相关的移除。第三个移除的好处就是它的车造成本很低。通常都知道一步，比如 SL 的从 SLB 上移除掉，我可以把权重直接调成为0，这时候它一瞬间就可以完成相关的操作。成本很低。但是前提也是有另外的它是否存在这些写死的熏制。比如我在马里就指向了IP，这样你移除掉了之后，它直接就报服务不可用了。这种情况下就会没有办法使用相关的移除。所以我们尽可能的在线上不要使用一些写死的 ip 地址，而使用一些域名，或者使用一些 DNS 服务，这种情况下，即使我们产生了移除的这样的一个操作，也不至于直接线上就开始报错。而通可以通过负载均衡来完成相关的倾斜。
