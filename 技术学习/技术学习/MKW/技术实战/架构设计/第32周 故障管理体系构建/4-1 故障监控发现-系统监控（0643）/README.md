hello，慕课网的各位未来架构师们，欢迎回来。在上一小节，我们为大家介绍的是发布前测试。发布前测试是我们面向于发布故障这一种我们最常见的故障的最后一道关，我们也希望把这道关守好，从而让我们在发布之后所面临的故障能够大幅度的下降。


在这一小节，我们开始聊一聊故障的监控发现。作为故障监控发现，不得不去聊一聊所谓的系统监控。我们这里用一个案例为大家去讲一讲什么样子的一个监控是我们希望的一个系统性监控。

案例是这样子的线上从昨天夜里开始，出现 g c 的次数变多，同时随着时间的推移， GC 的情况越来越恶化，频繁地出现负GC，系统体感明显卡顿，这是一个线上的故障。这个故障就是用户用起来感觉特别卡，说明这时候还没有完全形成一个巨大影响的故障，但它已经开始有了故障的潜质了。这时候我们肯定会有对应的一个监控。这里监控什么？按照这个场景，我们会有基础设施相关的监控，比如 g c 的减库， G7 的load，还有 s t w 时间，就是 stop the word 的时间。这三个都指向了 g c 的一个问题，因为 g c 监控本身就会去监控我们 g c 的数量，包括 young g c 跟 four g c。


stop the word 也是 g c 监控中的一环，只是它是单独拎出来的一个停止时间的这样的一个统计。而机器 load 则是在于平方的数据线，g、 c 机器的 load 自然会上升。

我们先去摘下来一台机器，比如从我们的s、l、 b 上或者n、g、 s 上把这台机器摘下来，作为一个现场把它保存住。其他的做一下集群重启，分批的一批一批的把集群做了一下重启，先让线上能够有恢复，因为重启了之后，整体的内存水位下降，整体的 GC 情况就好转了。这时候我们要继续去观察整个 g 群的g、 c 的情况，因为整个重启之后， g c 水位恢复了之后，整个的肌肉体感又恢复到了之前的状态。但是随着时间的延续，我们又发现 GC 的次数随着执行时间的退移，又开始逐渐的增多和恶化。但这个恶化的时间不是一下子就恶化了，它是一个漫长的过程。所以在这里我们发现我们还有几个小时时间可以去做原因的排查，

首先去做机器上内存的一个分析，因为那台机器当时的进程还在是吧，我们还仍然可以去看它当前的一个内存的情况，发现大对象占比是偏高的。我们又去看变化趋势，持续性的是一个增高。这里内存分析我们可以通过当时扎下来的现场变化趋势，可以通过线上仍然在跑的这些机器去看它的整体的 GC 的情况，发现整体的趋势就是这个样子。


我们又根据这个情况去想近期有没有发布，去做医药排查，发现前天夜里存在发布，我们就怀疑这个问题可能是发布引入。因为在近期没有其他的操作，没有其他线上变更，只有前天压力做了一次发布。所以这时候我们先做相关的一个直线做发布回滚。之后发现问题恢复了，线上的内存和 GC 的情况都不再出现，说明问题是由发布引入。


我们继续的去做这个问题分析，因为怀疑是代码的一个问题对吧？因为你像整体的 GC 变多，同时发布引入了这个问题，这肯定是代码的问题，我们就去做整体的代码的一个分析。最后发现代码是存在的一些问题，当大对象做到结合来在方法链中持续的调用，并且整体的链很长，大量的对象它经过了整体的 e 等待和这种央企的次数之后，并没有被回收，进入到了老年，从而导致了大对象在整个的内存占中占比非常的高。所以以在产生了我们分析的结论。

本身来说是从系统中的一些监控，我们去发现了一些技术设施层面上的问题，但是这个问题虽然表现出来的是一个显著的系统偏离，但实际上是一个代码的问题，那也意味着我们很多时候，一旦基础设施的一个野猪偏离，需要关注。第一，基础设施是否存在问题，但大部分可能事实表象。其次，我们还要深究代码的问题，因为大部分来说就是我们代码使用上的一些问题。在这种场景下去看，到底有没有在近期引入发布。如果有发布，大概率是因为发布中的代码带来这个问题。为什么这个问题在发布之前不会出现？正因为在测试环境，或者在预发布环境，我们不会有这么大规模的一个使用，也意味着线上它的一个内存变化。可能在测试环境，它是一个很较经历很久，比如好几周才能够体现出线上 1 到 2 天的一个变化。所以在测试环境，预发布环境，我们没有发现这个问题，我们有什么方法可以发现？其实我们可以在预发布去做一轮压测，针对于单台机器的压测，可能量还相对来说不会很高，但是这台机器面对压力下的整体的变化很快就会发现，所以整体的结论也是我们希望是以小见大。


我们通过这样一个案例，让大家发现系统上的一些监控，它其实最终反映出来的还是我们在研发过程中的一些问题，整体故障排查的思路也是如此。好了，到这里系统监控的这一个点就为大家讲到这里。在下一小节我们为大家讲一下业务监控，我们怎么样子通过业务监控能够去发现相关的故障，并且怎么样子去止血修复。各位未来交互师们，我们下一节课不见不散。
