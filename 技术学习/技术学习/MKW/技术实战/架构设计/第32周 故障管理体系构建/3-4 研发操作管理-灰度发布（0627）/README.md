首先它还是回到看我们已经非常熟悉的这样的一个发布链条，它肯定是从需求问题到变更，再到准入发布，在发布过程中进行相关的灰度，最终到达生产的状态。灰度发布也就是我们从待发布到全量发布中的 n 个环节，每一个环节进行相关的调整，最终顺利完成全量的发布。所以规则发布的定义就是在待发布到全量发布中，以一定策略进行逐步的发布的过程。

在验证没有问题的时候，完成阶段的移动，比如用户群体的扩大，比如功能的一个扩大等等。一旦出现问题，则叫进行回滚。它的目的就是确保每一个阶段都能够明确的验证功能的情况，在最小影响的范围下发现问题并致死，回过来进行故障预防进行结合。一旦在灰度过程中出现了问题，那就意味着它有可能继续恶化，形成故障。我们就要在这个时间点上，在用户范围内去明确是否存在着问题，一旦存在问题，马上回归。这时候我们影响到的只是我们已经灰度的这部分用户，不至于产生过大的全方面的影响。这样子我们故障的影响范围就被缩小了。比如我们才忽略了 5% 的用户的时候，我们就发现了问题，我们完成了回滚。我们最大的故障影响就是5%，它不像是我们一次性全量发布，我们可能一下子影响百分之百，影响就从一下子从百分之百降到了5%。而 5% 往往是我们相对来说更容易接受的这样的一个影响损失。

比如正如我们之前为大家介绍了那么多场景，这些场景是否会与这次发布相关联？同时，我们在整个的这样的一个线上系统中，我们的日埋点是否能够契合我们的场景，我们的定向监控，能否当我们的场景出现问题的时候，第一时间告诉我们，完成问题的发现与触达。同时能够以一个故障预警的方式触达到相关的研发同学，这些研发同学就可以完成相关的接受与处理。


所以这里需要关注的一些指标，比如像异常率，像异常次数，像整个异常的持续时间，影响的用户数，产生的止损金额，这些都是非常直接的业务影响指标。而这些指标可以帮助我们与场景结合，最快速的第一时间的发现问题，从而在灰度过程中就能够以最小的影响用户范围把故障发现出。
