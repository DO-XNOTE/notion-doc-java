即使故障的影响面其实不是那么大，比如只是有一些功能不好用了，但是却产生了相关的舆论影响。有些人因为这些故障开始去讨论，比如微博上就开始转发了等等，并且随着时间的推移，开始逐步的升温。最后因为舆论的扩散，导致更多的人涌入去使用相关功能，并且进一步导致故障扩大。同时，因为舆论的影响，即使故障影响力有限，而故障也会因为舆论的加成而导致故障升级。所以我们要做的就是，当故障发生的时候，一方面要控制故障的影响面，其中就包括尽可能的快速止血。


其次，针对舆论上已经对故障产生出相关的一些发酵的时候，要做好舆论的干预，避免舆论进一步的发酵，从而从舆论角度去提扩大故障。当然还有第二类叫做舆情故障，这一类其实是指舆务舆论导致的故障。

或者是一些功能上的理解，或者是设计上的，有一些方面导致在使用层面上会产生一些误解。而这些物件进一步的产生出了舆论，并且通过舆论的影响不断的扩大，最终由舆论反向的做了一个故障定级。

我们可以去做相关的一些关键词的抓取或者统计，既包括我们的产品方面，也包括跟我们的公司相关，也可以跟品牌相关，整体来说和本身事件相关的，以及延展出来相关的一些关键词的现在的一些热度。同时也包括一些关键人物，比如因为公司某些关键人物说了什么话，会被这个事情对号入座了。我们也可以去将相关的一些数据，包括关键词延伸之后，背后的一些数据进行提取。


针对相关的舆情进行抓取，包括我们的一些新闻资讯，社交媒体、传统媒体公司的一些留言，工商投诉等等。将这些信息尽可能的聚合起来，提炼出其中有助于我们进行舆情控制的相关信息，生成相关的舆情报告。其中肯定包括我们的一些舆情的数据，未来的舆情的一个发展的预测，以及这个事情就相当于是个突发事件了，我们针对突发事件做好前后中后的分析，通过这些方式帮助我们公司的管理人员进行相关决策，决定我们怎么样子对公众发声，去影响相关的舆论，从而逐步的将舆情进行降温。


是相关的影响过去，最终让公司回到正常的运作中。本身来说，不管是通过技术故障导致的舆情，还是通过舆情反向定义出来的故障技术，都能够以技术手段在里面帮助整体的世界能够更好的终止，让公司回到正位，这就是我们本身的可以去做的舆情监控相关的工作。好了，到这里故障监控与发现几个小节就带大家讲完了，在下一小节我们进入到故障定位与分析这样一个小节，在这几个小节里，我们将为大家聊一聊怎么样子，通过一些指标和一些手段，帮助我们更快速的定位出故障，去推进故障的执行。所以，未来加工师们，我们下一节课不见不散。
