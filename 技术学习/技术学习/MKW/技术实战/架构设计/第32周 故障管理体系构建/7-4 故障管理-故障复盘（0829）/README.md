当然首先第一步就是要做好细节的采集，将整个故障过程中所有的数据，原因及影响都能够记录下来。这样子在后续做复盘的时候，我们才能够有据可依，才能够知道当时到底发生了什么，当时的各种细节到底是什么。


其次，要做好过程的分析，明确整个过程中故障它到底是一个怎么样的演变过程，以及我们在处理故障的时候，我们又是怎么样的一个处理过程，从而来判断我们在整个过程中我们的行为，我们做法是否准确而有效。第三，我们要明确在故障复盘之后，我们要用哪些后续事项要做。比如故障引发的原因，系统的一些问题，我们是不是要做好相关的代码修正和升级等等，这些后续的彻底修复计划到底是什么样子，它又要该由谁去跟进，从而确保这个事情的落地。其实这就是我们在整个复盘过程中需要去考虑的一些范畴。

以及故障发现的我们的这些能力。比如我们的监控报警，我们的这种场景，红线的场景是否都覆盖到了，是不是我们的一个疏漏，还是它是在我们的预期的一个场景范围内，但是它的确是发生了。


其次，在故障发现了之后，有没有有效的通知到了相关的处理人，比如他场景负责人，系统的开发人，架构师团队的TL，有没有被集成通知到大家有没有第一时间接手并进入到故障处理？第二就是故障的止血，在整个故障的处止过程中，当故障已经通知到了相关人，接手了之后，是否有效的完成了相关的致谢，是否是通过预案，还是通过他根据当时的特征见招拆招想出来的这些方法，但体系是否产生了效用？以及他在这个过程中他是否做到了快速，比如他到底几分钟完成了止血？其次，他的止血的方式和他思考的路径是否是最正确的，他是不是蒙上的？还他的确是根据相关的表象准确的找到了原因来完成了修复。


我们不是来确定他行为是不是对的，我们更多的是来明确按照他之前的知识储备，按照我们之前的一些演练，或者是平时的一些积累，包括我们整个的故障发现与直接体系中的一些细节的设置，我们是不是能够有效地指导我们的故障恢复，而不适于的确是当时不知也不知道怎么做的，反正他突然就好了。但这样其实对于后续是没有任何价值的。所以我我们还是希望能够从中找到原因，吸取相关的经验。


而在故障恢复过程中，我们的恢复方法是什么？用了多少时间，是否有效的完成了故障恢复止血了之后，其实是保证它没有恶化恢复，还是需要它继续赶紧回到正常的状态。这里我们也要看我们的恢复方法到底是什么，它是否是真的是有效，而且是快速的。而在过程之后，我们也要看看结果。


结果就包括第一方面故障的影响，故障产生了什么样的影响，这个影响的范围是多少，运行的幅度有多大，它是否有存在的，连带的资损、客诉、舆论等情况，这些都可以来界定故障的影响有多少。这些也最终来确定了故障的定级。同时根据这些把我们等级确定出来，明确出到底故障的责任方是谁。


故障定责有几个方面要考的。第一个，它肯定是由故障的引发方来做，来背这个责任。比如 a 做了这个系统，但是 a 本身不会引发，但是因为 b 去引发了故障，责任方在于b。其次，历史的故障通常采用历史故障卸任背责。如果历史的故障当时没有出现问题，但是现任维护中，虽然他没有改动任何代码，但是仍然产生了相关故障，但也由现任的负责人来定责。

以及针对于产生了相关损失，我们要去做到损失的追回，将损失降到最低。这也是我们需要去做好的。因为你故障如果不恢复，你下次还有人会踩这个坑，还仍然会犯相关的错误。而损失来说，我们肯定是损失越少越好。既随着损失的追回，故障也能够进行进一步的降级，从而对于双方，对于故障的引发方以及对于公司来说都是好。
