在上一项这里，我们为大家介绍了一下流量故障。流量故障是指公司在大强大促之中，关键时刻面临到的流量高于了我们的预期，或者高于了我们的系统承载能力，从而导致了我们系统出现过载而无法正常的服务我们的用户。这种故障通常都是在很关键的时刻，所以它的发现很容易，但它的解决就很考验了我们有没有预留好对应的限流、降级、熔断等能力，从而更快速的去保护好我们系统，降低入口的流量，从而系统能够恢复其处理能力。在这一小节里，我们为大家介绍一下关于环境故障。

环境故障通常也包括很多，比如像服务器中间件、存储、计算这些非业务相关的服务出现故障。其实我们每一个系统都会依赖了大量的基础环境，而这些基础环境当然也不是一定百分百是没有问题，所以意味着环境本身可能就会出现故障。比如我们的磁盘坏了，我们的内存坏了，我们的网络坏了都有可能。


第二，我们本身服务其实是他的问题导致的故障，但是最终体现出来是环境本身不堪重负了。这时候其实就是很常规的，比如像运维变更的一些问题，运维去调整网络结构，报告人可坏了，导致服务访问不到了。这种是很常见的一种环境本身的故障。第二，不合理使用。比如在线上跑了一个特别好计算的脚本，把 CPU 打满了，整个的一个业务系统可能就跑不起来了。第三，压力过高。这种比如流量下我们的本身的机器过载了。这种情况下也体现出来的是环境故障。环境故障其实很多时候作为业务开发人员接触的不是那么多，因为环境通常由专线人士来运维，所以环境故障也会由他们来解决。但是对于使用方来，我们还要关注一个点，我们怎么去逃逸。比如当一个机房它断网了，我们怎么样子能够快速感知到它切换到另一个池塘去，从而让我们的系统从故障中跑出来了。


其次，很多时候表现是技术环境故障，但是实际上其实技术环境它本身没有问题，而都是我们的本身对于环境的使用或者代码身出现问题而导致的这些故障。环境故障怎么产生的？
