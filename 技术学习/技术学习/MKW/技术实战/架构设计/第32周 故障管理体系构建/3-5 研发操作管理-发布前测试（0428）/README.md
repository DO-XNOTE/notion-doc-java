第二就是回归型的测试。验证是用来验证我们要发布的这部分功能，而回归是用来验证那些我们不会去影响的这些功能。我们不希望把一些原来的功能影响到。通过回归可以帮助我们看一看他们是不是正常。通过这两类基本上就可以把整个的系统进行包括主。所以测试保证功能正确，是故障预防的主战场。这也是为什么有研发一定会有测试。两边是携手共进，共同来保障我们线上稳定质量的一个兄弟团队。

第一点就是我们在测试过程中 bug 是收敛的，我们不可能在线上还存在着bug，还在不断发现这样的一个状态去进行上线。我们一定是发现没有新增 bug 了，并且历史的 bug 已经开始进行相关收敛，或者是已经收敛完了。剩下的 bug 就是一些 fix in later 的这种，或者 want fix 的这种状态。在这时候我们才能够确保诶我们这个东西可以去准备上线了。毕竟在上线前需要 QA 去确认他们认为当前这个状况是否可以进行上线。如果他们确认没有问题，他们就要出具测试报告，在看到测试报告之后，研发就可以进行相关的上线了。


bug 收敛的目的是为了确保我们在上线之后不会再源源不断的 bug 出现，所以 bug 收敛是发布的前提，而 q a 确认它这是1，其实是一个环节上的一个点，这个点可以让我们确保整体功能是可用的，是有一个人他来负责，他能够以他本身的职业去作为整体的一个担保。所以最终的测试报告就是上线的一个凭证。
