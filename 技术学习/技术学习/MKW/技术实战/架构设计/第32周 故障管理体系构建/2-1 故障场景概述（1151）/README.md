整个的响应或者我们的能力架构其实也是对应的场景。第二，我们的业务也是多样，我们有各种各样的功能， ToB 的、 ToC 的每一种业务的一个点可能也是对应一个场景。第三，在复杂的架构、复杂的业务下，我们就会有多种多样的产品，每一个产品也会对应到第一个场景。这也意味着我们的场景随着我们的架构、业务和产品的增多，它本身的场景是成多样的。而我们其实需要去做的就是在这种多样的业务架构和场景中，明确出我们本身故障需要关注的场景。


所以这就是所谓的场景句话，要明确出我们故障要关注的到底是哪一个点？场景的多样化也意味着我们可能会出现的故障就会越多，因为你越复杂，它可能出错的概率就越高。任何一个出错都可能有更多样化的影响，这些都意味着故障越来越多。而通过故障和场景，我们也希望是能够明确出我们战略的故障到底匹配到了哪一个场景，进而我们能够针对这一个场景做更深入的探讨。


而这对场景这句话来看，我们也要明确出我们的场景它到底是一个怎样的分级。比如这个场景是一个很核心的关键的场景，还是一个非核心的场景，它最终能带来的故障是什么样等级的故障？只是一个 P4 的故障，还是 P1 的故障？其次就是场景的分类，我们要明确出它到底是影响 ToC 的，还是影响内部人员的还是 ToB 的，每一种场景可拿到的分级方法，它的监控方法都不太一样。


第三，要明确好对应的场景关联。我们面向于这样的一个场景，或怎么关联到对应的一个指标，这个指标又怎么样子反馈出一个故障。通过这样的一个方式，就能够让我们更好的去明确出我们的场景要怎么样去设计。

这三个项其实都是相辅相成，我们要去抓快最重要的一个重要程度，以及它们的影响面和影响程度这三个点，就能明确出一个场景，它到底会产生哪些影响，它的影响有多大。


而在重要程度中，我们就要去看它到底是否是对应的核心功能，还是它只是一个辅助功能，或者它只是边缘功能。因为制定核心功能肯定是必不可少的功能。这对于辅助功能来说，它是核心功能中可能会有相关的影响的功能，而对于边缘功能来说，可能是可有可无的，没有也有替代方案。针对不同的功能，它的重要程度当然就不一样。你的核心功能是缺一不可，但是你的边缘功能，我大不了先不用它，我用别的方式绕过它，或者是它本身产生的影响有或者没有，微乎其微。而这对影响面来看，它其实也涉及到几个点。
