由于没有考虑全面，从而使得产品在使用过程中存在着一些潜在的缺陷。这些缺陷不是因为bug，也不是因为代码没写好，而是在最开始的时候就没有考虑到可能会有这种使用场景，从而导致用户智商一用的时候发现不对劲了，出现问题了，流程没有办法走下去，卡在那里了。这种场景就会导致了相关的故障。


而这种场景其实也有一些特点，比如第一个特点难修复。设计的缺陷意味着我们要回到最早我们设计阶段去看它，所以也意味着可能很多后续的设计都要被推翻。这也意味着修复的难度非常大，这是修改面非常大，因为其整体性被打破。而且因为我们要修这个的时候就要往里加补丁。但是由于是设计的问题，所以我们在代码相关的编码的时候也会被其设计的思路影响，所以补丁很多时候也很难把所有的面全都盖倒，这里的补丁也容易引入更多的BUG。


第二就是修复的成本比较高，难修复，所以修复的时间就长，你长就意味着它损失就一直持续的在发生，所以它的成本就比较高。所以我们要明确出在什么情况下设计缺陷或发生。当然第一点就是客观的因素，

业务可能会有很多的面，每一个面都需要去考虑，而考虑的状态下就意味着我们很有可能疏漏了其中一个点。所以业务本身的多面性会对设计提出了较高的要求。第二，我们的场景可能也很复杂的特点，也要求设计人员全面的考虑了场景的各个细节。所以这些客观因素会导致他对于我们产品同学的要求会很高。所以就推出了我们的第三个原因，就是我们的人员能力可能是产生故障的一个原因。长期的复杂性，业务的多面性都要求设计人员去全面的考虑各个细节，这也意味着对他的能力要求是很高的。但是可能公司的产品同学并不具备这样的一个能力，这也就导致了他们可能会在设计的时候出现失误。除了客观因素之外，我们还有主观的因素，比如局部的调整，因为我们本身去调整一个点的时候，都是一个局部调整。很难说我们从整个产品的视角全面的去考虑这个问题，这时候就容易忽略到其他部分对于的影响，合伙者部分对于其他部分的影响，从而在设计的时候没有考虑全面，产生相关的缺陷。


第二就是设计视角不一样，这个产品它是个财务向的一个产品，而另一个产品是流量向的产品，但它俩产生了一些交集，所以每个人的视角不一样，是考虑的范畴都不一样，所以也就容易比较片面，最终也容易产生相关的缺陷。最后也和人员能力相关的。思考的一个全面性设计。人员的思考全面性，决定了他是否能从全产品角度去把控质量。如果不能，很容易产生相关的设计缺陷。但是不管怎么样，设计缺陷都是很容易产生的。我们要正视于这个问题，因为其主观和客观性的一个原因，所以大家要能够接受他客观的存在，并且尽可能的去快速的响应，去解决相关的一个设计的缺陷。当然要提到设计圈怎么样去发现，

当然如果能在功能回归层面上去发现，当然是最好，因为这时候意味着我们还没有上线，或者刚刚上线，我们也可以最快速，也可以最低成本去完成相关的申诉。


第二就是业务的监控，上了线了，业务就会报警，发现因为设计原因而导致的相关的偏离。比如我们本身是修改用户注册，原来的用户的注册比都在30%，在上了一个功能之后，突然间就掉到了10%，我们就会发现是否存在的相关问题，进而去看看是否存在的相关的设计缺陷。


第三就是用户的反馈。用户反馈使用产品的时候真不好用，他们通过客服来告诉我，其实是最直接的问题发现，但是有时候也容易藏在这种海量的用户反馈中，因为用户可能会有各种各样的反馈，反馈很难全部都到了产品的耳朵里，所以很直接，但是也容易被忽视。
