好了，默克网各位来架构师们欢迎回来，在上一小节点我们给大家介绍了一下环境故障。基础环境的故障其实不常见，通常来说一旦出现了相关指标的偏离，也大部分是因为我们代码不得合理使用而造成，但是一旦技术环境出现故障，就是比较大的故障。所以作为业务系统的架构师来说，我们要考虑的不是如何避免技术环境故障，而是要如何在基础环境故障的时候能够正常地逃逸出去，而其他的就是怎么样的让我们的代码能够更加优化，而避免由于代码的问题而导致技术环节的故障。那么在这一小节里，我们就为大家介绍一下关于应用的故障。

是指应用本身的配置、部署、应用之间的关系等导致的故障。那在这一类里面主要其实包括两类，第一类就是应用自身的问题，比如说应用的配置错误了，应用假死了才被进程出现问题，然后应用的部署的一些问题，当然还有应用之间的一些问题，比如说应用我提供这个能力有了问题，其他的应用没有办法调用我了。


第二就是调用关系的错误，我本来应该调 a 的，结果调到了 b 了，或者是说我对我的应用做到分组，我应该调到分组 a 的，结果调到了分组b，或者分组 a 不存在了等等，这是我们调用关系的错误。第三就是拓扑和不合理，通过不合理可能会导致某一些本应该是第一流量的应用被打到了高应用上，或者应该是边缘的应用被放到了核心的应用位置上，而边缘应用本身的保障密度都是不一样的，所以就可能会导致其出现故障。应用的故障一般不常见，但是一旦出现了比较难排查，主要是在于就是它的有时候的出现无迹可循，所以就不知道为什么出现这个错误。但是虽然说它表单白茬，但修复成本并不高，恢复也相对容易，比如像假死这种可能冲击一下就好了，只是在于说你如果你真的要完全跟解掉它，你需要持续的排查这个东西相对来说就比较难了。

因为应用可能会有很多的配置，包括代码的配置、中间件的配置、环境的配置等等，任何一个配置如果出现错误都可能会导致异常行为，比如说你数据库的配置配错了，你可能数据库就连不上了，本身应用所支撑的这种能力就不存在了。


其次就是比如说部署设计，诶，我们本身要怎么样子去考虑我们的服务的部署？而且由于我们大部分现在都是微服务化了，微服务的部署使整体的部署结构越来越复杂，这也意味着部署设计不合理就很容易会导致故障。因为我们现在很多时候，比如说就云原生了，我们在 K8S 写 ML 去做相关的部署设计，如果写得不合理，就有可能会导致应该高流量的，其实结果分配的节点就不够，而不应该高流量却纷纷到大流的机器导致了浪费等等。而这种情况下，一旦流量变高，很有可能就会出现故障。


第三就是链路的调整，因为链路也不是一成不变的，它也是随着架构的不断地演进，不断地升级，不断地迭代，一直在调整，这也意味着调整如果没有考虑好，就有可能会导致故障，它中间就有可能会出现链路不通或者链路打错。当然也有代码性的问题，代码问题一个导致一些底层的，比如内存泄露，比如for、 GC 等等，这一类就可能频繁的出现，就会导致应用级别的一些故障，比如说我们这个应用它的延迟就很高，或者应用它的那个出现了未响应的这种状态，那应用故障的发现其实这类会体现的比较清晰，也比较明确。

基本反映在能力、服务链路或者是应用的不可用。因此特定的异常其实是最容易去发现的。比如说你就报了一堆的服务不可用，或者是这种被响应的这种请求，通常可能来说就是一个应用的故障。那这类我们也可以通过应用监控去发现，比如说像中间件无法连接，比如运用地址不存在，比如说如果是RPC，那 RPC 目标方法不存在，然后connection， RESET 这种通常来说都是应用故障的一个比较清晰的代表。


第二就是说我们通过一些基础监控来识别，比如说像机械机器的一些指标， GVM 这些指标之前的反向来证明应用出现了某些故障，而应用故障的解决通常也比较简单。

同时比如说服务的重启，针对一些常规的应用故障来说，我们从重启就只写，然后包括我们通过重启将一些未持久化的一些更改，也能恢复到它原来持久化的一个状态，也能解决掉一些，因为我们不知道到底哪里配置被谁改了，但是实际上没有持久化，那这时候也可以恢复。当然也比如说我可以重新推一个镜像，那镜像内的东西肯定是我部署所有的东西，那在服务上被手动改掉的，就会被镜像重新覆盖掉，也能够完成相关的一个执行。
