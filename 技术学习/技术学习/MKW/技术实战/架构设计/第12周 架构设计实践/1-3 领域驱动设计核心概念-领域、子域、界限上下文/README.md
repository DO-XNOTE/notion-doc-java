比如你看到外面现在二手商品很火，二手商品这个大蛋糕我要切它一块，那这就是要进入到这个领域了，你要进入一个领域呢？你不能光开发一个二手商品的交易的能力的东西，那这个东西只能完成交易。但是如果你没有收费能力，你没有支付这个能力，所以你不能完成正常的支付。如果你没有人力资源能力了，那你就招不到员工，那你的平台就没无从搭起。


所以一个正常的平台是不是包含你业务的能力，后台的支撑，金融的结算物、信息流，还有什么资产流都要能够全部打通，那这就是一个完整的领域，也就是说你要具备完整的在一个市场里面赚钱的能力，有了这个能力，你所搭建的这样的平台就进入了这样一个领域。二手商品其实交易平台有点空泛，应该是更详细的，比如说二手车市场，这是一个领域，二手袜子市场也是一个领域，但是估计没有人愿意买，对不对？那除了这种领域以外，我们前面什么在架构师实践课程，在这个我们架构图的绘制课程给大家聊过一个什么宠物的托管领域，这也是一个领域，虽然很小，是吧？很多时候你可能就是开一家宠物托管店，但你也可以做成连锁，也可以做成电子化，也可以做成电商平台，这些都是一个一个的领域，在领域里面才有子域的概念，领域叫做 domain 的英文，那子域前面加个Sub Domain 就是比领域小一些的。


比如说二手商品交易这样一个平台里面，你是不是要付钱？有支付子域，同时你要把这商品列出来，是不是有一个产品或者商品的子域，还有你要有哪些用户是买家，哪些用户是卖家？是不是用户认证他的身份，认证他的权限设置？也有一些子域，这些子域合在一起才能承载这个领域，对吧？你如果一个领域只能进行订单的处理，没法支付，没有产品信息，没法让用户登录，也不知道哪个用户买了什么东西，那这几个订单其实是无意义的，对吧？所以子域合在一起形成一个领域的能力，领域对子域提出了一些要求，后面我们可以看到子域里面还分好几类，我们这里先快速的过一过，有领域切分成子域，有了子域以后很容易就导出这样一个叫界限上下文。


什么是领域上下文？很简单，把你企业里面所有子域前面的那个名词保留子域两个字，换成上下文就可以了。比如支付子域，支付子域是一个要解决的问题，那我这里面比较常见的方法就是什么？就是把支付子域用支付上下文来实现。所谓上下文是什么？就是有一个系统，这个系统跟别的系统打交道的，那叫context。那这个 context 其实就是一个环境，这个环境跟别人打交道，它就有一个上下文的关系，那这种 context 这种环境就叫做界限上下文。那子域跟界限上下文更好的一个定义就是：子域提出一个问题，比如我要提出支付，这样的能力该怎么实现呢？我搭一套支付的平台，这个支付的平台会跟产品信息打交道，会跟个人的这个用户认证信息打交道，那这样的一个跟别人打交道的一个独立的场景，里面的所有的功能合在一起就是支付上下文。这些功能解决的问题就是要实现能够支付，所以子域提出问题，上下文解决问题。
