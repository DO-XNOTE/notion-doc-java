领域层你一旦都是 get set 的方法，喜欢就是做 CruD 工程师，这种情况下你写出的代码基本上是属于贫血的。就是你在领域层，虽然你把领域拆得很好，但是每个领域里面其实没太多业务逻辑，它大部分就是在读写那些数据，真正的业务逻辑到哪里去了？到了上层由校长去管理学校了，他管理的很细很细，就是校长的压力很大，但是每一个班级就比较没有什么功能，这个班级里面学生就只会读书，也不会打架，不会打比赛，也不会去参加联谊了，就功能太单一了。


这种模型也很多，我们以传统的架构数据里面很多都是这样，甚至于 get set 方法都是public，是任何人都可以调用的，然后各个服务之间的互相调用，就用 get set 来操作。充血模型不是这样，充血模型特定叫说人话，它更符合领域驱动设计，用通俗的领域语言，或者说用通俗的大家说人话的这种方式来进行沟通，它可以有私有的成员变量，也可以有很多 set get，但是这些 set get 都是私有的方法会在外面封装一个比如 public 或者 protect 的这种方法来保护它，或者说来丰富它。


使得最后整个方法外面的用户调用的时候是一个动词名词结合的，待会我们看一下实际代码，看看怎么样去说人话。充血模型的特点就是它是以应用逻辑为主，因为它要做很多的复杂的逻辑操作，所以这些操作都是动名词结合的，这种动名词就会变成一个一个的方法，让实体让我们的聚合根去用这种方法来对外提供服务。它不是以数据管理为主的，如果是数据管理为主的业务，很多时候有可能还是用传统的贫血模型，我们来看一看什么样式充血模型，其实前面的代码里面我们应该已经看到过，这里只是方便大家再进行进一步观察。我们切来切去，终于把那个切到了我们这个代码，然后还是 backlog item 这样一个什么聚合根，这个班长下面我们可以看到它有什么，它有很多的私有的成员变量，然后还有更多的什么对外提供的公有的服务和方法。


public，比如说这里的什么assign，一个 business priority，其实本质上你可以看到它本质就是一个set，但是它封装了一下，它把 set 包装的用一个动名词包装了，这种时候当什么第三方的用户或者说是应用层去调用它的时候，你知道我是在定义或者指配一个业务的优先级，而不是set。除此以外，它可能封装完了以后会把一些事件的发布封装在里，看看其他的。比如我们要分配一个故事点，也就是分配着多少人多少天才能完成，其实本质还是set，但是它会把事件发布放在一起统一进行处理，然后封装一个说人话动名词，结合我们看一下 set 里面啥一个值，复制给成员变量，故意把 set 变成private。


为什么？因为不要让机器人说的话让别人能调用，只有说人话的那部分，我们的方法才会被外面说调用，这就是聚合根的操作。那我们回过去再看一下我们除了聚合根以外还有什么？还有一些值对象，还有一些实体。我们找一个值对象，孩子是以 business priority 为例，值对象是一本书，这个书就没有限制，就是基本上书的方法基本上都是人人都可以用的，它没有什么事物一致性的要求在里面，所以你可以看到它这里面很多方法，比如说计算优先级、计算成本，它就直接说了我是什么计算一个 cost 的percentage，直接把名称一眼就能让别人知道是什。然后它也是public。


我们再回去看一看有没有什么实体，也就是班级小组员，有 task 就是一个班级组员，我们回到头上 task 就是一个普通的entity，他不是班长，它也有很多它自有的成员变量，然后它对外提供的服务，我们挑一个经典的服务，我们找一找这里给一个任务分配一个志愿者，这是他一个经典的服务。


他这个服务是用一个方法来实现，什么方法动名字结合的一个很说人话的一个方法，但这个方法要protected，它大部分的方法都是让班长调用的，对外只有一个声音，就是班长，所以我们会把班级成员的方法都会变成protected，也就是在同一个包、同一个班级之内，它们是可以互相使用，班长可以调用他，其他班级的人会没法调用他的这个方法。而他方法的本质还是那个私有的set，以及发布一个事件，领域事件，我们后面会详细聊。


好，我们看一下是不是本质上还是一个 set 方法，那它这里可以把方法设成protected，其实也可以变成private。我更推荐什么把所有的 set getter 都变成private，然后让我们的实体对外提供一些 protected 的服务，让我们的值对象以及聚合根都对外提供 public 的服务。
切回我们的PPT，刚刚聊了什么样是充血模型，然后再跟大家传统的评选模型比一比，其实就是多了一层封装，多了一层说人话的概念。什么样的模型更适合我们，那大家肯定说了，面试总归说充血模型好了，也未必。我们要从这个角度来聊，就是一切的模型，一切的架构都没有赢，但应该是各有优劣的。

通常来看，贫血模型的好处就是上手快， set get 一写，很快就把这么一些这个数据库的读写能力写出来了，但是发展受限，就像打游戏一样，你这个游戏的这个英雄很容易上手，小鲁班开枪对吧？在王者荣耀里面大家都会玩，但是你要打得好，你要打成真正的王者还蛮难的。


这就是贫血模型，它比较适合应用逻辑，非常简单，是以什么以信息记录为主的这种业务逻辑你，可以看一个特点， CPU 内存消耗不高，但是什么硬盘读写或者网络读写消耗很大，这种就是适合于贫血模型的。哪种适合于充血模型？就是这个英雄上限很高，一些很难打的英雄，比如像李白就是有点难度的英雄，端的上手虽然可能难，但是未来的发展就比较好。那同样的这个领域里面整个模型开始上手有点难，但是未来有可能我们可以把它什么这个代码结构组织得非常漂亮，那这就是用充血模型的好处，它的文字描述很清晰，它的逻辑都包含在我们的领域之内了，但是它也有不好的地方，容易三高，什么高血脂、高血糖、高血量，其中一个高就是什么。


我们有大量的跳转，其实我们可以看到本来说一句话，我们通过从外面包一层，然后还发一个事件，然后他其实还是干了一件事情，代码其实行数变多了，并且经常有跳来跳去的现象，从那个应用层跳到领域层，领域同学跳到技术架构层，技术架构层又跳回领域层，这样跳的大家有点头晕对不对？刚刚上手可能会感觉有点三高，逐渐发展到一个比较强大的，或者说是比较庞大的领域架构的设计的时候，这时候充血模型就能展现它的优势，它适合于那种逻辑复杂，同时以信息处理为主体的业务。这种业务的特点就是它跑起来以后很占 CPU 内存，但是对 IO 读写未必那么夸张，那这是充血模型。


这两个模型其实我们聊完以后，大家想一想，如果让你去设计一个简单的，比如像之前我讲的那种宠物寄存系统，你应该用贫血模型还是中选模型？那或者是让你设计一套新型的金融衍生产品，从中获利，从中去牟取一些暴利，这样的系统又应该用贫血模型还是中选模型？都可以在 QQ 群里面跟我和其他老师来分享和交流。那下一节我们会重点聊一聊 DDD 的一个整洁架构，洋葱头模型，大家敬请期待。
