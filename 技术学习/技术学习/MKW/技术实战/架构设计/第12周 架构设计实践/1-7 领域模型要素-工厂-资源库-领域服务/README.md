哎，还记得吗？在我们开学的第一天，班主任现在先召集整个班级进行了一个大会，聊了很多内容，比如说我们什么我们的新的同学的这个学号，他给我们分配了，然后告诉我们，我是你们班级的班主任，我来统管你们班级，你们的这个班级从今天开始成立了对不对？然后有什么班级口号、学员注册，还给我们什么？发了一本四年非常实用的学生指导手册，对不对？这些都是班主任在干的活，那我们对应到领域驱动设计里面就是工厂，那工厂是用来创建这个班级、这个集合，以及它下面的一个一个的实体，给它们分配一些 ID 创建出来。


这些都是什么工厂要干的事情？那工厂干的事情通常可以用代码哪种方法来实现？可以用工厂模式，还可以用抽象工厂模式创建的模式来实现。那具体来看可以使工厂的类，也可以使用工厂的方法，云云总总，它总体只是只要能建出聚合，建出实体，你就完成了什么班主任的使命。


所以飞扬老师的班主任只有在开学第一天我看到过，然后我的四年生活当中是没有再看到过他的，我也忘记了他的名字，很不好意思。但是工厂就是干这个事情，那它是什么？它是相当于是学校跟学生，学校跟班级之间的一种纽带。如果我们把领域驱动设计的四层里面的第二层应用认为是校领导或者院领导的话，应用管理所有的这个业务的实体的这个交互过程，那他是领导层，那中间的领域层就是实实在在的学生，还有除学生以外，还有老师也是在领域承担的，这里就是第一位老师了，班主任，工厂模式的班主任。


那他呢？是学校跟什么学生之间沟通的纽带，那除此以外它还有很多复杂的过程，就比如刚刚说的什么给我们学生的辅导手册，告知我们什么整个的注册过程，号称班级成立了，设定班级的口号等等，这些复杂的过程统一在工厂类里面进行了封装和管理。这是工厂，我们下一个另外一位老师出现了，这叫什么？repository，翻成中文叫资源库，或者叫资料库，对不对？那我用校园生活来表示叫什么？

叫辅导员，辅导员这个名词是不是在学校里面出现概率比班主任高？我们初中、高中靠的是班主任，大学生活靠的什么？更多的是辅导员。


那一个辅导员通常不是光辅导这个班级，可能计算机系的所有班级都是由一个辅导员来管理，他辅导好几个班级。那这些班级里面学生的变动，这些班级的这个拆分，比如有学生什么到其他学校去交流了，那两个班级的学生都减少了一半，那这个时候把两个班级合并等等，这些操作是不是都是辅导员在做？那我们对应领域驱动设计里面就是资源库repository，负责很多个不同的集合的总体管理，它可以是一个集群式的管理，那么也可以是做一些这种特殊的，比如修改、查找等等。那它除了做一些这种集群式的管理以外，它有个很大的特点，就是什么每个辅导员都有一个黑本，这个黑本里面记录了这个班级发生的所有故事，尤其是班级里面出现什么打架斗殴，出现考试舞弊，他都记下来。


张三，一年级跟三年级打架斗殴各一次，李四，二年级跟四年级舞弊两次以上，好，毕业的时候你拿到了毕业文凭，但是有一个毕业评语随着你的这个档案出去了，就是我们的什么我们的这个辅导员写的。此人多次舞弊，多次打架斗殴，录用单位请注意。好，基本上是你是很难被什么好的单位录用的，就是因为你上他的黑本，那这个黑本其实通过持久化成，比如像Hibernate，mybatis 等等，把它什么把数据全部记下来，把你所有的行为都记下来，这就是repository。既记行为又管理什么多个班级，是不是比班主任的功能更强大？确实也是，辅导员是经常在我们身边能出现的，对不对？那他跟后台的比方像DAO， Mapper 这些，就是我们整个领域驱动模型四层里面，第四层基础架构层是不是有点相关的？对，它可以作为它的抽象，就是数据持久层的抽象在这里展现，在领域层做好了，然后底下的基础架构层是它具体的实现，你可以用 MySQL 实现，可以用 Oracle 实现，可以用缓存数据库实现，可以用 MongoDB 来实现。


那不同的实现怎么可以是五花八门的？那这种方法使得第三层跟第四层实现了控制的反转。那我们重点要聊一聊，就是 repository 相对于第四层来说更偏集合的概念，就是我要管多个班级，然后找到合适的班级，然后从这班级里面找到它相关的服务，通过跟这个班级的什么他的班长沟通来提供相对应的服务，那这是 repository 比较经典的操作，而底下的基础架构层只是实现最终的数据的落地、读写这些基本功能而已。


那相对来说资源库可能比工厂还要复杂一些，因为它是什么，它又管多个班级，同时班级里面的很多的什么增删改查的操作，它都需要和后台的持久化了机制的这个基础架构进行沟通和处理。聊完了这两位老师，大家想一想，学校里面除了老师，除了学生还有哪类人？

校领导、院领导是不是也是老师？对不对？那也算在老师这个行列，还有哪类人？校工，是不是？或者比如我们讲我们学工学农，可能碰到了校工，那我们军训碰到什么？是不是教官这类人？我们在领域驱动设计里面也有一个名称叫做 domain service 领域服务。


那它有什么特点？它无状态，所以无状态就是它从你生命当中一带而过，你很难记住学工学农的老师或者是教官的名称，你只记得什么什么什么什么一大队的队长，然后让我们怎么做，对不对？我们的教官让我们怎么做？你很难记得教官的名称，甚至你都不知道教官的名称，那这就是一种透明人。
这个人呢？你没法很好定位它，你通只能通过班级等方法来定位，同时它一带而过，它完成这个任务以后，你下次再也看不到它了。那这就是领域服务的一特点，它不像老师跟学生有一个完整的生命周期，它是一带而过，无状态的，那除此以外，它既不能算是老师，也不能算是学生，更不能算是书籍，对不对？好，那同时它要完成一些复杂的转换操作，比如把你跟什么一个农民进行转换，把你和一个军人进行转换，把你和一个工人进行转换，这种转换的操作就通过校工，通过教官来实现。


那具体来看我们代码里有哪些转换？用户登录注册的时候，这个密码的加密服务是不是一种转换？当飞扬老师在说我要上领域驱动设计的时候，你要多语言转换，转换到后面可不能变成飞扬老师在教嘛？老司机开车对不对？就是文字的转换。那除此以外还有多个实体之间的沟通，比如说我们会计准则里面最常见的什么有借必有贷、借贷 b 箱等，所以应收账款和应付账款对账的时候，两边是都有什么相应的操作的？你把整个这个服务放在应收账款这个班级里面也不合适，放在应付账款这个班级里也不合适，要放在它两者之间。那通过一个教官对吧？或者是一个什么一个这个教官的形式来进行处理就比较合适。
