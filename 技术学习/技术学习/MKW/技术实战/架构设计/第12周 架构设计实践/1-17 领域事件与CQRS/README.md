当你要修改一个内容的时候，你会发起一个命令，这个命令会经过应用层的，这个就是命令处理器，然后落到应用的模型，也就是应用层。我们的校长会说，嗯，找一个模型来处理，你这个模型只干修改这件事情，修改完了以后它会落到一个什么命令模型的存储单元上，那这个时候我们改完写库通常有两个方案，一个是平时我们用的传统的叫 change data capture CDC 这种什么写库和读库的同步机制，这是一种数据库机制。但我们的 CQ RS 里面不是这样做， CQRS 里面我们是用领域来实现写库和读库的同步。


怎么实现？命令模型会发起一个事件，然后有一个事件订阅方，也就是我们说王大妈寝室管理员，他会什么更新这个事件的内容，更新到一个读库里面，这次它不是更新到那个公告版了，它直接更新到读库里面。所以写库和读库的数据的更新是通过一个领域里面的事件来进行完成的。这就是 CQRS 跟我们平时的数据库的读写分离的区别。那这样的功能实现完了以后，基本上我们只要有修改都可以在写库里面得到。但是读库里面可能会有毫秒对分钟级的延时，因为中间采用事件驱动的风格。

事件溯源只干两件事，第一件事就是从左边往右边客户发起一个命令，我要修改聚合里面的内容了，那这个时候的聚合改完以后会发一个事件，那这个改完指示是什么？在 CPU 内存里面改完这个内容，改完以后发来事件说请把我改完的内容落盘。


那这个事件为什么还是有王大妈？寝室管理员接收到，然后写到那里，这次写到什么？写到我们的公告栏事件存储里面了，又是回到校园生活了，对吧？我们的王大锤是吧？他作为班长发起一个事件，然后有王大妈请示管理员把他写到我们的公告栏，那这个写完以后，大家要记住这个写永远是什么一条一条追加的，它不删除以前的记录，也不修改以前的记录，它是把每一个变更全部把这个变更的过程给增加了形式给记录下来。


所以通过这个事件存储，我们没法知道当前的最终状态，我们只知道王大锤说了什么话，一个话、两个话、三个话、四个话，其实我们没法确定到最终把王大锤这个整个这个篮球比赛这个活动变成什么样，我们不知道，我们只知道中间发生的一件事情，所以这个时候我们要需要有校领导或者院领导或者是校工来做一件事，什么事？把右边这个事件存储的内容抽取出来，然后按照一定的顺序进行处理，把最终状态给计算出来，然后对这个状态进行一个什么快照切片。


那把这个内容塞到我们的资源库，就是通过我们的对班级最佳熟悉的不是班主任，而是辅导员，通过辅导员这种形式把它落盘，落到这个可查询的库里面。那这个时候我们这个业务实现了两个功能，一个叫增绿颜色的增，还有一个叫查所有你有的变化，我们全部只增，不删不改，然后会把所有增加的内容堆叠成一个一个的快照，然后这个快照体现我们当前的状态，然后放到资源库里面可查，这个时候当左边的客户重新要查询当前我们王大锤这个班级最终的篮球比赛的结果的时候，就不需要去事件存储一条一记录一条记录去读取了，直接读什么资源库就可以了。


那一样的它跟普通的 CQRS 比，它也是最终一致性，但是它比普通的 CQRS 有什么好处？当你发生灾难的时候，原来的 CQRS 有可能它的命令模型里面的数据丢失了，就没法回退到什么老的数据。而我们的事件存储里面是有每一个时间点的数据的，不管你是发生灾难还是有的程序员尝试什么山库跑路，你都可以回滚到前一个状态，回滚到山库之前的任何一个状，因为所有的状态的变更都在我们这里有记录，也包含你最后那条山库跑路的记录，你只要回滚到这条记录之前的数据，重构出我们的资源库，就能够回滚到任何一个时间点。


所以 event sourcing 事件溯源既是一种熔断手段，也是一种逻辑保护手段，也是一种读写分离手段，也是一种领域手段。集这么多优点，医生是不是很酷炫？那我相信后面的让课老师会给大家更加精彩的什么讲解，通过事件驱动架构设计的模型和框架，让大家真正了解到怎么样去实现事件溯源。
