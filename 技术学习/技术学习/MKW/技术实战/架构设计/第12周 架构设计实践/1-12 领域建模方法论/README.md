对于领域的切分，传统的方法还有一个大家应该非常熟悉，就是面向服务的 SOA 模型。怎么说？一切以总线为基准，我在总线里面要完成异步的消息队列要完成，同步的 API 调用要完成服务的注册要完成我们的规则引擎，要完成事件的处理和框架等等。


在上面可以插把，比如像流程化的应用服务、第三方接口、交互式内容、信息管理、安全模块、建模功能等等，所有东西都是插拔在这一根总线之上，这种情况下是以一种总线为基石，然后进行插拔各种各样内容，相对来说我们可以很自由地进行业务的拆分，然后这个总线上就能够互相沟通、互相通讯了。

真实紫色建模其实就是红绿黄蓝分别。写纸条应该先写红色纸条。这里假设我们是有一个网上买东西的，下订单了，这个时候会有订单，会有网银记录，会有包裹，更会有一些促销记录等等，这些都是一些什么一段时间内发生的行为和行为的记录。这就是红色的，不是贵错100，而是下订单。绿色的纸条应该紧接的跟上。


绿色纸条什么意思？就是谁在哪里，然后被动作产生的对象，比如说是用户的账户，就是谁产生的这个订单的对象，其实就是一个图书，还有员工，在看那个促销记录，那这谁就是员工在哪里？就是地理位置这些信息就是绿颜色的纸条里面写的内容，所以不是半仙老师，也不是在女生宿舍前，而是某个用户在某一个地址要买一本图书。


下一步就是黄颜色小纸条，就是角色，这个角色是市场总监，这个员工是作为市场总监，所以他要什么查询一下整个企业的促销记录，他也同时是一个配货人，所以他要发一个包裹出去，所以不是作为校草，而是作为市场总监。最后一条就是什么蓝颜色，就是一个详尽的描述，有时候书有书这么描述，不是激情澎湃的，而是什么？一本书，写得非常好的一本书，这样的一个描述，把这四个串起来就是什么四色分析法，是不是很简单？小学生套路，这个里面一旦把一个很复杂的场景变成简单描述以后，基本上实体跟值对象都出来了。哪些是实体？一般性绿色跟红色都会用实体或者聚合根来描述，黄色和蓝色可以考虑用值对象，尤其是蓝色基本上都值对象，黄色还可以在子对象和实体之间有一个考量。
