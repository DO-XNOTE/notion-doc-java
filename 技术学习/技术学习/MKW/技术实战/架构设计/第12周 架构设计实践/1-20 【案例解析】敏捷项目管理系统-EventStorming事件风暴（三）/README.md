架起需求到落地的桥梁，构建 it 新蓝图。我是张飞扬，上一节我们完成了命令风暴，这节我们就开始进入事件风暴。第三步，发现聚合好。要说到聚合，必然是有一个特殊的标签的，这个标签颜色比较随意，大家只要记住黄颜色最多的用作事件，蓝颜色比较多的用作命令就可以了，那其他标签其实都是很随意的，那我这里比较喜欢什么？把我们的这种聚合采用紫颜色的这个来渲染好，紫颜色作为我们的标签，这里面到底有哪些聚合？其实大家通过一个语法的方法来观察事件，就是名词乙加个动词，命令就是动词加个名词，聚合通常就是那个名词。


好，我们直接复制一下第一个蓝黄里面都有的名词叫什么？叫产品是不是好？所以它整个事件和整个命令其实在讲的就是产品。那我们逐一把其他的各个东西也都标记出来，是吧？这个小组在说什么？其实就是在说小组对不对？一样道理，后面是什么成员，然后负责人等等，一路写下去，把名词全部找出来，同样第二排怎么样也要考虑。对，一样道理。第二排其实这里也是产品。好，我们也把产品列出来，然后把这一行也弄完整。


下面第三排、第四排都是一样，每个什么？每个这个蓝黄紫形成一个三角形，左蓝右黄上紫色。对，这样一个三角形就表示出我这个命令跟事件其实是附属于哪一个聚合的，这还不是完整的聚合，这是一个可能聚合的一个什么 candidate 一个选项，我们先把选项填完整，然后我们休息一两分钟来看一看选项弄完以后，飞扬老师怎么样一把就把聚合给实实在在创建出来的。


好。到这一步，我们已经建出了大部分的什么大部分可能存在的聚合，但是其中是有些问题的，是吧？我们是直接把每一个命令和事件里面的名词挑选出来，作为聚合的选项潜在的这个选择项，但是这里面存在很多问题，难道真的有这么多聚合吗？产品小组成员、负责人、发布冲刺任务优先级故事点标签任务、发布任务冲刺讨论，这些都是聚合吗？这就是关键问题所在，我们把聚合分得太细了，太细导致什么？导致有可能一个事物要跨多个聚合，比如说我们要对整个任务进行管理，也许我要确定一下它到底是优先级多高，然后分配多少的故事点，这是一个很常规的什么在一个页面里面完成的任务，但是我却要跨任务聚合，用兴趣聚合、故事聚合做三件事情。


而这三件事情通常我们在界面里是一把操作同时完成的，而不是打开一次改一下优先级，然后关闭，重新再打开一次改一下故事点，这样操作跟我们的实际用户体验差距非常大。所以这个时候我们考虑一下怎么样，是不是有些内容应该作为一个一致性的事物进行统一处理，比如优先级，所谓的优先级只是什么任务的一个属性，对它的什么实际聚合还是任务？同样道理，故事点是不是也是任务标签？是不是也是任务的一种属性？然后任务的发布，这说的什么？其实它有两个内容，一个是把任务放到发布里面了，另外一个发布接收了这个任务。


好，所以我们可以在这里认为是一个任务的能力，同时我们要在发布上做出对应的能力，对不对？把一个事件给接收过来，也是一个发布所要接受的能力。一样道理，我们的任务冲刺其实也是任务的能力讨论，确实可以作为一个独立体。好，我们再来看看下面两个任务负责人和任务协作人。本质上它只是一个普通的把一个任务里面安排一个负责人，在这个任务里面安排一个协作人，所以它的主体到底是负责人还是任务？其实也应该是任务，对不对？本质来看它其实都是一个 backlog 要完成的任务，然后它给这个任务分配了一个负责人，分配了一个协作人。


而不像这里这一小组的这个负责人可能就是一个产品的经理，或者是一个研发经理，他来管理整个发布过程，他这里面的成员就是作为一个普通员工来参与后面的冲刺和任务的安排，这两个才是什么真正核心的聚合的概念，而这里应该是任务。好，我把后面内容是吧边聊天边喝咖啡边喝茶给补全了，补全以后就长这个样子，下一步就是见证奇迹的时刻，我们既然已经其实把聚合的整体概念出来以后，不需要放这么多的紫颜色标签，我们怎么样？我们进行一个汇拢，怎么样？汇拢把相同的紫颜色标签给摆在一起，然后分隔开来。


这里因为不是真正的墙，其实只要什么摆的距离有点间距足够大，就表示风格。那我们这里找一根线来分隔好不好？我们来插入一个这种连线，这样用一根连线来进行分隔好不好？那这根连线左边是不是就是都是产品？所以我们把产品往上拉，然后把下面的这些标签可以拿掉了。


阜阳老师，这是恰恰很好，正好是什么一一对应，你看每一列刚好是一个聚合的名称，那在实际的项目过程当中，越是复杂的项目越混乱，所以通常我们需要把这三个东西移来移去，对吧？每三个一组铁三角，把一个题材移来移去，移出最后很漂亮的什么一个一个的分组。好，那飞扬老师，因为这里提前其实是老师偷偷做过准备的，刚好我们当前来看，不需要做任何腾挪，只需要把一根根的线，是吧？放上去，然后把多余的这些标签拿掉，就能够实现我们要完成的聚合的划分了。好，这里操作不多，所以我们就全部现场来操作，这里一大片的任务，对不对？这里有个讨论，所以我们把大量的任务全部删除，好，把它放在中间，这就是一个大的聚合任务。那是一个什么？对我们来说是一个比较复杂的聚合，这里不应该是任务，本来应该是讨论懂，然后我们实际上是因为它是一个聚合，所以下面的内容给删除了。基本上通过这样的划分以后，是不是我们的聚合已经出来了？事务一致性的聚合，有产品、有小组、有成员、有他的负责人，还有我们敏捷里面关心的发布和冲刺，以及很多具体的任务里面的内容还有讨论。这样我们基本上就把我们的整个子系统进行了微服务的拆分。因为这个子系统相对简单，所以我们没有什么，我们没有划分更多的层次，比如说我们在复杂系统里面可能会划分一个叫微服务层次，那这个层次我们后面有单独的一节，来看怎样去对应模块和包。


而在微服下面再划分聚合，我们这里简单起见就全部用聚合划分完了，在划分的过程当中，其实大家可以想象除了聚合以外，应该有些东西是跟我们有交互的，比如说这里的小组成员应该是什么，通过外部一个用户认证系统来获取的。那这种用户认证系统也有一个标签，我们这里给大家展示一下，叫外围系统，那我们用什么标签？这里也是比较随意的，我们可以选择绿颜色作为它的外围系统标签。所以我们可以理解到这个跟用户认证跟负责人认证的系统，应该有这么一个叫做外围系统。比如像是沟通系统或者是什么用户认证系统，这里应该是用户认证更合适一些。用户认证系统作为我们的一个外围系统。


好，刚刚也说沟通系统是在哪里？应该在这里的，我们既然说要强化我们的讨论区，所以我们在讨论区里面应该什么增加一些，比如说转赞评这样一些类似于微博的这种沟通功能，是吧？这个沟通系统沟通的管理，沟通管理系统也是一个外围系统，跟我们的这个讨论聚合紧密相关，那除此以外还有什么聚合？大家想一想，我们在代码开发的过程当中是不是可能还有一个代码仓库啊？刚刚其实我们图形化界面里费老师没有详细聊，其实我们是吧每一个这个具体的 backlog item 这个任务下面都可以选一选，我们这个任务对应的代码仓库的是哪一个仓库？所以代码仓库其实是一个潜在的跟我们直接相关的外围系统。
好，有了这样三个外围系统以及大量的聚合，我们其实大部分的什么微服务的拆分已经完成了，下一节我们就会来聊一聊怎么样用动态的方法把这些已经拆分的微服务给关联起来，使得它中间的是用 API 访问还是用消息队列访问这种交互方式给展现出来，同时把它里面的成员变量和方法给逐一的确定下来，大家敬请期待。
