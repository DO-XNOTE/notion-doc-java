架起需求到落地的桥梁，构建 it 新蓝图。我是张飞扬，上一节我们聊了聊贫血和充血，这一节我们来看一看洋葱圈模型。其实洋葱圈说白了就是一个洋葱头，右边那个是不是把一个洋葱头给啪一刀切开了，就长这样子，一层又一层。这个模型其实跟我们之前说的什么四层模型其实是一一对应的。我们来看一下这是两个模型之间是怎么样对应关系。

也就是我们平时 MVC，  c 端控制器层，恰好对应什么洋葱圈模型的第二圈控制器。重点它是用来实现一个接口的适配，它可以是一个 Google 的GRPC的控制，也可以是一个什么一个 rest API 的展现，或者是 RPC 一种展现方式，这是对外提供服务的一个主要层，真正整个系统要完成一个业务的用例，应该是通过谁来完成？当然是校领导了，不管你是办一个大型的运动会，还是搞一个大型的军训，或者是一个招聘活动，有校领导跨多个班级，跨多个事务一致性的组，然后逐一去进行业务的处理和事件的转发，最后完成整个用例的。所以我们用例模型里面的一个用例的真正的协调者当然是校领导。应用业务层。


第三层，领域模型层对应什么啊？当然就是洋葱头的最核心的那个实体。业务实体也就是其中的业务逻辑的所在，这一层是需要我们进行微服务的切割，切分成一个跟事务一致性相关的班级，然后在班级上面展现各种各样的能力，比如像班长能力，比如像普通班员能力，以及为班级配好他他的班主任、他的辅导员，还有一些校工共同来承担这样一个业务实体的能力。


这三层完全是依次跟我们洋葱头的这三层对应起来。


第四层恰恰是一个反转，我们的基础架构层本来在左边是最下面，但是在右边却是最外层，而不是最内层。为什么会这样？大家想一想，这就恰恰就是控制反转和依赖反转了，是不是？我们的基础架构说白了，什么可以用MySQL，可以用 Postgresql，可以是 no SQL，像MongoDB，甚至于是文件系统，对象存储都可以实现什么持久化，这个时候在业务实体逻辑里面，只要指定好辅导员说我把这个数据集合持久化就可以了，不用去关心具体使用 SQL 还是NoSQL 还是文件系统，还是对象存储，所以它的整个依赖关系是这样，洋葱头的依赖关系是由外而内进行依赖的。


也就是数据库会调用什么网关或者依赖于网关，网关可以依赖于什么？依赖于应用服务器，就是校领导，又依赖于各个班级以及班级相关的那些辅导员，那些班主任共同来承载这个什么学校的管理。这种情况下我们的业务逻辑可以自由发挥了，我不用关心我的持久化或者消息队列时采用什么样的技术，我想怎么发挥就怎么发挥，这给我们的业务逻辑的这个开发留下了很大的空间。所以洋葱圈模型重点聊的就是依赖。而左边的四层模型是给人一种感觉，就是它其实跟传统的这个三层架构，或者传统的四层架构其实是一一对应的，这两个其实是一件事情的两种不同描述，那我们是不是还有其他描述？当然有什么六边形架构等等，所有这些架构的描述的核心就是一句话，领域的核心的模型在整个的所有的模型当中是属于中间层，是最核心的，不要让它去依赖，而让别人依赖于它。


聊完了，洋葱圈模型是一个轻松的话题，下面是一个整个领域驱动里面的非常大的难点，也是关键点就是领域事件，不管是工厂方法里面，或者是我们的聚合根和实体的常用的对外方法里面，你都能看到领域事件的影子。下一节我们就会聊一聊领域事件的概念，以及看一看这个事件该如何从代码上进行开发，它的整个跳转逻辑应该是如何执行的，大家敬请期待。
