通用问题是什么？通用问题就是 do the right thing  right。那前面其实已经跟大家聊过， right thing 把你该做的事情做出来，那就是right  thing 而且你要把这个事情做得漂亮，就是right。


你该做什么呢？架构师就是应该完成业务和客户的需求，那同时要把事情做漂亮什么？特别是复杂的项目，你很难做得很漂亮，那怎么办？分而治之是吧？当一个大项目没法做漂亮的时候，我把它拆分成一个小块，然后我可以带领一个架构师团队，让我们的小兄弟们，让小伙伴们帮我们把每一个小块给实现，然后我把它拼接起来，不就完成一个大的复杂系统了吗？就是冲着这两点，领域驱动设计才这么火，才这么有意义。


因为领域驱动设计就是一个以客户和产品为导向进行业务拆分的一套架构设计思路。好，聊了聊这个架构设计的核心思路，领域系统设计的业务需求和风而知之，重点还关注一下业务需求。下一节会聊一聊风而知之的子域，借鉴上下文的概念。这一集我们先聊一聊业务需求。说到业务需求，飞扬老师又要拿出这张图了，对吧？

我们来看一下，最常见我们以前是 MVC 这种三层模型，或者是一些各种各样变种的三层模型。那领域驱动设计里面是四层。首先看第一层是不都一样？展现层-它是提供一些 API 或者是一些图形化界面，是对外提供一个封装的接口。那底下是应用层，传统的应用层里面既负责逻辑，也负责一些，比如说命令，一些任务的调度，管理一些这个编排策略，这里应用层只负责一些编排一些消息和命令机制，把逻辑下放要包到第三层。这什么层？领域层，也就是我们后面所有的领域驱动设计的模型，里面的核心概念都在这一层里面进行研发和设计。那再底下是什么层？大家应该能想象到什么基础架构的层次，包含一些caching、持久化、 DAO 相关的内容，一些消息、队列分析、发邮件、做日志、审计等等。


底层的所有的基础架构的内容放在最底下也可以，通过什么通过这个控制反转往上放都可以，但中间那层是我们的核心是吧？领域层是我们的核心，而应用层是调领域层，应用层里面可能有应用的服务，也可能有领域的服务。后面会聊到，那这些服务没法直接抽象成一个比较容易理解的领域模型，那这个时候可以放在应用层进行处理，而最外面的展现和接口是大家最熟悉的 Controller。


那这样一个领域分层模型会影响到后面我们的每一个什么模块，里面的一些因素和摆放也会影响到我们后面代码的分层，所以这里大家有一个感性的认知就可以，后面我们会通过代码的分层来真正深刻的理解这个领域设计的多层。

那除此以外，要讲到DDD，也要讲这样一个概念，就是**到底 DDD 在什么场景下用的最多？**这里是一张鲁棒图的三个小图标？那飞扬老师其实前面提到过鲁棒图一种什么？一种架构设计图的一种方式，这里借用这三个图标，左边表示跟用户相关，左边表示跟逻辑相关，右边表示跟数据相关，跟用户相关。


设计方法叫什么？通常我们叫做 smart UI，非常聪明的一种方法，一切以用户体验为导向。那这种研发方法比较适用于什么？我们的移动端的APP，很多移动端 APP 的厂商，它都是以这个移动端用户体验，以一个一个的控件为主导进行开发的，然后再控制后台的应用服务器该如何去支撑前端的这些控件和字段这些小模块的，那这种设计思路就叫以前端为主导的 UI 为主导的设计思路。


还有一种设计思路是以中间逻辑为主导的，比较经典的就是DDD， DDD 这种思路什么有个特点，它是一切就是以中间的这个逻辑抽象为主体，他不是那么关心后台的数据的存储，也不是那么关心什么用户的界面展现，他把你的用户的所有行为，用抽样逻辑把这个逻辑分成一个一个的模块，一个一个的小单元，这些小单元实现事务性，实现它的逻辑的功能的这个完整性。


通过这个逻辑来驱动后台的数据的存储，也来驱动前面的页面的展现，这是DDD，那后面以后台数据为主体的就是我们由 UML 的常见的设计思路，包含老师前面的什么架构师绘图章节跟大家画了很多图，是不是这些图里面你去考虑一下是不是有很多地方都有什么 er 图的因子，包含我们的部署单元，包含我们的这个数据单元以及我们前面的一个功能模块组件里面。


其实最后落地什么都是落地在数据，对所有那些图，其实很多时候都是考虑我要处理什么样的数据，我这个数据在传输过程当中应该怎么处理？我在最后落地过程中应该怎么处理？我觉得数据的处理单元一个 deployment unit 应该怎么样在我们的机器上进行摆放？所有那些都是偏数据的，是吧？而我们的领域驱动设计抛弃数据，抛弃界面，重点在看业务逻辑，所以它适合什么样的场景？**适合这样三个场景，一个是产品导向，这个产品不是说用户界面导向，而是整个产品的业务逻辑。**


比如说我要设计一套宠物的什么管理系统，那这个宠物管理系统里面有什么样逻辑？我是不是要让宠物洗澡？我是不是要让宠物吃饭？我还要监控宠物？通过一些中间逻辑层为导向，那这样的一个导向的产品通常是DDD。还有一个特点就是适合于中大型项目，就这个项目很大，大到就是我们通常两三个人没法在两三个月时间之内完成，那这个时候我们要进行拆分，但是拆分我们不能够随便让一个架构师从无到有的去进行拆分，因为太复杂，这个时候我要借鉴一些已有的思路，所以我找一些相关领域的思路。


你这些领域默认就有什么已经拆分成，比如说二三十个微服务了，或者拆分成这个十几个这个子域了，那这个时候你要参考这些子域微服的架构设计，进行你自己的什么每一个核心域的研发，那同时很多的支撑域，很多的这种通用域可以直接来借鉴。那这几个概念我们后面会详细聊，就是一旦碰到这种中大型项目，你可以考虑采用 DDD 领域驱动设计。


那还有一些就是当前我们什么 API 经济中台化，也就是很多内容，其实我们通过，比如像阿里云的平台，或者通过一些其他的，像的这个我们的后台的平台都已经完成包装好了。比如像支付交易，有一个交易网关的平台，比如像是我们的商品订单库或者商品中心都已经包装好，你拿来用就可以了。


那这个时候你可以把这些领域变成外包的，然后你只要重点关注你整个企业里面最核心的那些子子域，通过这种方式什么能够简化你的什么开发的工作量减，缩短你的这个研发周期，那这种时候也比较适合采用 DDD 领域驱动设计。好，那总体来看就是一要跟业务逻辑相关，二项目非常大的参与人员非常多，时间周期非常长，同时你能借鉴到很多的外面的中台 API 经济的那些现有的一些业务逻辑的这个资源，或者是一些服务，那这个时候你可以来考虑使用DDD。
