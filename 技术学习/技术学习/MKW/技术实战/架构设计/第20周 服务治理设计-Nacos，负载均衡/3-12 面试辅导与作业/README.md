如果你的负载均衡策略不能非常实时的去立即感知到你当前线程数还有 RT 的变化，那么我建议在这种超高并发的峰值脉冲模式下尽可能的就使用最简单的 round Robin。


为什么这样说？因为当我们在客户端的负载均衡上感知到你的线程数变化或者 RT 变化的时候，它有可能会有一定的延迟。为什么？因为我们本身就是依赖于服务发现来去获取到这些状态，那它并不是一个非常实时的一个状态获取的过程。所以说一旦它中间有一些延迟，当你在这种超高并发流量的场景下，你获取到的线程数有可能并不能反映它当前的真实负载，这种情况就有可能会造成一个浮物雪崩。


那这第一个问题，第二个问题，我们在选择负载均衡方案的时候，为什么会去选择客户端的负载均衡方案？那面试官问出这个问题的出发点其实就是很多微幅架构的技术框架，它都是采用的客户端负载均衡，那所以我们在回答这个问题的时候，从这样的两个点先后顺序来回答。那同学们记住面试过程当中不是说想到一个点，我就回答一个点，那这个就是知识点堆砌，我们尽可能把这个知识点组合成为一个脉络，比较循序渐进的这样一条脉络来回答，比较有章法。


这里我们可以先去肯定客户端负载均衡的优势都有哪些，那比如说你客户端网络开销比较小，客户端直接发起点对点的一个调用，这是其中一个。第二点，你在你的网络环境当中采用客户端负载均衡方案，并不需要一个额外的负载均衡器，并不需要额外的这样一个组件来部署。那还有一点就是说我们可以在客户端自由地定制这些负载均衡的策略，因为很多微服务的负载均衡都要求有一些像单元封闭，或者说是集群内的机器优先访问，或者基于权重访问等等这类需求，那我在客户端就可以做到这个定制化。

这两个是我们的大新应用当中非常常见的两个负载均衡需求，还有什么负载均衡的权重也把它考虑进去。 knacles 这里专门实现一个权重设置，不是空穴来风。阿里系当中确实也有这样的需求。对于我们的很多应用来说，你去申请的机器其实都是虚机。虚拟机，那虚拟机它的性能限制于很多因素，比如说你当前虚机是否是同一规格，那我都是一盒 8G 内存，OK，那这个可以没问题。但是如果你这个集群当中有一部分机器是两核16G，okay，那你的性能不同，自然而然我的负载均衡权重就需要给你做一些定制。那对于这类机器，你在启动脚本的时候，我可以去添加一些额外的设置，让你的负载均衡权重变高，尽可能的把请求打到性能更好的机器上。
还有一点，你不同机房，即使你的性能相同，但是你的网络环境不同，有可能你的网络带宽比较少，或者说你的机房物理位置离当前的中心化机房比较远，访问比较慢。这种情况下，我们也需要做一些权重的配置，尽可能的是在当前机房内访问，而不去跨机房。OK，那这就是很多通过负载均衡权重来实现的特殊的需求。


最后再跟同学们提一点，在超高并发应用当中，我们偏向使用的负载均衡策略恰恰是最朴实无华、平淡的 round Robin 简单就是稳定。那或者如果我们要使用一些偏门的算法，尽可能使用什么呀？加权类算法。为什么是加权类算法？其实就是为了防止服务雪崩。


如果我这样说，我基于当前可用连接数来做访问，可用连接数最小的机器，那所有请求都打到它。或者我用另一种方案使用加权，可用连接数最小的机器，它有更高的几率把请求打向它。同学们想，这两个不同的场景下，哪一个场景被容易被打崩？当然是第一个，那很可能把这台机器直接给它打挂掉。所以我们加入了一个加权，即使你当前机器可用雷接数最小，但是也不是说请求 100% 打响它，而是说它有更高的几率获取到访问权重，那这样的话就给你的整个集群带来了一个负载军风层面的一个缓冲，防止服务的雪崩。


所以说这几点一般来讲我们在做架构设计，或者说在面授过程当中都要去注意的超高并发应用，它会对你当前负载均衡提出一些额外的要求。OK，那说完这几点，我们接下来跟同学们就来布置两个动手作业了，
