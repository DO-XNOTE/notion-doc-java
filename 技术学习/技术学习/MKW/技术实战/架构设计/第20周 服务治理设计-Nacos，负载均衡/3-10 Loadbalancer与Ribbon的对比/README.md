那接下来的我们就要开始来批斗大会了，我们先来去聊一聊 Lord balancer 和ribbon，这里它真正的真功夫。那你既然两个都是负载均衡组件，那你底层的负载均衡策略都有什么样的不同？那 ribbon 我们知道底层有非常丰富的负载均衡策略，我们前面也已经学习过了七龙珠 7 个各不相同的负载均衡策略，可以应用在各种类型的场景上。与此同时，它也提供了非常易于扩展的一个接口 i rule 接口，我们可以去继承它来去实现自己的负载均衡策略。


那对于 load balancer 来说，我们这里真的是采用了一个极简的策略，无欲则刚，只有一个 round Robin 这样的一个负载均衡策略，那俨然一副光杆司令的这样的一个样子。所以从一个负载均衡组件它的负载均衡策略这样的一个重要指标上来看，毋庸置疑，那 ribbon 这里是处于一个更加有优势的这样一个地位，那如果有的同学这里硬找一些借口说我 load balancer，这里我也可以自己定制负载均衡策略，那没问题，我也可以定制和 ribbon 一样的策略，什么都听你的。


那我们接下来看一下负载均衡以外的以下功能，一些功能性的特征。比如说第一个超时ribbon，这里可以配置非常复杂的超时控制，那它可以去怎么样定义你的 connect 这个阶段的timeout，还可以去定义你接口处理过程这一段的timeout，也就是说你的一个从接口连接到接口处理这两步，它其实可以分别来控制它的超时动作，那这是它其中的一个优点。


另一个优点这里不得不提的就是细粒度的接口控制，我可以在 ribbon 的配置当中为指定的一个微服务，我来对它旗下的接口我做一个特殊指定的超时策略。那这个超时策略呢？我们甚至可以去指定我哪些类型的 request 需要应用，比如说我指定所有的 get 请求应用，这样的超时策略可以吗？okay，完全没有问题的。那与此同时，我还可以对接口的负载均衡策略进行一个详细的指定控制。


再有我也可以在这个服务层面去定义它的重试规则，那这都是细粒度的接口控制当中的一部分。刚才咱提到了一个重试规则，那这里就不得不提 ribbon 还有非常牛逼的花样重试的功能，那在重试过程当中，我可以去指定你在当前机器重试几次，你换一台机器去负载均衡到下一个节点再去重试几次，可以配置比较复杂的一个重试策略，这是他的三个本职工作。


除此以外他这边还有一个兼职就是服务炭活。我们前面学习到的 i PIN 机制就是 ribbon 的一个私活了，那从功能性的角度来看， load balancer 其实还处于一个比较玩具的一个状态当中，那跟 ribbon 比显然就是一个玩具，它这里可圈可点的地方有，这也不是没有，比如说它的缓存，那我可以把已经探明的服务把它缓存到本地那，并且设置对应的缓存策略。比方说我多长时间去做一次服务的拉取，从注册中心服务发现这里来去获取到最新的服务状态，并且我获取到之后我还可以去把它缓存，并且我可以去配置你可以缓存的capacity，也就是你的缓存容量个数。


不过这里如果咱想依赖缓存，后面会跟同学们来去做一个小的demo，这里就需要去做一个单独的依赖的配置，去引入一个特殊的客户端的缓存组件。从这两点来看，毋庸置疑，功能性的角度来说，那肯定 ribbon 这里大比分领先胜数。那最后我们说了硬功夫，我还要去看一看你的软实力，这个负载均衡组件它的软实力是什么呀？那说白了其实就是个人员跟什么的，人员跟 spring Claude 上下游组件之间的兼容的关系，那在 ribbon 这里已经成了 spring Claude 名义上的负载均衡老大个，那一提到负载均衡，你在各个组件当中它都默认地使用了就是ribbon，我可以说它是 spring Claude 当中的一个潜规则。


那如果咱这里去把它替换成 load balancer，那我们还需要在配置文件当中引入额外的引入 load balancer，因为大部分组件其实并没有依赖这个 load balancer。与此同时，我如果想让它生效，还需要去声明式的显示的在你的配置文件当中把 ribbon 给它剔除掉。那所以从这个角度上来看， ribbon 在方便易用性的角度以及上下游组件当中的协调性肯定是大幅领先于 load balancer 的。


既然我上下游组件兼容性更好，同时功能性还有底层负载均衡策略都是领先于 load balancer，那我们看着可怜巴巴的 load balancer 这边的几个屈指可数的功能点，那我只能说什么呀，有钱人的世界就是这么平淡无奇且枯燥。所以相信同学们，此刻如果让大家做一个选择，那大家心里应该已经有一个答案了，而且应该跟老师这里答案是一致的，那就是我们在这两者之间，**对于 ribbon 来说，能用一定还是继续用着，**
