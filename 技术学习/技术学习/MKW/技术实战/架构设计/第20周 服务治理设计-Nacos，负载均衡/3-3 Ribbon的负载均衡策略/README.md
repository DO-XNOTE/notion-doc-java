那同学们也许会有这样一个问题，假如我们现在当前客户端的访问量非常大，也就是高并发可能非常大，那么在这个情况下有可能有很多请求同时访问这台机器。


那么在 round Robin 这种负载均衡策略之下，我还能否保证一个顺序会不会有多个请求？在访问了一节点之后，没有去访问 2 节点，而是又访问了一节点，有没有可能发生这种并发性的问题？其实并不会，因为我们 run Robin 后面它的底层其实是采用了自旋加 cast 的同步操作，这也是一个非常静电的组合，自旋锁加上 cast 自旋锁同学们应该都比较熟悉了， a while true 循环就是一个很简单的自旋锁，或者是一个 for 循环不设置退出条件。那在 for 这个 while 循环以及 for 循环的方法体当中再去设置一个退出条件，也就是说退出条件在方法体当中，而不是在 while 循环的这个括号或者 for 循环括号里面设置推出。这就是自旋锁。听起来高大上，其实非常简单。那 case 这个就是面试当中经常考的一个问题，它在 Java 当中实际上是调用的底层的操作系统的方法，我们查看源码会发现它底层是 native 修饰的方法，那在操作系统层面 compare 比较加上替换 compare and sweep，那这个操作在操作系统里可以把它当做是一个原子操作，这是在操作系统层面来把控的。所以通过自旋锁加上一个 cats 操作，我可以保证以一个比较小的性能开销来去确保我在高并发场景下面做一个同步控制。那这套组合拳也是高并发下面非常廉价的一个线程安全手段，而且并不需要锁定任何系统资源。不过有好处当然也有坏处，比如说这个自旋锁操作。


自旋锁其实它在 for 循环这里没有设置退出条件，因此在它的方法体当中，如果这个 cats 操作失败，实际上它会再次做一个自选走到下一个循环当中，那这个情况对于 CPU 来说，实际上相当于让 CPU 空转了，因为它并没有产生任何有价值的业务逻辑，没有去执行这些业务逻辑。所以说在高并发场景下面，有可能造成一定的 CPU 资源的浪费。
