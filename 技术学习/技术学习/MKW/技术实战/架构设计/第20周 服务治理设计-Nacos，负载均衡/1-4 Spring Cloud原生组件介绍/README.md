它其实并不是开源社区，直接贡献了一款软件，它是一个叫做 Hashcop 的公司来开发的。 Hashcop 这个公司大家可能都没听过名字，不过记住一点就行了，这家公司可比 Netflix 要靠谱多了。那咱的 spring Claude 官方为什么推荐 cancel 啊？还不是因为 Netflix 公司这哥们不太靠谱，半路撂摊子跑路了。并且 Erica 2. 0 的开发计划又是被一再的搁置，官方这里才祭出了 console 来做一个多元化的发展。


其实从咱的这个稳定性的角度来说，有人卡依然是当前 spring Claude 当中应用最广泛、最为稳定的组件，所以同学们也未必急着要去把 Eureka 替换成console，那只要有人卡可以继续满足你现有业务的功能，那我认为继续使用是完全没有问题的。那对于新的业务项目，我们可以尝试着往 console 这个方向上来走。

咱老牌的服务容错组件大家应该都非常熟悉， Hystrix ，那   Hystrix  它的功能虽然也非常的全面，可是奈何它的主子奈飞公司， Netflix 公司半路撂摊子宣布停止了  Hystrix  的支持计划，不过它的原因其实很简单，因为  Hystrix  已经可以满足奈飞公司的日常的业务需求了。所以这个中原因并不是说  Hystrix  已经落伍了，而是说它的功能其实已经可以满足像 netflix 这种体谅公司的需求了。


netflixa 也是一个体量非常大的互联网公司，不过在 Netflix 宣布停止  Hystrix  维护的同时，它给我们也推荐了另一款服务容错的组件，叫 Resilience4j。这个名字读起来非常难读，不知道的还以为这是一款俄罗斯统治发明的软件。


它是起源于谁？它的架构思想设计都是源自于  Hystrix ，师从  Hystrix ，它这里来提供了函数式编程的一个体验，这是它最大的一个特点。同时它还具有服务容错的一些核心的功能，比如说断路器还有重试，那这里包括阻断式的重试，还有 ASYNC 形式的重试，也就是异步重试。与此同时，它这边还提供了限流以及信号量隔离。


不过从我自己的使用体验上来说，其实它并不如 high strikes 方便，因为  Hystrix  和 Feign 还有 spring cloud 的其他组件都有非常好的天然的集成，并且  Hystrix  是一个非常轻量级的组件，那它只是一个本地的服务容错，所以从我们的业务角度来说，几乎可以满足大部分的服务需求。


LLAMA 老师这里建议，如果同学们的需求仅仅是客户端的熔断和降级，其实用  Hystrix  已经绰绰有余了，并不着急于把  Hystrix  替换成 resilience4j。不过像 Netflix 公司，他在自己的新业务当中也做了一个尝试，是什么呢？也就是尝试迁移到这个新的组件当中。所以老师这里也比较推荐同学们在自己的新的业务项目当中使用 resilience4j，那或者使用另一款更厉害的组件 Sentinel。
