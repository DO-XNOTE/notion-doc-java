这个图是 Nacos 的多中心的运作模式，怎么体现出多中心呢？这个点体现得比较抽象，同学们看这里几张纸好像重叠的部分，这其实就是意思说是多个 Nexus 服务器，这就是它所谓的多中心。对于咱的微服务来说，两个角色，一个是 provider 服务的提供方，另一个是 consumer 服务的消费方，这个很好理解， consumer 调用provider，它提供的服务 provider 这里就要通过 neckers 这里提供的 open API 这一层，将自己注册到 Nexus 服务器当中。那我们 open API 这底层，它支持很多种不同的通信协议，比方说可以是HTTP，DNS， u d p , t r s 等等都可以，那这是底层的通信协议。


接下来我的provider，我要向 neques 注册什么信息？同学们，看这里啊，咱当中这块有个紫色的板子，恋爱色的板子，那这里面把我们注册什么信息已经说得很明白了，就是 name 报上你的大名。对于服务来说，服务的大名其实就是它的服务地址，还有服务名称。同样的，我的 consumer 其实也是作为一个服务调用者，也要把自己的大命同样的给报上去，注册到 NEX 服务中心这里来。与此同时，我们的 consumer APP 这里也是通过 open API 从咱的 NEX 服务器这里来获取 provider 注册进来的信息，这样一来他就知道我有哪些 provider 可以去调用。


那他们的物理地址分别是什么？咱这边说到现在都是说的这个表面功夫，我们再往里看，这中间这两条紫色的恋爱色长条，它们就分别代表了 Nacos 的两块核心的服务。左边的 Neques 作为配置中心的配置管理服务，右边其实就是与注册中心同样的一个功能，服务发现和 DNS 这两个大的功能，左右两点合起来就是咱这个 NEX 的左右护法构成了它最核心的两部分的应用，对于多个 Nexus 服务器多中心之间，我们是通过什么样的方式来做通信的？那就在这里通过这里底层的 consistency protocol 一致性协议来进行数据的底层的同步。比如说这里的一致性的协议，我可以使用 raft 协议，还有服务续约的同步，以及我们还可以去做基于这个RDBMS，就是关系型数据库，基于这种方式来做同步都是可以这个底层同步机制的模块，我们会把它包含在这一层上。

s p i 是什么意思？这个不是 spal，这是 spi service provider interface 服务提供者接口。比如说我这里跟同学们举一个例子，就帮助大家理解什么是SPI。咱平时连数据库同学们都会用GDBC，对吧？ Java 当中 j d b c 它就是一个编程接口，这个 j d b c 的每个具体实现，比如说 MySQL 的 j d b c 或者是 Oracle 的 j d b、c，实际上它就是一个 SPI 的功能点，不同的数据库的厂商会提供它们对应的 driver 实现，并且这个 driver 实现同一套 j d b c 接口，这个就是我们所说的 SPI 扩展 service provider interface。
