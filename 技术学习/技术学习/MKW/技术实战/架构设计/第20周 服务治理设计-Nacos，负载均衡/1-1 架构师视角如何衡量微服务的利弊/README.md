那么每当谈起微服务的优缺点，我们总拿出一个垫背的同学跟他做比较，那这个垫背就是单体应用，这里非常好理解，一个应用包罗万象，所有东西都在这一个小小的黑盒子里面。而微服务化正如大家想象的一样，它就像一个串糖葫芦一样，每个服务挨个调用一堆其他微服务来完成本来一个调用就可以完成的事情，那从这里听上去似乎微服有点那么多此一举。

诶，我既然一个单体应用可以完成的任务，我为什么要把它拆分成好多个应用来共同完成？难道是钱多的没地方花，要买服务器了吗？飞也那么单挺用它究竟跟微服务化比，它好在哪？我把这个单体应用它的特点总结归纳为一个词，叫做一桶江湖。而这个桶，同学们注意，这是桶娄子的桶。


单体应用它大发神威的时候是什么？是你在很小的小型项目当中，或者说在一个项目发展的初期，应用单体项目可以做快速的发布，快速的上线验证商业模式。那我的地盘我做主，就一个应用，想怎么改就怎么改，而当你的业务逐渐的变大的时候，如果还是采用单停用这种架构，那过不了多久你就会体会到一种感觉，就是你的代码逐渐的像变成一坨屎坑一样不能维护。这个场景可要比小便池里大便还要可怕的。所以到这个时候我们就不得不进行微服务化，把这大一统的江湖变成一个三分天下的格局。


OK，那么我们接下来了解一下微服务化它究竟有哪些优势。那同学们在这里忘掉你在网上可以找到的所有去评价微服务优缺点的论调，那这些论调都太书本化了，那我们用实际项目中的感受跟同学们去一起了解一下。哎，一个微服的架构它是如何体现自己的优越性的？那么我们首先先跟同学们定一个基调，那什么是微服务架构？那我给它起了一个大基调，那微服务架构在我眼里就是一个三个代表的精神，它分别代表着什么呢？

那这第一点不得不跟大家提的就是微服务，它是一种封疆大吏的指导思想，一种管理服务的模式。
那为什么说它是封疆大吏？我们把微服务架构和单体应用来比较，这就是一种分封制和集权制的根本不同。当我们去推行微服务架构设计的时候，自然就会形成一个微服务的团队，那通常来讲各个团队负责各自微服务领域的东西，那么你的订单域是由你的订单微服务团队来负责的，商品域是商品微服务团队来负责的。那不要小看这样一种分封的小地盘，它大大去提高了你技术团队的话语权，那这种小变化会使你的团队的运作模式发生翻天覆地的变化，那给了你的团队更大的发展空间，很多业务上的花头都可以在你的微服务领域当中来施展，那么使你的当前的这套小地盘上的业务可以做一个独立的演进。


那如果同学们去看阿里系内部的很多应用，那他们一开始从一个小地盘如何一步一步成长为一个中台，那其实就是靠这种封疆大吏式的微服务的划分，从曾经的集体所有制变成了当前的责任制、承包制，给集体打工变成了给自己的团队打工，那自然而然你团队的做事情的角度、思考问题的角度，还有能动性完全是不一样。所以我自己的角度出发，微服务化它第一个优点就是这样的一个精神层面上的建设。


OK，那第二点咱突出一个快上，快下什么意思？快上是指的部署那块，下是指的回滚，那和曾经的这种单体应用来相比，单体应用来说，如果我们要去部署一个改动，那其实是一个牵引发而动全身的动作，那么如果这个改动出现了任何问题，那么在回滚的同时我们也可能把其他模块的改动一起给回滚掉，那作为单停用这种架构，这是不可避免的。


但是在我们的微服务架构之下，咱的每一个应用模块都可以做一个独立的部署。那么即使你在这个模块当中发现了一个问题，那我需要进行回滚的也是当前我这个独立部署的模块，它并不会影响到其他模块的上限。那这样一种模式还有一个优点，那就是可以做快速的验证。通过像金丝雀测试、蓝绿测试等等，我可以在小范围规模内做一个很快的上线验证，去验证你已经上线的功能是否正确被执行。那这是一个非常显著的一个优越性。那同学们试想，很多单体应用它的发布周期其实是非常漫长的，那它们快则一个月，慢则甚至有可能是 6 个月半年的时间。那么你试想一下，半年时间好不容易发出了一个应用，但结果因为一些小的问题导致整个版本回滚，那这个损失是非常非常大的。那作为我们微服应用来说，它可以做很频繁的发布。像我们以前阿里系的应用，它在每周，通常我们会把周二、周四设置成发布窗口，那我每周可以做快速的迭代上线，每周两次，那么在平时还可以做一些线上的紧急补丁，这一个生产力的差别是非常非常巨大的。


OK，那么快上快下之后，我来再来看一个高起点的特性水位调配。那水位调配什么含义？比如说我整个集群有1万台机器，那我这里可以根据每个不同业务的忙闲程度给给它分配不同的集群水位。比如说我的搜索业务，因为在整个漏斗模型的最上部，它承接的用户流量是最大的，因此我在这1万台机器当中给搜索服务大概分配个 5000 台。


OK，那么接下来来你的订单服务还有库存服务，那我可能依据它的不同水位去给它做精准的调配，给它分配 3000 台，给最后的分配 2000 台，那这都是在微服务架构下面可以去大显身手的。但是同学们想一下，如果咱在一个单体应用上，那你还能做这种经济力度的扩容吗？你不能，那做的是大水漫灌形式的扩容，咱所有的业务代码全都部署在一个应用当中，那你的扩容其实就是针对你当前的这个单体应用做一个水平的扩展，这就产生了一个很严重的资源浪费。


同学们有没有感受到？比如说我当前的单停用只有最上部分的一个功能，我需要做扩容，但是我下面的几个应用其实并不需要做扩容。那如果我把这个应用做了一连串的水平扩展，那实际上我扩展出来的这些机器只有上方的这一个功能点，它是受益者，那下方的这些功能，这些并不需要扩容的功能，它们就造成了一种资源浪费的现象。所以说如果我们想要利用弹性计算去优化平衡你的服务器资源，那么我们必须要让自己的服务基于微服务化架构来搭建，那么这样的话才能去切实地提高你的资源利用率。OK，那咱这说了这么多的优点诶，还没有说完呢，我们还没有夸够，接下来我们再来看微服务化的其他的优点，

但是同学们想对于单体应用来说，它的接口的耦合程度非常高，比如说一个下单接口，我包含在一个 transaction 内，从头到尾一路数据库操作全部完成，那这整个下单链路就属于一个一荣俱荣、一损俱损的这样的一个流程，那如果是我们的微服务架构，那在这个链路上面我们可以做详细的调优。比如说我可以区分你下单链路过程当中，有哪些部分是处于主链路，哪些部分是属于可降级的模块，那在这里我就可以对它做细粒度的限流策略，或者是设置一些熔断降级开关。


我这里举一个最简单的例子，那比如说我在写数据库的时候，如果在下单链路当中，某一个非关键节点，非主链路的小环节出了错误，那对于单体应用来说，你写库失败直接一个 transaction 回滚掉。但是我们在微服务架构里面经常会有一些熔断降级的措施，比如说在某些微服的链路当中的一环里，那如果我写库失败，我可以做一个降级，怎么降级？把这个写库操作，我可以把它打到缓存当中，或者打到MQ，也就是消息组件当中采取一个异步写库的措施，那这就是一个很常见的降级补偿的例子，那你在单体应用当中，它就玩不转这么花哨的技术方案。


OK，那么最后我们再从一个大的层面，从业务解耦的层面跟同学们来分析一下。那对于微父来说，我的模块与模块之间有非常清晰的界限，那这就是我们常说的什么呀单一职责。不过在单体用当中，其实它里面的业务就像一锅什么一锅烂炖一样，所有的业务都冗余融合在一起非常的难以维护，那单停用并没有很好的践行单一职责，那其实当我们谈到单一职责的时候，它并不是我们从微观的代码层面分几个包，或者利用一些设计模式就可以做到的。
其实宏观层面的单一职责还有一个概念叫领域建模，那如何利用领域建模的理论来去拆分你的服务，将相关业务构建在一个领域模型之下，那这都是微服务架构所要做的事情。不止于此，微服务这里还可以去规范我们的数据和业务对象的访问。怎么说？对于单体用户来说，咱共同访问一个数据库，那所有的应用当中的业务逻辑都是直接在这个数据库上面进行一些读写操作，它并没有很好的界限的隔离，比方说我的下单接口会直接到商品的表彰中读取数据，那我的商品接口同样的也会到商品表中读数据，库存接口也会去读数据。



那么如果有朝一日我更改了商品表的一个字段，那么这个改动就会影响到单体应用整个链路上面所有访问我数据库的这样的业务，OK，但是对于微服务架构来说，这种问题并不存在，因为你的数据和业务对象，它都是隔离在了你的领域模型当中，那我的商品库、我的 DB 就是我自己的地盘，我来独享。


那我对里面底层的改动，那实际上只会反映在我的领域模型内，只要我往领域模型之外暴露出来的接口和业务对象，它把底层数据库的这种变更屏蔽掉，那么对于我的上下游业务来说，他们都不需要做任何的改动，这才是宏观层面的领域建模和单一职责要去考虑的事情。OK，那么我们这里讨论了很多的微富的优点，咱这节课的课题，同学们还记得吗？微服务的优点还有什么？还有一个缺点，那微服务纵然优点非常多，但是不可避免的它肯定存在一些缺点，那么接下来我们就来看一下微复的缺点都有哪些。OK，咱前面提到的这几个优点都体现出了一个概念。什么意思？就是我们前面提到的一个字，快。所谓天下武功，唯快不破，

这就是微服最大的一个优点，那快是它的优点，那它的缺点如果用一个字来去总结，应该是什么字？贵？微服务很贵，但是同学们，咱现在的互联网行业，它有钱，你贵，他遭不住，我有钱。那就像现在的家长，你孩子们上再贵的补习班，我都要掏钱给你上。钱不是问题，关键要把娃培养出来。和互联网行业一样，钱不是问题，你只要把业务做出来，用最快的速度把它给交付出来，那就是 OK 了。


那我们经常说的这个微服贵，它复杂，它究竟复杂在哪？那我们经常诟病的一点就是微服务的部署结构非常复杂，项目结构非常非常的复杂。这个点我们必须要承认，微服务确实有比较高的技术门槛。那试想一个单体应用，同学们只用掌握 spring boot 就可以把一个单体应用玩转，但是如果我们要学习微服务架构，自然要去学习更多的微服务架构的技术。


RPC 框架， spring Claude，包括国外的那套方案，GRPC，Kubernetes， Isteel 等等各种眼花缭乱的技术，那它的技术门槛肯定是提高一层的。不过同学们，我问大家一个问题，那这些复杂度究竟是我们应该考虑的，还是这些框架已经替我们搞定的？很简单，它所谓的复杂度，其实只是一个技术的入门成本，而它真正的这些部署结构、项目结构之间的复杂度，很大程度上其实都被我们的框架层面屏蔽掉了。比如说我们接下来要去学习的 spring Claude 框架，一套完整的微服务架构的方案，那一套服务治理理论，可以让你像统帅一样去统领三军，去管理成千上万的微服务的集群，而且他还附加了全套的微服务的技术解决方案，比如说负载均衡，再比如说降级、熔断、限流，还有还有我们微服务里面的一个配置中心以及链路追踪等等。基本上咱能想到的这些技术问题难点都被屏蔽在 spring Claude 的这一层。


那除此之外，对于部署结构来说，我们现在的大型应用通常都是使用 Docker 做一个容器化部署，可以快速复制水平扩展，做弹性计算，再结合了 Kubernetes 做容器编排，那整套部署结构以及项目结构的复杂度基本上全部被屏蔽在了各种开源软件这一侧。那对于技术人员来说，只要迈过这个学习的门槛，那我们就像真正的搬砖工一样，去拿这些已经垒好的砖块搭建我们的应用就可以了。那这是老师在做微服务架构微服务设计的时候最直观的一个感受。OK，那除此之外，那如考虑到硬件成本来说，那微服架构自然需要更多的硬件资源。不过在如今的云计算大行其道的当下，咱其实硬件资源的开销已经非常的廉价，那它不再是一个物理机器，而是虚拟机部署。那你在阿里云、在腾讯云上所买到的资源，实际上它是从一个硬件上面虚化出来的计算能力，那它的硬件成本其实并不高。


那我们在当下去实施一套微服务架构技术的方案，它的成本、学习成本、搭建成本，包括你的投入成本其实远远小于若干年前我们实施为服务方案。OK，那这就是摆在眼前的一个现实。那同学们如果在网上去搜索微富的优缺点，可以搜索出很多文章，五一例外都过分夸大了微服务的成本和部署结构等等。
