它没有很标准的正确与错误，它实际上是考察你的思考问题的方式，你的思维过程。那么对于吹水问题，通常我是用这样的一种方式解题的，这个解题的过程是首先从大局观上面层面来肯定你微服务的好处，那么再从细节层面去举出这里面的一些小缺陷，然后再把你前面表述的内容形成一个闭环，提出一个方案怎么来解决？那这就是一个非常连贯的思考问题的过程，它不止可以回答咱的这个微服务架构利弊的这个问题，同时也可以应用到很多的 open question 这种开放式的吹水问题当中。那同学们就记住这 3 点，大局观、细节闭环。


大局观就是夸夸团用来说这个方案的好处，细节，摆出某些要解决的问题闭环你自己的解决方案非常简单，比如说我们张口就来，从大局观层面来说，我们就要回答什么呀？面试官想听到的一个答案，那微服为什么好？老板为什么喜欢用微服务压榨团队？那这里我们就说出他们想听到的答案，比如说它可以大幅增加你的迭代速度，同时你可以对每个微幅做很细粒度的控制，比方说我的降级、熔断、限流等等都可以做很细粒度的控制。那与此同时你的每个服务模块都是由团队这边做封疆大吏独立发展，那不会像单体应用这样出现每修改一个地方就会引发牵引，发而动全身的这种效果，所以对于一些新需求的扩展能力也是相当快的。


那当然了，还有很多好处，这里同学们可以试面试的时间来去自己的往里面插播一些优点，从大局观层面来说，先肯定你的答案，先让面试官听到他想听的答案。然后我们这里不能一味当夸夸团，你总要提出一些问题，体现在什么呀？体现在某些小的细节上面，这里就表达出了你对这个问题的一些细节上的思考。这个细节不是抠技术细节， open question 这里基本上我们不要陷入到技术细节当中纠缠。我们这个细节也是相对来说宽泛一点的。


比方说你的微服务架构对你的运维能力提出了更高的要求，那你的应用部署、容器化以及你的网络层搭建等等，都是远比单停用要复杂的。与此同时，你需要更强大的中台业务以及各种外设。比方说你的 log 检索系统，你需要在每个微赋之间做打标，做检索，以及你的负载均衡，以及你的跨应用的分布式事务等等，这都需要是增强的一些细节。


还有一个很重要的点，你的微服具体实施效果怎么样？这里强依赖于你的领域建模，所以需要你的架构师或者有相当丰富经验的业务专家来去指导这个过程。与此同时对于技术团队来说，也提出了更高的业务知识构建的这样一个需求，那这是一些细节层面的问题，但是同学们发现不是技术细节，而是对大局观的一些补充。


那这里我们来提出问题，让面试官知道你的思考，最后再把这两者糅合进来，说出你的举措如何解决我们这里提出的一些问题，那这样一来，从这循序渐进的三个阶段，就会让面试官感觉到你是一个非常有想法而且非常有大局观的人，而且又懂得思考。okay，那这是第一个问题分享，接下来我们来看**第二个问题，这里问 WAF 的计数选型，你怎么来考虑？实际上这个问题几乎就有一个标准答案，就是 spring Claude**。

那你弹起微服务它技术选型其实远不是说你选择 double 还是选择某个 RPC 框架治理的微服务，它不仅仅是服务治理，而是说整个微服的平台的搭建。那现在目前市面上我可以找到的这种全家桶的微服务的架构框架，那真的就是 spring cloud 首屈一指。


不过我们这里不可能说 spring Claude 好，我们就用它，那你这里要让面试官知道它究竟好在哪儿。那我们这里夸夸团啊有很多的点可以来支撑我们的论点，比如说我们这里背靠大树好乘凉。首先你 spring Claude 它是直接来自于咱 spring 开源社区的技术贡献，那这个可以保证它在未来很长时间内依然保持非常旺盛的更新以及生命力。那从 spring Claude 的版本迭代当中，同学们就可以看到它的版本迭代非常的快，而些每个版本里都会有持续不断的新的功能加入进来。那与此同时我们还有很多的业界的大厂，比如说 Netflix 或者是阿里巴巴这些技术沉淀非常雄厚的大厂作为它的组件的供应商，那这就是它非常大的一个优势，那与此同时它提供了一个一整套的一篮子工程。


我们不用再去苦苦追寻我这个组件用什么，那个组件用什么？这组件与组件之间怎么来适配？ spring Claude 这里有它非常丰富的组件库，我们引入了 spring Claude 的 dependency 这个应用，你可以在其中找到你搭建一个微服务所需要用到的几乎所有的组件，从服务注册中心，配置中心、负载均衡，服务间的调用，以及你的网关层，还有链路追踪等等，还有你的消息驱动等等。这些所有组件它都可以通过一种最兼容的方式给你提供支持。


同时， spring Claude 现在也是业界不管国内还是国外非常主流的一种微肤的解决方案，那你的受众非常广，因此对于一些潜在的问题，那你被发现被修复的几率也就越大。所以你要想找跨 spring Claude 的点，那真的非常多。这道题基本上算是送分题了，接下来我跟同学们再分享一道送命题。第三个问题，我这里有的面试官可能会看到你当前项目使用的是Eureka，他会问到你为什么不用necklace，或者说你为什么不用concile。那这里其实体干，我们可以分析一下面试官想问的问题是，你为何不在项目当中使用最新版的组件，而使用一个已经停止更新，进入维护状态的Eureka？这个问题经常会被问到，因为我们很多的老系统在几年前做的系统，它直到现在依然是使用的Eureka，这是很正常的一件事情，那么我这里示范一个错误的送命答案，那就是我也想用necklace，但是我说了不算，这个领导也没有替换的计划。其实这个就是非常非常错误的一个示范。


面试官问同学们的问题是想听大家自己的思考，而不是同学用某些借口把它绕过去那一个正确的方向，我们可以这样回答，对于一些老系统是否需要去追逐新的业务组件，那这里你的回答领域回答方向就会非常的宽泛。比方说我做技术更新迭代的时候，我技术替换要思考哪些点？我是为了替换新技术而替换，还是有它真正的业务需求？那如果我在当前的老系统应用上面， Eureka 它的稳定性和它的性能已经完全可以满足我的需要，那我为什么需要替换 negos 呢？当你去思考替换一个新的组件的时候，你要思考它对你的业务可以提升多大的帮助，那它是否有一些新特性可以去大幅助力你的业务场景，那又或者说它可以大幅降低你的运营成本等等。


那我们在技术领域这里一个很大的忌讳就是为新而新，看到新的就用它，看到 spring Claude 出一个新版本就用它，那其实这是一个技术人员很容易犯的一个错误，而技术组件的替换可不是你技术人员的一厢情愿。你要去思考很多的问题，比方说你团队的时间成本，你的人力投入成本以及替换成本带来的运维成本，还有你团队的学习成本，那这些都是cost，都是花销，那与其对应的你的这个替换功能在远期可以给你的项目带来多少收益？这个是benefit，那我们要去平衡 cost 和 benefit 之间，去看一下你当前这个替换决定是否合理。OK，这个是正确的解题思路。
