Dubbo  它是一款 RPC 框架，咱不用被它的宣传标语迷惑，你看宣传标语提到是什么呀？ is a high performance，宣称它是一个高性能。其实所有 RPC 框架它都有这样一句标语，都说我很快，咱就当听一听就好了。除了快以外， double 这里主要提供了一些功能，是一个最核心的是面向接口，远程方法的调用。那有 RPC 使用经验的同学们应该对这一块比较有体会， RPC 框架只用声明一个调用方的接口，那么我们在方法里去做自动的寻址，从而调用到我的目标服务这一套框架。它背后的支持是什么呀？服务治理，也就是我们的服务注册、服务发现功能。


除了远程方法调用以及服务智力以外， double 这里还提供了一些智能容错，还有负载均衡的功能。负载均衡这一方面其实和 Claude 当中的 ribbon 有点相似， double 同样也去内置了很多种负载均衡的策略，也可以像 ribbon 的 iping 机制一样，去智能的去感知你的下游的服务节点，它当前的健康状况，从而减少你调用的失败率。
从架构层面来说， double 这里还有一个比较可圈可点的地方是什么？是它的可扩展能力，它这里遵循的架构模式是一种微内核形式的架构，采用了这种插件化的原子，你的这些核心的功能，像你的协议栈，还有你的序列化等等，这些点都被设计成了那种非常可便于扩展的点。在这些扩展点上，我们可以非常方便的去更换你内置的一些实现方案，或者我通过一些自定制提供一个外部的实现方案，都可以去安插在这些扩展点上。


这种架构模式同时也是我们现在主流的开源软件当中推崇的架构模式，所以老师这里也推荐同学们有时间可以去读一读 double 的源码，去理解一下这种微内核和插件式的这种架构原则，这个对你的代码能力相信会是一个很大的提升。咱提到 double 竟然是阿里系的出品，那同学们一定会想，哇，双十一诶，就是 double 在后面撑起了双十一其实并不是双 11 背后的功臣，另有他人。其实在淘系的上下游链路当中，我们在集团 nail 并不使用double，而是使用的另一款框架叫HSF，我们也亲切地称它为好舒服这个框架全称，其实是 high speed service framework，高性能的服务框架。这款框架之所以能在集团当中大行其道，其实是有一部分的政治原因的，因为这款框架是谁开发的？淘宝淘系，得益于淘系这个势力在整个阿里集团当中占有的话语权和统治性地位，所以在集团上下游的链路当中，都去使用了淘系的这样一款 RPC 框架HSF，它就成为了集团内部的一个既定事实。

尽管 HSF 也通过阿里云对外输出，但是从我自己的使用体验上来说，个人还是建议大家使用 double 框架，因为 HSF 相对它的依赖要求比较高，因为 HSF 这里还是依赖于很多集团内部的组件，比方说在早期的时候我还在阿里工作的时候，使用 HSF 的话还是要依赖于阿里的Tomcat，并且还要依赖于淘系自己的配置中心，这之间组件之间有很强的依赖。尽管后期 HSF 这里做了很多的努力来剥离依赖，不过现在还是需要依赖于 EDAS 这类服务发现的框架组件。


所以从开源的角度来说，我更建议大家去在自己项目当中尝试使用double，同时 double 也是国内项目开发当中的一个首选方案，基本上 double 已经成为了国内软件行业开发的一个既定标准，我们看完国内的情况之后，再带大家来看一下国外的情况，看看水深火热的资本主义，他们的主流技术方案是什么呀？这里冒出了一个鼎鼎大名的公司，谷歌。

谷歌这里厉害了，他也开发了一套自己的 RPC 框架，叫做 g r p c。那老师现在任职的公司就是使用 g r p c 来做开发的。通常 g r p c 这里会搭配一个描述性的语言，叫做 per to buff。 per to buff 这种语言实际上是用来定义一些接口，还有公共类的，它是一种跟平台，跟语言无关的一个描述性的语言。


对于 Java 来说，如果我们把 Protobuf 接口层定义出来，可以使用一些眉稳插件，根据 Protobuf 自动生成 GRP seed 访问层以及它的相关的类，不用你写任何代码，你在 protobuf 里面定义的所有接口以及类都会自动生成出 Java 的代码。


同时 g r p c 这里还提供了非常丰富的异步化的知识，在调用的时候可以自由的选择。我当前是一个同步阻塞调用还是一个异步调用， g r p c 通常还会将它与另外两种技术一起来搭配使用。 Kubernetes 就是其中的一种，咱也亲切地称它为K8S，也是谷歌去开源的一个容器编排的引擎，学过一期课程的同学应该对它都比较熟悉了。 Kubernetes 里面我们可以去做自动化部署的脚本，也可以去做集群的可弹性的伸缩以及容器化的管理。这两个都是谷歌自家出品的好搭档，两个搭档还不够，还有三兄弟， i still 是一个网格计算的服务框架，基于 GRPC Kubernetes 的服务集群去集成 s still 非常的方便。那我们这里只用加一个 sidecar 插件就可以为自己的服务来添加 s still 的支持。


i still 这里主要是提供了像负载均衡服务间的一些认证以及监控这些功能，同时可以去配置一些很复杂的路由的规则重试或者是 fill over 故障转移，对我们用户的访问流量进行非常细粒度的控制。这三兄弟基本上是国外的开源项目当中比较推崇的服务治理的方案，它在国内的应用非常的少，不过在国外却是大行其道，像老师所在的公司就是使用了这样一套云原生的技术实现。不过这三兄弟要想把它搭建起来，可比 double 它的难度要稍微高那么一个层次。如果你的团队当中没有很强的技术储备或者大牛的话，慎用这套方案，否则的话会踩坑。

我们的 spring Claude 为什么把它叫做微服务领域的集大成者？很明显的一个原因就是它不只是关注于服务治理，服务治理是微服务架构的基础，但是却不是微服务的全部。


整个微服的全貌还包含很多的附加的功能。我们这里 spring Claude 它打包了整个全套微服的解决方案，比如说负载均衡以及降级熔断，还有流量整形限流，还包括微服务的配置中心以及微服的链路追踪，并且还集成了相应的消息组件。除此之外，它还有分布式事物的知识，简直是包罗万象，无所不能。同时这个框架使用它还会让你体验到一个速度和激情。


那什么意思？不是说这个框架它的性能非常非常快，而是说它的更新迭代速度非常的快，基本上一年出一个大版本，自我更新迭代的生命力非常非常的旺盛，并且它的开源社区有非常多的活跃贡献者，绝对是一个非常生机勃勃的方案。不仅如此， spring Claude 在国内和国外都有非常大规模的成熟的应用，所以如果从微服务领域这个角度来出发，使用 spring Claude 是一个不二的选择。所以我们在微服治理这套大体系课程当中，也采用 Sprint 所有的最新组件，跟同学们去展示一个微服务架构它应有的一个全貌。

同时也是一个开源的框架。那像腾讯它有自己的框架叫Tars， Tars 也是它自身的协议栈的名字，基于 Tars 协议的高性能的 RPC 框架。最后像蚂蚁金服别出心裁了，它并不用集团的double，也不用集团的HSF，而是自己开发了一套 Sofa RPC。
同学们是不是有一个问题，这些框架它是不是大厂出品？没错，都是背景技术实力非常雄厚的大厂出品，而且也都是开源项目。咱在自己的项目当中能否使用这些框架？这里老师并不建议同学们使用这几个框架，为什么呢？它们虽说都是服务治理的轮子，但是轮子与轮子之间还是有很大差别的。


我们在去做技术选型的时候，其实有很多的考量点，首先它必须是一个开源的大作，这样的话才能保证你的当前使用的这套技术方案有足够的生命力去往后演进，不断的去做完善。除此以外，我们这里很重要的一点是要强调一个社区贡献，一定要去采纳来自于开源社区的代码贡献。而像我们上面提到的这几种方案，他们的主要贡献者是谁？还是商业公司自己。比方说美团自己贡献Octo，基本上绝大部分的贡献都是来自于企业内部，其实这就是一个风险点，打个比方，如果有一天我的公司说我不想去继续投资这个项目了。


在开源软件当中，我们见过这样的例子，比方说阿里巴巴 double 框架也曾经沉寂了很长一段时间，就是阿里内部宣布我暂停 double 的一些更新，直到近两年， double 才又重出江湖。除了 double 以外，摆在眼前的一个例子就是 Netflix 在初期曾经贡献了 spring Claude 当中大部分的核心的组件，不过在后期 Netflix 公司逐渐放弃了对某些核心组件的维护，比如说 Eureka 还有 high streaks。所以我们在考量选择轮子的过程当中，一定要去关注它是否有来自于社区的代码贡献。只有开源社区去踊跃参与，才能保证你这当前项目是一个有生命力的项目，可以持续地做演技。


另外一点也是很重要的一点，我们要去参考你当前所选择的这个技术栈，这个轮子是否是一个技术界的既定标准。比方说，如果我们选择了以上这三种技术方案，你就是处于一个孤立无援的境地了。不管是国内还是国外，这几种技术方案的应用并不多。开源技术其实也有点像股票庄家都喜欢什么呀，叫抱团，像我们国内抱团使用double，国外抱团使用GRPC，以及国内外通通抱团使用 spring Claude，使这类技术成为业界开源界的一个既定标准。


这样一来，一个好处就是你可以参考的例子非常多，它的文档也会非常的完善，你所碰到的问题大概率被其他人也都碰到了，既然使用你框架的人非常多，你框架当中所存在的 bug 有一些问题自然很容易被发现，这样一来就形成了一个正向的循环，使用的人越来越多，你框架当中存在的问题会得到更充分的暴露，进一步的修复之后会让你的框架越来越趋于稳定。所以这也是我建议的，让大家尽可能地采用比较流行的，成为既定的事实标准的开源项目。
