那我们首先来看基于客户端的负载均衡，这个客户端实际上就是指你这个应用的发起人，比如我的订单服务想要访问商品服务，对于咱的客户端这个负载均衡策略来说，就是由订单服务这里在本地来决定我想要调用哪一个商品服务，这个就是客户端负载均衡的表现形式，这种模式就是他俩儿单线沟通来去传递情包工作。那如果我们采用服务端的这种负载均衡形式，在他俩中间，我这要做一个中间的话事人的角色。


有订单服务来告诉一个独立的服务端的负载均衡组件，说我要访问谁，然后有这个服务组件来去代化，把这个请求根据它本地做的负载均衡策略来转到对应的服务。这种方式叫做是服务端的负载均衡。这个客户端和服务端这两种不同的负载均衡策略的方案实际上并不是让同学们来去做一个二选一的艰难抉择，我们没有这个需求，它们分别会应用在不同的场景里面，各有用处。接下来我们就先从客户端的负载均衡策略看一下，如果我想应用它，我需要什么样的先决条件。前面讲到你的客户端负载均衡是由发起方直接执行的。这里第一点，我发起方我怎么知道我的下限是谁？那这里我需要依靠什么呀？依靠一个注册中心，通过我的服务发现机制把所有的已注册的服务怎么样同步过来，花名册注册表同步到我的服务节点。这样一来，我就知道你有哪些下游服务可以被我调用。与此同时，我也要确保这个服务列表当中的机器的时效性，比如说你下线了，我要去感知到，或者说你在一段时间内不可用处于一个不健康的状态，同样的我也要获知到，这就是自动感知。自动感知实际上它是咱客户端负载均衡的一个优点，也是通过服务发现机制来去注册的。所以说对于这种方案来说，我们非常依赖于你的一个外部的注册中心，那因此它的应用场景大多就是说微服务架构的系统采用客户端居多，这是两点。


第三点它还有一个非常显著的优势，就是我们可以在代码当中非常自由地选择负载均衡策略，配置不同的策略，或者说我延伸我实现一种我自己的负载均衡策略，这个自由度非常的大，不用大动干戈的在服务端做额外的配置就可以生效，这是客户端负载均衡的几个比较鲜明的特点。


咱现在转向服务端负载均衡。服务端谁是服务端？订单服务、商品服务，虽然挂着服务两个字，但肯定不是服务端，咱们服务端通常来说肯定是一个单独独立的网关层组件，比如说像老师现在的公司就会使用硬件加软件结合的一种负载均衡策略，那就是F5。 F5 也是一个在国际上非常有名的一个网络供应商，很多外国的公司都会使用 F5 来做他们的服务端负载均衡，这是一个第二个，对于国内公司很多采用都是开源界的项目，不像国外公司这么豪气大方，那就是能不花钱就不花钱，所以这里经常采用谁 LVSGIP 加网关组件来作为一个单独的个网关路由器。


接下来咱就做一个简单的对比，看一下咱服务端单独把它拎出来的这样一个负载均衡的组件，和咱的客户端直接发起这两种方式有什么不同？比如我这里服务端，它第一个非常鲜明特征的就是说我要一个额外的网络组件，这样一来实际上也是增加了你的维护成本，你的部署成本，还有你整个链路网络层的复杂度。


第二个我修改一个配置，去修改你的负载均衡策略，包括说我是往集群当中去添加一些机器，这些其实都是需要额外的配置改动的。像我在用 F5 这种网关层的负载均衡策略的时候就是非常的痛，每一次改动开发团队是不能直接去做的，必须要去找对应的网络团队，对应的运维团队走很复杂的流程，非常的不方便。


第三点，你的网络开销相对较大，因为对于客户端来说，你的客户端负载均衡实际上就是一个点对点通信，直接发起，它只有一次网络请求，那么对于服务端这里我相当于把这个请求发给中间的这个网络组件，再做一层转发。这样一来你的网络请求次数多了，你的网络带宽占用了最显著的一个点，就是说你服务端的负载均衡策略，网络耗时肯定是要比客户端要大的，因为它多次网络转发，在时间上面肯定要花费更多的时间。最后一点自然也有它的优点，也就是高度可配的 LB 的路由。不同于咱这个客户端的负载均衡策略，客户端实际上即使像 ribbon 这种很成熟的组件，它给你的负载均衡这些策略，这些规则很多都需要去自己定制。但是如果我们是采用这种服务端的组件，比如说像这种商用组件， F5 里面的功能非常非常的强大，各种各样的配置链路。


其实不光F5，大部分你花了钱的商业版的服务端负载均衡的解决方案，那相比客户端来说，他们就是什么呀？就是人民币玩家，而且还不是小r，还是一个大r，毕竟花了钱的，所以说它的负载均衡策略，包括它的路由，还有各种监控、预警、高防等等一系列服务配套建设的会相对完善一些。


这是这两种方案跟同学们去展现的一个大致的对比，还有它们的特点。接下来我们来了解一下，在什么样的场景下，一般我们会倾向于选择哪个方案。前面跟同学们讲过，这个服务端客户端复杂均衡，它不是一个二选一，对于我们的客户端来说，

通常来讲咱的客户端负载均衡策略应用的场景就是微服务之间的调用，因为我们在微服务项目当中，本身你的一个系统就会被细分为很多个这样的子系统应用。那如果我们在这个系统之间不能直接的发起调用，而是要经过一层网关层，这么绕一下，一个调用链路下来，它有可能会访问七八个甚至十多个不同的微服，那你这个网络耗时是非常大的。


你像淘系链路，一个下单场景茫茫多的微服务，不可能说是让你每一个微服务调用全部经过服务端负载均衡绕这么一圈。所以对于微服务的解决方案来说，服务端与服务端之间，我们比较倾向于使用客户端负载均衡方案，像 spring Claude 也恰恰是这么做的。这样一来就说你必须要有一个什么呀？服务注册组件在外面 hold 住所有的服务列表，这是它的一个要求。另一方面，如果我们没有服务端的这样的服务注册组件，其实也是可以采用客户端负载均衡。


我们 ribbon 这边实际上也支持一个可配置的服务列表，那如果我们明确当前集群当中有哪些服务器，那么实际上是可以把这个服务器列表配置在每个客户端的，也能达到一个客户端负载均衡的这样的效果。但是我们这里并不太建议用它。为什么呀？因为你使用了这种方案，它脱离了你的注册中心，你很难知道某些服务下线了，某些服务它是否当前不可用，这个是很难去主动获知的，所以说容易产生一些网络异常。


所以说这种方式配置的客户端负载均衡策略，它的请求失败率肯定会非常高的，在你服务上下限的时候，那肯定会发起一些错误的失败的网络请求，那这是一点。另外我们来看一下服务端负载均衡策略会用在什么样的场景下？第一个跨网络的 zone 的调用，比如说即使我们当前是一个微服务的集群，在当前的一个 zone 里，我可以把它叫做 secure zone，这个 zone 一般会存放相对内部的一些很安全域的组件。如果我要访问一个 Web zone，那这个 Web zone 通常来讲我可以去部署一些对客户端直接暴露的服务，那么我叫它 external zone。


这些跨 zone 之间的调用，通常来讲我们倾向于什么？倾向于通过中间的一层服务端的网络层来这做一个代理，而不是说让这两个不同网络区域的微服务之间互相你来我往，这个是跨纵调用的情况。第二个网关层的组件，比如说我们用 Nginx 或者是其他的网关层选型，那么一个请求到达了一级网关路由之后，通常也会由网关层组件直接把它转向二级网关路由，经过很多级的网络转发，最终才会打到你的微服务的网关组件上面来。这是第二个场景。第三个硬件负载均衡器，就像老师前面说的什么呀， F5 这种软硬件结合的硬件负载均衡器，通常它都是使用的服务端的负载均衡策略的方案，这是一个客观现实。最后一点，我们这里不得不采用服务端负载均衡的策略。还有一个情况，就是说我不具备服务发现的条件，也就是说咱这个注册中心压根不存在，我没有搭建这样一个注册中心，所以这种情况下，我不得不借助一个中心化的负载均衡器来去做各个服务之间的转发。这种方案。在一些公司将自己的项目从传统项目拆分成微富项目之间的这个过渡过程当中，经常会存在这样一个过渡方案，在服务的注册中心没有全部集成上线之前，我们会采用这种服务端的负载均衡策略来去搭把手，帮他平稳过渡。这一节当中，对客户端服务端负载均衡策略的分析还有它的适用场景就跟同学们讲到这了。在接下来的小节里，我们就一起去深入地了解 spring Claude 当中是如何来做负载均衡的。**这里将跟同学们分别介绍一款 spring Claude 的老牌劲旅负载均衡组件ribbon，还有一个后起之秀，用来替代 ribbon 的 load balancer**。同学们，那我们下一小节再见。
