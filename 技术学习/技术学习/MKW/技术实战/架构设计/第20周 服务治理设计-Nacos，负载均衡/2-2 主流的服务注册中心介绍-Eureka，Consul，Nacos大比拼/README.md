在后面小节里我们也会跟同学们一起来手动手来操作，搭建一个 Neques 的服务集群。服务注册完成之后，作为一个上游应用，我要去对下游服务发起一个调用，这时候就要依托于服务发现机制，将注册中心这里所有的服务注册表给它拉到本地。这样我就知道我想要调用的服务，它们都有哪些可用的节点，这些节点的地址、 IP 还有端口号都是什么，然后我就可以去顺理成章发起一个远程的服务调用。那这次调用可不用再经过服务注册中心了，那是由端到端直接发起的一次调用。


在这个过程当中，任何一个服务还需要去将自己当前的状态及时同步上报给你的注册中心，这个流程我们就叫做服务续约，或者叫心跳续约都可以的，那在这个过程中，如果因为网络原因或者因为硬件故障等等，你的服务续约心跳暂停了注册中心这里一段时间收不到你的持续反馈，会怎么样？把你的服务给它下线掉，这就是服务剔除的一个流程，那这个我们也叫什么被动剔除，那同时我还有一个主动下线的操作，那这 12345 这 5 个步骤可谓是万变不离其宗。你只要是作为一个注册中心，怎么说这几个基本的功能还都是要实现掉的。

这个 p to p 不是那个炸雷的 p to p，它是 peer to peer 对等通信的这种方式，它采用了这种去中心化的结构，没有什么master， slave leader follower 没有这之分。那每一个服务器它其实都是对等的，角色，都是一样的。服务器与服务器之间，我们通过互相注册，互相备份来做一个高可用的架构。因此从性能角度来分析， Eureka 正是得益于这种 AP 方案 p to p 的对等通信的这种特性，所以它的速度犹如脱缰的野狗一样快，非常的快。那么 console 这里我们采用的是 CP 的方案， CP 我强调了一致性，牺牲了可用性，那这样一来，你在性能方面也就不具备什么优势了。

那雪崩保护是咱三高应用的服务注册中心当中的一个比较重要的特性，专门用来应对某些特别的异常情况，比如说我短时间的网络抖动导致服务异常下线，或者说我为了应对一个真实的服务雪崩的情况，打个比方，我现在 50 台的服务器集群发生了服务雪崩，那最后崩到只剩下三四台机器可用。


如果这个时候我的注册中心的服务列表当中把这些不可用的服务全踢掉，只保留可用的服务，那这可怜的三四台可用服务也有可能顶不住压力进一步挂掉。那所以就需要去做一个什么呀？雪崩保护什么含义？比如说我这边配置了一个阈值 50% 的机器，那如果我当前服务的可用性小于50%，也就是说你有一半的机器挂掉了，那么这种情况我就开启了一个雪崩保护开关。服务自保开关，我的服务注册中心就不会对不可用的服务做下线处理，那这个其实就是雪崩保护，同学们整明白了，那接下来我们再看，像 K8S 还有 double 的支持uriva，这里对 K8S 其实还是可以采用某些曲线救国的麻烦点的方式来做支持的。不过对于 double 来说， Eureka 表示 double 什么玩意儿没见过，不支持，那考虑到 double 在国内应用场景非常的广泛，所以阿里官方这里 double 自己去主动抱大腿提供了 cancelled 支持，那所以说这个支持这个 y 是 double 他自己开发的一个 console 的支持组件，那作为 spring 官方其实并没有说把 double 吸纳过来作为一个官方的 RPC 框架，因为确实国外的公司用 double 的非常少，只是国内使用 double 的会比较多。


OK，那对于一些其他的很重要的特性，比方说服务治理过程当中的服务剔除、心跳续约等等，那咱前面提过这个都是基础加点，你的主流的注册中心都具备了相应的功能，那如果最后我们要做一个应用选型的推荐，我个人建议是这样的，就是 Eureka 这里虽然具备很广泛的群众基础，同时也是一些 Laxy 的微服务应用所选用的组件，但是考虑到它毕竟已经停止了开源计划，所以我们从持续演进的角度来看，这里建议咱的新的服务应用尽可能地去转向后面的两个注册中心。
