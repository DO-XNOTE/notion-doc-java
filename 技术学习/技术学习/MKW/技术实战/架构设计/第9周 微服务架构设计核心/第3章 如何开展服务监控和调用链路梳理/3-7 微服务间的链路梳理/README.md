## 线上问题排查当中还有个内功的心法是什么呢？诶，这里就要提到一个叫 global trace ID，那它是做什么事的？它是从你的应用接收到了一个服务请求开始，直到它整个调用链路结束这从头到尾的所有流程，我通过一个全局唯一的 trace ID 把它给串联起来。



那我们在后面的微服的章节当中，将和同学们一道来看 spring cloud 当中的 sleuth 的组件是如何来做这种 global trace ID 的生成以及串联的。OK，那对于大部分的微服务应用，不管你是使用的什么样的服务治理方案，那它生成全局 trace ID 的方式都是万变不离其宗，大部分是通过在上下游调用之间，微服务请求的 header 里面加入某些特殊的参数来传递这个上下游的参数的。那么我们在做线上链路追踪的时候，其实并不只依靠这个全局的ID，我们还会在每一个调用的分区当中产生一个span。


i d span 是什么意思？比如说你当前的用户数据的调用是一个服务请求，那么在这个请求内的所有的业务逻辑，我把它划归成一个span，那这里面打出的日志记录都会带有一个相同的 span ID，那同理，每一个微服务的调用，在它的方法的上下文当中都会产生一个独一无二的 span i d。
那么我们的整个调用链路追踪就会生成一个非常清晰的路线，由全局的 trace i d 从头到尾串联起来，并且每一块应用，每一块方法调用当中，它都会关联到一个 span id。那么对于线上复杂问题的排查，
