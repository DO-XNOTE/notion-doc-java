比如第一个业务定制，第二个注解卖点什么含义？业务定制就是我们在开发阶段一种侵入性比较高的手工指定你埋点内容的一种方法。比如我们这里调用某些中间件，像阿里系的鹰眼链路追踪系统，那它在做业务定制化埋点的时候，就提供了这样一种侵入性比较高的，但是控制力度非常精细的埋点手段。


OK，那注解埋点是什么意思？其实它大多是运用了 Sprint AOP 的原理，在你的方法当中通过添加一些自定义的注解，去把你的方法请求的 request 还有 response 里面的数据，把它给抽取出来加到业务埋点当中。同样的我们也可以在数据库连接池层面，通过加上一层代理proxy，把数据前后的变更记录下来，这是一种注解埋点的方式。


那么我们从代码侵入、信息完整度还有精准度这个方面，对这两种方式做一个简单的比较。首先我们看业务定制的方式，前面说过这里为了达到精准的埋点控制，所以它需要在我们的程序当中去添加业务代码，那这个其实是你肉眼可见的一种侵入，在你的业务流程当中添加了一段跟流程并没有关系的一段埋点代码。OK，那我们从信息完整度和精准度这方面来看，你业务定制化的这种埋点方式可以随心所欲的让你在程序当中控制你想埋点的具体的数据，那我们可以把这个力度控制到你的核心业务数据当中，它不会掺杂一些干扰项，那么你做数据清洗的时候就会省事的多。那对于注解埋点来说，它的代码侵入相对于业务定制来说要小很多。这里通常来讲只是在你的方法层加一个注解，并且约定一些基本的格式就可以了，那与此同时它的信息完整度稍差一些。


为什么？你既然是作为AOP，那么你在整个代码控制的过程当中，你可以对它的入参出参做一些信息的埋点，但是在你的方法执行过程当中的一些信息，可能你无法很好的把它抓取出来，那它只记录一个，一出一进这两个动作就相当于你在宾馆前台做登记，那你这个人进了宾馆叫什么名字？登记下来，那你出了宾馆花了多少钱，在里面住了多少小时，给你记录下来。但是你在这段时间内在宾馆里面做了什么事情，那我就不知道了。因此这种注解方式，它在精准度上面相对于前面的代码侵入式的业务定制，那肯定会稍差一些，那这是后台埋点的作用方式。

同学们想它的作用点是哪里？既然是前端，那它的作用点就是前端页面上的一些控件，那这里包括我们在手机端 Mobile APP 上面的一些小按钮，这都是可视化埋点生命是埋点的作用点，那通常它是这样的一种响应模式。


首先你的控件拿到了一个用户响应，那比如说用户点击了下单按钮，那它戳了一下这个控件，紧接着就触发了一个响应函数。OK，这个响应函数控件的响应函数它做什么事？对于埋点的这个业务需求来说，通常它需要做这样几件事，第一个它要去定义一个埋点的开关，你当前的这个响应函数是否要做用户埋点？OK，那如果它这个响应开关是开启，那么我们接下来就去触发到一个埋点的函数，那在这个函数当中，我可以去定义当前的控件需要做哪些埋点的内容，那么这个埋点函数通常是一个异步响应的这样一个函数，因为它不可能阻塞前端的用户体验。它在触发了埋点函数之后，异步的向后台的埋点服务发送这样一条信息。
