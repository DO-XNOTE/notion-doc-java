紧挨着这个反向代理层就把你的请求转发到了 1234 这个四台服务器组成的一个集群当中。


那这四台服务器它里面部署的东西有什么区别？没有，完全没有，这就是一个典型的单体运用，把它做一个水平的集群扩展，每台服务器上面部署的东西都是完全一样的，同一个巨型的单体的war 包，那它们共同的连接到了一个后台的服务器集群当中，这就是上古时代应用的样子，那甚至在现在很多的大厂他们的遗留系统或者是巨无霸的系统依然保持着这样的结构。


那么我们说它的缺点，它慢在哪里？这里第一个缺点不得不说的就是牵一发而动全身，那咱的这每一个应用服务器部署的东西都是完全一样的，那你的每一个小的改动都很有可能影响到其他的功能点，那因此每次改动对应着你要做全链路的回归，这是一个非常耗时的一个过程，与此同时，对于这种水平扩展的单体用户来说，它的回滚是一个灾难性的话题。


你的一个功能，咱跟其他的所有功能一起打包，一起上线，同生死共进退，要生一起生，那要回滚，咱一块都要回滚。我好端端写的一个功能上线了，诶，结果因为其他的别人提交的一个小bug，导致你的功能也被一起回滚，OK？不光回滚是一个问题，那它发布周期也是一个不小的问题，那对于这些单停用来说，发布周期通常会非常的长，那少则两三个月长，长则一年，平均算是半年发布一次版本，那么对于这种迭代快速适应市场的这种需求是完全无法满足的。


okay，那最后一点也是被很多开发人员所诟病的，就是应用的复杂度非常的复杂。第一个由于我们是单体应用的，所以它这个代码混用非常的普遍，那么你写的一段基础代码有可能被各个服务方共同使用，你哪怕做一点微小的更改，也可能引发出意料之外的效果。OK，另外一点是你的数据访问非常混乱，比如说我们现在有一个订单表 order 表在你的数据库当中，那么它的底层的数据模型会被你这个单体应用里边各个服务所引用，你即便在你自己的这个 order 表里添加删除字段，也有可能影响到其他的服务。这就是在底层数据访问上没有做隔离的一个代价。


那我们叫这种应用给出一个很贴切的形容词，叫做使善，比起拼多多那种在小便池里大便的这种恶心劲儿，同学们是没有见过。那种几百万行代码的单体应用，那真的就叫一个史山，你可以想象一下一个开发团队整天在史山里遨游的这个心情，搞得这个上班就像上坟一样。那么我们摇身一变到了微服务的时代， 996 的年代，我们的这个应用的架构模式发生了翻天覆地的变化，同学们可以看到咱这里的每一个服务都把它单独抽成了一个独立的微服务，
