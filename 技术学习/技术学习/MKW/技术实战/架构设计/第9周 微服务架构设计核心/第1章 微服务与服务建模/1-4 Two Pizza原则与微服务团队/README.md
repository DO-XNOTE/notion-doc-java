那咱一些传统的公司呢？往往认为说是人多好办事，你的团队规模越大，你的领地越大，话语权又多。那咱的大佬贝索斯可不以为然，他认为你的团队规模如果过大，那么团队成员之间的互相沟通，他的沟通成本都会是个问题。那你整个团队的运行效率还是资源利用率的角度来看，它都不会是一个高效的一个开发团队。


那么早在十几年前，从硬件机器领域转向软件开发行业的一些巨头，比如说IBM，他们就提出了一个口号叫大象也能跳舞。那一时间蓝色巨人 IBM 的大象也能跳舞这个理念盛行，也出了很多本书。那我们如果用上帝视角从现在来看当时的那句话，用互联网的眼光来看，这句话现在就变成了一个笑话。


在互联网行业里，大象是不能跳舞的，你只有把你的这个项目，把你的职责划分，尽可能地细分到一个相对可控并且小规模的团队上，保证你每一个领域或者说每一个微服务领域的相应的微服务团队是一个小规模的团队，这样你才能去降低团队成员之间的一个沟通成本，同时也反映了我们互联网行业当中一个比较推崇的理念，一个精神叫做小步快跑。那由一颗小而精的研发团队来快速落地你的商业需求，做一些快速试错，通过这种小步快跑的快速迭代来逐渐的向前推进你的项目，这是互联网行业当中比较倡导的一个精神。

一般可以用 6- 7 位的核心的成员，一个精英化的团队，一个特战队，那么同时有一个leader，他并不是一个专门负责管人的这样一个角色，它是带头冲锋陷阵，一个文可上阵写代码武可下场跟产品撕逼的这样一个角色。


那么说一个子领域，比方说一个订单域订单微服务，其实不只是一个微服务订单，微服务后面有非常复杂的子域，一般来讲一个这么大的一块儿微服务，会交由这一个特战队来实现吗？不会的，我们往往一个组织一个部门，比方说像阿里系的一个事业部，那这事业部下辖下面有很多个这样的微小的微服务团队，那么他们每个人都会负责相应领域下面的一小块服务。


比方说你的一个订单域，订单和支付系统之间的集成对接，可能是由一个微服务团队来负责，那么订单的生命周期管理，也就是它的状态机管理流转这套系统可能是另外一个团队来配合，正是由这一个比较微小的微富职能团队来共同组建起一个非常复杂的微服务系统。那在咱每一个团队里面，咱的这种精英团队，一般来讲我们会做一个双备份机制。


什么意思？由于在微服务和敏捷的理念之下，我们通常来讲是重沟通轻文档，因此对你的业务知识的积累，对你的代码层面的积累，一般是把这个知识沉淀到了个人，而非是沉淀到了大量的文档上面。因此我们在这种快速迭代的业务当中，每一个开发人员个人在你的业务线上所积累的知识是非常重要的，快速定位问题，解决线上问题都需要这个人的深度参与，因此我们这里对每一个重要的功能点，我们会指派一个 feature owner，也就是这一个功能它的负责人是哪位。


同时咱的软件系统有单点故障的风险，那么人也有可能有单点故障的风险。咱比方说王大锤这个月刚认识了一个姑娘，那下个月就跟他结婚领证，他休婚假去了。第三个月王大锤的新婚妻子怀孕要生小孩了，可能就要休一个很长的陪产假，那么这段时间你的核心成员相当于是处于一个缺失的状态，因此我们会做一个什么呢？双备份，我们会指派一个总舵主和副总舵主这样的一个概念，那每一个重要的功能点作为一个团队的lead，或者说是你团队的一个开发经理 manager 这样的角色，一定要把这些重要的功能点保证有两个人可以去同时来做线上的支持，那这是咱微服务团队在快速迭代的互联网行业的业务当中所必须去考虑的。

第一，我们在互联网行业当中招聘的测试同学都是具备很强的开发功底的，他们会去致力于集成测试框架的搭建或者是效能平台的搭建。那么对于功能性的测试来说，我们一般不是指派到各个团队当中，而是作为一个中间的一个职能方，那他可能是去支持你当前事业部，或者是你当前多个团队的测试需求，那从集成测试的角度来衡量整个质量保控的计划，okay，那这里也相当于是给同学们传达这样一个概念，就是不会写代码的手工测试，那在现在这个行业下面是不可能存活很久的，那他的职业生涯一定是非常非常短的，代码能力是一个硬通货。


好，那我们这里每一个开发团队前面说过，他通常是负责一个子领域，一般是一个微服务下面下设的一个二级域。比方说订单微服务，那它下设的一个子的微服务领域可以交由这样的 6- 7 人的一个开发小组来完成。这个开发小组相当于是一个箭头部队，是贡献开发力量，也就是直接贡献开发力量的一个团队，他们就是构建大楼的搬砖的员工。你既然有搬砖工，那你自然要有一个设计的人员。那在我们一二线公司当中，你的设计人员是谁？同学们，就是正在听课的，你们谁？同学们，看这个两个箭头的天线宝宝，它通常来讲是一个架构师的角色，比方说在阿里系或者是蚂蚁里面，我们大量的采用了这样的一种架构，也就是跨团队的架构师的这样一个职责。


因为架构师前面一直提到过要有一个全局的视角，因此它要去跨项目，去衡量你整体的业务方向和架构等等的问题。因此我们在这里把它作为一个单独的职能部门，来做一个跨团队的技术方案的设计、跨团队的协调等等这样的一个角色。与此同时还有很多的其他的职能团队，比方说为你的后台微服业务去搭建前台页面，或者前台的手机页面等等这样的特殊方向的职能团队。
