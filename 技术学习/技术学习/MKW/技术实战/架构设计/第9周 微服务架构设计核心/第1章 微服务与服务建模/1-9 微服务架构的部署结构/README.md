大家都知道 Docker 是目前最流行、热度最高的容器化的技术方案，那自己的要部署的应用服务，我们把它打包成了一个 Docker 的镜像，那 Docker 可以把它看作一个小型的操作系统， Docker 的世界里是无所不能，你任何的应用其实都可以打包到 Docker 里面，这里不光仅限于订单服务，甚至我们的中间件、 Redis 中间件或者是 MQ 等等，那这些都可以把它打包成一个小型的一个 Docker 镜像。


OK，那么这么做的一个目的是什么呢？很简单，你的这个镜像就成了一个标准化的可复制的一种模板。诶，我们说为什么在全国各地的肯德基、麦当劳，你吃到的这个什么麦辣鸡翅，什么可口可乐都是同样的味道？那就是因为它们有一套相同的模板，可以做一个快速的复制。


okay，那Docker，这里也是一个道理，那我们把一个 Docker 部署在一台服务器当中，那么这样的一个最小规模的一个小型的操作系统，可以被快速的复制到其他的集群当中。OK，那么我们在真实的环境当中，远远不会这么奢侈的一个服务器就部署这么一个 Docker 镜像，而是说我们可以把不同的 Docker 镜像，比如说这里的用户服务以及订单服务，把不同的服务做成 Docker 镜像，同时部署在一台物理机当中。那咱有使用过云服务经验的同学们应该有这个体会，在你的集群上面，其实你是看不见物理机这样一个概念的，那你的所有的可操作的虚拟化单元，其实都是从你的物理机上而虚化分配出的一段计算资源。因此你的同一台物理机好似运行了多个操作系统一般，它可以同时跑起多个不同的 Docker 镜像。


那这些 Docker 镜像之间是如何做快速部署的来？这就要提到我们的 Kubernetes 系统，那它是咱现在当今最流行的容器管理服务作用就是用来快速的分发来部署你的镜像文件，比如说我们在双 11 这种场景下，可以去申请更多的物理的服务资源，使用 Kubernetes 快速的将你的 Docker 镜像复制到这些资源上，那在双十一结束之后，我们可以把这些镜像给它关闭，把服务器的资源返回去。那这样一来就形成了一种按需分配，我需要计算资源的时候可以做快速的部署，那么计算资源使用完毕，流量降下来之后，我又可以去归还你的计算资源。


所以同学们看出来了，**弹性计算怎么好？省钱,那需要机器你就要，不要机器我们就把镜像关闭，机器还回去**，那这就是我们在大型应用当中采用的部署方式，那么阿里系也是应用的这同一套部署方案。okay，同学们，那这一节的分享就到这里结束了，后面我们有专门的章节跟同学们去详细落地一整套容器编排以及容器化技术的方案，那同学们可以好好期待一下。好，这一节就抛砖引玉到这里结束了，我们在后面从微服务的稳定性以及安全性的角度跟同学们再做一番探讨。好，同学们，我们下一小节再见。
