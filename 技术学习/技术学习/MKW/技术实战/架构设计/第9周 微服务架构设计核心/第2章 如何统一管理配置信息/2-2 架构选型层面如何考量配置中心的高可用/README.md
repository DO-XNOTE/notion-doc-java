那它的配置项是从我们的一个配置中心当中获取到，并且保存在本地的。那同学们这里注意到，咱提到配置中心的时候，前面加了一个数量叫一个配置中心，那么假如这个配置中心发生了任何的故障，比如说是网络问题或者是硬件问题，那硬件问题其实在生产环境当中是一个非常常见的一个问题。那么对于大型集群来说，经常会有因为主板或者硬盘等等故障而宕机的服务器，那么如果我们的配置中心这里只有一台，那么当它产生故障的时候，就会发生一个非常严重的单点故障。因此从高可用的角度来说，我们对每一个服务都要去有一个backup。什么是backup？就是一个备选的方案。


好，那这里一个最简单的方式就是集训化。每道提到单点问题，我们第一个所要思考的就是你的这个技术选型可不可以支持一个集群化的扩展？其实这个集群化它的思想非常非常简单，就是让同学们做渣男、做渣女要多养备胎，一个不顶用了，我们可以换下一个。但是在成为这个渣男的道路上，我们有很多种不同的渣的方式。那集群方案我可以选择多副本，主从集群等等不同的方案来解决这个单点故障的问题。所以首要的我们考虑一个配置中心，就要从这个集群能力上面去看它的这个组件有没有集群扩展的能力。


那这是一个方面，另一个方面对我们的下游应用非常重要，当你的集群当中发生故障的时候，[我作为一个客户端，我怎么知道该从哪个正在运行良好的服务器获取它的配置属性？](/d7cbf4fb862c4c41ab73f7062bd87fc8)这就牵扯到一个话题，叫做故障转移，你的配置中心如何提供这种故障转移能力？同学们可能最先想到的是LVS，它是什么呀？前面提到过 Linux 上面的一种虚拟化的服务，在一期的架构师课程当中，也跟同学们详细的去落地了 LVS 的方案。


那么我们的配置中心这里可以应用 LVS 吗？当然可以，这是没有问题的。那在微服架构上面，其实我们有一个更简单的方案，它更天然趋向于微服务的价值观是什么呢？那就是服务治理。同学们有可能会纳闷，我们现在说的是配置中心，那配置中心和服务治理有什么关联吗？这服务治理听上去分明就是注册中心干的事情，它跟配置中心是什么关联？

那我们接下来就来看这样的一个例子，同样的假设，两个配置中心，或者说是一个配置中心集群，同学们想一下，我可不可以利用注册中心的能力，将你的配置中心作为一种服务来管理起来？未尝不可。


那我这里可以让配置中心把自己当成一个服务，把它注册到咱的注册中心上面去。那接下来的故事大家都知道了，我们的下游应用从注册中心这里获取到你配置中心服务的可用节点，然后它就像发起一个服务请求一样，到配置中心里面拉取数据，这样的一整个流程如行运流水，无丝毫障碍。那这种方案就是因地制宜，咱既然选择了微服务架构，**我就可以借助你注册中心的服务治理能力，将你的配置中心当做一个服务注册上去。**

那对于现在主流的配置中心来说，通常它其实并不是保存配置的地方，你的配置属性变更其实是由配置中心保存在其他的存储介质里的。


打个比方可以有外部的文件系统，或者说是GitHub，甚至我可以把配置属性保存在数据库当中，那这上面的这些数据源其实是它的数据副本的源头，我们的配置中心也是从这些数据源当中把配置属性拉取到本地，分发到各自的服务器当中，那它其实做的是一个分发的一个工作，原始的数据源还是保存在它该在的地方。


OK，那我这里假设上面的数据源发生了故障，无法再去获取更新的数据源，那么这种情况下怎么办？我们这里就要提出这样的一个思考架构的方向，就是说我可否在你的配置中心本地保存一个文件的备份文件备份里面保存**什**么内容，它就保存上次从你的配置中心上游的数据源获取到的数据内容，把它暂存在你的本地配置文件当中。那么这样一来，即便上游数据源发生了失效，那我依然可以保证正常的提供服务。


OK，那这种方案可以大幅提高你整体系统的可用性。其实我们在很多其他的业务系统当中也会利用本地备份，或者说是本地缓存来解决这个问题。非常典型的一个应用就是全局 ID 的发号器，那这是一个相同的架构思想。那什么是全局 ID 发号器？比方说你一个外卖平台，你外卖平台我们每下一笔订单，它要生成一个全局的ID，那么对于超高并发的应用来说，通常我就需要一个全局 ID 的发号器，那么这些发号器为了高可用的保证，通常它每次从数据源头申请号段的时候，它不是一个申请的，而是一次申请一段。比方说我申请 10000 号段到 20000 号段这1万个号，然后把它缓存在本地慢慢分发。这样其实有两个优点，第一个优点它不用频繁地到数据源当中读取号段。okay，这是一个优点，可以减轻数据源头的压力。第二个优点，它可以在你的数据语言失效的情况下，仍然保持一段时间的正常工作，直到把它本地缓存的号段发完。
