我的建议是你在简历当中写某一段工作经验当中最具有代表性的项目即可。那怎么理解最具有代表性？它可以是你做过最大量级的项目，可以是你遇到过最大挑战的项目，也可以是你第一次作为项目负责人主拎的项目。总之，你所写出来的这个项目一定是能够凸显出你哪一方面长处和亮点的。


最后一个点是对大家的一点点提示，很多的候选人在准备完简历之后觉得我就万事大吉了，但是其实并不是，简历写完之后给自己预留出来 2- 3 个简历里面没有的项目案例。为什么要这么做？是因为在面试的过程当中，总会有一些经验丰富的面试官会有针对性地要求你去讲另外的案例，用来判断你的这些案例的真实性和用来佐证补技术的能力。对于项目的理解，如果之前没有准备的话，在现场想不出来有特色的案例，可不是浪费了自己面试前准备的这些内容。


好了，关于简历到底应该怎么写，我们就在此告一段落。艾拉老师特意给听课的同学们准备了一个标准的简历概括模板，大家可以在课后问助教老师那边去领取，希望能够帮助你们更好的去梳理简历的思路。那接下来简历写完了，我们后面的面试准备要做什么呢？当然是做第二阶段面试过程当中的一些准备了，


那么有几个点是需要跟大家简单强调一下，对于面试这件事，它是有基本的面试礼仪的，其中需要大家注意的是，务必要提前 10- 20 分钟的时间到达面试的现场，

那么现在其实因为疫情的原因，很多的公司是用线上系统进行笔试的，那么如果是在线笔试的话，大家一定要提前准备好一张纸和一支笔，以防你在笔试的过程当中需要测算，但是平台却没有办法给你提供草稿纸的功能。

比如说第一点，有很多的候选人面试的时候过于紧张，对于面试官提问并没有很理解，但是他又不敢去直接问面试官，害怕面试评分产生影响，所以依据着自己模糊的理解开始回答，但是却达不到面试官想要的点上。千万不要做这样的事情。如果你对于面试的问题有疑问的话，可以直接和面试官沟通，只需要注意沟通的分寸和方式就可以了。


第二，有很多的候选人在回答问题的时候，尤其在技术的二面和三面回答面试官关于项目技术选型的问题的时候，回答的特别干，特别像从书本上背下来了一段话，这个时候面试官难有耐心的听你讲下去。对于类似的问题，我们一定要结合具体的案例进行讲解，如果有必要的话，可以让面试官给你提供一张纸或者笔去进行辅助的描述。


还有很多的候选人会在面试的过程当中急于想要表现自己，所以重复讲解很多的内容数不值。对于面试官而言，他需要你直接回答问题的重点，表述自己对项目的理解和思考，并不需要讲述这个事情的背景。好了，基本的面试礼仪和注意事项都给大家讲清楚了，大家一定要记住。

首先我们在一场 40 分钟到一个小时的面试过程当中，是可以拆成多个环节的，这个环节我们概括的分为几个部分，第一个部分肯定是笔试，笔试通过之后才会进入正式的面试环节，正式的面试环节会依据整体的面试流程拆分成自我介绍、工作经历，回顾具体项目的沟通和提问环节。当整体的面试环节结束之后，不是业务面试官或者 HR 送您离开，并让你等待后续的通知。那么今天我们就这几个环节进行单独的讲解和分析。


首先对于笔试的这个环节往往是代码测试，然后我们会发现中大型企业因为它对于人选的能力、模型要求都是在技术基础，尤其是在技术深度上面，所以算法和数据结构是我们逃不掉的。但是这个部分又是很多工作经验丰富的人选所忽略的部分，因为在实际的工作经验当中已经很久不用了，大家早就遗忘了。所以当我们决定要出来看看机会之前， 1- 2 个月的时间就要去刷一刷 lead code 和纽客。


我想 lead code 大家都很熟悉了，那我为什么会推荐纽客呢？是因为牛客这当下主流互联网公司都会使用的一个校招笔试平台，很多的应届生都会在网上进行刷题，所以牛客上有大量应届生的笔试题目，往往包含了常见的算法和数据结构，甚至于是计算机原理的内容。这里给大家分享一个小案例。


我在之前的招聘过程当中，有遇到过一个候选人，工作 8 年，干团队 3 年，早已不做一线工程师的事情，甚至于在技术架构的设计上面也做的比较少了，基本上是在做架构的评审。但是这个候选人非常的谦逊有礼，毫不介意面试架构师的岗位进行基础的笔试代码测试。而且在他笔试之前，我已经和技术的 leader 沟通得很清楚，因为候选人工作经验多年，可能很多的基础内容也已经忘记了。那么我们达成一致说，无论笔试的结果怎么样，都会让这个候选人正常进入到面试环节，万万没想到我铺垫了半天的内容竟然没有用上。


这个候选人在笔试过程当中，无论是二叉树的书写，正牌倒排的内容，都非常快速详细的提笔就写出来了，我们都非常惊讶。后来在我和这个候选人交流的过程当中才知道，他有一个保持了多年的好习惯，他每周都一定会拿出三天的时间，用起床后的半个小时，或者下班睡觉前的半个小时，在里的 code 上刷 3- 5 道题，这个习惯让他保持即便脱离了一线的爱马经验，他也依然能够保持自己在技术基础上的竞争力，是非常难得的。


那么当我们的笔试通过了之后，在面试的环节，根据刚刚拆解的流程一一进行讲解。在自我介绍的环节，千万不要去重复简历当中已有的内容，否则面试官会觉得，那我看你的简历就好了，你为什么还要跟我念一遍？在工作用力回顾的部分及时，需要我们用 5- 10 分钟的时间大概讲解一下每段经历选择的动机，每段工作自己的成长规划，反思是什么样子的。


占用时间最长的必然是具体的项目。往往技术面试官在沟通这个环节的时候，会沟通 1- 2 个项目，详细的拆解你的技术选型、技术方案、技术落地，甚至于调优和压测的结果。所以这个部分我们要预留出来大概 30 分钟左右的时间进行沟通。而这个部分也是我们接下来针对面试环节要重点讲解的内容，而往往一个完整的面试环节，最后一定是用提问环节进行收尾，但是很多的候选人会忽略这个提问环节，提前不做准备，或者只常规性的在网上刷一刷念经，找两个大家都常问的。殊不知在提问的环节是我们帮自己翻盘的机会。


40 分钟到一个小时的面试其实很快就过去了，在我们最后面试结束的时候，无论是 HR 送走我们，还是技术的面试官送走我们，我们都要对其表示感谢。对于 HR 而言，你的感谢是他坚持下去很大的一个价值了，那么你要相信你对于他的理解会为你后续的流程推进带来很大的便利。

第一点，地点，他有明确写道，我在某某公司任职的时候，这个就表明了你做这件事情的地点是什么时候，那么时间，嗯也很清楚。他说大概是在 2018 年 6 月份的时候开始启动的，同时也标注了做这个项目的背景是什么。他大概用一句话描述了这个项目的背景，也符合 star 当中 s 的部分。紧接着他描述了当时的情况和出现的现象是什么，就是用户高峰期的时候，后台系统会频繁的崩溃，那么后面的一切行为都是基于这个现象来进行操作的。


那么我们来看一下，候选人回答说他做了一些判断得出来了原因，然后推导出他要做的任务是什么，他要做的任务是要启动一个系统重构的项目，在这里艾瓦老师要暂停一下。大家会不会觉得在这里给面试官留一个伏笔，我想有聪明的同学已经想到了，我只说了判断是什么，什么原因导致的这个现象，但是并没有说我查这个 bug 的方式方法是什么。在这里我们留了这样一个伏笔，让面试官在我们描述整个项目之后，去针对这个点进行提问。那之所以跟大家强调这个地方，是想跟你们说，如果你们的面试准备足够充分的话，是完全可以预判部分的问题面试官会提在哪里的。这就是为什么我们要花费这么大的篇幅来讲解常用的面试方法。


好了，那么我们回归到这个案例本身，继续去做分析。在这个部分他明确地表示了自己在这个项目当中扮演的角色，合作的事情是什么，他是作为项目负责人做了整体的技术方案设计，紧接着他用一大段话来描述他是怎么给团队的人进行分工的，又是怎么做技术方案的评审的，有遇到了什么问题，他是怎么解决的？又在项目结束之后如何做验收的？那这个逻辑是一个非常完整的行动的逻辑线。


只剩最后一个部分了，就是结果描述。同样他大概用两句来描述了一下他做这件事的结果是什么。经过了 5 个月的时间，最终 QPS 服务的压测提升到了 1000 万，然后他告诉了我们这 1000 万所代表的含义是什么，是可以预估未来一到两年的时间用户并发量的需求。


那么在这个结果描述当中，它既用数据做了结果的支撑，同时也帮我们对于这个数据做了解读，我想大家经过这个案例的分析，已经能够理解这个 star 面试法应该如何应用了。好，那么常见的面试方法已经讲解完成了，
