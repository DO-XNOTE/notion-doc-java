启动过程组主要就是说制定这个项目章程，还有识别相关方都谁参与到这个项目里边。我在召开启动会的时候，人要召集全或者相关方重要的沟通一定要做到，包括这些参与方的参与度，有没有依赖性，对项目成功的影响力如何等等都要进行一个评估。许多大型或者复杂的项目也可以拆分成若干个阶段，这个也是启动过程的一部分，在每个阶段开始时候都对启动过程进行检查，这个更有助于项目的目标明确，并且目标是正确的。在下一个阶段也要进一步的审核，还有细化这个阶段的一个项目范围，我们再看规划过程组，可以看到在时间的一半在偏左，也就是说 15% 到 40% 之间，他的工作量是最大的。因为启动完之后我要做一个详细的规划，重点关注这个循环，谁的收集和掌握的项目信息越来越多，项目就有可能进一步的滚动规划。比如说前面项目的一个生命周期不断的发生变更，这里边我们这个规划过程和执行过程可能就是一个循环多次的一个情况，这也算是项目管理过程中一个渐进明细的一个体现。


总之规划过程主要的作用和目标就是确定成功完成项目，或者这个阶段的行动方案，我要怎么做才能搞定这个项目？规划过程也是非常重要的，凡事预则立，所以这个预非常重要。想收集需求、确认范围，创建WBS、分解进度管理，我要规划一下，也就是我们日常所说的开发任务，定义好，排列好它们之间的顺序和依赖，估算好他们的时间、工作量、成本估算预算的一个规划、沟通的一个规划、风险的一个规划等等，这些都是规划过程组重点关注的事情。


规划过程组的输出会给到执行过程作为它的一个输入。执行过程比较简单，就是真正的去落地，你对我们来讲更多的就是开发、写代码、测试、架构设计。执行过程组可以看到耗时是最长的，从最开始就慢慢进入了，一直到收尾结束，并且它在这个曲线里边的高度也是最高的，那项目的产品也都是在这个过程组进行产生，同时执行过程组也是项目经理还有项目团队投入时间和精力最多的一个过程组。在这个执行过程中可能就会有变更请求，那一个变更请求如果获得批准，可能会影响其他过程组，包括修改管理计划，规划过程组有可能再重新做一个动态规划，甚至新的目标基准也有可能改变。然后再说一下监控过程组，可以看到从最开始的时候就启动了，为什么呀？因为启动过程组也需要被监控，这里面包括审查项目进展、制定标准、监督和测量项目的一些量化情况、实际情况，并且要分析有没有进度偏差、预算偏差、成本偏差，还要采取一些纠正措施等加以控制，包括说实施一些纠正和预防的措施。那在监控过程组经常也听到说 PDCA 循环 plan 计划 d 度实施 c 检查， check a 行动 action PDCA 环，这个也是在监控过程组会有着非常重要的一个体现，所以监控就是监督和控制，让这个项目更健康。


说一下收尾，那收尾主要是说这个项目我要交付了，我要进行移交了，这个项目顺利结束，我要验证可交付成果相关的文档，并且做验收。收尾过程组一旦结束，就代表了我整个项目是结束了，这些过程组也都结束了。如果项目过大，整个收尾也可以不仅仅针对这个项目，也可以针对不同的阶段进行收尾管理，这个收尾过程总结经验，吸收教训，更新组织的过程资产，这个组织的过程资产就是例如知识库，这些都算。

那这个功能就是一个迭代模型，从最开始模糊不清，但是功能可用不是那么漂亮，但是能用一步一步做的美观，做的更贴近我们的目标。还有用户想要的，在这个过程用户也可能会给予反馈，比如说有用户反馈这个蒙娜丽莎的这个头发做成黑色的会更漂亮，于是根据大量的用户反馈机制，也可以不断的调整这个目标。


架构师现在更多的都是基于迭代模型来做，很多公司实际的工作中，如果接触的项目过多，其实是多个模型同时在用的。比如说我在做底层架构的时候，可能真的是用瀑布模型更好一点，我想的非常清楚，让我的地基更牢固一点，尽量少改变。然后像这种迭代模型可能坐在前台位置，它就是应付多变的一个需求，适应市场，适应前方的前沿变化。那在架构设计上我就要留出这些口子，把这个扩展性做好。
迭代和增量。简单概括一下，增量方法就是渐进的增加产品功能，而迭代是一系列重复的活动来开发产品功能，如果最开始就是画背景，从头一直画到手的这个部位，第二次我也是一样的，第三次也是一样，第四次也是一样。只不过我每次重复做的时候，让这个功能深度做得更深、更完善。


所以了解了这个项目的生命周期，还有项目管理的生命周期，我们也知道这个变更在什么时候，代价是什么样的。根据我们不同的特点，我们应该采取哪种开发模型去管理这个团队，或者建议项目经理采用哪种模型来做。这里并不是说选择一个一直做到头，这个还是说要综合的考量，前面也讲了重点要做这个平衡，极致的平衡。


那前面讲的瀑布型的生命周期，也叫预测型项目的生命周期，什么情况下可以优先使用？比如说我充分了解我要交付的产品，我就明确的知道我做的东西就是这个，肯定不会变了，并且我在这个行业可能有扎实的行业基础。例如说我是做 CLM 的，或者我是做订单系统的，非常了解，非常熟悉。


那如果说我一次付整个项目团队，或者整个商业计划，商业目标都能很好的达成，那像这种情况我们用瀑布型模型也是挺好的，说迭代和增量就这种敏捷型的生命周期最主要的特点需要快速变化，然后这个范围难以事先确定。那目前这种敏捷型的生命周期也是成为项目管理的一个新的趋势，尤其在互联网公司，变化太快了，不同的生命周期有不同的风险处理方式都是不一样，适用于不同的项目。所以又回到前面讲的孔子讲的中庸之道的问题，采用什么项目的生命周期开发模型，这个绝对不是说非此即彼的，一定要综合来看。
