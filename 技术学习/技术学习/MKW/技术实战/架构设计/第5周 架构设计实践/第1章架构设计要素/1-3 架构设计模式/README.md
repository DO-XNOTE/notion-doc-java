英文叫做 divide and conquer。是不是大家都很熟悉经典的算法方法，那这里在架构设计也普遍使用。我们来看一个实际例子，我们还是举生活的例子，这是一位音乐的发烧友，他想在自己家里面做一套 Hifi 音响系统。那如何去做这套音响系统呢？它给自己定义了，嗯，我要实现怎么的音域？我要保证我的什么每个音乐设备的这个主抗低于一定的值，我要在整个系统的当中设计一个很强大的这种什么功放系统，能够达到一个很洪亮的音响效果，那在整个选型过程当中，他就买了一些专业的，是吧？音箱，买了专业的调音设备，也买了专业的遥控器，但是发现什么整个我们的音响的这个功放系统，没有一个传统的功放系统能满足要求，或者是能满足要求的功放系统太贵太贵了，那这个时候他就决定自己开发一个符合自己要求的公放。


公放的旋钮是很容易买到的，公放的壳子也很容易买到，但是公放的电路板里面需要有一些定制化来满足自己的要求，所以他就开发了这样一套功放的 PCB 电路板，他买了传统的电阻，买了传统的电容，还买了一个很特殊的我们什么编解码芯片和一个放大器电子芯片，那这两个芯片其实就是如右图所示。


那这两个芯片其实特殊之处在于什么？特殊之处在于这两个芯片是不是传统的芯片，而是找一家厂商定制化从方队里面流片出来的芯片。那通过这种方式他就完整的把他自己的功放，把他自己电路板的需求实现，最后搭建了自己的 Hifi 音箱系统，大家想想他是不是在音箱系统过程，当他首先把这样的音响系统拆分成一个模块，其中发现某一个模块现在还无法通过传统的商业产品实现，他就找一套开源的或者自定义的解决方法来实现。


这个什么我们的这个功放，那供奉里面也有好多模块是现成的，唯一的电路板模块没法实现，那他才采用了什么自己定制化电路板，同时在电路板里面还发现一个核心模块，编解码模块和我们的功放芯片模块似乎不是很方便实现，所以你又用了一部分定制化的我们的方枕的流片技术，那这整个过程就是分而制的思想。当一个需求没法实现的时候，我们还是贯穿我们整个架构设计的目标。
do the RISING right？


当你 RISING 你的需求达不到的时候，我们就分解它，分解到一定的力度，刚好能有一个产品来实现，或者有一套开源方案来应对，或者是我们的什么软件工程师，或者系统工程师，或者是电路板工程师能很方便的来实现，这个时候就能把 writing 做对了。


那除此以外，第二个 right 是指方法论要选择的好，这个方法论什么是适可而止的？是自然拼接的，那这整个过程你可以看到它是什么，到了什么瘤片这个阶段就适可而止了，他没有去进一步去研究什么晶体管里面怎么样进行设计，这没有意义对不对？同时自然拼接这种拼接方法就是芯片该用电路板的形式进行组装，就用电路板组装，通过电路板来实现出来芯片和电容和电阻之间的沟通，而这个电路板该用什么壳子封装，该用旋钮进行沟通，就是如此做，该用遥控器来管理，该用我们电线对接我们的音响，就这样做一切采用最自然的拼接。


那我们的架构设计当中通常采用比如说 API 网关，或者是采用消息队列这种形式，把你一个什么低耦合的模块进行什么进行连串起来，然后把这些模块尽量通过商业化或者开源的方法来实现，实在不行通过定制化的方法，少量的编程来实现，这就是独耳 rising right。


那同时在整个设计过程当中考虑到什么，一旦无法实现，就把它进行切分而制止，一直分到自己恰好能实现位置就够了，不用再进一步细分了，这就是整个风而制之思想当中去贯穿了 do a RISING right，通过这样一套思路，大家后面的课程当中可以看到几乎整个架构设计的过程都是在分，一旦分到合适的点就停止了，不会过多的去切分好。

那右边是两张经典的架构图，右上角是一张企业架构图，从红到黄到蓝，这样三个大箭头分别体现什么？企业和业务的能力，那黄颜色的是什么？ it 的能力，那蓝颜色那个箭头体现了什么？是对于这种嗯需求的gap，就是我们的这个不可达的能力以及转型的能力。那这些能力通过一套循环，一旦什么出现gap，我们就进行转型，转型完了以后调整我们的什么企业的能力，最后改变我们的it，形成一个循环，有点像什么蛇咬尾巴这样一条循环。


这是一个企业价格，那一个项目，一个系统价格。如果你去为客户或者为自己的企业实现一个系统，通常可以分几个环节，了解需求、架构设计、开发实现和最终的什么架构验收。那这里有英文的内容比较多，大家其实不用关心细节，只要看这个箭头就可以了，每一个箭头看上去是正向，但是当什么届构设计最终完成以后，会有一个反向的大型的蓝色箭头，重新再去了解需求，不停的迭代你的设计，那后面的设计过程当中飞扬老师通常只带大家走第一个循环，也就是从需求一直到什么逻辑架构以后面的物理架构、最后的架构验设完成了。


但是一个循环，这里要强调一下，不是结束，而是下一个循环的开始，蛇的嘴会去咬蛇的尾巴，一个循环结束的时候会形成新的循环，那些新的循环会有一些新的迭代的要求，不停的更新你的功能需求，不停的去改善你的非功能性，把你的性能做得更强，把你的吞吐量做得更大，把你的安全性做得更好。通过这样不停的迭代，最终才能实现什么完美的音响系统，才能实现一套完美的 it 架构，真正具有匠心精神的 it 架构。好，聊完了这样两个核心的我们的设计的模式，那下一节我们会看一看架构设计的主要的输入是什么，大家敬请期待。
