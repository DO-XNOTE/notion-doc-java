三，做的，那相信大家在之前的项目经验中已经积累了一些比较成熟的思路和步骤，同时我们在前好几讲我们架构设计的原则、模式以及一些方法论，过程当中也跟大家分享了很多不同的思想和步骤。那这里其实一个体验是什么？就是通常飞扬老师会说的那句话，架起需求到落地的桥梁，那一个真正的架构设计必然从需求出发，到最后要落地，我们要看一看大家怎么样去分解这个需求，怎么样去在具体的架构设计当中去分而治之，去分解一个一个的模块。同时采用一种迭代化的思想，怎样逐步把我们的架构逐渐地落地，逐渐地细化，逐渐地可实现化。那同时在过程当中采用各种不同的方法论，比如这里比较推荐的架构立方体的方法论，通过不同的视角，通过不同的视图来展现我们架构的具体的优美，以及它如何真正能做到实现和落地。


这道题要拿高分，其实通常都是要结合实际的，光说理论其实很难打动我们的面试官，也要结合实际，可以通过有几个方向入手，一个是把需求讲清楚，我这个项目什么样的需求，我怎么样去理解它，同时怎么样拆解需求当中的功能性、非功能性。或者我们集中在什么整个架构设计的模块上，我们有什么？有应用模块、技术模块、功能模块、运行模块，还有逻辑视图、物理视图等等。


我们在整个不同的角度去观察这些我们的架构的时候，如何利用立方体的思路来进行转换。因为之前我其实立方体的具体转换思路没有讲太细，但是大家可以通过这样几个维度，尝试把一些思路梳理成不同的角度，然后通过角度和角度之间的映射来尝试去解答这道题，那相信大家在后面一个章节完完整整的实战一下这个架构思路，立方体的思路。


同时我们通过一个实际的点播平台的例子，再次实战一把，这时候可能会有更深一步的理解这道面试题，我们大家可以先在这里进行尝试，在之后的章节上完以后，还可以重温这道题目，再尝试用更深入的方法来解答它整体思路和步骤。好，我们来看第二道题，你在架构设计中通常要考虑哪些因素？

321，做到这道题相对前一道题稍微细一些，看一看考虑的因素，其实考虑的因素体验什么，就是你要考虑哪些东西作为你架构设计的输入，因为我们我们平时其实大家已经积攒了很多经验了，不管是成熟的架构的经验还是一些项目的经验，这个时候有不同的输入就会产生不同的结果。所以要考量的主要就是输入，包含像功能需求、质量限制，以及已经有的本企业有的或者是本人有的一些资产和技术，通过这些输入进行组合这些变化，最后形成不同的什么输出工件。


那这道题要拿高分有这样几个点，首先当然也是要结合实际讲一讲实际项目当中的一些方法，同时我们建议采用一些比较高屋建联的方法论，比如功能需求。飞扬老师前面提到的 w h 要分析一下到底是who， which 谁在用这套系统，他们是怎么样进行交互的，它用到的功能what？到底是哪一个功能？以及具体交互过程当中的一些我们数据流的进出情况，一些交互情况，那个号把这些讲清楚，就能把功能需求的实际的案例，实际的场景讲透彻。


那除此以外，质量我们也可以通过很多个，怎么不光是有架构设计维度的五大维度，这样五个怎么还有很多其他的，比如用户的体验的维度，怎么样才能够尽量的节省成本，怎么样才能够尽量的发挥我们的资产的重用，使得我们快速进行迭代等等。有很多不同的，怎么从好几个角度，可以从运行时的质量，也可以从准备时的质量好几个层次来分析我们非功能性的需求。那除此以外限制三角形，飞扬老师也给大家分享了两个限制，三角形，其实不光是这些，可以从项目角度分析限制，也可以从我们企业的架构角度去分析限，还有更多的不同的限制影响了大家。


通过这样的功能、需求、质量和限制，以及大家最擅长的已经是很成熟的技术，或者是大家了解了企业所有的资产，把它们组合起来，这些因素共同构成了我们整个架构设计的过程和最终的输出界。我们来看一看第三道题，你在架构设计中该如何分配工作时间？你平时是怎么花时间呢？

诶，这道题也是很常见的一道题目，通过这样一句话，基本上就能看出一个架构师实际的经验和他平时的工作习惯。 321 请作答这道题其实考察的点非常多，它既考察你的架构设计的过程，你在整个过程当中更关注哪些环节，以及考察你的输出。因为你的时间必然是要花在刀刃上，必然是有输出结果的，你到底在关注哪些输出，同时这些输出占的比例又是如何？另外也能看出你的这个风格，你是组成派还是决策派，大部分时间在做设计，还是大部分时间在开会讨论中？除此以外，你的时间管理你怎么样去管理一个项目，也能从这样一道题目里面得到一些小小的提示。那面试官通常就会问，这些开放性的题目，从多个角度来了解你的架构设计的深度和广度，那这道题要拿高分。


要这样既要结合架构设计的输出，因为时间必然是要为了什么输出做目的的，你可以分析一下你整个架构设计过程当中大部分的输出是什么？是规划还是设计？是为了测试和部署做准备，还是在考虑采购和外包的决策？那这些点都能涉及到，我们就能感觉出你是一个什么很资深的架构师，你设计的面很广，如果你永远只是在做设计图，你是由纯粹的组成派，有可能你在规划、测试，甚至于决策方面、采购方面了解得非常少，这感觉你的广度不够。另外通过你最关心的时间的一个点，就能了解你的深度。


比如说我最关心我的整个设计环节，我把我九成的时间都用在设计上，那么这个时候我们可以进一步聊一下你是如何进行设计的？你具体的时间又在设计当中怎么样进行，发挥出它的那个有效的价值？那这个时候我们就可以了解到你是组成派的一个自身架构师，然后你的深度到底能达到多深？那还有的决策派的架构是说我有一半的时间在采购，在外包的决策当中我要定那些采购项，定那些 PO seed 这个评判标准，以及进行考核和打分。甚至于我公司内部也采用那种内部采购，内部外包的形式分配到不同的组，我来其中负责整个所有的组件的拆分和负责所有的这个架构的评判，这是我 50% 时间要干的内容。


那这种决策派的架构师我们也会细问，以采购你的决策，你的外包决策当中的依据，为什么这样做？为什么那样做？你有哪些牺牲？你遇到哪些限制？来了解你在具体过程当中的深度？就通过这样一道开放性题目，基本上很容易知道一个架构师来面试的这个面试人员，他离我们所要招聘的这个岗位和广度和深度的差距。
