我们功能型需求矩阵通常有塞尔 3 列，一个列叫ID，那举例我们叫功能矩阵，英文名称叫 functional requirement，所以通常功能型矩阵 ID 都是 FR 开头的 functional requirement，功能新需求一，然后这里中间是什么？是功能信需求，然后后面是功能的具体的描述。

我们拿到什么写什么。我第一个拿到的其实就是什么叫 API 标准化这样一个需求，很明确的把这个需求聊清楚了，然后我自己就会说，我们所有的数据上传、迁移、下载、 BM 搜索的功能都需要支持 API 接口，能完整的通过 API 进行所有的调度、管理和控制，那这就是内容，我这里口述好，我拿到了第二个需求，对吧？第二个主要的很清晰的需求，什么是它的内容和边目？要进行自动化，叫自动化边目索引，然后和检索它有一点不同，边目是追加很多的不同的敏感信息，或者说是个人的标注信息。


比如古天乐的什么 2017 年的一个视频，那这个视频会描述这个视频的这个我们整个影片的名称，它的这个明星的名称名字，然后它这个创造的时间，我们文档的上传时间，我们最后边目的时间，那边我过程当中是不是有些这个比如说触犯了版权问题，都会在遍布过程中追加这些内容。


那索引是把所有遍布的内容进行快速索引，这样你查古天然就能查到，查 2017 也能查到，查敏感的一些关键词也能查到。那最后检索就是什么？就是直接在我们的一个检索平台里面，其实也是边牧系统的一个图形化界面中的一个环节，在这个环节里类似于谷歌搜索一样的，它提供了个叫什么叫做点评搜索，所有你点评平台里面的视频都可以在那个里面进行什么快速搜索，那搜索完了以后它就会自动展现具体的内容，那这就是自动化边目索引搜索的这个总体细节。
