第二张图，逻辑运行模型。那我们看看该如何把什么我们把技术元素加到我们的应用逻辑里面来形成真正的逻辑模型吗？好，首先把整个页复制到下面一个逻辑运行模型，这样一个页面当中，复制完了以后，通常我们每一页会做一些减法，也会做一些加法，那这里先要做些减法，我们把一个一个的这个部署单元拿掉。


因为什么部署单元和应用节点的关系我们其实已经知道，所以后续的部署过程当中，我们只要完成这些逻辑单元就可以了，那自然而然在逻辑单元上面我们就可以部署一个一个的小单元模块，我们把这些红色的单元拿掉，拿完以后我们还要做一个很关键的事情，什么 ALN 就是逻辑应用模块的代名词，所以这个逻辑其实已经不再适合我们这一张图了，所以我们要把所有的 a 给拿掉。


变成什么？变成一个一个的实际的**技术加应用的逻辑单元**，那个 a 应用已经不再适合我们整个这个描述。好，我们进行一下这个修改，这都是一些细节的修改，改完 LN 以后，我们现在每一个都是什么逻辑的这个单元了？这些逻辑单元只是什么应用需要我们安装的？其实在过程当中我们已经考虑了一部分什么技术，比如像容器是不是一个技术？中间件是不是一个技术？搜索引擎是不是也是个技术？那除此以外还有更多的技术单元。同时我们考虑技术的时候，还可以把数据中心的整个区块进行少量的划分。


我们首先看一下，如果我们把数据中心进行重新调整，比如说我们把制作岛和所有其他模块放在后台，我们叫什么？我们叫它后台数据中心，或者叫主数据中心这个概念，它就是主要的功能，它实现什么整个数据中心的核心的能力，前面可以加一个什么样的区域？我们前面可以换一个颜色，换一个，比如说橙色或者黄色这种不同的，这个区域表示还是在数据中心的机房里面，但是它整个网络划分，整个隔离的功能完全不一样了，我们换一种颜色来表示这个区域，比如用这种亮黄色，我们看一下这个区域叫什么？取个名称，我们前面已经有了 4 个区域，一是主中心，二是互联网，三是合作伙伴，四是总部。
那我们这里取一个名称就叫做DMZ，我不知道大家对 DMZ 了解吗？ DMAZ 就是一个什么是个隔离区，在战争的时候，两个国家之间有个隔离区，防止大家打仗。那同样道理，我们通常在数据中心里面也有个叫网络隔离区，这个区一个区域就够了，它能保护整个数据中心不被瘫痪。我们可以把什么应用界面放在隔离区里面，所以这样即使我们的界面出现少量问题，我们很多的用户还可以通过本地的缓存继续发放API，直接 call 到我们后台的服务器。那或者是什么，即使界面出现问题以后，我们普通的认证后台管理功能仍然能运行，只是对独立用户提供商、内容发布商做少量的什么服务降级，所以有时候我们会把最前端的页面等等功能放在我们的 DMI 去，而不是我们的主数据中心这家客户就是这样一个思路。


除此以外更关键的几个模块是什么？我们可以看一下，前面已经排到 13 号节点了，我们直接排 14 号节点。蜜罐系统不知道大家听过没有，蜜罐系统就是什么引用熊来偷蜂蜜的这个蜜罐，一旦熊来偷蜂蜜就会什么掉到陷阱里面了，那这样就防止熊来破坏家里面的这个光头强，可以用这个技术，对吧？防止屋里面的东西被熊出没的这个熊大熊二给盗窃了或者破坏了。那就是什么在家门口放一个蜜罐，引诱熊来投，然后底下放一个陷阱，这也是我们经常在 DM 在区做的事情，我们除此以外，还有我们最常见的什么防火墙的防御蜜罐，是引诱他进来，然后进行一些入侵检测，或者是一些入侵抵抗或者入侵防御。那这里的防火墙有好几种概念，可以是什么？我们最普通的我们的四层防火墙网络层，也可以是应用防火墙，后面会在安全章节详细来聊一聊，比如说防火墙蜜罐如何去跟我们的前端的界面进行交互，来进行安全的防御和安全的引诱的。除了这个 DM 在区的加了内容以外，我们后台区域其实也要加几个很核心的内容，比如我们的十六号节点时间同步服务时间同步符。在我们后台的集群管理里面最关键，比如说我们搜索引擎消息队列容器，每个服务器如果本身时间同步做的不好，很可能出现什么生存，出现类似于脑裂的现象，大家对事情的理解不一致，导致混乱，甚至于比脑裂还要严重。因为脑裂可以通过仲裁节点来控制，时间同步不一致，那都没法仲裁。但每个人的理解都不一样，对时间同步是一个很关键的节点，那我们也可以作为一个底层技术的核心。
