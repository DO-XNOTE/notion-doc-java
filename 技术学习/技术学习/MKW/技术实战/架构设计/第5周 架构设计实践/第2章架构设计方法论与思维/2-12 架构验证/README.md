大家仔细听飞扬老师的课，其实应该对 read 已经有所了解，是什么？四个英文单词首字母的缩写，这套弓箭也是一套方法论，我们看一看它是哪四个单词。


第一个就是风险risk，那在整个架构设计过程当中，一定要记录所有的潜在的风险，同时在我们开发过程当中，测试过程当中要验证我前面聊到的风险有没有发生，或者是被我缓解的，还是被我规避的，还是不幸的让它真真正正发生了。还有就是在我们开发和测试过程当中，有可能会发现一些新的风险，尤其是在我们系统上线的时候做集成测试，或者在审生产做上线之前准备测试的时候，经常会发现一些新的风险，这些风险一定要记下来，当真正上线出现问题的时候，我们可以回过头去看一下前面的风险规避策略。


我应该在实际风险变成问题的时候，应该怎么样去规避，或者说减少这个风险产生的影响？聊完了第一个风险risk，看看第二个假设， a 就是 assumption 假设。什么是假设？很多公司说是这样，嗯，我这个项目要成功，要 e 轮，是吧？我的 c 轮投资到位，我 c 轮我假设会有1亿元投资进来，比如这 1 亿美金的这个资金进入，我就可以把它项目做得怎么辉煌，我就可以把我的市场份量做了怎么大。唉，这就是个假设，通畅之中假设都不成立，对吧？不要说 c 轮资金可能不进来，连 b 轮你已经成功融资的这个 b 轮有可能最后资金到拨无法到位，我们的投资方，嗯，我只能给你 20% 的资金到位，另外 89% 什么什么什么条件导致你没有达到你的承诺，所以我没法给你资金到位等等故事。


那这只是一个资金的假设，还有很多其他的假设，比如说我们架构设计的当时采用的一个原则是什么？什么？什么，那这是依赖于什么？我们行业里面都是用这样的原则，那结果现在假设不成立了，行业里面所有的架构设计原则都改变了，这时候可能你就不适合用这套原则了。
