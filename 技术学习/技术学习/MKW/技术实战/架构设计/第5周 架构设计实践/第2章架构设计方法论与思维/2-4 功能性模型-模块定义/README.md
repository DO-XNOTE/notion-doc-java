内聚顺序内聚、通信内聚、过程内聚、时间内聚、逻辑内聚。而偶然内聚，他们的内聚性是由高到低逐步变弱的，同时他们的模块独立性也是由强转弱。难道左手与右手到底该用哪个手来摸这个模块内聚呢？来猜一猜是什么？当然是左手原则了，我们尽量倾向于什么？让模块内部的所有的功能全都是功能内聚、高内聚的一种形式，而不是偶然内聚。


那这样的这个设计出来的模块，或者这样切分出来的模块才更加什么有效、有意义？那有了模块内聚，自然会聊到模块间的耦合。我们从左手到右手，非直接耦合、数据耦合、标记耦合、控制耦合、外部耦合、公共耦合、内容耦合。还记得吗？他们怎么样？他们的关系是耦合度是由低到高，越发耦合，同时模块之间的独立性能也由强转弱。左手右手哪个手来摸更好？当然还是什么？我们的左手，是吧？非直接耦合或者实在不行，采用数据耦合的方式，使模块和模块之间尽量比较松散，千万不要采用内容耦合这种高密度的、紧密的耦合方式。


那通过这样的一种内聚和模糊耦合的定义，基本上我们大概能感觉出模块设计怎么样越形式更合理，但是就算在不同的这个力度上，我们仍然都可以实现高内聚和低耦合，那采用什么样的力度更合适？还是左手和右手？我们看看。

我们**假设以领域驱动设计里面的概念为例，后面会有单独的章节来讲领域驱动设计，**但是我们这里其实可以先抛几个名词，大家应该比较容易想象它是什么含义，整个系统，整个领域，那是一个的，我们也可以把它考虑成子系统子域，这是一种拆分，再拆分成更细是一个一个的微幅，那再细也许就是一个一个的数据的聚合。


那如果聚合里面还可能是一个一个的，什么应用的节点的实体里面有可能会有很多值对象，这样的一路拆分，到底拆分在哪一个力度？可能模块的划分更精妙，那从左往右力度是越发细了，本来很粗粒度的系统，子系统，到很细的值，很细的实体，但是大家有考虑到在这个划分过程当中，架构设计的时间也是越来越长了，你划分的越细，你设计的越精细，你的划分和划和设计的时间就花的越多。
