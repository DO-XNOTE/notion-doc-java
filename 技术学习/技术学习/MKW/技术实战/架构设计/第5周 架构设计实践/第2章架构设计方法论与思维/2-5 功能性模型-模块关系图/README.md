需求到落地的桥梁，构建 it 新蓝图，我是张飞扬。那上一节我们看了看功能性模型的第一件事情，定义模块，该怎么定义组件？那首先我们介绍一张图， AOD 架构草图，不管你是横向分层还是纵向分层，通常你可以把一个子系统分成好几层，然后这几层里面可以通过一些子系统来实现，或者是通过一些核心模块来实现。


通过这一这样一张很随意的手画的，或者是随便找一个工具画的图，基本上把我们主模块列出来是详尽的，各个子模块以及模块和模块之间的关系还没有足够细，**还没有到微服务那个层次。所以我们这里通过什么通过 ominigraphy 来进一步画一张叫 component relationship diagram 模块关系图，**那模块关系图通常是一画一张整个图的。


我这里偷懒了，因为时间关系，我们假设我们的什么，我们的宠物还是这个宠物的这个寄存小o，它需要有一个用户账号登录系统，那登录系统就是你可以登录它，然后你可以观察继承小窝里面，你继承了这个动物。如果你继承了小猫，你可以看到它，但你不能看到别人继承的小猫的情况，你也不能够跟别人家的小猫进行互动，这是一个基本原则。


那怎么样实现这样一个子系统？用户登录以及授权的子系统？我们可以这样来画，通过版型打开以后，我们找到什么 U M 2. 5，我们注意放大，看得更清楚一点诶，有没有 2. 5 图库里面有很多内容跟什么跟我们的组件，跟我们的模块相关的是这个叫component，其中这个是很经典的一个。那我们的这个模块的这个图，这表示一个模块和模块之间联系，通常用这个叫做use。好，我们把这两个拉出来，就把这个关掉。


如何来使用模块和模块之间的关系？我们假设这是一个整个子系统，就外面应该有个虚框，这个虚框我们偷懒就不画了。中间必然有一个什么用户的认证登录的核心的应用服务，我们叫它认证登录应用服务。好好，这是一个模块，那这个模块完成了大部分的什么我们的用户登录功能，那除此以外，这个模块是不是一个什么 Java 开发的小程序？暗示他对外是不是要提供一个外部服务？有一个外部接口，还对外暴露一些什么API？是啊，我们希望对外暴露一个API，甚至于这个 API 有可能是比较灵活的，像 open API，甚至于可以对第三方提供服务。那我们需要有一套，就是对外什么 Web 服务，把它给引申出去对外进行展现。那相当于是我们应用的展现层，除了这个层次以外，是不是后台要真真正正去记录这些用户信息？哎，是啊，那怎么样通过一个目录服务可以是什么？是 Windows 的ad，也可以是 Linux 的 l depth？那不管怎么样，它是一个目录的服务，我们就把它写成目录符，具体怎么实现我们不管，因为我们只管功能性，对吧？我们在模块定义这个级别，只管功能性好。有一个目录服务，那除此以外，我们的用户登录过程是不是要记录下来？万一有人投诉了，说你偷看了我家的小猫在洗澡。


好，那这个过程是不是有一个数据符要把它记录下来？这些都是什么？一个一个的模块，一个一个的组件，这组件上面有这样的图标，这是组件的标准图标。方块加两个小接口，这是组件标准图标。同时双箭头中间写一个component，这是组件的描述。就通常的方块里面一旦看到有component，像这个图标，就表示我在做什么？我在做模块关系图，或者叫组件关系图。那这个关系是通过 use 来表达的。怎样表达？用让 Web 服务器去调用后台的什么我们的 Java 服务器来实现什么 API 的访问？所以这种情况下应该是由上往下选择use，那同样道理，我们应用服务器是不是也需要通过这样的 use 的方法来调用后台的目录服务？好，又有个 use 了，就是它调用目录服务的功能use。


那另外除此以外是不是还要调用我们的数据库服务？那这个服务就写成数据服务？好，那连完以后大家是不是感受到了这是只是一个子模块，所以应该还有什么其他的服务会来调用它？对的，还会有其他的服务，调用它，这就是一个完整图的一个部分，这个部分只是完成了一个子系统，它实现什么用户的登录，同时对外提供一个 API 后台，是一个目录服务系统以及它相关的数据库服务。
