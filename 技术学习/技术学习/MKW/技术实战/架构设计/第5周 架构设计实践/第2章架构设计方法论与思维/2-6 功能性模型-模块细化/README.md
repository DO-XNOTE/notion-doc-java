看一看模块和模块之间是如何进行交互和互动的。那说到细化，我们必然会有什么？会有一个细化的输入，所有模块细化的输入不光是前面什么模块的定义，还有一个更关键的内容，就是用例模型，我们需要对每一个用例进行模块的细化，特别是那些核心用例，这些核心用例是影响我们整个功能是否能够正常的运行和验收的关键。对这些用例，每一个用例都要进行详细的模块细化，才能保证我们前面设计的模块的整体的划分以及模块的交互是正确的。

有些是鲁棒图，有些是实体关系 er 图，我相信 er 图的 entity relationship diagram 这种数据库的结构图大家都非常熟悉，作为一个软件工程师都很熟悉的，那鲁棒图大家熟悉吗？我们切到我们的这个工具，给大家看一下什么是鲁棒图，好，我们把这个 o minigrapo 重新打开，我们在之前的那个图的下面，我们画一下鲁棒图，上节这张尺码是指模块的这个关系图，那鲁棒图还是这样在我们的板型里面，我们把板型拖过来，因为我们是分屏显示，所以有拖拉的过程。


好，板型拖过来以后，大家可以看到有这样三个图表，这个这三个有点意思，这三个就是什么很核心的鲁棒图的核心概念。比如我们以前面这三个模块为例，我们完全可以这样画laundry，就是类似于接口图，我们可以叫什么？叫什么？ Web 服务，对吧？这是就是防止，就是相当于对外提供服务接口的图，然后对它上面画一根线，它应用过来的时候很可能是先到这个到了 Web 服务，而中间 control 是我们的核心代码逻辑功能，就比如说认证登录，这是一个逻辑功能，那这个应用的服务应用服务好这个核心的逻辑功能就放在它后面，那我们也用一根连线来连上去。


当用户的 Web 服务访问登录服务的时候，后面可能有两个东西来管理数据内容，一个是什么是目录服务，是不是它更偏于什么 entity 实体的数据的保留，通常有什么 er 图对应它的目录服务所要保存的数据的实体内容。除此以外，我们数据库服务，我们记录之前所有的这个用户的登录，什么尝试那个什么查看我们宠物的关系，这些数据库的这个服务也是一个很核心的。


什么实体的，这个鲁棒图的实体单元通过连线我们可以很方便的把它们分别的连接到我们的什么登陆节点上，就完成了这个鲁棒图的设计，这就是鲁棒图。那通常鲁棒图是用于什么？把一个详细的 user case 通过这种模块之间的转换，能够在一张图里面展现多个 user case 之间的关系，就跟类似的功能的模块的关系图一样，但这种过程实际上细节度不够，所以飞扬老师不建议所有的架构设计都涉及鲁棒图。
