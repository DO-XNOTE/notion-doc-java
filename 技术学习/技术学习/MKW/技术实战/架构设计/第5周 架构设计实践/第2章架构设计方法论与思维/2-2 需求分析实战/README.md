这里我会举一个例子，是什么是宠物的，是电的一个系统，叫做宠物寄存小屋，是一个虚拟的项目，我们来看一下如果我们要设计一个宠物寄存小屋应该如何来做？那首先是不是要有一个用户来访问我们这样套系统，那我先要画一个叫系统上下文图，看一下我应该做的内容是什么。
那除此以外，我们这个系统我们可以用一个标准攻击，就是这样的方块表示我们的系统好名称就叫什么宠物寄存小屋，除此以外我们想想看还有什么样的人会用到我们？我们有一个标准的客户，是吧？会把他的动物寄存在我们这里，是不是如果他寄存的时间比较长，我们还需要安排一个什么一个宠物保姆来帮助他监管整个这个过程？

比如说我们要给这个宠物熟放点食物，不可能因为一次性放满 7 天的食物是吧？会变质，每天给它放点食物等等。


那除了此以外，还有什么样的用户会跟我们的宠物小物打交道？大家有没有想到那就是宠物本身了，我们以小猫为例好不好？我们搞得生动活泼一点。我们以小猫为例，客户宠物保姆，小猫是给我们打交道的，主要的什么护人或者动物，那还有什么会跟我们宠物小物呢？打交道诶？我们可以举几个例子，我们想想看，如果这个宠物小屋是一个很高级的系统，可以让什么猫在里面喝水，可以在里面洗澡，是不是要能供水？能供电，那供水供电的系统该怎么画呢？还是用这种方形？我们写一个什么供水供电系统，但这样的方形跟我们的这个主系统是不是很像？会导致什么一部分混淆？通常我们阿拉比，然后教大家一个方法，把里面文字删干净，然后把这个框叠到下面去，看到没有出来一个扁扁的，旁边有两个小竖块的，这个就是通常在我们架构师的这个设计里面叫什么？就叫做外围系统，就是不是我们这个架构所要设计的系统，我们可以把它锁定成一个东西，这样的一个外围系统就是不是我们要设置，但是跟我们系统有关，是不是我们宠物寄存系统依赖于供水供电？除了这个供水供电系统以外，还有什么系统是可能被我们依赖的？哎，想想点宠物是不是要吃喝拉撒，它的拉撒是不是要排泄？我们可以写个稍微名词，好听一点叫垃圾箱，其实就是一个排泄系统，把宠物的垃圾自动的排泄出去，这样我们宠物保姆就不用关心什么洗澡，具体这个排便这个问题，只要往里面放食物就可以了。


那客户可以通过一些电子的产品，比如手机上可以打开一个 URL 来观察我们宠物的状态，那小猫可以在里面放进去，也可以很好的生活。我们要把这些嘴巴说的话全部画到什么实际图里面，我们可以这样简单来画，然后写一段文字，这段文字什么？客户是重点是要观察我们的动物，对不对？就观察这个字有点大，我们把字体稍微调一下，调到 14 好不好？观察这个动物，好，我们可以把这个观察复制几份，那宠物保姆说主要是什么？是放食物，那同样道理，我们的小猫主要是什么入住？还有很多其他的功能，我们这里简单写一两个功能，然后我们通过拉线的方式把它们关联起来，是不是就完成了整个边界上下文的描述？我们要设计的这个宠物寄存小屋，重点是要让用户能观察，让宠物保姆能放食物，让小猫或者小动物能入住，然后能提供正常的供水供电，让他洗澡，让他喝水，同时能让他们把它的排便自动的放到我们的外围的垃圾箱里面。


这就是我们这套系统的什么系统上下文，通过系统上下文知道谁关注我们系统，我们的系统又和其他系统之间是如何进行简单交互的就可以了，那光知道这些以后似乎对 what 跟 which 很清楚了，但是具体我们如何去实现它，或者说更详细的， what 到底在干什么？还不是够那么详尽？那这个时候我们就要对真正的 what 进行展示，采用什么啊？采用我们说的用例模型 use case model，把实际的用例给真正的讲清楚。


我们看看用例模型怎么画，还是这个版型里面？这个版型里面 UML 2. 5 里面有个关键图，我们刚刚没有拖出来，我们现在这里要拖出来。用例模型最关键就是个图标叫做 use case，其实就是用例，就是你要提供什么样的功能。比如我们以这个我们的客户为例，我既然把你的动物存在你这里存了一周或者一个月，我需要什么？我需要观察是不是观察我的宠物的状态？死了活了，到底是开心不开心？难过不难过？是不是要换一家宠物店？这就是它的主要功能，我们可以把它相关的这个客户复制过来，那它们之间是有一个关系的，就是什么客户要观察我这个宠物，对不对？好好，要观察我这个宠物，那除此以外还有什么样的功能是重点？需要我们这里强调的，我们这里除了观察以外还有其他的一些功能。


我们先跳过，我们先把刚刚这里描述的内容，就是系统上下文里已经有的内容，在什么 use model， use case model，用例模型里先画出来，然后再丰富画，好吗？我们复制粘贴，那除了这个用户以外，我们可以看到猫的猫本身是什么？猫本身要是不是要入驻就是一个基本功能，还有除了入驻以外是不是离开是另外一个功能？我们有入驻必然有什么？有离开，除了这个以外，我们的什么？宠物保姆是不是也有一些基本的要求和功能？


宠物保姆要能够什么？要能够放食物，提供食物，还有什么？还有我们这两个外围系统，对吧？供水供电的外围系统。好，我们一样，我们也是用这样的椭圆形的这种用力模型来表示它的这个供水供电的相关关系。我们要什么？真正的进行供水，然后靠的是什么？是我们的这个供水供电系统，是不是？


我们的供水供电？那除了供水以外，大家很容易想到就是什么供电，诶？这供电什么用处？这供电就是我们希望我们不要是直接对它提供热水，而是什么它本身有加热的功能，所以这里需要有一个供电的能力，能够什么把我们的喝的水和我们的这个洗澡水都给加热了。那除此以外还有什么？我们的排便的系统是不是要能够正常的处理它的小猫的排泄？我们把这个发机箱给拉下来，然后，我们正常的一个排泄，处理能够扔到垃圾箱里面去，这些基本功能是不是已经能完成？主要的动物生存的需求差不多，但是一个什么一个宠物寄存小物必然有很多丰富化的能力，比如我们客户观察这个动物感觉不爽，我们希望释放一个电子小狗来跟他互动，所以客户什么可以在远程等，虽然在这么几公里以外，或者几百公里以外可以放一个什么虚拟小狗来吓一吓小猫对不对？来跟他互动一下，那这是一个功能，那我们的猫除了入住以外，除了离开以外，刚刚是不是漏了很多功能？比如说什么？比如说我们吃喝拉撒，对不对？吃喝拉撒这个功能是不是没有花？那我们这里也一样。


好，那除此以外还有什么？还有洗澡对吧？我们猫之所以什么要有一个加热的系统是因为什么？它不光喝的水，是希望是热水，还要洗澡，我们要把洗澡的功能也给补上，我们再看一看还有什么？我们宠物保姆可能放食物以外，是不是不用管其他事情？不一定，如果要跟他互动，有可能还要一些实在的产品互动，比如我们要放一个毛线球来跟我们的小猫进行互动，那这个实际的毛线球的放置和收取是不是就需要靠我们的宠物保姆了？加上放毛线球，那这些基本上是不是就是我们整个平台所需要的所有功能？那通常我们会把所有的这种椭圆拉得比较近，比如这里没操作好，要把这些椭圆都靠近了，那这些所有的椭圆在一起就是上面这个系统的代名词。


所以系统上下文什么描述的是系统跟它外面的系统和用户之间的关系，而用例模型描述的是我到底系统要实现多少个 word 才能够满足所有的要求？依然把这些要求映射给所有的用户或者什么外围系统？如果有一个要求跟外面没有打交道，它就不是一个要求，那这里就是一个错误的架构设计。所以在我们需求界定过程当中，必然有触发源或者是外围系统，或者是外围用户，没有触发源的系统不是一个真正的需求。


聊完了用力模型，大家是不是感觉很简单？所有用力就是椭圆，外面系统就是这种什么两个大方块集成在一起，里外一夹，作为一个什么外围系统，用户就是一个圆圆的人头。那所有这些是不是只描述了一个内容，是吧？功能性需要，那我们的质量，还有我们的什么限制？该如何描述？正常情况应该是开个 word 了，好好来跟大家描述描述。那我这里简单起见还是在 Omini graph 上面。


那我们就描述一下质量，质量通常包含哪些？扩展性、性能、可用性？我们的伸缩性。好，我们假设以扩展性为例，那我们希望这个寄存小欧什么？可以管理什么？ 10 个小动物？本来刚刚设计的时候是管理一个小动物，我很容易管理 10 个小动物同时进行分隔管理。也就是说什么这个猫不会去影响另外一只猫，这个猫也不会影响另外一只狗，这是质量需求。


那我们还可以有什么质量需求？性能的需求。比如我们希望洗澡水在一分钟之内就能完成，所以它需要什么？它需要水加热很快，或者水可以预先加热等等，你这是质量要求，有什么限制要求，唉，限制要求也很容易，限制是成本小于 1000 元，因为我们宠物店其实投资都很低的，如果我宠物店里要放两三个这样的什么寄存小窝，能支撑二三十个小动物在这里面进行长时间的寄存。
