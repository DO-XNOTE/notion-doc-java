第一个，对于跨团队来说，如果咱这边都是使用的 spring cloud  这种服务发现机制，实际上同学们就可以像咱课程当中做的应用分层一样，我们单独分出来了一个这个 API 层。这里是定义什么呀？定义你所有的接口以及你接口当中所接收的参数，返回的response。对应的应用实际上就可以直接依赖于你的接口层，把你的接口层通过 Maven 我给它添加进来，这样的话我本地就不用对接任何的类定义in，任何的类我都不需要了，直接使用你 API 层定义的这些对象。不过在这种情况下，同学们记住，在 API 层这里，你一定要提供一个非常纯粹的 API 层。


什么意思？它包含的依赖要尽可能的少，不要包含一些无谓的依赖，这样有可能会给下游的应用造成一些依赖冲突，导致一些不可预期的结果。下游应用还要做一些排包，非常的麻烦。这是处理跨团队间 API 调用如何来去共享某些业务类的一个场景。使用 a p i 分层。对于跨语言的这种场景来说，比方说你使用java，提供了一套这种 rest API 的应用，对面是使用 C + +，甚至说我使用PHP，怎么办？它不使用你的 spring Claude 服务发现，那这种情况下，我们可以去考虑引入谷歌的 Pearl buff purtobuff。


同学们在前面章节刚刚学完，现在印象还比较深刻。 Pearl buff 我可以提供这种 pearbuff 文件，把它封装成一个项目下游应用，不管它是使用什么语言，只要是我们现在的主流语言 Protobuf 都有对应的插件帮它去生成对应语言所需要的业务类。尽管咱的 Protobuf 是谷歌在内部的 g r p c 框架当中使用，用来定义业务类定义接口的，但是我们也可以将它和份很好的结合起来，这是一个可以考虑的方向。


不过说实在的，跨语言交互一般都是这种跨公司级别的第三方对接，一个公司内的应用很少有出现这种跨语言的情况，所以基本上用第一个解决方案 NPI 分层就能大大减轻上下游的一个调用成本，这是其中的一个方向。第二个问题，我问同学们份的底层机制都有哪些？这个底层机制咱在课程当中跟同学们讲了一半，剩下的一半会在后面吉米老师的源码阅读当中跟同学们来去详细解释。关于分的底层机制，我觉得应该分成两个方向来去回答，首先第一个方向就是咱在课程当中带同学们去阅读的源码，那就是 contract 的加载过程。还记得它的特点是什么呀？祖孙三代这个接力带把这个 contract 给它加载起来，同学们结合咱课上讲的源码，可以再去仔细地研读一下，去 debug 去找一下线索来思考这样一个问题，为什么 contract 分为了contract、 base contract 还有 spring MVC contract 这三层它分别侧重什么？这三层之间都有定义好的同名方法，那么我们要去分析这些同名方法当中，他们完成的这些任务都有什么样的不同，这每一层他们主要侧重哪些方面，这个是需要让同学们自己来去摸索的，这是 counter 加载的底层机制。


剩下一种是调用时候的底层机制，那就是发起一个远程调用时，我的份组件如何去行使它的动态代理，来去代理我的这个应用发起的动作。这一部分我们在后面留给同学们一个课后作业，让同学们去来从源码当中来寻找答案。总结一下份的底层机制，这里我建议同学们分这两个方向来回答，一个是加载期的行为，另一个是运行器发起调用的行为，一个台前，一个幕后，

那如果你是两个写应用，上游这里调用下游应用的时候，第一个服务调用第二个服务它这里成功了，但是它这个成功比以往来得更晚一些，它在处理过程当中啪一下超时了，这时候上游应用会获得一个 timeout 的结果，这样一来相当于它会怎么样？第一步会回滚，它的当前的一个写操作会把它做一个rollback，但是实际上第二步个下游服务它这里是操作成功了，那这就产生了一个不一致性，这种不一致如何来去处理？同学们可以好好想想这里面的思路，那我给大家提供两个不同方向的思路。


第一个是最终一致性方案当中最廉价的一个方案，也是各大中小厂都广泛使用的方案，就是重试加补偿。重试很简单，我们这里其实 ribbon 还有份，它都支持一个源地重试，跨服务重试，在调用过程当中，如果发生 timeout 这种超时情况，可以直接发起一次重试，这只是能处理一小部分的异常问题。


对于那些业务层抛出了异常，比方说你的参数不合格， bad request，或者我在业务层当中抛出一个已知的异常，检查器异常抛出了一个500，这种情况下一个很通用的方案，就是说我在上游应用当中给你的数据做打标，我标记它异常处理的阶段，然后去创建一个批处理的job，那这个 job 会去扫你数据库当中的表，扫到什么标签它就在后台去触发这个重试的流程。


**别看这个方式看起来很 low 对不对？因为非常的简单，大部分是同学们以下就能想到的。但是这个方案在大厂当中应用也是非常广泛的，甚至蚂蚁这边很多金融级的应用，它在这背后的一致性方案也是使用重试加补偿的这种方式**。同学们记住，并不是说所有服务都要依靠分布式事务来去解决，一致性分布式事务会明显带来你的开发成本，还有维护成本的提高，所以说能用廉价解决方案的时候，我们尽可能的就不去开启分布式事务这个口子，这是其中的一个方向。第二个方向我就要给同学们说了，分布式事务，分布式事务它毕竟是最终一致性有问题的一个终极的解决方案。我们这里对于一些超高并发的应用，同学们这里要**记住不能使用 XA 的分布式事务，因为它会对你的连接资源做一个锁定。 XA 这里不适合于这种大厂的超高并发应用**，什么样的才是适合大厂的超高并发应用？我们在后面会跟同学们在分布式事务的章节详细介绍阿里巴巴的黑科技分布式事务解决方案Theta，在里面同学们将找到一种无感知的分布式事务方案，以及一个异常牛逼但是需要很大开发成本的一个方案，两个方案。
