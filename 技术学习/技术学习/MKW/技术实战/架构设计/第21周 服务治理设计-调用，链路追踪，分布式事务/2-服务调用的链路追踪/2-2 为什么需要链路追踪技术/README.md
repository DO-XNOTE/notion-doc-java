我调用你这样的一个状态，互相之间有一个循环依赖，那这个问题就产生在这个循环依赖的当中，我不知道怎么回事，有一个环节它掉了链子，导致了整个调用链路发生了异常。


那我这边我想定位到这次导致异常的这个调用链，那我当前这个服务在过去的一小时之内，我被调用了足足有 3000 次，那这弱水三千我只取一瓢，我就只想找到这个发生异常的这次调用来去深入的分析，那这就牵扯到一个定位问题的这样的一个过程，那为什么王大锤这里比较憋屈？哎，是因为这个问题并不像我咱这个图中画的这么简单，那咱这里 3 台服务器，那你调来调去，也就是每台服务器上面去查一查 log 就可以了。


不过在咱正式的大厂的复杂应用当中，每一个服务它肯定都是由很多个服务集群来组成的。那比方说那像淘系的核心链路，动辄大几千上万台服务器，那如果我们在这些服务器集群当中，我想定位其中的一个小异常，那么应该怎么下手？那假如我运气足够好，我一下就找到了这个问题的源头。比如说我发现了一个错误的日志，一个 error 日志，那这个日志就指向我当前的某一个服务集群当中的一台机器发生了异常。那么在这种情况下，我想去找到这条调用链路下面所有的上下游服务。比方说这台异常的机器，他把这个请求发给了谁，然后又经过怎样的周转，最后抵达了哪台服务，然后爆出了这个异常，我要把这整个链路给他抓取到，那现在问题就来了，我这个故事知道了开始我怎么知道结尾呢？哎，你集群当中有这么多台的服务器，那究竟我的下游服务，我调用的下游是他是他还是他？那这个问题就一下把王大锤同学给难倒了。


那所以说对于这种复杂的微复应用，如果你一旦发现了问题，想要做线上的链路的排查，梳理出整个链路，那这简直就是一个什么呀，还有一个玄学加上运气的一个过程，你要靠连蒙带猜。如果你的业务日志当中有打出一些业务主键，比如说什么主键，比如说订单的 ID 这类主键，那么以还有可能在上下游链路当中通过这个主键把它串联起来，但是假设不巧的是你的日志当中只记录了一些开始结束的一些信息，好巧不巧恰恰没有记录这类业务组件，那么你这个过程简直就是大海捞针。


对于一些稍大的集群，像上百台的这种集群服务，可以说你很难根据一个问题来去还原它的这个原始的案发现场，也就是它的整条调用链。OK，那这个问题有什么样的解决方案吗？那链路追踪就是这类问题的一个药到病除的良药。我们来看链路追踪是做一件什么样的事情，那这里我们把整个服务器链路给它打平，很容易的可以追踪到你当前的服务，它的日志文件，它调用的下游链路是哪一台机器，同时它又转发到了哪台机器，最终又回归到哪台机器？那这整个过程就是咱的调用链路追踪所要做的一件事情，那如何来做？那这里就有一些哎，黑魔法是要伎俩了。


咱在每次调用链路的一开始，通常来讲调用链路追踪的这种框架都会给一个链路的起始位置加入一个神奇的ID，叫 trace ID，通常我们也会把它叫做 global trace ID，它就是一个全局唯一的标记，它标记你整条调用链路所有环节，不观它是经过了哪些微服务，或者是调用哪些中间件。它只要在你当前的一次请求的调用链路当中，你就会在日志里找到这个 global trace ID，那这样的话，一旦我获取到了一行日志当中有 global trace ID，我就可以很方便的把它的上下游整个调用链路一网打尽，一下把它给拎了起来。
