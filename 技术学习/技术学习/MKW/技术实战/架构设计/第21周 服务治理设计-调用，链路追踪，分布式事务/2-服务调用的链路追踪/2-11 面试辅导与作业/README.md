比如说线上发生了一个服务调用的异常，那如果交给同学们，你应该是怎么来排查？实际上在大厂里面，这个都有很多固定的排查的方式，因为它们有非常丰富的中间件，像阿里当中使用的是鹰眼，那实际上这个中间件的功能非常类似于我们的sleuth，加上 zip kin 这个问题的回答点，我们就可以分成以下的三个部分。


首先定位问题线上故障，第一个我需要去尝试来获取这个故障异常调用的 global trace ID，比如说我去搜索当前这个用户反馈的一次 error 调用，或者我上下游业务方反馈了一次error，我在日志当中找到这个 error 它对应的 global trace ID，然后顺着这个 ID 咱去顺藤摸瓜就可以了。


使用 zip kin 将这个 global trace ID 直接定位到它的整个调用链路，那我们来看一下它这整个调用链路环节当中，它卡在哪一环。失败的这个环节，在 ZIP key 上面将会以一个红色的标记把它给标记出来，这就非常明显，可以一眼的看出它的这个症结所在。


除此以外，如果这个线上问题是一个性能方面的问题，那也很简单，我们在 zip kin 当中可以去查看你每一个微服务调用过程当中所经过的单元，它所花费的时间一目了然，以哪一个单元花费的时间最多，那它就是最有可能出现问题的这样一个环节定位问题的。之后，我们接下来就可以去使用日志检索方案。比如说我们本章节当中跟同学们介绍了Elk，或者我们使用 e f k 技术，进一步的去根据一些业务关键字或者 global trace ID 来去查找它详细的在每一个环节它打印出来的日志信息，这个是线上故障排查方案的一个回答的方向。


那接下来我们这里还可以问同学们另一个方向的问题，就是链路追踪技术，它的底层的技术原理是什么？那我们这里不光要去会用它，还要去了解一下它底层的原理。前面我们讲过，像 Sleuth 组件，它之所以能够在 spring Claude 的上下文当中，就是在整个全家桶的各个组件当中游刃有余，八面玲珑。它这里本质上是靠的一个标记加上适配器的这样的一个底层原理。标记很简单，我们就是打上 global trace ID， span ID， annotation 等等这样类似的标记，只用把它糅合进日志文件里面就可以了。


但是这个适配器就是苦力活儿所在了，那我们要在每一个组件当中开发它对应的适配，使我当前组件可以去接收上游调用方传过来的这些标记，比如说对于一些像 RABMQ 等等 MQ 类型的组件，你就有可能 sluice 把它的标记放到了 MQ 的 header 当中，我们的适配器就要是从这个消息组件的 header 里面去读取这些 span ID，然后来去看一下你当前是不是一个调用链的开始，但或者是调用链的中间环节。如果是中间环节，我给你生成一个 span ID，并且我生成一个 parent span ID 去指向你的上游调用来源，那它底层就是靠这样的一种方式来去完成的链路追踪的打标工作。


除此以外，我们这里的 ELK 方案还会对你的日志文件做一个分词构建索引，这个分词它底层会依赖于一些分词的插件，比如说英文分词、中文分词等等这种类似的插件，然后把它去构建在底层的 Elastic search 当中，并且使用 Kibana 作为一个搜索引擎的页面，来通过 Elastic search 这边暴露出的一些 restful 接口来去调用这些查询这些链路追踪的信息。

我们要充分发挥自己的主观能动性，利用我们章节当中学到的这些**基础知识，去网上，去开源论坛，或者去他们的这些开源技术的官方网站上去搜索解决方案，尝试自己把这个方案给落地。**
第一个方案就是去把我们的 ELK 方案当中的这个 logstash 前面再加上一个组件 file bit，因为 file bit 也是 Elastic step 这个套技术栈当中的意愿，那我们尝试使用 file bit 作为Logstash，还有我们的应用系统之间的一个桥梁，让我们的日志组件应用系统当中的日志组件可以经由 filebeat 将日志文件传输到 Elk 方案当中。这个是第一个动手作业，不会太复杂，也是蛮简单的。


第二个就是让同学们去把玩一下 Kibana 的仪表盘，那我们知道 Kibana 这里提供了丰富的日志检索的 UI 界面，以及一个非常功能强大的dashboard，在 dashboard 当中我们可以去自由定制各种各样的仪表，统计看板等等，这里同学们可以去 Kibana 的开源的资料或者是它的官方网站上面，通过一些技术文档来去了解学习一下如何在 Kibana 当中把咱的仪表盘给构建出来。
