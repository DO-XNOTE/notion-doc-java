那首先在一开始我这里要跟同学们抛出一个观点，就是大厂超高并发三高应用，它在一致性问题上面它有什么样的偏向性？那当我们讨论到一致性的时候，通常其实都是指带这种强一致性，那么我们在大厂的主链路当中，实际上对这种强一致性的容忍性非常低，它带来的一些资源性能损耗远远不及它带来的一致性的这种便利，那所以我们在这个问题上比较偏向于使用最终一致性。


OK，那了解了这个偏向性之后，我们就来跟大家去介绍这一章当中的主角 Theta 框架。那 Theta 其实除了强一致性的方案，它这里面主推的两个方案其实就是刚才所提到的最终一致性的方案。那在介绍方案之前，我们先来跟大家去了解一下 Theta 的历史，去看一下它的前世今生是如何由过去的蚂蚁金服，还有过去的淘系的分布式事务方案，逐渐演变成如今的这个 Theta 一篮子方案的。那再往后我们一起带大家去了解一下 Theta 是如何来拆解这个分布式问题的。


咱前面提到过 Theta 这里提供了不同类型的分布式事务解决方案，比如at、TCC、 Saga 等等。那么实际上这几个方案虽然底层的实现有所差别，但是它在顶层的框架设计上面，它在大方向价值观上面，它对分布式问题的拆解所用的原理是一致的。OK，那了解了 Theta 的以上内容，我们接下来就进入到咱的落地环节。

那这也是 Theta 当中默认推荐大家使用的一套无感知的分布式事务解决方案，那大家使用 Theta 这里它的神奇之处就是以不变应万变，你不用对代码做任何的业务逻辑上的拆解和改动，它就可以天然支持分布式。那我们这里首先分成这几步，三步。第一步我们先来了解一下 at 方案它的底层的原理，还有它的特点是什么。那这里将要连带着跟同学们介绍一个比较重要的知识点，就是分布式事务它的生命周期，那全局事务、分支事务它们之间是如何协同来做管理的？那这是其一。


其二，咱这里动手去搭建一个 Theta 的server，这个 Theta server 就像咱的注册中心、配置中心一样，它也属于一个中间件。那么我们搭建起 Theta server 之后，将要进入一个改造环节，那我们将使用基于 Neques 加上 MySQL 的方案来去实现 Theta 的 at 方案落地。这个方案是将咱的这个 Theta server 也做成一个微服务，将它注册到 Nequos 上面。与此同时，我的这些服务端，这些需要去应用分布式事务的服务，就到 Nequos 上面去寻找 Theta server 的服务，这个是和 necklace 的集成，这个 MySQL 在这里干什么呀？它在这里实际上不光是咱承接数据库底层的存储，它同时也是 Theta 控制全局事务分支事务回滚的一个要素。


我们将对 MySQL 当中的表做一些改造，插入一些 Theta 特有的表来去实现这个无感知的 at 方案。那 at 方案是我们淘系这里主推的一个方案了，那接下来我们将跟大家去看一看蚂蚁金服在自己的核心金融级链路上面的主链路当中，它的分布式方案是什么？TCC，其实 TCC 这里是一个由来已久的方案，发展了很长一段时间，那咱这一部分的内容分成两个部分。


第一节我们先跟大家去聊一聊 TCC 的原理，这个 t 和两个 c 分别代表着什么含义？那 TCC 这三部在分布式事务的一个生命周期里，他们会分别扮演什么样的角色？那这是第一点。第二点我们将跟大家做一个落地的实现，那同时这个落地实现也是基于咱前面的这个negos，加上 MySQL 这样的一个技术组合，来去把 c Theta 的 TCC 方案实现到咱的项目当中。


那最后一步我们这里要跟大家提及 TCC 当中的一些坑，因为相对于 at 方案， TCC 的成本是非常高的，那它需要你对自己的业务做一些拆解，对编码方式、对业务的逻辑实现做一些变革。那么在这个过程当中我们就要注意一些 TCC 当中可能出现的坑，比如说空回滚、悬挂、幂等性等等。那关于 TCC 和 at 方案的详细，对啊，它们之间的各自的优点、缺点分别是什么？我们到正课环节当中再跟同学们来揭晓。OK，那同学们，这就是本章课程的全部内容了，那我们正课环节再见。
