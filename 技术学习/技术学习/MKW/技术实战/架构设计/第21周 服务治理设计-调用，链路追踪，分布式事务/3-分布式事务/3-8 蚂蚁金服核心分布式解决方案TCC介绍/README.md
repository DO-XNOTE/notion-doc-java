那跟 at 方案同学们看出有什么区别了吗？咱的 TCC 它的一个业务流程的正向是分了两步，多了一步什么呀？多了一步资源锁定的操作？那从这里同学们应该可以看出一点端倪，咱多出来的这两步实际上跟 at 方案最大的不同还体现在这个TCC，它是一个有代码侵入式，也就是说我们需要自己编程来去实现的分布式事务解决方案。那你在自己的应用当中，你要去分别来调用这个 try 来去锁定当前的资源，与此同时你要去调用这个 confirm 来去将前面锁定的资源给它执行掉。执行正向业务这两步实际上都是一个代码操作，所以说它的这个性能比较快，而且它的这控制力度比较强的另一面也是有代价的，就是你的编程的成本比较高，你需要去自己来手动来实现。


OK，这里讲的是正向流程，与其对应的反向的 rollback 流程是一个 cancel 操作， cancel 这里执行的就是我们的业务回滚，那同样的这一步 cancel 也是需要我们自己去执行的，它可不是说我应用当前的 rollback info 来去自动执行一个 rollback 脚本，这个 cancel 是个要靠我们自己手动编码来去实现的一个动作。也就是说 TCC 这对应的三步都伴随着coding，那 TCC 的体验和 Theta at 的体验实际上就是一个手动挡和自动挡的区别。那你要是享受这个编程的快感，享受这个驾驭分布式事物的这种快感强迫症的同学，我建议大家可以去体验一下TCC，你就像拥有了整个世界一样，所有的环境都可以自由定制。那对于大家比较图省事分布式事物，我不想这样多余的控制，我就想一爽到底。那你这时候就要用自动挡 Theta 的 at 方案帮你解决这些层面的回滚，提交这些原子性的问题。
