那在这个过程当中，很自然的你的订单是强绑定到你的用户账户上面的，那么我在这个下单调用链路当中，自然而然我会去调用你的账户服务。那与此同时我要去检查你当前订单当中所下的这些商品，它是否还有库存，如果没有库存我会阻拦住你当前这个下单请求，那因此我们这里不可避免也会去调用库存服务，那其实我这已经是非常简化的一个下单场景的版本，那咱在真实的下单场景还有茫茫多的微服务。


如果你说到淘系的整个链路，你在一个下单从头到尾的场景当中，有可能要经历大几十个微服务，非常的庞大，那在这个场景下，我们一方面要去解决你分布式事物的一致性的问题，那么同时我还要避免尝试物，因为咱前面说过，像长事务这种 XA 的方案在互联网的这种高并发的场景上面天然不适合。
