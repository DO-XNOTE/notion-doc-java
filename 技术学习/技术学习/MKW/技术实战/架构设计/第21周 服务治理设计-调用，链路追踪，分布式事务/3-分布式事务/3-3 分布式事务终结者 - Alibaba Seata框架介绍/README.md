蚂蚁金服，它在处于上市风波之前，早在十几年前， 2007 年的时候他自己开发出了一套分布式事务的解决方案，叫做XTS，那后来也走了云产品化输出的路子，通过云产品输出了一款叫做 d t x 的这样的一个分布式事务的解决方案，那作为它的技术拓展的一个输出途径。


与此同时咱又有一个轮子制造者阿里系的中间件团队，那我们在业界当中听到的很多大名鼎鼎的中间舰，实际上都是阿里的中间舰团队这边来开发的。这个团队在 2014 年的时候也融入到了这个分布式事务解决方案的开发过程当中，他这边出品了一款叫做 TXC 的一款方案，从这个 t 同学们就可以看出这个 t 跟淘宝也是息息相关。


接下来它也走了同样的云产品化输出的路子，那作为一款叫做 GTS 的云产品，通过阿里云对外来输出故事又走到了 2019 年的 1 月份，那我们的阿里中间件团队将这款产品作为一个独立的组件，取名叫Fascar，自立门户了，这样一来，蚂蚁金服看了不乐意，说你阿里中间件团队老司机发车你不能带上我吧？毒越了与中越了，熟了不弱与众那所以蚂蚁金服这里高喊着，哎，老司机带带我，我也要上车。然后把自家的一个看家法宝 TCC 一起作为一个上门女婿送到了这个 fast car 项目当中。


那所以 TCC 其实是蚂蚁金服在它的核心金融链路当中独步江湖的一款分布式事务的解决方案，那然后在后面的一个时间点，这几款方案终于形成了一个大一统，打包成了一个叫做 Theta 的框架横空出世。这就是我们在这一章节当中想给同学们来介绍的这样一款分布式事物一站式解决方案，那同学们知道阿里系公司是一家很魔性的公司，它不光会给自己的员工们起花名，同时自己的项目名字也起得一个比一个带有魔性色彩。你像 Sita Feska，同学们，你说鬼知道他们是干什么的？ i 让你猜你肯定猜不出来，但是在阿里系里面，它起名有这样的一个约定俗成的一个规矩，就是你甭管这个项目它是干什么的，你起的这个名字要让别人不知道它是干什么的，这个名字就起得非常的成功。比如说我给同学们举几个例子，像迈巴赫、天梯、德鲁伊这些名字你听了你根本不知道他是什么样的中间将，但是隐隐约约觉得似乎有那么点厉害，所以结论就是给项目起名字是一个项目成功的关键，起名字要牛逼。

这里首当其冲的 at 方案，那 AD 方案也是我们 Theta 主打的一个方案，那它的默认选型会推荐大家使用的一个方案，它有什么好处？最大的一个好处就是无感知，你在自己的业务层当中不用写任何的代码，只需要少量的 annotation 上面的定义，它就可以为你的事物，为你的当前业务注入分布式事务的支撑，功能非常的强大，那这是 at 方案。


另外我们还有刚才说到蚂蚁金服这个上门女婿的看家本领，带着自己的法宝 TCC 方案跟 at 方案相比，它有一个明显的劣势，还有一个优势，劣势就是说编程非常麻烦，你要写大量的代码来做 TCC 的三阶段的准备工作、提交工作以及回滚工作，这都是需要代码来实现的，非常的重。那另外一方面它的优点就是性能非常的好，它的性能会比 at 方案提高一个档次，它也是用在我们蚂蚁金服的核心的金融级的链路当中的一个分布式事务解决方案。那与此同时，除了性能好，对你分布式事物各阶段的控制力度非常小，它是用代码来实现各种事物的回滚提交，所以你在代码当中可以做非常精细、粒度非常复杂的业务，那这是它的一个优点。


除此之外， Theta 这里还提供了另外一个解决方案，叫做Sega。 Sega 其实跟我们的 TCC 非常的接近，我们 TCC 这里实际上是三阶段的提交，后面在 TCC 章节当中，老师再跟同学们详细来说这哪三段。咱的 Theta 跟他比实际上最明显的一个改变就是说他这少了一步，他不用做 TCC 当中的这个t，也就是数据准备的这个阶段，直接就是commit。还有回滚，这里少一步，相对来说开发工作稍微地节省了一些实践，那这是 Theta 当中比较推出的主流的几个方案，那么我们既然谈到了已最终一致性方案优先，那么我们就不去来讨论 Theta 当中的这种 XA 的方案了，我们就只关注于以下的这 3 个方案。
