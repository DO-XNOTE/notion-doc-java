我们来看，那你一个微服务，它这里经过正常的生命周期就会有try， confirm 还有 cancel 这三个阶段，这三阶段大家都知道，我在一阶段发起一个调用的时候，首先调用的哪个方法？踹方法对不对？那如果同学们你试想一下，咱在这个踹方法的过程当中产生了一些异常情况，比如说你的丢包或者是连接超时导致 time out，在这种情况下，实际上你的踹方法并没有真正的去执行。与此同时，由于踹方法没有得到执行，因此你的全局事物最终会得出一个回滚的决议。


那回滚去执行什么方法？ cancel 二阶段必然要去执行这个 cancel 方法，这样一来，同学们想我就会产生一个什么样的情况，你的 try 没执行，而 cancel 执行了。同学们发现这里面的道理没有，轰回滚就是这样的一个原因。如果我们在代码里尝试基于 try 以执行这个语义来去做cancel，你在这里就有可能抛出一个异常，因为你踹方法其实根本就没有去执行。


如果你对一个没执行的方法做回滚，肯定会有一些不可预知的问题发生，导致你这个回滚一直不成功，一直的重试，这种问题有没有解决方案？很简单，我们通常的一个做法，还有业界的实践是在 try 的阶段，那我去做一些适当的手脚，这个手脚会去标记我当前的这个分布式事务在串阶段有没有正确地执行，那它可以简单地来说是一个标记，或者是一行表的记录等等。任何做法，只要把它当前的 try 阶段 mark 成功，我在 cancel 阶段就可以识别出这个特殊的标记，进而做出不同的判断。如果你的 try 已执行，那么我的 cancel 就相应地去执行回滚逻辑。如果try，同学们想 try 压根就没有执行，这个情况下我的 cancel 怎么样？啥都不执行，直接返回一个true，证明说我当前已经 cancel 成功结束这个分布式实物，这是空回滚的一个发生原因，还有它的对应的解决方案。

我们还把这个分布式事务的三阶段给它拉出来，try， confirm cancel。在前面跟同学们讲过，悬挂实际上和空回滚的这个打空箱是有异曲同工之妙的。怎么说？比如我们的一阶段调用，在调用到半路当中碰到一个拦路虎，这里网络发生了拥堵，导致你当前的这个请求我暂时还没有发出去，它卡在你的网络信道当中了。这样一来我自然而然会触发一个二阶段回滚。这个 case 是什么呀？我们的空回滚对不对？打空枪，那现在这个 cancel 怎么样？ cancel 已经空枪打出去了一把空枪。


接下来神奇的事情发生了，我的一阶段调用的这个网络阻塞突然又好掉了，你网络之前发出的这个包，通信包又抵达了 try 方法。这样以来就会发生一个什么呀？我们说的悬挂倒悬是什么含义？就是说你的 try 当前的 try 方法在你 cancel 之后得到了执行，这样一种情况，你的 cancel 实际上成功执行就代表着你的分布式事务已经回滚结束，那在分布式事务执行结束之后，这个 try 才到达这样一种场景，我们怎么来做？实际上有很多种防悬挂的理论还有方式，最好的一种方式实际上是在框架层面来去杜绝这种悬挂配置。这个方案实际上蚂蚁金服在它的TCC、 DTX 的云输出的技术方案当中已经实现了类似的防玄光，通过在 Facebook 的接口调用当中去指定一个特殊的参数就可以预防这个悬挂特征。但是在我们 Theta 的这个开源化的道路当中，这一步还没有很好地被支持。这个大方向实际上是已经定下来的，我们一定是要把这个房玄化机制放到框架当中，尽可能地减轻我在编程端的负担，这是其中一点。


另外一点，如果我需要在编程端去启动这个防悬挂配置怎么办？很显然的一个方案大家一拍脑袋就能想到了，就是说我的cancel。这里我要在对应的一些表上面做一些标记，或者是打一些相应的日志到我们的 MySQL 里面。这些记录会告诉我说当前这个业务组件已经完成了整个二阶段的回滚，那么我在 try 阶段就去检查。我在 cancel 阶段打上这个标，如果我识别出你当前的这个 cancel 阶段已经执行过了，我在 try 阶段实际上就不会再去执行重复的业务逻辑，这是一点。与此同时，我的 confirm 也要去检查这个 cancel 当中的点，如果发现它已经是一个回滚事务，那我 confirm 直接返回true，也就是直接返回 commit 成功，不去执行任何的业务逻辑，这就是我们的 TCC 的悬挂的处理方案。


在真实的业务场景当中，根据你的业务特征的不同，那你的房悬挂，那它和自己的业务流程是一个非常紧密绑定的一个特征，你就要去根据你自己的业务特点来去定义什么样的业务处理流程，才可以真正的防止这种倒选，防止悬挂的发生。这一节的分享就跟大家讲到这里了，我们了解了 TCC 框架在实现层面的 3 个坑，空、回、滚、幂等性，还有悬挂，那么我们这一整章的内容在这边就宣告结束了，就给同学们留下了非常充足的时间，去来把玩我们的这些落地小案例，再去对咱的 TCC 还有 Theta 的 AD 方案做一些更深入的了解。那我们下一章节再见。
