一个模块和另外一个模块之间的调用，这才是什么耦合产生的原因，对吧？因为只有两个模块之间有一些紧密的交互调用，才导致这两个模块之间出现了耦合。而如果去看它们之间的调用有哪几个方向？一个是看它们中间调用传输的数据量，数据量传输的越大，传输的频率越高，证明这两个模块之间的耦合度越强。


那同时要看一看模块和模块之间是传输普通的数据还是控制信息。当一个模块向另外一个模块发送控制信息的时候，那这两个模块之间的耦合性已经比那普通的数据传输要强得多。因为你控制了我，所以你的改动会导致我要做相应的改动。那除此以外还有什么？还有模块跟模块之间的接口是不是很复杂？我这个接口要传什么？要传十几个。嗯，不同的参数，有的参数是变量，更多的参数是一些复杂类对象的引用来作为参数来进行传输，这个就很不适合什么做模块之间的接口的使用，所以尽量减少模块之间接口的复杂程度，也是我们关注的核心点。

因为接口通常是定完以后很少改动，所以你两个类，比如说类 a 和类 b 本来是直接是调用类 b 里面的公共方法，那这种情况下你的方法有可能会变化很多。但如果你能抽象出一个长期不变的接口，那我只要什么调用你的这个接口，而你的那个类病人自己去实现这个方法，它既实现了什么依赖的反转，也实现了什么？应用和应用之间，模块和模块之间通过接口实现了一层解耦，防止什么任何你的微小改动导致我的代码要重新编译，那这是一处，然后第二处就是尽量使一个定义值在一处出现。


那我这里还要举一个很常用的例子，就是用户在我们的电商平台里有什么内部用户，有买家，有卖家，有这么多用户。如果每一个模块里都定义一次user，定义一次用户，那你就会出现，当你向对方传递一些用户信息的时候，你很可能是用自己的这个用户的这个对象的方式进行传参，那导致对方是无法能理解。


所以这个时候我们就介意什么建议，就是我们把所有的用户重新定义，比如用户定义成塞了，卖家白了，买家在买的领域里面我们就叫它白了，在卖的领域里面我们给它塞了，我们不要再叫用户。这样呢？你的类不会呢？有多重定义不会呢？出现嗯混淆的情况。那除此以外什么不要用全局变量？既然我们连什么命名都尽量分开了，那我们更不应该什么在整个的大的这个代码段里面，还使用全局变量来进行什么跨和模块的这个数据传输。那第四条原则就是什么？我们前面已经反复聊掉了，不管你是成员变量也好，你是成员方法也好，都尽量少用public，都用什么 private 把我自己的数据保护起来，对外减少暴露，减少接口。

那除此以外我们也建议多用一些设计模式，比如像是Adapter，对吧？适配器模式， facade 这种代理的模式，因为通过这种模式既可以实现类似于领域驱动里面说的反腐层，就是不管你怎么改动，我依然不变。另外也可以实现一些类似于什么依赖反转等等功能。


那通过这些实现的方式能够尽量保持什么？模块 a 和模块 b 之间没有一个紧密的代码的核心的业务层的耦合，了不起，是一些接口的耦合，那些接口通常什么是第一次制定完以后，未来可能很长一段时间都不会改变，这样就不会出现你只要改你的逻辑业务，就导致我要我的代码要重新编译，重新改写重新发布。那这种情况就太耦合性太高了。我们尽量通过一些设计模式减少模块之间的什么紧密关系。


第六条就是什么？我们其实模块不光是指领域模块、业务模块也包含，就像中间件模块、数据库模块都可以当成模块。那以数据库为例，尽量不要在应用里面直接用 CQ 语言进行什么增删改查的东西，我们可以用 DAO 层封装一下，那封装完了以后我们跟 DL 层之间是可以什么进行直接的文章，这种就是对象的处理的，那 DL 层下面再去封装不同的数据库，那如果数据库我们换了一个选型，或者数据库进行了升级，我们只需要改动 DAO 层就了不起就到此结束了。


我们对应用逻辑是不需要做什么相应的改动。那我们低耦合的第7条原则是什么？

是说如果你实在是需要调用，但你也用什么用 API 的形式，或者是什么调用它的什么公共接口的或者公共方法的形式，不要去直接访问它的内部的那些 set 的语句，对吧？就不要去什么采用内容耦合的方式进行泪和泪模块和模块之间的这个沟通。那同时我们尽量要减少控制耦合。就如果我和你之间传输一个普通的数据变量，这是很合适的。当我传输一个控制命令给你，让你去执行分支 a 还是分支 b 还是分支c，这就不是很合适，所以尽量采用一些其他的方法来规避掉这些控制命令的发送。


那最后一条就是什么要限制公共耦合的范围，我们前面也聊了，就是公共耦合，你可能要共享数据库，或甚至于说共享消息队列也能算是公共耦合，那有时候是什么必不可少。但是你尽量采用一些松散性的公共耦合，比如 a 向 b 传输， b 不会反写同一个公共区，写回a，那同时也尽量减少它的使用的范围，都用一些松耦合的一些调用，或者是远程API，或者说是一些简单数据传输的方式，减少那种强耦合，减少公共耦合的出现。

那除此以外我们还有什么？还有一个原则叫接口隔离原则，我大家还记得之前学的那个接口隔离原则，就应该能理解到，就是我们其实那个月租系统就是一个接口隔离原则的很好的例子。月租系统如果去跟水费计算系统沟通的时候，我只把用水量发给你，我不会把用户的住址，不会把用电量发给你。这样减少什么模块跟模块之间传输的那个数据，传输 DTO 的大小，减少它的那个接口的复杂度。同时我们对外的接口尽量越少越好，就类似于领域驱动里面说的六边形原则，一个模块对外只能提供什么 6 个不同的接口，用来调用别人或者被别人调用，这就是什么尽量对外暴露最小限度的接口隔离原则。


那最后一条内聚原则就是一切向功能内聚靠拢，向飞扬老师这种专一型的靠拢，坚决杜绝像什么大锤这种偶然内聚，非常之花心的这种情况，尽量越少越好。聊完了什么高内聚低耦合的一些原则，那下一节我们又回到什么王大锤时间，我们来看一看王大锤曾经在他的项目过程当中出现的两大问题，以及什么这些坑所产生的深远影响，大家敬请期待。
