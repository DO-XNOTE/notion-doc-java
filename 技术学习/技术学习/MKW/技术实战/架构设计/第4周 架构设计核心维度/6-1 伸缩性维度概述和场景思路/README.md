另外从方案上我们会看一下怎样实现网络入口层、应用层和基础架构多层的伸缩性，同时对于有状态和无状态又应该如何来处理？而之所以要把伸缩性作为一个独立的应用维度是因为什么？它看上去跟扩展性是不是很类似，但是他们的侧重点不太一样。扩展性是一个企业三年、五年、未来需要不停的扩展的过程，当中应该如何增长？高性能是指我们整个系统怎么样投入更多的资源，实现更强的吞吐量？而伸缩性是什么？是在业务低谷的时候，我要为业务的高峰做准备，这个高峰可能在下次的什么 1111 会到的，也有可能是下个月的月底的某一个特殊的热卖或者抢购的时候，会突发的增长。但当高峰过去以后，我们又需要把我的资源收缩下来，节省成本，成为一个低谷。就像这里说的，人生软有潮起潮落。比如我们在软件工程师的时候，有一段有一阵一直在写CRUD，写厌烦了，我们去搞点领域设计，去写一点 if else，那 if else 又写厌烦的时候，我们去搞一些架构设计，搞些云平台设计，做一些转型。


那逐步为自己积累成为架构师的一个潜在的可能是一样的道理，一个企业当业务在低谷的时候，要准备好有可能瞬时间未来，突然之间我们业务井喷了，爆发了，我应该如何快速的去应对？用一种伸缩性的思路去应对我们架构的变化？那同时当业务重新恢复到低谷的时候，不要节省成本，把资源快速释放掉，那这是伸缩性里面，快速的生长和快速的收缩就是伸缩性的关键点。

业务的爆发期，我们需要有一个伸缩的能力，那业务的萎缩期，我们也需要一定的收缩能力。什么时候业务爆发了最夸张？那当然是电商的秒杀抢购平台了。当然不管你是秒杀还是抢购，我们不需要把整个数据中心都具有伸缩性，我们应该什么减少我们的目标范围，我们把所有的压力或者能力要关注在我们的核心链路的热门热点业务上。比如大部分的电商平台都是允许用户提前把商品放到我们的购物车，那这个时候在 0 时零分零秒我们只需要下单，是吧？支付，那这两个功能只要能够支撑快速的伸缩就可以了，但飞扬老师也碰到过这样的客户，他不支持提前查询这个商品，这个商品很神秘，有一个广告，但你无法查询到了。


0: 0 零秒，你可以搜索到了，你也可以从商品导航栏里看到它，然后你可以点击查看商品详情，可以添加到购物车，可以下单，可以支付。这样从搜索到支付的整条链路都成为了它的核心链业务，所以它的伸缩性的要求就更高了，它需要更多的模块具备伸缩性，那除了业务以外，大家想一想，业务通常是可以是无状态的，但后台的数据必然是什么？它的状态的核心，那这些数据包含像秒杀系统它的商品信息，这些静态的数据，那它需要有大量的读请求去查询这些信息，那除此以外还有库存，还有支付相关的什么交易相关的动态数据，这些数据有大量的读和写同时产生。所以对于有状态的系统来说，对于后台的数据或者舰支队缓存来说，我们也需要具备很强的伸缩性。那总体来看，其实有这样一个思路，就是当你的业务需要伸展或者收缩的时候，通常你是在时间跟空间之间进行转换，进行抉择。


假设我时间很充裕，我这个业务系统虽然爆发性很强，但是它过程相对缓慢，它要十来分钟才爆发出来，那这个时候我完全可以让系统通过一个弹性的功能进行伸缩。一些虚拟机，花个几分钟时间把虚拟机建出来，配置好，然后准备好，为了十几分钟以后的爆发准备。诶，这种周期比较长的是吧？是用虚拟机的技术，但如果时间很短，那个时候你要预配一些虚拟机，预配一些服务器，然后采用一些容器化的技术，更快的秒级或 10 秒级的进行什么资源的快速的升级或者收缩。那除此以外还有没有什么其他技术？当然有，缓存的预热，我们服饰的预配置等等大量的不同的技术。是或者是牺牲空间，或者是牺牲时间，有没有一个在时间空间之间转换最灵活的思路？那就需要你具备红色的词，五个字，系统的伸缩性。一个系统它的伸缩性越强，它在时间与空间之间转换就越强，它能瞬间大量消耗你的 CPU 内存，长出很多很多容器，也能够快速什么消亡大量容器，释放你的资源。
