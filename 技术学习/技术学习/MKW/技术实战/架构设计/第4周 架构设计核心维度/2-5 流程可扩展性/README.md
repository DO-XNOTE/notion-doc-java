配置管理、变更事件告警等等很常见的什么流程管理，同时也跟我们的架构息息相关。那在 i 套基础上还有一套更加宽广的、更加宏大的服务治理和管理统筹的思路，叫ITSM，相对来说都是大型企业采用了一套管理方法和管理流程。那还有我们传统的企业里面，即使是 it 或者是什么汽车制造，或者是像摩托罗拉这种什么通信设备制造厂商，他们会采用什么利用 Sigma 保证质量、严控产品功能的一套我们的新的，或者说是一套非常酷炫的流程管理系统。
那还有什么？还有互联网公司作用最多的，什么 CICD 的，什么持续集成、持续发布的流程，也是不是一套管理系统？它控制什么？控制你代码的发布，控制你这个什么我们的弓箭或递交箭的准备，以及控制我们这些弓箭或者递交箭如何的发布到我们的什么不同的系统上，同时在这些系统上面怎样进行负载的掌控，怎样进行横向的弹性的扩缩容，那所有这些其实都是流程，有没有和我们架构是特别相关的？流程？有，是不是这个联合架构设计？ join the Architecture development，联合架构设计是专门跟架构有关的一套流程，他说的是不光是架构师，你还要什么？你还要找到相关的产品经理、项目经理，我们的这个研发的主力的成员，还有像 SRE 这种系统的或者是运维的专家，一起来什么进行一个产品从功能到非功能到落地到发布的联合的开发设计和那个design。


那除此以外还有没有什么重大的这个架构流程？有，是不是就是架构评审委员会 Architecture review board？那这个在不同的企业有不同的组织形式，比如像一些大型企业，它通常会有一个架构办或者架构组有专门的专家来负责整个企业的架构决策的评审。


那如果是没有这样的组织的企业，很轻量级的企业，它通常会什么？从核心的几个业务部门里面抽调他们的应用架构师，或者甚至是我们的开发的组长来形成一个虚拟的架构组，来评审一些重大的决策。这个也就是我之前聊到的什么决策派里面通常会做的一件事情，对核心的架构的决策和核心的我们的变更做出一个统一的最后的拍板，到底做还是不做？或者是应该选方案一还是选方案二？那今天是不是我们就要聊这些呢？诶，不是，因为所有这些方法，什么都是一些数，一些基本的方法。
