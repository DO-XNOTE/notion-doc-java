如果给你一套传统的单体架构，如何确保应用的扩展性来满足未来 3 到 5 年的发展需求？这里有两个关键词，一个是单体架构，一个是 3 到 5 年。大家还是一如既往321，请作答。


相信大家都能聊出一些什么，我们的扩展是吧？ XAYAZ 这道题如果是从单体架构和 3 - 5 年这两个体验来看，大家应该是更偏向于哪个轴？当然是什么 y 轴。那首先再说单体，自然是说到什么微服务的改造，受到服务的拆分。面试官肯定希望大家能聊一聊怎么破单体架构这个局。同时一旦说到 3 - 5 年，这就是个长期规划， x 轴和 z 轴都是短期见效的方案。而 y 轴才是什么？能把服务拆解开来，拆解它的依赖，拆解它的这种冲突，同时能让这个应用不停的迭代过程当中，系统架构的不停衍生的扩展性方法。所以这道题的面试更应该偏 y 轴，去聊一些具体的案例，聊一些你的想法。


同时在 y 轴这个大套娃里面，永远可以套中套娃 z 轴，也可以套小套娃 x 轴，所以可以以 y 轴为主体，也聊一聊 x 跟z，这样整个的这个面试的这个提供的这个答案就感觉很丰富，那有没有什么亮点？当然有，如果你能从系统、从网络、从数据，能把一些实际技术去聊一聊同事，不光是x、y、 z 这三个方向，你也可以能把系统跟数据分开来聊清楚。这样的一个矩阵式的疗法，就能体现你架构师的宽广，同时你有实际案例，你有实际技术，你也能体现架构师的深入。所以这道题是个发挥题，通常你能聊一两分钟就聊完，就证明什么你还没发挥到位。一道题目能聊个四五分钟，那差不多你就能把这个什么，你的经验、你的功力能体现出来了。

架构决策是如何考量扩展性？企业的不同角色岗位又应该如何来配合你的？唉，这里有好几个点，一个是决策，一个考量，还有一个配合好，我们还是32，请做答这道题目，看看是什么。是短短的一行字，其实考的点很多，我们看一下这种开放性题目，通常都是一句话，是一个什么，一个很粗犷的一句话，这里面其实是在考决策论的内容，考一考你是怎么样决策来保证企业的架构具有扩展性的？同时也问到了角色，这就是在问你们组织怎么样体现扩展性，又问到了如何去进行决策，如何去考量，这就是在考流程上。


所以我们可以决策论的观点，从组织上，从流程上，也从架构上，多方面去聊扩展性，怎么样才能让这个面试官眼前一亮呢？当然是要用一些方法论了，比如结合 rusty 矩阵，聊一聊你们企业的不同岗位怎么样配合起来来观察架构设计以及评估扩展性。同时从决策的角度来看，我们要什么？要定一些目标？ CMMI 模型你到底定在哪一层？是为企业制定好一个整体的战略，你又是怎么样去执行它？你是不是采用了 smart 原则，做到了足够定量的来考量和评估你的扩展性？这些什么都是一些亮点，通过一些方法论体现你们你架构式的这个资历，同时讲一些实际的案例故事，讲一些有深度的技术描述，来体现你作为资深程序员的功力。

这就是一道标准的计数题， 32 一请作答。那这道题呢？起点是什么呢？其实是什么？是讲数据的 x y 和 z 轴的这个扩展，你可以聊一聊 x 轴上怎么样进行水平的复制的扩展，也可以聊聊在 y 轴上我怎么样分库分表，我们怎么样把多个不同的库表采用一些中间件技术，或者是客户端技术或者是数据库技术来混编成一个大的这种关系型的数据库。


你可以聊一聊 z 轴上的我怎么样采用一些哈希曲模的分布的式的方式来实现，不管是关系型数据库，还是采用一些 no CQ 的原生的功能来实现整个应用数据的扩展线？那什么样的点才能让面试官眼前一亮？那我就要提一提了，一旦这种技术提，那必然是什么？先说一套理论，可能只花 1 分钟时间，然后要重点挑一两个案例，讲一下你熟悉的技术。比如你熟悉MySQL，你至少要把MySQL，对吧？怎么样做CDC，数据抽取？怎么样进行分表分库聊清楚。比如你讲MongoDB，那怎么样？分片怎么样？多副分怎么样？在实际案例中，考量其中的那些参数的选择，必须要聊得很清楚很深入。这种偏技术性的题目，通常还是希望在前面戴一个帽子，比如 XYZ 的这种理论性，然后聊一两个案例里面，把一两个技术聊得非常深入，这才能体现你架构师的功力。
