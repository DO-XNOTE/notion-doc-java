， C AP 理论里面的左下角的 a 就是可用性**availability，那跟它相对应的还有两个元素，一个是 c consistence 一致性，另外一个是 p partition tolerance 分区容错性**。那通常实现任何一个维度非常简单，实现两个维度稍微要有些架构施功力，实现三个维度那就是不可能了。


如果要实现可用性，如果是在 10 年以前，大部分的架构师都选择左边这个维度，也就是既高可用又强一致性，采用什么集群架构？现在很多的电商我们会追求什么？强大的扩展性，强大的并发性和吞吐量？很强的这个 IO 性能？我们的这个 QPS 跟TPS，那这个时候我们有可能选用下面那个，就是既高可用又分区容错的分布式架构，它会牺牲掉强一执行变成什么最终一执行，但是它的节点不会再像集群那样，通常是 3- 5 个节点，那么少了它会多少？几十个、几百个节点，是吧？很强大的高可用，同时能实现分错，但是一致性会略微降低。


好，我们也以这个思路，先讲讲集群架构，再讲讲分布式架构。看到集群架构，我们就是刚刚说的什么 c 和a， consistency 和 availability 都是它的强大的优点。有哪些层次能实现这个集群架构？

通常我们首选采用跟操作系统供应商相对应的这个应用集群。比如你 Unix 选的是 IBM 的 power 的AX，自然而然就会去选 power HA 这样一套集群，它能支撑上面。比如像阿帕奇服务器，他们开的服务器实现高可用，跨多个节点对外提供这个负载均衡服务，那后台的数据是完全强一致的。


如果你选的不是 Unix 而是Linux，假设你选的是 red hat，那通常就会选什么 red hat class Suite 这样一个天然的。这个 red hat 提供了这个应用集群软件，恰巧你用的是是Windows，而且你还有传统的 Linux 在同时在跑，这样你的导电台应用和你的 Java 应用怎么样共同实现？应用集群也是有方案的，这就是什么第三方软件，比如像赛门体克的 veratus class 的 Server VCS 就是一个很经典的方案，跨操作系统，跨应用集群，最终实现一个统一大集群，对外提供各种各样服务的跨节点高可用。


聊完了应用集群，通常我们在应用和数据库之间还有什么？还有中间件，不管是什么我们的 CACHE 的缓存机制，还是消息队列，还是 API 网关，这些都有一些中间件的集群技术。以传统的比如像 Oracle 的 Web Logic， Ibn 的 Web Sphere 为例，他们天生就是高可用，同时还能支撑强一致性，但是他们的节点数量通常不能达到太多，这就是中间件的集群。最难的其实不是应用的，也不是中间件。大家都能知道，集群里面要强一致，最终还要靠后台的数据集群。如果你选数据库，假设你是以传统的Oracle，会推荐你用 Oracle rack，它可以跨多个节点，后台有 ASM 这种就是数据的集群。同时上层还有 cluster 薇娅中数据库的集群，它们组合起来实现一个统一大数据库集群。


类似的，如果你选的是 DB two 的数据库， DB two 会推荐用 Po scale 这样一套机制来实现它的集群，那假设你是用的是文件系统，我们有一套很经典的叫做 general paralfisystem GPFS，它是强一致性，高可用文件系统，不像后面的像哈杜普那些。只是什么是最终一致性？它这个是强一致性的高可用文件系统。但如果你要实现强一致，通常不建议你采用太多的节点。

在集群跟存储之间通常是有一个联调配合，这时候集群会告知存储，存储也会告知集群发现主什么存储失败，这时候同步会自动停止右边那个备用服务区的服务器，加上存储会全自动的对外提供服务，所以在这个高可用集群里面，这种情况下我们的备用区自动对外提供服务，主节点会停止使用。


那这个时候仍然保持什么业务是对外正常的？当我们的主存储恢复完以后，我们可以通过人工的方式进行干预，保证我们的业务重新切换回主服务区，变成左边两台服务器对外提供服务，左边一个存储对外提供服务，右边成为冷备状态。好，这就是通常的我们一个系统集成商对外提供了一套 POC 的环境，然后让客户在上面实操，去感受一下，当任何我们的节点单点出现故障的时候，不会影响业务，同时的数据是强一致性，不会丢任何一份数据的。聊完这个以后，我们再来看一看什么分布式架构，分布式架构的核心是什么？
