一个用户通过浏览器把这个我们的访问需求发到了CDN，很不幸没有缓存命中，他又把这个访问需求发到了反向代理，仍然没有缓存命中。


他发到了我们的应用服务器，我们应用服务器让这个用户成功登录了，但是登录过程中发现一个小问题，这个用户怎么今天在非洲了？上次登陆好像他是在中国，他是个中国用户。好有点疑虑，那我们怎么办？我们通过 SMTP 服务器给他的邮箱发一封邮件，如果是一个同步设计会怎么样？当应用服务器找到SMTP， SMTP 成功地给用户发完邮箱之后，我们才返回给我们的终端，说我们发完邮箱了，这种时候可能用户还没法登录，要用户什么点击完？我这确实是本账号本人登录，然后才能够登录系统。这就是一个同步的设计。是不是全程都需要等待？等待所有的返回完成以后，才到界面上展现出我已登录成功。


如果我们换一个思路，可不可以这样，我们把同步变成异步？当我要发邮件的时候，我就告出 s m t b 服务器，请帮我发一封邮件，内容如何？ s m t b 服务器也就直接给那个用户发了一个邮件，不用等待邮件正确的让用户收到，也不用等待用户打开邮件，甚至于我们应用服务器都不等待 s m t p 服务器给出反馈，直接往它上面吐掉消息就结束了。


这样一个应用改造以后，是不是我们的用户登录的压力就小了很多？我们不需要有个长长的等待，不需要挂一些一个session，挂很多的这个缓存信息压在我们的应用服务器上，很快就把用户登录完成了。但是它对于用户的体验来说会有什么样的区别？有可能别人可以盗刷我的账号，是不是？所以异步什么是有妥协的？它是拿什么业务的时间性来做妥协，如果你的业务必须是同步的，要完全得到确认才返回，这种时候你并不能转成什么消息队列的机制。但是如果你在时间性上没有这么强的要求，比如说我用户让你登录了，没关系，我只要什么，在你后面支付的时候我卡住你，我认为你这个账号可能存在疑虑，我不让你正常的支付，是不是一样能解决防止别人盗刷买些这种非法的这个产品，或者是在网上挂假产品然后盗用别人账号的问题？虽然你用了我的账号进行了浏览，但是你并不能做很多出格的行为。


通过一个异步的方式改造了应用的逻辑，但是能简化很多什么很多 IO 的等待，很多这个 sql 的等待，很多 API 的等待的压力，如果我们把这套思路用到我们的服务器和服务器之间，是不是这样？应用服务器可以通过消息总线，比如像ESB，或者我们叫它消息队列，或者叫它服务总线都可以。


不管你是 ESB 还是 message bus 还是 message queue，不同的名称，其实作用之类似，都是可以把同步转成异步，然后让另外一台服务器来异步的消费这条消息，通过这种方案在业务上做少许的妥协，也许在性能上就有很大的提高。我们看一下异步到底能给我们带来什么样的好处。
