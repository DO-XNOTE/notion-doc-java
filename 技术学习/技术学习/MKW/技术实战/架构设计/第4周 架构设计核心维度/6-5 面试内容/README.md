大家一如既往三二一，请作答。相信大家对高爆发，尤其是在这一章节，是很容易摸索它的亮点或者它的关键点，它的体验就是要看这个爆发性，所谓爆发性就是短时间之内要增加资源，这就是伸缩性的一种体现。同时既然是爆发，那必然是什么？是处理一些秒杀、抢购这些高性能、高压力的业务要求，所以伸出性和高性能都是可以应对这种面试场景题的。


那如何才能把这道题目答得很有亮点呢？一如既往要结合实际，你要能拿到加分项，必然你是结合实际，聊一个实际的高爆发场景，你怎么样去应对其中的？通常一个实际的高爆发，从外面来看有网络层，从应用来看你有无状态，往后看你必然还是有状态的内容的，不管你是后台的分布式数据库还是其他的，就整条链路，从网络到应用到后台的什么？有状态的这个存储或者数据库如何来实现？一条链路都能支撑高爆发，一条链路都能支撑伸缩性，这是一个关键点。


那另外我们也通常在这种环节当中要聊一聊我在非爆发的时候怎么样已经在价格里面用缓存、用异步、用分布式实现了高性能的设计，那在这种情况下就是平时已经高性能，同时我的伸缩性很强，我能很快的增加我的容器数，很快的实现网络的这个盐脑手的执行、思考和切换，这种情况下才能真正实现一套高爆发业务的。

重点在于业界流行。32，请做的这道题就不像前一道题，一旦聊到业界流行，必然是要聊好几个架构，对吧？不同的架构他们怎么样用？他们的区别在哪里？这些架构必然是跟技术有关，所以通常都是要结合实际的技术的亮点来看一看这个技术能针对什么样的服务，和通过多个维度来量。那有一个维度就是无状态维度，那比如亮点的方法有伸缩性的功能模块，比如像容器化伸缩、网络伸缩，也有一个很知名很亮的 service 的概念，你能把这个讲清楚，基本上就把无状态应用的伸缩讲清楚。除此，有状态应用在不同的场景下，特别是在共享磁盘的时候，如何向有无状态应用进行转译，怎么把它的状态应用 Poly 到后面对结构化怎么Poly，对于非结构化怎么Poly，怎么样 Poly 到网络上进一步去降低我们系统的压力，进一步去提升我们系统的伸缩性。这些都是可以聊的。那要拿高分必然是要讲很多技术细节的，一旦是业界流行架构，这个架构必然是通过一些业界很流行的架构名词，比如像 Kubernetes 的HPA，横向的 Pod 的扩展性，VPA，纵向的 Pod 扩展性或者之前聊到的 Chrome P a 时间驱动的 HPA 等等。


以及像 k native 这种servist，那 k native 还可以跟什么 is still 这种 sense match 结合，跟 Kubernetes 这种集群化的容器化的编排工具结合，它们之间是如何之间互通的，它们是如何配合的，首脑也是如何配合的，这些你能讲清楚，同时你能结合你的实战的经验把它的原理讲清楚，同时把它们配合和使用的方式和注意点讲清楚。如果进一步，你还能聊一聊以前发生过的一些坑，在未来的实践当中应该注意哪些，那这道题你就把真正业界流行讲得很透彻。
