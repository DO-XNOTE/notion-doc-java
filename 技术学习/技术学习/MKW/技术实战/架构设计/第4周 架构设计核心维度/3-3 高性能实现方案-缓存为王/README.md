我们先从读跟写的角度来看，读缓存跟写缓存谁更重要？通常我们把读缓存叫做cache，写缓存我们也可以叫写缓冲叫做buffer。那读跟写我们压力怎么样？通常是 10: 1 到 100: 1，那自然读更困难，读也更重要，所以我们这一节重点讲的是读的case。那我顺便提一下写 buffer 的优化在哪里？写buffer，假设你每次写的是 100 字节的一个IO，我们可以把 10 个写 buffer 合成一个，也就是 10 个 100 字节合成一个k，一次性写进去，这就是写的缓冲的一个技术实现。


那读缓存其实是技术远远复杂得多，我们来看一看写缓存有哪三大技术，一是网络缓存，二是应用缓存，三是对象缓存。


而跟我们很多的这个应用架构师理解不一样，菲亚老师要**推荐能用网络缓存的都用网络缓存来解决，不行才用应用缓存，最后不行才是对象缓存**。好，那我们就来看一看原因，先看看网络缓存，

（Edge — 边缘， 边角）

我这里以那个扩展性架构的那张图为例，当用户浏览假设我们的 imock 网站的时候，他会去先经过一个CDN，然后再去访问我们的 Imock 的服务器，那最后才是从数据库里读数据。好，整个过程当中最好的情况是什么？恰巧这个用户以前访问过imook，所以它就有慕课网的这个静态的内容，同时它的 cache control 里面也设置了，是吧？允许把我们的 public 的这个 cache 使用到我们的这个浏览器当中，同时我们的这个静态页面还没有expire，同时还有一些 pragm 也支持这个cache，能够让我们能正常的使用。那这种情况下你浏览器就找到了内容，当你打开主页的时候，你根本不需要去访问慕课网任何的资源。诶，是不是很快也节省了我们服务器的资源？好，假设运气不好，这个人是第一次访问慕课的网站，那它怎么样？它就会把需求发到 CDN 上，所谓 CDN 下面会有一张图，看看什么是CDN。
