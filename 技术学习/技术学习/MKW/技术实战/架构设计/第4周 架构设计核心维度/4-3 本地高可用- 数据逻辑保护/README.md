他怎么说一个企业对外的业务可能发生短时的不连续，它的不连续 80% 经统计来自于有计划的事件，比如系统升级、打补丁、老系统下线、新系统上线等等，虽然 85% 的时间来自于它的影响，但是影响的金额并不大。


为什么？因为多数我们的计划类事件都是在业务最不繁忙，没人使用的时候，偷偷摸摸的在那里进行，那计划 y 事件如何？虽然时间只占了15%，影响面却非常大。而其中又有三种场景流程和人为的故障，比如有人删库了，这个故障占了 40% 左右，软件发布成一个错误的软件，导致业务没法使用了，又占了40%。


这两个问题在本地高可用能不能解？不能解对不对？本地高可用解决的都是硬件故障，像服务器单点故障、网络单点故障、存储单点故障。所以说我们在本地高可用投的钱其实只是做一个基础保证，但是真真正正最大的问题并没有防止住，对不对？好？ 85% 15% 是不是加起来就是 100% 了？我们这个饼图里面那个缺的那小片是什么？我们看一下。哎，自然灾害导致的什么？我们的容灾切换，所以什么？平时我们说两地三中心，四地五中心很悬，其实是做个样子工程。是不是我们大部分时候不希望采用我们容灾方案，即使我们把很多的钱投在我们的灾备中心，但我们也不希望真正去启用它。什么样的情况才是我们最要防治的？当然是什么？ 15% 里面的80%，这就是红框框标的地方，也是最应该投钱的，防止那些人为的或者软件的故障导致我们的数据丢失，导致的我们的服务不可达，这就是数据逻辑保护所要做的功能。好，我们来看一看逻辑保护我也分成了三块，这次我们不叫它三叉曲，而叫它三部曲。

那第一部曲逻辑错误防治，第一部曲就是要预防，所谓预防就是要平时经常做数据备份，或者是快照，或者是备份。同时我们尽量要让我们的应用以什么 n 和 n 减一共存的形式进行发布，这样可以防止应用的问题。通过数据备份能保证即使我们出现删库的时候，我们的应用也能够恢复到删库之前的备份数据。除此以外，我们的变更审核要非常之严谨，关于变更审核，**关于我们的系统应用架构以及 n 和 n 减一的发布和并存，我们会在 Devops 后续章节详细跟大家聊好。**


这第二个我们的布局是什么？就刷线和监控我们需要什么？我们需要做到全自动化监控，全自动化脚本，并且我们要界定出什么是系统的正常行为，什么是异常行为，通过这样的界定才能帮助我们比较方便的判断我们什么时候需要采取数据回滚，什么时候需要采取快照的恢复。
