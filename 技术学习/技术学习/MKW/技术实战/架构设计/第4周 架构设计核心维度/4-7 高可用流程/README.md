那同时谷歌这种很框架式的理论是没法落地的，那真正该怎么落地？谷歌也有一套思路，就是什么让更多的开发工程师去承担我们的业务的发布，我们的这个故障的，这个告警的应对，我们及时去反省我们的代码质量为什么这么差，然后快速的去提供bug， fix 所有这些文化通过下面这个机制来实现，所以说技术上我们要什么？实时监控我们的整个系统追踪谷歌最关键的四大黄金指标，第一指标是延时，这个延迟包含像是我们 API 的延迟包含我们什么？我们的数据库访问了 sequel 的延时，也包含一个IO，在磁盘，在网络上面的 IO 延时，那流量也要包含两种，一种是类似于IOPS，一秒钟你有多少个IO，有多少个交易，还有包含带宽就是我们一秒钟能跑多少个BYTE，在网络上面，在我们的光纤交换机上等等。


那错误也有很多种不同的统计方法，那有的部门他把只要除 200 以外都算是错误，还有的他认为 4XX5XX 才是错误，那饱和度也有很多个不同的这个维度，比如说网络的饱和度、网口的饱和度、我们磁盘 IO 的饱和度、我们 CPU 的饱和度和内存的饱和度等等，所有这些指标实时监控，不停地记录，最终就可以形成我们的这个目标了。


SLO 什么是我们的 service level objective 服务这个目标，它是在一定时间内、一定范围内满足一定规则的指标。那这里付杨老师举了个例子，半年内我所有的支付子系统的 API 的平均什么我们的 API 的错误率，就是 SLI 的这个错误率要满足环比下降10%，也就是跟上半年比，我下半年的错误率要下降10%。


这是什么？谷歌的一个相对，我们的 SLO 有相对服务目标，那当然你也可以定义绝对服务目标，比如我的交易子系统，我的所有的 l rate 要保证在 1% 以下，也就是我所有 API 平均来看，我有 99% 的时间都是成功的，只有 1% 的时间有可能出现 4 XX5XX 的问题。


这就是SLO。那很多企业会用什么 SLA 来进行沟通？我对你提供一个SCRO，如果我达不到，我就要什么惩罚我自己，我要奖励你，我要什么补偿你？比如什么几百万作为一个惩罚基金。那这种企业文化在谷歌里面是几乎是很难盛行的，除非你是买谷歌云的服务。那谷歌作为一个vendor，一个供应商，它会对你提供SLA，但是谷歌公司内部并不是 SLA 盛行的，它用另外一个概念来代替SLA，同样能保证我们 SLO 是可控的。
