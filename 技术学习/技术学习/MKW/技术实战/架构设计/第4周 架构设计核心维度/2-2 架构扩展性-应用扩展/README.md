它不需要什么依赖于本地硬盘的数据来存活，我很方便的把我的这个服务器做一个镜像，然后通过这个快速启动的方式起很多个不同的服务器。我的 CPU 内存很容易进行扩展，如果是Docker，那更简单了， Docker 我只要做一个image，然后我瞬间就可以起成千上万个容器来进行业务的承载。


那如何把业务的压力压到这些节点上？需要什么负载均衡器的配合？它可以是硬件负载均衡器，比如像F5A10，也可以是软件负载均衡器，比如像engine，或者是混编的，像是 Kubernetes 的Ingress，或者是 load balance。除此以外，负载均衡我们可以选用很多不同的策略，比如说Random，round，Robin，list， connection 等等，只要它能够相对均衡的把业务的压力压到后面扩展出来的很多的节点和容器上，就能实现负载分流了。


讲了无状态应用好容易，有状态怎么办？有状态有时候我们也有一些小小的方法，比如我们把状态里面的一些数据剥离到后台服务，可以变成后台的数据库，后台的我们的这个缓存键值对，比如像session，假设我要记录当前我的用户登录的这个页面里面贴的表单的内容，我可以把这个表单里面当前内容寄到后台的 Redis 的键值队作为一个缓存集群，这时候我前面的应用服器是不是就不用记录状态了？还有一种偷懒的方法，就是什么把一些状态扔到用户的 local storage，落到它的 cookie 里面，这样用户的浏览器里面记录了它的状态，也不需要我服务器去记录很多的状态了。


除了这些以外，大家也许会问说，假设我这个服务器要跑一个cronjob，那 cronjob 是不是要依赖于本地的什么 Cron Tab 文件？那也很简单，我可以把 CROWN job 里面的任务调度剥离出去，使我的应用只提供一个API，当时间到的时候，任务调度服务器来触发我的API，是不是就实现 cronjob 的任务分离？同样道理，有的小伙伴会说我的应用要写日志，我的 Vivlog 底下的日志怎么办？你不要写本地日志了吗？起到什么标准输出，然后让远端呢？我们的监控服务器去收集日志流，是不是也就是实现日志的无状态了，通过形形色色的方法，大部分的应用都可以变成无状态。一旦变成无状态，就可以利用 x 轴横向克隆。


x 轴横向克隆有很多优势，我们看看它如何打的一手好牌。它的第一张牌叫做弹性扩缩容，那扩容或者说是扩展性，说的是长期的一个能力。扩缩容是收缩性，说的是一个短期的能力。当短期业务压力大的时候，它可以快速扩容。当短期业务压力小的时候，它可以快速缩容，它也能适合短期需求。


除此以外，原来性能规划是很复杂的，每次要跑一个新的应用上升场，我们要准备一个类似的环境，上百台称之于上千台服务器。但是一旦我们学会了 x 轴横向克隆，很简单，假设我们要起一个什么以 10 万个 QPS 的这个应用服务器，在我们的嗯，准生产环境，先起一台服务器测一下，跑下来呦，效果不好。我这台服务器只能跑 1000 个QPS。那没关系，我做一个除法， 10 万除以1000，我知道什么生产系统，我要配 100 个节点，基本上就能承载这个线性的压力了。是不是很容易做一些算术好，这是对于性能规划的好处。


第三点是什么？是业务解耦？原来业务可能要考虑我是支付业务还是用户登录业务，可能要不同的扩展方式。对于 x 轴横向克隆没有这么复杂，不管你什么业务，原来你一个业务需要一个服务器就能承载。一旦有你的业务，未来要扩展 10 倍，我就买 10 台服务器，必然就能搞定了，是不是跟业务没有关系？那最后一点就是环境同构一般性，很多的这个同事都会抱怨说我开发环境测的好好的，我上生产就败了，什么原因？参数不一样，对吧？服务器数量不一样， CPU 内存不一样，而我们一个采用 x 横向克隆，我们就采用相同的一台服务器，生产上是1万台，我们开发测试环境就起一台，所以使得什么你的 GVM 参数也是一样的，你的这个应用的特征也是一样的，甚至你的配置文件都差不多。唯一区别就是数量，那是1万台，这是一台，这个环境还是相对容易同构的，这是 x 轴横向克隆。

我们不是用一个服务器做完整的整个主页的前端页面渲染，而是拆成登录页面、物流页面、交易页面、支付页面，不同的 UI 而访问不同的页面，每个不同的页面应用部署在不同的服务器上，这就是前端应用的 y 轴分割。


那后端也很简单，后端可以拆分成子系统，子系统还可以拆分成模块，模块还可以根据什么业务的这个交易一致性拆分成聚合，然后让不同的聚合或者模块运行在不同的容器，不同的虚机里面，是不是就能实现应用的解耦了，同时也很容易进行快速的 y 轴扩展。大家要注意， y 轴扩展不光扩展你的应用，也通常扩展你的数据，所以数据也要做 y 轴扩展。这个我会在下节重点聊怎么样配合应用做数据的 y 轴扩展。 y 轴扩展是打了好牌还是烂牌呢？好忧参半。好处是什么？好处是 y 轴分割服务不干扰。假设是两个服务，一个是用户登录服务，你有独立的登录应用，有独立的登录服务器，也有独立的登录数据库。那同时你还有个支付服务，你有独立的支付应用、独立的支付服务器和独立的支付数据库。当你整个支付服务崩溃了，不管你是代码写的太糟，还是服务器崩溃，这时候你的登录是不是完全不受影响？这就是 y 轴分割的服务，不影响特性。除此以外，资源是可以迭代分配的，特别是很多的小企业。


上云以后怎么样？我先写一个简单的什么交易服务，这时候是不是我只要起大概 4- 5 台服务器，跑 4- 5 个应用就可以了？后面我再开发一个购物车，我把购物车部署在新的服务器上，我再开发一个腾讯对接的支付交易系统。好，我部署在这个新的服务器上。我再开发一个支付宝对接的交易系统，我又部署在一个新的服务器上。是不是每次有新的需求来了，我代码迭代，我的资源也进行迭代，这就是迭代资源分配。
除此以外， x 轴。什么是不能存数据的？ y 轴没关系， y 轴可以存一些缓存，可以存一些本地数据，而且可以实现数据的一致性。因为相同的业务都把所有的数据汇聚在相同的服务器上了，所以数据一致性还是可以得到保证的好。最后一点就是它的坏牌了， y 轴就是因为跟应用太紧耦合了，所以导致它是应用业务耦合性。所谓业务应用耦合性就是原来扩展性，通常是有什么是用 SRE 或者是系统管理员来负责的，但是一旦碰到 y 轴分割，你就需要跟业务人员相关了，也需要开发来参与，因为每一个不同的应用你就要考虑。

好，我们看一看什么是大娃？大 y 就是 y 轴扩展，通常是最核心的一个扩展方法。所谓 y 轴就是把应用分成什么，比如分成用户登录，分成产品，分成这个物流三个不同的模块，根据这个业务的模块切分，是不是就是我们的 y 轴服务切分原理？切分完了 y 轴以后就开始进行 z 轴的扩展， z 又怎么扩展？把所有用户里面从 a 到 n 的我放到一套服务器上，把另外用户名从 o 到 z 的我放到另外一套用户上面，是不是这就是 z 轴的不同？根据用户 ID 的不同？好， z 柱扩展完以后，下面轮到什么了？大家应该想到什么？ x 轴了？是不是我们骑 4 台完全一样的服务器，专门去承载用户 a 到 n 的服务，当然也可以是 40 台、 400 台、4万台、 4000 万台，对不对？这么多台不同的服务器或者容器承载你的服务，是不是能够很抗压，同时我们也很方便上把 a 跟 n 的业务值扔到你这套服务器上，而不会去干扰 o 到z。


好，我们把左边这张图补完，这就是 x 跟 z 配合起来的，完成了什么用户这个领域的什么发布，中间那个产品领域怎么样？一样道理。我们把产品 ID 从 a 到n，然后用 x 轴理论发布到 4 台服务器上， y 轴理论切分成什么？ o 跟 z 放到另外 4 台服务器上。最后如果是库存模块一样，按照库存什么 SKU 的 ID 的编号，把 a 到 n 发配到一个服务器上，把 o 到 z 发配到另外一个服务器上。那这里的 a 是什么？比如那个方块里面的 a 表示代码， a 方块里面的 b 表示我们的弓箭， b 方块里面的 c 就表示我们应用。


c 是不同的应用存在不同的 y 轴的压力，而通过不同的用户 ID 产品 ID 库存 ID 进行 z 轴的切割，最后无脑的 x 轴横向扩展几百几千几万台服务器。好，这就是我们最常见的什么生活当中的应用的横向的纵向的 z 项的扩容和聊完了这个应用的套娃，后面就是什么数据的扩展章节，大家敬请期待。
