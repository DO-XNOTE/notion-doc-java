其实它的成本大家可以猜一猜会是什么样子？我们可以看一下数据中心 a 和数据中心 b 实现了一个双火，两个数据中心都能共同承载业务，如果有一个数据中心发生火灾会怎么样？完全由数据中心 b 来承载压力，所以数据中心 b 的资源配备必须能完整承载起所有的业务需求，所以它是一个 100% 满配的服务器存储网络单元，那同样的数据中心 a 也需要进行满配才能保证。


如果 b 出故障的时候， a 能完全接管所有的业务压力，那这样的情况下，我们算一算成本， 100 + 100 就是200，我们足足花费了两倍的资源来进行什么数据中心的准备。即使我们采用了双火，平时负载均衡，但是一旦发生灾难，我们为了防止这个灾难的发生，仍然要准备两倍的资源。

带着这样的疑问，我们来看一看多活 a 数据中心 a 到b，这是双火，然后他们同步的复制到c，使得 ABC 是一个对等的单元，这就是多活。如果我们实现了多活，我们猜一猜，还是发生火灾是吧？这会怎么样？我们让 b 和 c 什么呢？承载所有的客户压力就可以了。所以我们 b 是不是要准备 5% 的资源， c 要准备 50% 的资源，两个 50% 加起来刚好能满足业务的所有需求。那同样道理，平时 a 是不是也要准备 50% 的资源？这样即使 b 出问题，我们也能让 a 和 c 承载所有的压力。


需求单处理中心故障，在多活里面就是通过这样的方案来实现。我们什么成本的估算？如何估算？三活数据中心需要百分之五十乘以3，所以需要准备 150% 的资源，是不是比双活好像省了点钱？那如果 n 活怎么办？公式就是一除以括号 n 减一乘以n，如果 n 是3，那就是刚好是150，如果 n 趋向无穷大，那最后的结果就是 100% 的资源，所以多火反而省钱。


没有想到当我们搭建更多的数据中心的时候，如果只从服务器，从机柜，从存储，从内存来考虑，绝对是省钱的。但是如果你要考虑一些额外的，比如说我们的网络连接数，网络的价格，这个时候根据你网络传输的压力，根据你占用带宽的问题，也许价格会贵。所以通常我们多活里面 n 是什么？ 3- 8 之间是一个经常的情况，我们很少有 8 活以上的系统，那这样系统网络开销太高，而三活以下双活就太传统了，我们互联网公司是不是至少要保证三活？好，我们三八活聊完以后，我们看一个实际案例，

单元化的核心，我以导航企业为例，这是一家导航企业，它在对外的网站叫 Passport 点 AMAP . com，然后它导航到了三个set，这里三个虚框就是它的三个单元。注意，单元化里面不管怎么实现，通常都不会违背这样三条关键原则。第一条关键原则就是控制好路由，这个路由可以是什么随机分配，就是根据你的用户的ID，比如我把你发配到什么单元一，那你这个 ID 通常就会后续的所有任务都会在单元一。也有可能根据你的地理位置，像导航企业就选择地理位置，你经常是在说上海的浦东新区来进行什么，我们进去打车的，那你呢？导航的业务压链就会打到离浦东新区最近的那个单元，专门来处理相对智慧区域的这个单元压力就是根据地理位置，除此以外还有各种各样的什么哈希、曲墨，各种各样的处理方法，这就是我们的什么 z 轴分布方法来达到不同的单元。


那除了这个路由选路以外，还有一个很核心的内容就是单元封闭，一旦这个压力压到这个单元里面，通常每个单元之间有些相似性，比如你也有 Diamond 节点，我也有 Diamond 节点，你也有Redis，我也有Redis。所以当同一个单元进行交易的时候，尽量采用同一个单元内部的已经有了业务服务模块、中间件模块、数据库模块，不要把数据流再导流出去，这样会导致什么我们的业务就变得慢很多。所以单元封闭是在单元路由里面实现，通过好几种机制可以实现单元封闭，所以都会在 set 章节详细跟大家分享。


而第三个技术是最难的技术，就是数据同步，即使是你尽量都是单元封闭的，其实你的数据尽量都导流在一个单元里面，但是当你单元崩溃的时候，你仍然需要高可用，所以你的数据虽然不用什么强一致性，但需要最终一致性导流出去。怎么样导流更好？通常一个企业会采用多方导流，比如这里，它的阿里 SQL 的数据库进行了数据库导流，它的tail，我的缓存键值对，通过了 tail 导流，它自建的 Uralis 通过了有 Redis 的导流机制。不同的复制机制和导流机制能保证什么？我的数据有不同的延时，不同的这个最终一致性，但最终总是能够把数据以多副本多备份的形式发布到其他的单元，使得我为主单元，其他单元依然可以读到我上面的数据，虽然中间可能有小量的延时。这种数据同步机制不光是在这个导航企业，也不光是在用 set 的，比如像携程、微博、阿里面广泛使用。其实我们连可以拿银行做例子，我们要举个例子，就是宇宙第一大行，它虽然不号称自己叫 set 化，它叫多数据中心化，是不是很类似？那它怎么导流？以它的这个主中心的两个节点为例，它的主中心在同一个城市，两个数据机房，那这两个机房之间既通过底层的存储导流，也通过数据库导流，还通过中间件导流。


另外操作系统里面还有一些额外的监控和管理，使这三个导流能互相配合，是不是和这个架构图如此之接近？不管是宇宙第一大航还是我们的这家导航企业，只要你实现 set 化，实现多火，实现多中心化，整体思想都类似，数据需要导流单元尽量封闭，或者说数据中心内部尽量封闭路由，选择合适的路由请求策略，使得我们的业务能够尽量快速的满足用户的需求。
