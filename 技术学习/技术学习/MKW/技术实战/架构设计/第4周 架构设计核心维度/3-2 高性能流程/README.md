设计一套思路和目标，这个目标我外呼就是要来支撑一定的业务，包含像一定的UV，每天的这个用户访问数能够达到什么样的程度？能支撑一定的PV，每天的这个页面的浏览情况能达到一定的量，然后要支撑一定的业务量，除此以外， CPU 内存磁盘网络它的利用率要保持在一定的这个程度之下。另外我们还要用支撑一定的TPS，一定的QPS，能支撑一定的 session 和一定的用户的并发数。


这些业务的系统的性能的目标都是我们要在容量规划里面要提出的这个点。那除此以外，我们要选择一个合适的容量的规划的方法论来创建一些基线，同时对不同的场景定义一些水位。那这个具体怎么来实现？诶，我这里卖了一个小关子，留待后面范先老师的架构设计与评估来给大家仔细分析，如果有了目标，如果选择了合适的规划方法，是不是下一步就是执行？执行其实非常简单，无外乎就是我要施压进行压测，施完压以后我要做监控，来去看一看我实际测完的这个情况下，我的系统到底出现了什么样的情况，有没有出现拐点。然后我根据我测试的报告来预测短期内我应该什么？我要调整系统还是应该改变代码，长期内我如何去支撑像618、像双 11 这样的热卖情况？好，那我们逐一来看一看测试压力跟监控这两个环节。

它是用正常的压力来压我们的系统，所谓正常的压力就是我们在平均的每日或者平均的每个月的峰值，我们挑一个这个值作为我们长期的这个压测的这个负载，我们通常不会去挑。像618、双 10 一这种特殊情况，我们挑一个常见的压力的相对高值，在这个高值情况下，我们要去看一看 99. 9% 的场景的情况下，我们的机器的 CPU 内存，它的实际的利用率是什么样？它的 API 的响应时间是如何？它是不是足够稳定？所谓足够稳定就是指我的这个每次测的 CPU 的利用率变化可能差距在 5% 之内，今天测试 60% 的利用率，那明天可能是62，后天可能是58，不会出现今天20%，明天40%，后天 60% 这样波动的情况。同时向内存，不存在内存溢出，不存在一些系统的这个磁盘的这个使用率忽高忽低的情况，这些情况都是什么？都是需要回避的。


通常什么或者是我们的测试案例不精准，或者是我们的代码有bug，或者是我们的系统的稳定性有问题。好，有了这样的测试目标，还要有一些测试手段，这里面有七大手段，我简称叫七大姑，八大姨的七大姑。通常是你先拿历史作为基准，设定一个基本的这个性能测试的基准，然后你要挑选你的环境，跟生产越接近越好。如果你实在是没法挑选一个跟生产很接近的环境，索性你创建一个准生产，不管你是有多少个部门，你有多少个 QA 环境，你都可以共享那一个准生产来去跟生产更加接近。


而第三个大姑就是什么定义？你的测试案例要能保证我的测试的最终结果是相对稳定的。第四大的服务就是我们的执行要自动化。第五大故就是我们的整个分析都采用统计学的方法，做出比较精准的判断和预测。那第六点就是所有的报告不光要通知到人，还要录入到系统。第七点最关键我的整个测试过程是反复迭代，不停往复的。它的整个测试思想是什么？是负载测试，是一个长期的、永久的过程，当你开始测试以后，你的 QA 环境就一直在运行这样的负载，你的准生产也在不停的跑不同的测试，甚至于你的生产上也在什么有一定的压力，这些压力来自于你的测试，来自于你一些特殊的测试场景。

(

QA: Quality Assurance，**品质保证**

QA指的是确保软件产品符合规定质量标准的过程。这通常包括软件测试、代码审查、自动化测试、性能测试等活动，以确保软件的稳定性、可靠性、性能和安全性。

)


通过这种长期的负载测试，能保证我整个系统大部分的功能得到验证，大部分的什么稳定性得到保证，同时我的性能永远是可预测，当我发现我的 CPU 在不停的测试过程当中永远在增长，这个时候我可以预估到某一天我的 CPU 会崩溃。那这是通常是代码的问题，或者是机器老旧的问题，可以通过调整机器，调整代码来进行改变。好，这是一个常规的长期的负载测试。


那另外我刚刚也聊到了第二种测试，就是压力测试。

那第一境界是什么？叫一代渐宽终不悔，就是我们的性能监控，不管你胖瘦了也好，我都要把这个数据实实在在的记录下来。通常可以采用类似于普罗米修斯那种就时序数据记录，然后在时间维度，在一些这个键的维度上面进行数据的聚合，聚合完了以后能产生出大量性能指标的报告，包含像延时、CPU、内存等性能指标，那最后这些报告还能产生一定的报警。那这就是第一境界，一代渐宽终不混性能监控。
