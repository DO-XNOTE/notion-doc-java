整个标准从总体架构、敏捷开发管理、持续交付、技术运营、应用设计、风险管理、组织架构 7 个维度去衡量一个企业或者团队的 Devops 发展程度。通过标准，同学们应该至少有两点收益。一来，可以拿标准去对照，看看你的项目在这么几个维度， Devops 发展到了什么程度，哪些方面是值得改进的。


二来，如果你正好负责你们企业的 Devops 落地，也可以去参考标准，逐步的从某一个或者某一些方面作为切入点，并逐步深入，不断提升团队的 Devops 能力。这一段文档里面的内容 Devops 成熟度模型把持续集成这件事情划分成了 5 个等级，你可以看一看你的团队目前持续集成达到了什么层次，并去思考未来你们公司持续学生怎么样往更高的层次去演进。最后强调一下，希望同学们能够理解大木的意图。


我介绍还没有最终定版的成熟度模型，并不是希望大家以后去做什么 Devops 成熟度认证，我估计不久的将来，市面上就会有这种认证了。这种认证做或者不做大不觉得无关紧要。我贴标准是希望同学们能够参考文档的内容，看看自己团队在 Devops 方面的发展程度，有哪些方面是值得改进的。这样在持续改进的过程中，项目交付的速度、迭代的速度也都会更加的高效经济，这才是大木的本意。另外，
