他说 Devops 是 development 和 operations 的组合词，也就是开发和运营。它是一组过程方法与系统的统称，用于促进开发、技术、运营和质量保障部门之间的沟通、协作与整合。它是一种重视 DEV 和 OPS 之间沟通合作的文化运动或惯例。通过自动化软件交付和架构变更的流程来使得构建、测试和发布软件能够更加的快捷、频繁和可靠。

我个人认为 Devops 它具备这几个特性首先， Devops 强调人员、流程以及工具的协同。

第二， Devops 强调践行敏捷和精益，减少浪费。

第3， Devops 强调加快产品以及服务的交付速度。

第4， Devops 强调快速提供给用户需要的功能，并快速接受反馈。

大致上是 Devops 强调高效组织团队，并通过自动化工具协作与沟通，从而实现用更少的浪费、更频繁地交付更稳定的软件。其中，高效组织团队以及自动化的工具是途径，而更少的浪费、更频繁、更稳定则是结果。


你可以把 Devops 看成是开发、技术、运营以及质量保障这三者的交集。

前面说过 Devops，它是开发和运营的组合词。而在传统的写作模式下， DEV 它需要对开发负责，侧重于如何更快地实现业务需求。而 ops 则是对运营负责，侧重于如何提高软件的稳定性以及可靠性上。dev 和 ops 两者之间的分工以及目标不同，有时候就会导致利益冲突。开发它是由功能性需要来驱动的。所谓的功能性需要，一般指的就是和业务需求直接相关的那些需求。而运营则是由非工人新需求来驱动的，比方系统的稳定性、可靠性能等等。


所以开发人员他希望快速的开发更多的功能，并上线到生产环境。而运营人员则希望尽量避免掉功能变更，或者不要去经常变更，从而降低运营人员满足非功能引进需求的风险。而在某一个特定的时间段内，比方半年需要做的变更总量是固定的，而运营他又希望不要去经常变更，于是就会导致每次变更的规模会非常大。而变更的规模越大，设计的面也会越大，于是风险也会越高。


举个例子，业务上提了 n 多个需求，本身开发 2 个新功能就可以发布到线上给客户去使用了。但是运营人员他说才开发两个新需求，就要重新发布，这样会影响我的软件稳定性和可靠性，不妨再积累积累。于是一直积累到 10 个功能才发布。这样一来就延缓了开发人员将功能交付给用户的速度。

首先Devops，它提倡更加频繁的发布，作为一个功能，需求从提出到上线更加的迅速，能够更快响应客户的需求。每次发布包含的变更会很少，所以风险也更小。还用刚刚的例子，采用 Devops 之后，可能每一个功能做完之后就立即上线了，一个星期上线一次，或者两个星期上线一次，每次上线修改的东西也都非常的少，所以风险也会更小一些。


第二，Devops，它鼓励自动化，通过强大的自动化机制，能够保证任务的可重复性，同时也能够减少部署出错的可能性。在 Devops 圈子有非常多的工具，这些工具在稍后我们会详细探讨。


第三，Devops，它弥补了开发和运营之间的鸿沟，能够让开发和运营朝着相同的目标前进。OK，这是 Devops 带来的具体价值，也是 Devops 被越来越多企业采用的原因。好，这节课就到这里，谢谢大家。
