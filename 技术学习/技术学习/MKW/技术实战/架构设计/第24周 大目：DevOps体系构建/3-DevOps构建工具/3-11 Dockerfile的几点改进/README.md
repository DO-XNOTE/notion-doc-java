怎么样构建更加的方便。二是如何让你的 dockfile 更加的通用。比方你有 10 个 spring 步的项目，按照上节课的玩法，我们需要为每个项目编写独一无二的 Doc file，怎么样让 Doc file 的内容能够适应所有的项目。三是如何构建优化的镜像。所谓优化的镜像，包括了构建速度的改进以及应用性能的改进。先来看到第一点构建过程的改进。上节课我们需要先对应用进行打包，再把价包拷贝出来之后才能编写 Doc file 并去构建。整个过程还是挺麻烦的。怎么样改进这个过程？只要把 Docker file 挪一个位置，放到项目的根目录咖啡粘贴，根据修改 dockfier 的内容，把 Eurika 叉叉叉点价改成 target/ 叉叉叉点价就 OK 了。也就是利用 add 指令能够添加相对路径的特性去简化构建过程。

比方这里执行 add 指令会做成一个层，cmd 指令也会做成一个层。每个层你可以把它想象成是一个快照。通过这种方式可以实现层的复用。比方我们要构建另外一个镜像，这个镜像恰好也要执行相同的指令，添加相同的文件，这一层就可以复用。

写得非常详细。而 fat jar 会把我们自己编写的代码和项目的依赖放在同一个层，因为这里只有一个 add 指令。这样就会导致不管是依赖发生变化，还是你自己的代码发生变化，只要 feed 价发生修改，整个层都会变化，这一层就没有办法复用。但事实上，很多时候，可能我们只是修改了自己的代码或者是配置，项目的依赖并没有发生变化。你说在开发的过程中，你会天天去修改 pom.xml 吗？相对来说还是一个低频操作对吧？除非我们需要升级 Springboot 版本，或者需要整合新的框架的时候，才需要去做修改。所以我们可以改进一下。


比方把项目的依赖做成一个层，我们的代码和配置做成一个层。这样一来，只要依赖没有发生变化，依赖对应的那一层就可以被复用了，构建的速度也会得到提升。继续看这段话，他说重新编译的频率要比升级 spring boot 版本的频率要高。所以最好还是把你自己的代码和项目的依赖分开。这里的 jar files 可以把它理解成项目的依赖，放到 application classes，也就是我们自己编写的代码和配置这一层之前。这是什么意思？是我们可以 add 项目的依赖，
