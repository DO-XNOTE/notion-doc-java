这个话题我专门写了一篇手记，一起来看一看。**前面我们是用 kubectl 去管理项目的，这种方式相对于传统的部署方式来说，虽然已经比较方便了，但是依然不够方便，比方现在有一个复杂的项目需要用到。**

Deployment, service, secret, pv, PVC 等等各种资源，这些资源会分散在不同的文件里面，这个时候用 **kubectl** 去管理很不方便，我们会面临各种各样的问题。 而 helm 则是一个 Kubernetes 的软件包管理工具，类似于 ubuntu 里面的apt 或者 centos 里面的 yum。大家使用 yum 去安装软件的时候是不是特别方便？ yum install 就搞定一切对不对？用 helm 的也是类似的，它能够快速的查找下载和安装软件包。

说得更加通俗一点， helm 就是 K8S 世界里面进一步简化应用交付的一个工具。它的官方文档在这里，有非常良好的中文支持。

[https://helm.sh/zh/](https://helm.sh/zh/)

课上我们围绕目前最新的 helm 3.5.4去探讨。需要注意的是，不同版本的 helm 不管在架构层面还是使用层面都是有差异的。比方 Hem3 和Hem2，它有 Tyler 组件，而 Hem 3 根本就没有这个组件。我这篇文章能够兼容 Helm 3 点 x 版本，但是不兼容 Helm 2 点 x 版本。同学们今后阅读 Helm 相关的文档，一定要注意一下版本的问题。来看一下 Hem 的核心概念。


1 是 Chart: Chart 是 Helm 的软件包，类似于 yum 的 rpm  包，或者是 apt 的 deb 包，里面包含了一组定义 kubernates 资源相关的 YAML 文件。


二是repository。这是 helm 的软件仓库，本质上它是一个 web 服务器，里面存储了一系列的chart，相当于 yum 或者是 apt 的源。

三是 release 是 chart 的一个实例，一个 chat 可以多次安装，每次安装都会创建一个release。这三个概念还是比较简单的。

用一句话去概括 helm 将 chart 安装到 K8S 集群里面，每次安装会创建一个 release。** repository 则是一个存放恰当的地方。**

