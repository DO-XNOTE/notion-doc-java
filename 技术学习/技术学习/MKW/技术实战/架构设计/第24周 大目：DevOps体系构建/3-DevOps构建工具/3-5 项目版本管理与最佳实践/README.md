首先，版本如何命名这个话题，在技术选型阶段的如何选择技术栈的版本。这一节里面已经介绍了多种版本命名的方式，以及不同版本号所表达的含义。简单复记一下。流行的版本命名方式主要有两种

一是语义化版本格式，比如 1.2.3 点release。其中 

1 是主版本，表示第几代产品，数字不同，往往代表架构上的升级是不兼容的。

2是次版本，表示上下兼容的功能性新增

3是增量版本，表示上下兼容的 bug 修复，意思则是里程碑，表示版本的状态。

以 0 开头的版本也叫孵化版本，它表达的含义一般是低于 1.x 的版本都是正在孵化的版本，这种版本不建议直接用来生产，等它孵化完成之后，它会以 1.x 去命名。但是也不一定，有的项目的 0.x 版本也可以用在生产。一切看项目的规划，看它的项目 0.x 是孵化版本还是正式版本。

当项目发布完成之后，可以再做一下规划，如果下一个版本

假设是想做 bug 修复，就可以把版本晋升成一点 1.0.1-snapshot。

而如果下个版本我们是想做新功能的叠加，就可以晋升成 1.1.0-snapshot。

而如果想做架构上的升级，就可以升级成2.0.0-snapshot，并去继续开发，开发之后也是以此类推。 

从 snapshot 升级成 release ， 这是项目的版本。如何规划这节课就到这里，谢谢大家。
