来看第四步讨论并 review 你的代码。他说团队的相关成员可以去打开 per request，看一下 per request 里面包含了哪些东西去进行评审？如果对代码有疑问，或者发现有问题，比方代码的风格不符合标准，缺少单元测试等等，都可以提出来提。直接在 pull request 里面。据说这个代码没有单元测试，请补充。

最后，当部署完成，经过验证之后， pull request 的里面代码确实没有问题，就可以合并回 master 分支了。点击 pull request 合并请求，就可以合并代码。当合并完成之后，分支就可以删掉了。可以看到代码已经合并到 master 分支了。从中可以发现， GitHub flow 是非常简单的，它只有一个长期分支master。同时它的部署发生在代码合并之前。这也是很有意思的。为什么要这么设计？之所以这么设计，是为了在产生任何副作用之前，就已经知道当前的修改所带来的集成效果。比方经过部署之后发现刚刚的 sum branch 分支有问题。最严重的后果无非是把这个分支直接丢弃。不管怎么样，它都不会影响到 master 分支。而合并到 master 分支的代码都是稳定的，因为经过了部署的验证才会合并。事实上， master 上面的代码其实总是你线上部署的最新代码。这句话有点绕口，稍微的解释一下，大家想 some branch 部署到线上了，然后才会合并到master，是不是线上的代码和 master 是一致的？ Poly 另外一个分支，比方 sump range 2 部署了，发布到了线上，然后也合并到了master，是不是 master 的代码和 some branch 2 也保持了一致？所以我们说 master 上面的代码其实总是线上部署的最新代码。这是 get up flow。 get up flow 非常适合一些开源项目以及发布特别频繁的项目，比方一天就会开发好几个feature，发布好几次的那种项目。这节课就到这里，谢谢大家。
