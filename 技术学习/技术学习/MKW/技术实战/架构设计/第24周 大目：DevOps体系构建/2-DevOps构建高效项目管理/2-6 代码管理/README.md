业界有很多的版本控制工具，比方 Git, SVN,HG, CVS, RCS  等等。
这里有一篇文章，里面探讨了为什么要使用版本控制工具，以及各种版本控制工具的特点和相关的资源。虽然工具很多，但是就弹幕的经验来说，目前国内主要的版本控制工具是 SVN 以及Git。前几年基本是 SVN 和 Git 二分天下，而现在 Git 已经占据了主导地位。如果你的项目还没有使用Git，建议迁移的文档。在这里。

[https://git-scm.com/book/zh/v2/Git-与其他系统-迁移到-Git](https://git-scm.com/book/zh/v2/Git-与其他系统-迁移到-Git)


里面探讨了如何从其他版本控制工具迁移到 Git 的说明。比方从subversion，也就是 SN 迁移到Git，从 hg 迁移到 Git 等等，只要对照操作，就可以无缝地从其他版本控制工具迁移到 Git 了。所谓的无缝，指的是能够保留提交记录，并能够保留原有系统的账号。
这篇文档写的是非常详细的，如果同学们有这样的需求，只要对照操作就 OK 了。但需要注意，如果你所在的公司团队非常的大，项目比较多，如果按照这篇文档，针对每一个项目做一次迁移，还是比较麻烦的。特别是涉及到多个项目合作的情况，阻力可能会比较大。比方有一个超大的项目，整个项目有 500 号人在写作。这个项目也分成了若干多的模块，不同的模块使用了不同的版本控制工具。比方一部分用的是SVN，一部分用的是 hg，还有一部分用的是git。


**有一天你想做迁移了，有一部分人他用的 hg。他可能会说为什么要迁移？因为他可能已经适应了用 hg 的情况。为了减少这种阻力，你应该怎么办？我个人建议同学们可以把整个迁移的过程。比方 SVN 迁移到 Git 的过程写着一个脚本，并提供一个界面给各个项目的人员可视化操作。比方，在一个界面上，让负责迁移项目的人只要填入svn 的地址，以及迁移到的 Git 地址，还有 Git 用户的列表。最后鼠标点击提交由脚本自动化执行迁移的过程，从而实现项目的迁移。****这样一来可以提升大家迁移到 Git 的积极性，二来也可以减少迁移时遇到的问题**。因为有的人他可能并不熟悉Git，你让他去一个输入命令还是可能会出问题的。这是如何从其他的版本控制工具迁移到Git？

这是如何快速找到 idea 的快捷键的几种方式。讲到这里，有同学可能要说了，这节课的内容是不是有点 low 了？你管我是什么流派，能达到目的不就行了？而且还花了几分钟的时间去讲 idea 的快捷键，有必要吗？这里解释一下意图。大家想，你在提交的时候，如果每次用命令行去提交，你得怎么办？你首先得打开一个终端，至少花费 0.5 秒，用一堆的命令去进行提交。比方 get add 点儿。有的时候你还不能 get add 点儿，你得挨个的去添加你想要提交的文件。 Git commit 杠 m 写上注释 Git push 杠 u origin，跟上加提交的分支master。那这么一堆命令抄下来，小说得花费 30 秒了。复杂的情况下，你可能还需要合并代码什么的，花费的时间会远超 30 秒。而如果用idea，只需要输入快捷键，勾选、写注释、点按钮，整个过程只需要 5 秒钟。我们保守估计每次提交能节约 25 秒。每天你的代码至少要提交 5 次，这样一天就能够节约 125 秒，也就是 2 分钟多一点。


有同学可能要说了，两分钟能干什么无所谓，但是一年有 250 个工作日左右，计算一下，一年就能够至少节约 500 分钟。算下来是 8 个多小时，妥妥的一个工作日。这还是保守估计，实际结业的时间应该会更多。比方你要合并代码什么？其实大木通过这个例子是想同学们在成长的过程中也需要注意一下，**效率其实也是一种能力**。比方有两个小伙伴，一个小伙伴他是达到目的就 OK 了。而另外一个小伙伴则更加的追求极致。于是，追求极致的小伙伴，他会想办法去节约自己的时间。同理，在编写代码这件事上，追求极致的同学，可能每天又能够节约 10 分钟，这绝对不夸张。根据 Netflix 的研究，好的程序员，他能够创造的价值是普通程序员的 10 倍。


节约 10 分钟真的是小 case 了。问题来了，在编写代码这件事上，追求极致的同学，他一年能够节省出多少时间去提升自己？**希望大家要养成良好的习惯，尽量尝试用最合适的工具，最效率的方式做事情。**当然了，课上介绍的用 idea 去操作 Git 只是一个例子，并不一定适合所有的同学。如果同学们觉得使用其他的流派对你来说是最有效率的，你可以去继续使用，这样长此以往，你的成长也会更加的快。小处也不可随便，这才是大木的目的。希望能够给同事们带来启发。OK，这节课就到这里，谢谢大家。
