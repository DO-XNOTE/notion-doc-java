大致过程是这样的首先，由产品经理创建需求，不同的企业有不同的分工。需求也可能是由需求美媒，甚至是由开发直接去创建的。之后，开发团队指派，或者由开发自主认领需求，并在 red man 上完成指派。当创建需求的时候，也可以直接进行指派，看项目的情况。


接着，开发人员开发某一个需求的时候，就把需求的状态改成处理中，当需求开发完成之后，就改成已解决。最后，当需求验收通过之后，就把状态改成已关闭。演示一下这里我们使用两个用户，一是 product 小李，此小李他是产品经理，负责创建需求。二是 DV 小王，他是开发人员，负责开发需求。由于需要使用两个用户，所以我们打开两个浏览器 Chrome 以及Safari，在 Chrome 上面登录。 product 的小礼，他是产品经理，在 suffer 上面登录 DV 小王，他是开发。


首先产品经理小李，他经过需求调研之后，创建了一些需求。点击新建问题需求。需求一写一些描述状态新建优先级。高，这是一个高优先级的需求，指派给小王。你还可以去选择一下这个问题的类别了，不选也是 OK 的。创建这是第一个需求，再创建一个。需求2，描述这是一个重优先级的需求，没有做指派，再创建一个需求3，它是低优先级的需求，也指派给小王，创建。对于需求2，在创建的时候并没有去指派，可以由团队的项目经理去指派，或者是由某一个开发人员自主认领。点击右边的编辑，我们也指派给小王，现在只有小王是开发。之后小王他点击我的工作台，他一看我现在名下有 3 个任务了，我就看一看。我先做需求1，进入了需求 1 的开发。与此同时，他需要点击编辑，把状态改成处理中。当他把需求开发完成之后，就把状态改成已解决。最后有一天产品经理他验收了之后发现没有问题，于是就把状态改成关闭，这一个需求的生命周期就完成了。


此外，如果你的公司对工时有考核，还可以在上面录入工时。比方在产品经理创建需求的时候，就可以写一下需求预计需要多少的工作量。预计时间也不一定是由产品经理去估算的，而是由开发人员估算的。开发人员也可以去编辑并去修改这里的预计时间。当然还可以设置需求的开始日期以及计划完成日期等的，都可以提交。需要注意，这里的时间是预期时间，也只是一个时间上的评估。而开发人员他做完了需求之后，可以点击这里的登记工时，从而录入实际花费的时间。比方花费了 9 个小时。选择下活动是开发表示开发花费了 9 个小时。目前我们只创建了一个活动，实际项目里面可以创建其他的活动，比方开发、测试、修改bug、 code review、代码重构等等。创建。这样在耗时这一栏就可以看到工时了。不过目前来说，多数的互联网公司其实并不强调工时事情，更多的是结果导向，只要工作完成就 OK 了。所以工时同学们知道一下就 OK 了。OK，这是需求。

当有一天，开发人员将一个迭代的开发任务开发完成之后，就会进行提测，告诉测试人员说你们去测试。这个动作是在线下进行的，其实就是开发和测试打了一个招呼而已。之后测试人员进行测试，并提出项目中的缺陷。提出之后，缺陷的状态是新建，当开发人员修改某一个缺陷的时候，就把缺陷的状态改成处理中。当 bug 修复完成之后，就把缺陷的状态改成已解决。而对于某一个缺陷，开发人员经过验证之后，发现不需要处理。比方设计就是这样的。或者测试他提了一个不是问题的问题，这个时候就可以把它标记成无需解决。最后测试人员进行回归。所谓的回归，就是复测已解决以及无需解决状态的缺陷。如果发现某一个问题确实已经不是问题了，就把状态改成已关闭。而如果发现依然存在问题，就把状态改回处理中，让开发人员重新去修正。当然了，也有一些企业，它会做一个区分，当发现某一个缺陷依然存在问题，就把状态改成Reopen，也重新打开，这样也是可以的。看你们项目的管理习惯。


在理想情况下，对于某一个版本来，当项目达到可发布状态的时候，所有的缺陷都应该变成已关闭。但这是理想状态，实际项目中可能不是这样的，往往会遗留少量无关紧要的缺陷，没有能够解决的了。这个时候可以把缺陷留到下一个版本去解决。演示一下，我们使用两个用户，一是 test 小周，它是一个测试。二是 DV 小王，它是一个开发。有一天，项目提测了，作为测试人员的小周，他进入测试发现有缺陷了，之后就点新建问题缺陷 BUG 1，在描述里面写上复现步骤，也就是。按照这个步骤就能够重现这个 bug 了。这个缺陷非常的严重，所以它的优先级我们把它设成高，指派给小王看见，再创建一个缺陷。


bug 2，复现步骤， 8 个 2 不严重，所以它的优先级是低，也指派给了小王，还能完成。小王他来到我的工作台，发现我现在又有 bug 了。他说我先修改 BUG 1，然后把状态改成了处理中。等到处理完成之后，就把状态改成已解决。测试人员，他就可以去回归了。如果发现问题确实解决了，就改成关闭。如果发现依然有问题，就继续改成处理中。当然了，有一些选项溢出浏览器了，没有能够显示出来这个问题。假设它已经解决了，改成关闭，缺陷的生命周期就完成了。


同理 BUG 2，比方它也是先进入到处理中，他修改完成了，已解决，但是测试员回归了之后发现这个问题没有被解决，于是就继续改成处理中。由于这个问题的优先级非常的低，所以我们并不强调这一个版本一定要把它解决掉。你可以放到下一个版本去解决。我们可以点击配置，创建个新的版本，比方一点 0. 1。


创建这个版本标志着下一个迭代的进行回答问题，把这个问题它的版本改成一点零点一，表示这个问题可以在下一个版本里面去解决。当某一天我们发现当前版本里面所有的缺陷都已经变成关闭，我们就可以去发布了。发布完成之后，可以点击文档或者是 wiki 描述一下这个版本做了哪些事情，改进了什么，修正了哪些 bug 等等。最后再由项目管理员添加一个新的版本，进入新的迭代。刚刚我们已经用测试人员添加了一个新的版本了。在实际项目里面，一般建议由管理员去添加新版本。我们课上为所有的人员都分配了所有的权限，只是为了操作方便。OK。另外，我们还可以让版本库的版本和项目的版本给关联起来。比方当 read man 上面创建一个新的版本之后，我们就在 Git 上面拉一个新的分支，比方也叫一点零点一。这样就可以去对应起来了。不过这个对应 read man 他自己并不管，需要我们自己去控制。


最后还有两个问题需要说明一下。一是点击问题。当我们筛选状态是已关闭的问题的时候，点击应用可以发现没有任何结果。这是为什么？我们需要用管理员账号登录，点击管理问题状态，点击关闭，勾选一下已关闭的问题。这样 read man 才会知道问题状态等于关闭的问题是已经关闭的。从这里我们其实可以看出来， read man 的一个设计关闭只是一个显示，这里的关闭才是真正的状态保存。再次点击应用，可以看到已经能够筛选出两个关闭的问题了。这是第一，我们课上只谈论了问题的状态如何迁移。实际项目里面，你还可以额外使用百分比方80%，表示现在已经处理了 80% 了。当问题已经解决，就可以把它改成100%，这里就变成绿色了。通过百分比可以更加精准地控制问题的进度。这是用 read man 如何管理项目。总的来说，这节课的内容比较简单，之所以去做详细演示，主要是为了给大家一个直观的感受，从而方便从技术转型管理的同学。毕竟当工作年限达到一定的程度之后，会有很多的同学从技术转型成管理。这节课就到这里，谢谢大家。
