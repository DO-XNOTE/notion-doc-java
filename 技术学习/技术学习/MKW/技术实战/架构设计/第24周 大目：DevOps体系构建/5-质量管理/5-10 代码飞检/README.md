代码飞检是华为内部实施的人工保障代码质量的手段，在外界公司其实并不流行，所以这节课并不作为课程的重点，而只是一个扩展。但是大木之前在华为的时候确实从中收获了很多，所以觉得还是有必要拿出来分享一下。 这里弹幕找到一篇华为代码飞简交流的PPT，感兴趣的可以看一下。它原先是上传的百度文档的，但是百度文档它需要 VIP 才能下载，所以我把它上传到了 imooc Git，同学们只要点击一下就可以下载了。


OK，下面来看一下代码分拣是什么玩意儿，该怎么做。 简单来说，代码飞检，它是一种突击性的代码审计手段，它通过突击检查的方式来检查大家是否遵循了代码规范。这里的规范指的是华为代码质量均规。军规是没有办法通过工具自动扫描的，所以需要人工去审查。

这张图画的有点复杂，事实上，弹幕当初所在的团队也并没有完全按照这个图上来。一会来说，为了描述方便，我们把审查别人代码的人称作是飞检人，被审查代码的人称作是被飞检人。这个图画的虽然有点复杂，但是大致过程是这样的。首先，组织代码飞检。组织废茧的时间是随机的，从而达到突击检查的效果。之后通过抽样的方式确定好飞检的范围。接着就进入实施阶段，开始飞检了。飞检的过程中，由飞检人指出一段代码，并让被飞检人讲解，主要是描述一下相关的业务以及代码的设计思路。之后飞检人开始走读代码，并指出存在的问题，最后形成报告并总结弹幕。 



之前所在的团队实施飞检的时候，和这张图总体流程大量是一样的，但是有几点变化。

一是我们的飞检是团队自发组织的，不会有图像这么严格的策划，也不会有邀请专家分组评审方案，还要开会这样的过程。图上的流程相对还是太正式了一些。按照图上来玩，每年实施飞检的次数势必会比较少，毕竟大家都很忙，很难挑出一个时间相关人都有空。这是第一。 

第二，飞检人。让被飞解人讲解一下代码，再去走读代码。除这种玩法以外，我们还经常两两随机成对，相互阅读对方代码，并指出对方的问题。这样做主要是促进团队之间的沟通，同时也让飞检人和被飞检人的关系对等起来，互相玩大家来找茬的游戏。


再一个，在阅读同事代码的过程中，也可以熟悉对方的业务。长此以往，就可以逐步形成 backup 的关系。比方小王负责的一些模块，我正好也阅读过，我就成为了他的backup。即使小王不在，项目也不会停摆，业务也不会阻塞。


再一个，我们飞检的时候，也不会每次都编写审计报告之类的文档，这种文档主要还是用来做汇报的。我们团队更加关注的是采用代码飞减所带来的切实收益。okay，这是代码飞解。


通过讲解不难发现，代码飞解本质上可以认为是另一种形态的 code review。 最后，结合弹幕的切身体会，聊一聊代码飞减带来的收益。


首先，华为是大木带过的第二家公司，那会儿大木刚刚入行，在代码飞检的过程中，经常会找到可以借鉴的代码。这是第一第二，因为当时刚入行，写代码的能力说实话是比较差的。团队的师兄师姐们也都比较照顾我，会帮助我指出代码里面存在的问题。特别是我的一个秃顶师兄，他是团队的写码狂人，当看到我的垃圾代码的时候，有的时候会手把手的指导告诉我怎么样写会更好。从他身上我真的是学到了很多。秃顶的毛病，暂时还没有学到。OK。 


第三，我们当时的项目是一个大型的云计算项目，参与研发的自有员工加上外包可能有 300 号人，是比较庞大的。而那个时候 2012 年算是云计算刚刚起步的阶段，外面只能找到云计算的相关概念，至于技术材料还是比较少的，这意味着你很难在外面找到相关的技术材料作为参考。 在飞检的过程中，当发现感兴趣的技术问题或者感兴趣的业务，就可以问一下身边的同事。于是在一轮又一轮的飞检之中，能力也会有所提升。


这个能力包括了技术的提升和对业务的理解深度。特别是趁着过年期间，我把理解的业务串了一串之后，就逐步对系统有了相对宏观的理解了。至今还记得，当时我们的项目经理不是整个项目的项目经理，而是我们子项目的项目经理。他和我说诶，你过了一个年，好像不太一样了。意思是我对业务的理解好像加深了。当获得领导的认可之后，我逐步就成为一个核心成员了。过了半年之后，我就跳槽了。由于代码飞检给大木带来了切实的收益，我个人也是飞常乐意在团队里面去普及飞检机制。一般来说，只要是我带的项目，并且有时间，每年也都会组织几场飞检，同学们也可以在团队里面尝试看看，或许会有意想不到的效果。这节课就到这里，谢谢大家。
