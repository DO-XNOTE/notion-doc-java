首先团队要有统一的标准，并以标准作为基准。在此基础之上再去做 code review，否则很容易形成公说公有理，婆说婆有理的局面。大家真的面红耳赤，但是就是没有结论，本质上都是因为没有事先定好标准而导致的。


第二， code review 就是单纯的 review 代码，不应该掺杂其他的元素，像情感元素、绩效元素等等，通通都是要不得的。 code review 的目的主要是审查代码，并发现代码的问题，同时也能够实现团队之间的知识共享，促进团队成员的沟通。而绝不是吐槽大会说以缺陷和错误去批评他人。之所以把这一点拿出来讲，是因为一旦和绩效或者情感等等因素牵扯，就很容易达到不可预知的效果。


举个例子，小王的技术在团队里面正好是比较薄弱的， review 它代码的时候，如果掺杂情况元素，他被吐槽，一次两次可能可以忍 3 次， 4 次可能就消极了， 5 次 6 次搞不好都离职了。所以这样去做review，最后的结果是得瑟的人一直得瑟，憋屈的一直憋屈，对吧？OK。


第三， code review 不应该成为保证代码规范的手段。每个成员被 review 的代码，首先应该是符合规范的。 code review，它是一个人工手段，用人工去保证代码符合规范是非常浪费资源的。这种事情应该交给工具去做。比方 check style/pmd 等等。这些工具在 SonarQube 上面都有相应的插件，安装一下就 OK 了。 code review 的时候，应该把更多的精力放在代码是否正确上。

首先是功能性bug，也就是检查代码的功能是否正常，这是最重要的，也是其他一切的前提。

第二，逻辑优化。这一点常见于一些遗留代码，比方有一段代码写得非常的混乱，在混乱的代码基础之上去打补丁，可能也能够解决问题，并且工作量可能还小一点。但是如果时间允许，建议用比原先更好的代码。

第三，代码复用, 很好理解。

第四，检查是否存在不必要的代码，像这种不必要的代码，应该考虑把它删掉。

第五，检查代码的执行效率是否高效。

第六，检查资源占用是否正常。包括内存、CPU、磁盘等等各种资源的占用是否正常。

第七，检查资源是否有效关闭。比方数据库连接数据流使用之后是不是关闭了。

第八，检查资源池的配置是否合理，是不是过大或者过小。在架构设计期有讲过对象池、线任池、连接池如何调优，对吧？

第十、检查注释是否到位。比方是不是有代码看不懂，需要编写注释，或者注释写的不正确。

11、检查代码结构是否清晰。

12、命名是否规范。

13、检查单元测试的覆盖情况。

14、检查代码是否能够适应未来业务的变化。

这个很好理解，在互联网领域，业务需求的变化实在是太快了，所以很多时候光满足眼瞎的业务可能还不够，还要留有一定的扩展点，从而适应未来业务的变化。

15、检查是不是存在过度设计。

这一点和前面一条正好相反，我们鼓励提升代码的扩展性，但是并不建议过度设计。有点矛盾。因为如果想让代码的扩展性达到极致，业务不论如何变化都能够用这套代码玩下去。很多时候以牺牲代码的可读性以及易上手性为代价的。所以这两点放在一起来，其实要在过度设计和适应未来变化之间取得一个平衡。OK。

16、检查敏感数据是不是被加密处理了。

17、检查密码是否很好的管理和控制。这里的密码是广义上的概念，包括了用户的密码密钥，

比方 access key， secret key，还有token，比方用户的 access token， refresh token 等等。这些密码一般不建议直接提交的 Git 仓库，而建议用配置服务器去管理。比方 Nuckus。假设现在有一个应用的 access key 和 secret key，我们就可以把它存放到 Nuckus 。这样的只有 Nuckus 权限的人才能访问到 access key， secret key 相对就会安全一点。另外，你也可以把密码存放在数据库，比方用户的密码。总而言之就是密码要特殊的处理，不能随便来一个人都能看到。

首先建议沟通，而非对抗。前面说过， code review 的其中一个作用是促进团队成员之间的沟通，所以千万不要以吐槽的心态进行良好的沟通，可以让团队的氛围更好，大家更团结，战斗力更强。反之，相互吐槽会让团队的氛围变差，大家心不齐，就很难做大事了。


第二，不要自负。很多程序员比较容易自负，听不进别人的建议和意见，这个时候 code review 就容易成为口水战。这也是为什么前面说要统一标准的一个原因。一旦发生口水战的时候，我们就以统一的标准作为基准去评判到底怎么样去玩，对吧？


第三，建议小而多而非大而小。指的是单次 review 的代码量不要过多。你千万不要积攒了1万行代码才让你的同事帮助你review。相信我，你的同事他很忙，他绝对没有时间去研读你这1万行的长篇大论。这种情况下，对于超大的 code review，你的同事很可能扫一下不看了，直接给你通过，于是 review 就没有意义了。如果实际项目中你确实有很大的功能要做，建议分多次进行review。这样的每次 review 的时间都不会很长，你的同事能够腾出来这段时间。同时这样也可以灵活一些，你们甚至可以利用一些碎片时间进行 15 分钟 review 一段代码，OK。


第四，保持简短。这个指的是每个人的工作都很忙， review 的时间太长容易影响工作进度，得不偿失。所以 review 的时候应该快速的得到结果，不要陷入扯皮之中。


第五，不要太认识那个人。建议 code review 不要上岗上线，这样容易导致心智负担太重。


第六，三人行个人建议 code review 以三人为一组，也就是每个人提交的代码都有两个人去 review 它的代码。审查的人不建议太多。一是当第一个人审过之后，第二个人发现问题的几率已经变小很多了。第二个人复审一下，一般就 OK 了。再找第三个人，第四个人去审查，投入产出比就会越来越低。这是一方面，另一方面容易 3 个和尚没水池。


比方你一下找了 5 个人帮助你 review 代码，结果到最后没有一个人用心看的，都是一扫而过。因为都觉得自己随便看看就行了，反正自己审不出来，还有别人，对吧。而如果两个人一组，每个人的代码只有一个人去审查了，这样可能还是会有一些盲区，所以三个人一组是比较合适的，既可以避免盲区，也可以在投入产出比之上达到一个比较好的平衡。
