# 1-4 负载均衡（1350）

同学们大家好，这一章节我们来介绍一下负债均衡。首先我们来带大家了解一下什么是负债均衡，以及微服务的为什么使用负债均衡。然后给大家介绍一下负载均衡的一些场景以及算法策略，以及我们关注的加 RPC 场景下的负载均衡的基础选型。这里面我们主要关注的是spin，Claude， load balance 和奈飞的ribbon。最后给大家演示一下负载均衡的一些使用实例。


接下来我们来先看一下什么是负载均衡。负载均衡它是构建在原有的网络结构上，它提供了一种透明有效的方式进行水平伸缩，扩展服务器的网络设备的带宽，加强网络数据处理能力，增加吞吐量，通常可以提高网络的可用性。更直白一些说的话，负载均衡它的意思就是将我们的工作负载，这里面会包括工作任务、访问请求，进行一个负载平衡被分摊到多个服务单元，我们这里面会指服务器或者我们的主键上进行执行。通常情况下负载均衡是提升性能，避免单点故障的一些常用方案。
这里面看到对于通常的我们的服务方 provider 一，它在注册中心注册完成以后，那么消费方就可以获取到我们的服务列表。如果说只有一个 provide 一的话，这里面根本不存在负载均衡的概念，只有当我们的服务方它通过我们的横向扩容，服务节点越来越多的情况下，才会有我们负债均衡的一些存在的必要性。


现在我们看服务方有四个节点，消费方通过一个节点去选择我们的服务处理的时候，具体应该从 1234 这几个点里面选择哪个，这里面是需要我们提供我负载均衡策略，在这里面我们 consumer 方需要选择具体哪个节点，这个就需要我们的负载均衡策略来实现的。


回顾一下从单机网站到分布式网站，很重要的区别就是业务拆分和分布式部署，我们将业务拆分完成以后部署到不同的基地上，实现大规模的分布式系统。那么分布式和业务拆分解决了从集中到分布的一个问题，但是每个部署的独立业务还存在一些单点的问题，或者说是它的这个容量受限的问题，这里面我们要解决单点问题和访问统一的问题。为解决单点故障，我们可以采取容易的方式将相应的应用部署在多台机器上，解决访问统一入口问题。我们可以在集群前面增加负载均衡设备，实现流量的分发。


我们来介绍一下，对于系统的扩容可以分为纵向的垂直扩展和横向的水平扩展。那么对于纵向扩展是从单机角度，通过增加硬件的处理能力，比如说我们提高 CTO 的处理能力、内存容量、磁盘等方面来实现服务器处理能力的提升。通常纵向扩展的成本非常高，而且是有天花板受限的，它不能满足于这些大型分布式系统大流量、高并发海量数据的问题。通常我们会采用横向扩展的方式，通过横向扩展的某蒙 k 采用相对来说低成本的机器来解决大型网站服务的处理能力。比如说我们一台机器不能满足能，那么我们多增加几台机器让它共同承担访问的压力。这就是我们典型的集群和负载均衡的一些架构的场景。那么我们了解了什么是负载均衡，以及这个负债均衡它的工作原理。


那么我来看一下负债均衡它的一些应用场景是哪些？首先我们可以通过 DNS 实现我们的负载均衡，通过 DNS 磁线负载晶衡主要是为了解决全球地域性的多机房部署问题。我们通过 DNS 解析的方式引导请求到距离最近的机房的服务器进行我们的服务请求，通常它的优点是使用比较简单。我们的负载均衡工作交给 DS 服务器来处理，省掉了负载均衡服务器维护的麻烦，通常也会提高我们的性能。可以在支持我们基于地址的域名解析成距离用户最近的服务器地址，可以加快访问我们的速度，比如改善性能。


通常这些全球性的网站，比如像谷歌，我们在不同地区访问谷歌网站，它的响应时间是相差不大的。当然我们是通过会访问到我们相对来说比较近的一个节点。这些 GDNS 的负载均衡它有什么缺点？通常也就是它的可用性会比较差一些。 DNS 解析是多级解析，我们会新增或修改 DNS 后解析时间比较长，解析过程中用户访问网站可能会失败，扩展性也比较差。


DNS 负载均衡的控制权在域名商里面，我们无法对其去做更多改善和扩展，它的维护性也配比较差。通常我们不能反映服务器的当前运营状态，支持的算法比较少，也不能区分服务器的一些差异。也就是说，其实我们只是根据地域性指定了一个固定的机房。接下来我们说一下 IP 负载均衡是基于特定的 TCP IP 技术实现的负载均衡。 IP 负载均衡可以由硬件设备实现，也可以由软件实现。硬件设备主要是产品F5，软件产品有 SOSSA proxngs，其中 NGX 是我们接触相对比较多的负载均衡的实现。我们将要学习的 spring Claude， load balance 和 ribbon 也是属于 IP 负载均衡，基于软件的负载均衡器，它给了我们很大的自由度，可以实现各种算法策略来满足我们的一些业务场景。通常对于硬件负载聚合器，一般也就是F5，通常它的成本会比较高， OPS 同学可以关注，我们就不展开介绍了。


接下来我们来看一下负载均衡的一些算法策略，看一下常用的算法有哪些。首先是轮云随机追扫链接数和我们的哈希散列以及我们加权等等，还有一些其他的方式。通常这些负载均衡它默认采用的也就是轮询算法，保证我对于我们的服务节点的稳定的一个公平调用。对于轮询，它是将所有请求依次分发到每台服务器上，适合服务器硬件相同的一些场景。优点，通常是服务器请求数目会相同，它的压力负载是相同的。缺点，假如说我们服务器配置不一样，服务器的压力不一样，这就不适合使用轮询的方式了。另一种方式是随机，它会把请求随机分配到服务器上。


优点，当然是使用简单。缺点，它不适合机器配置不同的场景，因为这个随机性可以理解为是近似一个公平性的请求。对于最少链接数的话，它会将请求分配的链接数最少的服务器就是当前我们处理请求少的服务器。它优点是根据服务器当天的请求情况进行一个动态分配，它的算法会实现相对复杂，需要监控服务器请求的一些链接数，还有就是哈希散链，我们可能会根据来源 IP 或不同业务的一些哈希因子进行一个哈希计算，得到我们指定的一个IP。


优点，可以将相同来源的数据打到同一个服务器里面，它可以保自绘画的一致性，我们可以通过这样的转发相同的服务器实现这个绘画粘次。那么它的缺点是目标服务器宕机以后会话会丢失，并且我们通过哈希算法可能会有些机器它可能负载是不一样的。另外一串方面是加权，我们在轮询、随机和追扫链接哈希等上面的基础上，通过加权的方式进行负载的服务器分配。一个点就是根据权重可以调节转发服务器的请求数目。缺点当然也是使用比较复杂，当然我们还可以基于自己的业务场景实现个性化的一些负载均衡策略，比如说我们可以实现一个负载均衡，通过优先选择显然时间比较快的这些服务节点，作为我们下一次负载的一个选择。
