# 1-2 服务注册与发现-1（1238）

同学们大家好，这章节我们来介绍一下辅助处理发现，那还是设计了三部分来介绍。首先带大家一起了解一下什么是辅助处理发现以及微服务为什么要使用服务注册与发现。然后给大家介绍一下服务发现的技术实现，也就是注册中心它的一个技术选型，这里面我们主要关注于ZK、Irrock、 Cancel 和Nicos。最后来给大家演示一下服务注册发现的实例。首先我们来看一下什么是服务注册与发现。我们谈到注册与发现的话，我们首先需要有这样一个注册中心，那么对于注册中心的话，他关注的是我们的服务端和消费方他们之间的调用关系。首先我们的注册中心构建完成以后，我们可以启动我们的服务，也就是我们的服务方在启动的过程中，它要把本身这个服务注册到注册中心。对于我们通常的一个应用，它可能会有多台机器共同去服务，这样会把多台机器不同的 IP 和端口号作为它的一个唯一标记，注册了我们的服务中心。


对于我们通常所说的 user 服务或 order 服务，作为一个消费方，消费方如果说是需要一个用户服务，一个 user 服务，那么它需要通过注册中心里面去获取对应的 user 服务的列表。我作为一个消费方，我知道我需要调用我们的 user 服务，那么我通过命名去通过注册中心获取 user 服务对应的机器列表，我们就可以获取到它对应这里面注册的内容，通过注册中心获取到的我们的服务列表，我们消费方就可以直接去或调用我们服务方去执行我们的操作。


通过这里看我们回顾一下，首先是需要把我们的服务方注册到注册中心，同时我们的消费方通过我们的种种中心拉取我们的服务列表，然后来进行我们服务方的真正的一个调用。通过这里面我们可以了解服务注册，它其实就是将提供某个服务模块的信息，通常也就是这服务的 IPO 端口注册到一个公共的一个主键上，这里面是我们的公共主键。服务发现是指什么呢？服务发现就是指新注册的这个服务模块能够及时的被其他调用者发现，不管是服务新增或服务三处都能实现一个自动发现，那么就是说在这里面，假如我的服务列表新加机器了，或者是减少机器了，在这里面的注册过程都需要通知给消费方，让消费方对于对应的机器列表新增或移除。那么我们接下来看一下。


至于微服务为什么要使用服务注册与发现的。在早期我们处于单机服务时代，我们是这样的，我们作为一个服务，读取一个数据库，我们外网的这些数据去访问我们服务，整个过程就完成了，那么在这个过程纯是单机的一个效果。关于 user 和client，它访问我们 Server 直接指定 IP 和端口号就进行访问 IP 了，这个过程好像并没有涉及到我们服务注册发现相关的内容，因为服务部署相对简单，访问也比较简单，整个系统的运维也比较简单。通常在那个时候我们进行发布部署的过程中，甚至还有可能需要我们进行一个停服的操作。


现在时代已经变了，我们处于微服务集群的模块，我们的用户访问过程中，他根本不知道经历了什么处理才能真正到我们的用户服务过程。随着我们服务的增长的话，我们的机器的节点列表会突然增加，会有时候会收缩减少。那么在这个过程中，用户在访问应该访问哪个节点，这个过程就不是那么简单了。所以说在这个过程中我们需要有一定手段来让用户知道他应该去访问哪个节点，所以说这里面就涉及到了一些服务助推发现相关的一些中间件。


那么在这里面用户访问的过程中，他通常会首先访问到网关，那么网关他要知道他后面有多少服务的节点是可用的，它通过一定的策略进行负载均衡的选举，那么选中对应的这个节点去处理这样一个请求。这样的话由于机器的比较复杂，那么如果说我们每一次增加节点或三处节点都需要人工去在网关上配置的话，那么这个工作量可想是非常大的。所以说我们现在去为了让我们整个微服务集群它的节点更快速的让网关这一层发现，也或者说是他们在节点之间的调用能快速发现，所以说我们就有必要去使用服务注册与发现这样一个工具了。通过这样介绍，我们可以得出结论，微服为什么需要使用服务发钱与注册的工具呢？首先它是为了屏蔽进行解耦服务之间相互依赖的细节。我们知道服务之间的远程调用是必须知道对方的 IP 和端口信息的，我们可以在调用方直接配置和调用方的 IP 和端口，那么这种调用方直接依赖 IP 端口的方式存在明显的问题，也就是被调用的 IP 端口发生变化以后，调用方也需要同步修改。


通过服务发现，将服务之间的 IP 端口的依赖转化为服务名的依赖，那么服务可以根据微服务业务的标志，因此屏蔽解耦了服务之间依赖的细节是服务发现与注册解决的第一个问题。那么第二个问题就是微服务进行一个动态的管理。在微服务架构中，服务众多，服务之间相互依赖也错重复杂，无论是服务的主动停止、意外挂掉，还是因为流量增加对服务实现进行扩容，这些服务数据处状态上的变化都需要尽快的通知到背调方，被调方才可以采取相应的措辞。因此，对于服务注册与发现，要实现管理者服务的数据与状态，包括服务的注册上线、服务的主动下线，以异常服务的剔除。所以说服务发现将服务的IP、端口等细节通过一个服务名抽象给调用者，并动态的管理着各个微服的状态检测，状态更新，服务下线，服务上线，那么这就是微服务治理的基础。


接下来我们讨论一下注册中心的技术选型，那么在讨论注册中心技术选型之前，我们首先要想注册中心需要解决哪些问题，在这里面我们列举出了几点。首先是集群，我们要组成集群，这样单台出现故障，不至于我们的服务宕机。所以说我们刚才对于服务注册中心的理解，我们应该知道服务的注册与发现依赖的服务的注册中心，那么注册中心是非常重要的。另一方面就是通过集群它需要完成的数据同步，我们组成了集群，那么就需要数据同步，注册的信息在一台注册的其他机器上也应该能看到，不然的话一台挂掉，那么其他数据就丢失了。
