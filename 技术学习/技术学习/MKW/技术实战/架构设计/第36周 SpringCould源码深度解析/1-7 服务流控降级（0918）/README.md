# 1-7 服务流控降级（0918）

同学们大家好，这章节我们来介绍一下服务流控降级，还是划分为三部分来介绍。首先带大家一起了解一下什么是服务的流控降级，然后给大家介绍一下服务流控降级的常用技术方案，最后给大家演示一下基于 settle 实现的服务流控降级实例。那么我们来先看一下什么是服务流控降级。对于跟服务流空降级相关的内容，我们这里面主要涉及到限流减负，在限流减负的过程中，我们同时能做到流量整形，还有也就是我们要解决熔断与资源隔离，解决限流熔断与资源隔离，它主要是为了做到一个防止雪崩的问题。首先我们来看一下限流与减负与流量整形，就是随着微幅的流行，我们服务与服务资金的稳定性变得越来越重要。在应对秒杀、大促、双 C 618 等高性能压力的场景的时候，限流已经成为了整个技术方案的标配，为了保证系统的平稳运行起到了一个关键的作用。不管应用场景是哪种，限流无非就是针对超过预期的流量，这里面大家注意，一定是超过预期的流量，通过预先设定的限流规则，选择性的对某些请求进行限流。对于微服务来说，其接口请求可能来自多个系统，用户服务的接口会被多种内部系统调用，比如说 CLM 促销系统。对于服务于众多调用系统和应对的一些海南接口请求的微服务来说，接口限流除了应对上面提到一些大促的秒杀场景外，普通的系统服务其实也会碰到相应的一些挑战。


我们作为服务接口的提供方，是无法限制调用方如何来使用我们接口的。比如有一些调用方通过多线种批处理任务来请求我们的接口，或者是因为调用方的带有 bug 或者业务上面突发的流量，它导致这个接口的请求量突增。在过度征用服务线程资源而来自其他调用方的请求，因为资源不足，他不得不排队等待，那么微服务的整体的响应磁场就会变长，甚至超时。


我们为了防止接口被过度调用，需要对每个接口进行一个细粒的访问限流，也就是说我们的限流，它不仅是我们针对于大促双 C 618 的外部的这些高频接口，对于我们内部的系统接口，我们也应该做好自我防护，做好限流的工作。那么我们来看一下我们如何去防止雪崩。


对于这里面我们看到其实企业内部，尤其是物联网、易应用公司，它们之间的服务调用是非常复杂的。这里面我们可以看到通常基础的服务，像身份证校验，我们的加解密，对于敏感数据的加解密好，或者是我们底层的数据库炒作这些作为我们基础的微服务，我们在中间层的一些应用，比如说我们各个 BU 提供它的业务服务去做处理，最终这些服务暴露到我们的客户端。


比如说我们的APP，我们的3W，或者说是我们给运营同学用的管理后台去做的一些事情，我们看到当如果说我们这些基础服务有一个接口挂掉，比如说我们的加解密挂掉以后，它对各个模块的影响是怎样的？从这里面我们可以认为蓝色的这是我们是正常的，红色的是有问题的，黄色它可能是部分有问题的。从这里面我们看到当加解密接口挂掉以后，所有调用加解密的应用它都会受到一些影响。


这里面影响比较严重的是BU2， BU3 是部分影响，对于 BU4 它可以做到不受影响。在这一层中间层的这些影响它会传递到我们的上层，那么也就是传输到我们的客户端，比如说我们的 APP 三大客户管理后台它受的影响是怎样的？那么我们在这里面可以看到对于 APP 它的影响是什么呢？如果这里面我们 APP 做的我们的整个稳定性治理做得比较好的话，它的影响可以降到市场的一个位，而不至于像 BU2 它直接挂掉了，所以 3W 它可能处理得不太好。


如果说它依赖到BU2，它可能就处于一个不可用的中断状态。对于管理后台，它可能处理的方式就比较特别了，管理后台通常我们的降级方案会更容易一些。如果说我们底层接口失败以后，我们在处理的过程中一定要像 B O 一或 B O 四一样，我们尽量做到受影响。当然 BU1 是因为没有调用的接口， BU 4 它可以通过一些特殊的手段对于加减云接口进行了一些降级，像 BU2 的话，它可能是作为一个强依赖，它没办法做降级，那么只能就受到影响了。所以说我们可以看到当底层的一个接口出问题以后，其他上面这些服务都会出问题。如果说因为底层的家里面接口把我们的其他服务拖挂，最终导致我这一端的客户不可用的话，这就是雪崩的一个效果。我们怎么去防止雪崩呢？


既然你出问题了，我就不再调用你了，我会通过其他方式降级服务。另一种方式就是说当加解密接口出问题以后，它是什么原因出问题的？比如说它是因为 B U 二，它的请求量过不多，导致我们的加解密接口出问题，所以说因为加解密准，同时影响了 BU3 和BU4。这里面我们可以注意的方式就是什么呢？通过资源隔离我们来看一下资源隔离，对于熔断与资源隔离的话，熔断这里面我们可以看到在一些老式的电闸上面会有一些保险丝，当这个流量负载过高的时候，我们会断开保险丝来保护家用电器，当然这种电闸的方式使用的比较少了，我们现在的这种智能的电闸，它也是有过窄保护功能的，它可能不会看起来这么原始。资源隔离基于现在我们这个新冠疫情的影响，资源隔离大家现在已经很有感触了，就是资源隔离，我们通过方舱也好，医院隔离或制定隔离也好，避免了人与人之间的传染。


现在我们知道我们其实做服务流控主要是为了防止底下的某一个节点出问题，把这个影响放到了整个企业应用，防止雪崩这样一个效果。我们来接下来看一下我们服务流控实现的那些技术方案有哪些。在这里面我们列出了大家可能接触到的几个方案。首先我们看 his tricks，对于 his tricks，它可以说是出现最早的流控组件，当前它处于一个维护模式，它不太更新了，所以说作为 string close 的微服务组件中，它最早是 string close 的默认集成的一个流控组件。那么最原生的一个流控组件很多思路他还是有必要了解一下，比如说他提供的一些服务熔断主持故障、联手发音快速失败并迅速恢复或服务降级等等。


其实黑色 case 里面它定义了几个可用不可用还是灰度可用的一个状态，另一个是 resilice 错接。那么对于这个是 history 的停更以后， spring Claude 推荐的一个流控组件，因为它出来的时间比较晚，国内的一些使用量也很少，所以说它不作为我们的重点。另外这里面是 spring Retry，在流供领域， spring Retry 它实现的还是比较单薄，功能有限，这里面我们带特定的一些场合可以使用，但是如果我们作为一个流控平台的话，不建议大家使用。所以说这里面settle，它是我们学习的重点。


settle 诞生于阿里巴巴，它开完后在国内很受欢迎。 settle 以流量为切入点，中流量控制、熔断降级、系统负载保护等多个维度保护服务的稳定性item。它可以针对作为不同的调用关系，以不同的运行指标，比如说 QPS 并发调用系统负载等基础搜集资源的路径，并将这些资源的调用路径的以树状的结构存储起来，用以根据调用路径进行资源流量的控制。
