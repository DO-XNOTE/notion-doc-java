# 2-1 Nacos架构设计-1（1321）

同学们大家好，这章节我们来介绍一下 NICOS 的架构设计，我们还是设计呢三部分来介绍。首先带大家一起了解一下什么是NICOS，然后给大家介绍 NICOS 两个最重要的服务，分别是服务发现及管理，以及 nicos 的动态配置服务。那么首先我们来看一下什么是nicos，它本身是致力于帮助我们发现配子和管理，为服务 Nicos 它提供了简单易用的一些功能集，帮助我们快速实现动态服务发现、动态的服务配置以及服务元数据的一些管理。


为了更好的说明什么是Nicos，我们的学习了解以下几方面的内容，分别是像 NICOS 简介 NICOS 的架构以及它的一些核心概念。接下来我们会认识一下 NICOS 的源码。好，首先我们来看一下 NICOS 简介。 Nicos 它是帮助我们更敏捷和容易的构建、交付和管理微服的一个平台，它是构建以服务为中心的一个现代应用架构，比如说像微服务范词以及云原生的范词的一些服务基础。这里面通过了解 nicos 提供的各种功能来认识一下nicos。


首先我们看一下 nicos 提供的最重要的功能，也就是服务发现和服务的健康检测。 Nicos 它支持基于 DNS 和基于 RPC 的服务发现。服务提供者使用原生的SDK， open API 或者一个独立的 agent 的它注册设备之后，服务消费者可以使用 HDP API 的方式去查找发现服务。由于 spring Claude，阿里巴巴它已经对 nicos 做了一个完整的封装，所以说我们现在在开发 vivo 的时候，很少在手工的调 nicos 这些看的API。


对于Nicos，它提供了服务的实时的健康检查，这个实时的健康检查，它也是通过调用服务暴露这个端口，通过获取的内容去判断这个服务是否健康。这跟 NGX 它的 cos check 的原理是一样的。整个这个健康检查，它主要是为了组织这些不健康的主机，它暴露出来，避免了一些我们的消费端像这些不健康的主机发生服务请求。
Nicos 它支持这些传输层和应用层的健康检查，比如说传输层通常是我们通过 PIN 或 TCB 的方式调用。对于应用层，我们可以比如检查 STP 的端口以及 MySQL 服务的一种情况，或者它还支持一些用户自定义的一些健康检查。对于一些复杂的云环境和网络拓扑环境中，这些服务健康检查也可以做到的。 NICOS 提供一些agent，它可以通过 agent 上报模式和服务端主动检测两种健康检查方式。 NICOS 还提供了统一的健康检查仪表盘，帮助我们根据健康状态管理服务的可用性及一些流量的分配。


另一个就是 NICOS 的动态配置服务。动态配置服务可以让我们以中心化、外部化以及动态化的方式管理所有环境的应用配置和服务配置。这里面的环境包括测试环境和线上环境动态配置，它消除了配置变更时重新部署应用和服务的需要，也就是说，我们能做到动态的变更，不需要再重启服务了，也让配子管理变得更加高效和敏捷。当然，我们修改的是越来越频繁配子中心化，它可以实现无状态的服务变更，让服务按需弹性扩容变得更容易。 NICOS 提供了一些简单的UI，也就是我们 NICOS 的控制台，它可以帮助我们管理所有的服务和应用的配置。 Nicos 还提供了包括配置版本跟踪、灰度发布、一键回滚配置以及客户端配置更新状态的跟踪、站内的一系列开箱即用的配置管理特性，它也可以帮助我们更安全的在生产环境下管理配置变更和降低配置变更带来的一些风险。


另一个就是它的动态 DNS 服务，它支持一些权重路由，让我们更容易地实现中间层的负载均衡，更灵活的一些路由策略、流量控制以及数据中心内网的简单的 DNS 解析服务。动态 DNS 服务还能让我们更容易地实现以 DNS 协议为基础的服务发现，以帮助我们消除与私有服务发现 API 上的一些依赖关系。 Nicos 它提供了一些简单的 DNS API 来帮助我们的管理服务，一些关联域门的一些可用性的 IP 和端口的一些列表。


还有就是它的服务及元数据的管理。 Nicos 它可以让我们在微服务平台建设的视角管理数据中心所有的服务以及元数据，这里面包括管理服务的描述，生命周期，服务的一些静态依赖分析，还有服务的健康状态，服务的流量路由以及它的安全策略。另外一个是 NICOS 提供简单的安全认证， NICOS 它提供的这个安全认证相对来说比较简单，它主要是为了防止业务错用的一些弱依赖的健全体系，它并不是防止恶意攻击的强健全体系。通常我们认为 Nicos 它是一个内部的微服务组件，需要在一个可信的内部网络中运行，它不应该暴露在公网环境、房子带来一些安全的风险。


最后一个就是nicos，它客户端是支持多语言的。我们知道 NICOS 是基于 Java 开发的，所以说在 Java 领域很多同学使用 NICOS 对应的 Java 的，同时，由于像 go 语言Python，NODE，它的使用范围越来越广泛，其实 Nicos 也提供了这些语言的 client 端。


接下来我们看一下 Nicos 在官网的它的一个地图描述，从这里面我们可以看到，首先我们看到 Nike 的它一些特性，它是支持服务发现、配置管理以及 DNS 服务。这里面还有一些细节的内容，比如服务发现里面，它会支持一些权重管理、打标或者是优雅的上下线。在配置管里面，它会支持一些在线编辑，历史版本，一键回滚、灰度发布，还有一些推送轨迹等。


另外我们看到的架构图里面，它会涉及到像逻辑架构、领域模型以及部署架构，还有它的类似图提供了这样的一些设，大家可以在官网上去了解一下。另外它的业务大图里面主要还是服务发现和配置管理，那么在配置管理里面扩展了很多功能，这里面比如说像路由规则的配扩展配置管理以及限流规则的配置管理，还有像动态的数据源预案开关以及动态的一些UI，还有它的一些生态大图，这里面主要包括像它 nicos 支持string， Claude 支持double，支持K8S，支持一些 NODE g s 的一些生态。
另外我们这里面可以看到 NICOS 它的一些优势，首先它是简单易用的，它是经过几万人的使用反馈优化几代人的打磨。另外它的特性也比较丰富，其实跟 nicos 跟 u 瑞卡、 RZK 这些比较起来的话，它的特性确实比较丰富，使用也比较简单。另外它还是有一些超高的性能，它可以满足一些苛刻场景的要求，极致性的性能。另外它的容量是非常大的，它可以支持阿里的规模经济，造就强大的容量。另外它的高可用，它能支持双 11 不失败，造就了它高可用的一些认可度。至于它的一些战略，我们就不用花太多心思去了解了，不管是平台运营还是生态，都具有它的一些优势。


接下来我们看一下 NICOS 的架构以及它的一些核心概念。服务可以说是 NICOS 司机的一种公民， NICOS 它支持几乎所有主流类型的服务，发现的配置和管理服务我们可以理解为它是一个或一主的一个软件功能，比如说一些特定信息的检索，或者是一主 CRUD 操作。Nicos，它支持这些主流的服务，比如说像 K8S service 或 double GRPC，以及 spring Claude restable 思维。


另一个词我们了解一下服务注册中心，大家了解的比较多了，它可以理解为是一个数据库，是用来存储服务以及服务实例，包括它的元数据的服务实例在启动的时候，通常把自己注册到服务注册中心，并在关闭式的注销服务的客户端或消费端，它通过查询服务的注册表以及查找服务的一些可用实例。服务注册中心它可以通过调用服务实例的一些健康 API 来验证当前这个服务是否可用，也就是说它是否健康，能否接受并处理一些客户端来的请求。


接下来我们看一下服务元数据，服务元数据比较容易理解，它可以理解为是服务数据的一些扩充和描述，这里面可以包括一些服务的一些暴露的端点，以及服务的标签，或者是像当前这个服务它归属的业务线以及负责人等等这些思想。另外比较理解像服务的提供方以服务的消费方式比较容易理解的，那么服务就是服务的提供方，那么对应的消费者就是服务的消费方。服务提供方把自己注册到注册中心，服务消费方通过注册中心获取到服务并进行执行调用。好，接下来我们看一下 NICOS 配置相关的一些概念。在系统开发过程，通常会将需求变更的一些参数变量从代码中分离出来。独立管理，通常我们会把这些配置抽取到一个独立的配置文件，主要是为了通过不同的配置来适配不同的运行环境。那么配置管理一般包含在系统部署的过程中，由系统管理员或者运维人员完成这个步骤。就比如说我们上线的时候，对于 MySQL 的用户名和密码，很多情况下是不会开放到开发人员的。配置变更它可以理解为调整系统运行时行为有效的一种方式。


我们看一下配置管理，其实在数据中心中，系统中所有的配置的编辑存储、分发、变更管理、版本历史管理、一些变更审计等，这所有跟配置相关的活动，我们可以统称为一些配置管理。另外一种就是我们的名字服务，其实也可以称为命名服务，它那个是提供分布式系统中对所有对象实体的名字，关联到元素之间的一些映射的管理服务。


另外我们看一下配子服务，配子服务是什么概念？其实从概念上讲的话，它是在服务或者应用运行层，可以通过动态配置或元数据以及配置管理的一些服务的提供者。在这里面 Nicos 它就可以理解为它是一个配置服务。接下来我们看一下是动态配置和静态配置，可以理解为是个对立的关系，通常我们在程序里面配的对应的是比如说 application 的property，placement， YML 这些我们可以理为静态配置，通常这些配置的修改需要我们重启程序来让我们的配置生效，那 nicos 给我们提供的可以理解为是动态配置。动态配置也就是说我们可以在控制台上去配置这些属性的一些值的修改。我们可以通过一种方式，比如说推送或轮询的方式，把这些规则的配置推送到我们的应用程序里面，让应用程序里面可以做到计时生效，它这样可以避免我们程序重启。


接下来我们看一下 NICOS 的架构图，我们一眼看到这个架构图的话，我们可以粗略的把整个架构图划分为三部分，第一部分是中间这一大块，它可以理是 NICOS Server，另一部分我们可以看到是 NICOS 的控制台，也就是控制台操作。另外一部分是在这里面，不管是 provider 和consumer，它都可以理为 nicos 的 Claude 端去做的一些事情。我们先看重要的是necoserver，在 necoserver 这里面包含几部分，首先是这里面包含的是一致性协议，那么在一致性协议上面，它进行了 Nicos core 的一些业务的封装。基于 next core，它提供了两个非常重要的服务，也就是我们这里面的配置服务和我们的命名服务。对于这些业务的操作，它通过 open API 的方式暴露出来，通过 h t p 协议可以进行我们这些数据的一些读写。


那在这里面我们看到这里面对应的是控制台，它可以包含多种方式，比如用户的控制台和管理员的控制台，我们还可以通过一些特性定义一些自定义的控制台。这些功能描述清楚以后，我们看到对于 Server 的部署的话，它肯定不是单实例的部署，它可能需要多实例去构建一个集群，保证这个 next Server 的高可用。


不管是作为命名中心或者一个 configure 配置中心，它的依赖度是非常高的，它的高可用是非常重要的，我们可以看到这里面是层级的结构，可以理解为它是一个多实例的组合。那么在我们看它的 provider APP，它可以把 next client 端作为一个价包集成到我们的价环境里面，也可以采用这个赛道，靠这种比较鞭车模式进行跟我们系统的一个解耦的一个结合，那么这里面对于 consumer APP 也是这样的，我们可以通过 next Claude 端进行依赖，也可以通过 set core 模式进行一个解耦的依赖。在这里面他们都是通过对于相同这个服务名称的一些查找去进行一些定位。在这里面我们 Pro APP 可以把自己的服务注册到我们的注册中心，那么注册中心它会把 provide APP 的服务抽象成一个服务的名称。


在 consumer APP 它可以通过这个服务的名称查找到对应这个服务它的 IP 和端口号的列表，这样我们可以进行一个远程数据的远程调用，这个数据远程调用的协议其实 nicos 并不关心，这里面不管是 double 或者是 SDP 或者是GRPC，都是我们 consumer 端，它跟 provide 端直接的一个交互关系。
