# 4-1 OpenFeign架构设计（1341）

同学们大家好，这一章节我们来介绍一下 load balance 的一些常见的面试题。那么 load balance 作为负载均衡其实最常被大家问到的，也就是说我们负载均衡它存在的意义是什么？其实对于负载均衡来说，它的意思就是说将这些工作负载它可以进行平衡，分摊到多个服务单元上进行执行。通常情况下，我们使用负载均衡主要是支持一个横向扩展，那也能提升一些性能，主要是为了避免一些单点故障的一些产偶发。


其实负载均衡它的一个前提就是说我们是在使用集群。那么我们回顾一下从单机网站到分布式网站，它最重要的一个区别就是业务的拆分和分布式布，我们将应用拆分后部署在不同的机器实现大规模的这分布式系统，那么分布式系统和业务拆分解决了从集中到分布的问题，但是每个部署的独立业务还存在这个单点问题和统一访问入口的问题，那么我们为了解决单点故障，我们可以采用冗余的方式将相同的应用部署在多采集以上，解决访问统一入口的问题。


那我们可以在机群前面增加负载均衡设备来实现流量分发，那么这里面我们就提了负载均衡，它的意义是解决我们这个集群如何进行，有效的进行一个流量转发的一个过程。那我们再来去讨论负载均衡存在的意义，首先它一定是存在于集群上，那么单机上是没有负载均衡存在的意义，我们将同一些应用部署在多台机器组成一个处理机器，我们可以理解为多个实例节点，我们通过接受负载均衡设备分发的请求进行一个处理并返回相应的处，那么负载均衡这个设备我们通常是在客户端去完成。
它是我们将用户访问的请求根据负载均衡的算法分发到集群中指定的一台处理器，也就是说我们把一种把网络请求分散到一个服务集群可用的服务器上的一个设备也好，或者说是一台应用也好，那么对于负载均衡它的作用，也就是或者说是它能解决的问题，首先是它可以解决并发压力，能够提高应用处理性的主要是为了增加吞吐量，可以加强这些网络处理能，那么另外它还可以提供故障转移实现一些高可。这里面对于提供故障转移主要还是依赖。


当我们如果发现下游的某个节点它已经不可用的时候，我们可以通过我们负载均衡的列表把这个节点移除，那么我们可减少我们请求的失败率。那么另一种方式，我们可以通过添加或减少服务器的数，提供网站的一个伸缩性。那么其实对于互联网的面向 c 端应用，天然就需要支持我们这些伸缩性，那么在我们服务节点伸缩性的基础上，我们基于我们的负载均可以去动态的增加或减少节点，这也是跟我们服务注册与发现相关。


另外还可以支持一城的安全防护，通常负载均衡有可能它负一个网关城，那么在网关城去做负载均衡，那么网关城我们可以去做一些比如黑白名单，或者说一些权限校验相关的一些事，那么这些可以去理解为负载均衡它存在的一些意义。那么我们接下来看对于负载均衡一些具体的一些事情，比如我们常见的这些负载均衡的策略，它有哪些？对于常见的负载均衡策略，我们比如说可能有轮询、随机最少链接数，或者说是原地的一些散列，或者一些根据机器的情况进行加取，那么这里面对于每一个不同的算法，它有自己的一些特征，比如说撸轮询的话，它可以把所有的请求依次分发到每台机器，这样的话对于服务器硬件相同的这些场景的话，它可以做到一个最均匀也就是最平。


那么它的一个缺点，如果说当我们服务器的压力不一样的时候，其实因为我们给的流量是相同的，对于服务器压力不一样就不是很适合它，特别容易这个服务器压力，比如说它的负载小的这个请求特别容易压迫。另外一方就是随机，那么对于随机的这些负载均衡策略也有很多地方在，因为它实现比较简单，并且在它的缺点是什么？它并不适合我们对机器来配置不同的一些场，因为有时候我们会对机器做一些特殊的策略，那么随机的话就它太随机了。嗯，对于有一种叫最少链接数，或者说是我们对于一些加权平均数，这些都是我们需要根不同机器的负载情况去一些动态，那么不管是最少链接也好或根据负载也好，那么其实都会对这个负载均衡的算法提出更高的要求。比如像追查连接数，我们肯定需要计数，需要根据服务器大家的请求处理情况来进行一个动态分。它的缺点确实是算法相对比较复杂，我们需要监控服务器前几天的一个链接，那么对一次性哈希的算法，其实它是相对来说比较简。对于它的优点是对于相同的 IP 请求，它可以聚合到同一个服务器上，就是或者是同一个会话期，它会进行一个比如会话粘连这样一个效果。对于缺点它可能是目标服务器宕机后绘画会丢失。


另外对于一次性哈希的过程，有可能我们在选取的哈希它的这个区分度比较弱，那么它很会把一大部分内容集中到相同的几台机器上，会也就是说把我们的服务压力会做得不平均。主要是这一些算法，对于常见负载均衡，我们还可以去自己定义一些特定的规则，比如说我们这个规则可以去进行一些区分，比如说根据地区，或者说根据一些省不同的省份进行区分，或者说对于不同的人群进行一些一致性的依据负债均衡的策略。接下来我们来看一下对于 spring code， load balance 它的一些实现源。这个其实说起来我们在这里面再重复介绍一遍的意义不是很大，家去重复，去仔细去回顾一下我们在负载均衡源码解析里面的，那对于 spring q 的 load balance，我们要知道在 spring q 的设计的过程中，首先我们知道有 spring cloud，那么 common 里面定义了一下服务注册服务发现 load balance 和我们的熔断器等等这些概念的抽象，也就是定义出它的抽象以后会有各种不同的实现进行支持。
