第一种方式是单机房部署。这个方式下会开放多个访问点，比方美洲访问点、欧洲访问点以及国内访问点。但是机房只有一个访问点和机房之间会使用专线打通，各区域的用户都会通过访问点去访问应用。这种方式的好处是架构起来比较的简单，实施成本比较低，同时也可以保证数据的一个强一致性。但是这种模式下它的速度会受限于访问点和机房里面应用服务器之间的一个专线网络情况。所以这种模式一般会适用于要求数据强硬的性，并且目前没有拆分数据库的场景。
第二种是比较适用于海外流量不大的场景，也就是主要的客户在国内海外的流量很少，我们可能并不想分太多的精力专门再去搞一套海外版，于是就这样玩了。

这种模式下，会把应用系统以及数据库都部署到不同的机房，不同机房之间也不会去做数据同步，也就是数据彼此是隔离的。这种方式的实施成本也是非常低的，只是部署了多套应用系统以及数据库而已。但这种模式的不同区域，它的数据是完全孤立的。因此这种模式一般只适用于在国内外业务完全无关，并且允许数据孤岛的场景。就大木个人的经验来说，这种模式用的并不是很多。
