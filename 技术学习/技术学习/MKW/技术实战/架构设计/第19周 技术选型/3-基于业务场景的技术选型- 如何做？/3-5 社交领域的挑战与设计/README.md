此外，还以儿童为目标用户的小天才手表。讲到这里，可能有同学要笑了，小天才手表是什么鬼？这是乱入场次对吧？大家如果做个调研就会发现，小天才手表它是有很强的社交属性的，比方两个小天才手表之间的配碰就可以添加好友，而小天才手表里面的一个软件叫做微聊，可以做聊天语音、群聊的。可以说小天才手表在儿童这类群体里面是占有一定垄断地位的。

全称 http live streaming , http 直播流。这种协议是由苹果公司定义的，基于 httP 的流媒体实时传入协议。它的原理是把整个流分成多个小的文件来下载，每一次只会下载其中的若干个小文件，服务器端会把最新的直播数据产生新的小文件。这样的客户端只要不断的按照顺序去播放从服务器端获得的新的小文件，就可以实现直播了。


hrs 的传输内容包含两部分一是 M3U8 描述文件，它告诉播放器怎么样去播放二是 TS 媒体文件，里面存储的是流媒体 TS 媒体文件，里面的视频必须是 h264 编码，音频必须是AAC 或者是 mp3 编码。这点和前面的 rtmp 是类似的。 hls 的好处在于，数据是通过 http 协议传输的，所以不需要考虑防火墙或者是代理的问题。但是由于 hls 它是分段传输的，因为它是把整个流分成若干多的小文件来下载的，所以它的延迟会比其他的流媒体直播嫌疑高。

首先是高并发带来的挑战。一个热门的直播可能会有数百万人在线，而像春晚直播甚至上亿人在线，如何容纳这么多的人同时在线，这对带宽流量应用的高可用性、吞吐量、并发能力等等都是非常大的挑战。


第二是音视频处理以及传输上的挑战。工作涉及到实时美颜、音视频编码、视频推拉流、 c 端加速等等技术点，而每一个技术点都很有技术挑战。


第三是低延迟性的要求。其实是比较矛盾的，因为我们又希望传输要低延迟，同时又希望视频更加的清晰，所以需要在视频的清晰度和低延迟之间做好一个权衡，稍微获得更好的用户体验。


第四是终端适配的挑战。现在有非常多的直播终端，安卓 ios、 pc、 H5 等等，你的直播如何能在各种端上去适配也是一大挑战。实际项目中，我们常常通过转码、调整清晰度等等方式，从而满足不同设备不同清晰度以及流量的需求。
