并且物流业每年的发展速度依然非常喜人，依然有很大的潜力。虽然整个行业发展得很好，但是目前国内有数千家物流公司，主要的订单却都被头部公司抢占了。大木个人认为未来很可能会出现合并或者是联盟的现象。一来腰部以及尾部的物流公司需要抱团取暖，因此可能会多家公司合并。二来由于物流行业竞争日益激烈，很可能会形成阵营的格局，多家公司可能会形成联盟去对抗其他联盟。所以如果大家决定在物流领域发展，建议尽量去头部的物流公司，比方顺丰速运、圆通、德邦、中外运等等之类的公司，避免也会被吞并。当然了，这只是大木的一家之言，大家当笑话听就可以了，不要太放在心上。

在这里，我们不妨以物流的分支，大家最熟悉的快递业务展开。用户寄送快递的完整流程大概是这样的首先，用户去下单。下单可能是直接使用物流公司提供的 APP 或者是网页进行下单。


比方顺丰的 APP 也可能是在电商平台上下单，由电商平台去调用物流公司的接口完成下单。不管哪种方式，本质上是一样的。一个下单的业务功能。当当物流系统收到用户的订单之后，会生成一个订单，并收取相应的费用。接着，系统会根据用户的位置，并结合配送员的实时位置查找出合适的配送员。例如查找距离用户 5 千米以内的所有配送员，系统会把订单推送给范围之内的所有配送员。这些配送员收到通知之后，会进行抢单操作。一些平台不做抢单了，而是自动把订单派送给比较近同时工作不繁忙的配送员，我们图上是以抢单为例进行。看到了，当配送员抢单成功之后，系统就会把订单改为已接单。当然了，如果是自动派单，则是配送员确认接单之后，再去修改订单状态。


之后，配送员会和用户去联系，确认取货相关的信息，比方确认取货的地址是不是正确，什么时候方便取货等等。这一步不一定会有，看配送员的工作习惯。配送员取货之后，会入库分拣，并为货物规划出一条线路。这个线路的规划是动态的。比如正常情况下，这条线路是从物流中心 a 分发到物流中心 b 的，但是物流中心 b 爆仓了，就会发送到物流中心c，在物流中心 c 派送给收货人。当派送结束之后，配送员点击已经妥投，订单就结束了。订单结束之后，用户还可以去做评分、领积分等等操作。当然了，这些都不是主要流程，这是快递的大致流程，相信这个流程大家还是比较熟悉的。

我们可以把整个系统划分成平台能力以及系统能力两块儿。运力调控平台一是对运力进行规划和自动调控，二是运力的干预，也就是手动调控。 RBS 平台， RBS 的全称是 location based service，也就是基于位置服务。比方根据位置查找附近的配送员，可视化追踪货物的位置等等。


调度平台比如调度离客户最近的配送员，将货物调度到最近的物流中心等等。其他的比较好理解。再来看系统能力部分。绩效考核系统是结合评分投递的准时程度、投递单数等等对配送员进行绩效考核。 SRA 系统则是对配送员服务品质的一个管理。比方按照货物的紧急程度倒序排列，让配送员优先投递紧急的物品。再比如遇到投诉的时候，可以针对解决问题的效率进行统计。其他的比较好理解。


当然了，这个架构图里面，我们只画出了核心的部分，像网点管理、货品管理、商家管理、会员管理、积分管理、公告管理、报表分析、知识库、客服系统、开放平台等等。由于不在主流程里面，所以我都没有去画。


总而言之，物流系统主要围绕三件事去展开，一是便向用户提供服务，例如定价、交易、计算送达的时间等等

二是在成本、效率以及用户体验三者之间寻找一个平衡，计算出最合适的线路，匹配最合适的配送源三是为配送员提供辅助决策，例如路径推荐、 SRA 展示等等。
