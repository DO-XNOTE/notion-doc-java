大家好，我是大木。随着业务体量的增长，我们常常会引入缓存。总的来说，引入缓存的背景主要有这么三个一是数据量的增大，导致数据库访问响应比较慢，于是引用缓存去提升性能。第二是并发比较高，数据库压力非常大，缓存减轻数据库的压力。三是访问速度比较慢，确认缓存，提升访问速度，改善用户体验。

一是客户端缓存。对于 bs 架构的应用来说，客户端缓存主要分为页面缓存以及浏览器缓存两种。而对于 APP 应用来说，主要就是 APP 端上所使用的缓存，这是客户端缓存。


第二类是网络中的缓存。网络中的缓存主要是指代理服务器对客户端请求的数据的一个缓存，比方 web 代理服务器的缓存以及 CDN 缓存等等。第三是服务端缓存。也就是用在服务器端的缓存。如何确定使用缓存的位置？大木个人的原则是如果目前没有性能瓶颈，就暂时不要去考虑缓存这个事情，等到哪里慢的时候，再在哪里使用缓存。也就是头痛医头，脚痛医脚这样的一个原则。

从内存的位置又可以细分成堆内缓存以及堆外缓存。所谓堆内缓存，就是用 Java 堆内存来存储缓存对象。堆内缓存的优点是不需要实现序列化和反序列化，同时性能也是比较好的。它的缺点也非常明显。首先会对垃圾收集造成影响，特别是当缓存的数据量非常大的时候，可能会导致垃圾收集器停顿的时间变长。这个很好理解，在架构师一期的课程有讲过。堆里面的对象是由垃圾收集器自动收集的，数据量越大，垃圾收集器需要扫描的对象就会越多，这样就可能会造成垃圾数据器收集的停顿时间的增长。


此外，对于标记整理这样的垃圾回收算法影响可能会更大一些。因为标记整理算法分成了标记以及整理两个阶段，在标记阶段要做扫描，对象越多，扫描的时间可能会越长。而在整理阶段还要移动对象，让对象往内存的一侧去进行压缩，对象越多，移动的对象也会越多，这样造成的开销也会越大。不管怎么样说，当对应内缓存的数据量非常大的时候，会对垃圾处理器造成一定的影响。


第二个缺点是对内缓存的容量受限于堆空间的一个大小。第三，在我们使用堆内缓存的时候，一般会通过软引用或者是弱引用来存储缓存对象，这样当对应内存不足的时候，可以强制回收部分内存，从而释放对应内存的一个空间。在架构师 1 期里面也有讲过强引用、弱引用、软引用、虚引用这 4 种引用的特性，感兴趣的同学可以复习一下。
