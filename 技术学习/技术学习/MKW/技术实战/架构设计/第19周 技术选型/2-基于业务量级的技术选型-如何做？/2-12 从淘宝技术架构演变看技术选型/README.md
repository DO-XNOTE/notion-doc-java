所以在购买软件的时候，倾向于购买一款轻量级同时维护成本比较低的系统。针对轻量级，团队倾向于购买 LAMP  架构的系统，也就是用Linux、Apache , MySQL、 PHP 四件套去架构的系统。而针对维护成本，则倾向于购买一款带有源码交付的系统，这样后续可以自己维护。于是团队经过调研，选择了一款叫做 [PhpAuction](https://www.phpauction.net/) 的系统。这是一款拍卖软件。此外还搭建了一个 PHPBB 这样的一个论坛系统。这两个软件都是基于 LAMP 架构的。买完之后也不是直接上线的，而是在平台上做二次开发。比方做了交易系统和论坛系统的用户信息打通给运维人员，开发后台管理功能。另外把交易的类型从拍卖这一种增加为拍卖一口价、求购商品以及海报商品这四种。再修改了UI、界面等等。


其中做的最有技术含量的一件事情，就是做了读写分离，也就是写主库，读从库这样子。这个时期的架构图大概是这样的。其中的 Pear db 是一个 php  模块，用来操作数据库。可以发现这个架构是非常简单的。LAMP 这种架构的优点是不需要编译，发布也非常迅速。

而且 Pear db 还是挺强大的，它能做页面渲染到数据访问的所有事情。


值得一提的是，在 1.0 架构里面，团队采用了读写分离的方式去缓解数据库读的压力。但是现在多数的公司一般不会在初创阶段采用读写分离，而往往是利用缓存、 CDN 或者静态服务器等等手段，从而去降低 MySQL 的压力，后续才会去考虑读写分离。当然了，CDN、缓存等等在稍后也会出现。

另一个就是 LAMP 的组合，它广泛的用在各种中小型的网站架构上面，比方像大家比较熟悉的 WordPress 博客、 Discuz 论坛、 ECShop、商铺系统等等，都是基于 LAMP 架构的。大木在 11 年的时候有接到一个私活儿，大致是要做一个古董书藏品的电商网站，工期只有一个半月，考虑到工期太短了，报价也比较的低，从零开始做是很不上算的对吧。所以大木也是在 ECShop 的基础之上做了一个二次开发，花了半个月就搞定了。此外，直到现在依然有非常多的公司还在沿用经典的 LAMP 架构，这种架构特别适合创业公司早期在产品和商业模式的一个验证阶段，使用 LAMP，可以快速的实现产品原型，快速的部署，快速的上线，非常的高效。疑惑同学们，如果有一些需要快速验证的项目，不妨可以试一下 LAMP 的架构。

到 03 年底， MySQL 已经撑不住了。主要是因为当时采用的是 MySQL4.x 这个版本还不够稳定，流量比较大的时候经常会出现死机，于是他能考虑把数据库换成Oracle。之所以换成Oracle，主要有几个原因，一来是看中它容量大，性能好，稳定安全的特点，可以解决当时用 MySQL 的一些问题。


二来，众所周知，阿里巴巴当时有非常强大的 Oracle DBA 团队了，对 Oracle 的积累是非常深厚的，可以这样使用 Oracle 是当时团队最好的选择。但是换 Oracle 不仅仅是换一个库，它的访问方式、 SQL 语法等等都得跟着变对吧。再一个， Oracle 之所以并发能力非常的好，还是因为有连接池。而 php 它是放在 Apache 之上的，每一个请求都会产生一个新的连接，它没有连接池这种功能。这个时候团队又要做技术选型了。


当时团队找到两款 php 数据库连接池的产品，一款是 Bea 家的商业连接池，第二就是 SQLRelay。但是 Bea 的产品非常的昂贵，所以从成本上考虑，选择了 SQLRelay 的地址，在这里感兴趣的同学也可以看一下。有了 SQLRelay 之后，确实有连接池了，但是团队进入了一个新的噩梦，那就是 SQLRelay 经常会卡死。监管团队已经对源码做了很多的修改，但是还是没有办法彻底的解决问题，一旦出问题就得去重启状态。大量持续了半年左右，一直这样下去肯定不是长久之计，对吧？但大家想，数据库已经换了授课，人类的问题又解决不了，好像只能拿开发语言去动荡了。事实上也确实是这样的，

前面已经说了 MySQL 撑不住了，所以才换到Oracle。事实上使用 Oracle 也经历了好几个时代，数据最初是存在本机的，在本机存储顶不住了，于是就用了 NAS 作为数据库的存储设备，再搭载 [Oracle RAC](https://www.oracle.com/database/real-application-clusters/) 去实现负载均衡。等到 NAS 撑不住了之后，就用了 emc 的SAN存储，还是撑不住，于是我们就不得不考虑使用小型机了。到了这个时间点，项目已经用上了 IBM 的小型机， Oracle 的数据库以及 e m c 的存储。 IOE 的格局就形成了。



事实上，IOE 确实顶住了一段时间，但这个时候电商行业在国内已经开始盛行了。淘宝的流量直渐上涨，电商的人口红利也开始上涨了。随着流量的上涨，又面临着服务器和数据库的压力。从技术角度来看，团队面临着两个问题。首先是淘宝上商品的描述以及交易快照，里面的图片非常的多，迫切需要一款强有力的分布式文件系统去存储这些图片。存储初期使用的是 NetApp 公司的商用文件系统，因为省时省力。但是随着业务的增长，使用的存储系统也是在从低端到高端不断的升级。到了 06 年的时候，即使是 NetApp 公司最高端的产品，也没有办法满足淘宝网程度的要求了。第二是随着数据量的增大， Oracle 的压力也是非常大的。于是团队做了几件事情去解决这两个问题。

首先是自研了分布式的文件系统TFS，从而降低商用文件系统的巨大成本，也极大地优化了产品的体验。 TFS 的地址在这里，感兴趣的同学可以看一下。举个例子， TFS 上线之前，每个商品只能上传一张图片，大小还要限制在 120K 以内。
在商品详情里面，所有的图片必须使用外站的服务。当时发布一件商品是一件非常麻烦的事情，往往需要先把图片上传到一些在线相册或者是图窗上面，之后再在淘宝上面引用。而 TFS 上线之后，允许上传 5 张商品展示图，商品描述里面也可以使用淘宝的图片服务，所以这个提升是非常巨大的。


第二，前面使用了 ejb。使用 ejb 本身就是一个妥协的结果，随着 SUN 工程师离开之后， ejb 也就逐渐被冷落了。正好 05 06 年 spring 大放异彩，所以果断使用 spring 的 ioc 替代了 ejb 的工厂模式，精简了非常多的代码。


第三是引入缓存，从而降低 Oracle 的压力，并具提升性能。有关缓存可以分成两个时代，初期是基于 Berkely db 去进行开发的，把很多不太会变动的只读信息放了上去。而后来公司又研发了 Tair，底层可以支持两种存储引擎 MDB 以及 FDB，分别给予 Memecached 以及 [FireBird](https://firebirdsql.org/)，从而适应数据是否要持久化的需求。

数据库、服务器之间会共享资源，例如磁盘、缓存等等。当性能不能满足需求的时候，要依靠升级数据库服务器的 CPU 内存以及磁盘。数据库服务器一般会使用小型机，通过这种方式从而达到提升单节点数据库服务器的一个性能的目的。另外，你也可以增加数据库服务器的节点个数，依靠多阶段的并行和负载均衡，从而达到提升性能以及提升系统整体可用性的效果。


但是，当数据库服务器节点的个数增大的时候，节点之间的通信也可能会成为性能瓶颈。而且各个节点对数据的处理是受制于事务处理的一致性要求的。从实际的案例来看， 4 个节点以上的 RAC 集群是非常少见的。怎么办？要么把数据库服务器搬到性能更好的机器上去，要么再多买几台小型机。但是刚刚也说了，节点之间的通信也可能会成为瓶颈。这时候很头疼对吧？更可怕的是，眼下已经没有合适的商业数据库能够解决我们的问题了。你用 db2 或者其他的数据库也还是可能顶不住。这个时候它有两个选择，一是继续使用 Oracle 在现有的架构上进行优化，第二就是研发自己的数据库产品。正所谓成年人的世界不做选择题。这次的技术选型是并行的，总共走了两条路径。

DBA 和开发也是越来越有信心了，之后就用 DRDS 去做核心系统了。另外，在这个期间，淘宝走上了服务化的阶段，把整个系统进行拆分，按照用户域、商品域、交易域、店铺域等等业务领域进行拆分，陆续建立了 20 多个业务中心，比方用户中心、商品中心、交易中心等等。与此同时，团队的组织结构也发生了变化。这时候的团队主要分成了服务中心团队、中间件团队以及垂直产品团队。拆分之后，所有的业务系统必须使用各个业务中心提供的远程接口来访问，而不能够直接访问底层的数据库。而应用系统之间的调用主要可以分成 RPC 以及分布式消息。


有关 RPC，当时业界没有成熟的产品，所以团队自研了 hsf。这是一款轻量级的 RPC 框架，类似于大家比较熟悉的 DUBBO。而分布式消息内部研发了 notify 以及 MetaQ 这两款消息中间件。就目前来说， notify 已经基本废弃了，而 MetaQ 还在内部发光发热。而事实上，国内目前比较流行的 Rocketmq , 就是 MetaQ 的一个开源版。此外，在 4. 0 这个阶段还诞生了很多的服务治理相关的系统。比方分布式链路跟踪系统 ego i，类似于大家比较熟悉的 Sluth ， Zipkin 或者是 Skywalking。在阿里云上对应的产品是 aims。这是 4. 0 的加个演进。


到目前为止，从图上来看，好像已经解决了业务上的问题了，对吧？但是还是有一些问题的。比方资源问题。举个例子，一个现实的问题，随着服务器越来越多，机房可能都放不下了，这个时候应该建多个机房对吧？第二是容灾问题，一些天灾人祸在所难免，比方说发生地震等等自然灾害，或者机房用电量太高被限电，机房失火等等，都要去想办法容灾。


还有一个是扩展性的问题，有些业务不能只在一个地方部署，这样访问可能会比较慢，需要在多个地区部署。比方有些业务需要在海外开展，就有海外部署的需求。怎么样解决这几个问题？团队搞了一个单元化架构代优化架构。

简单来说，不把鸡蛋放在一个篮子里，我们把业务拆分成若干多个单元，并且部署在各个地方。每个单元你可以认为是一个机房。当某一个单元出问题了之后，也不会影响到其他的单元，只要把流量切换一下就 OK 了。


有关单元化架构在课程的第 11 周有详细的讲解，这里课上就不去展开了。值得一提的是，这些单元里面有一个是云单元，使用阿里云去交付应用，未来各个单元也都会逐步去云化掉。

纵观整个过程，我们可以发现，在淘宝发展的初期，相对更加倾向于使用商业化的解决方案，并在商业化方案的基础上去做定制。而后期，商业化的解决方案就没有办法解决需求了，他们会更加倾向于自己造技术。之所以做这样的选择，本质上还是一个成本问题。


使用商业化的方案和自己研发的一个成本大概是这样的，当项目体量比较小的时候，使用商业化的解决方案可以比较方便快速的解决问题。但是随着业务的增长到一定程度之后，商业化解决方案的使用成本就会比自研更高了。因为自己研发的软件会有更好的可控性，系统出了问题可以从底层去解决问题，系统的可扩展性也会更好。这节课就到这里，谢谢大家。
