请求打到负载均衡器上，由负载均衡器提供负载均衡算法，把请求打到多个应用服务器之上，应用服务器再去请求数据库。这节课我们就来详细探讨如何做负载均衡器相关的技术选型。


在做负载均衡器技术选型的时候，需要考虑两个问题，

首先是用什么类型的负载均衡器，第二是选择用什么负载均衡组件。先来看负载均衡的类型。目前来说，业界主要有三种方式实践负载均衡。第一种是基于DNS 的负载均衡。这种方式应该是最古老的负载均衡方式了。

首先，它把负载均衡的工作交给了 DNS 服务器，这样可以减少网站管理的维护工作。使用 DNS 负载均衡比较的灵活方便，简单易行，同时成本也比较低。第三，这种方式适用面也是比较广的，能够适用于大多数的 TCP/IP 应用。但是基于DNS 的负载均衡，缺点也非常明显。首先，DNS 一般不能反映服务器当前的运行状态，也就是某一个应用实例，即使挂掉 dns，也不会自动把实例给剔除。比方 IP3 对应的机器已经挂掉了 DNS 记录，只要在请求依然会打到 IP3 对应的机器上。


第二个问题，DNS 它是多级解析的，每一级的 DNS 服务器都会缓存 DNS 记录。 DNS 的服务器原理在稍后的章节我们会探讨到。这样某一台服务器下线了之后，即使我们把DNS 的记录给它修改掉了，比如把这条记录给删了，但是要想让这条记录生效，可能需要很长的时间。因为每一级的 DNS 它都有一个缓存，这个缓存更新的变数比较的多。


在 DNS 没有生效之前， DNS 依然会把域名解析到已经下线的这台机器上，这样就会导致访问失败对吧？所以实际项目里面如果使用 DNS 负载均衡，一般会保证 DNS 解析所指向的目标地址的一个高可用，同时目标地址也不能经常的修改。
