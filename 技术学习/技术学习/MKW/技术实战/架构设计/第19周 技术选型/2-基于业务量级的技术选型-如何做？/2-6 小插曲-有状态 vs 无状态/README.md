是不需要引入额外的组件，同时实现起来也比较的简单，只需要在负载驱动器上做一个配置就 OK 了。比方假设负载驱动器采用的是Nginx，可以在 Nginx 上面配置一个 IP hash，这样就可以实现粘性绘画了。粘性绘画的主要缺点在于它存在单点问题。比方这一台 web server 挂掉，这台机器上面的绘画就会丢失。所以在实际项目中实现粘性绘画的时候，往往还需要实现故障转移。怎么样实现?一个做法大概是这样的我们还是额外引入一个 session store。绘画不仅仅会存储在各个 web server 实例里面，也会在 session store 里面存储一份。当某个实例挂掉之后，请求会重新hash。比方 hash 到了这一台实例，这台实例一看，内存里面没有，于是就到 session store 里面获取。这样即使 web server 实例挂掉，也不会影响用户的请求。此外，粘性会话还可能会有多个实例之间不均衡的问题。比方 web server 实例接受了 n 多的请求，而另外两个实例则几乎没有受到请求。这是粘性绘画的优缺点。

主要是因为一个大型的分布式系统往往会包含很多的微服务。如果依然采用有状态的方案，假设采用了粘性会话，由于微服务架构的系统是由多个微服务共同组建而成的，而微服之间的调用很难去实现一个粘性的计算。而如果采用会话共享的方案，则意味着所有的微服务都需要使用同一个 session store。这和微服务分而致之的理念是相违背的，对吧？并且如果你的 session store 挂了，不管你的系统怎么样，分而至之都没有用，整个系统全部完蛋。

再一个，对于微服务架构的系统，如果你的 session store 想要做迁移，也会非常的麻烦，因为所有的微服都连接了，你要让每一个微服都去做修改吗？很难实施对吧？

第三是如果 session store 达到性能瓶颈，会影响整个系统的性能。

再来看一下如果采用会话复制的方式会怎样。采用会话复制就意味着在所有的微服实例之间都需要复制，会话复制的开销会非常的大，因为一个大型的系统动辄上百个微服务，上千个微服务实例对吧？

而是在用户登录的时候颁发一个token。这个 token 往往是加密的，之后用户的每一个请求都会带上token，你可以放在 header 里面传递， URL 参数里面传递，或者是 cookie 里面传递都可以。当服务器端接收到 token 之后，会校验一下 token 是不是合法，有没有过期。如果合法并且没有过期，就认为用户是登录的，否则就认为用户没有登录。你还可以在 token 里面存储一些不太敏感的用户信息，这样 token 解密一下就可以直接使用了。

但是无状态也不是万能药，一旦你把 token 颁发给用户之后，就很难去控制它的下线时间了。比方在颁发 token 的时候，设置了 token 的有效期是 7 天，但是你想在第二天就把用户强制下线，这是比较难实现的。而以前使用 session 的时候，相信大家都配置过他们tomcat session 过期时间，比方半小时，有一天你想修改成一小时，只要修改一下配置就 OK 了，非常的简单。使用 session 做强制下线等等功能也都比较的简单。这些功能在无状态的世界里面，药产时间会相对难一点。


当然了，也有一些公司在实现无状态方案的时候，并不是像这样服务器端不做任何存储的，而是把 token 也存储起来，用户请求的时候，传递过来的 token 会到数据库里面进行一个对比。不过这种方案感觉又不是那么的无状态了，倒更像是一个有状态的方案，大家说是吧？
