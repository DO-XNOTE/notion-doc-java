大家好，我是大木。随着业务体量的不断增大，数据量越来越大，业务上可能会有亿级、十亿级、百亿级数据量的查询需求，使用读写分离缓存水滴拆分水平，拆分之后可能依然顶不住。与此同时，业务上也常常会有全文搜索的需求，这个时候，我们会把这部分数据给拆分到全文搜索引擎里面去。


这样一来减轻了数据库的查询压力，二来在超大规模的数据量下，搜索引擎的查询效率普遍会比关系型数据库要快很多，从而有助于提升应用性能，提升用户体验。目前来说，有 5 种路径可以实现全文搜索。第一是使用数据库内置的全文搜索能力第二，使用和数据库深度集成的全文搜索第三，使用开源的全文搜索引擎。第四，使用商用的全文搜索引擎。第五字眼全文搜索引擎。下面我们来逐个探讨。先来看一看第一种方式。很多关系型数据库自身就支持全文搜索的能力。

比方 MySQL 的InnoDB、 MyISAM 都是支持全文搜索的。在早期， MySQL 的全文索引不支持中文，需要使用 mysqlcft 之类的插件去实现中文支持。而从 MySQL5.7.6 开始，官方推出了 ngrok 自带支持的中文。又比如 PostgreSQL它的 gin 索引也是支持全文搜索的。 gin 索引也不支持中文，需要额外安装 zhparser 或者是 pg_jieba 之类的扩展才行。

还有第三，由于搜索引擎和数据库是独立的两个软件，所以数据库和搜索引擎之间的数据如果保持一致也是一个问题。


有关这一点，主要有两个玩法，第一种是借助比如Logstash、input、 JDBC 之类的插件进行同步，第二种是应用在写入数据库的同时，也写到搜索引擎。第一种方式可能会有一定的延迟，第二种方式虽然一致性比较的好，但是会影响写入的性能。不管怎么说，数据同步总归是我们要考虑的问题对吧？OK，选型建议是可以放心使用，因为这是业界最主流的方案。
