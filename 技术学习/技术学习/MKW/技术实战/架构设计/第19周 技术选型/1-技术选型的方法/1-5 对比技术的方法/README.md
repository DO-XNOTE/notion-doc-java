很好理解。如果使用某项技术做出来的系统，维护起来非常困难，尽量不要去使用，尽量是比较保留的说法。括号里面是大漠的心声，关于这一点，大漠是有切肤之痛的。之前团队所在的项目，在实施一个中型项目的时候，选用了兄弟团队的低代码开发平台去开发项目。低代码开发平台有点类似于q**轻流**，

[https://qingflow.com/](https://qingflow.com/)

下面不妨使用清流进行类比，给大家看一看什么是低代码开发平台。使用了低代码开发平台之后，大部分的代码都不需要自己写了。对于前端，只需要在平台上拖拽组件就可以进行开发了。比方我们想开发一个表格，拖拽就 OK 了。想要添加字段，比方单行文字可以填入一些配置，就可以实时渲染，点击保存。而业务流程则使用跨流程度的方式进行编排，比方添加一个分支，再添加一个流程等等流程图无法实现的业务就 OK 了。


使用了低代码开发平台之后，我们的开发效率能够提升数倍。但是用了一段时间之后，我们发现低代码开发平台并不适合我们的项目，原因是在我们的低代码开发平台里面，一个技术功能发生变化，可能会引发数支柱的修改。而我们项目的需求变更也是比较频繁的，这导致需求一旦发生变化，我们就得到处修改。


此外，我们的项目也是比较复杂的，有些功能很难用流程图的方式去进行编排，而且即使编排出来了，执行的效率也是比较差的。这部分业务我们需要完全使用代码去实现。编造好之后，我们再把代码贴到平台里面去执行。这个项目的最终结果是开发人员怨声载道。到最后项目选择了重做，原因就在于在做技术选型的时候，没有考虑到项目后期的可维护性。


这里说明一下，个人并不是低代码开发平台这条路有问题，而是我们的项目不适合用低代码开发平台去做。事实上，我们也有很多相对简单的业务系统，在低代码开发平台里面运行的很好。事实上，低代码开发平台也是目前业界的一个热门和技术趋势，同学们也可以关注一下，这是可维护性。

也就是外部的机会正好是我们的内在优势。这个时候我们应该赶紧运用自身的优势把握住机会，对吧？这个也叫做 SO 战略。而劣势和机会组合，说明外部的机会是我们的劣势，我们应该弥补或者改善自身的劣势，并去把握好现有的机会。这个叫做 WO 战略。而优势和威胁去组合，说明我们具有一定的优势，但是存在着外部的威胁，应该运用自身的优势去规避掉威胁，这叫 ST 战略。


最后，威胁和劣势组合在这个方向，既是威胁，又是你的劣势，应该努力改善自身的劣势，同时还是想办法规避掉威胁。这是 WT 战略。那么简单来说就是一句话发挥优势，把握机会，改善劣势。规避威胁之后，再把你的结论填到之前的表格，填完之后，

团队应该增强哪些工作，哪些地方需要规避风险，哪些地方可以再去发挥优势，都会比较的清晰了。


可以看到 SWTO 分析法，它比较适合用来做一些更上层、更加抽象的技术选型。比方确定未来公司 5 年的技术方向，是用云原生的技术体系，还是继续使用传统的技术体系等等。值得一提的是， SWTO 是一个比较通用的分析框架，


它的适用面是非常广的，它不仅仅可以用来帮助我们做技术上的一些决策，像企业的战略规划分析，竞争对手个人的职业规划等等，都可以借助 SWOT 来帮助我们去分析。
