在这个阶段，首先要搞清楚两个问题当前遇到的问题是什么？是项目性能不佳，还是原来的技术方案满足不了业务需求，需要去做改造。技术选型肯定是要解决一定问题的，首先搞清楚当前的问题。第二，技术选型的目标是什么？是为了提升开发效率，提升性能，降低成本，或者是解决特定的业务问题。定好你的目标，如果有多个目标，可以排一下优先级，找到最为迫切的一两个目标。只有明确了目标之后，才有评判你做技术选型的一个标准。

明确目标之后，就可以做一个深度的技术调研了。在这个阶段，我们首先应该去识别是不是有必要采用额外的技术。在实际项目中，很多时候我们会为了解决问题而引入额外的技术，然而真的需要吗？举个例子，某一些团队发现数据库查询非常的慢，于是就引入了 mycat 去做分库分表，或者是引入一些 NOSQL 数据库，从而优化查询。但其实它们的数据量可能并不是特别大，只要经过一定程度的 SQL 语句优化就能够解决问题了，根本没有必要额外引入技术组件。还有的团队看到市面上一些技术方案非常的优秀，就生搬硬套在团队里面去实施了，结果实施出来的结果也是不尽人意。这说明有的时候技术就已经是一个伪需求。


当然了，在你在引入额外的基础组件以及不引入额外的基础组件之间做选择，本身也是一种基础选型。回答正题要知道，引入任何技术都是有成本的，比方学习成本、运维成本、团队适应成本、引入组件带来的项目复杂度等等。一般来说，如果能在现有技术的基础上能够想办法实现目标，就不要贸然的去引入新的技术了。这一点也是契合了奥卡姆剃刀原理的。正所谓如勿必要，勿增实体，这样才能做到简单有效。


顺便提一下，实际项目中，如果已经使用了技术a，而技术 b 和它解决的问题基本上是一致的。这个时候除非项目中确实有必要，否则就应该考虑在团队里面只使用 a 或者只使用b，而不是两个都去引入。还是那句话，引入任何组件都是有成本的。如果认为确实需要引入新的技术，就需要找到候选技术。也就是根据前面整理的问题和目标，去发掘有哪些技术可以去使用。
