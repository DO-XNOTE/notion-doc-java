第一，不尊重需求。任何技术决策都是为业务去服务的，满足需求是一切的前提。如果需求都不能很好的满足，接着采用了更好用的技术，做了再牛叉的技术方案也是白搭的。这个道理非常的简单，但实际上有很多的架构师在做技术选型的时候，却过于站在技术人员的立场，导致业务需求都不能很好的满足。希望各位未来的架构师们在做技术选型的时候，一定要立足业务，用技术服务好团队的业务。


第二，面向流行趋势编程。很多企业喜欢盲目跟风，什么流行上什么，觉得随着技术的流行趋势去选用技术准没错，但是却不去考虑是不是适合自己。大木把这种现象称作是面向流行趋势的编程。举个例子， 16 年那会儿，业界对于微服务基本上是一个吹爆的状态。各大技术网站、各种技术盛会，分享的都是项目原先怎么痛苦，采用微服之后怎么顺利。于是有一些企业盲目跟风，他们使用微服去架构系统，但其实他们连自动化的部署机制都没有，依然是人工运维。将项目拆分之后，开发人员是相对爽了一些，但却搞得运维叫苦不迭。好的是，这几年一直对微服的态度理性了很多。比方这篇文章。他说自己团队经过调研之后，认为微服不适合自己的项目，于是继续沿用了单体应用架构。记住，再流行再主流的技术或者是架构，也无法满足所有团队的需求。


第三，面向牛叉编程。也叫面向简历编程。有一些架构师在做技术决策的时候，不去结合团队当前的情况，而是什么牛叉用什么。至于引入之后解决了哪些问题，开发人员能不能 hold 住，会不会踩坑，用法是不是最佳，实践却不是很关注。他们更加关注的是自己在项目里面用到了什么牛叉的技术，并且积累了什么牛叉的项目经验。这样自己的简历更加的亮眼了，以后跳槽可以拿到更高的薪水。这种现象并不少见，我把它称作是面向牛叉编程。听这门课的小伙伴可能已经成为了架构师，或者即将成为架构师，希望大家不要去走这条路。

第四，过度考虑。很多人喜欢过度考虑，不可否认，我们都希望自己的系统 低耦合 和通配性好，可以完成任意的后续功能的组合。但是这种考虑也要有个度，过分考虑通用性、灵活性、可扩展性往往会导致工作量以及复杂度的猛增，特别是在需求变化日益频繁的今天。业务的发展可能和你最初的预想很不一样，你的过度考虑很可能都是无用功。目光长远归为了考虑是好事，但是看得太远可能不是好事。况且你看的还不一定准。架构是逐步演进的，不是一蹴而就的，你看到的往往不是终局。建议大家在考虑通用性、扩展性、灵活性的时候，**应当尽量的把额外增加的工作量控制在一定的范围内。**


第五，把看到的当事实。现在网上有很多的基础文章，里面携带着各种观点，这些观点非常的主观。比方一个 RxJava 的粉丝会跟你说，RxJava 有多好，设计思想有多么的优秀，开发起来有多爽，用它写出来的代码有多么的优雅。它不会告诉你。 RxJava 本身的学习曲线是多么的陡峭。而一个不喜欢微软的程序员，可能会把微软的技术体系说得一无是处。而一个吹爆某项技术的文章，搞不好其实只是一篇软文而已。千万不要把看到的内容直接当做事实，这样太傻太天真了，对吧？在做出你的技术决策之前，还是要亲自玩一玩，验证一下，这样才能有一个相对客观的判断。以上是大家做技术选型时候的常见误区，怎么样正确的去做技术选型？请听下回分解
