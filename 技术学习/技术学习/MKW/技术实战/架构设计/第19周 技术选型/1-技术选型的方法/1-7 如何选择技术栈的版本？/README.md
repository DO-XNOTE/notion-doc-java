大家好，我是大木，前面已经探讨了如何去做技术选型。宏观层面的技术选型一般是没有版本的概念的，比方恢复 1.0，恢复 2.0，没有这种东西对吧？但是如果你做的技术选型，选择的是具体的技术栈，选择好用什么技术还不够，还得确定好用什么版本。这节课我们就来探讨如何选择版本这个话题。软件版本是很多技术人员会忽视的东西，但是作为架构师还是应该重视起来。

选择不同的版本可能会有使用上的差异，比方 angular 1.x 和 angular 2.x 使用上的差异非常的大，哪怕认为这是两个技术框架都不为过。


第二，选择不同的版本也会有功能上的差异。现在版本往往有一些新特性，比方 jdk13 就引入了文本块，你这么玩就 OK 了，再也不需要像以前一样用一堆的加号去拼接字符串了。再一个使用不同的版本，在进户升级的时候也会有所差异。版本的升级也是我们需要考虑的问题。比方项目选择了 spring boot 1点x，后续想要升级到 spring boot 2 点x，这是一件非常痛苦的事情。


尽管官方发布了升级的教程在这里，

[https://github.com/spring-projects/spring-boot/wiki/spring-boot-2.0-Migration-Guide](https://github.com/spring-projects/spring-boot/wiki/spring-boot-2.0-Migration-Guide)

但是不得不说改动是非常大的，我们需要按照教程一步一步做下去。这个里面需要改很多的配置，甚至一些场景下还需要改代码。改完代码之后还不够，我们还需要做大量的测试才能支撑好我们的升级。而从 spring boot 2.0 升级到 2.1,   2.1 升级到 2.2 等等，就是非常轻松的，基本上改一下版本号就升级完成了。


说了这么多，大漠还是想要强调版本的重要性。怎么样选择版本？就目前来说，业界没有统一的标准，不同的人选择版本的偏好也是不一样的。这节课大木酱会给出几个我选择版本时候的原则，希望能够给大家一个参考，同时也希望能够带给大家一些启发。

 cglibrary no dependence 指定一个版本等等。这种方式版本由各个微服务自己去管理，如果操作不当，可能会引入不兼容的版本，对吧？比较好的做法是使用BOM 。BOM 的全称是 Bill of materials 物料清单，它定义了一整套价包版本的集合，从而让应用引入价包。依赖的时候，只要引入BOM，就可以放心地使用依赖的包了，不需要自己去指定版本号。这些 jar 包的升级都会由 BOM 的维护方去负责，并且 BOM 的维护方会保证定义的价包版本之间的兼容性。


如果大家对 BOM 比较感兴趣，可以看一下这篇文章，里面详细探讨了 BOM 的定义与使用。

[https://blog.csdn.net/LoveJavaYDJ/article/details/86594226](https://blog.csdn.net/LoveJavaYDJ/article/details/86594226)
