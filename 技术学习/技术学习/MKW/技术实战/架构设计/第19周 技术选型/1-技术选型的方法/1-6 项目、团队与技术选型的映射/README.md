从项目周期的角度来讲，可以分成短生命周期以及长生命周期型项目。如果项目的生命周期比较短，建议优先选择门槛低、简单易上手、开发速度快的技术。在开发的时候，代码也可以写得相对自由一些，甚至可以不强制大家遵循各种最佳实践，比方 code review、Checkstyle、代码规约、扫描等等，从而进一步的提升开发速度。这是因为项目一旦结束，代码就会直接被抛弃，精雕细琢的意义并不大。
总而言之，一句话，超快锰是比较适合短生命周期项目的一个选择。当然了，如果能够做到精快锰就更好了，对吧？OK，对于长生命周期的项目，

从项目的地位来说，可以把它分成边缘性项目以及核心项目。边缘性项目由于不是项目的核心部分，因此影响面也会相对比较小，哪怕出一点小的故障，一般也是能够忍受的。这种项目是比较适合作为实验田的，可以用来尝试一些比较新颖的技术或者方案，从中积累经验和教训，以便于之后在核心的项目上面逐步采纳。像早几年阿里去 ioe 的时候，最初也是用 MySQL 搭载 pc server。先在一些边缘型的项目上面试水，踩过坑，积累过足够的经验之后，再逐步修改核心应用系统的。

比方，边缘性项目 1 使用技术 a 项目2、使用技术 b 项目 3 使用c。这叫多线作战，全面开花。这种做法大部分认为在多数场景下是不可取的。这是因为即使是边缘性的项目，也需要有人去维护，你得有这样的人才。使用的技术种类过多依赖会给人才供应带来挑战，二来也不利于企业去做技术沉淀。

核心项目应该稳定优先，做技术选型的时候应该相对保守，优先选用比较成熟，同时团队已经积累了足够经验，并且有比较好的技术支持的技术。举个例子，在阿里内部，针对 MySQL 以及 MongoDB 有非常丰富的积累，出了各种问题都可以很方便地解决，并且还有非常棒的基础设施去支撑。


想要创建新的数据库，设置分库分表的规则分析、慢查询查询性能优化、表结构调整等等，都有非常方便的工具去支撑。想要做这些事情的时候，往往鼠标点就 OK 了。 开发人员不需要考虑太多，关注好自己的业务就 OK 了。因此，对于核心项目就会比较倾向于使用 MySQL 或者是MongoDB。


而对于其他的数据库，比方目前大热的 Clickhouse 核心项目，在做选择的时候就会非常的慎重，除非内部经过严密的调研认为确实有必要上，否则的话一般是不会采纳的。而且即使认为有必要，初期一般也会先在边缘型的项目上试水，而不会直接在核心项目中使用好。

这里说明一下，如果从项目的生命周期来看，核心项目几乎必然是长周期的，可维护性同样也是重中之重。而边缘性项目则可能是短周期的，也可能是长周期的，对吧？同学们在评估项目地位的同时，也可以根据项目的生命周期去综合考虑。

探索型项目初期往往希望迅速出成果，把项目投入到市场，快速进行试错。如果试错不成功，可能就会成为一个短周期性项目了，在比较短的时间内就会把它废弃。而如果经过市场验证发现大有可为的话，那么就会成为一个长期项目，甚至有可能把它晋升成核心项目。


总之，探索型项目拥有太多的不确定了。因此，大木建议在为这类项目做技术选型的时候，应该既要做的快速，也要考虑到可维护性。此外，如果没有明确的规划，在做技术选型的时候，可以优先保障系统的简单性，不做太多的预留。

再来从团队的维度进行探讨。有关团队可以从技术实力、团队规模以及组织架构这三个角度去思考。

首先，定好战略方向，战术方向以及技术方向。

第三，可以制定好团队的技术规范，让大家知道什么是不允许做的。比如禁止使用多个微服务共享一个数据库的方案，又比如远程通信必须使用轻量级并且能够跨平台的协议等等。既能够避免大家犯方向型的错误，也可以尽量帮助团队尽早的发现问题。

康威定律说过，组织的沟通方式会通过系统的设计表达出来。项目的架构其实是团队沟通协作的方式而产生的一个结果。技术选型同样受到康威定律的约束。通俗的说，想要什么样的系统架构，就得搭建什么样的团队，而有什么样的团队，就会搭建出什么样的系统。举个例子，如果你的公司前后端是独立的，两支团队很可能会选择前后端分离的技术方案。而如果前后端都有一个人去搞，不去做分工，很可能不会做前后端分离。也就是你们拥有了前后端分离的团队，于是使用了前后端分离的技术方案。反过来，如果你们想要搭建前后端分离的系统，就得组织前后台非遗的团队。


大家在做技术选型的时候，要去结合当前团队组织架构的特点，要考虑好你所选用的技术在最终技术架构中的位置，以及与当前团队沟通结构的一个匹配程度。否则即使你的团队采用的是一项非常先进的技术，也可能会翻车。OK，以上是团队和技术选型的一个映射，
