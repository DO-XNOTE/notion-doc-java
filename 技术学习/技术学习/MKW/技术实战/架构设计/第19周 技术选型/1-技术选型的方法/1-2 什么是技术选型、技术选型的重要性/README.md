它有狭义和广义的两个概念。狭义上的技术选型是指你们团队选用哪种技术去解决英文问题。比方选某个技术语言、某一个技术框架去开发项目。而广义上的技术选型则泛指项目实施过程中的各种决策。比方某一次需求里面，你们设计了两套技术方案， a 和b，选择了其中的一套。可以这样，在项目实施的过程中，我们所做的每一个技术决策都是技术选型。项目的整个生命周期都会伴随着各种的技术选型，是非常重要的。举两个真实的案例，考虑到企业的名称可能比较的敏感，这里我们就不提具体是哪两家公司了。

第一个案例，某一个 c 轮的跨境电商企业的架构师，他在网上看到 service mesh 的各种好处之后，决定在项目中引入Istio。并且他们引入 Istio 的过程是非常激进的，前期没有做太多的技术调研，直接在项目的开发环境中部署了Istio。但是由于团队并没有这方面的人才，在真正引入这门技术之后，才发现团队不能很好地 hold 住这门技术，一旦遇到问题，项目的开发进度就会延迟。经过很长一段时间的适应期之后，才敢把这套新的架构上线。整个过程中浪费了很多的时间和人力成本。在这个例子里面，问题的严重性可想而知。例子也告诉我们，在决定采纳某个技术之前，一定要做好调研，并且可以尝试先小规模的引入，并积累了相关的经验，经过一定的验证之后再去正式采用。


假设我是这家企业的架构师，由我来做技术选型，我会先搭建一个独立的环境，并去研究 Istio 的功能特性、常见问题。登记了一定经验，有能力快速解决各种问题之后，再逐步的把各个业务系统接入到平台上面来，从而降低风险。
