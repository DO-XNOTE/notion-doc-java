快照策略相对于备份来说更加迅捷快速，在一个存储设备里有磁盘a，这个磁盘 a 就是我们的数据库磁盘，或者是我们的文件系统盘。当有变化发生的时候，我们可以尝试对它进行一个快照，或者在指定的时间点，比如每天早上 8 点，每天晚上 8 点进行一个快照处理。那这样情况下我们在什么？在这个快照之后的所有改动，如果发现有问题，有人黑客进来了，做了一些删库删表的操作。或者由我们的 DBA 做了一些误操作，我们随时可以恢复到快照当时的数据，但是不是因为快照技术而会损失很多数据呢。当然当你回退到某一个快照的时候，也就意味着这个快照以后的所有数据你就丢失了。有没有更好的防御手段呢？当然有，叫 CDP 连续数据保护。当用户有任何信息变更的时候，它都会触发一个后台快，照这样它后台时时刻刻在生成新的快照，无数多份快照，这就是连续数据保护。


当你想恢复到某一个时间片，某一个时间点，你只要告诉存储设备或者是连续数据保护的工具，我要恢复到几分几秒，它就能够全自动地找到对应的快照，把数据恢复到你那个时间点了。但是大家想一想有什么样的坏处啊？是不是连续数据保护在什么？我们的数据存储和应用之间多了一道门槛，所以导致什么有性能的消耗，减慢我们存储的读写速度。除此以外，因为大量的快照需要保存大量的差异数据，所以对那些数据变化量比较大的数据库，我们可能占用好几倍的存储空间，所以各种方法是有弊也有利，但是传统企业或者是很多云平台用的最多的保护技术是什么？当然是备份技术，因为快照和连续数据保护都有什么厂商依赖，但是备份技术是一个通用技术，你可以把数据备份在磁带，也可以把数据备份的对象存储，只要符合一套标准的数据备份策略和技术框架就可以了。

那这个存储设备有些是天然可以支撑什么无服务器干扰或者无服务器参与的全自动数据备份，它天生可以支持把数据备份到代扣。那也有的是要什么要通过服务器的支撑来把数据读出来，然后再写入磁带系统，这样都可以。


总体来看就是我们很容易把数据库文件系统的数据塞到一个磁带里去，但是这是远远不够的，因为什么我们大部分时候，什么我们写完这个带库其实就不读了？当真正去把这个袋裤拿出来，把这个磁带，是吧？重新用机械臂读取的时候，我们会发现各种各样的故障，所以在一个真正的大型的企业里面，我们都会怎么做？会时时刻刻准备好一个什么恢复存储，这个时候我们左边在跑压力，我们的压力通过浮起和光纤网络把压力压到了我们的实际的生产存储上，但是后台在 total model 进行数据备份，同时每天都在尝试把备份的数据挑拣一些袋子，然后恢复到另外一台存储设备，然后通过独立的恢复光纤网络、散网络和恢复服务器来进行数据的恢复和校验。


那这样的恢复操作什么是每年一大列？这个一大列就是要对整个企业的所有数据做一个完整的演练，每季度一小列，这个小练是抽取某一个词语或者某一个域来进行什么功能和数据的演练。还有每周什么？每周一小册，我们挑一两个磁带，是吧？拿到什么会备份恢复的这个数据库和备份恢复的存储上面尝试进行恢复，然后从数据库里读出数据，看看我们最近的几天、几小时的数据是不是都能恢复出来啊？这就是我们平时经常说的，什么备份和恢复要双管齐下才能真正实现逻辑保护，这是什么？是一个备份恢复的物理部署图、物理架构，我们来看一看有什么样的逻辑架构在后台支撑我们数据的备份和恢复呢。

第二天又生成了一个新的PB，我们这里用一个大块表示新的数据，那第三天又生成了一个PB，那到第四天我们就有 6 个 PB 的数据了。那后台对付于这样的数据库，我们应该如何备份呢？那一种备份方法叫什么？叫全量备份，也就是第一天我备份3PB，第二天4PB，5PB，6PB，每天递增。大家想一想这样的备份策略有什么样的好处啊？一旦需要恢复的时候，我就可以找到那个天数备份的代库，直接从那个袋库里面的磁带里面直接把当天的数据完全读取出来，所以恢复时间很快。但是是不是我的磁盘或者磁带的容量每天都在增加？这样我的 it 的开销在增加，然后存储的空间我还准备很大的空间来摆放这么多的袋子，是不是很麻烦？那有没有一个稍微轻量一点的解决方案呢？当然有，那业界最流行的叫增量备份，增量备份的第一天其实跟前面一样，也是要做一个全量的，这是一个基准，这是一个基石。然后你每天只备份新增加的或者是修改过的那部分数据。


第二天我们就备份1PB，第三天还是备份1PB，第四天还是备份1PB，这种方式好省空间，我通过 5 个 PB 就可以随意的恢复到什么任何的前四天中的任何一天，同时我的恢复速度怎么样？很糟糕，为什么？假设第一天蛮好，是吧？ 3 个 PB 一下子就能恢复出来，但是第四天怎么办？我先要把数据恢复到第一天，然后我要追加EPB，第三天我要再追加EPB，第四天我再追加EPB，这样完整流程走完以后，我才能恢复出第四天的数据，所以增量备份花费的什么是恢复的世界？而我们一个 it 企业当出现灾难，当出现人员故障，当出现黑客攻击的时候，最希望的是什么？是能快速恢复，所以增量备份影响了恢复的时间，并不是一个最优的解决方案啊。


有没有在全量和增量之间有一种比较适中、比较中庸的解决方案呢？也有，这就是非常常见的差量备份和差量恢复。第一天，是吧？还是老样子。我们备份 3 PB 的数据，第二天我们跟增量一样只恢复或者只备份1PB，到了第三天我们就备份两 PB 了。那为什么要备份两 PB 呢？差量有这样的说法，我每次都是跟前一次全量进行比较，那前一次全量我们是备份了 3 个PB，那到了第三天我们追加的其实是两个PB，那同样道理，到了第四天，我们第四天跟第一天相比是不是追加了 3 个PB？所以我们要备份 3 个 PB 相对于增量了我们的什么磁盘使用率，或者说我们的磁带的空间占用量稍微提升了一些，但是我们恢复时间就很快，我们只要恢复到第一天，然后把这个差量一追，我就能直拿大脑。


你想恢复到第四天和想恢复到第三天都是非常快捷，通过插量我们可以实现相对快捷、相对省空间的备份方法。那是不是所有企业都在用插量备份啊？答案是否定的最常用的方法，飞扬老师可以这样给大家介绍一下，就是每月一全量，比如每月初我们做一次全量备份，然后每周一差量，就是每一周我们都跟第一个周，也就是一个月的月初的那个时间点进行一个差量的比较，然后备份，然后每天一增量，这个增量就是增在前一天上面。
这样的方法我们可以使尽量快速地恢复我们所有的数据，同时我们占用的磁带的空间并没不是那么大，但这是比较什么？比较非关键的系统，也就是边缘系统，真正核心系统我们有可能还是采用全量备份，每天都在备，也有可能我们会采用什么，会采用一周全备，每天增量的方法，所以方法很多种，根据我们业务的重要性，根据我们数据恢复的能够等待的时间长度来决定不同的备份策略。
