这适合于什么？适合于 a 和 b 之间的。我们私下进行沟通的时候，我怎么验证中间数据没有被篡改呢？我们两个人什么是可以通过这样一种对称密钥的签名方式？ h net 来进行处理？好 h Mac 四个字符什么含义我们先不讲，我们看看它的处理过程。


首先拿到一个铭文，这个铭文可能是几 KB 的一个 word 文档，也可能是几兆币的一个照片，或者是几 GB 的一个视频，不管是什么，我们直接在这个文档后面加上一串对称密钥。那什么是对称密钥呢？对称密钥就是在 AES 或者 DES 过程当中应该如何进行替换？应该如何进行轮转？是先替换还是先轮转？是先轮转还是先替换？他们有一串先后顺序，以及怎么样替换，怎么样轮转。这样一些详细的内容，我们写在一个小钥匙里面，这就是对称密钥，我们用这个钥匙触发整个 AES 的加密算法，就能把明文变密文，我们再重新再触发一次这个加密算法，就很方便地又把密文变还回了明文。所以在加解密过程当中是用的是同一把钥匙，这就是对称钥。好， a 和 b 现在是什么？是互相信任的双方，所以他们都有这把钥匙，所以 a 要做的就是把明文后面紧挨的就是把密钥贴上去。


然后让穆赫霍尔摩斯帮我们进行一个单向散列，所谓单向散链大家应该很熟悉的，就是什么 MD 5 还有呢？哈希算我们这次采用 SHA 哈希算法生成一个假设是 256 位的这个消息身份验证码 message authentication code m a C。好，用，哈希来生成MIC，这就是 HMI seed 成名的原因。
哈希 MIC 生成完了以后，这个 MIC 跟什么？跟我的这个整个文件打包成一起，然后发给b，那 b 怎么验证中间没有被篡改呢？很简单一样的，就 b 也有这个变成密钥，是不是跟 a 一样，把对称密钥放在这个原文下面，然后重新做一次？哈希，做完以后验证一下底下的这个 a 发过来这个 Mac 的验证码跟我自己算出来是不是一致啊？如果不一致有两种可能，一种是上面的文字被改了，一种是下面的什么签名被改，不管哪种被改就证明什么，这中间存在黑客攻击。


所以我整个文件是不信任的，如果没有被改，就证明了整段文字确实是 a 发给我的， a 和 b 之间、内部之间进行安全保密的沟通，这个时候要验证完整性，推荐就采用 h m a C 把对称密钥加在文件后面来进行单向散列的方式生成我们的签名。好，那这个 MSA 就是我们在对称加密过程当中的一种签名方式，


还有一种签名是更加常见，就是 a 和 b 之间不是互相信任。比如说 a 要把自己的文件发给当地所有的地区的人民，可能是自己的公司的人，也可能是企事业单位，也可能是一些安全监管机构。
