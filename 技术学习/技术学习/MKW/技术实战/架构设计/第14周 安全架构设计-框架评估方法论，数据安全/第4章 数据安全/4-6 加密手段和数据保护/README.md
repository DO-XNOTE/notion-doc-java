在讲加密决策树之前，我们先要聊一个话题，是不是所有数据都要加密？这个答案是否定的。大家还记得那个金字塔吗？四层结构里面最下面一层是什么？是公开公共的那个数据，这些数据通常是不用加密的，只有什么用户相关的数据，我们公司的内部数据以及跟支付相关的机密数据，我们才需要进行加密的过程。


当中第一个问题就是你的这个加密是否可逆啊？所谓可逆就是什么？我的铭文可以变成密文，那密文能不能还原成铭文呢？

多数情况下都是可逆的，但是也有一个例外。什么例外？我们的密码等等这些系统。只要用户输入完密码以后，我们会把它变成一个密文，但是下次用户再输入的时候，我们不会把密文还原成明文来比较，而是怎么样，而是把它的明文重新再变成密文。


然后比较一下我们原来磁盘里的密文和这次用户输入的铭文变成的密文是否一致，这种做法大家应该非常熟悉，就是单向散列，比如像 MD5 或者是 Sha 就是哈希算法。哈希和 MD5 是最常见的单向散列，来实现不可逆的这种数据的加密形式。这里要强调一下，我的整棵树里面，红色箭头表示否，绿色箭头表示是，也就是是否可逆啊？不可逆就是用散逆。那如果要可逆就进入下一个问题，你的数据是不是要保证完整性？也就是没有用户能够在里面进行串代，也要保证说我的这个用户的这个实际的信息，是吧？跟原来是完全一致的，这种情况下，如果你要实现完整性，也就是绿色箭头，我们就要采用数字签名技术。就如同前面聊的一样，熟人和熟人之间可以采用 h Mac 的签名技术。和陌生人之间我们通常采用 RSA 加 Mac 的这种签名技术来保证数据的完整性。


如果你不需要保证数据完整性，或者是已经通过数字签名完成了完整性的验证，这个时候的剩下的那部分内容该如何做呢？也就是红色箭头，我们进入第三个问题，你这个所要加密的铭文是不是一个小文件？所谓小文件就是几字节、几 k 字节、几百 k 字节，甚至于几兆字节都可以，但是几百兆、几g、几个t，那就不是小文件。


假设我的那个内容不是小文件，那这个时候我们进入红颜词否的分支，这种情况下只有一个方法是和对称加密，只有对称加密才是能最快速的处理那种大文件，而且是可逆的加密和解密，这对称加密里可以是前浪DES，也可以是后浪AES，大家可以自由选择。如果恰巧我又是一个小文件，比如几 k 的一个合同了这种此类文件我该如何进行加密呢？进入第四个问题，对方是不是自己人？所谓自己人通常就是自己公司的人，或者是跟我拥有同样的加密密钥的人，如果我的密钥可以共享给他，他也拥有我的密钥，这种人就是自己的，对于自己人怎么处理啊？当然是绿色箭头，我们采用对称加密，也就是前面的DS、 AES 这些对称加密，因为对称加密更经济、更快捷。如果不是自己的人，也就是他拿不到跟我一样的密钥，这时候我们就不得已而采用非对称加密了。我们怎么样？我们可以拿它的公钥进行加密。这个时候我把这个文件放在任何地方，别人都打开不了，我把文件发给他，他可以拿他自己的私钥来进行解密，这就是我前面举的什么飞扬老师跟慕课之间分发合同的一种方式。


这样一条路线是不是大家已经能够大致感觉到，尽量应该规避非对称加密的使用，多采用像散链这种比较快的，多采用像数字签名这种相对快的，以及对称加密这种对大文件处理，对复杂文件处理非常快的机制，而减少非对称加密的使用。聊完了这个加密决策树，这其实是一个总体思维，我们看一看这实际的语音平台，实际的数据库里面我们平时是怎么加密的？

这里我要隆重推出另外一个名称，叫做信封加密。这么多名称是不是太烦了？那信封加密呢？我简单来说就是这样，当你去加密一个内容的时候，你是不是有一个密钥啊？那密钥该怎么管理呢？我推荐是把它放在信封，然后给信封再上一道锁，再用一个密钥，也就是用密钥管理另外一个密钥的加密方法，这就是信封加密。那哪些地方在用信封加密？谷歌云里面所有的内容都规定必须全盘采用信封加密，也就是说它的所有数据全部是采用信封加密的。


亚马逊有一个推荐选项，可能是不加密，但是大部分的用户都会默认会什么采用它的另外一个选项就是可以支撑的全盘加密，因为这个加密是免费的哦。那我们的其他语音其实也有类似的，有的推荐说不加密，有的推荐说加密。但是基本上只要是加密都会支持信封加密，也就是用主密钥来管理数据密钥，用数据密钥来管理数据，那如果是我们的数据库，数据库里面经典像 Oracle 在 12 以后， Oracle 版本 12 以后，它号称叫12C，为了云而诞生的， Oracle 默认就是全数据库，底层全盘加密。如果是像 DB two，它支持像存储过程和触发器里面进行什么全盘加密。如果我们用 MySQL 怎么办？ MySQL 有可以应用层上用 encrypt 和 decrypt 进行应用层加密，也可以采用 MySQL 的变种 Merrid b，它支持表空间级的加密。


这么多加密手段，它的实现五花八门，但是比较漂亮的就是仿照我们像亚马逊，谷歌，像阿里云来实现信封加密，我们看看怎么来做。这是蓝颜色，就是没有加密的文件，红颜色我们表示加密完的文件好不好？蓝颜色文件我们先要找一个密钥，我们可以是对称密钥，也可以是非对称密钥。但是如果这个文件比较大，那我们通常都是拿一个对称密钥，然后对这个文件进行加密，加密完了以后它就会生成一个红颜色已加密文件。


这个时候我的妈，我这个文件是可以落盘的，我这个密钥当前还在内存里，假设我这个数据库崩溃了，比如 Mera db 我崩溃了以后，是不是我的密钥丢失了？这样我的密文以后再也读取不来，这种情况是要规避的。怎么规避啊？我们找一个主密钥，这个主密钥也可以看主密钥的尺子好像有点，这个齿轮有点多，对吧？我们这个数据密钥，我用这个量尺的这个卡通形象的钥匙来代替。主密钥通常是一个主密钥本，还不是一个密钥，是一个一密钥本。后面我们会说为什么是密钥本？主密钥可以用来加密我们这个两个尺的卡通形象，这个数据密钥怎么加密呢？就是采用对称密钥，比如说 AES 加密完了以后，把这个新的信封里面的密钥什么也放在同一个数据库里面，所以这个数据库里面，比如 My AI d b 里面既有我们加密的数据信息，也有我们加密的这个密钥信息。


这个时候整个数据库如果是重启一次没有关系，内存里面的那个原始的那个密钥没有了，没有关系，我完全可以依然从数据库里读出这个红色的加密以后的密钥，然后拿主密钥解密，就能还原回那个老的密钥，然后拿那个原始的密钥来解密我们的所有文件，是不是又能读出所有内容了？所以这就是一套以信封来管理密钥，以密钥来管理数据的一套机制。


为什么说这是一个信封机制呢？是因为主密钥可不是一个密钥，它是有一套功能的，所有的云平台，不管你是阿里、亚马逊、谷歌，其他云平台，只要是采用信封加密，都支持主密钥进行什么迭代和轮转。所谓迭代轮转，就是每个月我可以让主密钥的密钥换掉以后，如果你还是达到我的老的主密钥，我可以禁止你来使用，也可以允许其实可以同时支持多个密钥，也可以禁止老的密钥。然后再下个月我还可以换成其他颜色，假设我第二个月换成了黄颜色，当主密钥替换的时候，云平台会做一件事情，把原始的那个红颜色密钥读出来，然后重新加密，用新的黄颜色密钥进行加密，再落盘回去。所以这个时候我在数据库里面存的就是黄颜色的两个词的数据密钥了，一旦主密钥变化以后，数据密钥会相应变化。第三个月我变成了绿色的主密钥，这个时候我们的什么数据密钥也会变成绿颜色。


第四个月我们出现了紫色的主密钥，这个时候数据密钥也会做相应的改动。所有这些信封里面的密钥的变化我们不用关心，所有的密钥的生命周期的管理都是会有云平台来提供，或者会有强大的数据库，像Oracle、 DB two 都支持全自动地实现这种密钥的迭代，主密钥的迭代和数据密钥的重新的加解密，然后落到数据库里面。


就我们唯一要关心的就是当我们要提取数据的时候怎么办呢？很简单，我们拿这个紫颜色或者黄颜色或者绿颜色的这个什么密钥，然后通过解密的方式，那这样就是通过我们的这个主密钥解密，把它还原到内存里面来当数据库，比如 Oracle DB、two，甚至于 Mera DB 把这些密钥加载到内存的时候，它会变成什么？变回原来蓝颜色的那个密钥。


所以实际的数据密钥从来没有变过，变只是它的影子，它的那个信封里面的加密的形式，这个蓝颜色密钥依然可以把原来加密的数据给还原成原来的蓝颜色的原始数据，


所以整个过程当中数据是没有任何加解密的，你存一个 PB 的数据，这些数据不会在我们主密钥更换的过程当中出现任何加解密的情况。而且你的实际的数据密钥没有变，只是它的影子，红颜色那个密钥的影子在一直跟着我们主密钥的变化而变化，这就是信封加密。
