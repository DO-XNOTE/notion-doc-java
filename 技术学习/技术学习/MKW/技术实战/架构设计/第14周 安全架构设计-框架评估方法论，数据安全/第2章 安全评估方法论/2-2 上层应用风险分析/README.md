而是什么你这个应用有多少可能性，什么把你的应用的这个内部特征或者内部数据暴露给外部，这是暴露性。除此以外，一旦暴露以后有多少的财务损失？这个损失可能是什么真金白银的交易损失，也可能是相关的行业或者法律法规要求你的罚款损失啊。


除此以外，数据有多少敏感啊？到底能不能对外公开，以及本身我们数据要有什么样的高可用性是完全的什么 9999999 的高可用，还是只要有两个9、三个 9 就可以了？另外我对法务合规性有什么样的承诺？我要完成什么样的法律的要求？另外我本身数据是怎样进行分层的，以及我愿意在安全里面投入多少钱，这些都会影响我们整个应用风险分析的过程和决定。

最常见的思路是从脆弱性开始考虑的，就是假设我认为我这个应用企业里面的软件发布，比如有一个脆弱性，或者是我的本身的数据访问有一个脆弱性，然后基于这个脆弱性我们看一下有多少可能会出现真正的风险或者问题，那这种风险一旦真正变成问题的时候，对整个企业有多大的财产上、你名誉上的影响，然后把可能乘以影响，那基本上就能评估出大致的风险了。


对，每一个脆弱性都要评估风险，然后最后决定到底哪个风险是最关键的，要先处理。同时我愿意投多少钱来处理，我到底是接受这个风险，还是规避这个风险，或者是缓解这个风险？那这就是应用风险分析的核心思路，但是具体操作过程有好几种不同的建模方式。比如以风险分析过程为例，它重点强调是计算，如果你的数学公里特别好，就可以用各种各样的方法来带公式来计算每一种不同的脆弱性的详细的风险。那你比较适合采用这种方法，那如果你是偏评估类的，比如菲儿老师就是喜欢用威胁资产脆弱性评估这样一个套路。那后续呢？我也会跟大家聊一下，怎么用这个套路来进行什么威胁建模和威胁模型。


那这样的评估类呢？更偏什么？更偏打分了？ 12345 这种打分型的这种思路，除此以外还有失效模型。比如你的编程过程特别喜欢用什么 if else，那失效模型也是这样，它的所有考虑都是 if else，通过这样的考虑你也能算出我这个应用到底应该投多少钱，做什么样的应用，防火来规避什么样的风险，这几种模式都可以。


那飞扬老师重点会聊一聊我最擅长的威胁资产脆弱性评估，具体在评估过程当中也有很多不同的手段啊。我们的建模可以跟手段配合使用，比如说风险分析过程，你经常跟损失计算配合，通过一个底层很实际的一种计算方式，然后跟你的风险的整体的评估过程对接来完成一个基本的建模，也可以采用定性矩阵，所以定性矩阵就是不去算具体的多少美金或多少人民币，而是算一个总体的大概的这个影响面或者概率。然后跟威胁资产脆弱性可以进行配合，亦或是有些是愿意什么用无记名投票， Airfix 决策来进行评估，或者对策的评估等等，很多种不同的手段可以配合上面不同的建模手段。这里我标红的两个就是后续课程里面我们会跟大家实战的威胁模型的方法论定性矩阵加上威胁资产脆弱性评估。

首先看一下你们这个应用是不是跟法律法规相关，比如你是支付应用，你恰好还要跟银联对接，或者跟什么Visa、 master 对接，那你就涉及什么法律法规，其中的 PCIDSS 法律条例，那这个时候你就属于这个鸡蛋，叫法律合规。


鸡蛋还有的业务，它跟整个公司的财产相关，同时它一旦出现数据暴露或者数据信息泄露，会影响整个公司的声誉，那它就是属于财产声誉级的好。还有什么鸡蛋一旦你跟个人隐私相关，你跟 GDPR 相关，你就属于个人隐私。鸡蛋，这三个鸡蛋通常是放在一个篮子里，这个篮子叫什么名字呢？渗透测试叫价格评审或架构评估。这个渗透测试我们后面具体会聊，那架构评审其实就是之前我们也反复聊到过的什么。我们的架构是需要有一个评审委员会进行整体评估的，那个套路其实大家应该已经熟悉了，对不对？从什么？从扩展性、可用性、安全性等角度来综合评估价格。那这里一样还是走标准的架构评审的套路，但是要加一个渗透测试，这是为安全独门的一个秘籍，待会跟大家详细分享。


还有没有其他几单？当然有了，如果你对应用是对外的，外联的同时你是高危的，也就是说很容易被我们黑客攻击的。这样的应用就可以放在这个鸡蛋里面，或者是什么你这个应用它的服务是大量用户都会使用的，比如说用户登录和注册，是不是几乎每个用户都会用这样的应用？就属于大量用户的应用，可以放在这个鸡蛋里面。


还有什么应用啊？还有些应用虽然没有对外提供服务，但它是内部很敏感的。比如说我们什么内部的什么费用结算，或者是我们的内部的审计系统，这里面或者是有用户管理员的敏感信息，就登录删除数据库，删除机器获得的是跟我们整个公司的内部账务结算相关，属于内部敏感。


那这种情况下这三个鸡蛋我们推荐放在什么篮子里？放在威胁模型的篮子里，那大家要问了，左边三个基站难道就不用计算威胁模型了吗？也要左边三个基站是先放在渗透测试篮子里，处理完以后再扔到那危险模型篮子里，一个基站都经过了两个篮子。好，还有没有其他的基站和其他的篮子诶？当然有，同样是应用外联，但它很低危。比如说我们打开一个主页，暴露一些我们要卖什么样的产品，这个服务可能好像没有特别关键，同时也是外联的，那这种情况下我们可以认为是一个低危外联的篮子或者是鸡蛋。还有没有什么其他字段特定用户？比如说这个服务只给什么？只给电话中心的用户使用，或者只给我们后台的什么电话呼叫人员使用。那这种应用用户数少的不是几千万用户，可能只有几千用户。那这种小用户的情况下，我们属于什么？特定用户的基带还有没有什么知道？大量的内部的服务它，同时它内部还不是很敏感，比如普通的 API 调用，普通的这个数据的这个消息队列的应用，这样一些内部的服务通常你不容易被黑客攻击。你属于内联，而且你也没有特别多的 Pii 个人信息，也没有特别多的 PCI 支付信息，那这三个期待我们建议放在什么篮子里？放在应用安全测试的篮子左边， 6 个鸡蛋是不是也要拿过来呢？对了，一旦完成了什么渗透测试，完成了威胁模型，我们就应该统一进入安全测试，进行完整的安全测试了。
