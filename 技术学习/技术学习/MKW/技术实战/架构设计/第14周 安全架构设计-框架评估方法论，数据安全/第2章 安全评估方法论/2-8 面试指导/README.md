重点在于代码和依赖包，大家想一想， 321 做的是不是很容易想到就是我们的 s a s t 静态测试以及 IST 交互式测试，看到代码和依赖包必然会想到什么静态代码扫描和第三方依赖包，第三方库的扫描，但是不要忘了所有应用软件最后是发布在生产系统。


这个时候生产系统上是不是要有一套 IST 的交互方式来验证，不管是容器还是虚拟机还是 GVM 的，这个里面跑的所有的 Java 进程？是不是所有的用户认证，所有的防 c 扣注入、防XSS、防 CSIF 都生效了呢？这道题的题眼是什么？聊一聊实际工具，比如说 Sona Cube、 Soona link 这些静态的代码扫描，以及像 Sona type 的 Nexus IQ，以及接 frog 解蛙的 x tree 等等第三方库扫描的工具，或者是什么 Synopsis 的 seeker 这些 r a s t 的交互式工具。

 321 请做的。相信说到不同子系统，大家肯定考虑什么，是不是有很多个不同的情况，我们要划分成不同的，什么期待期站应该放在不同的篮子里，这个划分过程就是应用风险评估的总体思路，具体来看有些记得是需要渗透测试和架构评审的。有些基站只需要威胁模型就好了，更多的基站也许只需要进行简单的应用安全测试。


好这道题要拿高分，依然希望你能举个实际案例，在这个案例当中，你分析了哪些子系统，怎么样去对它进行分析，然后最后判定是用哪种方式来进行处理。如果你判定出来是需要进行威胁建模或者说是威胁模型的话，你应该怎么样做攻击术分析？怎么样进行具体的脆弱性和资产的评估？最终通过拽的分析判定出哪些攻击时就需要进行防御的，哪些是可以忽略的，以及应该投多少资金、多少人力在里面。这道题的重点是不同子系统，也就是什么我们要有一个整体评估，不是所有的研发经历，不是所有的经费都要平摊在所有系统之上的，是需要根据重要性，根据被攻击的可能性以及破坏性等等元素综合考虑，投有限的时间、有限的资产，到最可能被攻击的最弱的安全环节上保证整个业务总体的安全性，这才是什么？不管是风险评估也好，还是威胁模型的核心内容，我们要通过架构评审，通过危险模型，甚至通过安全测试和渗透测试，最终实现整个大的业务群的总体安全风险降低，这才是我们的总目标。
