大型应用它的网关层的构建从这个请求转发到DNS，然后通过 DNS 端的一个负载均衡，把它定向到我们的 LVS 虚 IP 的集群上面。


这个 LVS 咱在前面架构章节当中已经跟同学们讲过了，它这背后实际上是 Mapping 到一个真实的 Nginx 集群，它本身提供的这个 IP 独挡一面的一个虚IP。但是你一个请求被打到这个虚 IP 之后，它实际上会在背后根据一定的负载均衡规则，把它路由到真正的一个物理的 IP 地址集群上面。


那么我们 the nginx 集群这里，其实在一般的项目来讲，小型项目咱就搭建一个 Nginx 集群，在这扛着压力就可以了。但是我们在大型项目当中，实际上它会搭建多层的 Nginx 集群，也就是说我们的这层 LVS 以及后面的这个nginxon，你可以把它看作是一个整体。那么像这么牛逼的这个整体还有很多个，它们是多级 LVS 加上 Nginx 集群的这种架构模式，接下来这个请求才转发到我们的微服务网关。


咱这一节当中要给同学们介绍的这个组件， Gateway 组件，那实际上他承担的责任在这，同学们看到它在这里。一般来讲这类的微服务网关，它实际上是处于整个网络架构当中的下游链路，就是一个网络层的一个末端，它再往后这个请求就没有其他网关层来承接了，他们会直接的把这个请求转发到对应的服务上面去。比如说你是订单服务，你是商品服务。


那么在这个过程当中，有些同学们会有一定的误解，说我是微服务网关，我只要粘到网关这一层，那我就可以对外暴露，我把这个微服的网关层我暴露给DNS，打上 DS 的流量，直接打到微服网关可不可以？其实这样做也可以，但是同学们千万不要在高并发这种应用上面来做，因为我们的这个微服网关，它跟 Nginx 集群和 LVS 搭建的这套多级的对外网关还是有一定的差距的。


什么差距？主要不是说功能性差距，它是说你的性能差距，你跟 Nginx 这种高速网关在性能上面，你的单机并发能力，你微服网关是远远比不过 Nginx 的。所以说对于一些流量承压对外网关，那么我们尽可能的去选择使用 Nginx 集群来对外暴露我们的这个微服务网关，它其实它的作用点在哪？它跟 Nginx 比，它实际上一个很大的作用点就是说我自己的网关可以做一个服务发件，为什么叫它微服务？它自己这个网关它自己本身实际上就可以作为一个微服务注册到注册中心，与此同时还可以把注册中心下面的这些众多的服务的具体的地址花名册拉到他本地。


那么这样一来的话，一个显而易见的好处是什么呀？我当要增加一个订单服务的实例的时候，那我只管把这个订单服务怎么样加一，把这个加一的机器注册到注册中心好了，然后它就会自动的被我的网关层给它发现。那么与此同时我还可以去借助我们注册中心这里的保活机制，服务剔除机制来去做你当前服务的可用性检测。如果你自己的服务一段时间嗝屁了，没有，心跳了怎么办？你的注册中心就会把它给踢掉。


那么这样一来，我们网关层也会及时发现说你当前的服务在注册中心已经下掉了，不管你是主动的服务下线也好，或者已被注册中心主动剔除了。不管是什么原因，只要你的这个服务不在你的注册中心列表上，都可以把这种变化实时地反映在微服网关上面。所以说这个微服务网关，微服网关它实际上紧密地联系了你的服务治理技术。


那么我们在本节当中就使用咱的 neckles 作为它的注册中心，然后使用 Gateway 从注册中心这里来去消费相应的这些订单、服务、商品、服务的具体地址，从而完成 e 的服务的转发，所谓转发，实际上在这一层，在这个转发的节点上，本地是用了本地的负载均衡技术，这个是网关的通常操作。


你作为一个网关层，你转发请求薅羊毛，不能逮着一个薅，你要这个均衡，所以说网关层基本上这里都会采用负载均衡技术来去做这种转发，那么我们微服务网关这里自然而然你整个 spring Claude 这种架构，你基本上所有的组件都是实现默认实现了你 load balancer 或者 ribbon 的这种负载均衡技术，那么咱的 gateway 这里也不例外。

在我们的大型应用当中，这个网关层之间是如何来构建？这里同学们应该前面学了一个单元化的章节，对不对？我们当前单元的请求一般来讲会在网关层做什么呀？单元封闭，也就是说我订单服务这里，我要调用商品服务，我调用谁的，我调用自家的商品服务，我不会说调用另一个服务集群的商品服务。那么因此在这种架构下面，你的这个网关层以及你的服务注册中心，它实际上全部都是做了一个单元化。那么在这个单元化的架构当中，你从注册中心这里获取到的也只是你当前自己单元的这些注册列表，不会调用到其他的单元。


所以在这种架构下面，每一个单元机房实际上非常的宏大，你要确保每一台服务上下游的一套服务全部都在这个单元机房有一次部署，那这样的话你的单元封闭才会达到最大的效果，让所有服务都可以在你当前单元内完成它的调用，非常节省你的带宽以及你的网络消耗的时间。


这是一种方式，还有一种方式。从故障转移的角度来讲，假设你当前单元机房有一部分服务器挂掉了，那使你的这个服务你不得不去做一个跨机房调用。那么这种情况下实际上怎么办？这里在大厂当中其实有很多种不同的做法，其中一个做法是这样的，单元封闭到底，如果你当前单元不可用，那么我外来的流量在这个微服务网关之外的一层网关层，我就不打到你这个有问题的单元，直接做一个快速的故障转译，我把当前流量直接打到好的单元。


那么这样一来，即使你当前单元当中有几个服务还是可用的，那么它也不会承接外部流量，我外部流量直接做一个全范围的切流，这个方式非常的快，非常的见效迅速，那么也是很多公司采用的一种方式。那还有一种方式就是需要大家去从资源的角度，尽可能的提高资源利用率，那你当前单元当中如果没有团灭，还剩几个生力菌，怎么样？微服务网关这一层做一定的改造，那么在网关这一层，我们把这个注册中心从单元机房里面提成一个类似全局的一个注册中心，那么网关层在这里不仅可以获取到你单元注册中心，同时也可以把自己作为服务也注册到一个全局的注册中心，那么一旦它发生了异常，这个时候我们可以在网关层做一些业务开关，也好可以去指定你当前的这些服务的路由，可以把它路由到其他机房来。


那如果 a 服务挂掉了，那相应的我 a 服务的请求就把它路由到下面的好的地方， b 服务如果没有挂掉，OK，没问题，那么我们微服务网关继续在自己的单元化里触发这个 b 服务，那这也是其中的一个路子。但是相对第一种方案来说，其实这个方案明显增加了你的运维以及故障恢复的一个复杂度，所以一般公司来讲，选择方案还是选择第一种方案，直接切流的这种方案比较多，OK，那这是一点。还有一点就是在大型项目当中，我们通常会把自己的网络空间分成很多个zone，我们叫 zone 什么意思？空间，那这个 zone 它其实有一部分的应用，它相应需要去承建用户端的流量，那么对于这种应用来说，我们通常会把它叫做 unsecure zone，也就是说在一个非安全域，因为它可能来自外面的流量，外面的数据都会直接进到这个域里面来。


那么还有一层非常的安全的一个域，我们这里部署的都是一些非常核心的后台服务，比如说我的资金结算、对账等等，这类服务并不会对外部用户直接开放，而且其中的数据非常的重要，不可以对外泄露，那么我们在这种情况下会倾向于把它放到一个单独的 zone 里面。


secure zone，这个 zone 到其他 zone 之间的防火墙是一个关闭的状态。通常来讲，我们在这种网络架构情况下，如果你的 unsecurzone 的服务需要去访问 secure zone，我们需要把这个请求独立地提交给我们的网络运维团队。那对于一些比较大的这种上市公司、跨国公司等，他们有一个专门的团队叫Infosec，一般很多外企都会有这个团队，它是专门审核你的这种跨网络之间的请求。如果你能提供合理的解释，合理的业务场景，那么他们可以适当的打开防火墙，使你的当前的这个服务请求可以跨 zone 来调用。
