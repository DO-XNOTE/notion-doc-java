有一个是路由匹配的规则，还有一个是过滤器。OK，那我们挨个从上往下跟大家捋一捋，介绍一下。那首先这个路由是我们构建咱 Gateway 里面的一个基本的模块，那它有很多部分组成，比如说你路由的URI，也就是说我当前的请求要路由到哪个具体的服务，这就是URI。还有它还包括下面我们两个画出来的孔圈，一个是路由匹配，还有一个是过滤器，那么路由匹配我们通常也会把它叫做credit，也就是断言，为此断言，那么这两个部分加上像 URI 共同组成了一个路由，那么我们这里如果要设置一些路由规则是际上是这样的。在一个网关层面， 123 这每一个路由规则，它实际上都是由一个独立的路由 root 来组成的。那么一个请求它究竟会被匹配到哪一个路由？这个职责就交给了我们的路由匹配逻辑，也就是 with 断言，那它是这整个流程当中的第二个部分。


首先我们的一个请求，一个 gateway client，也就是我的业务发起方，这个图是从上往下看的，同学们，在这里我业务开始发起了一个调用，那这个调用首先我在 Gateway handler Mapping 这里将会经过一系列的谓词断言的匹配。那么我们 Gateway 这边强大的一个点就是它内置了很多个不同的谓词断言，那我们可以去通过你的路径，可以去通过你的 param 查询参数， cookie header 各种各样的形式去对你的当前请求做一个判断。那么每条路由规则我们说都有很多个不同的这种谓词断言的判断，那么如果你的一个请求，它当前通过了某一个路由的所有判断之后，接下来它就会被匹配到当前的这个路由里去，所以我们的这个位置断言承担了一个路由规则判定的角色。


那一旦咱的 handler mapping 这里完成了自己的职责之后，这个请求就会被丢到下面的这个 Gateway Web handler，那这个 Web handler 它实际上是会组装一个 filter chain，一个过滤器的链路，那么在这个链路里，同学们看，下面就是我们要介绍的第三个核心元素过滤器准确的说是一个过滤器链，那么这个链这里一分为二，同学们看中间这个链条一分为二，似有深意。


左青龙右白虎，大家可以看到，在你最终将要路由到的这个 URI 之前，你的请求会经过这一系列的断言，走这样的一个左边的流程。这个断言类型，这个 filter 类型，我们把它称作 Pre filter okay。什么是 Pre filter？也就是说，我在发起真实的一个目标调用之前，我执行的这些过滤系列，它是叫做prefilter。与此同时，我的目标请求当前路由的 URI 被发送了之后，我们接下来这个请求的 response 将经过另外一边右边的这条线，那这条线经过的这个些filter，我们把它叫做 post filter。


OK，那如果同学们这里尝试着去自己实现一个过滤器，咱的 gateway 的都提供了非常好的扩展点，你就可以自由地定义你的 Pre 阶段要执行什么逻辑，你的 post 阶段又要执行什么逻辑。那这个体验实际上也是一定程度上借鉴了我们 Zoo 的这样的体验。 Zoo 它可以通过继承覆盖一个方法指定你当前 filter 所要生效的阶段。与此同时，这个概念在我们 Gateway 这里也得到了一个体现，我可以在你的过滤器当中，通过把当前的业务逻辑定义在不同位置，决定他是要在目标请求发送之前执行还是之后执行，或者说之前之后我都各执行一次。
