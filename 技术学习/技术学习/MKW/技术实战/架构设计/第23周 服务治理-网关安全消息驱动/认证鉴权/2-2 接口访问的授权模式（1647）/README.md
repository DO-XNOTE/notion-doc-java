那我们接下来去连线了 Oauth 这个体验，大家肯定在平常做应用开发或者是去登录什么网站的时候有过这种体验。比方说我每次登录某个网站，它让我用支付宝扫码登录，或者让微信扫码登录等等，那实际上背后就是一个非常典型的一个 Oauth 的体验，那我们把体验用技术的视角跟同学们来去解释一下跳转、授权等等它背后发生了什么事儿。


那第一步，首先我们这里用户尝试去登录一个我想要登录的一个APP，那这个 APP 一看，我本地不保存你的用户资料，那我怎么来让你登录？一个办法很简单，我用第三方的授权，这个就是Oauth，我自己的这个应用本身并不保存这些用户资料，那这个用户他究竟是一个正确的用户还是错误的用户？由谁来管？由这个真正保存用户信息的 Oauth 的服务器这里，那么我当前的这个应用就给返回了一个URL，那这个第一步登录操作，同学们可以看它这一个叉 auth 的 URL 实际指向到哪？漂洋过海，我画一个圈指到了这里，实际上是把这个用户重定向到一个 oauth 服务器上面的一个URL，那这个 URL 后面同学们看有一个机关 callback 什么意思？这个 callback URL 我们在第二步将会用到用户在当前的这个跳转的页面上面，输入了自己的用户名密码之后，如果我当前验证成功，你这个用户是一个真实存在的用户，用户名密码都正确，姓名叫做法外狂徒、张三等等。


那么这个用户一旦授权成功之后，我的当前的服务器就会调用这个 callback URL，把当前的一个 token 给它返回去。返回到哪里？返回到我的这个想要访问的 APP 里面，所以 callback ULO 它指向的是我当前 APP 需要回调的一个地址，那在这个地址后面通过不同的 o Auth 的授权方式，那这个返回的东西其实是不同的，有的我们使用授权码验证的方式，那么它可能会返回一个授权码等等，那么我们这里呢？用户为密码提供完了之后，直接返回给用户的其实是这个 access code。


okay， access code 是什么东西？它是你当前的这个应用服务器，这个 oauth 服务器返回的一段可以去验证用户身份的一段code。那么我当前的这个应用拿到这串code，我怎么知道你这个 code 是真是假，对吧？我这个 code 可以伪造，甚至还有可能过期，那怎么办？这里就到了第三步验证token，我当前的这个服务接到了 callback UL 之后，我会把最后的这个 access code 给它拿到，然后再重新的发给咱的 oauth 服务器做一个 token 的验证，一旦验证成功，你这个 code 是一个完全正确的code，那接下来我就去可以获取到正确的用户资源。


那么这个用户资源呢？就打个比方，我可以是你当前用户在这个 Oauth 服务器上面注册的用户名等等之类的这种可以返回的用户资源。这就是我们 Oauth 服务的一个大概精髓在哪？精髓在于一个用户授权和真正业务的一个分离，业务层全部都在我当前的这个想要访问的 APP 里面，对吧？那我们授权用户保存用户这些资料，有专门的 Oauth 服务器来去做这种验证，那么因此这种方式实际上不光会出现在我们平时生活当中的扫码验证等等这些场景上，一般在这个大厂应用当中，我们也会使用这种Oauth，专门搭建一个 Oauth service 来做授权验证。那其他所有的服务我不用管，这些授权 API 之间的各种调用全部都交给 Oauth 来去授权，我只管自己的业务逻辑就好。

授权码 grand code 其实是咱的 Oauth 当中最复杂的一种验证交互的类型。那么简单来讲，我这边分成了a、b、c、d、e、 f 这 5 步，左边这个 client 我们可以把它想象为一个APP，那这个 APP 是我们想要登录访问的APP。


右边的这三个词怎么来理解？第一个 results owner，我们可以把它理解为前面这一页当中的这个人，也就是终端的用户 resource owner，它有的这个 resource 是什么呀？它这个 resource 实际上就是我的这个用户名和密码。


那咱在这整个过程可以分为这样几步，第一步用户这边打开了我的终端的 APP 还有应用，然后接下来客户端要求我用户给予他一个授权，这第一步就是我们的 client 这个 APP 去向用户来去获取授权，那么我们这个 authorization request 发出之后，当前的这个用户可以决定说我授权你本次的访问，返回给你一个对应的grant，这个 grant 就是授权的意思，可以把它理解为在技术层面上它是一个授权码 grand code。


那么接下来我这个应用拿到这一步当中，获取到了这个授权码之后，怎么办？向我的这个认证服务器去换取一个令牌，使用 grant 换取对应的一个 access token， access token 我们知道是可以表明一个用户身份的，也是我们 Oauth 当中的一个核心，那拿到了 access token，表示着我的这个认证服务器诶对你客户进行了验证，全部都正确，确认无误，这才把这个令牌发放给你。


那接下来客户端拿到这个令牌怎么来使用？去向咱的 resource Server 去申请获取什么呢？获取这个resource，获取资源。这里咱的资源服务器扮演的角色就是会去使用令牌去验证你的令牌是否正确无误。如果正确，那我就对你的客户端把这些资源给它开放出去，你就可以去获取到我允许你访问的这些 resources 资源。


这个就是 grant 的这样一个步骤，它主要就分为三步，获取授权码，通过授权码换取 access token，然后再使用这个 token 拿到我可以允许访问的这个资源，这个就是我们 Oauth 最复杂的一个授权方式，那听起来好像也没那么复杂，对吧？但是在落地环节这个还是有一点工作量的好。

这个简化模式看上去似乎比前面的一个模式更加复杂了，对吧？但实际上这个模式当中跳过了一步，同学们找你是不是发现 grant 这一步没有了？咱的这个验证获取到你授权码的这个步骤它不存在了，那它这里存在的是哪些步骤？第一步就是一个 redirection 的这样一个操作，怎么来操作？这里是客户端把这个用户去导向到一个 authorization Server，那么在这个 authorization Server 里面会去验证用户，来让用户决定我是否来去给客户端这种授权，那这种方式不会涉及到授权码的这种发放，没有这个 grand code 这一步的发放，那么在咱的这个 user authentic 里面，过程当中如果我用户给予了你的授权，那这种方式下我的认证服务器就会把用户导向客户端指定的一个重定向的 URL 里面。
看到这一步吗？伴随着一个什么呀？一个 access token 的访问令牌的发放，我们跟前面的这个步骤来比较一下，这里最大的一个不同就是缺少了这一步，没有通过 authorize request 来去换取 grand code，再拿 grant code 再去换取 access token 的一个过程，它相当于我直接把用户导向了一个URL，我让用户自己来去做授权，授权通过我就发放令牌，不通过我就不发放。


那么在我获取到令牌之后，我接下来就是一个 d 操作， d 操作这里是我的浏览器，或者是我的 APP 等等，我向我的资源服务器来去发出一个请求，这里你看它是提到 without fragment，就是说我在 c 步骤当中获取到的这个fragment，也就是可以把它理解为一个 hash 值，这个部分它不会包含在你当前的请求当中。这一步发给了我们的 client resource，也就是保存资源的服务器。


之后我的这个资源服务器会回访给你一个脚本，你看我们叫它script，实际上这个 script 是可以在浏览器上面来去执行的一个脚本。那么这样一来，我就可以从浏览器这边提取出相应的令牌，通过执行脚本来去提取令牌。并且我当前的这个 f 步骤将会把令牌再给发回给咱的这个客户端，那这就是一个完整的简化模式。


咱的这个简化模式它简化了哪些步骤？前面提到的一个步骤就是说我没有这个 grand code 交换 access token 的这样一个步骤，这是其一。其二它又有一个步骤缺失了。同学们可以发现，我的 script 这里获得了一个 access token 之后，同学们看他有没有再去对 access token 做一个校验，并没有 access token，我对所有当前访问者都算是一个可见，并且我拿到它的 token 之后，我不需要再次做一个认证。那它这两步骤是从咱的这个前面的 grant 模式当中把它剔除掉了，因此这是一个更加简化的方式。
