说到what，其实云运维内容特别多，覆盖面特别广。我们这里重点聊两个在云上特别需要关注的。一个是什么云上的软件的交付，怎么样把我的应用从一个代码转变成一个交付了最终的服务，一个真正能跑起来的应用程序。第二个是什么语音平台。你要做得好，必须要完成监控，尤其是混合云。我们不希望什么云上一套发布方法，我们数据中心的一套发布方法，而是希望统一的发布方法。同时监控也是整个业务链，可能是从云上调用云下，云下又通过消息队列传到云上。


不管怎么样，整条链路希望是什么？一个 trace ID 能够穿透中间件，穿透 API 网关，穿透公有云，穿透私有云，在整个混合云里面实现完整监控。这些功能其实都是要什么？要让大家首先要了解什么样是一套标准的交付套路，什么样是云原生最合理的监控手段。再去针对公有云、私有云或者混合云，做到完美的交付和监控。之所以也要把内容在前面三大块， i s pass。跟 SARS 讲完以后来讲是因为什么？是因为一个好的设计必然有一个好的思路，云原生就是一个好的思路。先想一想我应该把什么样的应用上云，应该如何改造云，哪些应用可以真正上到公有云、私有云或者混合云？第二步就是云产品，找一个合适的产品，把它拼装起来，把它用好了，从而实现我需要完成的功能。
最后就是云运维，光把一个软件发布在一个平台，或者光使用一个产品还不够，必须要什么？要持久化的，全生命周期的去管理产品，管理应用，如何去管理它，就是要考虑到像交付、监控等等的运维手段。只有云原生的架构，加上酷讯的云产品，加上很好的思维，很完善的云运维，才能够真真正正把我们的冰激凌吃好了，而不会拉肚子。


好。本章其实是一个比较小的章节，不会花特别长的时间。我们理论简单描述阿里云，挑两个比较经典的例子实战一下，让大家感受一下在云端的运维到底是什么样的一种体验，和什么样的本质。好，废话不多说，我们就直接进入正题。云交付。

何为CI，何为CD。我们以一个例子为例。这是某家企业的 CICD 的全景图。我们从左边看这家企业什么通过微服务改造，拆分成了很多个小的团队，每个团队拥有一到两个这种微服务。他管理这些微服，他也决定微服务在哪里进行什么代码的开发和管理。所以有的团队选了GitHub，还有的团队学了svn。没有关系，只要什么每一个微服务有独立的代码仓库就可以了。


通过一套自动化的流程，这里他选的是Jenkins，带很多 slave 的 Jenkins 把它全自动的什么走完，包括像 JIRA 这种什么审批流程，包括走完 sonocuba 代码审核流程，走完 jail unit 单元测试流程，走完结 frog x Ray 第三方库的安全扫描流程，以及走完解 Meta 这种性能测试流程。全部走完以后，他把所有的内容打包成弓箭。


弓箭发布在哪里？发布在接 Jfrog 的 artifactor 中间的一竖。接 frog artifactory。这是一个很知名的供电仓库，供电仓库里面可以支持像 Docker 的image，支持像 Maven Gradle 这种编译出来的价包，也可以支持 m p。 m 可以安装的什么 Nodejs 的软件以及 Python 的代码和引用，或者是 Kubernetes 打包出来的什么helm， chart 这些都可以放在什么接 frog actifactory 里面进行统一管理，管理完了以后靠右侧。


大家可以看一看真正的 c d 部署是从哪里开始。是从弓箭产生开始，也就是 c i 其实是什么？从代码的修改，代码的变动，一路上触发，一直到生成弓箭 CI 就完成了。 CD 是从弓箭仓库向右看结 frog archifactory 一旦有变化以后，就会触发CD。连续发布。
这个持续的发布会什么会用 Kubernetes 的套路，比如像 helm 发布到什么 K8S 平台上，变成一个一个的 port 变成一个的服务，或者通过 ansible sort of stack 发布到 link 系统里面，变成一些中间件，一些基于虚拟机的中间件。系统也有可能是什么调用一个 shell 脚本来进行什么生产系统的测试，性能测试，安全测试，冒烟测试等等。总体来看，右边的功能，它要实现铃铛机高可用，统一的管理，发布异地的复制，一键的容灾。这里没有提的什么，是最终要实现混合云的统一，cicd 流程的管理。这才是什么云平台cicd 的核心。
