对于阿里云，它的知识版本最主流的就叫 r d s 关系型数据库服务 relational database service。 r d s 是什么？是阿里云上卖的最火的数据库了，可以说它是什么？是在开源 MySQL 之上。它能支撑什么？能支撑安全的加固。比如它有事前的安全的白名单控制，也由事中的数据的 disaster 防御以及事后的审计工作。基本上你可以看上面介绍它能完成大部分的审计要求的 ISO 20000 的基本能力，以及像PCI， DSS 什么网络支付，金融支付的基础的数据保护要求。所以是一个企业级的数据库解决方案。同时塑造企业级不光要安全性，还要什么可用性。下面云平台的 MySQL 不管是哪个云平台，通常都会支持这样几种选项。一个是主备架构，主备架构相对来说是一个比较省钱的生产系统的架构，也就是一主一备的形式。还有支持什么多节点的，这种集群形式可以实现同城容灾，它每个节点里面会在不同的什么 AV 的波动，也就是不同的机房，所以在多个机房形成一个什么跨机房的集群，实现同城容灾。同时像 RDS 这样的 MySQL 数据库还能通过什么后台的那种 Binlog 的复制，变成什么异地的非同步的进行数据的传输，最终完成一个异地容灾的功能。


当某一个region，也就是某一个城市，所有的阿里云都崩溃的时候，仍然可以切换到另外一个城市的 RDS 数据库，从那里把应用起来。从而虽然间隔一段时间，虽然丢失一部分数据，但是最终仍然能够支撑业务对外提供访问。这就是r、d、 s 的MySQL。各个云平台都有类似的增强，把一个开源的解决方案变成了一个什么企业级的解决方案。但是不管怎么增强，它的主引擎是不是还是 MySQL 那几个主引擎，所以它的问题依然不变。单表是通常在几千万条记录以上就开始性能衰减，到了上一条几乎就处于什么半崩溃状态。所以 MySQL 的上限是吧，非常低。
为了解决这个问题，各个云平台都推出了什么 NO SQL 的解决方案？

 NO SQL，不是说不是SQL，是 not only SQL。在 seeker 上面做了一些增强，尤其是非关系型的增强，以及并发性、同步性、分布式的增强。比如亚马逊很经典的叫 Amazon Dynamo DB。
Dynamo DB 是什么？你可以当它是 MySQL 的什么集群版。它可以有很多个节点，实现一个集群式的很容易实现。上一条记录的数据库单表几亿条很轻松。同时它又是什么？可以支持那种列足支持一些这种，什么 b log， c log 支持那种类似于 MongoDB 的这种文档型数据。所以 Dynamiodb 实现了从 SQL 到分布式 SQL 到 NO SQL 的一个跨越。不光是亚马逊有此类套路，像谷歌也有很多套路，比如我们在 SQL 上做增强。


谷歌推荐的是Spanner，它后台是一个类似于什么大数据的分布式系统，但是向上面抽象出了一套属于关系型的数据库。但是 span 其实并不算特别秀对吧？它跟 Dynamo DB 相比是吧，没有体现谷歌的强大之处。所以谷歌针对另外一种场景分析性，针对交互型，针对非关系型，它提供了另外一套更强大的解决方案，叫做Bigquery。


Bigquery 可以说是业界第一的分析型 NO SQL 数据库，它支持什么？后台建数据仓库，把所有的关系性数据以及文档性数据全部收集过来，在数据仓库里面进行切片。同时采用类似于 map reduce 并行并发、高性能的计算方法，进行快速的数据读取。假设我们普通数据库大概有几 TB 的数据，你要读取里面相关的数据，采用一些复杂查询的语句，可能需要什么？可能需要数小时间。但 be query 不要说几 TB 几 PB 数据，复杂查询可能只要 10 来分钟就能够完成。


是不是非常非常酷炫？越复杂，数据量越大，别 query 越能充分利用它的强大计算能力。它不光是一个普通的数据库，而且是用强大计算能力、强大 map reduce 能力的 NO SQL 高端数据库好。有了亚马逊的 Dynamo DB 和谷歌的 Biquery 在先，我们阿里云又能玩出什么样的花呢？那就是 ocean base。闻名遐迩的 ocean base，轻轻松松实现什么单库数千一条，记录数百 TB 数据，极端情况下我们能完成是吧？单库就能完成几十万TPS，几百万QPS。这就是 ocean base。全球几乎是 number one 的什么交易型数据库，甚至于超越了 Amazon Dynamodb 的实际的应用案例。虽然分析型来说它比不过bigquery，但是在交易型来说，几乎就是什么排名第一的 NO SQL 好。我们要聊一聊像这种 ocean base，既能完成SQL，也能完成更多其他工人的并行的数据库，也要聊一聊 RDS 这种传统的 MySQL 的企业版升级版。下一节我们就来实战一下，大家敬请期待数据库的实战章节。
