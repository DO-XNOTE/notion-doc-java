我们先来看一下。第一种就是比较常见的，也是比较容易理解的，就是 event notification，它就是一个什么呢？那一个通知一样的东西，就相当于说事件发生了，现在我要通知一下外界这里有事情发生，大家不要觉得很简单，从表面上看很简单的东西，其实各位的逻辑可能是很复杂，正是它这种简单性掩盖了它的复杂度。


我们先来深挖一下这个简单的通知性事件它背后的含义和运行逻辑。简单来说，通知性事件就是一个源系统，这里的源不是原来的源，是源头的源系统，也就是整个事件的全或者出发点。源系统现在内部的状态发生了一些改变，这个状态它是一个统称的概念，它几乎可以包括任何一种系统的变化，比如是新增了一条记录，也可以是用户登录失败，还有可能说系统本身已经 crush 了，这都是一种系统的变化，现在这个系统他要把这种变化通知到外界，现在就是其他的系统，这个可能是公司内部、公司外部、组织内部这都是有可能的。这反正就是外部的系统，其他系统它可能会关心这个状态的变化，当然也可能他是不关心这个变化的。


但是我们要明白，在我们的 ED 架构里面，原系统也就是这个通知的发送方，他其实完全是不关心这件事情的。就什么意思？原系统它为发送方，它只关心自己是否发送成功。至于说这个接收方，嗯是怎么回事情，他要怎么处理这个元系统他不需要关心，同时也不受元系统控制，这一点它很重要，因为彼此之间没有任何控制关系，所以说他们的关系就比较松散，就是我们所说的没有一个很强的耦合度在这个里面。这样在任何一地方，不管是原系统还是这个处理的系统之间，它没有做出改变，只要你不改变这个通知本身或者通知内容本身的内容，那你想想它这两边是完全不需要做出任何一样修改的。


我改了通知，那可能两边有点影响，他如果改了原系统的一些状态变化的内容，或者我只是改了一下我处理的逻辑，其实对于这二者之间他完全是不用太在意的，这样子双方都不用考虑自己或者他人的这样一个什么相应兼容性，或者个可能。然后这个改变引起了一些问题，这就是驱动架构的一个好处，它的引入它是不会增加系统的耦合度的，它是一个低耦合的解决方案。也就是说如果你希望得到一个能够独立扩展的系统，

在这里这种通振系列还有一个很特别的地方，就是我们前面讲过了，原系统它不关心接收系统如何处理收到的这个事件通知，原系统自身也不需要关心，接收系统是会发挥一个响应，没有，比如说它不是像 GCP 一样必须是有来有回，否则这个它就不会工作。这里就是这个好处，发送方案它是无需等待响应，也就是他很快的可以去做别的事情，就我发完了之后就完事了，这个事情到我这里就算结束了。这样子对于整个系统的throughput，就是整个系统的这个处理的速度，流量也罢，都是很大的好处的。它是提高了整个 slow put，因为如果你有等待时间，那就意味有阻塞，那后续的事件或者请求自然而然它就要开始排队等待。但是采用这种通吃性事件 EDA 的时候，如果你在设计系统时对最初你的速度有要求，这种操作的架构模式它应该是非常适合这个场景的。

这种实景生架构也跟这个贝副本经常一起配套使用，当然二者它并不是一个鱼和水的关系，是可以分开独立的使用的，就是说只是提一下，就是这两者经常一起出现而已。


关于这点我们后面可以展开讨论一下，现在我们先来了解一下这种新的实验生架构。用英文讲这种事件叫 carrot state transfer，中文怎么能理解？简单的就是说事件它主要的是为了这个状态的转移而生。乍一听这个好像不是很好理解，有点怪怪的，其实要理解也不是那么难，下面我们详细的了解一下。这个事情是什么呢？这种事件它不同于前面我们讲到的那个同质性事件，虽然他们二者都有通知系统状态变化这个相同的点，但实际上确实有很大的不同，千差万别，可以说是同质性事件的内容，一般的相对来说都比较简单，就是前面讲的就是一个信号量，周杰伦的一首歌叫信号，大家听过了就我只是给你发了一个信号而已。


这个信号可能就是非常简单，可能就是一个 0 和一手电筒闪一下，它也是一个，因为它只是通知外界这个事情发生了而已。或者有一个很简单的一个像开关一样的这个效果，后续你怎么处理？需要什么样的信息，怎样支撑你这个运作，那不在它的考虑范围。但是我们的第二种状态转移的这种实现，它的目的是将原系统的数据和变化一并通知那些依赖者，这个依赖者就是什么？就是要使用这些数据的人，或者要依赖你这些状态的人或者系统。而这个依赖者在收到这个事件以后，他根据其内容要还原出原始信息，并且在本地生成一个相应的副本，这一点就是很重要的一个事情了，对吧？为什么说这个副本是这个新的事件驱动架构的这种关键点？为什么现在是这样子的？因为我们越来越多的系统它被设计分布式系统，那分布式的系统那个好处我们就不多讲，这也不是我们讲分布式的一个场景，后面的章节就是我们可能会一起来学习一下这个关于分布式的架构设计的一些内容，但是在这个地方我们就不详细展开。


但是有一点，然后我们都很清楚，就是在分布式系统里面，不同系统之间的再也没有办法直接来读取那个原始的信息的。就是你原来直接丢一个 query 或者这个SQL，直接去查一个DB，那现在因为这个微服化了以后，分布式了以后，那是肯定不可以的，对吧？大多数情况下我们都会采用一种远程调用的方式来获得自己想用的数据，那我们可以通过RPC，也可以通过 Web service，但是这都是要求源系统它是可用的，因为它作为这个唯一的这个事实来源，就信息的这个事实的这个source，所有要的这个信息的这个系统都要和它进行交互。那么在一些极端的情况下，如果这个元系统它不可用，那就不光是它自己的问题了，所有依赖于它信息的系统都会受到严重的影响。

我们可以直接读取这个副，这个什么情况？就是我们能够容忍一些简单的这个数据的不一致性的时候，此刻我们就可以直接读取副本。还有这句话再强调一次，前提是你能够容忍一定的数据的这个不一致性，这个数据的一致性问题，我们在后面的架构的课程里面也会讲一下这个数据一致性的问题，它是什么个意思？怎么回事？什么最终一致性？它强一致性，这个它是为什么？那我们因为都知道根据 cap 理论，那你最多得其二，通过这种相对牺牲的一些数据的一致性的性能，我们获得了达到了容错的目的，整体上提高了系统的可用性。
