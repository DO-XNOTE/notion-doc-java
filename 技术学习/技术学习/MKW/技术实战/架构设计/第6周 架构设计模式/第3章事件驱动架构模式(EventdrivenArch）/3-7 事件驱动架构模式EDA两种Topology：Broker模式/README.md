所以说我经常强调就说不同的模式你要套用在不同的场景下面，但是你套用的时候你要找到这个问题的关键点在哪里？就像前面讲的，我们处理的这个是一个很复杂的一个流程，你一看就知道说有第一步、第二步、第三步、第四步，但你说我就用broker，你这不自己找难受吗？而且这明显不合适，对吧？这没必要。

总之最重要的就是它是轻量级带，而且是没有业务逻辑的一个简单的分发，大家记住这一点，OK？同时在 broker 当中不同的这个类型事件怎么讲？就是说它是对应的一个单独的事件通道，这个和前面没有特别大的区别。从实践的这个角度来讲，所有的这个实验通道可能就是通过 q 或者topic，或者二者的结合来实现的。

下一个决心说我应该用这个或穷的，那在这个地方我个人觉得就是要判断这个，哎，响应这个事件，就外部传来的这个原始事件，它是不是需要一个很长的流程？是不是有很多个步骤来解决的？还是说它是一个很生产的关系，每一个的任务都是很小的，但是有很多个这个处理器愿意去处理它，但是处理的方式又不太像，这两种是不太一样的，要区分开。


关键你还是那句话，有没有长的业务流程来做这个事件的响应，事件处理器的通道的事件还是跟前面一样的，都是一模一样的，没有什么特别大的区别。它还是由事件处理器监听事件通道里面的事件，然后按照各自的业务逻辑去处理。这里就是前面讲的它没有整体性，我们已经讲过了，就是这个代理本身的话，它是没有业务流程的，这个中间缺少或者增加一个这个事件处理器，对整体流程它没有影响，或者甚至说这里就没有一个整体的这样一个业务流程，对不对？单独的这个事件处理器它只能影响自己，这一点是很大的一个不同事件处理器，在我们前面讲的每件事模式当中，如果你失败了，你可能会对整体流程造成一个影响。比如说你在房屋中介这个例子当中，那如果说负责贷款审批的这个中介人与业务人员，他没搞定银行，你贷款批不下来，你这个交易流程能结束吗？那显然不会了。



回到我们的这个收音机的这个广播消息这个例子当中来讲，主持人说请听一首歌，那法人说我就不听，另外人说我不但要听，我还要跳舞，这两者之间有联系吗？你不跳舞，你不听，会影响另外一个人的决定或者他的行为吗？那显然是不能的，对吧？甚至你啥都不干，他也帮你，没办法，你把收音机关了，他也帮你，没办法，因为这里不存在一个流程和一个协调的问题，所以说他们互相的独立性更高，互相完全没有影响。


这是一个非常松散的耦合，就是耦合度非常非常低。只有说前面的事件发到我们的 BROKER 里面， BROKER 通过前脑聚合发到对应的 event Processor 里面，这里不存在任何一个第三方来进行一个消息转化，也没有第三方，就是我们的中心控制点来判断这个消息应该由哪个事件做一切的处理，整个结构它简单通透，没有中心点，非常松散耦合的这样一个设计。
前面讲过的就是说在我们的 VDATA 当中，中间者本身它是一个生存来的点，他是知道这个事件就要举个例子，贷款买房，那贷款这个事件是由这个贷款这个 channel 才来完成的，然后这个 channel 里面的全部去听，那其实在这个上面就相对比较简单的，原事件一发，所谓的 channel 全都发一遍，你爱处理不处理不承认这个事情对于我们的 broker 的职责相对来说就是这样子的。比较简单一点。

我们再用一个生活中的例子再来理解一次，比如说我们的学校要举行一个会的，周六的时候，但是你说可不可能由校长来给每个人做到一个通知？给几万个人？一所大学里面几万个人每个人都通知一次，这个当然是不大可能的，对吧？那对于我们来说，可能一般都是宣传部门，他就把这个我会的信息它做成一个海报贴在了我们的食堂外面，这样每个去食堂的学生他就可以看到这个消息以后，那他就知道有这样一个事件，他就做出响应，但是每个学生他是独立的做出了决定是否要去参加这个舞会。


在这个例子当中，海报它就充当了这种代理的角色，它承载了一定的信息，告诉大家有这样一个事件，它对原始事件或者这个消息它没做任何的处理。比如说学校说我们周六要举行一个什么样主题的舞会，那这个海报上肯定几乎是原封不动的，对吧？你就把它传达出来了，然后每一个对这个东西感兴趣的人他去听，然后自己独立的做出了一些选择，说我要去参加还是不要去参加，这些人之间会不会互相干涉说你必须去参加，你必须不去参加，这不存在的。


而且同时每个人的机会都是均等的，海报一贴，他会不会强制说只有些人看得见，有些人看不见，或者选择性的让人不能看见，选些人选择性的让另外一部人看不见，你这不存在这个问题，对吧？他也不会说只有某人才能怎么样都存在的。这个地方相对来说就是信息是透明化。

首先来讲的就是谈都是监听这个 channel 里面的那个 event 来判断是否处理。这句话看的就很简单，其实你理解的话其实是有两点不同的，第一点就是都是监听 channel 的event，这点是它们的相同点。


不同点在于什么？第一在 media 当中这个 channel 里面的事件是经过 media 转化过头的事件，前面有讲过原始事件或者原始消息到了 media 手中，它根据自己业务流程和需要，它会把它转换成一个需要的业务对象或者业务事件，然后传递给它的channel。而 channel 里面的 event Processor 都已经处理，但是在这个 blocker 下面不是这样子，他接到什么就发什么，到了那个 channel 里面，那 event Processor 他来接受这个东西。


第二就是说判断是否处理，就一个 event Processor 能不能处理这个事儿，在这个地方它是有自己的判断的。就在 block 模式上面，在 media 当中它不一定是这样子的。为什么这么讲？就是说 media 它可以控制，说啊，比如说是某种事件的话，我就不发给这个channel，那进而控制这个interposure，它就根本处理不了这件事情，所以说这个地方有点不一样。
