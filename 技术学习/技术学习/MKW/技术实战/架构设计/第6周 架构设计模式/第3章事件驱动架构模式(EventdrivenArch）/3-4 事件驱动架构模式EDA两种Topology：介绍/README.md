前面我们学了一些关于 EDA 的基础知识和一些它的简单定义和一些现实的典型应用，这种应用可能相对来说是我们做一些初步的分析，但事实上就这个 EDA 市政系统架构模式，在真实的这个实现当中，或者说从设计的思路来讲的话，我们是对待进行了两种不同的实现方式，也就是说我们把它定义为有两种 top Lodge 就是两种 top 图就是通过不同的架构模式，虽然他们都是这种架构，但是因为他们的行为表现或者说解决的问题的场景还是有一定的差异。

它具体的这个含义还要怎么应用，我们在后面讲解，现在大概知道小 mediator 就适合哪种场景，你需要一个中间层或者中间节点来替你做一些coordination，就是协调协助，因为你整个这个运转的系统里面它是有一套流程的。记住在这种情况下你就需要 media 的模式这种topology，那另外一种就叫做broker， broker 就刚好跟它比较，反过来它是属于一种无中心也没有协作流程这样一种，相对来说就是比较目的单纯自由一些，相对来说它也不形成一个系统性的这个政务链。


所以说具体用哪一种，这个当然取决于你的应用场景，我们下面就详细的介绍一下这两种不同的 top 各自解决什么样的问题，我们应该如何使用它。嗯，首先我要强调一下，我们前面也讲了这个，不管是两种 top 里的哪一种，它都是解决不同的需求和被使用在不同的应用场景下面，本身它们是没有什么高效之分的，就不说哪个就很厉害，哪个就没有那么厉害，这不是这么是吧？只是我们在不同的情况下，作为一个架构师，根据我们的需求和目标来找到一个合适的模式而已，这一点大家一定要把握清楚，并不是说使用某一种就高级，一些从严，另外一种就不高级，这不是这个意思，他们都是平等的，只是解决不同场景下的不同问题。


复杂度、各自侧重点是有差异的。首先那我们就来学习一下这个mediator，也就是我们说的中间模式。

在这个没有进入这个具体真实细节之前，我们先从生活中的一些小弟子，哎，我们来初步的了解一下这个中介模式，它是一个什么样的场景。那你想想在我们中国，特别是一层的中介，在我们中国这个你是不是马上就是身穿西装，手拿传单这么一些形象？但是如果说我们这些人是生活在这个大城市，比如北上广深这种地方，不管是你要买房还是要租房，其实你如果能找到一个比较称职合格的这样一个中介，虽然说你付出了一些代价，这个代价可能是精力上的或者是金钱上的这种代价，但是你其实也是收获了一份安金。说实话不是为他们什么辩护，这是真实的。


为什么这么讲就要比如像我是一个码农，我们用码农只能是每天好好工作，像我们这个架构是课程升职加薪的，哎，年薪百万呐，首付也够了，现在我要买一套房，但我只是一个码农而已。那你说你要找一找这个系统的bug，做这个专业的这个技术分析见解，做架构设计，写两行代码。唉，这个我很专业。你说你现在让我去找一下市场上的房子，你说我这能做到这事吗？那我肯定显然是很困难的，不说有绝对不行，显然相当来说是非常困难的，对吧？就算是我又能找到一个行字，那你说我还要什么？法律文书，谈判各方资格，这些交易流程有哪些环节？什么东西乱七八糟的，你说这个都很困难，各整个流程当中有各种各样的文档准备，你要提交什么样的材料？那不可能知道的，因为你不是专业人士，除非你花了很多的时间专门去学习了解这件事情。


当然你这样子做，第一是不是够专业，哎？第二是不是够全面？这个你还是不是特别清楚，那对吧？这么讲，所以这就是一个我们社会的这个分工。要从社会上来讲的话，确实从社工的主题来讲，它就是个社会化的分工。每个人他有自己的专业，我们专心就是写代码，对吧？产出更好的应用，很好的系统。


这些地产中介他专门找房子，同时他们熟悉整个交易流程和各种各样的法律法规还有政策，对吧？这就是我们分工的不同，那所以说现在选好了我们这个价格设计模式，我们当架构师，升职加薪还要买房了，你马上走到附近的一个中介公司，这样就跟他说，相当于发出一个通知，是吧？本大爷有钱了，现在要买房，这个世界就是这么一个大概的意思，但是我告诉他是一个很笼统的这样一个事情，只是说，唉，我大约在什么时间唉？要买个什么样的房子，基本上就是这样子，就很简短的一个要求，对吧？你不可能长平到那唧唧歪歪的提出一道道的东西，那可能都是后续的事情，你首先就跟他讲说我要买个什么样的房子，对吧？很简单的一件话，理论上来讲的话，你这个时候你就其实可以回家了，你只是先跟他说我要买套房，当中介帮你找套房，对吧？中介促进你这个成交，你回家那剩下的这些工作在短期内肯定全部是由中介来完成的，因为你啥也不用干，因为中介还有很长的时间。


那中介的这个角度来看，他们收到了一个通知，就是由客户发起了一个通知，某某某，那要在什么地方，什么样的需求，买一个什么样的房子，大概什么时间节点，对吧？但是其实就是中介内部而言的话，你想它是一个公司，它肯定不是一个人的单打独斗。当你硬说我要有一种一个人的这种中介公司，那我有没有什么好讲？当然只是开个玩笑，他也不是一个人来完成这个通知所有的内容科学的处理，因为要完成那这个买房子的这个事情，它是一个相对复杂的这样一件事情，它需要很多内部、外部的人和这个部门来分工合作。


那这个时候中介的老板他整个是对这个所有的环节都非常清楚的，这不光是交易，就是上下游什么啊贷款什么全都懂。从他那他是知道的，他知道从第一步到最后一步要找谁来，大概要做些什么事情，但是他都只知道大概这个细节，他就不一定会知道了。所以说他就会按照他自己了解的流程，把你的购房需求拆解成很多个步骤，这些步骤它有可能是可以并行的。比如说一边找房一边帮你查询法律法规政策，这都是可以并行的。那有些是没办法的，那必须得串行。比如说你总得先找到房子，然后才能房东见面进行谈判，你不能说我直接就谈判，你跟谁谈对不对？这不可能的，这些事情都是这个老板来分配的，但是这个执行的过程就是说你决定某一个步骤或者某一件事，这个老板肯定基本上是不大可能会亲自下场的，哪怕这实际这个工作当中他亲自下场，他忙得过来吗？处理这么多的事情，对吧？不可能的，基本上他分配，他就是着重的分配执行，那就只是执行，就是部门里面的各种员工，他在这个什么组什么组，他自己因为他负责这法律文书给他负责交易给他负责这个资格审查，有的负责替谈判，也都进一步的去搞定这个贷款都是不一样的，他只负责分配，确保每一个任务都按时完成并且达到他自己的目的可能都是有它是结束了标志的。


而这个每到 0 到员工任务的线，员工他怎么办？他也有自己的专业，就前面讲过的，各自干各自的，各司其职，互相不干涉，也不了解对方是怎么完成他的任务的。就是说我如果是搞这个交易的撮合的，那我可能就是不管这个贷款的，一般来讲肯定是讲到小公司可能不一样，我只是举个例子，为了体现一下我们这个中介者模式的这个含义，它完成任务怎么完成的复观性。但是我们一旦就在这个老板的分配上，只要这个任务链条上的每一个环节团队完成了，那整个从最初发到通知到整个完成，唉，这个买方需求就已经实现了，对吧？同学们，你听一下这个例子，你是不是就对我们这个中建模式其实已经有了一个大概的认知？或者说你只需要自己是经历过这种买房或者租房的话，你都应该有这个感受，就知道这个流程是不是好像就真的是这么回事，哎？这个中介子模式好像，哎，蛮合理的，哎，感觉就是这么一回事情，哎，对不对？所以说中介的模式它是适合什么场景？就是当某个事件发生的时候，要处理这个事情，它需要一系列的步骤，因此它是需要一个居中协调的这么一个要求。

在中间这种模式事件队列，它是将原系统也就是生产或者发送事件的那个系统和事件处理系统的连接点，就是说它是怎么讲？就是说源系统和这个事件处理系统理论上它是不发生任何关联的，它在这个中间种模式当中， event q 就是扮演了这样一个连接点这样一个角色，源系统将它的事件就 event 发送到这个 event queue 里面去，这样它就会做到一个解耦的这样一个作用，类似于说我不需要知道你在哪里，但是我发到某一个第三方，剩下的事情就是交给你了，我其他不管，对吧？这就是一个窄耦合。第二也可以这种方式达成这个异步处理的这样一个能力。


从整体的这个系统来看，源系统它是只关心如何将事件发送的，事件对联，它要保证的事件发送本身是成功的就可以了，对吧？其他都不是他的职责。而事件处理不管他有多复杂，这个他也不关心原系统是怎么发过来的，都是这样子的，从哪里来的，什么时候生产的？他只负责从事件队列中读取事件，并且按照规定的流程来完成事件的处理。再来看事件队列本身，他负责将原系统生产的事件传输到事件处理器的系统中去，对如何发送和如何接收处理他均不关心，理论上实现了各自按照职责分工各自独立变化，也就是说，这是一种低耦合设计的典型特征之一。
