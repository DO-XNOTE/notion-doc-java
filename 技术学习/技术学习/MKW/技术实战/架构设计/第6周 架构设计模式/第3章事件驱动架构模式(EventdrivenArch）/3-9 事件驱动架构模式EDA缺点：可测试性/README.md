# 3-9 事件驱动架构模式EDA缺点：可测试性

学习过这个 ED 价格模式的这个优点之后，那自然而然我们就会问 ED 价格模式的缺点是什么？从我个人的经验或者从一些理论上、实践上来讲的话，可能有些时候反而是缺点更重要，因为不知道优点都会发挥它的长处来帮助你，如果你不知道缺点，可能会给你带来致命一击。所以说下面我们来了解一下第一，价格模式的这个缺点。


首先来看一下第一个缺点，就是它的这个可支持性，可支持性不是特别高，为想这么说我们找人看一下，单从这个单元测试上来讲，可能感觉差别不是特别大，为什么这么说？因为不管你是一个庞大的系统，还是像我们这种分布式系统，把它制成一个单体的组合在一起的这个完配式的分布式系统，对吧？它没有特别大的区别，顶多就是你要花费的时间和精力大小而已，但是从难度来讲，它不会有特别大的差异。但是如果从集成测试的角度来看的话，相对来说我们这种信训这个模式下的应用，它可能就会比较难一点。

不管你采用哪一种toplodge，不管你是 mediator 还是broker，它都是相对比较麻烦一点。为什么这么说？第一就是它的主键比较多对不对？但也是说所有东西都夹杂在一起，这个写起来是肯定很容易的，测试起来相对来说也会容易一点，因为你不需要写那么多 test case，对吧？你现在这样可能原来一个类搞定所有一切，简单粗暴，我现在变得优雅了，有很多个类，但是测试起来就累一点，要写更多的测试 case 对不对？你既然要集成在一起，它也更难，因为原来你是一个单体的，直接就搞定了。


现在我是一个分布式的，要把它组合在一起来测试，这个肯定相对说难度要大一些，组件越多，对吧？这是一个异步的，之前就有分享过，异步和同步最大的区别是什么？同步你是可以完全控制的节奏和它的这个相应顺序的，就是你的流程，整个 flow 都在你的掌握之中。但是一旦到了异步的话，这个前后执行顺序，什么结果，谁先谁后一类关系都变得复杂一点，你很难掌控它。所以说你在集成测试的角度那就很难了。举个例子就是你比如说我指望我发出一个event，然后马上我那边就会从数据库里面看到一条数据，你如果是一个同步的这样一个系统，那这次妥妥的必须的，对吧？那如果是异步的那个就不好说了。


如果说你通过一些网络传输发一条消息出去作为一个事件，对吧？什么时候那边收到消息进行处理，什么时候处理完，有可能真的完全不受你的控制。所以说这就是从集成的角度来看这个可测试性，相对来说在线循环架构模式下面，它因为自身的特点的关系相对来说不是那么优秀。


我们来了解一下 AJ 架构模式的另外一个缺点，可维护性。我们从两个角度来聊这个问题，第一就是从开发人员的角度来看这个问题就这个代码好不好读，好不好改，这是一种可维护性，同时另外一种可维护性。其实从运维团队的角度来看，这个运维团队是技术的，运维团队就说他们是帮助你去做上线，做deploy，做这个troubleshooting，帮你找问题的人恰好不好用。

OK，那首先来看我们，首先所有东西都是分布式部署的，这个可能对我们的整个 CICD 版本控制什么一切，甚至后续的这个技术支持，包括我们运营团队都是很大的挑战，所以相对来说可握性在这一块来讲还是要差一些。那同时我们这个 EJ 它是一个异步式的，基本上我还没见过同步这么搞的，就是它异步的，那对程序要求相对来说就要高一些，为什么这么说？因为就是说同步的更符合人类的一个直观的思维方式，因为我们所有的世界里面，这你不分开来看，它都是一个同步调用的过程，但是在这个程序事件里面，它其实是可以像异步一样的，那你要控制这个先后顺序一些流程方面就带来新的挑战，所以这方面对于程序的素养或者对这个程序的熟悉程度就有一定的要求。


相对本人这种普通，他的 word 那就不知道高明到哪里去了，对吧？所以说对程序员的要求稍微高一点点，这对可靠性也是一定程度的伤害，越复杂度越高，肯定读起就越不容易改，且也不容易，对不对？同时这里也提到了这个异常处理，为什么要提这个问题？就是这样子的，因为你是一个分布式部署的系统，那么分布式就带来一些挑战，比如说分布性一致性的问题，分布式一致性、分布式的这个事务处理这些东西怎么来？ handle 有任何异常处理而发生的，那怎么办？怎么来保持数据的一致性？你要控制这个事物，怎么控制这些其实都是一个很大的难题。


这一点，特别是这个异常处理，不过是说代码的可读性、可用性，还是从运维的角度来讲，好不好？来 debug 你这个系统好不好的？查这个数据的异常什么之类的，都是相对来说要不困难一点，比一个单体系统，但真的是要复杂很多的，如果大家有自己亲手开发的这种大型的分布式一致性，但是非常困难的，这样要分布式一致性是非常困难的，对不对？所以说它有很多要求，很多挑战，特别是当你系统一切正常还好，特别是在这个异常情况发生的时候，你要处理它，一个处理不好整个系统在这个可用性，还有这个准确性的数字一致性全都是错的。


代码的可读性前面我跟你讲过了，你有分布式的，也有异步，还有这个什么以是否一致性，在分布式的情况下的一个处理，还有这个数据一次性的要求，所以说它代码可能会变得比较复杂一点点，写得好和写得差这个差别就挺大的。这个对代码的可读性来讲也是一个很大的挑战，所以从前面两个角度我们讨论的因为这个分布式的这个问题，

就是说我们前面讲的这是一个分布系统对不对？然后由于它是一个消息或者说事件传递系统带来的这样一个分布式，它同时带来异步有可能是会提高这个性能的，这个是几乎毋庸置疑的，大家相对的赞成的应该是赚多数，我相信，那同时你想想你如果说有个消息传递的这个问题，那是不是性能也会受到影响？为什么这么说？你想一个问题，大家作为软件工程师还是架构师，或者说网络工程师，你消息只要发送，那肯定必定要经过这种异构的网络，特别是在现今这个互联网这个系统里面，这个网络问题可能是一个大问题，也可能没有任何问题。


什么意思？就是说你可能发送一个消息，不管你是通过 HTTP 发送还是通过这个 MQ Kafka 这种发送方式，不管你 UDP 还是TCP，只要你发送消息进行了一个网络传输，但这个可靠性还有它的延时，其实你不是完全 100% 的受你控制的。那如果举个例子，诶，我系统写得比较挫，我就一直超时了 10 分钟，我都一直没把消息发出去，这个消息一直事件就是没法传递出去，那你说这个性能它是上升的还是下降了？那肯定是下降了，对不对？那所以说就是这句话什么意思？就是说这种分布式的消息或者事件这种处理，虽然说在某种程度下通过异步性提高了我们的整体性能，或者说整体的这个support，但是事情一提两遍就来了，他如果说没控制好，没处理好的，很有可能就会成为你那个瓶颈，因为你反复传输一直没传输成功，或者你它一直被 broken ID 网络层的一个瓶颈，可能反而会让整体性能下降。


所以曾经大家知道吗？这是一个事物的一起两面，并不是它互相矛盾，它就是存在这样的好处，那也是有这样的缺点，所以说就看你怎么去规避这些事情，前面我们讲过，你不知道优点可能没太大的，要是不知道缺点这个就比较大的麻烦了，有可能。所以说这一点大家一定要注意，虽然说分布式带来的好处，但是它有它的问题，大家在实际的应用过程当中一定要注意这一点。
