我们举个简单的例子，比如说原来你是一个老旧的系统，是一种大一统的这种系统，大家所谓的功能装在几千几万个功能，全放在一个或者一个 jar 里面那种。


现在我们要微服务化了，来把这些功能都拆分出来，实现出大家都知道只要你拆分微服务，它就是上到别的平台上，那你自然就是面对数据库的这个分离，那数据库分离，这个怎么分离？这就有可能其中的一种可能性，我相信这个可能性是 90% 以上，就是你可能会面临双写，为什么会双写？所以大家就没想过，所以说我就是觉得这个题目是非常难的，就是贯从这个设计本身而言，还是从这个怎么讲实现层面都是非常非常困难的一件事情，很难做这个题就看你的这个思想能力，我在这里也只是结合我自己的经验和一些理论知识，给出了一些我自己的看法，希望同学们在课后有时间可以自己更深入地去思考一下这个问题。也欢迎大家在 MBOOK 网的交流群里面，我们可以一起探讨一下这个问题。


不能说我给出的答案就是对的，我一直强调这件事情，因为在不同的场景下，甚至在公司里面，不同的技术栈里面，可能这个方案都是会有所不同的，大家就是见仁见智。你如果觉得我这个做的还可以，并且给了你启发，那你就仿照这种解法，以后不管是做面试还是做这个实践的实施，你都可以采用方案，对吧？当然你可以根据你的情况做一些调整，如果你觉得我这个方案不太适合你所遇到的情况，或者你见识过更高明的这种做法，那也不妨我们在群里面一起讨论一下。


我们回归到正题，淘汰一个老系统要实现双写，那什么情况呢？意思就是情况已经说过了，因为我们可能淘汰老系统的时候，你要保证老系统它可能在一定程度上还是需要支持一些别的地方的这样一些功能。大家有没有想过一个问题，就这样子，举个例子，现在你要迁出，嗯，比如说我们现在功能是 100 个功能好了，由一个万来组成的，这肯定现在的社会上存在的可能性就非常小，很难做。那我们就要把微服务化，就意味着我们数据库也要进行搬迁，我们会兴起一个domain，在这个 main 下面有自己的数据库，自己的这个技术栈，也有自己的这个服务器。


部署的地方跟原来的地方肯定是完全要分离的，但是这个分离是不是就是这么简单？大家有没有想过问题？首先你这个老系统，因为老系统它是大家打在一起的这样一个 wall 或者一个账，那就意味着它以前肯定 99% 的可能性，它都不是走在这种远程这样的方式来使用的功能。


这话大家应该懂什么意思吧？就是说如果我们已经是一个微服务化的这样一个大型体系结构了，那当然你本来调用的是可能就是通过HCP、 GRPC 这种东西来完成了一次调用，所以大家不存在说什么。哎，我直接读你的一个数据库，我直接读你的一个表，这种事情肯定是不会发生的，这种情况他也没有这个权限来读。但是如果说你的过去的系统，它本身就是一个大一统的系统，所有的功能都放在一起的，那显然是不可能的。这种情况下，那这个东西你做法，你怎么做？那你就必须得保持向下兼容性，这个向下兼容性不光是指你功能本身的向下兼容性，这个功能本身向上兼容性相对来说你在拆分的时候，你可以通过这种流量切换控制的手段，你可以得到一些相应兼容性，包括当然也是你代码层面的一些功能上的兼容性。
我这里想谈的更多是一个数据层面的一个问题，功能我们可以在别的地方再探讨数据的情况是怎么样，就是我前面说过了，原因是因为老系统里面有很多人是直接来读取或者修改的这样的一个数据，这个问题是非常非常严重的。大家有没有理解我的意思？为什么？因为第一就是如果说它是直接在读取你的数据，那比如说你是一个 user 系统好了，举个例子，那人家直接读你，它就直接 query user table selection from user tables，那这个东西一下子就迁走了。


如果说你新增的用户或者改变和用户不再反馈的老表里面，它的功能是不是就受影响？为什么这么说？因为它的功能迁移的时间点可能跟你不一样，虽然大家都在做微服务环，但是大家上线的节奏，它这个是有差异的。你上线成功了，它就上线成功了吗？你就需要它就切出去吗？这不一定的。


所以说你要保持这种向下兼容性，就是说你的 user 服务相关的数据还是得在老的地方要有，是不是就是说你迁出去了之后还得念念不忘原来的地方，那你怎么把这个数据给它补回去？这是一个问题，大家记住，这是第一个关键点，就是说你要保持数据向下兼容，就是人家原来读你的表的时候还能让他读，虽然很恶心，但是我们毕竟不是空中楼阁，你必须得支持那否则这个网站就不 work 了。


那你怎么办？没办法，对吧？这是第一个gap，第二个更难的点在哪里？还是一样的，它是直接读取你的表的。它也可以直接写呀，我直接一个 SQL update 一下 user your name，你把文赖怎么样？没办法呀，老系统很多这种很不讲理的功能都有可能在你根本就不清楚它发生了什么，对不对？那怎么办？这种甚至可以说比第一种还要恶劣，因为它直接改你的表你都不知道，如果说搞得不好的话，那怎么办？那也很复杂。但是怎么办？我肯定要给他一个方案，那是不是就要把这种不受你控制的这个血操作也能够反映到新系统当中去？这个方向就反过来了。


大家有没有发现，一个是说你新系统迁移出去之后，你要保持像老的系统里面这个数据，他还能够更新，就是说你改了他也得改，他也得跟你同步。另外就是他先改的，他是这个修改的人，他改了之后你一夜的跟他一起改，大家懂了吗？这是一个双向不同的这样一个问题。这另外一个核心点，一个是你改让老系统能反映出来，另外是老系统被改了，要反映在新系统里面，就它两个数据要一致性，这个是不是又变成了一个数据一致性？分布式情况下这个问题相当难了。我还不谈其他从实验层面的问题，我单从这个设计角度来讲，你要完成这样一个操作就是非常困难的，那怎么办呢？那我就尝试来破一下这个题。首先来讲的话，不管你是真正改查的哪一种，那首先肯定就是有数据变化才有一致性问题。如果数据没有变化，那不存在嘛？那比如假如说一个配置文件大家都写成一模一样的，也就不存在这个问题。所以说我们讲的还是一个就是由变化引起后续的协议的问题的这样一个问题。


其实问题的本质的核心，那之所以这个数据会变，那肯定是有什么东西触发让它变了，那变了之后它又有什么？就是表示系统的状态改变了。还记不记得之前我们在定义这个事件驱动架构模式里面的时候，前面有讲到有两种东西，这从英文在本身的分类来讲的话，我们有谈过，因为它分成两种，一种叫做notification，一种叫 certicarol transfer，这种就感觉就是比较适合那种。就是说 aesthetic transfer 这种就是它是带着自己的一些更多的信心，直接可以广播出去的，你就可以根据这个信息重建数据副本。


但我们来看一下什么时候的场景，我们来简单看一下。第一，如果说是新系统的数据发生了变化，但如果说我有一个能力是把这种变化能够告诉原系统，元系统根据这个变化来把它做一个相应的修改，那是不是就保持一致？我们先从理论上不讲，实际就从价格层面来讲，这种可能性应该是指代的就是新系统里面的数据与 user 表加了个字段，做了一个变化，那把这些逻辑又反馈回老系统，那我每做一次个变化，就把这个变化变成一个 event 推送出去，让所有想知道这个变化等都把它拿到，并且在本地做出相应的改变。


这是拿到一个副本，强调它本地感觉好像也是有意义的把数据写回去了，那同时相应的你看如果是老系统变了，那就反映在新系统上，其实做法也是差不多的，理论上感觉这种就是可行的。因为你像揣测老系统，那肯定保持兼容性，这没什么好说的。所以两边好像采用这种事件性的价格模式都是可行的。


那我们就假定这个基调就是这样子的，我们把任何一次的数据库的改写变成一个事件，并且把这种事件所包含的所有的变化都发送到外面的一个广播体系，一般一个不管是 q 还是 bus 里面发送过去，然后对他感兴趣的人都可以去做出相应的修改，或者说一个存储，或者说保持一个副本的更新，这都是允许的。那所以说现在怎么做这个事情，这又是一个难题了。
为什么这么说？那你想想啊，在这个实现层面来看这个问题，假如说我们已经确定了是这种采用实现驱动架构模式的方式来完成，简单一点我们不搞得很复杂，就把这个假的，你是采用这种 broker 模式来复制这样一个数据，就你发消息，我这边收的消息，然后直接在本地保留一个副本，相对简单一点。然后我们就不采用 video Ator 模式，因为它没有特别长的协调什么之类的。


这种我们假定当然是实际情况在谈，我们先假定是相对，哎， Fox 这个在 vocal 模式下面，那消息来了之后你就纯诶，听上去好像挺完美的，但事实上是不是这样子这么简单？你想想任何一个通过网络传输的这样一个东西，它都有丢失的风险，那我们要做这个操作就面临什么几个问题。


第一事件这个修改的源头，不管是挠系统还是新系统，它是不是一定能够保证把这个消息发出去？这就是很大的一个问题了，因为我们强调的是一个分布式系统下面的一致性问题。那整个举个例子好了，原系统写入的一切都成功了，数据会 commit 啪，系统 crash 掉了，那你说你这个怎么能发得出去呢？那么肯定典型，如果你通过程序来操作，你典型的就发不出去这个消息了，对不对？这是第一个可能会失败的点。


第二点，就算是发出去了，它也不是直接发给你的，真正就是要修改的地方，它也是发到一个中介上面，这个中介不是指我们 data 的中介，而是指的就比如说是我们的 q 或者我们的 topic 上面去，它发过去之后，发过去之后能不能发送成功？这也是一个问题。因为有可能比如说是本身这 q 或者 topic 是不可用，或者我们的网络不可用，这都有可能它发送不成功。那即使前面没有crash，它发送成不成功，这也是一个问题，发生过去之后会怎么样也不知道，发送完了之后你也不清楚，我们就再假定前面一系列它系统没有 crash 发送，你发送成功了。然后你再说那接收方，比如说我感兴趣的是老系统里面我们参加了一个新的listener，我来 listen 这种变化，我来 listen 这种event，现在有一个从新系统里面发出来的一个 user 表的几个change， user 数据变了，现在我们要在 take 一些 action 做出一些变化来保留副本一版， i 更新一版。


好，我收到消息了，那我收到消息之后我系统就挂了，怎么办？那肯定是没办法保持更新，那我即使没有挂，我收到消息，我处理出错了又怎么办？或者前面处理一些正常之后 commit 都是失败的，那怎么办？这些问题都是非常非常复杂的，你把它放在一个分布式的这个系统里面来控制它，这个数据保持同步就非常困难。


所以说你做完架构的设计之后知道这个事情怎么做了，但从实现层面还有很多问题等着人去克服这里，因为我们毕竟不是讲实现的，当然我欢迎和大家一起探讨，我只是告诉你有哪些点，如果你是在面试的时候，你要把这些点都想到有问题有答案，这样才是一个完整的好的一个面试技巧。


首先你至少要把问题提出来了，然后再结合当时公司的这个实际技术栈来想想怎么解决这些问题，这些都是非常考验你的思维深度和你的这个思维逻辑的，逻辑不对，你什么都不行。所以我再就回顾一下这个题目，第一是想想怎么从架构，当然我们最后从价格里面是我们想要实现学生价格模式来保持这个双写的问题，找到了一个问题一些点。第二就是说是如何从实现层面来解决，支持这个实验分架构来达到这种分布式的一致性，这都是一些问题。比如说即使是前面这些过程你都做了答案，那你就想想你每天怎么保证说我新增的数据两边是对得上的？那你最后是不是应该还有一个是类似于对账号的系统或人为介入的系统这样一些东西？因为你如果是一些银行之间，比如说我们这种或者金融系统里面的一些东西，你要保持数据同步能保持你可能丢了一块钱，你当天那个账都关不了，这个就是很大的事情了。


从思维的或者从性能架构设计的完备性上来讲的话，你还可以看一下我们是不是应该还有一些类似于对账的这些东西，因为即使你感觉好像说每一条消息都没有丢，那是感觉而已，你又没法去数了，对不对？数了最终你还是得提供一个双方数据的这一个对比的这么一个报告一样的东西，告诉他今天这边新增了多少笔，那边修改了多少笔，修改多少，删除多少。这东西你要对得上，才表示你这个数据是完整的、一致的。
