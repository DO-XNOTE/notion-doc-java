第一点我们从这个整体灵活性上来看，系统的整体灵活性是指什么东西？就是说我们这个架构如果应用在这个系统里面，那么如果在外界的场景，不管是使用场景还是什么场景，它不断在改变情况下，我们这个系统本身能不能做出快速响应，这个快速响应不是说这个系统的本身的响应速度、时间就没掉的，但是性能我们这个端指的是说我们的开发者或者说这个系统本身的这个调整的速度能不能跟得上这个变化。如果说是联合性比较好的系统，就是开发者可以加快的时间内，针对这种时间场景的改变，就需求的改变做出一些调整来满足这个新的需求，如果说越快，那么这个指标就越好。


视线处理器本身在这个事件驱动架构模式下面，它的目的是非常单一的。理论上我们前面有讲过说事件处理这个目的最终是单一的，而且是内聚的，而且它是我们唯一的。而真正有业务逻辑的这个主角，它本身是高度解耦的，不依赖于任何的外界的这样一些东西，同时实现处理器之间也没有任何的互相依赖，可以是独立变化的，也就是说不管是我们的这个 broker 还是 Mediator 事件处理器，它是完全不跟这两者之间有强耦合的。所以说如果说外界业务需求有发生变化的时候，我们只要通过改变事件处理器，在有些场景下可能就可以达到这个响应业务变化的这样一个需求。也有可能如果是对中介模式来讲的话，并且为我们来讲中介模式是有这个一定的业务场景的，还有 BROKER 模式是没有的。


在中介模式上面，如果说这个整个业务流程要调整，可能 mediator 上是要有些调整的，但是因为我们的职责是分开的，所以说如果 mediator 上面要多一个流程，那么其实只是多一个点，多加一个对应的事件处理器，那对于已经成型的事件处理器和流程来讲的话，它是没有任何影响的。所以说这点灵活性是非常好的。同时这个代理模式前面讲过的和这个中介模式也讲到了，代理模式它的这个灵活性相对的更高一点。原因也就是前面有讲到过的，因为它本身是不具备中心点，而且也没有一个流程的这么一个概念，所以说它的调度相对说比较简单，也就是说如果说外交要改变的话，那你只需要改变我们对应的这个事件处理器就可以了，但没有其他的点需要改。


因为在这个中介模式当中，事件中介与事件这个处理器它是耦合的，当然不是这种强耦合关系，它是通过 event channel 来进行关联在一起的。当我们的 mediator 它把这个原始事件转化成一种新的业务事件的时候，发到 channel 里面，然后再跟我们的事件处理器两个进行一个关联。但是在代理模式下面它是完全解耦的。

在代理模式下面我需要先改一个中心点，然后才能改这个时间处理器，最后达到一个完整的这样一个业务逻辑的这样一个选择性调整，对吧？但是在这个中介模式下面就未必是这样子，因为这中间模式下面的话，如果说你要调整的是结合流程上的一些东西，那个可能相对来说改动就要大一点，因为它有一个中心点的调度的问题，但是在代理模式下面其实存在这个问题的，所以相对来说代理模式它是要更加灵活一些的，中介模式相对来说活动会相对高一点。


前面我们讲了这个系统的灵模性，现在我们来看一下 ETA 架构模式的一个非常好的这样一个优点，就是扩展性，就是我们经常讲的scalability。当对于，特别是我们像这种大型的互联网公司或者一些大型的应用，对这个用户的流量的增长，或者用户本身的增长或者数据的增长，那对于我们来讲最重要的可能就是可扩展性。如果 scalability 不行，那么这个系统的设计几乎可以说是失败的。当然如果说你说我用了这个单体小系统，这个可能不是你在考虑重点，但是如果对于我们目前来讲，我相信今天来听课同学大部分都是渴望进军互联网，而且应对一些高并发大用户这种场景，那可扩展性是你必须要考虑的一件事。

至少我个人认为你的耦合度千万不能高，而且最好是能够独立变化。显然我们的这个事件驱动模式是符合这个特征的。前面我们已经讲到的，就算是我们的这个 mediator 跟我们的 event Processor 之间有一点耦合度，但这个耦合度相对来说都是较低的，因为它也是只是做了一个什么呢？事件的一个转化， convert 发送到 channel 里面，然后由我们的 event Processor，就是我们事件处理器真实的业务逻辑单元去处理一下这个事件。其他的所有的地方全都是可以独立变化的，仅此一点点柔和度。所以说相对来说，即使是在 mediator MOS 下面，我们的弧度也不是特别高。


讨论这个 broker MOS 下面那几乎就是天马行空，爱咋整咋整，对吧？相对来说，我们的 mediator 模式下面可能独立性稍微要差一点点，也只是相对而言，其实它们都是 OPPO two，还是比较低的，而且也是可以得出变化的，这一点是我们 scalability 的一个前提。


我们再来看一下扩展可扩展性， scalability 分成两种扩展，一种就是横向扩展，横向扩展就是什么意思？就是说我们不同的这个点它能够膨胀成啥样？就简单地说这一个功能不能通过一些简单的方法，比如把它复制出来，就打造一个应对更多的流量，更多的用户这样的一个做法，这就是横向扩展。简单来说当然就是一个简单的说法，大家不要把这个定义限的太死，意思大家体会一下就可以了。


如果说不同组件，你在这种分布式的部署模式下面，前面我们已经讲过了，在事件驱动架构模式下面，你所有的组件其实都是分开的，你是 event q，我是 event Processor，你部署在哪里？我部署在哪里？我们的联系方式是怎么样的？其实未必都是在内存里面的，有可能完全是跨服务器的，通过网络连接的，都是有可能的。


不同组件的运行节点它是可以自行调整的。这句话就给我们的可扩展性能带来很大的一个好处。就是什么讲，比如说现在整个事件它处理得很慢，那是不是我调整一下某些 event process 的数量，就可以达到这个提高运行速度的这样一个要求，对吧？我们的 scalability 就上来了，就自然而然就可以应对着更多用户，更大的流量。


那比如现在发现这个 q 里面它赌的东西太多了，那我是不是第一把 cure 是不是要拉长一点？第二是不是我们要添加更多的组件来处理这个事情？就是我们的 mediator 是不是要增加一点，这样子我们都可以达到一个扩展器？但如果说你想想原来这些东西，如果说我们没有这种 EJ 架构模式，它就是一个揉在一起的一个 Monanic 的这么一个大的这样一个应用捆在一起什么的逻辑在一起，你要扩展起来那个可能性就相对较低一点，相对来说因为它太很大。


如果说你太大的一个东西，你要扩展它就比较复杂一点，如果说你把它拆成很小的部件，那么一个相对来说是独立变化的，那么是不是它的可能性就是通过调整数目的方式，就可以在某些需要的地方把它给性能提上去，把它 schedule 提上去。同时我们也有可以讲到有什么道理，就是说从 scalability 本身的角度来讲，它是可以自行拆分，什么意思？就是它自行决定我觉得我这个力度。


可能。比如举个例子好了，我们的 vtwitter 下面有很多个Processor，它在形成的步骤，这个步骤觉得我太复杂了，现在我要把它再细分一下，我把这个系统可能再简化一下，分成更细的力度，在我的内部里面再来完成一次。这个也是可以的，只要你觉得这个力度够细，能够应对这种场景，保持你的扩展性，那就是可以的。因为我们毕竟还是一句话是相互独立的，没有很强的耦合度的。两个 Processor 之间，所以说你自己爱怎么处理其他东西，也不能限制你。所以说这点就决定了我们的红线扩展能力，就是前面讲过的节点数量是可以进行调整的。其次我们可以按照功能或者一些属性成分之类的对它进行再度的切分，这样子从而达到一个可横向扩展性。


除了横向扩展性，我们有纵向扩展性。纵向扩展是什么个意思？纵向扩展就是说我通过把我的某一个组件的这个能力变强，从而达到提高整体性能的这样一个可扩展性。这个怎么来讲？就是说比如说我这个节点是要提供 machine learning 的，要利用很多很多运算的这样子的，比如像挖比特币一样这么一个，那超级耗能力的耗算力的，对吧？它可能是个 CPU 或者 GPU 密集型的这么一个硬件，需要或者虚拟机这个大家的matter，然后你就可以调整它对应的CPU、GPU，提高它的运算能力，对吧？那比如说我这个东西就是我做缓存的，或者我要 loading 很多数据在我的内存里面，那是不是我加它一下内存也可以提高这个整体的应对系统的流量？ soup 这种东西，或者我的把内存加大来提高整个性能，通过 scalability 来维持到这个系统的稳定性，对不对？这就是一种纵向扩展性，就是说我通过让原始的计算单元变得更 powerful 来达到一个扩展性。
