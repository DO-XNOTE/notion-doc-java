这整个你看中间的模式里面最重要的就是mediator，它做什么呢？中介的主要作用就是从事件队列中消费原始事件，并且根据所需的这个业务逻辑，按照一定的执行步骤将原始事件转化为业务事件，并将业务事件分配到对应的事件处理器当中。


从这个简单的定义我们可以看到中介者的主要职责就是分配和协调任务。按照职责分工，中介者他在这个体系里面他是唯一一个可以知道整个业务流程的组件，也知道每个处理通道他关注的是什么样的事件，这样他才能够按照业务逻辑来一一处理。这里有几个重点，第一，中介者它消费的是什么原始事件，但是我们后续会谈到这个 event Processor，就是真正的这个业务逻辑单元，它使用的却是业务事件。也就是说我们的中介者它是不一样的，它处理的是原始事件，而后面是一个业务事件，这二者之间是怎么转化的？当然是由 meditator 根据他的业务需求和步骤来进行转化的，这个就像买房子的这个需求一样，我对那个中介公司而言讲的时候我是要买套房，对吧？但是经过老板的这个消化，因为他知道各个环节他发出去的指令的时候，别的这个部门他收到的指令可能就已经发生变化了，就说可能 team a 他是负责找房子的，那这个时候他收到的一个而是event，那可能就是说去找一套房。


另外就是说你检查一下这个贷款资格，这个可能就是另外一个事件，也是完全不同的，对吧？但是这个转化是由谁来转化的？当然是由知道这个整体流程和业务逻辑的这个中间者，而且这个 code inter 就这个协调者最终我们的 meter 来完成的，对吧？他就把原始事件转化成了一个待处理的业务事件，这点很重要，大家一定要记住，也就是说最终我们的业务逻辑单元处理的事件和最原始的这个事件它不是同一个事件，在大部分情况下你说是不是直接透传过去？有没有可能？那当然不能完全排除，说一定不会，对吧？那大部分情况下肯定都是因为你分工切了小步骤，他所说肯定是会有一个视线转化的一个过程，这一点大家请要切记。


另外就是说前面也讲过了，就是我们根据不同的这个业务需求和各自的这个分工对应到我们这个里面。


在这个前面的图里面也讲了有 event channel，那这个分配就是这个事件，它根据不同的这个步骤，它会分配出不同的事件，就是前面讲产生带出的业务事件，对吧？他会把这个事件分配到了不同的 channel 里面去，就是前面讲的有的是找房，有的是负责贷款，就是不同的事件，就是不同的部门，也是不同的 channel 的意思，对吧？它原始事件结合这个交易流程，把它分配到不同的通道以及不同的工作组里面，去，执行了这样一个不同的操作流程和不同的这个业务步骤。


中介这个协调者他是知道完整流程的，但是在具体的每个步骤上他都不是自己亲自去动手的，这个是非常重要的一点。前面我也谈到了，如果你是一个老板管理一个公司，你肯定重点是管理这个公司，而不是去亲自做每一件事情。不是说一定不能，就像你写代码的时候，可能说这个类里面它大部分的 99% 的代码，它可能都是负责这个东西的，对吧？你说有一点点小小的其他功能，是不是完全不能？那当然不能，绝对这么说，在这个 mediator 当中也是这个中间的模式，说牛肉，作为老板，我不用参与到每一个亲子的环节，但是发现有些东西它是很简单，我随时就做掉了。


比如我复印一个文件，那谁做一下其实都可以吗？我很忙，那老板做一下也未尝不可，对吧？当然大部分情况下，那很复杂的事情肯定不需要这个 Mediter 来亲自下场。这一点以后同学们在这个设计也罢，实践也罢，想要注意一下。也就是说，嗯，我们的这个中介者本身它是知道步骤，基本上是不会自己亲手去执行的，通过这样一种方式就把流程和过程中的执行步骤它进行了一个切分，

那公司多开一个新的组，他就加一下就好了，这个东西不是那么的重要，对吧？那同时对于具体的这个处理当中步骤要实现，你说我在这个代码里面事件的时候，我用的Arco，现在切换到MySQL，那会不会影响整体流程？那当然是不会的，因为这个东西它就不存在这件事情，理论上流程是由 Mediator 来控制的，具体的处理逻辑是你自己的单元的事情，它不影响别人，对吧？因为这个中介他只关心步骤，就说是这份执行完毕，这份执行成功。但是如何执行？第一他是没有相应的这个业务知识来判断的，现实当中可能稍微有点差异，就是他肯定还是多多少知道一点的。第二，中介本身它也只是把这个消息就是我前面说的转化原始的这个事件的消息，把它转化成一个新的业务事件，把它转发到你的相应的消息通道或者事件通道里面去，是吧？有问题 channel 里面去，到底是哪个事件处理器真正在执行，他其实是不知道的。


为什么这么讲？因为就后面我们会了解到，就是说 event 的 Processor 就是真实的这个业务逻辑单元，他是关注着这个 channel 上面，他不跟中介直接打交道的，所以这就是一个解耦，就是说它的依赖还是没有那么强的，各自也是可以独立变化的。


我指的是后面的这个，就是各种Processor，它是各自独立变化的，完全不互相任何依赖，对吧？一个组件的改变，另外一个组件必须做出相应的改变，这种事情是不会在这种模式下面发生的。为什么我前面讲过他们互相不依赖，不知道对方的这个专业知识，我们这里就是将流程和步骤进行了解耦和异步化。当然这里的这个异步化它不是说一定会怎么样，这个不是强制一定说要异步化的，只是说如果可以的话你盘的进一步化而已，这就是我们的这个主角。哎， event mediator 就是我们的中介，它的职能职责和大概的要做什么事情，应该是怎么样的进行一个设计，就是这样子，前面我们在这个讲，嗯，这个有问题没提到的时候提到就是说它是把原始事件转化成一种新的业务事件转化给这个 event channel。

就业务小组里面，他有很多自己的员工，对不对？这些业务员，所以这员工他就从外界来看，所负责的事情都是帮客户做一件事情。比如说有一个贷款的小组，那对外界的人来讲，只要跟贷款相关的事情，他们都由这个组来搞定。就相当于说原始的买房需求经过中介老板的一个转化，变成一个贷款需求这个事件，就是说原始的事件是买房，现在是变成了贷款这样一个事件一起变了，那老板就会根据业务流程我们不说了，他会转发到这个贷款小组，就说这里有个人要贷款，大家就有这么个事儿，你们这个组来处理一下，就相当于说把这个消息从原始的这个买房的事件转换成了这个贷款事件。


同时这个贷款部门的人，所有的人都在，对吧？他就听到这个事件，或者听到这个消息，他们都在听这种消息，对吧？他拿到这个消息之后，他们就各自动脑，就我熟悉这个工商银行，我熟悉农商银行，我熟悉招商银行，那每个人他擅长的事情可能稍稍有不同，就是这样子，但是他们都是同一种类型消息。


那有个人说客人发出来需求，老板解析说我要在招商银行做一套按揭，我要在工商银行做一套按揭，让每个人对接招商银行的对接，工商银行的业务员就会去处理这件事情。在我们这个架构模式设计上来讲，相对说某一种类型的业务事件，它就会聚合在这一个 channel 里面，然后这个 channel 下面所有监听这个 channel 的这个 processor 就会根据自己的这个喜好或者判断，来决定自己要挑选哪一个 event 来进行处理，或者是所有的他都进行处理，对吧？就是这么个理论。


有些人就讲说这个不同的情况，它是不同的事件，也就是不同的监听，然后完了之后各自挑选自己感兴趣的事情处理。理论上里面有很多这个 event process 的事，它停留在同一个 channel 上面，但是他们各自的事情就跟这个图上一样，你做你的，我做我的，反正最终都是服务好这一个客户的一个通知或者一个需求事件，大家都是聚合的，就像那个 team 内部有一点点小小的竞争，大家都在这个团队里面监听这个找房子的需求。

简单来说就是真正最后一个环节，然后完成我们原始的这个业务逻辑的这个组件，就是说所有的前面的一切都说了这一切最终的这个业务逻辑单元来做准备的。嗯，系统启动的时候，他就把自己注册到自己感兴趣的这个实线通道上面，之后就开始监听这个通道内的所有的事件。但是事件处理器它并不一定会处理所有的事件内的通道里面，这个事件处理器会根据自己的这个业务逻辑来进行筛选。


其次为了保证这个主线的内聚性，因为前面我们讲过了这个 event Processor 它是互相无依赖的，而且相对说是没有这样一个耦合关系的这样一个组件，所以说每个事件处理器它都是可以独立运行的。什么意思？就是说如果事件处理器它并非依附于事件通道或者中介组件的存在，它自己是可以独立存在的。意义就在于它的职责是执行具体的、专业的业务逻辑，和事件通道以及中介组件是完全不同的。那此外这个事件处理的职责要尽量单一。为什么这么说？因为越是单一和简单的业务逻辑，就越不容易受到外界的影响，也就意味着外界的变化，它也不会引起这个事件处理器自身的改变。


理论上我们要尽可能在让个事件处理器尽可能的这个细粒度，当然这个设计也是要适度的。为什么这么说？就还是以中介这个例子来理解好了，比如说中介公司它有一个负责审批的小组，主要看市场的这个房子客户他能不能进行买卖。假定说这个市场上，假定市场它有 x 种，这个房产类型客户他有 y 这么多种，那为了达到最细溜的这个处理能力这个体现，那在这个组里面是不是需要 x 乘以 y 这么多个员工？理论上你当然是可以这么做的，可事实上你肯定看不见这样的一个公司存在，那为什么呢？因为随着小组的这个成员的增加，第一这个沟通成本它上升了，这是毫不犹豫的。


第二就是人工成本它自然也就上升了，那实际上一般可能就会成立 x 或者 y 这么多个员工的一个小组，那么各自负责某种类型的审批，这就可以了，因为这个过程我们来理解一下这个事件处理器的划分的这个力度就是什么呢？就是如果处理器太少，可能会引入额外的依赖或者耦合度，但是划分的这个太细，这个部署成本、测试成本、维护成本都会大大的增加，从整体的这个可靠度来显示，这样过于细粒度的这个设计也未必是合适的。


就是最后我们来看一下这个事件处理器，前面已经讲过了，互相是无疑地带的没有任何依赖，所有的处理器最后分工合作一起完整的这个业务流程把它给完善了。

所谓的业务流程，在这个时候只要所有的事件处理器全部都完成了，反馈给这个，嗯，我们前面讲的这个 media 就是中介这个事件，就是对真实的外部驱动整个系统运作的这个事件的响应全算是完成，他们互相是几乎彼此一无所知，达到了完全的解耦、独立变化的效果。


从整体的角度来看，完成一个完整的业务流程，就是我们前面讲的响应。这个背面的这个驱动事件它又是缺一不可的，因为整个流程都是由若干个步骤组成的，任何一个环境的缺失都会导致整体的失败。所以这些是实验处理器，它是既分离又统一的，就是它们聚在一起合力形成了一个交易链条，交易环节的这个完成度，同时他们也是独立的，一个个的存在，彼此之间是没有任何依赖，也没有任何耦合度的。


通过中介的这个协调和调配来响应最原始的这个驱动事件，它自身响应的是业务事件，我在想那边在这个地方事件处理器是最原子的，这个业务单元响应的是业务事件，我们整体系统应对的是外变驱动的那个事件，这一点请一定要注意，我再强调一遍，所以说以上我们讲的就是这个 event Processor，就是我们真实的实现处理器的这个职责分工以及他的一些注意事项，尽量力度细一点、职责单一点，这样子尽可能的不受到外界的这个影响和干扰，能够独立的升级变化。这就是我们事件处理器的一些核心要点。
