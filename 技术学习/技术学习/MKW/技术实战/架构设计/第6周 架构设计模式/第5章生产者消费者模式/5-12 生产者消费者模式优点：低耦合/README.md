# 5-12 生产者消费者模式优点：低耦合

相对较为概念的这样一个角度来而建联一下咱们这个的价格模式，现在的消费者，对吧？那我们呢再来看一下价格模式的这个优点和缺点。首先我们看一下非常显著的一个优点，之前在这个架构眼镜的这个图片当中我们有展示过它，其中也是第一耦合，对不对？首先producer、 consumer 和 container 它均可独立变化这个点，这个意思我不知道大家理不理解，就是说其实你现在怎么生产这个东西，你可以升级，可以改，它都是不影响 consumer 和 container 这个逻辑的。

同理也是你在 container 上增加一个节点，少一个节点就是存储，我指的是，那其实你说对两端的这个producer， concern 它有区别吗？理论做得好的话当然是啊，从实践层面来讲，当然我这个话是啊， container 里面加了个节点，存储空间变大了，你说这个 position 他能有感觉？完全不知道，我前面还讲过他那 potential 往里面丢了消息就好了，我管你怎么用那所有 consumer 他也不强依赖的，我只要有个数据就好了，我管你怎么丢进来的，那所以说他们的额度非常多，独立变化没有任何问题。



第二点就是如果说你记得在咱们那个架构模式引进的过程中讲这个问题了，对吧？最初它是一个直来直去的，我 a 掉 b 晚了，所以说你 b 不给我响应的时候，那我肯定就横在那里，那站在这个 consumer 这种模式下面，那就完全不是这样子的，他们是没有互相的强依赖玩气的。


虽然说 production 这种东西或者这是这个系统的源泉，但是你也不能说我 consume 就完全依赖于它，对不对？因为你有东西我就做事情，没有东西我就不做，这个依赖关系看怎么看，但至少我再看它不是一个强硬的关系，那反过来再看，否则对 consumer 更没有什么要求了，对吧？不过这句话也要怎么讲，辩证的看法，因为其实如果说你在实现层面就抛开架构看，因为毕竟我们东西要落地，按其实在实现层面来看，你说这两种有没有依赖，其实可能跟你想象中还是不太一样的，因为现实世界中是很容易打脸的。


嗯，咱们这个架构师就是在不断的打脸过程中成长的，我们本希望 Claude 可能本身是不依赖的，这是我们社区的好处都有，何况对不对？但是你比如要实现什么 consumer 这个transaction，在整个这个消费流程里面还是事务性变成唯一性，很多东西其实你要说完全无缘都是很困难的，这点我是做一个 best practice 给大家分享一下，但是是纯粹的抽象看架构，但确实要求是就是 115% 100，但是在实现层面其实这个挺难的。


我还所以说我们选我依赖，这就是我前面讲过的就无事物系统的时候，你对这三者都是没有什么约束的。但是如果说你一旦对这个农业有了事物这个点，你这个系统可能就会变得相对复杂一些，因为你要有事务的话，你就要保证这个很多步骤都支持原子操作，你否则就不叫事务了，对吧？但你要保证原子操作，比如说你 producer 生产的一个东西，我没有消费点，那怎么办？所以说这个时候你肯定必须得协调这样子，这一协调的话就不能说它唯一代了，对吧？但是没有事物的时候，唉，或者是你要求没那么高的时候，这是两者确实是互相不依赖的。然后那第二可以带来一个好处，就是我们前面讲过的生成者发完就说用完即走。像张小龙说的微信，我希望是欲望即者，对，是发生者或者发文记者。剩下的人就是 consumer 的时候，他还小时候， consume 的时候看出它不存在一个什么决定性因素，说你必须要几点几分，只要说实话，消息发出去了，这事就不受人控制了。你的职业就是发育城市出去，所以和得出来讲，那对于 consumer 来讲也是我只要把这个读到就可以了，管你什么时候发的。但是这句话还剩下，你也要辩证来看。


一般来讲的话，你肯定是希望说越早读到这个消息越好嘛，虽然咱们是个异步的非实时的系统，那还是越早发生越好，对不对？因为检得太晚了，这个可能很难受的，你就从一个人的体会来讲好了，你比如说下一个订单啊，半天不来给你做这个单子，你什么感受对不对？说虽然是异步，但有时候我们这个异步是迫不得已源于异步了。


首先还是尽可能希望说这个消息发出去之后，这个数据放进去之后，这件事情分配了以后要尽快完成，所以是个异步运作的过程，其实也是帮助了这个地耦合。但实际现实世界中还是希望虽然是一搏，也能够尽快把它执行起来，对吧？

这就是一个对第耦合的一个想法，如果说你它不是这种模式的话，那就彻底的失去任何意义，对吧？因为我们是通过这种分底的低耦合的这样一个设计架构，保证了二证的灵活性和扩展性，这才是我们这个模式中的一个比较重点。


第二个优点，我觉得剩下哪些不同的模式，它是支持或者倾向于是一个分布式部署，能够分布式部署的系统优势有非常多，那我们就结合这个现在的消费模式来讲一下这个分布式部署在这个模式上有什么优势？我这里说一下，我是尽量或者倾向于分布式部署。你说我在同一个 Java 代码里面，比如同一个不同的线程里面，一个发一个收，中间有个q，这个算什么？这也是一种生产的消费者，他认为你这我的分布式部署，废话，当然没有分布式部署了，对不对？但是我不说了，这是倾向性的。


你这同一个加完这个 VM0 跑来跑去跟我聊这个分布式，你不是自己反倒自己吗？没有什么意义的，咱们不要抬杠，分布式部署就是希望你在使用这种价格模式的时候，你尽量倾向于这么做，会带来很多好处。我们来看一个好处是什么，对不对？

第一我们尽量让这个，比如说consumer， container 都是分开部署，这点前面已经讲过了，对吧？那为什么这么说？首先把这个 consumer content 它自称真不是系统，那我们举个简单的例子好了，你现在订单在下单，在这个饿了或者那个团，我管你多少人在下单，就你不管有多少个生产者，对不对？ and 所有的这个客户就是他的producer，它是不是一个分布式的？就自身它也是一个分布式，那至于，唉，美团来讲的话，你想想这么多订单发过来，在这个系统里面存在，它自身必须得是一个分布式系统。


这个机构不用回答，但是答案是不言而喻的，这个 consumer 也是一样的。那这么多外包骑手，那随便哪个来接单，它不是一样的，没有特别大的区别，对不对？这个例子我觉得应该挺好理解的，所以这个制成分布式系统这个东西，三者各自可以成为一个分布系统，这个系统带来非常大的好处。


为什么这么讲？就是说你以后这样子在每一个地方，你都可以把它那个分布式系统的话，那它整个系统的这个可靠性，是吧？都上升了非常多。因为你在任何一个点你都可以做成一个分布式的话，那这个组装在一起，它整个系统这个扩容灵活度，包括你这个系统的扩容性什么，这些都是非常好的，因为它各自就成了一个分布式，在这一起组合成一个分散者消费者模式的这样一个架构，对吧？这一点大家其实要理解，各自有可以分布扩容是非常厉害的一件事情，这就是我前面的各自按需扩展，这个时候就是怎么说这个扩展是有横向扩展的，正向扩展咱就不细讲，反正大家懂的意思就是什么意思？就是说你在这分布式系统下面，你做了一个部署，

第一特别的优点就是可以通过生产者消费者模式来平衡系统参数。什么叫参数？这不知道大家对这个汽车这个差速器有什么了解？它的原理是什么样的？大家都知道这个汽车在直路上你去跑的话，那驱动轮的速度肯定是一模一样的，比如后轮驱动、前驱动四驱，不管怎么样，所以在直线上它跑的时候，你就可以假定它的速度其实是一模一样的，对吧？但是在转弯的时候，你想想大家，我相信听这个课程应该都是理工科的占绝大部分。


那你的物理知识有没有告诉你，在相同的速度，在不同的半径下面会发生什么样的事情？转弯的事对不对？那个受力就不一样了，对吧？这个时候如果你还要保证相同的速度，这个就很困难了。所以说如果说这个时候两侧的轮子它速度没有一个差异性，而是一样的话，可能就叫做汽车的这个失控。所以在后来目前发明了这个差速器，通过这个差速器让左右的两个轮子可以以不同的速度来转动前面一个组件，推动后面一个组件运作，但是如果二者这个处理的速度的不一样，会有什么后果？那其实整体的思路库存会降低，在系统层面来讲就是一样的值。


那怎么来解决这个问题呢？当然就通过这种差数器一样的东西，对吧？这就是咱们的生产者消费者模式的一个优点，平衡系统差数，就跟前面汽车的差速器一样的功能。首先就是前面我在举那个水池放水的例子的时候就有讲过，这个容器就是我们的水池，它可以平衡生产者的消费者速度，这个所谓的平衡是可以被打破的，对吧？我们尽量保证说如果的不要浪费，不要闲置，也不要溢出，通过这种方式来达到一个相对唯美的平衡点，来平衡生产速度和消费速度。这个其实有很多应用都是处于这个目的来使用这个生产消费者模式，大家一定要搞清楚。


第二你要调节这个差数，那比如说还有一种方式，就前面讲过我们是全部是分布式，这三者可以拆开的，那如果说你却发现这个 production 太多太快，那你就把数量减少一点或者增加一点，如果太慢的话，对 consumer 也是一模一样的，对吧？你可以动态调节这两者的速度、数量来反馈，来控制它。做这样子一个道理是你通过控制这些组件的速度，有些是打造一个使用速度、整体速度这样一个平衡点，这也是他们了不起的，对吧？还有就是通过这种参数你可以减少这个资源浪费。
