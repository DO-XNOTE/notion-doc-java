# 4-5 微内核架构优点

前面我们对 v 内核架构进行了初步的了解，下面我们再来看一下，就如果在实际应用当中，这个 video 架构我们要注意哪些事情，主要是我们会集中看一下它的优点和缺点，这样子我们才知道哪些是应该发扬光大，哪些应该是扬长避短的，对吧？简单来讲，第一个优点我个人看法是也是最重要的一点，或者说很多时候我们采用了这个 v 类和架构，它的主要原因也可能就是因为这个优点，就是它符合了这个开闭原则。


在这个 v 内和架构里面，开闭原则它主要是体现在以下几个方面。第一点，首先系统核心它是封闭的、完整的，这个话啥意思？

就它的优点里面就在于说在 UNICORN 架构里面，基本上你在内核层面的东西，或者你在核心系统里面定义这些功能是很少变动的，基本上最好是变动的不是那么频繁，这是一个基本要求。应该怎么讲这一点我相信不管大家对这个Windows、 Linux 这种内核系统的了解应该很深刻，这个 Windows 要升级一个版本，那可是一个大事情，对吧？这是相对就是说开发量是不得了的这么一个事情，所以说肯定核心系统的变化是相对较少，我们假定说它是一个 stable 而且封闭的，像大家比较实属性的 Java 程序员里面，比如是这个 spring 的这套框架里面，谈一个版本升级词，还很多问题都要解决掉的才会升级，对吧？所以说相对来说，我们可以认为核心系统的功能它会变的大致是相对来说封闭的。


这一点的好处是体现在整个系统的内核体验层，它是稳定的，由整体系统架构来讲，在内核这一层它是比较稳定的，插件是提供丰富的扩展功能，前面我们讲过的，对吧？第二点就是前面讲的就是插件系统，它是来丰富了这个功能，很内核、很稳定，插件很多样，很丰富，它来提供各种各样的这个拓展性就是从开辟原则来讲的话，对修改是关闭对不对？但是对这个扩展它是欢迎的。那在这个地方我们的核心系统的这个修改尽量是稳定的，不是那么频繁的，这是符合了关闭这个修改这一点。


对于说我们插件来讲，它是开放而扩展性，因为插件它只要遵循这个规范，你想提供什么样的一个功能，那都是没有任何问题的，对不对？对于我们来讲的话，重点就是把这二者区隔开来，核心系统里面的东西尽量少动，因为它一动可能引起。比如说你的这个系统或者这个开放平台里面有1万个东西已经进去了。


比如像 iOS 上面， APP 里面，你说你动一下，那影响了简直全世界的受影响，对吧？但是你说我要开发 n 多个APP，那随便你，反正影响不了别人，你爱开发啥，对吧？只要你不是病毒软件，其实随便开发，想开发什么都没有问题，对不对？首先这是两个不同的点，从开辟原则来讲，不同的两个角度来体现这个问题，就你使用这个微粒核架构，你想要产生的优点可能就是符合这一点的时候，你是特别喜欢它的。就是说内核功能很稳定，尽量少修改，但是外面你怎么折腾它都是允许的。
除开编者，最后就是整体系统，它其实是可以持续升级的。前面我们讲符合开闭原则，也不是说它就永远内核就一层变，这个肯定不至于的，对吧？不管是Windows、Linux、iOS、 Android 都是在不断的进化的，这么磅礴的系统都可以进化，但不存在说我们的一个整个系统就不可以进化，就不存在了，对吧？只是说相对来讲它是可以分别进化。


所以说从 Windows 这种大家比较熟悉在哪里讲，就你的 Windows 的版本可以从最初的window，现在都到Windows，是什么版本，对吧？无数个版本这么过来了，小版本端上的话不少了，对吧？也是可以升级过来的。那对于说我们系统而言，只要那你开放一个 spring 核心功能，基本上你还是要主要是 Bing factory，这是个核心，它永远是不大会变的，对不对？但是它又在变的是什么呢？你外面支持的各种各样的规范越来越多，

核心它也是升级，从最初的一个简称的 spring 框架，甚至现在什么 spring board， spring cloud，这都是在不断的变化的。


所以说大家理解这个所谓的开变就是内核不变，而它不是真的绝对不变。这个地方所谓的不变是尽量它就是 stable 的，这个不变的意思是插件就像你像在 spring 里面，它是一个像胶水一样的东西，大家想什么插件你都可以往上加的，对吧？对，就跟前面讲的变形金刚一样原则，变形金刚这个头脑或者躯干里面的这套东西不变，外面的壳装什么壳？它是可以允许的。
happy 原则的意思不是说真真正正就允许你内核一点都不修改，不是那个意思，只不过说相对来说它是改变，可能第一是 effort 比较大，第二就是说不希望你那么频繁地去变更内核，更多的是希望你把这个功能考虑的是不是要通过外延的这种插件来完成，尽量把你的这个细小的改动或者不是影响大体的改，我都放在你的扩展性的这个插件上面这一点大结论和开闭原则并不表示说内核就不可以变了。


之前千万不要误解，前面我们是讲过这个插件它是无法直接使用这个系统资源的，这个系统资源在不同场景或者不同的应用，它确实有不同的定义的，有可能说自己硬件，有可能说一些真正的一些比如像 DATABASE 这种或者一些敏感信息的这样一些访问接口，这都可能使用资源。反正不同场景下对它有不同的定义，这点大家不要限死，只有通过核心提供的这种接口，就是这种封装过后的接口来进行一个资源的这样一个访问的操作。而且这个核心系统和插件它也不是直接就是绑死在一起的，它是通过一些连接通信的方式来进行一些信息交流，对不对？所以说通过这种设计我们可以得到一个好的优点是什么？良好的隔离性。


那隔离性是怎么来理解这件事情？首先来讲在咱们这个架构里面，就为了和架构里面它的核心系统这个插件，它可以有能力把它关闭掉。

这一点我们举操作系统的例子来讲的话，就相当于说你如果在 Windows 或者 IO 上你发现有个 APP 有问题，比如 Hover 已经特别高，那很简单，你要直接把它给干了不就行？把它删掉就好了。


当然很简单，这是不是一个直接相当于关闭的这么一个感觉？那如果在一些比较好的产品里面，像我们先做了一些产品，如果做得好的话，这个功能你可以直接把它 disable 掉，它有问题它就把它 decip 掉，它就不会再影响到全局的这个系统的运转了，这点很优秀对不对？大家回顾一下这个跟我们之前在上一章节里面谈到这个防水烫设计，是不是有一点类似防水枪设计，就是说我这个防水枪有问题，我就把门一关，这样子别的枪就不会收烟。


在这个地方就是插件你可以插，可以拔嘛？对不对？你把这个插头插到插头上，然后把这个插座拔掉，它有问题的，不会影响插座的，换个插座你可以插，那换个插头也可以插，这就是一种可插拔或者可关闭加这样一个功能，把插件给关掉，它就不影响全局对不对？这点重要还是我们尽可能的保证这个触的传播范围是有限的，就相对说，比如说某一个插件你是通过连接的方式有问题，它如果说出了问题的话，它只会在什么插件内部来进行一个传播，就是它想整体影响到别的，我们要尽可能限制它这一点。当然这个话就咱们讲一下，从操作系统来讲，这件事情就相对来说比较玄妙一点，如果真的设计的那么好的，就不会存在病毒这种东西了，对吧？


那我的原因是说这个架构它本身的设计的初衷是要把各个插件之间的这个关系，它是隔离开来的，当它出现问题的，它不影响别的插件，也不会影响核心系统，那是我们设计的初衷，我们作为架构师肯定是希望说这个初衷是美好的，那达到一个层面。但是你说有人破坏这个规则，这个毕竟代码它不可能说完全没有 bug 的，很困难，对吧？它绕过了这些路径，它达到一个破坏你这个系统的角度，病毒你说它不符合，你是抉择吗？这是两码事情，就我设计成是这样子，你偏不走这条道，这个我是没法控制它的，对吧？这点就是大家一定要理解插件的话，尽可能把它的错在插件范围内限制住，就插件说两件，就是那插件本身你挂了你就挂了，可能仅影响你使用这个插件的，但是你不影响到核心系统和别的插件，这样子就不太好了，对吧？就我们希望是达到这样一个设计小说，就是要把它隔离开来。



还有就是前面我们也讲的说符合开闭原则，但是它其实是升级的话，我们其实是可以各自单独升级，这一点我相信不管你是使用 spring 这种项目的东西，还是使用什么其他的这种Windows、iOS，这个很明显对不对？当我们升级 Windows 的时候，那有些应用程序可能是兼容它可以直接就跑起来，有一些可能就它要求你必须得把它给搞定，重新升级，被新的这个操作系统版本，那个新的配套的这样一个东西，那对于 spring 来说也是一样的。


很多融入的这个 spring 里面的这个插件就各种第三方的开源的，要在 spring 框架里面跑得起来的话，它可能也不需要做任何修改，它就跑得起来，对不对？有可能他完全就跑不起来，他各自可以独立升级，教我这个操作系统，或者我这个 spring framework 这个核心我升级了，你升好了，那我配套的软件就是我的这个插件，我要可以独立升级，那也是没有任何问题的，对不对？这个不存在说已经可以，已经不可以，当然这个也谈到有兼容性的问题，这个你应该有心理预期的，比如说 Windows 到现在的话，你说最初的那个版本现在还能跑这等跑，那才是不可思议的，对吧？那肯定是不大可能的。
