为什么这么说？第一，首先你要保证你能使用这个架构的，是吧？你得知道你担心这个系统它是不是能够严格地区分出一种核心功能和插件功能，为什么这么说？有些时候你做一个东西的时候，就这几个新系统，还是改造个老系统，你会发现它所有的功能都是打通的，就一杆子捅到底的，不存在说。你可以很好地进行一个抽象，把它这个核心层和这个插件层它功能能够很清晰地定义出来，你要定义这个很清晰的层次它就挺困难的。所谓的功能就是你没办法说我从包里你就包裹它，必须得这样子，否则它玩起来就不够顺畅。你这个时候你要套用咱们这个维链和架构就挺困难，因为它就不存在一个核心与插件的没有一个主次，没有一个主角和配角的感觉。而所以说你应这种核心功能和这个插件功能就挺困难的。所以这个时候我就不太建议你使用这种任何架构，因为如果说你无法保证找到它扩展点和核心功能点，这点对于你应用起来就不是特别方便，可能也不是那么合适。


那其次我们前面也讲过，希望这个插件是可以热插拔的，有很大好处。前面我们讲过，首先你这个节省资源，你 bug 我就把它关了，就是有个隔离性的问题，当你有问题的话直接把它干掉。这个最简单的比如说我们设计插件的时候，想用你的时候就直接让你打开，或者是把你装载起来，不想用的直接把你下大，直接 on load，这是最理想的状况，但是是指这么说的理想永远都是什么样的，现实却是残酷的。


你要做到这一点，当然从技术层面上讲到，那当然也取决于你这个系统本身的复杂度，这个要不要难？还是说简单？这个取决于你的系统在 Windows 上面，你现在比如说要装载一个程序，再安装一下就可以过了，你要发现我有问题直接把它删了啊，那也很简单。但是你在 spring 里面，你要随随便便的时候，我要加一些什么东西，这个可能就没有那么容易让它要实现一个热差把。不是说绝对不行，但是你做起来肯定是没有像 Windows 的薯片输入薯片点亮一样，它只用搞定了。


在其他的一些系统，比如说开放系统里面，一个开放平台里面，如果把平台当成是一个核心，对于外面上传的东西，或者说它使用一功能设立一个插件的话，你说我要随时找到一个可插拔，这个应该没办法帮你改，因为外界对你的 11 代你是有既定的这样一个感觉。你把它弄上去之后，你说我要插，要拔，人家都已经开始用了，你把它插拔掉有点怪怪的。在 Eclipse 里面有插件，比如举个例子，它是依赖于我 a 插件是依赖于这个 VSVN 的，然后 VAGT 插件的，你把这个 Kitty 这个都干掉了，那别人也受影响，所以说时间有个可差，你想一想它到底有什么影响？你怎么能限制这个形式？这个可能比较重要。
还有就是说我们前面讲过插件跟内核之间它是有个通信问题和注册问题，你要针对注册问题，你要告诉他我是谁，因为怎么启动来的行业找到我类似于这样通信息，也就是说我们两个在哪里连接？连接完之后这个消息格式什么？怎么交换？类似这种东西你要针对你的这个场景，你要把它一定义出来，然后这些东西是不是够，如果被心怀不管的人拿的时候会不会有问题？所以这个对我们在不管是安全性还是性能封装度其实要求都挺高的。但是前面我也讲过说这个开发难度高，那是不是一个缺点？这个不好说，更多的我愿意把它说成一个挑战。


就像现在网上有一种流行的说法，这个劳斯莱斯太贵，不是它的缺点，那是它缺点，对吧？那这个方也是因为我们说的是开发难度高，这个未必就是这个架构的模式的缺点，那也可能是咱们的水平的问题，通过这个学习我们能够了解到它的优点和缺点在哪里，可怎么去实现这些东西？怎么能应用这个？这才是一个很好的解决方案，对吧？就像劳斯特斯一样，我买得起那就不是缺点，然后买不起那就是我的缺点，这点希望大家能够体会一下，好好感受一下，好好努力，争取找店买上劳斯莱斯好不好？还有一点我们觉得它的确认是可扩展性不高，

这个地方不是说跟前面的咱们灵活性一些东西，或者说可扩展性什么冲突，扩展性和灵活性之间，就你前面不是讲天花上去说这个价格有点在哪里？这个层准备收可扩展性不高，但是不要误解我们的这个意思，你好好理解一下，我这个话的意思是可扩展性不高，这个地方指的是怎么讲我这种价格，它的扩展性不是你想的那种，也就像叫 head scalability 这种。


一般来讲这个 video 和价格它用的东西相对来说是偏向于产品或者单体一点的。比如你只要不是受众开放式平台，它的可能性这个可能是讲到这种，就是说比如我性能有问题的，通过它来把它扩展一下，而不是讲功能上的不丰富而不可以扩展，不是那个意思，大家不要把这两者误会。


现在讲的说除了信念出问题之后，我要把这个产品通过什么方法把它进行一个scalability，之前我们在我们的这个分层架构还有信讯架构里面都有讲或者 scalability 的问题，这个来讲的可能性不是指功能的这个丰富程度的，个可能性指的是这个 scalability 的问题，因为这个东西它本身这个架构模式它更多的是用于产品开发，它不太在意这个扩展性。就对人它来讲它不是一个特别重点，因为它都是用一种产品的东西说重扩展性不在它考量范围内。就说是你硬要做，说你说我这个一个平台我现在就可以扩展，那是一个说法。


另外一种就是比如说更多的是你把这个另外架构模式你放在一个产品身上来讲，它可能并不是考虑说我特别擅长于通过 scalability 来提高性能这种东西，它可能不是它的目的，而且它的设计初衷它也不是这个，所以你硬要做这个事情，它这个不是不可以做，可能就是不是他特别优秀的这一些技能。


但是这一点有一个补充说明是什么意思？就是说咱们这个微内核架构，它其实是一个特别容易跟迭代的架构模式，两个融合在一起，就是 work together。嗯，我们要打出一定是新的高度，把这个架构提醒到。比如说我举个简单例子好了，你在分层架构里面，在某一层里面你再使用一个微联合架构。你这么说，比如你在前端，在后端这个 MVC 架构里面，你在这个 Mute 层的话，你说我使用 Voyager 架构，让它各种插件直接插进来，我这展示的方式完全不同，是不是可以的？通过这种方式它其实也可以扩展一点扩展性，那前端然后它就分离的，它跟别的模式结合在一起，当你别的模式可以有 scalebating 的，别的时候它不就随着它一起？ scalable 可能性不是咱们这个 winter 架构的一个主打点，这一点大家一定要切记。
我再强调一次，这个怎么可扩展性不是指的功能的投放软件，我讲的是scalability，通过 scalability 来 handle 更多这个流量什么指的这种含义，它不是一回事情，大家不要为。
