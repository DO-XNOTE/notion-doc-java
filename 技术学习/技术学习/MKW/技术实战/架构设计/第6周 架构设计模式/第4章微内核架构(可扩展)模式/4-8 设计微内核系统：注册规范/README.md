那功能就可能就说一般，就不是说有个硬性规定，说你只是说像比如 Eclipse 这种插件系统，它可能就相对来说更在意这一点，那就说我插了这个插件，我就知道你这个插件是干嘛的，对不对？对于用户来讲，它可能对于专项，比如说这种在内存是里面这种集成可差 80 多，可能相对来说这个功能也只是说供参考一下，但是对那种真正比如像要安装的软件或者插件，真正的要在 ecliptional 运行起的这种环境，那可能这个时候你要知道的其它功能，否则你就看见一个名字，你也不一定猜出来它是干啥的，对不对？而且你还有一些说明什么之类的，可能这个相对比较重要一点，在这种情况下当然只是位置这一点其实也相对来说比较好理解。


比如说你现在要把一个插件集成到我们这个核心系统里面去，对吧？我的系统核心里面它哪些点要保跟你保持有关联性？那首先来讲的话，现在如果说我们是动态启停你这个东西的时候，假定你是一个价，或者是你是一个class，或者是一个DDL，我要把它装载到我这个系统里面来，那我总得知道我你在哪里吧？如果你在哪里我都不知道，这可行吗？对不对？那像比如说有一些这个课插拔式系统，假如我们把他们 cat 这种东西当成一个课插拔式系统的话，那他是知道有些固定的目录的，他会去扫描这个目录，然后把它装载起来，然后跑起来，对吧？像 spring 的也是 subspring boot，它知道是大概的位置，我认为你应该在哪？从我可以把这些 librate 装载起来，但在 Windows 上就是你安装的时候，你就要告诉它你的路径在哪里，甚至这个系统当你要启用某一个转接头才能用， Eclipse 也是一样的，你把这个包下载下来之后，要把它做起来，这样才知道你的位置在哪里，否则它这个根本就没法让。


这一点我觉得大家应该还是比较想好理解的，对吧？不管你是动态加载还是静态加载，你首先都得知道这个东西在哪，你要是不知道它在哪的话，你怎么去加载它呢？这所以说我觉得这点还是比较简单，也容易比较理解的。而这个依赖是什么意思？就是说你插件可能会依赖于那什么样一些东西，这点你可以这么理解，当你使用一个新的东西的时候，就是现在要像这个系统新插入一个插件的时候，新插入一个插件，那比如像 spring 的话，那我作为一个 bin 对不对？我新一个 bin 要插入进去，那这个时候那我肯定有一堆的依赖，那我依赖于什么东西？我是不是要声明一次我依赖于ABCDE，什么whatever？那些 ABCD 存不存在？ spring been 怎么把 factory 整个起来，把它装载进去？这是另外一个问题。但是你首先你得告知是这个 Bing factory 我依赖于哪些？你都不知道他怎么给你把这东西注入进来或者 OP 才能够 run 起来吗？他不知道你的依赖。
对于我们来讲，它比如说有些插件像 Eclipse 里面它比较著名，就是它在 plugin 这个文件里面，就 sheet 定义说你依赖于什么东西？ Windows 可能有些程序也是说我必须依赖于什么东西我才跑到前，包括我 driver 我这个程序才跑到前，那这是一种。所以这个依赖我觉得如果让它定义，哎，有几大好处，第一就是知道说当前你新加入的这个插入的那个新插件，它要些什么东西，没有这个东西它根本就跑不起来，它也不会运行，所以这点对你来说比较好。第二你也知道说它最多它能影响到什么东西，你是知道的，就是这么个意思，所以我更觉得这个依赖也是比较重要的。还有一个就是权限，这个是我个人的一个想法，就是如果说你现在有一个插件版，它想要用一些功能，但是你的破坏程度可能是不一样的，我允不允许用我还是得知道的。


这一点我其实觉得在新式的这个软件里面相对比较多一点，比如像咱们的 iOS 安卓里面，你装任何一个新的APP，把 APP 用插件操作系统当成一个这个核心，系统核心的话，你会发现当他要使用任何一个东西的，他都必须告知你我要用哪这个权限。像安卓这个东西在安装的时候就会告诉你说是怎么样的， iOS 也是让你选择的远不允许用，对不对？这个就是很好的一个例子，你用哪些，不能用哪些我一目了然。这样子你如果要用，我就知道你用了这个权限，那如果我觉得你这个危害太大，那用户也是感知的到的，那我就不允许你用呗。那有些功能大不了接受不用，对吧？或者说我整个就不让你安装成功，因为我必须要有这个这点我觉得如果有了权限控制的话，当用户要使用某个插件的时候，那样或者这样操作系统上使用一个 APP 的时候，但你是很清楚的这一点。


同时作为这个核心系统的话，他也知道他要申请哪些权限，这个权限到底内部的实现是怎么样，控制怎么样的？他有办法能够控制住这个插件，他的还是这样，可能破坏程度相对于会较低一点，因为他知道他有哪些权限，他没有权限的地方他根本就打不通，他也用不了。那这一点，这点是很有意义的，就是说你声明的权限对终端用户和核心系统来讲的话，都是有很大的帮助的，有了这两点之后，你对它的控制力度更细了。嗯，集成那个插件核心系统，那很快就是来到了我们的这个通信的这一部分。为什么这么讲？就是说你有插件的，有核心系统的，我们已定义也集成规范、注册规范的，那反正他们二者真正要发生一些联系的时候，那肯定是通信的，对不对？他没办法直接就是说我的核心系统就天天把信息能够传递给你这个插件，系统，插件就能够把信息传递回这个核心系统，这个没有这个功能的，必须要定义一些通信机制，

怎么设计一个装载插件的这个过程大概呢？我们要注意哪些点？那首先来讲的话，肯定这个插件系统它也能够运作的话，前面也讲过了，不管是在核心系统里面它定义的规范，它有定义的一个注册规范，那我们也讲到在开发一个插件的时候，也必须遵循那个注册规范。你要提供一些注册信息，那我们就把这个注册信息其中一般来讲假定这个名字就叫注册表，所有的这个插件它可能有个注册表了，当然说 Windows 里面是叫注册表的，那别的地方是不是叫这个名字不一定，但是我们就用这个代称代这种作用就是提供一些信息，知道这个插件的一些注册信息的这么一个存储的这么一个空间里面，或者每个地方它的注册信息都在这个地方。


核心系统它启动的时候，或者是说在运行启动的时候，它可以通过这个注册表来查找到对应的插件，并且能够获取到它所需要的一些信息。不管是启还是停，或者还是装载这个插件，它可以通过注册表来获得。那这是第一点，就是说通过注册表能够知道插件的信息，这对于我们核心系统装载它这是第一个步骤，基本上那其次就是你装载的这些机制到底怎么去装载它？这个简单，从第一个角度来看，就是说在什么时机来装载这个我们的插件，这里大概一般就是启动期和运行期了。所谓的启动器就是说当这个系统一起来的时候，你的插件只要是注册过我的插件，我全都跟你一起起来，除非有特殊情况必须跟我一起启动。而运行期就是说即插即用的，我需要的时候我就装载你不需要的时候我就可能要 undo 的，你根本就不装载你了。
