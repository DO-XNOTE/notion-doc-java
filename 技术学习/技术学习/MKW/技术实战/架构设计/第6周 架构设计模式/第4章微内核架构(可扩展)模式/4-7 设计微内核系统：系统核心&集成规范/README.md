# 4-7 设计微内核系统：系统核心&集成规范

d 核心功能这点我觉得毋庸置疑，实现它的这个 MVP 这个话怎么理解？我们从两个层次来理解一点，就是说可能比如像一些偏业务的这样一些系统的这样一个开放平台的话，那你首先可能就是要把这业务点类似于像 workflow 一样的东西，你把这些整个业务流程全部梳理一遍，找到这些核心点在哪里，把它串起来定义在我们的核心系统里面。那这样子它的外延就是各种各样的插件，就具体实现某一个功能。比如简单例子，你先是下单，然后是审批，最后是出货，然后最后是送到买家手上，假定这样一个流程简单的实现了这样一个系统。


在这个系统里面，你这个流程的定义就是我们的核心，你要把它放到这核心功能里面去，但是具体怎么去下单，怎么去付款，这个就是一个实现它作为一个插件，这样子你就把核心功能和插件功能给分离开来了。那核心问题就是我们看前面这有一个工作流引擎一样的东西，那这样子就是这对于我们来讲的时候，如果你是开放的一个业务平台的这样子的，那你就首先要找到是不是中间有一些侧重点的这个工作流，你把这个工作流定义在这个里面，那这是一个方面。另外一个方面就是说如果你是一个纯技术的这样一个开放平台，那可能你就想想我到底要提供的是什么功能。那像 spring 的话，它很简单，我提供的就是这个类似于像IOP、 Bing factory、 IOC 类似于这种东西，这个就是它的核心。然后你核心功能出来以后，那插件怎么插？你就要遵循我的规范，怎么定义规范，我们后面会提到你遵循我的规范，然后你就可以和我集成在一起，形成一个新的功能。这一点就是说核心的就是相对偏业务的。


那再举个例子，比如说像是一个规则引擎，我们作为一个这个可插拔式的这样一个系统来讲的话，那对它的核心来讲是什么？它的核心就是这个推力集，根据你用户开发的插件，也就是我们所开发的一套路的规则，然后把这个规则插入到我们的规则引擎里面去，计算出一些新的结果，类似这样子就达到一个可插拔效果。


但它核心是个纯技术的东西，那业务逻辑其实反而存在。你的插件，它的核心一种就是那个规则引擎，类似于像 ESB 也是一样的 ESB 这种就是能够把消息进行路由或者 space 之间能够做这种 routine 聚合转化协议，这种东西，这是它的核心功能。那至于说你外面就到底这个系统或者这个消息到了之后，你系统做什么反应，那是反而是这种插件，所以这一点大家要清楚。另外就是说我们讲讲说你做出这个时候，你可以先把这个 MVP 给实现起来，让这个系统核心先跑起来，看一下有没有问题，设计上有没有咱们需要验证的地方。


第二就是说封装系统资源，这个前面我们明确的讲过，因为很多资源第一你直接让这个插件来访问的话，可能有一个破坏性的问题，因为你也不知道这个插线作者它是恶意还是善意，所以说你通过封装的它，至少说它能破坏的限度是相当有限的。OK，我可以这么简单地说一下，那比如说你硬规定说一个 Windows 程序它不能访问某某某一键支援，它肯定一直逾越不了，它逾越了那可能就是一个hiking，以这种做法，另外一层就是我们有个隔离的这样一个功能。前面就跟我们那个分层架构一样道理，你有了这个隔离，第一底层的变化和细节在更上一层次的他是不需要知道的。对他来讲他就是管一下一个初创的一个接口，这是最好的，这样子互相的变化没有一个非常强的依赖关系，这一点是非常好的一件事情。


通过插件的这个接口来访问这个核心功能提供的这个资源封装的话，是不是就是一种实现一点为抽象的这么一种感觉？实现有这个插件，我们的这个核心系统下面一点的，比如硬件系统里面的这种东西的话，它更多的都是一个被抽象过之后的接口了，这样子是不是就符合了我们的依赖导转原则？这一点也是非常重要的。那就像比如你开放了一个业务平台，然后你数据库它需要访问你的数据，等下我直接就把数据库暴露给它用。


当然不说不行，但是个设计是一个好的设计吗？你可以想象一下对不对？第一安全性没有隔离，第二这个来一个抽象层次，在中间的话，如果是任何底层有点变化，双方都会很不愉快，所以说真心没有必要这样子做，最好是通过一个接口隔离一下，这样是最好的。就对这个核心系统和底层还有包括插件层都是有很大的好处的，对于核心系统就是掌控就更好了。对于这个插件来讲，它实现依赖于抽象这样子的底层如果发生一个改变的时候，它是始终依赖于抽象的，可能它也是比较稳定的这么一个效果，所以说这点非常好。


另外就是你要有个开放的集成点，这一点就是怎么讲我觉得非常好理解。你如果说准备做一个可插拔系统或者一个微联合架构，那但是你说对不起，我不开放给你继承，那说难听点，你的这个逻辑不是前后矛盾嘛，你要做一个可插拔系统，人家不知道怎么给你插就没法集成，这个就是有点古怪了，对吧？没有这种做法，所以说你设计核心系统的时候，你肯定要预留一下这个集成点，这个接口，否则的话你就跟这个插座在墙头上，它功能什么都是好的，电也是通的诶，但是我觉得没有孔，

那我拿着一个插头我怎么能用你这个插桌上的功能呢？肯定是显得不行的，对不对？对于这一点的话，我相信很多人可能没有想到说还有这么一个说法，那其实你想想，你做这个可插拔系统也罢，微任何插入也罢，你就是为了让我们插件能够跟你一起协同工作嘛。那如果说你做了一个系统，然后人家根本没办法跟你集成，那不都白做了嘛。


所以说这点上可能大家误解有点多一些，但是我们再回顾一下核心系统这三大要素，我个人觉得第一就是从两个层次或者不同的角度，你定义核心系统的功能，这点要把它和插件要分离开的。前面也讲过，在开发难度当中这一点是很重要的，也就说一定要你能够识别出来说核心系统里面有哪些功能，对于我们来说是很重要的，否则你分解不出来这一层。第二我们这些系统里面的资源接口你不要直接暴露，这点非常重要。第三一点就是一定要有一个集成点，否则的话你要怎么可以进行插拔，这个就不存在这件事情了。第一步我们要设计的是一个集成式或者是一个开放式插件系统，有了这个系统核心以后，那核心要对外进行一个开放，允许插件技能，自然我们叫一个开放的规范。

因为没有规矩不成方圆，所以我们需要一个规范插件能够和我们进行一个集成。那所以说我们要定一些开放规范，那我们要注意些什么呢？我们主要看一下一般来讲的这种开放规范，它应该至少要有一下一些内容。


就是首先来讲了你这个核心系统要提供的一些接口，还有版本，这一点怎么讲？就是说很多人他不太理解，说这个版本有什么好讲的，那其实你是不是把这个问题稍微想简单一点，当然这个确实也不是什么特别大的事情，你稍微注意一下你就知道。但是我要提醒你的是什么？如果你是设计的一个开放平台或者可插拔式的那个系统的时候，特别要注意你对外提供的所有功能，特别是核心系统的功能，你一定要带版本号，否则人家跟你集成的时候，他不知道他到底要集成到哪一个版本上，因为这样子一旦错了，他提供的这个插件插进去的时候可能就不会work。


为什么这么讲？你跑一个 win 10 的程序，跑到这个 win 95 上能跑起来吗？这肯定跑不起来的，那你 spring 到一个低版本能跑到高版本，那可能general。怎样？那高版本能跑到低版本上面的一个 bin 吗？这个显然是不能的，所以这个版本其实你看上去是个小事件，但是很多人容易犯这个错误。


他对外提供的时候，其实你如果内核升级过一次，你不告诉人家你的版本深了，那人家怎么来配合你？他怎么知道这个兼容性测试要不要做？你看着其实我也不知道，那其实是挺重要的一件事情，所以这里我就提前先说一下，因为它是既简单又重要的一个事情，而且也比较好理解，就是只要你做任何一次内核升级的时候，你提供的接口的版本号都要发生变化。至于兼容性什么之类的事情，你要负一份责任，插件的人他也要负本责任，这一点其大家牢记，简单也好理解。


第二就是我觉得就是说你作为开放规范，你对外提供的一些东西里面可能要饱和那种类似于我们叫 context 或者环境变量这种东西，我这写的环境参数其实就是环境变量一样的东西，这个东西是什么意思？就当一个东西插进来的时候，它肯定得知道自己当前在什么环境下面，当前这个运行的这个 container 里面，或者一个什么东西里面，我得知道一些信息，这个信息谁来告诉我？肯定是开放规范里面要定义一些类似于 context 横向参数怎么获得的方式，你一定要定义出来，让人家这个开发者能够使用，那就是非常简单。


例子好了，你在 Windows 里面跑的时候，一个程序它肯定可以知道当前我是跑在什么系统里面，我的环境变量多了哪些，对不对？里面就是里面，你可设置一对待的环境变量，那同时来讲，你 spring 里面也是一样的， spring context 这种东西里面有各种各样的context，你全部都可以获得各种各样的变量和参数，你什么都可以获得到。因为这个好处就在于说你有一个类似于大一统的这么一个接口，你什么变量，什么样的环境相关的东西你都可以塞在里面，只要这个插件需要用它就可以去读得到这个东西。


不能说这是一个非常 perfect 的这样一个设计，因为少搞得好像太混乱了，而没有专用的这个对象来把它进行一次封装。但是毕竟咱们作为一个平台的开放者的话，你想到的这个情形和没想到的情形都会很多的，想到的情形教你简单一点，你可以用封装的方式把它实现出来，但如果没有想得到情形，那说白了这个环境参数或者 context 类似于这样一个 map 的这样一个结构，那其实对开发者是相当友好的。因为它第一可以自定义一些东西在里面，第二你作为一个开放平台的人，不便实现的功能的时候，你可以往里面塞一些参数，通过它来控制一些 be Harry 这个就是比较好用的一种手段。这个所以说你用得好的话其实有很大的帮助，但用得不好的话也可能造成一个泛滥，就是所有东西都往这个环境变量，或者说这个 context 里面塞，那人家去读的时候，这个可能就有点不受你的控制，甚至可能做一些不好的事情，这点反正就是你自己适度的控制一下，尽量不要滥用这个东西。但是我觉得作为开放规范，你一定还是要有这个东西。


第三就是这个类似于我们事件机制相关的一些东西，就是说当整个系统或者个插件，它都有一些事情发生的时候，你说这个时候你不发一个通知告诉外界，这个合理吗？这显然不合理。而且我们前面在开头就讲过，这种事件系统架构里面是有很多好处的， they do up，不管是要广播一个状态变化，还是只是通知人家说我发生了改变，那后续人家可以做一系列的事情。
这个例子也比较好理解，就是相当于说比较不管是 severate 还是我们这个 spring 的这个 container 里面都是这样的，你容器的这个启和停，你这个容器里面的对象的这个 initialize 成功，或者刚开始 initial 或者 destroyed 这生命周期，它所有的都有事件会 trigger 出来。然后如果说你监听的这种事件，接下来很多事情做起来就很方便。比如说你立刻成功一个病的时候，我要做某某事情，那如果说他没有这种通知事件，这个规范里面不含这种功能，你做起来你想想这个多痛苦，那你得跟这个 container 去做反复的交道，然后一个个地去试到底发生了什么，这显然是不太合理的，而且用起来不方便。


那像类似于 callback 钩子其实都是一样的道理，类似于说就是系统当中有什么事情的时候黎反正要告诉我，我现在对你这个事儿有兴趣，说白了就是这样子，就是说它都是一种类似于通知机制或者是一个回调函数一样的感觉。就是说当某事情我对它有兴趣的时候，你一定要告诉我，因为我要做一些事情，类似于这样子，从框架层面或者规范层面来讲，它只规范说这个时候会传一些东西给你，我会告诉你某某事情发生了。但是接下来这个逻辑处理是怎么样一个事情，那是你的事情，这就跟我们前面讲过的事件群众架构比较接近了，就是说有事情发生了，但是怎么来处理这个事情，那是你的事情，我只是通知你这个事情发生的。而且这个地方一般当你配合我们的 context 或者环境这种东西的时候，配合在一起你才有足够得的用途。
为什么这么讲？当一个事情发生的时候，你要知道这个事情里面那些context，你要是没有这个context，你要做很多事情都不方便。比如举个最简单例子好了，现在他告诉你 something happened，就是有个事儿发生了，他也已经通知你了，但这个事儿是个啥事儿里面？这个发生事儿的时候，当时的情形是怎么样？你不包装在这个 context 里面，或者这个环境变量里面，你说人家怎么去读到就是真正的listener，他做起来就很困难，这点我希望大家能够理解。一般你设计这种开放规范的时候，我都建议你设计一个类似于钩子、 callback 还是疑问这种东西最好都有，或者适当的实现一两种，这样子给开发者和你自己预留很多接口在这个地方，这样的灵活度诶高一些。


第四一点就是说你先集成规范，这个集成规范就要跟前面的好像，你说是不是有点重合？那不是那个意思，这个集成规范更多的就是偏向于说是一些业务。你简单的一点，比如说我们是一个 spring 的这样一个开发者套，那你肯定告诉你说你要注册这个bin，或者要启动这个bin，要启示这个bin，你要遵循什么规范？那我们的核心反应就是管理 bin 的，那现在你要跟我做集成的话，我要求你要做到 12333 点，你这个 Bing 才能跟我在一起，或者你要遵循什么语法，要遵循什么样的一个规则？这样病才能这样子，那样子，这个我相信大家可以理解。
