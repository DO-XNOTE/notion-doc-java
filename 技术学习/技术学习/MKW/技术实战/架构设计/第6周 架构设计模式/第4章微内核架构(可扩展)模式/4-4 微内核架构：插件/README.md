其实如果说咱们不采用这种威力和架构，那就是说所有人都打包在一起，那这些功能也罢，还是所谓的内核的功能也罢，还是由外延功能也罢，它就不存在这个区分了，对吧？所有东西都是由这个系统提供的，它都是一样的，但是我们因为某些原因，当然我们后面会讲到哪些原因来把这个内核和插件进行一个分离，所以说我的可以简单的把你解释核心系统的这一种能力的拓展，它方便做的，或者它做起来不是特别合适的就交给插件的事情。


那首先插件它本虽然说是一个能力的扩展，但是它必须得遵循我们前阵讲到的核心系统，它是定义规范的这样一些事情，对吧？那你作为一个插件的话，你必须要遵循核心一种所定义的规范，那否则到时候你是进入不了核心系统，也无法提供你的功能，这个是显而易见的。我认为第二点就是说你遵循了它的规范，但是你的职责并不是再去重新实现一次核心里面实现过的功能，你是要在这个资质扩展什么呢？就是实现你真正的业务逻辑，甚至有可能是对原有嗯，核心的一些功能的一些外延，就说有可能他原来做的不好的地方，你通过什么东西把它做得更好，比如原来它不方便呈现的东西也要呈现一下。


举个例子好了，在操作系统下面，如果说你使用命令行来呈现一个文件系统，那是没任何问题的，对吧？其实这样的功能，但是你作为一个插件，比如说我开发一个图形界面，能够把这些文件操作成更方便，这是不是一种插件与核心的这样一个区分？但是它只是一个逻辑的外延，对吧？那当然也有可能，就像比如说你在 Equips 里面哪实现一个插件是做这个版本管理的，对吧？ CVS 的这个大家就理解了，但是也就 Eclipse 本身只是编辑源代码，你控制不控制这个版本管理？它不控制的，那你通过插件来提供了一种这种 CS 的管理，那是不是就不一样了，对吧？这是它的专注的逻辑点，就在于说它是在做这个版本管理的，而你这个 x 本身是用来做这个源代码的编写，这是不太一样的，它是对它能力的一种拓展。你看原来也没有这个版本管理，你看通过这个插件我有了，对吧？但他们各自关注点其实是完全不一样，它的业务逻辑是这个版本管理 EX base 是提供框架和一些技术功能，对吧？这个还是不太一样的，它就能够区分一下这种二点细微的差别。


二就是说我们尽量要求这个插件模块是 single response Billet。首先第一，这当然可能符合设计原则，大家都知道，对吧？第二，可能说你更多的应该专注于你的这个独立的功能，你不要老是想要有的没的，就是你该做的不该做你都要做。这个在插件系统里面我觉得特别不合适。举个简单例子好了，比如说你说这个插件，我们那个插座在墙上又可以充电，还可以喷水，可以放火，这个好像是不是有点太怪了，对吧？这个对应在我们的软件上面，比如说你是一个少的游戏，你说我不行，我不做少的游戏，我做计算器，你说你硬要做，那当然不是不行，但这个是不是稍微有点怪怪的，对吧？我相信这个大家应该很好理解这个话的意思，就你以后自己一个系统的时候，你对插件的要求是尽可能的插件专注于他应该做的事情上面，不要去做一些不属于他的这样一些职责。


因为你这个核心系统只定义规范，那你里面具体实现什么东西他其实是不管的，但是从设计的原则来讲，你不能这么干，对吧？毕竟咱们作为架构是在这一点是很重要的，你要专注于这个每一个东西的规范。第二就是说你尽可能的，嗯，是通过核心提供的接口来访问这些像这种的资源一样的东西，这只是一个比方，并不是你就是操作系统就说可能有些资源性的东西是不方便直接暴露给你这个插件系统的。为什么这么讲？因为插件相对来说是比较浓隔离过的这样一个组件，因为对于他来讲，如果是一些不法的这样一些 hiking 行为发生的时候，如果你直接操作插件，因为插件是可以随便都可以开发的，在内核里面东西这个通过各种大厂的验证，或者说这个产品本身这个开发者的这种鉴定它是不会乱来的，对吧？那如果说你一定要这样子破坏这个规则，还真是不行，但是可能带来的后果就比较严重。


所以你作为一个架构师，你在设计这种开放式系统的时候，一定要注意不要让黑客能够很方便地钻到控制，反正这是一个双刃剑，你怎么用可能都会有问题，就你不开放可能有不开放的好处，但是可能也就限制这个开发人员的这个能力和手段，那你放开了感觉好像是不是有可能会被攻击，因为被病毒什么之类的侵入，是吧？这个都是一个双减，你要做一个很好的考量。

那还有就是说插件之间最好是不要有依赖，但是这个怎么说？就是说可能有点见仁见智，因为其实这比较难说的一点就是在于如果是用 Eclipse 来做比喻的话，这个就不太贴切了。就是你用 Eclipse 来做例子，一个 Bison 里面它是允许你插件它有这个所谓的机器点的，或者这个扩展点 extension point，它也允许有dependency，就说你可以一个插件依赖于另外一个插件。


我的个人理解可能在 Eclipse 里面，比如说举个例子到基础功能它也是插件来的比较活跃，图一个什么动画效果什么之类的，如果你不继承于它的这套 UI 体系的话，你做起来就很困难，对吧？这是一个例子，但是对于说那是不是所有的系统都这么做，可能就是比较麻烦一点，那比如就像是操作系统来讲的话，那如果说你插件之间有依赖，这个就很怪。


在操作系统上这个就不叫插件了，就叫应用程序、第三方应用，或者说，对吧？比如说我们有两个程序跑在 Windows 上面，但是两个互相干涉，互相影响，这个是不是有点怪怪的理论？他们应该都只依赖于核心系统提供的功能，互相都甚至不知道对方的存在，理论上当然还是句话，那你想想当年著名的 3Q 事件，就 300 大战腾讯的那个事，喝酒很喜剧了，本来理论上来讲的一个杀毒软件，一个通讯软件之间它有啥关系？但是它偏偏就是我要卸载，你要卸载我们两个只留一个，否则我就把它杀掉这个进程。


另外我把它的进程杀掉就杀来杀去的，这个好奇怪的，对吧？我个人觉得就是说如果我设计一个这种为何架构或者可查法架构的话，我可能是不倾向于插件有特别强的这个互相通信或者这个依赖的关系。这个话怎么理解？你还是稍微自己有一点自己的主观能动性，好好思考一下我讲的这个话对不对？我相信穆克王的同学们都是有独立思考的能力的，这句话反正我们可以放在后面，课后进行讨论，就说插件之间是否是有依赖，还是它只能依赖于核心？这件事情很难下定论的，各有优劣，只能说对不对。那还一个还比较重要的就是隔离性。你这么讲这个就有点像之前我们在那个番茄价格里面讲的这种，对不对？它是通过不同的分层的方式，它进行了一个隔离。那在这里就是说你其实有点类似的一个思想，就是你通过把它分成这个插件和核心，这两者其实就是一个简单的分层一样的思想，二者就是说它不在同一个层次上面，在这个地方稍微有一点不平等。


主要是可能有点像是插件它是依赖于核心的，就是核心理论上它是不依赖于任何插件的功能的，所以说核心的这个变化可能会影响插件，就是你很难把核心里面的一些问题隔离开来。但是对于插件而言，它其实基本上尽量要，就要让它在设计层面尽量它不要影响到核心。这个就跟我们之前讲那个防水仓设计可能就不太一样了，那防水仓这种设计里面，它其实每一个仓都是平等，它对不对？针对哪一个仓出了问题，我都把它关了就好了。


在这个地方就是说如果说从隔离性来讲的话，这个插件与插件这个可能确实是隔离的，平等的，但是如果你要从插件跟核心经验来讲，这个就不太抗得了，对不对？因为你的插件出问题，他可能自己就挂了，或者自己crash，他自己培训起来，但是他尽量你不应该去影响核心，这个就像操作系统的例子，比较推荐你可以应用出问题了。


比方我举个例子，我们这个 PPT 现在突然出问题了，那你说我要再把这个 Windows 都给搞挂了，或者把 macos 都搞挂了，这些人就是一个很差的设计的，那肯定尽量是不要出现这种情况，对不对？所以我们来讲，它就是说在插件系统大家可以简单地了解到，最重要的就是说白了它的作用就是什么丰富和扩展核心性的能力。简单来说它就是这一件事。
