首先来讲 u 层与 model 层，它是不直接通信的，都通过 presenter 来传递， presenter 完全把 model 和 view 进行了分离，主要的逻辑和程序都是在 presenter 里面实现的。 view 本身它是非常薄的，不部署任何业务逻辑，被称为被动视图，即没有任何主动性，但是 center 却非常厚，也就是它里面的逻辑非常非常多。


percenter 与具体的 view 是没有直接关联的，都是通过定义好的接口进行会员交互，从而使得在变更 view 的时候会导致 precenter 它不变，这样可以复用。不仅如此，还可以编写各种测试的view，用来测试这个presenter，这都是非常好的。
除此之外，相对于MVC， MVP 还有哪些优点？第一就是 model view 它是完全分离的，我们可以修改视图，但是不影响模型，同时可以更加高效地使用模型，因为所有的交互它都是发 APP center 层， model 层它是完全那无关的。


我们还可以将一个 presenter 用于多个视图，而不需要改变 percenter 的逻辑。这个特点非常重要，因为相对来说，业务变更的时候，视图层的变更，它的频繁程度是远远高于 presenter 层的。如果我们把逻辑都放在 presenter 群当中，那我们就可以通过脱离用户接口的方式来测试这些逻辑。比如你可以使用单元测试的方式 a test percenter，但是它当然也有自己的缺点，最大的一个缺点就是 view 层跟 presenter 层的耦合度相对来说是非常高的，这一点我们在使用当中必须要非常注意。

m m 的 view model 是一个值的转化器，你可以这么理解，因为这个 view model 它负责的是从模型中或者暴露，你也可以说是转化数据对象，这样子可以很轻松地管理，就是说你在界面上要呈现的东西和内容。在这方面， view model 它其实是做的，相对来说比原先我们在 MVC 中定义的这个 view 层上的东西或者逻辑是更厚重的，而且它负责的东西也相对较多一些。同时在这个 MVAN 架构下面，没有和 model 之间它没有直接的关系，都是通过 view model 来进行交互的。因为我们讲过了 view model 主要负责的所有的这个 view 上显示的逻辑，包括这些数据结构也是由 view model 进行转化的。
