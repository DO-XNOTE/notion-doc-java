# 2-6 分层架构的缺点：性能下降&成本上升

那前面我们都讲了很多分层架构的这个优点，其实大家对这个优点和好处能够中获得的这个收益让大家有一些人了解。但是其实缺点可能有些人反而是更重要的，因为你不知道缺点，那这个问题出现的时候，你不知道解决方案，这个可能会给你整个系统带来大的问题。那所以说我们先来看看第一个问题，就是说这个分成价格的一个区，就是比较业态功能的，可能是会导致一些性能上的下降。

大家是否还记得在前面的章节里面我有大概提到过，因为分析价格理论上你应该是只依赖于直接的sight，那如果说你要做一个跨层的访问，那其实这是不可以的，那自然就会引起额外的开销，因为层与层之间你本来原来糅合在一起，现在你要进行一个分层，那直接的引入的这个通信开销是毫不疑问的，自然对整体的性能可能是一个损害。那其次由于这个行程的信息它是包含在自己城内部的，不能够对外直接进行一个访问，所以说每一层当你要向下层传递信息或者向上层输出的时候，你都要发生一个数据格式的转化。


这个比较典型的就是比如从我们 MVC 来讲的，你从页面上输入的一些信息可能就是 user 原始的request，那你到了我们 data model 层要读这个 request 吗？这个可能性不太高，肯定都在 control 层对它进行一个转化，把它变成一个 data model 可能会接收到这样一个性质，然后之后当你再把这些 data model 存到数据库的时候，可能又是个 entity 的格式，在每一层到每一层间都是一个数据格式的转化的，所以这样子性能肯定也是会受到一些影响的。那同时还就是前面讲到就不能跨乘法分，因为跨程访问的话不太符合我们这个设计原则。


但其实是有些时候可能页面到数据库之间的信息全都长得一模一样的，比如说页面上有 10 个字段，而对应的这个数据库也就是完完整整 10 个字段，而且业务逻辑什么完全都是一样的，但是你就是不方便直接从 view 层跨到 data 层，因为这样是不允许的，那同时这样的电路的调用没有减少，自然而然它的开销是不能得到一个降低的。


因此我们有些时候就是前面谈到的，可能会因为一些性能的问题，我们会减少层与层之间的这种调用链路，可能会违背一些跨层调用或者数据转化或者信息泄露这样一些原则，这个大家就在实际开发或者架构设计当中灵活掌握了。这里我觉得很难给出一个统一的标准答案，自己觉得说是通过这种减小层或者减小数据转化带来的收益是大于我们付出的代价就好，自己可以在实际项目中进行一个一一评估，不要做出一个统一的标准答案。我们现在在讲另外一个缺点，就是这个老板们可能必须要需要考虑的，作为架构实战的人，你也需要去衡量一下的就是这个开发成本的上升。

那首先来讲大家都知道这个分层的好处是不言而喻的，但是分成了之后，特别是随着咱们这个系统的复杂度越来越高，那所以说每一个点都要求越来越高深的这些复杂度的处理技巧，所以说在每一个层可能都会有一些专业人士来提供支撑，所以现在提倡的一种 full STAG，虽然说是有，但是我感觉不是很多公司都会存在 Foo stake 这种开发的一个结构和架构组织，所以说复制这个少而且很难培养，导致很多岗位可能会出现一些重复建设。所以说这个人员成本可能是会有一定的影响的。


第二个就是说因为你分成层以后，不同层那自然就有不同的组织。前面我们已经讲过康危定律了，那不同的组织你只要有了人的参与，它就需要一个沟通，只要有沟通就有沟通的成本，那个沟通成本随着你系统的划分的粒度越细，你的组织越多，那你这个成本就越来越高，这一点也是大家需要注意的。还有就是当外界提起一个变更的时候，这个变更虽然我们通过分层来隔绝了一些层与层之间的影响，但是有可能这一个变更它其实是要一杆子捅到底的，从 UI 到 database 全部都要修改，但是你是无法直接一个人搞定这一切的，所以说你很多层都要参与，很多人也要参与，因此你一个小小的变更可能就要比原来的这种意难词解决方案要麻烦很多，因为它需要很多层的参与。所以说这种分层架构带来的好处是灵活性。

now 相应的一些付出可能就是我们开发成人的成本上升，还有这个不同人、不同专业组织之间的这样一个交流成本、沟通成本也是在不同的上升。所以说这一点我们作为架构师一定要取得一个平衡。是不是在有些情况下，我们可以尽量的消除某些层，在这些层消除之后带来的便利性是可以高过我们原来对这个组织结构也罢，还是架构的隔绝性也罢的一个要求。就说我们的付出是大于了我们的回报，还是小于了我们的回报，这一点是作为加分必须要去衡量和平衡的一个点。
