仔细看你会发现一个很重要的规律，就是圈的最外层是偏向于实现，越往里面就越偏向于抽象的规则。我们来看一下这个同心圆的构成。在最外层，也就是蓝色的层，都是属于一些框架、工具等等，我们都会把很多细节放在这一层，但是不会有很多代码逻辑在此，更多的代码都是关注于如何与内存通信，这种代码名称之为胶水代码，因为它关注于两层之间的融合。也通这种细节分布于外部的做法，可以让整体系统不受细节的干扰，保持自己的独立性和抽象性。


再看第二层绿色层，这一层属于主adapter，也就是将不同层之间的数据进行转化，以方便不同层之间能够互相交流，简单理解，这就是这一层里面将比较抽象的应用程序逻辑和使用的数据。通过这种转化，让外层实现层可以理解，同时也可以将实现层得到的信息转化为业务逻辑所使用的数据结构。


再实际一点说法，比如说我们数据的各种映射，RIM，包括呈现给用户看的数据，它的数据结构可能是不一致的，在这里完成了一种数据之间的转化。再往里，嗯，你看是我们最核心的 use case，所谓的 use case 也就是各种各样的业务场景，所以在这一层里，主要就是封装和实现了大部分的业务逻辑规则对内还支撑了企业的业务规则和法律法规，对外沉浸了外部用户输入并解读其含义，可以说是很多系统的核心。作为架构师，我们希望这一层能够在外部 UI 不停变化时对内的影响降到最小，又希望业务规则改变时不用改变太多的用户行为，所以它非常关键。


最核心的一层是entities，这一层的体现企业里的各种业务规则。这里的 entity 具备一组数据结构和方法，但是却可以通过这种方式封装了各种各样 high level 的抽象规则。它在整个应用中具备最高权限，其他的层都要为这一层服务，在这一层的东西尽量不要被外层的改变而影响。

具体依赖于抽象这种模式做相对比较 high level 的层。定义之后，再将不同的组件按照功能分配在不同的层，比如有的层主要负责具体的业务实现，有的层负责展示信息给用户或者接受终端用户的输入，或者是提供接口给第三方服务。这样就可以开始把系统内的组件按照各自不同的功能和属性分配到不同的层里面，而且每一层都是高度内聚的，因为它们具备相似的趋同性。


那前面的思路主要是按照功能来划分层次，但是有的时候如果觉得按照功能划分比较复杂，比如都是一个系统内提供相似的功能，那按照功能划分的时候，界限就比较模糊，这个时候我们不妨换个思路来看问题。前面我们有讲过康威定律此时可以发挥它的作用，因为功能模糊，但是组织架构却是很清晰的，或者说用户的画像群组是很清晰，这个时候按照不同的组织架构或者用户画像，还将功能归类到不同层，这就是相对较好的一种做法。但是需要强调的是，这种做法并不一定适合所有的场景，而且应该是尽量少采用的方式。

再想想我们的研发体系里面是不是有各种不同的组织，按照这种组织把它映射到软件架构上，每一层具体应该负责什么？如何实施？是不是又相对有一个不同的概念的定义？按照这种方式，我们一样可以定义出一个分词架构。

那首先我们可以根据每层的需要各自选定，这是因为我们前面有讲过，每一层都是各自独立，并且是自制的类聚的，所以说理论上你每一层选择了什么样的技术栈，是完全不用考虑其他层对你的限制和约束的，所以这点非常重要，大家其实完全是可以按照自己的理解来选定自己的技术栈。


第二就是说因为这么多分层架构出现以后，有很多实现得非常好的功能和模式都已经被很多开源软件以及各种产品所实现出来，那你可以参考和借鉴它们的实现方式，找到每一个的优点和缺点，与自己当前的实际场景互相弥补和契合，选定一个自己的技术方案。那最后也有可能说我们受限于自己公司或者这个组织所在的这个数局限，也就是说比如说你有些不同的部署方式要求你选用某种技术栈才能够起来。比如说我们公司只能用Java，那你肯定只能选用 Java 这个或者跟 JVM 兼容的这样一些语言，它的层次架构都会受此的影响，那比如说我们全体都是用 NODE js 或者Python，那可能又是另外一幅场景，其实就是说相同的架构，相同的职责，那其实实现层面的技术选型是完全不一样的。
