那首先我们再来回到目的本身就 spring called hatract 它的工作原理是怎么样的？在讲这个原理之前，我觉得先大家应该了解一下这个 hatch 它为什么会出现。在说这个问题之前，我们先了解一下一个典型的这个，特别是麦克斯瑞斯这种价格模式下面的一个问题，那假定说我们在整个系统里面有 n 多个服务，从 1 到n，那这个其实在你的真正的这个业务场景当中，你的服务肯定是由 n 多个服务串起来的，形成的一个功能这个应该没什么疑问。但是对于在实际的运营过程当中就不像画图那么简单了。那任何一个服务因为成了微服务以后都不受全局掌控，它都是由自己这个豆瓣的人来支配的，所以说他随时有可能会崩掉。


唉，出于什么原因不知道？因为你作为一个调用方，比如说是你去调别人的一个服务来形成一个新的功能，那别人崩掉了，那你在不知道的情况下会怎么样？大家可以想象一下，那肯定你的服务也会不行，那再来看一下这个问题，就会形成一个什么，是不是一个连锁反应，因为你钓别人，那别人有可能钓你啊？这样子，就这个系统就是怎么样就崩掉了，就是所谓的雪崩效应，也就是说当雪崩掉的时候，没有一片雪花是无辜的，就这个意思，因为有一个坏了，那可能你写 n 多个都坏掉了，那只要坏掉了，那任何一个钓你的又坏掉了，那再依赖于他的又坏掉。这是个元素反应，也就是个血崩效应。那所以说 hashtrigs 它的出现，那就是为了防止这种雪崩效应。它的主要作用体现在什么地方？就是说第一它是帮助我们形成一个断路器，这个断路器就是我们整个设计思想当中的一种 feel fast，什么叫 feel fast？就是说让它尽快的断掉，尽快出售的时候整个就直接熔断掉，就跟你们家的那个保险丝一样，你可以这么简单理解，就是如果你们家里面有这种跳闸，那是不是保护了屋子里面的所有的其他电器？如果说你不跳闸，这个电流越来越高，对引起的短路越容积越来越重，就所有的电器都会烧掉。


那在系统当中就相当于说有一个服务崩坏掉，没有关系，那它只要断开了这个段落开关，闭合了这个段落开关就会打开或者关闭这个功能相当于把有一个服务器从系统中隔越掉了，当它出现的时候，它自身是保护起来了，那不至于让所有的系统全部都死掉在这个地方，唉，所以这样子就防着雪崩效应。


那这个东西它怎么做到的？第一就是我们讲到的所谓的资源隔离，就是我们跟分层价格里面有讲到的，你要实现这样一个东西，可能有一个层的问题，那就是在服务当中每一个都是不同的层里面隔绝开来。哎，还有就是我们说的防水仓、涉企，还有前面讲到这个熔断企，这几者综合在一起形成了这样一个功能。那这里我们可能跟我们这个分层架构相关，比较紧密的是这种熔断或者是防止仓，但其实整个的这个功能是也是挺复杂的，我们可以简单的分析一下。


那前面也讲到说，嗯，你要保证一个防水箱工作，那首先来讲它要对它分配不同的线程池，应该说是不同线程池里面就是说我跟外界服务在进行交互的时候，因为你通过这个 hetrics 的 commander 来配置了一个交互，对吧？那这样子它可以帮助你观察各种这个人。


有术语叫 metrics metrica，它来看你这个服务的这个服务状况，那通过这个关注给metric，那度量你这个里面的东西有没有问题，那同时在你对外进行调用的时候，它是需要一个线程来的。这个时候其实如果大家有去仔细了解过 ATRIX 的这个线程模型，也是非常有趣的，这地方我就不深入展开，但是我希望大家如果有兴趣或者有空闲的话，去了解一下 hashtricks 的这个线程模型，它其实设计的非常好，这里从重点我们先讲什么呢？就是当它服务对外进行调用的时候，它形成了一个不同的性能池，这样子保证了每一个对外服务，如果一个不可用，它不会把整个都在资源池耗尽，导致所有的服务都不可用。


ahead 就是说是不是都要的所有的交流节点，这种 potential 的 field 的一个点，哎，把它隔绝开来，在当有一个点 fail 的时候，是不会影响别的点，这第一是错误隔离，第二就是进入防水上设计把各个这个线程调度的分开。OK，那再讲了前面我们的这个断路器这块的角度，从断路器的角度来讲，其实它也相对来说不是那么复杂，它的整个原理其实是相当于我们前面讲的它有个metrics，那 metrics 就是一个度量，就是衡量你现在这个对外服务的调用的某一个点它是不是有问题？它如果有问题的话它会怎么样？它没有问题而怎么样？此外还有一个功能就是相对于我们叫 fall back，所谓的 fallback 就是什么意思？举个简单例子，那比如说你现在在调一个第三方服务，这个服务成功了，那自然一切都好处。但是平时可能我们也会有些 default 的实现。


我觉得如果说调第三方这个服务有问题的时候，我们是不是从 cache 里面去拿，或从什么数据库里面去查一下？从一个 fallback 就是什么样？相当于是一个 lead alternative，就相当于你有一个其他的选择，我不掉远程，我掉本地行不行？我拿缓存行不行？我从准备别的文本里面去读行不行？唉，就是这样的一个，这次尽量让第三方出现问题之外，我还是能够提供一些意料之中的这样一个备份，那就是一个备胎了，就是你还有一个备胎可选，当一个女神走掉的时候，你还有一个备胎可用，就这样子一个意思。


那它这个做法是怎样的？首先来讲到它这一个Magic，就是相当于说它会通过一些窗口事件，这个窗口事件有很多种，这个人也不叫他们 window 的意思，这个我就不详细的讲了，大家回去研究一下。所谓什么滑动窗口，各种窗口在实现，就是在判断这个窗口期内这个服务的错误的次数有多少次，超过一定限度的次数就是这个是可配置的，超过你这个次数或者这就判定为错误会是怎么样，它也通过可以判断说你超时多少，就为一个fail，然后超过一定的次数，那整个服务他就认为你是不可用的。


简单的说就这么简单，其实整个过程就是这样的，就是说其实你在使用Atrix，在你的方法上了你都要调用的地方，你会加一个 a Hydro 的commander，然后这个 commander 它其实相当于是一个切面一样的东西，然后它就可以先检查你这个，那当你这个运气的先检查你的这个熔断器的开闭状态，如果是开启的话，那它就是执行一个降级服务，那直接就调你的 fallback 方法。
那如果说它熔断器是关闭掉的，那就说是正常情况，它会检查当前线程池是否能够接受新的请求？如果线程池已经满掉了，那它一样也会执行降级服务，那就是走到你的 fall back 路径里面去，如果线程池能够接受请求，那 hatchix 它就可以执行调用具体的业务逻辑，那如果执行失败，那它一样会再看一眼你的callback。要就是 fall back 这个方法，看有没有办法同时把你这一次错误的情况上报到那个 metrics 里面。就是我们前面讲过就是这个通过滑动窗口，各种窗口服务的 timing window 来衡量你这个服务的健康状况的。


这个 metrics 如果超时的话也是跟前面一样，就不管你是得到一个 error 还是得到一个超时，它其实都会把这个情况返送给那个 metrics 里面去判断是不是要打开或者关闭这个熔断器，然后如果说一切正常，那当然那直接返回正常结果，如果不行的话就是降级服务，这个也很简单。


总之这里有两个关键点，一就是说你要知道在这个 ATRIX 里面它有一个 metrics 来度量这个服务的健康状况，这个近况状况它是一个 time window，就它是在一段时间内你错了多少次或者超时了多少次，它都认为你是个错误，它会把它进行降级，然后会怎么样？如果说这个错误的次数到了一定限度，超过一个阈值，它整个就判定你服务不可用，直接下次进来的话，你这个因为段子形打开了，这个时候他就直接向你的那个保险丝烧掉了一下，整个就垄断了，他一下子就不会让整个正常调用服务，直接给你 fall back 掉。


这样就是一个非 fast 的策略实现成功了，那在这个情况下，那我们熔断器的目的也就达到了，它是保护了整个服务的可用性，牺牲的单点就是这样子一个思想。前面我零零碎碎的讲了很多点，但是总体归纳起来就是最重要的两个点，一、这个断路器它本身是怎么工作？2、防水舱是怎么设计的？所以说我个人觉得还包括就前面讲的线程池，还有包括它的滑动窗口 Matrix 的实现，都是非常有趣的结果。
