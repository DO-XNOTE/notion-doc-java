# 2-2 层的定义&何谓分层

就像我们开篇时讲的一样，分层这个技能是一种很古老的智慧，并不是什么新兴的思想。在古代思想家的认知里面，他就已经知道把具有相同特征的东西归结到同一层次，再将不同层聚合在一起，形成一个整体的系统。那在我们的信息世界里面又是怎样的定义层的？

基于理论和实践层，就是软件中的一个逻辑聚合单元。


我们这么想一想，假定现在我们的一个系统不管多小，它里面至少也散乱着几个到几十个组件，而庞大的系统可能会成千上万个模块和组件，如果我们没有办法对它们进行分组管理，你可以想象一下那是一个什么样的设计，都是一群木头苍蝇一样，会是一个多么混乱的系统，你也可以想象一下你所在的公司或者学校是不是大家都挂在同一个老板名下？那显然并非如此。


所以我们都需要对一个庞大的系统按照一定的逻辑来进行组织，比如说按照功能来把组件分层，每一层都由不同的组件组成，每一层都有自己特定的功能，然后再加上一层组合在一起，形成一个完备的系统。我们说对层系统来分层，其实就是把众多的组件按照其特征分配到不同的层里面，这种特征可能是功能，也可能是组件本身的属性，这一点并没有运行规定一定要按照什么逻辑来进行分层，但是只要你能找到这些组件的共通性，就可以认为是他们在同一层里面，而我们作为架构师的职责是什么呢？就是从杂乱无章的系统中找到不同组件的共性，然后通过共性定义层的含义，最后把组件分配到不同层，最后它们在一起形成一个完备的系统。


那么我们现在知道了层是软件的逻辑组织单元，那么分层架构模式又是什么呢？

简单来说就是将系统按照职责和功能来进行划分，从逻辑上来讲，把整个系统划分成为多个不同层次，整个系统可以认为是由多层次组成的一个整体。大家请这里注意一下。我们的定义是将系统分层架构却并非被局限于软件。虽然我们大多数时候谈论分层时都是在说软件的架构分层，但其实分层架构也是可以针对硬件来同时进行的。其次，所以说软件分层是没有定数的，或者说很难来价定如何划分。可是有些基础规则是必须要遵循的，比如最重要的一条分层架构规则，就是层与层之间只能上层依赖于直接的下层。这句话有几个含义，就是只能是上面的层依赖于下面的层，不能反过来让下面的层依赖于上面。或者我们可以这样理解，在分层架构里，架构是在定义层的时候，要让上层尽可能的变成抽象，越往下就越要偏向于具体。或者也可以这么说，越往上的层越不被任何技术约束，越往下的层却需要绑定具体的实现细节的技术。


这也是 solid 设计原则中的依赖导转，即实现依赖于抽象，而非抽象依赖于实现，这点很重要，很多的新手架构师很容易在这一点上犯错，拿着锤子看啥都像钉子。一谈到架构设计，就喜欢说某某某框架或者某某某公司的某产品适合我们这个场景，这是典型的抽象依赖于实现，我们要尽可能避免这种做法，搞清楚架构中的抽象和实现，不要混淆，这一点非常重要。现在再说第二点，就是不可以跨城访问，但是这一点其实在实践中是挺有争议的，因为这种不可跨城访问这实践中带来很多其他的开销，这一点我们会在后面的讲解中做详细的分享。现在我们就先偏理论一点，将这个定义严格的认为是不可以化成访问，这是为什么呢？因为我们分层的目的就是要将不同池子的东西组合在一个层里面，每一层对上层提供访问接口，同时也屏蔽了下一层的相关信息给上一层，以此达到一个清晰的信息交换过程。如果层与层之间可以随意访问，就不存在信息屏蔽了，那么分层的意义何在？同时跨层访问的做法也违反了接口聚合原则，因为一旦跨层访问，那么你的依赖就会发生改变，可能会依赖一些你不需要依赖的东西，这是不符合我们的设计原则的。但是在这里我们也先把一个伏笔，大家想一想，我为什么会说这是理想状况，实际情况又会什么样的挑战，让我们愿意违背这种原则，获得一些什么样的东西？

不知道大家有没有救人过？装修的时候你会知道，如果你们家要刷墙的话，它并不是一次性搞定的，它是先要对墙做各种各样的修整，然后再涂上不同层次的颜料，最后形成一个整体效果，这其实也是一种典型分层架构。
