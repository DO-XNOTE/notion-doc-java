# 2-8 如何设计分层架构：代码抽象&集成

不管我们架构理论上有多复杂，但是最终都要落实到代码上，那么分层架构对代码的指导意义和在首先是层与层之间必须通过接口进行交流，每一层的内部实现无论多复杂或者多简单，对外暴露的永远是接口，而实现细节必须要隐藏在接口之后。

如果这样的层直接拥有了实现的知识，那么后续如果实现细节有所改变，势必影响其依赖者增加了整个系统的额度，而是我们要竭力避免的事情。


在划分层的时候我们也提到过，在设计的系统时，每一层都要尽可能的只依赖于其直接的下一层，而如果将这种架构设计映射到代码上，那就意味着代码不能进行跨层调用，而如果使用的语言自身就是支持可见性控制，比如 Java 的类型日词可以控制 class 的访问权限， class NODE 的向上委派机制可以保证正确的类会被加载到 GOM 里面。


再比如说，在常见的 MVC 架构当中， u 层的代码是不应该直接访问 model 层的功能的，必须通过Controller。如果 view 直接访问了我们的 model 层，形成了二者的强耦合。

如果 model 层发生了任何的改变，势必影响终端用户的用户体验。这显然不是我们设计的初衷，而是应当遵循架构指导原则，每一层的代码只能通过接口访问自己的直接下一层。


最后，不管是从架构设计的角度还是实现的角度，我们都需要为层与层之间的集成找到一种方法，但是在开始集成之前，我们需要先完成测试。

从代码的角度来说，就是完成单元测试，从系统角度来讲，就是完成模块的功能测试之后才能谈到集成，因为从实际工程角度来看，如果单体工程都不能保证其正确性，那整体就是无本之木。所以请牢记在集成之前先完成单体的功能测试。说回集成本身，所谓集成，即两个层之间的交流就是前面所涉及的接口被调用。如果层与层之间是分开部署的两套系统，那么集成方式就要考虑是同步还是异步。如果是同步，我们可以考虑HCDPR、MI、 RPC 等等。如果是异步，那我们可以考虑MQ、Calfcar、unit、 bus 等等。


但如果同一套系统的集成，多半是以内存的形式，不管是方法还是函数的同步或者异步的调用，在确定了接口信息和集成方式之后，自然就要开始做系统的集成联调，这种联调一般都是以集成测试的方式来完成，我在这里简单的介绍一下集成测试。

首先根据系统的 use case，我们设计对应的 test case，通过这种方式完成系统整体性的连通性、功能性的评估。之后我们要关注非功能型需求比整体的性能和吞吐量等等。因为一旦有信息交换，整个系统就多了一个可能失败的点，这个点它可能会引起系统的整体的失败，其功能可能不满足定义的层接口，也有可能是因为接口的请求次数过多或流量过大导致的通信失败。关于集成测试的相关知识，我这里只是点到为止，如果各位同学有兴趣，请自行查找资料，更进一步的深入了解。
