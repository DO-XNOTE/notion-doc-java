当然咱们这里不是真正的网络协议解读，我也不是专业来解读这个东西的，我们主要是讲的架构，大家更多的希望是通过这种经典应用来了解这种架构思想，被运用到各个不同层次，不同的系统上面的一种体现，来从中获取这种共同的抽象知识的掌握。


那大家可以看到我们在这个网络协议层里面把它分成了七层，从最底层的这个物理层到最上层的应用层与层之间实现了一个格局，通过一些接口来进行交流，也就是说你在 HECP 层做各种什么 Web service，各种服务的时候，你是完全不需要考虑底层的硬件是什么的，这样子从而实现了一种层与层之间的屏蔽。这也是我们分层架构思想之中的一种体现，就是层与层之间是可以通过接口来进行信息交换，而互相不需要知道实现细节。下面我们再来看看第二个经典应用。

同样也是一个两层架构，这个地方唯一的变化就是由 c 变成了b， c 端开始之前已经知道了是跟操作系统绑定的一些副客户端，而最大的一个改变就是把 c 变成了Browser， Browser 就是运行于各种浏览器的 show 客户端。


这个怎么理解呢？就是说因为在 CS 模式出现不久以后，大家发现一个问题，就是这个应用的更新很麻烦，每次那个时候也没有现在 APP store 一样的东西，互联网也不够发达，所以说你要去拿到这种软件的更新升级，都是通过物理方式来进行的，或者邮寄一个硬盘给你，邮寄个软盘给你或者到分销商那里去拿到这种最新的软件。


所以说这个 system 架构不足，就被 BS 给替代了，因为有很多程序它可以 user 气场直接运行起来，那浏览器它一般来讲实现了统一的规范，所以屏蔽了操作系统，而且你任何在 Server 端发生的一些改变，它都是可以直接马上反馈出来的，就是你的应用做了一些业务逻辑的变更什么之类的，那没有问题的，它直接可以在 brower 上呈现出来。


所以说 s 端还是 Server 没有什么变化，但是可能跟它的协议有一些变化，但是就其实逻辑是没有大的变化，最大的变化是由副客户端变成售客户端，也就是我们的 brower 这样主导，当然同时还是一模一样的，他们还是和之前的网络架构没有什么太大的区别。

所以这个时候大家就发现一个问题，就是那随着这个使用的人数越来越多，这个计算机不再是神秘的东西了，那所以说企业里面的应用就越来越多，而且也越来越复杂。


然后这种 it 的这个开发者在当时就发现这个复杂度越来越高，要用传统的这个简单的这个不管是 CS 还是BS，好像做起来都有点复杂了，那这个时候结印就逐渐出现一种风潮，就是变成了典型的三层架构。所谓什么三层架构，第一层就是


presentation，就是展示给用户要看的东西，也就是通常我们说的UI，那这一层就专门用来和用户交互和呈现信息，然后有一个叫 domain model 的东西。 domain model 就是什么意思？就是把企业里面的这种各种业务规则想要的这种功能，就是通过 DOM model 来体现的。那同时还有一层就是存储层，也是我们所有的data。就前面你讲的，不管是这个交互还是说这个逻辑，最终数据必须得持久化，把它存起来，这才是很多应用的关键和核心点，甚至在很多早期的系统里面，这个 domain model 就相对是比较弱、比较薄的，它更多的都是通过 SQL 把这个数据固化到我们的数据库里面去。所以就是一个典型的三层架构，就是通过这种方式给设计出来了。那同时就是说在于这种架构模式以后，发现你带来了很大的好处，那有意思，就这个东西分的就越来越精细了。
