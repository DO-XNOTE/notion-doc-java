# 2-6 持续交付体系-开发模式（1104）

hello，慕课网的 QQ 未来加工师们，欢迎回来，在上一小节里，我们为大家介绍了关于交付质量相关的内容。交付质量是我们需要考虑在整个持续交付过程中，我们如何保障从需求到最终的产品上线验收，这样的过程里，通过我们的努力能够让我们的产品的质量得到相关的保证，从而让我们的用户可以更好的使用我们的产品。那么在这一小节里，我们将来聊一聊关于持续交付过程中几种开发模式和这些开发模式本身的特点，和我们如何通过选择相关的开发模式来提升我们的研发效率。首先来看几种模式，

第一种就是主干开发模式，这种开发模式基本上都是基于 master 的开发，它需要我们所有的研发人员将代码提交到主干，然后基于一个分支去进行相关的一个 feature 的发布，这时候线上会有大量的开关，这些开关都是在我们的主干中，从而来保证我们不同的功能之间的切换。


是通过这个开关来保证这个的好处是什么呢？好处是主干的所有的代码都是最新的，并且分支上相关的代码都是稳定的，不会因为主干上发布到分支上之后，影响之前的一些分支发布，那也意味着它的回滚可以非常的快速。而所有人代码都是最新的，它的好处也是我们在基于某个节点去发布的时候，我们就不会再有合并代码这样的一个困难了，因为每个人都的代码都已经提交到骨干了，只要针对当时节点去发布即可，而针对还在开发中的则通过开关去关闭，防止它透出到线上之中。所以这里带来的一个负面的一个点是什么呢？就是如果开关没有管理好，可能就会有一些开发中的一些功能被漏到线上去，这是我们不希望的一些状态。第二种叫做 GitLab flow 开发模式，这一种开发模式其实是这几年开始逐渐新出来的开发模式，它其实和一些分支模式是比较像，但是有些区别，这里其实认为是说我通过一个开发的一个分支和一个 release 的一个分支，这两个分支来进行相关的切换，它是来确保日常开发的，而通过 master 是最终上线使用的分支。


这边的特点就是我所有的新分支，我会基于主干，基于 master 去拉取出一个新的一个这样的一个分支，然后开发都会在这个新的分支里，并且不不同的环境会有独立的分支。比如说我的日常会有一日常研发力分支，然后我的预发布会有一个预发布的分支，如果我要有项目环境，有多仓环境，那每套环境会有一个独立的分支，这时就会出现我开发可能是基于我本身那个开发的一个分支，比如说我开发就叫 new feature 这样的一个开发分支，然后提交到日常上的时候，它会有一个日常上的这样的一个 release 的一个分支，那所有的代码是要合并到这样的 release 里面去的。


而到最终发布的时候，所有的发布变更会合并到主干，然后通过主干去发布，这样的一个好处是什么呢？这样的好处就是多环境的时候可以并行，并且大家可以完全不受影响，并且因为分支的隔离，那所以环境之间也不会受到对方代码的影响。好处就是可以多个需求并行，可以不同阶段的需求也并行的去开发，然后最终根据上线的情况来进行合并。


上线，但是缺点是什么呢？缺点是会经历各种各样的合并，在哪？比如说从开发到各种环境上又一次合并，不同的分支都提交到环境上，又有一次合并，好最终提交到 master 上的时候还会有一次合并，所以这里的代码合并会比较多。还有第三种也是以比较传统的分支开发模式，这种其实就是通过 feature 和 bug 的来拉取这样的一个分支，然后有对应的 release 和master，那它也是一个基于的分支去开发，然后分支需要在发布之前合并到主干，通过主干去进行开发，那我它通常就是基于一个feature，就一个我们一个功能，然后去拉取这样的一个分支。所以这个更像是以一个以功能为主要单位去开发的。但是开发之前需要先把它合并到主干，所以我们最终肯定希望是我们能够更加灵活的让我们的持续交付过程中更并行。


而针对代码合并这个点，那则是相对来说它比较低频一些，尤其是面对我们提交代码，我们开发过程中调试的这些工作，它肯定是很低频的，可能在整个的一个开发周期中可能会经历到 4 到 5 次，最多 5 到 6 次的一个合并，但是本身消耗的时间其实相对是可控。那针对这个点，我们在持续发布过程中就需要去考虑我们怎么样子去做好这个选型。那肯定是说考虑持续和持续发布对于代码和开发和部署有更多的要求，对于代码的合并反而它可以接受一些妥协。这里考虑就是如何选择一个合适的开发模式，去促进取架构的同时降低开发成本。

它可以在代码提交到这个仓库之后就会触发相关的部署。那这时也意味着如果我们的环境和我们本身的代码之间结合是有一个很紧密的一个结合，那它就会要帮助我们来完成这个点。所以代码合并相对来说是比较频繁，但是常规来说就只要二者代码没有冲突， Git 可以帮助我们去完成这个合并。


第二我们也需要这样的一个拖沓环境去持续测试，比如日常我要预发布项目环境等等。那这样也意味着我们需要多个开发分支并存，而非完全基于某一个分支完成所有的工作。另外一个点就是我们也可能会有多个版本并行的进行开发，那这里也意味着我们针对这些版本之间有一个灵活合并的诉求，我们不是说一个分支一条肝痛导底，更多的还是说我们同时有多个并行版本，那这些多个并行版本之间可能会进行灵活的合并。比如说我开始是 a 分支和 b 分支，然后部署到了一个日常的一个环境，那这时候 a 和 b 就要完成合并，而这时我们的预发布环境中是 a 和 b 和 c 和 d 四个分支同时合并，那这种状态下意味着我们要能够灵活的进行相关的合并，而避免说我需要手动的去拷代码，或者手动的去完成分支之间的拷贝合并，如果那样子的话就会带来很多很多的麻烦，这也意味着我们的整个的一个思路就很清晰了。

然后在我们整体的开发过程里进行不断的开发，然后最终在合并上线到线上去讲这个过程。那正如左边的这张图，他从 master 去拉取一些新的分支，然后这些分支进行开发 deploy 之后，最终会进入到我们的 priori production 这样的一个环境，染完后在测试完毕之后，最终才会去部署到我们的 production 这样的环境。


所以在 master 的分支外，我们其实不同的环境有不同分支，就是我们刚才讲的 get up flow 的模式中的时候就讲到了，然后每一个环境分支相对来说是固定，然后动态的将我们的开发分支合并到这个环境分支中进行部署，而开发分支它就会由每个研发同学自己去创建，然后它就在上面去修改相关的代码，然后合并对应的环境分支。
比如说环境，开发环境那可能就会有一个开发环境的一个 release 的分支，然后到预发布环境就会预发布环境的一个 release 分支，然后再到发布的线上，之后最终合并到 master 上。只有紧急的一些情况才允许跳过上游直接合并到下游分支，比如说我们从开发分支直接就提交上线了，这种情况下只有紧急情况才允许，否则的话它很容易出现相关的集成的问题。

正如左边这张图，我们常规我们不管怎么开发，我们比如说我们基于 master 去开发，当我们需要去发布一个稳定版本的时候，我们肯定需要去从主干上去拉出一个分支，这个分支去承载了相关的稳定的一些代码，那这个代码其实通常来说就不会在大规模更新了。比如说就是我们之前想的那些tag，其实就有点像这种感觉，对吧？就针对某一个发布，我们会打一个tag，那这个 tag 通常来说就不再不怎么再更新了。但是如果要修补bug，也可以允许将某部分代码合并到这些分支，并且更新对应的小版本号，那这时候我们的 2 - 3 stable 和 2 - 4 stable 其实是两个并行前进的版本，这两个版本的代码可能会发生了某些的不一致，因为其本身支持的特性和相关的 bug fix 都会有区别。


这种状态下就是从 master 然后到一个stable，然后再从 stable 去拉一个分支去修改相关的一些bug，然后再把 bug 合并到这个 stable 的这样分支里。通过这种状态相当于是我们在持续的开发的过程里，然后针对某一个节点下，然后去发布的一个稳定版本，然后这个稳定版本就处于一个独立的分支，然后再通过一个小范围的一个迭代，这是主要是 bug fix 的迭代，这样的话就能够完成我们面向于不同的大版本的一个发版这样的一个诉求，所以这样子是我们基本上就把我们在整个研发过程中所面临的这几种开发模式都给大家介绍，本身也结合着持续交付这样的一个思路去重点讲解了 GitLab flow 这样的一个开发模式。所以这也是我们在目前常规的持续集成中，常规的持续交付中最常用的这样的开发模式。
