我们先来聊一聊持续集成。神经集成是一个已经有好多年的一个概念了，

在这个概念里基本上就是我们研发它会去提交，它会对应的代码提交到我们的代码仓库，比如说我们的 GitLab 里，或者我们的 ICV 里也好。但目前大部分用的 GitLab 在提交完之后会有对应的触发我们的 CI 平台，比如说我们的Jenkins，在这个平台中它就会去执行相关的build，然后 test 和 package 编译、测试和打包完成打包之后就会把对应的包提交到我们 ARTIFACT 的reportionary，就是我们的这个包仓库提交到仓库之后，并且通知我们的研发，我们打包已经成功了，并且把对应的包的地址发送给研发，研发这时候就可以拿着这个包去完成相关的部署。这样的一个过程就是我们的持续集成的一个过程。


在这个过程里其实它强调的是开发人员提交了新代码之后，就会立刻的进行构建和测试，这样子的话就能够在第一时间完成相关的构建，并且因为我们的代码是持续提交的，那也意味着我们提了个代码就会发现我们是不是构建失败了，我们测试有没有通过可以持续的异步的在 CI 平台上完成相关的工作，并且根据测试结果我们可以确定新代码和原有代码是否能够正确的集成在一起，所以它叫做持续集成，意思就是我们持续的进行代码的集成，在持续集成这个概念并且在各个公司实施了好几年之后，逐步的开始诞生出来是一个新的概念叫做持续交付。

在前半程的过程是很像的研发提交代码到代码仓库，然后触发 CI 平台的对应的编译，然后测试和打包，在这个完成之后就会将这个包提交到我们的一个包仓库中去，然后 CIP 它又进一步的去做系统的提升测试，然后去做我们的用户验收型的测试。


再往后就是我们要把它发布到我们的生产环境中去，当然这个发布过程是人工发布的，这里所有的过程都会依赖着我们的这个包仓库，因为它相当于从包仓库中把这些对应的包拿过来，然后完成对应的验证。而持续交付就会发现它比持续集成如果多了系统集成、用户验证以及人工发布的这样的过程，并且在最终它会将结果都会返回给研发，那在这样的过程里，我们可以发现它屈指是持续集成的一个下一步，并且他强调的是不管怎么更新，我们的软件都是随时可以交付的，因为在整个过程中它已经完成了系统集成和用户验证这样一个测试，我们拿到的是一个随时可以交付的这样的一个产品包，它就会比持续集成更往前一步。而再进一步就是现在各个大厂正在去努力去做的这种持续部署。


持续部署其实跟持续集中非常的像，它在前半截，甚至说基本上在部署之前都是一致的，


都是研发提交代码，然后触发相关的集成，包括构建测试和打包，然后进行相关的系统集成测试、用户验证测试和最终的上线发布。但这里强调的一点是它是自动的，它不再像直接交付一样是一个手动的一个发布，那在这个过程里它也会将所有的结果返回给研发。


可以发现持续部署其实更像是一个在持续交付之后，代码通过评审就会自动部署到生产环境，目的是可以随时部署税入，迅速投入到生产阶段。但是持续部署在现在其实使用的还是相对比较少，因为自动部署到线上有的时候还是有一定风险的，


所以目前更多的还是从持续集成向持续交付发展，那未来我们也要向持续部署去进步，

进一步的进军目的也是为了能够更好的更快速的完成相关的上线与交付。现有的我们的微服务容器化、云原生这些都可以帮助我们把持续交付这个点做得更好更透，因为像之前我们还需要人工去往线上去进行部署，这样的一个过程其实是不太利于持续交付进行集成的，但是目前微服务容器化之后，完全可以通过生成对应的容器镜像，然后交由 K8S 这种云原生平台完成相关的一个交付部署，就会显得更加的容易。


所以持续部署在云原生时代下，它即将迎来它的一个新的时代，所以主流公司目前已经从持续进程进入到了持续交付的时代，但是测试和部署还是需要有规律干预，其实目的很简单，就是我们还是要保障线上的稳定可靠、安全生产，我们不太敢完全撒手让所有东西完全机器化，我们还是需要有人来介入，人来检查。


实际上持续部署仍然是一个未来的趋势，也就是在云原生这样的一个趋势下，未来是属于持续部署的。好，这就是我们这一章主体为大家要讲解的内容，后面就会围绕着持续集成、持续交付、持续部署进行展开，来讲解持续交付的过程、交付的质量、交付的周期以及相关的一些模式的探讨。我们这一小节先到这里，让各位架构师们，我们下一小节不见不散。
