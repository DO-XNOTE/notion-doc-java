根据我们的灰度目标去制定我们的灰度策略，在灰度目标下去选择，我们要验证这个目标我们需要哪些用户群体，当然比如说像全量，像新版本发布这种从 a 版本到 b 版本的演进，那我们的用户群体也是需要说我们要明确出我们第一个阶段需要选择哪个用户群体，圈好我们的用户之后针对这些用户群体去设置好他们相关的一个灰度定向。


那这里通常我们用的方式就是去明确他们的cookie，从而在他们的 cookie 中带上特定的标识，这样子的话我们就能够在服务端去知道这些人他是一个灰度用户，我们要针对这些人去给他们展示相关的新版本，然后根据 cookie 中所带的这种标识来做好相关的路由，来明确我们到底是版本 a 还是版本b，并结合着机器占比来进行双重数量的一个约束，在这种状态下最终来获取我们灰度的一个结果数据，并结合着数据之间的对比来明确我们版本之间的演进本身是 OK 的，新版本的功能使用上是没有问题。


当然我们还额外考虑像灰度时间、灰度覆盖和灰度的异常。灰度时间是为了能够确定我们采集到的足够的数据，比如说我们在某一个阶段只回了 5 分钟，那可能我们还没有采集任何的数据，甚至连异常都没有报上来，我们就向下一个阶段去走了，这时候就可能导致其实已经发生异常，但是我们还不知道。所以有充足的灰度时间才能够保障我们有充足的数据进行相关的决策。


其次就是灰度的覆盖，我们灰度的覆盖功能范围要尽可能的是覆盖我们变更的相关部分，当变更比较大，我们灰度没有办法覆盖所有的变更范围的时候，我们一定要覆盖灰度中最主要最核心的那部分，这样子的话我们的灰度才有意义。否则的话如果灰度覆盖不到大量变更的一个范围，可能就导致灰度没有问题，但是实际上发布之后反而出了问题。


最后在灰度中出现的异常，我们需要人工判定，或者根据提前设定好的预案去判定是否需要回归，因为有些异常可能是一些常规的，或者是一个不注意导致线上故障的异常，那这些异常有可能会让他在下一个版本的修复，也有可能是说我们马上要修复，这就是马上要修复的，就意味着我们要进行相关的回滚。所以通过这些策略我们就能够明确出我们当前的这个阶段是否还可以继续往下一步去走。

这里灰度目的肯定是我们第一步的，然后围绕着我们的商业策略、功能体验和新版本发布，这是三种很常规的灰度场景，就像超越策略，我们要明确这个模式下是否能为我们带来收益。通过灰度可以进行相关的测试，通过对比来比较我们新的模式和之前的模式是否能够更优，而功能体验一方面可以通过少量灰度去采集相关的数据去验证这些功能是否正常，而另一方面也能够通过 AB test 确认我们这些功能是否是我们需要的，是否是用户需要的。


而针对新版本发布当然就是一个发布的一个过程，我们避免新版本带来一些问题，同时还有一些定向的一些服务，比如说我们某些服务只针对了其中的某一类用户群体，那我们需要灰度就是意味着我们要针对这部分群体进行验证，避免节目发了之后才发现针对这类群体存在的问题。


当然跟功能体验对应的 AB test 和一些高危试错是什么，这个发布我们已经知道它有具有非常高的危险性，它很可能发布就报错，就是说就出故障了。那这时候我们通过灰度先用小流量去验证一下是不是它真的存在问题，也有可能它本身没有问题，只是我们过多的担忧了，这些都是我们本身的一些灰度的目的，围绕着这些场景去设计好我们对应的策略，包括应用的灰度、链路的灰度和前端的灰度。那这是三种实际上落地的一些方案。


应用的灰度是我们有两个应用，比如说我们APP，我们可以用都不同的版本。通过这几种灰度法主要还是在于a、 b 的 test 和功能上的一体验，可这对链路的灰度则主要是一方面可以逐步的切换，另一方面也可以做成 AB test。而前端的灰度更重要的就是在于功能体验了，我们看看这个版本的发布，你看一是否产生了所需要的这种效果，第二方面也可以做二者功能体验上的交互使用一些验证。所以不管是应用链路还是前端，它都可以兼顾着验证性灰度和探索性灰度两种灰度的工作。

就是我们所谓的灰度量。在验证一些灰度，我们要验证功能是否正常，行为是否符合预期，这些都是我们需要验证的一些点。那在这些我们就要看短期上我们需要采集哪些数据指标，比如说像异常的数量是否灰度，发货了之后就出现了一些异常报错，用户的流量是突然增加了还是突然下跌了？要交易的情况，我们这个发布了之后，用户的交易是否出现了断崖式的下跌？这些都是我们去发现问题的最主要的场景。


因为很难说一个新版本发布之后流量会暴增，交易会暴增，这些人来说都比较难，因为增长是一个缓慢的过程，但是下跌可能就是个断崖式暴跌的状态，因为一旦它出错了，用户不可能往下走了，整体的流量和交易都可能会下跌。所以我们短期内关注的这些主要是去关注问题，去发现问题，然后去回滚对应的灰度，那这对长期来看，因为它这个长期的过程也说明我们在短期内没有出现错误，我们去看长期情况下这个版本的影响，那这里就包括用户的使用情况，比如说我们的用户新增的情况跟原有的这样趋势是否是增加了？如果发现我们用户限制度变少了，反而说明我们本身的一个体验或者是功能不如之前了。


其次就用户时长到底是变多还是变少了里也需要针对我们本身的产品的情况去看。比如说如果我们是一个比如内容型的这种APP，那我们肯定希望用户长越好，但是如果我们是一个工具型的，让用户用完即走，那么肯定是希望用户使用的时长更短，因为他用的越短说明他的体验越好，他使用越方便，他更愿意来使用我们。


当然第三还有用户的留存，这个就是非常关键的了，不管用户新增是多少，如果没有留存且没有任何意义，如果即使用户新增缓慢，只要我们留存高，我们仍然可以积累起大量的用户割掉。所以我们来看看新版本的用户留存是否比原来更优了。其次就是营收的变化，这部分就非常直接了，就直接体现在我们财务报表中，看看是不是营收增加了，以及利润相关的变化。通过这些能够长期起来去看我们的新版本是不是比老更好、更优，是不是满足了我们新版本发布的预期。好了，到这里关于整个策略我们就已经为大家介绍完了，策略重点还是什么呢？第一，明确灰度目标，选好人群，定好我们的发布节奏，确定好发布时间，灰度的覆盖，然后关注灰度的异常，明确我们的灰度目的，来看看灰度实际上的效果与我们的目的是否契合。并且从短期和长期结合着用户数据去看一看我们的灰度是否达成了预期。
