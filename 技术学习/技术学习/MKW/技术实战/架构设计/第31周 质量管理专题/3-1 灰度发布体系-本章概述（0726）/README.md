它的目的是为了通过同时的部署多个版本来验证两个版本之间是孰轻孰略，或者是从旧版本向新版本演进的过程中，新版本是否出现了一些错误，从而能够快速的发现问题，解决问题，或者在进行二者同时的这样 AP test 的这种灰度中能够去帮助我们去验证到底哪个版本更好。


那所以它本身那个过程都是在于 a 和 b 两个版本之间的比例，通过不断的调整最终达到全量上线这样的一个过程，那在这个过程里它能为我们带来什么样的价值？第一可以降低风险，第二可以控制相关的影响，第三能够帮助我们更好做快速恢复。怎么去理解这三个点？因为我们两个版本同时在线，所以我们影响到的只是部分的用户，所以这个包不帮助我们降低了风险。同时因为我们可以在灰度过程中持续的跟进，持续的盯着我们的系统去验证我们的系统，所以我们也可以在第一时间去发现问题，所以这里就是对应的快速恢复，降低风险和快速恢复都共同贡献给了控制影响。那么在这个过程中我们就需要看我们要怎么样子去做。

我们面对的什么样的客户群体，以什么样的方式交付档位，他们交付给他们我们的功能，然后这里就涉及到了我们要针对用户的特征标识进行确定，然后通过把我们的系统灰度给他们，并且在用户侧我们去看用户的一些反馈，从而来完成功能性的验证。


如果验证没有问题，比如说我们在北京区域的整体的灰度看，哎，所有的反应都正常，那我们去进行阶段的移动，将我们策略中第一阶段宣告完毕，然后移动到下一个阶段，最终来调整我们用户群体，最终使我们的用户群体达到百分百覆盖。那这就是我们整体的一个验证协会中的一个流程。在最终我们目的肯定是希望能够正常的完成 100% 补，但是一旦出现问题，那我们要进行相关的回滚，从而确保我们在最快、最早、影响范围最小的情况下完成公共。所以我们也要确定每一个阶段都能够去验证功能。
