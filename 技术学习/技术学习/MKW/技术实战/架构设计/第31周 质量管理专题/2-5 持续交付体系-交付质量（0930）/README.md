# 2-5 持续交付体系-交付质量（0930）

hello，慕课网的各位未来架构师们，欢迎回来，在上一小节里，我们为大家主要介绍的是需求周期，需求周期是指从需求提出到最终验收完成的整体的一个周期。那我们本身能做的是什么呢？如何在持续交付这个体系的一个支撑下，然后进一步的去提升我们的研发效率，然后降低整体研发过程在需求周期中所占好的时间。这样子的话也意味着我们可以用相对固定的时间来承接更多的需求本身可以并行的一些事宜，因为不同工种同时在合作，这些并行的事宜就可以帮我们争取到更多的时间。


那么在这一小节里，我们将继续沿着交付体系往下走，我们既然聊不完周期，我们下一步一定会在聊这个周期内我们去做的这些事情，他们的质量究竟如何去保障？首先我们来还来一首打油诗，

就是产品一直都很浪，需求一天一个样，明天就想我上线，用户有没有放心上？那这个我想是每一个技术心中都有的这样的一手打油诗，我们很多时候都不希望产品一直去改需求，尤其是那种一句话，需求或者是明天就要上线的这种需求，这种紧急的需求对于我们来说通常都可能会带来的大量的不确定性，甚至说为线上引入故障，那么我们就需要去考虑在效率和质量之间孰轻孰重，那就是我们在整个交付质量的过程中需要去考虑的很重要的一个点，在效率中我们就有这样的词序作为它的指正，世上无坚不摧唯快不破，那效率就是快的一个最有力的支撑。


并且还有这样的一句话，就是世上只有第一和无名之辈，那也意味着我只有越快我才能够占据越有力的位置，而剩下的慢的人可能就会被人遗忘了。但是它从质量层面上也会有这样的一些佐证，比如说客户，第一，我们要保证客户的利益，客户所拿到的这种产品的质量是第一位的。第二就是十次满意也留不住一次失望，你给用户提供了十次满意的服务，只要有一次他失望了，他可能就转身离去寻找你的竞争对手，从而为我们的产品带来不可估量的一个损失。


因为如果用户大规模的流失，很可能产品离死也就不远了，所以这样也就引出我们今天的主题，就是交付的一个质量红线，

我们需要保证我们本身的一个交付质量，它的整体的偏离是可控的，并且没有对线上引入大规模的故障，否则的话就意味着这次需求本身带来的价值可能远远不如其质量出现问题带来的损失。


那么在偏离这个层面上，我们要考虑，第一，实现偏离需求，它所描述的这样的一个产品愿景和我们本身实现出来的一个产品实况，二者是否是一致的，这个实现的一个过程有没有发生偏离。第二，验收的过程是否充分且完备？产品有没有验收？业务有没有验收？他们验收的过程里有没有充分的考虑到用户的场景，考虑到各种各样细分的这样的一些角落，如果没有，也有可能在用户使用的时候和我们实际做出来的结果可能就是不一样的。而故障这边的话，除了代码问题、数据问题、配置问题、环境问题这四大类主要引起相关故障之外，还有可能就是因为产品设计不大而导致的线上用户带来的大规模的投诉，大规模的流失这些非因代码原因而引入的故障，那这些都需要考虑在内，从而去明确我们每一次迭代交付出来的产品，它的质量是不是都可以拍着胸部说没有问题，只保证这一点是我们所有的决策都需要去紧密合作才能达到的。


并不是说我研发，我好好写代码，我测试，好好测就一定能够保证所有的质量，一定是从业务到产品，到研发到测试，甚至到后续的运营，大家齐心协力共同的努力的结果。那在这样的一个背景下，我们就会理出我们职业交付质量的一个思路。


第一个就是业务方他们也需要在这一中去承担一部分的责任，就是这里的责任，不是说他们背锅，是他们也要用他们的努力来保证整体的一个质量，

而这对开发来说必然是我们要去做好整体的开发工作，而测试也要把测试充分的完备的进行，从而保证我们的产品在交付到线上的时候，他是经历了过了我们的努力仔细的考验过的，并且在运维同学他也要保证我们的上线的这样一个过程是一个顺利、完整并没有遗漏的这样一个状态。


所以这四种角色在整个质量交付过程中，都需要去可充分的参与之中，并且大家齐心协力达成我们的效果。那这里从第一步，从业务到开发这一角度，我们就希望用需求拆分小步快跑。需求与产品他们共同去明确出来的业务一定是拆分到相对细的力度，从而能够支撑小步快跑。这样子的话，一旦发现问题，可以快速地纠正错误，调整方向。比如说我们引入我们的 Scrum 体系，通过 Scrum 来完成我们本身的一个迭代的一个管理，专注于迭代的一个质量。而在开发到测试这个环节里，就只要需要进行持续的测试，每一个迭代都与上一个迭代进行持续的集成，并且在集成之后进行系统和用户层面上的测试，那这里就需要将持续集成进来，快速的发现问题，快速的修复问题。而在测试到运维之间，尽可能是通过研发到运维这个自动化，而非靠人去兜底，那这里 DV OPS 可以帮助我们更好的完成产品的部署。


从开发到部署上到线上，其实不需要一个跨越角色的一个充分沟通，而由开发自身来完成相关的上线，通过这三项的共同的一个助力，可以帮助我们来保证我们持续交付的一个质量。所以从管理的视角去看这个交付过程的一个质量，我们其实需要的是什么？第一，过程项稳定，这里包括几个点，比如说交付是一个及时的一个状态，

然后各角色能够去辅助检查，保证我们的质量是一个经过多个角色验证的这样的一个状态，避免因为报某个人的当日的状态不好，或者因为它本身的粗心大意而导致引入相关的问题。


第三，就是将我们的灰度发布这样的一个能力进行充分的覆盖，确保我们发布的每一个重点的那个功能都能够进行相关的灰度，从而让我们能够以最小的代价去发现问题并且修复问题。而在过程稳定之后，我们也要考虑我们整体的交付质量的这个流程是否是一个合理，因为只有一个合理的流程才能够更好地帮助我们来完成相关的质量开发，那这里其实就第一点就是我们自动化是否有可充分的覆盖，自动化可以帮助我们降低人的投入，同时也可以减小人个人的状态和个人本身的一些特点对其产生的影响，那机器说话自然不会撒谎。


第二就是相关的一些静态扫描，它可以帮助我们发现一些我们平时没有注意到的一些角落，比如说我们的编码风格，比如说一些可能潜在的 now potter，比如说一些安全漏洞，它都可以帮我们发现出来，从而改善我们整体产品的质量。当然整体的集成测试也是非常关键的，我们必然要保证我们当前的版本和上一个版本是兼容的，从而让用户能够更顺畅地延续他们的产品使用思路，并且在上线之前去设置相关的卡点，将安全漏斗、潜在风险和代码规范中存在的严重的问题进行卡住，让我们研发同学能够确保代码上到线上的时候一定都是质量和效率双重保障，所以交付卡点也一另一方面可以去确保我们的整体的过程稳定，这样就形成了一个正向的循环，可以帮助我们保证我们的开发节奏，辅助质量保证，并且通过卡点来杜绝相关的状态风险。最终让我们的线上质量可以做到最好。


好到这里我们这小节的内容就完事了，我们这一小节希望问大家从管理思路结合我们针对于整个持续交付体系中各环节的一个理解，最终确保我们在整个持续交付过程中能够让我们的质量达到一个最好的一个状态。在下一小节里，我们就为大家聊一聊开发模式，不同的开发模式与持续集成、持续交付怎么样子去结合，最终能够为我们带来什么样的一个价值？好，各位未来架构师们，我们下一节课不见不散。
