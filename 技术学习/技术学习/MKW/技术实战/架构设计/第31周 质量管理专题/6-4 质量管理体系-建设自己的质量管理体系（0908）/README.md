那当然也包括我们需要去考虑我们怎么样子在当前的质量管理体系中去推动质量的升级，最基本的就是从我们最传统的代码质量向服务质量去升级，因为代码质量还是太基础，而服务质量才能真正能够体现出我们线上用户感知到的服务它当前的一个质量情况的这样的一个指标。


从测试本身来说，也要从功能测试逐步的向快速恢复去发展，功能测试是在线下的一个保证，而快速恢复只是在线上的一个保障，会帮助我们去预防线上的故障。因为光宠物的测试不一定能够保证我们一定线上突出问题，只有说出了问题快速恢复，它才让我们能够承受最少的损失。所以这也是说我们要从线下质量向线上质量发展，而从模块质量向链路质量去发展。


当然像百度这种体系也可以说从模块质量向领域质量去发展，这样子的话我们整个的一个质量升级就从原有的偏向于线下的，偏向于代码质量的这种纯测试的工作，面向于服务，面向于客户，保证线上快速恢复以及全链路质量的这样的一个质量保证，这样我们肯定是希望我们的质量能够越来越好，我们的缺陷越来越少，质量的和管控和保障可以做得越来越好。

包括我们的质量规则和质量礼物层规则，也包括我们本身的一些规范，以及我们本身面对于质量所设想制定下来的一些约束。而流程就是这些约束在最终以一个 SOP 的形式固化到我们执行过程中，那这就是流程，这二者都会最终体现到我们的场景中去。


我们理清楚这个质量场景之后，还要将其场景做对应的句话，包括我们的场景主体用场景的任务、场景环境和场景的变化，这四个都是什么概念？常见的主体，我们这个场景面对的是一个什么样的主体？它解决的是个什么样的核心的一个问题？那这就是我们本身需要去考虑，需要去明确出来的。


比如说我们的电商系统，我们的下单就是一个场景，那这个场景中的主体是什么呢？其实是我们的下单用户他的场景任务。什么场景任务相当于就是能够顺利、准确、快速、便捷、流畅的完成它的下单，那场景环境就是它所处于的各种各样的电脑环境，我们的网络环境，包括我们的服务器环境，整个串下来的环境的一个组合。还有环境变化也是指所有的环境都不可能百分百不变，我们服务器压力也在变化，我们的链路也在变化，我们网络也不停地在变化，我们甚至可能会有一些细节的一些故障，有一些损坏、有些压力等等，都有可能导致我们整个场景在发生的细节上的变化。而这些变化如果我们没有把它控制得很好，或者是说在这种变化下，我们本身没有能够正确地满足用户的一个期望，那它就可能走向一个出现故障或者出现客诉的这样一个方向，而这些都是我们需要去把控好的这样控制点，所以是一个质量体系，就是要探讨我们到底要在哪些点去做好我们的质量控制。

而研发流程与灰度发布体系就包括我们从需求实现的问题恢复，从而引出的我们需要去做的一个线程变更，然后从线上变更来越过我们的准入卡点，一旦有哪些不满足准入卡点的，我们要在这里卡住，这是为了降低一些发布到线上去的不确定性，然后最终进入到发布环节。


而发布就要进行对应的灰度，灰度这里就意味着我们通过一个过程来分阶段分批的完成这个发布，从而降低我们去验证线上可用性的成本。那整个一个过程就是从待发布到全量发布中的这个 n 个环节，这个环节可以由我们资金域控制，它到底是按百分比还是按用户还是按地域等等都有可以，它是一个群体，一个标记的一个过程，我们通过将明确我们的用户群体，逐步的去增加这个比例，最终达成全量发布就是我们的目的。


所以我们整个研发流程意味着我们要控制、变更、准入、发布与灰度的种种每个环节，最终让我们可以更好地将我们修改的这个部分发布到线上去，并且控制我们发布的内容与发布的过程。
