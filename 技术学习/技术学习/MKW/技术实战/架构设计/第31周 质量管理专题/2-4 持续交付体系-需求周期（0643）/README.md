需求周期是不是就是我们的开发周期？明显不是的，那什么是需求周期？其实是指的需求的一个生命周期，我们需求通常来自于，比如说运营他们提出的用户需求，比如说产品针对于产品形态的对应的一些产品需求，比如说公司围绕着战略去拆解下来的这种战略型战术性的需求，以及技术内部需要去提升整体的代码质量、架构质量的技术需求，那这些都是需求从新建开始，然后进入到需求的评审，它包括我们的技术方案，包括植物各的需求它延伸出来的视觉设计，然后就会进行相关的排期开发。


开发完成之后就会去进行对应的测试，测试完了就进入到待发布的状态，进行相关的灰度发布，并由灰度发布验证没问题的时候完成相关的上线，那这样的一个过程就是一个需求从新建到上线全部的生命周期，这也意味着我们的需求周期所需要消耗的时间，就是从需求新建那个时间点到需求发布上线这样一个时间点。当然在上线之后我们也要做好一点，就是需求的验收，因为如果需求验收没有通过，那也意味着它要在下一个节点上重新进行相关的开发。而验收通过之后，彻底需求就进入到归档状态了。

我觉得这里其实没有太多优化空间，更多的还是从开发测试、灰灰度以及最终的发布上线，这样的一个过程里我们是不是有更多的可以去优化，可以去挤出来的时间？当然最终的验收肯定不能少，那在这里我们需要关注的就是我们在虚线框框里面的这部分，在这部分里我们的目的是很明确，就是怎么样子去缩短这个交付周期，也意味着这样的一个时间轴里，它会牵扯到开发构建、单测部署，然后测试与交付，这是整个的一个过程，这个过程中每个环节应该都不能省。


那我们有没有办法能够去尽可能提升这个速度？那其实这块儿就是我们在整个需持续交付中需要去考虑，也是持续交付为我们带来的价值。首先开发这个时间肯定少不了，但是我们是否是说在两个版本我们可以并行的去开发，然后在第一个开发完毕之后，第一个就可以开始它的构建单测部署、测试与交付。然而在并行的第二个版本开发完的时候，我们也可以进行对应的构建单测部署、测试与交付。这样子的话，原来完全并行的两个版本就可以最大程度上的完成交集，就可以最大程度上的并行进行，并且由于及并行进行可以缩短的时间其实是很多的，因为在这里只有开发是研发所需要投入最多时间的，而构建单测部署这些都由持续交付平台它来帮我们完成了，而测试是由 Q a 介入进行，所以整体的在差异化角色和系统去支撑的部分我们完全可以并行挂掉，所以这里基本上到了最后整个去牵扯到时间分派的更多的还是在于开发的时间。

那就涉及到单一迭代决定的整体的时间周期，以及交付的一个质量管理。那为什么说交付质量管理也会影响到我们交付周期呢？因为交付质量决定着我们迭代重复的次数，质量越好意味着我们可能一次迭代就可以完成，但质量不好意味着我们需要多次迭代，每一次迭代都需要额外的沟通管理时间，也需要额外的持续集成的时间。第三就是交付的流程管理，因为流程越顺畅越清，让那我们整体的流程迭代时间就可以越整体可以越快捷，这样也意味着我们的交付周期管理，就是我们整体质量管理中去做好周期性管理的很重要的一环。当然还有另一环就是我们是对需求的一个质量管理，因为我们这一章节也是就需求的周期，我们在整体的需求周期上去看，也可以去考虑说我怎么样子去提高需求的质量，然后从而降低我们的迭代次数，比如说需求本身的质量和需求尽可能的清晰明确，这样的话可以降低需求的沟通成本。


第三就是需求契合现状，需求过于的跳出现状，比如说需求太未来化了，或者需求太老旧了，可能需求最终所需要发货的时间都会更长，那在这样的一个状态下，我们希望需求能够更好地契合于现状，从而能够更快速的进入开发状态。这样子的话，我们在需求前置的那部分时间就可以进行最大幅度上的优化，从而降低我们整体的一个需求周期。好，这一节到这里就结束了，在下一小节中，我们将会为大家聊一聊在持续交付过程中，我们怎么样子去提升我们的交付质量，从而让我们的持续交付流程可以更加的快捷，更加的稳固。好，各位架构师们，我们下一节课不见不散。
