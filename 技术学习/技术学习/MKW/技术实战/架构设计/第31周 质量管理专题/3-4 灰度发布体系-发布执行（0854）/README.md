帮助我们支撑好我们的灰度能力，它提供了各种多样化的工具以及对应平台能力，比如说像 AB test 平台，它就是一个很好的能够帮助我们做好a、 b 两个版本同时发布的用户去验证收集相关的对比结果这样的一个平台。


当然也一些 dvops 平台本身也带有一些灰度的发布的能力，但是这类灰度发布的能力基本上就是面向于无差别的以服务器数量为灰度的这样的一个灰度能力。当然对于合规的一些后端服务，通过这种方式去验证一次发布是否存在问题，其实已经足够了。所以如果只是面向于这种无差别的灰度流量的去验证，可以常规的来使用 DVOPS 平台去完成，它的成本与实现都是最小的。


这里通常会有一些方法，比如说使用 Beta 环境，它可能就是一台机器或者两台机器，这样子的话我们可以很好的去查看这两台机器它本身的是质情况以及它的使用情况，从而来验证我们这次发布是 OK 的。当然还有一些像元数据的平台，它能够帮助我们去管理好我们灰度发布出去的这些元数据，通过元数据这种标识的方式，能够帮助我们更好地管理定向用户，他访问 cookie 中对应的标识，对应的灰度能力是什么，从而借用委员数据平台的这种能力帮助我们管理好我们要将哪些内容展示给哪些用户，这样子的话它就可以很好的来实现一些细粒度的灰度能力。


同时业界也有一些开源工具能够帮助我们做好灰度与灰度，比如说有一些分组引擎，它能够帮我们更好的将服务端做好相关的灰度分组，并且针对于某一个请求可以指定到对应的分组上，从而达成了一个灰度分派的一个效果。还有一些灰度框架能够与我们的应用提升起来，更好地提供灰度能力。同时大厂来说一般都会自研灰度工具，如果作为一个技术能力比较强，并且人力资源比较充足的公司，各位架构室也可以考虑去自研这样的一个灰度工具。从简单上看的话，灰度工具可以使用一些开关来实现差异化的这种灰度开启女士关闭，从而来完成一些比较轻量的灰度能力。

既有我们本身的灰度平台，比如说无论是 Nginx 也好，还是 CLB 也好，还是说我们自己建的一些工具，抑或是一些云平台，然后帮助我们做好版本 a 和版本 b 的路由，并且明确我们对应版本的信息占比，最终去采集到相关的一些数据，并且进行数据对比，明确我们本次灰度在这个阶段下用户的一些反应与反馈。并且围绕着我们之前的预期判定是否满足我们的预期。如果他的目前状态是比较满足，那我们就要去看我们在当前整体的预期偏离幅度不大，或者是没有偏离，那么这时候我们就要去调整我们的流量，从而向我们的灰度的下一个阶段去走，并且在每一个流量调整之后都要去验证对应的结果以及流量验证。在这个阶段如果出现了问题，或者出现严重的偏离，那么我们就要进行回滚。这里也意味着我们每一个流量调整之后都需要做流量验证和偏离的验证，目的是为了防范那些在小流量下表现正常，而当流量放大到一定阶段的时候，就开始出现井喷式的报错，从而影响我们的线上服务。


那么如果流量验证没有问题，随着流量的不断调整，我们就可能去完成我们的灰度，从而达成全量发布。当然这是一个以流量调整为主要策略的发布过程，

这时候就是意味着我们要做好以用户灰度为主策略的这个执行。在这种状态下，我们的前半截其实是一致的，我们仍然要确定好我们的用户，并且在他们的库位中设定车定标识，借用我们的路由将两个版本同时部署在线上，并且勇于对这个机器占比来确定其流量的分派，最终来验证最相关的也是用户数据和判定用户预期，并且围绕着人群进行相关的验证。


验证在当前人群中我们的结果是什么样子，如果出现了严重的偏离沟异常，那么我们就要进行回滚，但是如果验证是通过的，我们就需要进行人群的调整，放大我们的圈定用户，进一步的验证在更大的人群范围内，我们的灰度产品是否仍然能够满足我们的预期？这时候就形成了整个的执行闭环，不断的验证，不断的扩大人群范围，最终达成全量的发布。


不管是以流量为主节奏，还是以人群范围为主节奏，我们的目的都是在每一个阶段进行相关的结果验证，如果出现偏离就叫回滚，如果没有偏离，一切如预期所预设的那样，那我们就去扩大流量或者扩大人群验证，进一步的走到下一个迭代，最终完成全部的用户或者全部流量的发布，从而完成全量发布。


那么这就是我们在执行整个灰度发布过程中的两种类型下的两种方法，在这两种方法基本上你可以全面的覆盖我们常规的灰度发布，所以我们可以使用这两种方法结合起来，不断地去优化。在这个迭代过程中，我们怎么样子能够更好地验证结果，更好地扩大范围，更好地完成灰度执行，进一步的促进我们的质量管理，让我们的产品质量可以越来越好。好到这里这一小节的内容就结束了，在下一小节里，我们就将针对流量灰度这个过程中的流量分级做一下探讨，看一看我们怎么样子管理我们的流量分级，从而让我们的灰度过程可以更加的平滑。各位未来教育师们，我们下一节课不见不散。
