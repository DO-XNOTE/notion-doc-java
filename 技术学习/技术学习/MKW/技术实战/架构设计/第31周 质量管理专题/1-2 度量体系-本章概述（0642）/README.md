既从需求和问题出发，从而引发变更，然后进行相关的开发测试，最终发布，从而影响到我们的生产环境，这就是我们整个的一个从需求到线上的这样的一个研发流程。那在这个流程中我们也会经历哪些？比如说像需求方面我们会去考虑需求相关的数量，而在开发过程我们需要去跟踪我们的代码，去看看我们的都有哪一些bug，有哪些开发缺陷。而在发布过程中则要做好发布的跟踪。而在生产过程我们就需要去看线上问题，那这些就是我们在发布这样的一个过程中最需要去关注的那些数据，也是这些数据可以让我们去看看我们的同学，他们在整个工作过程中他们面临的需求的情况，他们的执行的开发的情况，他们在进行上线这样的一个过程中的执行的情况，以及在线上真正运行的时候它的代码的质量到底如何。

包括我们可能用多个维度去呈现，要横向的去做对比，尽可能的一不了然和可视化，能够帮助我们更好地了解我们当前的这些指标。它的一个现状，我我们的质量水平发现的现状，我们度量数据它处于什么样的一个状态，同时还会有报表，因为大盘它更多是一个更容易看、更可视化的，而报表只是个更清晰、更完整的，它需要去明确多样化的数据聚合，然后进行多维度的分析去跟分析，更看看各个维度我们到底做的怎么样。然后你同时能够支持上下钻和上卷，能够让我们从一个维度展开，又能够聚合到我们一个维度上去，从而让我们更好地来了解我们当前的数据现状。

然后根据度量诉求去明确和设计，我们都要度量场景，在这个场景下再去制定出我们的指标体系。因为如果我们连诉求都没有，我们上来就拍脑子，去拍这样的一个指标体系，可能拍出来的它没有任何意义，因为我们不知道这个数据它要代表的是什么样的意思。但是如果我们从管理的视角去看这个东西，我们一定是说管理者他心中会有个想法，我希望这个数据能够帮我支撑哪些场景和哪些诉求？它能帮我解答什么问题？它能够有对我有什么样的管理支持？那这些就是这些指标它本身的价值。所以根据指标体系我们才能够去看我们的研发质量，包括历史的数据、现状和将来可能会产生的这样的一个变化。历史和现在是我们没有办法去改的了，对吧？但是未来其实是我们可以去改变的，那所以我们所有的事情都是说希望能够根据历史和现在的情况，然后去看我们未来会往哪个方向去发展，在这个过程中我们是否还能做一些事情去改变这个发增的走势，可以让它变得越来越好，而不是让它变得越来越差。


而针对指标体系来看，也要从多个维度去看，比如说从时间维度，那它就是个我和我的一个比较，


我过去的我和现在的我和未来的我们之间的一个比较。而另外一个就是横向的对比，就是我与他这个比较都在同一个时间节点上，我团队和大锤的团队到底谁做得好？那这就是一个横向的一个对比。在这样的一个情况下，我们就能够通过时间维度的一个纵向对比，和通过跨团队或者是人与人之间的一个横向对比，来完成多个维度之间的一个比较，从而能推动人员的成长。


人家实业维度我与我肯定要比昨天的我做得更好，那未来的我要比今天我做得还好，那这个过程中我就是在成长，而在横向对比中我要比大区做得好，而那大学看我们主要超过他了，他也要发愤图强，奋发努力，他要比我做得好，从而推动了互相的一个成长。而在研发质量它能够帮助我们更好的去看我们当前的项目管理属于什么样的一个状态，我们的项目的质量，我们项目的整个管理的过程中，是否能够有力的推动我们项目的执行，那这些都可以从研发者当中看得出来，而这些也最终也能推动我们团队人员的成长。所以我们的目的其实很明确，
