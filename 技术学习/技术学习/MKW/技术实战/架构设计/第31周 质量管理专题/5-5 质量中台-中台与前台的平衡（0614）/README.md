# 5-5 质量中台-中台与前台的平衡（0614）

hello，慕课网的各位的架构师们，欢迎回来。在上一小节里，我们为大家介绍了一下一个中台应有的一些能力，以及我们如果要去思考要不要中台，我们需要从哪些角度去看，是否能够去支撑，去完成，去面临的一些挑战？我们是否能做好，能不能协调对应的矛盾，如果能，那就坚持往下走，如果不能，我们就要去思考是不是还有去建设中台的价值和意义，还是说只是一个资源的投入却见不到回报？那在这一小节里，我们去聊一聊怎么去平衡中台与前台本身呢？我们这是一个中台，还是为了让它能够更好地去支撑前台，能够去产生对接业务的价值？那如果中台和前台之间的矛盾我没有办法去平衡，这也意味着我们很难特别好的去支撑好前台业务，


那所以对于前台来说，他们最重要的还是客户价值，他们希望能够更好的更直接的去支撑客户，能够为客户带来价值。

最终来看他们是一个业务驱动的，并且他们在面临客户需求，面临这些外部行业不断的变化，他们的变化也是很频繁。所以对于前台来说，它是以需求驱动的一个研发模式，它们本身来说是很需要贴合这个需求，一旦需求有变化，就要快速的响应。并且针对于业务有太多的业务定制，每个业务都有自己的特点，我们需要针对自身的这个业务去做好对应的定制点，从而能够让业务更好的更完美的为我们客户服务，同时因为其快速变化，同时变化频繁这样的一个特点，所以它是一个既要又要还要的一个状态，既要能够贴合业务，又要系统稳定不能出问题，还要更好的用良好体验，用起来很容易很舒服，让用户觉得很好用这样的一个状态。所以对于业务的高速响应和对于客户企业的高追求，使得前台非常贴近客户与业务，也意味着要求更多，变化越快，同时有更多的定制功能，但这就是前台的一个特点。

它的价值是在于前台是否产生的价值。其次它本身来说它也不会完全的面向于业务去开发，更多的是面向于一些复用的功能去做开发，所以它这个复用驱动而非业务驱动。第三点，它虽然也在不断的变化，也在支持好各种各样差异化的高速变化的前台，但是它更终的是要求找变中的那个不变的点，需要变中求稳的这样一个状态，那在这样一个状态下，也意味着中台它为了能够更多的支持好钱的价值，它需要支撑更多的钱财。它是一个多头复用的状态，它复用的越多，价值就越多。


其次，针对于复用驱动，我们就要处理出一些通用的流程，这些流程是放自四海而皆准的， a 来也能用， b 来也能用， c 也能用，这样的一个状态它的价值才会更加的大。同时做好业务隔离，谁怎么变都不影响其他方，这才是真正能够变中求稳的这样一个特点。


所以中台对于多个前台复用的特性，使其思考整体设计和增量功能的时候，不是以某一个前台方为主的，而是更多的去思考怎么满足更多前台的诉求，从而提供更通用加能够支撑更多的前台的这样的一个复用能力，并且支撑好业务隔离，从而保持整体的一个问题。这是中台的一个特点，其实我们能够发现它跟前台是有表现出的差异的，不像前台那样很直接，我业务要什么我就给什么，我用户要什么我就给什么。而对于中台来说，它要考虑的是这个流程我能不能更通用一点，这个功能能不能支撑更多的潜在方，产生更多的价值？不知道业务隔离是不是很明确，很稳定，我能够保证大家都稳定。


所以中台的要求其实很高，它对于整体的技术要求和对于通用的能力的理解都是远高于前台的，但是前台又有太多贴着业务贴着贴近用户的了，他对于行业对于用户的理解远不是中台，俗称平语。

而将差异化功能定制都交给小前台去处理，这样子的话在大中台侧去避免重复的造轮子，而在小前台侧则能够更多的去提高能力的复用。当然它也有对应的问题，就是我的轮子到底是不是适用于所有的前台，我的边界到底在哪里？谁来做什么？以及整体的周期的问题。


这也是我们在上一小节聊这个僵局的时候很关键的一点，谁的应该往前放，谁的应该往后放，从而能够让大家都产生最好的价值，支撑更多的业务，这就是大中台和小前台之间的一个矛盾与协同。好了，到这里我们这一小节内容结束了，整个中台体系也为大家介绍到这里，所以质量中台也是我们希望能贴合着我们的核心业务的质量，去做好相关的能力，抽象建设出可以重复使用的轮子，支撑好更多的前台，能够让更多的前台他们的业务的测试，他们业务的质量保障都能够复用我们的能力，从而帮助我们更好地支撑好我们公司的质量管理。
