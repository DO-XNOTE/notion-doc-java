# 1-4 度量体系-缺陷管理（1045）

hello，慕课网的各位未来架构师们，欢迎回来。在上一小节里，我们为大家介绍了一下在度量体系下的一个需求管理。需求管理本身从需求规划开始，到需求执行，最终到需求的结果与复盘，那么目的是为了在整个的一个需求实现过程中，我们能够对其过程所消耗的资源和其最终的结果进行相关的管理，最终来促进我们能够更好的更高效的转化率，更高的来实现这些需求。当然在这一条件里，我们那将为大家介绍是我们怎么样子进行缺陷相关的管理。

比如说缺陷的分布，缺陷的密度，分布和密度其实是两种不同的概念，缺陷的分布主要是围绕着我们在不同的缺陷中它大概出现的情况。就比如说我们有十种不同的缺陷，每种的数量，那整体的一个分布情况，那这是缺陷的分布，而缺陷密度是相对它本身来说，比如说我这个模块它有多少个缺陷，那这是这个模块儿本身的缺陷密度。


而缺陷的趋势是指在时间的维度下，我们拉一个轴去看一看这个缺陷它在之前这个出现的数量和现在出现的数量，来推断它未来出现的数量，以及说在有这么多缺陷的情况下，我们是不是哪些缺陷是我们漏测了的，而最终它发布到了线上，并在线上产生出了相关的危害。围绕着这些我们能够对于缺陷本身的情况有一个基础的一个度量数据，在这个度量数据下我们就需要做好缺陷的管理，那这里第一个就是要改进我们的开发过程，那第二个就是去提高我们的产品质量。改进开发过程和提高产品预售本身就是相辅相成的，但是二者是谁也得不能少的，一个也不能缺的这样的一个过程。

它能够明确的计算出各类缺陷的分布情况，我们可以通过类型、严重情况等等进行排序，然后就能够比较客观的去看我们应用的质量。比如说我们有一个应用这个用，比如说我们的购物车应用它现在有多少种缺陷，比如说有共同使用上的缺陷，有体验上的缺陷，然后有一些是属于偶发的一些场景叫触发的缺陷。那针对不同的缺陷，那最终我们就可以通过多个维度去看，能够明确出我们这个应用它本身的质量是怎么样。而缺陷密度它是要以从已知缺陷处除以产品规模，我的产品规模越大，那我的缺陷肯定就会越多。所以通过这个就能够明确出我的单位规模下我的缺陷是什么样子，从而能够明确出我们的产品它的机制质量是什么样子。


有了这个指标，我们就能够将两个不同的规模的产品进行比较，从而明确出质量的情况。当然我们还可以用另一个指标来去评估缺陷密度，比如说欠行代码缺信率，也叫欠行代码 bug 数，这个是在各家大厂都非常用，也非常适用的一个度量指标，它能够帮助我们去评定我们每一枪代码中会出现多少bug，从而明确出在整个这样的一个产品规模中，我们的密度到底是什么呢？它可以帮助我们明确出代码质量与应用质量之间的变化一个情况，所以这个也能明确出一个研发人员他的代码质量到底是什么样子。


第三个就是缺陷的趋势，各种状态下的缺陷的一个数量，它都会反映出我们的性缺陷数、被解决的缺陷数、遗留的缺陷数的趋势，从而了解出缺陷是否及时和全面的被解决了。当然也可以通过这个趋势来去来确定我上个月、这个月、下个月以这种时间轴上的这样一个缺陷的一个情况。当然最后还有我们提到的漏测率，非 QA 测试时发现的缺陷数，除以 QA 测试时发现的缺陷数，这个就能够明确出是什么呢？是我们的 QA 同学，它对于指导的一个把控，漏测率越高说明 q a 把控力越弱，这也是我们可以针对性的去提升的一个指标。

比如说我们当前版本的密度数是否小于我上一个版本的密度数，如果是，那就是说明我们整体的质量可能是在提升，但是这里我们还要去看一些前提，比如说我们测试效率降低了吗？我们测试投入有没有减少？为什么要提这个问题？因为提这个问题是用来确定我们当前版本的密度下降，适用我们的投入少了，效率低的产生，还是说其本身的质量在提升？如如果说我们这版本的确测试效率下降了，测试投入减少了，那这里就说明什么呢？说明代码质量可能没有任何变，甚至它变好变坏我们都无从去度量。我们要明确的是我们需要增加测试的投入，同时加强研发测试过程管控。因为我们随着测试投入的减少，那也意味着我们没有办法再像之前一样发现出那么多问题。但是如果是否，那也意味着我们整体的产品的质量在变好，向更好的方向去发展，这也是我们希望看到。


当然随着这个迭代我们就可以说，唉，这个版本发现的确是运动下降了，那后面我们逐步的降低了这个测试的投入。但是在降低测试投入的同时，我们的产品的这个缺陷密度也没有发现大幅度的变化。通过这二者之间的交替进行，从而来一方面提升我们的质量，另一方面可以降低我们测试投入。


另外一个视角就是如果我们当前的版本密度大于了上一个版本，例如这时候我们也要统计一个这个问题，我们测试效率提升了吗？我们测试投入增大了吗？来确定是否是因为我们投入了这些资源，从而来提升了相关的版本密度，但如果是，其实我们是发现了原来没有发现的那些问题，所以我们整体的质量是提高了一个更好的保障。但如果是否，那就说明我们代码质量在恶化，我们需要加大人员投入。而而针对缺陷的状态来看，我们可以用一个图来很好的去理解整个状态的一个变化。

首先我们任何一个状态选择是new，然后我们的研发就会看到这个bug，然后他就去看，唉，我这个 bug 要不要修？如果发现这个 bug 本身就是已经去看这样子，它不需要修，那我们就可以直接把它标记成 want fix，但是如果说是这个它不是不修，但是一看诶产品设计就是这样子的，你不能说这个我知道是错的，只 by design 那这种肯定是就是没有问题的。


当然从另一个方向去看就是会发现，唉，这个问题的确是一个bug，那我们就把它 asend 来标记这个 bug 被认领了，然后加取来明确我是不是要现在要修。如果我现在不修，我要把它标记成later，我们在未来的某一个版本里去修。但是如果它这个比较严重的，我们就要现在去修复它，然后在修复完成之后去将这个 bug fix 掉。当然 later 的这个 bug 早晚有一天也要把它 fix 掉，而不能说一直挂在那无休整雷特过去。


然后在 fix 之后 q a 就要去验证，如果这个问题没有被修复，它就会重新把这个代码给它 Reopen 掉，在 Reopen 之后，那研发看真的没有学掉，它重新要回到这个 design 的这个状态，重新进行相关的修复。但是如果它已经彻底验证被修复了，我们就可以把它 close 掉了。在这种等状态下，这个 bug 就完成了，它的生命周期，它的使命就结束了。我们就可以在我们整体的一个项目度量中去看我们各个状态的 bug 到底有多少，这必须随着时间的维度逐渐往前走。我们看看每一种状态的 bug 数量的变化的一个趋势，最终来评定我们整体的一个质量情况。所以这里我们可以将至此。


那我用数件分了 4 个区，我们把它分区就是 want fix 和 by design 的其实就是特区，它是一个具有特权的，也不需要被修复的这样一个区域。而 new 就属于新发区，就是新发现的这些区域。然后中间的 design 的fixed， Litter 和 Reopen 属于执行区，在这个区域内，代码是在被修复的过程中，而最后一个 closed 就仓库取，这时候我们所有的缺陷就被收拾到仓库中去，留给后人去进行相关的分析与评定。
