# 4-3 产品评测-badcase挖掘（0904）

hello，慕课网的各位未来架构师们，欢迎回来。在上一小节里，我们为大家讲解了关于竞品评测的那些事儿，怎么样子去分析两个产品，我们要从哪些方面去做好相关的产品分析，最终明确我们做好评测的灵魂。三问，评测的目的是什么？评测的范围是什么？评测的结论是什么？通过这些最终还是要将其他人做得好的地方吸收过来，提升我们本身的产品的一个质量，改善我们产品的功能，最终做到一比之长，改善自身这样一个效果。那么在这一小节里，我们会为大家讲解的是 Badcase 挖掘。

 Badcase 的话其实通常是指一个不符合用户心理预期的产品输出结果，用户在使用我们的产品的时候，他通常都会有一个预期，比如说他点击一个内容，他希望出现的是内容的正文，但是如果出现的是个白页，那这其实就不符合他的预期了。比如他去点下单，他希望跳到的是一个下单页，但是结果却跳到了整个的 APP 首页，这也是不符合他预期的。


但是这些行为通常都会藏在用户海量的行为之中，并不是所有的行为都出现了这样的badcase，但是我们就需要在这种排量行为中把它定位出来，找到那些实际上的结果和预期结果不等的这些行为。这些行为就意味着它就是一个badcase，那我们把它发现出来，把它解决掉，让大家都回到一个正常的、良好的符合预期的一个输出的这样一个状态，这就是我们 badcase 挖掘希望去做的事情。那所以这里问题又来了，我们怎么去发现这些实际结果不等于预期结果的这些 case 呢？这里当然是一需有一些难点的。


我们在排量用户的行为中怎么样子找到存在的问题，一直来说都是一个比较大的挑战，因为我们需要面临着海量用户和他们的海量行为，

如何在其中去做好对应的问题发现，并且最终把这个问题修复闭环上到线上去，让后续的用户不再去经受这些不满足预期的效果的行为，这是我们在 Badcase 挖掘希望达成的一个目的。


那这个问题我们怎么去发现这些问题？我们怎么去找到这些Badcase？难道靠 QA 人工去一点点测出来吗？我想这肯定不是一个很现实的一个行为，因为当陆奇出现的概率并不高的时候，靠人工去一点点的发现出来基本上不太可能。那靠众包外包来解决呢？众包外包的确可以降低我们本身的一个人力投入，并且它也能起到一些海量的测试这样的效果，但是作为这种状态下，其投入于产出也未免太低了一点，所以它也并不是一个最好的这样一个选择，那剩下的就是一个通过大数据来挖掘了。我们怎么样子通过用户的行为日志找到一个badcase，它就是我们最好的一个选择。那这里也出来了，就是既然我们要通过日志去挖掘，那我们的挖掘方法都有哪些呢？

通常会有两类的挖掘法，第一类我们要通过数据标注，然后去建立好一个挖掘模型，最终通过这个挖掘模型和算法分析的结合起来，找到对应的Badcase。那这也意味着数据标注其实是一个比较大的成本，而且模型训练的迭代也是对于技术较高的一个要求。


还有第二种就是通过日志挖掘，这里会记录用户的一个行为，通过用户行为挖掘，找到用户行为记录下来的日志中能够体现出来的那些不符合预期的一些行为，比如说在行为延续这个点上，我们可以挖掘出来的一些现象，当每个人的行为都是 a 到 b 到 c 到d，结果有一个人 a 到 b 就没了，消失了。


这里肯定意味着他到了 b 之后，他往下走面临到了某些问题，不管是他自身的一些意愿上的问题，抑或是他的产品使用上的一些问题，那这里就是我们可以去挖掘的一个点。第二就是行为时间，我们本身的平均使用时长是 40 秒，但是一个用户他 15 秒就离开了，针对这类用户，我们其实是需要去关注他离开的原因是什么，是不是遇到了某些功能上的不可误，当然还有就是行为漏斗，我们每一个功能往下走的时候其实都会存在一定流失，但是出现了某一个 a 功能和 b 功能之间出现大幅度下降，那这也意味着这里的漏斗出现了一个快速的紧缩，我们就需要去看一看是否在这里遇到了一些功能上的不可用。通过这些行为的挖掘，可以帮助我们比较低成本的找到一些在本身使用过程中可能出来现在问题。


当然还有第二种就是我们通过预计结果的一个挖掘，因为我们可以记录下来用户本身时的一个实际结果，那针对这样的一个功能，我们也能够去制造出一个预期的结果，我们就可以针对实际结果和预期结果进行对比，如果不一致就说明存在的偏离，那这也就是我们可以定位到的功能上的问题。


第二还有就是针对于我们本身已经在代码中或者在逻辑上判断已经存在问题的地方，我们抛出了一些异常，针对这些异常做好相关的监控，可以帮助我们更好地找到汽车存在的问题。当然，通过算法模型的一个挖掘，二者之间其实并不矛盾，因为日志发掘也往往给算法提供了基础数据，而算法也可以在日志挖掘挖不出来相关的 Badcase 的时候，通过算法帮助我们更好找到这些Badcase，并且能够帮助我们在后续过程中去修复掉它。


所以 Badcase 它很多时候我们在挖掘出来之后需要对其做好定级，

需要你明确它到底是个严重的 Badcase 还是重的，一会是轻微的，为什么要做好定级呢？因为不同级别的 Badcase 影响面可能不一样，针对严重的，我们可能马上就要去修复它，但针对轻微的可能可以排到下一个版本中去修复。所以一旦定级了之后，我们就可以针对不同的 Badcase 进行分级处理。


但是不管怎么样子的等级，我们都需要对其进行问题的定位，找到问题出现的原因，然后分析其原因，看看我们要怎么样子去做好对应的修复，明确出一个修复方案和修复排期，最终在代码中解决掉这个问题，然后上线修复。这样子的话，一个从 Badcase 到上线的这样的一个过程就完事了，但是这就结束了吗？并没有。上线之后的产品仍然要进行相关的 Badcase 的挖掘，继续去探寻是否还存在着其他我们有发现的问题，通过这样的一个闭环就可以不断的提升我们的产品的质量，从而让我们的产品在用户侧流失，可以更少带来一些我们本身增量。那这也意味着算法训练、大数据和日志分析就是我们做好 Badcase 的三个利器，它能够帮助我们更好的完成整个 Badcase 修复的闭环。


那刚才也提到了Badcase，它其实有的时候是可以帮助我们去提升用户量的，或者提升我们一些收入， GMV 都有可能提升，就整体的一个产品的这样的一个状态，它是可以提升的。
