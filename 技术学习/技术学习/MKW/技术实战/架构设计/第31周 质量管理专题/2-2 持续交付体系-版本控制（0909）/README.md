比如说频繁的代码冲突、不兼容的版本、循环依赖的版本以及杂乱无章的分支，这些都有可能为我们带来研发过程中的种种的困难，而这些开发也痛苦，管理也困难，因为在我们的日常管理过程中，我们又涉及到更多的应用，更多的人，那就可能会有更多的代码冲突和各种各样的分支。


那在这种状态下我们又要怎么去管理好我们的版本呢？老师这边引入了三个要点，第一个点就是需求任务关联，第二个点是版本开发规范，第三个是语义化的版本。什么是需求任务关联？就是我们真正开启一个分支的时候，其实是要和一个需求和一个任务去做关联。这样子的话我们就可以围绕着一整个需求或者一整个项目，将所有的分支管理起来。所有的分支之间是有关联的，他们关联到了解决了是一个什么样的需求，解决的是一个什么样的任务，修复的是什么样的bug，通过这样子就将这种没有意义的版本和分支关联到了一个有意义的任务池上，进而进入通过版本开发的规范来约束好我们怎么样子去命名我们的版本，命名我们的分支，最终来达成这样语义化的版本。


每一个版本本身具备着相关的语义，我们看版本就能知道它大概是一个什么样的一个语义。


而在整个开发过程中，我们其实要做的就是做好 CI 的一个持续交付，那首先肯定还是回到了我们这样的一个开发的一个流程，就是从需求实现、问题修复到变更，然后到开发、测试、发布、追踪到生产的这样一个环节。

在这样一个环节中我们一定会建立着我们需要去开启一个版本，在这个版本中它关联着开发代码的一个版本，因为不是只有一个人在开发，肯定会有好几个团队，甚至说一个团队好几个人去做整体的一个需求开发，所以多版本一定是并行开发的。然后进入到这个持续交付体系的时候，肯定需要有一个最终部署到这个仓库的唯一的一个版本。因为不可能说我一次一个流程中有两个版本，那样的话它就混乱了，肯定是最终有一个唯一的一个版本，这个版本是最终部署到仓库中的。


而到了线上之后，线上运行的这个版本，那也是要有明确出来的一个版本号，我们就可以知道我们线上使用的什么样的一个版本，然后它关联起来的前面的一系列事情是什么，而我们提供给我们的数据团队，提供给我们的使用方的则是我们的二方版本。我们的大方包具备的一定版本号，他们可以通过版本号来不断地更新我他们需要的能力。


当然最后还有一些其他的资料的版本，比如说我们的文档的版本，我们一些图片的版本，把我们有一些流程的版本，这些版本也是要关联到我们整体的版本大图里面来的，通过这些版本可以帮助我们更好的管理好这个文档，定一些定义一些高阶级脚本这些资料。所以这么多的版本都意味着我们要在整个的直交付体系中去管理好这些版本，否则的话，这么多的版本一交叉在一起，一定是一个混乱的一个局面。

那每个人会提交到它这个一个仓库，本地仓库，然后进一步 push 到我们的中央仓库，那这里意味着大家要写好这个 commit log。每个人在提交的时候要写明白我这次提交是做了哪一个事情，然后关联了哪一个需求，然后解决了什么问题。当然如果你要简化，你至少也要写清楚本次提交是解决了是做了一些什么样的事儿。然后在版本控制之后，我们就会进入到这种多分支的开发，因为可能好几个人都在做同一个应用的相关开发，那也意味着他需要去与做好对应的一个开发的一个分支的拉取，这里可能就会有我们的主干开发模式跟分支开发模式，不同的模式可能会有不同的使用方法，但最终的目则是大家能够协同好的完成对应的开发。


而分支模式也会更有，像比如说按发布去创建分支，我们这次有一个发布，这个发布会牵扯到好几个变更，那我们把这个变更都放到同一个分支里去，这是按发布创建分支，还有是按功能特性去创建分支。比如说，唉，我的 a 需求对应的功能特性，我创了一个分析 b 需求对应功能，又创建了分支，在这时候发布的时候，我会将 a 分支和 b 分支放在一起，二者一块去发布。


这个就是按功能特性去创建分支，不同的创业分支的方法就会有不同的用途，但是比如说我们如果按照常规的通过需求这种的走这条发布流程去创建的配置，更多的还是以发布去中的分值，因为这样的话我们就可以在一次发布的时候将所有的东西发布掉，而更多的按功能特性去创建分支，一旦发布合并的时候就会牵扯到大量的代码合并，甚至有的时候出现循环依赖的时候就会非常的麻烦。

我们的版本开发才会更有节奏。在这个里面首先要明确是版本的格式，通常来说就是用主版本号、次版本号加修订号作为一个版本格式就是非常合适的，因为它能够帮告诉我们的主版本号是什么，因为通常主版本号都会是一个大版本，而修订号则是针对这个次版本的不停的一个修补。比如说我们像大家都熟悉的 Java JDK 1. 71. 81. 9，这个就是主版本号，然后它后面还会有一个小版本号，这个版本号就是代表了它的次版本号，而最终在次版号后面它还会挂着一个，那个 update 多少，那个其实就是一个修订号确定好通常就是针对一些 bug 里面修复了。


当然本号的递增规则，通常来说主版本号它是当你做了一些不兼容的 API 修改的时候，或者你做了一次比较大规模的功能调整的时候，会通常来递增一个主版本号，而这对次版本号就是你做了一些向下兼容的小功能的一些新增，它的改变幅度并不大，更多的是在原来上做了一些修正，或者做了一些调整，或者做了一些功能新增。
那这时候修改一个字段问号，当你做了一个向下律的问题修正的时候，它本身来说不引入不懈新功能，只是做 bug fix，那这时候针对修订号进行递增，而语义化版本我们更进一步的去看的话，
