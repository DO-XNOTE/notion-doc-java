# 2-7 持续交付体系-变更发布（0955）

hello，莫克网的各位未来家伙吃梦，欢迎回来。在上一小节里，我们为大家介绍了几种不同的开发模式，以及如何将开发模式与持续交付集成起来，从而能够帮助我们改善持续交付过程，提高持续交付的效率。那么在这一小节里，我们就进入到了持续交付体系的最后一节，叫做变更发布。不管我们前面怎么去开发，怎么去迭代，最终我们的目的还是去发布我们的变更，将代码发布到线上，这样子才能够完成我们本身发布的目的。

我们要考虑最终的在这个变更发布过程里，我们怎么样子去管理好我们的变更发布，从而降低因发布而引入的故障，因为保超过 60% 的故障都是发布引入的，所以发布过程的管理是保障质量的一大要素，也是最重要的一大要素。如果在前面都做得很好，但是在最后一环节掉了链子，那可能你前面做的所有的心血都百分。所以这里也会需要去聊几个点，比如说在发布这样的一个触发机制，以及整个发布的一个过程管理和围绕着发布的整体的一个数据度量。我们也是希望能够将整个的发布流程，然后发布卡点以及灰度发布都结合进来，从而来看整个的一个持续交付的一个发布过程是要怎么样子去管理。

一种是通过githux，比如说我们通过 commit 他所在的钩子，然后 push 到了我们对应的仓库，然后仓库它在感受到这个钩子之后去执行到对应的发布，所以这种情况下适它适合于哪些场景？比如说第一就测试环境，我们测试完全提交完了之后，可以由提交的这个触发，然后去完成对应的一个日常的一个发布。


也可以说是我们提交了之后会有一个单独的一个部署的一个按钮，这个部署按钮就会帮助我们完成相关的 e 系列的部署，包括代码的拉取、构建、打包以及提交到对应的开发环境，然后完成相关的部署。它同时也适用于像自动化环境以及像一些内部体验环境，这种不会对用户产生相关影响，同时由自动化能带来的提效就可以提高我们的工作效率这样的一些场景。


当然还有第二种就是通过手动触发，那手动触发它是一个由人发起这样的一个节点，所以这种场景通常来说它会比较安全，因为在点击这样的一个触发这个过程是我们人能够确认并且能够明确其不存在的相关的风险，所以这种适合于正式环境和灰度发布。而因为灰度发布本身我们需要人工去验证，然后去进行整个发布阶段的一个迭代，所以它也尽可能是手动的触发。还有一些就是流程化，它可以通过人来进行流程中间某些节点的确认，最终完成流程的向下走，最终完成整体的发布。

然后准入会帮我们进行代码扫描、 code review 风险的扫描，这些安全相关的扫描，通过来这一部分的卡点，帮助我们把一些低级的或者是一些藏的比较深的一些问题找出来，然后进入到发布的一个过程，在这个发布过程中进行相关的灰度，那灰度的整体的过程就是我们在发布这个过程里从待发布到全量发布中间会经历着几个阶段，那这几个阶段就是我们进行选择进行有针对性的一个阶段性的迭代，从而能够保障我们治理发布的安全。


另一方面也能够让我们最小成本的来验证出相关的错误，所以灰度发布也是我们在整个过程中需要去关注，也是可以帮助我们最大程度上降低因为引入了其故障而带来的损失。那么在这个环节里我们就是最关键的，我这个框住的就是我们最关注的发布过程，然后在发布之后就会上到生产环境，那则就要对其进行相关的验证。那么这个整个的一个过程就是我们这个发布流程，我们所需要做的也是在变更、准入、发布、灰度这四个环节，做好相关的过程管理，这对每一个点都能够有我们针对性的一些管理策略。

并且整体的发布相牵扯到的人员也会比较多。首先是需求的变更方，无论是业务需求还是由组织战略带来的自运营的一个技术需求，抑或是技术内部的一个更新迭代的，这样的技术需求都会有需求的变更方，再有对应的研发同学作为需求的变更开发方。然后在真正上线这个环节，通常来说是由开发方来作为变更的执行人进行变更上线，尤其在当前的第一个 OPS 这样的体系下，在这个环节里最终完成发布之后会有对应的验收方，通常是变更的需求方去进行验收，当然变更的开发方也要进行开发层面上的验收，而在变更执行过程里也会牵扯到向变更的审批人和相关的受影响方，那他们则是一个需要去理解为什么要执行相关的变更，从而明确变更的执行。另一方面也是我们要告诉变更可能会受影响的相关，让他们去关注他们的系统，避免因为变更为他人带来故障。


所以在变更这个节点上，我们需要管理好所有的人之间的协同，从而让信息能够在这么多方之间能够充分的挪动，最终让整个变更过程可以更加的顺畅稳定。

那变更度量这些关注的数据中，第一个就是发布的次数，发布的成功率，回滚的次数，发布相关的事实就是我们来看我们这一次发布它本身的一些成功的情况。那通常来说发布因为我们有各种环境去保证了吗？发布通常来说不会有问题，但是发布为什么还会有成功率？因为有可能会跳过某些环境，从而导致我们代码本身有问题，亦或是在代码合并的过程中会发现一些问题，那这些都可能会影响我们的发布成功率，所以它通常也代表着一些低级错误。


那回滚次数则是代表着我们发布中出现了哪些，我们在发布过程中就发现了线上问题，以及在发布之后发现了线上问题产生的回问，那这些其实都是我们非常需要关注的一些指标。针对灰度发布来说，灰度的接入率与灰度的回滚次数则可以让我们知道我们到底有哪些应用能够进行相关的灰度发布，以及灰度发布又有帮我们提前发现的哪些问题？紧急发布和非发布变更次数，这两个则是在面向于非常规的发布流程中额外的一些发布的带来的一些指标。紧急的发布不是我们希望的，那并且紧急发布很有可能会带来一些我们没有经过充分测试而导致的故障，那所以我们需要去度量它，并且尽可能减少景区发布。而对于非发布的变更，我们也要去进行相关度量确保。我们为什么要做太多的一个线程变更？是我们的后台不够强大，抑或是我们的代码本身存在着大量的一些设计不合理，通过这些帮助我们更好的来确认我们整体的一个发布质量，并且也能够去促进我们的发布复盘。从复盘角度去看一看我们在整个过程执行中到底哪里做得好，哪里做得不好，并且进一步的提升我们的研发质量，然后推动有生改进，最终促进我们的人员成长。当然我们的发布质量它也会出现在我们的发布复盘中，帮助我们来完成整体发布的回顾。好到这里为止整个持续交付体系就为大家介绍完了，那我们希望的是大家能够在整个过程里明确哪些是我们作为一个架构师，作为一个未来的 TL 需要去关注的那些点，从而帮助我们缩短交付周期，提升交付质量保障发布过程，最终为我们的线上带来更好的更稳定的这样一个状态。
