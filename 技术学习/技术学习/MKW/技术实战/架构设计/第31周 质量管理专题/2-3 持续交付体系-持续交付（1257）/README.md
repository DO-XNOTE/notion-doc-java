# 2-3 持续交付体系-持续交付（1257）

hello，默考网的各位未来架构师们，欢迎回来。在上一小节里，我们为大家介绍了我们在研发过程中可能面临着各种五花八门的版本，既包括了我们在研发的这种代码的版本，亦或是我们要上线的这种线上的产品的版本，还有就是我们在平时日常工作中会有一些资料，一些脚本的版本，那这些版本可以帮助我们区分差异化的这些能力和功能，但是它也为我们带来种种不方便，比如说混淆，比如说管理上的困难和混乱。


那么更好的进行版本的管理可以帮助我们更好的来完成我们研发过程和上线后产品管理。那么虽然业界帮我们提供了版本管理的这样一套模板，但是秀女自己的能够帮助自己提高版本管理的这样一个设计才是最好。那在今天这节课里，我们就将为大家讲一讲持续交付中最核心的一点，就是我们怎么样子去做好持续交付。

就研发它将代码提交到代码仓库中，比如说GitLab，那相关的 Hawk 触发之后它就会触发到 CI 平台，比如像 Jenkins 去进行相关的持续集成，那这里持续集成就会包括常规的持续集成三要素，编译、测试和打包。


编译和测试帮助我们验证我们的代码可以与之前的代码完成持续的集成，并且能够基础的测试下保证其通过，从而验证我们的常规的代码是 OK 的，然后在打包之后会提交到对应的包仓库中，并且进行持续的系统集成测试和用户验证型测试，在这两个测试验证之后，就能够来产生出一个可以交付的这样的一个包了。


当然在目前常规的区域交付出过程中，我们的上线发布过程还是需要人工触发的，那在这里整个过程完成之后，都会给研发返回相关的结果，通常是以邮件的方式。那在这里我们就要提出第一个问题是如何推进持续交付，因为整个流程大家看还是比较长的，并且其整个的组织过程是相对比较复杂的，其实最麻烦的一点在哪里？最麻烦的一点还是如何进行更好的系统集成和用户验证型测试，并且在这里的测试如何能够自动，这个是我们的持续交付过程中非常关键的一点。

我们需要做的就是怎么样子尽早的发现问题，并且尽早的解决问题，那这也是持续交付的目的。我在这个过程中我们首先要做的一点就是单测的覆盖，因为只有我们测试覆盖特别全面，那也意味着系统能够帮我们做的就越多。而第二就是说把持续集成这个流程给做好，如果持续集成都是没有做好的，那持续交付本身的基础就不复存在。然后第三就是我们的交付能力，这里的交付能力也意味着我们整体的环境串联需要做好。更细一点去看，单测的覆盖是代码基本可用的保障，如果我们单元测试的覆盖率不够，那说明在基本保障的这一块我们做的是有缺失的，我没有办法保证我们的代码是一定可用的。其次，持续集成是持续交付的基础，如果持续集成没有做好，持续交付是不可能做好在最后交付环境与整体运行的保障，其实这是我们在持续过程中需要去做的这个关非常关键的一点，所以整体来看，如果我们从质量管理的角度去看持续交付的话，就意味着我们要尽可能的推进单词的覆盖。并且第二在整个研发过程中一定要把值域集成做好，能够保证我们的代码是个一个可以制集成，可以通过测试的角度的状态，并且最终来完成我们的交互。

首先要把核心覆盖率要保做好，确保核心都能够被高度覆盖，保证核心的可用。而针对于非核心，那我们可以选择性的覆盖其关键的功能，针对其他的在人力和各方面的考虑下可以胜任妥协。而针对单测覆盖率也需要去一定的保证，否则的话为做了这么多单测都是一场空。


其次就是在持续集成的借力，持续集成其实现在有很多的这种能力我们是可以借用，比如说像开源，像 Jackins 这种就是很通用的这种垂直升能力了，包括还有一些面向于垂直场景，或者是面向于特定的这种功能，像 team city，像这个 GitLab CI 是 GitLab 提供的一个持续集成的一个能力，它可以跟 GitLab 天真的集成。所以如果使用 GitLab 来进行代码的管理，那使用 GitLab CI 就可以非常好的介绍了。


那除了向开源借力之外，我们还可以向服务借力，因为开源它虽然是免费的，是开源，但是我们也要基于它做大量的定制和配置，包括我们本身的机器资源，也需要大量的投入。但是我们就可以考虑去借用一些福利。就比如教育云平台，那它可以帮助我们管理好整个持续集成工具，管理好后面的执行的机器，整个的一个管理都由他来帮我们完成的，那我们需要做的就是花一点钱。


其他就是经验建立，可以去参考一些大厂，他们去怎么去自建的自己的持续汽车平台，因为基本上所有的大厂都是有自己的持续汽车平台的，那我们就可以参考他们的经验去建设我们自身的平台。第二就是我们结合其他人针对开源定制的经验，我们也可以借鉴这些经验来打造我们自己的一个平台，并且和我们自身的相关情况做好相关的适配，确保做出来的区域集成是符合我们的现状的，跟我们流程是契合的，这样子的话我们才能够更好的来完成持续集成。当然这里借力开源是最便宜的，也是最省形成本的，但是我们需要花费时间，花费精力去维护，而借力云平台可以帮助我们轻量级的来完成持续集成的建立。所以在不同的公司发展阶段，可以考虑采用不同的方式，比如说公司现在处于一个非常快速的发展期，那所有的时间都要投入到业务开发中去，那我们借用云平台是最好的，因为虽然我们花一部分钱，但它为我们争取了保护的时间。但是如果我们现在已经进入了一个稳定成熟的阶段了，那这时候逐步的去建立自我的持续集成平台，这是非常重要的。

那这里也意味着我们要管理好我们的多部处环境，它是我们持续交付的基础。在这里我们通常也会有像日常环境，它包括我们的开发环境、项目环境这种日常开发过程中使用的环境，还有我们的预发布环境，它是一套与线上配置基本一致，仿照线上这种行为的这样的环境，这是它我先服务与线上是相通的，从而能够更好地还原我们代码发布到协程之后的这样的一个效果。


还有我们的生产环境，通过这几套环境来完成我们本身的一个多环境的部署，生产环境中还有我们对应的 Beta 环境，它可以帮助我们小流量的完成对应的验证，那这些都可以帮助我们在交付之前可以自我的完成相关的验证。而基于这些多部署环境，那使用 d v OPS 可以帮助我们更好地来完成指引集成测试跟用户验证性测试。并且第一个 OPS 可以打造一个高速响应的文化，它不再需要我们开发完了之后再通知运维去进行发布，我们可以直接通过 DBS 平台来完成研发，就能够在开发后第一时间完成部署。


而 DVOPS 它可以打破这种运维和研发之间的障碍吗？就可以让我们的直接的沟通的成本降的最低，并且能够更高效的来完成相关的一个交付，持续交付的全周期可以无缝的衔接，所以 DVOPS 也是持续交付在这几年逐步的火起来的。这样的一个最主要的一个原因，云原生加 Devops 共同建设出了目前的世纪基层的行业级的这样的一个方案。总体来看，在整个持续交付过程中，我们期望其带来的结果第一个点就是快速，它能够帮助我们更快速的完成代码的集成和交付之前的验证。
