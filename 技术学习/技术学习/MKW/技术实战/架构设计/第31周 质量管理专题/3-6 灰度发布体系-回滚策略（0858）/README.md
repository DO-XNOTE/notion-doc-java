我们很有可能在整个过程中就会去发现一些异常，对，而这些异常有可能会牵扯到一些不可用的状态，或者会引入协商故障。那么也意味着我们在整个发布过程中要持续关注这几个点，比如说我们的高危监控，比如说我们的业务监控，看看业务当前的指标有没有发生断崖式的下跌，同时关注机器上的一些监控，比如说有没有出现大量的异常日志，大量的这种 now pointer 或者 class cast 之类的这些日志，以及我们的中间件是否开始出现一些链接时被打爆了等等这种状态。


那一旦出现这些问题，都意味着我们进行灰度的相关代码是存在问题的，那么也意味着我们要赶紧进行灰问，但是这个过程也意味着我们不是只在某一个阶段进行监控，而是在我们每一个阶段都要去做好相关的监控，因为灰度发布中的每一个阶段都有可能导致大规模的异常，那也意味着不管是 5% 还是95%，我们都要时刻的关注这些监控，直到我们完成百分百发布并且没有发现异常，那整个过程我们才算是完成。


在发现异常之后，我们就要进行回滚了，回滚的目的是为了快速止血，所以我们要先回滚再排查，先把血止了再说，然后再去查是不是存在着相关问题。而在这个阶段，如果我们先排查，可能排查所花费的时间就把宝贵的时间浪费了，线上的损失会进一步的扩大，整体的影响力得不到控制，所以在止血完成之后，我们再去进行相关原因的定位，并且在定位出来的结果在下一个迭代中把它修复。

那也意味着我们要对我们所关注的这些异常监控项要格外的上心。比如说高危预警，通常高危预警它都是我们基于某一个场景去梳理的，比如说我们的下单，它整体的一个指标其实是我们只有一个预警，那一旦加微加单只成功率跌到了 50% 以下，那就已经绝对说明下单这块儿出了大问题。所以某些我们认为不可接受的场景去进行梳理，我们就能够理出了一系列的高危预警，并且将这些高危预警转化为我们认为可以快速止血、快速恢复的预案。


同时围绕着这些场景，我们要做好日志的埋点，我们要看看线上有哪些地方，我们要打好相关的日志，以及有一些定向的监控，比如说我们就针对下单的这一个服务做一下监控，来监控其实成功率的情况。还有就是我们本身将这些结合起来共同挂正成一个故障的预警，我们可以提前设定好，如果我们这个跌到多少，或者我们的问题出现多少，这时候它就可能会演化成一个故障，在这种状态下就可以前置的去发出相关的报警，下单成功率跌破了30%。是否是一个故障，那这里就需要我们结合着业务的情况，结合着整体的场景，然后将高危预警转化为故障预警，并且在整个定向监控这一个点上，我们要监控好整体的服务的，比如说异常率、异常次数。针对于低频的这种服务，二者要结合起来去看，如果你这服务只有 10 q b s 出现两个，那它就已经跌破了 20% 了。但是其实异常次数可能只是一个抖动，但是如果你的服务每秒都是上千甚至上万，那我们去通过异常次数去评估，反而它会陷入到一个不那么合理的一个境地。所以我们要将异常率大于多少，结合的异常次数大于多少二者并行去看，才能验证出一个明确的一个错误。


并且在二者同时出现的情况下，我们还要去看它的持续时间，最终计算出一个影响用户数，并且算出我们的资损的金额，这些都是我们需要去定向监控的指标，也是这些帮助我们在场景化的高危预警中，明确出我们的故障应该是什么样子，我们怎么样子界定出这个故障预警？

比如说一级的这个异常就是灰度中可能会导致故障的重点异常，高位异常，如果出现较多，那我们就需要马上回滚。比如说像 now pot 这种，其实它就属于一级异常，因为一旦出现 now pointer，就说明它出现了我们未预期的这种空指针，这种空指针通常都会直接阻断我们这里的功能，从而导致用户的体验受到较大的影响，所以这类是我们需要重点关注的一级异常。


而针对比这一级异常要弱一些的二类异常，那它就是灰粒度中可能会出现一些体现系统不太健康的异常，或者偶现的奇怪异常。这些异常可能会偶尔爆一下，比如说爆出了一个 spring 容器中的内存回收的一些问题或者之类的，但这些它又不足以导致整体服务出现问题，用户可能还能正常使用，但是我们需要持续关注并且关注其统计的一个值，一旦它出现大规模出现的时候，那说明系统可能马上就要崩溃。


虽然说用户的词还没有感知到，但是它已经是一个出现故障的前兆了，那这时候我们就需要马上硅谷，同时我们还有三级异常，三级异常是指灰度中出现的常规的业务异常或者其他不会恶化的异常，那在这种情况下，本身因为其不会影响业务或者影响的非常的少，那在这种状态下我们可能只需要抽样去看一看，来看一下异常对战它的异常是在什么情况要爆出来可能就只是一个用户的偶刷，或者是因为恶意的请求的一个打过来。但有可能那在这种状态下就根据我们抽样的结果来决定是否对我们产生巨大的影响。

我们也要在回滚之后去进行明确的问题定位，在问题定位之后也要做问题的修复，并且围绕着整个过程做一下问题的复盘。那么在修复之后也意味着我们要在下一次的灰度的过程中，重新把本次修复的内容进行定向的灰度，去验证我们是不是确认完整的修复了这个问题，如果没有问题，那我们就进行全量的发布，所以在整个过程中会发现回滚也只是我们问题恢复的第一步。在回滚之后，我们仍然有很多事情要做，并且我们最终的目标仍然没有变化，我们的目标仍然是全量发布，所以在回滚之后，我们需要去问题定位，问题修复。问题复盘，目的是为了让我们这次回滚中发现的问题得以充分的解决，并且再通过下一次的灰度进行问题的验证，最终完成全量的发布。
