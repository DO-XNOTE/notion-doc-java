那发布和代码本身在层次，在那个研发的过程中，它俩是一个连续的，我们代码写完了，我们后面会进入到相关的发布的一个环节。所以发布的一个跟踪度量其实也是围绕着以人为核心，重点去考虑如何去度量整体的一个发布过程，去评定发布质量，从而让我们的发布过程可以变得更好。这样。


那针对发布度量来说，我们肯定首先要有几个指标，比如说我们的发布次数，发布的成功率和回滚次数，这是最直接的，我们一周要发多少次？这些发布都成功了，没有几次是回滚的。为什么是回滚？那这个就是我们最基本的一个度量。第二呢就是办公灰度和紧急发布相关灰度的接入率，灰度的回滚次数，灰度是在发布之前我们想要用小流量来去吃错的，那灰度的接入率也意味着我们有多少能够具备回滚能力，而灰度回滚次数是有多少，我们在灰度期间就发现了问题，并且完成回滚了，通过这样的方式能够帮我们快速发现代码中的问题。


而紧急发布则和灰度则是一个相对称的，因为灰度是我们想用小流量逐步的是去试探，其实它是降低发布的速度，但是紧急发布通常是为了解决线上已经存在的紧急的这些问题，然后我们要赶紧去发布，那这也意味着线上的确出现了这种紧急情况，我们需要去考虑快速的修复它，而不再去考虑更多的就是我们减缓发布过程来去降低发布风险，而采用的是快速的发布来解决线上已经存在的问题。


针对非发布变更次数，比如说我们的线上配置变更，比如说我们的 DB 变更，这些是不是一个发布的过程，但它的确对线上产生了影响。考虑这些的目的也是为了能够更好的来判定我们线上的这个稳定性与质量，因为发布质量本身是就是影响这些的，通过这些指标我们能够发现哪些是我们研发质量不足而直接导致的这种异常流程，通过这些能够倒过来推动我们去做好我们在整个研发过程中的管理，因为发布它的成功情况其实是一个很明确的一个在发布前的准备工作和针对发布我们人员的一个态度的一个反馈。

而准入通常来说它都不是一个单点，它更多的像是一个链条，它会由各种各样的节点来串联起来，最终形成一个准入链条。而这个准入链条也可能不只有一条，它可能会有多条，多条的目的也是为了我们能够将各种各样的准入节点灵活性的组合，最终选择最适用于当前这种发展阶段的这样的一个准入门槛。


在这样的一个状态下来明确出这样的一个准入清单，并且通过这个准入清单来推进我们来完成整体应用的一个准入卡点，而我们通过相关的接入率、 block 数和跳过数来去反向去看我们这些准入在真正的发布过程中，它到底有没有被严格的执行，从而来明确出我们这道保障这道门槛有没有真正的发挥起作用。
