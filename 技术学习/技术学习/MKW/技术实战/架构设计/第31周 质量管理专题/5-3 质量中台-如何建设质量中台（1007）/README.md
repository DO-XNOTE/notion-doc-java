# 5-3 质量中台-如何建设质量中台（1007）

hello，慕课网的各位未来教师们，欢迎回来。在上一小节里我们为大家介绍了一下什么是中台，我们用了一个例子开始为大家讲解了一下，就是怎么从一两个独立链路演化成一个中台，他的思路是什么？为什么会要这么去做？后面也讲了之前的困境的思考，以及说我们怎么样去做一个中台，中台它能为我们带来的一个价值。


那到这一小节我们就中台这个点去聊一聊怎么样子去建设一个质量中台。首先我们还是要看一看常规的中台建设有哪两种思路，

第一种就是说我们基于我们的核心业务去升级成一个中台，第二种就是我们我们直接以一个中台为定位建设这个中台，这肯定是两种不同的思路，前一种就相当于是从一个叮复制出了一个，对吧？以它作为底座，上层在逐步的去支撑更多的业务。


第二个就是说我上来就是要把这个中台给建出来，所有的业务在改造成以这个中台之上去走。这两种肯定有不同的差异。比如说核心业务升级这块，它的思路就是我们把最主要的核心业务为出发点，然后去扩展核心业务，提升了它的抽象能力，增加多业务支撑的一个扩展能力，从而形成中台。也意味着我们将核心业务中提出的核心的东西提炼出来，抽象成一个流程或者一个链路，在增加对应的扩展能力和多业务的支撑能力，这样子的话它就形成中台了，因为它本身的这些能力还在，我们其他的可以针对扩展点对其中所做一些调整，做一些扩展，最终就是它慢慢的就下沉成了一个中台。


那第二个种就是我们设计的时候就纯粹仲裁定位，这时候就会全面的去考虑复用扩展、业务隔离，所有东西都在设计之初就考虑好了，所以这个时候它的诞生之时即为中台，那这二者到底应该选哪一种？这个就让我们后面一点点为大家揭晓。那首先我们看就是无论是哪一种方式，我们希望达成的目的，第一就是能力的抽象，第二就是业务的支撑，第三具备相关的扩展能力，第四就是隔离能力。这四个点其实很好去理解，我们只有做好能力抽象，它才能够逐步的内聚，形成一个个积木，这时候他才能更好地进行复用，不然的话大家都耦合在一起，你压根就抽不出来的。并且无论他要怎么抽象，无论怎么调整，你最终的目的还是为了更好的支撑好我们的业务，能够让我们的业务在这些做好的积木之上良好的运转，并且通过扩展能力将一些差异化的能力进行支撑，而差异化也要做好相关的隔离，不能 a 打到b， b 打到a，一旦打串了，整体来说就是一个灾难了。

我们先往后看，先来看一看，就是我们应该怎么样子选人去做这个事，那它也有两种选择，第一种是使用专门的团队去做，第二种使用业务团队去做。当然还有一种思路就是专业门团队加业务团队共同，那这就是所谓的共建。这种情况下其实专门团队它能带来什么呢？它能够去做很多通用化的建设，因为它跟业务本身来说不强相关，他们就是说我们专门的中台团队可以去做很多底层的通用化的这种轮子。而业务团队其实他要做好的业务支撑，因为专门的仲裁团队他肯定不能像业务团队那么理解业务，所以他们的通用化建设出来的这些积木到右腾的这边就需要做相关的调整扩展，更好的去支撑业务，跟业务百分百的做好契合，这样子的话它才能产生其价值。第二就是专门团队把差异化隔离给做好，而业务团队去做这种差异化的定制专门团队他们在架构上要去考虑好我 a 和 b 之间怎么样子彻底的隔离，避免 a b 互相影响。而业务团队就是 a 去做 a 的定制， b 去做 b 的定制，将其贴近业务的这些定制点给实现出来。


而专门团队在模型层面上也要去做好对应的模型抽象，将整体的链路流程能够抽象的很干净，而业务团队则要反击向而行，他们要在专门团队抽象出来的这种抽象模型之上去做好模型的计划，真正的与业务进行百分百契合，如果走到业务中去，并且能够更好的去支撑业务，所以总体来看就是重大的这个业务团队其实还是需要共建去中台的。


双方的侧重其实是不同的，关注点也不同的，但是目的其实是一致的，就是共同的以最小成本去支撑业务。所以这里我们就会发现，有的公司可能就专门用团队去做中台了，而有的同学就是想我就在用我团队去做中台，然后向其他团队去推，这二者都会存在的相关的一些困难。因为业务团队你很难去做好特别多的通用化建设，而专门团队又很难去贴着业务去做事情，所以只有二者进行更好的合作，二者本来进行共建，最终才能更好地去建设我们的中台。

首先第一点就是要去分析我们的现状，很多公司现在都有一些工具平台，还有各种各样的工具，比如说质量，中台可能有这样的各种测试工具，比如说单测的、自动化的、 code review 的等等，那这些工具它的现状是什么样子？其实是我们要在中台之前先去做好分析的。


第二就是各家业务的发展情况，我这个中台我预期要支撑多少个业务？三个还是 5 个还是 10 个？那这每一家业务当年的发展阶段是什么样子？是成熟期了还是刚刚开始？某一个阶段它所需要去做改造的成本、差异化的定制成本都不一样，所以当然是针对这些刚刚诞生的这些业务去做中台化建设推广肯定是最容易的，而对那种非常成熟的业务，你再让他花大精力、他人力去改，很多时候其实是比较难的了。


第三就是如果公司之前做过中台了，我们还要去看看历史中台是什么样一个状况，他是还活着还是已经死掉了？为什么死掉？当时遇到了什么问题？遇到什么困难？我们这次是不是可以解决这些问题，可以规避这些困难，最终能让我们农产能够顺利的建设下来。第二就是要分析我们当前各个业务系统他们的债务情况，包括代码质量，现在是不是有一些重复的轮子，这些轮子有多少？他们的差异有多大？他们可复用的东西有多少？第三就是针对不同业务的质量定制到底有多少，他们到底自己又在上面加了多少逻辑，加了多少代码，加了多少功能才能满足其业务诉求？那这些是否都需要沉淀中台中去，还是中台以什么样的方式去支撑这些质量定制？这都是我们的债务分析中需要明确，也需要去定下来到底我们哪些要先解决的，也是后解决的这样优先级。


当然不管是怎么样一个债务，我们最终还是要回到说我们到底怎么投人，各个团队能够投入多少的人手，投入多少的精力，投入多少的资源，这些都是我们从整个组织架构层面上需要去看的，从而当中确定出来我们在建设这个车的时候有多少可以投进去？如果真的是大家都捉襟见肘，那这个中台也只能是雷声大雨点小，喊的震天响，到最后落不下来。所以我们还是要回到怎么样去定位这个中台。

通常我们还是推荐以核心业务为内核去支撑类核心的业务。为什么说不从头去建一个中台？因为如果我们从头建一个中台，则意味着我们核心业务回头保证迁移的成本其实是很高的，如果核心业务都不切上来，你支撑了一堆零散的业务，其价值也不够大。所以以核心业务为内核去支撑内核心业务这个方法我们就可以牢牢的把我们核心业务绑在车上，而这时候也意味着其他人可以尽可能地贴近我们的核心业务，复用我们最多精力、最多人力去保障的这个核心业务，所以大家的质量和大家所需要的资源也能够得到进一步的保障。


第二它也能够最大程度上的复用核心能力，所以一些类核心它本身跟核心很像，如果完全从头建设，可能又投不出那么多资源，但以这样的方式我们就可以最大程度上去复用我们的核心能力，然后再去针对业务的差异化去做好相关的定制。第二就是我们还是要有一个独立的这样一个中台团队，它的实现虚线双线回报就是实现，就是你直接的有一种组织架构的汇报关系的，那需要有独立的中台团队，而虚线则是由业务团队，他们的人也要能够参与进来，结合着怎么样子将中台和业务更好更紧密的结合，最终就是执行路径了。


那就是以核心业务为出发点，深度建设平台。因为我们强调了我们的定位就是以核心业务为内核的，那所以我们的本身执行也是如此，以核心业务为出发点，然后进行差异化的梳理，我把其中可以复用的部分我们就成为中台，这样子的话，对于其他的业务我们就可以扩展的方式接入我们的核心业务。


如果没有差异化，那就是和业务去走，如果有差异的就会走到对应业务的扩展点，从而来完成其差异化的逻辑，并且我们的扩展点可以挂好对应的业务身份，这样子的话整体来说我们的逻辑就不会走串，最终能够保障出我们差异化的业务之间相对来说稳定隔离。到这里我们怎么样子去建设一个中台，基本上就清晰了。首先我们有两种方法，但是我们这里选择的还是以核心业务为内核去建设中台。其次我们要有专门的中台团队和业务团队，二者去共建，这样是我们的组织保障。第三就是我们的执行路径，就是要梳理好差异，将核心的业务抽象出我们的中台能力，差异化的业务去做好对应的扩展，最终完成相关的隔离，形成我们最终的中台。这就是这一小节内容。那么下一小节点呢？我们就为大家讨论一下哦。开始聊的那个矛盾就是我们真的一定要中台吗？好，各位未来加工师们，我们下一节课不见不散。
