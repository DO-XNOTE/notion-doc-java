# 6-3 质量管理体系-百度质量管理体系（0758）

hello，慕课网的各位未来加高师们，欢迎回来，在上一小节里我们为大家介绍了一下阿里巴巴的质量管理体系。阿里巴巴其实是一个更偏向于商业，偏向于业务这样的一个公司，那所以它整个的一个智能体系也更加的贴近它的业务，所有的内容还是以业务为主，然后以几个重点项目为牵引，再加上近年来于阿里云的一个蓬勃发展，而带来的公司的基础设施逐步的聚拢到阿里云而带来的整个基础设施的一个质量体系的一个提升。


那作为一个和阿里有着截然不同的定位以及有着比较大差异的百度来说，我们也要从它的业务体系去看一看。

它是一个独立部门的，它这个独立部门来支撑全部的一个质量业务，所以在这种组织架构下，它有什么优势和劣势？它的优势就是在于它是很容易做好跨产培训协同的，因为大家都是一个部门的，我做了一个东西，我旁边的一个同学就是测另一个部门的，我就跟他说，你看这个东西很好，大家一块协同去用，甚至说一块协同去做这个事儿就很容易的一种推动起来。


第二就是便于工具的传播，因为在这样的话，部门中有专门的工具组去做工具，并且因为是一个部门，所以大家很容易的进行协同讨论，然后工具的传播也会更加的容易，并且专业的路径有更加清晰。因为有这么一群人，整个的一个专业梯队是一个很鲜明的一个侵略，而它的劣势在于，第一，它的业务贴合度会比单独的要低，因为两边儿肯定会有两个大老板，那两个大老板也意味着其肯定在任何的一个管理上或者是信息传递上还是会有一点点偏离，哪怕这大老板之间达成了一致，到了下面仍然会有一些偏差。所以业务的贴合度肯定不如每个部门的子团队来支持当前部门要高。


第二，就是差异化的一个发展要小。为什么这么说？因为大家都在一个部门里，因为所有东西都拉通了，协同了，那整体来说，个性的东西就会变少，而大与统的东西会更多一些，当然人力的专注度也会变低。为什么这么说？因为如果我原来是一个部门的组团队，我只管我们部门就好了，但是现在我并不是，我可能今天要管这个部门，明天要管另一个部门，我整个的人力可能会被分散到其他的不同部门去。这个部门今天有测试需求，我就投这里，那个部门明天有测试需求，我就投那里。所以整个的一个人力收入度会具有一些下降，因为其人力的充分复用而带来的一个人力的分散所导致的。

第二就是在于一些泛质量的领域，比如说线上的业务监控，比如说质量的评测，众测、数据众包这些也是百度在这些走的很远，探索了很多的。比如像众测，这是个近 10 年的一个产品，并且这个产品已经非常成熟了。百度的很多产品的一些很细节的一些测试都是有众测的参与的。而数据众包其实就是更多的一些，通过大量的用户的一些数据反馈，最终让整个数据变得更加的精准。比如百度地图就是用那个数据众包帮他们提升各种 POI 的精准度。当然还有一块有估充效率，那包括像灰度发布、持续集成、持续交付等等，那这些也是百度去花了很多精力，那整体来看的话，就是在本身的软件执行和工程效率层面上，百度是走得比较远的。

然后再到泛质量领域的这样的一个不见不断演化的一个过程。然后其细节上也是从代码质量到服务质量进一步的去达到服务效果这样的一个测试过程。因为代码质量本身来说是最容易的，但到了服务其实它就有跟阿里的安居生产概念上是去契合的了，就不仅仅是在于代码市场本身，也要看它对用户的这种服务是否是满足一定的质量要求，然后最终看服务效果，因为你光本是服务本身没有问题，它并不代表最终效果好，而要从效果本身反推着代码和服务本身的提升。
