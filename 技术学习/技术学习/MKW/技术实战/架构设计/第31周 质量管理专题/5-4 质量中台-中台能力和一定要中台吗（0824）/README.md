# 5-4 质量中台-中台能力和一定要中台吗（0824）

hello，默克网可以为大家国师们欢迎回来，在上一场节里，我们为大家介绍一下常规的建设质量中台的一个思路，这里我们肯定要贴着核心业务去建，然后有专门的中台团队以及相关的业务团队进行共建，最终以核心业务为基准，通过梳理出与其他业务之间的差异点，将差异点通过扩展点的方式接入，从而将核心能力下沉成一个中台。这样子的方式可以让我们的中台和核心业务紧密的结合，作为更好地支撑公司的核心业务和内核心业务，从而更容易的能够去落地这个中台。


那么在这一小节里，我们要聊一聊的就是中台应该有哪些能力，

以及我们在这些能力之上去看一看，真的要建一个中台吗？首先看看中台的能力，中台其实有很几大类能力，比如第一类就是基础的支撑能力，这个能力也意味着我没有支撑能力，我的中台其实就是说空。第一点就是它是否能够支持核心业务，它的核心业务支撑能力是不是有的，这是它最基本的一个支撑能力。第二个就是抽象通用能力，针对我们的这些业务，它是不是有可以明确的抽象出来的一个通用能力？它有没有将我们的流失在我们的链路抽象成一个可以支撑类核心业务的这样的一个能力。 GV 只有抽象出来了，我们才能够在其中的某一几些点上增加扩展能力，否则的话大家都是耦合在一起，其实并不具备扩展性，那也意味着我们其他业务想要接进来也难度非常高，甚至说毫无可能。


第三个就是扩展能力的开发，我们是否具备一套架构、一套体系去支撑扩展能力的开发，并且扩展能力可以跟我们的中台非常的结合起来，包括扩展的管理，包括我们整体面向于整个流程上和扩展之间的这个关系的编排，那这些都是意味着我们只有建立好了这些能力，才能够支撑起我们最基础的一个中台。


第二叫做多业务的一个支撑能力，比如说多租户的一个能力，它是否能够支撑好我们内部的租户和外部的租户共同使用我们的平台？当然更多可能中台还是面向内部，那也意味着我们不同的业务，他们本身的业务身份是否能够在我们切片上做到差异化的管理，从而来建设出我们的多业务身份能力，这样子的话才能够基于这个业务身份能力去做好相关的隔离能力。


每一个业务本身也是需要保护的，我们不能说因为有了中台，你所有东西完全的有中台那一套，而是在于每一个业务过来，他要按根据其业务的特性，将中台的能力与其业务本身能力结合起来，形成既特有的链路，而这个链路跟我们本身的核心链路和其他业务甚至其他租户的这样的一个业务都是要隔离好的，避免之间的功能点互相的串，因为一旦串了就可能其行为是偏离预期的，那这样的话整个中台就失去意义了。


当然还有一些附加能力，比如说我们的流程编排，我们是不是说能够用一个低代码，这是无代码的一个流程编排的方式，帮助我们更好地将扩展点和我们本身的能力进行编排，最终形成一个不怎么需要研发人员，就能够编排出一个和原来核心业务很接近，但是又有差异的这样的一个业务链路。


其二就是一些垂直的一些中台，因为我们本身的核心业务可能会有很多的领域，但是有的时候一些垂直的业务领域，他们本身对能力的诉求其实是很聚焦的，就聚焦在某几个领域上。比如说我们一个商品中台，它就聚焦在商品的管理，创建商品所能不能提供出来的这种跨地域的展示、 CDN 加速等等，它是很聚焦的这一个领域，但是它就会有各种各样的面向于这个领域的一些细节的一个诉求，这也是我们的中台是否能够在其抽象出来的模型上，能够更容易地组建出这样的一个垂直中台，帮助一些更聚焦、更垂直的领域去完成其所需要的这样的一个能力。


这些就是我们作为一个中台最起码所具备的能力，如果这些能力不具备，整体的状态往下去走就很艰难，因为大量的能力都需要去改代码，就变成了原来业务系统写代码，现在变成了中台系统去写代码，其实它能带来的复用和价值也就变弱了。


所以这也就引出了我们很多时候所面临的僵局就是在于人力投入，是否是会出现各个团队与中台之间的一个矛盾，团队说我没有人，中台说我也没有人，这个时候这个中台怎么去建？其次就是边界的车皮到底谁来干？

怎么干？这一个功能到底是中台团队来干还是业务团队来干？这个边界到底该怎么分？这个蛋糕要怎么切？谁干多少活，谁拿多少力？这都是要明确出来的。然后还有一个很大的一个点，就是响应速度，因为仲裁说我要优先做这一块，而业务说，唉，我们这个特别着急，下个月就要上线，到底谁能先做呢？因为我们通常意味着对于响应速度都希望它能最快的去响应我的速度，但是不可能我几十个去几百个，需求谁都是 P0 的，谁都是 Dev 的，也意味着肯定有的要快，有的要慢，但怎么样子去平衡这个快与慢的关系？怎么样子平衡中台与业务之间的优先级关系，其实都是在整个状态建设中需要思考，也需要去明确的。


当然最终一点就是满意度的一个问题，到底业务侧满意吗？中台侧被认可，这个其实是我们考核一个中台，判断一个中台是否有没有价值最关键的一个点，所以意味着这一些点都容易把我们中台拖进到这样一个僵局中。人力的，那我们有没有保障？边界是不是清晰？响应速度是不是足够快？大家是不是都满意，这些都决定了我们的一个中台它是否真的能够活下来，这些都是我们中台面临的情绪。

合作的矛盾是不是日益突出？合作的效果怎么样？到底是好还是坏？我们是否面临着更多的投诉和抱怨？这些都是中台所需要面临的情绪，我们有没有能力去调和好这些情绪，有没有能力协调好这些矛盾，最终让我们整体的中台能够基本上的走，如果我们做不到，就会听到大量放弃中台的声音。


为什么要做中台？就是重复造轮子，我们原来在业务中写代码多好，还快 3 天的事，现在你们要折腾两个礼拜，这是这种声音听起来真的很难受的。还有一种就是自建能力的期望，为什么一定要用你们中台，我们自己建一个就好了，还能贴近我们业务，我们还快，我们还稳定，你们中台三天两头的升级，我还要跟着你们去折腾，你们夜里要发表我还得来这里跟你们耗着。其实这些都会有大量的听到这种声音，其中归根结底市场部还是项目的支撑达不到预期，这些就是我们到底要不要建中台，


我们就看一看这些将军这些抱怨的声音，我们真的能不能协调，能不能扛得住？在这中能不能推出更好的方案，给出更好的解决办法，去缓解各个矛盾，去划清楚边界，去投入更足的资源，去支撑好各个业务团队，让最终整体的一个中台能够真正为业务带来价值。


这时候中台的其存在性和必要性就毋庸置疑了，但是如果我们做不好，就意味着中台一定会面临着各种各样的阻力，面对各种各样的困难，而这些每一个困难都有可能把我们的中台给击垮，所以一定要中台码，这个答案是需要各个架构师自己去想明白。在我们的公司，在我们的团队，在我们的现状中，我能不能解决到这些问题，如果能大胆的去往前走，你的中台能够产生应有的价值，如果不能就要思考我要不要顶着去上我的中台能不能扛得住这么多的压力，
