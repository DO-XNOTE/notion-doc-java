# 3-7 灰度发布体系-发布总结与复盘（0706）

hello，慕课网跟未来架构师们，欢迎回来，在上一小节里，我们为大家讲解了一下关于一旦灰度发布中出现问题，我们应该怎么样子去监控，以及在什么样的场景下我们应该去回滚，并且在回滚之后我们又要做哪些事情，从而让我们本次灰度的目的能够得以达成。那么在这一小节里，我们将进入灰度发布体系的最后一节，也是针对整体的发布过程进行一次总结和复盘的过程，这个也就是我们这次为大家介绍的发布总结与复盘这些内容。那这些内容呢？我们希望大家能够将整个灰度发布过程中我们学习的这些内容都进行相关的总结，最终明确我们在做完一次恢复发布之后，我们有哪些需要去回顾？有哪些需要去复盘？


所以发布总结其实是我们对一次的灰度发布过程中的所有的细节进行回顾的这样一个过程。所以前提也意味着我们要能够将细节都能够采集下来，全面而准确地梳理灰度用户使用情况的信息。

因为我们灰度发布一定是针对了某一些被我们圈选的灰度的用户这个用户群体去进行灰度的，所以我们要采集他们，比如他们使用的时长、使用的异常的情况等等，来明确出他们使用的到底怎么样，然后针对这个过程进行相关的分析。这个过程就会包括两个方面，一方面是我们整体发布的过程，而另一方面是用户使用的过程，将这二者过程都结合起来进行相关的细节分析，看一看发布过程中有没有哪些点是我们做的好，做的不好？用户使用中他们有没有遇到什么问题？他们整体的使用是否流畅，升幅和预期，这些都是我们在过程分析中需要去看的。


在过程分析之后进入到我们最终的结果判定，看一看这次灰度发布，第一它是否达到了我们本身要全量发布这个的一个预期，第二，它是否达到了降低发布风险，减少 bug 带来的影响这样的一个目的，如果它做到了，那说明灰度发布本身产生价值。如果没有，那就意味着我们要去回顾，要去反思我们整个灰度发布过程来看一看是不是有哪些地方我们没有做得好，所以这也意味着我们要进行灰度的复盘。


复盘的复盘哪些方面？第一个方面就是过程，去看一看我们在整个灰度发布过程中，整个的灰度过程是否顺利，

有没有遇到流程性的问题，是否可以优化的空间。对于过程来说，通常我们都是按照常规的灰度流程去执行的，所以如果在过程中我们因为流程性的问题而产生出来的一些故障，那其实这一块是我们尤其在复盘中需要去关注的。这也意味着我们在下一次灰度的时候，我们是否可以规避掉这些流程性的问题，避免因为流程不够友好而为我们带来一些在灰度过程中的不便利性。


第二就是整个灰度的结果的一个复盘结果是否符合预期？如果符合当然是最好，但如果不符合记忆产生的偏离，但这个偏离的原因是什么呢？我们有没有什么办法去避免这个偏离？其实这也是我们在灰度过程中需要去关注的非常重要的一个点，就是每一个灰度发布迭代，就是我们从这一集到下一集过程中是否能够达成我们的预期的这个这样的一个效果，是否产生了相关偏离？每任何的偏离都意味着我们都需要去关注它真正产生的原因，从而帮助我们避免掉因为某些偏离实际上是一些故障的前兆，而忽略掉这些前兆可能意味着我们后续就会带来一些故障。


第三就是行为，这里的行为主要还是面向于后续的行为计划，谁在什么时间点要完成什么事儿，解决了什么问题？这也意味着我们在整个复盘过程中，我们看过去的那已经过去了，我们没有办法去改变它，但是未来是我们可以改变的，所以面向未来我们就需要去进行未来的规划，做好一个行为计划，这也是我们刚才所提到的这个整个行为复盘所需要去做的事情。


最后一个，那我们在整个灰度发布过程中，我们的收获又是什么呢？这些收获它是否能够转化为一些沉淀，复用于其他的项目或者后续的发布？因为每一次的发布我们都在积攒的相关经验，那这些经验都可能在未来的一些发布过程中，或者在其他项目的发布过程中帮助我们少走弯路，帮助我们更好地达成结果，那所以这也是在灰度复盘过程中我们需要去看，所以一个复盘不仅仅是去看我们那里做的不好，更重要的还是要看我们那里做的好，而这些好能够帮助我们在未来可以一直好下去。

快速的去定位问题与恢复我们的系统。这样的一个过程也意味着我们灰度发挥的目的也是为了在从零到百分百发布这样的一个过程里，怎么样子可以更好的来完成风险的控制以及损失的降低，并且能够快速获取用户的使用情况，帮助我们完善产品功能，提升产品质量，那所以这也是跟我们质量管理结合得非常紧密的原因，我们也希望可以通过这两类灰度，比如我们验证型灰度，它能够通过小流量、通过分级验证，通过快速的感知和回滚去帮助我们验证我们每次要上线的功能，其实正常的其实不存在问题的。
