# 5-2 质量中台-什么是中台（0949）

hello，鳄慕客网各位大家国师们，欢迎回来。上一小节点，我们开启了一个新的章节，就是质量中台这样的一个大章节。那么质量中台其实是一个近期比较争论的一个话题，有的公司说你一定要做质量中台，因为它可以帮助我们提高迭代速度，可以帮我们增加我们的复用度，降低我们的资源的消耗，人力的诉求，一方面它也能够帮助我们更快的完成创新，迈出我们的业务。


但利益也有一个话说，我们为什么要用中台，其实它很多时候是浪费人，浪费时间，浪费精力的，到了最后反而也支撑不了业务，出现了这样两种完全截断不同的声音。但是不管怎么样，我们还是要回归到我们公司自身，回到我们项目自身来看一看我们的项目，我们的公司是否适用于这个质量中台，所以还是要回来看看到底什么是中台，这就是我们这节的内容，

一套是卖服装的，它是我们一个公司a，假如说是 a 公司早期一直的一个主营业务，那在这个业务上我们就建立起了我们的交易系统的商品系统，我们的库存系统、订单系统和履约系统，它就牵扯到了从最早的用户下单到最后的发货，全整条的一个电商链路。


那么在这条链路之上我们已经跑得很成熟的时候，我们又开始去卖了我们的第二款业务卖电器，它本身也是有这样一整套的，从交易、商品、库存、订单到入履约的这样一整套形态。但是它其实跟服装是有差异的，比如说服装会有尺码，有这个颜色，但是电器很多时候可能没有，它可能是型号，那比如说服装它的履约其实通常时候就是寄出去就好了，走个快递，对吧？但是电器它很多时还有安装，它的售后也是不一样的，它本身的商品的一些情况也是存在着差异，但是其中又有一些东西其实是类似的。


所以这时候我们是不是在这样的独立链路之上就去考虑，说是我们可以建立一个中台，这个中台能够去支撑服装电商加电机电商两套电商系统，所以这时候就变成了服装和电商，他们都共同复用于整体的一个电商中台，而在这个电商之上会有对应的服装扩展和电气扩展，像交易、商品、库存、订单和履约都会有对应的二方面的扩展。


为什么要有扩展呢？因为其本身肯定会存在差异，哪怕我们作为技术沉到最底下的，他们的出用户字段可能也是不一样的。如果完全通过扩展字段，通过 JSON 去做二者的差异化，很多时候也意味着我们会难受，因为所有的东西，第一它不方便我们在 circle 中去做相关的处理，第二就是我们有些涉及到 JSON 中间的细节字段的，也不是很容易去处理。所以就意味着我们要有两个扩展去实现对应的差异的情况，然后再有一套公用的中台体系去支撑二者，但是同时我们还有一个业务身份去区分来的这个订单，这个请求到底是服装的还是电器的，通过这样的方式就能够帮我们识别出我们到底要经过哪个扩展来完成相关数据的一个补齐。这时候我们的电商中台概念其实就已经很清晰了。


让我们通过这样一个案例可以很直观的来了解一下我们的中台它是一个什么样的一个定位，其第一点就还是要复用，就是我们在这五个关键环节，看看哪些环节是最大程度能复用的，而哪些关键是我们有比较多的定制的，但是无论相似度有多高，肯定都会存在的一些差异，这些差异意味着我们要做好相关的差异化的一个扩展，并且二者之间你要有一个隔离。


不能说我服装的链路走着走到责定机构展去了，那肯定就出问题了嘛。比如说我们那服装走到了一个上门安装的履约，其实这个东西让用户看起来就觉得很夸张、很滑稽的一个逻辑了，所以我们还是要做好对应的隔离。但无论是复用还是差异还是隔离，我们最终的目的是为了提高我们的迭代效率、研发效率，能让我们更好的去支撑服装电商跟电气电商两套这种体系。所以了解了一个什么书台，我们还会要来反过来看一看就是什么是我们最常规的一个困境。


我们在平时工作中一直强调着，我们尽可能要避免去重复去照这个轮子，但是这个轮子有的时候它是避免不掉的，很有很多方面原因。

比喻说历史的原因，我先建设了服装电商，然后再去建电信电商，等我开始做电器的时候，我服装已经很成熟了，并且历史悠久了。在这种状态下，我其实并不希望在服装链路里去大改特改，我肯定想我全部新人一起一套，这样的话我新的就会很干净，而且我可以用一些最新的技术，我也不会影响老的这个业务。通常来说新的一摊子事儿可能是另一个团队在做，老的一个团队肯定不希望我新的团队过来咵咵把我老的改的面目全非。所以这里也意味着因为历史原因，选型侧重业务的原因等等，还是容易去重复建设基础设施的。第二个就是差异化的发展，我的服装电商可能与发展很成熟了，我改动的并不大，但是我的电器电商，我是我新起来的，开着有五花八门，千奇百怪的需求都提起来了，那这时候我为了匹配我对应的需求，可能就意味着我要在电信电商里做大量的调整、大量的修改。为了匹配这些需求，就可能导致这多个平台进行重复的建设，它又有差异化的一个发展。


而且横向的这个扩展也是我们要考虑的，因为你最早做服装电商的时候，你可能并没有考虑它未来会扩展到电器这边去，所以没有这个规划的你肯定是没有这个规划的，这是横向的协同，其实就很费劲，交叉的扩展能力也很弱，要么对老的做大改，要么在新的上去建设新的，一旦建设了新的轮子就建起来了。


当然还有一种就是差异化的业务逻辑，就会导致大量的分支串在里面，我有大量的 if else，一个代码 1000 多行，里面十来个 if else，这样的扛不住，那这种情况下一不小心就可能改出问题了。而在这种情况下也希望我们二者的逻辑独立开来，这就是我们很多情况下为什么会重复造轮子的原因，也是我们在思考我们的业务发展的时候常规的一些思考困境。

当然最早的肯定是从后台直接就触达到前台了，我们后台管了对应的DB，那前台就做对应的任务处理。


但是随着中台的概念的引出，就意味着它会在前台和后台之间产生了独立的一个环节，独立的一套系统，这套系统能够去支撑好公共的这种系统能力，提供可复用的这样的一个能力，这时候前台和后台就会依赖这些能力去分别做建设。但是我们也肯定希望我们这个中台不仅仅为一方前台，也为一方后台服务，我们肯定是希望中台是最大的层层能复用的，所以它就会去支撑好多个前台，从此支持好多个后台，最大程度上去复用我们的中台能，也意味着我们前面的聊的那些困境就是说我们怎么样做一些协调？服装电商和电竞电商这二者前台能力和二者本身具备的后台能力之间的矛盾，怎么样子通过一个中台最大可能的去复用。
