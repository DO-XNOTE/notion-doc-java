# 1-5 度量体系-代码跟踪（0732）

hello，木克网的各位未来架构师们，欢迎回来，在上一小节里，我们为大家重点讲解了关于缺陷的管理。缺陷就是bug，每个人在写代码当中都会经常性的遇到它，那我们怎么样子去评定我们的代码中之间缺陷它的数量，从而反映出我们代码的质量？因为本身代码质量你说要用风格之类的可以是吧？但是最直接的就是缺陷，你有多少bug，那就说明你的代码质量当前的一个情况。


当然我们也目的是说在提升代码质量，降低测试投入这两条线上都要去两手抓两组，并且两手都要，从而来提升我们的质量的同时还降低我们的资源投入。在这一小节里，我们将继续延续我们质量管理中关于为度量管理延展出来的相关的内容，那我们这一节讲的主要是代码跟踪。什么是代码跟踪？那我们先来看看就是质量管理它本身包含的几个面。首先就是质量管理，包括代码的一个质量管理，

这是跟我们代码相关的。第二就是研发效能的管理，那包括我们之前的需求，然后 bug 其实就算是研发效能的管理。然后第三就是人员的成长，那在这些里面都是我们需要去非常关注的。


那代码跟踪其实就是将代码质量和人员成长结合起来，代码的质量可以通过 bug 去反馈，但是代码的跟踪是说在这个代码随着不断的人在不断的提交，不断的处理，然后围绕着代码本身之外，我们再考虑代码所牵扯到的所有的相关人观测进来，最终形成就是代码跟踪。


而这块儿的话，我们最强调的第一个是人，第二就是代码，第三就是问题。那这就是我们整体的一个三要素，人写了代码产生出了一些问题，然后再由人去修复可以产生的新的代码。这样整体的一个闭环就是我们对于代码跟追踪这样的一个概念的一个定义。而关注人也是我们提高代码质量中最关键的一点，也是整个你不良体系中最关注的，毕竟管理我们管的不完全是那些事儿，我们管的更多的还是人，我们希望更多的还是人员的成长。

通过这些来评定出我们的一个代码的质量，当然还包括我们之前讲的我们的缺陷的数量，我们的缺陷密度等等，这些都可以体现出我们的质量。而在代码质量过程中，我们细的去看，就是包括我们代码规范、代码复杂度、测试覆盖率、 CRR 率这些是我们通过客观层面上围绕着 bug 之外，围绕代码本身的一个质量评定的一个标准。代码规范，我们是否符合常规的规范，是否会有一些容易出问题的这种代码在里面。代码复杂度过高意味着维护性就越差，那测试的覆盖度，我测试覆盖的越多，那也意味着这部分代码的可信赖度就更高。而 CR 率则是通过人作为最后一道防线来看一遍代码，来确保这个代码本身的质量，那这些都可以帮助我们严控代码质量，确保代码的稳定。


同时我们可以对代码做静态的扫描，而这些静态扫描包括规范，你看上面提到的规范还有性能，因为有一些是可以通过扫描扫出来，我们提前发现这里是一个慢代码，那对于这块我们就可以提前置性的做修复。还有就是潜在的异常，比如说我们定义了一个对象，然后去调对一个方法，但这个对象如果它有可能在某个分支内没有被初始化，那它就有可能导致 now counter，或者这个对象是由上一个方法返回的，上个方法在某一个代码分支中可能会返回null，那也意味着在针对这个对象调用方法的时候可能会出现 null point，这都是我们在代码扫描中可以提前帮我们扫描出来的。


第二就是 code review，再加上就是风险扫描，风险扫描比如说像漏洞，比如说像低版本的扫波动，这个就是安全同学会帮我们配以这个规则，它可以帮助我们发现我们在代码过程中的一些漏洞，而这些漏洞可能会为我们带来比较大的风险。低版本的扫描它可以帮助我们一是识别出那些有漏洞的版本，第二就是识别出那些基本已经没人维护或者是已过老的版本，及早的升级到最新的版本，而避免产生相关的问题。


最后还有关于 SQL 的风险扫描，我们在整个代码扫描中，我们也能够识别出我们对应的这个SQL，第一是否有可能有所有注入的风险，第二是否是慢SQL，通过这样的扫描可以帮助我们在提前去对其进行修复，防范于未然。

这些就意味着我们的工作范围会有差异，我们生产角色也有差异，有没有研发，有测试，有运维、有做中后台的，有做前端的等等。每个人的智能都有差异，但是最终的这些都决定了什么呢？决定的我们的代码产出是有差异的，我们不能说把一个 q a 同学和一个研发同学放在一起去评定他们的代码，我也不能把一个越位同学和一个前置同学放在一起去评定他们的代码，那这些也都意味着我们所有的这种代码度量都是一个参考，我们不能拿他说，哎，你代码写 1000 行，他代码写 3000 行，他就比你牛逼，他就比一好。不能这么去看所有的东西，不能一刀切。


更多的还是说他在他的工作范围内，他的生产角色，然后他的工种角色职级都影响了他，最终的代码可能会和你有差别，那这时候根据其这种差异性，然后来更多的去评定人量和质量本身的关系，最终还确定我们在整个质量管理过程中应该怎么样子去把握好其中的度，去把握好让每一个人都能够做得更好，让每一个人都能有成长，同时还要考虑其本身的发展路线与成长空间。


好到这里这一小节就结束了，代码追踪本身就是一个需要长时间关注，并且与人结合非常紧密的一个事情，管理好人，管理好我们的代码，才可能让我们的整体的研发工作更加的顺畅有效。好，在下一小节中，我们将从代码层面进前进一步来聊一聊关于发布跟踪，在发布这个过程中我们现场的那些人，那些事儿。好，各位大家伙士们，我们下一节课不见不散。
