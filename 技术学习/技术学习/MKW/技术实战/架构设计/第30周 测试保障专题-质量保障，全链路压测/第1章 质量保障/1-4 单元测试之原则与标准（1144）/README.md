# 1-4 单元测试之原则与标准（1144）

hello，慕课网的各位未来加工师们，欢迎回来。上一小节里我们为大家介绍了测试驱动开发，通过先写测试再写代码这样的一个方式，能够更好地帮助我们管理我们的代码质量。在写代码的同时，我们就已经能够完成了我们单元测试的覆盖。但是也因为各方面的原因，国内其实使用 TTT 的并不多，当然有一些误区，当然也有一些历史上的一些原因，一些卡点，但是这些都可以通过一些方式去克服，包括在未来我们的架构中，我们是否也可以尽可能地使用测试驱动开发去开发我们的核心模块，确保我们的核心可以被其覆盖住。


那么在后面的几节，我们都将围绕单元测试做一下展开来的讲解，在这一小节里我们重点大家讲解的是单元测试的原则与标准。大家都知道单元测试是什么，那单元测试有什么好处？它需要按照什么样的原则去落地？首先就让我们来看一下单元测试它的好处都有什么。第一它是一个自我 review 的一个过程，因为通过单元测试，我们能够反过来去看我们的代码设计是否合理，我们的代码实现是否正确，整个一个过程是一个自我 review 的一个过程，我们通过 case 反向来看我们代码本身需要实现的各个功能是否能够正确的执行。

其次是一个修改的保障，因为我们是在一个不断的持续的迭代过程中，也意味着我们对于代码也是在持续的进行修改中，那通过有单元测试的一个保障，它可以帮助我们避免产生扩散性的影响。什么是扩散的影响？就是你改了 a 类，结果 BCD 都被影响了，因为 BCD 被影响了，那 EFG 也被影响了，这样的话影响就随着这个调用关系，随着这个依赖一步一步的扩散，那通过这个单元测试，它就可以保证我们的影响范围是可见的。我们知道哪些被影响了，哪些没有被影响，所以它也能够帮助我们避免一些未知的功能影响。我们都以为我们只改了a，那对于 z 这么远的应该不会有影响，但实际上它受到了影响，那单元测试就可以帮助我们发现这些我们本来没有预期的未知的过度影响。


另外单元测试还可以作为一个传承沉淀这样的效果。比如说我们新拿到了一个代码，我们可以通过看单元测试的代码去理解这个代码它的面向的什么样的场景，实现了什么样的一个功能，它可以作为我们看代码的一个功能考验。其次，我们在学习使用这些框架或者使用一些架构的时候，使用其他人的产品的时候，我们可以通过看它们的单元测试来理解我们要怎么样子去使用它们，所以它也是我们去学习使用的一个方法。考因通过这些它就可以帮助我们更好地来完成相关的传承和沉淀，因为我们可以把一些从代码中比较难看出来的东西在单位测试通过 case 的覆盖让大家知道，哎，我都考虑到哪些边界，我考虑到了哪些特殊场景，我的测试数据应该是什么样，以及在整体上我们额外关注的场景都有哪些，我们曾经遇到的过问题都有哪些？都可以通过单元测试来沉淀下来，那说完它的好处，那我们就正式的来看一看单元测试都有哪些原则。

a 就是指自动化，它是指我们的 case 需要能自动化的运行。第二是指独立性，是指单元测试的 case 之间不能有相关的依赖。第三是可重复执行，也意味着我们的每一个 case 都可以重复的被执行，那它也对我们的 case 的书写提出了更高的要求，比如像一些插入，删除，那这一类要如何重复的去执行呢？那这就需要我们在单元测试的书写过程中去考虑到这些范畴。


第二个原则是 fast 原则， fast 原则是由五个字母组成，那它也意味着对应着五个条目。第一个条目 fast 意味着我们的 case 要既可能快。第二个，隔离 isolate 也意味着我们 case 之间要具备隔离的效果，而不能 case 之间有相关的依赖。其实这个也对应了 air 的你其中的独立性原则。


第三个，可重复，这个就跟 air 的可重复事情其实意思是一样的。第四个，自我验证，那它也是指我们在 case 执行的过程中，它能够自行地完成其正确与否的验证。最后一个是及时，也意味着 case 的执行结果要很及时的能够反馈到最终的一个结果中，以及 case 中出现的问题也要能够及时的反馈出来。那通过这两大原则就可以帮助我们更好地去理解单元测试，去写单元测试的case，那我们就展开来的去看一看，比如说像 air 这个原则。

二热原则，第一个就是自动化，自动化是什么呢？它是我们的 case 是需要全自动执行，非交互式的。什么是交互式？就是你点一个，它执行一个，那这是交互式的一个执行，所以意味着我们单元测试要尽可能地全自动地执行，我们只要启动了它，那所有的 case 都由程序自己来执行。这样子的话，它可以帮助我们最大程度上避免需要交互式的不断地去触发，而引入了大量的工作量，就是独立性单元测试用例之间不能相互依赖，也不能有相互的顺序。


这个点其实是指我们的 case 它较尽可能能够把单独去跑，也有可能是被自动化的去跑，而且因为其不相互依赖，也没有先后顺序，则意味着我们的 case 不管在什么情况下，以什么样的顺序去执行，它都能够完成其要完成的这样的一个使命，它都能验证其对应的这个生产代码是否是正常的。这样也就帮助我们降低了 case 的管理成本，因为 case 之间一旦有依赖，就意味着它的依赖需要先执行，然后它才能执行。而先后顺序也意味着顺序之间不能被打乱，那也意味着我们要需要额外的精力去管理我们的执行顺序。所以这独立性可以帮助我们尽可能降低 case 的执行成本，那它也就更容易地去自动化了。


第三个可重复执行单元测试必须是可重复执行的，不能受外界环境的影响。我们一个 case 可以重复执行，则意味着我们不管它执行一遍还是执行 100 遍，其效果是一样的，其最终产生的数据也是一样的。它就是就对我们对 case 的书写是有要求，但同时它也可以降低执行的成本，因为其可重复执行则意味着我们在任何时刻去触发它的时候都不会产生副作用，这个就是口重复执行带来的最主要的一个价值。

第二个，隔离性测试用例之间是隔离的，一个测试不会依赖另一个测试，那这个其实跟独立性是有相关的，我就不去过多的去讲解了。也是希望 case 之间相互隔离，意味着其相互影响会更少，其执行的管理会更成本会更低。可重复其和可重复执行是一样的，重复执行组织能得到相同的结果，多次执行不会有任何的副作用。
好自我验证可以验证其测试的结果全部通过时给出简单的 OK 报告，当他们失败时描述简明的细节，这样子的话我们就能够很快速的明白我们到底是执行成功了还是执行失败了。所以自我验证性也可以帮助我们从我们需要去一个 case 去看你执行结果的这样的一个工作中解脱出来，我们可以直接从最终的报告中去看看我们是否全部执行成功了，有多少个 case 失败了？他们失败的细节是什么？通过这样的方式可以更快地完成 case 的一个检查。


当然最后一个是及时的测试是及时的，程序员在代码上线之前及时的编写它们，以防止bug。那这里提到的是编写，当然也要说明 case 的执行也要是及时的，我们要尽可能快地、尽可能及时地完成 case 的编写与执行，所以在上线之前可以帮助我们验证我们的代码去治疗。

单元测试其实有三大标准，第一大是可靠性，第二是可读性，第三是可维护性。单元测试其实是代码和生成代码一样，也要保证其整洁，那也要保证其可靠可读。我们代码也不能说我们单元测试代码就可以有bug，单元测试代码也不能有bug，并且其也要能够满足相关的标法规范，以及满足相关的代码整洁，使其具备较好的可读性与可维护性。然后每一个测试一个断言，尽可能地降低断言的数量。断线越多则意味着我们需要关注的复杂的这样的一个断线场景就越多，我们需要投入的精力就越多。并且在单测中，单词的名字就像注释需要有完整的信息，这样子的话，可以在看整个单词列表的时候，我光看名字我就知道这是个什么case，它带来的什么样的结果。


当然，我们也不要过于执着地去追求太高的测试覆盖率，因为测试覆盖率你想要把它提高很高的一个程度，其实它的工作量也是很高的。所以选择一个能够保证你代码质量，同时又让你能够具备高效的研发效率的这样的一个覆盖率是最合理的一个覆盖率。到这里我们这一小节关于单元测试的优点，其原则和其标准都为大家介绍完了。单元测试本身也是需要有精心维护，精心去编写，它也要能够体现出一个代码其实本身的质量，所以我们不能说为了用单元测试去兜我们生产代码的质量，那单元测试本身的质量就忽略了，所以这一点大家也是要牢记在心。
