尽可能的去将场景覆盖全是开发自测过程里需要非常关键的一环，因为你一旦漏了，就有可能有相关的bug，是非常低级，非常粗浅，并且它到了 Q a 那边，那在场景里面我们要考虑第一个就功能，我们这次新增加的这些功能是否都自测到位了。


第二是边界，对于一些很常见的边界，我们不要求开发自测把所有的边界全部都抠细，但是我们对于常见的边界是要能够完成相关的自测的。第三性能，这里的性能叫做基本上的一个自测，我们首先要保证它不至于太慢，不至于每次使用都超时，基本上处于可用的状态，这是我们一个最起码的一个保障。还有最后一个就是异常，就是我们该不出异常的，不能出异常，该明确返回异常的这个异常要明确出来，这就是我们在研发自测考虑到的全部场景。


当然我们还要延展下来看，我们也要考虑到我们的数据的一个制造。我们测试需要哪些基础数据，这些基础数包括维度数据，包括事实数据，它是一些最基础的保障，我们能够完成相关测试的一些数据。其次就是我们的测试接口所需要的一些测试的一些数据，比如说一些样例数据，那这些也是我们需要在开发测试的时候准备好，并且提供给 QA 的。


二就是我们开发自测的这样的一个过程，我们要把它记录下来，到底哪些是我们测好的，哪些是我们测出问题的，哪些是遗留的，哪些是后面再看的，这个过程我们要简要地记录下来。我们也许没有说像 Q1 的同学一样有非常完善的工具去记录它，但是我们至少对这个过程大概心里要有个底，并且要能够有一个对应的结果。
我们知道我们的测试哪些是 OK 的，哪些是不 OK 的，我们告诉 QQ 同学你可以测了，不 OK 的我们还要再看一眼，针对其中要修复的问题要及时的修复，不能说哎，我拖等到给了之后我后面再去修，真的要为了进度要有这种场景下一定要给 Q1 说清楚我残留了什么问题，这些问题我会在什么时间点以什么样的方案去修掉它。在测试的过程中先注意这个点是先跳过还是用什么样的方法先规避掉这个问题，在后续我修复完之后，你再来去做相关的一些事情。


第二就是我们所有的这些东西都要在提测的时交付出来，告诉 Q a 我们都干了什么事儿，我们验着什么样的点，你 Q a 就可以根据这些情况来设计他们要做的这个工作，这样子的话双方在这个结合点上就可以配合得非常密切了。自测也不是一个漫无目的的一个行为，

它也需要预先的去计划与设计，包括我们在估排期的时候，我们也要估出来一个自测的一个时间段，我们要决定我们什么时候能开发版，有什么时候进行自测，什么时候进行联调，什么时候开始提测，这块儿都是我们要明确好的一个时间。我们不说自测不给自己留时间，自测是要给自己留出好充分的时间，甚至要留出一点儿 buffer 的。


所以我们要制作一个很明确的一个自测计划，其中包括我们自测的一个范围，我们自测要考虑到哪些case，我们自测的使用的时间在什么时间点，要花多少时间去进行自测，然后还有我们的自测顺序，我们先测什么再测什么，这是一个测试计划，在这个计划中提前花一点时间去整理出来，可以有助于我们完成我们的测试，没有相关的疏漏，并且可以更好地管理好这个进度。确保我们可以正常的交付。然后在自测中我们要考虑到我们的联调功能，以及说非联调的我们内部的一模块内部的一些自我功能，像这些都要自测考虑得到。然后针对测试数据要进行做好相关的一个准备，包括基础的维度和实时数据，也包括我们测试的这些 case 数据，用例数据，这些都是我们在自测的过程中需要去考虑好的。


第二要去 mock 好对应的异常，比如说我们要针对联调过程中其他模块儿返回的异常做相关的定制的，那这类的异常，因为你在自测有的时候可能对方还没有就位，那也意味着你要对这个异常做好相关的mock，从而能够让你能够完成后续的测试，从而先去验证好我们的代码是否OK，最终再和大家一块去联调。

并且我们要提前向 QH 要到我们的测试case，他们会针对整体的功能做好相关的 case 设计。这里他们会考虑很全面，比我们考虑的很更全面。针对边界，作为新人场景，作为分支，他们都会去理得很干净，理得很清晰。然后我们可以结合他们的测试 case 去设计我们的自测case，这样子的话我们就可以更好地完成相关的覆盖性的分析。当然因为他们的 case 很全，所以我们如果完全按照他的 case 去执行一遍，可能所花费的时间也会很多，所以我们可以针对其做好相关的演支，从而将注意力集中在最核心的点上，然后也尽可能的去做好相关的覆盖。


第二就是自测的数据要尽可能的符合真实场景，一场测试场景也要覆盖。你的数据和真实场景是尽量是一样的。不要是说你正式场景都是一个大整数，你测试的都是小整数，那这时候到了线上的时候可能就会出现问题。比如说你线上的整数都是 8 V7 的，你测试的是123456，那到一段线上一看，你8位报错了，截断了，转类型转换出错了等等，你在 123456 时候根本就测不出来，所以这里是要考虑的一个点。


第三就是在自测中发现的问题，你要马上去修复并且验证，不要攒一堆修，因为你都自测了，我们自己的写的代码自己去测试你测出来的一个问题，你马上把它修了就好了，你要是攒一波儿的话，就有可能你当时的脑子很清晰，这些问题你到攒了后面再去看的时候忘掉了，修的手就可能出现遗漏，甚至说反反复复的修了发现还存在问题。所以发现问题马上修，趁脑子里所有的东西都还很新鲜的时候把它修掉，这样的话也可以最好的来完成相关的修复。

你不是说我们自测把每个点测完了就完了，我们还是要走一遍我们这次功能的这些功能，确保说我们开始自测改了一部分，那后面测出一些bug，又改了一部分，结果把前面的改坏了。


你不知道这个是在最后回归的时候起到的一个作用，自测也要按照我们最早开始盘的那个计划去完成，因为你只要按这个计划去走，你才能够确保你后面可以准时的去提测。当然自测也可以通过一些度量，通过一些数据错放之区自驱动这个自测迭代。当然这个点不是必须的，因为针对这个点你若再单独去建一条度量体系去做的话，其实成本也是挺高的，但是因为都是我们自己去写的代码字宇去测的，有些数据可以自己的去简单的去维护一下，哪怕在纸上写几个数，你也能够很好地去管理自己，将这些过程更好地管理起来。自我的不断地迭代，也可以帮助我们更好地去做好这个事儿。


那所以作为研发自测来说，它本身是一个针对研发在完成代码之后，马上就跟进的一个阶段，这个阶段能够是我们在交付之前给予 Q a 的一个承诺，我交付给你的东西基本上没有问题， Q a 再去更细节地去研究，去测试，而不是说我们完全写完代码直接申请测试。我们都不自测，那测试就会发现一对一的问题，然后再答过来，我们再去改，改完了他再测，又发现了一堆问题再打回来。


这个反反复复的过程其实是很花时间的，因为包括沟通的时间，包括反复的流程的时间，包括我们代码不停地修改的时间，这个时间浪费掉真的是太可惜了。好，那到这里我们这小节内容就结束了，在下一小节里，我们就继续往前走。一个我们的测试阶段，从研发自测之后，我们肯定是要提测了，提测给 Q1 的同学之后， Q1 同学就要进行相关的集中测试，他们会针对所有的代码，所有的功能完成一轮甚至多轮的集中测试，最终验证测试的代码没有问题。好，各位架构师们，那就我们下一节课不见不散。
