回归测试是也是我们在整体测试过程中不可或缺的一环，所以并且通常在技术测试之后，我们就会介入到一个回归测试来验证我们所有的功能是不是OK，那么回归测试就是指代码改动之后，对于整体功能或者可能被影响到的功能进行验证，避免引入问题，其中重点有点标大了，对吧？就是代码改动之后，这个是在回归测试的切入的时间点，对于整体功能或者可能被影响到的功能进行验证，那这是回归的范围，避免引入问题，这个是回归的目的。



所以我们就是在什么时间点代码改动之后，对什么东西回归可能微想的功能或者整体功能，这个是要根据我们回归的这个场景的一个剪枝裁切之后来确定我们的回流的范围，目的就是确保这些公地方没有问题。为什么要进行线上的回归测试？

因为有的时候我们发现往线下好好的发布就出了问题，这里就会意味着我们可能会有哪些会在线下线上产生差异，比如说配置，线上的开关没有开启，又或者是我们发布完了没有去改这个配置，线下是有这样的配置，但是线上没有这个配置。然后第三就是相关方未改革发布，你跟别人一块儿合作的一个项目，你发布了他不知道，那他用起来有问题，你用起来也有问题，因为你的二者之间相互是互相依赖的，所以这里就会存在着一个配置相关的一些现场问题。第二就是数据可能会有脏数据没清洗，因为你在预发环境进行测试的时候产生了一些脏数据，这些脏数据透出到了线上，那它可能就会存在问题了。第二就是测试数据的透出，测数据也是脏数据中的一种，但是它是由于人为造出来的这些测试的数据，这些数据也是不能露出的。第三就是我们在上线的时候可能也会涉及到d、 b 的一些变更，那这些表结构是否修改了？现阶段表有没有创建起来？这些都是我们在上线的时候需要关注的。这些点除了配置数据之外，还有环境，比如说像用户版本的一些兼容问题，用户他在之前是个什么环境，那你发布了之后是否对它原有的环境有兼容？第二就是线上线下可能也会有环境差异，线下的机器可能不如线上好，线下的 DB 跟线上也有差别，那等等。包括你线下你可能改了一些基础性的一些内容，比如说你改了系统的版本，改了某一些基础的库的一些版本，到了线上你忘记了这相关的一些环境的一些变更，那可能就会导致上到线上发现出现了问题。还有一些就是你线下其实你不光是你只有你的内容，你还有其他人也在开发中的内容，它可能对你产生影响，所以你可能不知不觉依赖了别人开发中的内容，而你不知道，那在这时候你在线下完全 OK 的，到了线上之后它的功能还没有上线，那你这时候就除了相关的问题了。所以种种的原因都可能会导致我们线下好好的发布了就出现问题，这也意味着我们也要针对线上做好相关的回归。

第一就是可能被影响到功能，这个功能往往我们可能心里是知道的，我们会根据当前的这个它影响到的范围，比如说都有谁用了，依赖了我们，那这块儿就能知道我们可能联想的范围。就比如举一个例子，你是一个电商系统，你修改了商品模块儿，你是怎么都不可能影响到最终的那个线下发货的那个系统的。就你们俩之间本身就不太可能有交叉，因为它更关注的是物流单你关注的是商品本身，那二者之间其实存在着天然的隔离，它就可能不是被你被影响到的功能。
第二就是核心链路场景，即使它不在你评估的可能被影响的功能范围内，但是因为其它是一个核心的链路场景，它是我们最重要的，面向于客户的这样的一些场景和链路，我们也需要对其进行回归，确保发布了之后我们的主链路是 OK 的，是没有问题的，我们可以正常的使用，这是我们最主要验证的场景。


当然还有我们红线的场景，我们发布栏之后，我们要过一下红线，看看是否违反了某些业务红线，因为一旦违反红线，它给我们带来的可能都是故障和损失。 l 就是历史的一些故障场景，针对前期发现了一些故障，那我们针对一些修复的，我们看看这个上线的这部分代码是否在这个历史的过程场景中可以非常好的活下来，这也就是我们需要进行全链路回归的一些场景，那当然我们基于全链路回归还会有一些方法，比如说基于修改变的回归，我们到底改了什么东西？它的延展出去的一大片，那这些都是我们修改面的一些回归，那基于风险的回归，风险回归就是我们不管你改了什么，但我知道我当前系统存在的某些风险，我要针对这些风险做一轮回归，确保不管你怎么改，我不能在这些风险上栽跟头。


还有就是基于验证的回归，我是以有一个指向性目的的一个回归，我来验证你这发布了这个事之后，这个事儿是不是真的好了，那这就是基于验证的回归，那通过这几种思回归，我们用目的很清晰，就是希望保证我们的线上一定是好的，通过回归的方式就可以让我们从祈祷它线上好的状态变成我们确认它线上是好的，变成这样的一个状态。

它的第一大成本就是工作量巨大，它令你要把所有的东西都验证一遍。第二就是很枯燥又乏味，


因为每次回归干的都是同样的一些事情，反复地进行验证。那所以当你有一个大版本需要大规模回归的时候， QA 同学心中的第一想法肯定是，别让我打死他，他又给我找了一堆的活，我这一讲座全都要加班，这就是他们心中的一个非常明确的，而且有非常痛的一个点。


所以我们怎么样去解决这个问题？第一我们要把场景具化，尽可能的让场景能够聚焦到某一个点上，这也意味着我们在基于修改面的修改，这个时候我们怎么样子能基于修改面将场景进行具化，到底告诉他我到底改了什么？我影响的是哪个场景？然后在围绕着这个，围绕我们修改面做常见检制，把一些我们明确没有修改的，不会影响到的整个链路都不可能通的这些场景都给它剪支掉，这样子的话就可以变得很少了。


然后最好的还是通过自动化测试来完成自动化的回归。所以这里就想起当之前老师在给大家讲的，包括阿里的这种基于线上的这种DOM，它来采集线上的这些数据，然后在线下进行回放，这种方式进行测试其实就是一种非常好的自动化测试。当然我们也可以有很多方法，因为如果我们没有办法做到像阿里的 DOM 一样这么强大，但我们也可以，比如说我们采集它的输入和输出，然后去验证我们在这个输入和输出下是否是 OK 的，通过这样的方式来降低我们回归的工作量

我们回归到底是在线下好，还是在预发布好，还是在线上好？但是我想一想我们回归的背景是什么啊？


我们回归的一个前提肯定是我们的功能修改肯定基本上就完毕了，我们的 bug 已经收敛，已经修改完 b 了，那这个是我们两回归的两大前提，当然还有回归一个条件，就是环境稳定，且与线上基本一致。


那通过这 3 个点来看，基本上线下就可以被排除掉了，因为很有可能会有一些没有修改完的代码也混在里面，包括其代码的 bug 也不一定完全收敛了，因为多个人都在开发都会可能会有影响，并且它的环境与线上环境也存在着较大差异，所以它不太适合进行回归。而真正预发布和线上则可以进行相关的回归，因为预发布和线上本身是比较接近的一个环节，并且预发布它可以确保我们不会影响线上，我们可以很轻松的在未发布的时候完成所有功能的回归，然后在上到线上之后，再在线上进行一次线上回归，去验证我们所有的发布点，这样子的话我们就能够更好的完成相关的互关工作了。
