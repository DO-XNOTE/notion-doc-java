# 1-6 单元测试覆盖率评估（0934）

hello，慕课坊的各位未来架构师们，欢迎回来。在上一小节里，我们为大家介绍了三种常见的基于 Java 语言的测试框架， g unit， springunit 跟 test NG unit。其实这两位大神的一个作品，它虽然有了很多年的历史了，它的经久弥新到现在还有很多人在用它。当然更多人还是围绕着 spring 结合起来的一个 spring g unit 去使用的。这 unit 也有一些使用 test n g 来完成更复杂的一些场景的一些测试的设计。


不管用哪种工具，只要能够达成，我们去管理好我们的case，我们去运行好我们的case，都可以达成我们想要的结果。所以黑猫白猫只要抓大老鼠就是好猫。那在这一小节里，我们将去针对这个 case 的覆盖做小讨论。我们 case 覆盖它要到底覆盖到什么程度？我们先要来理解，就是我们到底要评估哪些指标，比如说这个指标就代码覆盖率旗下都有哪些细节的覆盖。比如说第一个点语句覆盖，它是针对了我们的所有的语句到底都有多少的覆盖，那其中通常也有两种更细的说法，一种叫做行覆盖，

一种叫做段覆盖。因为语句它其实是比方法更当一层的这样的一个覆盖率，那么行覆盖可以精确到我们最终代码行每一行它是否被覆盖了，那这堆段它就可以有一些代码片段，比如说像if、像 while 这种，其实是一个片段，它有大括号就框起来可以帮助我们更好地将几个代码促成一个段，然后去看这个段覆盖。它可以帮我们度量代码中每一个可执行语句是会被执行到了。


第二个叫做判定覆盖，它也叫分支覆盖，或者叫做边界覆盖，是指什么呢？就相当于我们有一些判断分支 if switch，像这一类分支就可以去通过这个覆盖率去看我们的 case 是否将其那细节场景都考虑到了。它可以度量程序中每一个 Candy 的分支是否都被测试到了。比如说你有一个很大的一个switch，那是否 switch 中每一个 case 都被测试到了？这这个就可以通过判定覆盖来计算出其覆盖的情况。然后第三个是条件覆盖，它可以度量每一个子表达式结果 true 和 false 是否被执行到了。这里就越来像是我有大量的if，那我们每一个 if 中的某一个 true 和false，我可能是if，然后里面是有或有括号有语，那这里就可以看每一个词表达式true、 false 是否都被测试到了。


最后一个是路径覆盖，它是录出来度量函数的每一个分支是否被覆盖到了。通过这四大类的覆盖就可以基本上把所有的代码、所有的场景都覆盖到。那么实际上在我们的工作中，我们其实还有很多的就覆盖率工具去进行评估的，

因为靠人去评估肯定不现实，你怎么知道我到底每一行到底有没有被覆盖到？所以通过覆盖率工具可以帮助我们更好地去度量我们的覆盖效果。


业界比较常见的就是这三类覆盖工具，这Coco， Emma 和 Coopera 这三类其实在业界都还有人用，但是现在用的最多的还是这个Jack， Coco 就是Jakako，它可以更好地去管理这个，并且最重要的一点是什么呢？大家可以看到这里它还在持续的更新中，而站在后面两种其实基本上已经停止维护了。


通过我们看着我们表格中的这几项就是jacoquotter，它们的原理其实大部分都是通过 ASM 去修改这个字节码，然后从而来完成对应的插桩。通过 AICM 这种在运行时注入和修改炸包，直接改 class 文件来说，肯定是在运行时注入更好，因为它对我们的原始文件的入侵就更小了。


第二就是在覆盖力度上，可以发现就是他们常规来说这种行类方法指令分支都是能够覆盖的。然后在 Emma 和后面这个Copiter，那可能还有一些其他的，比如说像那个基本块的一些覆盖，像一些分支的覆盖，那这一类其实在平时过程中也用到也不多，我们更多的还是参考着这个行覆盖率和分支覆盖率。然后在插桩那种方式它也有 on the fly 和 offline 两种。 on the fly 就是你在运行的过程中你也能把这个东西插进去，但是 offline 它就只能是在你离线的状态下去插入。性能上来看的话在 Gecko 也是最好的执行命令方式上也可以多种方式都去执行，在集成上也可以跟 jackets 做非常好的集成，可以在 CI 的结果中就能看到我们的这个测试的覆盖率的情况，并且这个报告也可以具备一定的实时性。所以整体上来看的话，这 Coco 肯定是当前首选的一个覆盖率的一个工具。


然后说完这个覆盖率工具，也说完了，我们有这么多种覆盖率的一个覆盖力度，那我们就来真正的开始了解我们最关键的一个点，我们的代码覆盖率是多少才好。

但这个故事老师觉得非常的好，所以也就把它给借鉴过来了，他就是说有一个大师，这个大师他能够很清晰的知道我们每一个应用应该覆盖多少，他是很经验丰富的大师。那所以几个程序员就过来问这个大师。


第一天他来了一个程序员，他就问大师，我准备写原来源测试了，我该针对什么去做代码覆盖，我要覆盖多少呢？大师回答说，不要担心覆盖报告，只要写一些好的 case 就够了。程序员微笑鞠躬，然后离开了。然后晚些时候，第二个程序员又来问了同样的问题，大师指着一口锅说，我应该在那口锅中放几粒米。看起来很困惑的程序员回答道，这怎么可能告诉你呢？这取决于你要给多少人做饭，他们有多饿，他们的饭量有多大，你是不是还有别的吃的，以及你到底有多少米，才能决定你要你放多少米吧？大师笑笑说，没错。


第二个程序员微笑鞠躬，若有所悟的离开了。在这一天将要结束的时候，第三位程序员来到了大神面前，拼接询问关于代码覆盖率相同的问题， 80% 一点也不能少。大师用非常严厉的声音回答道，并把拳头砸在了桌子上。第三个程序员也微笑鞠躬，然后离开了。在最后的回复之后，一位年轻的学徒转到，这位大师，今天我无意中听到你用三个不同的答案回答了关于代码覆盖率的相同的问题。那为什么针对它们，你要有 3 种不同的答案。大师从椅子上站起来，说，来来来，和我喝一杯茶，听我说一说这个事儿。在他们把所有的杯子都烧倒满了茶，大家坐下来开始喝的时候，大师开始回答。


第一个程序员是个新手，刚刚才开始接触测试，现在他有很多代码没有测试，他有很长的路要走，这时候过于专注代码覆盖将是一个令人沮丧也无法完成的这样的一个事儿。他应该更多地去关注怎么样去写好一个代码和写好就可以问的case，这个对于它的长久的发展都是最好。而第二个程序员，它在编码和测试方面都非常有经验。当我回答它问答你应该放多少米的时候，我就帮助他意识到了必要的测试量取决于许多因素，他比我更了解这些因素，毕竟这是他的代码，没有单一写的答案，而且它足够聪明，可以很好地处理好这件事儿。


我明白了，你现在学图说，但是如果没有一个简单答案，那你为什么回答第三个成学员 80% 一点也不能少呢？大师笑得非常大声，以至于他的肚子都在不停的抖。他说，第三个程序员只是想要简单的答案，即使你告诉他了这个答案，他也不一定会去追随他。那你告诉他多少又有什么意义？你还不如告诉他一个比较高的数，让他尽可能的多写一点测试case。年轻的徒弟在沉思中慢慢地喝完了他们的茶。好，这也是我为你们讲解的这个故事。那你们能告诉我覆盖率多少才是好的吗？到这里，我们这节课就结束了，在下一节课里，我们就要进入一个新的测试阶段，开发自测。来聊一聊作为开发的我们要怎样测试好我们的代码。各位未来架构师们，我们下一节课不见不散。
