# 1-3 测试驱动开发（0801）

hello，慕课网的各位未来架构师们，欢迎回来，在上一小节里，我们为大家介绍了一下关于代码可测性以及如何提升我们代码的可测性。那么在这一小节里，我们将衍生出来去讲一讲测试驱动开发。那什么是测试驱动开发？测试驱动开发就是先写测试再写代码，所谓的 test driving develop，通过测试来驱动我们的开发，那它与上什么样的一个过程，让我们来看一下。


首先就是我们要先写这个测试，它就和我们传统的开发有区别了，是吧？

我们传统的开发肯定是先写开那个业务代码，或者先写我们的代码，然后再去针对代码去写对应的测试。但是测试驱动开发它的方式就不一样了，他要先写相关的测试，然后写好测试之后再去写代码，这个过程中就会意味着我们先去写这个单元测试，然然后去运行这个单元测试，这时候必然失败，你想你连代码都没写，单元测试肯定跑不起来，所以所有的 case 都是红的。


接下来我们就围绕着这个测试去写我们的生产代码，写完代码之后，然后我们去运行相关的一个单元测试，如果这个测试还是失败的，那我们就要去修改我们的代码，直到让这个测试成功，这个 case 通过了，这时候这个 case 就是绿了。那这时候也意味着我们相关的代码就写完了，在这个时候我们还需要做什么呢？还需要可能需要做重构，因为我们为了能够通过测试而写的代码，可能还会存在着一些问题，一些细节要调整，一些不规范的地方也重新规范起来，命名要调整，注释要梳理，然后这个结构也要调整。


那重构完了之后，可能这个单元测试就回到了这个失败的状态，那这时候我们就要接着去调整我们的生产代码，然后直到让这个单元测试通过这样的一个方式，就可以确保我们每一行代码都在我们的单元测试覆盖下，并且我们重构也能够有一个单元测试作为我们的指明灯，来告诉我们这个重构版的代码是否是 OK 的。


通过这样的一个红绿重构这样的一个循环，就可以帮助我们更好地来完成我们代码的一个书写的测试驱动开发在实际上用的多吗？其实在当前中国其实用的并不多，其实大家都会存在着一个误区，他会觉得我开发机的压力很大，你这时候还让我先写测试，那这不就给我增加工作量嘛？

它会影响我的进度的，而且现在我们的 QA 帮我兜这一切，所以我不想通过这样的方式来给我增加工作量，给我增加这个时间。


并且我们现在也不是说一切都能够特别顺畅的使用测试驱动开发，比如说我们现在可能连单测都没有，我们都要重新去补这个单测，整个业务代码几万行，从头补单测，光补单测要花了几个月，那这个过程怎么办？我的需求还在提啊，我的这个进度还要正常保障，所以我没有办法完全通过测试群众开发来缺乏我整个架构的开发。
其次历史债务也很高，补测试把测试全跑通，把测试所有分支都覆盖，这个工作量就很大了。那我用测试驱动开发，我所有的重构所影响的面，我都要把测试补起来。这个所牵扯到的根据历史债务的偿还都是非常的大的，同时改造的成本也很大。而中国国内的现状就是这样，实施 TTD 的很少，大部分没有单测，甚至说单测还是候补的，大厂会好一些，小厂可能很多连单元测试都没有。


单元测试其实能够帮助我们改善代码的质量，但是因为上面这些误区，还有一些历史的原因，还有大家一些变习惯的原因，这一块儿其实在国内落地其实真的落地得很不好，那测试驱动开发其实它也有具备三个定律，我们可以通过这三个定律能够帮我们更好地去理解这个测试驱动开发。

比如说定律一，在编写不能通过的单元测试之前，不可以编写生产代码，这也意味着我们所有东西要先写测试，再写生产代码。定律二，只可编写刚好无法通过的单元测试，不能编译也算不过。它的意思是你的单元测试这个 case 也要有限制，你不要去一下子就铺了很多乱七八糟的case，你写的就是针对你要考虑的那个点，你这个场景下刚好让这个编译无法通过，那这样子的话就可以驱动你写后面的代码，当然如果代码它不能编译，也算不能通过，所以这样的一个方式就可以去让这个 case 红掉，然后去推动去写对应的生产代码。


那定律三就是只可编写刚好足以通过当前失败 case 的一个生产代码，这个就跟定律二去结合起来了，对吧？你定律二去写了一个刚好无法突破的单元测试，那定义 3 就去针对这个刚好无法通过的当前测试，去编写足以通过当前失败 case 的生产代码，这样子就可以每次都是一个小步法，我们通过小步法高节奏这样的一个迭代速率去完成一个很聚焦的一个单元测试，并且通过这个很聚焦的单元测试去完成很聚焦的生产代码，从而完成相关的一个测试驱动开发。


所以测试驱动开发本身的一个思想其实很清晰，就是先写测试再写代码。但是测试的驱动开发这个 TDD 3 定律告诉我们的是什么呢？一定要先写单元测试，这是定律一给我们的一个约束，那定律二就告诉说我们在单元测试的时候也要聚焦只写刚好无法通过的单元测试。这里一定是理解什么无法通过是当前这个 case 无法通过了，那围绕的是我们新增的这部分的业务诉求，那定域 3 则是围绕着定域 2 产生的单元测试去完成相关的生产代码。测试驱动开发可以帮助我们更聚焦到我们本身的需要完成的这个功能场景，因为 case 是覆盖这功能场景的，它也够帮助我们去覆盖各个边界，所以通过先写测试，可以更好地让我们理解我们当前的场景，理解所有的边界，理解整体的业务的一个诉求，一个产出。通过这样的方式更好的去完成代码的一个书写，那也通过这个红绿重构这个过程来保证我们整体的过程是在一个 case 的一个约束之下，从而让我们的代码可以更高质量地来完成。并且因为你是写的case，这也意味着你的代码的可测性天然得到了保障。虽然说很多公司没有去用TTDR，但是大厂现在也在逐步的往上去倾斜，很多大厂比如像阿里，像百度，他们对于这个单元测试的覆盖率要求很高，那所以在这种高覆盖的要求下，很多团队也开始逐步地使用测试驱动，他们会先写测试，然后再去写生产代码，从而一方面能够天然性地保证我们的单元测的覆盖率，其次也可以更好地来验证我们的代码的质量。
