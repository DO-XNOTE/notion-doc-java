# 1-8 集中测试（1705）

hello，慕课坊的各位未来架构师们，欢迎回来。在上一小节里，我们为大家讲解了研发自测这个阶段里面我们需要做的事情，我们需要定好一个自测的计划，然后去抓好自测的执行，去做好自测相关的一个功能覆盖，造好相关的数据，然后并且最终要做好一个结果，就是我们的自测尽可能的确保我们所有的代码，所有的功能自测通过。然后接下来我们就可以提测了，把相关的代码和功能交付给 QA 同学，让 QA 同学介入进行相关的集中测试。


那也就来到了我们这节课为大家讲解的内容，集中测试。集中测试是什么呢？

集中测试就是指 QA 同学他们集中地进行功能的测试验证，这样的一个阶段，它通常都包括哪些内容？比如说小冒烟测试。什么是冒烟测试呢？它又源于说最早的地形工业上的一些集合成，比如说，哎，我的几个电器接在一起了，我就插个电看看它能不能用，如果它冒烟了，说明它烧了，这就坏了。


那所以我们的网页测试也是指我们把它跑起来，接下来去做一遍功能的一个使用，看看它是不是能走得下去，能不能通，如果它能通，那就说明其贸易测试过了，然后接下来就可以做更细节的一个功能测试了。因为猫眼测试只是个体验上的一个主流程的一通过，而功能测试就要根据其细分功能一个点的结营测试，然后甚至说我们要经过多轮的这样的功能测试，才能最终的将所有的功能确定其 OK 了。然后再往后就是回归的测试，我们需要针对修改的内容以及被修改的内容进行相关的回归，确保我们整个系统整体上来说是 OK 的。


当然集中测试其实也是个很花时间的一个阶段，那在这个阶段里面，我们那个时间和资源主要在项目初期确定下来，并且保证投入。我们在项目的 kick off 的时候，我们就要知道研发花多少时间，测试花多少时间，那测试谁来测？投入多少个人花多少时间，这是在立项的时候就要明确下来的，并且在研发提测之后，测试要保证其资源和投入，确保其能够完成相关的集中测试。


而在集中测试中重点关注几个点，第一个是 case 的设计，这块儿可以在进入测试之前前置的完成，因为在研发进行开发的时候，测试的同学可以根据需求来完成相关的 case 设计。第二就是 case 设计中需要考虑到我们尽可能将边界做好覆盖，并且这里的边界覆盖的思考的一个全面度要比研发更广，他们要去抠很多研发没有注意到的细节边界，并且确保在这个边界下这个程序是 OK 的，能正常执行的。然后针对场景要去考虑到做剪枝，因为有的时候一些场景是不可能被触发的，那这类场景可以被剪枝掉。场景剪枝的目的是什么呢？是为了降低场景的数量，从而来提高测试的效率。当然如果有一些场景真的没法剪掉，那么是否可以考虑把它做场景合并，把一些场景合并在一起去测掉，这样子的话也能够降低测试的工作量，通过这些方法以某些工作前置某些工作，通过一些方法把它剪枝掉和并掉，从而降低掉在真正集中测试这个阶段里所需要投入的人、资源和时间。

冒烟测试是指每一个版本或者每一次需求变更后，在正式测试之前对产品系统进行一次简单的验证性测试。所以这里看老师把关键词都放大了，需求变更，那这是我们要冒烟的那个点，那在正式测试之前是我们冒烟的时机，然后做一次简单的验证型测试，这是我们冒烟要干的事儿。


所以我们的贸易测试的验证范围其实是很清晰的，它主要是面对主功能、主场景、主链路，还有一些预置条件。那这些就是我们要进行这个验证型测试，进行这个冒烟测试所需要考虑的一个范畴，它不会有太多很细致的一些边界测试，或者一些很细的功能性测试，它注入的目的就是说，哎，我先试试看我能不能一下子跑下来。如果我连跑都跑不下来，那你说明你自测肯定是有问题的，因为我如果能够顺畅地跑下来，那就说明，诶，至少第一你的自测没有问题，你的代码大概上没有问题。

第一方面基本的一个功能验证，它需要验证功能是否符合预期，那其中就包括我们的功能的验证，一些流程的验证和体验的验证。我们功能上用起来，我们做了一个手电筒的应用，那它是不是能点亮？是不是能关闭？是不我能调？是不是能调亮度？那这就是一个功能性的验证。


其次就是流程性的验证。哎，我是不是能先做第一步，再做第二步，再做第三步，这个流程能不能走下来？还是说我走完第一步流程就飞了，我第二步也触达不了，它也结束不了？那这其实就是问题了。第三就是体验的验证，我们是否能够用下来？这个按钮是不是真的能看得见？是不是到这该出来的时候出来了整等这些就用户体验上的验证也属于基本的功能验证范畴。在基本功能验证之后，我们就要去做这个数据结果的验证，我们要验证全流程的数据生产和展示正确。比如说我们从商品的一个上架，然后到用户的下单到出库，然后到发货，然后再到结算，我们整个链路上整个围绕这个商品数据是否生产和展示是正确的，那这些就是我们的数据结果的一验证，它要包括在整个功能去生产数据是否是正常的，看功能能否正确地生产出相关的数据。


第二指出过程的数据处理是否正确。我们每一个功能它肯定有的时候要经过很多的过程，比如说我上架一个商品，那它可能要先要去填商品的信息，然后再去调整商品的状态，然后再把它显示出来，收藏、传图片等等。那在这个过程中，每一步它这个过程数据是否是正确的？最后一个就是显示的正确，我们如果数据都正确，显示不正确它也是有问题的。那显示正确除了数据本身的正确之外，还有它既显示的形式与格式是否正确。它的比如说它的样式居左居右、颜色以及它的比如说小数点的位数，你这个大整数，它们那个逗数号的标识是否有，那是否会变成了一个科学计数法等等，那这些就是数据显示上的一个正确与否，这也是在功能测试中需要去验证的，在数据结果验证之后就进入到了一个探索性验证，他就要去探索各种边界和异常值，看看在边界值下这个系统反应是否是正常的。


然后针对某些功能是否具备幂等性的一个正确，我反复执行它是否都能够正确的来记录相关的数据，正确的去更新，然后针对一下异常值处理是否正确，我是不是我写了个异常值它就挂了？我补写一个库存，我选了个 -1，它是不是能够校验出我这个 -1，我不该往里录，还是说我 - 1 进去之后他挂了，甚至他是真的出了一个库存为 - 1 的这个一个订单？那这些就是我们在探索性测试中需要去验证的一个功能，当然除了这些之外，我们在功能测试中还需要做好很多事情，比如说像我们的测试日报，我们要能够明确出我们每一天的一个测试进度，因为要组织协调整体的 bug 的技术与跟进，有 bug 要及时反馈，

为什么是在功能测试之后才叫做性能？因为在功能测试的过程里，研发还会去不断地调整它的代码，过早的做了性能测试，也意味着后面还需要反复的去做这个性能测试，才能确保其最终修改的代码，其性能能够满足我们的要求。在这个过程里我们就要去验证大数据量、大流量和高频操作下性能是否会能满足预期。那大数据的大流量跟高频操作都是指什么呢？都指压力，数据量大，那 DB 压力大，我们的代码压力也大，大流量我们的机器压力大，服务压力大，高频操作，其实这个就意味着很多事情它在设计上是否面考虑到了。面对这种压力，那我们就在压力下去看我们的系统性能、系统体验和系统行为。


系统性能是指在这种压力下，我们的系统是能保持一个什么样的一个性能？它的性能是否开始出现异常了，是不是开始变慢？这是第一类，第二类系统体验就是我们的系统看是否体验会出现问题，它开始卡顿，然后它的交互出现问题，甚至有些数据开始显示不出来了，那这就是体验。


那系统行为就是在系统在性能和体验之下，那系统就开始呈现他们的行为走向，它到底是流程正往下走了，还是报错了，还是卡顿了，还是说超时了那种的这些行为就是我们在压力下的一个系统表现，通过这三类我们就能判断出我们的结果是否符合预期。如果结合符合预期，那说明我们的性能测试达标了，如果我们结果不符合预期，那就说明我们需要去做好相关的改进。当然这也就刚才说的，我们的性能优化通常放到最后要先保证功能，再去优化性能，否则的话就可能真的是优化了性能，结果又反过来还改了代码，改了代码之后又要重新去优化这个性能。


然后在整个性能优化过程中，我们要结合全链路压测去进行极限性能的探寻，因为有的时候我们自己去造了一点流量，造了一点数据，它还不至于能够达到真正的类似于线上的这种效果，因为我们还是基于单个模块，我们要看整体的一个效果。也许你单个模块谁都不超时放到整个链路上，因为超时的扩大，超时的叠加，最终链路超时，它在单个模块下大家都正常，但是整个系统层面上它出现了问题，那所以也意味着我们要结合全链路压测进行极限性能的确认与调优。在职业竞争下去看看系统的我们的表现、体验和行为是否符合预期，哪里不符合预期改哪里，最终让我们整体系统能够达到我们所需要的这样的一个性能预期。
