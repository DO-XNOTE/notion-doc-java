大数据现状深思到底是研发做得不够，还是测试做得不够，抑或是在整个研发流程中存在着某些问题？那作为架构师的我们又能去做什么呢？所以我们就要和大锤一起进入到这一章的学习。

然后 QA 同学才会进入集中性的测试，他们通过多样化的测试手段，对我们的代码，对我们的系统功能、视觉等等进行一个周密的集中的测试完后，并且针对历史的一些功能，针对我们这次没有改的功能进行相关的回归。


基产没有改为什么还要回归？因为你不知道你改了的部分有没有对机产生影响。即使单元测试可以保证代码层面上没有产生影响，但是在业务层面上，在流程层面上仍然要进行相关的回归，确保我们没有从其他地方，就是单元测试没有覆盖到的地方，影响到了其他的功能。当然还有一些新型的测试方式，这些测试方式能够帮助我们更好的完成相关功能测试，比如说像众测，可以发动更广大群众的力量来完成一些单独通过研发，通过 QA 没有办法完成的相关的测试的功能，还有一些探索型的测试，我们打破原有的测试的这种步骤，用探索型测试去发现一些用传统的测试方式不容易发现的问题。


这样子我们综合性的能够对我们的代码进行一个更深层次、更透彻的测试，从而保证我们上线的每一行代码都是健康的，我们是整个架构仍然保持它的生命力。所以在这一篇当中，同学们也要跟着大锤一起去理解不同的测试阶段。

比如说像单元测试，它是判断各个独立单元是否符合功能性诉求，大家要注意这个里的重点关键词，功能性诉求。所以对于一些非功能性的诉求，包括一些整合性的、集成性的一些，其实单元测试是没有办法盖住的，它更多的还是看独立的单元。


什么是独立的单元？类是不是独立的单元？方法是不是独立的单元？可以说我们最小的最细粒的独立单元就是方法，而针对类来说有可能一个类很大，所以是说它作为一个对象来说，它是一个独立代孕，但是它仍然可以被细分，那这样子的话我们更尽可能的去做好最细粒度的单元覆盖。


第二就是集成测试，集成测试是集中测试中的一个环节，那在这个环节里用来判断软件是否能够适应运行环境与基础设施，以及多个组件之间是否能够正常的协同工作，这个是在最起码的基本功能保障之后，我们再去看我们的软件，它的运行情况，它和基础设施之间的配套，我们的基础设施是否要升级？我们的基础软件是不是要升级？我们依赖的基础库是不是要升级？那多个组件之间是否能够协调工作？我们依赖的其他的二方库和三方库是否和我们的架构能够正常协同？我们读作多个组件之间，多个模块之间是否能够正常的协同工作？那这个也是我们要考虑的很重要的一个点。


而针对回归测试来说，它可以判断软件的其他部分是否被影响或者符合协同性的这样的功能的诉求，所以这个点是我们去验证我们线上其他功能没有出现问题的一个很关键的一个点，通过回归测试可以非常有效地帮助我们验证我们在上了线之后，我们是否全局上、全面上都是 OK 的。


当然还有最后一个就是验收性测试，这个测试是由产品和业务来完成的，他们用来验证系统是否符合客户的诉求，是否契合他们最早提出来的这个需求，从而来确保我们这次的迭代是否是有效。那通过这些不同的测试阶段，最终能够保证我们上线的这个代码，我们上线的这个架构是健康的，是正确的，这个也是我们在整个这篇一篇章中希望为大家去带来的主体的内容。那话不多说，就让我们跟随大锤一起进入质量保障的海洋，看一看在整个质量保充体系中对都哪些内容。各位架构师们，我们下一节课不见不散。
