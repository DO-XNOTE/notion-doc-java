# 2-4 链路改造之代码改造（1705）

hello，慕课网的各位未来加工师们，欢迎回来。在上一小节里，我们为大家讲解了一下在压测过程中我们的目标是什么。首先这个目标其实要平衡，要明确我们的一个匹配以及隔离。什么匹配，真实场景与测试场景的匹配，真实数据与测试数据的匹配。同时我们在整个过程中还会有很多的小目标，保证我们的目高目标，我们的系统的稳定性，我们各个监控指标数据一致性，这些都是我们的小目标。但最终的目标就是探取我们的极限，明确极限的性能，去做好我们的容量规划，那这就是我们整个压测的一个大目标。


在这一小节里，我们就进入到了真实的压测准备环节，第一环节就是做好我们的系统改造，做链路改造的时候第一步是什么呢？是做代码的改造。为什么要改造代码？为什么不能直建那个线上系统？就用，这就是我们在上一节课的目标中平衡性中提到的那个很关键一点就是隔离，我们要能够将测试流量与正式流量做好相关的隔离，

所以压测流量和压测数据以及真实数据和真实流量，他们二者之间应该是天然隔离的。不能是说我们将压测流量毁在真实流量里面去压，我又不能够说将压测数据写到真实的这样的一个数据表里面，否则的话它就会产生大量的干扰，影响我们支持的用户，甚至产生出相关的故障。所以做好相关的隔离能够帮助我们把这个影响给隔离，好，我们不影响正常的数据，正常业务以及正常的用户。


其次我们针对这个压测行为也能够产生相关的隔离，压测的流量不会走到正式的业务上，而且正式的用户不会访问到压测数据，这样这些完全隔离到了之后，那正式的用户还是完全不受影响。第三就是整个的一个制造过程和清理过程都能够非常容易的完成，因为有独立的数据，独立的业务流本身来说是可以很容易的去 mock 的，我们相关的数据也能够很容易很清晰的把它干掉。那这就是为什么要做隔离，以及为什么要做这个代码改造。接下来肯定就聊，就是既然要做这个了，大家已经明确下来了，

我们肯定第一步我们要明确出怎么样子去标识的这个正常流量跟压测流量，如果连标识都没有，我们是没有办法区分这二者流量的，就没有办法做任何的一个隔离。


所以在二者之间，肯定在他们到负载均衡再往业务系统程序打的时候，我们需要在压测流量上带有特殊的标记，这样子的话在每一个应用上我们针对这个链路 agent 就能够明确出它到底是一个压测的流量，还是一个正式流量。这个其实就是我们从架构层面上的基于应用之间交互的一个思考。


除了应用架构之后，我们还可能会有一些，比如说像数据库，我们肯定要开始做最基本的一个有正常表，也有针对专门存储这种压测数据的影子表，那这两者之间肯定存储的就是我们对应的一个支持数据和对应的压测数据，还有一些除了数据库之外的其他的一些存储。比如说像 Redis 可能会也会，这对应的正常存储也会有对应的影子存储。那表很好理解，就是我们影子表可以完全是另一套表，比如说表上加前缀，通过这样的方式能够明确出它这是两条完全不同的表。


但是对于 Redis 这种是以 k v 形式的怎么办？答案就是针对 k 加前缀，我们针对一个 k 做固定的一个前缀，这样子的话我们就能够明确出这些 k 一定是压测流量，那它也意味着我们在应用的这种中去使用这种中心件的时候，也要根据这种流量动态的标分别的路由到对应的正常存储和影子存储出去。


并且在应用之间我们还会考虑到一个就是我们会有二方调用，也会有三方调用，那也意味着我们有些流量其实是要明确的打到二方或者打到三方的，但是也有很多场景就是我们不希望将流量打到二方或打到三方，那这时候也意味着我们针对二方和三方，我们要做对应的mock，将他们本身可能会处理返回的结果直接通过 mock 来返回，这样子的话就我们既能够保证整个链路的完整，又不至于影响二方和三方兄弟们通过这样子就能够完成我们整个的代码改造。所以整个架构的思路就是这样子的了。


明确架构思路之后，我们就要来看看若真的要改，我们要怎么改。

第一针对这个流量要做改造，那肯定流量第一步就是流量的进入，第二步就是我们在整个流量的传递，因为进入是我们通过压测工具，比如说像这 matter 它压进来了那些流量它要怎么进来？其次就是针对这个流量，我们在应用之间要明确出它的传递过程中怎么样子，保证这个标记令人存在而不丢失。


第三就是针对这种外部的一个 mock 要怎么去做？这个流量进入其实有很多的细节点是我们需要考虑的，比如第一个大点就是登录问题，因为很多时候我们的一些接口是要有登录态的，比如说你下单用户不登录，你就不能让他下单的。但对于压测来说，每一个压测前面都跑一次登录其实是一个麻烦的一个事情，并且它也会带来很多的一些问题，比如说我们怎么造那么多的用户？第二就是针对正式现场来说，用户不会在下单的前一秒才登录，他们会有大量的前置登录的一个过程。那如果我们在压测的时候，每次压测都登录的话，其实它跟真实场景又不契合了，那也意味着我们既要考虑它要匹配真实场景，也不能因为灯问题去卡住我们的流量接入，所以这也是我们链路的流量改造，所以这块蛋白感受中很关键的一个点。


第二个点就是流量的重复使用，有些场景下流量是可以重复使用的，但有些场景下原来我们系统就做了限制，比如说限购一个用户最多买 5 单，但是你在压测的时候这样就不合理了，除非你是要压这个限购这个点，如果你压的是下单，那我们肯定希望我们一个用户我们能下1万单甚至下 10 万单，这样子的话我们可以用最少的用户制造更多的这样的一个测试数据。


所以针对压测链路将某些流量限制和提防做一些事做一样对应的一个放开，从而让我们的流量能够重复性使用，这个也是我们在流量进入环节需要去考虑的。当然还有一个就是 CSIF 的验证，因为很多时候我们提交订单，提交这种功能都会有 CSIF 的这种校验，而避免这种跨站点的这样的一个攻击。那通过在压测流量标记下跳过这个验证，能够帮助我们更好的来完成对应的一个流量结构，否则的话每次还要去 mock 这个token，其实也是个很麻烦的一个事情。


那说完流量的进入，接下来去说这个流量传递，其实最共为关键的一点就是模块之间这个压力测流量的标识不能丢。其次就是流量除了在模块之间会传递，它也会通过中间件进行相关的操作，所以中间件我们也要有对应的压测流量标识。然后第三就是数据层面上它也要有对应的一个操作表示，通过这样子能够来完成整个的一个流量传递。而到外部其实就包括外部的实际调用以及外部返回那个mock，这就是我们在整个的流量改造层面上需要去关注的这些点。并且我们也要考虑到我们对于外部用户做好相关的保护，因为外部流量它的成本承受的极限不是我们的控制，比如说我们系统能承受 3 千克BS，但是外部系统商它只能呈现 1000 QPS，这时候我们既要围绕着当我们的商品是一个外部系统商提供，并且需要外调的时候，对于整体链路就要有对应的限流保护，从而保护我们系统上不因我们的流量被打爆。
