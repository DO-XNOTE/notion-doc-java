# 2-9 压测设计目标设定（1301）

hello，慕课网的各位未来架构师们，欢迎回来。在上一小集里，我们用五字箴言来讲解了一下关于场景设计的相关的一些点，那 5 个字就包括全变、真、准和稳。通过这五个字很形象地将整个场景设计中我们需要考虑的方方面面都包括了进来。那么在这一小节里，我们依然是压测设计，但更多的是聚焦在我们在设计一次压测的时候，我们要考虑本次压测我们要设定一个什么样的一个目标，它和之前全链路压测的一个这个大目标其实是有差异的，另一个目标更接近于整个事情。我们要做好全链路压测要做什么？但是这次就是说我们怎么样子通过设置一个合理的目标来驱动我们做好一次压测，那就让我们话不多说，进入到整体的课程，首先我们还是要明确一下整体的目标的一个基调，就我们设计这个压测的时候，我们要考虑我们以什么样的一个出发点去设计这个目标。

我们要想我们要针对这个目标做好对应的思考，就是我们这次压测到底预期是什么？我们要的是什么？其次就是我们在做这个目标设计的时候，我们一定考虑我们是在一个周期性压测的基础之上去做的，就是我们这一次目标的设定，只是围绕着周期性压测中的其中的一次压测，但也意味着我们上一次压测的结果会作为这一次压测的一个参考，并且修正我们对于这次压测的一个预期，然后逼近我们整体压测的一个大众标。


那也意味着我们在这个压测的一个目标的设定过程中，其实是我们在逼近我们大目标过程中的一个里程碑，那也意味着我们要做好在这次工作中的一个平衡，包括我们的目标基本上是可达成的，因为我们这个大目标不一定能达成，但是针对这一次压测，我们的目标要有能达成。


所以在极限法就是我们去探索我们系统极限的时候，是要尽可能的逼近极限，也尽可能的逼近我们这次为我们设定的这个极限，同时目标也要有一定挑战，它是一个不那么容易完成的。比如说我本身我的系统上次 1000 q B s，我这次1050，那这个目标太没有挑战了。但是如果这次1万，其实这目标就不太可能完成，所以我们设计一个可达成但是有挑战的变起来会来跳一跳，能够得着的目标它才有价值。


那是有挑战，也意味着我们要针对这个挑战，我们要去验证它，验证我们在这个目标流量下我们的表现，在二者之间寻求一个平衡，既能让他有挑战，也可以让他尽可能的可达成。当然也可以是一个略微达不成的目标，但是它能够帮助我们去逼近我们的极限，或者验证我们在目标流量下我们整体的一个表现，这就是我们在思考这一次压测执行过程中目标的一个基础的一个出发点，那在定义这个目标的时候，这个关键词非常重要，就是目标一定要纯粹，

就是我们的目标不要太花了，不要太复杂了，不要乱七八糟，那样的话就意味着我们的力量都散掉了。


我们尽可能的是让我们目标很纯粹，然后尽可能是一个单一的目标，这样子的话我们就能够排除到其他的干扰，把我们所有的注意力聚焦在我们这次要达成的目标之上。然后可以确定我们所有的力量都是让向这个目标去进军，然后确保目标单一且能够尽最大的力量去完成它。


当然目标纯粹也可以帮助我们更好地去验证我们的优化，它可以有多种的对照，比如说多种对照，我们针对一个问题或者是一个瓶颈做了优化，然后我们就要去验证我们这个优化是否产生价值了，也意味着我们同时对两组做同样流量的压测，来完成多主的对照调整，和对照组之间来进行对比来看一看是否其真正完成了优化。还是说虽然说做了优化它其并没有带来任何的改变，甚至说还走开了，倒车便退步了都有可能。


也可以针对历史做优化，我上一次是 1000 Q B s，压出的指标是这样子，我这次改了之后我又是 1000 k p s，我的指标有了大幅的改善，甚至说我可以压到 1500 QPS 了。那这就是一种历史和当前的一种对照，通过多种对照的方式能够帮助我们去验证我们的优化结果，也可以帮助我们去确认我们优化的确产生了这个价值。


其次除了验证优化之外，我们还可以通过压测去验证一个可用性，它包括多种场景的一个可用，比如说我们来契合这样的正逆向那样一个复合场景，比如说我们设定了一个相对有限的库存，使得我们拥有大量失败的流量，那在这样的一个状态下来验证我们整体系统是否综合上是可用的。并且针对于，比如说我们还输入了一些错误的流量，比如说伪造了一些攻击流量，那我们在这种干扰下是否具备强容错这样的一个效果，通过这样的方式来验证我们整体的系统的可用性是否达到了我们的预期，第三就可以探寻相关的极限，我们去验证我们对于这个系统我们施预设的那个极限是否可达到，然后明确出我们如果达到这个预期我们需要多少资源去做相关的保证。然后当它契合真实场景的时候，是否能够满足我们对于这个场景的一个期望，从而探寻出系统它的这种极限下的可能性，以及在极限下是否能够达到我们的预期。


其实这里就可以发现，在通过验证优化验证可用到最终我们来验探我们的极限的时候，这时候我们就开始逼近了我们整个全链路压测的一大目标，我们去看探索我们的极限可能，我们以及去验证我们的极限表现，从而去确认我们需要多少资源，那在做这么多时候我们也要考虑到我们的目标也是能够要，需要能够覆盖的。

所以我们要做场景分析与场景拆解，去做这种主场景的确认，最终明确就是场景的目标，这也契合了我们上一小节去讲场景设计的相关内容。


我们场景设计了那么多，其实目的就是为了我们能够明确出我们每一次压测的一个主场景，并且去制定出场景的目标，然后这些场景去看我们能否覆盖住我们所有的分支，然后制定好每一个分支的目标，从而来达成一个目标的覆盖。因为我们要把整个系统所有的都覆盖其实是挺难的，但是通过这样的一拆解法可以帮助我们完成目标的覆盖，而分支目标就会映衬到执行链路的目标，因为每一个场景分支最终都会映射到一个执行链路，甚至是说一个链路它能够覆盖出多个场景，而通过这样的方式来完成链路的覆盖。


当然链路的目标也意味着我们要做好到链路的分析。链路分析和场景分析是相对应的，每一个场景其实最终映射到一个链路，那也意味着我们将场景分析中那五字箴言帮助我们理出来的又好又全的各个种种场景拆解，映射到我们这个链路拆解上来，然后最终来完成链路的覆盖。

当然它是会在我们完成了一个纯粹的目标这个单一压测之后，我们需要去做的一个事情，它去验证这种多样化的一个压测场景，包括多链路的协同、正样的混合、多样化的行为。通过这种多样化来验证我们系统在接近真实的场景中是否能够正确的来满足用户的诉求。


其次就是我们去调整我们的行为，包括我们针对行为增加一些波动，有一些正常的行为，也有一些不正常的行为，比如说就有一些恶意的行为，或者有一些刷单的行为，我们能否在这些行为中仍然保证我们系统的稳定。其次就是我们整个的行为的比例是否是偏信于真实行为的，并且在真实行为上可以有一些波动，从而提高我们系统在这种行为比例的一个容错性，当然行为比例最终会转化为这样的一个流量比例，它就可以帮助我们从压测入口上去做好相关的调整。当然无论是多样化还是行为调整，我们其实希望的还是说它尽可能的是真实，所以就可能映射到混合目标加真实场景，甚至说我们在之上再增加一些波动，增加一些容错，最终让我们的压测能够更全面的覆盖我们的系统。

当然不同的压测，比如说只读压测和读写压测，这个调整的思维也是不一样的。针对只读压测来说，因为大量是读，我们目标更多的还是在于极限的探寻，那针对读写压测来说，我们的目标除了极限探寻之外，我们还要考虑一致性、稳定性等等。


而针对场景，我们也有很多调整的一些思路，比如说关于流量的一些递进，还有流或者是流量的陡进，刚才还是100，现在直接 5000 了， 100 起到了一个预热，而 5000 则通过这样一个陡进来验证我们系统在面临这种流量突然变化的时候，能否自动的调整来适应这个流量的变化。而起始高亮和衰减则是映射到大抢的这种场景。秒杀倒计时结束，用户突然涌入大规模开始抢购，第一波用户消费完毕，好的商品已经被销售一空，这个流量开始衰减，非常契合于这种秒杀情况下，热门商品和非热门商圈混合的这种场景，无库存、高流量，我们的库存开始就很少，一下子售空了，然后流量仍然还有很多，不停在试探的去买。那这种场景下也是我们一种可以去探寻的一种场景。


当然我们也可以根据系统表现去调整，但是这个前提是我们的系统不大规模出错，因为系统一旦大规模出错，你再去调整它其实没有意义了，你只能是停止压测。当系统不是大规模出错，它开始出现一个按比例的这种错误的时候，体则意味着系统进入了一个极限下，在极限下我们仍然可以持续的进行压测，然后不断的调整我们的流量，去看一看在这个压测的这个极限状态下，我们是否还可以有很多更多的信息可以获取得到？然后根据系统当前的表现，我们各种指标，我们的CPU，我们的内存，我们的load，我们的网络的traffic，我们系统的 IO 等等，通过这些来动态的去调整我们的压测，一旦发现问题马上停滞，确保我们的系统不会被我们的影响，从而影响了正在使用的真实用户。


说了这么多，其实就是希望我们能在每一次压测过程中，都根据实际的场景，根据我们系统的表现，不断地调整我们的目标。然后在完成这一次压测的过程中，能够让我们获取更多的信息，能够让我们发现系统中更多的问题，让我们知道我们整个这个周期性的压测，我们下一次压测的目标又要朝哪个方向去走？那这就是我们这一次压测所达成的一个效果。好到这里为止，这一节课的内容就结束了，那在下一节课中，我们就结束了这个压测设计的这一小节，我们进入到了压测执行，聊一聊在压测执行的数据准备中，我们需要做哪些事情？好，各位未来加工师们，我们下一节课不见不散。
