# 2-11 压测执行之压测执行（1352）

hello，默克网的各位未来架构师们，欢迎回来。在上一小节里，我们为大家介绍了一下关于压测之前的数据准备，包括基础数据准备和流量的数据准备。那没有这些数据，压测无从谈起，所以做好数据的准备，还有一个准确的能够体现出这次压测所对应场景的目的的，那才是一个好的数据。那么这一小节我们就进入到压测最正式的一节，就是压测的执行。其实压测的执行本身来说是为了针对本次压测目标的还原，我们要明确这个目标是什么，然后去朝着目标去努力，去执行，去做，

第一个点就是我们的工具要怎么选型，很多人来说，如果公司已经选好了我们对应的工具，那当然来说就直接用公司已经选好了，就不需要再额外去选型了。但是如果说本身公司还没有一个明确的压缩选型，你现在是第一个吃螃蟹的人，那么你就要针对这个情况去做好用工具选型，毕竟工欲善其事，必先利前器，有了一个好的工具，才能够更好地进行压测的执行。


第二，要明确压测的执行方法，以这样一句话，同时追两只兔子，将一有所获。这什么意思？就是你同时有两个目标的话，你觉得哪个目标都完成不好？同理，压测也是一样，我们在执行的时候一定要将之前预设的目标很明确的拿出来摆在前面，告诉我我这次就要干什么事，我先把这事干完了之后，我才会轮到下一个目标。那这样子的话我们就可以明确出来，我们的这次执行过程中，同一时刻我们只会有一个目标。


第三个也是比较重要的一点，就是我们要明确我们的这个压测的一个节奏。俗话说你必须有一个很好的节奏，先开箱再瞄准，那它指什么呢？就是你先进行执行，然后你再去调整你的方向。但是也有另一种说法说你要先瞄准再开枪，这样子你才能够更好地找到你的目标。哪种是对的？我觉得两种可能都有意义，但是在真正你们不清楚你能达到什么样的，尤其是在你第一次进行压测的时候，你并不知道你的系统能达到什么样的效果，那在这时候线开箱再瞄准，可以更好地帮助你们更快地来感知到我的系统当前处于什么样的一个状态，它能到达什么样的极限，然后再有针对性的去进行压测的时候，那就是先瞄准再开枪了。

压测工具目前来说还是比较多的，但是我们首先在选工具的时候要考虑几个点，第一个点就是我们的工具能够帮我们更比较真实的去制造相关的流量，如果工具它不能够帮我们制造真实的流量，它的流量都聚焦在某小部分机器上，或者它的流量是偏的，或者是它的流量本身来说不稳定，那都可能会导致压测不出我们想要的结果就是压测它要能支持这个高QPS，如果我们的工具没有办法支撑一个高QPS，假如说我想要压 5000 QPS，但是我的工具到 1000 就压不上去了，那这个工具本身来说就没有任何意义了。


第三就是易用性，一个好的压测工具，它一定是拿来就稍微配置一下就能用，而不是一个非常繁琐的一个配置文件或者非常繁琐的脚本。那样的话也意味着我们在做压测的过程中会遇到很多很麻烦的事情，折腾几次之后大家的积极性就被打击了。当然还有一个点非常重要，就是能省钱就省钱，毕竟很多压测工具它在制造高片 QPS 的时候，它其实是很花钱的，那么意味着我们要综合起来这四点去考虑我们的选型，当然除了这四点之外，我们要考虑其他的几个点。比如说我们公司的场景是什么呢？它是要针对一个简单场景去压，还是针对一个复杂的场景去压？这个也是要考虑的。


第二就是压测机的资源，就是我们在制造这些QPS，我们也是要叫机器的，那这些资源我们能有多少？我们是否能保证我们有足够的压测机可以帮助我们去制造这个高QBS？第三就是我们在执行压测这个过程里，我们常备资源到底有多少？比如说我们这个压测机它是一部分资源，然后我们在平时我们这个链路上要达到一个 QS 的时候，我们又有多少资源能够放到整个链路的一个部署上去，那这些都是我们的常备资源，这些常备资源才能决定我们能动用多少资源进入到整体压测上去，才能知道我们能在这些资源上能够压出多少QPS。


假如说工资的常规资源很少，大家就在线上进行这么多，那你想要去网上压的时候，就意味着你要到处去借，也意味着整个压测过程会有一个很长的前置的挪用资源的一个过程，因为别人可能还要从线上把他资源卸下来，然后再给你，你再去部署，然后再去压测，压完之后还要还原回来还给他们。那这也是我们在压测工具中需要去考虑的一个点。


如果我们的肠胃资源充足，那通常来说就是部署压测就好了，如果我们肠胃资源不充足，我们的压测工具最好能够和我们相关的资源调整工具去结合起来，当然这也可以我们人去定义对应的SOP，去完成相关的压测执行。当然对应着复杂场景，我们就还会有简单性的压测，哪种会压测会是简单性的压测？比如说验证型的压测，我们只想用一些小流量去验证一下这个功能它是否是 OK 的。还有就是说我们想要通过压测来让我们的线上系统预热起来，那这些都是简单压测，那它就不需要有那么多复杂的考虑，更多的还是在于怎么样把这个点做好，完成当时这个小目标就考了，这也意味着我们针对这种功能性验证性压测适合我们，针对这种追求高 QBS 探索系统极限的实际上就不一样的。但是如果我们的工具二者都能兼备，那当然是最好。并且如果我们再有一些压测，它的诉求非常少，比如说刚才我提的那个，我这是一个预热，我只要 50 Q B I C 就够了，我把整个阶段区预热一下就好了。那本身来说这种压测它所需要的资源和针对我们线上并不需要扩容，也意味着它对于工具的要求就会非常的少，我们可能用一些开源的，或者用一些自研的就非常好用了。


那么在这些前提之下，我们就去考虑我们的工具，就意味着我们有几种可以选择的方式，比如说第一种就自研工具，我们自己去研发一套我们压测工具，通常来说大厂或者一些相对比较大的公司可能会选择这条路，这条路的背景是什么呢？就是你要公司有钱有人才能去自研，因为你要需要疼人，你还花很多钱，但是它的好处是什么呢？好处是你可以针对你们要的这种复杂场景，你针对这些资源去做对应的定制，我能够针对我们公司的业务特性去定制我们的压测集群，去定义我们的这样的一个面对这种场景下我们这里的压测链路，那这样子的话就意味着我们能更好的来完成对于压测。但是不是所有公司都这么有钱，有人能够去做自己的这样的一个资源工具的，那也意味着使用开源就会是一个非常好的替代方案。


开源很多时候也意味着它能够实现很多比较简单的压测，但是这对特别复杂的压测，用开源就意味着你要有很多的二次开发、二次加工、二次配置的这样的一个过程，否则的话用它很难直接就达成对应的这样一个效果。那这时候就有第三条路就是购买服务，针对公司一些关键性压测，我们比如说一架很高的QPS，那我们要自己去背这种压测机的话，其实是一个很大的一个成本。然后用开源的话，因为业务很复杂，所以它的整体的之间的串联关系也会很麻烦。那在这种场景下用买一个服务，比如说阿里云的服务就会非常方便，因为它能够帮助我们解决所有的问题，虽然说花一些钱，但是也比自己完全去自研要省钱，同时也能够达成我们的诉求，那这时候就进入到了我们真正的压测的一个执行。

然后去提出我们的压测需求，我们要做一个压测了，为什么要压测？er，哪个场景？我们希望压测什么样的效果？那这时候就要需求做一个相关的评审，让业务团队的人以及压测执行团队的人参与出来。


比如说我们的研发，我们架构师，我们的 QAMTL 都参加这个评审来评审这个压测需求是不是能够满足我们当前的一个预期，然后针对我们的线上系统去做对应的改造，因为你压测就意味着我们之前提的我们的链路要改造，中间件要改造，然后我们所有的线程相关的这个代码要改造、存储要改造，针对这些东西要做好对应的压测改造。在这个过程里把整个业务链路梳理清晰，要把这个数据依赖也梳理出来，然后完成我们刚才说的这些代码和中间件的改造。


这时候我们的线上系统就位了，在这种就位的状态下，接下来去制定我们的一则一次压测计划，因为我们在整个压测需求中，它意味着我们在达成这个目标的时候，它不是一次就能完成的，它是一个完整的一个计划，我们要周期性的或者经历过多少次压测，最终达成这样的一个效果。


在压测计划提出之后，然后大家对这个压计划做一个简单的评审，来确定压测计划可行，前面又出现相关的遗漏，然后就去做好相关的压测准备了。包括记住数据的准备，流量数据的准备，其中基础数据可以每次都复用，但是流量数据可能针对自己的场景去做好对应的一个制造，然后去做好对应的执行。这时候我们就要针对那些我们比较关注的指标，要实时地关注其变化。当系统出现负载过高，或者数据大规模报错的时候，我们就要去调整我们的压力，比如说系统一切良好，甚至 load 也没有变高， CPU 没有变高，那这时候我们肯定是参照整个的一个执行路径，然后就一步一步的去增加我们的流量。如果当系统开始出现报错，我们的 loader 标到了10，然后我们的请求开始超时了，中间某些比如说 d b 开始报错了等等。那这些时候我们就要去调整我们的压力去往下走，从而让我们的吸头开始往恢复，已经降低压力了，它就开始用钱恢复回来，然后从而找到一个开始出现少量报错，但是不至于压垮我们系统，找到这样一个点，然后去观测我们整体系统的一个表现。


当然也有可能是一旦开始压测，马上整个系统就挂了，开始出现大量的超时，大量的异常，那这时候我们就马上要停止压测，然后去检查到底是什么原因导致了我们的系统出现了问题，在压测执行之后就会出现一个压测报告。这就是我们针对整个压测的一个过程进行一个梳理，把所有的数据记录下来，把所有的现象记录下来，把所有的问题记录下来，然后进行相关的复盘，聊一聊为什么会有这些现象，为什么出会出这些问题？那这些问题后续要怎么办？谁来跟进这些问题？如果它和我们当前压测计划有变有些冲突，那我们就需要调整计划，然后重新去做评审，然后重新去做准备，为下一次的压测做相关的调整。


当然也有可能我们在复盘之后发现，诶，这次我们要怎么做？我们相关的结果也得到了一个很好的一个认证，那这这也意味着我们这个压测报告就发出来了，我们本次的压测会话完成了，当然也有可能这个压测报告值，这一次压测一次值，压测出的报告，我们还要多次压测，最终发现我们这个报告达成了我们的规划，然后我们就要重新去做一个新的规划去树立我们是不是要有个一个新的场景，然后去到达成一个新的目标。


所以在整个压测的执行过程里，其实就是从压测规划到压测报告这样整个的一个周期，它中间可能会有很多的闭环，比如说从压测计划到调整计划，这里可能需要经过率很多次，这意味着我们周期性的压测的时候，当我们上一次没有达成目标的时候，每一次就是在于我们执行完之后去复盘，复盘去改代码，去改我们的系统，然后重新去调整计划去执行。


直到我们的系统达成了我们的预期，这时候就意味着我们就要去调整我们的亚瑟绘画了，这就是一个相对简单的一个压测执行的过程，大家要先理解整个过程这样的一个走向。在下一小节我们就会去聊细节的关于监控与调整的一些事情，我们去聊我们到底要监控哪些指标，然后在什么情况下去做怎样的调整。好了各位未来架构师们，让我们下一节课不见不散。
