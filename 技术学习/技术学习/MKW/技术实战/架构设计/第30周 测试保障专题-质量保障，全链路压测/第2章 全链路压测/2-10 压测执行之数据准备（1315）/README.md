因为没有基础数据就没有整个场景所赖以生存的这样的一个数据架构，那也意味着我们要在压测之前把所有压测需要依赖的这种基础设计去准备好。


首先我们要契合这个场景，去构建这个基础场景的一些数据，包括我们所依赖的这种元数据、维度数据，这些数据需要从正式的这种表中同步过来，因为这类数据通常来说它都没有太大的变化。正式的和压测的从数据的构成来讲是一致的，甚至数据的内容来讲也是一致的，只需要将其同步到对应的影子表中。


而针对依赖数据来说，某些依赖数据则是跟业务相关的，那它则需要通过比如说我们的压测工具，比如说我们带压测开关的后台去创建出来，从而来完成相关的一些依赖数据的一个建设，就是流量数据有了支撑起所有商品的这样的基础数据，那我们也需要去构建一些打入的这种压测流量的这种流量数据，而这些流量数据需要去契合用户请求。


我们围绕这个场景下，那用户会以怎样的一个请求数据的格式，以结构进行相关的请求调用，那我们就意味着需要去制造相关的压测流量数据，那这里第一关键的就是我们的用户数据需要去构建出来，包括用户他的登录以及用户他本身的相关的身份，以及他相关的权限等等，这些需要在压测之前全部都准备好。


其次就是场景的请求数据，这一类则跟场景强相关，比如说他做的是一个衣服的下单数据，那一定是针对于服装的那个商品的相关的一个基础数据，需要围绕着基础数据、中学数据、维度数据和相关的这种实体，依赖所有的业务数据相结合起来，共同构建出这样的场景数据。所以通过手去创建这些数据肯定不现实，我们肯定要基于相关的工具或者后台来完成相关的数据的一个 mock 与创建。然后还有就是关联一些信息数据，就是我们这个请求它肯定会关联一堆的ID，那这些 ID 都需要从影子表里把我们这种压测相关联的这些数据全部集合起来，共同构建出一个合法的请求数据，这个数据能够在我们的整个流量拓扑中从头流到尾，并且能够正常地完成其业务功能。


那么在基础数据中我们就针对这个压测场景搭建的这些基础数据做一下展开的讲解，这里我们会有几个点需要去关注一下。

而且所有的场景数据都需要记影子表或者影子存储，也意味着像缓存，像数据库，像中间件，只要我们需要它来支撑我们整体的一个压测场景，那我们都需要提前建设。比如说我们在 Redis 中有一些长缓存，而这些缓存是可以构建出我们的压测场景的，那也我们也需要在压测之前将其预热到缓存中去，从而能够完成相关的场景的支撑。


其次压测场景的数据需要得到保障，元数据的建设也需要完整准确，这里需要关注的就是数据的可用性与完整性，也意味着比如说我们依赖了一些地理位置，那我们可能就需要将各所有的城市线，这种城市的国标码都需要同步到影子存储中，整个的这个数据要需要完整。它不能是说，唉，我压测只有这些，那我很多同学都缺失，那这种情况下也意味着即使你的链路跑下来了，它到时候跟真实线上的一个去比较的时候，它也是存在哪些问题的。第三就是数据需要跟线上的数据量级一致，我们不求它数据量完全一样，但它数据量级要是一致的，你不能是说我的压测里就是几十条数据，而我线上有近百万条数据，那这时候它的 DB 的性能是完全不一样的，也意味着我们针对于这种维度和二方依赖也需要跟线上的量级尽量是一致的。


最后就关于分布，就是我们的数据的分布要可以线上一致，比如说我们地域单元化了，我们北京区域和上海区域和成都区域，它是三个区域单元，那每个区域单元可能就会有面向于当前区域的不同的这样的商品和库存。那也意味着我们在压测的时候，我们的压测影子是存储中的，相关的数据也要跟线上在地域分布上是一致的。并且比如说在场景细分发之后，在不同的场景下，它的对应的数据分布跟线上也尽量是一致，这样子我们就能达成什么呢？就是我们的压测数据也是具备很高的真实性，并且因为所有的数据都在影子存储中，所以它也具备了非常好的隔离性。


然后我们额外还要注意就是热点也要分散，它跟线上尽量也是一致，线上可能将可能热点的商品打散到了不同的表中，那在影子存储中也需要将它打进了不同的表中，从而来换取同等的这样的一个热点效率。

那我们就要考虑压测要不要覆盖，如果压测覆盖，那需要二方去建立相关的数据，并且匹配我们本身的压测数据，而如果二方mock，那依赖方则进行自我数据的mock。还第三种就是压测方去mock，也意味着流量不达到 r 防，完全由压测方自身内部消化对应的一个请求，所以这时候就压测方自己去 mock 二方可能会返回的这个数据来完成业务的下行。而除了二方之外，还会有三方的数据，三方的数据基本上来说对于外部的流量是隔离的，所以数据需要进根据请求 mock 返回，而且并且 mock 仅限于压测链路，对于正式链路来说不应该有外部的mock，否则的话也意味着它可能会为我们带来一些线上莫名其妙的一个行为，甚至说带来一些故障。


我们需要有一些外部触达的配置，允许配置外部数据的触达，这是什么呢？这是说当我们跟三方进行联合压测的时候，我们就允许我们的压测相关的请求能够打到第三方，因为这个时候是双方合作的一个结果，所以外部触达也是有必要的。

所以我们首先要做好就是压测的用户数据需要建立好，需要在用户中心中去创造压测专门使用的数据，并且创建虚假的登录态。这里为什么说是虚假的动物态？因为它有很多的定制，像在本篇章之前也提到了，我们针对用户数据来说，需要去构建一个长效的一个登录态，甚至说这个登录态只能用于压测和真实的这样的用户动态是存在的差别，所以我把它称为虚假的登录态，就是系统特殊性的压测用户ID，并且并不校验用户信息。


这类就是指我是一个压测，但是我不在用户中心中去生成了，压根儿连用户这个 ID 都是假的，都是我这个业务系统自己去生成，但是我业务系统不把它传到用户中心去校验，也意味着它是什么并不重要，它只是了作为这个用户标识。其次就是压测的业务数据，针对业务数据来说，我们需要根据压测影子数据来生成压测流量数据，也意味着我们要需要基于基础数据去产生流量数据，流量数据和基础数据需要去强关联，其次就是根据实际请求数据去替换ID，也意味着我们这个场景下用户实际操作后产生的这样一个数据结构，我们要把它拿过来，它能真实的是一致的。然后其中的所有的相关 ID 我们要替换为影子表中的这个ID，然后数据能不重复，我们尽量不重复，这样子的话能够匹配出现成的这个场景，所以不重复性和偏用户属性以契合。

比如说针对于无限压测和有限压测在数据量上就会存在差别，你无限压测肯定数据量就非常大的，我的库存 10 个亿，我会永远压不完，这就是个无限压测，而这对于有限压测来说它就会面临着我会有一部分失败的请求，因为我的库存就1万，我当我库存受弃了之后，后续的所有的请求就应该失败。如果请求成功反而是个问题，并且我们需要在后续去校验这个一致性库存未消耗完全链路一致性，要通过库存消耗完了全链路一致性，也要通过这样的一致性检测，才能够保证我们整个系统在数据的一致性上能够通过我们的考验。


其次就是数据的准确，第一点就是压测的这个场景数据是要完整的，第二就是压测请求数据也要准确，这也意味着对于基础数据和对流量数据都需要尽可能的完整与准确，这一点值和数据量中的有限，下次的一致性检测二者是相辅相成的。其三就是数据也不重复，我们尽量不，因为我们要有大量的数据重复，我们希望在链路上做一些定制，然后让这个重复数据能够正确的下去。这个其实对于我们来说并不好，因为它只是减小了压测的一个复杂度，但是却为我们的链路中引入了定制。链路的定值在未来就可能会产生坑，因为压测的人可能后面会有变化，会有转岗，会有离职，但是后续接手的人并不一定知道，他因为为了降低压测的复杂度而修改了链路，在链路中引入了定制需要额外的操作才能够开启或者关闭，而这是一旦出现问题就可能为线上带来故障，所以这是我们不希望的。


当然数据创建和使用其实也是很复杂的，所以我们也希望通过构建一个数据工厂来生产一些数据工具，包括一些一键式的工具来帮助我们去制造这些数据，它能够让我们把一些数据能够复用起来，让我们快速地生产出一次压测所需要的这些数据，从而降低数据储备的一个工作量与复杂度。
