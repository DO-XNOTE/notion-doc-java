# 2-5 链路改造之中间件改造（0704）

hello，慕课榜的各位未来架构师们，欢迎回来，在上一小节里我们为大家介绍了关于流量相关的改造，那流量相关就包括流量的介入，流量的传递以及外部的保护。流量进入是整个压测过程中非常关键的一环，因为只有流量能够正常的进入系统里，这来我们才能够完成后续所有的事情。而流量传递则更多的聚焦在应用之间，模块之间，关于这个子调用的流量的一个传递。那么在这一小节里，我们就将补齐上一节课没有讲到的一块关于中间件相关的一个改造，那中间件其实主要我们常用的就是这三类，缓存、消息队列和异步任务。那这三类对大家来说都是非常熟悉的，

包括缓存，像Redis，像一队列，像卡夫卡，这些都是我们的日常非常用的这种中间件，那围绕着这些中间件我们也要做相关的压测改造，为什么呢？因为我们也需要将我们的流量打到对应的缓存中，或者对应的中间件中，那其他的模块从中间件中读取的时候，在压测流量下也只能去读取压测相关的这种数据，而不能产生图串了。


你不能说是我 a 往中间线里写了一个数据，我 b 去读的时候就丢掉了压测标，然后读到了正式的数据，那这样的话可能链路就报错了，所以这也意味着我们要针对全链路做中间件相关代码改造。那改造主要也是在于第一个点就是隔离压测数据与正式数据有差异化，因为像缓存，像消息队列这种，它跟数据库不一样，数据库是一个单独的表，作为差异化的一个存储，那它是个非常清晰的具隔离性的存储。但是将缓存这种来说，它并不具备独立的空间，那也意味着我们要在 k 上去做一些事情。
而第二个也意味着传递，因为中间界也会牵扯到流量的传递，比如异步任务，它也会牵扯到你将任务提交上去的时候，它再转发出去的时候。压测流量的异步任务也要带着压测标，从而能够让下游在处理的时候也是按照压测的逻辑去处理。所以这就是我们在这一小节中需要让大家明白的，我们要怎么样子完成中间件的改造，从而能够在压测流量下正确的使用这些中间件。

像缓存我们更多的是以 k 作为我们的主要的数据标识，那也意味着我们在压测的流量下，我们需要增加固定的前缀，而正式流量我们就不需要。这样子的话，面对于这种固定前缀的k，我们就要剥离前缀，然后作为一个 k 的一个使用，但是在存储的时候需要把这个前缀给加上来，而这个环节里面我们需要对 client 做一下包装，为什么呢？因为我们肯定不希望我们在代码里充斥着 if else 这样的判断，那我们将这 client 做了包装之后，大家都用同样的一个 client 就可以完成，整个世界是透明的。


在这个克拉的包装中，我们第一要做好链路的判断，根据的 local 中存储的特定标识来判断是否是压测。第二就是针对数据操作，那肯定包括读和写，对于读来说就是我们将读取到的这个值，然后玻璃前缀返回。这也意味着我们在读的时候，我们要根据我们的 EI 代码里传工作的 k 再加，如果判断有这个压测标，那我们就要往前面加上这个固定前缀，然后去缓存中去读，读到了之后真正返回给业务代码的时候，将这个 k 要剥离掉前缀。而针对写的时候，我们要根据判断当前是否是 2366 自动增加这个前缀。


通过这样的方式其实也意味着整个业务代码在使用中间件的时候，它完全是透明的，它不需要关注我是否是在压测流量去使用这个中间件，我仅仅在于我引用到了支持压测的 client 包即可，那间接着去讲，

那在中间线中也会牵扯到这个数据传递，通过中间件触达到下一个逻辑节点。


那也意味着我们在中间件，比如说像业务任务、消息队列的时候，我们也要做好这样的 client 封装。为什么还是client？我们也是希望我们在使用异步任务，在使用消息队列的时候，我们也不希望做太多的 FLS 判断在这里，因为它不完全是一个 k 了，它肯定是一个提交的一个对象，那也意味着我们要做好这个数据的封装，把它做一个信封，上面盖上压测的标识，然后将所有的东西封装到这个对象中去，然后在链路中状态。


也是说在链路读到这个消息或者业务执行这个任务的时候，根据这个对象中的标记还原到链路中去。这样子的话也能起到针对这个压测流量业务代码不需要去感知中间件上的差异化的逻辑，而由中间件的 client 来完成这个事情。当然我们还有更多的多样化的中间件，

当然也意味着针对这个中间件，我们要结合它的数据存储的协议和格式来完成这个数据存储的一个隔离。这里最重要的比其实主要两个点， k 隔离和空间隔离，或者是围绕着这种，嗯，没有 k 也没有空间，那我们就是套数以格式再包装一层，这样差异化存储之后磨灭便于创建也便于销毁，不会污染真实的数据。


其次在整个链路状态传递中也要注意，就是压测标记需要封装好，链路状态需要还原，从而能够将压测信息沿着业务链路不断下行，传到下一个节点。所以其实针对中间件来说反而比较简单，我们做的就是一劳永逸的事情。我们将 Claude 的包封装好，比如Redis，就那么多API，然后将其封装好，然后传到，我们在仓库中，然后大家引用正确的版本即可。所以中间件改造就是这么简单，那使用起来也没有任何的感知，那肯定是我们也是最希望的结局了。


好，那关于中间件改造就讲这么多，在下一小节中呢，我们还继续聊链路改造，聊到的是数据隔离，流量产生了隔离了，中间件也隔离了，我们的数据也必须要隔离，通过将这些都隔离开，我们才能够真正的将所有的业务在压测和政策下完全的隔离开来。好，各位未来架构师们，我们下小节不见不散。
