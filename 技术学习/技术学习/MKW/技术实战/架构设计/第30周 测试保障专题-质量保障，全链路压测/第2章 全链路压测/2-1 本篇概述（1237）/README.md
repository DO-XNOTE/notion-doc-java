首先还是从请求出发，我们要去制造一些线上的这种流量请求，然后它可能就会打到我们的入口服务a，这种入口服务a，然后会调到相关的 b 和c， d 可能还会继续的调到，比如说服务 d 则会调到外部去了。比如说我们的商品它本身的服务提供商，有一部分商品是外部的提供商提供在我们平台挂货售卖的，那这时候我们的最终库存扣减就要扣到外部的系统里去了。


但是我们肯定觉得我们在压测的过程中，我们肯定不能把支持流量打到外部依赖里去，不然的话也许我们没挂友商挂了，因为他不知道我们要压测，其实我们提前通知他压测了，压测所制造的这些数据对他来说也是一个干扰，不管是数据的是准备还是实际上压测带来的数据结果对于他们来说都是一个负担。那么我们是否可以说在最后的时候将外部依赖 mock 掉，这样子的话整个过程就不会是影响到我们的外部系统了。所以中间的这一些环节就是我们全链路压测的重点。我们要探寻从服务的ABCD，它们是否能够承载如此高的流量，他们在流量达到极限的时候表现出来的服务的一个状态是什么样子的，有没有出现一些问题，有没有报一些错误，那这都是我们在全链路压测中希望去探索，去发现的。


当然除了服务之外，我们还希望去探索我们的数据库，包括数据库的响应能力，它在这样的压力下是否仍然能够保证良好的性能？我们的缓存，我们的缓存设计是否合理？我们的缓存是否能够承受这样的压力？以及我们整个的一个消息体系，我们的Kafka，我们的 Rocket MQ、 rapid MQ，他们是否能够在这种压力下仍然保持着良好的一个重大率以及其效率？当然还有一些异步任务，这些异步任务是否能够正常的执行，

其次就在极限水位下我们系统运用能有哪些表现？它是表现的一些正常，还是它已经触发了某些降级？还是说它已经开始出现一些问题了，用户的体验受到影响，甚至连成单都已经成不了了？那这就是我们需要在流量评估这个环节去发现，去探寻。第二就是去做问题发现，借着极限下的表现，我们去找我们系统中是否存在一些问题，那比如说它是否存在了某些隐患，这些隐患只有当系统达到瓶颈的时候，达到极限的时候它才能够真正的爆发出来，将这些隐患挖掘出来，干掉他们。那这也可以保证我们系统在流量来的时候，不被这些隐患拖到故障中去。


其次就是瓶颈的调优，因为每整个系统来说，可能只有某些环节存在着较大的瓶颈，那针对这些瓶颈的调优可能就能极大的改善我们系统的性能，那压测也是帮助我们发现这些瓶颈，然后去针对性的解决这些瓶颈的过程。当然还有就是容量规划，因为我们已经评估了我们的容量，我们也通过问题发现找出了我们的问题，并且把它们修复了，然后我们就叫规划，我们的线上到底要放多少资源，我们要放多少的机器，我们要用多强的数据库，甚至说我们是否要将我们的一些资源做拆分，然后做扩容。这些都是在容量规划这个环节我们需要去考虑的。


其次就是限流的设定，资源已经到达极限了，我没有更多的钱，甚至说有钱也买不到更多的资源了，那在这时候我们怎么样子去设置我们的限流？通过一个限流数值的设定保护我们系统，它最多就这么多了，你再来我就直接给你报错了，用户明确的告诉他，你现在钱包拥堵，请用户稍微等一等再试，通过这样的方式再给一个用户一个友好的提示，从而保护我们本身的系统不因过大的流量而出垮。
