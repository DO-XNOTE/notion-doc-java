其包括整个压力的一个变化过程以及在压力下的一个系统反应。然后要明确这个异常和问题，要明确出在整个过程中出现了哪些异常，在最终的压测结束之后，他留下了哪些问题需要去跟进，然后压测的数据也要相关沉淀下来，包括我们是不是每次都在比上次要好，我们是不是在进步？其次就是一些复盘的一些资料都要留下来。


所以这里的压测数据沉淀不是指我们在压测准备的时候的那些数据，而是指每一次压测过程中我们都沉淀下来了一些数据，这些数据能够证明我们压测在一次比一次好，以及后续改进的凭据，能够明确出我们到底有哪些事情需要去立一个专项。比如说关于性能调优，数据库慢、 SQL 慢、代码算法性能不够，那这些可能都需要立一个专项去调整。然后针对其他的要明确出优化方向，我们到底要朝哪个方向去优化，从而让我们的系统变得更加的完善，变得更加的快，那这些就是一个压测报告所需要报告的内容。那我们更细的去看就是一个压测报告会从压力的场景，到列入状态，到压力信息，到压力的调整以及最终的异常信息。

我们要在压测报告中描述出本次压测我们的目标场景是什么样，让看报告的人能够很清晰的知道我们这次压的是什么场景，要压测范围是什么，所以它就要包括我们的业务场景，我们所压测的对应的链路，我们压测的模式是什么？这里的模式比如说是一个阶梯性上增流量的，还是一个瞬时流量起高然后下降的这样一个模式，等等。然后我们这次压测的目标是什么？压测持续了多少时间，以及相关的执行人，那通过这些就可以让看报告的人能有一个很直白的信息的一个获取，它能够知道本次压测的背景以及其主要的一个目的与执行情况。然后就会牵扯到在压测场景之后的我们的链路状态。在链路状态中重要描述出串联链路的压测情况和主要的业务指标，
