那我们先来看一些数据，就是在大锤的公司，双十一的时候，用户量是平时的 5 倍，那订单量是平时的 10 倍，请求量是平时的 15 倍，为什么订单量的倍数比用户的倍数多？因为每个人买的更多了，那为什么请求量更多？因为每个用户在买的时候他都还要反复的比较一下，他要看这个，看那个，他看得更多，所以请求量是大于订单量，订单量是大于用户数，是这样的一个状态，那在这样的一个状态下怎么样保证我们的系统不挂？这是架构师最大的一个挑战，所以对于大锤来说，它就发出了这样的灵魂。


三问，我们的系统能承载吗？我们要扩容多少资源？我们的限流应该说值多少？其实这个对于他来说目前都是不知道的，因为谁也没有经历过这么高的流量，谁也不知道我们要扩多少容，谁也不知道我们要设置多少限流，两眼一抹黑的情况下去赌这个肯定不是一个明智的一个决定，我们肯定要把这个数据拿出来，我们要知道我们的系统极限是什么？我们当前的资源能承载多少的流量？如果我们要达到预期流量，我们要扩多少揉，我们要在其资源水位下去验证这个流量，那这都是我们在做全链路压测的时候，我们需要去做的事情。

而且我们要做哪些事儿？第一还是这个容量的一个评估，明确极限水位，明确系统在极限下的表现。第二就还是去发现我们的问题，因为我们没有经历过那么大的流量，所以一旦流量到了或者接近那个水位的时候，可能很多之前没有被发现的问题都被发现出来了，然后系统中存在的一些瓶颈也随着流量的增加开始出现，如果不解决这些瓶颈，我们是没有办法达到这个流量的，扩再多机器可能也没有用。所以最终围绕着评估好的这样的一个水位，以及根据极限表现去明确的安全水位，再去把这种隐患解决掉，把这瓶颈解决掉，最终才能够产生出我们的容量规划。
我们要买多少机器，用多强的DB，多强的缓存，以及我们的限流要设置到多少，那这就是我们刚才那个离婚 3 分钟的两个问就在这了，我们要扩大扩多少机器就是我们容量规划所要明确的，我们的限流是多少也是我们的容量规划所要明确的。那在容量规划之前，我们依旧要做好我们的容量评估和问题发现，那这就是我们在压测过程中短期的或者是说局部的目标。


那进一步的去看压测其实也是需要一个平衡的，所有的事都是既要又要，还要，我们既要能够去达成第一个关键词匹配，也要去做好我们的第二个关键词隔离。

什么是匹配？什么是隔离？对于匹配来说我们要能够匹配真实的场景，但是同时我们还要将这些真实场景转化为我们的测试场景，因为我们没有办法真的在真实场景中去演练，我们一定是在测试场景中去模拟这个真实场景，然后去完成相关的压测的，那同理也需要有对应的真实的数据，和我们的测试数据是相关是匹配的，我们不能够造一些跟真实完全脱离这种数据。


我们有没有办法拿真实数据直接来用？我们肯定是要围绕着我们的测试基准的这种基础数据去制造相关的流量数据，去尽可能的让我们测试场景契合我们的真实场景，让我们的测试数据契合我们的真实数据，但是这二者之间一定是什么呢？这是第二个关键词，隔离的压测数据和真实数据之间一定是隔离的，压测流量和真实流量之间也一定是隔离的，所以就会发现我们既要有它俩很像，我们又要让他们隔离，为什么呢？因为只有匹配我们才能够真实的去发现在未来要达到的那个场景里，我们的系统表现是什么样子。而隔离它才能够帮助我们不影响我们的真实用户，不影响我们的真实功能，同时也能够更容易的去做好相关的清理。

我们需要逐步的去探寻我们系统所能承载的这样一个极限，所以我们每次是要明确在当前压测里面的我们的末高目要是什么，要逐步向目标去迈进，这一次的目标可能达不成没有关系，我们发现了问题解决了问题也可以在下一次我们就能达成。下一次没有达成也没有关系，我们只要每一次都比上一次好，我们总有一天不会达成。


其次就是系统指标，我们要明确压测中系统指标处于一个合理的 Vue 范围，我们的 load 不能超过5，好，我们的 CPU 不能超过80%，那这些都是我们针对系统指标的给出的一个目标范围，那如果我们在压根的时候这个指标直接爆表了，那也意味着我们本次压测就发现这个问题，也意味着我们在后续的 IK 群中就需要去解决这个问题。所以在下一次压测中，我们还是要更将我们的系统指标约束在一个合理的目标范围。


其次就是整个这个稳性能，稳定性的要求我们在这个流量承载水平和系统处性能都要处于一个稳定的一个状态。我们流量一直是 1000 QPS，保持住的时候，整个的系统性能是一个稳定的，它不会因为时间的延续而开始变差，这个是我们要达到的一个目标。进而还有一个就是在数据层面上的这样的一个目标，就是数据的一致性。
大流量下全链路的各模块的数据最终是一致的，我们不能说流量一大，我们从一致性就被破坏了， a 模块 b 模块最终出于对不齐了，那这也意味着在真实流量来临的时候，这就是个灾难，因为我没有办法明确这个数据到底应该是什么样子，最后靠人去盯，那这是真的是一个灾难。


最终还有一个就是系统的恢复性，压测停止后，我们系统能否快速的恢复到压测型的状态，它自身是否具备这样一个快速恢复的能力？这个点不仅仅在压测的时候，它是我们考核点，它也能够让我们感知到在未来流量来了又走了之后，我们系统能否快速的恢复到流量来之前的一个状态。


因为只有恢复到这个状态的时候，我们才能够根据后续的流量来去规划我们整体的资源容量，我们是不是可以缩容了？双 11 过去了，我们是否都缩容了？如果系统恢复不回去，那意味着我们一旦缩容它就会出现问题，这都是我们在整个压测过程中需要去考虑的，那所以这些都是我们在压测的小目标，细分领域下的目标，


而针对这些目标我们也最终能够明确出压测场景，那它就是我们的在场景选择上的目标。

第一重点业务场景，第二热点业务场景，第三就是关联业务场景。重点和热点当然不用说了，一个是我们系统本身我们对其认定的这样的一最重点，第二个就是用户他们用脚去选择的最热点的这样的一个场景，这两个场景可能往往是一个，这也有可能会有一点点偏差，以及在两个场景它所关联的一些业务场景，比如说想下单可能就会牵扯到我们会将一些消息推送到履约系统，或者去推到配送部门，那这些就是配送部门，它就是关联的这样的业务场景要明确相关的场景边界。


压测它也要明确这个边界是什么。我们不能说一切胡乱压，一定是要明确这个边界是什么样子，确定好我们到底要压哪个点，通过单点来明确出在这个场景下，我们这个压测它达成的一个结果和这个场景是关联的，明确的，而且是一个很聚焦的这样的一个场景，这样子的话压测结果才有意义。


第二就是场景细分，我们一次只加一个点，它可以帮助我们排除干扰，明确在这样一个点下它的局限性的表现是什么。当然除了细分之外，我们也要做混合，细分是能够探寻单个链路、单个场景的，而混合它能够帮助我们尽可能去模拟现实，来明确在多个场景互相干扰下整个系统它的一个表现。


好，那通过上一页和这一页这两页 PPT 其实就是我们的整个压测过程中很明确的两个目标。第一，通过不断的去探寻我们的流量、系统指标性能稳定性、数据一致性和系统的一个这样的一个恢复性。通过这些就是我们在每一次压测中需要去关注，需要去达成的这样的一个目标，所有的目标头上都挂着一个大的目标，就是我们本次要达到的一个极限以及极限下我们期望的表现。那关联出来不可能是空哑，关联出来的业务场景就是我们对应的场景目标，这二者结合起来就是我们整个全链路压测所明确出来的这样的一个业务加技术结合起来的目标。
