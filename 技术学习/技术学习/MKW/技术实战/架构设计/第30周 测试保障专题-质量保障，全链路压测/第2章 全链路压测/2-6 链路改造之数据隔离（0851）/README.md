# 2-6 链路改造之数据隔离（0851）

hello，各位未来加工师们，欢迎回来，在上一小节里，我们为大家讲解了关于链路改造中的中间件。改造其实是个一劳永奴的事，我们更多的还是围绕着 Claude 包的一个封装，将处理好的 client 提交到公司的仓库中去，然后大家就可以引用正确的版本即可。这样的方式既一方面可以一次改动处使用第二个方，也可以避免了在链路中做大量的判断，降低了改造的工作量。

那作为数据隔离都包括哪些？第一肯定的基础的数据，其中基础数据就包括我们一些在线和一些离线的这种基础性的数据，保证我们所有的业务系统能够正常的运行起来。其二是中间数据，中间数据是指我们当流量传入进来之后，所有的写在中间状态的，比如人类存中的，像缓存中的，还有一些就是我们业务中间状态下写入存储的一些数据，那这也是我们的中间数据。第三类就是结果数据，结果数据是我们最终完成了整个的一个业务场景之后，针对十一结果的一个数据展现。以及我们可能还会有学生交互，比如说下钻呐，我们的上卷通过这种交互的方式来进行多样化的数据处理，那这些就是结果的数据。那针对压测来说，我们这三类数据都要完成隔离，这也意味着我们每一块都要针对性的去看，

比如说针对基础数据，而针对战线存储来说，我们大部分数据还是 MySQL 中的，它是通过影子存储这样的方式来完成正式压测二者的隔离。针对这二者革命来说，表结构其实二者是一致的。而针对数据来说，那只会有正式的和影子两套，通过数据的差异化来完成差异化的表征，也因为其影子存储的存在，所以我们也可以实时的完成对应的交互。


而针对 ODPS 这种离线数据来说，它建立完整的一整套影子下的这样的一个数据计算，其实是很复杂的，那么我们也不希望花这么多的工作量去建的话，我们其实只要建好最终的产出的这样的影子存储，然后压制机不进行生产，所有的数据前置性的准备好，在数据准备阶段完成一次性的工作，把所有跟压测相关的结果灌入到对应的结果表中，然后提供给在线系统去使用即可。同时压测的相关的数据也不会周期性的介入你离线计算，避免一方面浪费了我们的算力，另一方面阐述出 5 位的这样子结果。但是通过这些我们也需要在准备阶段把基础数据准备好，它是支撑我们在压测的时候整个场景的一个支撑。

我们首先就要有一个库表同步的工具，因为针对 MySQL 来说，它的正式和影子是分离的，但极有可能我们正式表改了结构，其实对于影子表来说他并不知道，所以可能两边结构就对不齐了。还有一种就是正式表套有大量的数据，现在影子表你去创建这数据很麻烦，是不是？对于有一些基础性的数据，我们可以一次性的从正式表里同步到影子表里，但是二者需要做一些差异化的区分，比如说 ID 段发生了变化，比如说类型发生一些变化，从而能够完成二者的隔离，同时又降低我们创造数据的一个基础的工作量。


这里主要包括两类，有一个就是同步表结构，像我们很多时候建了一个正式的 user 表，我们建立一个压测的 user 表，那其实这个结构完全可以通过同步过来，还有索引，因为可能随着我们线上的不断优化，我们给正式表加了索引，影子表也通过这个方式一键同步。


其次就是数据，这里不是所有数据都能同步，但是很多数据其实是可以复用的，比如像维度数据，比如说它有一些关联性的数据，甚至像维度数据，它叫国家的这一类中国标码完全可以从这处同步到影子，因为二者之间没有任何的差别。还有一些就是通过一些关联，我们可以 ID 发生变化，但是数据的含义是类似的，那有一个工具就比我们通过人去一点点去创建要容易的多了。其次还有像一些后台的影子开关，这些影子开关可以帮助我们通过界面，通过我们的软件功能来读写影子数据，这样子的话它就能够更好的让我们去维护我们的基础数据，从而完成一统的这样的一个影子数据管理。通过工具加后台关，然后后台关一旦开启了，我们就可以在后台上去管理我们的数据，通过这样的方式，很多基础性的数据就可以在压测之前很容易的就完成了。当然这里也是要考虑额外的几个点，比如像热点的分散，我们的库存，那肯定牵扯到商品，那商品如果没有做好热点分散，所有的热点商品都在一个库里，那压测出来的结果也不一定是对的。


其次就是大表的一个管理，因为压测表特别容易产生大表，尤其是我们做一晚上的压测，可能产生了几千万甚至上亿的数据，如果没有针对这个大表做好相关的管理，没有清除掉历史压测数据，可能压测出来的结果也并不一定能够真实反映出线上系统它真实在流量下的表现。所以影子的数据也需要持续的优化，从而能够保证我们在压测的时候能够真实地体现出线上的真实场景，并且和业务链路可以保持一致。

第一类，比如说共享内存，那因为压测带标，所以我们在整个链路中需要去针对压测链路做好相关的判定，这样子的话就可以避免说我们在共享内存中压测链路影响到了正式链路，或者正式链路影响到压测链路。而针对静态数据我们还是要尽可能去避免修改静态数据，因为静态的变量它是被所有链路共享的，如果你在压测链路上你去改了一个 static 的一个变量，那很有可能正式链路在来的时候就读到这个东西了。所以我们针对压测链路，如果真的要用它，就必须针对压测链路单独设置静态变量。所以最好的解还是说我们压根儿就不操作静态数据，因为操作静态数据它可能真的是埋下了很大的雷，而且这些雷一旦出现了问题也很不好去排查。第三就是肢体缓存的数据，就是尽量做好压测的缓存的隔离，然后针对对应的 k 做特殊处理。这个就其实在中间件环节我们讲到了本身来， client 做了包装之后，这一块对于我们的业务代码来说，它本身就是透明的，当然最后一环就是针对这个结果数据我们除了看之外，
