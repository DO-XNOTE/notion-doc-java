OK，除了这些维度以外，咱这里还列出了一个领域模型的划分，这个知识点在张飞扬老师的领域建模当中会着重的讲解。那同学们学完了那个课程之后，结合我们这一章当中跟同学们介绍的其他维度，来来一起充实这个话题回答的方向。OK，那这是一个话题，第二个问题呢，有些面试官比较喜欢就一些高并发场景，同学们发问，比如说如果遇到突发流量，你怎么来处理？什么是突发流量？就是可能在某一个时间点，一个特定的时间点，突然从 0 达到一个很高的数值的这种突发流量。比方说双十一的下单场景，OK，那在 0. 0 秒的时候会有一个非常高的流量峰值，那么对于突发流量的解决方案，我们在前面章节当中也有详细的介绍，这里一个比较核心的方式就是流量整形。流量整形是最有效的一个方案，因为它直接应用在了突发流量发生的那个档口，它是在你整个流量路径的最入口起点的地方，把它给处理掉、消化掉。OK，那么流量整形，那么这里就可以就限流和流量预热两个方案来进行讨论。从这里可以看出，其实问的每个问题它都有我自己想要听到的某些答案，所以回答面试官的问题就像做阅读理解一样，这个题有两个主要的关键字，一个是流量，第二个是突发。


OK，那如果我把这个问题点回答到了这个流量路径的后半部分，比如说我想采用弹性计算，增加你集群的计算能力，增加它的机器资源去消化这些流量，那这个答案可不可以？当然可以，它其实就是高可用高并发当中的一个处理方式，但是对于这个问题来说并不是非常的契合。


所以从这个提问的方式，流量和突发，那我们可以揣摩一下，其实面试官想听到的一个方案是从流量的整形这个方向，在流量的入口处，怎么样用最经济有效的方式把突发流量给解决好？OK，那接下来再分享第三个面试问题是微服务的一致性怎么考虑？那咱的微服务被拆分成了很多个不同的组件，组件和组件之间如果产生了不一致，数据不一致应该怎么来处理？OK，那这里我们就可以搬出这样几个方案。


第一个是 base 理论， base 理论咱前面有学习过，相对于一致性来说，它是可用性优先的一个方案。最终一致性，同时最终一致性也是各大一二线互联网公司在自己的核心高并发应用当中采用的一种方式。那既然是最终一致性，那么它最终一定要一致。怎样来保证确保最终一致呢？很多种其他的方案，比方说我可以上分布式事务，我们在后面的小节当中将会落地一个阿里巴巴的 Sita 组件，来跟同学们去演示分布式事物的方案。OK，除了这个以外，我们还可以使用分布式锁，或者在你的接口层面，在业务逻辑里确保它的幂等性。同时我们还可以用一些补偿 job 来批量的去查漏补缺，来补齐这个数据不一致的问题。OK，那这是微富一致性当中几个比较主流的方案，那你把这几个方案说一说，面试官对你刮目相看，知识面很宽。好，我们再来看最后一个面试问题。比如说我现在面试的这个团队，它有一个APP，这个 APP 发版非常的频繁。OK，那面试官问了，我的手机 APP 升级了版本，但是我的老的应用还在用户手机里面跑，

服务的拆分，那其中这里应用的比较多的是领域模型。OK，领域建模，那这部分请同学们结合张飞扬老师的领域建模的课程，那在学完那部分课程之后，相信大家对领域建模都会有一个比较系统的认识，然后再来应用到服务拆分的领域当中。OK，第二个问题，我这里问同学们，假如咱脱离了分布式事务，你还能想到多少在微富环境当中保持一致性的方案？为什么我说要脱离分布式事物？其实分布式事物我这里跟同学们讲一个事实，它即使在一线大厂当中，它的应用也没有同学们想象的这么广泛，为什么呢？这里后面注意到一个是分布式事物，它的实施的成本，还有它本身对你业务可能带来的复杂度。从这两个方案来考虑，我们如果能用一些比较廉价的技术手段来确保一致性，那么本身是并不倾向于使用分布式事物的。


OK，那么我们后面会通过 Seata 里面的 TCC 方案，也就是蚂蚁金服的核心分布式事务解决方案，使用它跟同学们做一个落地，让大家体会到一下什么是给业务带来的复杂度。OK，那至于成本，其实 Sita 框架里面也提供了一些对业务无侵入式的方案，但是毕竟它需要部署一些中间件来保证一致性，所以对你应用的成本也是一个增加，这个问题就靠同学们自己去摸索了。好，那最后一个小作业，我们这里去做一些涉猎研究，一些目前公开的技术文章分享等等，去了解熟悉一下大厂里面的微服务实践，他们是如何来拆分自己的服务的？因为领域模型它其实是一个比较理论化的东西，理论的出发点还都是很美好的，但是应用在实践当中，我们总归会做一些妥协，所以让同学们去研究一些公开的技术文章分享等等。这样的话，即使没有在一二线互联网公司工作过，那么呢，也能对他们的微负拆分与技术演进知晓一二。就相当于咱没吃过猪肉，看看猪跑总是可以的。对啊，好同学们，那这一章的内容就到这里结束了，我们下一章节再见。
