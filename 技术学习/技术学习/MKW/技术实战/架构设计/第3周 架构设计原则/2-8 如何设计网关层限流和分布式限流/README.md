那第一个访问频率，第二个 IP 和连接数，那我们把这两个象限合并在一块儿来讲，那这两个是我们最常用的限流手段，既可以去根据你服务的 query per second，也就是 QPS 的数量来做一个限流，也可以去设定 IP 维度的限流。那在我们正式的线上环境里，其实通常会设置多个限流维度，那这些限流维度共同来作用。比方说我可以限制一个限流规则，它约定了每一个 IP 每秒访问的频率要小于 10 个请求，并且这同一个 IP 的并发量小于5，那这是一个限流规则。然后我可以再去做另外一个维度的限定，比方说每台硬件机器，它的 QPS 最高承接100，连接数最大只能保持个300。那在真实的线上环境里，我们正是用了这种多层复杂规则共同作用的限流措施。那它就像一个木桶理论，你木桶当中最短的那一个木条决定了你的水位会到达哪里，因此这些限流规则共同作用的时候，那你的用户请求如果触碰了任何一条限流规则，那这条请求都会被限制住。


okay，那么接下来看还有什么规则，哎？黑白名单，那黑白名单是我们很多大公司应用里面非常常见的现行以及特殊放行的手段。那而且这里有一个特殊的地方，黑白名单它是处于一个动态变化的。那举个例子，比如说老师现在是做的支付行业，那支付行业政府部门往往会从合规的角度定期给企业输送一些黑名单限制，那么你处于黑名单上的商户将不能去执行收款、付款等这样的业务流程。那这里是一个例子。


还有个例子就是我们的大型应用经常会受到低 DOS 攻击，比方说某一个 IP 或者某一个 host name，它在一段时间内频繁地发出大量的访问请求，那这就很可能被我们的系统识别成机器人的攻击或者是 dedogs 攻击。那么这时候我们通常会将这个 IP 加入到 IP 黑名单中，那限制他的访问，这就是咱平常经常说的叫什么风IP。那如果记有写过爬虫程序的同学应该有这个体验，很多同学会到知乎上的一些著名的问题下，比如身材好是什么体验？这些话题下爬美女图片，那么你写的爬虫就要怎么样定时的更换IP，这样的话才能绕过它的一些黑名单。

第一个网关层，第二个是从业务层进行限流。那从资源成本的角度来看，同学们觉得哪一层限流它的资源消耗是最少的？网关层对吧？一定是在网关层，因为网关层相当于所有业务流量的一个入口，如果你在入口层把这个流量卡住，那么后台就完全不用响应你这个请求，否则的话你在业务层实际上是需要去对这个流量做一个判断响应来决定它是否要被限流的，因此在网关层做限流它的成本是最低的，但是相对于控制力度来说，通常来讲在业务层限流你可以做更精细粒度的控制。


那我们来看，在网关层访问频率、传输频率、 IP 连接数、黑白名单，这都是非常常见的限流手段。那这些是通常我们采用什么样的技术来实现？最常用的 Nginx 和 Lua 脚本？那这是一个网关层限流的铁搭档，那同时这个组合也是在业界经过真正的实际考验的比较靠谱的一个组合。所以建议同学们在网关层选择限流维度的时候，可以考虑使用 Nginx 层，那它在以上的访问频率、传输速率、连接数、黑白名单等这些维度它都有相应的命令可以去支持。


那当然了， Nginx 只是一种选项。那通常来讲，我们也可能在自己的业务系统中采用其他的网关，比如说像 F5 也是一个非常著名的网关系统，还有像阿里系使用 MTOP 这样的网关层接口，那都是可以达到这样的限流效果的。那这里其实并没有太多的技术含量，完全是依靠你自己的压测的数据结果，以及你对当前业务的理解，根据业务容量来估算你的限流的这个数值。OK，那业务层限流这个玩法可就多了。


同学们，看，业务层这里也有一个网关，那我们把它称作什么呢？业务网关，比方说 Gateway 和Zoom。哎，同学们可能会奇怪，咱前面有一个 Nginx 网关，怎么后面还有一个 Gateway 呢？那为什么我们需要多层网关系统？同学们，这个不用感到奇怪，那在真正的大型商用级应用里面，通常我们是会搭建很多层网关系统，但每层网关它的性能和应用场景都是不同的，像Nginx，我们往往会把它放到比较外层做一个高性能的网关转发，那这里并不会牵扯太多的业务逻辑。而到了业务网关里，像Gateway，那这里我们就可以做一些很精密度的控制。比方说我可以给它做什么呀？不光是限流，还可以给它做降级、熔断，以及非常复杂的谓词判断具有什么特征的请求。比方说你 header 里有什么数值，你的 request 里面有什么参数这类请求我可以把它路由到哪一个微服务组件当中，那我可以做这种非常精细粒度的控制，因此你在业务系统当中的每一层网关，实际上它所要解决的问题是不一样的。


OK，那这里是业务层限流的业务网关，以及我们还有什么呢？还可以借助一些限流组件，那这里我们有很多种不同的选型，比如说现在风头正劲的Sentinel，阿里系的一个非常厉害的流量整形的控件，那么我们在后面的微附章节里也将会跟同学们详细的介绍 Sentinel 的应用。
OK，还有我们可以自己编写一些限流组件。使用什么呀？像 Redis 和 Lua 脚本这样的一个好搭档， Redis 它本身就内置了对 Lua 的知识，那我们在一期架构师课程当中同样也是使用 Redis 和 Lua 这个组合，跟同学们去开发了一个自己的限流控件，通过一个 annotation 就可以给任意服务指定限流规则，非常的轻量级。


OK，那这两个组件Sentinel、 Redis 加Lua，它体现了一个什么呢？同学们，前面我们在开始章节的最开始跟同学们提的一个关键词，分布式限流，它的一个特性是什么？中心化。可以看出Sentinel、 Redis 加 Lua 都是独立于你微服务之外的，那它俩就是一个中心化的限流组件，那可以去汇总你所有服务集群当中的当前访问的数量。那如果我们这里在业务层里面想做一些轻量级的限流，我不想用一些分布式的组件怎么办？那很简单，你可以用一些单机组件，比方说gullava，那瓜里面也提供了非常轻量级的限流的工具类。


OK，那以上就是我们在分布式以及微服务环境当中做限流的可以去考虑的应用点。那从这里我们可以看出网关层和业务层之间的区别，我们前面说到过，那就是你的力度，那你在业务层可以做更精细力度的控制，我们对两者的定位是不同的，网关层我希望它达到的一个效果叫什么呢？ fast fail，快速失败、高性能网关，我们不要牵扯太多的业务逻辑，而在业务层里面，我们可以对一些被限流掉的这些请求做一些降级、熔断等等非常精细粒度的异常处理。
