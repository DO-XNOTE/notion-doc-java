跟你的这个卖家的付款动作息息相关。因此对于卖家支付这样的一个action，我们要做出一系列的响应，去调用上下游的系统来做各种清算的业务知识。比如说哪些系统？我这里随便举一些会计账目的子系统，那这也是相应的微服务，还有什么呀？报表系统，还有你的卖家支付，如果涉及到了退款等等，那我要在自己的账务清算系统当中触发退款流程的返清算。


那咱互联网行业的财务清算系统，它这里需要把这个支付信息这个具体的transaction，把它的数据同步给各个的上下游业务系统，那做数据的冗余，那把它异构到各个不同的子系统微服务当中，那从这一个需求上来看，非常的容易简单并且清晰。那因此只要我们的业务方配合一下，那卖家支付的这个微服务，只要相应的调用下面这几个微服务去调用一把就好了，对吧？看起来非常的简单，但是这个现实不那么的美好。


那有的时候当你对你的兄弟团队说，兄弟请配合我，他们往往就给你说一个字，我配不配合，那这种事情同学们在江湖上漂，你早晚都会摊上这样一种事情，那在互联网行业可能都会比较好疏通，大家都是自己人。那如果你在一些像国企，甚至说在外企网难，这些跨团队的沟通将是一个非常麻烦的事情。那如果你的业务方完全不配合你做这样的改造，一点办法都没有。比如当时他的卖家支付平台给我们的说法就是，我不想对接这么多复杂应用，提高我自己的业务模块的复杂度，我不想这么做。


**同学们，这个案例如果摊到你的头上，在业务方不配合，不给你喂数据的情况下，你怎么来平地起高楼，把这个问题解决掉，这里就用到了我们之前所学过的一个知识点，同学们还记得吗？怎么样把一个问题以大化小，然后去划分职责，最后把它细化，方案解决出来是什么？三段论对不对？**同学们在翻到最早的架构设计章节，去复习一下三段论，那我们用这个三段论去跟大家尝试分析过分布式事故的问题，也分析过天猫的双引擎回归测试框架。那这里呢？哎，老师就跟大家去看一下我当时去带的团队，那是如何来用这样的一个架构的思考方式来解决这个问题的。
