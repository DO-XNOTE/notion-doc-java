从业务角度来出发，这个非常好理解，比如说你设计一套电商系统，对吧？你总有各种各样的像商品服务、订单服务、库存服务或者是营销优惠服务这样的子业务系统，那这每一个服务背后代表着自己的业务体系，并且各个服务与服务之间有自己比较清晰的界限和上下游的划定。OK，那这是从业务角度来思考。


还有从功能维度上来说，比如我们这里拿其中的一个服务向营销服务来打个比方，那说到营销，我们在功能维度上可以把它划分为两个方向，一个是营销优惠计算的规则，第二个是营销优惠券，那这两个业务是可以属于不同的业务线。因此从功能维度上来说，那我们比较倾向于把这两个不同的业务拆分开来再往下细分。比如说我们把刚才说的这个营销优惠券的服务再往下分一个粒度，那么从功能的角度上来说，我们可以把它分作劵模板、优惠劵模板的创建以及领劵服务，那用户从你预定的劵模板当中去领劵，那当然能领多少，那是否有资格领？这都是在劵模板上面的元数据信息来划定。


那以及最后一个服务核销，就是当你的优惠券应用在一个订单上面，那我要把你这个优惠券用掉，那这是核销服务。那么从功能维度上来说，我们只是把它简单的分成了这三块功能，但是在这三块功能之后还有非常更多细分的服务。OK，那这也是做服务拆分的一个维度。


功能维度，最后还有一个维度，从资源维度来考虑，那这个可能同学们平时没太接触过，那当我们需要从资源维度来对服务进行拆分的话，那说明你这套系统已经比较牛掰了，你的用户量基数、访问基数已经到达了一定的层次，这时候就要从资源维度来入手，把一些访问量比较大的服务给它隔离出去。


哪些访问量比较大？比如说你的高频场景和低频场景之间的区分，那是什么叫高频场景？那很简单，商品搜索。那商品搜索看似是处于商品服务这个业务维度，但是它是所有电商场景当中的导流端的入口，访问量是非常大的，因此我们把它定义为一个高频场景。从这个角度出发，这一部分业务将要独立的从商品服务当中拆分出来，独立提供这种服务，那这是从资源保护的角度出发，不希望这些高频场景它过多的消耗资源，然后影响到其他的业务，那同样的我们这里还有其他的一些类型定义，比如说 i o 密集型和计算密集型。


很简单， IO 密集型就是输入输出流的密集型。比如说你在发布商品的时候，那上传的商品的视频，商品的图片，按理说它也是属于商品服务的一部分，但是它其实并没有任何的业务逻辑，它更多的是会耗费你的网络带宽进行这些流媒体的传输，那或者是文件二进制流的传输，那这类业务通常我们也是倾向于把它独立出来做拆分。再往下看计算密集型，那计算密集型通常不会涉及过多的事物输出，也不会占用过多带宽，但是它需要大量的计算资源。


计算资源是什么？内存、 CPU 这类服务单独拆分出来，我们在资源划分的时候比较方便，有的放矢，那所以这些场景主要处理的都是一些特长生，那在某一个方面要求非常苛刻的这样的服务，那比较有针对性的隔离这部分业务，给他提供相应的资源，不至于让其成为整个链路上面的资源瓶颈。


那以上这几个维度的拆分是为了什么呀？那其实一谈到那微服务这种拆分是为了什么？我们通常会讲为了快速迭代，当然了快速迭代和可扩展性其实是其中的一部分，但是一线大厂有一个不同的维度来看待这个问题，什么维度？这么做究竟是为了什么呢？
