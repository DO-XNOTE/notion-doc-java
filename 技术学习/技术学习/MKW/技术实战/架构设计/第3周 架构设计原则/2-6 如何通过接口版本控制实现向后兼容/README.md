APP 的发布不光伴随着他的客户端的版本升级，同样的也会伴随一些业务方向的变化。比如说我接下来跟同学们举这样的一个场景，让大家理解一下，为什么我们需要做向后兼容，那在这三个版本的 APP 当中，我们都实现了下单接口，咱微服务强调快速迭代、快速验证商业模式，所以我们在最早期的下单接口当中，只实现了一个最简单的下单版本，那就是单品下单。


什么意思？去浏览商品详情页，点击立即下单，那你的这个订单只会包含一个单个商品。OK，那在往后的一个版本里，我们做了一些优化，除了单品下单以外，我们还支持什么呀？单门店下单，只要是在这个门店内购买的所有商品，我们都可以把它做成一个订单。好，那这是第二个版本的内容。


最后一个版本我们上了一个下单接口的完全体，那就是真正意义上的购物车，那你可以实现一个跨店下单，然后针对每个门店，我们后台的服务可以去生成子订单，给到对应的门店。OK，那这是伴随着 APP 升级背后的业务升级。接下来问题就来了，尽管你的 APP 升级到了最新版 1. 4，但是依然会有很多用户使用你早期的 APP 版本，那么对这部分用户来说，你的接口必须提供一个向后兼容的特性，也就是说他们的老的业务流程要依然可以通过你的后台服务正常访问。好，那接下来的问题就是同学们看这三个服务下单接口，它所对应的业务逻辑那可是千差万别。那假如我们在后台只提供一个下单接口，同学们可以想象你在这个下单接口当中如何处理这些所有的历史包袱，那给大家 3 秒钟时间想一想，三 21 有什么好的想法吗？不管你们有没有老师，有一个比较好的想法，那就是除了 if else 还是 if else 来实现，这会带来什么问题？同学们，第一，你的这个接口就会变得非常的臃肿，那你的改动很可能牵一发而动全身，并且你要保证你的所有的测试用例可以把历史的情况全覆盖。


这样一个巨无霸式的下单接口代码，同学们想它是不是有违我们非服务架构设计的理念？在不同的业务版本之间，我们要做一个职责，清晰的划定它能不能做到，不能，完全不能。那这里一个更好更优雅地提供向后兼容的一个方案，就是我们开篇所提到的 API versioning，通过 API 的版本来将不同的访问请求路由到不同的历史接口当中。

咱首先 RPC 接口来说，那 RPC 对接口版本的控制是实现的非常到位的，不管你用像 double G RPC 或者是像 HSF 这些 RPC 框架，那它在接口定义层面都可以去指定你当前的接口版本。那就是在下图当中我们看到的version，你的微服务模块可以同时声明多个接口的不同版本。接口调用者在发起调用的时候，它可以通过这个 version 字段来指定一个版本，从而调用到它想要调用到的那一个特定的服务。OK，那这是 RPC 接口的使用方式，开源的 RPC 接口基本上都实现了这样一种套路。


好，我们接下来看对于 HTTP 接口来说，我们怎么来设置呢？这里有两个维度，第一个就是pass，也就是访问路径，第二个是header，这背后有一个简单的方式，还有一个难但是比较优雅的方式。那我这里把两个方式都跟大家说一下，但是更建议大家使用后面的一个优雅方式。


好简单的方式是什么呀？同学们，这里可以看到怕和 header 当中，我们只要使用path，那可以在 API 的访问路径后面加上一个版本号，比如说 checkout 斜杠V1，那就是下单的 V1 版的接口，后面跟V2，那就是 V2 版的接口。这样的话你的 APP 端通过指定你不同的路径就可以访问到对应的接口。


这是最简单的方式，但是似乎看起来有点粗鲁，每一个新版本新业务的发布还要去约定这种路径，这种方式可以工作，但是似乎并不是那么优雅，还有一种更优雅的方式是什么呢？我们借助下面的header，让咱的客户端在发起 HTTP 调用的时候，在 header 当中指定一个接口的版本号，指定好了之后如何来路由到对应的接口，这里我们就可以应用到业务网关了。比如说后面在微服务章节里将跟同学们学习的 Gateway 这样一种业务网关。那在业务网关层我们可以做一些谓词断言等，根据你服务请求当中的 header 来决定我需要把你路由到哪一个具体的业务方法当中。这种方式相对前一种来说稍微会复杂一些，但是这种调用方式显得更加的优雅，那么以后如果你要做老接口的下线，其实只用在网关层当中，把这个接口下线掉也就可以了。
