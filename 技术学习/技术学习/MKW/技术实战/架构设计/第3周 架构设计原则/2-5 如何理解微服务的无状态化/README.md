好，那所谓微服务的有状态是指你的服务上面有这么一个buff，什么buff？也就是说它是上下文依赖的。这里最老生常谈的一个话题就是session，一个请求过来，我要从当前这台服务器的上下文当中获取你访问请求的session。那这样有一个什么问题？哎，同学们，假设我们现在是一个单体应用，你有一个用户来登录它的session，保存在当前的机械，这都完全没有问题。那假设以后你的访问量增加了，我把这个单体应用扩展成了一个水平集群。同学们想，你的 session 如果还存放在单机，当你的请求被转发到集群当中的其他机器的时候，是不是你的 session 就丢失了？因此，这种上下文依赖的有状态的服务其实是缺乏水平扩展能力的。


那如何将这个有状态的服务改造成无状态？非常简单，你的 session 可以存在 Redis 里面，对吧？存在一个非本机上下文环境的地方，那么你就解决了有状态的问题。老师这里再跟大家举一个例子，让同学们对有状态无状态有一个更深的理解。比如说单机缓存，这个单机缓存是在超高并发场景当中经常使用的一个例子，当你的中心化缓存顶不住压力的时候，比如说我们这里有一些热点数据的产生，热点数据就像秒杀商品或者是明星出轨这样的头条新闻，它会迅速的成为一个热点，这时候有可能你的外部缓存甚至都顶不住这种压力。那么我们会构建本地缓存的方案，那这是对热点数据的一个处理的流程，但是问题恰恰就出现在这里。


本地缓存什么意思？用户在访问服务的时候，通常我们会根据用户的 ID 或者是商品 ID 等等做一个hash，根据这个 hash 的值，我们来决定把你的请求路由到哪台机器上面来做，那这个是一致性哈希的方案，也是我们在互联网场景下经常用的数据分发的方案。


假如我们有一个像 Redis 之类的中心化的缓存来获取这些信息，那么不管你把请求路由到哪台机器上，它都可以从 Redis 当中拿数据，对吧？这个没问题。但是假如现在你的这个商品突然成为了热点数据，那么呢？相应的我们就会通过一些后台的 job 等等去构建一个本地缓存，这个缓存只存在于你本地这台通过 hash 路由道的服务器上面。


OK，那假设以后你的一致性算法发生了变化，或者你的集群当中添加了新的机器，这样一来你的访问请求可能就会路由到其他机器，这时候问题出现了，你的其他机器上面没有去构建这个本地缓存，那可能就访问不到你的数据。这个我们就把它叫做上下文依赖，也就是有状态的服务。


在微服设计的这个理念当中，我们要强调服务一定是无状态的，为什么？这个是完全从水平扩展的角度来看的，那一个有状态的服务如果应用在微服务环境当中，在弹性扩缩容的阶段，你就会受限于这种上下文的依赖，导致服务不可用。因此我们从哪些角度来解决服务的这个有状态的问题？从这几个角度在数据分发、处理以及存储这三个环节要做到一个要求，什么要求？

就是任意节点的可达性，你的这个用户请求，它不管是落在你服务集群中的任意服务节点上面，它都可以去完成这个任务。那在分发、处理、存储三个环节当中，每一个环节都要考虑到它的扩容可能带来的问题，那这是我们在设计以及实现后台微服务当中需要重点关注的一个问题。

应用无状态就是前面说的，比如说我的一个商品中心，它可以任意的部署在集群a、集群 b 上一个无状态应用无状态服务，我部署在任意节点上都可以去正常的提供服务，这个是应用无状态，与其对应的是配置管理的有状态。什么意思？通常来讲，对于一个单元化或者是服务集群来说，它的配置属性是不同的。比如说你当前集群的 DNS 的访问地址，集群的入网地址、出网地址，这些有可能是不同的，这个不同其实就是一种上下文依赖。但是我们在设计的时候，一定要把这种配置管理的有状态和应用的无状态之间划分做一个很好的隔离。那通过什么办法？那就是通过我们的配置管理，通过一些像配置管理中心之类的中间件，为不同的集群以及单元来提供不同的配置文件。
