就是说如果我的系统在当前时候不能给你很好的提供服务，那我就退而求其次，用一些降级后的结果返回给你，把你打发走，就是这么简单。降级，通常来讲我们分为主动降级和被动降级。那主动降级怎么来做？很简单，开关推送，这个开关是一个动态的业务开关，通常由我们的配置中心来推送过来。那在后面的微附章节里，我们还会深入介绍如何使用 spring Claude 当中的 config 配置中心来推送动态的属性变更。


那动态推送的开关怎么来使用？给同学们举一个非常非常简单的例子，比如说我当前商品详情页访问压力非常大，那这详情页当中有一个优惠价格的显示，那这个小模块有可能因为访问压力过大而不能很好地响应请求，那这时候怎么办？可以人工的控制，主动降级，推送一个业务开关到我们的手机 APP 或者是你后台服务。


那在你去渲染详情页的时候，那详情页的接口就可以去绕过营销优惠计算，默认返回一个原价，那手机 APP 也可以在上面显示，请到结算页查看优惠信息。那这就是一个降级的例子。那我们再来聊一下被动降级怎么来考虑呢？通常一个接口它的运行状态有很多的维度，比方说你的响应时长超长，或者是你在运行期间发生了异常，这两种异常情况我们可以在接口层面采取一个被动降级，那把这次用户请求导入到降级的业务流程当中来做返回。那还有一个就是接口的失败率，那如果真听到失败率过高，我们也可以在程序当中设定一些被动降级的规则，来把这个业务导向降级。流程里我们最后一段还有一个叫熔断保护，同学们知道熔断和降级之间的关系吗？简单来说熔断其实是在降级之上的一个更加雷霆更强硬的手段，那如果你的服务在过去的 10 次请求当中，比如说有 8 次都被降级了，那这个失败率80%。我这时候就可以开启一个熔断开关，那熔断之后的服务就会在一定的时间内自动走向降级流程，那当你经过了一段时间之后，那可以去放行一个请求，去发起正常访问，那如果是一切正常，那么接口返回到正常运行的状态，那如果依然被触发了降级，那这时候继续保持熔断状态。okay，那这就是降级和熔断之间的切换。等后面的微服务章节，我们将跟同学们去讲如何利用 spring Claude 当中的组件去做精细粒度的熔断和降级配置。



好，那这里跟同学们一带而过，那咱继续往下看。现在同学们了解了降级是怎么回事儿之后，那我跟同学们讲一个比较现实的例子，降级在业务当中是怎么玩的？比如我这里拿一个稍微复杂一点的多级降级的例子，多级降级什么意思？就是一错再错，你降级业务当中依然返回异常，那么有可能进行二次降级。打一个比方，我的业务流程首先从 Redis 当中读取一个信息，那么假设在这一步，因为 Redis 的种种异常导致接口响应超时，那这时候我可以主动导向第一层降级，也就是从 local 的缓存当中读取，那比方说 e h、 catch 等等本地的缓存技术，那如果在这里面没有查询到数据，或者是有一些异常发生，那这时我可以再把它导入到二级降级流程，那也就是静默处理返回默认值。


那这是一个多级降级的例子。那同学们在做高可用系统设计的时候，在你的关键主链路上一定要去思考，当你的核心主链路发生异常的情况下，你怎么办？那规划你的降级流程，这也是阿里新应用当中对你主链路的保护手段，这是一个潜规则，所有主链路当中都需要去思考这样一个问题。

那网关是什么呀？ Nginx 网关，大家都用的，这里面有个 limit 模块，那我在架构师一期课程当中也有这样实际的例子，怎么利用 limit 模块从访问量、并发量等等不同的维度对后台接口进行限流处理给。


那除了Nginx，还有像我们如果使用了像spring、 Claude 这类微服务的架构体系，那这里有一个服务网关，比如说Gateway，那 Gateway 呢？这里提供了使用 Redis 加 Lua 脚本进行限流的这样一种手段，不光是 Gateway 当中可以集成这样的功能，我们自己也可以去使用 Redis 和 Lua 自己编写一个限流控件，那这也是在一期课程当中，老师带大家手把手做过的一个内容，那网关层限流是最有效的在源头上读漏的这样一种手段，那除了网关层限流，同学们想一想我们还有没有其他的限流方式？比如说业务层如何来限流？那提到业务层很简单，这里第一个就是大家一拍脑袋就能想到的本地吸引流。


使用 Grava 这样的工具类，在你本地代码当中使用一些漏桶算法、令牌桶算法等等来做本地的限流，这种本地限流我们也把它叫做单机限流，是一个很轻量级的限流手段。那对于分布式应用来说，我们可以利用像一些 Sentinel 等等的分布式组件来做分布式系统的各个服务之间的限流。
OK，那这里具体你要给自己的服务设定什么样的限流的阈值，这个值就是通过压测来测出来的，不是一拍脑袋想出来的。因此你像阿里新应用当中才会这么的重视双 11 之前的全链路压测，它会把你的核心主链路当中每个模块的负载给你压出来，这样的话比较方便去统一规划如何来通过限流保证系统的可用性。

除了你的代码应用和配置中心以外，我们还有什么需要回滚？给同学们再提一个点，大家可能没有意识到 CICD 的 pipeline 这是什么东西？持续集成的这个工具，你如何将自己的应用部署上线？以什么来策略来部署上线？是蓝绿发布还是金丝雀测试等等，那这个我们把它叫 CICD 的pipeline，为什么它也需要版本控制和回滚？你发布到一半如果需要回滚怎么办？那这个是很常见的一件事情，有的时候很多细微的 bug 是不好用金丝雀测试来发现的，经常会出现到你发布到一半，突然业务响起了警报，你不得不叫半途而废，把这个上线了一半的应用全部回滚，那这时候就依赖于你的整套持续集成环境有一个部署回滚的这样一种功能，那通过快速回滚这条路子来保证你系统的可用性。
