那对于一些变更做起来非常的容易，与此同时也非常方便的可以进行自定义化。那它是通过什么样的设计思想途径来达到这样的效果，我总结下来主要是有两点，第一点是定制化，第二点是默认值。这两点分别对应着共性，还有个性定制化定制的逻辑，这个是个性默认值约定俗成的一种规范，那么我们叫它共性隐藏共性突出个性，是这类约定大于配置框架的一个中心的思想。比如我们这里用的大名鼎鼎的框架，它整天都在用的同学们可能还没有意识到， spring boot 它就是一个非常到位的践行了约定大于配置这种思想的框架。


怎么滴，我说同学们还不信吗？我们这里就挑几个点来看一下，比如说它的环境变量的配置，那环境变量系统参数，它的配置文件是什么？ application 杠叉叉点 yaml 或者 bootstrap 杠叉叉叉点yaml，那后面的叉叉叉其实对应的是什么呢？是profile。


什么是profile？就是你的环境隔离的参数，那如果你后面跟DEV，那相当于你这就是开发环境，你后面跟production，那就是生产环境，随着你当前环境的参数的不同，它会对应的去加载不同的文件。那这种形式就是一个典型的约定大于配置，通过定义一套潜规则，在规则的定义方和使用方之间达到一种默契，不需要额外的配置。


OK，那举完这个例子，我们再来看一下定制化的需求怎么来办？比如说我想在环境当中配置多数据源，按照平时我定一个 spring data source 就 OK 了，那如果我这里想要定制多数据语言怎么办？这里就需要一点点额外的配置。在 spring 下面的 data source 下，我还可以去指定不同的标签，这每个标签都可以是一个独立的数据源。然后在声明连接池的时候，我可以很方便的用一个注解加载以指定标签名开头的配置属性。这就是 springboot 为定制化的内容所提供的一个便利，非常的灵活，可扩展。


OK，那对于一些既顶的一些行为，我们这里通过默认值体现在哪里，同学们有所不知。咱天天在用 spring boot，但是同学们有没有研究过 spring boot 背后的starter？ springboard 框架很多组件它都有对应自己的starter，而这每一个默默无闻的 STARTER 背后都会去加载对应的属性文件当中的属性，那如果属性没有定义，要么对应的功能不加载，或者指定一个默认值，这就是我前面所说的共性的部分。通过定制化的个性和默认值等等默认行为，约定的共性共同完成了约定大于配置这样一种体验。


okay，那这是框架层面。对于一些技术选型层面，我建议同学们一定要沿着这个方向，尽可能的在你的项目当中使用约定大于配置的这样的基础组件。打一个比方， spring data，同学们相信都用过，那我们再也不用像 my badcase 一样编写复杂的 SQL 脚本。一些简单的SQL，我们可以直接定义一个接口，就把它给声明出来，这里接口的名称，它包含了 SQL 的关键字以及你数据库实体类它里面对应的属性。那有了这两点的支持和约定，我们就非常轻松的可以达到一个共性的部分。对于一些异常复杂的需求怎么办？那它的查询非常长，如果用这种方式的话，你可能这个接口的名称就要声明半个屏幕怎么办？那很简单， spring data 它也提供了标签，可以去执行自定义的一段SQL。


从以上两点的例子都可以看出，约定大于配置的框架，其实就这两招，通过共性和个性来满足你的需求。那在更新迭代、业务飞速发展的互联网时代，我们讲究一个字快，快字当头。因此这种约定大于配置的框架的思想将在现在以及未来很长一段时间内占据主导的地位，那这种框架就提供了一种轻量级的接入方式，这个轻量级可不是说它功能，轻量级功能度是非常强大的，它的轻量级体现在接入成本上，代码量减少，轻量级的配置，这就是它的轻量级。那这种框架能力总结起来四个字就叫拆箱即用，
