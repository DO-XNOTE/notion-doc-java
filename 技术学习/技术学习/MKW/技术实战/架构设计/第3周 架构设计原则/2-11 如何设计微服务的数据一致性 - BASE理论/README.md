那从系统设计的角度来说，以上这三个方面结合起来，实际上就是在可用性和一致性之间的这样一个平衡。而 base 理论则倾向于在系统架构层面保障系统的可用性，而在一致性方面，并不强求强一致性，而采用了一种叫做最终一致性的解决方案。

那所谓的在高并发场景下面，那我们前面经常提到这样一个词叫主链路。主链路规划实际上就是使用各种手段来保障你的主链路处于一个可用状态。这里的手段我们前面多次提到过，比方说你可以采用降级减轻主链路的压力，还可以采用弹性计算，把非主链路的计算能力给它做一个缩容，并且把这个机器的计算资源放到你的主链路上面，让你的主链路可以承接更高的访问压力。


同时你服务请求的增多必然伴随着你的响应时间的增加。那这里我们可以在负载均衡层或者是你的熔断降级的这一个层面设置一些规则，使你服务的最大可以承受的响应时间保持在一个可控的数字内，这样你的用户会感受到你的服务还处于一个基本可用的状态，没有说要崩溃到你要等待十几秒，依然没有任何响应，那其实这就是我们主链路规划的一个作用，使你的整个的服务保持一个最小的可用程度，那么在这个最小可用程度下，你可以让用户完成你的主链路上的主要核心业务。


那另一方面，基本可用既然是一个主要侧重于可用性的这样一套理念，那么对一致性要求上面，我们肯定是不可能选用强一致性的，那一个强一致性能系统必然会牺牲一部分的可用性，因此我们这里采用的往往都是最终一致性的方案，但是体现在基本可用这个维度上面来，你要保证自己的系统在最终一致性这种一致性手段下面依然可以正常提供系统功能。为什么这么说？那就要牵扯到第二个维度了，软状态 soft state，

**那 soft state 实际上也叫做一个中间状态，这个是我们互联网行业应用当中一个非常非常正常的一种现象。**


为什么呢？我们打个比方，假设你当前有两个数据副本，那这两个数据副本你可以把它去看成一个是缓存，一个是数据库。好了，OK，那同时在一开始这两个副本当中都有一个对象叫马冬梅，那数据库里也叫马冬梅，缓存也叫马冬梅，没有什么问题。那我们进行到后面有一些业务流程对这个数据做了一些修改，诶，然后你会发现这两者之间变成了一种不一致的情况，你的数据库里面可能叫马什么煤，那你的缓存当中又叫马东什么，那这两者之间会产生一个短暂的不一致，也就是中间状态。在未来的某一个节点下，他们会再次回到一个同步的状态，那这整个过程其实就是一个最终一致性的过程。在未来的某个时间点，那你的多个数据副本当中的数据会保持一致，而在这个中间过程当中，你会发现它存在了一个中间态，那这个中间数据不一致的状态，我们就把它叫做中间的这个软状态。


看到这里同学们应该就能理解，前面我们说基本可用状态下，在最终一致性的情况下，我们要保证系统功能可用，那这就是为了保证我们的系统，我们的业务逻辑在这种中间状态下仍然能正常提供服务，所以 base 理论当中它的每一个要素都是环环相扣的。那接下来我们再来看这个最终一致性是怎么个体现法。okay，最终一致性往往是我们高并发系统当中一个不可绕过的一个话题。那老师这边狠话就撂下来了，你在高并发系统当中没有什么第二选择，通通都是最终一致性，
