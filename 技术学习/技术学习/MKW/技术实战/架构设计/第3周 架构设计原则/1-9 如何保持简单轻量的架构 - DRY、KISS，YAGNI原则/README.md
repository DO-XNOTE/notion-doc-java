那这是软件开发过程当中的一个基本原则。那同学们在自己的项目中有没有这种感受？真正厉害的大道至简的方法，不是添加越来越多的东西，增加越来越臃肿的代码，而是拒绝诱惑，甚至是砍掉功能，实际上这就是做减法。做减法又是什么意思？从两个方面来说，第一个就是如果别人已经有了这个功能，那么我们就不要去重复的去做一个新的一模一样的功能，那这是第一点。第二点，在自己的业务逻辑中，尽量把公共的部分识别出来，抽取成公共的组件包。那很多开发人员都有这样的一种毛病，就是非常的喜欢造轮子，热衷造轮子，总以为自己写的才是最好的，最适合项目的。那其实老师这里更提倡另外一种观点，什么呢？叫不劳而获。有的时候真的不是越勤奋越有效率，而是越懒惰越有效率，这个懒惰是体现在哪里？意思就是尽可能的少写代码。OK，那这里就强调一这样几点。


第一个比如说代码复用，那在你项目当中，我们可以把公共的组件部分沉淀下来，组成工具包，那甚至是一些Lib，可以供所有的兄弟项目来一起调用的。我这里跟大家打一个比方，比如说审计功能，有的时候我们需要对一些核心的业务链路做一些审计log，我们也叫 audit log，那每一个业务，每一个微服务都有自己去做审计 log 的一种方式，那其实这就是一个很好的切入点。在你的上下游链路当中，只要是需要记录 Audio log 的地方，我们都可以使用统一的工具包来处理。


同学们可以看到很多的公司当中都有一个基础中间件部门，那这个部门的存在是为了什么呀？其实就是为了减少我们每个业务部门重复造轮子这样一种资源浪费。那有中间件部门还有 platform 团队，可以去抽取一些公共的组件。再往下看，前面我们提到了工具类实，实际上在大家的项目当中，我们也经常需要使用一些小工具，那大部分开发人员比较喜欢自己去手写，这里老师提醒大家一下，你基本上你在项目当中所需要用到的工具，基本业界上面都有一些相应的开源项目。那比方说我这里可以随口举两个例子，像 gullava 工具包，那谷歌开发出来的这样一套工具包，可以满足你很多项目当中的小工具类的需求。


另一个 Loon book，一个可以帮助你产生简洁代码的一个插件，那像这类第三方的开源工具包比比皆是，同学们每到想要用到工具类的时候，不妨去看一看有没有可以让大家不劳而获的机会，那就是别人已经造出来的轮子，我们可以拿来就用。那再往下看，我这里跟大家提一句，叫常扣的是魔鬼，那这是什么意思？很简单，你的一个方法如果写得非常非常长，那么别人对你的重构的动机就越低，那碰到新的业务开发新的业务需求，需要加入的时候，往往就是打补丁。


那老师曾经认识一位在军工行业工作的朋友，他就告诉我，在他们研制的武器当中，火箭弹发射，火箭弹发射的一个闷方法有几千行，你动都不敢动，根本就不敢修改。那我们在工作当中尽可能的找一些机会，把自己的长代码给他处理掉，给它重构掉。而有些项目它会在自己的代码编译过程当中引入一些代码质量检查工具，那对于代码超过一定行数的方法，它们会产生出严重的警告，那这就是一个消灭常 code 的一种方式，那通常来说这种消灭重复的过程往往伴随着设计模式的引入，那同学们可以去查看开源项目的软件代码架构层次的设计，这里面就大量应用了设计模式来减少代码的重复。那它不是像我们平时写业务代码一样，就像记流水账一样，一个非常长的代码块，而是利用设计模式将这些功能分门别类地打散到了各个的小的组件上面去。OK，那这里讲的是第一点，减少信息重复。

往往我们去做一个软件系统架构设计的时候，都是从设计角的视角来出发，那希望它无所不能，包罗万象，有非常多的功能。但是更多的时候我们应该从一个使用者的角度来思考，你如何让自己的架构设计变得简单，足够易用。


那这种设计者与使用者之间的思维差别就非常像，我们学车的时候是选择手动的还是自动的？那很多人说要学就学手动的，对吧？开起来有快感。但是请问同学们，现在你还会买手动挡的车子吗？那既然不会买，那为什么要去学手动挡来自虐呢？那理想情况下是你有了手动挡驾照，你也能开自动挡的，现实情况是什么呀？是没人会买手动挡的车子。


所以我们去思考架构设计的时候，也不要单纯的是从一个架构设计者这个造物主的角度来出发，所以做架构设计一定要从实际出发。我这里跟大家举一个例子，假如从框架层面让你去研发一个框架，抛掉这些功能点，我们不谈你的一个框架怎么才是一个好的框架呢？那这第一点我就要提无感知，无感知是什么意思？你的业务部门接入你的框架是一种无感知、无侵入式的接入。那这里的一个反面教材，就比如说像Struts，它是一个很早期的上古年代的 MVC 框架，那使用 Struts 它的侵入式就非常非常的强，你的代码从前到后都被这个框架强绑定住了。但是我们反过来看，如果大家去使用 spring Claude 的组件，比如说我的底层使用的是Eureka，那如果将来的某一天我想把它换成Nacus，那听起来是一个底层框架的大改变，但是体现在业务层面，体现在大家的代码层面。难不难？不难，非常的简单。那这就是一种无感知，你的业务层代码并不知晓你的底层的逻辑的细节，而这是很重要的一个特性。


那再来接入成本，你研发的一个框架使业务方投入的接入成本有多大？这是衡量你一个框架能否推广开来的一个很重要的一个方面。比如说像阿里系的框架，阿里系中间件，你就可以切身体会到它的接入成本非常非常的低，那几乎不用你写任何的业务代码，而只是要极少的配置，甚至是零配置，使用这种约定大于配置的理念，甚至不需要配置，你就可以把它集成到自己的项目当中。


那最后一点，从代码层面来说， clean code 什么意思？代码简洁之道，那有专门一本非常有名的书，就叫 clean code，老师这里也推荐同学们都去读一读。其实当同学们的技术能力达到一定层级之后，再往上发展，往往靠的真的不是技术，不是这个数，而是什么呀？道你心中的架构之道，那就是你对代码还有架构设计的一种感悟和体验。


OK，那最后对于业务层面来说，保持简洁。有这样的一个方式，就是一句话描述。那如果你的方法可以用一句话很简单的把它的基本的最主要的功能描述出来，那么它就是一个合格的足够简单的方法。如果你要像写千字文一样，非常繁文缛节的这样的描述语言，才能把它的功能说清楚，那这种情况下怎么办？拆分，把它拆分成多个方法，那最后再带大家看第三个非常绕口的这样一个词， YAGNI: you arengonna need it。你根本不需要这样做。

什么意思？避免过度设计，那什么是过度设计？比如说你的应用现在还是处在一个单体应用的阶段，那么这时候我就考虑分布式事物该怎么做，那或者说你的数据库还是一个单库的时候，我就在思考如何去做数据异构，甚至是你的业务逻辑在并不复杂的阶段的时候，你就再去思考如何去上一个工作流引擎或者是规则引擎，那再比如现在图形数据库非常的火，那你就在想方设法的把图形数据库引入到你自己的项目当中，那这些是什么呢啊？说好听点叫面向简历编程，说难听点就是什么脱裤字放p，多此一举。


我们提倡面向未来的架构是什么意思？是去规划你未来技术发展的路线，当这一天需要到来的时候，你有能力去实现它，而不是说你现在就把未来不需要的东西去搬到自己的项目当中。很多程序员都容易犯这个错误，而且犯得非常的严重，你花了很多时间做了很炫技的技术，但是到头来其实对业务未必有多大的帮助。所以这也就是为什么我在之前的章节一直跟大家说的，技术服务于业务，怎么服务于业务啊？就是说如果这个功能点没有在业务需求当中体现出来，那我们就不要在代码当中把它体现出来，避免一厢情愿的过度设计。OK，那这一节我们跟大家介绍了三个软件工程领域当中保持简单架构设计的规约准则，那这些规矩其实就是为了达到一个很简单的目的。That is try everything you can to keep it simple.
