食之无味，弃之可惜。那既然弃之可惜，说明还有那么点用处，对吧？有什么用处？那这里最大的一个作用就是在上古年代那些 SSH 盛行的年代里面了，做视图层的分发，那这个功能是在 Controller 层里实现的。比如说我现在有一个叫 login Controller 的类，那它专门用来处理用户的登录请求，那比如说一个用户他正常登录用户名密码输入正确，那这里应该跳转到一个叫 login view 的页面，那这是由 Controller 来控制的。那如果它登录不正确，这时候我们应该给它跳转到一个 error 的页面，那这是在早期 MVC 模式当中的试图控制的，这个功能其实是在后端来控制的，它不像如今我们在前端来控制用户的这种交互行为，那这一个视图转发的功能，我相信在大部分的新项目里面都是见不到它的踪影的。


okay，那所以 Controller 的功能就去掉一半了，那还有什么？还有剩下来的一点是什么呢？封装返回对象，这是 Controller 里面大家非常用的一个功能，把一个对象封装成一个标准的返回的格式，那仅仅为了这样的一个功能，我们把自己的代码当中平白无故地加了一层 Controller 层。
同学们，你说这不叫鸡肋，那什么叫鸡肋呢？有的同学可能会说了，我的 controller 里面还会做一些数据验证，我还会聚合很多个service。那我问大家一句，聚合验证数据验证难道在 service 层里面不能做吗？难道 service 层之下就没有任何其他细分的层次吗？难道我们不能运用设计模式再将 service 以下的逻辑进行封装吗？那为什么我们不能去弱化 Controller 的作用呢？

okay，那我们微服务的核心是什么？微服务的核心就是服务service，对不对？我们的所有的后台的业务逻辑，它提供的其实是一个数据服务。


那从服务的角度来说，第一，我后台的接口是视图无关的，我不用关心具体要转发到哪个前端页面，我只关心我能处理好我自己的业务逻辑。那前端的转发不在我这一层来控制，因此你的关注点就落在了什么呢？服务本身，你自己的核心的业务逻辑上面了。并且我们在所有的微服架构里面倡导大家去使用 JSON 的格式来返回数据。很多项目里非常倾向于把所有的后台数据都封装成一个统一的数据结构返回给前端。所有服务返回的格式都是完全一样的，就那老三样 status code 有时候也叫 response code，还有一个message，剩下是什么呢？一个entity，那这个 entity 里面其实才是我们业务逻辑返回的最终的 JSON 数据。


那我这里就请问大家一句，返回成统一的数据格式，是为了服务于你当前要做的业务，还是服务于你前端的视图层，如果你是为了视图层做这样一种统一的适配，那么实际上你就把前端应该去处理的业务的关注点引到了后台服务的本身，对吧？那我们前面强调什么？你自己的这个微服务，其实只关注你自己的业务逻辑，其他的通通和你没有关系。


因此在真正的微服务架构里面，我们怎么样返回数据？我们返回它最原始的 JSON 数据，而不用做任何的无谓的封装，那这个也是在阿里系项目当中我们践行的这样一个准则，那它不要放在 RPC 这一层， RPC 只是单纯的业务数据，你可以在外层再包一层，或者是你在网关层对一些特定的 URL 的一些访问做这样的数据的视图封装。

第一个 API 层，第二个 service 层，最后就是道层和 repo 层，那 repo 就是 repository GPT 通常使用的一种叫法okay，那 API 层为什么我们把它单独抽出来？而且这个抽并不是说在你的当前的项目里面建一个文件夹，它叫API，而是说你把它单独抽成了一个 Maven 的子模块，而是说你把 API 专门定义到一个 Maven 里面的子 module 里面。


为什么我们要这样做？因为 RPC 框架的核心其实就是依赖于API，那同学们如果之前有使用过 double 经验，应该对这个问题体会的非常深刻，你的 double 是怎么调用的？引入对方的 API 接口层对不对？那后面我们在 spring Claude 的教学环节中，也会跟同学们阐释这样的一个理念，通常来讲，你在阿里信用当中，我们单独抽出一个 API 层是给谁用的？其实它并不仅仅是给我们的 service 来继承的，它是给你的其他上下游业务进行调用的，它们需要依赖于你的 API 来发起 RPC 调用，因此接口层我们单独把它抽出来，OK，那接下来 service 层就像老师前面说的，这个 service 层目的很单纯，它没有controller，没有这些跟仕途层纠缠不清的情感纠葛。那它只是关注于自身的业务，返回你最原始的 JSON 对象。
