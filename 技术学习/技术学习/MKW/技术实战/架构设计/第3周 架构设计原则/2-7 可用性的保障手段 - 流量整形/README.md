它这里面就有两个角色，分别是令牌，还有一个桶。OK，那这个令牌和桶之间是个什么关系？我们来看，令牌是这个算法，它定时发放的一个通行证，如果一个用户请求过来，他需要获取到通行证才能进行后面的访问，那从哪获取？当然事实从桶里获取。


OK，我们在尝试把令牌放入桶的时候，你的桶的容量是有限的，那如果他的桶已经满了会怎么样？你当前的令牌相当于被流放掉了，浪费掉了，对吧？他不会放到桶里。而假如这个桶里面的令牌是空的，或者桶没有被装满，那这时候这个令牌就会被放入到令牌桶中。OK，那接下来一个访问请求，一个真实的用户访问请求过来的时候，我们会设置一个排队的机制，那当然在令牌桶算法当中，这个排队队列其实是一个可选的，那它是什么的队列？用户请求的队列，他们都在这排好队等待去获取到令牌了。那此刻如果你的桶里有令牌，那会怎么样？我们会看你给这个请求配置的权重。比如说下单接口，我每个请求我要配置两个令牌，那对于添加购物车的请求，我每一个请求我要配置一个令牌，那它的权重是不同的。OK，那如果你的桶里的令牌数量满足当前请求所需要的个数，那么这时候它就会从桶里把你的令牌给拿到，然后放心请求去执行后面的流程。那如果没有足够的令牌，那这个队列就在这排好队继续再等。这个就是令牌桶算法的基本工作原理，它有这样几个特点，第一，令牌是匀速发放的。


第二，桶具备堆积令牌的能力。通过这两点同学们有没有看出来，它这里有一个堆积能力令牌桶算法里的桶，这里可以堆积令牌，因此它在你流量整形的过程当中可以处理一部分的突发流量。这个很容易理解，比如说你当前的下单接口，它每调用一次需要获取一个令牌，如果你的桶里它的最大容量是 50 个令牌，也就意味着假如现在下单请求有一个突发流量，那么它可以一次性的把你这 50 个令牌消耗掉，这个是令牌总算法流量整形的一个特点，可以去打出一部分的突发流量。


OK，接下来我们再来看另外一个不同的算法，天下武功殊途同归。那另一个算法和令牌筒算法也大体相似，但是细微之处有一些区别。OK，那是什么算法？漏桶算法。

那咱先来说相似的地方，咱这个漏桶算法相似的地方就是它依然这里有个桶，这个桶还是桶，但是令牌这里不一样了，前面是这样，往桶里放令牌。令牌是什么呀？同学们，还记得吗？令牌是你用户请求的通行证，那把这个通行证放到桶里，让每一个请求到桶里面，自己去拿搂桶。算法不一样，它不一样在哪？同学们，你看它放到桶里的是什么？它放到桶里的是请求，而不是令牌，是这个用户的真实请求。OK，那我们来看，假如当前的这个桶数量已经满了，没有额外的容量，那你的用户请求就会被丢弃，那如果你这个桶还有盈余，咱这个用户请求就会被怎么样加入到桶里？好到这里原理还和令牌桶算法比较类似，那接下来就不同了，咱这个漏桶算法这里一个特点是什么呀？它漏呀？怎么漏啊？同学们，往这看，你的用户请求它会通过一种匀速流出的方式，那流出的请求实际上就是可以去执行的请求。漏桶算法通过控制请求的流出速度来决定你在单位时间内有多少请求可以被执行。


同学们，从这里可以看出漏桶算法在流量整形这里的一个特点是什么，那就是它可以将突发流量把它整形为一个匀速的队列，这样你的后台的服务就可以一个匀速的速度来消费你的用户请求，OK，那这是漏斗算法和令牌桶算法的几个不同之处。目前业界常用的流量整形的算法大部分都是要么漏桶，要么是令牌桶。


对于这些流量整形的算法来说，我们主要关注哪两点？第一点，你对突发流量的处理，所谓流量整形，其实就是整的谁突发流量短时间内的用户请求的爆发，这样场景你怎么把这个流量做一个整形？这是我们关注的第一点。第二点是什么呀？堆积能力，这里我们关注两点，第一点是用户请求的排队队列，那你经过多长时间排队之后丢弃你的用户请求，这是一个关注的点。第二点是什么？同理的令牌或者同理的请求它的堆积量，那这是在这两个算法当中我们需要调整的参数。那怎么来调整？根据什么来调整？当然是根据具体的业务，对吧？OK，咱这一节差不多了，跟同学们介绍了流量整形的业务含义，以及两个经典的流量整形算法。OK，那在最后给同学们留一个小的思考题，换大家来热身了，
