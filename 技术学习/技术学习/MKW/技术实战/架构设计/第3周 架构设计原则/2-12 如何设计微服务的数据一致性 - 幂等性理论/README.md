打一个比方，你在购物车内做一个下单操作，那它会生成一个订单，那如果你手抖了，啪啪一下了两个订单，那么我们后台业务系统依然只会给你生成一个订单，那这个就叫做对相同资源的操作，返回相同的业务结果，那这就是幂等性。


OK，那为什么我们要强调这个幂等性？为什么在微服务架构系统下幂等性如此的重要？两个原因。第一个原因就是我们前面提到过的最终一致性，那在微服务架构下面，很多的最终一致性方案都是靠两个手段来保证的，分别是补偿还有重试。那补偿和重试就是我们所说的换个姿势，再来一次，你的用户请求我通过重试或者是追加补偿这样的形式来确保你的业务最终把它给执行掉了。


那在这个场景下，假如你的业务执行的过程当中，接口超时了，那么这时候你会获得一个失败信息，并且发起另外一次重试，而实际上这个底层的超时接口有可能还在执行，并且已经执行出了业务结果。那在这种情况下，如果你的接口没有实现幂等性，那么你下次重试的时候就有可能把这个业务执行过程重新执行一遍，那产生不一样的业务结果，OK，那这是最终一致性。


还有一个情况就是异常访问，比如说下单接口，那如果你在你的用户终端没有做好很好的前端控制，那有可能用户就可以重复的点击下单这个按钮，导致重复下单，那这都有可能造成异常访问。因此这两个场景决定了在微服架构下，你的业务接口一定要去保证一个幂等性，那我们接下来怎么保证幂等性呢？咱尝试着从增、删、改、查四个方面来跟同学简单说一下。比如说 insert 操作、增加操作，

这里很著名的一个场景就是账号注册，那像这类业务场景下面保证幂等性的操作非常简单，因为它们往往都有一个唯一的业务约束。比方说你账号注册里的邮箱，它就是一个天然的唯一的索引，那通过这个唯一约束我们很方便的就去可以保证在这个场景下的幂等性。


OK，那对于其他一些场景，比如说创建订单怎么办？那你的订单号是在你创建订单的时候才生成的，那这时候实际上并没有一个唯一的业务约束，那通常对于这种方案，我们往往采用一些隐藏表单和分布式锁的技术。比如说在你进入下单页的时候，我在后台为你生成一个隐藏的token，并且把它传给前端页面，那么前端把这个 token 附加在你的表单当中，把它作为一个隐藏的属性，在用户下单的时候传到后台。


OK，那我在接收到这个属性的时候，我会通过一些分布式锁，那可以是 zookeeper 等等各种手段，将这个通过隐藏表单提交上来的 token 加上一个锁。这样当你第二次下单的时候，如果使用了相同的token，那你就会发现当前的这个订单已经被锁住了，因此不能重复下单，那这是一种方式。当然还有其他很多种方式，但是总体的思想基本上都是使用业务为约束，或者是利用这种分布式锁来去锁定住资源。那么呢，对于 update 操作道理也是相同的。


比方说你个人信息的修改，那在个人修改的时候，有可能你打开了多个浏览器窗口，那在第一个窗口里面你修改了自己的姓名，那在第二个窗口里你修改了自己的性别。假如我先提交了第一个窗口，那你的姓名此时被修改了，而当我提交第二个窗口的时候，你的性别被修改的同时，大家看这个名称姓名是不是也被覆盖了？因为你第二个窗口上面的姓名还是之前修改之前的姓名，那我们并没有去从最新修改的数据当中重新捞一遍，然后再做覆盖，那这种情况下，通常我们就采用一个乐观锁的控制，那乐观锁同学们在项目当中应该都有使用过，那就是在你 update 数据的时候，那将乐观锁的这个 version 把它放到 where 词句当中，并且每次 update 去把乐观锁加一。


那这个乐观锁是我们在读取信息的时候，要从后台获取到，并且透传给前端页面的，因为它每次修改都会自赠。那么当你在后面一个页面修改的时候，那你的底层服务就会发现你当前的乐观锁的版本号已经落后了，那你需要去重新刷新页面，那通过这种方式来控制你的接口的幂等性，OK，那还有一些其他的场景，比方说并发修改，那假如你的这个并发修改是带有业务组件的，那也非常好办。
