那与组合和聚合相对的，那就是继承，那相信同学们应该听过这样一句口号，叫组合优于继承。诶，你说这个继承面向对象语言开发出来的新技能，熟着用着不香吗？那为什么说组合优于继承？其实任何的技术它有自己好的一面，那必然也有自己的弊端，继承用的好可以去增加你的可扩展性，但是用的不好而经常会产生类爆炸这样一个现象，大家都滥用即成，任何的抽象都要把它单独抽出来一层，甚至说有些开源项目这一点做的也并不好。


那继承结构层次过于复杂，那我们说其实继承的滥用它违背了一个开闭原则，为什么呢？因为我们看继承它是一个类与类之间的树状结构，那这个树状结构从封闭性的角度来讲，我们看这个父类的细节发生的变化，同样的也会在子类当中有所体现，那其实这个算是一种打破封闭性的一种体验。


而我们做微观层面的代码加固设计，通常是更偏向于优先使用组合或者是聚合的，那你不论是组合还是聚合，它其实都是对你的持有的对象的一种引用，那我们在自己的业务逻辑内部去调用你引用的一个对象的方法，这样一种结构方式是可以降低耦合度的，为什么？因为这样以来你可以明确每个对象之间的职责，而不像这种继承，而继承是一种结构化的一种设计思想。你滥用继承，那其实不光是类爆炸，也会打破这种开闭原则，是子类与超类之间的耦合度就会非常的挤。

好，那我们第一个例子就跟大家去通过一个组合原则在代码中的经典运用的一个方式，跟大家看一看组合和继承有什么样的不同。那所谓继承，同学们知道是一个树状的结构，对吧？所有类一层一层，像叠罗汉一样层层嵌套。那同样的组合，我们的层次结构其实也是一个树状结构，但是它并没有继承这么复杂的层次关系。比如说同学们看这里我们定义了一个叫 component 这样的一个接口，那它是一个组合对象的顶层父类声明的一个接口，可以看到它这里有几个方法，一个是添加，一个是 remove 删除，还有一个是 my service，就是它自己的业务逻辑。那这里添加和删除什么呀？那我们接下来往下看就知道了。


好，我们来看啊，这里我声明了一个类，哎，这个类叫叶子live，那 live 类呢？其实是继承自这个 component 类的。那它实现了哪个方法？它具体实现了这个 my service 这样的一个自定义的方法，它是这个树状结构的叶子，我们说叶子有什么特性？同学们，叶子它就是一个最底层的节点，对吧？它下面不再勾三搭四的去挂一些根节点了，那它就是最底层的一个。


那我们怎么把这些底层的叶子节点给它组合起来呢？那我们就来看另外一个结构了，哎，走这一个结构，它叫组合，同学们可以看出一点端倪，对吧？你看它这里旁边有一个什么？有一个 list component，它这里就是它要组合的内容。这一个组合的类，它自己内部维护了一个 component 列表，那通过添加和删除，它可以控制这个列表内的元素的新增和移除。


而由于这个叶子节点就是咱这个左边的树叶构件和右边的这个组合构件，我们可以把它叫做树枝，树叶和树枝的关系，这两个同样都是继承自component，因此这里就发生了一个很巧妙的组合的方式。你右边的这个结构件，它其实是可以把live，也就是数列构件以及另一个结构构件统一都加到这个列表当中的，因为它们都是继承自同一个接口。那通过这样一种组合模式，你就可以把一个原本需要很复杂层级的继承关系转变成了一种层级非常简单的组合关系。OK，那我们通过一个例子，实际的例子来跟同学们展示一下。好了，那我们这里就用慕课网的课程这个例子跟大家去阐述一下这个组合结构是怎么来使用。

另外一个是什么呀？树枝构建，也就是咱的组合这样的一个节点，以及最顶层的这个抽象构建，就是 component 这样的一个接口层。那对于怎样客户端的程序来说，我们可以使用一种非常一致的方式来处理这些对象，并不需要去关心你要处理的是一个单个应用还是一个组合的应用，那这是一个优点。还有一个是你如果要在这个组合内加入一个新的对象，非常的简单，那你客户端需要引入新的代码吗？其实相对来说并不需要改它原有代码，那你只需要去声明一个新的类，一个叶子也就可以了，这就满足了。前面说到了一个什么原则，开闭原则，添加新的应用并不需要改变原有的代码，那只用这样通过扩展的形式来满足新的业务。OK，那前面的扩展是通过继承的方式来扩展，那我们这里就是通过组合的方式来扩展，同学们觉得哪种更轻量级组合，对不对？所以说在你考虑使用组合还是继承的时候，记住前面提到的一个点，你要明确你当前想要扩展这个对象，是不是一定是超类的完全子类什么意思？也就是说你超类的一举一动，任何改动都需要去原封不动的体现在你这个子类里面，这种才是一种非常强的这种血亲关系的父子类，这样的情况下我们才建议你使用继承。而对于那些逻辑上并没有很强的这种关联关系的类，我们更倾向于让大家使用组合的方式来去搭建这样的结构。
