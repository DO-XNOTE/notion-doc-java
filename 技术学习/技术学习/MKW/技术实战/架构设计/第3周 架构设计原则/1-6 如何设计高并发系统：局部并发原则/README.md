那咱是采用异步的方式还是采用同步的方式？我相信大部分的同学应该是采用的同步调用的方式。那这一招在你一些小型的业务里，或者是并不复杂的业务系统里可能还能玩的转，但是当你到了真正的高并发系统当中，这一招肯定不管用了。那比如我们这里给同学们举一个例子。什么例子？

如果大家做一些毕业设计之类的这种难度的项目，那它的下单接口其实非常简单。但是当你去做一些一线大厂的下单场景，比如说你在淘系链路当中下单，那这背后的微服务。我的天哪，几十上百个微服务的调用，那不信我们就随便挑几个来说。


首先你下单这里要获取商品信息，对吧？商品信息从哪来？那当然不是从同一个数据库，他要调用商品微服务的接口，那这个接口背后还要去同步的从一些像库存之类的这些其他微服务当中聚合信息返回过来，OK，那这是获取商品信息，还有你要锁定库存，你的下单链路很长，它花费的响应时间很长，所以在一般来说一开始要锁定你的库存，等到下单完全结束的时候，你已经买了这个订单。付完款之后，那你这个库存才会被真正的扣减掉。


那我们再往后看，生成付款链接，那这是谁的事？支付系统的事，又是另外一个独立的子系统。再往后你这个订单要送到哪里？你要获取地址信息。很多同学们可能有点奇怪，为什么要获取地址信息？那你地址信息在数据库中不是一条记录吗？它有一个 ID 对不对？直接拿 ID 就行了，其实不是你的地址信息，它是会变的，那在你下单的时候要拿什么地址信息？当然是要拿你下单当时的这个地址对应的快照，那即使你以后这条同样的地址记录变掉了，你从铁岭搬家到了老鳖湾，但是在你下单的时候，一定要去拿当前下单时候的这个地址信息保存下来。那再往后看，你生成了付款链接，那如果你的账号设定了是自动付款，那后面还有一个自动扣款的流程，对吧？那这也是需要跟支付系统来打交道的。


OK，还有一个很重要的地方，优惠营销计算。哇，那这一块后台的服务就可复杂喽，你像淘系里面有一个小几十号人的团队在维护着这套营销优惠计算引擎，后面又是一大坨的微服务。好，再往下看，创建订单必不可少，对吧？与其创建订单对应的是生成订单的快照，你在未来的任何时间去回来查这个订单，我不需要再去重新做一些聚合，把所有信息拿过来，而是直接给你一个快照版信息就可以了。还有在所有的流程完成之后，你也付款了之后去释放库存或者是扣减库存。那释放是什么含义？释放是一种补偿。那通常来讲，在高并发系统当中，尤其是一线互联网系统当中，我们采取了一致性，叫最终一致性，所以它去伴随着大量的补偿的业务逻辑。那这个我们后面有专题再跟大家深入讨论啊，咱先关注这个买买买啊，那以上这些步骤你全部完成了之后，我才会认为你是完成下单，那这里跟大家看的只是很少的一部分的操作，那其实在它背后的，你整个的淘系达人下单链路，它背后的微服务数量非常多。

那如果我们在这些**服务调用上面全部采用顺序执行的方式，同学们可以想象一下你的接口的响应时间会被拉长到多大的一个数字，这样的话肯定会影响到你主链路的整个的系统的性能、吞吐量等等。**那下单场景只是其中的一个例子，那像老师以前在阿里面做商品中心，我们知道像商品的发布，同样的也是会有一个非常长的链路，调用很多的微服务来做这样一件事情，那这些场景它本身调用链路很长，这是无法改变的，因为它的系统复杂度高，所以微服务拆分的相对比较细，那因此我们就要在业务当中应用局部并发原则，把一些可以同步执行的东西给它放在一块儿执行掉。

这部分信息咱是不是可以放在一块来获取，它们三个之间是没有相互依赖关系的，对吧？我们可以把它一次性的获取局部并发，拿到这些信息之后设置一个数据检查点，如果在这个检查点上达成了一致，我们可以继续往下走，那么接下来才进入到第二个步骤，比如生成付款链接，计算你整个订单的优惠信息，并且去真正的锁定你的商品库存。


同样的，做完这件事情以后，我们也要设置一个数据检查点，为什么？因为你这里面每个步骤都有可能失败，那失败过后有两种方式来处理。第一种方式什么重试，那重试你就要保证自己的接口有幂等性的，保证，那幂等性对有工作经验的同学应该并不陌生，那后面我们也会有相应的章节再深入探讨。除了用重试来保证，还有一个什么最终一致性的补偿对不对？那么就有可能你需要把你锁定的这部分库存，把它再释放掉，变成可用库存，那在这个数据检查点 2 之后，我们同样的做接下来的步骤，生成订单快照创建订单，真正的去释放或者扣减库存。从这里可以看出，我们把之前一个顺序执行的链路分成了三个等份，那这三个等份当中每一块儿里面的信息，其实它们相互之间都没有依赖，他们是可以并行来执行的。这个就是在超高并发的一线互联网大厂当中，在业务层面的局部并发处理，这样一来你就可以去大大缩短你接口的响应时间。但是对于一些不能做并发操作的一些关键步骤，比如说你设置了自动付款，那这个付款应该是在前面所有任务完成之后的最后一步，那这个迫不得已咱就用顺序的方法给大家去处理掉，从而完成整个的下单流程，那这里是从架构层面跟大家去解释这个局部并发原则，但是从最基本的技术层面，同学们知道怎么做吗？非常的简单，咱去抛掉这些响应式编程或者RX Java 等等这些并发框架来说，如果单纯让大家从 JDK 原生的类里面，同学们应该知道该怎么做，那异步的方式， future 接口、 completion service、 executor service 等等。


那当然了，当你到了一些数据检查点，你可以用后台做 count h 或者是一些山栏之类的这样的类来做一个同步的阻塞，那这都是 JDK 里面给大家内置的。但是咱在自己的项目当中其实有更多的选择，我们可以用一些现在的像响应式编程，还有一些像异步并发框架来处理这种局部并发的场景，那这种处理方式从代码层面来说会显得更加的简洁，更加优雅。**那后面我们也会有单独的章节跟大家去分享响应式编程这一块怎么来处理异步的消息**。


那再回到我们这个话题，那很多同学在工作当中参与的系统项目往往并不是这种超高并发应用的业务，因此对局部并发的必要性可能并没有很深刻的认识。
