对于这种问题，我建议同学们分两个层次来说，首先我们从全局视角来给它列出你的当前项目架构里面的核心模块，它们组件之间是如何来划分的。那么在这一个层次上面，我们运用了哪些架构原则？这样的话它就比较清楚地了解到你的系统它的顶层设计是什么样的。


接下来我们再从一个局部视角跟它去详细的讲解一下你所负责的这块业务，那它里面的模块的分层，这里既可以说你当前业务的微服务的划分，也可以说你的每一个具体的服务，它的一些核心的模块，那这些模块上你运用了哪些架构的原则来去做设计？这部分的内容我们就可以去套用前面小节当中跟同学们一起分享过的架构设计的一些原则，以及一些经典理论。


OK，那这里回答的时候不要忘记，不光要说出运用了哪些原则，你还要明白为什么这样用。那我们接下来看第二个问题，对于高并发业务的场景，那如果交给你来设计，你有哪些思考的方向，那我们这里首先可以明确一下，现在一线互联网公司的这些高并发场景，那他们都有一个大方向的前提。第一个微服务化，那这个我们在后面的微服务小结当中会跟同学们在深入的探讨，除了微服务化以外，在高可用也是去支持高并发的一个前提，因为业务的可用度 availability 这个是放到首要目的的，其次才会谈到如何支持更高的并发量。那么对于高并发业务，前面也跟同学们去分享过一些比较通用的一些手段，比如说限流、降级、熔断，还有利用 MQ 做削峰填谷，那分库分表加上数据异构的方案，以及缓存或者更高级一点的分段缓存。包括后面我们还会跟同学们分享缓存在高并发场景下面有哪些不可靠的因素，来怎么来避免它。


最后局部并发，咱本章当中跟同学们讲到的，如果你的接口中有可以去并行执行的业务，那么这部分就可以做一个并发执行，那这是应用层面比较宽泛的一些设计理念。那么具体到细节如何来落地，我们这里不能泛泛而谈，要用一些具体的场景来举例。那么对于这类面试问题，通常面试官会给一个具体的方向，比如说我想让你去谈一谈秒杀场景，怎么设计红包场景等等，那这些场景其实基本上面试过程当中都说烂了。


其实问到这些场景，大部分的面试官他其实自己并没有去设计过这类系统，那他所能接触到的也只是网上的一些理论，**那么对于有经验的面试官通常不会去问这些你可以在网上搜索到的烂大街的资料，它会提供给你一些场景，比如说它当前项目当中负责的一些模块，来让你去设计高并发的架构，或者去说一些比较冷门的一些方向，比方说点赞、关注等等这些你在网上不一定会准备到的材料**。那这部分内容就要靠同学们在平时的工作当中多积累，多去关注一些有价值的公众号等等，去积累一些这些技术视野。

很多技术经理都喜欢问这个问题，脱离了一线开发很久，但是又想展现自己是个懂技术的人，会这样来问，那么一般我们这里并不建议同学们去回答这些 AOPIOC 这种小学生都能去给他倒背如流的问题，这些话题咱可以一笔带过，然后把这个你的理解升华一下。


我们来谈一谈设计理念， spring 的一些设计理念，那举出一些具体的模块组件，它如何实现这种插件式的开发？它是如何来做抽象来实现这种底层组件可替换的这样一种插件式编程，以及现在非常流行的约定大于配置的这样的理念。具体举出一些例子，这样的话你回答这个问题的角度就远远高于 AOP LC 这种低级问题。

找一些相对并发量比较高的业务来去做一番分析，那么这也是为你们以后的面试做一个储备。那假如被问到你当前的业务当中有哪些是高并发的场景，那么你这里就可以比较从容的来回答。


当然了，回答这个问题的时候，同学们可以去自己的添油加醋，把一些并没有用到的技术给它运用进来，这都是可以的，因为最熟悉你的业务的是你自己，而不是面试官。第二个问题是一个批斗大会的话题，去了解一下。我司是同学们在任职的这些公司，他们的 platform 团队或者是 Infra 团队，那就是平台的团队，他们封装的某些蹩脚工具，那如果交由同学们自己来封装，如何运用一些架构设计的理念或者说是设计模式来运用一些当下比较流行的设计理念，比如说约定大于配置。


那我这里建议同学们基于某些具体场景去真正的动手实现一下，因为大部分的同学平常接触的都是具体的业务，相对比较偏CRUD，那我们适当的接触一些平台组件，框架层面的开发，对提高自己的架构水平都是蛮有帮助的。那最后一个话题，


我们建议同学们每天不要多去关注一个技术文章，每天做一点的积累，去了解各行各业当中的高并发业务的解决方案。比方说在微信里面的发红包，以及春晚全民抢红包，或者说全局发号器，在互联网公司是非常重要的一块业务，比如你的订单号生成，比如你像微信当中每一条点对点消息的 message ID，消息ID，那么后面还有像老生常谈的秒杀，以及一个非常类似秒杀的业务，那就是抢单。平时就多花点时间去积累这类高并发业务的解决方案。okay，同学们，那这一章就到这里结束了，哎，我们下一小章再见。
