查询就像它的字面意思一样，只是获取一个查询的结果，与之相对的是命令，那命令是对当前的对象的状态的一种修改，那它本身是一个非常上古的软件架构理念，但是在如今的超高并发的互联网行业，它又有新的不同层次的解读。那曾经查询命令分离其实只是一个简单的职责分离，比如说在接口层面将查询和修改从功能上区分开来，那这个是大家都知道的方式。但是在如今的超高并发的互联网行业，应用的访问压力非常大，那我们对查询命令分离这样一种设计理念又有了新的一层的认识。


为什么呢？同学们想现在的查询其实就是读那命令，就是所谓的写,那读和写在超高并发的场景下，它都有什么样的底层的不同？我们说互联网的架构设计当中都有这样一种现象，叫读多写少，

大部分请求都是围绕在读数据而写数据其实相对于读来说，它的并发量是相对较小的。因此这个命令查询分离这样的一种设计原则又有了不同维度的解读。


比如说读写分离，那我们把后台的数据库分为读库和写库，**写库负责落地用户的写请求，也就是命令，那读库为了承接用户的查询请求，那读库和写库之间我们通常采用 binlog 的同步形式**，把数据从写库同步到读库。那这样做的一个显著的根本的目的是什么呀？那读库的横向扩展，那我们通过定向增发给这个读库的集群添加机器，那么可以水平扩展你读库的承载能力，那这是第一个用处，那第二个用处就是在超高并发的这些电商场景下面保护写操作，因为我们是处于一个读多写少的这样一种情况，那么读多就意味着你的读库被打爆的概率是非常大的。那一旦你的读库被打爆，我们这里不会影响到写库，也就是说你的写操作是安全的。一旦碰到这种极端用户场景的情况，比如你的数据库连接池，你的读库连接池被吃光了，那这种情况只是会短暂的影响你的读服务。


OK，那这是读写分离的一个应用呢？还有一个场景是什么呀？我们同学可能平常并不常听的叫数据异构和数据聚合，业务系统背后的数据库无非就是那些像Oracle、 MySQL 之类的这样的业务数据库，但是这些数据库其实它的并发能力是非常弱的，并不能很好地扛住这种海量用户的请求。
怎么办？我们往往会把它做一些数据异构，比方说采用同步的形式，那同步 binlog 到一些像 Elastic search 等等的这种搜索引擎的中间件上面来，然后在那里做一些像数据聚合的查询操作等等，那一份数据往往会冗余在多个地方。这是我们互联网行业当中非常常见的一种场景。那同学们可以看到读和写其实是在不同的数据源，那在之间我们并不是一个强一致性事物。互联网行业更偏向于什么呢？最终一致性就是说在未来的一个时间点，我最终可以保证你的数据达到一致，但是在短时间内我允许这种不一致的情况下存在。


okay，那上面这两个例子就是在互联网场景下，我们如何把查询和命令分离开来，满足高并发用户访问量场景下的读操作以及保护写入操作。okay，那这就是 CQS 的架构设计的理念。那我们接下来看第二个 SOC separation of concerns 关注点分离。

何为关注点？我这里跟同学们举三个方面的关注点，第一个是横向业务，第二个是纵向结构，第三个是无感知，那分别是从三个不同的维度来切分你的关注点，比如说横向业务，这是我们最好理解的一个方面。这里就伴随着向服务拆分，模块拆分，从横向的功能性业务层面将你的关注点切分成更小的模块。那相信到这里同学们应该可以看出关注点分离其实是一个单一职责的扩展，减轻系统模块间的耦合的这样一种手段和架构理念。


okay，那什么是纵向结构呢？其实是代码内的微架构，比方说我们早期推崇的 MVC 这样一种架构，或者是你的业务代码当中有一个 manager 层或者是 DAO 层，以及像 spring data 这里我们有的 repulstery 这样的一层内部的结构。


那通过这种纵向结构的拆分，我们把每个功能点都分别类的放到了对应的层次领域当中，那在过去很长一段时间的架构设计理念的发展过程当中，那我们也一直是朝着横向业务拆分以及纵向结构的划分这个领域来推进经济化的架构设计。近些年随着约定大于配置这样一种理念的发展壮大，我们的关注点其实有了一个更新维度的方向，那就是无感知。什么是无感知？无感知这里并不是说什么都不做，而不是说无微而至，而是说我们把你的关注点所关注的这样一个业务，通过无感知的形式把它嵌入到你自己的业务当中。比方说我这里举几个例子，切面 AOP 就是一个非常好的无感知的这样一种集成方式。我们这里大家都使用过 spring 的 transactional 标签，事务性管理以及我们的权限控制，这两个其实都是我们业务的关注点。业务的一个功能点，但是它并没有体现在各自的业务层代码当中，对业务层来说，这两个功能点就达到了对业务层透明，但是你的关注点又从业务层分离了出来。


OK，那除了切面还有其他的吗？代理也是一种模式，比如说我们经常使用的代理数据源，那在原先的数据源的基础上，我们通过代理来为它添加更多的功能。比方说我们阿里系出品的分布式事务框架Sita，那它就是通过代理数据源来实现了分布式事务的控制。那我们从这个角度来说，分布式事务是不是一个特定的功能领域？那通过代理数据源的方式，我们把分布式事务的这个关键点、处理点从你的业务代码当中分离了出来。


OK，那还有没有其他的分离关注点的这种无感知的实现？比如说我们整天用的 spring boot 对不对？那它为什么可以在极少量的配置下，就把你的整个应用给它全部启动起来？同学们有没有读过里面的一些代码，一些 starter 它是怎么来启动这个应用的？所以大家到后面源码阅读环节，记得要仔细的跟老师一起去读这些开源项目的源码，那相信你会对这个架构设计有更深层次的认识。


okay，那我们这里你说项目启动、项目加载，这是不是一个关注点？是没错，但是它跟业务相关吗？并不相关，我们怎么把它分离呢？那像 spring boot 这里的一个方向就做得非常好，那它把这些封装到了 starter lib 里面，那在应用启动的时候，通过一些约定大于配置的这样一种形式，那加载默认的配置文件初始化框架启动项目，达到了对业务层的无感知，并且同时将应用启动的这样一个功能点分裂了出来。


okay，那以上三点，切面代理和幕后操作是我们做框架层面设计的分离关注点的三个非常优雅的手段。其实这也是开源框架未来发展的一个趋势，对业务层的侵入越来越少，让业务层越来越无感知。OK，那这节的内容就到这里结束了，**我们分别从命令查询分离以及关注点分离这两个层面跟大家做了架构设计方面的探讨。那下一节里我们跟大家去聊一聊未来框架的一个发展趋势，那就是约定大于配置的这样一种理念。**OK，同学们，那我们下一小节再见。
