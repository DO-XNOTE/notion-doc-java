# 8-4 Tomcat源码详解类加载器（1420）

同学们大家好，这章节我们来学习一下汤姆开的核心源码解析的第三部分就是汤姆开的类加载器，那么说到汤姆开的类加载器，我们先要了解一下类加载器的一些机制，另外我们会给大家介绍一下双擎尾牌模型以及唐 k 的类加载器的一些特征。对于类加载器的话，我们先去了解类加载的机制。是这样的，我们 Java 编写的代码，对于 Java 代码，它会首先通过 Java 编译变成对于 class 代码，那么代码编译的结果从本地的机器码转化成直接码，那么这个过程虽然说是存储格式的一小步，却对于编程语言是一大步，这样我们可以做到了一次编译多次运行。


加瓦虚拟机，它把描述这个数据的 class 文件加载到内存，并对数据进行校验、转化、解析和一些初始化的操作，最终可以形成能够为虚拟机直接使用的加压类型。那么这就是虚拟机的类加载机制。那么虚拟机的设计团队，它把类加载这个阶段，通过一个类的全限定名称来描述这个二进子流这个动作，这个全限定名称比如它包含对应的包结构和它的类名，这样他把这个二进制流放到我们加 2 虚拟机的外部去实现。它这样做的好处可以让我们应用程序自己去决定如何去获取所需要的类，那么实现这个动作的代码模块，我们称之为叫类加载器，那么这里面我们在想类和类加载器的关系是这样的，类加载器它首先只用于实现类的加载动作，但它在加化程序中却取得了一个远远不限于这个类加载的阶段。对于任意一个类，都需要由加载它的类加载器和这个类本身一起去确定一下它在虚拟机中的唯一性。


每一个类加载器都拥有一个独立的类的命名空间，这句话说的更明显一下，就是比较两类的是否相等，那么首先这两个类必须在同样一个相同的一个类加载器下，这样的比较才有意义，否则的，即使这两个类是来自同一个 class 文件被同一个虚拟机加载，只要它们类加载器是不一样的，那么这样得到的两个类它也是不等价的。
接下来我们来看一下双亲委派模型，那么从加尔虚拟机的角度来说，它只存在两种不同的类加载器，一种我们称之为启动类加载器，也就是 Boost travel class loader，那么它是由 C + + 语言实现，这里面这个是虚拟机自身的一部分。另一种就是其他的这些类加载器，那么其他这些类加载器它都是基于 Java 语言实现，那么对于独立信息，外部微信前的它都会继承于这个抽象的类，也就是加尔浪的 class loader 这样一个类。


对于加尔开发人员的角度来看，类加载器我们可以去划分的更细一些，通常我们对于它的化班次可以分为启动器类加载，也就是 trouble class loader。另一个是扩展类加载器，也就 extension 的 class loader，另外一种就是我们的应用程序类加载器 application class loader。


最后就是我们可以通过 application class loader 继承我们实现自定义的一些 class loader。这里面启动类加载器我们可以看到它会通常会加载 GRE 下面内部下面的 RT 点zar，通常是它只指定这一个扎包，当然它我们也可以通过一些参数，比如说 boot plus pass 参数指定加载的接入性。


那么对于扩展类加载器，它通常加载的也就是 GRE 内部下面EST，也就扩展的包下面的这些所有的这些大文件，应用程序加载器它就比较容易理解，它其实加载的内容都是对于 class pass 下面的一些内容，那么卡里帕说指定这类库，当然开发者也可以直接使用这个类加载器。如果说应用中并没有定义过自己的一些类加载器，一般情况下这就是程序中默认的一个加载器。当然我们可以通过 weekend class 指定一些加载器，指定我们自己目录相关的一些内容。


那么了解完这些加载哪类加载器之间的关系以后，我们再去想一下双擎委派模型，那么对于双晶美元模型的话，我们可以理解这个整个图的结构，它就是一个类加载器的双晶美团模型的效果。双晶美元模型它要求除了我们这顶层的这个启动类加载器之外，那么其余的这些类加载器应当由自己的父类去加载器进行加载，这里面的类加载器父子之间关系，它并不是以继承关系去体现的，它是以我们的程序组合的方式复用负累加载器的代码来完成的。比如说当我们在使用应用类加载器加载一个类的时候，我们首先去看一下它的扩展类加载器里面是否存在这个对象，如果扩展类加压对象里面不存在这个类，那么他就看一下启动类加载器里面是否存在这个类，如果也不存在的话，那么这会儿我们才会使用到我们应用程序类加载器的方式来进行加载。


那么类加载器的双星美元模型它是在 GDK 1. 2 的时候被引入，并不广泛运用所有的这些加法程序中，但是它也并不是一个强制性的一个约束模型，因为在 Java 设计的过程中，它是支持作为一个推荐给开发者的一种类加载器的实现方式。这里面那个我们的双侵委派模型，它并不是一个强制性的约束模型，而是一个建议性的类加载的实现方式，就是在加 2 中，大部分的类加载体都遵循这个模型，但也有一些例外，到目前为止，双期美化模型有过三次大规模的被破坏的情况。第一次是在双期美外模型出现之前，也就是说 GDK 1. 2 发布之前，那么第二次是这个模型自身的缺陷所导致的。我们说双亲委判模型很好地解决了各个类加载器基础类的统一问题，那么越基础的类越应该由上层的加载器进行加载。基础之所以称之为基础，是因为他们总是被用户代码调用的API，并不是绝对。如果说基础类调用户的代码会怎么办？这个也是有可能的，比如说一个典型的例子就是 GNDI 服务，那么 GNDI 现在它已经是一个价的标准服务，它在代码启动的过程中去加载，其实在 GDK 1. 3 的时候其实放进到了 RT 点g，但是它需要调用由独立的厂商实现并部署在应用程序 class pass 下面的 GNDI 接口的提供者，我们称之为SPI，就是 service provider interface 的代码。
那么但是启动类加载器它不可能认识这些代码，因为这些类不在 RT 中，所以说启动类加载器它又需要加入怎么办？就是为了解决这个问题， Java 设计团队又引入了一个不太优雅的设计，我们称之为线程上下文的类加载器，这里面就是设的 context classloader，这个类加载器可以通过加尔让色的类的 set context classloader 方法进行设置，如果创建线程还未设置，那么它将从副线程继承一个。


如果应用程序在全局范围内都没有设置过的话，那么这个类加载器默认使用的是应用程序的类加载器。那么现在我们知道了，有了线程的 3 下轮加载器，那么 GNDI 服务使用这个线程三下轮加载器加载所需要的 SPI 代码，也就是父类加载器请求子类加载器完成这个加载的动作，这种行为实际上就是打破了双擎有限模型的惩治结构来逆向的去用了调类加载器，实际上它就已经违背了这个双擎违背的一些一般性原则，但就是确实是业务有这样一个场景我们也只能去接受。所以说加尔中所有的 SPI 的加载动作基本上都是采用这种判断的实现。


另一方面，其实我们在 Java 开发的过程中，为了实现热插拔、热部署或者是一些模块化，也就是说我们添加一个功能的时候，我们不去重启我们的服务，只需要把这个模块连同它的类加载器 h 换掉，又实现了一个代码的热替换的一个操作。其实对于代码热替换，我们加码程序中基本上有一个基本的共识，对于 OSGI 类加载器它是值得我们学习的，如果说我们弄懂了 OSGI 的实现，那么可以说是对于 Java 类加载器的这个精髓就已经掌握得非常好了。


接下来我们来看一下汤姆 case 的类加载机子，那么我们先想一下对于汤姆cat，如果使用默认的类加载机子有没有什么问题？比如说对于汤姆cat，这是一个 Web 容器，那么它要解决一个什么问题？首先一个 Web 容器它需要部署多个应用程序，那么不同的应用程序可能会依赖同一个第三方类库的不同版本。比如像我们的spring，我们的Jackson，我们的 j 森处理的一些操作，它不能要求同一个类库在同一个服务器里面只有一份，就比如说我们的一个 template 部署两个哇包，这两个哇包依赖了不同的 spring 版本，那么所以说这样的话，因为我们这个相同的 JVM 里面出现了同一个类的多个版本，那么因此要保证每个应用程序的内裤都是独立的，太太能正常运行。所以说需要保证我们不同的，我们的外包里面的内裤的信息是进行一个隔离的。


那么部署在同一个 Web 容器中，相同的类库，相同的版本当然是可以共享的。比如说像我们一些公共的一些信息，比如说我们为了减少汤姆派的加载的部署，我们可以都把它放到汤派的对应的 lab 包下面。否则的话，如果这个服务器有 10 个应用程序，那么要有 10 份相同的类库加载进虚拟机里面，这也是增加了我们这个内存的占用。


对于外部容器，它也有自己所依赖的一些类库，不能用于应用程序的内裤进行混淆。比如说基于一些安全考虑，我们应该让容器的内裤和程序的内裤隔离出来。比如像容器下面，对于 template 下面的内容，我们可以称之为容器的内裤，那么我们放到对应的外包下面，或者说 Web APP 下面，对应的 Web info 下面的lab，它可以是我们应用程序的一个类库，就是 Web 容器。


它还要支持 GSP 的修改，也就是说我们在写 GSB 的时候，我们可以动态修改这个文件，我们只要重新刷新，那么就能看到我们修改的效果，我们知道 GSB 它在处理的过程最终也需要编译成卡死文件，还能在虚拟中进行运行。但是程序运营后修改 GSP 当然是非常常见的，否则我们在用 GSP 过程中，它的一些优势也就不存在了。所以说 Web 容器它需要支持 GSB 修改后不用重启，也就是可以重新加载。


那么我们看到这几个问题，他们开始如果使用默认能力加载机制行不行？当然是不行的。我们看第一个问题，如果我们使用了默认的类加载器，那么是无法加载两个相同类库的不同版本的，因为加载的过程它都会委托它的父类去查找，那么如果说有一个类已经找了的话，它就不需要再找了，它只能去使用已经加载类库的这个类，那么这就是当我们默认的内加载器，不管你是什么版本的，只要是你的全限定类名，也就是你的包名和类名如果是相同的那么它就只有一份。


接下来我们再去讨论一下关于 GSP 的问题，那么通常我们的 Tom cat 的文件 GSP 我们可以直接修改，及时生效，那么这样的话就涉及到我们对于这个类也就可以动态加载，那么我们是如何实现？对于 JSV 文件我们可以是动态修改， JSV 文件其实本身我们也可以理解为它会动态的变化与 plus 文件，那么如果修改的话，它的类名还是一样的，那么类加载器会直接去获取方法中已经存在的，那么修改的 GSP 它可能不会重新加载，那这个怎么办？所以说我们需要去采用直接卸载掉这个 JSP 文件的类加载器，所以说你应该想到了每个 JSP 文件都对应一个唯一的类加载器，当一个 JSP 文件修改了，那么就直接去卸载这个 GSP 里面去重新创建加载就可以了。所以他们 cat 现在是怎么实现的？它确实也是去这样去实现的。


那么我们看这个图，这里面像我们的启动类加载器、扩展类加载器和应用程序类加载器，跟我们加尔默认实现是一样的。那么对于他们开的它实现了哪些？它实现的像 common classloader，这里面肯定 liner classloader 以及 third classloader，这里面还有 YVP car 的 loader 以及 GSP 的loader，这里面它每一个 card loader 分别起的作用是什么呢？比如说如果说我们对于 Web last loader，它们开的自定义的加载器的话，它已经加载，像这些比如 comments service 或一些共享，像 Web AP， Web info 下面的话，对于我们 webp 类加载去进行加载。


说对于我们的 common cast loader，它是汤姆 cat 最基本的类加载器，加载路径中的 class 可以被汤姆 cat 容器本身以及被各个 Web AP 进行访问。那么 containener loader，对于 containener class loader，它是它们开的容器私有的一个离加载器，那么它加载的路径中的 class 对于 Web v p 是不可见的，所以说这里面大家一定要区分出来。


另外一个 third class loader，我们可以理解它是各个 Web APP 共享的，那加载器加载路径的 class 对于所有的 Web AP 是可见的，但是对于他们开的容器不可见，我们去细想一下这些划分，那么这里面的 Web AP class loader 它的特征是什么呢？它是各个 Web 私有的类加载器，也就是说每一个我们的一个瓦包，它对应一个 VIP 的 class loader，那么加载的 class 只对当前这个 Web 容器是可见的，那么 GSP 的话，它的作用区间更小，只有这当前这个 GSP 去使用。所以说对于 common class loader 能加载的类，它都可以被 cleaner class loader 和 third class loader 所使用，从而它实现了一些公共类库的共用。但是对于 Calendar classloader 和 third class loader，自己能加载的类，对方都是相互隔离的。这里面的是 YVPE classloader，它可以使用 seed class loader 所加载的那些类，但是对于每一个 Webp class loader 实现之间，它也是做到相互隔离的。
