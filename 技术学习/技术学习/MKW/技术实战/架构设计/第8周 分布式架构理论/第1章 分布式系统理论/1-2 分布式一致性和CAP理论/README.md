这个 c 是代表着一致性，那这个一致性就是所谓你分布式系统当中，它有很多的数据，备份，主从备份等等。那么在同一时刻，此时此地此刻此秒，你的这些所有备份当中的同一份数据，它的值是否一样？等于说是你从任意一个备份当中访问这条数据，应该可以获得一样的值，那这个我们叫做强一致性。


这个 a 代表着可用性availability，那体现在分布式系统当中。可用性是指你的集群当中有任何的一部分节点，它出现了故障无法访问的时候，那你集群当中剩下的节点仍然可以去继续提供服务。那这个指标在大型分布式应用当中非常的常见。比方说我们经常会用你的应用可用度有几个 9 来衡量你这系统的稳定性。


那最后一个 p 分区容错性。这个从字面意思看不太好理解，那它代表什么含义？简单来说，它是指你的分布式系统当中的这些节点，如果经过一定的时间内仍然无法达到数据一致，那么就意味着你的集群当中发生了分区的情况，就像脑裂一样，一分为二，那在这种情况下，你就必须在前面我们提出的这个一致性和可用性这两者之间做一个二选一的选择。那这就是我们前面说到的 cap 理论当中的小纠结。


什么纠结？从一致性和可用性这两者之间选择一个，然后结合分区容错性，这就是这个 3 选 2 的由来。为什么只能 3 选2？一句话来解释，就是说你如果要保证系统的强一致性，那么必然就要在可用性上面打一些折扣，做一些牺牲，体现在咱具体的应用层面，老师这里跟大家随便看一个小例子，我们在分布式系统当中经常有这样的一个主，从节点的设置，

一个主节点下面挂着节点一节点2，那通常这有点像 must slave 这样的结构，那主节点负责写入。而从节点主要从主节点同步信息，以供下游的服务集群来读取这些数据。


那这里节点一下面我们挂着服务集群123，那节点 2 下面同样的也挂了一堆服务集群，那这幅图体现出来的就是一幅岁月静好的图片，那我们的集群从任意节点下面可以读取到相同的数据，那因此我们这个集群之间可以去互相调用岁月静好，对吧？但是谁都不知道明天和意外哪个先到来，有的时候是意外先到来了，主节点挂了，无法向从节点写入数据。


这种情况下，我们假设你左边的有一个服务集群，它失效了，那他想将这个失败信息上传给主节点也不可能了，对吧？因为主节点挂了，那这种情况下，他失效的这个信息能不能同步给节点？当然不能，那在这种情况下，你的服务集群与服务集群之间的调用就可能访问失败。


这个场景就说明了在分布式应用当中，如果不能保证一致性，那么将会产生什么样的后果。所以一致性同学们是分布式系统设计当中一个非常重要的环节，那比一致性还重要的可能就是可用性了。前面说到它俩是二选一，水火不容，你注重一致性，那么你必然要损失一部分的可用性。那如果你以可用性为先，那么恐怕就不能保持强一致性的约定。那我们怎么在这之间做一个平衡呢？有什么手段方法吗？哎，有，这就是我们这一节接下来要跟同学们介绍的主要内容，
