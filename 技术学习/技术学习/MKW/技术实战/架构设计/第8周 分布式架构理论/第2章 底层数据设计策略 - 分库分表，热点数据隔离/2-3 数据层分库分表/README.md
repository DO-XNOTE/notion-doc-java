垂直分表和水平分表。


OK，垂直分表，这里怎么来理解？我跟同学们打一个比方，那因为老师这经常做商品中心相关的业务，咱就拿这个商品来说，你商品上面有一些字段，比方说 name price，还有那种 blob 类型的description，那它是商品的富文本详情，或者它是商品的主图，那这个主图通常也是二进制的这样的大的字段。


okay，那对于这些字段来说，我们的垂直分表的含义是什么呢？我们一看便知，那就是把一个表按照字段，我们把它分到不同的表中，那每一个表中存储其中的一部分字段，比如说这个经常查询到的字段，像 name price，我们可以把它放到商品的主表当中。


那对于一些不经常查询，并且非常占用 i o 的这样一些字段，比方说 Claude 类型， blob 类型的这种字段，我们把它单独抽出来，这里了 description 商品详情给它抽成一个叫 product underscore text 这样的表， image 商品的图片我们也把它抽到对应的表当中。那对于大部分业务来说，你只用到商品的主表当中就能获取到这些信息。


OK，这个就叫做分表的垂直拆分。那咱们在实践领域当中，就像之前说的，把这些 blob club 类型的大字段，那种 IO 效率比较低的字段，把它分到单独的表中，与此同时一些相对冷门的字段也可以把它给拆分出来。那垂直分表相对看起来非常好理解，实现起来也非常的容易，没有什么技术难度。


那水平分表又是个什么鬼呢？比如说我们这里分了三个商品表， product 01 和22，那这三个表当中它们的字段都是完全一样的，但是不同的数据，不同的商品会落在不同的表中，那怎么来区分哪些商品该漏在哪些表中呢？这里就要谈到水平分表的这个路由规则。


比如说我们常用的有两种，那第一种我们对于商品表是经常采用这种手段，你的商品 ID 做一个取模，比如咱现在有三张表，那你的这个模就是3，你商品 ID 取模之后的值，那这个值是几？你就对应的把这个商品存到哪张表中。这有一个显而易见的好处，就是你的单表的数据量会显而著地降低，这样的话就能整体的提高你的查询效率。OK，那这是一个维度，商品 ID 去取末。


还有一个维度，我们通常在业务中会应用在订单表中，比如时间分片，你过去一年的订单，过去三个月的订单，过去 10 天的订单，它会根据你的时间片来分到不同的数据库当中。那这两个业务上分表的维度都有自己的优点和缺点。比如说我们通过商品 ID 取模的形式这样来分表，一个显而易见的缺点是什么呢？你不好水平扩展，假设你现在是三台机器，大家相安无事，但是当你增加到第四台的时候，同学们，你增加到第四台，你这个膜变掉了。这样一来，你相同的ID，因为它的膜变掉了，你取膜的时候就可能被路由到不同的表当中，而这个表原本可能并不存在你要查找的商品信息。


因此对于水平分表，通过 ID 取模的这种路由方式，每次水平扩展都要伴随着数据的迁移，怎么迁移？我们后面的小节再说。好，那这是它的一个缺点，我们再来看订单时间片，那咱根据时间分表没有这种问题，对吧？你随着时间往前走，那你每次都是新加一个表，比方说当前这个表是 1 月份的订单，那到了 2 月份，你再添加一个表，是 2 月份的订单，那看似没有这个问题，但是同学们有没有想到你根据时间分片，有可能在某一个时间段你的订单量非常的多，比如说双十一对吧？那你这当月的这个数据有可能比过去 6 个月的数据都多，因此时间片分片它也有一个缺点，就是数据分布可能会导致不均匀。
