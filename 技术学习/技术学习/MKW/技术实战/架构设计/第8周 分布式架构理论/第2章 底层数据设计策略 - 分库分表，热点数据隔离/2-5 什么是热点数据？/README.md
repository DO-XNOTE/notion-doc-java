321 坏了，系统炸了，那为什么到 321 之后系统才炸？其实这就牵扯到一个前端调优的小手段，则往往在 321 这个倒计时的过程中，大部分请求其实是被前端拦截住了，它并不会实际的到达后端。而当你的这个真正的请求到达后端的时候，短时间迅速涌入的大量请求去频繁地访问一个热点的商品。


有的同学说了，我用缓存来扛可不可以？当然可以，但是在真正的热点数据面前，你想象这是一款享场在拍卖的全民商品，在这种数据量下，即便是缓存也是扛不住的。在下一节里老师再跟大家具体讲为什么缓存扛不住，那从当前的这个场景来看，我们预先其实是知道它将要有这么一款热点商品的出现，因为对于秒杀活动来说，我们要去提前报名，那这个对我们的系统设计流程来说叫预知的热点，我可以对它做一些提前的预防措施，对不对？具体有哪些预防措施，我们在下一节当中再跟同学们细说。


其实在老师眼里可以预知的热点根本就不是什么技术的问题，但是互联网场景下对这种预知热点的数据有一套非常成熟的应对措施，那在老师眼里预知的热点，可以预料到的热点，其实都不存在什么技术问题，但是有这么一种热点它是不可预知的，那可能会对系统带来非常大的灾难。即便是现在一线二线的互联网公司碰到这种事情可能也会栽一个跟头。
