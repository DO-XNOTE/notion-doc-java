那从数据层面我们可以把它理解为底层的数据层面，这里提到了底层数据可以有很多种的方式来支撑高并发，还有高可用的应用，比如说你的读写分离和集群的扩展，这是从我们的数据库物理库的层面来做的一个高可用保障。


OK，那么同时还有主从的备份，这个相当于是在数据库的层面做一个 Dr 容灾。okay，那从业务层面来考虑，我们可以做一些数据的异构和分库分表的方案，来支撑你的海量的数据查询请求。那么我们这里还可以去回答一些缓存相关的问题。结合业务可以去看某些一致性要求不高的数据，我们可以把它落在缓存当中，尽可能地降低底层数据的访问压力。因为我们的底层物理库，不管它是 Oracle 还是 MySQL 等等，那这种关系型的数据库，它的并发能力其实是非常非常弱的。


相对于咱的缓存或者是一些 no SQL 的数据库来说，结构化的数据库在性能、吞吐量各方面上来说都是要逊色一点的，那么这里是底层数据，那还有我们可以就热点数据做一番探讨，这就相当于把咱这个话题做了一个升华，那热点数据热点方案就是区分一个高并发和超高并发应用的一个分水岭。


那么问题又来了，对于这部分的热点数据，我们如何在应用层面做一定的防范，避免我们的应用被某些热点数据打穿。那么这里将前面课程当中跟同学们介绍了几种方案，比如对于可以预知的这些热点数据，那么我们可以提前采用热点库，或者是构建本地缓存加多机缓存的机制，那在热点数据的流量到来之前，我们可以做足准备。


OK，那么这是一个方面，那对于动态的热点数据来说，你无法提前预知的数据，那么我们这里就可以应用到前面说的数据侦听来，从咱的这个接口参数聚合出来的分析结果，或者我们日志埋点的统计，这里动态的分析哪些数据即将成为热点，然后再将这个热点数据把它通过推送的方式。
比如这里可以应用MQ，那推送到整个集群当中的节点，在节点接收到这个通知之后，采用预知热点的方案在本地构建缓存。或者也可以在推送到节点通知之前，先构建起热点库，然后再推送到节点，那么每一个应用一旦接收到这个通知送达，然后就可以去转而从热点库当中获取数据。
OK，那这是预知热点还有动态热点的两个思考的方向，OK，那接下来我们再来问同学们最后一个问题，**那分库分表之后，如果我的业务量再次翻倍，那我需要做扩容应该如何来处理？**那在**前面章节当中跟同学们讲到了这样几种方案，就可以帮同学们从容的应对这个问题。**

去了解一下分库分表它具体应用的某些业务场景，那因为大部分同学可能知道分库分表是什么业务含义，那它能解决什么问题？但是这个技术可以套用在哪些具体的业务场景里？场景化的能力，希望同学们可以通过一些开放的资料，自己去做一些调研来去了解一下这背后的应用场景。


OK，那这是第一个作业，接下来一个是希望同学们可以搭建一个非常简单的小 demo 来处理热点数据，那这个小 demo 是在咱枯燥的理论学习之外，给同学们添加一些动手的小环节，那它主要做哪些事？这个 demo 只有这么几步，第一步它可以通过 MQ 接收一个消息推送，那我们假定这个 MQ 的消息推送，它会推送一个热点的ID，我们可以把它当做一个商品的ID。


OK，那么接下来推送到这个热点 ID 之后，我们的具体的应用就从 Redis 集群当中获取到这个数据，并且把它缓存在你本地，也就是你当前服务器的 local catch 本地缓存当中。那这一步要求同学们预先在 Redis 当中把这个热点 key 给它先设置进去。


OK，那第二步完了之后，第三步我们可以写一个API，那么这个 API 对于某些特定的ID，也就是经过消息推送的ID，它不会去到 Redis 当中获取到数据，而是直接从本地缓存当中获取，那这就是一个简单的处理热点数据的一个小demo，一个非常廉价的技术方案，同时在生产环境当中应用的也蛮多的。


不过这个方案它其实有一个小坑，诶，什么坑？我们这里通过一个思考题来让同学们去想一想如何来趟过这个坑。我们在接收到一个动态的一个热点 ID 之后，肯定会通过某种机制缓存在本地，这样的话咱的第三步中的接口才会知道这是一个热点ID，那我需要从本地缓存当中获取，而不是到 Redis 当中获取。因此你本地你的 Java 的 JBM 当中一定有一个存储结构来保存了所有热点的ID，对吧？那么问题来了，假如我的应用如果重启了，那你 g v m 当中保存的这个热点 ID 是不是就消失了？因为它是相当于在内存当中保存的，这么一来，当我在调用第三步的接口的时候，那我们的服务就无法预知你的这个热点是在本地已经构建好的缓存，还是需要从 Redis 当中获取。那么这个思考题教给同学们处理的方式非常非常多，但是老师有一个要求，同学们自己独立思考，可不要到提问区去问老师应该怎么做。那这其实也是锻炼同学们解决问题的一个小测试。OK，同学们，那这一节的内容就到这里结束了，我们下一小节再见。
