那在主库下面依次裂开sleep，这三个是读库，那主库和读库之间通过一种叫做 binlog 的机制来做备份replication。我们对所有数据的写入操作直接打到主库上面，那读取的操作 read 操作则是打到它的读库上面。那这种业务场景下，随着你的 read 请求越来越多，比如说前面咱的商品中心，那你的这个业务平台上面有了更多的客户，那大家都来去查看你的商品，那自然而然你的这个读取请求就会变得增多。那这种情况下，我们通常只需要去通过水平扩展的方式来把它的读库增加新的节点，就能非常容易的去解决这个性能瓶颈，okay，因此这就是读写分离的目的。


通过读写分离，大家可以看到还有一个额外附赠的好处是什么呀？咱的这个读库如果挂掉了，因为你的读取压力特别大，把整个读库集群打崩了，那这种情况下它会不会影响到你的这个 master 节点呢？并不会影响你的写入请求，依然可以去正常的操作。那这是读写分离的一个显而易见的好处，那这里说的只是一个表象，其实咱们读写分离开来的一个原因就是它们俩会相互拖累。

当你在 write 的时候写入一条数据的时候，其实它在后台的数据库层将会给它上一个排他锁，那排他锁就是我们俗称的斜锁，那当一个数据被打上了一个斜锁，那么直到这个锁被释放之前，那你都不能去修改或者读取这个数据，那因此才说它是一个排他锁。


OK，那对于 read 请求来说，它就温和了很多， read 的请求会给一个数据加上共享锁，那共享锁我们把它称作读锁，那这个读是读书的读读锁，OK，那一个数据被加上读锁之后，那你其他的事物只能对它加同样的读锁，而不能去给它加一个排他锁。那这就保证了你这条数据只能被各种共享的线程来读取，而不能被修改。


那从上面的这个描述当中，同学们能看出这两个锁之间有个什么关系？它们之间是一个互相竞争的关系。OK，那如果我们人为的通过这种读写分离的方式把这哥俩给它隔开，那这两大了老死不相往来，互相见不着，那自然而然的就消除了这种竞争，那这就是我们读写分离性能比较高的一个原因之一。
