那么一个应用它调用缓存的这样的一个过程，通常就是发起一个请求去获取缓存，那如果获取不到，我们从数据库当中直接获取，并且把它写入到缓存当中，那在这个过程里，缓存通常是一个分布式的集群架构，那它里面会保存正常的普通数据，那同时也有可能会有一些数据变为热点数据。


那如果我们的一个访问请求打到了 Web 服务器里，然后这些海量的请求都被打到了你当前缓存上的热点里，那这种情况就容易造成缓存因为它的毒压力过高而导致服务不可用好。那这种情况下，我们怎么样来通过多级缓存来解决呢？同学们也许猜不到这个多级缓存把它放在哪儿，咱一般可能会以为在当前的分布式缓存当中再加设一层缓存，那其实多级缓存还真不是这么玩的，它是放在哪儿的？我们来看变长，诶，你的 Web 容器腾出了一块儿区域。


好，这就是你多级缓存放的地方。什么呀？ a g v m OK，我们也叫它堆内缓存。那因此对于热点缓存数据的访问，通常首次从缓存当中获取，获取到之后，我们会把它保存在你本地，作为一个堆内缓存。如此一来，你只用从当前的 JVM 当中获取这个热点数据，它可以大大的降低分布式缓存的一个压力。OK，那这是一个应用它的 QPS 从低到高的一个必经的过程，从分布式缓存的方案过渡到分布式缓存加本地缓存。


好，那老师这里跟大家提一个问题，本地缓存它存在 GVM 里面可能会引发什么问题？假如你当前保存的是一个大对象的热点，它的容量可能会非常大，那由于它是占用的 GVM 的地盘，你在人家的地盘上倒腾，那自然而然就有可能导致一些 GC 现象，频繁的GC，那这是不是会影响到你当前机器上运行的程序？肯定会影响你服务的响应实践。那对于这种情况，我们来怎么改善好？这里就要把 GVM 咱称作堆内缓存的这样一种本地缓存，替换成一个堆外缓存。


OK，这个堆外缓存是拿 JVM 来比划的。 JVM 之外的堆是什么呀？你的内存区域对不对？那在 Java 当中是可以去操作读取 GVM 以外的内存区域的，那这样的一个过渡，从单纯的单一的分布式缓存过渡到分布式缓存加堆内缓存，在最终过渡到分布式缓存加堆外缓存，那这三步走是你应对高并发场景的一个非常对症下药的解决方案，那在业界的应用也非常广泛。

咱的每个 tear 服务器，我们可以把它去当作一个 Redis 服务器，其实都是缓存了，那每个 TIER 服务器里面它都有一个 hot zone，就是存放热点数据的地方，那它扔一个数据服务器，它的 hard zone 里面其实存储的热点数据都是一样的，那这之间也没有什么权重之类的关系。那你的客户端对热点数据的读取，拿到一个 key 来读取数据，那这个读取的请求它会随机到任意一个你的数据服务器的 hot zone 区域。


OK，那这样的一个热点数据请求就被散列到了很多个节点，你整个集群当中的很多节点都会来承接这个请求，那我们叫这种方式叫做热点散列。OK，那热点散列技术其实还依赖于一些额外的配置中心，在我们的这个客户机发起请求之前，它会去获取到咱这个所有的 TIER 集群当中的数据路由的关系表，与此同时你的这个 hot zone 的节点的位置信息也会同步的获取到。那获取到了之后，我的客户端就会认定其中的一个 hot zone 作为一个固定的区域来做数据的读取。这个过程它一旦认定了一个 hot zone，它就不会再去松口了，不会改变。那除了有一些特殊情况，比方说你的这个机器数量发生了变化，那你散列的配置也相应的发生了变化，那这种情况下它可能会换到其他的 hot zone。


但是在这个过程里，你的一个服务器始终都是访问一个好自动的，那你的客户端是如何知道你当前要访问的数据应该到 how zone 里访问，这个情况就靠你的客户端与服务端之间的持续反馈，当你收到服务端反馈给你的信息，说当前的这个 key 叉key，它是一个热点。


OK，那在你这个当前热点生效的期间内，你的客户端在访问这个 key 的时候，就会首先的尝试从 hot zone 节点当中去先获取OK，那如果你的 how zone 节点不存在这个数据怎么办？那就到正常的区域当中把这个数据给它读取出来，那这种模式我们来理解一下，就非常的像二级缓存，对不对？OK，正常区域和 hot zone 形成了一个二级缓存的这样一种架构，那咱读取到了这个信息之后，不能把它放着不管，对吧？我们要把它去更新到 hot zone 里，那这个过程其实是一个异步的过程，第一步写入热点数据，OK，这个过程就这么简单，那对于一些其他的服务器，同样的也是执行相同的一个流程来获取热点数据。
