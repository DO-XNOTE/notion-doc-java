接下来应用系统往往会先去到缓存当中，看你的数据是否存在，如果不存在就到数据库里把这个数据读取出来，这是缓存最通用的做法，那这个流程完全没有问题。但是对于一些热点数据来说，这里就有坑了。比如说我现在有一个商品是热点商品，它在缓存当中并不存在的原因有很多，有可能是缓存过期了，或者是缓存还没有被加载。这个时候假设所有用户都来访问这个热点商品，那么由于缓存不存在，这些用户请求就会直接的落到你后台的数据库当中，这样一来你的热点就会穿透缓存，直接打到后台的数据库当中，这就是叫做热点击穿。


OK，我们有什么应对的方法吗？当然有，这里的应对方法有新的招数，也有之前老师在前面的课程当中跟大家讲过的，我们一道来复习一下的。第一个防御方式就是要调整你的缓存过期策略，这里一个最通用的做法就是永不过期，你的数据一旦从数据库加载到了缓存当中，那我不给它设置过期时间，对于热点数据来说，那它永远都会打中你缓存当中的数据不会穿透到后台的数据库，那这个其实我在这里叫它初阶玩家的玩法，高阶玩家怎么玩？我们有另外一套方式，高阶玩家前面老师是不是跟大家讲过 Canal 的一个用法？

我们可以利用 Canal 做一个数据异构，然后把缓存当作一个读写分离的架构，也就是说你读取永远都从缓存当中读取数据，而缓存和 d b 之间的同步，我们采用 Canal 做增量数据的同步，这样一种架构来说，你就可以把缓存当做是一个读库。这个玩法是一个全量同步缓存的玩法，也就是说你数据库当中的值会被全部的同步到缓存当中。


okay，那除了这个玩法以外，我们还有一些其他的，比方说热点缓存策略。前面老师在一节课程当中跟大家有说过，热点缓存，我们可以通过分析系统的调用日志来甄别出热点，然后将你的热点访问请求打到一块特殊的缓存区域，或者是特殊的缓存库，这样一来你的热点数据也就不会影响到普通的数据。你在普通的缓存库当中依然可以设置过期策略，在热点库里你可以设置不同的缓存过期策略，比如说永不过期，这样一来也会节省大量的缓存空间。


OK，那这是热点缓存策略。还有一个也是比较常用的，像 Redis 当中，我们可以使用一个互斥锁。什么意思？比如说你在缓存到期的时候，我会使用互斥锁给你的这个 key 加一个锁，那同时期的其他热点请求到来的时候，由于咱的这个互斥锁已经被加到了缓存上的这个key，因此其他的请求不会去做重复的读取，这样可以去显著地降低后台 DB 的承载的读压力。

这个玩法其实也是比较初解的玩法**，高阶玩法，这里就要打一个时间差，也就是说我不会在你将要到期的时候才给你上锁，而是在你将到期的前一分钟，前两分钟设置一个时间差，提前给你加上一个锁。如果其他请求过来的时候，发现你当前这个锁已经处于锁定状态，那说明在到期之前已经有一个请求去后台数据库来获取值了，那么这些其他的热点请求就可以大大方方的从缓存当中把这个数值拿出来。那这提前上锁的这个请求，它可以去做一个异步的缓存刷新，然后在缓存刷新结束之后，再把这个锁给拿掉。这个是我们在互联网场景下面更常用的一种玩法。**OK，那这就是缓存击穿的主要的解决方案。我们再往下看，接下来带大家去了解一下什么叫缓存雪崩。

假设我们当前的缓存当中有P1、P2、 P03 这三个数据，这三个数据我们注意看后面有一个时间，这个时间是它们的过期时间。缓存雪崩是什么含义？它是指当我们在缓存当中的 key 在一个相对集中的时间，集体团灭，全部过期了，这时候你的应用系统的所有请求就会直接打到数据库当中，直接把数据库打崩，这是缓存雪崩的一个表现形式，那么它的应对方案也非常的简单。


这里第一个方案，我们同样的可以去调整你缓存的过期策略。早安，两分钟之前才刚刚讲过，就在缓存击穿那里，我们的这个缓存过激策略可以采用同样的一个策略。OK，那还有没有其他的解决方案，咱可以在这个时间上做一做文章。既然它在同一时刻过期，那我们在设置缓存的时候，可不可以这个样子去把它的过期时间做一个散列？它的过期时间由基础时间，比如说 5 分钟过期，好，这是基础时间，再加上一个动态的一个散列时间，比如说我在后面加 1 分钟到 10 分钟之间，随机的给它取一个值，那么这样一来你的缓存的过期时间就分散开来了，它不会在同一时刻产生集中过期的这样一个现象。这是缓存雪崩的两个处理方式。当然了，我们在真实的生产环境上，有时候也会采用多级缓存来预防缓存雪崩的出现。比如说在 Redis 后面我再加一个 MEM catch 其他类型的缓存，做一个多层兜底方案，降低缓存雪崩的发生几率。好，但老师在这最后还要跟同学们做一个额外提醒。


大家可曾听过缓存预热其实在像双 11 之类的场景中非常非常的常见，也就是说你在业务开始，比**如双十一倒计时 321 开始的那一刻，在这一刻之前，如果我的缓存当中什么数据都没有，其实它是一个变相的缓存雪崩场景，你为了构建缓存，导致所有业务请求在同一刻都打到了数据库。**因此在像双 11 秒杀场景之类的这些业务里，我们通常会**采用一个缓存预热阶段，也就是在业务正式开始之前，把你的缓存给它构建起来**。

我现在卖两件商品，这两件商品分别是 P01 和P02，它在缓存当中有，在数据库当中也有，那这时我们的终端用户偏偏不想买这两个商品，我**想买P03。 P03 是个不存在的商品，因此你从缓存当中永远无法获取到它的值，这样的话，你的请求就会被直接绕过缓存，尝试从数据库当中读取。但是由于 P03 在数据库中也不存在，那因此它在缓存当中也无法构建出来。那一些不怀好意的用户也好，黑客也好，他就可以利用这个机制去把你的访问请求直接打到后台数据库里，然后把数据库给打崩。**那怎么办？好吓人，似乎没有解决方案，哎。
