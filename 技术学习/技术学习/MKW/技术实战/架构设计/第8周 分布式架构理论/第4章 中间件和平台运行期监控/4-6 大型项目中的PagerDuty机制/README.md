那一个故障单到达了我们的处理人员这里，那处理人员这里要发送一个指令，这个指令是我们做这个故障响应当中一个非常重要的步骤，叫 a C k 它就是 acknowledge 的意思。也就是说我知道这个问题了，朕知道了，不要再来烦我了。那么从这一步开始，你的故障响应机制就不会再层层往后传递到这个人这里，就由他来接手来处理。


OK，第二个时间点，我们这里去标记一个叫 rude cause，也就是说我发现了这个问题的根本原因。到这一步问题已经清楚了，接下来就是一个打补丁上线的过程，那最后把这个问题解决好。在这整个过程当中有三个时间点非常重要，分别是你问题发生的时间，以及你的团队去 a C k 响应这个问题的时间，再到最后他部署上线的时间。那通常这三个时间点我们会用来记录你解决问题的一个绩效，你从发现问题到响应这之间的时间差，以及响应之后到解决问题这之间的时间差，可以去评估你的当前这个故障单的影响范围以及影响的时间，那这些指标将会最终被用来你的故障单定级。


通常一个故障之后，我们还会去开展一个叫 RCA root cause analyst，也就是根本原因分析的，会来总结沉淀你这套应急响应过程当中的一些问题，以及如何避免同样的故障重复产生。那前面说的这几个时间点，还有你影响的客户的范围，那最终都会被作为一个故障单定级的指标，那同样还有一个最重要的指标在阿里系里面就是谈虎色变，一谈到这个指标，大家脸色就变得正经了起来，这个指标就叫资损，那资损是你做故障定级的一个最重要的一个指标，那就是给你的业务带来的多少资金的损失。那它可以有直接的资金损失，也可以是你通过对故障影响的评估，间接估算出来的一个资金损失。OK，那到这里咱们在大型项目当中的应急响应模型就跟大家分享到这里了，接下来
