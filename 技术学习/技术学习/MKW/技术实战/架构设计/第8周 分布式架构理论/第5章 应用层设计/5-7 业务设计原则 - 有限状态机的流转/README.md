那一个订单下单完成之后，那通常它的状态就会被置为待付款这个状态。随着一个付款动作的完成，支付宝、微信各种付款方式完成付款之后，那你的订单就会变到一个待收货的状态，随着你的收货完成，订单履约步骤完成之后，那么订单就会被置为 complete 这个状态，那这是一个非常非常简化的一个订单的状态流转。那从这个图里面我们可以看到它的这些状态其实是预定义好的，并且随着一系列的判断条件，它们这些状态之间的流转的图序是一个固定的业务流程，那么通常我们就叫它为有线状态机。


OK，那在真正的业务里有线状态机其实并没有我们图示的这个例子这么简单，那往往会伴随着非常非常复杂的状态流转。那单就一个订单履约，那它其中都有很多的子状态，那咱平时做一些自己的demo、小系统、小玩具，或者做一些课程设计，毕业设计，一个订单数来数去就那六七种状态，但是在大型的互联网应用里，那订单的状态可就茫茫多了。


如果让我们设计一个这样复杂的订单的流转系统，我们可以去把这个模型做这样的一个抽象有限状态机的四要素，哪四个要素？首先第一个当前的状态，也就是整个流程前后状态流转之间的初始的状态。那第二个环节是触发事件，一个状态从当前状态移到下一个状态，它需要经历一个事件的触发。


OK，这是第二个要素。第三个要素是咱主要的业务流程所在的地方，那就是在你触发事件到下一个状态转变之间，这里有一个响应函数。OK，在你的响应函数做完响应之后，那么我们的这个状态就会最终抵达什么呀？抵达你的目标状态，OK，这是一个有限状态机，两个不同状态之间流转的四要素的条件。
