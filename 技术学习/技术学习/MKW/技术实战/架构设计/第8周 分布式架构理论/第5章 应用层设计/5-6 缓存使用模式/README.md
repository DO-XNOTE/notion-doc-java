那三两句话，把这个问题回顾一下咱的程序，第一步，写数据库，写完数据库之后，谁去来同步缓存？我们程序可不做这件事儿，这个麻烦的事儿交给咱的 Canal 去消费数据库的binlog，然后把它发布到下方的缓存当中。


这一步有两种做法，一种是我们自己写 Canal 的 Java 端的监听程序，然后根据一定的规则，一定的 key 发送到缓存当中。那**另一种更常用的做法，或者说是更高可用、更可扩展的做法是你的 Canal 这里发布一个消息，什么消息？ MQ 消息？OK，然后让下游各种业务方，他只要需要消费这个 MQ 消息，让他们自己写一个消费者，然后根据自己的业务逻辑，你该往缓存的存，你该往 ES 里面say，你就say。那这种应用方式更可以去把应用间做一个很好的解耦，同学们如果面试当中问到这个问题，可以采用这种方式，绝对是阿里系里面的标准做法。**


OK，那在同步完缓存之后，你的应用程序这坨屎一样的代码就可以从缓存当中直接读取数据，那这其实就像老师前面章节说的，这看上去好似一个读写分离的架构。那如果我们想这样做，有一个前提，同学们看这个标题，全量缓存意思就是你所要缓存的这个数据，或者说是它插入数据库的增量数据，在咱的这个缓存和数据库当中都应该存在，那就说你有我有大家才能有。


咱平时使用缓存大部分是带有过期时间的，但是在互联网场景下，那种超高并发应用里全量缓存的场景非常常见，那如果说我这个公司比较穷，咱是 18 线互联网公司，我没有钱买这么多缓存资源，我不想做全量缓存，我想做一个有过期时效的这种非全量的缓存。那你的代码和缓存数据库这三者之间就有一个复杂的三角链的关系，你要控制同步了，我们这里就讨论这个场景，在讨论之前，老师先把这个复杂度给它降低一点。

双删其实是一个非常廉价的一个手段，但是其实在像大厂应用当中，一般绝不会考虑这种方式能用并发写的控制，那么我们就不用采用双删策略。这个过程非常简单，就是说在你将要写数据库的时候，首先淘汰缓存，然后再写入数据库，写入完成之后做一个延迟删除，也就是稍微等待那么一小段的时间，再次把缓存淘汰掉，那么这种情况下就可以保证你完全在这段时间内没有脏数据，缓存中的数据会等待下一次缓存加载的时候再把它构建起来。
