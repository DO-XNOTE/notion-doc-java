**第一个就是长任务，第二个叫非实时。**什么意思？字面上就已经说得非常明白了，这个长任务就是指执行的非常长的一段任务，那非实时它的意思是指这个任务执行，它对实时性的要求并不高。OK，那这两个任务如果想利用消息组件解耦，它有一个必须满足的前提，那就是这个任务是可被异步化执行的。也就是说你当前的主任务逻辑允许把接下来的这个长任务或者非实时的任务作为异步执行，那这是它的前提。待会我们会通过一些实际的例子跟大家看一下这两项任务如何来做组件间的解耦。


好，那最后一个场景就是大家最老生常谈的发布订阅的场景，那发布订阅这个应用非常广泛，那总结起来什么样的业务它适合使用发布订阅？我这里跟大家总结了一句话，叫多下游相同语义，乍一听不太好理解，有点费脑细胞，其实它就是指多个下游需要消费一个相同语义的情况。那接下来我们就从这个常任务先开始，跟同学们看一下咱的实际的项目，实际的业务场景当中有哪些此类的长任务。


那第一个跟大家分享的，在**电商平台或者是支付平台当中，这个业务非常的普遍，**比如说我们的电商平台，支付平台，它都有一个商户的数据，

这个商户简单来说就是你的卖家，通常来讲我们卖家更更新了自己的一些信息之后，比如你的住址或者是你的付款方式，又或者是你商家数据的营业执照之类的变更。


那么这类变更通常我们会在后台触发一个风控的自动评分，它所需要的数据源来自四面八方，并且它的模型非常多，把一个个人数据跑完，整套风控模型也是一个相当耗时的任务，那这类任务我们可以把它归类为一个长任务，那通过消息组件来触发这个任务的执行，OK？还有比如我之前做商品中心比较多，在商品中心这个业务当中，你商品的编辑还有商品的发布动作其实是一个非常复杂的流程。用户在前端页面点击一个发布，通常我们不希望让用户等待整个发布任务结束，而这时候我们直接给用户一个返回响应，告诉他你的商品发布正在进行当中，请稍后来查看。


那在后台我们会执行一段非常长的商品发布流程，把商品数据同步被各个的下游的业务方，这个流程像当时我在阿里系的应用当中，这个流程我是把它作为了一个异步化的流程，由消息组件触发的一个商品发布的长任务。OK，那还有一些场景，比方说咱在很多系统当中都会看到一个报表，系统用户提交一个报表生成的申请，报表不用多说，大家就知道它的生成是一个相当耗费时间的一过程。通常这类任务我们约定俗成的也会把它作为一个长任务，发送一个消息组件到后台的报表中心，由报表中心的业务来消费这个消息，最终产生一个报表供用户下载。


总结一下这类业务的一些特征，第一个就是时间耗费非常长，第二个它内部的业务逻辑非常的重，那基于这两点我们可以看出来，如果能把这类业务通过一个消息组件来驱动，那么将大大提高你的系统的处理效率。但是它有个前提，咱前面也提到过叫可异步，也就是这类长义务，它必须在用户体验或者你整个业务流程当中是一个可以被异步执行的步骤。


为了达到这个异步执行，我们往往要在交互体验上做很多的变更。比如说前面的这几个例子，向商品中心的商品发布和报表的生成，那么我们这里需要让用户感知到这不是一个等待即返回的流程，而是一个后台执行的流程。比如说报表，我们公司现在的报表系统提交了任务之后，它可以通过一个页面去查询你当前报表的生成状态，最终它可供下载的时候，将提醒你当前的报表已经生成，可以下载。这就是为了让这些长任务异步化而做出的交互体验上的变革。

我来跟同学们看几个例子比。比如说咱的用户验证，你在登录任何网站的时候，需要去输入一个手机验证码，当你点击触发发送手机验证码的时候，其实在后台它就是通过消息组件来做的手机验证码发送，而不是一个实时的接口。

那我再跟大家举一个我老本行的业务，我一直在做支付，那在支付行业当中，咱有一个叫付款回调的接口，比如说同学们在线下买东西，我在将要下单的时候，往往要调用一个第三方的支付。OK，这个第三方支付可不是你自己的系统，它是第三方的支付供应商给大家提供的业务，那这个流程通常是你触发一个支付请求，同时你提供一个回调的页面，当你的第三方完成了这个支付动作之后，它将会去调用你的回调接口，给你做一个付款成功的通知。那么在这个流程当中，咱第三方业务处理付款的这个过程，那是一个同步调用，但是在处理完成之后，它的第三方付款业务在调用你的回调接口的这一步，我们往往会把它作为一个消息队列，采用异步化的方式将当前的支付状态回传给你自己的业务。OK，那这类场景下我们也有两个标签，第一个是轻任务，同学们可以观察以上的这几个业务场景，它所要完成的业务都非常的轻量级，可以在相对来说比较短的时间内完成，与此同时它对你时间的容忍有一定的容忍性，它可以容忍短暂的一个时延，它对实时性的要求没有那么高。那对于这类场景，老师的经验告诉大家是能异步就把它给异步化。


这里再跟同学们提一个点，异步化其实目的不仅仅只是解耦这么简单。我们知道消息组件，消息组件其实是一个可靠性非常高的组件，与其你直接发送 API 调用或者 RPC 调用，消息组件的消息送达率以及它的容错能力、重试能力是比你业务层直接访问 API 保障性要高得多，这也是我们在实际的业务场景当中拿捏是否使用消息组件的考量的因素。
