第一步是由一线的执行者去提交变更，提交一个变更申请，它包括了你将要执行的 SQL 语句，以及你 SQL 语句你预计会被影响到的行数。那这两部分信息你通过一个内部软件提交给谁？来提交给前面所说的这个 owner 来进行什么呢？进行一步审核操作。


那审核其实还分两步走，如果你影响的数据比较少，比方说你的这条 update 语句只会影响几百个，或者是 1000 出头这样些数据量，那这是少量数据下，我的应用 owner 就可以直接给你审批通过。那如果你当前的 owner 他自己想要提交数据变更怎么办？通常我们有一个准则，就是必须你的变更要让另一个人去审核，那么你才能提交。那如果你是 owner 自己提交的怎么办？要找到你的主管，也就是这个 owner 的主管来审批一下，就可以去执行了。


那同学们要问了，如果是马爸爸来执行怎么办？哎，马爸爸没有主管哎，这个也许他自己就可以直接执行少量数据，但是如果马爸爸他没有主管，他想执行一个山库跑路的动作怎么办？诶，这个情况下我们的数据变更流程也可以堵住这个漏洞。山库跑路它影响的数据量比较多，当马爸爸这里要去执行一个大量批量的删除操作的时候，那这个请求让他自己主管批也没用，那必须把这个请求再路由到你的DBA，让 DBA 做一个最终的审核，那之后才能去执行 update 或者是山库跑路的操作。OK，那这个最后的这一步，那就是 step three，也就是执行。那当我们提交的数据变更工单被审核通过之后，那么你可以去自己的手动去执行这条命令。好，那这就是 1233 步走的数据变更的策略，那因此从这里我们可以看出，在阿里系的集团内部，它的权限是非常开放的，
