也就是说你一张表里的一个字段，它应该是一个非常细粒度的原子字段。打一个比方，你的电话号码通常有一个国家的编码，像中国是加86，再加上你后面的一串数字的号码。在电话号码这一栏，如果我们把这两个字段放在一个 column 里面，其实它并不能说是原子字段。比方说我们可以把它拆分成一个 country code，再加上一个 telephone number，这样的两个字段放在一块儿，那这就是第一个范式所要宣传的内容，很扯淡，对吧？第二个范式和第三个范式其实更扯，这两个范式其实说的是一回事儿，那他非要把它扯成两个范式，那我们可以把它放在这里一起来理解。


二三范式说的是你数据库表当中设计的字段应该是主键相关，不要做冗余。 okay 什么含义？比如说下面这样一张订单表，它有这么几个字段，你的订单的 ID 是什么？他什么时候下的单？这个订单花了多少钱？然后他买了什么东西？同学们，看这里的前三个字段，这三个字段是订单相关的，那后面的字段其实是商品相关的，订单和商品之间的关联。那因此在数据库设计范式里，通常我们会把它这样区分，一个是订单表，另一个是商品表，然后再把订单和商品通过中间的一个关联表把它关联起来。那在经典的数据库范式设计的指导下，我们设计的表。同学们，一般来说，你 10 张表中只有 4 张表是业务表，剩下 6 张全是关联表。


数据库设计范式之所以是经典，因为它也具备一些时代特征，但是在如今的这种超高并发的互联网场景下面，数据库范式基本上没有什么用武之地。可是这数据库范式几十年的经典，说废弃就废弃了吗？**老师这里要跟同学们表达一个观点，就是尽信书不如无书，那每一个理论它又都有自己的时代特征，像曾经奉为经典的牛顿三大力学，对吧？那现在也并不是无懈可击。因此曾经被奉为经典的数据库的三大范式，在互联网的超高并发场景，其实基本上算是一文不值了。**


我们在互联网场景下面有一个特征，就是你的数据会被大量的冗余到各种表当中，那如果用数据库的三大范式来做设计的话，我们会把这个冗余全部剔除，在表与表之间做一个中间的关联表来做关联。


而在互联网场景下面，我们往往会采用很多的大宽表做冗余，

这样的方便查询那关联表我们其实并不建议去大量使用，因为在真正的超高并发应用下面，我们知道做 join 操作其实是一个非常耗费数据库资源的一个操作，我们通常在你的查询语句当中是只允许做一次，甚至是不让你做join。


所以如果同学们用 Hibernet 记住在这种超高并发互联网场景下面，一定要去关闭 one too many many too many 之类这样的桥接，不要让它在加载数据的时候主动带出一些关联表，其实这也是 harbonate 在阿里系应用当中应用非常少的一个原因，省了写 SQL 的事，但是你并不好去控制你的 SQL 语言。


如果你的底层设计非常复杂，确实要用到很多这种关联关系。老师在前一节当中讲过，对于这种有复杂关联关系的这种结构来说，我们可以做一层转化，把它去用作图形数据库来做这个查询。那图形数据库的特性同学们可以翻看前一节课程当中复习一下。对于大家来说，可能这是一个新的名词，但是在互联网的大型应用当中，像阿里系应用当中，图形数据库的应用非常非常的广泛，你越大型的应用，你数据之间的这种关联关系也就越复杂。那如果我们用传统的这种 DDL sticker 这种模式来做结构化数据建模的话，那这种关联关系其实并不是很好扩展，那我们用图形数据库可以很好地解决这个问题。


OK，那另一个解决思路是什么呢？我们可以去做大量的数据异构，把你的业务数据异构到很多不同的存储介质里面去，支持不同的查询维度。并且互联网场景里面有大量的快照信息需要保存，快照其实就是你的商品和订单他们的版本控制，这里就牵扯了很多的冗余数据，因此你在超高并发的互联网场景下，数据冗余是根本无法避免的。相反，我们会使用冗余来提高你的查询效率，而数据库的经典范式在这里只能成为拦路虎，阻挡我们向前进步。


那**互联网场景下面这个数据异构快照冗余其实是非常普遍的，比如说前面老师也跟大家举过一个例子，一条订单记录，那它可能被异构到多个不同的库当中。**

比如说我根据订单 ID 的维度做一个订单表的搜索，那这里可以很好的用分户分表来解决。那如果我想根据一个用户维度查用户过去的所有订单，通常我们这里还会把它异构到一个根据用户 ID 维度做分库分表的这样一个查询库当中。而且对于一些需要根据时间来搜索的，通常我们还需要根据订单的时间做这样的一个分库分表。


OK，这一条订单记录单单从底层的查询维度这里就被冗余到了多个库当中，这还不够，那为了满足前端的一些非常复杂的查询，那我们还需要把订单数据同步到搜索引擎当中，比如说 Elastic search、solar，还有阿里系当中应用非常广泛的 open search，这些都是我们的搜索引擎。那数据 EGO 的方案其实已经有非常成熟的解决方案了。比方说老师前面跟大家去讲的Canal，它这里就可以从一个订单库主库当中把你的数据变更同步给其他不同的存储介质。那在后面的数据库设计章节里，我们也专门安排了是一个业务场景的落地。


OK，那这是数据异构冗余。我们前面提到还有一个versioning，也就是版本控制这一说，比方说你的订单的快照，那就是你这个订单当时下单时候的样子。与此同时我们每个产品，每一个商品去做发布更新的时候，我们也需要保存商品的历史记录，这个叫版本控制。



比方说一个典型的场景订单快照，假设我们运用数据库设计的这种理念，你的订单快照通常是你的订单的历史版本，然后中间再做一个什么呀？再做一个关联表，关联到你商品的历史版本记录当中。但是在我们互联网场景下是不是这样，其实并不是OK，往往对于一些个订单的快照信息，则会在这里做一个聚合冗余大量的数据，比如说你营销计算出来的结果，还有你商品库的历史快照，以及你的下单用户库的用户信息等等，我会把这些信息全部聚合过来，通通把这个历史数据冗余到我的订单库里面。那这就是我们数据冗余的一个典型的场景，完全打破了数据库的三大范式，为你的业务开造。


其实数据库的三大范式，它是一个乌托邦，是个非常理想情况下的状态，但是往往这个理想、这个情怀要为现实所让步。**因此在我们互联网场景下面，老师以贯倡导的包括前面的章节，一直跟同学们去清醒催眠的就是业务第一，这么快怎么来？**
