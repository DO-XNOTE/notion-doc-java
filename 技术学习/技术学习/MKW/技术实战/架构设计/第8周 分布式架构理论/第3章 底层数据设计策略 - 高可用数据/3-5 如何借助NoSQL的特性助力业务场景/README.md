这个墙头草就是MongoDB，那我们常说 MongoDB 其实是摇摆与 SQL 和 no SQL 之间的一种形态，它是最接近 SQL 的一种。


no SQL 的数据库，那它的一个特性首先就是面向集合。怎么来理解？我们来看 SQL 数据库里面，我们面向的是什么呢？面向的结构化数据，面向的是数据库的表，这张 table OK，那一个 table 里面有一些记录， record one， record two，那同一张表里的不同记录，那它的结构也是相同的。
对比到 Mongo 这里来说，它面向的可不是table，它面向的是一个叫做 collection 的东西，那 collection 里面它保存的实际上是一条两条的文档，而这个文档其实是有唯一的标识，那这个标识是由 MongoDB 这边来生成的，那所谓的文档其实说白了就是一个 json 格式的数据。


我们知道 Jason 其实是一个相对来说 freestyle 的东西，它没有像我们的 MySQL article 这种 sickle 类型的数据库的结构化表的约束，因此我们可以在一个 collection 当中加入非常多变、结构多样的文档。那从这里我们就可以理解到 MongoDB tar 的一个很重要的一个场景应用，就是有复杂模型变更的这样一种需求，你的文档里面的字段经常会发生变更，那这种情况下就非常适合 Mongo 来操作。


OK，那还有你的这个文档，其实它是一个非常大的存储对象。OK，这种大对象来说，如果你把它放到 SQL 这种数据库里面就不太适合，那你一行记录当中包含的字段过多，但是我们在一个 json 格式保存的这样一种结构下面，它就非常适合这种大对象的存储。并且 Mongo 这里还有一个优点，那它的主从和 sharding 都是内置的功能，**其实我们在 SQL 当中能做的业务很很大一部分都可以迁移到 Mongo 上面来。**
