从技术层面来，我们一般把它分为冷备和热备，这个冷热之间的区分就区分在它备份的这个时间点上。对于冷备来说，通常我们是采取一个固定的时间做一个备份动作，那它不是一个实时的备份。这样的一个技术方案实施的成本非常低，而且操作起来也非常的容易。但是它有一些固有的缺陷，比如说可能产生数据丢失，进而导致数据不一致的情况。那如果经常玩游戏的同学们可能会有这样的体验，比方说有一天游戏公司突然宣布所有游戏回荡到前一天的某个版本，那它这里使用的回 down 的这个数据，通常来讲就是你每天定时启动的冷备数据。


在生产环境里，冷备往往是应对一些极端的，最坏的一些情况，那不得已才会去丢失部分的数据，采用冷备的形式来恢复数据。那从业务的层面角度，我们哪些数据可以去做这样的一个冷备来，比方说你的历史的订单，那历史订单指的是你订单的快照信息，这是在你下单的当刻就已经定格的信息，它可以去做一个冷备，还有业务报表等等已经生成的报表。那这些时效性并不高的历史数据，我们可以采用冷备的这种形式。


OK，那热备和冷备之间的一个比较就是它的备份的时间点，那我们说热备基本上是一个偏实时性的备份，那说到热备它有哪些方式的方法？也是两种主要的流派，分别是同步热备和主从热备。

那如果我们开启了 MySQL 当中的 master 和备库的半同步，那当你在 master 提交了一个事物之后，那 master 并不会真正的把这个事物完全执行掉，而是会先阻塞一段时间，直到你的这个备库。 master 同样确认你当前的这个事物已经被接收了，也就是说它已经被复制到了这个备库当中，那这时候 master 才确认你的这个提交是成功的，那否则你这个半同步就会被中断，那这是 MySQL 的一个内置功能，那同样还有一个功能 failover，也就是我们常说的失效转移，通常会简写成FO。


那在 MySQL 当中，如果你的这个主库挂掉了，经过配置的 failover，配置就会相应的生效，把你的这个访问流量切换到它的备库当中，那关于这个方案的实施的细节，我们会在后面的数据库设计章节当中跟大家去详细的落地一个案例，那这也是一个在生产环境当中应用非常广泛的一个方案。这个方案和强一致性的分布式事务方案有那么一点相似，因此对你的这个数据库的执行时间会稍微的多花那么一点。


我们可以把这种半同步加 MySQL 失效转移的这个方案应用在一些对一致性要求非常非常高的场景下面。那在其他场景下，从咱的这个性能上面考虑，我个人比较推荐同学们使用哪种方案，而前面咱说到的黑魔法 Canal 来做你的数据镜像的备份等等。那在稍后的一个小节里，哎，老师通过一个小 Demo 跟大家看一下 Canal 是如何来在本地执行的，它又是如何来消费你数据库提交的执行记录的？OK，同学们，那这一节的内容就到这里结束了。那跟同学们简单介绍了我们的数据备份以及阿里系的数据备份策略，还有相应的备份方案和失效转移的方案。那在下两节当中，我们就着重看一下 Canal 这个黑科技是怎么来玩的。除了数据备份以外， Canal 还能应用在哪些其他的场景上？好，同学们，我们下一小节再见。
