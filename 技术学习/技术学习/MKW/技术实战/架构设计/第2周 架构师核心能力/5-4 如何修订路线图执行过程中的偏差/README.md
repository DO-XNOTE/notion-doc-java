竞对就是竞争对手。这里跟同学们举个例子，在支付宝和微信的支付市场白刃战的阶段，微信他悄无声息地使出了一个杀招，这个功能点扭转了白刃战的双方局势，大家平时肯定经常会使用这个功能的，微信红包这个功能一上线之后，微信支付的新用户数量有了一个很大的拍摄，甚至连马爸爸都不禁惊呼微信这一招叫偷袭珍珠港。因此后面支付宝没过多长时间也推出了一个红包的功能。


那从这一点同学们可以看出什么呀？你的竞争对手，它的路线图和业务的发展方向，将对我们前面讲到的你当前公司业务还有技术架构，路线图的长中短期路线产生非常大的影响。恐怕支付宝的架构团队怎么也不会想到，突然间接到了一个政治任务，实现点对点群发口令红包等等各种的业务场景，那这就是对竞对业务的变化所做出的反应体现在了你的技术层面，那从另外一个方面来看，市场环境是不是也可能产生变化？

举一个例子，在互联网领域非常的明显，很多的成长非常快的公司，在它初期业务跑马圈地阶段之后，在国内市场会逐渐达到一个饱和。体现在我们的这个运营层面，就是新用户的增加指标趋于平缓，然后你的获客成本在不断的增加。因此在这个阶段很可能你就要去思考自己的业务方向，那从获客获得新用户转变为做服务或者拓展其他市场。


比如现在各大的互联网巨头，他们都有自己对应的出海战略，比如说像支付宝，那它过去几年在推行一个出海战略，它的业务逐渐的从国内的市场慢慢的向东南亚等周边国家去扩展，寻求新的业务增长点，那作为我们架构师来说，竞争对手的业务玩法有可能并不好把握。但是我们当前市场的业务变化是可以去透查出来的，就要去培养你对业务的敏感度，能让你洞察业务的趋势，掌握架构调整的主动权。那这就是我这一章里一直在强调的，从架构师视角培养你对业务的敏感度，对业务的思考。

我们玩游戏都知道你的游戏为什么好玩，它并不是说主线剧情非常的好玩，它是说你还有很多的支线任务可以让你探讨，增加这个可玩性。


那同时体现到技术路线里面也是一样，它也分为主线任务和支线任务。这就像在游戏当中的体验一样，你在玩的过程当中选择不同，有可能导致结局不同，正派结局、反派结局、大欢喜结局、悲情结局都有可能。所以在技术路线的演进过程当中，我们也要做好切换赛道的准备。


什么是切换赛道？比如说技术选型，通常我们会做多个 POC 项目，前面提到过 POC 是什么概念验证这样的一个含义， proof of concept 概念验证。那再比如说像重大项目的技术架构方案，这可不是纸上画出来的，那叫纸上谈兵。我们这里要通过实际的快速验证，做一些概念方向上的尝试。


在这个过程当中我们就要去识别出一些支线任务，怎么来识别？我们叫它 plan b 是什么意思？就是说在做技术架构路线图设计的时候，通常我们要让你知道有哪些备选方案， plan b 在关键的时间节点有哪些可选的其他路线可以走。同学们可曾想过为什么要这样设计技术用的好好的，为什么要替换？这就是同学们用程序员思维来思考问题所得出的答案。


我再从其他视角给同学们举一个例子，这个例子非常常见，比如说你们公司的业务一直构建在 Oracle 出品的 Web Logic 这种容器内，那这个容器功能当然非常强大，但是它一年动辄百万千万的授权费用，很大可能在你们公司业务萎缩或者说财政计划紧缩的这些财年，公司突然说打算砍掉这个预算。这个例子我经历过非常多，不单是体现在 Web Logic 这种服务器上，还体现在很多的运维知识系统上。你想你们公司的整套系统一定有从外部采购的部分，而不可能全部自建。既然牵扯到了采购，那你必不可少的就预算上的考虑，以及各种各样影响你做决策的这些因素，所以有经验的架构师往往会为自己怎么样留一条后路。


那你要清晰的知道你当前用的技术栈，它用了什么技术，同时它的成本、史诗成本、运维成本目前是多少？并且更重要的有哪些的替代方案，那这些替代方案所能支持到的业务效果和它对应的成本又是什么？这是作为架构师你要了然于胸的。所以我才一直说架构师是一个什么角色，一个大家长的角色。


你要是碰上那种天天坐着不动，等着业务部门提需求，给你喂需求吃，等着上面指派任务，那这种架构式的人设和咸鱼又有什么区别？那碰上这种万年潜水老甲鱼的老干部，你们就让他回家赶紧抱孙子好了。那到这里我们说了市场环境竞对策略的对路线图的修整，那这个作用力，同学们想一下作用力来自哪？

对应到技术路线图的修正，我个人经验来说，季度和半年度的审查其实就足够了，那主要也是针对业务目标的修正来做相应的技术层面的规划。所以你看一提到技术栈、技术架构和技术路线图等等，这个业务目标是逃不掉的。一个原始驱动力，那除了业务目标以外还有其他什么？前面也一直在强调的有一个容量规划，那像双 11 这样的业绩压测目标等等，以及你估算当年的业务目标量，去计划你的应用资源的调配。还有前面我们一直提到的一个词叫稳定性，就是保证系统不被流量压垮，很简单，但是这个流量的水位是怎么样定的？集团制定了业务目标，还有像压测、双 11 等业绩目标来决定的，那在阿里系项目里面，大家就能体会到稳定性的重要程度，甚至在集团内部有一个特别的行动小组，叫什么呢？叫稳定性团队，专门根据业务容量规划来做架构调整、弹性调整、全链路压测。随着你业务规模的不断扩大，在稳定性上投入的资源是会飞速增加的，这是一个很花钱的方向，同学们可不要小瞧他啊。
