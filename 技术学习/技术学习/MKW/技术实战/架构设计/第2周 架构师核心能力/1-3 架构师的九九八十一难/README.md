那蚂 super Mesh 是蚂蚁金服这个金融级云原生中间件，它是主要来做这个分时复用的资源切换。分时复用我们顾名思义就是说同一个资源，它其实在不同的时间是可以在多个应用之间进行一个复用的，那这项技术其实可以大大的去提高资源的使用效率。那以往我们用来做分时复用资源切换的这个整个过程大概需要在 4 小时以上，那在 super Mesh 的助力之下，只要不到 5 分钟的时间，那这项技术其实在早在 18 年的时候，双十一就已经应用到了咱的这个生产环境，在当年就是一战成名，大显神威。然后在次年也就是 19 年的双十一过程当中， SOFA Mesh 基本上把咱蚂蚁金服这个核心的支付链路全部覆盖，那为了体现高并发和高可用之间的关系，我这里再跟同学 share 两个数字。第一个是咱刚才说的 19 年双十一蚂蚁金服全链路上了 super Mesh 之后，那在这种场景下，它的峰值的 QPS 已经达到了千万，非常恐怖的一个高并发数据，那它平均的响应时间只有短短的 0. 2 毫秒左右，那通过这种直观的数据，相信同学们对高并发和高可用有了比较直观的认识。

那对于一些成熟的业务，我这里举一个例子，咱阿里集团有一个兵来将挡水来土掩的一个非常厉害的中台业务系统，叫UMP，是营销优惠计算引擎。咱去玩过淘宝的同学们都知道，在淘宝上面，你可以去做非常复杂的营销优惠的规则，跨店满减、满赠、满折、优惠券、送积分等等，这些需求来自于四面八方的事业部，他们对接的都是这一套 UMP 系统，一套由模板驱动的优惠营销计算引擎。他和这些各个事业部八方诸侯对接的，那简直是如行云流水，无丝毫障碍，为什么？因为它从宏观层面来说做了很好的架构模型的抽象，那在微观层面，代码级别的架构也是非常非常的精妙。后面我们会再跟大家去分享一下这里面的一些门道。所以从上面两个例子大家可以看出，可扩展性其实是一个非常讲究实践的一个话题，它远不是只停留在 PPT 上的这样的空想幻想，而是需要去真正的实际去走到你的代码当中，有丰富的经验做模型抽象，才可以做出一个非常好的具有高可扩展性的这样一个架构。好，那接下来的一个话题是咱系统的安全性，

我这里讲几个让大家听上去非常肉疼的数字，来感受一下什么是安全性，这里就要说到 d DOS 攻击，这里我先来举两个大家都听过的非常有名的公司，第一个Amazon。那 Amazon 在它历史上所接受到所经历的最大洪峰流量的 DDoS 攻击，它的流量是 2. 3T 比每秒钟。那我再举一个例子，这个产品大家肯定都用过， GitHub 在历史上经历的最大洪峰冲击大概是 1. 35 TB 每秒钟。


有的同学可能认为安全性离我们真正开发人员还很远，但是请同学们记住，咱现在是以架构师的视角来看待这样的一个系统，那你就不得不考虑安全性的问题。像阿里每年都会受到这种大范围的 d DOS 攻击，那它自己也沉淀了一些非常优秀的中间件来防御这些问题，比如说阿里云的SCDN，那 SCDN 其实是基于阿里的飞天平台的一套高防的系统，它可以去运用一些深度学习的算法去生成一些安全情报，可以快速的落地一些安全策略，并且可以非常智能的去识别这种 d DOS 的大规模攻击的手段，然后通过切换高防的 IP 来完成这样的流量清洗的工作。


那老师这里列举的是从外部来的攻击，那同时我们还要去预防内鬼，那什么是内鬼？就是自己公司的员工山库跑路的事件可出现了不止一次了，所以从安全性角度来说，作为一个架构师，你还要去考虑如何控制内部人员的访问权限。比方说早期有一些员工，他会在内网当中用天梯去爬数据，然后到外面倒卖给卖家，这样造成了用户的安全信息泄露。所以从安全性的角度来说，我们不光要做用户鉴权，这个鉴权不仅体现在员工账号的权限，还包括 API 的鉴权，防止 API 的恶意调用。同时还要做审计机制，这个审计是指内部的 it 审计，你的员工在内部系统当中做的每一个动作都是应该记录在案的。

这个成本它可以体现为直接成本，比方说你集群当中的计算资源，你的机器数量，对于这种硬件类型的直接成本，我们可以利用前面提到的弹性计算去提高资源利用率，降低直接成本，那弹性计算相当于从宏观层面来解决直接成本的问题。


那么从微观层面，也就是咱的具体的业务实现来说，我们可以去强调单机QPS，提高单机的QPS，提高你的接口性能，是可以显著降低你的服务集群中的机器数量。那咱们每台服务器其实都是钱，那阿里内部的事业部也是用的阿里云，但是我们还要跟阿里云进行结算，这些可都是白花花的银子。当年我做业务容量估算的时候，我对一个微服务申请了 20 台服务器。啊，我把申请打到 P9 那边， P9 就问了我一句话，你知道一台服务器要给阿里云多少钱吗？我听了这句话我就心里有谱了，我说得嘞，老板您歇着啊，我回去给优化一下我的代码餐，只要 10 台就够了，所以说业务流程要扛压，从微观层面你就要去对你的代码做极致的性能优化。
