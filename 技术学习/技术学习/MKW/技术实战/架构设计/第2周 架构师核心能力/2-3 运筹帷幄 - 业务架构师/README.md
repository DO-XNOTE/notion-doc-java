那在互联网和电商行业，我们也有一些老生常谈的领域，比方说是营销域、优惠计算这方面的，那还有商品域，一个电商业务的核心，所有玩法都围绕在商品层面，那半仙老师的技能加点主要是在支付领域、商品域还有营销域这三个方向，可对这三个领域可谓是略懂。那老师就随便挑几个领域，跟大家去稍微深入的聊一下，让大家换一个视角，从业务视角来看一看这些领域内的知识点

我们通常来讲首先有一个叫 enroll 的这个概念，比如用户在支付之前需要在你的中间系统上面去登记他的卡号，那我们教这个过程叫做enroll，相对于国际清算来说，这部分通常会涉及到一分钱测试，那他需要去通过扣减一分钱去验证你的银行卡的真实有效性等等。包括你的卡号哪些信息可以保存，哪些信息需要保存到第三方。


每个国家都有自己不同的合规的要求，尤其是像欧盟国家，所以这一块是一个非常具有业务性导向的这样一个领域。那再往下走是收单，那收单这里就和支付的钱息息相关了。比方说像蚂蚁金服，它这边主要是上海团队做这个外部的收单系统的接入，像阿里集团的各个事业部都要对接这样一个收单系统，那把这个接入的复杂度屏蔽在了收单这一层，像杭州也有一个非常大的技术团队，大本营也做底层的收单，还有一些分账等等。


那与此同时每笔交易它绝不仅仅是一个总价，打给你多少钱，那就是代表这个交易的全部不是的。那比如像跨境电商一笔交易中它包含有物流有关税，那关税还分国税，还有本地税，还有增值税、优惠券、折扣服务费、银行间手续费等等。每一笔订单的背后，它的会计账目准则、精准核算，都需要把这些钱分拆到各自的项目当中。所以如果一个架构是你只懂技术，而不懂这背后复杂的业务，能做的来支付领域的架构吗？绝对是不可能的。所以这从侧面印证了你架构复杂应用的前提就是要非常的了解这里面的业务。OK，那再往后就是咱的 settlement process，那这是一个清账的过程，通常是多方清账。OK，那支付交易的生命周期，我们用最简单粗暴的形式给它分为 off 授权交易，还有 capture 付款、扣款发生，还有 refund 退款以及void，在一定的窗口期内我可以去回退这笔交易。那在这之外通常我们还有像转账、 transfer 这样的生命周期，所以这里跟大家展示的是在支付领域最浅显的业务知识。
那你作为一个业务架构师，你要怎么做？挖地三尺，每一个流程，它的有限状态机，它图的流转，每一个业务你都要在自己的脑子中有非常清晰的图表，这样你才能去和业务部门去谈业务分析需求，OK，那这个例子呢？是技术领域偏向的业务架构师。

那营销领域它的玩法是纷繁复杂，那经常去网购、用淘宝的同学应该知道它背后有很多复杂的营销规则，比方说满减、满折，你满多少赠多少产品，还有送券、送积分等等，非常的复杂。那同样这些不同的规则背后还有一个范围，约束什么范围？你的这个营销规则可以应用的范围，它是单用户范围还是单个门店，或者它是一个跨店的营销优惠规则，甚至是全场折。最后它还可以去约定可以使用这个营销优惠规则的商品的范围，也就是说从用户层面、门店层面还有商品层面都有它的约束规则，那这里都是后台玩法。


对于前端玩法来说，你的营销计算出来的规则，它需要做透出，透出是我们经常用的一个黑话，它是指把这个营销信息透传到你在下单主链路中的关键环节，比如说主搜商品搜索页面、商品详情页、购物车页面和最终的订单结算页面，这里都要求你做商品优惠信息的透出。最后一点才是最复杂的，怎么去计算核销你的营销优惠规则？那一个商品可能应用多少规则？那这之间规则与规则之间的优先级是什么？你如何使用一些叠加规则？那你在这些种种规则当中，你如何去给用户显示一个最大的优惠？最优惠的一个规则，这在算法层面简单来说就是一个类似背包问题的解决方案，但实际上比背包要复杂的非常多。


从以上可以看出，营销优惠其实它的范围非常小，就是一个营销优惠的计算对不对？但是它的业务玩法非常的复杂多样，比方说淘系的营销优惠对接 40 多个事业部，底层就是一套模板引擎，那它必须是一个高度可扩展的优惠模板引擎，技术底层的建模非常复杂。可能做过电商系统的同学知道，像营销优惠应该使用模板这种技术来做，但是如果没有在像阿里这种体量这种复杂度的营销优惠场景下面玩过这个业务，那么你能设计出来的营销系统优惠券模板这些大部分来说只是玩具，所以这里强调一个词是什么呢？
