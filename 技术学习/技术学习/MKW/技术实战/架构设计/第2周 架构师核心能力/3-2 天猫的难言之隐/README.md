那我们还有一个特色是什么呀？咱国内玩家的电商业务的复杂性非常非常高，说是全球第一点都不为过。那在这一节讲的这个问题就是一个具有非常复杂的业务场景，一个应用就是营销优惠计算的服务。大家经常网购应该有这种体验。我们的国内的电商平台，它的营销导购这些优惠计算场景异常的复杂。这些玩法你放在国外了肯定玩不转的，因为老外他们没有我们中国人这么好的数学基础，他们 100 以内的加减法都要用计算器，你让他们玩这种叠加优惠，跨店满减，这相当于你让留学生来参加中国的高考一样，考完一场考试他们就只答对一题，那就是第一道题，哎，你的姓名叫什么？不过这种业务放在国内那就不算什么了，反而可能会增加用户的粘性。

这个故事叫做迈巴赫的陨落，迈巴赫是早期的天猫线的营销优惠计算引擎，那说到这个早期的营销优惠业务非常的简单，比方说就两块，一块是满减，另一块是优惠券。那这一个模型如果让大家来做抽象非常好做，因为优惠券它在本质规则上它就是一个满减规则，对不对？满多少减多少，满多少可以用某个劵，它在背后的逻辑上本质是一样的，那后来添加了满折这样的一个功能，打折跟满减不一样，那打折减掉的金额它是一个浮动的，但是本质上它依然是一个满减的一个模型，那实现起来业务上并没有太多的难度，对不对？可是再往后面发展，这个事情就不一样了，那玩法越来越复杂，出了新花样，比如说满送，对不对？这就是新的玩法了。


那送人头、送上瘾，说来就来，满几送几这个玩法就跟前面啊都不一样了，送什么呢？它不光送商品，还可能送积分。这个玩法又变得复杂了，它发展到后来，事情就变得越来越复杂。你的营销优惠规则优惠券它有使用范围，比方说跨店或者它限定了一定范围内的商品，那这都是从单个营销优惠规则角度出发。如果你们把它合为一个整体，那它还会增加什么的复杂度？**增加优惠营销核销的复杂度。
核销就是计算的意思，这里面就涉及了你的计算的最优解，就是一个背包问题，**前面有讲到过，还有拆单，你的优惠计算条目最终要拆分到这个订单上的每一件商品上，具体到每个商品分单多少扣减的金额，这个是优惠的拆单不要小看这个，实现起来非常有难度。那你拆了单的商品，我说不要就不要，我可以退单，退单就牵扯到你用掉了优惠券如何去还原，退了之后你的当前订单不满足优惠金额了怎么办？那这都是非常非常复杂的业务逻辑。好，我们讲到这里呢，迈巴赫还暂时没有陨落，但是它离陨落已经不远了。我们看上面的营销优惠规则，随着你的业务线越来越复杂，产品种类越来越多，你的营销优惠引擎变得越来越笨重，最终你整个系统的技术架构层面会变得非常非常的复杂，复杂到你甚至都不敢去添加新的功能改代码。
