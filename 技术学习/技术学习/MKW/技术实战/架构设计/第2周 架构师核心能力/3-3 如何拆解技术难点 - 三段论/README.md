那同学们会心想，唉，老师你说的这么轻松，但是我怎么感觉自己完全无从下手呢？那为什么无法下手？因为这里面最难的就是拆解问题的思路。就像我们平常读书一样，咱经常说一万个读者心目中就有一万个哈姆雷特，那同样做架构师也是一样的，每个架构师思考问题，解决问题的方案和角度都是不一样的，并且还谁都不服谁，这就像微服务拆分一样，你的设计、你的服务边界设计的好坏将直接影响到你后期的扩展，还有主链路的规划。


所以面对同样的问题，你的分析过程、思考过程，也同样决定了你解决问题的这个质量。
那作为架构师来说，当你拿到一个复杂问题，最重要的一环是什么？第一就是以大化小的能力，这是一个抽象能力。那怎样把一个问题抽象成什么程度？

我们来跟大家讲故事，定义几个故事的角色好不好？对于全局式的分布式事务来说，我们定义这样一个角色TC。前面我跟同学们讲过去定义角色，我们要从三个维度来描述这个角色将要完成的工作，那这三个问题分别是你是谁？你的边界是什么？你的功能要完成什么功能？那对于 TC 来说，它的角色是一个协调器，那它的边界是什么？它的边界就是独立于各个分支式事物的一个全局的中心化的中间件，它的功能就是在全局层面去负责事物的协调，这个协调包括去控制每一个分支事物的提交，还有回滚，并且记录每一个分支事物的状态。那通过这样一个中心化的事物协调器，我们就可以在全局层面把控分布式事务。


那对全局式分布式事务的职责领域界限搞定了，那接下来对于分支事务我们怎么来控制呢？这里我们定义了两个角色，第一个角色是 TM transaction manager，同样的我们也来问他三个问题。**第一，你是谁？ transaction manager，他是一个全局式事务的事务发起者，那这个事务发起者往往是这个全局事务当中的第一个分布式事务，由它来发起。那它的边界是什么呀？就是你整个全局式分布式事务链路的起点。第一个分支事物，OK，那它做什么呢？很简单，就是注册全局式事务，发起一个全局式事务的上下文。那有了这个发起者，同样的我们也要有一个执行者。执行者是谁？我们叫他RM，那 RM 是 resource manager 的意思，资源管理器，因此他的角色定义就是分支资源管理者。那它的边界是什么呀？从角色我们就可以看出来了，它既然是管理分支事务的，那它的边界肯定是各个分支事务，那它的功能是什么呢？就是注册分支式事务，负责当前分支的提交和回滚。**


OK，那到这里咱的职责领域也划分完了，那最后一步就是分层构建细化的方案。
