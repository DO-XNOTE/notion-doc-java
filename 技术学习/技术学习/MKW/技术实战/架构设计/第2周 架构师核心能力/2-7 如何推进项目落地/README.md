听上去非常的接地气，那在互联网公司里面这个词非常的常见，这也是我们一直强调的，如何把你的想法，把你的架构项目，把它落到实处做出来，这才是最重要的。因此，所以为什么马爸爸这么风光？因为他下面有一群非常勤奋努力的小伙伴，能把一个云里雾里的需求，一个战略拆解成技术方案，把它最终实现。所以你指望业务部门做这个事情吗？不可能的，这个就是我们架构师做的事情。所以这一节我们就把目光集中在落地这样一件事上，来聊一聊这是怎么回事？好，前面有一节我们说过，架构师是一个三位一体的角色，**懂技术、懂业务、懂管理，那所以说项目管理还有项目协调能力一直以来都是架构师的一个非常重要的能力素质。**那在老师的工作实践当中，我认为落地层面比较重要的有这样三点，第一点是优先级priority，第二点是时间表timeline，还有一点是协调能力coordination。

那我们接下来用什么方式来展开讲这三点？我们就用阿里系的超快猛的项目推进方法来跟大家比划一下。好，第一点，优先级。说到优先级，很多同学一脸蒙圈，觉得每一个项目、每一个功能点在你眼里都很重要，对不对？


其实都重要，它往往就代表着都不重要。比方说就像我们小时候考试老是画重点一样，你**看一个同学，他画的整本书都是重点，全是花花绿绿的重点划线，那他大概率就是个学渣，学霸一般是很云淡风轻的。**

那这种情况就像人生一样，不能每天都是高高光时刻，所以我们一定要去识别出哪些是重要的东西，哪些是不重要的？OK，那重要的东西是什么呀？比如说你的主链路上的业务对不对？非常重要，如果这上面一旦有一环不通畅，那么它会影响你整个业务，那这些就是重要的事情。简单来说就是没有这个东西你的业务玩不转。所以你定优先级就要有这种大开大合，敢于舍弃，不要把所有东西都认为是非常重要。


OK，那么这样我们就可以去识别出 P0 级别的需求，这些最关键的业务功能点，然后在这些 P0 级别的功能上面排坐次。那排什么坐次？那就是按照先后顺序，先后实现的顺序排列一个顺序表，哪些是先决条件必须要先做好的，哪些 P0 是强依赖于一些前置条件，必须让上游业务先做好的梳理出一个先后依赖的关系，然后再怎么样，然后再根据优先级把接下来的P1、P2、 P3 这些级别的重要的功能点依附在这些 P0 级别的事项上。


这样你**就有了一个非常清晰的群落聚集的一个图片**，对不对？每一个重要的功能点它都有一些相应的辅助功能把它包围住，这就形成了一个可以打包的项目交付包。那接下来我们根据前面定好的这个优先级去做一个时间表，什么样的时间表制定项目的交付节奏，那根据什么节奏来？那就是根据上面我们定的这个priority，每个 P0 我们都可以把它作为一个小的里程碑完成一个交付一个，这就是一个最浅显、最简单的持续迭代的这样一种概念。
