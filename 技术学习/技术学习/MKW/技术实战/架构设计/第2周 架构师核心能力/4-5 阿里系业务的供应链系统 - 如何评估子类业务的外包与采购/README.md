以及其中的风险隔离这个因素，那第二个是研发资源的一个平衡的问题。那最后一点非常重要，是同质业务和核心业务的PK，那从这几点同学们可以看出，它既兼顾业务，也兼顾了一个我一直强调的一个中心点是什么？钱对不对？钱？即使像阿里这么有钱的公司，它内部孵化的核心业务也不能抛开钱来为所欲为。那这一节里我也会从阿里系里面挑一个核心业务，跟大家看一下它的内部系统研发之路，经历了哪些阶段，在自建系统和外包采购之间来来往往做了几次演进，那通过这个例子，大家来探讨这样一个决策的过程，所以同学们听这个课还能听到非常多的阿里系的宫廷秘史。那接下来我分别就这 3 点跟同学们去深入的探讨一下。


第一点，投入收入曲线这里要跟大家去 show 一幅图 321 走，诶，这里的蓝线代表着是外包采购，那这个橘黄色的线打是自建系统，大家可以把这个横轴把它想象成一个时间序列，随着时间序列往前走，那这个纵轴 05 时这个数字我们可以把它看作是你的投入。那对于外包系统来说，你可以看到它的投入曲线相对的平滑，对吧？一个成型的系统买来即用，所以在初期的接入成本往往会相对较高，那在初期接入成本以后，就可以让业务产生一个稳定的收益，也就是说你的系统可以用在真实业务中，可以让业务跑起来。但大家不要小瞧初期的接入成本，比如说有些重量级的一些产品，像 Webologic 之类这种服务器，或者 Oracle 的数据库，他们的费用对于一些大公司来说，动辄是上千万起，非常的恐怖。但是自建系统的往往并不便宜。同学们，我们要考虑，自建系统是一个从零搭建的系统，所以它的投入成本应该是这样的，你的团队的投入还有乘以它的一个开发周期。


我们打一个比方，如果你的一个购买的采购的产品，它的初期投入费用是 200 万，看似很高，对不对？假设我用团队来自己去搭建他，那可能你会招聘一个 6- 7 个成员的开发小组。假设每一个成员的成本是 20 万，并不是一个很高的数字，但由于从 0 搭建的系统，所以它的开发周期是个悬而未决的数字。


这就带来两个影响，第一个影响是你无法准确的估算初期的接入成本，所以在有些情况下，管理层会倾向于外包采购，因为它在初期的接入成本是一个可预估的数字。尽管这个数字并不便宜，但是与其去承担一个不确定的开发周期带来的风险，管理层面大多可能会倾向于一个可以预估的初期接入成本这样的一个方案。那这是第一点。


第二点，你的开发周期决定了你的什么呀？你的业务产生效益的时间点。我们从这个图里可以看出，对于外包采购来说，初期接入成本之后，它就可以让你的业务产生正向收益。而自建系统来说，往往需要经历很长的开发周期，因此你业务产生效益的这个点，这个时间点往往会比较滞后。


那在 it 业界摸爬滚打的老法师都知道，软件工程自己开发东西，它有很大的不确定性，你开发周期的长短受到很多因素的制约，它并不能很好地把控它的精度，而这个时间它的长短影响着你什么时候可以开展业务。因此在评估外包采购和自建方案的时候，我们有一个很重要的一个时间点，这个点是一个拐点，叫什么？预估的拐点就是我们计算出来一个大致的时间范围，在一定的投入之下，经过多长时间之后，我团队自研的项目可以去替代外包采购项目，并且在成本上建立一个优势。这是预估的拐点，它有助于你去评估这个时间线对业务产生的影响，以及你的整套方案它需要做的投入的成本，通过这个 PK 来决策你什么时候开始投入资源做自研，以及在未来的哪一个时间点去替换外包采购系统，从而产生正向的收益，

我们来分析一下，对于新零售行业来说，供应链系统是不是核心系统，绝对是它最核心的一个业务。前面老师提到，技术团队应该关注到自己的核心业务铺到这上面，但是在实际情况下，我们这个例子当中，一开始这真的不是自见的，为什么呀？因为它有其他的因素制约。同学们还记得老师说过，架构是一个平衡各方利益的这样一个过程，它从来就不是一个只分对错或者黑非即白的决策过程。好，我们再来往下看，究竟有哪些因素的制约，使技术团队不能直接去打造这样的供应链系统？第一，当时的技术团队规模 20 人左右，并且即便他是马爸爸的头牌的创业部门，但是他的资源投入其实也是相对有限的，因为一个新兴的商业模式，我们要做到快速试错，不可能给你无限的资源。


而供应链系统，尽管它是核心业务，但是它背后的业务逻辑非常的复杂，它是其实是 ERP 系统当中的一个部分，在有限的技术团队的资源下面去投入到一个如此宏大的业务系统当中，确实不太现实。因此把有限的团队投入到这么宏大的系统当中，考虑到这种不确定性，你的投入收入曲线它的平衡点可能会在未来的很远的这样一个时间点。那与其承担这个风险，不如在初期去选择一个投入曲线相对可预估的采购系统，尽管它看似贵，但是它是一个内部创业项目在初期的一个局部最优解。

我们接下来看一下在整个供应链系统的演变过程当中，我们采取了几步把它替换成了自建系统。那这里有一个时间表，我们在第一步的时候采购了一个外部的 ERP 系统，在业务开展的头八个月，我们使用这套 ERP 来跑业务，那第二步我们要逐渐的把这个业务的主动权拿回来，掌握到自己手里，因为毕竟它是一个核心业务。


所以第二步我们采用了天猫的供应链系统，集成到了一个中台系统当中，那这套业务系统我们使用了一年有余，同学们可以看到它的时间线是有重叠的，前面 ERP 采购快速验证了商业模式之后，就立即上马了内部项目的搭建和接入。那随着业务的发展，新零售是一个变化非常快的业务领域，它的内部的玩法层出不穷，中台式的天猫供应链系统有时候并不能满足如此具有个性化的局部的业务，所以在接下来一个阶段，我们采用了自建系统，同样的它们之间的开发周期也有一定的重叠，道理都是一样的，那便是系统的替换成本一定要考虑远期效益的这个平衡点。即使我们心里清楚早晚都要换成自建系统，但是早换可不一定比晚换要好。
