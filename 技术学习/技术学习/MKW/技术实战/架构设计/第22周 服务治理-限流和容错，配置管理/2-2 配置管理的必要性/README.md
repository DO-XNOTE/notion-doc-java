它这里面的弊端也非常的明显，比如说你在代码当中的配置非常不好维护，东一块西一块，没有一个统一的地方对它做管理。


第二点它的职责不分离，你把一个配置项放到你的业务逻辑当中，这就是傻傻分不清楚，没有很好的践行这种单一职责。然后安全性还不高，你的用户名密码都被泄露了，这个时候山库跑路的可能性就会大大的提升。那还有很不方便的一点就是写死在代码当中的配置，它是属于一个静态编译的过程，你每次涉及到你的配置变更，实际上都是要把整个代码全部给它重新编译一下，打包，然后再发布到线上，正式运行生效。整个流程非常的冗长，非常的麻烦。后来我们逐渐地习惯把这些配置相关的东西把它放到配置文件当中，那会儿还没有yaml，那会儿大多也没有problems，多的是什么呀？多的是 XML 这种类型的配置。当时的老板经常给下属说，你们把这个配置给它抽出来，实际上抽是从代码当中抽到这个配置文件里，那就是当年的抽出来。这种情况下，相对于写死在代码中，自然是有一些优势的，但是缺点依然掩盖不掉，比如说你安全性低，对吧？那就像你穿了马甲，我照样认识你。你写在代码里，写在配置文件里，那你把这个用户名密码都是明问保存的，对吧？不管在哪都是程序员可以一手接触的，这个就是不安全。


另外一点，你的配置文件实际上也是一个静态编译的过程，它把你的配置文件随着你的源码一起打包，然后一同部署。你每次做改动都是需要做这样的一个宠，编译也非常的不方便。那么后来当我们有一些动态需要变更的属性的时候怎么办？当时很流行的一种做法，甚至现在有很多公司在使用的就是通过数据库存储你的代码、业务逻辑，每次执行到一个点，从数据库里动态的去读取这个值，然后加载到上下文当中使用。


这时候如果你要对这个值做一些更改，非常方便，只用把数据库做一个更改就可以了，对吧？但是缺点也非常的明显，如果我们是从静态文件当中读取这些配置项，那实际上速度非常快，相当于内存读取了。但是你从 D b 这里读取，很容易造成一个瓶颈，因为 d b 它往往是我们整个系统应用当中最薄弱的一环。那么如果一些高并发应用尝试从 DB 里每次执行业务都要读取一次，这个连接数 DB 的资源都无法很好的保证，后面就会做一些相应的增强。比如用什么呀？比如加一层缓存 catch 加在 DB 层和这个应用程序之间，每次从缓存当中读取一下，如果缓存中的数据太老，再从 DB 里面刷新，这种方法一定程度上可以解决这个问题，但是对于你的代码也是一个额外的逻辑操作，也是非常麻烦的。除此以外，你这个整个配置读取的链路当中，相当于又一个DB，又一个 catch 两个中间件，增加了你的失败的概率。 OK 实际上也是并不好的一种方式，不过它的优点就是可以动态的去加载参数，而且它解决了这个安全性的问题。


除了 DB 以外，其实还有一种途径，环境变量。我在启动的时候可以通过你的启动参数将一些特定的配置加入到你的启动上下文当中。对于一些本地的像密码等等，也可以通过这种方式通过环境变量传入，这样一定程度上也能解决安全性的问题，但是你每次需要去变更属性的时候，依然需要做一个重新的部署，重新传入这个环境变量，与此同时你重新部署的这个环境变量，它应该包含在一个部署脚本里，对吧？所以你还要有相应的团队来去动态的维护，你这个编译脚本也是非常麻烦的。


就是总之你这以上这 4 条路很难有一个方案，可以综合以上几种方案的所有优点，同时又没有对应的这种缺点。我们现在碰到的一些问题，我把它总结一下。现在配置管理有这样几个问题，

第一个，你的配置文件格式不一致，有的时候你是yaml，有的时候你是practice，甚至有时候你是很老旧的XML，那你很难做到一个统一管理，那这是一点。第二点，实际上大家不要把豆包不当干粮，觉得配置文件就是代码以外的一个配置，实际上你的配置项它也是代码，所以说你也要像代码一样，可以有一个对应的像回滚功能、版本回溯功能，你要知道你之前发布了什么历史记录等等这些功能点你都要把它给配齐。OK？还有我们的配置项通常分布的非常零散，有的你加在 DB 里面加载，对吧？有的你是放到配置文件当中，有的你可能写在代码里分布非常零散，不好做一个统一管理。那最痛的一个痛点非常麻烦的就是它是静态化的配置，咱每次需要去让一个变更生效，那要重新部署，甚至说要重新的编译再部署，非常的麻烦，那我们有没有办法可以解决它？当然就是我们的配置中心，那在引入配置中心之前，我们先来跟同学们看一下咱配置项都有哪些业务类型的分类。

一块是静态配置，另外一块就是被称为叫动态配置。OK，静态配置非常常见，比如说我们的 DB 连接池、 DB 的用户名、密码等等这些不变或者说很长时间内不会改变的这些数据，我们把它叫做静态配置。与此同时，类似的配置还有像应用名称，对吧？以及你做一些方法调用，或者和外部供应商对接，你总要交换一些密钥，对吧？这也是很长时间不会变的内容，我可以把它划归为静态配置。不过像密钥、密码等等，这里还有一个额外的需求，就是加密，你一定要让加密存储，这样的话别人即使脱库或者用任何方法拿到你的原始配置之后，他不知道你对应的保存的这些密钥密码，它的原文是什么。这是我们数据安全当中很重要的一个环节，也是配置中心需要保证的一个功能点，那这是其一。


动态配置这里那大家非常好理解了，都可以举出很多例子，比如说你的业务规则，哪些业务规则？我们3T、 nil 这边前面章节学到的存储的各种限流、降级、准入、热点规则等等，这都是动态变化的业务规则，那么这一部分它需要在方法的运行期生效，我不希望让应用重新部署，我希望让它跑着跑着过程当中我就可以动态改变。
除了 Sentinel 这类 rule 的规则，其实还有很多类似的功能，比方说我可以人工的设置一些功能开关，新的应用上线之后，我可以人工控制这些新功能的开启关闭，或者说我设置一些人工熔断开关，做些蓝绿发布，做些数据源切换都可以。那甚至说有些金融级的应用，它需要去实时的反映你的股票价格的变化，或者说你当日的汇率的变化，那这些数据源都有可能是动态的，需要在你的业务当中动态的体现这种变化，那么这是我们配置中心所需要处理的两种不同的配置类型，动静结合加上一个加密。


从以上情况下不难看出我们为什么需要一个配置中心了，那一个配置中心可以为我们这个配置业务带来什么样的优点？比方说第一个业务分离中心化管理很简单，你的配置项不应该牵扯到自己的业务逻辑当中，这样一来，通过一个配置中心可以将你的业务执行的这种核心的逻辑编程的代码与它对应的配置项，把它隔离开来，这就是一个职责分离另外一块中心化管理，你的相应的配置可以在一个统一的中心化的这个节点上面，或者说是这样的一个集群上面做管理，不用说像以前一样分散在各个地方。那你通过一个统一的中心化的配置中心的节点，就可以去管理所有应用，所有配置项的变更推送等等。那这是一点，还有几大好处，就是说在你的配置中心里可以设置相应的访问权限，控制谁可以访问你的配置中心。当应用当你的配置发生改变，这个改变是一个错误改变的时候，我有能力做一个快速的回滚，把变更后的正确的数值再推送给各个业务方。那这是一点。还有我希望让我的配置项像代码一样，可以被版本控制工具给它管理起来。很简单的一个，就像 Git 一样，可以查看所有的历史记录等等，这个都是配置中心所需要考虑的功能点。


最重要的一个，我的配置中心一定要带有一个动态推送的功能，因为现在电商业务它的花样非常多，各种业务玩法纷繁复杂，你有一个动态推送的这个功能，就可以赋能很多的业务应用，很多的场景就可以玩得转。除此以外，还有咱前面提到的加密解密数据安全的范畴。OK，那么从以上这几点来考虑，我们使用一个动态的配置中心，那就可以完成上面的这些属性配置文件的保存的需求。从以上的几点来考虑，假如有这么一个组件可以完成上面吹出来的这些钮，大家何乐而不为的？为什么不用？OK，那这就是这一节当中想跟大家分享的内容，为什么我们需要这样一个强大的配置中心？在后面的小节里，我们跟大家一起去了解一下现在主流的配置中心， spring Claude 当中提供的这些配置中心组件。OK，同学们，我们下一小节再见。
