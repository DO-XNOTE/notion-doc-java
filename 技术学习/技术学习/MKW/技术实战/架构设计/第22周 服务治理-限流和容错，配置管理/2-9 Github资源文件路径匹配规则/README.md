下面的这个profile，它实际上是我们可以把它看作是你的环境隔离。那咱同学们在前面 spring page 当中应该看到这一个属性， spring 下面的一个点有一个 profile active，前面通过这一个属性来去配置了你当前的应用所处的一个环境变量。那么上面的 application name 加上profile，这两个是构成咱 GitHub 文件名当中最重要的两个部分。


那下面的这个 label 是什么呀？它跟咱的 GitHub 是有点关系的，它这边对应的就是一个branch，我们之前默认的 branch 是master，那后面在 2000 年咱统一都改成了men。所以大家这里要注意一下，如果同学们是使用的老的 GitHub 的repo，它对应的 master 应该指定为对应的label，那如果是我们新创建的repo，那么你的默认的 branch 实际上主 branch 就是men，那这里要做一个新老的区分，那这里很多同学容易搞错，经常会说，诶老师为什么我跑这个 config 程序，我发现我拉不到文件？其实这里一个排查的点就是看一下以当前的GitHub，它的这个label，它默认的主分支和你本地是否配置的一样，OK，那这是三个元素属性，那我们接下来看一下 GitHub 当中这里的文件命名规则。

咱们前面跟同学们讲过 application 和 profile 这两个，它是构成了最核心的文件名，那么我们可以给它起名叫做 application 杠 profile 点yaml，或者我们如果不用yaml，我们把它写成properties，这也是 OK 的，没有问题。那一个典型的例子就是application，我可以把它替换成应用名是restroom，那对应的这个profile，我把它替换成杠dev，这就是我们 restroom 在 dev 环境下面的对应的配置文件。


那如果我们这边还设置了 search path，在你的 GitHub 当中前面还会 Pre append 一个文件夹儿的名称，那么咱在前面的配置当中跟同学们也实际的演示过，使用这个 search path，那么 config 组件就会根据你的应用名到相应的目录下面去寻找它的这个 GitHub 的配置文件。


那这里如果不应用名称其实也是可以的，那 search path 在我们的 config 组件当中，它是一个占位符的，同学们可以在这边编写自己的匹配规则，那这是咱们的 application profile 部分，那 label 这里代码分支的名称，这个也很重要，那如果我们不用 profile 做环境隔离，实际上也可以使用 label 这个属性来做环境隔离。
