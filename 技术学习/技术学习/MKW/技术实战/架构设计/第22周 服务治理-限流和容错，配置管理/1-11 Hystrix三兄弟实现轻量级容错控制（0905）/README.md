这三个微服务它们背后都集成了 high strikes，那么这些服务背后通过 high strikes 去托管了某些接口，去设置了降级流程也好，熔断配置也好等等，他们这些服务实际上都会将自己作为一条服务记录，注册到咱的这个注册中心里面。


注册中心这边它可以是Eureka，也可以是Nicos、 cancel 等等，我们题材不限，那这是一个咱目前服务集群部署的一个样子。那么接下来 Turban 这边的角色怎么来定义？我们的 Turban 这里实际上也是可以去集成注册中心的，那么它从注册中心这里做什么呢？我本地在 Turban 这里，我去会配置我所要监听的这些服务的名称，比如叉还有 YY 这两个服务，我需要去监听。那么这两个服务它背后对应了有多少台服务器？这些服务器它们的低质又是什么？它从哪获取？就是从注册中心这里，通过咱的服务发现机制，把对应的我配置的所需要监控的这两个服务，把它对应所属的这些 client 全部给拉倒，OK，那拉到这些 client 之后，他这边做一件什么事？同学们知道阿里系一直有一个非常坏的传统，叫做周报。传统每一个小组成员去在每周的周五花很长时间洋洋洒洒去来写周报。这就正如我们的 high strikes 在这边给每一台机器去发布它的接口调用信息一样。那我们可以把它看成一个周报。 Turban 这里相当于什么角色？相当于一个团队的leader。他做了一个合并周报的操作，把接下来这些所有我所监听的具体服务的列表，通过 high strikes 这边暴露出的接口。你集成了 high strikes，就可以通过 high strikes 的一些 actuator 端点，将自己的调用信息转化成一个 JSON 的数据对外吐出，那么 turbine 这里它实际上目的就是为了获取到这些我们的 high strikes JSON 数据。


你有一份Jason，我有一份JSON，他也有一份JSON，这些 JSON 就相当于各个的周报，每个小组成员的周报汇总到了咱的 Turban 这里，然后 turbine 这边就产出了一个大周报， turban 这里就会从服务的维度去展示一个聚合的信息，这个聚合的信息它实际上跟我们每一个 high strikes 吐出的这个 json 结果一样，咱 turban 这边吐出的也是一个 JSON 结果。


只不过这个 JSON 结果是汇总计算了咱监听的所有节点的信息之后，展示出的一个聚合的报表，那么这个报表给谁看？就给我们接下来的这个 dashboard 来看，你要在 UI 上面显示出一些图表统计分析的信息，那它自然需要去监听一个数据源，那我们 dashboard 上面的数据源，它的数据的源头可以从两个地方来，一方面它可以去从 Turban 这边来获取，但它这边获取到这个数据源之后，展示出来的就是一个经过聚合之后的仪表盘信息。


除此之外，我也可以针对于单个hashtrikes，我可以去监听，去查看当前单个的 instance 节点下面它背后的接口调用信息，那这个信息数据源不管是从 hashtrigs 的节点，还是从 Turban 这边来的聚合信息，那实际上它这边去监听的都是一个URL，一串网址，那这串网址实际上就是 high strikes 和 Turban 背后集成的一个 actuator 端点，暴露出了一个端点。那通过监听它，我们就可以去获取到这些服务，调用它背后的成功失败各种状态信息。
