那其实际上并发数控制，也就是说线程数控制是为了保护我们的这个业务当中的线程池不会被高访问的流量慢慢耗尽。打个比方，我当前应用我有一个下游调用，但是这个调用因为某种性能问题，导致它的响应时间会拉得非常长，那么这样一来，对于我来说调用它花费的时间长，那会导致我当前的应用吞吐量变低，那所有的请求都在这里积压起来了，那后面的请求每次一过来实际上都会吃掉一个线程来在这边等待。


那因此对于这种方案一个简单的解决办法就是通过线程池线程隔离的方案，这里像 high streaks 也是使用这个方案来做这种防雪崩异常情况的发生。
不过这种方案有好处，自然而有缺点，它其中一个最明显的缺点就是你当前的线程创建的数目会比较多，那因为你在线程上下文切换，它实际上也是一个很耗资源的操作。如果你的应用它是一个延迟很低的应用，打个比方，我的应用只是 20 毫秒的返回等待时间，但是你给我打了一个线程池，这样的话，实际上它这个线程之间切换的时间都会耗费非常多的时间，不太划算。因此我们这里对于 Sentinel 来说，线程数判定实际上并不意味着我要创建和管理一个线程池。不这样做，我只是说简单的来统计一下当前这个请求你上下文的这个线程数目，那也就是有哪些调用，多少个调用现在正在执行。如果你这个调用的线程数去超过了我的阈值，那么后续的新请求我就直接把它拒绝掉了。这种方式实际上更类似于咱在 high strikes 当中的信号量隔离的方式，那这是它的线程数判定的其中一个不同点。
