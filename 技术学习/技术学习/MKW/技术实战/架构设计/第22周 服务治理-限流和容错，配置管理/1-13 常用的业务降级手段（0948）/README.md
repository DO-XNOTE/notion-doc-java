那什么是静默降级？很简单，就是说我什么都不干，那在对应的 fall back 方法里面，我就返回一个空结果，比如说我可以返回一个 none 值，或者对于一些 list 的返回要求，我这里可以去新返回一个空的list，那这是我们目前最常用的一种手法。那同学们可以思考一下，我们降级熔断的目的是什么呀？因为咱的这个调用的方法，调用的资源在一段时间内频繁失败，导是被打到了熔断逻辑，因此最常用的一个场景就是这种快速失败。那你快速返回一个null，由它对应的上下游的业务，你谁调用我？你自己要做好什么呀？防御性编程，你要注意，我可能返回一个空值。那这个也是目前在线上环境当中，我们最常用的一种手段。那有的同学可能会有这样一个问题，那你静默降级返回空，我直接在代码层面我写一个 try cache，把你的所有方法包裹起来，一旦发生异常，我也返回空，这样是不是可以？那其实 try catch 可以解决这种异常的情况，但是对于这些超时的情况，那你 try catch 实际上是处理不了的，对吧？那再有一点，你引入了这个 try catch，实际上它跟你的正儿八经的业务逻辑风马牛不相及。那你把它引入进来，相当于引入了外部的一些dependency，相当于这种外部依赖被打破。那我们在程序设计的时候一直在讲开辟原则，单一原则，所以说在这种情况下，我们尽量的不要使用 try catch 来去处理这种降级流程，那我们把这个主导权，这个相应的降级逻辑就交给Sentinel，交给 high strikes 这样专门处理服务容错的组件就好了。

这时候返回一个空可能就不太适合。那对于这种场景，我们通常会能设置一个业务可接受的默认值，我们就设置一个默认值，那这种处理方案就像营销优惠计算，我可以返回一个 0 return 0，也就是说我当前计算出来的价格是他没有参与任何服务，任何优惠计算。但是同学们想，我本身这个商品哎，确实参加了优惠计算，但是你显示出来又不参加，这样是不是会降低哎？你客户的购买率，这肯定你的服务被降级了，对终端客户多多少少肯定是会有一些影响的。那对于这种情况下返回默认值如果跟跟正式的值有偏差怎么办？有时候我们也会采用一种默认加标记的方案，比方说如果我这个服务调用被降级了，那么返回默认值的同时，我还会返回一个特殊的一个标记位，那就是这里的一个小感叹号，那这个感叹号呢？可以是记录在你返回的 response body 当中的任何一个字段里面。


那么记到这个里面，我们前端或者是我的上游调用方，根据你的这个小标记，我会做出一些额外的一些业务逻辑。比如说我在前端页面显示出这样一段话，优惠计算以结算页为准，本页面不会计算优惠价格，那可以给用户这样一个解释，那他就会理解优惠依然生效，但是你需要到结算页当中来去查看这个最终的优惠，那因为前面课程也跟同学们讲过，你的这个电商下单就是一个漏斗模型，你越往后你的流量越小，你越往前像搜索、像详情页，那都是流量很大的地方。
