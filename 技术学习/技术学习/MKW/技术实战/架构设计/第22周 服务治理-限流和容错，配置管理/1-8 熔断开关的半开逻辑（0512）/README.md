什么情况下这个降级流程会达到一个熔断状态，那就是我们的失败达到了一定次数，我这里乘以n。但是实际在配置这个降级逻辑的时候，我们可能会从不同的维度来去做判断，比如说当前的异常数的比例对不对，还可以去根据它的响应时间， response time 来去判定。如果你的慢请求达到了一定的数量，那我可以认定你当前应该是处于一个熔断开关开启的状态。一旦我当前这个服务熔断状态处于开启，自然而然咱的相关请求发到这个服务之后，它不会去真正的执行它底层的正常方法，而是说做了一个叫快速失败，直接转而执行它的降级流程，这就是我们的一个正向流程，从方法正常调用达到降级，多次失败达到熔断这样一个正向流程，那我们逆反过来，也就是说从熔断状态我们如何来恢复？那这个问题在咱们的不管是Sentinel，high， streaks 这些熔断组件当中，它的流程实际上是非常相似的，可以就说是一模一样。


当你的熔断器开启了一段固定时长之后，这个时长我们可以把它称作为叫时间窗口。那我们说人在空窗期容易寂寞，这个熔断方法也是一样的，经过一段我们指定的空窗期之后，他这里就怎么样打到了一个半开状态。什么叫半开状态？熔断状态这里其实要么开启，要么关闭，那这个所谓的半开状态，实际上它是处于一个尝试态。什么意思？就是说我们在空窗期的寂寞结束了之后，我这里会进入一个一支红杏出枪来的尝试状态，这个阶段下我虽然大方向还是处于熔断的，我这里会偷偷摸摸、悄摸摸的发起一个测试请求，这个测试调用如果他这边调用成功，那么我们的熔断状态就会从当前的这个开启状态变为一个关闭状态，这是我们熔断开关恢复的一个条件方向。测试请求调用成功，那反过来，如果咱的这个当前测试请求调用失败，那么不好意思，谢谢，再见。


你依然是处于一个熔断状态，然后窗口即时这个空窗机开始做一个重新计时。如果用线性的图表来表示，实际上就是当前处于一个熔断态了之后，我们在接下来的一段时间的空窗期时间往前走，往前走，在这一段空窗期内，它依然是处于熔断态的。但是在这个空窗期结束的这一个点，我们怎么样在这里执行一个半开逻辑，那根据这个半开逻辑的执行结果，我们才来决定它是否继续处于另一个新的熔断空窗器？还是说我这边会去把这个熔断态直接打断，恢复到正常状态？所有的方法经过这样正常调用的流程，那这一节的内容非常短小精悍，就跟同学们分享到这里了。那我们在接下来的小节里，将跟同学们一起去 Sentinel 当中，实际的去尝试配置一个降级规则，然后观察他的访问行为。OK，同学们，我们下一小节再见。
