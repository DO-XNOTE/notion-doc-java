不过随着后面的历次双十一的磨练捶打， Sentinel 它自己的应用场景，在整个 13 年到 17 年这内部的双十一各种链路的捶打之下，它已经完全成为了一个基础模块。那作为一个基础模块，你要去服务于整个集团业态的各个事业部，这每一个事业部又有他自己非常独特的业务场景，那所以经过这四年的研发，还有这四年的发展， Sentinel 这里已经积累了非常多非常丰富的生产级的实践，它的业务场景已经覆盖到了几乎可以说所有核心的主链路这里都是接入的Sentinel，所以它在内部实际上已经有非常广泛深厚的群众基础。

在介绍它实力之前，我们想先给同学们了解一下 3 TL 这里的一个很关键的概念，两个字，资源。OK，那 stantinal 当中它的作用，它的所有限流、降级、黑名单、访问控制等等这些作用点实际上都是作用在资源这两个字上的。资源同学们知道非常的宽泛，你可以是 Java 当中的任何内容，比如说我可以是一个接口调用，对吧？或者我是一个 RPC 调用，那也可以是一段接口当中的代码块，我都可以把它叫做一个是资源。Sentinel，这里实际上通过两种方式提供了对资源的控制。第一种方式，我可以使用 Sentinel 的开发的核心的注解， Sentinel API，这些注解加在我的接口层面、资源层面上面，就能和你的代码执行流程以及它的大盘监控形成一个非常好的联动。这是一个 API 接口的形式。另一种形式就是我可以在代码当中，我去声明相关的 Sentinel 的控制点。这是一个编程的方式，一个 coding 的方式来做 Sentinel 埋点。通常来讲，这种方式灵活度实际上更高，你可以控制的力度会更加细致。当前方法内你可以划分很多个代码块，甚至对这些代码块，我可以应用不同的 Sentinel 的限制策略，这都是可以的。


不过我们这里建议同学们采用的方案实际上是第一种，这一种可读性更高，代码侵入更低，就是 Sentinel 的 API annotation 的形式，那通过一个注解定义相关的资源，在咱的大盘监控，就是咱待会儿会跟大家看到一个叫 dashboard 这样一个大盘监控，在上面去定义相关的规则应用到它的接口层面注解上的资源里面。
