通常几句话也说不下来，但是可以通过这个问题来考察你对配置中心的一个全局的了解，以及你的解题的思路。那么我们这里如果想要入手这个问题，实际上就是从需求层面入手，最简单任何的问题。我们先理清需求，那比方说你可以跟面试官之间做交流，你需要我设计配置中心，那么你的业务想要达到哪些需求，那这个是可以去交流的。或者你可以直接给面试官说出你自己当前对配置中心的认识有这么几点，比如说几个简单的需求点，环境隔离、多元数据获取这些配置项，还有你运行期的属性动态配送，以及一个在大厂里面绕不开的问题，那就是安全。


那在这个过程当中也可以跟面试官有个互动交流，因为我们通常面试不是说单口相声，而是一个对口相声，你是这个主讲，相当于你是逗，跟那个面试官是捧哏，所以说以你说为主。但是就是有些东西需要跟面试官做一个交流，这样面试效果会好。你比如说当前这些需求点，如果面试官还有一些补充的话，那我可以一起纳入来考虑。


那么针对这几个需求点，我们就可以去挨个展开来说，比如说像环境隔离，那么环境隔离的手段就是应用profile， active 或者namespace，那这里我们同学们其实可以跟面试官来探讨，就是在 config 组件里或者是 necks 组件里，我们底层是如何通过一个配置项来最终把它映射成对应的配置文件的这个寻址地址的。那实际上这里的底层实现非常的简单，无非是加入一个参数，把它拼成一个最直接的一个文件的字符串来去获取，那这个没什么难度。


那再往后像多元，那多元这里就牵扯到一些架构层面的事情了，比方说你一个框架里，你要从多个不同的目标类型的存储介质当中来获取你的属性，那有的是数据库对吧？有的是GitHub，还有是其他版本控制工具，那如果你一套工具，你写一套完整的方法，那实际上从架构层面来说并不是一个很具有可扩展性的这样的方式。
那我们这里实际上你在底层实现就可以应用一些设计模式，比方说 Adapter 模式，这就是一个比较方便的模式，为每一个不同类型的存储介质，我给它开发一个适配器，那么这些适配器可以去继承一个统一的接口，那这样的话以后你有一个其他的源数据库，对吧？那你就可以去将这个源对应的获取数据的这个逻辑封装成一个Adapter，那么它也是继承了统一的接口，非常方便的扩展。


并且我们这里可以连带着跟面试官说出我们的 autoconfig 这种自动装配，当在配置项当中看到哪些属性，以什么样的字段开头的属性之后，我会采用哪种装配方式？那这个自动装配这一个知识点也是 spring boot 很重要的一个知识点，就可以通过这个多元这种方案来去把它给带出来。


所以像面试过程当中就像高手过招，有来有回，它不是说你当前面试问题，你就围绕着这一个面试问题来回答，你要把自己懂的其他的知识，就是不经意间诶把它给带出来一下，比如像这里 auto config 你就可以带出自己对 springboot 的一个理解，对吧？这样的话面试官就会觉这小子有点料，你通过一个问题可以旁征博引带出这么多其他的知识点，这是很好的一个面试效果，那再往后向动态推送，这里我们可以采用很多种不同的方案来，你本地可以去监听一些动态变化的事件，我可以监听消息组件来获取这个变动通知，重新拉取。我也可以在自己的上下文当中去监听 spring 自己的，像这种refresh， event 等等都可以。或者我可以像 neckles 一样，我去做一个长连接，长连接到谁？连接到你的对应的配置中心上面了，这都是可以实现动态推送的一个过程。然后获取到之后如何来刷新 Sprint 上下文，对吧？这一个点也是可以带出来 springboot 的一个知识点，刷新上下文当中的属性配置，okay。


最后安全的 in ETI，其实安全分为两部分，一个是数据安全，还有一个访问安全，那么对应数据安全这个议题来说很简单，就是加解密，对吧？你的存储的属性，我为了不对外泄露，我用一种最安全的方式把它给加密，然后我在读取属性的时候把它给解密，那这个加减密呢？同学们其实就可以去参考 spring Claude config 组件是如何来做的，它的对称密要加密，非对称密钥加解密等等。那么这里如果时间足够，你甚至可以稍微唠两句，你对这个非对称密钥这个密钥加密过程当中的理解都是可以的。


那么这里除了以上列出的这些简单需求点，像我前面提到的跟面试官多一些互动，看一下它还有一些额外的业务场景没有，然后你再基于它的反馈来做进一步的推理介绍，OK，那么除了这些功能点，我们还有一些其它的附加点。比如说你的配置如何来做持久化，持久化这里有很多方式了，你持久化到数据库当中，持久化到本地等等。


那这其实可以验证你一个方案设计能力的全面性，那你不能是只完成了业务需求，业务需求其实只是你整个方案冰山上这里冒出头的这部分，那实际上你在这个海平面以下，这才是冰山下面的一个很宏大的部分。也就是说你用到的功能它需要很多支撑模块来去支撑它。那这些部分我们在设计方案的时候要一并把它考虑到。比如说你的属性持久化就是其中的一个部分。那另外你有了这些功能，酒香不怕叫个深，你怎么让别人来去应用这种功能？那么这里就涉及到一个 API 如何暴露出去，那一个常用的方式就是springboot，它支撑的 actuator 端点，你可以去通过这种统一的方式来去把它给暴露出来，或者你也可以自己开发一个Controller，暴露出自己的接口都是可以的，很多种方式。


除了这以外，版本控制配置中心，咱说过配置也是代码，对吧？所以你一个配置，它如果配置出错了，对你系统的伤害有可能比你一个 bug 还要多，还要严重。所以说作为配置中心来说，我们在设计的时候一定也要把这个版本控制给它考虑进去，包括你如何来去做一个快速的回滚等等，那么把这些东西都谈到了，那么这一个面试问题设计配置中心，我相信就答得非常非常到位，非常好了。OK，那这是第一个问题啊，接下来我们再来聊一下
