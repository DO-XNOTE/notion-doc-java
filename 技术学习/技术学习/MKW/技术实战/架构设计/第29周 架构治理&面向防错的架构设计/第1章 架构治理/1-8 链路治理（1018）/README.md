我们自然也没有办法对其做相关的治理，因为我们连哪里出了问题都不知道，通过有点瞎猜或者通过很直觉的方式去感知这个东西，很多时候是会走很多弯路的，那么其实业界有非常多的可以做这种全链路的，日志也好，做 trace 跟踪的也好。其实是有很多的这种开源，比如像阿里鹰眼，比如说像阿里鹰眼，虽然它不开语，但其实它是做的比较完善的一块。


还有就是阿里云的 Charchase 也叫链路跟踪，它是阿里云上提供的一款产品，如果你是基于阿里云体系构建的你本身的基础设施，那使用这个会非常的方便。当然美团点评也开源了一个 cat 组件，基于这个组件也能够完成相关的链路跟踪这样的一个效果。当然还有业界开源的 zip Keen 和 Google 本身提供的defer，但是 Google 的 defer 目前只看到论文，却没有看到开源的产品，所以目前主要能选择的还是 Zap king 以及美团点评的cat。


如果在阿里云体系下，也可以使用 XTRACE 这个链路跟踪的工具，它能够帮助我们去明确我们的微服务调用链，知道从入口到最终返回整个的一个微服务的执行的情况，甚至它也可以帮我们去明确相关的子调用的情况。所以它会包括哪些点？包括对应的应用的调用以及应用内方法的调用，还有中间件相关的调用，以及一些存储相关的调用，比如说像MySQL，像 Redis 相关的调用。


那通过这些它就能够帮助我们知道整个的一个链路，它到底执行了哪些，运用了哪些方法，调用到了哪些中间件，哪些存储，并且知道其在在整个调用过程中它基于了哪个区域？当然如果你是单元化的，就可能会有对应的单元所处于的全国的区域，然后所处于的机房相关的执行时间以及执行的RT，这些可以帮助我们来确认我们的方法调用是否出现了跨区域、跨机房等耗费时间比较长的这种请求，抑或是帮我们去分析我们整体调用的时间以及 RT 的情况，来确认我们到底整个链路中慢慢在哪里出了问题，那卡在哪里都能够通过这个来识别出来。


那么有了这样的一个工具，我们就能够很好地认知到我们的链路到底哪里有了问题，哪里需要去提升，通过不断地修复，然后不断地再去看我们整体链路的一个调用情况，从而来明确我们修复是否真正产生出了它一对应的效果。所以在这里我们需要做几个点，

第一个点就是我们可以通过这个明确出我们的需要，这里的链路里就会包括需要治理的场景和整体的一个基于用户什么样的行为下产生出来的整体的链路调用链。第二则是链路的健康相关的一些信息，包括整体的成功率，也包括子调用的成功率，还有对应的RT，然后还有链路评估去明确我们整体的链路它是怎么样的一个追踪，并且针对我们希望定位的一些问题做好链路染色，这样子我们就能够很清晰的知道到底我们的链路现在处于什么样的一个状态，它是否健康？它哪里需要治理？需要怎么样的治理？而这里就意味着这些点都是影响到我们链路治理整个过程的这构成元素，它帮助我们的无非也是问题发现和问题修复。


通过链路跟踪来明确我们链路中存在的问题，并且针对性地进行修复，再进一步的通过链路来进行问题的验证，这样我们就能明确出我们链路中存在的问题，并且明确出我们的问题是否智力并已对修复了。当然最好的还是基于全链路压测进行压力小的链路情况去查看，因为很多时候一些问题只有在高压情况下才能够出现的比较明显。在链路治理中，哪些问题是我们要高度关注，

高度注意的点的第一个点就是链路的平滑，我们要确定我们的链路它是否属于一个平滑的状态，在压力下是否会一如既往地表现出是一贯的一个表现，还是说当粒来临的时候，它就会出现有些请求，是一个很好的一个效果，请求时间短， RT 也比较短，成功率又比较高。但有的时候它就会开始抖动，出现大量的毛刺，出现一些长RT，甚至出现一些请求漂移等情况。


第二是明确一些子调用的一些管理，针对子调用来明确出我们哪些子调用是健康的，哪些子调用是存在问题的。因为我们整体链路是由各个子调用串联起来而构成的，所以一旦子吊用出现问题，那它就成了我们整个链路中这个木桶原理中的个短板，针对子吊用做细节上的修复和治理，就可以帮助我们提升木桶中最短的那个板，从而让我们的链路可以更加的高效。


当然也要针对整体的链路成功率做相关的治理，确保在一定压力下我们能够达到对应的一个成功率。那在这里我们就去细细看一下。比如像链路平滑，我们就要关注这个流量请求的漂移，因为当压力比较大的时候，出现了线程比较激烈的竞争，那么这时候从一个子调用在折成下一个子调用的时候，就因为线程的竞争等待就可能出现请求的漂移。在这些点就是我们需要去关注，去优化，去明确我们怎么样子可以优化我们的线程模型，让请求可以更高效地响应，而不至于等待太久。


其次就是请求的毛刺，毛刺通常是在某些数据、某些场景、某些行为下，会出现响应时间比预期要长，甚至要长很多，这种情况，那毛刺肯定不是我们希望的这样的效果，所以我们就要针对出现毛刺的时候分析整条链路，去明确毛刺到底是因为网络原因，因为代码原因、因为场景数据原因抑或是因为线程竞争等原因。通过明确出的原因进行专项治理，可以有效的缓解毛刺，让我们的链路更加平滑。


而在此调用这里第一个很重要的就是不合理的循环，比如说我们在循环中进行了一些长 RT 的操作，那这些都会导致整体的链路执行时间会被拉得很长，因为每一次籽调用都会有意味着有建立通信，然后拿到数据再销毁这样的一个过程。所以将长 RT 的事情一次性的执行完，再在内存中进行相关的循环处理，可能会比在循环中进行挨个的查询，挨个的计算，然后再统一合并，前者肯定是会更优一些。


第二就是针对不合理的子调用进行相关处理，因为有些子调用其实可以不使用，或者是它的位置可以进行相关的调整，抑或是可以异步化，通过这样的一个方式，能够将整体链路的 RT 进行进一步的下降，确保我们的链路更加高效，更加稳定。第三就是针对过慢的子调用，又针对专项进行处理，如果子调用本身可以做优化，那就要对子调用的内部性能做相关的提升和优化。如果子吊用没有办法做优化，那就看是否可以异步化，可以并行化，通过异步和并行的方式来降低对应的执行RT，从而缩短整个链路的执行RT。


第三就是链路成功率，链路成功率整体展示会有一个指标，这个指标其实很容易感知到，但是我们需要关注的是是否是因为局部的超时，或者因为局部的长 RT 而导致全链路的超时来明确这一类场景，在这类场景下就需要针对局部进行性能提升，来保证我们链路的成功率能够不断地提升。


二就是针对中间件的成功率做相关的定位，因为有的时候中间件可能会出现大规模的失败，这种场景可能很多，比如说像网络原因，比如说像某些场景下中间件请求不合理，或者中间件压力过高等等。那针对这种做针对性的修复和改进，来保证链路的一个成功率，当然这些都有可能是由压力导致，那我们也需要结合全链路压测来查看一下我们在压力下整个链路的表现，然后再结合着链路平滑子链路治理，共同去贡献给链路成功率。


好了，到这里为止，这一小节的内容就宣告一段落。针对链路治理是一个长期的事情，我们需要不断的迭代去明确出链路中存在的这些问题，然后针对问题各个击破，提升链路的可用性，同时降低链路的消耗时间，最终让我们的架构能够承担更大的压力，并在大压力下表现仍然良好。那么在下一小节中，我们将继续专项治理之环境治理，跟大家聊一聊在我们的各这个环境中，我们要怎么样子去做好相关的治理，能够让我们的研发、测试和线上运行都处于一个健康的一个状态。好，各位加油师们，我们下一节课不见不散。
