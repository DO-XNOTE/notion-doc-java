那升级也意味着我们在不断地往里引入各式各样的新的功能，新的能力、新的周边的依赖、新的兄弟团队等等。这些架构的升级都能为我们带来很多好处，比如说我们的用户体验上升了，我们的架构的并发提升了，我们具有原来很多不具备的功能了，但是因为我们的改动，那也有可能会为架构带来额外的风险。


同时在架构的升级过程里，我们也要考虑曾经在架构中已经引入的那些风险，我们又要怎样去治理它，怎么样子让风险化为无形？这里就意味着我们要去提出这样的问题，我们的架构中都有哪些问题？围绕着这份这些问题之后，我们再要去想我们在整个架构的升级和治理过程中目标是什么？围绕着这个目标之后，我们再要去思考我们的架构该何去何从，我们的架构要迎来怎样的发展，它曾经历史上的那些问题又要怎么样去其解决，这些都是我们需要考虑，并且它们也都是无时无刻不在变化的。
当然除了这些之外，我们还要面对哪些变化？比如说行业也在不断的发展变化。你想今年疫情太多的行业因为疫情被改变了，房防疫工作的落实不再允许大规模的人群聚集，这些都意味着很多原来线下非常热的，非常火的这些行业都被受到了影响，而原来在线上相对可能没有那么火的，反而因为疫情的原因在线上非常的火了，那这些都是行业的变化。其次就是技术的进步，像这几年技术在飞快的发展，无论是云原生还是人工智能，抑或是之前火过的大数据，以及后续可能还会继续突破的种种技术，这些技术都可能为我们带来架构上的变化，可能原来我们的架构是基于微服务去建设的，现在我们就可以基于云生技术去重构我们的微服务，这些都能够为我们带来架构上的升级，带来很多的好处。


其次还有业务的发展，我们做的事儿都在不断的变化，我们也希望让我们的业务有不断的发展，那它的发展也会反向的推动我们去创造很多变化。当然不可避免的还有人员的变迁，铁打的公司流水的兵，每个公司都面临着人员不断的变迁，在这种变迁的环境下，我们要拥抱这种变化，

并且能够让这些变化让我们能够越变越好，所以架构变化是不可规避，也是我们需要去拥抱的这样的一个点在架构变化中可能会有环境的变化、代码的变化和依赖的变化，这是从底到上的三个层次的变化。
环境是我们的架构运行处于其之上的这样的一个载体，而代码是我们的环境的这里面的一些水泥，它把我们所有的依赖的外界的这种库，或者内部的二方的三方的这些服务，以及我们自身写的这种基础的代码或者是业务代码粘合起来。当然也有刚才提到的我们依赖的其他人的这些能力，可以帮助我们快速地实现相关的功能与迭代，那这些都是我们将要面临也要去拥抱的变化。


在这些变化中都有可能会引入相关的问题，代码可能有bug，环境可能会被某些升级破坏或者存在一些兼容性的问题而依赖，也不能百分百保证它一定是好的，这些都有可能带来各种各样的问题。当然我们要基于这些问题去做好我们的架构治理，从而带来架构层面上的进步，并且进一步推动架构的变化。在整个架构的变化中，我们也要考虑到的是我们相关的功能的开发，所希望的面向于某个特定领域或者特定方向的架构升级与架构重构，以及发现的这些问题的修复。所以我们可以发现在整个这样的一个迭代过程里，它是我们架构进步的源动力，就是不断地发现问题，治理问题，推动架构进一步再进一步的促进架构本身的变化，包括环境的变化、代理的变化、依赖的变化，让它们都越变越好。所以一个好的架构不是天然就设计好的，也不是说一蹴而就的，它是通过不断的迭代计划而来，不断地去发现问题，去治理问题，从而带来架构相关的进步。那么在整个的一个治理过程中，就意味着我们需要明确我们的驱动力到底是什么，是什么一直驱使着我们不断的前行。

当然第一肯定是业务驱动，我们的业务在进步，业务的进步就带来了公司的进步，带来了我们大家的一个进步，这里是一个很大的驱动力。其次是技术的变革，每一个领域的技术发生了翻天覆地的变化，这些变化都会反过来作用于整个行业的进步，无论业务进步还是技术进步，最终决定的是人才的一个进步，通过有我们自身的成长，以及我们不断地引入外界的优秀人才，这些都可以帮助我们让我们的架构变得更好。


这三类就是我们主体的三大驱动力，也通过这些驱动力不断地驱动着。除了架构的一个升级，除了我们功能的开发之外，它也驱动着我们不断地进行架构的治理。我们要考虑在业务不断的变化之上，怎么样保证我们的架构不因为过大的流量变化，不因为流量的剧烈的变化，它突然暴增，突然暴跌，通过这些导致我们的线上问题，那这些都是我们需要去治理的。


其次就是我们本身有一些落后于整个行业的这些技术，这些能力，这些功能是否也需要不断地重构，不断地革新，跟上行业的发展而避免一旦当相关的技术不再被行业支持的时候，它可能为我们带来的难题就会比当前去治理掉它更难、更大，但最终也是希望我们不断地让我们的人在其中得到成长。
