说起这个数字化，我想就是很多时候我们可以这么去定义，它就是数字化的，有时候是可以去定义业务，也能通过数字化去定义架构。我们想要让业务能够更明确地去知道它当前的一个发展情况，我们也想让同学可以知道在未来我们怎样期望它往哪个方向去走，怎么样子让它往那个方向去走？这些可以通过数字化、量化的去推动这个事儿。

但是数字化是要超脱于度量体系的，因为度量体系更多的还是更关注于我们需要关注的那些指标，而数字化则不仅仅在于指标，他要考虑的东西远比指标要大得多。因为我们很多时候是要想要将业务能够数字化，向架构也能够数字化，这里的数字化的概念就会远远的大于度量体系。


我们可以细的去看，比如说数字化定义业务都有什么呢？必须第一点，业务全面在线，这个怎么理解？如果我们有一个业务是在线下，需要人自己按照一个流程去做的这个事儿，其实就不是在线的一个事，它属于一个离线的一个事儿。那这也意味着 g 相关的这个过程是不受数字，就是不受我们的系统去把控的，也意味着它本身不在我们的数字化范围内。


其次就是变化的自动适应，我们的业务会有很多的变化，我们的环境有很多的变化，我们的团队、我们的人也会有很多的变化，那在这些变化中，我们都能够通过数字化的方式去适应这些变化，让这些变化都能够在我们的系统中，在我们的架构中去体现出来，并且用架构去支撑其发展、支撑其变化，那这个就叫做变化的自动适应。
还有就是自动化智能化，这就包括我们的工单体系，我们的很多流程，在线的这种流程，还有我们的一些机器人，一些 RPA 辅助的一些工具，那这些都能够帮助我们更好地去完成相关的业务。当然还有最终的我们这个效果度量指标，它是我们能够更加明确、更加数字化的，更加具体的去看我们的执行结果，执行效果通过度量指标能够很清晰的知道我们当前到底做的怎么样，那么这些就是我们所谓的数字化去定义业务。当然还有我刚才提到这个数字化定理架构，在架构层面上第一个点就是要架构，它要有能够具备内饰感知的这样的一个点，就是架构它本身可以知道我这个架构自己是什么样子的，我内部是不是有什么风险？我是不是遇到了不好的一些状态？那这些呢？都可以通过数字化的方式去感知的到。


第二就是问题发现这个和安全生产就会比较契合了。第三就是架构的特征量化，我们架构会有很多特征，这些特征很多时候是我们需要去关注，并且需要针对其变化做相关的调整，那么这些时候量化就能够帮助我们更好地去做相关的事情。最后也是质量的相关度量，这些都可以帮助我们知道我们质量到底怎么样，我们是否还有需要去提升的部分。


总体上去看，数字化更多的不在于说我们到底是要多少个指标，而在于通过信息将一切串联起来，将一切链接起来，它能够将所有的东西都信息化，这这个其实就跟企业架构数字化其实是有一点相契合的。我们希望将整个公司，比如说我们都要知道我们架构可能包括 it 架构、应用架构、企业架构，那这些架构本身来说都是希望将我们整个公司的这些信息能够线上化，能够数字化，能够信息化。那同理我们的架构的数字化也是希望我们将架构的一切都能够信息化，从而能够让我们更好地去感知我们的架构。所以数字化也是充分发挥存储、计算、大数据与智能的前提。那我们的架构其实也就是执行着这个存储、计算、大数据和智能，那所以数字化也是我们架构去充分发挥其威力的前提。

因为架构其实都是我们的研发人员去研发出来的，这个视角里我们就会有很多偏向于执行，偏向于架构的创造的这样的一个部分。其次是使用视角，这部分可能更多的面向于运营，他们使用我们的系统，使用我们的架构来执行相关的事宜，来完成我们的业务。


还有一个就是面向于我们老板，我们的主管面向于公司层面上的管理视角，它能够通过这个视角去看架构能够为公司的整体管理，为团队管理带来什么？当然我们可以细的展开去看，在研发视角下，研发肯定是第一大项，我们需要做coding，就要做架构设计，那这些都都是我们要在研发层面上去做好。


当然做完研发就有我们的必然的凶力，就是测试，还有相关的运维，包括从线下到线上的过程，以及在线上的一个保持，那这些都是我们需要去考虑好，需要去数字化的点。那这些反过来去结合我们的之前的这个安全生产的这些思考，也就是说包括我们在研发测试，在运维流程里，我们都是需要有系统去支撑，那当然现在业界有大量的系统去支撑着整体的研发测试，包括线上运维，现在被甚至说我们的 DV OPS 由研发来运营，它包括云原生这些很多都是研发去直接做了运维的工作。那这些通常来说都是一个在线的信息化的一个过程，原有的那些完全通过手动一点点发布上线，手动在线上不断的去运维的这些事儿，都在慢慢的退出历史舞台。


所以意味着在我们日常的架构的一个工作中去考虑在这些范围，也尤其在研发视角上，是否我们还会有一些数字化没有做到位的，还有很多需要人去大量兜底，需要人去参照一个 list 不断的去离线执行，需要人去过做这个过程的这样一些点，如果有尽快的解决掉它，因为它就是可能为我们带来架构风险的一个最大的一环。


第二首先是使用视角，更多是面向产品运营以及我们的用户运营，这是两类方向的运营，他们都是运营，当然还是最重要的就是我们的用户他来使用我们的产品产生相关的价值，而从管理视角上看就是经理。还有我们的 BI 就是他们是以管理视角去思考，然后提供给管理人员以运营支持决策，以管理决策的这些内容。当然还会有财务，因为它需要从财务的角度去看整体的一个架构，需要从管理的视角去理解我们做的这些东西，做的产生的成本和收益，从管理层面上是否是合理的。


当然不管怎么样看，多样化视角关注肯定是有差异的，但是无论差异多么的大，最终还是要以人为本，他都要关注的是研发同学，我们的运营同学，我们的管理同学，通过不同人各司其职都去做好自己的事儿，然后共同的去促进整体的一个架构质量的不断提升。那也意味着我们要从这三个视角去看，我们的研发整个过程是否全都数字化了，我们的整个的使用过程是否也数字化了，我们的整个管理的过程是否也数字化。将这三个最起码最基础的视角去看待我们的架构，当前的数字化生物如果有做的不够的，尽快地去调整，尽快地去做改变，让这些没有我们能够数字化的点，都能够在数字化的大浪潮里可以快速上车，快速前行。好，这一小节相对来说就会比较简单，那在这一小节中，我们也希望大家是什么呢？我们不会需求大家告诉大家怎么样去做数字化，我们需要告诉大家怎么样去理解这个数字化，更多的是要回过来去思考我的整个过程里，研发视角的过程里，使用视角的这个过程里，管理视角的这个过程里，是否还存在着大量未数字化的部分，存在大量人工兜底的部分？但如果有，这也是我们面临的一大挑战，面临的一大风险，快速解决，快速落地。
