# 1-6 依赖治理（1421）

hello，慕课网的跟未来架构师们，欢迎回来，在上一小节里，我们为大家开启了架构治理这个篇章中的专题治理，并且为大家介绍了怎样的去针对性地治理我们的现场性能。性能都是在高流量下才会爆发出来的这种问题，所以针对高流量下关于相关的缓存和流量的准备工作，以及针对流量进行相关约束，包括流量整形和限流降件等流量控制工作，通过这些手段从而让我们的线上具备可承受更高流量冲击这样的能力。这是我们希望我们架构所具备的，也是我们在架构的性能治理中所需要关注的那些点。那么在这一小章中，我们将为大家继续我们的专题治理。这次的专题讲的是什么呢？讲的是相关的依赖。

大家互相紧密合作，产生出合力，以团队精神拿到最夺人胜利，这都是我们在获取胜利的一些常规的一些方法。那在这个方法中也意味着我们就要使用其他人的现提供的相关代码或者能力，那么这些对于我们来说就是相关的依赖，同理我们也会提供给他们相关的能力，我们对于他们来说也是依赖。但是我们也要认识一个问题，就是依赖它也不是尽善尽美的，当然优质的依赖会提供更多优质的服务，它也能够帮助我们最大程度上提高我们的研发效率，以及提高我们快速达成目的这样的一个效果。但是同时依赖它也会存在着一些问题，在这些问题中我们就需要好针对性地去进行相关的治理，从而提高我们的架构质量。


当然依赖必然能带来研发提效，因为相关的东西我们不需要再重复建设了，但是依赖它也需要不断的升级，同时也在不断的升级，也意味着我们也要不断地去跟上依赖的升级，去对相关的依赖进行相关升级，从而来享受依赖能给我们带来的最新的这些特性。但是升级就意味着有变化，就可能引入风险，而且依赖的本身也可能会出现故障，比如说你依赖了上游的一个巨头，比如说用户在下单的时候，它除了依赖商交易系统之外，它也要依赖用户模块，那用户提供的能力就可能就是交易系统的依赖。但是这时如果用户系统出现故障了，那么交易系统也是必然会面临着故障，所以这就是依赖可能会引入的相关故障。


并且依赖也可能安全漏洞，他们也是写代码，他们也是常规的一个开发，所以他们也可能会有安全漏洞，那他们的安全漏洞就有可能会有人顺藤摸瓜，顺着他们的漏洞入侵到了我们的系统里，这都是我们需要考虑的以及依赖本身的机会成本。我们用 a 还是用b，它们可能都会有各自的优点，也有各自的问题，到最后我们使用了一个依赖，它的相关升级和替换的成本也是存在的，这些都说明了一个问题，我们需要在线上进行持续的依赖管理与治理，确保我们线上所使用的这些依赖和我们作为依赖本身提供给他人来说，都要面临着针对相关的风险要进行梳理，针对相关的问题要进行解决，从而让我们的线上能够更加的健康。

依赖肯定也会面临着依赖的分层，因为有一些是我们直接依赖的，当然也有一些是我们间接依赖的。电商模块儿依赖了用户模块儿，那用户模块儿就是电商模块儿的直接依赖，但是用户模块儿又依赖了一个分布式的数据库进行相关用户信息的存储，那么这个分布式数据库可能就是我们的间接依赖，同时还有循环依赖。订单模块依赖了商品模块儿，商品模块儿也需要从订单模块儿这块儿去依赖相关的关联的订单数据，那这里就存在了订单模块跟商品模块之间的一个相互引，循环依赖。从这三个点能看，直接依赖通常是我们很容易去感知到的，而间接依赖很有可能会是我们在架构管理中会容易出现疏漏的那环，而循环依赖则由于其循环特性而导致其升级的过程非常的麻烦，并且很容易就出现问题，因为双方都存在着把升级内容升级的时候很容易形成一个升级死锁，所以针对性地去解决循环依赖和降低间接依赖可能带回来的风险是我们在依赖分层中所需要考虑的。


其次就是依赖保护，因为除了我们要用它之外，我们要预防它本身的故障对我们带来的影响，所以这里我们需要做好熔断的保护。当某一个依赖出现问题的时候，我们要将故障的影响范围控制在一个较小的范围之内，而不至于整个系统都因其连累而垮掉。有的对的熔断也意味着我们需要考虑在这个依赖它本身出现问题的时候，我们要有一个兜底方案，通过兜底方案来避免所有的用户都被出现故障的依赖所影响。其次也包括依赖相关的安全，我们要持续关注依赖本身存在的相关安全问题，而快速的进行升级响应，避免因为依赖本身的安全问题而导致我们的系统也存在相关的安全问题，避免一些非法分子通过依赖本身的漏洞顺藤摸瓜来从而掌握了我们系统的一些权限。所以这 r 加起来就是我们要做的依赖治理，它都会包括哪些方面？比如说像临时包的问题是因为临时包可以在开发环境就被deploy，这样子的话就会导致其稳定存在的一些不确定性，很可能你用着一个版本，在正在现场跑着一个版本，突然有一天被一个人意外的 deploy 了，那可能就会带来一些灾难性的问题。


其次就是依赖的升级，针对一些万年没有升级的一些陈年老依赖，要看一看是否可以允许升级。而针对一些正在快速迭代的一些依赖，我们也要能够不断地去享受其提供的最新的能力。万年老依赖贸贸然升级，很有可能会导致其新版本跟老版本不兼容，因为期间隔了太多的版本。而针对于快速频繁升级的这种依赖，这个问题就会少很多，因为每一次版本的更迭，它都会告诉你你需要怎么去变化，并且大部分版本来说它会保持向下兼容，或者需要其新版本提供出来这些能力，我们可以针对性地做相关的升级。


其次就是依赖冲突，很容易出现无依赖两个不同的模块，这个二者从是有一个不同版本的间接依赖，那这时就会出现相关的冲突，有的时候的依赖冲突很容易解决，只要排掉其中一个包就好了。而有的时候因为其两个版本的不兼容性，可能导致其依赖冲突非常难以解决，甚至需要依赖方进其中的依方进行相关的发版，从而来解决相关的依赖冲突。


而依赖瘦身则是指很多依赖，包括一些间接依赖，其实我们用不到的，在这种情况下它就会无形中扩大了我们的包的个大小，通过一旦收帧，可以很好的降低我们包的大小，从而让我们的部署更快，启动更快。当然还有一些私有依赖是我们手动 deploy 到我们私有端的仓库中去的，这种一旦在发到线上去的时候，如果在线上仓库中没有，极有可能导致问题，或者当我们把相关的依赖交付到使用方去使用的时候，他们手里也是没有这些私有依赖的，这时候也会导致使用中出现问题。


所以针对私有依赖需要进行相关的治理，确保其不会影响到依赖本身，也不会影响到我们的架构本身。

所以在依赖管理中，我们需要尽可能的用正式包替换掉临时包，这就是专项的 Snapshot 治理，我们要看看我们的依赖的这个 Marvel 里面它到底有多少 Snapshot 包，哪些是可以保留的，哪些是要治理掉的。其次就是一些版本问题，是否有漏洞，版本有过往于第一的版本，有已经研发者不推荐的版本，那对这些问题版本进行相关的升级，产品升级一定要小心哟，不要说觉得升级完什么都不用管了，升级很有可能会引入一些问题，所以在升级完之后一定要做好相关的测试和灰度，避免因为升级而引入故障，这样的话真的就得不偿失了。


其三就是因为一些包的它的依赖的能力问题，我们可能要做到新的要升级，历史的要清退掉，比如说一个包它本身的能力已经我们不需要了，那我们就把它清退掉，如果包它不断地提供新的能力，我们需要使用其新的能力，那我们就要做好相关的升级。这时就我们就意味着我们要做好依赖的一个功能的升级与改造，而针对间接依赖和循序循环依赖就我们就要做好相关的冲突的管理，避免因为循环依赖而导致升级出现死锁，也避免因为渐渐依赖中版本冲突而导致线上出现多套类或者不兼容的类。
当然我们在常规情况下也会认为依赖是个上下兼容的，但是有的时候可能也不一定，可能两个版本之间存在较大的变化，这时候就需要专门的去针对性地进行相关升级，这东西都是我们的一些稳定的保障，同时我们本身提供依赖的时候，我们也要做好向下兼容，这个是我们作为依赖的一个基本要求。


说了这么多，其实也是希望我们在使用依赖和提供依赖的时候，都尽可能的让我们的依赖更正式、更稳定，更向下兼容，从而能够让我们的线上可以质量更好。这里就简单的针对两个点做一个延伸上的讲解。

而这种非常复杂的这种关系也可能意味着我们光解决一个依赖就要一天甚至一周的时间，那他对我们肯定是一个非常不爽的一个事情。麻烦本身也提供了一些，比如像 Mobile per 这个插件儿，它能够帮助我们去分析整个的一个依赖数，但是它很多时候也不是一个万能百分百的，还是需要用 Marwin 自身的这个 despendency tree 去查看相关的一些类，然后去定位到我们到底哪些冲突，然后一个一个的排掉，再重新去build，重新去一次到去探索一下我们中还是否出现了某一些依赖的冲突。
