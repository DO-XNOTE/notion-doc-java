# 1-11 风险治理（1257）

hello，慕课网的各位未来架构师们，欢迎回来。在上一小节里，我们为大家介绍了关于技术债务的治理。技术债务其实就像一座大山，它一不停地压在我们头上，让我们喘不过来气。它一方面影响我们当前的工作状态，另一方面也会影响我们后续架构的发展。同时也因其债务性，所以它会不断地向我们收利息。这个利器可能是在研发过程中不断变慢，不断让我们难受的这样的一些方面，也包括了它可能会为我们带来线上故障的这些方面。所以我们要针对企业进行相关的治理，选择一个合适的时机与方法，最终可以去降低技术债务，并且减缓其对我们的影响。


那么在这一小节里，我们将进入本一篇章的最后一节风险之旅，跟大家聊一聊关于风险这个点我们怎么样子在架构设计中去考虑进来，同时也通过风险的治理，通过架构的迭代来减小我们架构中存在的这些风险，提高架构的质量。当然我们必须要承认一点，就是风险无处不在，

不是说我们今天不动了，线上不动了，还有代码写了都不改了，它就没有风险，风险这就跟意外一样，它有的时候真的是我们不知道它什么时候就会到来。同时也因为我们的线上也在不断的变化，我们的架构在迭代，我们不停的有对应的新功能，那也意味着风险也在跟着架构去迭代。


就融图在之前安全生产的时候，首先讲到可能 60% 的线上故障都是由变更引入的，那么风险也意味着在架构的不断迭代，不能开发中，不断的新功能引流中，它也在慢慢的迭代陈降，那么在这里也意味着我们要去持续的做好风险治理，不是打一耙子就跑的这样的一那个事情它一定是一个随着架构的迭代不断地去进行的一个过程。
那在治理中我们一定要贯彻我们常规的这套方法论，就是发现之后我们先要对其风险可能存在的隐患做改善，比如说做止血，只有最快速的改善才才能够保证我们最快的从风险所能带来的损失中卷脱出来。当然我们也要在我们的下一个迭代中去考考虑将这个风险给它治理掉，然后再通过验证明确我们线上不再具备这个风险了，这样子我们就是形成了一个治理环，通过这样的不断的一个发现、改善、迭代、智力验证这样子的一个过程，来将我们线上的风险逐步的消除。所以当然我们要明确就是我们到底具备怎样的风险感知能力，

不是说放在那里它就会告诉我们他们有风险，一定是我们做了某些事情我们才能够知道几个风险。当然有的时候价格式的第六感也是一种非常好的风险感知手段，有的时候真的是老师也有这样的经历，就是诶，我感觉好像线上哪里不对劲，但是又说不出来，所以我就去挨个的去排查，果然让我发现了其中某些机器的负载不正常，然后再顺着负载顺藤摸瓜，然后找到了一些有问题的代码，而这些其实它还没有完全形成大规模的破坏。


因为那些代码是一些新开发的代码，它会导致一些查询变慢，而这些查询变慢会间接性的影响到我们的数据库，所以当它达到一个阈值，可能数据库扛不住了的时候，它才会爆发成故障。而在这之前我们如果发现它，解决掉它，它就会在我们的处理中要消散于无形。所以在风险感知这个点上，我们还是要去做很多事儿的，肯定是要架构来告诉我们它出了一个风险，所以这里我们要做好风险的防控，同时也要做好治理流程。风险防控这个点上我们需要明确出架构中它当前存在的那些问题，通过我们的报警机制来报出来，然后再围绕着我们的治理流程去针对这些可能存在的风险做好相关的。这里在这个过程里架构设计、架构原则和架构模式，这个就是我们在这门课中为大家反复强调的一些基本功能、基本概念和基本方法，通过这些手段来完成架构的相关治理，并且在线上做好相关的验证。如果不验证，那我们治理其实也没有任何意义，因为你不知道它到底属于什么样的一个状态。我们一定是要明确我们到底存在了哪些风险已经在治理中，哪些风险已经治理完毕，一定得验证完毕。

比如说组合、聚合、复用，通过这些思路来将我们架构的一些模块进行重组，其目的是为了防止因为其代码重复性或者代码的层次结构不合理，从而导致的一些因代码引发的一些问题。其次就是我们也是最常聊的这几个架构设计的原则，通过最小化我们的功能，让我们代码最简单，让架构更纯粹，这个一些都是可以尽可能的降低架构风险的一些原则。其次这是关于状态的设计，因为状态往往是引入一些风险的很关键的一个点，我们要明确我们的架构中的一些状态到底在哪儿管理，由谁来管理，其次是我们怎么样子，尽可能的让我们的服务都是一个无状态的服务，因为无状的服务也就意味着其具备副本性，它可以通过扩充机器来扩充能力，那这都也意味着我们可以通过扩充机器来解决流量问题等这种非常棘手的问题。


当然还有一个最大的也就是数据性能，其包括大 key 和大表，这些都会围绕着数据的查询与存储性能，以及它在流量下是否会为我们带来的相关风险。这里我们就要去做好相关的热点的一些处理，包括热点的分散，然后针对数据热点如何去打散，以及做好相关的数据冗余，因为数据冗余有的时候可以帮助我们少查一次库，少查一张表，也可以避免我们为了一条搜过去查出来，而是在代码中采用 drawn 的这样的方式，因为 drawn 本身来说它就会有比较大的一个性能影响。

其实风险治理也跟安全生产息息相关，安全生产更多的可能围绕着是线上的一些质量问题的快速发现与修复，而风险治理它一定程度上是安全生产的前奏，我们可以在安全生产进行实施之前先感知相关的风险，并且在安全生产关注的现场故障的点之前去落一些事情。当然也可以结合着安全生产的这些工具和安全生产的思路，在治理线上的质量的同时也治理我们的线上风险。那全链路压测、故障演练、应急预案和相关的安全与容灾，这些都是进行风险治理非常好的工具，我们也可以通过原全链路压测去发现我们的链路中存在的那些风险点，也许它还构不成故障，但是在未来的某一天它可能会导致一些问题，那我们可以提前去修复它，防止这只羊被偷走。


当然像故障演练也属于是一种通过各种手段来发现问题，我们同时发现故障的同时也可能会发现一些构不成故障的风险，那这些也可以随着优先级去修理它。当然应急预案、安全和容灾这些针对线上的故障做的相关的一些后续的止血与恢复的方案，在风险中也可以同样的去落，但是前提是我们肯定是要做好相关的优先级，明确出到底哪些是也最早要处理的。当然故障肯定是最早要处理的，但对于风险来说，我们可能就需要一个时间，比如说一个月或者两个月，在这样的一个过程里，在避免其恶化成故障之前，我们把它做好相关的修复。还包括像一些架构宣讲，那这些属于是一个文化，或者是大家对于这个架构做好更深层的了解，对于这个架构知识的学习，这是这一类的这一类相关的事情。
