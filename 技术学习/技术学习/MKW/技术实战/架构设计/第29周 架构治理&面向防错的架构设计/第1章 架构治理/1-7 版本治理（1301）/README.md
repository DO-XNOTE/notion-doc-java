# 1-7 版本治理（1301）

hello，慕课网的各位未来架构师们，欢迎回来，在上一小节里，我们为大家讲解了一下专题治理之依赖。治理依赖既可以为我们带来研发的提效，但它同时也可能会引入一些风险，所以我们在我们的架构中需要去把控依赖本身的风险，一方面保护我们的架构本身的质量，另一方面也通过依赖的管理来完成依赖相关的瘦身，以及在研发过程中，依赖如果出现冲突怎么样子快速的解决。


那么在这一章中，我们将从依赖的版本延展出来，延展到整一个架构的版本去聊一聊关于架构的版本之力。架构本身也有多样化的版本的概念，

比如说像它本身也会有对应的一个版本的一个发布，我们也在不断地去发布我们的版本，就像乌邦图这种开源的系统一样，它也会有长期知识版，也会有开发版。那像我们的Java，它也会从 Java 7、 Java 8、 Java 9、 Java 10、 Java 11 一步一步的发布上来。


那我们的架构本身也是如此，我们会有一个开发的版本，我们也会有对外的版本，那这些都是我们对应的版本发布，同时也针对在线上跑着的这些版本做好一个运营，因为当客户接入了某一个版本之后，在他们的认知里可能就停留在了那个版本之中。而这时我们在那个版本之后的所有的迭代，可能他们会有感知，也可能他们压根儿就不知道。在这种情况下我们就要做好版本运营，针对需要提升的版本需要去通知相关方，让其能够升级到最新的版本。这两个点我们就要做一下详细的一个介绍。
比如说要版本发布就可能会有明确出来的发布周期和版本力度，我们的一个周期到会怎么样子？是不是一个偏固定的周期，还是一个偏自由的周期？是不是每一个月或每两个月会有一个版本，亦或是高频的发布，每半个月就有一个版本？版本的力度有多大？会是一个大版本还是一个小版本？每一个版本之间是否存在大量的差异性，这些都是我们在整个版本发布过程中需要去考虑，需要去告知我们的架构的使用方的一些信息，而这些信息也会在版本运营中去告诉相关方，同时版本运营还要关注我们的架构当前的使用方，在使用我们架构时，他们认知中的架构的版本的分布，比如说到底有多少不同版本的二方包还残留在其他的架构中。


同时我们在研发过程中我们是否存在的并行版本？并行版本有几个？哪些版本是我们要维护的？哪些版本是我们目前正在主力开发的？这些版本中它们的生命周期又是什么样？它们未来的方向又是什么样，这都是我们在版本运营中需要去考虑的。当然还有一些长尾的版本，某些应用或者架构，它集成了我们早期的一个版本，并且在之后它就再也没有升级过了，他们的结合告诉我们他们的架构封板了，他们不再研发了，那这时候怎么办？如果我们希望他们能够做相关的版本清退，那我们又要怎么样去推动它？其实这些就是我们在版本治理中需要去考虑的一个问题，版本是需要有意识的维护与运营的，它可以帮助我们控制我们的架构对外部的影响，因为我们的兄弟团队能感觉到的就是我们在对外发布的这样的一个的又一个的版本，所以我们要正确的认知架构的版本。


当然架构依赖的版本也是我们需要去关注的，其中包括我们自身对于依赖的一个升级，依赖中可能会有差异化的依赖，

比如说像业务依赖也有基础依赖，那业务依赖肯定是随着业务的能力的升级而升级，而基础的依赖则参考其对应的发版而进行相关的升级。但同时这里也有可能是我们根据我们自身的情况选择不升级，这是很常见的一个事情，对吧？我们依赖了它的一个版本，我们觉得它版本够用了，它即使再有很多新的花样，如果我们用不到，那其实没有意义去升级。这是一个很常规的一个升级思路，而且我们也没有意识地去说我们这个东西要升级了，只有当发现了一些问题，我们才会有意识地去做相关的升级。在这里也主要注意就是谨防升级引入兼容问题而道出骨折。我们原来依赖这个版本噢，他告诉我们你们要升级，你们不升级不行了，我们老板们要不支持了，那这时候我们就一定要升级了嘛，对吧？要不然我们就会面临着他们不支持这样的一个悲惨的境地。


那这时候升级的时候发现，诶，新版本和老版本不兼容了，就比如说像 MQTT 阿里云的新版本和阿里云最早的版本其实 API 都扁了，不可能兼容，那在这时候我们就要小心地以去做好相关的升级，升级完之后要做好全面的回归，避免我们发现以为它是兼容的，而实际上它不兼容就会导致了这块功能出现问题。这又起了骨折效应，原来的骨头是连着的，咔，不兼容了就 6 楼走不下去了，骨折了，这个真的是很形象的一种疼痛。


其次还会有涉及到依赖的版本统一，因为我们架构可能会有不同的模块，那不同模块可能都会有相关的依赖，那这时候多个模块依赖了同一个依赖它的版本就可能会存在着散列，有的模块依赖了是它的 1. 0，有的依赖 1. 1 好，新起的模块依赖它 2. 0。这时候后续的随着架构的不断迭代，它的版本的管理就存在的问题，那所以这里在架构的依赖版本治理中，我们需要对其进行统一，避免因为媒体版本差异而导致能力不一致。


我们在新模块开发了之后引入了它 2. 0，然后我们都觉得大家都是这个能力，其实有的模块还有了它 1. 0 的能力， 1. 02. 0 能力发生了变化，那这时候体现在用户那里感受到就是，诶，你们的架构怎么一会儿这是这样，一会儿是那样的？这里不应该是一样的吗？这种就有可能为我们带来用户侧的客诉，或者是在使用上理解不一致而引发的一些故障。


第三句依赖移除就是我，我们有唯一的不想用了，那这里也需要是有很多注意点的。首先移除要灰度，我们觉得，诶好像这个依赖我们没有用到，我们把它移掉，但是也有可能我们框架中有一些历史上的一些代码其实用到了这一块，只是因为最近没有人请求到这一块了，而导致了我们真的把移除了之后就会出现依赖缺失。并且也要考虑到间接的依赖移除也可能导致能力产生偏移，因为比如说我们依赖了两个二方包，它们有间接的依赖，我们做了一下牌包，解决了一下冲突，那这时候它们本身依赖定义上以为的这个能力是依赖它 1. 0，而我们因为解决冲突的时候把它改成了 2. 0，这时候它的能力就可能产生偏移，而这种偏移有的时候是超出了依赖方的预计，也超出了我们的预计的。它就有可能带来一些奇怪的现象，甚至是故障，所以这都是我们在做依赖移除时需要去关注的点。

比如说我们真的一个 1. 5 的一个版本，我们不希望用户再用了，因为它本身其二方包中就存在的一些漏洞，或者存在这些问题，但是这个又是之前花了大精力、大价钱去推广的，那我们这时候也要花大精力、大闸钱去做好相关的清退工作，避免我们这边停止服务了，但是对方却告诉我们我们还在用，你怎么突然没了？这种问题其实是不是我们希望遇到的？所以这里第一要做好广泛的通知，针对使用方，你要告诉他们，我们班们要清退了，你得升级了，在每一天之前你一定要升级，我们最多只支持到哪一天把这些通知要通知到位，并且尽可能的去通知到我们所有的使用方。
比如说无论是私信也好，电话也好，内部邮件也好，通过这种方式让所有的使用方都能够知道，并且明确这个信息。避免说，诶，我们以为发了邮件大家都知道了，实际上有些人压根不看邮件，这些人就没有收到相关的信息，到最后的时候就会发现通知存在死角，通过多种方式来确保不同的人都能够知道相关的信息。其次在清退的时候也要做好电商的灰度，我们先能够针对某一个使用方说，我们把相关的授权相关的知识都移除了，看一看它是不是有问题。通通过这种定向的灰度，我们能够在最小的影响范围以下去明确清退带来的这样的一个损失和成本，最大程度上降低其可能带来的一些损失。然后在单项的灰度验证有效之后，我们再逐步的扩大通知范围，再逐步的 1 + 1 加的做好相关的清退，让他们迁移到最新的版本，和他做好相关的联调，确保其能够正常的使用。



然后它在这整个流程过程中其实是一个很漫长的一个过程，因为你要通知到它，然后你这边要给它提供替代方案，它要根据你的替代方案做好相关的升级，升级完之后确定没有流量了之后再给它切割掉。这里就需要一个流程的一个驱动，比如说它的研发流程中也存在的某些卡点，那我们其实可以在智能中加入到一些卡点，告诉他这个版本必须要清退，否则你不让你发布了。卡发布了这种方式是非常有效的，可以让大家都升上来的。


然后在清退之后我们需要一个静默期，这个静默期是我们本身不再提供相关服务了，但是我们的所有的服务部署，所有能力还活着，还存在。在这种情况下，我们可以防止有一些人说我已经没有应用了，突然有一天他一个应用炸失了，然后又很着急的情况下，这时候就出现了一个不屑故障。那我们有了静默期，就可以保证在这个期间内用户突然间的访问我们也能够正常响应，然后再根据流量的使用情况确保比如说我们在三个月或半年内真的没有流量了，我们就可以把它下线掉了。所以在这里我们也额外的要讲一个点，
