# 1-4 问题发现与异常治理（1041）

hello，慕课网的各位未来架构师们，欢迎回来，在上一小节里，我们针对架构的这个技术栈统一为大家做了一下讲解。技术栈统一不是一个蹴而就的事儿，但它也是一个很重要的一个事儿，因为统一了之后能带来很多的好处，比如说关于基础的积累，比如说关于人员的一个互相学习成本的降低，也比如说关于传承，关于交接，关于人员变化。在这种场景下，我们的一个容灾能力、承受能力更强，避免出现断崖效应，因为某些人的离职而导致了整个架构的垮掉。

但基本上问题发现主要还就是四类，第一类，基于异常的发现。第二类，基于日志的发现。第三类，基于数据的发现，最后基于反馈的发现。那让我们回顾一下这 4 类发现都是什么样子的发现。基于异常很简单，我们的程序爆了异常，我们的线上的巨容架构里爆了一些异常，无论是用户行为导致的一些参数性的异常，抑或是我们本身的代码出了问题，出了一些 NPE class cast 这类很常规的这种异常，那它都是我们线上的一些问题。那针对这类问题是指向非常清晰，非常明确，也非常容易能发现，在发现之后相对来说也相对比较好处理，但它的处理更多的是基于代码层面去做调整这样的一个处理。


而基于日志的发现是要通过日志中的一些特性，然后进行对应的统计分析，这类的可以有助于什么呢？有助于第一类就是它不是所有的链路都在线的，它是主流，中间有断开的链路或者是有离线的部分，那可以通过日志来完成大家的一个串联。第二是针对日志其实是可以能够将很多不属于异常的，但是是属于一些不正常的业务逻辑的，在这一页通过日志的方式能够完成问题的发现。第三类，数据的发现。数据发现的主要是在数据的对账，数据的巡检，通过数据一致性和数据上下游链路的准确性来发现问题，最后反馈码这个就是最常规的了，别人告诉你有问题，那我们就发现了我们的问题，那通过这四类我们就能够快速的从我们的架构中发现是否存在着某些问题，然后根据这些问题去做相关的定位与止写，然后最终修复相关的问题。


但同时这四类也是我们去在架构治理中需要去关注的四点，我们要去关注我们架构中存在的这些问题，并且将这些问题快速的修复，从而清除相关的风险，并且提升架构的质量。所以问题发现更偏重于我们怎么样子去发现问题，然后偏重于安全生产这一环节，但是我们还会有很多问题，它其实是面临着我们要在架构的治理过程中不断的去逐项的提升的。

比如说像基础设施的问题，那就会牵扯到基础设施的升级，其中一项最容易的包括我们的扩容适配，当然也包括一些基础代码的改进，我们是否在最底层的网络层的代码写的有问题，我们在 IO 层的代码写的有问题，我们在这块儿做相关技术代码的改进是否可以极大的程度上提升我们基础设施的能力？这块其实就是在我们在架构治理过程中需要去考虑，需要去落地的点，扩容生沛肯定是最容易的，这相当于是用钱来换技术设施。而基础代码改进就相当于我们在保证当前的基础设施投入的前提下，然后通过我们本身的技术能力来提升相关的基础设施。


所以这一块其实是很多大厂，尤其像阿里这种，它都要求说我今年，在明年我业务增量百分百，但我的基础设施的预算只涨30%，你需要额外的 70% 通过技术的生机来实现。那这都意味着我们在架构升级的时候，也要考虑到基础代码、基础能力的不断的改进，从而来降低成本，并且提高我们的架构质量。


第二类，系统异常问题，这类一问题相对来说比较容易发现，也比较容易定位，那我们要做的就是异常清零，针对于线上的问题，时报时修报出来的问题就要针对性的在下一个点中把它修复，避免线上这些爆出来的异常，可能现在它还不是一个故障，但它未来哪一天可能就演化成了一个故障。而针对一些误报本身就不是问题，但是因为某些策略的原因它被爆出来了，那这类都是干扰，我们要针对物保做相关的消除，避免在后续继续干扰我们。而针对性能问题，我们就要去做好性能优化，比如说像慢SQL、慢代码、慢数据，那这些我们会在后续的性能优化，在性能智力在那个环节中为大家做相关的介绍。


当然最后还有业务的问题，就要做业务的校准，比如说些逻辑审核的存在一些问题，那我们在这里就要需要对整体的业务做校准。第二类就是逻辑上没有问题，但是这个流程比较复杂，所以有人用的就不对，那我们就小学 SOP 化，让大家能够参照一个标准的 SOP 来完成相关的业务执行，从而避免出现相关的问题。那在这样一些整体栏上来看，就是我们要通过这些问题的发现，然后去进行相关的治理。而在治理的过程中也意味着完成了架构的迭代，通过迭代完成相关的治理，并且在通过治理去推动我们的架构能够向更好的方向去迭代。所以架构的健康其实是非常关键的。

上，之前的问题都是我们在去感知架构这样的一个健康，然后从健康角度去不断的推进架构的健康的水平的提升，但同时也要考虑架构的容灾这块儿，就是在架构健康之外，额外的要去为架构赋予的生命力，这样的健康是在常规水平上它能保持的一种状态。而容灾是在非健康结果，它承受着某些不该它承受的压力，它仍然能够表现出良好。其实这对我们的人脸是一个挑战嘛。


我们人常规来说，唉，是一个可以过得很开心，很健康，但是我们真的处于一个非常大的压力之下，或者我们遇到了一些灾难，我们是否仍然可以安全的度过，没有任何损失？自身的自身还成长了，那这就是我们自身要做的容灾，那架构也是同样。当然还有包括架构的可用性，我们要为了追求架构可用性，是我们更向用户的一个承诺，我们告诉他，我们可以承诺你我们架构的可用性，但通过这个承诺反向的逼迫我们自身对于架构不断的提升，那无论在这前面三项，我们目的都是为了让人员能够成长，让我们的团队同学能够成长，让我们自身能够成长。因为只有人成长了，前面的三项才有落地的可能性。如果人没有成长，不停地去机械性的去做前面三项，也最终没有办法为架构带来质的提升。


所以在架构治理的过程中，我们要做好问题的发现，从发现了问题之后明确相关的监控报警，然后去做好快速的恢复，并围绕着这类恢复的手法做相关的架构预案，通过预案的不断地明确，并且围绕着故障演练去验证我们的预案是否能够真正的解决问题，并且在故障演练中也继续去发现那些我们在线上正常运行中没有发现的问题，发现一些在极端条件下，在一些灾难发生的时候，我们是否能够快速地感知，然后快速地触达报警？然后快速地通过预案来完成恢复，然后再在后续去做更进一步的迭代与升级，那这都是我们在整体的问题发现与异常治理过程中需要去感知的。


我们在故障演练中去发现我们系统是不是还爆出了更多的异常，那这些异常我们要针对性地去做修复，做相关的预案去预防，真正当灾难来临的时候，我们是否真的能从灾难中逃出生天。
