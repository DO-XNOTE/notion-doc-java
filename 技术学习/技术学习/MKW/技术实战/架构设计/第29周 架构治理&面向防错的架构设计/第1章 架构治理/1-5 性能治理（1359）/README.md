# 1-5 性能治理（1359）

hello，慕课网的各位未来架构师们，欢迎回来，在上一小节里，我们为大家介绍了问题发现与异常治理，围绕着安全生产的介绍的那些点，我们进一步的展开了一下。在安居生产那个大篇章中为大家介绍的问题发现是将整体的从线上的这种稳定性保障到整体的架构升级的，围绕整个思路的这样的一个问题发现。那到了我们这一篇章，我们的问题发现则在这上延展出来，我们怎么样围绕着发现出来这些问题去做对应的架构治理和迭代，从而能让我们的架构在不断的异常治理的状态下，达到其想要的这种可用性，以及架构健康与容灾这样的一个特点。那么在这一小节里，我们就将进入架构治理中的专题治理的第一环，性能治理。性能其实是我们架构非常关键的一个点，


但是我们通常也有另一种说法叫做我们要先完成功能，再考虑性能。为什么呢？

因为你即使在开始考虑了很多性能，后面也有可能随着功能的迭代，你的性能还是会下降，你还是要考虑重新去优化你的性能，并且过于的执着于性能，也有可能会导致整个设计及扩展性、兼容性都存在问题，而导致后续的功能并不那么容易完成。


还有一点就是功能才是最重要的，哪怕你性能不达标，你功能完了，你可以先用你的性能慢慢优化，你开始就那个非常牛逼的性能，功能缺失这东西不及格，所以就是新完成功能在考虑性能，但是性能不是不要考虑的，有意识的在设计的时候就考虑性能，可以大大的降低后续的改进成本。


如果在开始的完全没有考虑，代码都是堆在那里，那你在后面再去完善的时候，你就要有很大的重构工作量和很大的性能改造的一个工作量。但是在开始设计的时候，尽可能的在完成功能的同时把性能提升上来，然后在后续专项去做性能治理的时候，它的成本就可以大大的降低。


但是针对于不同的流量，其实性能的体现也是差异化的，比如说在 d 流量下，这性能的问题压根就不凸显。你想如果你的一个系统一天的 QPS 就几十分到小时都分不够，那你分到秒是更不用说了，在这种情况下这个性的问题完全不 care 你，只要你能在两三秒内把这事儿完成了，用户感觉能用就够了。


但是在高流调下它就会出现木桶理论，任何一点的性能问题都可能会导致你整体的不可用，那整个系统的性能就由木桶的最短的那根板决定。所以这时候也意味着我们要针对这个架构的性能治理，就是找到我们整个架构这个木桶中短的那几根板，然后把这几根板给补上，这样子我们的木桶就能装更多的水了。

首先第一个要考虑就是说我们怎么样去解决这个流量的问题，因为流量可能会很高，或者流量可能会不均衡，那在这种情况下怎么样子去解决它？第二就是围绕着热点问题，因为热点就是性能的杀手，凡是你的架构中存在热点，无论是数据热点或是代码热点，都有可能在这个热点上你就是栽跟头，所以解决热点问题就解决性能问题的一大半，这一点真的不是瞎说的。


第三个就是数据的问题，因为你在高流量下，你的数据也是要考虑很多的，不是说我只考虑我的架构，这可能我的代码，我还要考虑我的数据分配，这个其实跟热点一个是结合的，也要考虑我们在进行数据设计的时候，我们的缓存的 k 怎么设计是吧？我们的数据库的表存储怎么设计？因为数据也会导致整体性能出现问题，所以在这个点上也绝对不能放松。


当然最后还有一点是往往是容易忽视的，就是日志打印，不是所谓的日志打的越多越好，你讲所有的日志数武器打印出来，它也是要消耗性能的，而且尤其是在并发很高的情况下， IO 是很大的一个瓶颈。那怎么样子在日志打印这块儿做一下细节的优化，就去可以极大程度上提高我们的系统的性能，这个点是小成本而高收益的。


好，那我们展开来去看，面对流量问题，最常规的就是缓存的建设嘛，这个在各位架构师给大家在架构设计的时候，其实我讲这个点已经讲得很通透了，我这里就不再赘述。但是缓存建设绝对是去扛流量的最关键的一个点，但是还有第二点就是缓存预热，你建立了缓存，但有可能在建缓存的那一瞬间，你的系存就被打爆了，尤其是在面向于这种突然就汹涌而来的这种大流量的情况下。但也意味着什么呢？意味着我们在面对这些流量之前，要去做好缓存的预热，将我们的缓存中把这些该有的 k 都给它建起来，该刷新的中超时间的刷新。好，这样子当流量来的时候，我们万事俱备，就等它流量来了。


第三就是针对流量做整形，因为流量本身可能不均衡，上一秒还特别多，下一秒就成第一步了，那在这种情况下，上一秒对系统压力很大，那下一秒明明还有富裕的性能，但是却没有流量进来了，等在这时候做流量做一个整形，把峰值削填到谷里去。这样子的话在高峰的时候我们还是能够可以扛一扛的，在低谷的时候我们也不浪费我们的资源。


最后还有就是异步化，很多事儿我们不是需要非得需要同步解决，我们完全可以把它异步化，通过异步来解决同步代码或者同步请求段内所必须要执行的任务数，而将一些可以延后的、推迟的不那么着急的事儿都放在异步化，用异步的一个代码去处理，放在一些用现成的或者用异步框架调度，让这些事儿最终能够完成就可以了。
而面向于热点问题，主要还是热点数据和热点代码，针对热点数据要做热点分散，不要让所有的请求都打到了某一张表的某一行字段上去，大家都去改，那就很容易去排队去扎堆了。不和质检代码也是一样，线程池可能很快就满了，那我们将机器可能分散，这样的话可以最大程度上保障我们的系统，我们的架构在热点流量的时候不出问题。


第三，数据上就是刚才提到的大key，我们有一个很大的缓存的key，不管它是特别热的 key 还是特别大的key，你一个缓存下存了一个key，它肯定是不合理的。之这种情况下就要根据场景去做拆分，然后根据这个冷热的这个缓存去做对应的治理，把不该长期缓存的内容到时间就给它清掉，这也意味着我们要做好长短缓存，对于一些更新比较不那么频繁的，我们就可以用长期的缓存，对那些更新比较密集的这种数据，那我们就要做短缓存或者做实时更新的缓存。既数据更新了，也有个任务可以把缓存给更新掉。


通过这种方式来保证我们的 case 设计上是更合理的，并且在表上要有做查供词，大表一定要拆，确保我们的表不要突破 MySQL 的本身这个性能的瓶颈。因为 MySQL 来说 q 表能过百万，它的性能就开始下降了，所以尽可能的将表作分散做分控，分表也是为了让我们每一个表的性能能够在其最优的这样的一个位置上。


还有我们的性能杀手慢 circle 一定要发现一条解决一条，但是注意不要在治理 MySQL 的时候不小心引入故障这一个点，可是一定要记住在上面栽更多的人可不是一个两个了。而在日志打印上打印性能这块还是有很多地方可以挖掘的，对于线上运行的环境来说，有一些不该打的认知，那就不要打了骂，把关键的日志打出来，能够帮助我们做相关的分析结构了。而对于线上出现问题，可以再通过流量回放再去看一些日志，然后去发现对应的一些问题，这样的话其实是最好的了，因为现场那些打印的那些日志本身来说，你也不会去现场一直的去盯着看，因为它滚动的每天还是很多的，并且我们也可以做一些日志异步化，比如像 log BUG 2 它就可以去做这样的日志异步打印。本身它是一个环境队列，也不担心它有存在线程池的问题，虽然说可能在一定程度上会丢一点日志，但是日志丢了本身来说问题并不大。

我们肯定有一些事儿是在流量到来之前就去做的这种准备工作。还有另一类就是在流量来的时候我们要去做的流量约束。三，热点数据缓存键、缓存预热，这些都是在流量来之前的一些准备工作，我们要在流量来之前把热点数据都分析好，该分散的都要做好分散，而这对缓存来说，做好超时管理，哪些该长缓存的就好好把它的缓存时间扩大。而哪些该要注意去更新的，就要及时的做好缓存的刷新，长端缓存以及实时缓存的这种建设。


而针对链路预热来说，就需要在流量来临之前做好整个链路的这样的一个缓存预热，把链路的缓存建立起来，然后在缓存建设的时候，包括长段缓存和缓存预热的时候，把这个超时间给它好好的刷新一遍，不要在流量高峰的时候抢着缓存过期了，那这时候重建缓存的这个时间就可能把我们的系统击穿，这就是缓存击穿。
而在流量约束中，主要还是在流量整形和流量控制，流量整形是哪些？将令牌桶这种通过持有一个令牌的方式来将流量进行缺峰，而流量控制最典型的就是限流了，而流量整形中通过将流量进行缓存下来，然后在后续再逐步的释放整治消费，也能达到流量者进行这样的一个效果。


而针对流量控制熔断也是一种对于我们系统做保护的这样的一种行为，那么这样子我们就从准备工作到流量约束执行来完成了我们整体的一个流量控制的这样的一个选型与技术实施。从流量约束到流量控制的这样的整个的一个过程，面向于架构中可能存在的这种问题做相关的治理。那老师这里提供的这些都是一些常规的一些方式，当然这些方式平时也是非常好用的，各位同学，各位架构师也可以考虑根据自身的情况去额外的增加一些内容，或者做一些更细的选型，比如像限流可以有对应的一些开源的组件的选型，然后比如说像预热这些是不是我们有更好的预热工具？甚至说可以考虑用像全链路压测这种方式，跑一个小并发的压测，把唇全都建立起来。

除了刚才提到可以在一定程度上帮助我们去预热缓存之外，它还能有什么作用？它还可以帮助我们去发现木头最短板，因为我们可以自己去制造这个流量，去探索系统到底哪里还不行，探索系统在极限压力情况下到底哪里表现的最不好，哪里开始报错，从而去发现目录的最短板，然后针对这个短板进行专项提升，然后提升之后重新再进行一次压测，这样子不断的迭代，不断地将木桶中最大板发现出来，一点点的往上修。


到最后我们就可以达到我们的一个预期，所以每一个性能问题都是大问题，我们要用小步法高频次这样的一个迭代速度去完成我们整体的一个性能问题的发现与提升。所以压测可以帮助我们发现系统瓶颈，以及去探索极限下的系统的各个细节表现，然后能够帮助我们明确我们性能改进收益的最大方向，这个就是我们不同的最短板，然后进行架构治理的实施改进，并且验证其结果。


怎么验证，继续通过压测去验证，这样子不断地反复迭代，当我们真正迎来线上流量的时候，我们就可以做到胸有成竹，我们可以知道这个流量大概是多大，那到我这，我每一个应用会处于什么样的一个状态？当然压测也需要考虑多样化场景，因为现实中的场景可能比压测要复杂得多，所以在压测所能达到的极限水平下，往下调一个百分比去设置我们的安全水位，做好系统的保护，也是我们保证系统的性能的网络好到这里我们这一小节就结束，这一小节就针对性能这个专项会给大家做了一个介绍，性能我们主要考虑它是在高流量下才会去重点关注的一个点，那我们就要考虑流量热点以及在这两个之下我们到底要怎么样去做。
