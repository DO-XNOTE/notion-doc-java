hello，慕课网的各位未来架构师们，欢迎回来，在上一小节里，我们为大家讲解了一下关于架构数字化相关的事情。数，数字化和的度量体系是一个包含的一个关系，度量可以做是数字化的一部分，但是数字化的概念要比度量要大得很多，它其实更多的不在于最终的效果指标，也不在于过程中是否我们有一些指标去监督我们的过程。它更考虑的是将所有的行为线上化，将以人为文，考虑所有的信息的串联，信息的链接，那这些是我们在数字化中需要考虑的一个点，那也意味着我们要在我们的架构中去考虑是否还存在着数字化程度不够的一些点，这这是我们需要去提升的一部分，这也是在架构治理中面向于数字化这个点，我们需要去治理的那个环节。
那么在这一小节课中，我们将围绕着技术栈这个话题去聊一聊。通常一个团队或者一个架构的技术栈通常由两个人决定，一个是技术的leader，另外一个就是战国师。因为技术 leader 他通常代表着他能够有很大程度上的一个话语权。而架构师因为他本身是架构的一个设计师，那他也会有比较强的一个话语权。那在这二位的合作之下，一个架构的基础站会怎么样子决定？大部分情况下会在公司的基础约定之上，

再结合着这二位的擅长的部分，然后定出了对应的选型。比如说这两位它就会勾，它别的都不会，那基本上这个选型也就是围绕着 go 的这个生态体系去落地的，不可能或说两二位都不懂Java，然后他们又选了一个 Java 的这样的一个选型，除非是在某一些条件下被迫做的这样的选择，否则的话大概率不会这么来玩儿。


所以从选型上来看，这一块儿就会出现了一个技术万花筒，每个人看到的可能都不一样，甚至有一点变化了之后，整个的一个视角可能都会发生比较大的一个变化，所以也就意味着由人来决定了最终的选型，就可能会导致在这个万花筒中他们只看到了其中的一面。并且这个万花筒在整个团队上来看，每一个擅长某一个领域的基础 leader 的架构是决定了万货虫的一个方向，那随着万货宠的转动，这是整公司的这个运转，看到的技术选型可能就会五花八门了，那这种情况下技术选型有的时候就会成了一个护城河，护城河护的是谁？一方面护的是城内的人，那另一方面他也可能护的是城外的人，假如城内都是猛兽，是不是城外反而是安全之地了？所以这里也看这个护城河也有两面性，但是它也会导致什么呢？导致第一，内部割据，因为不同的人获得领域不一样，那所以就会产生会某一些领域的人，他们之间形成了一个割据，而不会这个领域的人就会选择以他们的擅长的部分去形成他们的一个领域割据，在这种情况下外部是难以进入之内的，所以呼声格就做了一个内部的一个保护。



然而除了这个之外，它也可能会带来什么？带来从整个层面上的一技术混乱。因为各自为政了，每一个城都用护士和保护起来自己，并且成与成之间的合作阻力也会变大。因为你用的是go，我用的是Java，如果都用一些通用的一些协议去沟通，可能还好一点，但有的时候就会出现二者的一些不协调，或者因为某些原因需要去某一方做出妥协，那这种情况下合作阻力自然就大了。所以不统一的技术栈会带来什么样的一个问题？

第一，这个学习成本高，第二个学曲曲线相对陡峭。有人会讲每个人都用自己擅长的基础里，为什么还说学习成本高？因为这个学习成本不在于当前的学习成本，而在于从整个公司层面上的一个学习成本。大家为了能够合作，为了能够协同，公司层面上多个团队之间的合作都会导致你需要额外去学很多的东西，那这时候这个学习成本自然而然就水涨船高了。


而在这种情况下，整个的基础设施的距离也是不足的，因为每一个人他们都用自己的方式去做了对应的基础设施，我们团队用go，我有一套 go 的基础设施，你们团队用Java，有一套 Java 的基础设施，他们团队用的 C + +，有的 C + + 的一套基础设施。


在公司真的成了一个巨头的情况下，比如说你像阿里巴巴，像腾讯，你想他们有近 10 万人，那他们有足够的人力去打造每一个技术栈的一个基础设施。但是作为一个中小型公司，比如说一个创业公司，它可能整个公司也就在 100 人，那在这 100 人情况下，再有大量的差异化的技术栈，就会导致技术设施根本就没有足够的人力投入进去打造一套大家都能用的基础设施，因为差异化太大了，这时候的距离就会产生不足了。


第三就会引入大量的重复性的建设，而这些重复性的建设就是我做了一个基础的日志类，你也做了一个基础日志类，二者因为不同的语言，它一定有了大量的差异化，这个日志打上去的，上层想要统一去看的时候又发现因为二者打的一些方式的差异，又会出现了大量的改动和定制，那这些都会意味着大量重复性的建设，最终还会引起评价标准的不同意。


到底我写的这个东西和你写的东西两套基础体系，两套选型，哪套更好？大家怎么样去评价呢？很难去一碗水端平，很难去一刀切，那这些都是技术栈不同意所带来的问题，在这种情况下，这时候大家的相互沉淀其实是没有办法形成距离，而传承也相对单薄，比如说整个团公司就两个人会有够这两个离职了，那他做的这些事怎么样子传承下去？他的交接倒是问题，这也意味着这时候就带来了断崖效应，两个人一走完了，这时候出现了断崖了，这就意味着我们要在一定程度上尽量的让我们的技术栈从多样化变变成一个统一化，尤其在一个创业公司或者在一个中小型团队，

第二，改造成本。之前技术栈已经差异化了，那不同的模块可能已经使用了不同的技术去落，那这时候大家统一之后也意味着会有这样的改造成本。第三，沉默成本。有些东西已经真的你投入了精力，投入了思想去做，但是这时候因为你换了技术栈，那之前的那些心血都白费了，它整个的东西都扔掉了，这就是一个沉默成本。还有最后的就是管理成本，我们在做整个的一个事情，推动整个的这样的一个统一，这样的一个大事上以及向上沟通，让老板能够知道我们做的是价值的意义，以及向下的一个协同，让大家能够在某一个时间点去完成整体的统一，这都是相关的管理的成本。那但是这些成本它真的是我们值得去投入的吗？这里其实是要根据公司当前的情况，在公司越早期进行投入，成本越低，收益越大。在公司越月晚季，我们就要考虑将这些成本尽可能地平衡其收益与成本，如果能够平衡，甚至收益大于成本的时候，我们这事就能干，如果收益小于成本或者成本太高了。


比如说之前就有个很牛逼的算法，这个算法是一个大牛来做，大牛已经走了，我们已经没有任何一个人能够把它改造成另一种语言或者另一个框架了，那这里的沉默成本就会非常的高，这都是我们需要考虑的点。当然我们还要更细的去看，比如说沟通成本，也包括团队内和团队之间的这样的一个沟通成本，因为大家都要统一，而改造成本包括现有系统的改造成本和基础代码的一个改造成本，业务系统通常来说工作量会更大，而基础代码的改造成问题在哪里？是在于它的切扯面很广，它可能会牵一发而动全身，那所以这里都是要非常小心。


就比如说大家都用的是 1 j b k，但是其中有一个模块用的是 UTM 8，贸贸然从 UTF 8 改成GPT，就可能会导致之前数据的乱码，这些都是基础性元素的改造，它的统一的成本其实远比我们把它改怪的相比要高得多，因为它可能带来的辐射性的影响是比其本身的改造成本更大的。


而对象与管理成本，其包括推进技术上统一落地的这样的一个推进的成本，还有我们的技术栈也要迭代之前的那些我们废弃掉的技术栈，那肯定有可能不管它了。但我们统一的技术栈我们要有个清晰的迭代路线，到底怎么样子，和行业内的结合，和公司内的结合，再和团队内的相关结合，从而完成整个竞争的迭代。


而面向于沉默的就是其他的框架和代码的一个投入，已经投入的精力还有利益里面非常要考虑的就是人，因为你要转型，比如说团队之前有很多写 C + +，然后现在大家都转渣了，不是所有的人都能够成功转型，其中一定会有人会转型失败，那这些转型失败的人他们在这里就没有存在的空间量，那所以怎么办？他就流失了，这种情况下我们就必然会承受着这样的人员流失这个成本，而这个成本其实是沉默性的，我们只能重新去招啊，原来这个人他真的走了，就再也回不来了。

可能短期成本高，收益小，但长期来看收益很大，那这事就值得去做。但这车可能就变成了一个重要不紧急的，跟随着项目的发展，跟随业务发展不断的去完成就好，而是可能短期的成收益特别的高，但长期来看它并没有很多的收益，那这时候也可能是打一耙子就撤了。因为我们把短期收益拿到了之后，它既然没有长期的收益，那么其实就没有长期做的意义。


但是其实也有一些特殊的要有考虑的，比如说公司的现状，我们公司的发展的情况是什么样子？公司现有多少人？你真的像阿里这种公司，它基本上不太可能把所有的基础上统一，但是它可以把一些核心系统，把某一个领域内的系统统一，比如说电商系统，它可以统一都用Java，阿里云相关的可能会有一些用Java，但是底层的面向云原生的当就要与行业接轨，那肯定就是go。然后一些更底层的一些基础的数据库操作系统，那一定是 ZA 或 C + +，它不可能用 Java 去做操作系统，那这种都是行业性的一些现状，那这时候就意味着行业技术栈也会决定我们到底使用哪种技术栈，当然也会有技术栈优势，比如说这个东西就是用 Java 特别好，那你非得用 go 其实是强扭的瓜不甜呐。


我们还是想要吃甜瓜，而不是非得要吃一个葫芦种的瓜，当然也决定了团队的水平与成长，现在团队人良莠不齐，做家暴的人就是一些人学不会，那他们就是可能就需要保持他们在当前的这个技术战场去继续去做，然后直到完成人员的汰换。而另一方面就是你在换技术战的时候，也可能会有人跟不上这个节奏，它会掉队，它会离开，这都是我们需要考虑到整个技术上统一中的一些元素和一些特殊的点。


好了，到这里为止，关于技术栈统一的事儿就讲完了。技术栈统一其实是一个既可能是很大，有可能是很小的事儿，因为说它大，因为它的影响面很大，说它小，它又是我们在工作中很小的一个环，因为可能如果当曾旭在初创公司，可能一下子大家都同意了，这是一个约定的一个事。但是不管历史怎么样，我们可以让未来尽可能统一，让未来大家的劲儿能使到一块儿去，这也是我们在架构治理中，在架构中去查看我们那些五花八门儿的技术之后，去考虑未来发展中非常关键的一点。好到这里这一小节就结束了，在下小节中我们将去聊一聊问题发现与异常处理。这一节其实跟安全生产真的是太像，所以我们的内容和安全政治社会有大量的雷同。但是大家也可以将思维从安全生产那种以稳定性为第一要素的转变为以架构本身能力为第一要素的视角，去思考这个问题，可能会有不一样的收获。好同学们，我们下一节课不见不散。
