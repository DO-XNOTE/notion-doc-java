# 1-9 环境治理（1101）

hello，慕课网的各位未来加工师们，欢迎回来。在上一节课里，我们为大家讲解了一下关于链路治理相关的事情。链路需要先去感知明确入我们链路当前的状态以及存在的问题，然后再针对其问题做专项的修复。链路的问题无非在于链路的执行过程中它的是否够足够平滑，这才是在整体的链路从整体上去看的一些事情。然后我们的子调用的情况，这是针对存在问题的链路，我们要去看其子调用是否存在问题，是否存在性能问题，存在成功率的问题。然后再围绕着这些去看我们单一链路它的成功率上，体现到子链路体现到中间件上之后，是否存在着因为局部的长 RT 而导致整体链路的超时而降低其成功率这样的一个问题，通过这三个纬誉度去明确出我们在链路治理中可能会存在的问题，并针对性地进行提高和治理。这就是我们上节课为大家讲解的内容。

如果我们可以把我们项目可以分为人或场的话，那环境基本上就是我们所谓的场。人通常都是你我他。我们的资深，我们的研发同学，我们的运营同学或通常就是我们的架构，那场就是我们的架构所运行在其上，这个载体，在这个载体之上，它也是需要相关的智力与健康保障的，只有载体健康了，我们的架构才能够活得更随意、更轻松。


那环境都有哪些？第一就是机器环境，我们线上的机器和我们测试环境，这些机器是否稳定，机器中基础的能力是否稳定，那机器环境之上具有我们的开发环境，这是我们干活儿的地方，是否能够为我们提供高效的支撑，这是我们的效率的保障。还有就是应用的环境，那就是我们线上真正执行的所处于的，它具备应用之间调用上下文的这个环境。那围绕着这些我们就要明确出我们环境中的问题，并进行针对性的治理，包括环境的使用率、环境所处于的压力、环境当前的成本以及环境的合理性，这些都是我们在环境治理中所需要关注的这些主体的点，那在做治理之前我们肯定要做好环境的分析。

它是属于一个偏闲置的状态，还是有一个偏过载的状态？偏闲置意味着资源浪费，偏过载意味着压力过大，一旦有点风吹草动，可能它就崩掉了，那这种情况也是不允许的，所以宁可闲也不要过载，但是闲也要有度，因为它是钱呐。


第二就是环境的敏感性分析，我要明确在整体流量压力变化下，环境会怎么变，比如说流量增高了，那环境的压力会怎么变化？当流量增了一倍，环境的压力的，环境的load、 CPU 内存是不是也是同等增幅？还是说它增加的更多还是更少？通过这样的方式我们就能够明确出我们去增加机器，它给我们带来的边界效应是什么样子的。当流量在不断的上涨的时候，我们是否需要增加机器以及增加机器，它是能够为我们带来的压力下降的一个幅度都是可以通过敏感性分析来获取的。


第三就是弹性收益，因为我们的业务很难说有一个业务它从早到晚到夜里，甚至一年 365 天完全是一条线，它一定是有高峰有低谷，那这时候我们就要明确出我们的低谷我们需要多少资源，我们的高峰我们需要多少资源，然后去明确出在高峰的时候我们的成本是多少？在低谷的时候成本是多少，然后是否是想我们可以去预测我们的低谷，我们也可以去预测我们的高峰，然后针对预测的结果进行动态的资源调配，在高峰的时候我们就砸机器往里砸资源，尽可能的保证我们的业务平稳，在低谷的时候我们就缩容降配来节约成本，说不定一年就能省出个几个亿。


那这些都用在我们的新业务拓展上，它不香吗？最后一点就是环境的统一，因为我们可能线上有各式各样的环境，每一个应用它所去搭建的这种基础环境可能得也会有差异，但是只要它是标准的环境，少量的差异问题并不大。就怕是有些非标的环境，比如说常规的环境都是容器化的，但是有几个就是直接在裸机上过了Tomcat，那这些非标的环境意味着在后续管理的时候它都是额外的成本，我们需要小心谨慎地进行相关管理，避免因为我们以为它是镜像，但实际上它不是。镜像。上去运维的时候发现一些工具不生效了，我们动态缩空容不可能。那在这种情况下，我们就是意味着需要针对这些非标环境进行相关的统计分析，然后来明确是否进行环境迁移。而在具体的环境里，我们也要考虑有一个点，就是怎么样能够让我们的环境更稳定、更复用？答案就是镜像。

使用镜像能够将我们的基础系统、常规的依赖软件和部署结构进行约定，并且让每一个镜像都是使用同一套的这些基础依赖和部署结构，这样子就意味着在整个过程里我们可以有一个可复用的这样的一个结构，并且将这些结构完整的聚合起来，运用到差异化的不同的应用里去。这样子的话我们的管理成本就会下降，因为环境是可复用的，同时它的稳定性也在上升。因为同一个结构适用于多个应用，那这个结构一定是我们经过了多次验证得出来的结论，其稳定性一定会比那些一次性临时堆建出来的环境更稳定。所以有镜像驱动部署可以便于我们把整个的基础环境进行统一，统一了之后它的管理成本自然就下降下来了。


当我们需要管理一个环境和我们需要管理 50 个环境的时候，一个环境把它做好的成本一定远低于 50 个环境，都要做好当，所以当你环境多了的时候，这里的管理成本一定是超出线性增长的。那么针对这些我们就要做好整体的环境治理确保。

第一，环境是稳定的，我们要打造一个相对稳定的镜像，这个镜像经历了我们线上多轮压力的一个确认，确认它能够支撑出我们多样化的线上环境。其次它要具备一定的自制能力，出了一点小问题，它能够自己去容灾，能够自己报警，然后做流量的保护，做逃逸。比如说基于云原生的，它就可以基于一些服务治理相关的能力来保证我们的环境其具备一定的自制能力。第三，要有做差异化管理，虽然说我们可以用一套统一的镜像，但是上册的应用依赖的相关内容一定也是差异化的，怎么样做得到差异化管理？怎么好把握差异化这个度其实是我们在环境管理中非常重要的一点，过于的统一化可能会意味着很多环境中存在一些用不到的这种软依赖或者软件。那我们做好差异化，明确出不同的诉求下环境应该怎么样的管理，甚至说沉淀出一些套餐，让大家使用不同的套餐，不同的镜像，来确保我们的差异化可以得到满足，同时管理成本在不断的下降。
