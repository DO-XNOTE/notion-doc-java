# 3-10 防错思维与故障预防（0925）

hello，慕课网的各位未来架构师们，欢迎回来。在上一小节里，我讲解了一下关于故障案例，以及从案例中如何去复盘，以及和防错的几大规则进行关联这样的一个事儿。在这一小节里，我们就进入了防错设计的最后一节，就是防错思维与故障预防。无论我们之前讲了多少案例，其实最终来说我们都还是要落回到我们的架构中，还需要围绕着这些思维，围绕着这些点，围绕着这些前人已经研究明白的这些理论，把它和我们的架构融合在一起，形成实际的生产力，它才能发挥起真正的价值。所以也就是这一节我们希望和大家一起去聊的相关的内容，

也是我们作为的架构师，我们也都是人，我们所面临的所有的客户，他们也是人犯错。人犯错这个点更多的还是在于人，因为机器它不会犯错，它我只会按照我们的指令去一点点去执行。但是人他是会犯错误的，原因有很多，有主观的，也有客观能力。


正如我们在之前讲犯错误的十宗罪的时候，其实上面的五宗罪基本上就是偏主观的，包括粗心，包括焦虑、烦躁，这些都属于是主观的一些原因。当然还有客观的原因，比如说像经验的缺失，那这些处于是一个我们在没有达到所要求的这种经验的情况下，我们客观的一种状态。


所以我们要认识到人其实是构成整个世界的基础，当然人的多样化也是构成世界多样化的一个基础，但它同时也是导致人行为的不确定性的最主要的一个原因。每个人都是一个独立的个体，每个人都有着自己的一套思维，自己所处于这个文化的一个范围，自己的习惯，自己的认知，而这些都能导致了你的行为是由你的各种条件、各种背景已经以当前所处于的环境所决定的。并且信息传递和信息理解的偏差也是客观存在的，你不可能说一定能够做到信息百分百的有效传递。


每个人对于一个同一个东西的理解也是不一样的，我对于一个点理解和你对于一个点的理解本身可能就会存在的差异，哪怕我们孩子坐在一起去沟通，都有可能有产生相关的理解偏差，更何况是通过系统这样的一个中间产物来完成双向的沟通。针对于更多的点，像明天和意外，其实我们真的不知道什么时候一定会先来，也不知道哪个会先来。意外有很多种，黑天鹅也是种意外，就比如说一个公司，它们通过两条光纤来保障公司的一个网络的可用性，通常来说一根挖断的概率就非常小了，结果当它两条光线同时被挖断的时候，这种真的可只能说是黑天鹅了，因为它的概率实在是太小了。但是它的确发生了。在这种情况下，我们其实不知道哪种会先来，也许某一天艺人的一些事件就会导致流量的激增，种种的客观的我们没有办法控制的现象，都是我们需要提前去考虑，在我们的架构中做好相关的一点一滴，从而让我们能够扛得住这最终的这些片场。所以彻底防做是可能的吗？



它是不是一个可以实现的？照滚，即使日本说我们零缺陷可以达成，美国也有零缺陷管理，但是零缺陷真的达成了吗？

我想也并不是，日本也有车，大量的车辆召回美国也是。所以彻底防错其实是不可能，也是不现实的，不太可能说是把所有的东西都做到彻底，做到百分百，因为我们还有人的元因素在这边，我们还有人的多样性，我们还有客观的一些原因。所以世界的多样化与人的行为的不可预测性都决定了我们没有办法去控制所有一切，但是我们可以去努力做好，去减少犯错的机会，去增加犯错的成本，同时让犯错有兜底。通过这些让我们尽量是不出事儿和犯错是两个概念，我们允许犯错，但是我们尽量让错误最终不演化成故障，这里就是我们可以去做到的，也是我们一直努力去做的一点。减少犯错的机会，那它就不会有那么多的机会去犯错这个错误。增加犯错的成本，也意味着在他拘泥的这个过程里，他会选择低成本的那条路，那他大概率就不会去犯这个错误了，这也是一个变相的降低犯错概率的一个点。同时犯错有兜底，即使犯了错误，也可以通过兜底这个方式来让这个错误大事化小，小事化了，最终没有产生实际上的损失和伤害，这一点其实是我们所希望的。

关键功能肯定是关键点。其次用户的关键行为，比如说下单，这就是一种关键行为，你像其中有一些哪些不是关键行为？查看库存，如果这个商品有库存，哪怕库存这个接口出现问题了，降级掉了，看不到这个库存，其实并不影响他下单，但是像下单支付这些关键性的行为是一定不能够损失的。第二就是我们要明确我们的行为预期，包括正常的行为预期和行为偏离的可能性，我们会对一个用户他使怎么使用？我们的架构有一个预期，当我们的功能支撑它，预期行为之外的所有的行为都可能会导致偏离，那这些偏离的可能性就有可能会引发某些错误性的行为，而这些错误型行为可能继续演化就形成故障。当然这些都会由我们的结果预期所去影响。


当我们的正常结果预期用户达成这样的预期的时候，也就认为用户是正常使用的这个系统。但是也有可能用户不是按他的预期去完成这个结果的，他拿的是另一个结果，这时候我们就需要有一个最差的后果估计，因为最差的后果估计是用户他经过各种各样的使用，最差的情况下能够达到这样的一个效果，那在这种情况下，我们就要根据这个最差后果反向去推我们是否能接受这个最差后果。


如果能接受，那OK，如果不能接受，我们怎么样子消除这个后果，让用户在他的行为路径上，在他的关键点上不具备演化成最差结果的这样的一个可能性。所以总体上来看就是我们要通过我们的防错原则，在我们的体验交互和行为预设这两个点上共同推进架构改进。

这时候我们要基于我们线上发现的这些问题和故障，以及包括从他人那边学到的这些问题和故障，促进我们的架构改进，并且由它来指引我们的架构设计。我们要建设好基于问题和故障的逃避能力，这些逃逸能力可以在一定程度上帮助我们从故障系统逃逸来做到，即使出了问题，但我们不出事。将这个逃逸能力涉及到我们的架构中去，并且围绕着行为预设做好行为分析，明确出用户的最差后果，在这个最差后果上去评估我们是否能够接受，不能接受怎么样通过架构设计来改变用户的行为路径，从而调整用户的行为预期以及用户的行为偏离，最终让用户的行为在我们的把控范围之内。


并且通过架构设计也进一步的提升体验交互，让整个的体验如斯般适用化，同时用户又感觉到非常的愉快，并且他们不会因为体验交互中存在的问题而陷入到一些不知所去，不知所踪，在这种他们没有经验的前提下，仍然能够更好地使用我们的产品。防作设计整个是一个很大的一个概念，但更多的我们是要考虑怎么样子从错误中去学习，然后一点一点地围绕着我们架构所提供出来的能力去设计我们的行为预设，设计我们用户的期望结果，在围绕着防错的这样原则是否能够打造成一个尽量让用户不犯错的系统，并且在错误无法避免的时候，我们有缓和，可以逃逸，能够尽可能的降低我们的损失。
