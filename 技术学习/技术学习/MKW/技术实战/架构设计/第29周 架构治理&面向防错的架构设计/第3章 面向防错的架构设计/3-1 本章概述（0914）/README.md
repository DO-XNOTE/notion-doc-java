在世界一流顶级架构师茶话会的第二季里面，我们重点讲的是哪三个点？联调，卡点，还有中台。其实这些都是以围绕着人，以人出发，以人以为本去探讨在合作中，在可复用的这种资源中，我们如何去协调，如何去构建，最终达到一个提升质量的同时，也保证我们本身的架构的合理性，以及我们最大程度上来复用我们人的这种资源。那么在这一小节里，我们就要开启一个新的篇章了，在这个新的篇章里，我们叫做面向防错的架构设计。防错是什么？我们防的又是哪一类错呢？人非圣贤，孰能无过？那每个人都有可能会犯错误，那就让我们一起探讨一下，在架构设计中我们又能做什么？怎么做？从而让我们能够更好地来降低用户可能犯错误的机会，从而降低我们所面临的损失。

我想 P1 对于大家来说都是一个很熟悉的一个级别，通常来说它就是指在一个企业中最高级别的一个故障了。那我们来看一下这个故障啊。线上支付吊起失败，支付成功率下跌70%，未全部下跌，因为部分使用余额整理持续时间 30 分钟，这个相当于比较严重了。如果我们是一个小厂，可能每天交易额都很少，那可能影响不大，但是如果我们是一个还不错的一个企业，我们每天都能到达百万甚至千万级别的交易额，那下跌70%，至于 30 分钟这个损失也是很可观的了。


所以在面对这样的故障，我们可首先肯定要去找一找故障的原因是什么。离开原因发现非常小的一个问题是，在进行支付配置推送时候，推送的 JSON 缺少了半拉引号，导致了解析的失败，从而在失获取支付配置的时候程序报错了，因为程序在获取支付配置报错了，所以支付成功率哗就下来了。但是使用余额部分没有影响，因为余额部分不需要调起支付。


从这里我们就可以看到，这是一个很小的一个失误，但这个失误带来的损失却是很大的，我们先不说我们接下来要做什么，但是我们先从这个现象和原因中去分析，是不是我们哪里没有做得很好，是不是我们有该做的地方，我们有做到位？让我们带着这些问题跟老师一起往前走。

比如说第一时间有没有做校验，我们在界面上有没有很明确性或者误导性，它是不是能让用户知道我一定是这么用的，或一定我一定不能那么用？那我们在功能上有没有明确的操作路径的指示？我们的下一步，下一步应该是什么？还是说我们这个下一步需要人去手动的去切换，那这些都是使者我们该做的事，到底有没有做好？是否没有做好的部分形成了问题的触发器？而并且在出现问题之后，我们有没有一个好的缓冲方案，有没有好的回滚方案？我们的开弓有没有回头箭？我们有没有后悔药可以吃？这些都是我们在架构过设计过程中需要提前去思考，需要去明确，我们在架构中这些地方，我们该做的有没有做好，我们不该做的是不是我们做了？那当然归根结底代码有没有写好，如果代码没有写好，那肯定会出现各种各样的问题，但是种种之前的一切都可能会导致线上问题的频出，那我们的目的是什么呢？我们作为架构师，我们肯定是希望让我们的线上能够更稳定，更安全，

但是出了问题再去修复，其实就像是亡羊补牢一样，羊丢了，我们再把这个围栏补起来，目的是什么呢？目的不是去惦记那只丢了的羊，而是去想我们怎么样让剩下的羊不丢。


当然后续的事情也就可以成为防范与未然，但是我们有没有办法在丢这只羊之前就去做好一些事情，从而让第一只羊也不丢？这个就是我们在这一篇当中想要为大家讲解的这个点。当然无论是亡羊补牢还是防范，意味着我们都要去从错误中思考，要去想这些错误它能给我们带来什么，是否我们的架构可以根据曾经犯的错误去做好相关的改进升级，而且如我们不从这里去思考，不去做改进，是不是也意味着我们的羊还会再丢？我们的损失还会一再出现？当然在架构升级过程中，我们有没有做好价格的防措，我们是只是在不断地堆砌功能，还是我们也在做功能的同时想好了怎么样去防错？那当然如果在这个防错都没有做好，我们也会面临着用户会再次犯错误这样的情况，那我们也会蒙受一战的损失。


而在做好防磋之后，最终我们的架构质量有没有提升？如果架构质量提升了，那说明架构改进和架构防措都做到位了，但是如果质量并没有提升，那说明其中必然有地方没有做到位，而且并且在整体架构质量没有提升的时候，我们再去写新的功能及新的架构，就有可能还会犯同样的错误，蒙受同样的损失，那这些都是我们不期望的。所以在这门课里我们就和大家一起去探讨我们怎么样子能够更好地做好架构的防措。
