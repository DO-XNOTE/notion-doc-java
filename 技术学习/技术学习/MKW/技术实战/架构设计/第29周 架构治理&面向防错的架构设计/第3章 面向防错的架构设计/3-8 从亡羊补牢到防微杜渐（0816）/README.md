# 3-8 从亡羊补牢到防微杜渐（0816）

hello，慕课网的各位未来加工师们，欢迎回来，在上一小节里，我们为大家积列举了一系列的案例，这些案例都是防错设计在架构中的一个应用，其实重点还是讲了三个原则，杠跟自动可相符。其实这三个原则在架构中还是蛮常见的，当然其他还有很多原则也都会在架构中有所体现，但是在这里我们就不做那么多讲解了，因为要讲的东西真的是太多了。那在这一小节我们就为大家在从架构中的案例再衍生出来，我们去思考一下。都讲了这么多案例了，从工业加案例到软件的一个应用的案例，它架构中的案例我们怎么回来去思考最开始提出的那个问题，到底防羊补牢和防微杜尽之间怎么样子平衡？怎么样子去发展？这就是我们这一节要讲的内容。当然首先还是说我们可能遇到的都是问题，线上的问题包括线上的一些问题和线上的一些故障，那这些可能会为我们带来损失，


那其实就是我们在之前犯的错误，错误就导致我们的羊给偷走了，被线上故障这只狼给偷走了。

偷走了之后其实我们真的就带来了一些损失。我们损失了什么呢？既损失了我们的用户，包括我们有一些用户可能跑了，他会觉得你这应用这么难用，我不在你这玩儿了，会把你 APP 都卸载了，或者是我们损失了用户的一些体验，就会觉得，哎呀，我其实觉得你们挺好用的，紧接组这么难用啊？第二就是损失联资金，这个是经常会有的损失，我们订单出现问题，该给用户发货的没给用户发，那个得补给用户，发多了的我们还没法要回来，那这都是损失。


当然还有就是口碑，原来辛辛苦苦营造的啊，你拼到这么好用，这么稳定的一些口碑没了，大家都觉得，诶，你是有可能犯错误的，你上次就犯了错误这个梗他们能吹一年。在这种情况下，其实真的是我们要真的要好好的去反思，所以也意味着我们要先去执行，然后再去复盘，先让损失停止，在一这一刻所有的用户体验都回来了，然后接下来我们去复盘我们这个过程，看看这个过程里我们哪些是没有做好的，然后再针对这些没有做好的点去做好相关的改进，那所以这里就串联起了把。


一旦出了问题，我们肯定丢了这只羊，到了这种改进的时候，是不是说我们不光丢羊的这个地方改，没丢的那地方我们也改，这样子的话就做到了，其实是防微杜渐。所以故障复盘与改进重点还是向前看，我们不要光盯着这个故障，故障已经发生了，一反复的强调责任，反复的强调谁该背锅没有任何意义，我们需要的是什么呢？接下来要怎么做？我们线上可以杜绝这些问题，我们有没有一些原则可以使用？我们有没有一些设计方案可以落地？我们可以让这个问题断根，哪怕断不了根，我们可以去缓和，我们可以有提示，有警示，我们可以用一些产品设计的思路，减少用户犯错误的机会等等。


所以不要总去想丢了的那只羊，关注的还是还在羊圈里的那些羊，包括不属于这只狗的，我们另一个羊圈里还有羊，是吧？所以要考虑的就是我们现在手里还在我们手里的这些羊，那这些就所谓的防眉突见了。所以本身来说，从亡羊补牢，它本身就算是一个后置的行为，因为它相当于是我们从故障再到恢复再到改进，其实这是一个故障发生后的一个事儿，我们一定是要向防微杜渐的这样前置的一个方向去走，

那大家就要需要我们架构师了是吧？我们需要在里面针对我们的架构做相关的调整，做相关的改进，去降低犯错的概率，减少犯错的机会，同时让错误是有兜底的方案的，前两者肯定是最主要的。


犯错的概率降下来，那用户就不太那么会犯错误，减少犯错机会，当他有撞上，他会犯错误的概率了，但他不具备犯错误的机会，自然他也没有办法犯错误。真的二者瞪没有兜住，我们还有一个错误兜底，我们有缓和，我们有降级，我们有一些其他的产品方案，能够当用户犯了错误的时候，帮他把这个底兜住。这样子的话它最多只是体验上有那么一丢丢的，不那么友好，但是实际上不会对它有本质适应的影响，这样的话我们的底就兜住了。所以通过这样的方式，我们就能够将后置的亡羊补牢，逐步向前置的防微杜渐发展。


那进一步的看故障它本身来说其实是面临我们需要去修复，然后加复盘，再通过这两二者之间加上故障本身发生之后去思考，

我们思考的是什么呢？首先第一点就是行为预设，怎么去预设用户他要执行的这样的一个行为。其二就是对应的一个期望结果，用户他做这个事儿，他期望的结果是什么？这个网就是用户基本上的全部的一个信息了，他会是怎么样去用这个系统？他用这个系统希望达到什么样的一个效果？这个就是我们在设计这个的时候的考虑。当然通过这两个我们就能够预测出它的偏离机会，我们觉得它应该是这么用这个系统，但是我发现系统并没有把其他的路堵死，它还有很多的机会能够点别的按钮，能够往别的上去走，那这也意味着它在这些点上存在着偏离机会，它能够按他的意愿或者不按他的意愿发生了一些偏离，这时候就会导致这个结果其实并不是他期望的结果，也意味着这个就是我们需要去重设计的点，我们要针对这个点，针对这个用户他可能会产生的偏离机会进行除设计。


然后这个点当然就是我们防错的点了，我们从设计的过程就是怎么样子将它可能犯错误的这些地方进行防错设计，并且通过将故障中我们学到的那些信息融入到在偏离机会发现之后的重设计中，思考架构的重设计应该往哪个方向去走？我们应该怎么样子去设计我们的一个架构，从而达成防错的效率，用户的使用成本下降，我们用户的犯错误的概率在下降，他犯错误的机会在减少，哪怕他真的犯了错误，我们有东西能帮他兜底，这就是我们在重设计这个过程中需要去重点考虑的。


当然还有一个就是什么呢？既然有了行为预设，那我们就要能够让用户将他的预期行为锁定，他在用这个系统的时候只能按他的预期行为去走，他没有其第二条路可以选。就比如说老师最早给大家展示的那个电商平台上买东西，选了一个尺码之后，他就让我一定按他的预期行为去走，他一定会去选他有货的那一几款，不可能让我去选他没货的那几款，这也意味着我出现偏离的机会，通过产品设计被杜绝掉了，这样子的话就不会出现大规模无货商品被售卖而导致的履约上的风险。那这也是我们防错设计所能达到的一个比较好的效果。


好这一小节的内容其实比较简单，也比较聚焦，我们希望的就是在故障发生之后，我们再进行只写复盘这个过程里，那它就是一个亡羊补牢的过程。在这个过程里我们重点是向前看，关注我们还在羊圈里的这些羊，而不是关注那是被丢了的样。然后在整个过程里，我们要思考用户的行为预设，思考用户的期望结果，控制用户的偏离机会，针对这些进行重设计，锁定预期行为，避免偏离机会出现导致故障，从而达成相关的防措。这就是我们也希望大家在架构中，在大家的产品中去落地的相关的能力。那在下一节课中，我们将围绕着一些故障进行相关的分析，看一看我们怎么样通过防脱智力可以让架构更强壮，让架构能够从这些故障的这种死亡危机中涅槃出来，然后真正爆发他新的生命力。好，各位架构师们，我们下一节课不见不散。
