那老师这里在架构中讲的重点是一个熔断机制，比如说某一个目标服务出现特别慢，或者有大量超时，这时候我们把这个服务就熔断掉，然后对于后续的请求直接返回降级内容。


一方面我们可以让用户有一个比较好的一个体验，另一方面也可以快速地释放资源，当目标用服务好转的时候，我们就恢复调用。这种状态下我们第一我们对用户是一种保护，第二对于本身出现问题的服务也是一种保护。因为此时如果还有大量的调用，大量的重试，不断地打到这个服务上，这个服务所承载的压力会越来越大，本身它已经进陷入了一个非常慢或者有大量超时的临界点了，再有大量的请求过来的时候，就很有可能就把它打爆了。所以这里就意味着我们要做好双方的保护，然后起到整体的一个作用。


在这里有怎么样体现出一个自动？比如说我们客户端在调用服务a，也在调用服务 b 这样的一个同时，然后服务 a 出现了大量超时，那这时候如果我们还是这么继续去调用，那服务 a 可能就会出现故障。我们需要在客户端上自动的针对于服务 a 和服务 b 中的 a 进行降级，然后通过把降级值返回给客户端，这时候就可以起到针对于服务 a 的一个熔断机制，那这个机制一定是自动触发的，我们配置比如说超时率达到多少这样一个规则的时候，它自动触发降级，这样可以不需要人介入的时候保护我们线上的系统。

那这些其实都是一个相符的一个效果架构中，我们其实在责任链模式就是一个很好的相符的一个体现。责任链模式我们或者说它是两种模式的一个集合，一种是相符这个原则，另一种则是顺序的原则。责任链它规定好了一组链式的节点，每一个节点其实是一个处理者，只有相符的，满足某一定条件的才能够进行处理，而且处理后立即返回了，这也意味着我不相符的，其实我是不会处理的。


5 who 交给链的下一个节点，所以第一它满足了相符这样的一个原则。第二顺序，因为它是一个节点，另一个节点这样的一个处理的，它也能完成对应的一个顺序这样的一个原则来保证防错。所以看我们看右侧的这个 UR 标的图，它基本上就是客户端调用这个抽象处理者，它有一个 next handler 的这样的一个下一个处理者去进行执行的这样的一个接口，这样子的话我们具体的处理者排排一坐，那我就一个一个的去处理，满足条件的时候它就会处理，不满足的时候它就直接调用到下一个处理者了。这样子的话就是如果全部相符，那我们就返回失败，其实是就是一个很典型的这样的一个相符的一个案例。
