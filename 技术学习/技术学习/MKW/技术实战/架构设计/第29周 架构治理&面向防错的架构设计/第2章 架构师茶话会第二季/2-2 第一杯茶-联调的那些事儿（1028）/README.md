他现在负责的是公司 ToB 业务这一大块儿的一个架构，那么对于他来说，刚刚结束了一个大型的项目，那么就我们就有请何大亮为大家讲一讲他们在这个项目中经历了哪些事儿。何大亮在这个项目中，他们经历了一个跨团队多角色的一个大项目，这个项目基本上画了整个 b 端，甚至包括中台，包括前端与端等等，基本上是 ToB 的一个旗舰型产品，那在这个产品中我们的目的其实很简单，就是我们要准时按时的完成版本交付。因为这个事儿决定了公司层面上在 ToB 战略上的一个打法。那所以在我们接近联调的时候，作为架构师的我和几个团队之间的架构师以及TL，我们做了一次对焦。


在这个对焦里我们先把整体的联调思路与时间点确定了下来，大家到底要怎么联调？因为项目比较大，牵扯到了很多的模块，牵扯到了比较多的角色，那模块之间的依赖也比较复杂，所以我们先要针对整体的一个链路顺序，整体的依赖顺序进行相关的一个对焦，明确出来联调的先后顺序，明确联调的范围，然后最终确定好联调的时间。然后在对焦之后我们做了相关的约定，包括流程上怎么走和联调过程中我们应该怎么样具体去执行这个联调。那在此这些约定可以帮助我们有效地减少更多的变量，从而向大家的注意力，让大家的思想都能够达成一致。


在这个之后我们就做了很多联调相关的准备，包括数据层面上的准备，以及我们要准备很多的说明，比如说我们的模块儿，它本身这次实现了哪些功能？我们暴露出了哪些接口，哪些 RPC 服务？它都有哪些参数？这些参数都是干嘛用的？它可能会面临什么样的场景？异常的场景是什么样？将这类说明尽可能地详细地明确下来，这样子的话，各个团队、各个角色在拿到自己的文档的时候，他就能够知道我依赖的什么样的一个功能，这个功能能为我带来什么样的一个能力？它又有哪些参数需要我传递它同时给我返回什么样的结果？并且它可能会有走多少种异常错误码是什么？它有没有可能有一些在某些极限场景下带来的一些就是异常的行为，那这些行为我要怎么去降级？针对这些就有可以有了一个相关的一个设计，然后在最终完成相关的执行，就包括我们联调这个验证，以及当发现问题的时候，我们要及时的反馈，将问题反馈到这个问题对应的模块儿的一个owner，从而快速的来完成一个功能，它的联调验证以及反馈。


这个是我们在进行联调的时候首先去做的一个事情，就是把整个的一个流程，整个的一个思想做了一轮对焦，其这个事儿花得很时间很快，除了准备和实行花的时间比较多，但在前期对焦和约定上，我们其实花了不到一个小时，所有的人都明确了大家都回去进行自己相关的准备，以及为后续的执行做好相关的预先的沟通以及针对说明把这个规范格式进行要求了下去。


所以这个点我觉得是我们在联调这个过程中做得比较好的一个事儿，并且在这个事儿上我们也是获得了比较好的收益，就是在后续的准备和执行过程中，我们没有遇到比较大的阻碍，其实坐联调而已，我们也都称为他会坐在一起干活，因为有的时候要坐在一起，两个人才能够非常快速地得到相关的反馈。

但是很多时候大家都很忙，每个人手里都很多的项目，其实坐在一起干活有的时候是一种奢侈品，我们总觉得，哎呀，我想要把你的时间跟我的时间完全拉齐，我们能有一个大块的时间能坐在一起，把这个完全调好，真的是一件很奢侈的事情。有的时候就从要从同步变成异步，我去调好我的东西，然后我给你一个文档，你去照着我的文档先去调，发生问题的时候给我留言，发现看到留言之后我给你反馈，双方根据这个不断的调整自己的工作的一个优先级，调整这工作的顺序，将自己同步的执行和异步的消息反馈与调整做好相关的协同。所以这个其实是一个很需要考验大家技术含量的一个事儿。


其次就是联调一些，重点在于我们信息的记录传递，并且完成信息的一个履诺，这个信息记录与传递我想大家都很容易去理解，这个信息履诺是个什么呢？就是你给我一个信息，我告诉你一个信息，我会这么样子去使用我的一个接口，那我就要的确的把我的架构调整成是这个样子，然后你按照这个文档去做，唉，发现跟文档上保表现一致，如果发现不一致就意味着我要去做好相关的修复。


这个就是所谓的信息理论，三个就是快速反馈和反馈落地是关键的。我发现了问题，我第一时间就要告诉你有什么什么问题，可能是你哪里出了错时，数据问题还是代码问题，还或是一些边界值，一些异样值没有考虑到和原文档不符等等。那你在收到了这个反馈的时候，你要快速地去做好反馈的落地，当然最起码是日清，就是我今天提给你的所有的问题，你今天都要把它处理完毕，不然的话我明天我再下一步去做这相关的事情的时候，我就遇到卡点了，因为这件事情你都没有往下去落，我就没有办法继续往下走了。


所以这个点快速反馈和反馈落地非常的关键，当然就是我们在联调过程中肯定也会分很多种情况，大家很难说完全节奏都能拉成完全一致，

这也就会有外外部就位了，也有可能是外部没有就位，那在这两种情况下，我们就要差异化的做好相关的处理，目的是什么呢？目的是说让我们能够继续往下做我们的工作，而不至于说他们就会我就不做了，那回头他就为的是我来不及了。


那这种事儿其实就都是一些问题，那我们肯定是说要做好数据和验证这两个层面上的一些前置的一些事儿。比如说当外部就位的时候，我们就位首先可能就有对于有对应的约定数据了，也有可以做相关的实际调用，这时候去验证这个结果。但是在外界没有就位的时候，我们就需要去做 mock 这些数据，把数据先 mock 出来，我们先能保证我们自己是 OK 的。虽然我没有实际调它，但是我可以验证在这个 mock 数据下，我自己是不是 OK 的。这也意味着我们在验证这个过程里，我们重点是为了验证功能的一个准确性，有逻辑的一个准确性。所以即使没有外部的一个实际调用，只要我们能够明确出这个数据是真实的，和实际是一致的，那我们就能完成对应的验证。
