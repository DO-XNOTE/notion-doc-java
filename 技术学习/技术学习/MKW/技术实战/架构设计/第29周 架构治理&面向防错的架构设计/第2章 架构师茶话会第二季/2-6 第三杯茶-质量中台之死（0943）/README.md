# 2-6 第三杯茶-质量中台之死（0943）

hello，各位未来架构师们，欢迎回来。在上一小节里，杨大洋同学为我们介绍了雅在他们公司关于流程卡点以及卡点中关于质量与进度之间的一个平衡。质量和进度其实就无非对照着我们的管控能力和我们在平时去追求这种快速迭代的这种期望去制定的一个平衡。本身来说，从长期来看，我们其实肯定是做到两手都要抓，既要能够保住我们架构的质量，另一方面又要完成快速地迭代。但是从短期上看，技术债务会决定成本，那也意味着如果当债存在较多的债务的时候，我们直接一刀切地想要去从这个弱管控马上就能够强管控的状态，它会为我们带来很大的一个抵触心理，以及在平时执行过程中的会有强大的成本，那所以在其中找到一个平衡，逐步地循序渐进于渐进一方面逐步提高公司的质量，另一方面又提供员工的人的能力，让我们的同学能够成长，最终达到一个优质质量和快速迭代两手都抓两手都硬的这样一个结果。


那么在这一小节里，我们将邀请宋大美同学为我们分享一下关于质量中台的那些事儿。宋大美同学是一个中型的场子，他们的场子还可以 1000 多人，他在其中也是参与，在一个团队中担任了一个架构师，然后今年公司来推行了中台战略，它作为团队的架构之，也作为了整体公司中台的一个建设者，一个接口人参与到了其中，那就有请他来为我们介绍一下关于在他们公司这个质量中台中他们经历的那件事儿。


好，有请宋大美。大家好，我是宋大美，我很荣幸来为大家分享关于质量中台的这件事。整个分享我也把它分成两部分，我看前面的同学都把它分成两部分，那我有两部分，第一部分我把它取名叫做质量中台之死，其实也是我们之前在做的一个质量中台，其实我们原来最早就是沉淀了一些工具，这些工具其实包括了我们有一些研发框架和测试框架，这些框架可以帮助我们很好地去做相关的研发迭代和测试工作。

然后我们也有我们的脚手架，这个脚手架可以帮我们生成一些代码，从而提高我们的研发效率。


还有一些其他的工具，比如说像静态代码扫描，像这个单测的这种工具，像一些 MySQL 和扫描的一些工具，那这些工具其实我们都有，然后有我们也在至上去建立了几个质量平台，这几个质量平台包括了几块。第一个就是测试执行平台，它能够帮助我们更好地去执行一些case，比如说一些病情化，当当数据量比较大的时候，它帮我们提高业执行速度，也帮我们去管理相关的case。然后以及我们的质量管理的一个平台，它可以将整个的一个质量工作都串起来，从需求数到 bug 的跟进、 bug 的处理流程，然后以及线上的关于这种故障的管理，然后变更的一些管理等等，那这些是我们的一个质量管理平台，然后我们还有一个测试流程平台，它和测试执行平台是连在一起的，它帮我们去圈好我们的流程，做好流程编排。这样的话我们就可以用最小的成本来将各个模块之间的 case 串在一起，从而来完成整体流程的测试。
其实这几个平台还是比较好用的。然后到后面公司决定向大厂学习，我们也要去建对应的一个质量中台，然后希望通过质量中台的沉淀，能够进一步地将这种质量的工具和质量的平台成为一个质量中台，变成一个可复用的能力。然后由各个部门他们想要根据自己的情况可以去定制对应的前台，然后来完成面向于具体业务、具体场景的一个差异化定制。


其实这个初衷我讲大家都认可，包括我们所有同学都觉得这个是一个对的一个方向，我们也愿意去来建设这个东西，所以这个中台这个概念就开始在公司就沉淀下来了，我们也管它叫大中台、小前台，这个就跟行业中所有人面对于中台这个定位是一样的，我们期望的也是针对大中台来说，核心功能收敛，然后对于小前台去做这种差异功能定制，目的也很明确。

因为所有的工具都可以在中台中沉淀下来，然后大家各方面能力都可以贡献到中台中来，这样的话就能够用一套轮子来解决所有的问题，提高能力的一个复用。但是它随后也带来了一些问题。就比如说第一个就是轮子的适用性，因为这就跟比如这个轮子一样，自行车跟汽车能用一个轮子吗？用不了。
