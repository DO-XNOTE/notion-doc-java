说到功能分解其实不难理解，就是如果你有了需求，如果你已经把需求抽象成一个用例，是不是我们每个用例或者每个需求都可以拿一个模块来实现？但是这样的稍显复杂，因为我的模块太多了，我的需求可能成千上万，一个系统可能有几百个、整几千个，这种具体的需求和用例，但都用一个模块来实现，太复杂。


就通常我们会用一个模块来承载相类似的一堆的需求，那这个模块设计的多大？其实就是什么内聚和耦合要考虑的，那说到这个模块的尺寸是不是有一个名词非常之熟悉啊？高内聚低耦合，说的就是尽量让模块内部沟通非常之紧密，同时模块和模块之间很少的去沟通，这样保证我的模块可以独立开发，独立发布。


那到底怎么样才能做到呢？大家想想有什么设计原则？如果是什么？是经常是做一些软件设计，是吧？已经是一些资深的这个我们的程序员、资深的工程师已经参与过一些少量的架构设计呢。脑子里面是不是会跳出这几个单词啊？叫solid，是五大设计原则的首字母，那这里我就不详细去聊 solid 了，卖个关子，因为在后续我们在导论里面的那个设计原则章节的半仙老师会给大家详细聊一聊什么是 solid 原则？哪五大原则来支撑正常的一个我们的什么，我们的类，或者我们的一个我们的模块是怎么来进行开发的？那除了这五大原则以外，脑子里面还有没有其他的原则？通常我们什么，我们高内聚还有三个原则，而低耦合也有三个原则，这些原则也会在半新老师的课里讲。


我在架构师维度那个章节里会重点挑其中两个原则， deal 和原则，讲一讲稳定依赖和稳定抽象。那所有这些原则抽象起来实现什么呢？实现就是把所有的那些需求落地成一个一个的模块，而落地完了以后是不是够了？这个模块只是划分了，是吧？那怎么去实现那个模块呢？自然而然就要用到上一个大章节我们的架构风格，一个好的架构风格什么能保证什么？我们这套思路所实现的质量？同时一个好的架构风格也有很多的参考架构和参考代码，可以快速的帮助我们落地。


什么业务需求？好？我这里要考一考了，如果我们把模块拆得分得很好，然后我们要找架构风格了，你脑子里会印出哪些架构风格？有哪些？是不是张飞扬老师讲的那十几个风格先要跳出来，当然它不能涵盖所有的设计风格，但至少是什么是非常主流的那些，比如说你要把整个框架设计成有层次感，你是不是会用层次结构风格？如果你是什么应用的顺序调用就是什么主程序、子程序风格，如果你是要更偏向于实际，更偏向于世界和生活，很有可能就是什么面向对象风格，那同时你的数据流怎么样进行流动啊？也许是批处理，也许是管道程序之间怎么样沟通，也许是精神通讯，也许是事件驱动。


以数据怎么样在模块内部进行共享，也许是数据库，也许是超文本，也许是黑板，这种共享模式如果是一些特殊的代码，怎么样让它运行起来是解释器合适还是基于规则的系统啊？所有这些选型完以后，基本上每个模块的大体框架就完成了，这个时候拿我们的软件模板来填充。所谓软件模板，就是把所有的小软件怎么样进行快速拼装，来形成什么我们的风格，来完成我们的什么模块。就是有些现成的什么架构图的、现成的文档、现成的代码的，就直接拿到那些现成的东西快速复用，这就完成了 ABSD 的大部分功能。


为什么说大部分呢？还差一个，对吧？还差最后一个，也是最关键的基石递归，为什么 ABSD 跟其他的架构设计不同的点就在于他强调什么每次功能分解、风格设计和模板使用完以后没有结束进入下一次功能分解，只有不停的迭代，不停的递归，才能真真正正的把一个很复杂的系统逐渐的什么简单化成子系统，把一个子系统变成模块，把模块落成类，或者是落成我们的实际代码。

这是个小圆圈，然后我们可以用一些概念的子系统把它给拆分开来，这时候还是什么很模糊，对不对？我们还没有想到具体的风格，我们也不知道具体的模板，我们稍微拆分一下。


拆分完以后下层的风格向上，这个空心箭头我又有考点，如果大家对 UML 是吧？统一这个模型这个语言很熟的话，这种一个箭头，然后这个箭头是空心的，这样的箭头表示什么含义啊？表示什么子系统？是上面系统的泛化，它可能继承了主系统的这个类，也可能实现了它的接口，对不对？是泛化继承实心等等概念。
好子系统去实现一个上层的主系统，那同时我们也许会找到一个现成的模板。比如说用户登录就是最典型的一个例子，用户登录注册用户管理系统很多的，这个我们的 it 系统里面，我们的程序员不会自己去重新实现它，重新造轮子，而是什么用一个现成的开源的系统，或者是一个线程第三方系统来支撑它，这就认为它是用了一套模板，这套模板什么直接去什么去构成我们的这个系统，这个菱形箭头就表示什么，就表示一种形成，这种形成是什么呢？是聚合，对不对？如果是 UML 模型很熟悉，就是聚合好，我们用好几个模板去聚合出这样一套嗯，系统来完成某几个子系统的需求。


好到子系统这个层面，我们可以再进一步进行拆分，拆分成一个构建，而这种拆分也是什么？让构建去什么？去泛化我们的子系统去实现，或者说是什么去继承我们子系统的那些类。这些构建以外，我们也可以拿一些现成的模板，那把这些模板直接去去堆叠起来，去形成一个个子系统。好，基本上把构建这一层做完以后，已经完成了我们架构设计的主体内容就是前三步。


第一步拆解到第二步构建的时候，其实这些构建里面已经有一些风格了，对不对？这样构建是用什么？用黑板来进行沟通？那就是黑板风格这两构建之间是什么？是事件驱动的互动的方式，那就是事件驱动风格，当构件和它的构建的边界定义清楚以后，风格已经很清楚了，模板已经用起来了，这个时候就到下一步，我们把实际的内容和逻辑概念之间是需要什么？是不停的演进的，这个虚箭头就表示什么？一种演进，一种什么？一种延伸，一种递归，那通过这样的字顶向下完成以后不停的演进递归，就能实现我们整个 ABSD 的核心思想。


好，聊完的核心是不是感觉还稍微有点什么假大空啊？那我们后面一个章节详细的聊聊它的什么实际六步走，聊完六步走大家再回过头来看这个，这张图应该能更加理解。这一小节，我重点强调一下考验大家几个知识点，一个是对上一节我们架构风格的理解，一个是对后面我们会重点强调的什么我们的设计原则。第三个点就是什么，第三个点就是我们要了解到很多的什么，我们的代码都是可以用模板的方式快速的重用，很多的什么，我们的架构设计也可以重用的，所以架构师永远在什么，永远在拼装一些现成的东西，复用它，同时剪裁它。好，这一节就到这里，下一节敬请期待。
