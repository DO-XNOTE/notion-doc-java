想出来没有？我们有五大设计风格，下面又有两到三个小风格，所以一共有十来个风格。所以我们不能一个猜，我们要一枪命中这道题非常有难度，那我来给大家讲讲这道题的体验其实是什么？那面试官希望听到的是事件驱动软件架构风格，或者是事件驱动类似的思路。如果是说你对俄罗斯操作系统不熟悉，你也可以用 Linux 或 Linux 来做类比。你想一想看，当一个用户执行一个命令的时候，为什么它能够成功的执行？原因是什么？是后台必然是有些机制，而 Windows 的机制其实就是以事件驱动为主旨。


如果大家对注册表很熟悉的话，很多的行为其实都是通过注册表，通过它的系统的回调函数提前挂载在了操作系统上，当这些系统被提前注册完以后，它就是一个事件驱动的平台，当任何一个比如你打开控制面板，这样一个事件被鼠标驱动了以后，提前在操作系统后台挂载的很多的回调函数中的控制面板，打开函数就会进行操作，它就会去展现控制面板里能提供了各种功能，这就是 Windows 操作系统它的一个特征。


它就是一个事件驱动型的操作系统，如果你对 Windows 很了解，你就可以从 Windows 角度去侃侃而谈。如果你会熟悉 Unix Linux，你发现其实 Unix Linux 很多时候未必都是以回调，以事件驱动为主体，你就可以把它领到你熟悉的去操作系统方向。所以这道题既是在面操作系统，也是在面我们对于事件驱动这种架构的熟悉程度，同时你也可以引申出去，比如说我对应用系统里面自己开发的时候，实现驱动用的不多。但是我跟第三方对接，我基本上都是通过回调函数这种方式来进行异步解耦，而不是采取同步等待第三方系统给我答复。这样的这个交流解释就可以深入，你可以深入到某一个业务系统，比如支付系统怎么样进行回调等，减少等待的。


那这样的交流从操作系统到应用系统到实质业务系统就是一个什么？给面试官有感觉，就是我真正的是在面一个很资深的架构师。它有一套设计的风格，同时它也有很宽广的知识面。好，第一道操作系统的设计风格题聊完，你答对了吗？如果没有答对，没有关系，我说过三道命中，两道游戏，下面两道努力，

这种时候你用什么样的架构风格来开发这套服务器的处理软件？一个 Web 的应用服务器的处理软件，处理包头和报文的，大家想一想什么风格？321，请作答。想出来没有？唉，是不是有好几个风格都有点像？我有点犹豫，是不是看看我们其实考官希望听到的是什么风格？这道题在出题的考官其实希望听到的是说这个风格叫管道过滤器风格，他希望是说什么是能够听到这样思路，就是我们的这个架构师的这个面试者是通过一些分层的思路。


但是不光局限在分仓，而是什么对于每一层采用合适的过滤器直接进行处理，甚至于采用一部分流式的，因为当 Web 服务器的客户端进来的时候，它的数据流不可预期的，所以你没法提前调度，你是在不可预期的情况下，数据率有可能很大，采用一些流式的思路管道过滤器就是最合适的应对这个问题的办。但是同时我要强调一点，一个产品不是一个架构风格决定的，它是应该有多个架构风格共同承载的。


所以这个时候我们既可以一个思路，就是在管道过滤器上深挖怎么样把流式讲清楚，当你对流式处理不是那么熟悉的时候，我们完全可以这道题眼中的附带加分项为引申项，它的加分项的点在哪里啊？在分层你能感觉出来吗？ HTTP 是一个什么？是一个四层、五层的结构，而 SOAP 也是一个六层、七层的结构，中间其实是有一些什么分层的感觉的，就是报文头解开以后里面还是应用层的报文，这时候里面再处理是不是有点分层的感觉？就你完全可以不围绕题液，我们围绕加分项，因为这种实际的应用都是多层风格的堆叠，我在强项上，在分层的上面我可以详详细细来聊一聊怎么样在不同的应用之间进行沟通的处理。比如说我的认证层，我的校验层，我的网络层，我的这个加密层怎么样在业务层上层分离。
