 i m Dex 矩阵，相当于是一个整体企业架构当前面临的各种各样问题的一个核心矩阵。这个矩阵相当于什么？是一个比较静态的？就是你虽然还在不停地叠加它，但是相对静态 read 矩阵是一个非常动态的矩阵，那这个矩阵其实用 4 个单词 risk 的那个RNN。


第二个是假设 assumption 的a，第三个是问题的 i 和第四个 dependency 的依赖的 d 形成一个RAID，不是那个磁盘正面的 RAID 是四个英文字母的缩写。就每一个决策其实你都要去考虑一下具体的 risk 跟 issue 的区别是什么？ risk 是尚未发生的， issue 是已经发生的。


assumption 跟 dependency 的区别是什么？ dependency 是尚不确定的，尚不是已经确定。比如说我们前一个矩阵里面有这样的约束叫做，嗯，企业架构规定了我们必须使用开源技术框架。那这就是已经确定的，那就是写到 assumption 里面，因为企业架构要求开源，所以我们不能，是吧？我们不能去做一套自己的这种闭源的系统，我们必须用什么开源的 Jenkins 来进行 CICD 的自动化部署自动化这个编译，那这就是一个assumption，那如果是dependency，就是假设我们这个公司还没确定两个的决策，就它就dependent。


什么dependent？架构组或者架构办，对于这开源、闭源进行最终的决策，定完以后我们再进行选型，那所以依赖就是尚不知道 assumption 是已经知道那 issue 的问题就是我们在选型过程中我们假设用了Jenkins，我们发现 Jenkins 跟我们的 Docker 几乎不能兼容，出现了兼容性，这就是 issue risk。
就是，嗯，我们上前还没有尝试使用Kubernetes，所以有可能 Kubernetes 跟 Jenkins 之间是有冲突的，那就是risk，那把这些都记下来以后它会制约你的决策，同时它也会优化你的决策。怎么说？制约就是因为他们不能实现，会导致你的什么决策师，你的你之前的决策是无效的，同时优化你的决策是什么？有时候那些 issue 问题被解决了， risk 问题被降解了之后，你这个决策就很明显的二选一，就不用选了，有一个自然就是什么，就因为它的质量更好，因为它的实现更简单，它的所有风险和问题在被解决的时候，它的所有的这个依赖条件都会确定成假设，最后形成最终的决策。

当一个电商平台从一个什么单体成为分布式，成为什么百亿、千亿级的这个交易量的大企业的时候，它的扩展性永远是摆在第一位的。它可以牺牲各种各样的期待，比如说 CIP 理论里面牺牲两方来实现一方，对吧？这些都是很多时候都是为了扩展性，那性能跟扩展信息息相关，是吧？性能其实在于什么？吞吐量在于它的 IO 的承载能力，在于它的响应延时能力，所以它的性能那高可用就是当我的一个节点或者一个应用的软件出现故障的时候，有没有单点故障？我到底可以承载多少个酒的高可用？安全性也是所有真正的企业级是最关心的。


如果说你是测试开发可能不关心，但是你在企业生产的地方所有的安全性有没有黑客攻击？怎么样防御攻击？我的本身数据有没有被什么被污染、被破坏、完整性破坏的可能性？这些都是什么安全性的重点？那耦合性指什么？指说我这个应用和另外之用之间是什么？是松耦合还是紧耦合？他们是不是能够什么组成一个新的实体，重新去构建我们的类？还是我们要尽量的细拆？对，这些性能决定了我们架构设计的总体的框架。如果在前期这些方面考虑失误，后期你即使改动代码，即使反复测试，很难去改变你的整个这个产品的特质。质量是什么？是架构设计的核心，其中这五点是核心中的核心。除此以外，我也聊聊这四个。


嗯，质量特性、伸缩性看上去跟扩展性很像。扩展性是说我能用1万个节点来承载这个业务，伸缩性是我能在一秒钟之内从 10 个基点扩展到1万个节点，就在瞬间的一种这种生长收缩的能力。这是伸缩性，易用性跟可用性看上去很像，其实不然。可用是availability，是说我什么时候能对外提供服务？易用心说是usability，就是我对外提供的服务是不是友好？假设我一直都能对你访问，但是你这个界面，嗯，普通用户根本看不懂，无法操作，那它的易用性就是0，是吧？可用性是100，所以那其实没有用，因为用户没有法得到他所需要的产品的易用性也是一个。嗯，产品就是比较重要的点，但是它这个可以在后期进行改良，不像前面 5 个那么关键。


除此邮箱可维护性影响的是运维，影响 SRE 的操作，那可一致性影响的是我这个功能不能什么摆脱硬件软件的依赖。那种，比如像容器化就是一个特点，就是可一致性很强，我能从雷凌系统到 Windows 系统到 Max 系统，我能从这个 IBM 服务器到这个。嗯，我的戴尔小型机也可以到云平台的一个虚拟机，它可以飘来飘去依赖一句，那好，这些以外还有两点，我也是希望大家在架构设计中考量了这两个性，考虑的是你架构的性、可操作性。


就是我这个设计完美符合大厂要求，但是什么开发需要1万个人，天，那好像不适合你们中小型企业吧？对不对？那同样道理，可重用性是什么？可重用性就是诶，我这个架有架构设计的很好，我开发个 5 个月能开发完成，但是我一次又有个新的需求来的时候，我要重新开发一套系统，结果这套所有的价格要重新修改，全部重新设计。


那你是不是能够设计一个可能需要六七个月完成的架构，但是能够重用，对吧？这就是可重用性，因为这些所有质量因素都算是相对和谐，我们外面应该还有更多，你把这些都考虑到架构设计里面才能什么？找出所有的对你的项目产生约束的，不管你是质量约束还是嗯我们的这个公司的约束，还是生态和圈的约束，把这些约束考虑进去，你才是一个真正的架构师，你设计的架构才是真正可行可执行。下一节我们会去聊一聊组织和架构，聊一聊架构和组织和实现之间的关系，相信更精彩，请期待。
