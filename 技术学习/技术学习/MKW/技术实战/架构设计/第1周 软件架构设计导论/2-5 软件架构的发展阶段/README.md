还是现在比较流行的 javapython 还是代表未来的 go 语言，这些语言之间其实什么都有互相的限制，也有一些基本的约束。通过这些语言来实现我的应用，就自然我会受到它的约束，也会被它所掌控。所以这是第一代，基本上软硬架构就跟着语言走，语言能怎么实现我就怎么做。


而到了第二代，有很多大师，把一些抽象的内容结构变成了数据结构。不管是我们什么在大学期间还是我们在这个就业以后，我相信大家都在开发过程当中反反复复在推敲或者重新理解每一种数据结构。在这种数据结构的认识当中，其实什么我们就在实现一个业务的抽象，那除了这种结构化的抽象以外，还有一种模式性的抽象。


那海外通常我们叫什么叫 gun of four 四把枪？我们国内就叫什么就叫设计模式，四人帮对不对？那都是指这四个人他们开发了一种什么23个模式来完成常用的软件设计。那在这之上，马丁福勒又推出了几种新的模式。除此以外，像领域驱动里面又推出了几种新的模式，像23加几将叠加出几十种不同的设计模式，基本上涵盖了我们软件设计所需要的方方面面，有结构了，有模式了，是不是对复杂逻辑？我们还有一些算法，大家如果去经常面试，其实就是经常在刷算法题。那算法题很多人说算法题就是什么造火箭，我进了这个公司以后，其实我就是煮鸭蛋，我从来都不用算法。那其实不是这样。大家想一想，如果你是在做谷歌和百度的搜索引擎，你必然是用算法。但是如果你是一个简单的业务查询，里面是不是也有一些算法的影子在里面？虽然你没有在推敲一个算法的优劣，但是一些算法的思想和影子永远是在你的代码的研发过程，在你的架构设计过程中保持不变的。


所以这些基本上是什么？是对语言的一种抽象，也是过去一个整个软件架构里面很核心的内容，但是到了第三代以后会又发生一些不同了。第三代主要特点就是整个业务逻辑变得更加复杂，你没法用一套结构、一套设计和一套算法来实现整个业务框架。你更需要关注软件架构的拆分。就是前面说的组成派的思路，就是你要把你的业务拆得更细更精美，它中间的沟通才是你的核心关注的点。那这种情况下，最经典的语言出现了 UML 它不是一个完整的编程语言，而是一种画图的语言。那我在后面的章节里面会带着大家在相关的地方会给大家看一些这种 um 我相信大家很多的架构师或者说很多的资深的研发的工程师都很熟悉 UML 了，但是一些细节点我到时候会跟着大家包含在一些习题里面，我们面试题里还包含在我们的课程里面，稍微给大家再交流一下。


那除了像 UML 以外，还有像 cbm 这种就是也也是 component based model 的一种大型的企业级的功能分组模块模型。通过这种模型，基本上可以把一个业务组把一个整个企业的需求全部拆解成一块，然后再用语言和我们的结构化设计来完成。那这就是软件设计的过去。那今天又发生什么不同呢？我们看下一页。


那今天最大的一个不同就是多视角，那之前其实都是什么？一套方法论一套视角相对单一，但是在当前这个环境，你会发现什么？每个人的角色会有不同，屁股决定脑袋，所以每个人的观点差距很大。

这个时候如何要很好的让架构师跟产品经理跟研发工程师跟运维公司配合起来？你既要考虑逻辑对吧，去完成产品经理的需求，同时你也考虑物理去跟 SRE 去跟运维工程师对接。你既要考虑应用去完成业务需求，你也考虑技术去实现中间件和底层的数据库。你既要注意功能性，你也要注意非功能性部署、监控这等等的要求。


所以架构是多视角，整个架构设计的多视角成为了当前架构设计的一个核心点。那在此基础上，衍生出了很多个体系框架，像这个 ABS D 就是基于架构的设计框架，这种框架其实更强调什么？比如在早期就用架构的思路进行衍生演进。还有像 DS A 就是特定领域架构设计，强调我要注重领域模型，同时我要结合这个领域里面常用的架构进行复用，然后有少量创新。那除此以外， at 更是多视角的一个特征点，就是它把整个架构用三个视角来进行展现。这三个视角形成三个维度，最后形成一个旋转立方体。这我们后面再后续章节这三个框架都会有很详细的介绍，大家不用急也不用慌。

那这就是我的预估了。首先我的预估第一点是云化是不可避免。因为现在有很多的架构设计，它一开始不是从需求出发，而是假设我把我的所有业务全用某酒公司的 SARS 平台来搭，我还缺哪些？就这种思路其实也是可以的，就是我能提供哪些跟我需要哪些之间的 gap 和之间的差距，才是我需要研发的内容。所以这种思路其实就是大量的资产复用 ，SARS 是最容易资产复用的。


因为 SARS 平台基本上从底层的服务器到中间的中间件到应用，都是由第三方帮你做了，你只需要租用它的平台实现少量定制化，就能完成你的功能需求，你更关注你的核心欲，你把你的支撑欲，把你的这个通用欲全部都剥离出去。那这是一种另外一种就是 pass 可能我的业务需求定制量很大，我没法完全把我的业务都交给第三方，所以我仍然需要我来定制我的应用，我来设计我的应用，我来开发和运维。但是我可以把我的所有中间件、所有数据库基本上全部抛离出去，简化我的运维过程。就这种 pass 也是一个经典的架构模型。那最少复用的其实就是 ISIS 只是把服务器让什么云平台提供了，这上面你要搭自己的数据库，你要运维自己中间件。所以 is 是一个最少复用的云化形态，但是仍然会什么会产生一定的限制，因为 is 也有供就是不同的供应商的限制，那 pass 限制更多， SARS 的限制就更加多。所以云画既带来复用的好处，同时也断来限制。


我举一个例子，假设我是金融云，那这个金融云要求所有的用户要三方认证，要短信认证、要微信认证，还要有一个特殊的令牌工具认证。任何你去看云上的这种就是用户认证系统，他都没有既能实现令牌认证，又是微信认证，又是用户认证，三方全认证才能通过，几乎没有这样现成产品。


这时候就如果你一开始是传统的一个金融运营商，你现在想变成一个金融云，你就很难直接拿一个 SARS 来改造，而且 SARS 平台它相对都封闭，你很难去改造它的接口。所以这种时候可能 pass 就是一个很好的选择。所以在被云平台限制还是能反复运用云平台的或复用云平台的架构之间做一个博弈，是未来架构的一个可能很多的需要做决策的点都在这里。另外一个是演进型架构，就是我们现在很多传统的不管是大泥团是吧，还是一团大江湖，你已经在这样了，你不可能我投个一亿元，全部重构一遍，很少有这个 CEO 这个魄力的拍板。通常怎么样呢？我先投个100万，我改造几个服务试试看，如果改造的好，我就逐步在改造。那这种投钱的方法其实是什么？就是眼镜式架构。那这眼镜式架构也有三种，一种是说我投了一个整个业务群，一次性开发完，然后改接口，把它完全的融合进我原来的整个企业里面。这种接口有可能涉及到同步到异步的改动，有可能涉及到五六十个模块之间的相互耦合，非常麻烦。所以拆迁者用的相对少，后两个用的相对多一个修圣者。修圣者就是我就在你原来的服务上增增改改，实现新的功能。那绞刷者是说我又起一个新的一服务，但我不是起一个服务组，我只起一个服务，然后把这个服务跟你那个原来服务搞得还蛮像的，然后一一步一步的去把你的替换掉。


我在后面的章节会详细讲这三者之间的区域还如何来演进展的架构。所以这里简单跳过，基本上可以感觉出来未来的架构是什么。至少是张老师的敷衍。老师心目中架构就是能宠用的会更加多宠用化，然后能不替换的也不替换化，这样就是小的改动，但是步子走得更快。来跟上我们业务的需求，我们业务的需求通常不是什么跳跃式，而是小量需求的变化。所以我要让我们的架构能够支撑这种小量的眼睛式变化，同时也要让支撑那种把很多的服务复用，用云的平台来复用它来减少我的开发成本。那这就聊完了我们的主体这一章节，主体内容后面就是关于小节，关于一些面试期的沟通，那相信这才是真正大家期待的是不是。
