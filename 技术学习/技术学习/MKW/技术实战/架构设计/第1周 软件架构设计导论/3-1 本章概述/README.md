先看一看这个电商公司十年之前它的早期架构是什么样的？它其实是一个什么类似于意大利面的那种，就是大江湖、大泥团，它是一个什么大巨型的这个架构，所有的这个业务全部混编在一堆的服务器里面，这对服务器没有角色之分，上面跑着整个业务系统的所有的代码，就像是一堆大泥团的这种架构，那之后它逐步什么把这个架构稍微分拆了一点点分成，比如说几个所谓的玉或者所谓的子系统，但是很粗糙，可能只有买卖、用户搜索这样几个主要的子系统非常的粗糙，然后上面的代码是这样跑的。
你看跑火车，所以跑火车就是说我每个月我发一辆火车，把这个所有的这个代码发布一遍，当你能赶上我这辆火车的时候，你的代码比如说在开发测试环境完成了，你就可以在那个 git Hub 上面什么 merge 到master。那这个时候你赶上火车你就发布了。如果你差了几分钟，那你刚好就看到火车在你面前什么呼呼疾驰而过，你就没有追上火车，那怎么办呢？再等一个月，等到下个月的发布场景的时候，你再来追这辆下个月的发布火车吧。

他是这样，用户他去登录服务器，那他因为经过了微服务改造，所以他前后端分离了，所以他每次首先访问前端服务器，然后会触发一些阿贾克斯call，或者是前端的这个叫 API orchestration 编排组合，然后去 call 后端的这个API，那调用后端的 API 以后，前后端和用户之间就形成了三位一体的互动，那这个之后后端这个 API 又经过了进一步的这个微服的拆分，使得什么服务和服务之间尽量解耦，采用了消息队列这种异步的这个通讯机制。然后我们发现数据库的写库其实压力有点大。那这个时候怎么样去让数据库能够更强的具有读写的能力呢？我们把它实现了读写分离，把读库拨出来，同时搭建了毒库的集群，使得毒的能力的明显增强。因为一个实际的电商系统，其实血是远远没有毒的，交易来多的，所以增强毒库的可扩展性能，很好的缓解我们什么我们的服务器端，我们的这个写数据库的压力。


好，这个做完以后在前端再加了个CDN，加了个大缓存，那这个缓存很就近用户，如果你这个用户是在上海，他那个 CDN 就访问上海的电信、移动、联通的机房。所以很多的缓存的内容直接从这个就近原则就获取到了，那同样的北京、长沙、深圳的同学或同事自动就从它就近的那些 it 机房获取到什么静态数据。
对，很多的这个前端服务器的那些CSS、 JS 被缓存到了就近城市的一个 CDN 的这个 edge 就是边缘节点，这样做完以后是不是可以搭建一套实现蓝和绿两套的这个系统？那任何一套系统它都可以什么完全独立的运行？那同时它可以运行什么不同的代码来进行快速的周期迭代？那这样什么？是不是就不需要赶火车了？我们可以在每天或者每周都制作一些这个发布周期，在这个发布周期上大家应用可以自由的发布到我们的服务器上。


只要你负载均衡的 CDN 能够很方便的切换每一个微服务的指向，你就可以有的应用服务在这个上面，那个一套系统上有的应用服务只在下面那套系统上就有的应用处于蓝，有的应用处于绿，没有关系，因为我们 CDN 可以很方便去指向这些业务。同时业务和业务之间采用一种比较松耦合的方式，你就实现了这种蓝绿发布，同时又实现了服务器的微服务改造。那这样改造完以后还不够，我们发现后台服务其实还是有点紧张的。那怎么样加一些缓存？这些缓存通过大量的这种 API 的缓存命中，使得服务和器和服务器之间、服务器和数据库之间的压力进一步减少。同时我们需要提供一些大数据的能力来增强我们的业务的分析能力，来揽客来获取到更多的这个用户。


就通过这个大数据分析，再通过强大的搜索引擎，我们逐步把数据库的一部分功能剥离出去了，那很多的分析业务直接落在大数据上，那很多的这个主页查询，很多的这种什么目录查询，很多产品查询都自动的转向了搜索引擎，那搜索引擎通过这种很强大的这种分布式能力或并发能力，减少这种数据库的这种单点压力，并发压力，这就是五年之前微服改造以后，这个电商企业它的那个 it 状况。
