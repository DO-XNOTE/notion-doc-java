那这个范围其实就是我们的问题域和问题子域，我到底是要做一个 c to seed 公司，还是一个 b ToC？我到底是要卖我们的二手车，还是要卖二手书？定义清楚这个领域的范围之后，下一步是什么？定义特定的领域的元素，你这个元素包含说哪些呢？包含我要设计的产品，我要卖的一个渠道形式，以及什么我的这个产品的这个价格定位，或者是我的这个用户特征，所有这些都是领域的元素。


那如果是大家对商业其实很熟悉的话，在我们的 marketing 市场营销里面好几个p，是吧？什么样的用户的person？什么样的价格？price？那什么样的这个促销方式？promotion，对吧？什么样的地址去买place？这些东西都是我们领域特定的元素，把这些元素考虑清楚以后，同时也要考虑一些 it 的元素，综合起来也到了下一步，我们就会定义出完整的整套在领域里面设计的需求以及它的约束。


诶，是不是跟 ABSD 很像了？对，到了这个设计过程的时候，开始掺杂进一些我们很实际的需求了，比如我这次要卖的是什么？是给家用防止某某疫情的这种口罩，那这个设计的需求就很详细，所以这是一个具体功能的，而约束是我这个口罩希望是什么？是我们的每一个用户每天都可以抛弃型、日抛型的。


那这样呢？这个产品就不能太贵，是吧？不能超过 5 块钱，那每天花 5 块大家还能接受，每天花 20 块，好像这个日抛型就有点贵了，那这就是个约束，有了这些约束以后，我们就会形成某一个特定领域的实际的这个架构了，这个领域还偏什么？还偏空泛一些。所以我们会希望什么在这个下一步过程当中会生成或者说搜集更多的可用的产品单元，把这个架构更丰富化，它到底落地的每个构建该怎么样开发什么样的这个具体设计全都给设计出来，同时把它的拼接，对吧？就是每个构建和构建之间，我们这个每个组件和每个组件之间它是怎么样一个互通有无的关系？他们是怎么样形成整条领域链的？给思路出来。


当做完一次领域设计以后，当然不够，我们需要什么不停的并发的、递归的、反复的去迭代，去螺旋式的上升，这就是领域的DSSA，整个的这个设计，或者说是建立过程当中主要的五大步，那这是一个动态描述，我们怎么样来进行设计的？其实关键点大家能理解到，是吧？基于前面的领域分析，我们还沿用这个领域分析，然后把它给细化到某元素。


这个时候我们在引入我们当前的项目所需要的一些具体的功能需求、非功能、新需求，然后找一些什么我们常见的那些架构的构建，找一些这个架构的框架，快速的去像拼积木一样去拼，这其中是不是也有很多思想是跟 ABSD 一样的？这些思想其实就是经典的架构思想，那有些不同的地方，偏领域的，偏我们的整体的元素的这个特点是领域驱动或者说领域设计所特有的。


好，把这个动态过程讲完以后，我们看看静态来看最后会出现哪些弓箭，或者说是哪些递交件，对吧？我们要做一个设计，做完以后一定要有个产出，这产出可以落在文档上，也可以是拍张照片，是吧？记录下来都可以。那有哪些我们的弓箭递交界呢？首先我们对整个领域开发这个环境，我们的架构师应该能产生以下这些东西，能产生一些参考结构。

这参考结构可以帮助我们真正的在这个实际的开发环境当中去使用。除了这些参考结构以外，还有一些参考需求，那这些需求是来自于前面的分析的。四大金刚的四种人帮我们来分析了，我们所以可以把需求盯得很清楚。在于这四个人的分析的基础上，我们又有了一些对本项目，对于本次系统和要求的一些功能性和非功能性加进去，这样就是有了一个传统的分层结构，也有了我们很具体的参考的需求。


之后我们会有一个基本的架构设计图，有一个总框图或者有些类图会设计出来，同时我们会把领域的模型，这里聊的领域模型可能是什么？是可能是个六边形模型，对吧？就是某几个领域模块，怎么样去跟别的模块沟通的？六边形模型有可能是一个金字塔的市场模型，具体模型我们在领域驱动建模里详细来聊，看看会生成什么样的模型。


不同的领域模型都可以在 DSSI 里面作为弓箭输出，输出完了以后还会有一个标准的开发工具他会指导，比如说我在这套开发工具是以，比如说印特利介的进行代码开发，然后我们以什么 JVM 进行代码运行，同时最后以容器的方式进行代码部署，整个过程当中我们会引用到一些实际的这个代码框架。


这代码框架可能用一个模板直接可以导入英特利界，那这些都可以认为是一套开发工具，它帮助我们完成了整个从设计到落地的到部署的实际环节的所有功能自动化。那它都算是开发工具，当这些做完以后，进一步考虑，其实什么是在比较偏设计到偏代码之间，我们完成了五大弓箭，最后还有一个弓箭是把我们的实际的代码和最后的运行给连接起来的。这就是什么实地化的体系结构，就说到这个有点虚，就说到落地就是什么？就是一个一个的快速上线的 POC 项目，就是你通过我们的架构设计完成以后，除了这些工件以外，应该有一两个落地的 POC 项目， POC 项目就是验证我的原型是可以运行的，我们要在实际的应用环境当中去真正把它给运行起来，这个时候我们才认为 DSSA 的设计完成了。


如果你的全部都是落在纸面上的弓箭，我们不认为 DSS 的设计是完整的，必须有一些这种实例化的真正的能够可运行的代码，或者说是功能模块出来了，然后以一种原型机的形式在我们的实际应用环境当中跑起来了，验证完了部分难点或者说有争议点，这个时候我们认为 DSSA 的设计过程完成了。

因为 DSSA 对实现几乎没有提特别高的要求，因为它是一个偏领域的设计的思路，所以对实践它只有这样三个简单的提示点，一个提示点就是能复用的都复用，说白了就是什么能不开发就不开发，这是 DSS 里面实现了一个主要的思路，那第二个思路就是什么？一旦你开发以后，你要把你的开发件变成可复用的，那下次有其他行业或其他公司，或者你的其他项目来用的时候，你就可以复用了。


最后一点就是这个整个过程一定要进行不停的螺旋式的迭代，跟前面的分析一样，跟签名的设计一样，都是螺旋式的。那开发怎么螺旋式迭代呢？其实很简单，你的所有的开发运行过程。什么是满足标准的 CICD 持续集成、持续发布的？那这个时候你的代码的改动自动会推送到我们的准生产环境，同时每一次我们的发布最好都是主链发布。所以你这样的人，你的应用代码永远是在向前演进，不要老是回退，对吧？同时你要有一条很清晰的代码发布线，我们的代码的演进线也保持我们的应用是不停的向前演进的，同时在整个过程当中，我们每一个节点的版本都有一个四记录，比如说用GitHub，那你在每一个大的这个节点上你就会打一个分支，是吧？你会把当前这个版本或者打一个标记记录下来。


如果是说到某一次眼睛到一定程度你只能卖口罩的时候，你可以回退到之前的某一个版本，把这个版本变成什么可以卖，什么可以卖我们的钢笔、铅笔另外一条线的这个分支，通过这种不停地演进，同时在核心的这个关键点做一些记录的方式，能够实现领域的分叉管理。当这个领域你在不停的前进的时候，你可能不够抽象了，这时候你可以回退某几个版本，进行分叉的管理，这种螺旋式的或者说是分叉式的眼镜迭代，也是什么 DSS 里面强调的复用的基石，我就是因为经常有什么不同的节点上的不同的架构都记录下来，所以我可以在不同的层次上向其他的行业，向其他的特定领域进行衍生。
