所以我们就拿类比，就拿我们的莫扎特一生当中他所追求的六大目标来看一看我们的架构应该如何来追求它的完美。


那首先第一条，莫扎特他永远把他的音乐作为一个桥梁，这个桥梁是他和所有的听众之间的桥梁。有人问莫达特，为什么你这么天才，只要坐在钢琴前，就能很方便地谱写出新的乐章。那莫德他的回答是什么？他平时经常去听不同的这种风格的这个音乐家的作品，去了解观众的需求，去听观众喜欢听的曲目，他去感受当时这个匈牙利或者说是整个音乐之都，所有那些能够产生共鸣的音乐是什么什么样的东西，客户需要，然后他把它融合到自己的钢琴的音乐曲里面。


这跟我们其实 it 设计很相像。当你在做 it 设计的时候，你要了解客户的需求，它的产品需求是什么，也要知道我们的产品经理希望的产品定位是什么。出类旁通的。我们应该是拿行业的解决方案，拿我们竞争对手的产品的内容来进行类比，看一下什么样的架构才能真正满足我们自己企业里独有的这种功能模块，要出众要出条，同时也要能够什么，也要能够承载基本的那些功能，业界里面基本的一些功能都要在我们的产品里得到实现。所以这就是大众的需求和某些我们的听众的需求实现一个完美的融合。


这就是莫德特曲目的特点，它阳光明亮而又穿刺性，它既满足了大众的需求，又有一些特殊的亮点来满足某一种某些品位的听众的需求，它就是一个很好的桥梁。那除了桥梁以外，莫泽特还有个特点就是他谱写的钢琴曲，他不是谱完就结束了，他会自己演奏。同时它的曲目很多的时候是什么？是一个钢琴协奏曲，它不是一个奏鸣曲或者说叫做钢琴独奏曲，它需要有一个乐队，有一个管乐队和弦乐队的配合。那这个时候他会把他的曲目发给他的最好的伙伴们，让他们在我们的钢琴的这种大会上去展现，去实现他的曲目。


他会作为一个主要的观众去旁听自己的曲目，在观众心目当中是怎么样展现出来的，它的这种反响是什么？这跟我们什么？这跟我们架构师要关注我们设计的软件怎么样，研发之后，怎么样上线，上线完以后，在运维过程当中，我们的功能性是不是达到了我们产品经理和客户的需求？我们的非功能性是不是能够实现高可用、高并发、高吞吐量，还有很强的安全性？所有这些其实都是架构师在整个架构过程当中要追求的。


我不光是完成架构设计，我要追踪整个产品的线，从他的需求到我的设计，到后面的开发和上线，以及后面的长期的运维，以产品的角度要贯穿它，你既是一个什么一个跟客户和产品经理的桥梁，也是一个指引后面研发和运维去把整个产品线贯穿落地的一个指引的工作。这也就是穆罗德特做的比很多钢琴作曲家要强的地方。


除此以外，你去看莫扎特还有个特点就是它的每个乐章大乐章都很长，但它把它切成了独立的小乐章的时候，小乐章主题非常鲜明，同时在小乐章里的每一个旋律都让人记忆犹新。那他之所以能这样做，其实大脑里面一直在什么在切割。当他有一个完整的故事的时候，他把故事切成一段一段的乐章。而在每个乐章里面有起承转合，很多不同的旋律使得大家一直处于种激情饱满，同时呢又很亢奋的状态，这种状态不会丢失。有些乐曲就是中间很亢奋，后面尾部深得很轻柔缓淡。而莫扎特不是这样，它的每一个乐章，每一个大章其实都有高潮叠起的地方，这就是一个很好的把一张很长的卷轴进行了分割。


我们其实架构师在设计的过程当中其实也是一样，你在分割的时候，你不应该把某些功能模块割的太小，而另外一块一些功能模块割的太大，这样就很割裂。你应该指我们比如领域驱动设计里面的领域划分，还是我们传统的 SOV 里面的 service 的划分，相对均衡的每个服务之间都有它的亮点，每个服务都能成为一个主域，核心功能，只有这样的划分，这样的展现才能真正体现什么架构的美。


那除此以外，除了分割以外，莫扎特还有一个特点就是交互性，因为是钢琴协奏就会有很多的声部。然后有些声部是由管乐和玄乐来完成，而主声部通常是由他的钢琴来完成的。但是在某几个乐章的时候，有时候钢琴会停下来，完全让管珏乐发挥它悠扬的特征。这种交互这种声部和声部之间交叠，乐章和乐章之间变化，它这也进一步使架构使它的这个乐章得到完美。


那我们在设计的时候其实要关注，就是你分割出来的模块是静态的，你如何让这些静态的模块之间以动态的形式之间交互，我们可以选择消息队列这种异步的交互方式，也可以选择什么 RPC rest API 这种同步的交互方式。在交互的过程当中你可以考虑有时候像 MVC 那种就是你对外提供一个 controller 然后对内使用 module 的方式，你也可以用其他的一种方法来实现什么来实现那种 reactive 的交互有很多不同的思路，不同的交互的形式使得你的整个架构会非常之灵活。这也是我们在新型架构当中，你越发会发现主模块的业务模块可能越来越单薄，但是外面有什么防腐层，有什么 adapter ， 有什么法萨德很多很多不同的这种设计模式的层次，使交互更加的完美，更加的解耦。


好。除了以上两点以外，还有两点是摸在他做得非常好的地方一点就是他永远是在思考中挣扎，在决策中使整个乐谱趋于完美。如果你有幸看到它的原稿，你会发现莫达特作为钢廷天才，他第一遍其实已经写得非常完美，但是他每次都会修修改改改一些很大的地方，同时也会改一些很小的细节。

这跟我们架构设计一样，我们既要去决策一些难点，比如说你选开放式架构还是封闭式架构，你是开源的还是闭源的？你是采用那种就是上云的那种 pass 的那种堆叠的方式，还是用自研的，完全用一些开源组件来实现高可用分布式。有很多时候其实决策非常之艰难，而且影响非常之大。


除了这些大的影响以外，还有一些小细节，一些非常小的这种，比如说类似里面的一种接口的改动或者是类的形式的改动，甚至于是数据库里面某一个表和另外一个表，它之间紧耦合还是松耦合的一个小小的改动，都会导致你整个架构的运行状态、运行性能差距千差万别。那就是这些细节上的决策决定了你这个曲谱是完美还是拙劣，你的整个架构是真正能高可用高并发还是只能在小厂里面运行，就这种决策其实也是非常之关键。那除此以外，莫扎特它的整个曲调，它的整个一生其实都是在眼镜，你可以看它前期的曲目是阳光的热情的。但是后期自从它的整个经济状况下滑以后，它曲目里面有一种悲壮的成分悲凉成分。但是仍然它的主基调还是那样前进。前进在前进。这种眼镜其实在我们架构设计里也是一样。当客户需求变化的时候，其实你什么你应该维持你整个业务架构的一种总体基调，但是你要跟着客户的需求进行少量的变化。就比如说我们拿那个传统银行业为例，银行业安全性的数据保密性，然后合规性这些都是银行业很强大的质量要求，非功能性要求。但是当互联网银行这种新型业务驱动出现的时候，你要把你的应用放在公有云上，互联网化，对外提供 API 经济。但是你仍然要保持你原有的基调前进即使悲创也要前进，就是这种六种思维的碰撞和互动下，莫德塔才成为了什么一代音乐巨匠。
