比如说我们举这个类似于这个滴滴这个打车系统的这样一个打车公司为例，在五年、十年之前，这个打车公司是什么？是模仿传统的这种就是电话呼叫公司，它什么是把它所有应用系统做成一个中间那个六边形蜂巢这样一个功能，它是一个什么整合体？然后对外它会怎么样？它会既提供像这个右左上角的API，左下角的 UI 也会提供一些，比如像数据库访问提供一些额外的这个适配功能去跟第三方对接。那所有这些功能其实什么都是什么融合在它的整个业务体里面。


而业务体没有按照实际的这个功能拆分里面这个业务的运维人员，通常比如像上面的数据库有DBA，那 API 有这个研发人员对外的这个各种各样的那个接口，有什么对外的专门做应用接口相关的负责人来负责整个 it 还是按照传统的，是吧？运维开发、产品管理、业务管理这样的这个 it 结构，这样组织结构适合传统的这种什么应用，变化不大，但是希望就是以技术为导向。


懂数据库的人，不懂网络的人，不懂开发，以这样为导向的一个结构，但是随着业务需求的变化，比如像滴滴这种，就是软件迭代非常快，它有打车是吧？有拼车，还有专车，这种快速迭代的功能出来以后，每个业务模块需要快速响应市场需求，快速开发、快速测试、快速运维，还是稳定上线。


那这个时候原来的传统的这个组织结构好像有点跟不上业务的需求，就出现了，像右边这种把部门分成什么专门管和人沟通的，就是说乘客部门，然后和车沟通的这个什么，就是驾驶部门和路面沟通的，就是什么地图部门和告警沟通的这个通知部门，和支付沟通的支付部门和记账沟通的 billing 部门，那这部门里面既有什么业务人员，又有什么产品开发，也有架构师，也有运维团队？所有这些不同的这个部门之间，它内部有很有各个的什么技术栈，那形成了一个什么统一的接口？它对外可能是一个 API 调用，可能是一套规范协议，也可能是一个图形化界面。


当用户出现任何额外的需求，或者出现任何这个 bug 的时候，很快的响应给我们的系统。而系统的这个业务人员和产品经理响应给我们开发做出快速的这个新的这个功能的迭代或者是 bug 的修正，然后发布出去，然后进入新的运维。那这种组织架构的变化导致我们什么？导致我们希望我们的 it 架构也都相应的变化，同时 it 架构的变化又推动了组织架构变成那种就是竖颈式的，一个小竖颈。

那贝索斯每次看到一个会议室里面，这个会议室里面要定的披萨超过两个，他就会，嗯，感到不是很满意。那原因是什么？是因为贝索斯很抠门，不是因为他感觉一个披萨通常在美国，大概这个不是胖子来吃的话大概是 6 个人，两个披萨就是 12 个人的披萨团队，他的沟通成本是 12 乘以 11 除以 2 已经是个什么，就是数百条沟通线，就当你这 12 个人不在的一个会议室里面，你们要通过电话，通过这个微信群，或者是通过 slack channel 这种方式沟通，就已经形成一个上百条的沟通线的。这个错综复杂的沟通之前聊到过什么多渠道沟通导致什么信息失真，这个问题就会出现。当你人数超过 12 个以后，几乎没法保证信息是完美的沟通的，那这样的团队是无效的或者是低效的。


所以贝索斯认为一个团队超过一个 Dozen 的人就是 12 个人，一个 Dozen 超过一个 Dozen 人就不合适，那少于 6 个人感觉好像太少了，这两个披萨通常就是 6- 12 个人之间，刚好能吃饱，这样团队在一起工作非常高效，即使远程办公也不会有太多的影响，所有的服务也应该切分成，刚好让这样个 6- 12 个人团队能够完美胜任的地步。这就是微服，这就是符合什么组织架构，那同时微服要去符合业务组织架构也符合业务，这样三者之间形成了一个互动，出现了微服务化、团队缩小化、业务导向化这样的趋势。
