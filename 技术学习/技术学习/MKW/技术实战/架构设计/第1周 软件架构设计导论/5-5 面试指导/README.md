大家是不是聊了很多点诶？聊出点就能跟面试官互动了，但是这道题光点还不够，最好能把点串成一个线，形成一个完整的流程。如果大家是结合自己的实战来聊也很好，但是记住最好什么有一个方法能帮你串一下，否则你的点就有点支离破碎。面试官不容易。什么不容易？把握你的核心的思路。


假设我们用 ABSD 来串我的一个项目，或者说用 ABSD 理论来聊这个题目的答案，那通常会怎么聊？先聊一下需求是吧？怎么样把握什么核心的功能性啊？非功能性需要，然后把需求变成层次化，变成模块化。那之后在设计环节我们会用到一些什么架构的设计原则？我们会大量使用架构的风格去找一些模板，帮助我们快速地进行架构设计，那之后有一个标准的文档化复审的过程，在实现的时候我们会用构建库里面的很多构建，帮助我们快速的什么进行这个代码的开发工作。
而所谓的项目完成以后不是结束，而是一个什么新的迭代的开始，我们通过多次迭代演化的方式，把一个需求或者说是一套需求完全的这么实现出来。那这就是一套方法论，有这样的方法论支撑，你讲内容每个点就会比较亮，它就会串成一条线，也容易什么面试官去理解你的思路，这道题有没有加分线啊？当然有。在你讲整个设计流程当中，如果你有设计原则，同时你能讲清楚为什么要用这个原则，诶？这就是加分。另外架构风格的飞扬老师讲了十几个架构风格，你为什么选这个？在这个模块当中，这个模块为什么又换了一个风格？能把风格讲清楚，以及风格具体怎么样采用模板来落地讲清楚。


唉，这也是一个亮点，那最后一个亮点什么是演化？有很多的这个架构式面试，大家都听到我原来怎么设计，最后怎么落地，哎，结束了少一句话之后，我又什么调整了哪些需求？然后进行了进一步的演化？我的这个架构什么成为了一个比较通用的方法呢？又给什么更多的项目进行复用？唉，这里是不是就生生不息的循环起来，这种感觉更好？这就是证明什么？你是一个什么成熟架构师？你会复用一些原来的知识，然后在原来知识上不停的演化和迭代。

如何在架构设计中选择合适的架构风格和软件模板。诶，这道题呢，是不是没有像前面那道题这么开放，稍微收紧了一些，要聊一聊风格，聊聊模板。


好， 321 请作答。如果你的答案里面聊的都是风格，比如说飞扬老师讲了十几个风格。如果你在答案里面聊的都是模板，我要用了 spring boot 的框架作为我的模板。我用了什么网银交易中间件作为我的模板，甚至于我用了某一个库、某个包作为模板。那这道题其实你被误导了，你答题答偏了，这道题问你如何选择？是什么是一个？什么？是个决策论？对不对？决策论？而决策论的题干通常是什么是决策的原因？我在我们这节 ABSD 课里面的讨论里可是重点提了一下整个 ABSD 是什么驱动，是需求驱动，所以他的决策所有原因都在自己需求。


正如我之前的那个实际案例里面聊的内容一样，一个需求什么是功能性？需要我要实现什么银行系统里面的证券交易功能？一个是质量需求， 96.95 的高可用，以及什么有些限制，比如银监会的限制，正是这些需求堆叠起来，导致你必须选用某些风格，同时你必须什么使用某些模板来加快你的开发进度？所以为什么选择风格啊？为什么使用模板是很多时候都是需求论导向的？这就是一个决策论里面的需求驱动想法，如果你从这个角度谈，那你就切中了题干，基本上能聊得比较畅快了，那同时也希望什么加分项，就是你在聊的过程当中不要只是聊需求，同时你要把这个需求对应的风格讲出来，为什么是选这个风格，而不是那个风格？同时这个风格底下有哪些模板？你经常使用的这些模板有什么好处和劣处？好讲完这些以后，基本上什么一条线，对吧？从需求到风格再到模板，已经讲得很清楚，所以这道题大家是不是也答得不错啊？如果没有考虑到需求去中，请看一下我 ABSD 本章概述章节。

能不能大致画一画？赵启在面试时候有时候会问，有时候不会问，此类题目叫什么？叫画图题，通常面试官不会像笔试这样出一道这么详尽的题目，它会口述表达一个意思，然后让你的画画图来聊一下，你对这种情况怎么设计。


既有创建新订单，也要更新订单，同时也希望在过程当中能核查一下这些用户到底存在不存在这种情况下，这几个用例你该怎么设计呢？这道画图体其实看上去很简单，其实有一定难点，我们 321 请做答，大家是不是做出来了？难点在哪里啊？难点在描述清楚一个东西的关系。


其实有很多种描述方法，通常碰到这种用例，你看到有用例的词样，或者说是我要有这样一个功能，通常在面试过程当中有听到有功能用例，用哪种描述关系比较好？用 UML 的模型里面的描述，比如说常见的关系有什么？有包含、有扩展、有泛化，当然你也可以用什么我们 Java 代码里面，比如说继承实现也可以，但是通常大家在描述用例、描述需求、描述功能的时候，是在需求分析那个阶段，所以这时候用包含扩展泛化这样的 UIMR 描述词更能体现就是架构师的那种特点，相对抽象一点的特点。


那这道题如果要打高分，通常在聊的过程当中有一个黑板，或者有一个这个白板，我们会拿一支笔，对吧？在旁边画一些我们的图，把我们的思想描述清楚，还有分析一下怎么设计更好，可能碰到哪些坑对不对？那这道题的答案到底是包含还是扩展，还是泛化，还是什么其他关系啊？核查用户账号到底和他们什么关系呢？我们看三张图，看完这三张图你再回来看，其实答案就呼之欲出了。

这两个之间有什么关系？通常我们认为是这个从右边往左边指扩展，也就是考试是个基本功能，你就考就好，但是有时候你要通过充钱的方式来帮助你考试，这是一个扩展，在某些时候我需要扩展到另外一个层次上去。就是扩展好，我们再举个例子，假设我们要注册一门课程，有两种方法，用电话注册，一种网上注册。这三个 UML 的用例时间什么关系啊？应该是这样叫泛化，泛化就是用细节的东西泛化了它这个抽象的东西，那如果是我们 Java 里面可能很简单，也就是什么？其实就是继承我的这个电话注册类。什么继承了什么课程注册的这个母类，对吧？我是你的子类，那一样我可以换一种方法来继承你，用网上注册的方法来继承你的课程注册。


好，泛化扩展中包含，其实经常就是我们在用例，在我们平时的这个 UML 画图里面经常用的这个 3 个关系包含的是虚线加一个这个书名号include，那扩展也是虚线加一个书名号，而泛化是一个实线带一个空心箭头。前面我的课程里面，大家如果回过去看，有大量的实线带一个空心箭头，其实那些都是泛化，都是继承。
