那这个架构最后落成一个文档，同时还要经过一定的复审。那这里的复审是指内部的架构师、评委会的复审，以及外部的，比如像产品经理、业务方，或者是在行业专家、领域专家的复审。那最后是什么？我们要把这个实际的架构真正的代码实现，实现完了以后就进入什么架构演进不断迭代的过程，这个跟普通的软件工程没什么区别。那重点提它这里有两个循环，叫小循环。他是强调什么？就是在架构设计过程当中尽量提前的去什么用复审的方式发现一些问题，同时还有一个大循环，就是在代码实现以后重新去评估需求，看看有哪些需求没有被命中，或者需求又有了什么样的变化，来什么来改进我们的架构。那这类流程不管你是小循环、大循环，还是具体细节，和普通架构没什么区别。


那 ABSD 到底亮在哪里呢？我这里从 4 个点来进行分解，需求设计、实现和演化。这四个点各有一些小亮点，使得 ABSD 跟其他的架构设计略有不同。

比如一个 Excel 表格或者是一个 word 文档都可以。我们叫它需求库，然后把它提取出来，通常需求提取出来有一个什么逻辑？分拣架构设计分层很复杂， ABSD 不是这样做，它叫什么？叫小鸡啄米，小鸡快跑，就是我们用很小的动作来完成一定的工作量，然后不停的迭代这种思路。我们看看怎么样小计划，它直接就生成类图了。当需求感觉出来大概要做什么的时候，我们不经过复杂的设计分层，先全部把相关类图全部都画出来。有一个需求我就给你设计个类，全部画出来，不管你对错，先画出来再说。然后再把类进行分组，看看哪些类可能用一个模块来实现。分组完了以后，我们再把这个类具体变成可以打包的构建。那这个时候其实没有代码设计，它只是什么在逻辑上要设计一些一类的样子，它们的关系，然后它是怎么样组合成包的？在这设计完了以后，这一套东西会以什么一个有标识的构建记录下来？我这又需要哪些？列一些列表或者有些图来描述，我们后面会看一张真正的类图，画完以后就拿去什么把需求评审一下，这个评审还没有特别去聊功能设计，只是聊聊看。

这是什么？这是唐老鸭的类图，你可以看到从上往下，首先我们是一个动物，是需要氧气和水的，然后它可以什么？它可以泛化，或者在我们的这个 Java 里面叫什么？叫做那个，我们可以用一个鸟来什么来继承我们这个动物这样的一个类，对不对？就相当于是一个它，鸟是动物的继承，然后鸭子又是鸟的继承，那最后唐老鸭又继承了什么？鸭子这个类的功能就这样层层泛化，或者是什么层层继承。
