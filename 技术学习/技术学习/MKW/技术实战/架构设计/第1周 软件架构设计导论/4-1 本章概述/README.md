首先what？什么是主流的软件架构风格呢？我们这里要介绍五大架构风格，分别是数据流风格、调用返回风格、独立构建风格、虚拟级风格和仓库风格。那这五大风格是 Gallon 和 shore 两位大师从众多的软件的这种体验软件设计的框架中提取出来的，可以说基本上涵盖了架构设计的方方面面，所以是一个很经典的一套理论。那有了这样套理论，如果我们用在实际，就会提高我们的架构质量，使得它能够靠拢一个已经很成熟的架构设计，减少很多错误发生的可能性，同时可以重用很多代码，重用很多框架，所以大大提升我们的效率。


那我们如何来讲这五大风格呢？那这次和前面有一个不太一样的地方，这次是什么理论？结合实际这个理论就是我会把五大风格通过五个图文 MD 分别详详尽尽的给大家阐释每一个大风格下面的子风格，每一个子风格的特点，然后整个大风格里面的 e 和同时还有各种各样的实际场景，什么样的场景下用哪一种风格更合适？那通过这样 5 个图文， MD 基本上让大家能够了解到这个风格的特点和什么时候来使用。同时图文 MD 有个好处，就是大家可以在地铁上看，可以在出租车上看，也可以在闲暇时候反复的去温习这五个风格。


对后面的所有的包括架构设计，包括我们的什么架构的治理，也包含我们的很多的评审工作，其实都会围绕这五个风格，所以说是非常关键的五个点。好，那我们这里呢，还是给大家普及一下这 5 个点的侧重点，方便大家在后面看图文 MD 的时候有一点感觉。

大数据世界里面常用的一种风格，那它也包含两个子风格，一个叫批处理，那批处理风格如果大家用过批处理，比如说我们跑一些狂 job 或者类似的一些任务，都是一些批处理的任务。批处理任务并发性强，时间点和控制点强，同时数据流也非常大，那这是它的特点。


另外一种风格叫什么？管道过滤器风格，那大家如果是在雷克斯里面跑过一个竖杠 pipe 管道服也跑过其他的像是阿帕奇的streaming，或者跑过一些流式处理，通过这些管道和流的感觉，你其实就能找到数据流，那种时时刻刻在流动，时时刻刻在处理的感觉。这两种感觉共同形成了数据流风格。

我们看一下它三个子风格，你就可以了解它的本质。它第一个子风格说的是什么？一个程序经常是划分成主程序和子程序，主程序去调用子程序，子程序每次完成任务以后返回给主程序，那是不是就是我们最常见的面向过程的程序调用？那有了面向过程，大家很容易想到第二个子风格，其实就是面向对象，所有的内容全部封在对象里面，不管你是封在成员变量还是方法里面，它以世界为模板，以一个人、一个动物、一个实体为模板，开发我们的软件。


第三个风格，那就是在主程序、子程序和面向对象上，在高屋建林重新进行了解耦，形成层次感。那我们有很多不同的层次，比如说我们有网络有 7 层，我们那个服务器有 BS 结构、 CS 结构这种客户端服务器、浏览器服务器的层次分层。我们的 Java 代码里面可能会有MV、 seed 分层，也可能是领域驱动设计的四层分层，我们用各种各样的、竖的、横的层次结构，这些其实都是什么调用返回风格？所以调用返回风格其实说的是一个主系统怎么样去解耦成子系统的一种风格？

这个名称其实也有点难，我们来这样理解，一个构建其实就是相当于一个模块，那这个构建和构建之间应该怎么样进行沟通呢？这就是独立构建风格的两大风格了。其中有一种是什么进程间通讯，不管你是rest、 API 还是RPC，你每次是一个过程去调用另一个过程，一个进程去访问另外一个进程，一个模块去靠另外一个模块，这种有指向性的就是进程通信风格。


另外一种是事件驱动风格，以消息队列为主，我们广播一些消息，然后让这些消息在这个消息池或消息队里面流转，谁爱监听它，谁就监听它，谁爱处理它，谁就处理它，消息的这个产生者和消息的消费者他们之间并不互相直接沟通，他们也互相不了解，那这种风格就是事件驱动的系统，那么后面我们的领域驱动设计里会详细聊什么样是事件驱动的一种架构思路，什么样是读写锋利的一种架构思路，这些都是非常吻合事件驱动系统风格。

，是不是我们一台一台的物理虚拟机呢？其实不是，它比物理虚拟机稍微小了一点。还有什么虚拟机啊？你想想看是不是最容易想到就是GVM、 Java 的虚拟机？对了，这里的虚拟机其实就是一程序的定制化的处理系统，那这个系统呢？可以是我们的 GVM 解释器，也可以是其他各种各样的。


你自己开发了解释器，你自己开发了语言的处理器，或者是其他的一些解释处理功能，那它只要是能对一些套定制的程序或语言进行合理的承载和处理，那它就是一个解释器风格，那与它相对应还有一种类似的风格，也是承载一些特殊的需求，一些特殊的程序，但是它是什么？它是偏向于决策性的，很经典的就是我们的规则引擎。


当业务人员希望什么？希望提供一个很灵活的，比如说我们的什么，我们的这种就是优惠券政策，我们的这个报价策略的时候他不会是希望他自己写 if else 与居时性，因为业务人员并不是 it 人员。那这个时候我们提供一套规则系统，让它能够拖拖拉拉就能把 if else 在我们的界面里给表示出来，并且能跟最后台的实际的业务系统对接起来，完成我们的优惠券的设置，我们的定价系统的设置，那这样的规则引擎也是一种风格，那就是基于规则的系统风格，也属在什么虚拟机风格这种定制化程序的解释执行的大类之下。
