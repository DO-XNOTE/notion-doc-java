那他呢？是看，嗯，我们的同行都在上云，而且你们都做得风风火火，是不是我也如法炮制，把我的 it 系统搬上云上呢。那这家企业开始什么尝试？联系一些跟云有关的企业，同时采用了一些容器化技术，那它采用的是 cloud foundry 的容器化技术，然后通过它进行了一部分的改造，开始出窥云尽好，我们看看它第一步是怎么样找出一支强大的队伍，把它打造成完整的Devops，这种价值，这种文化的团队。
然后这个团队里面有开发的，有运维的，也有发布的这些不同经验的工程师，打造成一个团体。然后他们共同基于 CO 方锥的开源的、容器化的解决方案打造一个集监控、集中间件、集基础架构、自动化于一身的私有云平台。好，那这章节我会聊到好几个名词，比如像私有云、公有云、混合云，大家应该是听说过，没有听说也没有关系，因为下节我们会详细来仔细介绍。好打造完这样一个平台以后，时间一晃已经一年过去了。那之后是不是应该我们把很多应用搬迁到云上？那这家企业怎么样来搬迁应用呢？他选中了一个很重型的方法论什么事件风暴？那这个事件风暴什么用了？它来驱动领域驱动设计的整个架构设计。同时他挑选了一个比较强大的团队，我们姑且叫他机械化成团队，或者叫机械化部队，是整个公司里面精兵强将最多的一个小组。
那这个小组把他们的这个应用全部采用 cloud native 云原生的思路进行开发，整个团队用什么？用极限编程的方法所有任务细化，然后以什么？以测试来驱动，同时每一个任务都是结对编程，两个人同时来开发，你开发我帮你查错，我来开发，你也帮我查错。
这样一套什么完全是完美型的 Devops 文化的这样一个团队，一个开发方式，大概又用了小半年的时间，终于把第一款小应用、第一个小领域发布到了我们这套 Paas 的私有化云平台上，欢欣鼓舞，整个团队，整个公司都士气大涨。我们上云非常成功，经历了一年半的痛苦，终于看到了成功的希望。然后下文大家猜一猜情况什么？急转直下，当其他的应用陆续上云的时候，我们发现，哇，机械化部队的应用好像跟其他应用不一样。他们天生就有点解耦，他们天生就做得很完美，同时他们的团队本身就是很云化的一个强大的团队。
那其他团队怎么样？很多团队还属于小米加补枪机械化团队是用坦克跟飞机，其他团队是小米加步枪，同时很多的领域，或者说很多的子域，它的后台数据库不光仅耦合一套数据库共享表，同时数据库里面还有很多存储过程，触发器外键非耦合度非常高，而且在事件风暴领域驱动设计的时候，每一个部门是独立的，自己做自己的。事件风暴导致什么？你的那个领域解耦和我的领域解耦不一样，所以这个表呢？你也不要，我也不要，大家都不要，都指望对方出现了很多混乱，这是小米加步枪。
还有一点就叫病号伤员，所谓病号伤员其实就是一个英文单词叫 Tech debt 技术栈。大量的企业其实都有它自身的技术在那有些传统的，比如说不适合 spring boot 的发布方式，比如 yeah 这种传统的软件包发布方式，以及很传统的老式的这种什么 Jenkins 1.0 这种发布方法，什么通过集中化的 CICD 团队来进行发布和测试？没法做到分布式发布、自由发布等等，有很多老旧的东西跟新的整个框架、新的平台进行对接的时候，产生了很多冲突。
但是你不能全部依赖于新平台，因为什么我们的系统是在线运营的，每年有几百亿的这个资金在这上面流转，那这个时候你要保证所有的功能向后兼容，同时你所有的发布也不能干扰原来的系统，这个时候对整个的系统的运维产生了很大的压力。同时因为什么？因为我们已经跑了一年多的，什么这个 cloud foundry 容器化平台，这个时候你会发现 cloud foundry 的版本发生了翻天覆地的变化，大版本升级平台大版本升级是一个很大的工作量。同时大量的应用还要上这个平台，在上的过程当中还有一些什么为应用做的定制化的开发？我不喜欢用 Rocket MQ，我喜欢用 Rabbit MQ，我不喜欢用卡夫卡，我喜欢用 active MQ。不同的这种中间件要载到这个平台上就产生了很多冲突，然后多个领域之间还要争到底用什么样的消息中间键互通才更合适啊？等等等等，整个团队什么反而被我们这套 Devops 团队的 Paas 平台给拖垮了？整个企业的发布节奏反而变慢，自动化没有提高，运维没有，轻量级发布并没有变敏捷。嗯，上云的所有价值似乎都没有达到。那这个时候会发生什么样的事情呢？其实前面已经说到过了什么？有人，对吧？上马，然后带动我们上云下云，但下云的时候必然什么，有些老大们就落马了，就出现了大换血、大换旋。


完了以后新的领导团队重新评估了，评估完了以后，嗯，不光是业界在上云，其实我们也是需要上云的，我们需要更快的敏捷，我们需要业务的解耦。因为之前的一年半我们已经明显感觉到不够敏捷，不够解耦导致很多问题拖累了应用，拖累了业务。

那这个时候如何用新的思想而改变原来的上云的套路呢？那当然就是什么找到合适的架构师了，就像飞扬老师这样的架构师就在这个企业里面隆重登场，然后我们重新换一套思路来上云怎么来弄呢？先不尝试去搭一套平台，也不尝试去找一个机械化部队，而是这样我们整个公司横向的、很均衡的来评估一下我们整个架构在什么样的级别，我们有什么样的业务需求，需要怎么样的敏捷，需要我们的云平台提供什么样的SLA？什么样的QOS？评估完了以后，梳理完了以后，我们对自己这个脉基本上已经把握清楚了，那然后做第二步开始做方案设计的时候要仔细选型，到底是选公有云、私有云还是什么？还是混合云，我们选的是先混合再公有，通过混合云这个过渡阶段，最终变成公有云，从而实现什么所有的安全放在云上，所有的运维也放在云上，降低运维成本，降低服务器的什么一次性购买成本，那整个过程当中要向前兼容，对吧？或者也叫向后兼容，就是像之前的这个系统兼容，所以采用混合云逐步升级、逐步迁移的方案，中间没有私有云了，之前搭的那套私有云完全铲掉，完全抛弃它，反而变成了一个技术债。
好，这样一套方案设计完以后，我们怎么做？我们开始分阶段的勇敢的实施每一个阶段都是什么？有一部分业务通过开发，通过准生产先搬上去，然后把生产也搬上去，逐步的一点一点的变成混合云，最终向公有云的整个完整状态进行转换，在过程当中每一步都是有完整测试的，所有的生产投产之前都有很精准的测试，给什么？给团队信心？同时我们不是在找机械化团队上云了，我们是每个团队都有几个微服务，同时或者说是并行的在向云上迁，这样保持什么不要出现。
大家是小米步枪，有一个团队是坦克飞机，这个什么，这个是团队是不能很好的配合的。我们应该怎么样呢？所有的小米的步枪换成半自动步枪，然后所有的半自动步枪全部变成连射机关枪，大家每一个冰射手上拿的都是机关枪，是不是战斗力反而更强大？好，到了第五步，就是什么一个系统的上线切割，在切割过程当中我们采用一些双写，采用一些什么黑启动等等方案，尽量保证业务感觉是透明的。
但是其实系统是在逐步迁移的，到了最后就是打通上下游，打通云上和云下，实现整体监控，我们在云上搭建一套新的监控平台，包含性能监控、链路监控、日志监控、业务监控，已经把它们串联起来，完整的实现混合云的总体监控，降低成本，降低运维的投入，同时增加发布和迭代的速度。
好，聊完了，两种不同的思路在一家企业里面先后都实现过一个是什么？嗯，好，大喜功，搭大平台，找精英部队上云，另外一个先整体把握整个企业的节奏，同时了解你的实际需求，找到合适的云平台作为你的目标，逐步转型进行切换。好这两个上云下云的案例，其实最主要想告诉我们，就是先思考再动手。好，下面才是真正的思考的核心，就是你要看一看到底应该上什么类型的云，以及语音平台，有哪几种类型呢。语音平台的分类介绍就是下一章节的内容，大家敬请。
