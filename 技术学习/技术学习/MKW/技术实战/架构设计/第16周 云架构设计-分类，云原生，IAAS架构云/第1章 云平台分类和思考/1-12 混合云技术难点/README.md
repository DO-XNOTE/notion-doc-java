也要慎重地使用混合云。我们看一看首先的难点在哪里啊？在安全性既是混合云的优势，也是什么混合云的难点，那哪些难点呢？统一的账号管理就是一个很难的点，比如我们传统的数据中心大部分都是用什么？用 Windows 的ad，或者是用 Linux airdap 作为什么用户认证登录的核心，然后把这套系统跟我们的不同的平台进行对接，实现单点登录。
但是一旦我们进入了混合云，公有云平台可是有独立的用户账号管理系统的，怎么样实现公有云和私有云的共同的单点登录？这就是一个难点。同时单点登录通常还不是一个因素，就不光是有密码就可以登录的，在某些场景会触发一个什么叫多因素登录，通过你的邮箱，通过你的这个手机的短信的这个注册的内容，或者是通过你手机上装的一些多音素的APP，拿到一些 token 来混编的方式来确定你的身份，最终允许你进行单点登录。
所有这些复杂的功能，云上是一套，云下是一套，一旦两套脱离，就没法很好地完成混合云了，所以必须通过一些技术，比如像Samo，比如像是 Oauth 等方法来什么实现真正的统一身份认证、联合认证、联邦认证和单点登录。对，这是什么云平台？特别是混合云平台的一大难点必须要克服。除此以外的某些合规性的场景会导致我们没法把业务放到公有云。比如说假设我们是跟海外的一些支付系统相关，尤其是跟几大什么信用卡系统相关，像Visa、 master 等等，这个时候你所有的交易系统必须要合到什么 PCI 审计的规范？那这个规范很复杂，在传统数据中心你可能花个几个月或者半年、一年的时间已经搭建好了，但这次你一旦上了公有云，一旦进入混合云，你要让云平台提供相关证书。
同时你的云平台的新的搭建的行为规范，要重新去找一次这些安全的审计，而这些安全审计有个特点，它特别强调隔离，云平台很多的隔离是不规范，达不到指定的什么 PCI 支付牌照要求的。这个时候你要有大量证据的证明，我在云平台用了某某某最佳实践实现到足够级别的隔离，所以一旦碰到支付系统上云，难点其实很大。
除此以外，我们通常整个的业务系统和什么财务报表系统都有外部审计。

这些审计一旦发现你的系统上了公有云，它会提出更苛刻的要求，所以对混合云的合规其实是一个很大的难点。那除了这些基本的支付和财务相关的信息以外，个人隐私的合规也是一个难点。比如以 GDPR 为例，当年 GDPR 当中隐含一个条例叫什么？叫欧洲信息岛。所谓欧洲信息岛就是我希望欧洲用户的大部分数据会保留在欧洲某一个城市数据中心。那这个时候你自建数据中心很简单，你在那边搭一个数据中心，保留它的某些业务数据就可以了。
但是你一旦上云以后云平台不一定在欧洲的指定城市都有它的数据中心，导致什么？你某些跟GDPR、跟个人隐私相关的业务没法上映，虽然新的 GDPR 已经没有这部分要求，但是各个国家有各个国家的规范，有些国家还是对什么个人隐私有了指定城市、指定国家数据的保护要求，那这个时候混合云的处理会相当困难，就大家可以考虑还是通过一部分什么私有云为主体，公有云来削峰来填谷的方式来实现混合云。这是对某些特殊的隐私相关的业务系统跟支付相关的业务系统的一种安全性的。
什么妥协好？除了前面聊到的这些以外，还有一个大的安全难点，就是网络隔离，传统我们的网络什么是独立的一套网络？我们可以找一些什么海利陆战队退伍军人来进行保安工作，从而实现一个完整的物理隔离。但是上了公有云以后，物流隔离不是你管，是云平台厂商在管，那这个时候你只能实现软件隔离，比如像防火墙，比如像跳板机、堡垒机等等这些操作手段。其实什么是有一些限制的？它没法实现类似于物理隔离的强大隔离，所以这个时候怎么样去向物理隔离靠近？怎么样实现更加的安全和强大的什么网络防御能力，也是对我们混合云的一大挑战。
好，除了安全性以外，运维其实也有很多挑战，我们来一一分享。第一点就是发布的挑战，原来不管你是用 Jenkins 还是什么Git， actions 等等各种套路，通常你是一套 CICD 发布到一个数据中心。但是一旦进入混合云以后，你有多套数据中心，至少私有云，公有云就是两套，这两套你是用两套CICD，两套这种弓箭库还是一套呢？那妇炎老师跟大家强调一下，必须用一套。
而且如果你是什么，两边的应用是什么？是相同的应用发布，不做这个应用的这个垂直划分的话，所谓的应用版本也要同进同退，那这种情况下才能真正实现一套混合云，提供一套完整的企业服务。所以怎么样把 CICA 打通，怎么样实现版本同进同退，也是我们运维的一大难点。
那除了发布以外还有什么监控？不管你数据流是往云上流还是往私有云流，你的什么 trace ID 必须能穿透中间件、穿透消息队列、穿透 API 网关、穿透VPN，能够实现完整的链路追踪。除此以外，你的日志，你的用户行为追踪也都要能够什么？统一混编到一处付杨老师推荐是统一混编到公有云平台，因为公有云平台的监控套路更成熟、更规范。好，那除了统一发布统一监控以外，资源调配也是一大难点。原来我们是什么？我们是 it 系统，是有一个统一的采购计划，所以你的容量配得很漂亮，也很规范。一旦上云以后上云是很弹性伸缩的哦。所以每个部门研发部门是很自制的，就导致什么经常会有容量爆料、钱超标的情况。就这个时候需要有一个架构师团队通过每季度审核、每年审核来整体把握资源调配的规范。另外弹性伸缩是必须要设置好上线下线，同样对部门要进行考核。
另外还有很多的云平台资源，比如说是强大的 no SQL，它能够支持多个什么领域的不同的业务系统，多个数据库共享一套Nosql，但是当某一套 Nosql 里面的某个业务进行，比如说跑批处理，而且恰好没有索引的时候，有可能导致一个业务群影响了其他业务群，这个时候怎么样对云上资源做 QAS 的隔离也是一大难点。
所以这些都是真真正正飞扬老师在实战过程当中发现过，没有统一发布，没有统一监控、没有资源调配，出现了大量的问题，所以强烈推荐在运维层必须要有什么强大的思维，必须要有强大的 Devops 文化，才能够真真正正上混合云。好，聊完了这么多难点，运维也好难，安全性也好，是不是就打了退堂鼓呢？大可不必。下一节我们要给大家重塑信心，举一个最难的例子，什么金融行业怎么样上混合云的例子，而且是一个很成功的案例，来跟大家分享一下上混合云的心得，大家敬请期待。
