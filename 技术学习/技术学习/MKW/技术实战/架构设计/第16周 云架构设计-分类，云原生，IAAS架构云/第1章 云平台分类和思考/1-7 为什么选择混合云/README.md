在架构设计维度的和架构设计里面当时聊到过的一个企业，它一开始所有的业务流是流向自己内部的数据中心的前端服务器、后端服务器、数据库和缓存，但是跑了几年以后它发现，嗯，云上现在好像好东西很多，我是不是利用云的安全的这个强大能力，把一部分业务放到云上，然后我利用云的什么原生的这种弹性伸缩容器化这无氟化能力进行一些业务的扩展。
那除此以外还有什么？还有两更强大的有用的功能，那一个功能是什么？是这里批处理引擎所谓 p 出的引进是什么样的特点呢？通常白天他不用跑业务的，在晚上 12 点到凌晨 2 点之间完成所有的金融结算、账务审计等等功能。这个时候并行性非常大，不是几十个节点，而是几百个节点，几千个节点在那里跑批处理任务，但是时效性很强，就那么两小时给你跑，跑不完你就放弃了，不能再跑。
有后续两点钟以后有其他的安排，那这种时效性、并发性很强的任务就是属于什么叫周期性应用？它不适合放在我们数据中心，如果你要放在数据中心，你要为它额外去买一个几百上千台服务器，放在那里，平时都不用，就那么跑两小时，很亏，成本很不划算。那这个时候放到云端就很方便，每天 12 点的时候准时也及时启动了。
跑完任务以后人家机器自动的就关机了，那这是周期性的应用，那除了周期性应用以外，试点应用也经常是放在云端的所谓试点，比如说大数据，比如说人工智能，那这家企业就是它是什么？从传统的在电商平台向一些智能化、用户感受化的电商平台做转型。这个时候他要用到一些 machine learning 人工智能的引擎，也要用到一些大数据的功能，所以他选了一家什么云平台里面对大数据做得特别好的谁？谷歌的 biquery 是大数据平台什么？一个民生非常响亮的这个大数据分析引擎，同时它的 machine learning 提供了很多类似于 Alphago 的这种人工智能功能。这些功能都敞开了，让什么让你云平台的租户来使用？所以为什么不把所有的这些试点应用，把那些周期性应用搬到云上？但是我的核心业务仍然在云下呢。这就是一个企业的例子。

也就是经常你要横向的去观察多个不同的边界上下文，不同的子域之间的成本，整体把控整个企业里面的核心技术栈。那这个时候你经常会和 c 什么什么o，会和一些总监们去聊一个这个问题，我企业到底现在有多少现金流？我应该怎么样去做投资？那很多的数据机房是怎么投资的？年初 p 一笔款项，然后三四月份这笔款项到了，可以买大量的硬件设备了，五六月份这个设备一次性采购。
所以这就是 CAPEX 一次性投资成本，那我们原来的传统的数据中心全是这种 CAPEX 一次性投资，三五年淘汰，然后第二年去买一批另外的设备，也是一次性投资，就每年都有一个投资的高峰点，可能是年终的时候，年初的批这个budget，批预算，年终是个高峰，年底是什么？是想办法把这个原来的这个预算全部花销掉。
那除了这种投资情况以外，还有什么样的投资情况？持续性投资，比如说我们的水费、电费、云员工的工资等等。另外一个新的持续性投资在云平台里逐渐出现，那就是云平台租赁费用，这个租赁费用就没有一次性投资了，你不需要一次性去买一堆服务器，运行 3 ~ 5 年，你是什么？每时、每分每秒在云上租设备，这有什么样的好处？ OPEX 是可以跟业务绑定的，比如像我们新冠期间，我的业务从原来的，比如说，嗯，千万级的业务，对吧？每天千万级收入变成百万级收入，缩水了 9/ 10。那这个时候我的是不是 CPU 内存也可以缩水 9/ 10？那原来的投资其实是收不回来的，在数据机房里买完的设备你可能退不出来的，但是在云平台收入在减少，是吧？我的资源储备也可以减少。
当什么我的情况、经营情况提升的时候，我收入在增加，我的什么我的压力在增加，这时候我的资源也可以增加，所以完全可能可以来应付一些经营性风险。特别是类似于新冠期间，类似于一些什么金融风暴的时候，这个时候我可以通过云平台的租赁实现资本的成本控制和我的业务成正比的方式来渡过难关。所以如果是说成本控制来看，飞扬老师很建议什么我们采用更多的持续性投资，也就是租赁形式的投资来消减经营风险。

还有一个关键点就是平滑转型的核心就是混合运，如果一家企业传统的数据中心它未来要上公有云，那其实不应该是一步我要的就变成公有云，那这样风险很大，一次性做数据迁移非常难，整个数据中心瞬间转到公有云上也非常困难。
那比较好的方法可以学什么？可以学我国的这个航天视野和我国的高新科技，它能做到什么？既是能上九天揽月，然后我们能发火箭，同时也能下五洋捉鳖，我们还能什么让潜水足潜到足够深的深度？这样两个思路是在什么？我的国的整体科技里面是同时进行的？下海和上天同时发展，那同样的道理，我们数据中心其实应该什么既能够实现，我们很容易转向公有云的能力，同时也要保持，什么时候当公有云价格很贵了，是吧？当这个阿里云很不负责任来一个什么所有资源价格翻番这样的行为的时候，我可以及时的下五洋捉鳖，把所有业务快速转型回我的数据中心。
我有一个伸缩的这个渠道，有了这样一个渠道，你就可以跟云平台去谈判，说我要降低你的这个收入，或者说我要降低什么？我的这个成本打个九折吧。 9 折好像不好，我们打个 7 折好不好？ 7 折好像，我感觉又不好，我打个五折。我有很多的谈判空间，我可以伸缩的上云下云，上云下云不会很被动，我可以通过这种上云下云的灵活性的，既可以去谈价格，也可以去实现一些什么云平台的转型。我可以从阿里云转到百度云，那这个时候我先下来，再上去就可以换云平台了吗？通过这样的什么混合云的搭建，我给我的企业增加了很多的灵活性，同时我们也为什么我们的架构师自己的职业发展找到了一个安全的港湾，这句话怎么说呢？当你上云的时候出现了一些问题，某些服务上不成功了，我就可以说，嗯，我要实现混合云的机制，所以某些服务就放在云下，另外一些服务放到云上。
当那些服务整改完了以后，我又可以发出一个什么一个通稿，我又把怎么困难的一套服务发到了云上，所以通过这样一个混合云的概念，使自己什么整个上云的过程变得什么可以更加的长，中间可以有更多的故事可以讲，也为自己什么职业生涯留下一个安全性的保障。好，聊完了平滑转型，聊完了混合云的各种各样的价值试点、周期性以及削峰这样的这个处理能力。那我们来想一想，如此有价值的混合云是不是很容易实现的？答案是否定。从下面几节开始，我们来聊一聊混合云当中要了解哪些知识，又要注意哪些难点？好，大家敬请期待。
