都叫。 net 地址转换。 net 就是什么网络 network address 转换 translation 同样都叫地址转换，这种叫做全地址转换。又有一种说法叫做 proxy 模式，代理模式。所谓代理模式，大家应该很容易理解，还是原地址发给目标地址，从上面开始发，我原地址发给目标地址，这个目标地址是个VIP，对吧？ DNS 解析好，我们把 URL 发给 VIP 负载均分期，一看到这个VIP，他就把这些数据包给截下来，然后往后台发。
包的时候怎么发？这样发，以我的名义，以这些负载均衡器的名义发给这个后台服务器，后台服务器是不是回包也是这样回，我把我的包回到这个负载均衡器就好了，我不用回给客户端。那这个时候我的 LBS 这里有个映射表，原地址原端口映射到目标地址目标端口，这样一个映射表回来以后就是目标地址目标端口还回原地址原端口，所以它有一个转换，它会把这段包重新发还给客户，然后转换成用 VIP 回包回给什么 client IP 一套这个映射转换协议方在这里，那这样的好处有什么好处呢？是不是很安全？我的内网服务器是没有公网地址，也不需要写很多的VIP，起很多的alias。



那同时我的整个网络中间所有的穿透信息全部在这里，我在这里面可以架WAF，可以架防火墙，可以架迪道斯防御，我可以防住所有往内的和往外的数据流量。所以这套方案是什么？是最安全的一套方案，那同时它还可以提供什么？还可以提供更多的灵活性的处理，比如说这里我可以把一些什么，一些这个基本包里面的一些什么一些 TCP 的特殊的工具的内容进行分片处理，打包转发给一些这种特殊的，比如说是我们的这种 IDS IPS 入侵检测、入侵防御系统等等，有很多的套路可以在这里做，甚至于在这里还可以起一些这种 TCP 的特殊包处理。
比如我知道你是消息队列，我会对你这个协议的消息队列做一些特殊处理，就类似于 HTTP 负载均衡器，或者说是 HTTP 反向代理可以做的什么特殊的针对 HTTP 投的特殊处理，我可以针对你 TCP 协议里面的某种协议做特殊处理。所以这套方案什么是安全性最强大的一套方案，也是飞扬老师非常推**荐的一套方案，同时灵活性也很好。

那怎么转换呢？我们看一下目标地址，是吧？是黄颜色，原地址是绿颜色啊。这张图用来自于另外一套百度图库，风格略有不同。那依然还是什么发包？客户端发包给服务器，绿颜色就是原地址， CIP 是原地址， VIP 是服务器的目标地址。发过来以后，我把这个包转给后台服务器。我转的时候有一个特点，我不把这个地址替换成DIP，而是保留了客户端地址。那这个时候服务器接收到这个包，回包的时候回给谁啊？是不是回给黄颖子部分CIP？我的回包不是回给DIP，要回给CIP，但是因为路由协议这个 IP 我找不到，这是个公网IP，因为路由协议我把它变成一个什么类似于 Router 的机制。这个机器对内网对外网来说可能是类似于一个路由转换或者是网关，所以这种情况下包还最后还是回给了他，由他帮忙去把这个什么，把这个包最终转向CIP。
所以这整个过程当中可以看到这台机器什么？有没有一个 VIP 榜单？上面没有，有没有公网地址啊？没有，依然很安全，但是它在整个传播过程当中，这台机器不是代理，这台机器是类似于一个路由器做路由转发，所以在后台机器依然什么能够知道客户端地址，所以在这里我依然可以做一些，是吧？双层防火墙，第一层防火墙在这一层我作为一个防火墙防止某些什么 TCP 连接的攻击，然后第二层防火墙是在服务器层，我还可以再做一层防火墙，对于某些特殊协议我也可以防止这个 IP 地址的攻击。
双层防火降机制很适合于哪里啊？很适合于支付网关支付系统安全性很强。同时对于某些特殊的支付协议，虽然也走的是四层TCP，但是它上面的报文协议是一种特殊的协议。这种协议在负载均衡器进行处理有点麻烦，这个时候可以把这种特殊协议的安全加固放在后台服务器来实现，就对于金融行业这种方案是很适合的。
但是非常可惜的一点是什么？我们公有云平台里面，很多云平台都不支持这种模式，这种模式在哪里用的比较多？在我们的私有云用得比较多，因为它的整个搭建过程采用类似于路由器协议，相对是有点复杂的。比如在私有云大得非常多公有云里面，比如说飞扬老师比较熟悉的像谷歌云是不支持这种模式的，但是已经有团队在跟谷歌云切磋或者交流，看看什么时候能把这种模式追加上去。在阿里云，飞扬老师后面会跟大家来一个实战，感受一下此类模式是怎么样酷炫的来支撑阿里云平台的真正的这么负载均衡的强大功能。那这里还要最后提一下两种。 net 机制，基本上都是实现了网络的安全，但是牺牲了什么？吞吐量所有回包全部经过这个负载均衡器导致什么？它只能支撑大概小百万级，或者说大几十万级的这个并发，不像前面的什么直接路由，是能支撑大几百万的这种并发能力的。‘

那这两个是什么功能呢？其实就是前面那张图刚刚没有跟大家聊的，当你去建一个 SLB 的实例的时候，通常你要指定的不是后台一个机器，而是一组机器。这组机器可以是一组Pod、一组容器，也可以是一组虚拟机。我以一组虚拟机为例，后台如果是指向 10 台虚拟机，把它定义成一个叫什么，一个叫应用组a，那这个时候我会建一个 SLB 的实例，指向这个应用组a，当应用组 a 里面机器跟这个 SLB 连接的时候，它会问一下你该用什么样的方式来进行健康检查？通常网络层是用什么？用 TCP 协议？如果是 HTTP 层会用什么用？可以用 API call 的方式来进行健康检查。
当你能确定这个机器的什么网络能连接它的这个 HTTP 的请求返回值是 200 的时候，我们认为这个节点是健康的假设。以网络层为例，我们是以 TCP 的连接为例，我们尝试去连接，比如说它的 443 端口。当我们发现 10 台机器里面有 3 台软没法连接的时候，负载均衡器会自动做一件事，什么事把它从轮询算法里拿出，这样的话数据包只发给剩下的 7 台节点，这三个节点不会再收报了，一直到什么健康检查后期发现重新活跃的时候，再把它加回轮询算法当中。
那这种情况也适合于弹性伸缩，当我们后台的什么实例的集群进行扩展，从 10 个扩展到 100 个的时候，它会什么？会自动地向 SLB 发起一个 API 请求，说我的后台的实例从哪 10 个扩展成哪 100 个了？那这个时候前端的 SLB 自动会把后续加进来的九十个节点加入人群，通过这种方式我们可以很快速地实现几个节点到几十个节点到几百个节点，快速弹性伸缩以及负载均衡的压力完全能够转换到这些被伸缩出来的节点上，最终实现整个资源组、整个业务的弹性伸展。好，聊完了这些主要的内容以后，我们看一看啊。下一节我们会从另外一个角度来聊一聊我们的互联，就是安全防御，大家敬请期待网络连接入口分流之后的进一步的我们的互联网中间的混合网络，混合云的关键难点安全防御，大家敬请期待。
