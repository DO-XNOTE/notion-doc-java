但是有一天一个企业说我两个数据中心其实最好可以变成一个数据中心互通有无，方便他所有内部服务去进行沟通？那这个时候有一个科学家想出了一个方法，在内部网络之上架一个虚拟的VLAN，大家知道 VLAN 是什么？是网络层的一个划分概念，但它只在一个大的网络内部好划分，没法穿透整个大的数据中心网络。


我们在上面打一些标签，把这些标签表示一些特殊的含义，从而虚拟地构成一个新的Vline，是不是这样就可以把不同网段、不同数据中心互通了呢？当然可以。这是什么？曾经非常流行的一个大二层解决方案，但是它有很多问题，比如说性能上在什么每个子网之间进行加解密，处理完以后我们要在上层还有一次封帧，还有一次加解密的处理，所以性能大打折扣。同时 Vxlan 要求什么互通有无的信息量非常大，导致没法进行很好的扩展，两三个数据中心可以 10 个、 20 个就很难做到了，特别是当前单元化流行的时代，两三个数据中心可不够哦。


所以 Vxlan 受到了一些限制，那我们来看一看另外一个多数据中心的主流解决方案，那就是什么三层路由、网络路由交换协议？

BGP 其实是什么？是 20 年以前就出现了一套协议，它这里面其实理论上来讲有两个子协议，一个叫内部 BGP i BGP，还有一个外部 BGP e BGP。然后每一个内部 BGP 形成一个自治域，整个自治域内部所有的路由协议是互通的，那在外部可以选择性地把某些路由转移给对方。比如说 1 号数据中心把一部分路由转移给 2 号，同时把一部分路由转移给 3 号，那 2 和 3 之间也有一些路由转换，最终能实现指定的几个小子域，或者是指定的几个子网之间形成什么大互联的概念？这恰恰满足了我们什么多数据中心单元化，以及云上和云下、云上和其他云之间的互通要求。


所以当前基本上 BGP 一统了云平台，尤其是混合云平台，大部分都采用 BGP 网络层进行互联，它没有像前面那种 Vxlan 大二层需要有额外的开销，相对来说开销较小。路由协议比较简单，当然也有一些限制在我们网络设计的初期就要求什么，每个子网之间的网段划分非常清晰，两个数据中心之间不能有什么网络的重叠，同时路由协议哪些推送过去，哪些不推送过去，要有很明确的定义，所以这对什么架构师提出了很高的要求，我们系统架构师或者是应用架构师要了解网络，同时能够配合底层的网络工程师实现整个数据中心的整体设计可不容易哦。所以三层结构固然酷炫，但是对我们的架构师提出了什么很高的理论性和实战性的要求。

云平台几乎什么都已经把这些 BGP 的路由规则，一些基本的网络划分变成了一些很容易理解的名词，帮助我们架构师即使不懂网络，只要你读懂了云的那些名称，也能够了解它的工作原理。比如说我们这里以还是以阿里云为例，阿里云选用了叫 VPC 的概念，这个概念其实来自于什么 AWS 云，所以我这里用了一张图，这张图既适合阿里云，也适合亚马逊云，那就是整个数据中心用总的一个云框架来包含，那其中每一个什么子网全部叫做VPN，或者叫每一个子数据中心都叫VPN。这个里面承载的一个内容就相当于是什么我们平时说的单元化的一个单元，或者是一个子数据中心，在里面我们可以创建一些小交换机。


这些小交换机在里面再划一个更进一步的子网，我们叫它SUBNET，可以看到这里面 10.0.0 是一个子网， 10.0.1 是另外一个子网。所以 VPC 本身就是个虚拟的网络，其中还可以划一个 SUBNET 更小的网络之间的应用该如何沟通呢？我们以这个为例，这是一个什么 ECS 的计算节点，那这有可能是一个什么？是一个数据库节点。好，这个计算节点跟数据库节点之间该如何沟通呢？通过什么路由协议？就是我们刚刚说的 BGP 路由协议进行沟通，这是在子网内部的，或者说是在 VPC 内部的，那可以理解成是IBGP。好。如果是跨 VPC 甚至于是到互联网之间沟通，我们就使用什么 e BGP external BGP 外部 BGP 进行互联，那最终的结果就是不管你是内网IP，对吧？还是公网IP，只要什么我们路由协议互相到了以后，即使你是，虽然你的又是10.0.0.5，通常是在内网才能访问的。但是我们互联网上其他的数据中心也能跟你进行方便的互通，那这样就把什么，把整个数据中心扩展到了混合云的框架里面，使得 VPC 跟下面，跟下层的这个物理机跟我们数据中心内部的机器都能互通了。


那中间是条什么渠道啊？是什么VPN？就是我们前面介绍过的什么 Ipsec VPN，实现了数据中心和云平台的互通，云平台内部通过 VPC 的一个底层架构设计，实现了各个子网之间的内部的 BGP 互通。大家不需要理解 BGP 的内容了，只要知道一切皆可通就可以了。


一切皆可通是不是有些安全隐患啊？当然有，那为了防止这些安全隐患，我们在什么安全域里面想出了一个名称，叫做 security group 安全组。什么是安全组啊？是对实例来说，也就是对我们的 ECS 计算节点来说，每一个计算节点你是不是上面要跑一些应用服务啊？这个服务必然有它的端口，也有它的 IP 地址，同时也有它的协议。比如你是跑的是 TCP 服务还是 UDP 服务？那对端是不是也有一个 IP 地址？有一个端口来访问你啊？所以源地址，源端口、目标地址、目标端口以及跑的协议，这样五元组形成了一个 security group 的规则，这个规则会说某种情况下我允许访问，也就是某种机器可以访问我这台物理机的某个端口。某种情况下，五元组匹配以后是不允许访问，所以出现了黑名单和白名单。


好，这就是访问物理资源的一个思路。黑名单，白名单，那另外还有一个思路是什么？所有内部资源如果想访问外网，那其实是很简单，是通过它的公网地址，这弹性网络就是公网地址可以访问，但是大部分数据中心是没有这个公网地址，在这种情况下该如何考虑数据流呢？嗯，这里有一个经典的思路给大家看一下，就是什么所有进来的数据流走SLB，然后到 ECS 计算节点。
那这个在上一节已经给大家看过了，那后面一句没有说当我没有公网地址的时候，我的物理机是没法访问互联网的，不能访问谷歌，也不能访问百度，那这个时候怎么办呢？我必须建一套这个叫做什么？ VPC net，就是地址转换协议的意思，有一套地址转换协议，把我内部的 10.0.0.0 的网段转换成公网那种，像类似于 0.0.0.0 什么什么 100 点多少、 32 点多少、 96 点多少等等等等不同的公网地址，把内部地址变成公网地址就可以。
什么向我们的什么公网发起数据请求去访问，各种各样像新浪、百度，对吧？我们的微博等等不同的功能的网站，所以通过这样一套转换形式，所有数据流进来走 SLB 出去走MIT，完成了数据的流转，那有数据流，必然有管理流。如果我们尝试管理一台 ECS 怎么样？我们也需要走 SLB 吗？不需要。
大部分的数据中心，或者说是大部分的云平台都有一套内部的映射逻辑，这套逻辑只要你满足说我的安全域 security group 设置的比较好，我就可以让外部的某某节点可以访问我这台数据中心的内部的instance，即使它没有公网 IP 也没有关系。这就是在云平台里面，或者在各个数据中心里面额外产生的一条管理透明的数据流。
这套透明流跟大家什么不公开机制不公开，但是思路很清晰，只要在 security group 里面描述说允许它就可以，什么通过管理节点尝试可以访问这个节点？好，那管理流有没有往互联网访问呢？当然没有，出去的数据流。我们统一认为什么叫数据流？那进来的有两种流，一种叫管理流，一种叫数据流。好，我们聊完了，进来的什么 security group 的管理流，以及通过 SLB 到 ECS 最后到 net 的什么数据流。哇，听得是一头雾水，是不是没有关系，如果大家对网络知识没有很熟悉，后面我们的时。实战会带着大家走一走，给你感受一下什么是 security group，什么是。 net 再加上一节已经聊到过的SLB，我相信是吧，通过实战，大家可以真真切切理解什么是互联网，或者说是混合云中的网络沟通。
好，刚刚其实聊到了什么？聊到一个概念叫安全组，但是安全组只解决了一部分问题，我们在云平台里面安全有两个功能，一个叫防火，一个叫防盗哈。

什么叫防火就是什么？我内部一个机房里面如果着火了我怎么办啊？我要担心这个火蔓延到整个机房，所以我要起一个叫 ACL 防火墙，它防的目的是一处的问题，不要衍生到其他处，也就是我在数据中心内部在交换机层形成隔离。我可以出现后端域、前端域以及什么 DNZ 隔离区域等，这些不同的域之间的网络默认是不通的，只有你设置 ACL 白名单才能够互相访问。
这就是内部的防火，那还有一个就是防盗，所谓防盗就是防外面的黑客、防坏人进来。那怎么办呢？就是我们刚刚说的 5 元组，你要知道对端的 IP 地址和端口，然后限制它只有某些 IP 地址端口白名单里面的人才能够访问我的内部数据中心，大部分的云平台安全组都是白名单。也就是说默认我是不让你来访问我的，除非你能拿到一个合理的身份，你是什么数据中心的管理员，或者是什么安全组的管理员，或者是运维人员，才能够到数我的数据中心里面来拿取实实在在的内部信息。


好，有了 ACL 防火墙，有了安全组，既实现了内部的什么安全隔离，也实现了外部的什么防盗防破坏，是不是就非常安全了？差不多，其实安全的里面的内容非常多，只上一个章节，大的安全章节跟大家聊了很多，那我们这一章重点在什么？重点再实战一下网络以及网络中用到的一些小的安全套路。好，下一节就让我们来实战吧，看一看怎么样让我们之前搭建的那几台机器能够真真正正地实现远端管理，以及真真正正地实现访问互联网，大家敬请期待。
