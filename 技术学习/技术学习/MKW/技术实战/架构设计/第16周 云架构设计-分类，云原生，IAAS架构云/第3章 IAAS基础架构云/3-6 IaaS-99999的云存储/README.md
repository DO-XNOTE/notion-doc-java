你是 arm 芯片也好，你是大型机也好，各种各样的系统，其实所有的数据统一来看归成两类，一类叫非结构化，一类叫结构化。那什么是结构化呢？就是我这个数据有些特征，它的长度未必那么长，但是它整体特征几乎保持不变，一条记录也是这个特征，两条记录也是这个大概类似的特征。这就是结构化，那非结构化的内容其实在世界上更常见，照片、视频、声音，这里面蕴含了很多信息。也许你拍的是某一个人，也许拍摄者是另外一位，同时它可能是某年某月某日产生的，又是在某地产生的，很多维度来描述同一个内容。所以你很难用一个结构化的表单，一个结构化的信息来描述这样一张照片，这样的情况下我们就统称为非结构化。


对于非结构化，当前最火的就是左上角那些技术对象，存储文件系统， Linux 很常见的，什么EX3、EX4。Windows、fat、NTFS，大家耳熟能详的文件系统，而在网络上，在云环境当中，通常是网络文件系统。比如像NFS，Samba， CIFS 的天下，除了这些传统的或者说是新颖的文件系统以外，还有一部分什么分析型业务，通常用什么？通常用并行文件系统，比如像 HDFS 跑大数据，跑流处理，这些都是非结构化的数据，就是右上角那些跑数据库的，跑 no SQL not only SQL 不光是关系型数据库，但总体来看它们都是属于数据库类型，都是有一定的结构，不管你是列族也好，键值对也好，甚至于缓存系统 Redis 也算是什么结构化数据？对于结构化数据最好的解决方案当然是快设备，大家可以看一下当前各种中间件、数据库套路，后台挂的是不是都是一块一块的硬盘采用分布式，采用一些仲裁采用一些这种并发性的能力，充分发挥快设备的优势，从而实现结构化数据的快速搜索和查询。


对于非结构数据，刚刚其实已经聊到了一种，就是文件系统，这是什么？这是传统的主流套路，而在云上更强调尽量采用对象存储来替换文件系统，因为它们都是非结构化数据的有效载体。聊完了几种存储分类，我们来逐一来看啊。块设备文件系统和对象存储在云上，我们可以举例以阿里云举例，看一看它做得如何呢。块设备来看，

不管是阿里云还是什么云，所有云几乎都建议以，如果系统启动的时候肯定会用到系统盘，同时有些系统还会用到什么？用到数据盘等等，这些盘不管你是什么盘，都尽量不要分区。我们传统老套路什么把一个硬盘隔成好几个区，某一个区做boot，还一个区做root，还有的区做swap，是不是这种套路在云上都不合适？云上尽量不要分区，因为云上的磁盘有快照。有什么有这种镜像？当你有分区的时候，你的快照镜像重新回滚，或者说还原回去的时候，有可能存在问题，而且磁盘可以扩容吗？缩容扩容缩容扩容，文件系统可以跟着缩容扩容。但是你的什么分区表可不一定，所以尽量不要分区， LVM 是坚决要杜绝的。什么叫LVM？逻辑卷管理在传统的这个磁盘系统里。里面我们有时候后台挂了三四个盘，然后建成PV，用逻辑卷管理形成很多个虚拟磁盘。这个时候当有一个盘进行了什么，快照回退另外几个盘就会和这个盘什么变成数据不一致，导致整个文件系统崩溃，在云端发生了太多太多实际此类的案例。


所以付阳老师强烈推荐不要再用LVM，不要再用传统套路对待云上的块设备了，云上的设备同时还要有个注意点，就是要独立系统盘里不要放任何数据信息，比如像什么Redis，Kafka，你这些中间件你没法做到完全的云原生。所以你会有一些盘，但这些盘千万不要落盘。


在系统盘上必须独立的有一个数据盘，而且这个数据云盘和计算节点也独立。什么叫和计算独立呢？通常先创建数据盘，再创建计算节点，最后挂载数据盘，这样的就表示什么？当我删除一个计算节点的时候，我的数据盘肯定不会被拨起，不会被删除，我会保留原有的所有数据，这就是盘和计算节点独立。有的一些小伙伴他不熟悉，我创建系统的时候，我就创建两个盘，一个 boot 盘，一个 data 盘。那我删除的时候，很多云平台就会默认把数据盘也删除，导致很多问题简单独立，还有什么统一？这就是云平台的快设备的特点。什么叫统一呢？我们云平台一定要追求一个叫不可变基础架构 immutable infrastructure。所谓不可变，就是当我启动的时候它是什么状态，后续就一直保持这样的状态。不要什么启动的时候挂的盘很少，然后在后台起一个什么 cronjob 去mount，这是不合理的。所有挂载的内容要在 etcfs Tab，如果是 Linux 的话里面统一展现，需要挂载就启动的时候都挂载好，后面不要再挂载卸载，这样会导致很多问题。


聊完了，快设备的一些思路，大家感觉什么？简化对吧？不要分区，不要做一些额外的挂载操作，不要 LV m，然后该独立就独立。简化和独立是快设备什么永恒的真谛？聊完了快设备，我们来看一看文件系统又该如何处理呢？

对于文件系统，第一原则就是能不用就不用，因为什么非结构化设备对象存储完全可以胜任所有文件系统的套路全部可以用非结构化的对象存储来应对。第二条原则就是假设你玩不转对象存储，你一定要用文件系统，那你要考虑一下文件系统的容灾，比如 rsync 文件系统的备份，比如采用一些收费软件netbackup，以及云原生应用。尽量把什么文件系统抛离出去，变成远端文件系统来进行使用。网络访问有很多的难点需要攻克，比如说我要访问远端文件系统，这个远端在同一个数据中心，同一个 VPC 里面，那还好设定一些什么内部的 SL 规则就可以了。那如果在 VPC 外面，我要考虑什么？ Nat 去访问它，同时还要通过 VPN 去穿透各个网络，非常复杂，尽量建议什么不要采用文件系统啦。

叫什么？叫比纯金还纯的存储。为什么这么说啊？因为亚马逊在第一天推出对象存储的时候，就推出了一二三四五六七八九十一个 9 的高可用的对象存储，当然是数据持久化高可用，不是数据访问高可用，所谓数据访问高可用，就是有多少时间我可以保证能够读写它大概是在 5 个 9 左右，不管是阿里云、亚马逊云大概都在 5 个 9 左右，但是数据持久高可用，就是讲我有多少数据有可能丢失， 11 个 9 的概率是不会丢失的。


好纯呐，比纯金还纯的对象存储不光可用性高，还有很大的特点就是非常灵活，我们的代码大部分代码都会什么都会，用SDK，对吧？调用底层的 API 来访问对象存储阿里云、亚马逊、谷歌云。任何一个云平台都有它自身独有的SDK，你只要调用这套SDK，使用它的库很方便就。可以什么读写修改，控制你的对象存储资源了。


另外如果大家不是用代码，而是什么习惯用？比如像控制台，比如像命令行完全可以使用，甚至于我们有很多的驱动可以什么把对象存储变成文件系统，完全取代文件系统，还有可以用 FTP 的传输方式。还有一些知名的网盘软件开源的，比如像Nextcloud，它支持什么亚马逊、百度、谷歌云的对象存储，它直接封装成网盘，你可以在里面传视频、传照片，做各种各样的文件分享，是不是很酷炫啊？除了灵活性和高可用以外，对象存储还有一个很大的功能点，就是什么？它的功能远远比文件系统完整很多，它有访问控制，非常精细的访问控制，可以到一个一个的bucket，也可以到一个一个的对象的访问控制。


除此以外，元数据描述我们在架构师实战课程里面是不是聊过一个例子？媒资库，我们把视频以对象存储保存，然后我把所有的音频文件，所有的那些这个文字，以及什么对它的描述，是哪个明星什么时候演的，甚至于一些用户的评论全部可以写在什么元数据里面。这样呢？你只要找到这个视频，就找到了跟这个视频相关的所有的一些描述性信息，这就是对象存储的一个强大之处，它身上是吧？是带元数据的，跟我们文件系统里面的一个文件不一样，文件系统里面带的元数据只是什么时候创建，谁创建？然后没有一些额外的信息，但对象存储信息丰富很多，另外对象存储天然支持版本管理，只要你激活了一个bucket。


所谓 bucket 就是对象存储的目录的概念。对象存储是两级管理，一个目录，一个内容，一个bucket，一个对象，只要激活了 bucket 的版本管理功能之后，你所有的增删改查每一个过程全部有什么？有版本控制，就像你去修改一个什么在线文档一样，不管你是 word 文档还是Excel。如果是在线文档，微软的在线文档或者谷歌的在线文档，只要是在线文档，每一个修改它都会记录下来，你可以回退到任何一个时间点。即使你这个文档删除了，你仍然可以回到某一个时间点，把它快速恢复出来，这也是对象存储的一个亮点。


最后就是对象存储支持那种类似于键值的存取形式，对象存储文件的名称就是键，它的内容就是值，一个 bucket 里面它的文件的名称都是独立的，相当于是一个键值对的关系。聊完了这么多是不是哇手很痒，我好想玩玩对象存储好，我们下面就有两个小demo。第一个Demo，玩一下云上的块设备和文件系统。第二个 Demo 就让大家真真切切感受一下对象存储的简单方便，大家敬请期待。
