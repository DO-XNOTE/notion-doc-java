总结
会议讨论了关于云原生中中台化的相关思考和原则，具体如下：
日志流原则：
日志汇聚难点：节点和微服务增多导致大量日志需汇聚，且多维度分析困难，日志流可能占大量资源。
解决思路：减少不必要日志，做好处理分析，通过归档备份减少在线日志数量，应用开发时日志统一输出到标准输出，由日志平台完成收集、分析、处理。
作业管理平台原则：传统作业管理方式不适合云原生，云原生作业应将其变成普通服务，如消息的consumer、API服务等，通过云平台定时器触发执行。
认证和授权原则：业务平台的相关处理应通过外部认证平台或服务网格等方式简化，将认证验证、用户登录、审计过程抽象化并剥离。
监控遥测平台原则：监控系统能帮助保持应用健康检查、合理扩展、追踪业务出错并及时告警呼叫，应用要做好简单埋点等基础准备。
API经济原则：系统层或底层的API中台仍可在公司内部推广，开发新功能时移动端优先，后台封装好API供前端调用，业务部门应实现SLA以考核服务质量。

那它什么样的思考呢？我们来看一看我们越来越多的节点，越来越多的微服务会导致什么？越来越多的日志需要汇聚，就这个时候怎么样去汇聚这些日志节点？把这些所有的，不管是系统也好、应用也好，所有的这种层次的日志进行汇聚，这就是一个难点。
那同时我们在绘制的过程当中还希望能做到多维度的分析，我们可以在系统层做一层分析，在应用本身做一层分析，在外部的调用这一层做一层分析。另外还可以在业务上面进行拔高来进行分析，所以不同维度的分析也是一个难点。另外随着什么微服务化，我们的什么服务改造越来越多。所有的服务自身都是有日志的，所以日志流的占的比重可能会越来越大。甚至于飞扬老师还碰到过这样的企业，整个企业里面 API 还包含消息队列，所有的数据流的流量都没有这个企业内部的日志流的流量来得大。
那这种企业情况下，其实日志流已经成为什么？最占带宽、最占资源、最占存储空间的一套这个系统了，该如何运行好、管理好这样的日志流呢？这就是一个难点。那我们的思路是什么？减少不必要的日志，同时做好强大的日志处理和分析。并且通过归档和备份的方法减少我们需要管理的在线日志数量，这样既可以省成本，也可以增加日志的有效性。
那本质上来看，其实我们在应用开发的时候要关注什么样的点啊？要关注以下几点，首先所有的日志不要去写本地硬盘，有的日志什么写到 var log 底下，有的日志写到 OPT 目录底下，这些都不合理。不管你用 log4j 还是什么logback，其实你在它的配置里面都可以看到，通常我们是默认是输出到标准输出，同时会写一些不同的文件，这些文件在指定的目录底下，这在 cloud native 里面可不适合。 cloud native 要假设什么本地没有硬盘，所以所有日志统一就往标准输出进行输出。整个输出过程当中不要有任何的代理。我们既不要装 Splunk 的代理，也不要 Elk 里面有任何代理，完全就是铺到标准输出，这样我们应用就完成了日志流了，后面的其他的所有的内容通过什么来帮我们完成啊？通过日志平台，我们要强调平台或者中台，他们来完成日志的收集、分析、处理。我们只需要所有应用，只要把日志吐到标准输出完成。
不管你是容器的标准输出还是虚拟机的标准输出，整个过程不代理、不保存、不处理，三不原则，是不是这样就轻量了很多，这样就很方便地进行弹性地伸缩了？

甚至于我们在那个标准的 twelve factor 里面，他推荐的方法现在都不流行了，传统的方法比如像什么，比如像是写一个cronjob，然后指定具体某某某某时间跑某某某某作业。这是一种或者高级一点用，通过spring， batch 等等方式，底下可能是有多作业并行，还会有些什么一些锁，什么 queue 的锁。
然后这些作业有个统一的管理平台，但是大家想一想，这些作业既然要发布到不同的节点，同时它还有一些锁的干预，所以它是不是能够弹性地伸缩？未必，它不一定能够很方便地在任何节点进行伸缩。所以说这种作业其实不是 cloud native 的作业。那什么样的作业才是 cloud native？整理一下，是这样，任何的 cloud native 的代码都在一个仓库里包含什么作业，所以我们要把作业变成一种普通的服务，我们这个服务可以是一个消息的consumer，对吧？就是接收者处理器，也可以是一个 API 的标准服务，也可以是我们的作业调度服务。那这个调度服务如果是把它开发或者说是什么打包成一个API，那这个时候作业会有一个标准的叫定时器，定时器会触发这个API，整个过程不是写一个 Cron job，也不是去写一个 spring batch，而是写一个普通的 API 服务。那这个。
的服务有外围的，比如云平台，每个云平台都有它的什么定时器？这个定时器是跟格林威治时间进行精密校准的，它可以很精准地在某时、某分、某秒或者是某个指定的间隔触发你一个 API 或者什么向你促一个消息，你可以从消息队列里去接收一个消息，或者是从什么物联网去接收一个事件，亦或是从 API 的角度去被别人调用一次。一旦被别人调用到了以后，你就执行相关的作业任务就可以了。
所有的内容都是什么偏 API 的，或者是偏消息处理的，或者是偏事件处理的？你不需要在你的业务本身去写一写我应该在什么时间，怎么调度，我应该是不是用某某框架？不需要，你就是一个简单的API，简单的消息处理代码就可以了。这是作业管理 cloud native 比较推荐的一个方法。

大家说，哎呀，这我听得很多了，这不就是中台化的思考吗？所有的业务平台，我要对外提供一些基本的API，能让什么第三方快速调用，或者能让公司的子公司进行快速调用。现在阿里好像说这个业务平台中台化已经不火了，那我们现在应该如何来准备呢？那飞燕老师可以跟大家这样交流啊，业务中台即使不火，其实什么平台的？就是系统层的中台，或者是底层的 API 的中台，其实仍然是可以在公司内部进行大力推广的，我们可以把一些基础服务在公司内部变成API，甚至于可以把一些这种基础的这种平台化的服务也用 API 来提供出来。比如你的 CICD 发布内部的一些基础的这种作业调度都是API，这些其实都是中台化思考或者是平台化思考的一个亮点。
另外我们现在的大部分企业都是叫什么 Mobile first 移动为先，当开发一个新功能的时候，首先考虑这个功能不能在移动端客户端能够进行发布，那这个时候我们就会什么重点？要看一看基本上所有的移动端套路都是什么？后台调的是API，不像我们的这个 Web 端有前端代码和后端代码，那移动端，移动端其实是什么？是部署在客户的那个手机上，那这个手机后面调的其实每一个都是API。所以我们在后台上应该封装得很好，既可以适用于 Mobile 移动端，也可以适用于什么我们的业务系统的前端来调用，这样的后端就是以 API 经济为核心的一套强大的平台化思考。
那除此以外还有一个很关键的点，就是每一个业务部门，特别是架构师团队应该带领整个业务部门实现SLA，我这个部门所有的 API 的可用性性能和业务的这个功能性都是有标准的，每年都可以实现自身的考核通过这样的 SLA 能够什么强化突出本业务部门相对于其他部门。
服务质量是不是比别人高啊？我的这个被别人调用的次数是不是比别人高？我的带给公司的业务价值是不是更高一些啊？那希望就是以 API 经济为主导，以前面的什么 14 个 cloud native 的原则为支撑，最终实现我们整个应用是架在 Iaas 和 Paas 这些什么公有云或者混合云之上，而通过我们强大的研发团队，通过我们强大的整个的平台化的思考，最终实现在公司内部搭建出自己的 SAS SaaS 软件。
