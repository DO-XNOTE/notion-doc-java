# 2-3 云原生应用-弹性思考

---

会议讨论了 cloud native 的实现应用弹性的三个原则。具体如下：

- **无状态进程**：服务器启动后应能拔掉硬盘且功能正常，所有结构化和非结构化数据不应放在本地，如将静态文件放对象存储，Nosql 缓存等放外置中间件和数据库节点，避免应用共享磁盘，将本地缓存改外置缓存，实现应用无状态。
- **同步并发**：不同应用可在横轴负载类型上通过线程、进程（含容器）、虚拟机、沙盒技术等方式进行弹性伸缩，根据隔离性、安全性和性能选择，多线程资源消耗少并发强。
- **快速启动、优雅关闭**：应用要能随意抛弃，前提是无状态且无依赖，启动时可不依赖数据库、缓存和中间件，能独立启动并重试，云原生应用启停应在秒级或 10 秒级。优雅关闭时应避免不合理代码设计，无法优雅关闭可强杀。
- **原则共存的意义**：上述三个原则共存才能实现强大伸缩性、快速启停和应对出错场景。

起需求到落地的桥梁，构建 it 新蓝图。我是张飞扬，上一章节我们聊了聊 cloud native 的三大原则，注重在CICD。那这一章节我们来聊另外三个原则，重点是看看它如何实现应用的弹性。好，首先是第一个也是最关键的原则，就是无状态进程，英文原文叫做 Stateless processes。
什么是无状态呢？国外有好几种比喻，什么奶牛和宠物。这个其实我们听来感觉怪怪的，用最简单的话比喻就是公鸡和母鸡。母鸡你是什么？用来生蛋的？所以你不能够随便杀的，今天我杀一只母鸡，明天就没有蛋了，是吧？这一个月都没有蛋吃，所以母鸡不能随便杀，但我买一只公鸡回来，我不指望你生蛋，所以买一只就杀掉这种公鸡就是无状态，启动完用完就扔掉，不用关心这个鸡扔掉以后的后续影响，这个鸡就是用来吃的，那母鸡不一样，母鸡买完以后它会不停地生蛋，所以它上面的数据是有用的。


那我们换 it 的方法来说，就是看一看你有没有内容写在硬盘上，那像这里下面那个不是数据库，我们是当成硬盘来看，当一个服务器它要去连一个硬盘，通常只有一个场景认为是正常常就是启动。也就是当我服务器从硬盘启动完以后，我应该可以把硬盘拔掉，这个时候整个服务器的应用所有的功能都正常，数据库的读写、什么业务的响应、操作系统的运行、文件的访问都要正常，如果不正常就认为你是有状态，那怎么样才能做到硬盘拔掉还正常呢？就是这样，所有结构化数据不放在本地，放在什么外置的数据库里面？比如MySQL。
所有非结构化数据不放在本地的文件系统，尤其是动静分离里面的什么静态文件，比如 CSS 文件、 js 文件，千万不要放在本地硬盘上，放哪里呢？扔出去，扔到外面对象存储上，把照片、视频全部扔出去，这样我的服务器里面其实没有什么任何文件需要保存的，我读取和修改全在外置。

同样道理，有些 Nosql 有一些这种缓存信息，有一些那大数据信息全部扔出去，都到外面的中间件和数据库节点进行运行，这个时候每一个服务器就像图上面这样，我想从两台变成 7 台就变成 7 台，想变成 70 台就是 70 台，想变成 700 台就是 700 台。当我要关闭的时候，你指定数量你说要关闭 600 台，我随便挑 600 台。关闭。整个业务是没有影响的，因为所有的数据都不在这里，它不是一个生鸡蛋的母鸡，而是一只公鸡，待宰的公鸡，这就是无状态应用的思路。


那其中的核心什么？就下面那句话，整个过程当中所有应用不要去共享磁盘哦。不要像 Oracle rack 那种集群式软件，什么共享磁盘保存心跳信息在硬盘上，硬盘崩溃了，整个什么 Oracle rack 集群就崩溃了，千万不要这样。那除此以外，我们应用过程当中很多的，比如像是session，对吧？这种什么，这种 session 的信息通常是放在本地缓存，这个时候我就建议把本地缓存改成外面的，比如像Redis，我们的这个外置缓存系统等等，不同的选型 culture base 等等。很多的选型通常把那种核心的缓存、核心的数据库、核心的文件的内容全部往外扔。那这个时候你的应用就是无状态应用，你跑的这个服务器就可以是容器，可以是虚拟机，也可以是云平台的任何一个资源，可以变成service。

这张图其实也是张经典的并发性的图啊。我们看横轴，横轴上面是什么？是负载的不同类型，比如有 Web 应用的，有中间的工作节点，就是做计算的 work 应用，还有后面的什么时钟应用，做一些什么任务调度的时间触发器，这样不同的应用它本身在 y 轴上可以很方便地进行 scale out 弹性伸缩，这伸缩过程可以用线程来伸缩。


原来我是一个线程，当我压力大的时候，我会变成 10 个线程，也可以进程来乘坐这个进程其实也包含了容器，我们每个容器其实就是一个独立的进程，当一个容器压力大的时候，我们就可以不同的进程时的状态运行成多个容器。在安全领域，特别是支付领域，我们通常要求隔离性很强，线程跟进程的隔离性不够，这个时候我们或者采用虚拟机，或者采用沙盒技术，或者采用安全容器啊。这三个技术其实都是属于沙盒的不同的衍生，它跟进程隔离不一样，越往右也就是沙盒隔离的隔离性高于进程，隔离性高于线程，但是并发性业务的性能来看，多线程其实最少的资源消耗最强的并发，所以同样是什么向上向 scale out 进行扩展，根据我们的隔离性、安全性以及我们的性能可以斟酌着采用。
多线程，同一个容器里面多线程或者是多进程，多容器或者是多沙盒、多虚拟机、多安全容器的方式来选择不同的并发，从而保证我们云原生应用能够很强的concurrency，并发性好。

但是在英文里面一个词就能够表示出它的本质含义，大家能猜出这是什么词吗？这就是什么？这就是随时可以抛弃的那个中文的翻译disposability，这个什么概念呢？那我们举个例子，如果你是去吃我们的自助火锅，这个时候火锅里面的每个菜你可以换，但是当你点了个辣锅的时候，你发现，哇。这锅太辣了，这锅好像不是上海的辣度，是湖南的辣度啊。我吃不下，这时候怎么办呢？你没法把锅抛弃是吧？自助火锅通常换锅还是要额外收费的，就这个时候你就很苦恼，你想。
嗯，早知我就吃这个了，自助烤肉，当我吃一块烤肉的时候，我吃了它感觉不爽。我可以把这个烤肉什么放一放，待会再吃，或者我就咬着牙先把这烤肉吃完，不要浪费，不要罚钱，然后我可以再点下一串烤肉，有没有更优雅一点？那就是自助冰激凌了，我们的每一口、每一个球都是不同的风味，随时可替换。
disposability 就是说我想替换就替换，像一张餐巾纸，一个冰激凌球一样，可以随意地抛弃。那在实际的 it 过程当中怎么样？首先你的应用要是无状态，这才能随意抛弃吧。如果你本地的硬盘关键性很强，当我把应用服务器给干掉，把你的硬盘给毁坏的时候，其实你对业务造成了影响，所以无状态是前提，其次是无依赖。我们经常在做容灾项目时候会发现当我把我的整个业务中心交给运维团队，尝试进行一个容灾切换的时候，他们通常都会以随意的方式来启动应用，没有一个先后顺序，然后会经常导致整个业务启动不好，这个时候我们就怪运维人员了，你们怎么连基本的先起数据库，再起中间件，再起后端应用，最后起前端应用，这么简单的逻辑都不懂，这是他们的错吗？不是谁的错，是我们研发团队用我们应用架构师的错。
每一个 cloud native 的云原生的应用在启动过程当中，它可以不读数据库，不读缓存，也可以不跟中间件打交道，它可以独立无依赖，启动之后它才不停地去跟不同的中间件、不同的数据库进行沟通，当初错的时候，应用本身还可以进行重试，这样每个应用的启动可都是无依赖的，这才能实现真正的快速启动。
除此以外，每一个应用本身不管你是容器也好，虚拟机也好，其实很多时候是在起应用本身。如果你一个应用是在 10 分钟以上，那我可以告诉你，你离 cloud native 云原生还非常远。我们要做到云原生应用本身的启停应该是在秒级或者是 10 秒级的阶段，这样的应用才是能真正地快速启动，才能真正地优雅关闭。
说到优雅关闭，经常有这样的应用，就是我在关闭的过程当中我支持 service stop，但是这个 service stop 过程尝试什么清缓存、清我们的这个队列等等的方式，处理完了以后才进行应用的删档，那这个时候可能花炒几分钟时间，在系统里面可能会有一个额外的什么这种启停脚本，当我两分钟之内没有优雅杀到的时候，我就 care 9 了，这种情况下整个我们应用的关机服务，这个服务其实没有任何价值。因为每次它都没有完成优雅地关闭，而被什么系统里面的那些调度器给直接什么杀进程了。
那这种情况告诉我们两点，一点就是似乎你的停止的这个操作，你的这个代码设计不合理，需要改进。第二点就是什么，既然你不是淑女，你就不要装淑女，你的关闭直接 K29 强杀就可以了，因为本质上你最后还是在走强杀的一条路，所以不管你是怎么样，希望你是能够快速启动，也希望你能够快速关闭，在过程当中尽量的优雅，这才是我们 disposability 要说的核心。
内容好，要弹性，就要能做到上述的三个原则，只有这三个原则共存的情况下，我们才能真正做到很强大的伸缩性、很快速地启停，以及能够非常快速地应对不同的出错的场景。好，聊完了我们这三个关键的原则以外，下一节会聊另外 4 个原则，这四个原则偏重于不同应用之间的解耦，大家敬请期待。
