总结
会议讨论了实现应用解耦的四个关键原则以及相关内容，具体如下：
依赖明示：发布的应用工件应明确指明所有依赖，包括外部软件包和第三方库的具体版本号。可使用 Maven 的 POM 或 gradle 指明依赖，在操作系统层面用 dockerfile 提前打包所需运行时环境等。
配置和密码分离：应用代码应做到放之四海而皆准，将环境变量、密码等敏感信息从代码中移除，放入集中化的环境配置管理系统，并对密码等核心内容加密。
后台服务：将数据库访问、与第三方沟通等后台服务剥离，采用网络连接方式与第三方服务沟通，并做到完整的熔断和降级，使应用专注核心业务价值。
端口绑定：通过物理端口和虚端口的映射，实现一台物理机运行不同应用，需有服务注册和发现机制来解决端口映射和访问问题。
解耦与云原生应用：上述原则旨在解耦应用的各方面，使微服务真正落地成为云原生应用，部署在公有云或混合云中。

叫 dependency management 怎么样来管理不同的应用之间的依赖？我们想一想，除了经常我们业务是本身，是吧？对第三方库第三方包有依赖以外，是不是对系统还有些依赖啊？比如我们要在安装我的应用的时之前要做什么？apt、get、rpm、yum、 brew 等等，装一些，不管你是装 Java 代码的运行时环境，对吧？还或者是装一些这个开发环境，是吧？ JDK 等等，甚至于装一些中间件，或者装一些简单的什么配置、管理、服务、小工具等等，然后再去部署我们的应用。
那这个时候其实你对环境有要求，所以你就不单纯我们希望我们发布的业务或者我们发布的应用，不管你是采用容器的镜像还是一种什么工件形式，这个工件里面能明确地指明我所有的内容，不管是外部的一些这个软件包，还是外部的第三方库，它的具体的库的版本号也要指得很清楚。我们可以用什么？可以用 Maven 的 POM 或者 gradle 来明确指定我们的 dependency 的依赖哪些第三方的库或者包，以及其中的版本号。那在操作系统层面，我们可以尝试采用像 dockerfile 这样的形式在我们的 doc image 里面提前打包好所有需要的运行时环境的库包、第三方的依赖等等，所有这些东西全部打包在一起。这个时候你就没有借口说，哇，这生产系统的环境好像跟我原环境不一样，所以我的发布不成功，或者我运行起来有问题，没有环境的问题作为借口喽，一切都在你应用代码的什么配置文件里进行了指定。

可以更清楚地描述成configuration， credential and code separation，就是应用代码和我们的配置文件和密码是分离的。首先你要把应用代码做到什么？放之四海而皆准，这个四海包含你的开发环境、测试环境、生产环境，要同一套代码，也包含给你公司用，或者给你的合作伙伴用也要同一套代码，那这个代码可以相当于是我们的开源软件，这中间没有任何跟你公司特有的环境变量、环境参数，也没有公司内部的什么跟存储相连接的什么密码、用户名，或者连接数据库或者中间件相关的一些用户名密码等等。
所有那些敏感的什么token、 credential 全部要移除，这个代码是完全开源化的，可以给第三方直接使用的，然后把跟系统跟环境有关的内容放到了环境变量里。这个环境变量注入有很多种方法，可以是直接什么用不同的这个环境参数来注入，那这个参数可能来自于操作系统本身，也可以用什么外部的一些这种什么，比如像 spring cloud config 等等方式向系统注入，或者用什么 Kubernetes 里面什么config、map、 secret 等等方式向应用里面注入环境变量都可以。
那除此以外，我们要对最核心的内容类似于密钥、密码内容进行再加密，这个加密是防止公司内部的人员能看到，不光是外部人员看不到，我们公司内部人员也看不到，那最终能实现什么？一套完整的开源精神，所有方圆老师这里，对吧？团队开发的代码都要能够随时随地把我的系统开源，也就是老板一句话说，嗯，今天我开心了，我想把我 Git Hub 上或者 GitLab 上所有代码全开源，可不可以？你要毫不犹豫地回答，完全可以。
那这时候之前要做的工作就是所有环境变量要提取出来，放到一个集中化的环境配置管理系统，可以是阿波罗，可以是 spring cloud config，或者是 Kubernetes 的配置系统等等，都可以有一套中心化的、集中化的方式进行统一管理。
另外你的所有的跟密文、密钥、密码相关的内容全部采用一些加密的方式，比如像经典的 Hashicorp vault，这个 Hashicorp vault 就可以跟 spring cloud config 配合实现什么密钥的配置管理，然后动态的加载解密，然后传输给应用，当成环境变量传输给应用等等这些机制全部可以在一套集中配置管理里实现，最终你的代码是完全开源，完全可以放之四海而皆准。

但是却对什么解耦产生了非常大的这个影响。我们来看一下是不是很多服务我们用的是 80 端口，当你开发完以后，你这个应用跑起来也许就是 80 或者是 8080 端口，那我如果有 100 个应用，那我怎么办呢？通常是吗？我们大部分的架构是这样理解的。既然我应用统一用80，就用 80 好了，然后我公司内部的 100 个微服务就用 100 个 80 端口的应用，然后这每个应用跑在唯一的一台服务器上，也就是假设服务器 a 跑了，应用 a 了以后这个服务器 a 是坚决不能再跑应用 b 的，为什么？因为它产生了端口争执，是吧？所以应用 b 必须在服务器 b 上跑。
所以这里面有好几种思路，一种是服务器名上就把应用的名称写进去，那这样呢？这个服务器永远只跑应用 a 或者应用b，还有一种就是什么 CMDB 叫配置管理数据库，它这里管哪些应用，跑在哪些服务器上，当前是什么状态，然后方便运维团队进行统一管理。这些都是老旧的套路了，我们在 cloud native 里面从来不这样使用。
那怎么使用呢？是这样的，我们有一堆服务器的，我们不关心这个服务器之间有什么区别。当有应用来的时候，每次绑定的物理机端口是不同的。假设我应用a，我绑了服务器的时候占掉了 80 端口，我同一台服务器完全还可以跑应用b，但这个时候通常采用容器化的这种 IP 端口镜像技术，使它运行在 81 端口上。我代码里面大家还是写的一样的，都可以是 80 端口，没有关系，但是真正绑定的时候你就会绑定一个其他的端口。
通过这种物理端口和虚端口的映射实现什么？一台物理机可以跑不同的容器，可以跑不同的应用，那这里面最大的难点是什么？就是端口映射了我到底什么样的服务，应该占物理机的什么端口呢？当我应用要访问这个服务的时候，我怎么知道应该找到如何的端口呢？这就需要有一套什么服务的注册和发现机制。比如像 Eureka 是可以来做这个事情，比如像 NACOS 也可以来做这个事情。还有很多的套路，比如 Kubernetes 的什么service，它虽然没有一个标准的服务，也没有一个标准的发现。但是他告诉你我的服务a，你要访问我，你就知道我的短名，比如我的短名就叫 application a，只要你要访问这 application a 这个URL，只要你对它写一个HTTPS，或者是对它写个 HTTP 冒号斜杠 application a，它自动就会把这个映射，把这个什么 URL 映射到对应的物理机的端口上，从而映射到内部容器的 80 端口上，这整个全过程全部由什么底层平台来实现，或者由服务注册中心来实现，从而解放我们的应用。
当应用调另外一个应用的这个端口的时候，不要记住它的端口了，也不用记住应用里边的具体的什么 IP 地址都不需要，只需要知道对方的应用名，对方的服务名，对方的这种服务发现注册的名称就可以了，这就是端口绑定已经全自动化的一个端口映射的功能。好，聊完了前面 4 个原则，其实大家想一想，这些原则是不是都在解耦啊？不管你是解耦应用和它的配置，还是应用和它的数据库，亦或是应用和应用之间，都是在实现解耦，通过强大的解耦才能使什么使微服务真真正正落地，成为 cloud native 的云原生应用才能够真正部署在公有云？部署在混合云当中。好，聊完了解耦的 4 个原则，我们后面还有几大原则，我们统一把它归类，为什么中台化或者平台化的思考。大家敬请期待。
