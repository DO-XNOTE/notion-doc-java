请作答，相信大家都能够什么？想到前面菲儿老师已经重点聊了 4 个点，把那 4 个点一一举出来不就可以了嘛？一个是无状态应用，无状态进程。所有的什么业务都要什么？把状态剥离掉，把你的缓存，把你的 session 剥离出去，把你的数据库，把你中间件的依赖都剥离出去，所有的这些中间件数据库缓存都不要写在本地硬盘上。甚至于你原来对于文件系统的依赖也要变成对于对象存储的远程依赖。另外，同步并发，不管你是多线程、多进程、多沙盒，只要你能够很方便地进行横向扩展，那你就能够体现强大的业务弹性。
除此以外， disposability 怎么样？我像什么吃冰激凌？我像扔废纸一样可以随随便便地去什么？去停止或者启动我的所有服务，要快速启动，要快速关闭，如果可以的话，在关闭的时候稍微提供一点点优雅。这就是什么整个题目的题眼，如果要拿高分，依然是要聊实际案例了，聊一个你所擅长的领域的服务，当时你是怎么样把它进行改造的？从有状态变成无状态，又如何通过容器化，通过编排技术来实现同步的并发，实现对于业务的 CPU 内存对服务响应时间的追踪。这是眼，然后怎样通过智能的大脑，比如规则引擎、人工智能，这是脑，最终调用容器编排平台的API，比如 cloudfoundry 的API，比如 Kubernetes 的API，也比如 Mesos 的API，从而实现实例的弹性伸缩。
脑眼手的配合可以实现并发的强大能力以及申诉的快速响应，这个时候整个应用最终是秒级启动还是分钟级启动，是秒级关闭还是分钟级关闭，都要能够把实际案例讲清楚，有真实的数字，也有真实的数据。通过这样的情况，我们就能够给面试官一个感觉，你真真实实是玩过云原生的，而不是只在理论上。

另外作业调度中台，这个中台应该提供一个定时器，不需要由每个服务自身来运行。定时器有一个统一的定时器，然后它通过 API 或者通过消息队列的方式去调用每一个后台的服务，从而触发整个作业的调度和管理。另外安全中台是什么？是。认证授权，每一个服务都需要有认证授权，不管你是JWT，不管你是Oauth，还是各种各样的认证的方式，你都需要有一套完整的认证授权的体系才能够保证整个数据中心的安全。另外还有一个监控中台，这个监控中台还不光是日志监控，我们需要遥测所谓的什么链路的情况，我们要遥测完整的性能的追踪，还有知道每个 API 的延时以及什么互相之间的关联关系，这样几套中台合起来就形成了我们的技术中台，技术中台还只是一个支撑，跟 IS 跟 Paas 配合起来，再加上我们的标准的技术中台最终实现什么？以 API 经济为主导的应用中台从 IS 和 Paas 走向 SaaS 服务？是不是 4 道面试题完全就围绕了前面 15 个原则的四大分类一一展开？大家如果碰到此类面试题，是不是心中已经有数了？可以妥妥地讲清楚理论，再结合自己的曾经参与过或者自己主导过的项目实际来把什么理论落到实地。
