那么这些场景里所有的搜索其实都是有分页的，那分页的目的就是为了让搜索结果集尽可能的小，返回结果的时间花费也尽可能小。但是由于咱的主搜是一个公共模块，那你作为这种中台类的产品，咱无法约定业务方以什么样的姿势来调用。


你比如说我之前在对接 open search 主搜的时候，我有一个场景非常的庞大，那我希望去调用出当前门店的所有商品，那对于一些淘系商家来说，每一个商品它都有一个门店 shop ID，那这看起来没什么问题。那么对于某些特殊的业务方，比方说像阿里系的一些自有门店，那它的一个商店就像一个大卖场一样，它下面的商品可能有几十万个，那这样一来，哎，你的主搜就有可能不堪重负，它一次要返回很多的商品，这是其中一个场景。


还有的时候某些业务方，诶，我希望去做一个门店的盘点，我想看一下当前门店在一个时间段，比如说2月 1 号到 2 月 28 号这个时间段，所有商品操作记录，或者所有商品上架下架的操作记录。OK，那么对于这些场景来说，它有一个共同的特点是你的结果集会非常的大，并且作为一个主搜类的中间件产品，你去返回这些结果机所花费的时间也就特别的长，就是说你的 API 的 RT 会变得非常的大。那么对于这类你无法预知的一些查询维度，我们作为一个中台类商品，如何来规划它的降级方案？

那我想导入它从 2021 年2月 1 号到 2 月 28 号所有的上架商品，那毛姑姑这些商品至少得有上 10 万个。那这里的问题就集中在时间长，那你的主搜接口响应时间非常的长，那对于业务方来说，诶，你的时间长究竟要长到什么一个度啊？我究竟是等还是不等？这就是一个问题。那么被动降级是什么含义？我这里跟同学们举两个例子，等待接口返回这个过程就像我们和女神约会一样，那这个接口就是你要等待的，女神对于我们某些业务方来说，人非常的好，就像个备胎一样，说没事我就愿意等。你不在，我不在，你什么时候返回，结果我什么时候响应业务，那这种模式完全没问题，业务方愿意等，但是有的业务方不好惹。我管你是女神女屌丝，你今天对我爱理不理，明天就让你高攀不起，那我不愿意等你。


如果你超过了我本地的超时限制，那么我就做一个重试几次，还超时怎么办？降级熔断之，这就是我们所说的被动降级，它发生在哪里？哎，发生在业务方的这一端，由业务方来决定你接口多长时间不返回，那么我把你作为一个超时来处理。那这样做有问题吗？对于业务方来说没什么问题，他愿意怎么做？但是对于我们的主搜业务这类平台业务，那就存在一个问题，你的重试OK，这里就会对你的中台业务带来一个性能损耗，那么反过来业务方发生了降级和熔断，反过来会骂你的平台方，你的 open search 号称什么？什么样的性能？为什么我的搜索条件这么慢才返回？那当年我就跟主搜陷入了这个撕逼过程当中，你要推举自己的 open search，但是你又没有能力快速返回，那你让我们怎么对接？那这就是一个不好调和的一个矛盾。那被动降级，那我们换一个思路，被动为主动，对于这类平台类的业务方，我们采用一种不同的方式主动降级。


怎么说？主动降级？我作为主搜平台方，我来约定一系列的性能规约，比如说我这里做了这样一个约定，结果即要在 200 秒之内返回，并且每次查询我最多返回给你 500 条的数据。那么如果你不满足这两个条件，说明要么你申请的数据条数分页参数过大，要么呢？你的查询结果，查询维度范围太宽泛，那么这两种情况下，我都要求你去重新定义你的分页还有查询条件，以达到一个什么效果。少量多次，就像糖尿病人一样，少量多餐。那如果你想今天吃得饱，你想查询到足够的数据，那么你分多个批次从我的接口当中来获取。


通过这种主动的规约，我们可以约束业务方在集成我的接口服务的时候，它的查询维度，它的对接方式，进而可以降低我的平台接口的性能损耗。那通过这个例子，同学们应该可以了解到像主搜或者是中台类的这类商品，平台类的这种商品，你不好预测你的所有上下游调用方会以什么样的方式来查询你的接口，因此我们比较建议的方式是对于这种开放接口来说，如果它会返回大量结果集。
