第一方面是从硬件层面如何来监控你当前服务器资源，那么调优思路这里，我们从 JVM 这里出发，从 JVM 的角度跟同学们聊一聊调优的一个思路，OK，那么首先咱先从这个硬件资源来看，通常来讲我们硬件层面其实监控的指标无非是这 4 种，逃不过这四种。


那第一个 CPU 的使用率，那接下来是内存以及当前硬盘的存储空间，那最后一个是线程数，我们在生产环境当中这四个指标看，我们这里打出了一行字，生产环境全监控，OK，像包括阿里系的应用，它也会严密的监控你所有集群环境当中每台服务器的这些指标。比方说最常见的，当你的存储空间满的，那么它就会发送短信通知到你的应用的owner，这里告诉你存储空间即将用满。再比如线程数过高的时候，你当前应用的负责人也会收到一个报警信息，提示线程数已经超过了多少的阈值，请去处理一下。


OK，那么这些指标对我们的这个性能调优有什么指导意义？很简单，我们先从这个线程数这里跟同学们去讲一下。那我们为什么要做线程监控？这里它监控的是什么内容？这里主要有两个，一个是线程的数量，第二个是线程的状态，线程数量很好理解，线程状态是什么呀？就是要去监测一下你各个线程当前都处在什么样的状态之下。


通过线程数量、线程状态这两监控，我可以去识别出你当前应用是否有些线程没有被回收，或者你在应用里面没有去使用线程池管理你的线程，还可以做一些死锁的检查。那如果线上收到了线程数量这些报警其实是很好定位问题的。真正难定位问题的是什么呢？我们的 CPU 使用率和内存这两个资源，我们应用的监控策略叫热点监控。热点监控是什么意思？我会监控你当前 CPU 资源的占用，以及你当前内存当中的一些大对象，还有数量非常多的类的对象，这我们把它叫做热点，所以这个热点监控是枪打出头鸟的过程，谁占用的资源多，我就去花精力排查，你看你是否应该占用这么多资源。那我们通过什么手段来排查出这两个问题？这里就要应用到我们的 GVM 的分析，那么对于有一定开发经验的同学，相信你们一定去处理过线上的一些 GVM 的问题，比方说最常见的 out of memory，那当你一个线上的故障用到 GVM 这个手段来分析的时候，那十有八九它发生了什么呀？一个内存的泄露，OK，那么大部分团队面对这种内存泄露，是用肉眼代码扫描机的方式，漫无目的的做 code review，实际上这种方式对于一些简单应用，你可能可以一目了然，瞎猫碰上死耗子，直接找到那个出故障的点。但是对于一些大型应用，复杂应用，漫无目的的 code review 并不是一个很高效的手段，这里就要去结合使用 GVM 分析给你指一条明路。


好，那么我们这里就从内存泄露这里个角度来出发，看一下我们的 JVM 分析这里有哪些常见的问题，以及这类常见问题故障排查的方向。

OK，那么内存泄漏会发生在你的 JVM 堆栈里的各个的地方，那么这里也比较常见的一类问题，在我的老年堆这里发生了内存泄漏，就是 hip space，那老年堆的问题排查相对来说比较简单，如果我们观察你的内存波动情况，会发现它大概是这样的一种情况，线性的这样一个波浪形式的上升，最终你的内存在某个线上面，这里被耗光了。


OK，那它就像你炒股的一个波形图一样，是一个逐渐上升的过程。那么为什么这里有阶段性的高点和阶段性的低点？这里就是 GC 前后的一个比对， GC 前处于一个高点，每次 GC 后这里会到达一个低点。OK，那对于这种情况，我们不可能用肉眼来比对你 GC 每一秒的变化，那我们这里怎么做？通过 g seed 快照来做一个比对，比对你发生 GC 前后之间的内存信息的变化，这里就可以揪出哪些是可能导致内存溢出的对象。


OK，那这是老年堆的一个排查方向，我们这里还有一个 palm generation，这是咱的持久待。OK，持久贷相对来说并不容易发生内存泄漏的情况，那么如果它发生内存泄漏，其实大多数情况下是有两个原因导致的，那这里就不用同学们再费心排查了。那大部分情况，经验之谈告诉大家，咱在这个区域里面，内存泄露大部分是两种情况。第一种，你的整体应用的上下文当中使用了很多个 class loader，或者说咱大量的应用了反射机制。OK，那应用反射这个很好理解为什么 classloader 可能引起 palm generation 内存泄露，因为这跟 class loader 的底层机制有关系，即使我们加载一个对象，同一个类型的class，那么不同的 class loader 它都会把这个 class 加载一遍，同学们可以去观察。


实际上当你起一个应用的时候，可不是说你的应用只起个 class loader，它有很多个 class loader 都会一起启动 GVM 的classloader，容器 classloader 等等。那咱以前在刚刚进入程序员职业的时候，经常被面试问到什么双清委派模型对不对？这其实就跟 classloader 有一定的关系。好，那么这是 palm generation 的一个内存泄漏排查的方向。


除此之外，我们还有一些比较常见的内存溢出的情况，像 Stack overflow，这个不用多说，大家应该都知道到，只要功夫深， Stack 想把它给怼爆那还是相当容易的。递龟经常就会出现这种递龟把 Stack 给它挤爆掉的情况，okay，那么除了 Stack 以外，还有一个比较常见的 error 是什么呀？也是 out of memory 的一种，它是说不能创建一个 native THREAD，这个意思是说操作系统目前并没有足够的空间创建新的线程。


OK，其实它的原理很好理解，那咱都有操作系统，对吧？ OS 操作系统，那么在你的操作系统创建一个线程的时候，它其实不仅仅是在你的 Java 的堆中给它分配一片内存，那么操作系统它本身也是会分配一些资源来创建线程的。那么因此为什么我们在监控硬件资源的时候，会要监控你的当前的线程数量，那这就是因为当你的线程数量达到一个很大的数字之后，那么它不仅会消耗你 GVM 当中的资源，同时也会吃光你操作系统当中的资源。


OK，那么我们操作系统和 GVM 之间的资源其实是一个固定的操作系统多分一点， GVM 就少分一点。对于这类问题的解决方案有两个，第一个是治本的一个方案就是去排查为何你的线程数量过高，去找出线程数量过高的可能的原因，那么这是第一个。第二个是治标不治本的方法，怎么治标？就是我们通过 JVM 的一些调优参数，减少你单个线程资源的大小，这样的话就可以给你的操作系统和 GVM 分配更多的线程数量。


OK，那么我们看了这几种常见的内存溢出的情况，我问同学们，当你发现线上的这类内存泄露的情况，你应该怎么来处理？同学们能想到的第一个方案是不是多给它分配内存？咱平时经常做的 GVM 调优当中，最常见的一个手段就是多分配 prime 内存，多分配你最大可用的内存等等，其实这种方案也是一个治标不治本的方案，那我们正儿八经想把这个问题解决掉，就要上到 GBM 的分析，OK，那么我们在一期课程当中其实跟同学们有很多的篇幅跟大家去讲解 GVM 分析的过程以及相应的工具等等。
