那么后者这个数字是什么含义？比如说 TB 99，它就是指你 99% 的网络请求，它的平均的响应时间，这个指标是代表着你系统的稳定性。因此我们在三高系统搭建的时候，首先就要强调一个稳定性，那很简单的一个道理，你系统不可用了，再高吞吐量有什么用，对吧？因此我们在做架构设计的时候，欲速则不达，那求稳才是一个真理。

稳定性，这里资源是一个大头，那我们从微观的角度来说，从咱编写代码业务逻辑的角度，减少资源开销，有这么几点可以来考虑的。第一，降低锁的竞争。那一提到锁，很多同学都认为我应该上synchronize，那 synchronize 里面各种的底层机制背的滚瓜烂熟。但是同学们，咱从这个微观角度出发，如果咱去观察一些开源软件的项目，其实里面很少，或者说几乎不用synchronize，因为为什么呢？ synchronize 其实它的开销并不低，那只是显示的，降低了我们调用的成本，但它背后的原理，那偏向锁，轻量级锁，重量级锁之间的这些升级其实都有相应的资源的消耗。那尽管这背后也做了一些优化，锁膨胀之类的优化，但是它的开销依然还是比较大的。


我们在本地可以用另外一种方式，比如叫 case 加线程出让的方式，那这是一个非常廉价的一种本地的锁机制， case 同学们应该都比较熟悉， compare and sweep 比较，然后替换，那么线程出让就是 thread  yeld命令。那么在后面的源码环节，我们也会去解析这样一种小的本地锁的调优技巧。


那如果咱说有些地方不得不用到锁，那我们这里建议同学们把锁定的资源的范围尽可能的减少，你锁定的范围越小，你产生竞争的概率也就越低，所以我们这个目的是尽可能少的减低净争。那么除了锁以外，我们这里资源层面还有一个很耗资源的操作，就是线程，那我们这里要想一下如何来通过节省线程资源提高稳定性，这里面首先应该想到的一个操作，一个必备操作就是尺化线程尺，那打个比方我们在数据库层面，你如果每次访问数据库新建一个连接，那么你的这个 session 的建立开销这都是非常厚重的。


那如果我们有一个尺化的线程，那线程尺这样的话你就可以节省很多线程的创建开销，以及握手等等资源的消耗。那这里不仅体现在数据库连接池上，我们在任何用到线程的地方，尽可能的把它做一个线程池化，本地池化，或者是全局的线程池化都可以，这是一个基本的操作。


OK，那除了池化以外，我们这里还有一个操作叫线程隔离。那同样的一期里面，我们通过 high streaks 的线程隔离跟同学们演示了这样一个技术方案，通过线程隔离方案，我可以为你特定的一类资源隔离出来一部分线程，那么操作你资源的请求不管再多，它所消耗的都是这个隔离区域的线程，那一旦它突破了瓶颈，那你的线程不会去抢占其他资源的线程。这就是一个线程隔离操作，给指定的资源分配数量一定的线程资源。


那么最后一点，从资源角度，这里还跟同学们谈一谈服务的无状态化。在微服务的架构下面，我们尽可能的要保持所有应用的无状态化。那无状态化很好理解，比如session，如果你的业务中有通过session、HTTP、 session 来获取状态，那么它就是一个典型的有状态化的服务。如果你是一个本地 Tomcat 级别的一个session，那么你的用户请求如果当前落在你这台机器上，那一旦你下一条用户请求落在另一台机器上，那你在原先的 Tomcat 里建立的 session 那就丢掉了。


因此服务无状态化就是为了解决类似的问题，比如说刚才谈到的 session 问题，我们可以用什么？用分布式的session，可以把 session 存到 Redis 当中、 spring session 等等各种组件来把它从本地状态剥离成无状态的服务，这样是为了满足什么呢？来满足一个弹性的需求，一旦我的集群需要扩容或者需要缩容，那我们又叫它快上快下。一旦有这种操作，那我就要首先保证你所要弹性的服务一定要是无状态的，这样的话它才能去承接你的线上流量。


OK，那以上几点是我们从资源的角度和同学们聊的稳定性保障手段。那么最后我们再来看以稳定性保障的另一个利器就是异步化。

那前面已经跟同学们讲过很多节的异步编排的内容，那这种用空间来换时间的操作也是在我们生产环境的大型应用当中，经常会使用到了一种技巧，那除了异步编排，我们这里还有一个状态机的推进，那状态机加数据库保存状态信息，那这种方式将接口的及时调用，把它演变成了后台的状态机的推进服务。那么这就有点类似于我们在后台跑的 back and job 批处理服务，比方说你的订单的付款动作完成了，那在这一步完成之后，其实我是往状态机上面推送一个event，一个事件，那由状态机在后台将订单状态推向下一个状态，然后执行相应的逻辑。


那除了异步编排状态机以外，这里还要跟同学们提到一个点叫缓存的更新，那缓存和数据库的一致性一直是业界一个热门的问题，那这里缓存更新如何来做异步化？一个最简单的方式，也是阿里系应用当中非常用的一个方式，就是将缓存更新的这部分任务而下放到 Canal 这一层，那通过监听 MySQL 的 binlog 来做这个缓存化的异步更新，那其实不光是阿里，很多的一二线的互联网大厂它都有自己相应的中间件来做这个操作，那原理都是类似的。这里不仅能降低业务层的复杂度，你更新缓存不需要我在业务层里面做控制，那么另一方面还可以提高你的稳定性，一方面来说你的业务越简单，你的稳定性相应肯定是越高的。另一方面把这个缓存更新的任务从变化比较频繁的业务层把它下放到了相对稳定的中间件这一个层面，那这是我们在生产环境上经常用到的个缓存异步更新的技巧。


那除了这一点，我们异步化还体现在一个队列上面，比如我可以在本地做队列，或者通过中心化的 m q rabi， m q rockemq 做消息队列，把可以解耦，可以做异步化的任务丢到队列里面来执行。那么以上这几种异步化的方案，同学们在去执行的过程当中一定要去注意这样几点。而异步化就代表着你的业务有可能会要进行重试，其实应该说不可避免的它一定会有一些重试的操作，因此我们页面下面标注的这两点一定要考虑在内，第一个就是咱刚才提到的接口幂等性的保证，第二个一定要去建立一个重试机制，OK，做补偿。那关于三高系统的几个稳定性保障思考的点就跟同学们分享到这里了，那大家课后可以拿自己的业务去思考一下，看如何利用这些点来去提高自己系统的稳定性。好，那我们下一节呢，再跟同学们分享一点线上异常回退相关的知识。好，同学们，我们下一小节再见。
