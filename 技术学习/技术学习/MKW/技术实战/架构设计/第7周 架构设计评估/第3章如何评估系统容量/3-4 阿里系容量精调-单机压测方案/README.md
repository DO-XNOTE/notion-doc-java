比如说Gemeter、 load Runner 或者阿帕奇的AB，那我们就在线下环境当中使用这些工具来压出系统的容量。那不过这种传统的方式它有非常大的性能偏差，那么因为我们大多是采用的线下机器，和线上机器的配置、网络环境等等都有很多的差别。


另外一点，我们所使用的这个测试数据的样本往往是使用一些脚本或者人工编写的，它并不是真实的线上流量。那么因此基于这两点，如果采用这种传统模拟请求的方式，可能会产生较大的性能偏差。打个比方说你梦中情人，你预想的你的老婆是王祖先，那么最后结婚之后发现诶是王祖蓝，这个偏差就非常大。所以我们阿里系里面的单机压测场景需要去基于什么呢？基于第一个一定要是在线上环境真实的线上机器当中做压测。第二个它使用的压测的数据样本要来自于真实的流量，和用户的用户行为一模一样的数据。那么我们这里来看有三种形式的线上单击压测的场景。

第一种场景我们叫它流量复制来，怎么说？非常容易理解，那我在线上的机器当中去复制你真实的用户的访问流量，然后去拿出一台影子机器，那这台机器其实也是线上机器，不过我们把它当做一个测试机，它不再接收线上流量，而是去重播你前面录制的一个真实流量。那么这样一种场景有这样几个特点，第一个场景非常的真实，线上机器线上真实流量，那这个完全可以满足我们的要求，那它还有一个非常大的弊端，同学们想想是什么呀？以真实的流量重播，那么这样的话，如果你没有做好数据隔离，那它会影响你的线上数据，因此这里需要去做一个数据隔离的方案，那比如像阿里系当中它有一个测试标的透传，可以去很好的解决这个问题。


那但是对于我们一般的中小公司，那可能没有这么厉害的框架，那就要去好好思考一下怎么来做这样一个线上隔离的方案，因此它可能对于我们自己的业务来说，还需要一定的编程工作，在业务当中去处理这个场景。那在阿里系里面，其实流量复制也有比较广泛的应用，比如在很早之前的章节里跟同学们分享的天猫双引擎回归测试框架，那这里其实就是复制了真实的流量，然后做了一个流量重播。那我们就这个方案流量复制的方案来看，如何才能解决这个问题？假如我不想要做数据隔离，有没有一个廉价的方案？当然有了，包括阿里系也是使用了一种比较廉价的方案。是什么呢？流量转发，
