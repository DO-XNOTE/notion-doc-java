这里每一个资源都要保留它所有的历史发布记录，这样做的目的是什么？是为了让它可以快速回退到上一个稳定的版本，那这就是线上的快速回退的历史数源的一个根源。


那在发布之前，我们不仅需要做单元测试、集成测试，还有一些跑回归测试等等的批量测试场景，确保你发布的内容无误。但是这也不能保证线上是一定没问题的，所以在我们的版本发布的过程当中，我们还要做金丝雀测试，还有灰度发布，那金丝雀测试就相当于鹤立鸡群，那这里使用一台线上机器部署你的变更的代码资源，然后观察它的运行状况是不是和预期一致，那这个是近似决策史。


那我们做灰度的时候，可以做蓝绿发布，然后分批上线的这样一个方案，比如像阿里系的 a one 系统，它在发布每一个代码变更的时候，都会让你选择一个发布策略，那通常我们是会选择批次发布，在第一个批次结束之后等住一段时间，等人工验证好了之后再去进行接下来的批次发布。
那么同时对于一些页面的资源，那因为会影响到用户的前端的体验，那对于这类静态资源或者是用户体验上的变更，我们通常会要做一个 AB testing 来测试你的页面流程变更是否可以达到预期的效果，那因为有些变更甚至可以会影响转化率，所以我们在做过 AB testing 之前是不会把它全量部署上线的。


那么这个 AB testing 就要求我们在网关层，或者是在 APP 层，或者在网关后面有一个分流器，它可以去切用户访问的流量，那将一部分流量导向原先的老代码，那么将另一部分流量导向到新的代码，那么这里不仅说是静态资源，也包括后台的实际提供服务的接口，都可以做这样的一种 AB testing 的测试。


那咱这里说的流程主要是对代码变更，还有配置文件中心化的，配置中心的属性变更等等。那么对于数据变更或者数据订正，尤其是数据库表结构的变更，我们也要同样遵循一个一样的流程，那么保留你的所有版本控制记录，还有发布历史。

很多同学们觉得我写的 CQ 语句做一些数据订正，那这个不算写代码喽？不会影响到线上的一些业务流程，对吧？但是实际上咱的DDL，也就是数据模型上的变更，还有数据订正，这个也是代码，那不要把豆包不当干粮。


SQL 也是一种代码，那么我们在整个线上发布的流程当中，还要遵循一定的规范，而且是很特殊的流程。比如说我们的 SQL 脚本，通常我们 DBA 会规定让你提供两种类型的脚本，一种是全量脚本来，比如说我全量脚本当中有建表语句， create a table one、 create table two，创建 index 的语句，还有去初始化一些数据种子的语句，这个是全量脚本，它用在什么地方？那比如说我有一个新的数据中心上线，它需要重新搭建整个数据库集群，那么这种情况下它就可以去直接执行全量脚本，把所有的表数据，这些 DDL 的数据给它创建出来。那全量脚本其实就是从 0 到 1 去搭建整个数据库的环境。比如说我想再多要一个测试环境，或者我想在开发人员本机上去跑出所有的数据表，那么就需要用到这个全量脚本。那与此同时我们还会用到一个脚本，叫做增量脚本。比如我们第一次上线的时候，我们的数据库的表有 table one、two、three，有三张表，那随着我们的业务发展，我在每次上线的时候都会对这个表的 DDL 有一定的更新。比如第二次上线的时候，我去把 drop table three 这第三张这个表给它 drop 掉了，同时我修改了第二张表的一些字段，然后在第三次最后一次上线的时候，我又添加了index，然后插入了种子数据。


那增量脚本这些中间态的变更，比如说 table three 这里的变更，它并不会体现在这个全量脚本当中，因为全量脚本是最终形态当前从 0 到 1 搭建数据库所要执行的脚本，那么我们增量脚本包含了每一次的变更，那为什么要保存增量脚本？因为这是数据 DDL 的一个版本控制的手段，那比如我们当前的系统在这样一个分水岭上，那么以上的这些 SQL 全部都是运行过后的。


那么当我想做一个数据模型的 DDL 变更的时候，我只用执行对应的第三个数据发布的版本，这样就能达到类似我后台版本化部署的同样的方式，可以很快的回退到上一个稳定的数据结构版本。OK，那么这里说到了回退，但是这个数据库这里一旦把这个 SQL 脚本执行了之后，我怎么把它回退？那它可不像我们的代码版本，可以直接部署上一个稳定版本。那么 SQL 这里回退方案是怎么来执行？是这样的，我们要求所有数据变更都要同时提交两个版本的SQL，一个版本是要待执行的脚本，也就是我们这里看到的增量脚本，那它是一个正向执行的脚本，那么还有一个反向脚本是什么呢？回退脚本一旦线上出了一些问题，那么我们就会去执行这个回退脚本，来把最新提交的这个版本的 DDL 变更，把它回退回去。那正是通过这两个脚本，再加上我们的增量脚本的版本控制，才实现了数据模型变更的版本化部署。
