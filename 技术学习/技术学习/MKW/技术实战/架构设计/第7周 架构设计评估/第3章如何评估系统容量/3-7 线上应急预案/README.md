分别是预防问题、发现问题，然后快速响应。那么重中之重，大部分的精力我们花在哪一个地方？第一个环节预防问题，这里花费了最大的人力和物力，因为在一开始的环节你投入的越多，那么你最终产出的效益其实是越多的。那么这里经过前面几节的学习，相信大家对如何来预防线上问题都有非常深的体会了，那我这里也分 3 个小点跟同学们来过一下。首先第一点是精准评估，

那么我们这里将通过全链路的压测，还有日常的压测性能基线等等，对你的应用集群的承载能力做一个容量评估。并且在最终的全链路测试里面，我们再做一个精准的容量微调，从而摸底到你系统的承载能力。那么从数据和服务的角度来说，我们有很多的技术方案来去优化你当前的架构应用，比方说缓存的应用，还有像数据异构如何来提升你的查询效率，还有我们的前面跟同学们介绍过的异步编排，那这三个都是在互联网大型应用当中大显神威的几招，那在说到底层数据，我们也是大量运用了分库分表，那在阿里系的应用当中，即便你一开始一个事业部的业务量非常小，那么你在设计表的时候我们也要在一开始就要上到分库分表的维度。这样当你的业务量最终上来的时候，再去接入分库分表，就是一个成本比较低的一件事情。


那么最后我们这里还有一个DR， DR 是什么意思？容灾，做数据机房，还有你的服务机房都要有这么一个容灾，以便它可以做故障转移，快速切换机房网络。那第三点就是你系统的极限，那么我们从三个方面上来解决。第一个限流，那去限定你当前可以访问系统流入的流量，那这个限流会根据你的弹性扩缩做一个对应的变更。那限流和弹性扩缩它的出发点一般都是根据你集群当中的水位，应用当前的承载负载的压力来决定的。那这里还有一个很重要的手段，就是降级和熔断。那么降级熔断这里我们不仅要求你的服务设置了自动的降级策略，同时也要在各个关键服务上面设置手动的开关，那以便我们可以用人工干预的方式在第一时间把问题阻断住。那以上几点就是我们从预防问题这个角度出发做的一些准备和方案，那么接下来发现问题这里非常简单，我依靠的是什么呀？只此一家。监控预警，那通过你集群的当前的水位监控，以及你主链路上的核心应用的 QPS 和响应时间的监控，那么我们可以去知晓你当前服务的承载的压力，那进而触发警报，或者是触发一些自动的降级限流等等。那么从另一个角度，我们也可以通过一些人工植入的一些业务埋点，那这里可以去监测一类特定的异常情况，或者去分析某些热点数据，进而触发热点数据的隔离方案。
那么从基础平台的角度来说，我们在阿里系的机房当中都会去密切关注你每台机器的CPU、内存、硬盘以及线程数，那当某些机器触发了这些阈值的时候，在一定范围内我会进行一个自动的重启，去尝试着自己恢复这个问题。那如果重启解决不了，这时候就会发出短信通知当前集群的负责人，业务方的负责人。那最后一点快速响应，在发现问题的第一时间，就使用钉钉的招牌功能，夺命三连钉，把能钉到的人全部钉上线来解决问题。


那这个过程我们前面说到比较人性化的一个方案，是设计一个应急预案的三极响应，那不像阿里系这么简单粗暴，把一个团队的人都给盯上来，那除了这个三级响应之外，我们在每一个环节里面还会设置一个FOFO，是什么呀？ fill over、故障转移，那通常这个 f o 会连带着 D2 放在一起。考虑这里，把这个相应出故障的机房的流量把它给卡掉，转移到哪里？转移到第二机房，容灾机房，这就是故障的快速转移。


那除了这些，我们还有一个手动降级，那因为工程师通常是最了解你业务的人，那么在自动降级发生之前，我如果人为的判定你当前服务在未来很短时间内有极大可能降级，那我不妨预先把它通过手动降级开关给它降掉。那除了这些以外，我们这里还有一个叫以赛代练。什么是以赛代练？就是这个容灾演练，那集团三天两头就会做一些卡段机房的操作，迫使你的业务方倒逼你的业务方去做相应的应急预案，那这个容灾演练是强制性的，对集团的各个上下游的业务都会强制的执行。除了这个以外，还有很重要的一点叫做 RCA 复盘， RCA 就是 root cause analyst，我们叫它 learn from experience，就是从经验当中学习到技巧。那么通过 RCA 复盘，我们可以去总结这个线上故障的问题，来防止这个故障问题的二次发射。那么在这整个过程当中并不是所有问题都严重到需要盯其整个团队来看，需要做故障转移的。

第一个是影响的范围，影响的这个范围这里是指最终受影响的终端用户，那这是从用户维度来出发的。另外一点还要从钱的维度，我们在这里可以去预估接下来一小时到 2 小时的资损情况。那我们为什么会从两个不同的方向上来出发？打一个比方，比如我们的广告系统，如果广告系统挂掉了，其实它并不能影响终端用户的购物行为，它只是会影响到投放广告的这些大商家，那影响的范围其实非常少，但是它可能会产生很多的资损。OK，因为用户每点击一下你的广告，你都能收到一次钱，所以像阿里妈妈这样的广告系统都是集团的现金奶牛，即使它没有影响到终端用户，但是它的故障每一个故障都不是小故障，那背后都有人背 3. 25 的。那么至于影响的范围，这就更不用说了，毕竟电商公司客户至上，那如果客户受影响的范围比较多，我们也会相应的把这个定级定的比较高，那么级别的范围我们通常会根据P0、 P1 这样来标示。那如果是 P0 级的故障，我们通常要是在最快时间第一时间内响应，那么 5 分钟的时间是一个相对来说比较合适的时间，那可以在第一时间把所有相关的责任人全部叫起来去看待个问题。不过响应并不是说修复，而是说我有对应的团队已经开始在看这个问题，这个叫响应OK，那么相应的 P1 级别我可以设置是 15 分钟内得到响应，再往后依次时间递增，像P2、 P3 我可以设置的时间相对拉长一些，那么这里具体的时间还要根据你当前的公司的业务以及用户体量来决定。


那像阿里系的大型应用来说，通常给我们 P0 和 P1 时间响应的时间都不会太长，如果你放着一个 P1 故障一段时间不处理，那它的资损可能就会使这个 P1 故障上升为一个 P0 的故障。那么接下来我们再跟同学们聊一聊大家比较关心的一个话题，就是故障预案。

我们从咱的这个业务方来考虑，也就是我们的具体的服务业务部门的开发人员的角度来考虑如何来设置一些故障预案。那第一个我们需要考虑的必然是人工降级方案，这个是我们故障预案当中和你的开发团队是最为相关的。比如说我这里跟同学们举几个在电商场景当中非常常见的故障预案，那就是读写操作，一个不落下降级、异步写，还有读操作降级，那么读操作降级这个非常好理解。比如说我们这里的多级缓存，那你后端服务如果访问数据库有问题，那这里可以退而求其次，从缓存当中去读取。


那第二个血降级可能同学们并没有太接触过，那什么是血降级？打个比方，像一些超高并发的场景下，如果我做通常的持续写，那么像数据库当中持续的修改插入记录，可能会导致数据库不堪重负，那么这种情况我们一般会怎么来做？我们降级到缓存更新，把这个写操作落到缓存当中，比如像 Redis tire 当中，那落到缓存里之后，我们再通过一个后台进程去缓存里，怎么样把这个数据捞到做一个合并，然后再把它写入到 d b 层里面，那么这样的话你就可以大大降低 d b 层的并发连接数以及写操作更新操作的次数。


这个就是异步降级写那这类读写的降级方案，同学们记住，通常来说尽量应用在对一致性要求不高的一个场景上，那么这个降级险，其实这种方案就是什么呀？最终一致性的方案，那么同学们想一下，我把这两类归为了人工降级，难道自动降级不行吗？其实就像我前面说的，在线上故障预案当中，人工降级的口子一定要把它留下来。


那么有这样的几个考虑点，第一，我希望是让技术团队可以第一时间做出反应，永远给我留一个人工操作的口子。为什么这样说？因为有些业务你可能已经预知到它接下来一定会达到一个降级，但是现在却又没有触发自动降级。所以就像前面说的，我这里可以进行一个人工干预，把它做一个提前的降级，这是一个方面。第二个考虑点，我可以主动的对一些边缘业务做降级，尤其在双 11 大促的这些环节，我根本不等它自动降级了，我在一开始就把它直接降级掉，把这些计算资源给它贡献给主链路，那么这是人工降级的几个方案。还有一个功能性降级，这个大家理解起来非常简单，比如说我的账单模块，那账单的发送其实需要这么急吗？需要你下单之后立即发送吗？并不用，我可以把这个模块做一个功能性降级，等到我的双 11 大促流量顶过去之后，我再把它开放出来，然后通过一种补偿的形式，再把账单发送给对应客户的邮箱，那这就是我们说的功能性的降级。
那么除了功能性人工降级之外，咱还有一个客户端的限流开关，那么这也是我前面说的要自动和手动两个互相结合。那么在限流这个领域，我们不光在网关层，或者是说在一个中心化的像 Sentinel 这样的组件当中做限流，咱在客户端也可以做一些限流开关，那么这些开关有统一的配置中心，把这个开关的值推送到咱对应的服务器群里，然后就可以触发客户端的限流。


那这个推送我们可以采用现在主流的 spring Claude 的 config 组件，那每一个大厂也都有自己的配置中心，都可以完成这个运行期的动态推送。那咱在后面的章节里也会跟同学们去落地一个 spring Claude 的开关推送的案例，那最后还有一个小配角要登场，故障预案当中的其中一个就是在大促环节下，我们要禁止爬虫，那爬虫这个东西其实是个双刃剑，一般我们会允许搜索引擎对我的网站的功能进行一定程度上的爬取，那提高我在搜索引擎当中的排名。
