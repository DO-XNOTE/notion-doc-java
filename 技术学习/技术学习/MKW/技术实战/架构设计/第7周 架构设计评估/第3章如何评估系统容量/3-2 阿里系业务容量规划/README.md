OK，这四个步骤分别是流量预估、系统容量评估、全链路压测以及最后的限流降级预案。那么这几步当中，全链路压测已经和同学们在前面的小节里分享了它的概况，那么我们这里就其他的几个小节跟同学们去探讨一下这些环节是做什么事儿。


okay，第一个环节，先头部队流量预估，它做什么事情，它就是从历史数据做一个回测，给出一个大大致的流量的估算，那这个预估是业务流量的预估，那它的输入条件有两个，第一个毫无疑问是线上的历史数据，OK。第二个是我的模型预测，那通过我的预测算法以及历史数据，这样就能得出一个相对比较准确的业务的访问量。不过预估毕竟是预估，谁也不是那个神婆神算子，一旦我预估出来的流量和我真正双 110 点的峰值的脉冲误差非常大，怎么办？这个情况历史上出现过的，有一届双十一，我们预估的流量大概我记得是 14 万 QPS 左右，那最后真正的实际流量是16，那么峰值甚至达到了 20 万，OK，那 20 万和 14 之间差了非常多，将近40%。


okay，一旦产生了这种比较大的误差，我们怎么办？我们可以在其他的环节做一个修整，在哪个环节？在这里限流降级预案，在这个环节当中可以把超出我流量预估，超出集群承载能力的流量给他拒之门外。OK，那么这是咱流量预估环节所要做的一件事情。再到了第二个环节系统容量评估，这个环节我们来做什么呢？那我们想前面获得了我的预测的流量，那么这个环节我就要根据你的预测流量的结果，然后把这个流量分配到各个的业务系统当中，那么这里其实就是评估每一个业务方所需要的装机容量，一个预估的初始装机容量，那么在这个阶段会得出一个比较精确的你的集群机器所需要预置的数量，那么这个数量通常是通过业务容量除以你的单机吞吐量来做的决定。


OK，那么这里单机吞吐通常来讲，对于咱集群上面的业务，不管你是不是主链路，一般来说我们都会搭配一个两核 8G 的这样的一个老爷机，但是这台机器不是一个正儿八经的物理机，那它是通过阿里云的计算能力虚化出来的一个相当于虚机的这样一个概念。那咱为了获取到你的装机容量，那必然要进行一个什么呀？一个单机压测，OK，单机压测从这个字面意义上非常好理解，它就是为了衡量压出你这个单机它的承载能力，那单机压测为了保证你和线上环境的匹配真实性，所以一般来讲是直接采用线上的机器来做一个单机压测，这样的话能保证你的这个压测结果不会与线上环境产生较大的误差，从而影响到你整个容量，规划你的装机容量的准确性。


OK，那么关于单机压测的部分，我们会在后面容量微调的小节当中跟同学们分享阿里系的单机压测的一些场景，还有步骤。okay，那接下来预估出了你每一个集群，每一个业务方所需要的装机容量的大致数据之后，我们接下来就需要通过全链路压测做一个调整和微调。


那前面跟同学们已经分享过全链路压测的概况，还有大致的流程，那么我们这里知道这是一个集团层面的集团军大练兵，那上下游所有业务方几乎全部的核心系统都会参与其中，那么它这里的一个目的就是尽可能的模拟双 11 时的场景，那同时像峰值、脉冲这些场景也都会在全链路压测当中把这个流量打出来。

对于限流来说，它的作用点大部分是应用在请求数量。在这个方面上，我们有不同的限流的方案。比方说有一种刚性的限流，非常的刚，直接刚正面，它叫抛夫弃子。什么含义？比如说我的当前请求大于1万，那每秒 QPS 大于1万，或者我的 RT 大于一个预定的阈值的时候，我这里对后续的请求就直接咔断。


OK，那么这就是一个非常刚性的限流的措施，也是我们采取的最多的一种限流方案。它在某些极端场景下，只允许你的流量一个恒定的速率流向你的肌群。OK，那这是一个方案。还有一个是相对来说，我们叫柔性的一个限流方案。什么意思？当来访的请求大于咱的这个阈值的时候，我并不急于把它直接卡断，而是让它在门外候着，比如说把它加入到一个队列当中，那么这其实是一个削峰填谷的一个操作。OK，那么在这个情况下，那你的请求实际上并没有被丢弃，当后续的流量逐渐下来的时候，那你的集群的消化能力会逐渐的把这部分在队列中候着的这部分流量给它处理掉，那这是一种柔性的方案，那与刚性方案比，它有一个非常显著的特点。比方说我抛夫弃子，我每秒钟打1万QPS，就是1万实打实的，那么你当前集群的处理能力和你的装机容量是相匹配的，那如果我采用了这种柔性的方案，即便我们给它做了一个限流集群，当前的这个流量压力减少了，但是我可并不能把它的容量减小。因为为什么呀？因为我在门外候着的人少了，不过我在队列中，后者的这些请求可还多着。我需要集群保持当前的这个容量规划，保持当前的这个装机容量，去逐渐的消化我队列当中这些请求。那因此你是否可以去做实时的缩容，那还要去看你队列当中的请求堆积量。okay，那除了这种柔性之外，我们还有一种柔性是叫部分流控。怎么来理解？有的请求是在我的网关层就直接把用户给卡死一道门，谁都进不来，那对于部分流控来说，其实它是流控结合降级做的一个措施。那比方说我们当前有些下游应用处于一个 RT 非常高的一个状态，那下游应用其实是被上游应用调用的。


OK，那如果我这里下游应用失败了，但是我上游有一个主链路当中的模块依赖这个下游应用，那主链路可不能把它给限流住，怎么办？这时候我们可以采取一个部分流控，那有主链路这一层，也就是它的一个上游的应用主动的对下游应用进行一个降级，或者对下游应用发起一个流控，那从上游应用这一层就直接卡断了下游应用的访问，做这样一个降级处理，或者采用一定的速率来访问下游应用，那这就是我们说的部分流控也是柔性限流当中的一部分。OK，那这一节内容就跟同学们分享到这里了，我们跟同学们大致了解了阿里系的容量规划是怎么回事儿，里面的一些主要的步骤。那么在下细节里，我再跟同学们去分享一些案例，然后看一下容量精调部分是如何来操作的。好，同学们，我们下一小节再见。
