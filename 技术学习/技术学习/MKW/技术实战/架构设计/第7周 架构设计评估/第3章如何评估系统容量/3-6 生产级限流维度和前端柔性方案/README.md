一个是限流，第二个准入，第三个是整形，这个整形不是整容整形，这个是流量整形。那咱先从限流维度来跟同学们讲一下，咱一期学员相信对这个限流都比较熟悉了，我们通过Nginx、Redis、 Lua 等等各种技术跟同学们展示了网关层限流的一些落地方案。


那么从大方向来说，限流首先最主要的是一个 QPS 的限流，那么 QPS 这里 query per second 可以是一个全局维度的限流，那同时也可以是针对某一个接口，某一个特定的 URL 路径的这种限流。除了这个以外，我们还可以在用户维度做限流，比方说用户的ID，还有一些统一的资源标识符这个维度，那最后我们也可以是基于一个 IP 维度做 q p s 的限流。除了 QPS 以外，我们通常还会做一些connection，也就是连接数或者是并发线程方面的限流，那限制你这些维度可以保持 active 的连接的数量，或者这些维度上面的并发线程数量，那除了这些以外，我们在网关层的后面通常还会设置一个统一的中心化的分布式限流组件。比如说像 Sentinel 这边可以是支持调用方限流，比如我可以决定哪一个上游应用可以调用我的次数更频繁，也可以对热点做一个特殊的限流规则，那么这个部分我们会在后面的微附章节里跟同学们落地 Sentinel 的具体的案例。


那么除了限流以外，准入这里也是网关层的一个非常重要的功能。那一提到准入，我们最先想到的应该就是黑名单和白名单，哪些应用会被黑名单阻拦住，哪些 IP 端口会被阻拦住，然后又有哪些应用 IP 等等会被白名单放行，那这是网关层需要搭建的一个功能。除了这个以外，我们通常还会设置一些 DDoS 的高防，比如阿里云这里就提供了专门的高防服务来防止这类网络攻击，自动识别一些恶意的攻击流量，那么最后还有一个流量整形的功能，那这个整形其实不光是说在网关层，我们在中心化的一些流控组件上，比如 Sentinel 也可以做这个流量整形。这里流量整形一般来说分为这种几个方式。


第一个我们叫它预热，流量预热或者也叫它是冷启动，那这个很好理解，我们在流量峰值脉冲的时候，不是把所有流量一次放行，而是在一开始放行一个比较低的流量，然后逐渐的升高，给后台系统足够的预热的时间，不至于后台系统被这个峰值脉冲一下给打崩。那么除了这个以外，还可以去做一个匀速通过。那匀速通过这种流量整形的方式，它的关键点是什么呀？是记录你上一次请求通过的这个时间，比如说我在这个点请求通过了，那我设置的匀速率是多少呢？是 100 个请求每秒钟，那么我在后面就会以毫秒为单位来计算你每个毫秒可以去通过的请求数量。那比如拿这个例子来说，每秒钟通过 100 个请求，那意味着 10 毫秒才能通过一个，所以我的下一个可以放心，请求的时间点就设置在 10 秒钟以后。


那么如果你的请求提前到来，那在第 7 号秒到来，那么这个时候我可以做两种操作，第一种我可以把它直接丢弃，或者说我可以去做一个排队机制，那么排队匀速也是 Sentinel 提供的一个流量整形的功能，我可以把这个当前先到达的这个请求先给它暂存起来，匀速放行的节点时间到达之后再把它给放出来。


那么这是几个流量整形的方式，那么流控的具体的内容我们也会放在后面微附章节，跟同学们去做一些详细的落地。那么除了网关层流控，我们在生产级别通常也会应用到一些向后端降级的方式，

其实这个并不是网速问题，那移动运营商就表示，那这个锅我可不背。那这是什么原因？这是我们的 APP 里面做的一个小机关，前端柔性。那咱去买股票都知道股票交易有这么一个 t 加一，那也就是隔日买的这样一个概念，那同时前端这里也是一个类似的概念， t 加0，那这个加 0 可不是当即就可以生效的意思，这个加 0 其实是一个前端的小柔性。那么当你在点击这个按钮的时候，实际上请求不是立即发出的，它会随机在 0 秒 T0 秒之间做一个随机的缓冲，那它会延迟这段时间再发出真实的请求。那么假设这个 T0 秒呢？我们把它设置为大概 0. 3 秒好了，那么当全国的用户都在点击这个按钮的时候，那么他们的请求就会被散列到 0 - 0. 3 秒这之间来发出。那就这么一个小小的改进，我就在前端做了一个错峰处理，大大减轻了服务端的压力。可不要小看这 0. 3 秒或者这 1 秒，那它足以把一个峰值的脉冲流量把它的数值降低 2 到 3 倍，小改造，小成本，大收益。所以这种限流在 APP 端把它做上去是一个非常性价比高的技术方案。
