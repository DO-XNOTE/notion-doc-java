这里就可以用老师前面一节当中给同学们去过的一个整个淘系链路的容量预估，容量精调压测和全链路压测的这个流程，比如最开始的时候如何来做业务容量的预估，以及我们全链路压测前的一个非常重要的环节，单机压测。那么单机压测这里我们提到了几个场景，同学们可以再去回想一下，回顾一下如何来做单机压测。


那流容量预估这部分这里有两个不同的方向，一个是对于老业务已经在线上的业务来说，那我们可以去抽取线上数据，加上使用一定的模型来做回测，那预测出老业务当中的漏斗模型，在每一个环节里大概需要分配多少的计算容量。那对于新业务来说，只有看我们的产品运营团队对新业务给一个初始化的预估，那么老业务和新业务结合在一块儿，最终就输出了一个模型，这个模型就是用来接下来的环节容量精调环节，通过全链路压测去找到系统当中的所有的瓶颈，那这个瓶颈可以是中间件、业务、团队等等链路上的任何环节，然后来做相应的容量调整。那这个问题同学们还可以再延伸一点，因为全链路压测是要模拟真实的一个大促场景，那么在这个场景之下，我们要让流量真实的是从全国各地的节点打入到我们的后台服务的。


那么最后根据前链路的结果，我们在做相应的不同集群之间的水位调配，设计自己的降级熔断策略以及添加限流。那么把这部分全部配置妥当之后，我们就来换个姿势，再来一次，再去重新压两到三次，那通过这种方式使你的全链路稳定性提升一个很大的台阶，那这些内容去应对面试官的这个回答绰绰有余，因为大部分即便是阿里的面试官，他也未必有全链路调优或者是参加双 11 全链路压测的这个经验。


那么我们再来看第二个问题，那第二个是从架构层面来回答的一个问题，就是**如何来提高你线上资源的利用率，**那这个问题也是一个 open question，那看你个人的知识积累，还有相关的架构经验。

那我们这里可以从以上的几个点来回答，比如最容易想到的一点，监控应用，当前的负载能力水位监控，然后利用弹性计算将闲置资源分配给水位较高的集群应用当中，那这是一个层面，基本上讲到这里其实也就足够了。因为我们这个问题的题干如何提高线上计算资源的利用率，其实就是一个资源复用，那如何节省线上的服务器？我们不用非常详细的去讲代码层面的一些调优，这种比较open，比较上层的问题，我们就用上层的解决方案来回答，那其实这个点就已经达到点子上了。那如果我们后面还想再戏虐一下面试官，再去秀一下面试者的优越感，我们可以跟他们讲混合部署，那像阿里系的混合部署的策略，我们跟他去聊上几句，那么基本上面试官就会对你刮目相看了。那这个混合部署我们可以去参照前面小节当中老师跟同学们分享的这个内容，那么我们继续往后说，那现在又来了一个新的面试问题，也是从架构层面和工程层面两方面考察同学们的。那这个问题就是说如何防止流量超过系统的承载能力，那么这个问题同学们想，我不是说当前流量已经超过了系统承载能力，那已经超过承载能力，那没什么好说的。添加限流规则、打降级，OK，那只有这几种方式。但是我们这里题目问的是如何防止流量超过系统承载能力，也就是说当前还没有超过，给我们足够的时间让我们在代码层面、宏观、微观层面都做一些调整。那么所以面对这类问题，往往优秀的答案其实并不是回答几个点，比如限流，比如降级，这些点上的知识很多人都知道，那我们怎么来把这个问题回答全面？那就是形成一个完整的解决方案，由点即灭，把整个解决方案从前到后串联起来，形成一个海陆空三军混合作战的一个方式。


那么我们看这个题目如何来去拆解，比如说我们从网关的入口层面可以这样做，回答一个限流，OK，限流是正确的方向。那么我们有哪些限流方式？在网关层添加的限流，以及服务内部分布式组件的限流，还有客户端本地限流，多个维度来共同防止流量超限。那为什么只有网关层流量是不够的呢？因为你很难预估在真正的线上环境会出现什么情况，即使你网关层服务添加了限流，但是因为你内部的微服务的调用关系非常复杂，它也有可能在服务集群内部的微服务互相之间调用的时候，把某些服务节点容量不高的节点打崩，所以我们要在服务内部通过分布式组件还有客户端再多加几个限流的维度。


那么另一方面非常老生常谈的降级和熔断，那通过监控预警去密切的观察你当前机器的硬件资源、网络资源，尤其是它的响应实践、 QPS 等等，来及时的把一些服务给它降级掉。那打开熔断开关，这里有两个方式，一个是自动降级，还有一个是手动降级，那我们都要把这两个点给达到，那这两点基本上是后台服务所需要关注的内容，那我们在这里还有前端不要忘记用户的第一层入口其实不是网关，它的第一层入口是什么呢？我们前面刚提到前端的柔性，那可以利用前端 APP 在前端做一些小技巧，那这里可以把用户的流量打散，做一个错峰和削峰。


那除此以外，在前端柔性之后，用户流量到达网关层的时候，我们还可以在网关这里做一个什么呀？一个流量整形，把它做预热或者匀速流量，然后对于一些多余的流量，我们可以采取一个排队的措施。那么最后我们这里不要忘了，弹性计算是高可用的一个非常重要的技巧，那么基本上每一个和高可用相关的问题，都可以把弹性计算给它放上去。那咱从以上这几点，这就形成了一个什么呀？一个完整的闭环的解决方案，那基本上这个问题防止流量系统超载这个问题，从这些答案上出发，基本上可以说是天衣无缝了。
