还是说我在底层的一些数据模型上面，每个不同的服务之间，它的耦合都是非常紧密的，那这样一来，你的底层的谁？ Oracle 他这里就吃不消了。强如Oracle，你的底层连接尺在这个量级下面也会变得不可用，不够用了，数据库就达到了一个自己的性能瓶颈容量的极限。


那尽管我们这里采用了像主从分库、分表这些策略，但是由于你上层应用的耦合紧密，导致了你很多不同应用其实访问的都是同样的库表，那么在这种情况下，如果要去增大你的集群的吞吐能力，靠去扩展上游应用给业务系统来增加服务器、机器资源，那是完全不可行的。那为什么说业务系统无法继续扩容？那其实这个就像一个木桶理论一样，你的木桶它不是由一圈木板把它拼起来围起来的吗？那你最短的那块木板就决定了你的桶子里面你可以盛多少的水。那对于我们现在的这个情况来说，最短的一块木板就是它底层的这个数据库，那么你即使再把应用集群扩大，那你底层数据库的吞吐能力就摆在那里。


所以我们才说上游应用系统的扩容其实无法解决这个问题，那臃肿耦合紧密的业务层应用，加上一个达到瓶颈的一个数据库，那么这样就像一个什么呀头重脚轻的大头娃娃一样，那走着走着就要跌倒了样子怎么办？那这时候头痛医头，脚痛医脚这个法子其实在 it 领域并不一定是行得通的。
那同学们想，我们就拿唉这个交易系统来说，那交易系统它这里一方面来说它和商品系统耦合非常非常的紧密，那与此同时它又会和像支付宝、像淘宝这种终端应用平台绑定的也非常齐，那我们在当前的这种业务体量下，那这就成了一个滚雪球的效应，你这种巨无霸的打包起来的这种应用结构，那你这些服务抱团耦合一个滚雪球效应，你今天压垮DB，明天不知道压垮什么其他中间件了。


所以我们必须要把这个应用的耦合这部分问题，把它解决掉，我们整体的业务，整体的架构、吞吐量才可以继续往前走。所以同学们，我们这里要做什么呢？没错，服务拆分，OK，所以同学们，你们看像淘系业务，它也不是一开始就是高大上的微服务架构，那这些高大上的技术其实也并不是说你越早用，在项目越初期用就越好，而是说你在什么样的时期来做什么样的事情，把握好自己的这个项目的节奏，利用好现有的所有资源，不要去做过度设计。你应用在什么样的体量，我们就做什么样的架构设计。你并发数量3000，并发数量3万和 30 万，那是完全不同的设计，这里我们不讲究过度设计。


OK，那说回到拆分了，那怎么拆？那这也是个问题，那我必须把你底层的这些基础服务做一个继续拆分，你从底层开始向上扩容这样的方式，从底层开始动刀子，那你上层的服务应用才能继续扩展，那才能去容纳以后的增长潜力。OK，那我们这里对于业务拆分来说，同学们想我们对微服务的理解，对微服务的理念的认识是什么？
