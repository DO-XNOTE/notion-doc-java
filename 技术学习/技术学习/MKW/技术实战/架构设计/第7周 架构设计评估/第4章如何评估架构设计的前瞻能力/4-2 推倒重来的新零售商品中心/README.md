那什么是可扩展结构？我们来听这个故事，那之前老师在前面小节当中也跟同学们分享过，那大家知道以前我们最早做商品中心的时候，它线下只有一家门店，所以它的这个商品和门店之间的关系非常的简单。那这种关系也是淘系统当中通用的一种关系，就是一个商品它背后对应的一家门店，这个概念很好理解。


那张三在淘宝上开家淘宝店，李四开家淘宝店，即使他俩卖的商品都是同一个，比如都卖同一款绿帽子，那进货商都是一样，价钱都是一样，但是这也是两个不同的商品，那这个概念非常好理解。不过在新零售的业务当中，这就出现了一个瑕疵，为什么呢？因为新零售业务它的这个供货商是谁？是一个中心化的供货商，那它的门店通常也都是自有门店，那这样一来你的每个商品它就可以在多个门店当中售卖，并且它确实是同一件商品，那这就对我们的商品中心产生出了一个很大的冲击。


为什么呀？因为我同一件商品，今天在这个店当中，我的库存数量是 3 个，那么在另一家店当中，我的库存可能是一个，因此我们的商品中心就要做一层改造，那把一个商品发布到不同门店，但是在这一层中间，我们在这里每个门店会管理自己对应的库存数量，那这个主商品依然是一个商品。


好，那么这种结构短期内可以解决库存的问题，但是长期我们来看又发生了新的问题。比如说我的每个商品，我在不同的门店当中，我的定价策略还不一样，并且我的宣传图文也可以不一样，并且我在每一个门店我参与的营销活动我也可能不一样。那这样的话，如果我还保持一个商品这种结构，那么完全不能满足这种扩展的要求。所以我们几乎是在三个月的时间内，把整个商品中心重新改造了两次，由 1 到 2 再到 3 这样的一个版本。
那一个主商品下设三个不同的子商品，那这每个子商品在淘系链路当中，其实它都算是一个独立的商品，只不过在我们的这个业务里，把它抽成了主商品、子商品这样的一层结构，然后每一个子商品对应的发布到它所属的这个门店，那么这样一来我就可以在淘系的上下游链路当中，营销链路、商品链路、搜索等等对这些子商品加以个性化的定制。


那我们之所以在短短三个月时间内重做两次，是因为一开始在架构层面上面，我们没有梳理出这个业务模型，那么这就让我们明白了一个什么道理，在做架构设计的时候，一定要去先了解，去分析自己业务的特征。第二，不要基于经验主义来设计。那所谓经验主义，其实有电商行业的经验，这固然是好事，可以指导你的架构设计。但是一旦走入了经验主义这个误区，那事情并不总往好的方向发展。比如说在一开始做这个单商品、单门店这种对应关系的时候，淘系的整个业务链路全是这种架构设计，那这种结构我可以在淘系当中玩的转，那我理应在其他电商行业当中都可以玩的转，这就是基于这种经验做出的底层的数据模型的结构。


那么在这里犯的一个最重要的错误是什么呢？我们没有去好好的去了解你的业务模型，那这就是架构层面的一个懒惰。如果我们去了解业务模型，从线上业务走到线下业务，这里面一个很大的不同是什么呢？我们去想家乐福、沃尔玛这类门店，它是典型的线下业务，那它和我们的新零售业务的模型就非常相像，那它就具有这种主商品、子商品的业务模型概念，那和我们线上的电商业务，一个门店管理自己的商品这种模式有非常大的差别，所以在这个过程当中我们也消耗了大量的人力物力，短短三个月重做两次商品中心，来弥补一开始架构设计上的缺陷。


所以同学们以后在做底层的架构设计，这种数据结构设计的时候，一定要尽可能的去想象一下可扩展性。那这个可扩展性怎么来想象？那就是要去尽可能多的深入了解你的业务模式，业务模型，来设计你的底层数据结构。OK，那么这是第一个例子，那接下来我们跟同学们分享第二个例子，它跟微服务有关。那我们这里第二个很惨痛的经验教训就是微服务化推行的时间过晚。

怎么说？那一开始为了快速上线，为了快速验证商业模式，我们不无可厚非的采取了一种超快猛的开发模式，很简单，所有的应用全部打包在这一个外包里面部署，那这一个外包里面所有商品、订单、库存相关的任何的服务全部都在里面，所以它是一个典型的单体应用。那后面随着业务的快速发展，以超乎想象速度的发展，因此微服务化进程箭在弦上，不得不发。但是我们在做微服务改造的时候非常的痛苦，从一个单体的应用当中把所有的服务剥离出来，非常的痛苦。那么在这个过程当中怎么样来减轻这种痛苦？一个很简单的方式，那就是即便我们是用单停应用，也要对它进行模块化。


咱就拿电商项目来做举例，那一个单体应用可以很明显清晰的画出几个大体的模块，比如说商品，还有我们的库存以及订单服务，还有订单后面的履约相关的服务。再往后你的这些打折类的服务，营销优惠的这种服务，以及购物车服务，还有你商品的主搜业务，最后你后台的业务系统，那其实这样一划分，你可以把一个单体应用划分成很多的模块，那这些模块在你的单体应用当中应该是一个什么样的形式存在？是不同的包名吗？一个 Java 目录下面 12345 并排出一堆包名，并不是。


同学们这里我说的模块化是什么意思？把它从 palm 级别给它抽出来，这每一个单独的模块，我们把它抽成一个子项目，紫的module，它也是一个 Maven 的项目，只不过我们在这些所有的子项目，这些 palm 文件的顶层，我们把它拎出来一个负项目，那这个负项目来做什么事儿？负项目可以做成一个单体应用的打包，那他把每一个子项目的 palm 文件全都给他包含进来，那么这样一来，我们就可以通过一个主的 parent palm，把这个所有的 service 全包含进来，打包成一个单体应用。


那这样有一个什么好处来？很简单，有朝一日我们要做微服务改造的时候，同学们 parent palm 直接抹杀掉，把所有的微服务改造基于每一个子模块，直接原地冻牵，原拆原件，这个模式可以大大的减少你微服务化的成本。试想如果同学们这里的单体运用是在一个 palm 文件下面，根据文件夹来命名区分每个模块，这样的话你要做大量的工作，把这些文件夹单独拆分出来，成为一个单独的应用，这个不光会消耗大量的时间成本，同时你在做变动的时候，你这相互之间错综复杂的依赖关系你怎么来处理？那如果我们在一开始把这些都区分成了 palm 文件，咱把之间的这些依赖理清楚，尽可能的只引入一些公共组件，比如说我这个子的palm，下面我还可以再分一些具体的 palm 文件，把其中的公共模块，需要别的项目引用的模块，我单独放到这个子 palm 下的子palm。那这样一来我在其他的这个项目当中子 palm 当中去做变更，不会影响到其他服务，那这会大大降低你在微服务改造过程当中产生的冲突。

那我们一开始是使用的一个单体应用，前面跟同学们讲过，那它后台的数据库也非常的简单，就是一个单体加上主从库。不过在淘系的巨大流量的加持下，加上手淘的引流，那我们这个业务发展得非常非常之快，所以匆忙上阵去做了分库分表，那这个过程其实非常的被动，那我们在上分库分表的时候，也是短暂的做了一些业务的下线，那也影响到了线上的业务。


那我们如果以一个比较有前瞻性的架构眼光考虑这个问题，那就可以这样做，毕竟淘系的流量是非常大的，这个我们可以预知到，那么在这种大流量下，其实可以去尽早的去考虑来做分库分表，那么如果我们过度设计这是一种资源浪费。没错，这个老师承认，因此我们可以这样做，我们可以做假分库，也就是说实际上咱把分库分表的这个中间件，把 sharding j d b c， proxy j d b c 这种中间的 framework 把它上去之后。但是我们底层的数据库还可以，在业务量比较小的时候，我不上很多个分库，或者是很多个分表，我就是一个单库，可以吗？完全没问题，等你的业务量上去之后，我再一变2，然后 2 变4，根据你的业务量逐渐的扩增分控分表，那这样一来你的业务在前期就有了一个很好的扩展点，那么以后如果业务量扩增，我只用改哪一层，只用改中间这个分库分表的这一层的逻辑。


像分库分表这类技术，它有一个特点，就是初期在项目刚开始的时候，你的改造成本是最低的，那当你的项目真正到了一定的规模，你再匆忙去上线分库分表，那么这样一来你的改造成本也很大。所以对于一些可预知的业务量会有爆发性增长的业务，那比方说现在大厂推行了一些新的业务方向，那它很可能就会迎来一个流量爆发，那对这类业务，那我们可以尽可能的在项目初期把这些扩展点预先的考虑到加到项目当中里去，那这是跟同学们分享的第三个例子，那接下来我们再来看最后一个例子，那叫万事有始也有终，一个终止状态。

那这终止状态背后的故事是什么？我们这里就拿商品发布这个案例来跟同学们讲一下。那商品发布它牵扯到的上下游链路非常的长，里面的坑也非常多，因此我们在这个发布过程当中采用了异步化的方式，那一个商品发布请求是在后台通过异步 MQ 来触发的，那在这个过程当中如果有任何异常，我们会在商品的一个相应的位置打上一个标记，比如说商品发布淘宝服务失败，那是一个错误码一，那商品发布库存失败是错误码 2 等等。
这种失败的异常同时在后台使用一个补偿 job 批处理来去做这个补偿，重新触发商品发布的流程，那么这个触发动作也是发到 MQ 当中进行重试，那这看起来是一个比较完善的技术方案。不过这里有一个什么问题，对于一些数据异常，如果没有在前端做很好的校验，那把它提交到了后台，这样的话你重试失败了之后，你再次重试它还会失败，因此它就会永远处在这个失败的过程当中不停歇。这样一来经过一段时间之后，你线上的 MQ 它的积压就会变得非常非常多，你会发现它每时每刻这个 MQ 当中都有大量的 case 在往里面跑，那他们这些商品发布请求就是不停在重试，然后不停在失败，那么这里犯了一个什么样的错误？我们做技术方案架构设计的时候，同学们一定要记住，我们一定要设置一个终止状态，任何方案要设置一个退出条件，否则他就犯了一个。我们学写代码的时候最简单最容易犯的一个错误叫死循环。那因此这里一个破局之道是什么呢？死性队列就可以很好的解决这个问题。如果你重试几次依然失败，那么就把这个商品发布请求打，打到死刑队列当中就可以了，由人工来排查。


那这最后一个例子也告诉我们，做一个前瞻性的架构设计，安妮要考虑两个点，你不光要考虑失败补偿，那这个点很重要，考虑到非常好，那同时你也要去考虑到这个链路的闭环，也就是它的退出条件。所以我们说方案设计、架构设计一定要是一个有始有终的过程，有时候终点反而更加的重要。
