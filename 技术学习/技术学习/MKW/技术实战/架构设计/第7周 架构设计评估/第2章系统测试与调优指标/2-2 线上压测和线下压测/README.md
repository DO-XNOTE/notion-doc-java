咱一说起系统压测，哎，就会把它分为两个方向，以一条线为界，把它分为线上压测，还有线下压测。那咱先不管线上线下是什么含义，那么压测来讲统一个目的就是做稳定性和有性能的测试。那么压测这里有几个大家都比较熟悉的要素。首先我们得要有一个压测方案，那么在方案当中去指定我压测所需要压下去的指标，比如它的QPS，流量等等，那最终产生一个压测的报告，报告中我需要体现出在当前的这个压测指标下，那你的接口所返回的响应时间，那响应时间我们这里要统计出你的平均时间最大最小时间，那么计算一个方差okay。那除了响应时间以外，还有一些像机器的负载、 QPS 等等指标都是要包含在你的报告当中的。那么对于大型应用上来说，主链路上面压测必不可少。每年淘系的双十一都是对主链路的一次重大考验，两个月的浩浩荡荡的全链路压测，都是为了压出你的主链路的性能指标。


OK，那么我们就这两个测试方向，线上和线下来跟同学们分别来探讨一下。首先咱们来看一下线下压测指的是什么？那线下压测非常简单，

通常我们可以在本地使用 g meter 打出自己的压测流量，在本机上进行测试，那么通常来讲它的测试范围也比较小，比如我可以测试某一个特定的接口，那甚至是可以测试我应用当中的一个小模块，或者我自己封装的一个小型的组件，那通过这种小规模的压测来测出你的接口它的性能，或者说是组件性能来做这种细粒度的调优。


OK，那么这种测试方式有一定的局限性。首先显而易见的环境上面有差异，那我们在线下执行压测的机器，你的硬件资源首先就跟线上真实环境当中的服务器是不一样的，那你测出来的线下的性能指标和线上服务器相比有一定的不一致。第二点，你压测环境当中数据库的数据量也不一样，那么线上的库一般我们会做主从备份、分库、分表等等，那么线下库一般也就是一个这样的单库的性质，并且每张表每个库里面的数据量，我们的线上环境是要远大于线下环境的，那么因此你某些接口在线下环境可能性能非常非常的优异，但是一到线上就挂，为什么呀？因为你的数据库性能不一样，你线下数据库的访问压力还有数据量给数据库带来的这个承压都是非常小的，因此它的性能报告未必准确。


OK。再有一点，你网络部署的环境也不同，那线下环境，你的接口与数据库等等各个服务之间的网络通信的时间和线上环境相比，肯定是存在不一样的。但因为你线上环境往往组件与组件之间的交互要经过好几道的防火墙，那即使是内网，这里也有几个网关需要来跨越，OK，那么这是环境上的差异。


除此以外，我们在测试场景的多样性上面也是有所不同的。那线下测试一般我们是自己造一些测试的样本，那跟线上流量的多样性程度是完全不可比的。线上环境有用户的真实的行为，那么在线下环境很难去模拟这样的测试样本。因此对于线下环境来说，一般我们就找一台和线上服务器性能相近的机器来简单的压测一下单机的性能。那总的来说，线下压测只能提供一定的参考性，属于小打小闹，那我们正儿八经的是线上压测，OK，那线上压测我们就要真刀实枪的上一些真功夫了。

第一，我们先从这个流量上面给同学们做一个分析，那这个流量就是指咱线上压测所需要的数据样本。OK，这里的一个方式是采用模拟流量，那模拟流量是什么意思？比如说我可以使用一些工具来模拟生成一些流量，打个比方，我现在一键生成 100 万个商品来供你的线上压测做测试，那么这些商品都是使用一些随机属性生成的多种多样的测试数据。


OK，那这种方式叫做模拟流量，那它可以是通过我们的程序来自动生成的，或者是人为的加入了一些加工。那通过这种方式打造的流量，它可能有一个弊端，就是有可能会产生热点，为什么呢？因为我们数据的样本的种类如果比较小的话，那么它有可能产生比较相似的数据。比如说我淘系商品有十几万个商品类目，那么我做模拟流量测试的时候，有可能我只挑其中的个别几个类目来创建商品，尽管商品属性不一样，但是它的商品类目是一样的，这就是有可能造成一些热点，也就是商品的类型过于集中。


OK，那么为了解决这个问题，通常我们会采用一个线上导流的方式，那把线上用户真实的访问流量作为你的测试样本，那这里有很多种途径，各个互联网公司也是自己大显神通。比如前面老师跟同学们去分享过的一个天猫的双引擎测试，同学们还记得吗？用这个例子我们学习了分析问题、解决问题的三段论的手段。


OK，那它就是一个可以从线上将用户真实流量导入到影子机器的一个测试框架。OK，那除了这个阿里系的高大上的应用以外，咱平常在外面还用不到怎么办？这里不用着急，我们也有一些非常用的工具，比如说 TCP copy，那通过这类的开源工具，我们可以很方便的把线上真实流量给它 copy 过来，甚至我们还可以做一个什么呢？流量放大，我们可以把线上的 1% 的流量通过这些工具把它扩增到 2% 3%。


OK，那么这是从线上数据的样本层面来考虑的两个点，那我们接下来再从压测的作用点来考虑一下，那作用点有什么不同？我们的线上压测其中有一个目的，我要压测出你的单机的处理能力，那么这里它的作用点就是单机，这个单机就是指集群当中的一台服务器，我对它进行玩命的压测，我从而估算出你这个台单机服务器它的能力的极限可以到哪。那针对压测结果做一些 GVM 调优，单机调优，争取把你的单机性能尽可能的调优到极致。


那么单机压测这里有一个弊端，同学们想到没有你的单机后台访问的这些数据库资源，中间件资源，那它们在单机压测的场景下面未必会成为一个瓶颈，但是这些中间件资源一旦放到集群环境当中，就有可能成为整个集群当中的瓶颈最薄弱的一环，所以除了单机压测以外，我们为了验证整个集群的负载能力，我们还要做集群或者是全链路的压测，那从而整体的评估你整个集群的负载能力，进而找出你整个链路环节当中的这个瓶颈。
哪一个模块，哪一个具体服务是整个链路过程当中的吊车尾卡脖子的模块，那我们可以对它进行比较有针对化的调优和优化，那在后面的小节里我们会有专门的章节对全链路压测做详细的介绍。OK，那么接下来那我们从一个更高的视角来看，线上压测有几个点非常需要注意的，那这也是作为架构师，你在去设计你的全链路压测场景的时候，必须要考虑的几个点，比如说第一个，那在我们做写测试数据的时候，也就是说在真正的生产环境当中去写入测试数据。那么这个情况下我需要做一个打标操作。


很多同学可能不知道打标是什么含义，那在阿里系的应用当中，上下游全链路测试过程当中的所有的测试的门店、测试的商品下的测试订单，我们这里都会给打入一个标记，一个特殊的测试标，那这个标记相当于是什么呀？相当于是个通行证，那么你在整个链路的上下游环节当中都可以很方便的识别出你这个数据是测试数据，它不是一个真实的商品。


那么当有一个外部的真实用户尝试去购买你这个商品的时候，那可以对它进行一个错误处理，报错，那这是你做测试方案架构一个非常重要的一个点，要叫什么呀？叫生产数据隔离，那把测试数据和生产的真实数据隔离出来，所以这里一定要设计一个打标识别测试标记的这样一个框架系统。
OK，那除了这一点以外，我们还要去注意读、写接口场景要去混合来处理，这是什么意思？就是说咱在设计你的测试样本的时候，记得要将读接口和写接口这两个场景混合起来做测试。为什么这样说？因为在很多的业务下面，读和写之间它们有一些一致性的要求。某些服务当中可能对你操作的资源做一个什么操作？一个加锁操作，加一把锁，那么这样一来某些写场景就有可能因为这个并发锁的存在而导致失败。那如果想暴露出这类情况，那我只测试读接口或者只测试写接口都不一定可行，那必须要把读写接口，把它像真实场景一样糅合进你的测试脚本过程当中，那才能尽可能暴露更多此类的问题。


OK，那么最后一个需要同学们注意的一个点是什么？在设计你的数据样本的时候，一定要做到数据散列。OK，散列是什么含义？我这里给同学们打一个比方，那比如说我是一个资深的电商用户，我经常去下单，有些订单是一年以前的，有些订单是昨天发生的，那么我在真实场景下面，我很可能继续搜索过去很久之前的一个数据，也有可能去搜索近期刚刚生成的一个数据，那么在我们的业务当中这种老旧数据我们通常叫它冷数据，那么最近生成的这些相对来说是热数据。


OK，那如果咱在准备测试数据的时候没有充分做到散列，那你准备好的这些数据很可能就是这些相对来说热的数据，那么这并不能很好地反映你系统的真实的处理能力。那么因为你的冷数据这一个部分没有被 cover 到，所以在做测试数据准备的时候，去做这个方案设计的时候，一定要考虑做尽可能多的数据散列，把每一个不同的样本集的数据都把它准备到位。
