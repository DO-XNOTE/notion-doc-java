OK，那压力测试它有一个非常显著的特征，一提到压力测试，同学们就应该想到的，它是一个循序渐进的过程。什么循序渐进？流量，你跟系统施加的这个压力，它是一步一步，这个得寸进尺的这个过程。那通过这样的一个过程，我们要衡量出一个性能变化的一个曲线，OK，那从这个曲线当中我们要抓捕两个点，哪两个点？第一个你系统的最佳状态，第二个点你系统的哎，崩盘点，OK，为什么需要抓取这两个点？同学们看一个很简单的曲线，当你的流量逐步增加的时候，你的系统所能接受的QPS，所能去承接的并发量、业务量，它大致是一个这样的曲线，一开始是逐步上升，然后相对平缓，到达一个点之后就开始逐渐下降，然后直到系统彻底崩盘。


那么随着这个流量逐步增加的这个过程，我们希望获取到两个拐点，哪两个拐点？第一个性能达到最佳状态的这样一个点，那第二个点就是系统崩盘的这个点，那通过这两个点我可以清晰的知道系统在什么样的压力范围内是处于最佳的运行状态，那么在什么样的压力下，那会处于一个崩溃的状态。


那我接下来用一个案例跟同学们形象的比喻一下什么是压力测试。这就像有些头铁的同学们以身试法去摸电源线，想测试自己身体的对电压的负载能力，那你在一开始测试之初没有压力，没有通电的情况下，那对自身是一点影响都没有，然后随着这个电量逐渐增高，到达了你的人体所能承受的安全电压的时候，那这个时候就是一个疯狂秀操作的这样的一个拐点，那处于你整个人身体的最佳状态。那这就有如像啊接受了辞报步兵杨永信教授的这个电击疗法一样，整个人都感觉精神了，这就是你身体的最佳状态。那接下来当我把压力，把电压逐渐增高的时候，这样就到达了人体不能承受的一个点崩盘。那结果怎么样？还能怎么样？去西天取经呗。


okay，那这个就是我们压力测试的一个目的，找到这两个拐点，确认系统的最佳状态，还有导致系统崩盘的这个负载压力的点。OK，那就是这两个点了。好，我们再来看一下**与压力测试相对应的这个负载测试，**

它是什么目的？同样它这里也是一个流量逐步递增的一个过程，逐步增加对你系统施加的访问流量的压力。那么这里有一个不同的情况是它会标注一些性能的指标。什么意思？比如说我这里约定你接口的响应时间要小于 100 毫秒，还有我可以去约定你的系统是占用网络带宽的，传输速度要在 100 兆以内。OK，那么接下来我还可以去做一些要求，比方说你在我的云服务器上面，你所占用的总内存要低于一个阈值，或者你的 CPU 使用率，那你的算力要在一个范围之内。OK，那么这几个同学们看起来像什么呀？像一条红线，那这个红线可不是阿里的 P9 一碰就没的红线，那是结结实实的打在这里的不能逾越的一些鸿沟。那么我们的负载测试是为了测试什么呢？在某些指标之下，我要测试在临界资源的时候，也就是说当我的流量逐步增加，使得我前面定的这些指标当中的一项或者多项达到了一个临界值的时候，你系统所能承载的一个最大的负载量，okay。但是这里要跟同学们强调了一点是什么呢？但负载测试并不是压力测试，它的目的不是要把你的系统给压崩盘，压到不可用，那么它到哪里为止？就像高手过招，点到即止，在危险的边缘疯狂试探，但是不让自己陷入危险，它到达你的临界值的之后，那么就点到即止了。OK，那这些临界值呢？往往是你的资源的一些限制，比方说我限制了你在我的整个云服务器上面所能访问的内存，所分配的内存总共有多少t，那么当它达到饱和的时候，即使你的集群当中还有多余的资源，但是我不希望你的业务侵占更多的内存。那么这就是咱们负载测试的一个目的，那测试你的某项或者某几项资源达到饱和状态时候，你系统所能承载的最大负载。
