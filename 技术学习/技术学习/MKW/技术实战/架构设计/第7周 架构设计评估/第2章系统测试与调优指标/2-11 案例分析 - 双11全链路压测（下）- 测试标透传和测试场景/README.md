okay，那么对应到压测平台这里，压测平台是使用我们前面数据工厂这里生成的数据来根据约定的流速还有量级，从各个的不同地理位置的节点向我们的业务方打出自己的流量，那这两者之间的关系相当于数据工厂是提供弹药，那么压测平台就是专注于打输出，一个辅助一个 DPS 的定位。

那网关层这里是咱正儿八经的线上组件，一方面去承接咱的线上流量，正常的用户访问的流量请求，另一方面承接我们的压测流量。压测流量这里有一个小机关，就像前面一节当中跟同学们分享的，它有一个什么呀？有一个测试标记标识了你当前的这个流量是来自于压测平台产生的流量。那么这里在我们的网关层还有高防的组件里面，我们就要去做这样的一个适配，去更改咱的 d DOS 判定，或者说是放宽某些安全防护策略，不然的话这个压测流量很容易被识别成是一个攻击流量，那这里是要对压测的流量标记做的一个放宽处理。


好，那接下来这个流量从网关层入到我们的中间件，那这个中间件可以是业务层、网关或者是 RPC 等等各种类型的中间件，他们都有一个共同的能力，测试标记透传，这是在阿里系当中每个中间件团队做的一个比较大的改造，可以从上游接收到测试标记，并且透传给下游的谁。


下游的业务应用方，那咱的业务方在拿到标记将要去落表存储的时候，是否需要自己来决定该存到哪个数据库里，其实并不用这么麻烦，我们在数据层的路由上面将通过中间件来做这样的一个判定，来判定它是向左走还是向右走。对于一个正常的用户访问流量，我们就出门左拐把它放到线上的 DB 或者是线上的缓存组件当中，那么呢，如果你是一个带有测试标记的流量，那么出门右转只走不送，把你去存到对应的影子区域里面中来，那这个影子区域是什么呢？我们来看下一幅图，这个影子区域听起来虚头巴脑的，是不是就是一个空实线呢？并不是。

那么它和正儿八经的线上 DB 缓存，它们之间的表结构是一致的，比如我们线上的 MySQL 是什么样的表结构？影子区域当中也是保持同样的DDL，那么还有一个特点，它是一个可擦写的 d b，并不是一个只用来做只读验证，同时它也是一个可以做写操作的一个区域，在很久以前没有影子 zone 的时候，为了不影响线上测试，我们只能去做一些读操作的验证。那么这里引入了影子区域之后，我们就可以在这里面我的地盘我做主，可以去随意的擦写读写，okay。那么在去做咱的全链路测试之前，我们还要对影子区域做这样的一个预操作，就是系统缓存预热。


为什么的？我们的双 11 大促之前，在0点之前，我们通常是要在各个中间件当中去把它的一些预加载的信息先加载起来，比如说我们的营销优惠计算规则引擎，它背后就要做很多营销规则的预加载，那么体现在影子区域里面也一样，它要模仿咱正常业务的相同的业务路径。


因此这里不要忘记在双 11 链路压测的时候，一定要去做好系统缓存的预热，那么咱双 11 压测究竟压什么样的场景？我们这里跟同学们来分享这样几个场景。第一个重中之重，我们首要当头的任务是做峰值的模拟，它模拟的是什么呢？它模拟的就是你在双十一 0. 321 哨响的那一刻，巨大的流量洪峰的冲击，那这就是一个大醋态的压测，用来观察你系统是否能扛住这一波洪峰脉冲。除了这个以外，咱还要去做一个降级演练。什么是降级演练？这里面导演组给到我们两个不同的剧本，第一个剧本是限流降级的演练来模拟，当你的用户请求大于你的限流阈值，或者触发你的降级条件，或者是我们在人工降级的时候去来观察你的系统的表现。OK，那这是一种降级演练的方式。第二种降级演练是演练失效转移。什么意思？比如说我模拟当前的缓存突然变得不可用，那你的应用方如何来做一个失效转译转到缓存的备库当中？再比如咱的数据库主库机房突然连接中断，那你这个数据库层面的失效转译如何来做？这是我们降级演练当中的主要的两个剧本。


okay，那除了降级演练，咱这里还有一个拐点，压测。看到压测两个字，咱就不由得这菊花一紧。压测要压出你的性能拐点，就代表着你要把一部分线上业务要把它压到自己性能的极限，那么这个步骤当中我们需要适当的放宽你的限流和降级的阈值，这样的话才能把你的请求打高，不然的话你的测试流量都会被拦在网关层之外，被限流掉，那是无法来打出你的性能拐点的。
