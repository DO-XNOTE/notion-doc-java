一个商品它有一个主商品的ID，比如这里是10001，那那么这里既然提到了主商品，难道还有子商品吗？我们往下来看，那除了商品以外，咱的电商业务还有一个概念是门店，比如这里我的新零售业务开展了三个线下门店，分别是门店123，那么这三个门店都有自己对应的 shop ID，也就是门店ID，我们把它也叫做 123 okay。


那么咱的这个商品发布在新零售业务当中比较特殊，它每次发布之后，主商品会连同着发布到不同的门店，那每一个门店都可以对这个商品设置特殊的定价策略以及优惠营销策略。那么从这个角度来看，是不是说咱的主商品在每个门店都相当于是一个独立的商品？没错，你的主商品发布到哪个门店，它就会自动的生成一个子商品的ID，那么这个子商品 ID 才是正儿八经的淘系商品ID，可以在淘系的整个上下游链路当中的各个业务当中使用，OK，那么现在问题来了，我们的APP，手机 APP 这里我们实现了一个商品查询的功能。


那么这个商品查询实际上它查询到的是主商品的ID，那查询到商品之后，我是不是要进入到商品详情页，在详情页里有营销优惠、库存、价格等等，那么这里的商品同学们看是子商品，okay，那么我怎么从商品查询的主商品链接到子商品？这里就牵扯到一个手机 APP 上的定位功能，它会会根据你当前所处的位置定位到最近的一个门店，那么通过你的主商品 ID 还有门店的ID，它就能找到你对应的子商品 ID 是哪个。


OK，那么业务是这样玩儿的，但是后台实现我们是怎么实现的？通过缓存来保持主商品和子商品之间的链接，那么我们看咱在阿里系里面用的都叫 tear 缓存，那这个缓存当中的 key 就是主商品的ID，加上门店的 ID okay，那么它的 value 就是子商品的ID。


所以我们在商品查询出的结果，每次用户点击查询结果当中的一个商品图片的时候，后台业务会根据当前商品的主 ID 和当前离你最近的门店的那个 ID 来找到对应的子商品ID，这样的话我们就去调用淘系商品的详情页接口去渲染整个页面。OK，那这套业务看起来没有什么问题，技术方案也非常的合理。那接下来的问题，这个故障出现在什么情况下？

同学们在自己的项目当中应该经常去写集成测试脚本，每上线一个大的功能改动都要去做回归测试，那么它其实目的就是一个功能性的测试，确保你的新功能没有出现任何的异常故障等等。


那么压测阶段同学们都知道它是为了去压出你的性能拐点，那么在压你这个拐点的过程当中，这两个拐点，那它的异常经常容易被忽略，因为系统在压力比较大的时候，自然而然的就会发生一些响应超时等等异常，因此压测阶段并不适合去来发现功能性的一些bug。那又到了最后的稳定性测试阶段，咱前面一节刚刚提到过，它是让你的系统保持一个稍微比较恒定的压力，在一段较长时间里持续的跑这个结果，看这个长时间里面你的性能是否有波动，是否会触发一些系统的资源瓶颈、性能瓶颈等等。然后根据方差、标准差的公式找到它的指标偏差。
