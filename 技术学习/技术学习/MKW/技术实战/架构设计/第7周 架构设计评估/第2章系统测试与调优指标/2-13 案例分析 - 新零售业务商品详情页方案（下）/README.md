商品详情页作为主链路当中非常重要的一环，我们自然是采用了弹性化的部署的架构，那么弹性化这里有两个东西必不可少， Docker 和Kubernetes。对于 Docker 容器来说，我们这里的所有应用都是做成 Docker 来进行部署的。


那么在机房的搭建里，我们用到了前面章节当中跟同学们提到了阿里系的一个标准，两地三机房。还记得这个两地之间要间隔多少吗？阿里系的一个标准是说要间隔 1000 千米，同时我们这里还允许了一种手工弹性，当我需要为自己的机房申请新的机器的时候，带有所在部门的分组的PE，只要审核过后可以达到一个分钟级上限，那么这个快速上线就得益于咱的 Docker 容器的快速的可复制化的部署。


那与此同时，为了实现自动化的弹性，我们这里还在关键的主链路上面申请了一个弹性机房，那弹性机房就会用到前面小节当中提到的弹性计算，会根据当前主链路当中的应用水位来自动的做弹性扩缩容。那么这里有一个小的知识点，对于主链路上面的环节，一旦有一个环节上了弹性，那么最好整个主链路都要把它做成弹性，为什么呀？因为其中的一个环节它可以根据当前流量做弹性的扩容，但是其他的环节如果没有这种弹新能力的话，恐怕要被有弹新能力的主链路给打崩。所以我们这里要求主链路上的关键节点要谈，大家就一起来谈。


那我们的这个机房设置也是采用这种两地三备份的失效转移方案，保证在其中一个机房发生了最坏的情况下，我可以快速的做失效转译，把流量切换到其他机房。OK，那么对于数据库来说，我们也是采用了类似的方案，当主库发生异常的时候，我们的失效转译方案可以切成从库小三转正，从库转主库，也可以在我当前的数据机房 down 掉的情况下，迅速的切换到备用的机房当中，那这两点是非常概括性的架构模式。那么在后面的专门的容器化章节里，会有张老师跟同学们详细的介绍容器化的落地方案，同学们可以期待一下。

这个缓存它前置到有多靠前，我们接下来看当你的用户流量抵达我们的网关层的时候，当然这中间其实还有一个 CDN 层的缓存，那我这里就没有把它画上去。咱假设这里是网关层，那这个网关层，那它首先会去获取什么呀？对于一些读服务，我们要获取一个网关的本地缓存，不同的公司这里采用的黑科技不一样，但是大部分公司网关层是使用 Nginx 作为第一层的 Web 服务器的网关，那么大家都知道 Nginx 其实并不执行什么业务逻辑，那它只是单纯的把你的用户请求转移到后台的服务器，不过在这个过程里 Nginx 可以做一些小手脚，它可以把这些数据在本地的缓存起来。


那么在以后，如果咱们的用户访问了相同请求的数据，那么就可以在网关层直接返回，那么就不用去请求后面的服务了。这样呢，有两个好处，同学们能不能想到是哪两个好处？第一个应该非常容易想，就是减轻后台服务器的压力。那么第二个同学们还能想到有其他好处吗？第二个不太容易想到，假设当后台的服务器宕机的时候，诶，同学们想后台服务器宕机了，但是我网关层如果还是运行状态，那我本地的这个缓存是不是还可以去撑一会儿？那这个网关层就相当于一个缓冲地带，它不会使你的服务完全的 down 掉，可以提高服务的可用性。


OK，那关于 Nginx 的部分我这里就不展开来讲了。 Nginx 其实也有一种缓存的持久化的能力，缓存的数据持久化到本地磁盘上之后，那你服务器重启之后依然可以去构建缓存。那么当本地的这个网关缓存不存在之后，我们这里会去导向。首先把请求导向 tear 或者 Redis 缓存当中，那么对于阿里系的应用一般都是使用的tear，那对于大部分的公司应该是使用的 Redis 作为这一层缓存的中间层，那么这里还可以采用一致性哈希的方式。那一致性哈希就非常像我们数据库的分库分表，通过对你的请求做哈希运算，把它特定的路由到一些特定指定的机器上，这样的话可以在一定程度上提高缓存的命中率，节省缓存的存储空间。


为什么一致性哈希还可以节省存储空间，还买一送一这样的好处吗？因为一致性哈希它只在我哈希值指定的那几台机器上面构建缓存，不用在整个集群当中构建。就是这样一个很简单的原因。好，如果咱的中间的这一层缓存层没有获取到数据，那么我不得已就要把你的用户请求正儿八经的导向咱的这个业务服务当中。那在我业务服务里，同学们想是要直接访问数据库吗？那在咱的大型应用当中，可不能直接去在这一层直接访问DB，那这是我们写毕业射击的玩法，访问 DB 之后，如果存在构建到缓存当中，那在真正的商业级应用，往往这里还要再加一层，我需要从本地缓存里尝试读取这个数据，然后再返回给前端。


那么这里为什么我要加这么一层？同学们能想到它的用意吗？其实这里不光本地缓存，我甚至还可以再加一层 Redis 做兜底。为什么呀？因为是为了防止某些热点数据的击穿，在这一层当中我可以把这个本地缓存的失效时间设置的非常非常短，比如我 Redis 里面设置 2 分钟过期，那我本地缓存可以只设置一个 30 秒，它就是做了这么一层兜底，防止万一有一些热点数据没有来得及导向热点库的时候，直接打造了后台流量。


那么我这里可以保证在每台机器上面可以让这个热点数据构建起本地的缓存，加上这中间一层其实会稍微拖慢那么一点点的响应时间，但是一定程度上加了一个兜底方案，可以大保障你后台服务的生存性。OK，那么如果这面几层全部都没有读到怎么办？丑媳妇儿早晚要见公婆。你怎么说？这里总该去访问物理数据库了，对吧？那么在这里访问物理数据库，我们记住要通过某种分布式锁的机制，使同一时间只有一个或少量的几个数据可以去真正的访问到后台的数据库，去查询同一个ID，那这是保护后台数据库的一个必须的步骤。okay，那么读到库之后，我们这里要做一个反向的构建，从网关缓存这里回溯到咱的业务服务的这条路径上面，我们把这几个不存在的缓存都要把它构建出来，那么通过这样一个架构方案，我们就实现了一个缓存前置。同学们可以看到在你的流量真实打到后台业务服务之前，我们已经有几层缓存在这里做一个兜底，那这就是做大型应用设计架构的一个考量的点，就是说我们没有必要把所有流量都承接到业务服务这一层，业务服务不应该是第一层承接流量的组件，它应该是处于一个相对靠后那么一点的位置。
