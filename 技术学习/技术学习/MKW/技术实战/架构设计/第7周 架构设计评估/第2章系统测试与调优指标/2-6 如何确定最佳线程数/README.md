第一，当我增加线程数的时候，增加你线程池当中线程个数的时候，如果你的业务的表现是这样的，接口的 QPS 不变，也就是说你接口可以同时在每秒当中处理的请求数量并不会发生变化。已经到头了，不会再增加了，那甚至你每个请求它的 RT 处理时间会变长，那么这个情况下就是一个信号发出了。也就是说，你增加线程数实际上并没有对你业务的吞吐量带来什么变化，甚至反而会降低你业务的响应时间。


OK，那么在这个时候你就要注意，你当前测试用的线程数可能已经超过了最佳线程数。okay，那么这是一个条件。第二个条件，当我去减少线程数的时候，那么如果我的表现形式是这样的，接口的 QPS 下降，也就是说你减少线程，那我整个集群服务的每秒钟可以处理的请求的数量，也就是我的服务的吞吐量下降。那么这就是一个另一个信号，告诉大家你的集群性能还有提升的空间，也就是说你的线程数还离最佳线程数还有一些位置。OK，那么就是通过这两个条件，我们**可以借助于性能测试去压出你真正的最佳线程数，那这就是我们在阿里系的实践当中得出的一个架构指导，如何从架构层面去确定你当前业务的最佳线程数，就是用这两个维度来做衡量**。OK，同学们，那这一节里针对最佳线程数的理论和实践相关的分享就到这里了，我们在后面的小节里继续对接口性能调优的话题做探讨。好，同学们，我们下一小节再见。
