# 1-20 响应式架构本质：定界一致性

想要是价格本质的最后一个特点，这个特点很多地方我都没见到有人讲过，但是我觉得这一点是我自己的一些心得体会，也希望通过这个课程跟大家分享，一起来探讨一下。这个东西叫定界一致性，这个话听上去比较让人疑惑，但是我相信你听完我这个讲解，你就会明白这句话的意思。首先大家都知道响应式架构天然是分布式的，需要建立一种层次结构来处理这种失败，由于父节点来处理子节点的失败，同时可以向上归因。这句话是什么意思呢？

就是说因为响应式架构天然建议也是把这个人分布式的，这样子可以达到多种多样好处前面我已经再想讲过它的好处在哪里，我就不再一一的赘述，同时大家也知道你做成分布式，那就有存在一个问题，因为彼此之间它不是一种强烈的像内存之间这种异代关系，所以说当它发生错误的时候，你很难去捕捉和处理。但对待错误的处理方式，在响应时间当中自然就会有一些不一样。


相对在内存里面的这种异常处理，或者 l 的这种处理，那肯定是不同的，那这种分布式并且是响应式架构的情况之下，如何来做这种错误的处理呢？这里有一种概念叫做层次结构，这个层次结构它是一种父子关系一样的，相当于说你在整个架构当中把不同的处理断分成不同的片，一样的，你把它归上片以后，你定义一个父子关系，类似于由 a 派生出了 b 这样子，你把 b 当做 a 的子，如果说子在这一段出现问题的时候，你要把这个错误向上传递告诉他，至少让他能够知道哪个节点上发生了错误。


以此对整个架构，不管是从监控的角度，还是从错误的链条的环节，一些技术手段来讲都是一个必要条件。所以说在这种方式下，就相当于说子节点发生错误的时候，不仅你要在自己内部形成因素的机制，同时还得通知自己的复节点，这样让整个系统处于一个良好的消息传递机制管控范围之内，然后把这种分布式系统当的不特指响应式这个地方其实都是相对说比较泛的一种说法，不过这响应是在上面体现的比较明显而已。


将这种系统拆解为更小的模块，在小模块中实现强硬致性而最终达成最终一致性。前面讲过的是错误处理，那在错误处理的时候我也讲过，在那一步你最好是能够实现这些错误处理，同时你也要把这个错误告诉你的父节点，那在这种拆解为小模块之中，小模块里面，但肯定要达成一种强一致性。举一个简单例子，好的，比如说你是在一个电商系统里面，你付钱了，下单了，希望看到各种状态变化，扣钱、走账这种东西都实现，对吧？但是你只要是分布式系统，不管是否是响应式的，都采访这个特点。 at 分布系统做的强力执行是非常困难的，如果做不到，我们把它拆解为不同的模块，比如说是支付模块，订单模块，这两个模块它各自里面都是有自己的独立的数据库，或者是其他的存储状态的，这样一些机制也不用纠结于是否真的是哪种数据库，我们就认为它是有一种可能存储状态的占用机制，对不对？那你不能说一定保证支付完成之后订单的状态就变成一个你特定的状态，对吧？但是我可以要求说至少在支付的这个模块里面，然后这个系统里面它的状态是必须得一致的，不能说是我扣了钱，这个状态它却没有变。当然这是指支付的状态，钱已经扣掉了 500 块。


支付的状态表示支付已经完成，这个账它得记下来，它支付系统内部一定是强一执行的，但是你说支付完毕以后，订单的状态一定会变成一个你想的状态，比如说是下单成功这样一个状态，它是不是一定会变成一个下单成功状态？当然最理想的状况肯定是这样子，如果是在分布式系统当中，它稍微有所延迟，这也不稀奇，甚至最终他没有达到一个你想的状态，这都不奇怪，因为消息传递可能会丢失，有可能在另外一些你的数据中出错。


所以说支付成功未必代表着订单一定会成功，但是它最终的一致性是体现在哪里？比如说订单最终处理成功了，那一切都好，订单的状态、支付的都是一致的。这是一种最终一致性，所谓的最终一致性，我记得在之前的一些课程我有提到过最终一致性强调的是什么？最终这二字最终是一致，只不过中间花多长时间也不确定而已。然后再回到前面这个话题，你如果说订单系统出了问题，支付失败，但是最初支付成功，怎么样？支付失败，那就是通过前面这种说法，第一你把错误传递出去，让上游戏讨他知道这件事情出了问题，要把他状态也支持失败，把钱再退回，客户也达到了这种，因为都失败就是这样子。


所以说这里两个重点是在于小模块，它是强，一直在整体系统上，它是最终与执行。这一点对你设计分布系统，不管是否是响应式架构系统，它都是有很大的指导意义。你对自己的 SOA 或者 Qos 这种质量要求的话，在这个前提下保证一下就可以了，不能说在分布系统下面我一定要保证一个强制性，我不能说你绝对做不到，但是你也得考虑一下这件事情是否可行性有那么高的，特别是在有限的投资情况之下，在真实的环境系统你做到这个有没有那个必要？这一点我希望大家制作我系统设计就是时刻提醒自己单模块能强一致，在整体系统上做到最终一致性即可。


同时我们也比较推崇使用这种 DDD 云驱动设计的方式，将行为和相应的数据进行切片，把每一片晒进自己的事物边界，也就是前面说的这种小模块里面，我们要对这些东西分片，你就要对它进行切分，你怎么签？这个就是很科学的一件事情，所以说一般情况下，个人的经验或者我学习过的人，我推荐是使用这种临生的方式，你给他做一下事物的边界。


所谓的事物的边界有一种说法叫帮助的context，就是说你讲这一种类型归在一个边界，在另外一种类型归在另外一个边。前面我举的这个订单支付，这就是一个典型的按照领域驱动设计积分者的两个边界，其实在一个边界之内它是有自己掌控的，超出这个边界的事情就不归它进行掌控。但是在这个边界内它怎么玩都是它领域自制的，你无权去干涉他。当然你说遵循一些大前提什么，但那是其中之一，这就不需要再进一步阐述这个问题。


对于事物这种把行为和数据进行切片，就是我们说的领域包装完了，在不同的两个短慢，也就是两个领域之间进行通信的时候，就发现一些交互它跨过来的边界，只要跨了边界，很多事情就不受你控制，就跟前面我们讲的这种聚众一致性和在单个边界内它是强一致性，价值目的和设计需求都是一模一样的。


所以说在这个地方我最后强调一遍所谓的定检一致性，最重要的是强调是利用一些领域驱动设计的方法，这是基于我个人的一个推荐，你如果能找到一些这样的方式，我也不能说它就不行，但是在业界主流认可的范围内，可能领域驱动设计是一个相对比较推崇的方式，你使用领域设计的方式对你整体系统不管是否是使用小程式架构，这都是很有用的。把它进行一个定界，就是给每一个模块划分它的职责归好，它的数据在哪里，在不同的领域之间要求最终一致性，在同一领域要求强一致性。
