# 1-5 响应式编程的主要模式

前面我们有谈到响应式编辑只是一种相对抽象的概念，其实它有很多种实现手段，本身它只是一种架构设计的原则。当然任何概念或者理论要落实在这个实践上，其实都是需要相应的实现手段，所以一般来说在实践中强势架构都会有一些继承的模式，相当比较通用的一些模式来实现，那我们简单来看一下，依然就是这个观察者模式，

这个是经典模式之一，简单的说在这个详细编程当中，我们就是来观察这个流针对它的变化就是我们的响应，以此形成一个响应式系统。


前面也讲过这流的变化，什么叫流的变化？就是你可以把这个流当成一种事件流或者消息流或者数据流一个统称，对吧？那只要这个流当中我们还有不断的新的东西进来，那我就可以看到这个前后的变化，观察这个流，监听这个流，我就可以做出一个响应。只要是礼物上有东西，我马上做出响应。这是一种观察者模式，就观察这个源头，然后根据这个源头发出的事件信息来做出相应的小悟。


第二一种类型就是事件驱动模式。什么意思呢？就是通过流中的事件来驱动整个系统的运作，所以这个流由序列的这个事件而组成的，就是一个时间序列事件，流就随着时间的流逝，这个不断有事件被广播出来。那你这个事件成为整个系统的一个运转的推手，让它来驱动整个系统运转。就是说当系统当中有任何事情发生的时候，这个事件都会放到一个流里面去，然后我们这个行事系统就去监听流里面这些东西，就是所谓的事件在这个地方，那这个事件从源头上出来的时候，它会被绑定到一个要做的动作上面，就事件它跟它想要做的动作是绑定的。因此当一个事件被抛出来的时候，他要做的这个动作又要做的响应，或者说是就会自动被触发，就会被执行。


其实这个前面在讲我们这个所谓的事件或者消息，在迭代的课程当中其实也有提到过，这只是学而的差异的。后面我们有地方会专门的讲解一下这事件和消息之间这个差异在哪里，请同学们留意。最后一种，也是相对说比较常见的一种模式，这里大家可以这么讲，我们这里讲的这种模式都是相对说比较模式中的模式。什么意思？就是说其实后面在课程中有提到很多更细节的，就是更具备执行力的模式，这里讲到这几种模式就相当于是比较抽象的，更高层次的这一种抽象的一些模式。就是后面提到那些小细节的，更偏向于执行层面的一些模式，是指这些模式的子模式。什么意思？就是说这些模式是最高层次的抽象，下面各种更细节的模式只是它这个上面的一种更具体的一种变种而已。


最后谈到这种发布定义的模式，这个就是虽然常见的一种 design pattern，对吧？什么？它发布的定义者什么意思？就是说我知道你这个账会发生一些事情，那我就订阅你，当你有任何东西出现的时候，你把它告诉我，这样子我就会做出我对应的反应，通过这种 power SUB 的方式来实现了一种响应式编程，这也是比较常见的一种模式，对吧？我再次强调一下，这里讲的这三种模式是最高层次的这种抽象，后面我们会讲到一些更细节的一些模式，它其实都可以被归类到这三者当中的某一种。这里只是给大家提供一种思路，或者一种高层次抽象的概念，不要和后面这种更细节的这些设计模式，就是在这个响应式架构当中它的模式搞混。其实它们虽然都是同源的，但是抽象层次是有所不同的。
