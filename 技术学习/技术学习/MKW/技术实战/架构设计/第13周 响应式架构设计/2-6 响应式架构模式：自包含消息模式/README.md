# 2-6 响应式架构模式：自包含消息模式

自包含消息模式，这种模式要解决的问题其实非常简单，大家一看就一目了然，只是平时可能在做设计的时候思考的深度没有到这个地步，但是我觉得这种模式都是一点就通的。让我们来看一下消息传播的上下文在哪里保存最合适？大家有没有想过为什么需要这个模式？所要解决的问题主要有几点。第一点，比如说现在我们在一个系统当中发出了一条消息，然后系统出某些问题，我们要重新再尝试发送一次。那这个时候你想在消息的传播上下文，就是相关的一些信息它应该放在哪里？这样付出到代价最小，类似于这种问题就是通过自包含这种模式来解决。

我们来看一下问题的表述，在自包含消息模式里面，每个消息将包含处理请求以及了解其响应所需的所有信息，也就是说一个消息它不仅包含了自己请求本身的一些信息，也包含了这一次请求响应所需要的相关上下文信息，都把它包含在这个消息体里面。请求包含处理它所需要的所有内容，因此在发生故障之后，它完全可以重新处理，它无需系统从头恢复，这点就是最重要的一个点。如果说相对发送的信息比较少，也许在大多数情况下它都是可以运作，这几乎没有什么好疑问，但是一旦系统出现问题，你要进行二次处理或者重传，有些信息可能随着时间的丢失，它已经不见了，这个时候你要从系统上再恢复它就比较困难。但如果说你的消息本身它就包含了这种信息在里面，但你直接重新发送消息即可，这样子不仅节省了时间成本，也节省了资源成本。因为你要把这些信息从系统中恢复出来的话，你是要花费计算资源的，特别是一些相对较困难或者运算量比较大的一些，这样子全部都节省下来了。


所有什么呢次包含消息，对于你在出现故障的时候重新处理，这个特别有意义。大家都知道 http 请求， http 本身就是一个无状态的请求，就是你所有的请求信息，那响应信息留得类似于词包含一样所有请求的信息里面有它所需要的一切信息，用这样子的类比的方式，不知道大家有没有更好的理解这个自包含的意思？它的适用范围相对还是比较广，希望会话协议尽可能保持简单，并且不维持上下文的状态。


如果说你希望是得到一个无状态的服务，那么你尽可能让你传递的消息都是自包含消息，这一点大家可以这么来理解。如果你的服务要保持一个状态，那么你要付出的额外代价非常非常多，因为这个维护状态本身就是很大的一个开销。你想这么多请求过来，如果说你每一次请求的上下文之间有第一次、第二次之间互有联系，你要把这种信息保存起来。特别是如果说我们面对的用户量到了一个非常夸张的地步，那你这种信息全部要自己维护状态的话，你的整个成本，对，资源成本、维护成本，包括你代码的成本都是非常高昂的。


所以说更多的时候，现在不管你未来做负载均衡，还是降低服务器开销，还是说其他的一些原因，你尽可能推崇的是这种无上下文状态的服务，推崇的是这种，而你要支持这种不维护畅想状态的这种符，那么自爆狠消息是你很好的一个帮手。那有什么情况下不能使用这种方式呢？我总结一下比较简单的一点，就是可能这个信息量过大，不适合传输，如果实在太大，这个确实是可能不太适合你使用这种模式。其他的你都酌情考虑。我个人认为现阶段对这种无状态服务，也就是说不需要维护上下文的服务需求还是挺多的。那在这种情况下，私包含消息跟它几乎是配对出现的。
