那肯定你的发送方和结束方至少是要隔离起来的。你说发射方和接收方放在一起的这种，除了在链接端，那在正式的设计当中或者实现当中这么做的可能性还是非常低的吧。那你说我们一定要这么做，当然不是不能，但是这种思想或者说架构模式下面基本上是不可能推荐你这么去完成的，所以说你只要使用消息驱动的方式在 100% 的首先就在主键之间建立了边界，不存在说我们彼此融合在一起，这就为我们后面的第一实现松散和大小的基础，你只要没有那么紧密的结合，那自然就是松散的，对吧？这个耦合之间关系它自然就变了，因为一个是发送方，一个是接收方，就是最起码要具备的两个组件，所以它们之间就不可能是因像方法级别或者内存上的这样一个强合一代关系。


那隔离呢？就跟前面这个来建立边界是一个意思，你有了这个边界，那这样大家互相隔离开了。比如说你的这个消息的发送方出了问题，然后会导致这个接受方也会crush，这个可能性非常非常低，对吧？所以说隔离性本来是这样的，那位置透明性可能大家相对来说会有一点点疑惑，这个是什么意思？其实位置臭明星后面我们会有一些详细的讲解是什么意思？但其实提前说一说位置臭名就是这么讲，你作为一个发送方，你只管把这个消息发出去，至于说谁在哪里诶？消费这个消息或者来被你驱动，其实你必不需要太关心，首先你们的关系是从松耦合的，你作为发送方进到你的这个单一直就是发送消息就够了。


所以说这对于发送方来讲，谁来消费？它在哪里消费？它什么时候消费的？它是无所谓的，这就是一种透明性。那反过来看，消费的就是接受方这一段也是一样的，它根本就不关心你是哪里来的，他只是关心说我是不是收到了一个消息，我接下来应该发起什么样的一个任务，做什么样的事情，对不对？也就是彼此都是痛，而且隔离的，这是一使用这种消息驱动的方式去达到一个很明显的好处。


前面已经讲过了，就是通过这种发送方、接受方按下去模式，实现了一个很好的隔离和自我的这样一个封闭性。这个时候的封闭其实你如果用这种类或者是面向对象的方法来理解，可能就是说 good sponsored 是都一样的，对吧？所以说他们彼此之间的这种隔离性、依赖性就发生了改变，而且是一个良性的好处的改变，而不是说朝着向往的方向，这一点不言而喻。

第一，有没有可能有一个监控，可能要监控这个消息，保证去把 b 给重启起来，对吧？这是一种可能性。另外一种，也就是 b 他有没有可能提供另外一种方式，直接通过这种消费消息的方式，而不是直接调用的方式来完成这种？反正这都是觉得一个例子，怎么能理解作故障违法，就是通过将故障转移或者风险转变到别的地方，或者以另外一种方式来化解了这种故障，这就是这种现在驱动也可以提供的一种功能。


此外通过对系统中的这个消息队列进行整形监控，必要是施加反压力，实现负载管理更大的弹性和负载控制。这个事情就是说只要你使用了这个相遇驱动这种模式来设计你的系统，你就有可能会出现速度的这个不均匀性。我们在后续的讲解里面会详细的谈一下这件事情，如果说消费者太慢就会形成一个不一样 bug pressure 这个在工位上有很多讲解，我们后续的章节里面有一个专门重点就是讲这件事情。因为在这种消息驱动的系统里面，特别是比如说以及我们这个强势编程里面，这种异步消息特别多的情况下，如何处理这个被压其实是非常重要的一件事情。这地方我先不详讲，这里提一下意思。

什么意思呢？如果在系统中出现了这种生产者、消费者的速度的不匹配，那我们必须要通过一定的手段来进行控制。这种控制可以按照系统设计者最初的愿望来进行一个更平稳的方式，有点像流量转型那种感觉，就是你可以通那么快的控制住消息的消费的速度，而不是被动地去接受这种一个现状，就是说你用了更多的控制手段，让这个系统运转的速度变得更快一点，还是变得更慢一点，所以这是一个非常重要的手段。


对于架构式来说，在你的工具当中，这种类似于像流量整形的工具，但是越多越好，这样你可以实现消风填谷嘛，对吧？先把消息再澄清一下，那我愿意什么时候让你消费就上去消费，让你快点快点，让你慢一点就慢一点。所以说这种东西你系统的整个前面讲的错，不管是弹性还是这种速度的变化，你都是比较更受控制一点。所以说对架构师来讲，你掌控性的能力也就更有把握一些。这种根据我们开篇在讲宣言的时候讲过，就是由于这个摩尔定律的，这个不说失效，就是说摩尔定律它不再主导近些年这个硬件的变化，但不是不主导，只是说他可能就不再按照沃尔丁流量去发展，对吧？重乌阻塞通信确保了消息的接受者在活动时再消耗资源，降低了系统的开销，并且更加高效地利用了这个资源。


前面我们讲开篇讲这个，就是小程序宣言的这个来龙去脉和历史的时候就讲过这个问题。最初我们整个软件都是被硬件强力支配，软件更像是跟硬件配套的这么一个东西，到后面独立发展出只有一个完结学科也罢，还是一个工业级的东西也罢，对吧？它其实跟硬件是强相关的，它的后面当我 2.0 是想以后我们的硬件这个发展，它进入了一个瓶颈期间的东西，那这个时候就出现了多核这种东西，这种价格，那我们要是利用好这个多核，那势必就不能使用传统那种那个 request youre responsible for 等待的模式，因为这个是多核，它其实闲着的。因为很多在我们的这个应用系统里面， Crystal persuade 一个 THREAD 对应了一个请求，但如果这个时候你停着，你有几个，你就可以用几个，那等着的话就太浪费了。所以说后面慢慢就起这样说，无阻塞的这种方式，我发给你一个请求，然后我不会再等待你，等你这边有了结果以后告诉我，我来消费这个 response 就好了。


哎，这种可以更好地提高整性的throughput，也可以把整体的性能往上升，因为资源它本来就被尽善尽美地死掉了，不存在说浪费。原来就是不管你是多少个和多少个CPU，那你可能都会在这里等待。而现在是处理完一个请求，马上就要可以处理下一个请求，而不存在做中间的等待时间，做 CPU 或者硬件上就不存在一个浪费的问题，最大的程度减少系统资源共享的堵塞。这就是跟前面讲到这种伸缩性、弹性、低延时、高吞度量，或者是它的一个有力的支撑。因为你在这个如果有真用的时候，前面我们在讲这个产品都有讲过，如果有中央瓶颈，就是这个征用资源的时候，你要做的这个弹性是相当困难的。对，但是如果说我们用这个资源，只是我现在发一个消息就好，至于测试后续怎么处理，我不需要阻塞。


这样子相对说真用的这个现象虽然还是有，就有可能他不能同时向他发一个消息，但是你发消息不用等待响应，已经没有独占这个资源，那别人就可以与你共享这个资源，所以说它这个征用性就下降了，整体系统增速性、延时存储量都得到一个提升。这种设计手段就是消息这种设计手段大大的提高了以上这三个系统的信任指标，所以这个采用消息机制，在这种新型的应用当中是非常有意义的一件事情。如果整体都转向这种系统或者这种架构方式，那是不是对我们整个设计思想和架构风格的一种转变呢？所以说这个地方我们又回答最初的说法就是响应式宣言最重要的是指导我们发生一种架构风格的变化，就属于传统的这种请求等待，但是我们机器越来越强，让我们观念变化为我尽可能地利用好硬件条件，不让它做过多的等待，提高资源的利用率，但是它的做法跟传统可能就不太一样。
