# 1-14 响应式架构本质：消息与事件

响应式宣言之后，我们来看看基于响应式宣言构建的系统，它都会有哪些特点。这个系统的特征有很多种，但是我们加入几个最根本的与其他系统的一个区别来研究一下，有助于我们了解这种强势架构的设计的特点，以及以后你要使用一种强势架构在你的系统当中，那你应该注意些什么？跟我们想想架构的本质的目的就是这，所以说第一点我们来看一下，我们要严格的区分事件与消息这个东西吗？为什么这么说呢？虽然说这个事件与消息在大多数的情况下是难以区分的，但是其实在响应式渲染当中对它们是进行了严格区分的。

首先来说响应式编程，它专注于短时间内数据内容链条上的计算，因此倾向于是这个事件驱动，对吧？但是想的是系统，它是关注于通过分布式系统的通信与协作所得到这个弹性和可靠性。这就是用消息来驱动的第一点，像咱们对于事件来讲的话，之前我有分享过世界它跟回调这件事它都是绑定的，就回调这种方式，不应该说这种事，就事件它一般与回调都是把柄的。所以一般来讲你可以说事件就是与回调这种机制它是捆定在一起的，回调就是用于响应某个事件的过程，回调与事件生产者是绑定的，这种绑定机制当然在大多数情况下都是工作的，但是也产生了自己的局限性，也就是说你在这有一个绑定的过程，我们叫做强耦合。


因为你把这个事件的响应方式和事件本身在这种情况下进行一个绑定，这就可能会带来一些局限性，但是消息它完全不是这样子的，消息系统提供了一种方式能够让你发送消息给某个特定的接收者，然后由接收者做出一些响应，就对于这种事情它的生活活动，那相对来说跟回调与事件中绑定的做法来讲不可同日而语的消息系统就提供了一种更加灵活的解耦的方式，对不对？前面我们也分享过，而且你要去消费这种消息有很多种方式，但是如果你是用了时间，一定要这种方式的，相对处理的手段也比较单一，消息天然就是可以替代这个事件的，事件在系统中的传播，你可以就认为是消息它沿着这个处理单元链的一个转发，这件事情怎么理解？也就是说事件可以认为是消息的一种，自己消息它是可以有一个更广阔的一个包容性的，你要在达到事件的传播效果或者回调这种方式的话，其实你是更为消息的，对，你包装有消息的方式来替代事件，但是反过来有事件来替代消息，这就是做不到的。所以说在这个响应式的系统里面，我们就倾向于 show 消息，而不是使用事件。


这个原因的区别就在于此，一个消息就是 ETL 被送往一个明确目的地的数据，一个事件就是达到某个给定状态的组件的发展与信号。就这么区分，这个消息驱动的系统中可寻址到的接收者，等待消息的到来，然后响应它，否则它就一直保持休眠状态。在一个事件驱动的系统中通知的这个监听者，他是被绑定在消息演示上的，这样当消息被发出的人他就会被调用，这就意味着一个事件驱动的时候专注于可寻址的事件源，而消息驱动系统专注于可寻址的接收者，这两者就有挺大的区别的。而且基于这种架构模式，消息驱动的系统它相对来说是更灵活，耦合度更低。
