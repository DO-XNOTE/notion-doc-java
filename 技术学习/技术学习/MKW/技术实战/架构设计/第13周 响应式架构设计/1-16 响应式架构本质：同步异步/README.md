# 1-16 响应式架构本质：同步异步

我们再来看响应声架构的另一个本质问题，就是同步于异步，这个毫无疑问大家都已经知道，我们想要下个讲究的肯定是异步的单一，那为什么呢？首先同步的话需要双方都同时做进行通信的这个准备，这一点其实你只要是对 HCP 这种消息协议比较熟悉的话，你很容易去理解这件事情，对吧？如果让你发送一个消息，那你必须要 server 的是 already 的，然后你才能给他发消息。同时你词其也要是reason，你只能处理这个响应，否则的话你这件事情它就无法进行下去。

所以说对于同步来讲，第一个问题是双方必须要ready，这个函表是另外一个什么含义呢？它其实是一个阻塞的问题，双方在发起一个请求，我要得到一个资源，或者我要得到一个确认结果。那都是我向你发起一个请求，然后我就开始等待，等待到你给出我一个结论，就得到我的响应所要的东西，然后这个事儿才算完。


就是这样一个 request response 的模型，就在这种同步通信或者同步请求的情况下，得到了一种意想不到的后果，就是必须得等待，有时候是个阻塞，对吧？因为你一直会等着这个响应回来，这个响应要是不回来，我这个事就不算完，这次请求就不算成功。这个大家应该很能理解这件事情，所以说有些时候，特别是在系统的实际的这个代码编写或者系统设计的时候，你都会设我最长的超时间是多少，过了这个我就不再等待了。
但是这种做起来就怎么说是可以实现的，但是其实在有些情况下就会给你带来很大的问题，你说这个超市时间设多长合适呢？这就是一个玄学语言，有些时候你说系统里面我们都规定不超过 100 毫秒，那你敢这么做吗？所有的都不超过 100 毫秒，稍微波动一下，超过了 100 毫秒变成 101 毫秒，这事就算 seal 了吗？这也不一定，所以说这种通不通信虽然还是目前，当这是我个人的一个观测或者的结论，还是占主流的百分之八九十的系统，还是肯这是以同步的为主的，对，不？是在现在咱们这种 HTTP 的为主导的这样一个系统层次架构里面，那肯定基本上都是用同步请求居多，所以说同步它有它的好处，就是你可以马上拿到你的响应，如果在一切顺的情况下，怕就怕在这种要等待、要出差的情况，第一浪费资源，对吧？第二你很难去判定这个事情是超时还是很仍然在计算，这件事情就比较可怕。


所以说使用同步，你得考虑你在收获和与你的付出的代价到底哪个高，包括前面我们就有讲到说这种多核系统里面尽量不要让它组成在那里，而是应该利用多核，尽可能不让他空闲。如果在阻塞的时候我们去准备这件事情，当然这是另外的话题，这个同步环境就是有他自身的问题和他的不足，虽然他有他很多的好处，对吧？它对于异步来讲的话，它可能就不是这样子的，就像前面这个例子所说的一样，发送方它可以随时发送，它想发送就发送，它完全不需要考虑接收方市场准备就绪。


坦率地说这个话虽然是这么说，或者我可以说这样一个考虑，说他是否接受准备好了没有这个都不管，当然这个相对比较理向化一点。因为在这个课程当中主要是偏向于理论，但从实践角度来讲，大家肯定还是有所差别的。因为一般比如说是一个异步系统里面，特别是这种以消息为主导的系统里面，你就会知道当你发出一个消息的时候，它是一个 attention 的这么一个概念，它到底把这个消息或者这个系统里面保留多久？这其实也是一个问题。


当然我们这个地方主要谈理论，就理论上其实你是不用太考虑多大的问题，因为你也从理论上来讲它，你单一系统它不可能宕机 24 小时，对吧？说稍微说你在不是一个特别天亮的系统里面，这运售量不是天量的，保留 24 小时的这样一个数据，我觉得也不会称其为一非常大的一个运营成本了啊。


当然这个稍微有点题外话，理论上发送方他是不用再考虑说，唉，接收方是否ready，就是说甚至有可能斜视方都没有，那我就往这一个方向发就可以了。举个例子好了，比如说 Kafka 的系统 potential consumer。但是这种消费关系这个事他可以很早就开始发，但一直累积在这个开发的 topic 上面，而且他发存起来，到时候把它读出来就可以了。另外一种就是 q 里面，你一直往里面 q 里面发，它也可以放在哪里，也可以持久化，也可以放在内存里面。


当然前面我们讲的这个到底要保留多久？这可能是一个目前不管卡不卡还是MQ，它都会有这个问题。当然这个是题外话，是更多的实现层面的问题，我们如果只从架构的角度来讲，抽象一点的话，理论上你的发送方式不用太考虑。诶，技术方是否准备好了？也就是说我提前一个小时发你一个小时以后这个消息的接收方才开始上线，才开始 consume 这个消息，那也是可以的，甚至在很多新系统上线的时候，这种做法非常常见的，因为前面发出的消息有可能是有问题的，我们要先检查一下或者测试一下这个消息出去的时候有没有问题，然后我们可以先发消息，然后用一些别的手段来检测下这个消息正不正常，后面等真正的这个 convert 上来之后再开始去消费这个事情。


这也常见，所以这就是异步给我们带来一种好处，当然这只是好处之一，更多的还是包括前面我们讲的这种发送方、接收方的这种隔离性更好，他们从空间实际上都有解耦性，这种其实对于我们整体的系统设计都有非常多意想不到的好处啊。当然是不是说异步就这么完美，那肯定不是，那异步的这个处理前这个就比较麻烦的。


第二呢？也就是跟前面讲的一样，你发出去了，那你都不知道这个消息消没消费。再有一些比较关心说它到底是否是被消费过，这个就比较麻烦点，有的系统的或者有些机制可能会给你发一个类似于 ACK 的这样一个东西告。唉，这个消费消息出来了，我已经消费掉了，有的人还有可能出现这样说，发送者跟消费者必须类学生有 transaction 的这种功能，这个东西都相对说比较复杂一点，也顺起来也比较困难一点。


所以说异步也不是说没有它的问题，同步也不是说没有它的好处，这个更多的还是看你的需求。异步就是根据像我们这类型讲，它其实时间和空间都是有一个解耦度的，比这个同步这种做法肯定是要好很多。就从解耦这个角度来讲，时间和空间上都得到解耦异步，它带来的问题当然也是有的。所以说选择是同步还是异步，那我就给大家讲一句，在非这种响应式架构里面，你是有很多选择的自由的。你就根据你说采用同步获词的好处多还是采用异步的好处多，你就可以决定。但在响应式架构里面，你采用这种同步的方式应该是非常别扭，所以说一般在响应式架构当中肯定都是采用异步的这种消息通讯云机制，所以说咱们使用这种异步的通信方式或者消息交换的方式，那好处肯定是在这种小式架构里面相对较多的，就正如咱们前面讲的这种异步调用的方式，事件驱动的这个响应式编程，它侧重于时间上的解耦，从而在技术层面提供了一种更高性的并发方式，而它的范围也限定了它在空间上实现了解耦是有一定的这个困。


消息驱动的异步性也使得这个响应式系统既能够在实践上解耦，还具备空间解耦的能力。服务间不仅可以通过这个消息队列来实现分布式协作，还可以根据负载实现单个服务的弹性伸缩，从而实现详式训练中的这个Elastic，也就弹性的这个能力。这个之前我们在讲这个详式训练中是谈到个数 instance 的，这个数随着外面这个负载能力动态的变化的适应能力怎么样，就叫弹性。那如果说我们的消息速度消费的快一点，个数就会少一点，消息消费慢一点，个数就多一点，所以说通过这种方式异步的这种消息交换方式实现了这种空间时间的解耦，同时也实现了这种弹性。所以说在响应式价格当中更推崇的是异步的这种方式。
