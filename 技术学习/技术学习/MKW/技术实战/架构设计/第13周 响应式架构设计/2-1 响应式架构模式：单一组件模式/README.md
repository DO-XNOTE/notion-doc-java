含义，组件模式相对来说是一个普适的原则，由 single responsibility 最基础的几个设计原则派生而来。我们首先来看一看什么是单一组件模式，它主要解决问题。我个人看法是比较有相对的普适性，几乎无论什么样的系统，你都会遇到这种问题，当然这里并不特指是在响应式架构当中会出现这种情况。其实任何系统都有可能会出现这种情况，这种问题可能如何来解决？它的那单一组件模式可能相对来说在很多情况下它都是可以适用的。


那我们再看一下，如果系统多个功能或它执行的功能非常复杂，需要分解为不同的组件，这个问题是什么意思呢？其实也非常简单，它想说的就是如果你要去面对一个非常复杂功能的系统，你怎么把这个复杂的系统在一个合适的力度之上分解成不同的组件？单一组件模式就是对它一个最好的解决方案，这种提前 pattern 来看一下，首先就像我已经提到过的，它是由 single responsibility principle 派生出来的。


设计的时候你为每一个组件分配了职责，但是你尽可能的让每个组件只做一件事情，而且把这件事情要做得非常好。就像设计一个类的接口一样，这个接口如果被设计出来以后，它尽量只会为一种原因去改动，这就是你那个评判标准，那对应的这个单音组件模式，其实你稍微过渡一下你就懂，它的意思是比如说这个组件里面它所操作的对象、所承受的数据、所具备的功能，都应该只围绕着一个核心的领域对象事件来进行操作，你要解决的每一个问题，或者开放的每一个接口，接收的每一个消息，发生的每一些都与此应该相关。如果在这个主线内部与此相关性较弱，那就应该把它放到另外一个组件当中去，这就是说单译时的含义。


对组件层面而言，在计算器科学领域里面有一个非常好的解决问题的思路，叫做方而治之。其实单因组件模式就是对方而置之的一种实现，它们都是较为抽象的一种说法，所以单一组件模式也是分而置之这种原则的一个实现就是把一个复杂的问题分解为相对较小的问题，这个我其实也很好理解，因为当你在不管是一笔产品经理还是和开发者领导你去谈论的时候，你得到的永远都是比较复杂的一个问题，甚至一个比较抽象的问题。因为他们可能表示的比较笼统，这个时候你要听懂他也比较困难，更比方说把它分解成一个系统，要设计成一个系统，这个时候应该怎么办呢？那你就应该把这个复杂的问题变成不同的小问题。针对每一个小问题和系统中的组件来进行映射，这样子相对来说，你就把这个复杂的问题简化成不同的简单问题，再组合在一起形成一个完备的系统。


在你实际的这个使用单一组件模式的这个项目当中，你应该怎么去操作呢？这里一个建议就是使用递归的方式不断地划分这个责任。前面我讲过单一职责，你尽可能要让你的这个组件内部的职责趋于统一。你最初的这种就像愤而至之的道理一样，刚开始的时候你划分出来的东西可能相对力度较粗，里面包含的也比较大。这个其实就像画一棵分类树一样，我个人的一个建议是在你不愿意动手做更多实际的工作前，用 mindset 这种工具。
诶，先把思路理清楚，通过不断递归的方式把每一个组件的粒度都画得相对较细，这样子有便于你理解整个系统和跟系统投射的实际项目当中去，那它到什么情况下才是到了一个合适的力度？这个其实有一个最佳实践，就是前面我讲了，你就把它想让你再定义一个接口，这个接口里面如果要进行修改，它只会因为一个原，比如说我们有一个订单的组件，在这个订单组件里面任何一个操作所用到的他如果要修改，他只能是因为订单的需求发生变化。


多少人说因为支付的条件发生了变化，所以我这个订单系统要修改，如果是这样的话，那肯定是你的系统它划分的力度、层次上有问题，这点其实是一个非常好的指导原则，就是你确定你在这一个组件里面所操作的对象，或者所建立出来的领域对象相关的，不管是事件还是功能还是数据，它都必须得是围绕着这一件事情在做设计。
