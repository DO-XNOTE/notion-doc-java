# 1-22 响应式架构优点：高效

强制价格的另外一个类就是高效，这个高效它是有很多层次的含义的。首先我们从这个程序的撰写，一般的讲师架构的或者小式编程，那你使用大部分组式、函数式编程的公式，前面我有给大家讲过，在这种和我们传统的图形机式的并列使用方式不同在于它是以结果为导，诶，它要的是结果，那它的编写方式跟咱们命令式的投屏机的传统方式有所不同，但是这种方式是有的程序，因为它是直接描述结果的，所以说它写起来比较容易，比较方便，也是某种程度的一种高效。当然这个是有前提的。如果说你没有将你的思维方式或者写程序的这个方式从传统中命令行式转变为这种函数式，或者说是以结果为导向的这种形象方式。那这个高校可能就要打一个问号，但是就从书写习惯那各个方面个人看法是函数式编程写法是并列式的，要简洁很多，所以说我认为它也是一种高效的体现。

第二就是咱们说的非主塞的异步操作道理都是非常浅显易懂的。异步操作它不用去等待结果，所以说减少了类似于像 io wait 等待 response 这种我们的异步消息传递机制。你只需要在想要执行某一个方法的时候发出一条消息就行了，而这个消息发出去之后，你居然不用等待它的结果就可以了。因此通过种种方式也提高了效率，所以也是一种高效的体现。


因为我们是一个异步，所以说你的方法执行还是变成了发送消息，而且请注意这里的发送消息是不用等待的，只要发送消息就可以了，不用等待发送消息之后对方的执行结果，这个跟方法执行就是有一个很大的区别。以上的种种除了这个函数式编程之外，其他的两者对这个 CPU 的利用能力都是一个很好的帮助。我们说过，在摩尔定律逐步开始失效的情况下， CPU 的发展率其实就变现了。多核的要求的程序或者软件在上面运行时，它的善用多核这件事情，而不是向传统的大 g 时代发。一台机器做得很强劲，现在是不管说分布式也罢，你落在单个计算上，还是希望你尽量的利用多核 CPU 的能力，这是前面讲的这种信仰，减少阻塞，提高并发性。所以说无论是响应式架构还是响应式编程的方式，这种高效也就是要追求的目标之一。
