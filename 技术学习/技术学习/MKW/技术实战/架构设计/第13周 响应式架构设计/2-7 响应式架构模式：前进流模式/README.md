# 2-7 响应式架构模式：前进流模式

下面我们要讲到的模式，前进流模式，其实就我个人而言，我甚至都认为它不算是一种博士，更像是一种思考的方式。为什么这么说？想一个问题，前进流模式它想解决的问题是什么？我们先来看一下如何避免在使用其他的内裤或者模式时引入到了不必要的访问开销。这个坏一看是不是觉得这很惊讶，为什么会有这种情况？你现在看这个问题你会惊讶，但是各位不知道在做设计的时候或者写代码的时候，有没有思考过一个问题，你所使用的内裤，你所使用的模式，你是否有十足的把握掌握它的每一个细节？你使用的某一个内裤，它有可能会去访问一些你想问不到的东西，因为你对它不够充分的了解，你想有没有这种可能性？在 Java 当中至少我就知道，比如说你使用不当引起的这种错误或者性能开销其实远远超过你以为的东西。如果是你要使劲用一件武器之前却对它不够了解，那么后果可能也不是你所能控制。

为了解决这个问题，这一个模式它不是一种模式，而是一种思考方式，或者是直到你做设计的时候，编程的时候的一种思想，减少中间环节，尽可能让信息和流直接留下目的地，在一些不必要的情况下，你就不要引入第三方内裤，而是直接使用一些代码，当然你这个代码尽可能地写得简单优雅就好，这样子它是需要中间不必要的性能开销和资源浪费，就是这么简单。


它适用的范围我觉得应该是适用于所有的场景，就像我说的，我觉得它不像是一个模式，而像是一种思考方式，使用前进流来优化整体资源消耗或者响应的等待时间，更方便的为服务之间的交互创建一些快捷通道，不要在不必要的情况下使用一些过度设计。很多人现在有一种炫技的想法，我个人的一个简单的看法是，在面试的时候，你尽可能把问题复杂化一点，有可能对你有帮助。但是在实际工程中，如果你使用过度设计的话，只会给你带来坏处，没有很多好处。一个定律叫奥卡姆踢刀原理，如非必要物真实体与这个前经理模式又异曲同工之妙。在不必要的时候，千万不要过度设计，引入新的不必要的性能开销。其实我认为它并不是一种模式，而是一种思考方式或者设计方式的指导原则。
