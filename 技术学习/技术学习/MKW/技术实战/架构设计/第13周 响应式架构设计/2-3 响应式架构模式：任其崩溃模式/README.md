# 2-3 响应式架构模式：任其崩溃模式

任其崩溃模式，大家听见这个词是不是觉得比较搞笑？这种感觉其实它的含义很简单，大家看一下就知道了。如果说在我们的程序中，有些组件有些不常见的处，一旦发生，它将会导致整个系统的工作不能继续往下进行。比如可能你认为它是一个很关键的组件，但是我们要捕捉这种非常罕见的错误也很困难，而且可能要花费极大的代价。当然说你说能从 root cause 就是根源上把这个问题给解决掉，那当然最好。但如果说在很多系统里面，其实很多问题已经隐藏得很深，或者说我们也没有必要花那么大的代价来解决问题，因为毕竟可能一年出现一次，那你说我要投五六个人去解决它，花一两个星期。

这个其实从架构的角度，或者说从系统工程的角度来讲的话，其实不是特别合理，当然说那是最好的解决方案。这个打个问号，因为这个毕竟咱们作为一个商业公司的是要考虑这个成本的问题的，一旦这种问题它应该怎么解决呢？我们的这个题目就已经表达了我们的观点。任其崩溃就是让它出问题，没关系，就让它出了问题，我们再来解决这个问题。怎么解决呢？每当检测到组件有故障时，并不去尝试修复它，这个资源和数据直接重新启动释放它所占用的资源，这就叫任其崩溃。你出了问题你就出，没有关系，我让你出了我也不担心，因为我直接让你重新启动一下，不追求一定要找到你的这个根源，也不尝试从中间状态进行恢复。


为什么这么说呢？前面我已经讲过了，有些问题可能一年出现一次，这个肯定没有必要花很大的大家去解决这个问题。如果是这个问题很频繁的发生，那对，当别人你必须要找到根源，一年出现一次，一个小时出现一次，这是完全不同的两种故障率。新的实例当中，因为它是干干净净的，所以说它不存在任何脏的数据，或者之前的故障的一些后果就是故障机，因为系统直接从正确的原始状态开始，也不存在脏的东西在里面影响它的下一步决定，只是把过去的东西都清空了，简单来说你就可以认为是一个电脑重启来一次它里边所有的一些脏数据都不存在了，有什么依赖关系呢？

首先它可能需要一些心跳的检测的功能，因为你怎么知道某一个组件或者某一个地方它出了问题，所以说你要使用这种模式必须先能够发现故障才能采取行动。


上级组件需要发送虚拟请求，可能就是一个PIN。它的目的就是检查下属是否正常工作，由于这个检查机制比较频繁，我们很多地方都把它叫心跳，这个大家应该都知道的，对吧？副节点就单副检测下级节点的这个功能，只要发现任何一个点有问题，那你就把这个点给重启起来，之后让它恢复到一个工作的正确状态。


下面还有一种叫主动故障信号组件，可以自己处理它的内部错误，并将这种错误上榜，这种相对来说是不算非常严重的故障。而这种故障是在单一组件内部可以被捕获，你把它简单的理解就是如果你一个 Java 代码的话，我们以 Java 代码为例，你能够拆开去做这个异常，并且把这个异常觉得是可以处理掉，我们常说的吞掉不用往上抛，也有可能把它抛出去通知你的上一级。


比如说我调用一个第三方服务，假如说数据库好了，我要访问一次数据库，哎，没访问成功，那这个时候我有两种做法，一种做法那我就把这个簇一直往外抛，因为我这个从数据库是一个必须的。另外一种就是数据库之外可能有一个备份，比如说是有一个缓存从缓存中读得到，那其实也是 OK 的，那这种情况下你只要把这个处向上报一下就好了。


可能抛出一个消息，发出一种通知，到复节点说这里有故障，无论从哪个方向你都是一件好的事。第一系统的监控层面或者任期崩溃模式，它都是可以通知父亲点错来重启，是不是我自身有什么问题？这样子，只要让它重启一次，可以去回到原始的干净状态，这都是可以的啊。这种任期崩溃模式其实有一种变种，这事情简单的讲一下，就是叫做定时重启，比如说这个系统，我知道它一天在几点几分会发一次故障，可能某些未知的原因我作为一个简单的处理手段，定时重启这种有的地方把它叫起搏器模式，当然这个是相对有局限性的，因为毕竟你重启这种主线，它可能都是会有对系统产生一些的影响，当你在重启的时候，这个系统可能会你有一阵不能工作，或者说它处理的有异常的。这种情报器模式并不是所有的场景都适用，但是迫不得已的情况下，你可以采取这种模式。
