总之 publish 在这个地方扮演的角度就是内容的生产者，它就是牛，因为我们操作都是string，就是牛里面要处理的东西。我记得在之前的讲解中有说过，牛本身是一种操作规范，它并不是真正包含你们的数据的，它只是提供有操作方式让你能够操作流里面的东西，并不表示流里面它自身是一个container，它并不是一个概念。


而这个我们要告诉你的是publisher，也就是发布者，它是事件流中的事件的来源，发布者它的作用就是在发布完以后，它要把这些数据流同步给注册的订阅者。也就是说 publisher 发布的这些数据，它的目标就是给它各式各样的订阅者，让订阅者能够消费这些消息或学习事件，并做出自己相应的处理，这就是整个 publisher 的规范，这些东西相对来说是抽象一些，但其实已经在往实现这边走了，因为它已经规定了 temperature 它要做什么跟我们的相对更抽象的这个设计模式和我们之前的响应式编程的一些思想，那已经是在逐步落地的，你可以认为这种规范其实就是逐步在把我们之前谈到的架构层面的东西落地。


publisher 它还有一个职责，就是要确保每个订阅的 subscriber 的方法间隔按照顺序调用。什么意思呢？就是 subscriber 上面它是有各式各样的方法的，而这些方法它其实跟 publisher 里面的各种事件是有关联的。也就是说 publisher 它有很多事件会发出不同的事件，其实对应的 subscriber 它就应该有一个相应的响应的方法，就是这么简单。而且这些方法它不是随便乱调用。比如说hungevent，就是当这个事件出现的时它应该做什么处理呢？比如它结束的时候应该做什么处理？
