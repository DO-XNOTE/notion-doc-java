# 2-4 响应式架构模式：断路器模式

下面我们来看断路器模式，断器的模式其实大家应该是比较熟悉的，如果你对 spring cloud 这套组件都非常熟悉的，你就知道中间这个断论技能是讲得非常多的， trickers 里面所提供的这种 secret break 就是做这种模式的一个实现。当然我们在这种详细架构里面就它的存在也不奇怪。所以这种模式我们相对花的时间少一点，因为我个人觉得它应该是一个在现在相对于来说应该是几乎处于一个常识的这个状态。

我们首先来简单看一下当系统中部分组件失效的时候，如何保证整体系统大部分功能不受影响。就是如果说系统有一个组件出了问题，我们第一要如何防止它将这种错误传播到更广的一个范围之内？这第一点，第二点如何保证在一个小节点或者一个小功能出问题的时候不让所谓的受牵连？这个含义是什么意思？如果说我们系统中有主流程、有子流程，当主流程出问题，那肯定这系统就没什么用了，对吧？但是如果说系统中有 1, 000 个功能，它可能主流程占到50%，那剩下的细枝末节的很多效果，如果我们的系统服务1亿用户，但其中有一个功能只有 2, 000 人用，那么显然这 2, 000 个人用的那种功能就是我们的支线功能，那么这种功能出现问题时，整体系统应该不受特别大的影响，如果把一个小的事故由于没控制好变成一个大的事故，这就是系统架构设计上有问题。要解决这种或者防微杜渐，或者是把小问题永远控制在一个相对较小的范围内，千里之体溃于一切。如果说系统只是一个小问题，我们把它控制住了，那整体系统没有受到特别大的冲击，也还是一个好的架构设计。


断路器模式就是适用于这种场景，尽可能让小功能不要影响整体系统。小功能的问题限制极小的范围内，这就是多人气的模式，想想你们家里面保险丝它不就这个作用，如果有一个电器它出了问题，把保险丝给熔断，如果不熔断就会导致火灾，或者整屋的电气全部被烧，其实效果和作用都是差不多的。


断路器模式它一种运营场景是在长时间的故障情况下，它就中断用户的连接保护服务。这种条件是什么意思？当系统一直持续出问题的时候，但是用户还是不断地涌来，如果你不断开这种连接，那系统的用户越来越多，或者并不会减少，那么系统里面出现的问题就会越加累积，可能会导致更加严重的问题。所以在适当的情况下不让用户访问系统，也是对系统造成保护，当这个时候对用户体验肯定是定伤害的。不过为了整体系统的安全性和可用性，我们必须牺牲一定的用户体验，这也是相对可接受的一种方案。


总之，我们的目标仍然是保证系统不完全受损，让它尽可能的恢复。因为举例子来讲，一个系统它有问题，可能让它恢复个 5 秒钟，把当前积累的任务处理完，它就会恢复过来，但如果说你没有这种保护模式，用户越来越多， CPU 也过载了，内存也过载了，全部都crash，最后整体系统完全不可用，这是大家最不愿意看到一种场景，只要满足一定的条件，你就断开用户的连接，让用户不能持续地访问，以此来保护系统不整体受损。
在响应的式架构中，当消息的接收者过窄或其他方式失败的时，从一个组件到下个组件就会被中断，因为响应式价格这个大家都知道它是一个通过消息驱动的系统，那如果说在这一个链当中你把它当成一个调用链的话，我们简单思考一下，如果把它当成一个链，那这个时候从 a 发送给b， b 说正常情况下 b 再会发送一个 c 也出了问题了。


b 这个点，那 c 肯定也是不会说的， b 就把断课了，这个调用链条这个时候是因为它如果有问题你还忘了发，那你怎么能保证不会引起下游的一个问题？比如说第一 b 本身就是被很高的流量给击穿了，它出了问题它还疯狂地往 c 发，那是不是导致 c 也不可用？而 c 其实又在承接一些其他的功能，这样子其实就无法保护c，也无法保护 b 自己。所以与其如此，还不如就停止发生消息。


在故障点的这个地方，那其他至少说还可以正常工作，接收者从可能的复杂引起的故障的恢复，这跟前面讲断开这个用户的连接是一样的道理。用户的连接在这个时候断开和从负载引起的组件上的崩溃一样的道理。因为一个组件陈述了太过高昂的负载，导致它不能工作，这个时候如果我们系统检测发现比如 CPU 过高、内存过高，这个时候短暂的让它从服务当中移除，这样子它可以把前面应该消费掉的计算完毕的任务给计算完之后再恢复正常，能够接受新的请求，这样子就是对它的一个保护，可能短暂的有些功能受影响，但长此来看的话没有问题，就是短暂的不可用，换来更长治久安。


还有一种相对较简单的断路器模式的应用，发送消息的人，他决定我不再等待消息。为什么呢？可能我们有一些判断条件。举个简单的场景例子，你发送一个消息，我整个计算任务希望是在 100 毫秒完成。你可以设置 101 毫秒的时候超时，我不再等待，直接丢弃当前请求，也不再等待。您只要超过你的预期，那可能他返回给你的结果，或者是从整体的正确性还有 SOA 层面都不再满足整个服务的要求了。这种时候与其继续等待空耗资源，还不如就此结束当前交易，这当前这个调用其实可能对服务层面而言，它有没有什么特别的含义呢？还不如就直接放弃等待，这就是几种常见的这个架构模式的应用。它的范围两个分离组件通信的情况，如果故障不会向上下游传递，减慢其他组件的速度。因为在前面我们已经分析过种种场景了，比如说自身计算得太慢，那就不要拖累上家，如果自身负载过高，你向下家施加的压力也更大，下游系统它也承受不了这种压力。按与其上下游受自己影响，还不如把自己从服务矩阵中移除，等所有的故障恢复或者当前积累的任务完成以后，在接入的矩阵当中为整个系统提供服务。
断路器模式就是为了保护整体而牺牲局部的一种做法，只要你的系统中可能出现这种小故障传播引起一个级联的失败，让整个系统全部都失败的情况，那你就尽快在系统中引入断线模式。
