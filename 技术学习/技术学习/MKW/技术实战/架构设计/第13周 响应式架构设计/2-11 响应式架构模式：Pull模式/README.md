首先第一种 pool 模式 Pro 模式其实相对比较常见，而且在前面课程中我们有简单的提到过铺模式来应用场景也相对比较广泛，我们在此回顾一下。首先 pool 模式它要解决的问题是什么？在一个响应式系统里面，其中有一个挑战就是如何平衡消息的生产者与使用者之间的关系。其实前面我们在讲被压的时候，已经简单的讲解过这个问题，而且也给出了一些方案， pool 就是其中的一种解决方案。来看一下 pool 模式，让消费者向生产者请求批量数据，而非多次单条请求。

我不知道大家有没有使用过 Kafka 和MQ？ Kafka MQ 这就是典型的两种不同的思路方式来解决这种问题。如果大家都熟悉了，就知道 Kafka 是使用 pull 模式，也就是说你可以设定一次 pull 多少条就拉取多少条从服务器上面。而类似于 MQ 的这种技术一般都是以推送为主。而 Kafka 是完全不同的一种解决方案，它使用了 pull 模式，自己设定要 pull 多少数据就 pull 多少数据，这样子让消费者端就 consumer 有更多的灵活性可言。同时也不要一次一条，但是去拉很多次，因为显然你每拉一次数据那都是有资源消耗的。所以说为了高效率一点，整个系统运作的时候最好建议是一次拉多条，而非每一条拉多次。


第二个问题是当生产者比消费者快的时候，系统以 pull 模式运行，消费者在每次提出请求时，都会从生产者那里批量的提取数据。就是前面我们有讲过，不管你本身的消耗速度怎么样，但是出一个更高效率的系统，整体来看你每次拉取多条数据都比你每次拉一条，但是拉多次划算。从整体的角度来看，因为每拉一次有可能存在这种连接，建立各种其他的消耗。但是如果说你每次拉多条，那就把这个事情给节省下来，从整体上来看，资源的效率、利用率都高很多。


当生产者比消费者慢的时候，系统以 push 模式运行。生产者不需要等待消费者的请求，而是直接将数据推送到消费者，就像我刚刚讲过的例子一样，很多 MQ 类的技术实现就是以 push 的模式进行运行，它将自己生产的数据推送自己的消费者端，

以 pool 模式这样一种方式构成了一系列相关组件之间的流量控制关系，通过有工作需求，消费者将瞬时加工能力告知生产者，也就是当你作为消费者端的时候，不管你的消费能力是多少，你按照你自己的能力，从服务阶段或者从消息的生产者拿取自己能够处理的数据，而不要过载一此实现了一个非主塞的异步通道。


通过这样一个通道可以沿数据处理链传输变压，通过这样的一种方式，我们分享过你自己能消耗多少就拉取多少，就不存在你自身的 buffer 会溢出的情况，因此笨压在这种情况下几乎是不会出现的。以及我们还价的被压的这个问题，因为消费者按照自己的能力上限从生产者处拉取数据，因此就不存在一个被压的问题，所以 pull 模式是解决贝阿的一种很好的方式。如果大家在做系统设计的时候怀疑消费者的处理能力是低过生产者的，那么可以尝试使用 pool 模式，根据自身的处理能力来拉取数据，因此避免了系统的被压问题。
