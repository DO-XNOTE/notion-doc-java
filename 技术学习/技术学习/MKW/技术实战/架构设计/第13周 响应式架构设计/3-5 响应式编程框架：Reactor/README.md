同时除了 Mono 和 Flex 这两个 publisher 实现之外，诶也有一个叫做运算符，运算符是不是大家听上去和前面讲的 Arex Java 里面操作符一样的？我再提醒一遍，如果你是对 Java 的 stream 编程方式很习惯的话，那你看到这东西是完全不会陌生的。而且特别在 Reactor 里面，这种感觉特别明显，比 RX Java 没有特别大的区别，思想其实几乎都是一模一样的，这一点大家请牢记，在听这个课程的时候，你对于 Java 的这个 stream 流式编程和 Lambda 这种表达式如果还不是特别熟的话，我个人建议重新去复习一下加瓦巴的语言规范。


重新重点看一下安巴达和刘氏在操作这种了解以后你来写我们安克特程序的时候，你相对会轻松一些。当然并不是你必须要做，但是你做的话你可能就相对会觉得有点突兀，而且你自己理解这些写法你也觉得有点别扭。虽然说他们其实很多时候都是一种语法堂，但其实在你写的时候会觉得很顺手，这是一个小小的建议啊。


从实现层面来讲，如果说你只是听咱们这个课程，并不是要去做真正的利用 react 做开发，只是理解这种异步编程思想，或者这种流式编程思想的话，你了解一下 llama 达和这个流式编程也是很有好处的。这里作为一个小小的建议，其实 spring 和 Reactor 都是同一个大公司下面的两个非常出名的开源项目，所以说 spring 跟他的之间的关系非常紧密， spring 5 开始起就开始支持这种响应式编程，当然 h o 应当的它是基于 react 来实现，至于说它具体是怎么完成的，后面我们会简单的讲一下。


Reactor 里面有一个概念叫组装，不要再说一次，如果你对 G28 这套东西很熟的话，你就知道这个组装是什么意思啊。我们来看一下这个概念，将要把里面的stream，如果你对它数，你就知道这个概念是什么意思。在 Reactor call 里面有 Flex 和Mono，还有运算符，你在它上面直接调用方法进行操作的意思，它是不会触发任何行为的，用命令式的，或者叫原始一点 Java 编程方法稍微有点区别，就你直接在它调用方法，你以为这个会是生效，对不对？其实不是这样子的，这点就是你如果熟悉家伙的流失编程，你就知道你一点都不会惊讶，而是返回了一个新的 Alexa 或者 Mono 的实例。然后你可以在这个实例上继续添加其他的运算符，最后把这些运算符构成一个算子链，也就是说你通过一系列的操作，把原始的 publisher 上面的数据进行各种运算操作之后，你将要得到一个结论，但这个时候其实并不是真正开始作用，就数据也不会流动，方法也不会真正被触发，这就是它的奇妙的地方，这个过程就被叫做组装时间。


就这个时间段它叫组装时间，就你只是可以想象成你在组装一个流水线，这个流水线你通过各种各样的方式，你有各种各样的流程，对不对？这运算符和最初的原料来源就是publisher，不管是 Flex 还是Mono，这两个就是你的来源，对吧？这是原材料，然后你设定了一堆的流程，每个步骤就是代表的工序，最终你得到一个成品效应，但是并不是说你有了原始的进料口和中间的操作流程，这个产品马上就会出来。没有按下最后那一个开始伸展的开关，在这个地方就是你还没有进行最后那一次触发。


所以说我再给大家讲一下组装的含义，是你组装了一条流水线，但不表示这条流水线马上就会开始生产，所以你要把组装流水线和流水线开始生产产品这两个概念分开。在流式编程里面，这两个点非常不同，也非常重要。有些时候如果说写的时候写错的话，你永远都在组装，而永远不会生产出你想要的东西。这一点是非常重要的一个概念，大家一定要搞清楚，我们再来看数据，它是不会流经管道的，因为少了赘以后按了一下生产开始的按钮，因为触发数据流的不是管道的声明，而是管道的订阅。还是回到我刚才流水线的地址，进尿口又有流水线把整个都组装在一起了。但是投了原材料，你不按下这个启动开关，它也是不会生产产品的。


对应到咱们这个 reactor 编程里面，就是你有了原始的这个publisher，有了这个事件员。但是你虽然组装了这个pipeline，就这个管道你已经组装好了，可是你只是要不订阅的话，后面的一切都不会被触发。在 Java 的流式编程里面，如果你对流式编程很熟悉，你就知道它是需要一个事件员，中间有一个组装，最后你要有一个触发器，不管你是结束它还是开始它，你都要有这种终止操作一样的东西。你要开始触发这件事情，就是开始触发这个 pipeline 的数据开始流动，所以说你没有最后那一个动作的话，你前面就是只在组装，只有真正当订阅开始以后，这个地方的订阅就相当于说是触发开关，它表示出具要真正的开始流动了，原料投进流水线开始要真的造产品的，经过各种流水线的工艺来造出一个真正的产品。


当你开始调用了 Mono 或者Flux，它们构成的这样一个算子链的一个定义方法以后，才真正开始发送传输性，通过算子链向后传播。指导员的 publisher 为止，只有开始订阅，然后它会一直回溯到真正的那个publish，就是事件源去告诉他现在你可以开始传输数据了。这个又跟流水线有点稍微不同，流水线的话可能是正向的，就是从原料开始那样开始说那天可以生产的，但是你可以把它这么想，比如说一条流水线是在一个可见范围内，对不对？然后在终点出产品这个地方有一个闪光按钮，你按一下这个按钮，守在投料口的那个人就知道开始生产了，他把料往下一投，流程就开始启动，整个产品就会开始生产。


Flex 其实它就是 eractive stream 里面的 publish 的一个标准实践，它就表示一个异步序列的 0 到 n 个数据或者事件，它是一个长江的一个流，流里面有很多可供你操作的这个数据来源。但是我再强调一遍，流里面并不是表示是个数据的容器，它只是提供一种可以去操作数据的范式，这一点我一直在这中反复强调，大家不要把这两者混淆。

Flex 就是一个标准的publisher，它就是有多少个发射项目的异步序列。这个无需再强调，总之你可以把它当成就是我们的 event source，所有的事件或者数据的源头，由它，你把它想象成所曲线的第一个原料区，它在这个地方开始触发后续的一些生产，只有这个原料投进来之后，你的流水线唉才有意义。


因为你虽然说启动了，如果说你没有原材料的话，那个流水线也是空转的，对不对？那这个地方就相当于流水线开始生产了，但是你得要投颜料，这个 flexure 或者后面的 Mano 就是这个原料，只有他们不断的往里面输送，然后你的流水线的产品才会源源的不断的被生产出来，可以选择完成信号或者初步终止。


这就是前面讲的，当你在开始传输以后，那你最终这个事要结束的话，它肯定得有一个标志，就跟启动信号一样的，对不对？那你要结束这些事情，或者说告诉他出错了，我们就不再继续往下玩了。你类比成，嗯，流水线就是假如说我们每天 12 点就要下班了，要去吃饭了，那这个时候你怎么样？一切顺利，那肯定就不存在就出啊故障，今天上午我们就到这为止，就给他打一个信号，然后原料区就不再发送原料流水线，你就停下来了，就不再继续生产品了。他只要把前面的数据处理完毕，就是这种圆的处理完毕，这条路线就展示关闭这样的唯一不同的就是你下次再有使用者，你这个 pipeline 可能要重新组装一次，由这个flanx，它组成的这个标准的数据流就是这个pipeline。最终你在处理过之后，你要发出一个信号，告诉它是不是要完成，或者是出现了错误，我们要终止当前这个数据的传输，这一点与 reactive streams 它也是一致的。这三种信号它可以转化为下游定库的不同的方法的回调啊。


正常的一个传输就是我们一直在处理，刘正阳一直在处理，那就是 on next，这个大家应该很容易理解，就是你处理当前的这一个数据，处理完了又来一个数据，就一直来数据，对吧？这个就是一个标准的处理，那 uncomplete 就表示好，所有东西我们都传输完了，我今天比如说只进了 10 吨的原料， 10 吨原料都处理完毕了，那就uncomplete，或者像前面讲的一些其他的原因，我们要结束了这个事情就要uncomfort，让它结束，你整个流上就让它结束，然后出了故障，比如说流水线上某一个环节有个工艺它有问题，这个时候出了故障了，那就是 on error，这个时候就会触发敬业者的 on error 方法，这三种方法都是反映在对订阅者的一个回调上面，但它其实都是来自于 pipeline 里面的这样一些事件。

react 既是传送弹，又是一个装备工或者机器人原材料从源头就是publisher，它流出经过每一道工序的加工，最终才能变成一个成品，等待被推送到最终的消费者，也就是 subscriber 来消费。


在 regular 当中每个操作符。它的作用就是对 publisher 发布的事件或者信息进行处理，并且将它包装成新的一个publisher，形成一个链条。在我们开篇的时候就有讲过 negative 的特征就是可以形成一个算子链，尽管从 negative streams 这个规范来看，它并没有指定任何的运算符，也是我们的operator，但是 react 的核心价值之一就是提供了非常丰富的运算符，从简单的转换、过滤到复杂的编排和处理，每一个方面都有提供相应的 operator 操作，它这样的好处就在于作为开发者，在开发不同的程序功能的时候，可以用现成的这种 API 直接进行操作，而无需自己额外添加相应的功能。


第一规范了相应的操作，第二，省去了不必要的开销。因为这种事情无论你是哪一种操作，它 99% 都是一模一样的功能，一模一样的 input output，所以通过这种标准化的 operator 为开发者节省了大量的资源和实践。 Reactor 它一共提供了 11 种类型的操作符，分别是提供转化功能筛选功能组合功能，还有判断条件时间统计匹配分组事件调试和一些不太方便分类的其他组。这 11 类操作符如果单看他们已经很强了，但是你如果在实际的使用中再进行组合，那基本上千变万化，可以组合成无穷无尽量功能。
