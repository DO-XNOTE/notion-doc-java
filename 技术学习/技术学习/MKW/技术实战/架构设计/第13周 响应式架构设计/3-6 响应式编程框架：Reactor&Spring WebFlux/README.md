首先它是响应式编程，响应式编程里面首先就是基于事件的这样一个驱动，这是大家都知道。而且它这个是叫 event loop，就一直有一个数量不太多的线程数在这个地方来循环处理接收事件，那请求第一步到达我们的服务器，那这个服务器它就把它当成一个事件，而且它会有一个固定的callback，也就是这个地方的请求相对原始的请求是有点不太一样的。这个地方请求和响应它不是像原来一样，你发送一个请求给我，然后你等着这个响应回来就可以了，这个地方请求它一来，它就马上返回去了。


但是唯一不同的是什么？它会有一个callback，比如说你请求完好，你回去继续该干嘛？但是你要留下一个尾巴，就是我们的一个回调函数或者一个回调链，或者一个回调地址都可以。可能不同的时间手段稍微有差异，这也无所谓，反正大家懂这个意思，就是它会有一个回调的尾巴在这个地方，然后它把事件就发到 event loop 里面。也就是我们有一个很小的线程池维护者这个所有的事件的响应，当它拿到事件以后，你先抛开这个callback，就先看到事件这个主体本身它可能有些信息。根据这个信息它每一个 event 都会有相应handler。如果说大家之前有认真去研究过Nginx，就知道它的 event loop 几乎原理就是一个，就是基于事件驱动的这样子提高效率，因为这样子它只要达到相应的事件，把这个事情交给handler，它就可以去处理另外一件事情。不用像之前的这种同步调用的 blocking model 一直被 block 住，直到上一次结束这个线程才能归还回去。而这个地方不是这样子。因为他是马上把这个事情交给了 Handler 去处理， Handler 开始去做各种各样的处理，那就看底下是谁，如果说是有这种响应式变成 Q2 没有的话，他就只在这一个地方被 block 出，然后得到这个结论作为另外一种 event 再回来。这个时候他看到这个 event 他就知道，哦，原来是要去调用回调函数了。


现在我的这个结果出来了，我现在要把结构返回给最初的调用者，它不是直接向传统的 request response 这种返回方式，它不是这种方式，它是通过一个回调函数去触发相应的事件。这个大家对回调函数理解的话，其实理解其实是一点问题都没有，所有的事件它发送到服务器端的时候，它都给了一个后手，就是叫做callback。用 callback 可能有很多种不同的实验手段，这个不同的技术有什么差异？我这里就不展开讲，大家有兴趣的可以自己去研究一下到底怎么来实现的。这个callback，这不是我们要讲的，重点在这里，它在这里所有的事件都跟一个 callback 函数或者 callback 的一种手段是关联在一起的。这样子当你最终结果出来的时候，因为它之前调用完之后，它马上就返回去了，不存在的时候在等待的问题。所以 third 你继续处理，你处理好了之后把这个结果告诉我就可以了。现在他把结果来了，然后做一个 call back，然后你原始的这个调用者他就会有相应的处理操作。因为 call Mag 它被毁掉的时候你自己应该干嘛？那你自己处理这个跟服务端有什么任何关系的？它只是负责把要回调的内容和信息告你，并且启动这个回调它调用就可以了。他只做这些事情，至于说回调函数里面是什么功能、什么结果，他是不关心的，所以这两者也从这种层面上把它分离开了。


当然更重要的一点就是退出调用的它就已经返回去了，它不会一直等的你后 b 端的业务逻辑执行完毕，因为它已经返回了，然后当你真实结果出来之后，你再给它做一次 callback 执行，完成了整个调用。当然说你说时间是一样还是不一样，从客户端的感受来讲，它的时间是变快了，因为它直接就返回了。其实从整体的处理速度上面，比如说只是单从处理这一件业务学的这个时间上面，他其实是没有特别大的区别的，唯一不同的是对用户来讲的话，他的感受是提高了，因为他马上就返回去了，就是这样子的。
