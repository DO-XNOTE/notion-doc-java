# 1-15 响应式架构本质：消息传递

今天我们再来看响应式架构本质的第二点，消息的传递其实是基于说我们上一个特征，我们更倾向于在小程序架构中，或者响应是系统当中才有消息机制，对吧？那你要了消息，那在系统中发挥之后就必须要被传递，从一个生产者传递给一个消费者。这个就是一个很重要的特点，你不要小看这个消息的传递，因为其实无论从概念上还是从技术实现层面消息的传递，这都是面对很多挑战。我们先从概念上来看一下这个问题啊。首先消息在传递的时候它有两个含义，第一，如果有一个消递出来了，那肯定表示在系统中肯定有什么状态发生了改变，或者有某种事情被触发，这个事件被触发了以后，由于之前的一些契约、一些定义，表示要把这个事件给广播出去，以此向外表达一个有状态变化或者有事情发生的这么一件事情。

这个其实在我之前的课程当中就是讲到这个事件驱动系统这种其实我已经有讲过，这种原理都是一模一样的。第一是讲的是事件本身为什么会出来，或者消息为什么会出来？第二就是这个消息你就不要单纯看它只是一个字面上的这个消息，这个意思，其实它是绑定了很多信息出来，除了说你的这个事件本身或者消息本身而言之外，那在这个消息体当中它也是包含了这件事情发生的前因后果，也就是我们所谓的上下文。


通过这个上下文你可以推出很多事情，做出相应的这样一些操作。所以说消息本身和消息它所涵代的上下文，其实在这个消息传递的过程中，它都是被附加出来的，对你后续的所有的操作对接收者来讲它都是有意义的，甚至在消息传递本身的过程当中，也会出现这种利用消息上下文来进行一些额外操作的东西。


例如说可能在有些消息系统里面，它可能会进行一些路由规划，那这个路由规划它就会基于你的上下文里面的一些东西来做一个物流的规则。六证路由规则来实现一彻底的功能，或者说消息当中，比如说我们有一些特殊的逻辑，在路由过程当中你可以把它过滤掉，有些消息根本就不用特定到达某一种特定的目的，这都是有可能的。所以说消息本身以及消息所含有的上下文，它都是对我们整体系统后续操作有影响、有意义的。这就是消息传递的其中一个含义。


如前文所说，消息其实也可以被认定是一种信号量，这个其实很好理解。就是我前面说过这个道理，当你系统中某一种状态发生了改变，那你要告诉外界，那你就要发一个消息出来，或者你把它叫一个事件也发，我们就把它叫做一个消息好了。在这个消息它就表示有一些事情发生了，对外你表达了一种含义，说 something happened，你让人家知道这件事情，我们简单的理解这样一个场景啊。


比如说现在在一个电商系统当中有个订单被取消了，那如果说我们假定是没有这个消息传递这种机制的话，那你怎么办？你就只能自己不停地去扣一个API，或者不停地去做一个查询这个订单的状态有没有发生改变，对吧？那这种方式当然你说它能不能起作用，我不能说它不起作用。但是在现实中目前这种做法的几率是非常非常低的，因为这么做太浪费资源，而且太低效，也不实际，不够实用，这可能是非常非常古老的做法了。


现在的系统当中这种做法应该是少之又少，都是肯定在这个订单的状态发生改变的时候，都会发出一个广播消息，以此来通知 something happen，在这个 happen 就是什么这个订单被取消，那这个订单被取消的信息传播出去的，它可能会表达两个含义，一订单某某某订单被取消了，这是第一，第二这个订单为什么会取消？可能还有跟这个订单的相关信息则会被异常广播抓取。这就是我们在第一点里面讲的这个上下文信息，所以这些信息被广播出去，然后所有只需要对订单被取消这件事情感兴趣的接受方，他都可以做出对应的操作。所以说这个地方消息它是两层含义，那前面你就成 young 本身表示是一个信号点，告诉你订单被取消了，你应该做一些事情。


第二就是消息体本身里面它是带有上下文的一定的含义，你要从这个含义中推出更多的信息，做出相应的操作。第三一点就是一样的道理，跟前面我们举的这个例子非常接近，生产者他把消息发送给这个对该消息感兴趣的消费者，再消费消息。好，简单来说就跟前面讲的还是一样的。我作为生产者，根据我一定的条件，我可能规则一把，而不可能还是代码，它会把有事情发生，这件事就是 something happened 这件事情。宝宝出去，那我们说的稍微那个细节一点，就是可能这种事情它会对于一个主题，就是我们所谓的topic，对吧？或者对应的q，它会把这种消息扔到这个里面去。如果在这个 exercise 里面，可能它就是某一某某的一个东西的一个信箱什么之类的，它发到这个里面去，然后接收者它就会做出对应的行为，就是消费者消费这个消息，这个消费消息和生产消息，这个在我们之前讲的事件驱动里面其实已经有讲过了，这个时候我也不再详细的去讲啊。
