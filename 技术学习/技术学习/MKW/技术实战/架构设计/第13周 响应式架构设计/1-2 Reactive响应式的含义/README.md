不管是响应式还是反应式，下意识的第一件事情可能要去了解这个响应式它到底是什么意思，对不对？OK，首先从英文来讲，因为前面讲过的歧义，我们先说下英文是reactive，对吧？ reactive 它是什么含义呢？日常的例子，我们对这个响应式它有了一些初步的印象，当然这些例子说的这个和我们实际在 it 系统当中所指的这个响应式，那肯定是有非常大的区别。所以现在我们就探究一下这个 negative 或者这个响应式它到底是什么含义。

这里所在刺激就是说第三方发给你的一个请求，对吧？要进行一个快速的响应，这种就是通过前面的例子和你思考一下你自己平时做过的或者用过的一些系统，对这个世界进行快速响应应该是有很明确的一个了解，对吧？第二个就是对世界它要立即响应，因为其实后面学习大家会告诉你，其实在这个响应式编程里面最重要的就是一系列的事件，每一个组件都是对书的一个事件，它做出一个响应。这就是这个系统存在的意义，事件驱动的自然特性使得这个反应能够立即加载实施，这就是对事件产业的这样一个含义。


也可以通过这种方式避免对共享资源的征用，实现可扩展性，因为你对响应事件来讲，就是说如果有一系列的事件通过一个时间流进来到这个系统里面，但是我处理它的时候是一个一个的处理的，不是一拥而上，以此达到这样一个避免竞争的这样一个可能性，而且提高了这个资源利用率，对吧？三一点，对，失败也要立即想这个可能有些同学听到这里会觉得有点奇怪，失败有什么好响应的？那如果你是这么简单理解这个响应式编程的，或者响应式价格的话，那你结果你的认知真的是有点肤浅。


因为在一个正常系统当中，对正确的输入我们做出正确的响应，哎，这里说当然他才没什么疑问，对吧？但是你对一个设计良好的系统，或者一个设计良好的编程方式，他对错误或者失败这种意料之外的东西或者异常，他也是需要有一个很好的处理方式的，就是你要保证在错误处的时候你不会引起系统的崩溃，你还是要给客户或给你的调用方一个适当的响应。


要做到这一点，那就要求什么我们在任何系统的一个级别，因为在这个响应式系统里面其实是分层次的，对吧？在任何一个级别，它需要建立一个能够实现了失败之后能够恢复的那个弹性的，就我不怕你有问题，但是你有问题了之后我系统不会挂，我还是可以在这里为你提供别的服务，对用户立即响应。


这个就是基于以上几点，就前面这三点的一个综合结论，因为我们毕竟最终收到用户提供服务，对吧？只要你前面三点做到，那自然而然的话，对用户的立即响应这个也就做到在这里这几点是相辅相成的，对吧？从这个角度来看，首饰大家对这个 relative 或者这个小诗的理解就不要太字面化，它其实是分多个层次的含义的。


他在用户层面的话就是说无论在何种工作负载之下，就这个时候可能系统压力已经非常高了，但是在这种压力之下我也还是能够满足用户对这个时间响应的一个要求。比如说线的一个常见的说法是，一旦用户访问你这个网站或者使用的一个东西，你要加载的时间超过了 3 秒左右，几百用户扭头就走，但这是大部分的情况，那所以说我们要达到一个响应式的这样一个编程习惯和模式，以及这样一个架构的目的，就是为了尽快给用户一个响应，让他能够尽快地看到他想要看到的东西，提高产品的竞争力。所以说以上几点大家可以听听和你平时做的系统，或者说使用的系统这个互相印证，有没有想到说哪些场景或者哪些系统已是见过听过他使用了，可能是使用了这种响应式编程，或者采用了一下应该是价格这样的一个设计指导原则。
