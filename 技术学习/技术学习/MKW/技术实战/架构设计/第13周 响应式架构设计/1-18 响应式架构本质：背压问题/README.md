# 1-18 响应式架构本质：背压问题

下面我们讲一个相对较难的点。什么点呢？就是叫流量控制，流量控制其实在很多地方都会有，但是我们这个大家讲它不光是流量控制，还有流量控制比较附加的叫做被压的一个问题，在详细架构当中谈到的这个流量控制略有一点点不同，这个不同知识在于哪里呢？它有一个概念叫被压，作为一个非常重要的处理的问题来体现在这一点上，其实在别的系统设计架构当中可能不是那么太在意这件事情的处理方式和手段，但是在响应式编程或者响应式架构设计的系统当中，对被压的处理是非常看重的，为什么呢？因为它全体的这个整体价值都是基于异步消息的，因为你其他系统可能未必有这么严重的一个依赖关系，但是对于响应式价格来讲，它因为太过依赖于这个异步消息，所以说他对这个被压的控制是非常重要的一个策略的手段，在很多系统实现的时候，所以他很看重。

我们现在就来看一下这个问题，流量控制本身是什么意思？这个还是相对比较简单的流量控制，就是调整这个消息的传输速率，保证这个接受者不会过窄的过程。说白了就是我们达到这样一个流量控制手段，你要把下游都撑爆了。诶，你有什么情况会发生呢？如果是出现了这种情况，消息的发送者被通知说你要放慢速度，发送者就在承受力量，这个其实是什么情况？因为在咱们这个诊断里面，在流处理的时候，生产速度大于消费速度，这个时候消费者的 buffer 一出就是在承受不一样，

它主要就是生产者，消费者之间的这个速度不均衡导致的，这就存在这个被压的问题。那么在响应式编程中，无论是哪种语言或者框架，在设计的时候就必须得去考虑如何处理这个不一样，所以过来后面的课程更加容易理解。


我们这里先讲解一下 impression 并不光是指想是编程独有的，其次它也不是一种机制，也不是一种策略，它是一种现象，在数据流从上游生产者到下游传输的过程中，上游生产的这个速度只要一旦大于这个下游溢出，它就是 bug pressure 编程中的这个 bug pressure，它其实来自于工程概念，这个原因什么呢？就是在管道的这个运输当中，只要气流或者液流，如果管得突然变细，或者一个很急的这样一个弯头，大家见过自己家里的下水道吗？就你知道如果说你们家下水管它有很弯很弯的，但上面可能即使你下水抖的时候那个入口很大，但是通过那个地方的时候它其实很慢的，这样子你会发现你框在一盆水倒下去，没有你想象中的那么快。


上面看你这个孔很大，其实下面它一因为有弯头，二看它下面是窄的，所以导致你上面的水一直流不下去，这个时候它其实就在承受一种不一样，为什么？因为你上面倒进的水比下面流水的水多，对不对？就这个意思，那所谓的这个被压其实也有一点，从词语上面翻译上就有点问题。
它本来也不是说这个back，它不是背的意思是向后等于是back，对吧？就是向后压的压力，因为从下游承压的地方记得过它是一个承压的地方，对吧？因为自己消费不掉，可能这个压力就倒推到上面去了。比如它其实形成一个向后的压力，这个地方的后，其实就是当时生产者往下丢东西的那个地方。


就这个意思，其实程序开发中咱们讲的这个 back pressure 和城市 back pressure 非常相似的概念是数据流产生太多而导致了这个下游 buffer 溢出。重点不是说上游生产速度大于下游的这个消费速度，而是在于说下游的buffer，它有溢出壁压和buffer，它是一种相互相成的概念，只有设置了这个 buffer 才会有这个被压的出现，只有设置了 buffer 它才有这个风险，因为你没有设置它就不会溢出，你在没有溢出的这个情况，它就不存在一个被压的问题。


既然如此，为什么还要设置 buffer 呢？这个很简单，因为我们知道下游的这个消费速度，它小于这个上游的生产速度。就像我前面举张往下水袋里面倒水的例子一样，你上面的这个漏斗口很大的，你可以倒一盆水下去。但是你家的下水道下面出水口它能用一个盆刺那么大吗？那显然是不可能的，比如说你的下游的消费速度，小生产速度，但为什么要丢弃这种事件呢？因为处理不过来，所有把它放到 buffer 里面，还有也有可能事件本身是可以丢弃的，由于这个消费的速度和生产速度，设置 buffer 由于一些外部条件的限制，这个 buffer 它也不可能无限制的上升。一旦出现那个备样，就是从处理的这个能力加上这个 buffer 里面全都满了这个你在系统里就没有别的选择，你只能把它再往下的水就丢掉。


如果说你的现实世界中就有类似于像这种，比如你加的这个，像它堵住了，那你再往里面一直管着一个户口是什么，那水就全部溢出了，这个新增出去的水就被丢弃了，就是一样的这个道理。这个逻辑就是这样，再强调一下，意思就是说为什么要设置buffer？ buffer 就是为了说尽可能的容易多一点，所以说我消费的慢一点，就像有个蓄水池一样，对不对？蓄在这个地方，等我这边处理完了我再来拿。但如果说实在是你生产的太快，我消费的太慢，这个 buffer 它也没有用，被承包了，那这个时候就只能丢弃时间，新来的东西全部给丢弃掉， buffer 里面清空了，或者我原本的处理完了，才有可能积攒新的东西，这就是一个 back crash 概念，肯定是编程系统里面切记不要用这种，上游的生产速度是否会大于下游的消费速度，判断这个模块或者这个系统它会是否有被压的问题，因为这个实际的系统只要一旦运转起来，当你说我提前经过预判不大可能。


但是事实上在现实场景当中总会有你想不到的情况出现，所以说生产速度一定是有可能大于这个下游的消费速度的buffer，它是永远需要的。所以只要你的上游生产出多，不会快到把这个系统搞崩溃，那么就不用设置 buffer 的上限，这样子也不会存在一个被压的问题。比如说系统产生的总有个极限，只要在这个极限范围内我的 buffer 都没报，那这个其实就是无所谓的，但这种情况相对可能不是那么多。


第二，只有上游的这个生产速度，有可能把整体系统都给弄没办法工作了，就是整体系统这个响应都完全没法响应的，这个时候你才需要把这个东西给丢掉。前面第一点是说的是，但是如果说他再怎么往里面填，你的那个 buffer 都是足够的，不足以让他把系统给弄崩溃，其实都是 OK 的。那在什么情况下我判断说他要丢弃这个时间？只有说他实在吃得太满了，这个系统弄得无法工作，无法响应了，这个时候新的事件就只能丢弃，这个时候需要设置一个 buffer 的上限，当 buffer 有上限的时候，笨牙存在的出现的可能，一旦被压出现，则 n 选择丢弃新的世界。


当然这个后面我们的实现当中会讲到有很多种不同的丢弃这个事件的策略，在那这次我们在详解，在这里我们就不详解说到底采用哪些策略来丢弃这个新的时间。最后如果上游的生产速度可能让这个系统无法工作，但是你又不能丢这个怎么办呢？那就证明说你这个事情系统设计上就有问题，你就不应该有这样一个问题出现，就是说你要么就不应该需要 buffer 把这个系统搞崩溃，这是一种方法，就是我们说的第一个，第二个就是你来了世界，这个世界我可以丢了，让系统不会崩溃，你说我就不让它崩溃，但是我也不想丢弃，那行不行？这个问题其实应该说不是用工程方法来解决，而是你设计程就出了问题了，你在技术层面你要做的就非常困难，这种情况下如果一旦你发现你系统分析完之后出现这样一个状况，那你就要回去从头看一下设计者是不是哪里有问题。


所以说 buffer 这件事情我们在这个被压之间可以说是成队出现的，只有 buffer 满了，你才会成为被压，有了被压你就要把它丢弃。如果你又存在说我不想让这个生产者生产的东西太快，导致我下游这边出问题，最后导致整体系统的崩溃，例如说我又不愿意丢弃新的这个世界，这种系统你设计上出了问题，当然这个前面讲的是一种，我就用变压的方式来处理这种，那另外一种方式其实就不太一样，就是有一个动态的这个 pull push 也是一种解决问题的思路啊。

这个相对来说是怎么样？比如说那慢的话，快的话它处理方式不一样的，假定 n 是这个消费者的这个处理能力，对不对？如果说我知道我只能处理 n 拉的少，那我就把它拉少一点，在基于这种破的模式下面，相对就比较简单一点，你知道自己每秒钟能处理 10 个请求，这就是对你的 n 十，那你就只拉 10 条消息嘛？这样子你拉多少你去处理多少，就虽然它生产的很多，但是你是取用拉这种模式的话，这个问题其实也不存在，所以说从某一个程度就规避了被压出现的问题，这种其实也就提供一种新的锁，就是前面我们讲的，如果你发现又不能丢弃，又要不让他崩溃，就要换个思路了，是不是之前这个设计上有问题？这里我就提供了一种新的思路。


比如说你知道我这个生产者的这个速度是很难去控制的，甚至说我必须得让它生产那么多，那你消费的这个问题是可以受你控制的话，那也很简单，知道我每秒只能处理多少，那我就只拉多少。就这样子，这是一个 poor base 的点，对于你这个 consumer 使用能力比较差的情况下，就处理能力比较慢的情况下。那另外一种就是我这个消费的比较快，第一次请求没有收到就发出更多的请求，然后 publisher 就根据不同的 subscriber 满足需求，这种就是一个 pop sub 的这种另外一种实现方式，发起一次请求，我马上又发起一次请求，不停地发这样子，因为我消费的快，对不对？
