# 1-19 响应式架构本质：目标透明

下面我们来看一看，想是架构本质的另外一个特点叫做目标透明，其实目标透明在前面的一些讲解中，其实我有间接地提到过这个，他说好理解也是挺简单的。简单来说就是目标透明表示作为发送者他是不需要操心谁来接收，谁来消费这句书发送的消息，但是它背后的含义其实有很多层次的，那我们逐一讲解一下。


未知透明星是一种属性，就是无论你接收者在哪里处理用于发送消息的源代码看起来都是相同的，特别是你希望解耦合，降低依赖，不要让接收者和发送者之间产生直接的联系，那会是透明性是非常重要的，因为对于发送者来讲，他完全不用关心是谁在消费这个消息，而对于接收者来讲，他只管消费消息，至于是谁发的，从哪里发的，什么时候发的，他都不用操心这一点二者的关系就变得相对较弱，虽然说从整体系统运作的原理上。这二者是有必然的联系的，但是从设计的理念来讲的话，他们之间的合度很低，这样子依赖关系就必然不会那么强烈。

对整体系统的解耦，或者说我要做一些替换升级，甚至扩容弹性在这种变化都是比较有利的，所以说目标透明性在整体系统之间中是非常重要的，我们尽可能去达成这种，并不是所有的地方你都可以使用这种技术应用组件以显示效益，传递定义的统一的方式相互交互，这个就是讲的所有的消息发送的时候它都要发送，然后你所有的交互都是以消息的形式来完成的，所以说这种都是有一种目标透明性的特征的。


你通过消息的方式来跟下一步进行通信，或者对外发起一些要求，或者表示一个信号量，也或者说是系统状态的一个变化，甚至有可能是把事件帮助你的消息对外发送，这种我们统一为全是作为消息发送，但是消息发送前面讲过了，目标透明性对它非常重要。因此在系统设计中它是统一的我们的交互方式对整体的系统设计这样标准它相对标准化以上的，对吧？就是把整个系统的这个操作模式都给固定化了，以此来摆脱了上下游的很多强依赖关系，这跟我们的目标透明性的要求和后果是一吻合的，其目标是将本地与远程交互通用，抽象变了消息表达的进行统一。这就是我们前面已经提到过理论上响应式架构的或者响应式系统里面。


这样的系统里面所有的交互全是通过消息来完成的，但是有可能有些消息其实从实现层面来看它是一个本地消息，有些其实你部署到这个接触者，他可能实在比较棒，或者说你在这个 q 或者是这种消息队列什么之类的，它其实不跟你在同一个网都是有可能的，但这只是从实现层面，而对于发送者来讲，我们的抽象做得好的情况下，它是完全区分不出来的，它发送的消息也是发送一个本地的接收对象，还是一个远程的接收对象。
这样子如果说你要做系统迁移升级这种东西的时候，你就不会犯难，因为理论上如果说没有这种目标透明的属性这种做法，那你想想这个系统会变成怎么样？那是不是你发送远程消息和发送本身其实相对差别非常大的？就以一个你比较熟悉的例子，味儿不宜言，对吧？如果说你是发一个本的消息，有可能你只是进行一个方法调用。如果说你是发送一个远程消息，那你可能要做的是什么？什么 HGP client？或者什么 soft Messenger？或者 listed for the json question。


所以说你的其实是没有统一的，但是在响应式架构里面，很多语言或者框架层面他就给了你一种幻觉，你写程序的时候至少说你是没有这个感受，就说我这个到底发的是远程还是本地？因为你对外看来，或者在写代码的人来看，他就是发送了一条消息，至于这个消息发到哪里去，他是完全不用在意，他也不知道。


目标透明的一个好处，将本地和远程的这个界限，或者说代码层面的东西传的比较好，封装的比较好，看上去差异比较小，为你系统整个升级有很好的一个基础，将所有的消息传递视为可能的远程消息，可以任意重排组件而不必更改其代码。这个和我前面讲到这个道理是硬不一样的，因为理论上你是无需关心它到底是本地还是远程。所以说当你发送消息时候，你的代码都是很随意的这样子，因为写出来代码都是差不多的，按理来说发送消息你也不用太在意强烈的这个先后顺序，因为在这个响应式架构里面都是异步的，所以说你愿意先发什么后发什么。


其实在这种价格模式下就没有那么重要，因此给你带来另外一种模型，就是你可以自行安排这些组件到底如何先发哪个，后发哪个消息，都是可以这种给你系统带来更多的灵活度。所以整个目标透明性，在我个人看来它就是为整体系统提供两个好处，一对于发送者来讲，它不用关心目标方式在哪里，它对外的代码写出来的方式就比较接近或者相似，因此你也不用太操心，所以说抽象层就做得比较好。第二就是统一的这个消息的传递方式。因为你无论是本地还是远程，因为目标透明了，你就不用太关心，除了它的标准化，这种标准化也为你将来系统升级或者在改造提供了一种隐形的好处，就是你不用过于在意它的细节，因为抽象层面做得非常好，都是标准化、统一化的，你不用关心它是远程还是本地。因此目标透明性在这个响应式架构的程序设计里面其实是非常重要的一个属性。
