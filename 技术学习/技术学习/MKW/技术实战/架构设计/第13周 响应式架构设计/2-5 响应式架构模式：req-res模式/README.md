这种模式我不会花太多的时间来讲解，因为这种模式其实已经非常通用的一种模式，我们简单的从理论上来理解一下这些，其实对于去理解系统，一般还是你自己做设计，一般都是有很大的好处的。我们来看一下 request response 模式，它主要解的是一个什么问题？其实我不知道大家有没有平时在使用这种模式，或者平时自己写这种程序的人，有没有想过为什么会这样的 quest response 模式？这个问题其实很简单。假如我们有一个系统，系统里面有组件 a 和组件b，其中 a 是知道 b 的，但是 b 是不知道 a 的，这其实就有点像是一个客户端服务器的模式的这样一个初始条件，就他们最初的状态就是这个样子，其中有一方知道，另外一方仅此而已。然后客户端需要走第一步，因为服务端他并不知道客户端在什么时候需要什么服务，大家明白了吗？那 customer response 就应用在这种场景下面，在一个系统当中有一个组件知道另外一个组件，而且它知道服务端组件它是能够提供一些功能的，但是作为服务端本身而言，它是不知道客户端什么时候需要它，或者需要什么样的功能，这两者它是不知道。

因此才出现了 request response，也就是客户端他在需要的时候发出一个request，然后服务就知道原来你在这个时候需要这种服务，因此我就经过计算给你一个response，在请求中肯 link 得包含回信地址来接受这个响应。


这个其实道理很简单的，不管你用现实时间的一个类，比如说写邮件，还是你写程序的时候，http， request response，你在发起家底层依赖都是依赖于 TCPI p 的话，那你想一想 TCPI p 是不是会告诉你我们会建立一个通道，使用这个通道来进行通信？这种通信是包括客户端向服务器发起请求，同时请求完毕以后的响应也会沿着原通道返回来。


第二，这个请求者他必须得知道服务的提供者的地址，不管是哪种形式，因为可能在不同的协议规范当中，他的地址表现形式是不一样的，他都得知道这种目标地址，然后才可以把请求发送给这个服务端，也就是这里所谓的响应者。响应者就是能够给他提供响应的组件。响应方，也就是所谓的服务端提供这个响应能力的这个组件使用相应请求中包含的地址信息向请求者发送响应。


这个我们在第一点就已经讲过了，你必须给人家一个回信地址，简单来说就是这个意思。这种模式就像我开头讲的一样，其实是比较常见而且比较普适的，只要是在这种一方知道另外一方的情况下，都是提出请求，需要响应才得到结论，同时表示请求已经收到和执行。所以说只要是这种场景，你知道一方的地址，也知道它提供你所需要的功能，那使用请求响应模式是没有错的。
