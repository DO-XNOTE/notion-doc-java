一般来讲在一个领域任何一门技术有一项趋势出现苗头的时候，肯定是百家争鸣、百花齐放。在刚开始的时候，每一家或者每一个开源实现都是有自己的想法的，带着自己不同的目的的。


可能有公司利益的驱动，也可能是项目组本身的一些要求，所以说它实现方式方法未必就尽如人意，各有各的优点，各有各的缺陷，但是当一门技术到了一个成熟度以后，如果已经是被广大主流人群，或者是广大开发者，或者是各大公司联盟对这个成为了一个认可，一种主流技术了，那这个时候市面上必定会出现一种规范来规范行为。

为什么需要一种规范？简单来说就是如果各家都有自己的独立实现，那么互通有无这件事就很困难，如果说在尽可能的范围内照顾到彼此的利益，在某个抽象层次上面大家达成一个共识，在实现层面各自又有不同，这样是皆大欢喜。我们一起来看一下这响应式编程这种方式下面有什么样的规范帮助大家来统一思想。


那现在最出名的可以去 reactive streams 这个规范基本上现在就是一种共识了，只要你是提到做响应式编程。当然我指的是绝大部分主流的情况下，它起源就像我前面所说的一样，就是为了统一各种不同祥式编程库的设计，因而催生了 reactive streams，就是因为最初各方的利益没有互相兼顾，其次因为每家公司最初做的时候也不知道未来发展方向在哪里，也没有把它直接就成立委员会什么之类的。


对 Java 人脸的话一般一个规范追求起草，这个肯定要提交给 GCP 这种规范组织来进行审核，各种在这之前其实这种技术已经出现了苗头，到最后等到家务委员会来介入的时候，那就相对来说是这一门技术已经到一个成熟度了。就像我前面讲的一样，所以说在前面百花齐放以后，现在我们要进行一个统一思想的行为，这就产生了 negative strings，它就是一种标准化的技术。
简单来说就是要为各大厂商提供一个标准，大家都在这个标准之下各自进行实现，而这种标准的好处就是业界之间互相交流或者探讨，完成技术的时候可以互换逻辑，这种标准化的技术它就是个泛化，在 Java 当中或者在别的语言当中来实现这种响应式编程，用于异步顺序和强制非主塞回压力处理潜在的不守约束的数据流。


简单来说这个地方你可以把 reactive stream 看成两个词，一个是stream，大家都知道 Java 现在是非常推崇流式编程的， Java stream 是有专门的撰写规范的，只是针对 Java 里面自己的stream，比如你可以把connection， list 什么之类的对这个 stream 在进行操作，在那个上面它是一种流式规范，然后你跟 negative stream 结合在一起就变成了响应式的流式操作。


所以说就把二者的好处都结合在一起了，规范它是由四家提倡的最早的公司来提出的一个共同规范，由 Netflix red hat、 Twitter hat。它定义只定义了非常简单的四个相互关联的接口，用来表示反应是流必须提供的功能集。只看这个接口或者只看那个规范本身而言的话，它提供的功能是非常简单的，它就是为了规范这些行为和接口，让大家能够在彼此的实现之间互相选择、互相切换、互相支持。


基于这种 interest stream，有阿卡react，RX， Java works 这些东西都是前面所说的不同的供应商。但其实他们都遵循了相同的 negative streams，这就为我们广大开发者提供了更多的选择，我们既享受了面规范或者面向结构变成了好处，同时也可以相对自有的在不同场景先选择一个你认为最适合你场景的实现方案，而且这种规范和实现分离的好处也为将来规范本身的进化和各大厂商自己的独立升级打下基础。
