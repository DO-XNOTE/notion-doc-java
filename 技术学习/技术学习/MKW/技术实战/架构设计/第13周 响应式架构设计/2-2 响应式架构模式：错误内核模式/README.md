# 2-2 响应式架构模式：错误内核模式

首先来看一下错误内核模式它要解决什么样的问题？我们知道响应式架构它是具备故障分层处理的能力的，之前我们有讲过，会有赋予此证涉及关系彼此被不同的责任层次进行了一个隔离，但是它的后果就是如果组件发生了故障，也就是所谓的parent，就是附节点，它发生了故障，那其所属的下属结构或者下属的节点将会受到后续重启启动的影响。


如何将所有的内容重置为已知状态，并且保证不丢失中间的更新？因为我们要求响应式架构它的能力其中之一就是有高较高的弹性。男性就要求当系统出现故障的时候能够快速恢复，但是恢复的手段之一，你要可能会面临着将某一个组件重启，那如何保存？在重启的时候状态不丢失，就是这个意思，在这个地方出问题和模式要解决问题，与此却是相关联的。因为我们进行了这种父子一样的设计，在这个沙发当中如何保证父节点在重启的时候并向子节点，而这子结点能够读取到重启之前的种种上下文关系？这就是一个挑战。让我们看看这个模式它是怎么做的，简单来说就是在这个错误的层次结构当中，有风险的操作都委派给这种，下一集就是它叶子的时候，要把所有的与之相关的上下文或者说状态保持根部路附近系统在进行一些相对可能出错几率较高或出故障率相对较高的操作的时候，你要先将状态更新到一个根节点上去。


这样的好处就在于，如果说不管是叶子节点，因为它还可能有自己的叶子节点和当前节点出现问题的时候，你还可以从根节点中把它恢复出来，虽然说这样可能会有一些资源或者说性能上的微小损失，但是为了保证整个程序付出一定的代价，这是必不可缺的。世界上没有免费的午餐，大家应该都知道，你要保证状态不丢失，那你就必须得把状态给它存起来。危险操作的时候把状态更新到一个根节点，这个根节点它是不会随着应用的重启或者说恢复这个弹性一系列操作受影响的。就这样一种地方，你可以把状态存起来，后续的操作可以从中恢复到当初的情绪状态。那就是这个模式的一个操作方法，主要功能就是把系统的操作约束集成到基于责任的问题分解中。这个话的意思就是说如果说你在不同的地方可能会有不同的问题域，也有不同的解域，但是你分解这些问题的时候，你把风险较高的和较低的分解到不同的组件上去，不同的领域问题上去。


同时你要把这种问题都归集到相对较窄的一个范围内，与咱们之前讲过的这个错误隔离有点只是类似，不是完全相同的，类似的一种死角疤问题和故障率尽可能的相对约束在一个教养范围内，就是这样一个含义，它一般适用于什么情况呢？首先系统由不同可靠性的组件组成，并前面有讲过在这种不同的操作当中，它各自的这个风险或者故障发生的可能性是不大相同的，那如果有的组件相对较高，有的相对较低，那你就自然应该明白把风险委派到哪些地方去。故障率严重性不尽相同，以此把错误的可能性在较高的地方做一些风险控制操作。就是像我前面讲的，把它的系统状态在某些故障率可能较高的地方保持到根节点中去，这样当它后续恢复中，可以从根节点把状态恢复出来，让程序能够保持运转。
