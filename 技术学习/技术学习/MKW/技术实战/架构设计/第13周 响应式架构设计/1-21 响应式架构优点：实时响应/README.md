# 1-21 响应式架构优点：实时响应

前面我们了解了响应式架构的概念和一些基础的理论知识，让我们对它有了一些好奇心。现在我就要问大家一个问题，你们有没有想过为什么要使用响应式架构，或者采用响应式编程，或者设计一个响应式系统？有没有哪些原因呢？当然你说我平时为了自学或者公司使用的这种方式，那这个没什么好说的，完全就没有意义。那我们要从内心自己来分析讲师价格给我们带来些什么样的好处，但是意义何在才是我们真正要追寻的东西。那我下面就基于一些过往的经验还有理论知识，总结出了几点和大家分享一下。

这一块就当我抛砖引玉，大家可以共同探讨。首先响应适应用程序都是实时的，就这实时响应性这个是最为看重，所以说它才被称之为响应式系统。前面我们在讲基础知识的时候已经反复强调过这一点，这样设计出来的系统让客户他有实时协作的这种功能，就是有一种感觉是我有什么变化就马上获得了他的结果，这种感觉。当然这是一种感觉，所以说感觉其实也就是用户体验，就是在这种情况下，响应式系统里面用户的体验非常棒，非常好。


说一说作为我们一个开发者或者一个架构师，就设计出这样的系统来，让用户觉得用户体验很好，就是这样子让客户有实时协作的这种功能应用，与使用者建立了一个开放、持续的对话，通过响应获得了互动体验。很明显就是之前我举过的一个网站的例子，比如你在网纸上输入了一堆的东西，然后马上给你组成一个下游的东西做出计算，你随便改一个东西，马上给个对应的响应。


这种体验其实你在一些本地工具当中比较容易获得，这其实也是一种流式计算，或者说是响应式架构的一种体现，所以在这种很下面它就给你一种感觉是非常舒服的。那相对来说传统的做法，比如说举个例子，你是一种公式计算好了，你要先填两个数字，然后点一下计算可能才得出一个结论。那如果说你变成说你这里面随便什么数字，它就直接可以算出最后的结论，中间只要你不改的前提下，随便怎么改，马上出一个结论，这是一种响应式的体验，它时时改变，让用户觉得它时时在跟服务器进行交互，说这种感觉对于用户体验层面而言的是非常好的一种。


那咱们做软件的，不管你是互联网软件还是这种公司级、企业级的软件，那都是为了服务用户的。只有这种感觉，让用户觉得好，用起来有感觉的系统才是好的设计更有效率，长效连接，具备解决问题和完成任务的协作性，不管是分布式系统还是响应式这种，它都是通过消息机制互相传递，互相通知。


应该如何协作？通过多个组件的协同效益达到一个最终目的，特别是大型系统或者复杂系统里面，这种效益越发明显。只有当这种高效率而且是长效链接的这样一种模式才对我们的客户提供更好的服务。通过分布式的实时的计算，然后协同效应得出一个问题的答案，给用户的体验相对应就是前面举的例子一样，它让用户觉得自己的整个操作过程很顺遂，而且得到的响应的是非常实时。还有之前在第一章讲的时候就说过，在响应式架构之中，要求是对错误作出响应，有可能是已经出错，但是你还是得告诉客户要响应客户如果说应用程序长时间没有响应，不管他是因为错误的原因还是真的是在做技能，但是可能由于程序的 bug 的问题导致他一直无法得出结论，这不可用。这种不可用不是从系统层面的衡量，一个不可用，而从客户的角度来也不可以。


我记得这已经成为共识了，就是当用户刷新网页超过 3 秒钟，他就会认为网站不好用，这个系统不行，所以说我们提倡在响应式架构模式下面这种架构方法，下面就是不管你出了什么错，一定要对客户进行响应。如果说你说这个系统会死机，那就证明你要么不是响应式架构，要么就是你的使用方法有问题。因为我前面已经分析过了响应式价格式，它如果有一个点有问题，必定会有接替它的点，或者有重启它的点，所以说即使它出了错，第一它会报错。他会告诉监控系统或者也有可能是用户或者他的parent，就是他的父节点都有可能把这种错误传递出去，但是他必须要做出响应，这才是一个符合响应式价格的要求，这也是响应式价格能够给我们带来一个好处。
