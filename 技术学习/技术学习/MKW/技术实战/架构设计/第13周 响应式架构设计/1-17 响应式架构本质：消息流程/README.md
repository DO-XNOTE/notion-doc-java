# 1-17 响应式架构本质：消息流程

我们再来看想要试驾和本质的另外一个特点，就是消息的流程。消息的流程是什么含义呢？消息流程就是可以对业务流程建模或可视化，以此避免限制可伸出性或者弹性。什么意思呢？消息流程它从某个侧面反映了你整个系统要完成的业务逻辑，那在这个业务逻辑当中你就要进行梳理。那知道说哪些地方需要的自身资源多，哪些地方所需要的计算资源少，以此来分配资源，而同时也在这个过程当中梳理出了一条完整的业务线，把消息流传递在这个过程当中得到了一个展现。

通过这种方式，第一是帮你理清了业务逻辑，第二也可以知道在这个业务链路当中，哪些点可能是你的一个未来潜在的瓶颈，那这种瓶颈上你怎么能实现整体系统伸出性和弹性的这样一个支持？这就是说我们消息流程最重要的一个功能，帮你梳理业务链，找到每一个点上可能的资源消耗，或它自身会不会成为一个瓶颈，以此不影响整体系统的这个弹性。


第二个就是消息传递路径要短，但是这个要分两层含义去理解，大家不要看见这个说要短，它就按照最短的路径去算，对，这个肯定是基于事实，而不是基于这个马定律的这样一个偏差的理解，它的本意是什么意思？如果说咱们有多条链路就实现一个目标，那么你相对来说应该在大部分情况都是选择最短单的条路径，对吧？因为从逻辑上来讲，可能如果是多条路径达到相同的目的，理论上你就拿球，这是没太大的关系的。


而在现实事件中，你经过的点越多，出现失败的几率就是不断的放大，对吧？通过概率论，大家知道如果有 n 个节点，那只要通过的节点越多，你这个思维的几率就是越来越大的，所以说这种情况下你是要选择一条最短链路的，因为不管是重新产品本身，还是说从这个效率来讲的话，肯定都是走最短路径来的。


但是这个前提是得满足你的业务规则，这个业务规则前面讲这个链路上它的点它是不能乱的，如果乱的话你这个最短它就失去了意义了。所以说这个东西的最短的前提是保证你的业务逻辑它不受影响，而且能够更高的提高整体系统的这个稳定性和可用性，才去选这点这一条路。


否则大人都知道，只要学过小学的都知道两点之间距离最短，对吧？但是在实际的这个系统工程当中，对，最短的可能并不一定是你怎么说每次都能够选择到的。我们只是说用这个原则来指导我们如何选择一个链路的长短，尽可能的让消息沿着一个方向流动，就是说白了要把这个数据的逻辑的目的定，这个方向一直向你前进，你的流动，这个信息的流动始终朝一个方向去流，并且最终是要被消费者消费的。这个话其实简单的理解的话就是什么意思呢？你的发消息的得到发现只朝一个方向去发消息，就是你所有的不要形成一个来回的这样一个闭环，因为只要你形成一个闭环，就有可能会导致一个死循环。如果说你在这一个大型系统当中，你特别要注意一件，这样是一个事情，你要把你的架构图画出来的目的就在于什么？你要保证这个消息之间不会形成一个死链路。 a 发给b， b 发给c， c 再发给a，这不就死了嘛，就相当于说这事就无解了，对不对？所以说你最终的逻辑执行力要看一下他是不是最终的消费者，这个仙女系列挪到他的有没有在对外广播？他在对外广播下的消息者会不会又回到最初发消息的这个地方？这个往往很体现架构师的这个职责。


坦率地说，与这个小程序架构是不是潜力的相关，这都未必，因为在所有的价格当中，只要你是这种倾向于传递的，不管你是不是小游戏价格，其实你都应该避免这件事情。就是什么意思？在一个大型系统当中，你首先要确定这个 component 之间的这个依赖关系，不要形成一个依赖闭环，在响应式架构当中，因为它都是异步消息传递的方式来驱动的，那 100% 单的要避免这个问题，所以说它提出的原因就是尽可能让消息沿着一个方向越多，把它流到你最终的消费者来，就是我们所谓的数据的目的。


然后我们来看一下一条批处理来发送回置消息，来保证数据流的精。简单来说就是在有些系统里面就是消息传递这个流程过程当中它需要一个消息发送成功的这样一个 ACK 一样的东西。当然这个有点偏向于这个实际的这个技术实现层面。因为其实从小平时架构本身而言的话，它并不对最终做出一个明确的规定，这一点更多是从实践层面的考虑的，一个 best practice 就是最佳实践，因为在有些系统当中，你要确保整个消息发送的成功性，在一些我们叫 missing Critic 的这样一些系统，就是相当于说比较重要的系统，在银行交易或者说一些证券交易当中，可能它也是需要采用这种异步消息，在小公司架构这个时候，但是你对这个消息，因为有些消息可能就是你滚不出去。


至于说有没有人接受成功，我发送成功没有，它可能不是那么在意，但是在有些系统当中它必须得确定说这个消息我发出去之后它是真的到了目的地，但是否被消费，那有另外的检查手段，但是本身消息这个发送他成功了没有？这也是一个问题，对吧？如果说我发了 100 条消息，是不是每一次发一条他就给我一个a、C、k，那就不一定，有些时候他在某些系统，他倾向于说可能 100 条发完一次，我发一个 ACG 这样子的，他就把前面 100 条都确认掉了。当然这个跟你整个你要求回执的时间一定关联性。你比如举个例子，你的发送方说我在 100 毫秒内都没收到一个ACK，那我就会重发这种消息。


说简单一点就是这个发送回置最大的目的就是确认发送本身成功的，我要不要再重发？但是你怎么来做这个回置的发送，其实也是有讲究的，这个更偏向实践层面一点，你只能认为叫这是一个最佳实践，未必在你的系统中一定要实现这种方式只是一个最佳实践的推荐而已。那最后就是前面跟我们第一点讲得比较接近，就是我更推崇使用 DDD 的模式来定义这个业务流程，并保证消费流程与它一致。在 DDD 当中我们首先会对这个叫朵曼也好了，别在各个朵曼之间这个边界，它的消息传递就是我们互相的依赖关系定义好了之后，当你要去完成任何的一个事情的时候，不管是在跨界的还是跨界里，你把这个流程定义好，这就是一个业务流程，就是一个业务逻辑，对吧？然后得要保证你的消息传递它是按照这个方式来完成的。
如果说是进行了一个 crush domain，就是跨界的这种一个调用的话，那你也要遵循这种规则和逻辑。这个跟第一点是相辅相成的，因为这个消息的流程它就是体现了这个业务建模的过程。那我推崇 DDD 的这种方式的，你只要定界了各个domain，就有各个不同的领域的话，对这个消息的传递的方式就跟这个领域之间成一类关系，它应该是一一印证的。不可能说我在系统层面没定义出这样一代，但是你的消息却发过来了，如果这么做， 100% 的要么就是消息搞错了，要么就是你的一代关系定义错了。这二者是相辅相成的，不可能说我没有定义一个跨界的依赖，你却发生了一个跨界的消息。第一，理论上架构这么设计上肯定是出了问题。第二，如果说你发成功了，那他也很好奇，因为理论上，从实践层面来看，我们先不谈架构，谈一下实践。如果说理论上两个不同的领域，就两个不同短面之间它应该是不会有依赖关系，就不会开防火墙。如果你把这个消息能够发出去，那就对你这个系统的内部管控说实话也是有很严重的这个纰漏的，我们不多说这个实践层面的问题，意思就是什么意思呢？就是说你通过 DDD 的这种方式定义了这种跨界的这些依赖关系，或者内部的依赖关系，那么你的消息流的话，肯定是它的框子里面不可能跨出这个框子， ATD 没有画出的线，你的消息流通不过跟它相冲突，有的话那肯定要么是在 TTD 关系问题，要么是你在消息的实现层面的人出了问题。所以说大家可以通过 TTD 这种手段来实现，帮你规定好。
