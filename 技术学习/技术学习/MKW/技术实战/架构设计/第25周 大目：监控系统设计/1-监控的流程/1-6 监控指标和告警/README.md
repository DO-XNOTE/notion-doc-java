通过延迟的大小，我们可以分析应用的性能以及用户的使用体验。


通讯量是指系统当前的流量，通过流量层面的监控，我们可以衡量服务的容量需求。这里的流量是一个广义上的概念。比方对于用 rest API 通信的微服来说，流量通常指的是每秒的 HTTP 请求数。而对于 Redis 来说，流量则指的是每秒请求 Redis 的次数。对于操作系统来说，流量则是指每秒的带宽传输量。


好，接着看错误。它指的是系统发生错误的请求。相关的指标通过错误可以帮助我们排查异常的情况。最后是饱和度。饱和度主要是指我们要监测最能够影响服务状态的受限制的资源。比方有一个 IO 密集的应用，它的性能受内存影响最大，我们应该主要关注系统的内存占用情况。而如果受磁盘影响最大，我们就应该更加关注磁盘的爱物状态，因为当资源占用达到饱和之后，服务的性能就会明显的下降。此外，还可以通过饱和度对系统进行预测，比方预测磁斑 3 天之后就会满了，我们就可以提前扩容。

一般来说，告警有 5 点诉求一是丰富的告警渠道，能够支持邮件、电话、短信等等，前面说过。二是丰富的告警模板，告警的内容可以配置。三是高度的可配置性，比方配置不同类型的人员收到不同的告警。四是告警分级，指的是可以按照问题的严重程度以及紧急程度去推送告警。比方不严重的告警推送邮件，而严重的告警打电话并发送短信。最后一个诉求是易用性，易用性包含了多层的含义，比方告警规则的配置要非常的简单，告警的内容要简明易懂，重复的告警要能够自动精简，不要重复的推送，尽量减少冗余的信息。这是常用的监控指标以及告警层面的 5 大诉求。好，这节课就到这里，谢谢大家。
