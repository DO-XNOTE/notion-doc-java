一是丰富的监控指标，这个很好理解。二是灵活的数据聚合与分析。监控系统要能够对收集到的原始数据进行分析、计算、汇总，从而得到我们关注的数据。比方分析 API 的调用日志，从而计算出最近 5 分钟的错误率、错误次数、成功率等等。再比如分机应用的带宽使用情况，获得带宽的分时，按天的带宽使用波动，从而监测带宽是否发生异常。


第三，监控系统要能够具备功能强劲并且人性化的dashboard。这一点很好理解，因为我们要通过 dashboard 去了解系统的运行情况，提前发现问题。而出现问题之后，我们也需要用 dashboard 去辅助解决问题，所以 dashboard 的功能性以及易用性都是非常重要的。第四，监控系统应当具备灵活的异常检测能力。也要能够通过配置检查数据所存在的问题。比方当 API 错误率达到了20%，就认为 API 是存在异常的，而 20% 的指标可以灵活的配置。

当检测到异常之后，需要推送告警。告警的时候，可以设置灵活的模板。比方对于不同的人群发送不同的邮件。举个例子，比如对于刚刚的错误率超过 20% 这样的一件事情，不同的人关注的点是不一样的。对于 API 负责人，他需要关注错误率以及错误调用相关的机器，所以它的告警模板就可以。比如这样。某 API 的错误率已经超过了百分之多少，持续了多久了？涉及了哪些机器？这样方便 API 的负责人快速进行处理。而对于项目负责人来说，他可能并不关注微观层面的异常，说某个 API 的错误率达到了百分之多少。而更加关注宏观层面的问题。比方系统发生了什么问题，出问题的点是谁负责的。所以它的告警模板就可以是这样的。故障点是某某， API 负责人是某某某。以便于项目负责人去和 API 负责人了解问题的处理进度。

OK。第六，监控系统应当具备丰富的告警渠道。比方邮箱、电话、短信、钉钉、飞书等等。第7，监控系统自身也应该是可扩展的，这样才能满足业务上不同的需求。第8，高可用是指监控系统自身也需要实现高可用，否则如果监控系统自身发生问题，掉链子就会比较尴尬了。最后，故障复盘。当问题修复之后，我们可以针对这次故障进行复盘，做一些知识沉淀，防止以后再犯同样的问题。当然了，复盘并不一定要做成监控系统的一项功能。在监控系统里面，要去做一个模块，专门去记录复盘的细节，这个是没有必要的。复盘的方式可以很灵活，比方开会探讨，编写文档到 wiki 里面。这里只是当故障解决之后，可以做一些总结和沉淀。OK，以上是监控系统必须具备的能力，这节课就到这里，谢谢大家。
