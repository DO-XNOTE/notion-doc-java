一是使用体验无疑是非常差的，我们可能又要在不同的系统里面跳来跳去。

二来监控数据的整合、分析以及聚合会很不方便，因为不同的系统没有打通，会存在数据孤岛，而且即使打通了数据也不一定能够聚合。

因为不同的监控工具使用的存储可能是不一样的。监控工具 a 可能用的是 ElasticSearch，而监控工具 b 可能用的是 open tsdb 这种情况下，要想做数据聚合无疑是非常困难的，再一个学习成本也会非常的高。我们需要掌握各种工具的使用，所以在企业内部一般会采用一款监控系统，把各个维度的监控都给串起来。我们课上采用的是Prometheus，翻译着中文叫普罗米修斯。这样一来，所有维度的监控数据都会汇集到普罗米修斯这一个系统之上，这样就不会存在前面说的这几个问题了。


说明一下，同学们不要误解监控系统这一个阶段就讲 Prometheus 这一款监控工具了。不是这个意思，而是课上我们会以 Prometheus 为主线，各个维度的监控都会集成到 Prometheus 上，或者和 Prometheus 进行打通，通过这种方式去实现全方位的监控。OK，有关 Prometheus 在架构师 1 期的调优阶段已经有过简单的入门了，这里带同学们稍微的复习一下。

这张图组件有点多，我们简单分析一下。首先中间一块儿是 permefius 的 server 端。它是 Prometheus 的核心，负责数据的采集、存储以及提供 API 去查询数据。 retrieval 表示检索 tsdb 表示时序数据库用来存储数据。http server 表示提供 http api 从而方便其他组件和它交互。左下角的 Prometheus targets 用来采集监控指标，并把监控指标暴露出来。


Prometheus server 端会从各种 exporter 里面获取监控数据，并且这种获取是拉取的方式。 server 单拉取各种 export 的数据。左边的 PushGateway 主要有两个作用，一是有些应用它并不是长期运行的，而是运行完就停止了。如果我们也想对这种应用进行监控，就可以利用 push Gateway 让你短期运行的应用把监控数据推送到 PushGateway 上，然后 server 单再去拉取 PushGateway 上面的监控数据。这是第一个作用。


第二个作用是如果你的 exporter 和 permission server 端不在一个局域网，或者是因为防火墙的原因，可能导致 server 单没有办法直接拉取到 exporter 上面的交控数据。这个时候也可以让各个 exporter 把数据汇总到 PushGateway 上去，只要 PushGateway 和 server 进行通信就 OK 了。


okay 上面的 service discovery 表示服务发现，用来帮助 server 端找到其他组件。比方找到 PushGateway，找到exporters。 permefius 支持多种服务发现方式，比方利用 K8S 的服务发现机制，利用基于文件的复发现机制等等。右上角 alert manager 可以用来实现告警通知。 AlertManager 它支持多种渠道。


最后右下角是 Prometheus 的可视化，你可以使用 web UI 或者是 Grafana 等等工具去实现数据的可视化。OK，这是 Prometheus 的架构。从中可以发现， Prometheus 的组件虽然非常多，但是每个组件的职责还是比较清晰的，而且协作起来也不复杂。


OK，复习完 Prometheus 的架构之后，来搭建一个 Prometheus 的 server 端，把核心先搭建起来。
