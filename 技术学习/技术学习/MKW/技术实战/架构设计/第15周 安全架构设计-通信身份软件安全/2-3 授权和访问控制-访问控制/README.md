知道你需要知道的，那反过来就意味着你不知道什么。这句话好绕口，那我们来举个例子，比如飞扬老师去登录我们企业的这个系统，后台有个叫 back office 后台管理系统，当我登录进去以后，我可以看到整个企业里面所有的应用的流转关系，以及哪些地方出现了熔断，哪些地方能正常沟通，哪些地方出现了降级，这个效果不错，但如果我想看一看奥罗克数据库的 AWR 报告，我想了解一下我们的 AI 的模型的训练情况，怎么都看不到呢？那我就提了个功能需求给我们的什么后台系统管理员说能不能帮忙开发一下这两个基本功能啊？然后他给我个回复是什么？阜阳老师，因为你是架构师角色，所以你看不到。


如果你是 DBA 角色，你会看到什么？什么？什么？这不就是 AWR 我们的实际的工作负载压力报告嘛。如果你是什么数据科学家，你会看到怎么这不就是最经典的Tableau，什么数据训练和预测模型的登录界面吗？噢，原来如此。我是什么不在此山中，所以看不到此山的风景，那一样的道理，我们很多的这个外围登录系统，当你用错误的用户或用错误的密码登录的时候，他不会告诉你用户不存在，也不会告诉你密码不正确，而是给你一个奇奇怪怪的返回，那这个返回的暗示什么？暗示的其实是你权限不够，或者是你什么，或者是你账号登录有误，但不会具体提示你出错在什么地方，这样有效地杜绝了黑客的攻击的可能性。


所以对于我们，特别是后台系统，我都建议所有的这个设计方法都应该知其所需。也就是按照你需要知道的内容展现给你，但并不要告诉你你不知道哪些内容，这样可以减少黑客什么登录到后台系统，进行什么层层攻击的可能性。好，第二条访问控制原则叫最小权限原则。前面其实已经跟大家分享，默认安全管理员是不能访问任何资源的，他也不能授权自己来访问资源。这就像什么军队的一个虎符调用一样，各地的藩王是不能直接调用各地的军队的，是吧？虽然各地的军队的将帅都驻扎在各个藩地，但是藩王不能直接调用，他应该怎么样？由皇帝下一道谕旨。某某藩王，你调用本地军队去对抗隔壁的什么什么国家，那这个时候藩王会拿到一片虎符，他这片虎符去跟当地驻军的这个元帅的虎符对起来。当两个虎符对起来的时候，就可以真正调用军队了，所以通过这种方法防止驻军的领袖人调用军队，也防止藩王，是吧？能够起兵反叛。


好，这就是我们平时企业里面常用的最小权限，多人职责合并，或者是管理员跟什么，跟安全员互相配合，才能够进行资源的完全掌控。好，那除了知其所需，最小权限我们还有一个很大的安全准则，就是默认拒绝。所谓默认拒绝就是什么？就是要通过一个白名单来进行管理，而不是黑名单。所有不在白名单管理范围之内的都是默认就是黑名单。比如说我们用户访问里面的 ACL 怎么设定的？如果你没有设定相关的用户或组，我默认都是拒绝，这是一种方法，或者防火墙规则里面要设计成全部默认，否定全部拒绝。


只有以下几种，原地址、原端口、目标地址、目标端口的访问才能够正常进行。那除此以外，我们的 job token 进行什么 API 的授权的时候，有些企业是规定拿到 job token 时授权是为空，也是不做任何权限的复制，没有任何能力，除非他在 payload 里的 scope 写清楚我是某某服务的这个什么 read 或某某服务的 write 才能够对这个服务进行 get 或者 post 请求。


那同样道理，很多的云平台，我们是当用户注册完，登录完以后，我们会 assign 一个普通的role，这个 role 是没有任何访问权限，所以你登录云平台是看到是悬空一片。那这个时候当你 assign 了一个，比如说 project reader 项目阅读权限，你就可以看到项目里面的各种资源，那这就是默认拒绝。

比如说我在网络的云盘，或者是什么百度知道里面发布了一条消息，这条消息我可以发布给或者共享给我的好伙伴们。那这个共享给谁、发布给谁全部是由我个人来掌控的，我企事业单位是不会统一掌控。


那就是 d a、 C 自主访问控制，这种访问控制适合于个人资料的，不适合于什么支付信息，不适合于公司内部敏感信息，所以通常敏感信息用下面两种访问控制，一种是 MIC m，就是 Mandatory 强制访问控制，其实用到的场所非常多。那最经典就是什么？就是我们 link 4 里面什么 04 里面的那种，比如像 change mode 来进行什么文件的权限控制，唉，是不是付杨老师打错了，哪里有996，也没有857，是不是？那正常应该是多少啊？应该是这种，是吧？千金 mode 754。唉，这才是正常的嘛。


那 754 就是当前文件所属用户是可以可读可写可执行，但是组内用户是只能读，只能执行，不能改变，那外面用户就只能读了，对不对？这就是一些常见的一些强制访问控制，这是在集中的，在整个操作系统，或者在整个企业内部总控所有资源的掌控，它的授权，对不对？那还有比较经典的另外一种方法，也是更加推荐的方法，尤其是现在，是吧？现在这种云平台时代常用的方法叫角色防控制，不是根据文件，不是根据一个 SL 列表，而是根据你的角色来授权。比如我们在数据中心里面，一个用户是在某一个组底下，这个组就会自动有哪些权限，一旦你要访问这些资源，你直接只要加到对应的组就可以了。


那在语音平台呢？我们不叫组，我们叫什么？叫角色？一个角色具有哪些权限？我们要访问这些权限对应的资源，我们只要把这个用户加到对应的角色里面就可以了，这也是最常见的一种访问控制方式，也是我们最主推的一种访问控制方式。好，除了这种访问授权的控制分类以外，还有一种分类方法叫什么？叫做集中式和分散式的访问控制分类。
