第一， where 子句里面用不到的字段。这一点也不一定。比方覆盖作用就是一个反例。假设现在有条语句 select a，b， c from t where a 等于叉叉叉， and b 等于 y y y。理论上我们应该是创建一个组合索引，作用在a、 b 两个字段上面。但是我们现在只查询了a、b、 c 三个字段，如果创建的索引只撞在 a 和 b 两个字段上，查询 c 字段的时候，还是要回到表数据查询。


覆盖。索引是我创建一个索引，索引包含了查询结果里面拥有的所有字段，也就是a、b、 c 这样的。在查询的时候只要到索引文件里面查询就 OK 了，不需要回到表数据文件里面查询，从而提升性能。也就是我们需要创建一个作用在a、b、 c 三个字段上面的索引，这一点是和原则相违背的。


第二点是表的数据非常少，这种场景下一般是不建议创建索引的，因为创建索引会带来增删改的开销，而数据量非常小的情况下，创建索引带来的查询性能提升并不明显，所以就不需要多此一举了。第三，有大量的数据重复。

选择性低的字段不适合创建索引。比方性别这样的字段，它的选择性非常低，不是男女或者是未知这种长期上就不适合创建索引。索引的选择性越高，查询效率也会越好，因为可以在查找的时候过滤更多的黄。最后，频繁跟进的字段。如果创建索引，要考虑索引维护的开销。索引的本质其实是一种以空间换时间的手段，创建索引之后会带来增删改时候的维护开销。所以，如果你的字段更新非常频繁，就需要注意了。如果可以，建议不要在这个字段上面创建索引。好，这是索引相关的几个结论。这节课就到这里，谢谢大家。
