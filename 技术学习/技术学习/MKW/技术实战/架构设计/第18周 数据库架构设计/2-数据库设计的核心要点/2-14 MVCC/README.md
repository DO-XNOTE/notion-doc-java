它可以使得支持行锁的数据库引擎不再单纯地使用行锁进行数据库的并发控制，而是把数据库的行锁和行的多个版本结合起来。这样只需要很小的开销，就可以实现非锁定读了，从而大大提高数据库系统的并发性能。前面讲过，对于普通的 select 语句，也就是没有加 for update、 for share、 lock in share mode 这种 select 语句，都是非锁定读的行。这是 MVCC 的概念。多数数据库都是支持 MVCC 机制的。实现的原理细节上可能会有一定差异，但是总体来说是类似的。对于 Innodb 存储引擎，它的 MVCC 是这样设计的，它会对每行数据添加 3 个隐藏字段。

一是 DB_TRX_ID 占用 6 个字节，用来记录最近一次对本记录插入或者修改的事物ID。对于删除操作， Innodb 也会视为是一个 update 的操作，只是会额外的修改删除标记。这个删除标记叫做 DELETE BIT。

第二个隐藏字段是 DB_ROLL_PTR 占用 7 个字节，它会指向当前行的 undo 信息，也叫回滚指针。

第三是  DB_ROW_ID 占用 6 个字节，这是一个随着新行插入而单调递增的行ID。如果当前表没有聚簇索引的话，innodb 会直接使用这个字段作为聚簇索引。而如果存在聚簇索引的话，这个字段则不作为聚簇索引。MVCC 和 UID 这个字段关系不大。这里我们顺便普及一下。

一是快照读。普通的 select 语句会使用快照读 Repeatable read 这种隔离级别。事物开始之后，第一条 select 语句会创建一个快照，并读取满足可见性的数据。之后这个事物会一直用快照直到事物·结束。这也是 repeatable read 能够解决不可重复读的本质原因。


对于 read committed 的级别，事务开始之后，每条 select 的语句都会重置快照，也会重新创建一个新的快照，并去读取当前满足可见性的数据。所以 read committed 它没有解决不可重复读的问题。
