大家好，我是大木。这节课来探讨数据库的概念结构模型在数据库架构设计的基本思路。这一节，我们已经知道数据库概念结构是什么了。回顾一下。概念结构设计是指通过对用户需求进行综合归纳和抽象，从而形成独立于具体数据库管理系统的概念模型。也就是把需求分析阶段得到的应用需求抽象成概念模型。概念模型是连接现实世界和信息世界的桥梁。一个好的概念模型应该具备这么几点，首先它能够真实充分的反映现实事件。比方概念模型能够非常好的表述用户对数据的各种要求，并且表述事物之间的联系。

第二，要易于理解，以便于用它和不熟悉业务甚至不懂技术的小伙伴去交流意见。第三，要易于修改。实际项目中，我们常常要对概念模型进行修改和增补，所以易于修改也是非常重要的。概念结构设计是整个数据库设计的关键，所以可想而知它的重要性。OK，下面来普及概念结构设计里面的一些基本概念。首先是实体。实体指的是客观存在并可以相互区别的事物。实体可以是具体的人、事物，也可以是抽象的概念。第二，属性。属性。指的是实体所具有的某一个特性。一个实体可以由若干个属性来刻画。


第三，关键字。唯一。标识。实体的属性集称为关键字，也叫马关键字。它具有唯一性和最小性这样的两个特性。第四，域。

er 模型是一种比较流行的概念模型，他用 er 图来描述现实事件。玩法是这样的，先去分析实体之间的联系，之后这句话 er 图去表示实体之间的联系。好，下面来看一下实体机之间的联系有哪些。两个实体之间的联系有三种一对一联系、一对多联系以及多对多联系。先来看一对一联系。如果对于实体积 a 中的每一个实体来说，实体积 b 中至多会有一个实体与之联系，反之亦然。据说实体积 a 和实体积 b 之间有一对一的联系。记作一对一。比方，一个班级只有一个正班长，而一个正班长也只能在一个班中任职。据说班级和班长之间具有一对一的联系。

基数约束是对实体之间一对多以及多对多联系的细化。参与联系的每一个实体类型会用奇基数约束来说明实体类型中的任何一个实体可以在联系中出现的最少次数和最多次数。基数约束会用 Min 点点 Max 的形式去表示， Min 大于等于0， Min 小于等于 x 星号的话，表示无穷大。其中 Min 等于 1 的约束叫做强制参与约束。也就是被施加基数约束的实体类型中的每一个实体都需要参与联系，而 Min 等于 0 的约束则是非强制约束。


比方，在学生和学生证之间的联系中，一个学生必须拥有一本学生证，只能属于一个学生。因此学生的基数约束是 1- 1，学生证的基数约束也是 1- 1。而学生和课程之间的联系中，一个学生可以报名若干多门课程，报名某一门课程的学生可以是0，也可以是无穷大。因此学生实体的基础约束是 10- 20，而课等的基础约束是 0 到无穷大。

首先是合并 er 图，从而生成一个初步 er 图。之后我们会在初步 er 图的基础上进行修改和重构，并去消除掉不必要的冗余，从而生成一个基本 er 图。在生成初步 er 图阶段，由于各个子系统要解决的问题不同，设计人员也可能不同，所以很可能会导致各个子系统之间的 er 图之间会存在不一致的地方，也就是冲突。合并 er 图并不是简单地把多个 er 图拼到一起，你得想办法消除掉各个 er 图里面的冲突，从而形成一个能够为整个系统所有用户共同理解和接受的一个统一的概念模型。
