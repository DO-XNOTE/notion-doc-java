这些**原则可以指导我们去实现高性能数据库架构设计**。OK，首先是表结构设计原则。


在表结构设计方面，可以遵循这么几个原则。首先是注释原则。注释原则是数，我们设计表结构的时候，所有的表以及所有的字段都必须添加注释，以便于后期维护。比方 test Innodb 表，他就没有添加注释。我们可以写一下，这个字段是ID，这个字段的注释是名称，表也要加上注释，比如 Innodb 引擎测试表execute，这样后期维护的时候就会比较方便。


这个注释原则虽然简单，但是有很多企业很多的项目，它其实都没有去遵循，导致后面项目维护起来会越来越困难，很多时候可能只能看代码，甚至靠猜，这一点我相信有很多有精力的小伙伴会深有感触，对吧？这里我们把注释放到表结构设计原则的第一位，用来强调一下，很多时候规范是非常重要的。

第二，表数据量原则。当单表数据量达到 800 万的时候，可以考虑分库分表，这里的 800 万是一个经验之谈。如果你的项目没有分库分表的计划，可以把一些不常用的数据通过归档的方式存放到其他的数据仓库里面去，从而减少我们 MySQL 的数据量。


第三，表字段原则。首先不保存大字段，不要在数据库里面存储图片、文件等等大字段数据，否则话对数据库的压力， IO 对应用的序列化、反序列化等等都是一个挑战。一般来说，建议把图片、文件等等存放到比方分布式文件系统里面去，这一点是比较好理解的。

最后是数据库架构设计的原则，我们要选择合适的数据库架构和方案，在要考虑的问题很多是选用一主一从、一主多从还是多主的方案？采用读写分离双写还是多写分固分表？是借助中间件还是直接在应用层上去做分固分表的规则又是什么？高可用的方案又该怎么设计？是使用 3M XC 还是MGR？如何实现容灾？这些问题我们都要去考虑，并去选择合适的架构和方案。这些内容在咱们这一阶段都会详细的探讨到。


总之，最合适的架构其实是结合业务场景、软件特性以及实施成本的一个平衡，我们要综合各方面的因素去选择合适的技术解决问题。实际项目中这节课探讨的所有原则都可以活学活用，在必要的情况下甚至可以违反。因为原则只是一个指导，实际项目里面遇到的情况会复杂很多，合理采用反模式往往会有意想不到的效果。好，这节课就到这里，谢谢大家。
