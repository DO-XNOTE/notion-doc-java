列出了不同的工种所关注的点。比方，对于开发来，它会关心一个数据库是不是容易维护，性能是不是好，是不是足够稳定，是不是足够易用，是不是容易扩展，是不是足够安全。而对于d、b、 a 来说，它会更加关注数据库的可维护性、性能稳定性、可扩展性以及安全性。而对于采购来，它可能只关注采购的成本。比方数据库是不是收费，搭建数据库需要多少台机器，需要多少带宽，使用多少内存和磁盘。而至于使用层面的事情，采购它可能完全不care。

PPT 下面来分享一下大木所在团队的数据库技术选型，给大家一个参考。首先是关系型数据库，我们只用了MySQL。 MySQL 基本上是互联网应用必备的一个数据库了，在我们内部的是 TDDL 加 DRDS，不过本质还是基于 MySQL 去做的。大家可以把这两个玩意儿理解成是为 MySQL 实现了高可用以及分库分表一堆东西的一个整合型产品。

我们的项目关键型数据库同样也使用了 Postgres SQL，不过只是应对地图以及 GIS 等等场景， postgresSQL 在这一方面做得非常的成熟。虽然其他的数据库，比方MySQL、 elastic search、 MongoDB 等等也可以实现继承，但是使用起来都没有 postgres SQL 方便，同时功能也没有 postgrass SQL 强大，所以我们在这一块儿使用了 postgrass SQL。


第二块是 key value 存储。我们使用的是Tair，它的功能类似于Redis。本质上的 Tair 其实是多种 value 存储的封装。在 Tair **内部，集成了rdb  、mdb 、ldb  等等存储引擎**。其中 rdb 是基于 Redis 做的，mdb 是基于 Memcached 做的，而 LDB 是基于 LevelDB 做的。集成这些引擎之后，会对外提供统一的API，从而方便程序员操作。同学们可以把 tire 理解成是一个集成了多种 value 存储的一个多合一产品。我们使用的时候，只要使用统一的API，比如 get put，就会把数据存到比如Redis、memory、 cache 等等的存储引擎里面去。


当然了，除此外项目还使用到了一些自研的存储和分析系统，比方 opensearch、ODPS 等等。这是我们团队的数据库基础选型。最后做一个预报，有关技术选型在咱们课程的第 19 周还会专门探讨，到时候内容将会更加的系统科学。这节课就到这里，谢谢大家。
