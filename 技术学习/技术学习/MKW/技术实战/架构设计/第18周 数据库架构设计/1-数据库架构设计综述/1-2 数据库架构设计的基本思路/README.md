数据库架构设计的基本思路大概是这样的整个过程包括了需求分析、概念结构设计、逻辑结构设计、物理结构设计、数据库实施以及数据库运行和维护 6 个阶段。


当然了，在开始设计之前，还可以做一些准备工作。比方选定参加设计的人员，包括了系统分析人员、数据库设计人员、应用开发人员、数据库管理员以及用户代表等等。再一个，如果应用的功能比较复杂，还可以挑选好数据库设计的工具，从而提高数据库设计的质量，并减少设计的工作量。


目前数据库设计的工具有很多，例如功能超级强大，市面上也超级流行的 power designer，又比如 MySQL 自带的 MySQL Google bench，再比如开源的数据库建模工具 PD man 等等。这些工具都可以很好地满足我们的需要，并且各有特点，同学们可以结合团队的喜好以及熟悉程度去选择。当然了，如果项目比较简单，你不用工具也是 OK 的。这是一些准备工作。好，下面我们来过一下每一个阶段要做的工作。

需求分析阶段是整个设计过程的一个基础。需求分析做的是不是准确以及是不是充分，它决定了数据库设计的速度和质量，而且在一些极端场景下，如果你的需求分析做得不好，可能会导致整个数据库设计的返工，甚至是重做。需求分析阶段是整个数据库设计里面最困难也最耗时的一步。在互联网项目里面，我们的需求往往是非常不明确的，同时需求的变更又非常的快，这一步就更加困难了。所以可想而知需求分期阶段它的一个重要性，对吧。在需求分析阶段，大船可以分成四步明确目标、调查需求、分析需求以及确认需求。

我们做需求分析，它的目标是通过分析现实世界要处理的一个对象，比如组织、机构、部门、企业等等，并充分的了解原系统的工作情况，明确用户的各种需求，在此基础之上去确定新系统的功能。通过调查，我们应该获得用户对数据的几点要求。首先是信息要求，也就是用户需要从数据库里面获得的信息的内容以及性质。


第二，数据要求，也就是在数据库里面要存储哪些数据。第三，处理的要求，用户要完成的数据处理的相关功能，还有安全性要求等等。OK，这是需求分析目标。再来看一下调查用户需求。

在调查用户需求的时候，可以这样首先调查组织机构的情况，包括了解组织的部门组成情况，各个部门的职责等等。这是因为在做数据库架构设计的时候，可能会涉及到多个部门的协作，因此我们需要知道一次架构设计需要涉及到哪些部门，每个部门做什么的。


第二，调查各个部门的业务活动。包括了解各个部门输入的是什么数据，输出的是什么数据，怎么样去加工数据的，输出数据的时候输出的格式是怎样的。这一步是调查的重点。接着，在熟悉业务活动的基础之上，协助用户明确对新系统的各种要求，包括信息要求、处理要求、安全性要求等等。


第四，确定新系统或者新业务的边界。在这一步，会对前面调查的结果进行初步分析，确定哪些功能由计算机完成，哪些功能由人工去完成。由计算机去完成的功能就是新系统应该去实现的功能了，对吧？好。在调查的过程中，我们可以根据不同的问题和条件去使用不同的调查方法。

常用的调查方法有这么些一是开调查会，通过和用户座谈去了解业务活动的情况以及用户的需求。第二，跟班作业，通过亲身参加业务工作去了解业务的活动情况，请专人介绍，比方请外部顾问或者是询问熟悉业务的人。第四，你还可以设计调查表，让用户去填写。如果你的调查表设计的比较合理，这种方式是非常有效的，而且成本很低。最后，可以查阅记录，也就是查阅和原系统有关的一些数据记录，需要去了解用户的需求。在调查用户需求的时候，往往需要同时采用多种方法，但是不管使用哪一种调查方法，都必须要有用户的积极参与和配合。用户的参与和配合是非常重要的，否则很可能会偏离用户本身的一个需求。

所谓数据字典，就是指有关数据的描述。

[https://baike.baidu.com/item/数据字典](https://baike.baidu.com/item/%E6%95%B0%E6%8D%AE%E5%AD%97%E5%85%B8)

可以认为，数据字典是数据的元数据，这个数据字典一般会在需求分析阶段建立出来，并且在数据库设计的过程中不断地去修改完善。有关数据字典在百度百科有非常详细的描述，这里强烈建议同学们看一看，特别是这一段。


**数据字典会包括数据项、数据结构、数据流、数据存储以及处理过程 5 个部分。**其中数据项是数据的最小组成单位，若干个数据项就可以组成一个数据结构。数据流是数据结构在系统内部传输的路径。数据存储是指数据结构保存的地方，记录了数据流的来源或者是去向的位置。最后处理过程是数据的处理逻辑，还是很好理解的好。这是需求分析阶段。需求分析阶段完成之后，就到了概念结构设计阶段。
