大家好，我是大木。这几个来探讨。 MySQL 复制的方式有很多，比如异步复制，半同步复制，GTID 复制等等。稍后我们都会探讨到这几个来探讨。


第一种方式异步复制。异步复制是 MySQL 默认采用的方式。所谓的异步是主节点写入 binlog 文件之后就会返回客户端，而不去考虑 Binlog 是否已经完整的同步到从界点，是不是已经完整的存放到从界点的 relay log 中去了。它的优点是性能好，没有阻塞，毕竟是异步的。但缺点是如果某个时刻主节点发生了宕机，而主节点上已经提交的事务还没有同步到从节点，如果强行把从节点晋升成为主节点，就会导致新的主节点数据不完整，也会丢数据。OK，这是异步复制的优缺点。下面话不多说，我们来搭建一个异步复制。


这里我准备了一篇文章，里面列出了 MySQL 复制的各种方式的步骤，我们不妨参考这个文章去操作。首先在这里列出了 MySQL 各种复制方式下的配置，同学们可以看一下。我准备了两台服务器， 235 是主节点， 236 是从节点。那一会儿在演示的时候，同学们可以关注这里的主节名。 master 是主节点， slave 是从接点。OK，下面来看一看异步复制是怎么玩的。


搭建 MySQL 复制需要满足这么几点条件一是确保主从节点操作系统的版本与位数一致，特别是位数里标一个节点是 64 位，保持一致。第二，要确保主从节点的 MySQL 版本一致。顺便提一下 12 两点要求，即使不一致，也不代表搭建会失败，有时候还是可以搭建成功的。不过建议大家尽量养成习惯，保持操作系统的版本以及位数，还有 MySQL 的版本一致，不要给自己挖坑，OK？第三，确保主库已经开启了Binlog。当然了，对于 MySQL 8. 0 默认就是开启的。


第四，建议从库也开启Binlog。并且建议开启 log slave updates 的参数，方便后期扩展架构。这是什么意思？我们来看一下 lock slave updates 的作用。是否要将从主节点上获取的数据变更信息记录到从节点的 binlog 文件里面去。


这个配置对于级联复制的扩展比较实用，比方原先有两个节点， a 和b， a 是主节点， b 是从节点。后来我们想要把它扩展成a、b、 c 的级联复制架构。 a 是主节点， b 是中间节点， c 是从节点。也就是 a 是 b 的主节点， b 是 a 的从节点， b 是 c 的主节点， c 是 b 的存节点。如果 b 节点没有开启 log slave updates， b 节点的 binlog 就不会记录从 a 节点上获取的数据变更信息。这样 c 节点就没有办法完整的同步到 b 节点的数据。


顺便提一下，如果你使用的是 gtid 复制模式，当然了， gtid 复制我们还没有讲到，后面会讲到对于 5. 6 的话，这个参数必须要开启，否则的话会报错。而对 5. 7 则可以不开启。实际项目建议开启这个参数。OK，接着看。第五，确保主从节点的 server ID 不重复。这一点非常重要，一定要保证 server ID 的唯一，否则搭建就会失败。最后，可以根据项目的需要选择合适的 Binlog 格式。我们课上使用的是 MySQL 8. 0 默认的row，当然了，你也可以使用 statement 或者是mixed。做好这些必要条件之后，来做几点准备工作。首先，编辑主节点的 MySQL 配置文件。 copy 主节点编辑 i 进入编辑模式，添加这一段粘贴。其中我们把主节点的 server ID 设成了 1 log_bin，用来开启主节点的 bin log，并且指定 log 的文件名前缀是 mysql blog。当然了，这个配置也可以省略。

[https://dev.mysql.com/doc/refman/8.0/en/binlog-replication-configuration-overview.html](https://dev.mysql.com/doc/refman/8.0/en/binlog-replication-configuration-overview.html)

MySQL 8. 0，默认开启 Binlog 的同时省略 Binlog 的文件名前缀就是Binlog。okay。我们设置了 Binlog 的格式是 row 保存。接着修改，从界面配置，找到Mysqd，复制粘贴。这里我们设置从界点的 server ID 是。 2 再次强调，每一个节点都应该拥有唯一的 server ID。开启 Binlog 设置 Binlog 格式，开启 log slave updates。刚刚已经讲过了。最后开启了 relay log recovery。配置的作用是如果 relay log 损坏是不是要自动修复。建议开始好保存。
