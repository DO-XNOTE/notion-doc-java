多元复制是 MySQL 5.7 引入的，它的玩法大概是这样的将多个主节点指向到同一个从节点，从而实现数据的汇聚。这里的主节点是完全不相关的数据库节点，节点之间不会有任何的数据同步，不是前面讲的多主。可以这样认为主一是微服 a 的数据库主，二是微服 b 的数据库主三是微服 c 的数据库里面存的表以及数据完全不一样。

主要有这么几个一是数据分析团队可能要分析多个微服务的数据，我们可以使用多元复制，把多个微服务的数据复制到同一个数据库里面去，这样分析起来就会方便很多。


二是用来做数据备份。随着微服的深入人心，一个大型的项目往往会有 n 多个微服务， n 多个数据库实例。以前我们只能为每一个数据库实例创建一个从实例，从而实现数据备份。而现在，我们可以把多个数据库实例同步到一个数据库实例进行备份，这样可以减少维护的成本以及运维的成本。


OK，下面来看看多元复制怎么搭建。首先需要注意的是，多元复制的场景下，多个主库的名称不能相同，否则会发生覆盖。也就是如果你的主库 1 存在 test replication 库，主库 2 不能存在同名字的库。确保这一点之后，就可以去配置了。首先在从接点添加这两个配置，这两个参数是用来指定在哪里存储同步信息的，它的值可以是 table 或者是file。

如果配置成table，会把同步信息分别存储到 MySQL 库的 slave master info 表里面去，以及 slave relay log info 表里面。而如果配置成file，则会把同步信息存放到 master info 以及 relay log info 这两个文件里面去。不过基于文件的方式，如果服务器意外关闭，而主从之间的同步信息还没有来得及更新，就会导致数据丢失。所以建议把两个参数都配置成table。
