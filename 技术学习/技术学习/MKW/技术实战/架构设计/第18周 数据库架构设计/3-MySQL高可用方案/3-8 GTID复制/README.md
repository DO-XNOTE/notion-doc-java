接着在每一台机器执行 GTID mod。 of permissive。 GTID Mod 有 4 种取值是 of off permissive、 on permissive 以及on。默认的是off。 off 是说在你设置 GTID mod 等于 off 之后执行的新事物。是非 GTID 的。从节点也只会接受不带 GTID 的事物，如果从节点接收的事物是带有 GTID 的，就会报错。

第二是 off permissive，它指的是在设置成 off permissive 之后，新事物是非 GTID 的。从节点可以接受带 GTID 的事物，也可以接受不带 GTID 的事物。 on permissive 则是说新事物是 GTID 的。从这点既接受带 GTID 的事物，也可以接受不带 GTID 的事物。


最后是on，指的是新事物是 GTID 的，从这点也只接受带 GTID 的事物。如果你传递的事物不带 GTID，就会报错。 GTID mode 设置的时候是有顺序的。比方当前 GTID mod 的值是off，它得先转成 off permissive，再转成 own permissive，最后才能转成on。同理，如果现在是on，它要先设置成 on permissive，再设置成 off permissive，最后才能设置成off。你们的跳跃比方从 off 直接设置on，这样是不行的。

好，回来，我们首先把它设置成 of promising，这样之后的事物依然是非 GTID 的。但是从这点，它既能够接受带 GTID 的事物，也可以接受不带 GTID 的事物，特别，粘贴。再之后设置成 on permission，让新事物以 GTID 的方式执行。从这点，既接受带 GTID 的事物也可以接受不带 GTID 的事物，copy，粘贴，粘贴。接着使用这条命令检查不带 GTID 的事物是不是已经复制完毕了。 copy 粘贴在重借点也粘贴。

从中可以看到每一个事物里面都会包含一个 gtid。之后当这一段 binlog 传送到从接点并存到 raley log 之后，从这点会自动先去执行 set GTID next 等于叉叉叉，也就是这两sql。


GTID next 指的是下一个 GTID 值，也就是下一个事物是什么。 SQL thread 读取到 GTID 之后，会对比一下本地的 Binlog 里面是不是已经有 GTID 了。如果已经有，说明 GTID 的事务已经执行过了，直接忽略。如果没有，从节点就会执行这个事物并记录到 Binlog 里面去。这里我们发现了 GTID，复制它的从接点也是必须记录 Binlog 的。这是 GTID 复制的原理。简单总结一下，


其实就三句话

一是 gtid 用来标识一个事物。

二是从库会对比传过来的 gtid 是不是已经在 Binlog 里面了，如果已经在就直接放弃，不在就去执行。

三是使用 GTID 复制之后，我们就不需要知道 binlog 文件名以及 position 是什么了。
