它最大的特点是基于 paxos 协议，解决了异步复制以及半同步复制等方式存在的数据一致性的问题，所以非常适合金融行业使用。我们知道金融行业它对数据一致性的要求是非常高的。


OK，下面来探讨 MGR 的原理。为了讲解清晰，我们不妨先来复习一下前面的异步复制以及半同步复制。

异步复制是主库执行产生 Binlog 并异步复制到从节点之后，主节点就提交了主节点，他不会关心从节点是否已经收到了主节点的blog，哪怕从节点接收失败，主节点也不会感知。于是就会发生数据不一致的问题。比方主节点和从节点的数据有延迟。另外，如果主节点发生宕机，并且 Binlog 还没有来得及被从节点接收，之后，我们用主从切换的程序把某一个从节点晋升成主节点。这个时候新的主节点和老的主节点的数据也是不一致的，会丢失一部分数据。即使你用 MHA 去尽量保证数据的一致性，特定场景下还是可能存在数据不一致的情况。比方某个时刻主键点宕机非常彻底， ssh 都没有办法连接上了。同时主节点上面的 binlog 还没来得及被从接点接收，这个时候依然会丢数据，还是存在数据不一致的现象。


由于异步复制的这种问题， MySQL 以插件的方式支持了半同步复制。有两种模式。我们这里聊的是 MySQL5.7 引入的新模式，也就是无数据丢失的半同步复制。这种模式下，会先在本地产生Binlog，然后传给从库存库，收到主库的 Binlog 之后会写到 relay log 里面去。在通知主库说自己已经收到 binlog 了，主库会等到至少一个从库的 ACK 消息之后才会提交。半段不复制。它能够保证至少一个从节点收到 relay log。这种情况下，如果你用 MHA 去实现主从自动切换，确实不会有数据不一致的情况，但如果你用其他的工具去切换，比方自己弄一个 

HA Proxy 虚拟 IP 漂移，没有 MHA 那种 binlog 对比以及恢复的机制，在存在多个重叠点的时候依然可能丢数据。还是会存在主从数据不一致的情况，对吧。

在 MGR 里面，若干个数据库实例会组成一个复制组，组里面各个节点会维护各自的数据副本，彼此之间依靠分布式一致性协议通信。分布式一致性协议其实是 Paxos 协议的变体事物，必须经过组里面大多数节点的决议并通过才能成功提交。通过这种方式去实现了数据的一致性。比方图里面 member 123 组成一个复制组。 Consensus 层是一致性协议层，在事务提交过程中，要求一半以上的节点决议通过，才允许后面的操作。客户端发起一个新的事务，会先在本地执行，执行完成之后会将产生的复制写集广播到其他成员上去。这里的复制写集参考了前面 PXC 的概念。之后每个节点要做冲突校验，如果检测成功，事物可以提交，否则就回滚。这里的冲突校验和前面的 PXC 也是类似的。从这个图我们可以发现MGR，它是先在本地执行之后把事物广播并去验证冲突。


本地验证通过之后才会产生binlog。其他节点验证通过之后才会将事物写到 relay log，如果验证失败的话，就直接回滚事务了，不会产生 relay log。而半同步复制则是先有 Binlog 再传给其他节点的，这是一个比较大的不同。OK，了解 MGR 的原理之后，

首先MGR 会对组内成员的 MySQL 版本进行排序，选择最低版本作为主节点。实际项目里面，我们应该尽量的保证所有的节点版本一致，这样维护起来更加的方便。因为不同版本的MySQL，它的特性是不一样的，这也是为什么要选择最低版本作为主节点的一个原因。你选高版本作为主节点，可能有一部分特性。低版本的MySQL，它不一定能支持。

OK，尽量保持版本一致，这是一个原则。接着看，如果有多个节点同时使用最低版本，则去考虑每个成员的权重。你可以使用 group replication member weights 去指定。权重值是 0- 100，默认是50。会选择权重最大的节点作为主节点。如果有多个节点的权重值同时最大，则通过 server u i d 系统变量去计算。它会用字典排序取 UUID 最小的成员作为主节点，那 server UUID 的话，可以通过 show variables like UUID 获得。OK，这是单种模式。

这里我准备了 3 台全新安装的服务器。主机名规划是这样的。最后来做几点准备工作。首先要配置主机名，这是因为 MGR 成员之间需要通过主机名通信。使用 hostnamectl set-hostname 就可以配置主机名了。配置完主机名之后，还要修改每一台机器的 /etc/hosts，在里面添加这些内容，这样机器之间才可以通过主机名进行通信。配置完成之后，需要退出一下 ssh 并重新登录，这样主机名才会生效。最终要达到的效果是在任意一台机器使用主机名 ping ，其他的机器都能拼通。比如在 S1 上面 ping S3 , 能 ping 通， ok。



第二是开放端口，需要开放 3306 以及 33061 端口。 3306 是 MySQL 服务的端口， 33061 则是 MGR 的端口。第三，关闭SELINUX，前面已经出现过多次了。第四，需要安装MySQL。这里安装的是 8.023 版本。为了节约时间，这些准备工作我已经事先做好了。之后来搭建单主模式的 mgr 集群。我们规划 S1 是 primary 节点， S2 以及 S3 是 secondary 节点。下面来配置 单主模式。

首先执行 systemctl stop mysql.service 停止MySQL，停止S1、S2、S3。最后修改 MySQL 的配置文件，在里面添加这一堆的内容，配置有点多，我们来过一下。

首先，这段配置前面课上都已经讲过了。需要注意的是 Binlog format 一定要设成 row, MGR 必须要使用 ROW 这种模式。 

Binlog checksum 对于 8.0.20 以及之前的版本，必须要设成 NONE 。这是因为八点零点二零以及之前的版本默认会用 CRC32 进行 checksum，而  MGR不支持 checksum。到了 8.0.20 开始  MGR 它已经支持了 checksum 了，所以你可以省略，也可以继续把它设成none。 


disable storage engines 把其他的存储引擎都禁用掉。这个配置刚刚已经出现过了。 transaction write set extraction 用来指定记录事务的算法。建议使用 XXHASH64。当然了，对于 MySQL 8.0，默认就是 XXHASH64。这一段则是 MGR 的核心配置。

首先， plugin load add 用来指定 MGR 的插件文件是什么？ group replication group name 用来指定复制组的名称。值可以随便写，但是一定要是一个 UUID 的形式。你可以用 select UUID 生成一个 UUID 贴到这里来。此外，在一个  MGR集群里面每一个节点的 group 和 replication group name 要相同。


group_replication_start_on_boot 用来指定买服务器启动之后是不是要自动启动。主复制一般不建议开启，因为开启配置在故障恢复的时候会存在坑。 group replication local address 用来指定 MGR 的本地地址。 group replication group seeds 要来指定组成员的记者列表。 group replication bootstrap group 用来指定是不是要开启引导模式。引导模式是用于第一次搭建  MGR 或者是重建  MGR的时候使用的。一般来说我们会把配置设成 off，当要做引导的时候，用 set global 命令把它设成 on
