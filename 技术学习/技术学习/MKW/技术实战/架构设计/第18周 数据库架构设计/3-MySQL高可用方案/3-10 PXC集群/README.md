okay 普通 ppt 我们接着看另一个节点。收到复制写集以及 GTID 之后会进行校验，如果没有发现冲突就执行应用以及提交。反之，如果有冲突，就取消这次事务。这个过程简单来说就是数据合并、校验以及提交。为什么存在一个合并？这是因为 px 截取里面的每一个节点都可以做写入。比如节点 1 提交了一个 update 的操作，把数据 a 改成 b。而节点 2 与此同时，也提交了一个 update 的操作，把数据 a 改成c。这是有冲突的，冲突需要合并之后才能知道。


OK，这是两个节点的情况。假设现在有 3 个节点，第三个节点也是类似的，它也会受到复制写及 GTID  并做验证。这个时候比较有意思的是，如果两个节点都验证通过了，而第三个节点验证没有通过怎么办？ PXC 是这样设计的，它会把验证不通过的节点踢出集群，这个节点会自动的 shut down，也就是会把节点 3 踢出集群，节点 3 就 shutdown 了。说的通俗一点，节点 1 以及节点 2 都不接纳，节点 3 把它踢掉了，节点 3 就社会性死亡了，于是原地爆炸下 shutdown 了。

这里我准备了 3 台全新安装的机器，主机名和 IP 在这里已经装好了，操作系统是 Centos7.8。我们来手把手的安装一个 3 节点的 PXC 集群。需要注意的是，在实际项目里面，你的集群节点至少应该有 3 个，最多不要超过 8 个。


为什么最少是 3 个？其实两个节点的 PXC 集群也可以正常工作，只是一旦集群出问题，就可能会出现脑裂现象。比方两个节点之间的网络断了，两个节点都会认为另一个节点挂了，自己是 OK 的。对于 px1 集群来说，这种情况下，整个集群就没有办法正常工作了。你输入任何的命令，返回的都是 unknown in command，为什么不要超过 8 个呢？我们后面再说。

PXC 只支持 innodb 引擎，而且所有的表都必须要有主键，这个局限我认为绝大部分场景下都是可接受的。因为实时项目里面我们主要使用的数据库引擎就是Innodb，同时我们的表 99. 9% 的场景下都是有主键的，对吧。

第二，由于 PXC 要保证数据的一致性，所以多节点在并发写入的时候，锁冲突以及死锁的概率比其他模式高。并发写入指的是节点1，再写入节点 2 也在写入。比方节点 1 在操作 ID 为 5 的一行数据，节点 2 也在操作 ID 为 5 的数据。这个时候就可能会有冲突，对吧。脱离还有思索。还记得前面介绍的死佐的例子，同学们可以去类比一下。


第三，由于采用的是强一致性的方案，而且是同步写入的，所以写操作要在所有节点都执行成功之后才算成功。于是写入的效率往往取决于集群里面性能最差的节点，会存在短板效应。也就是木桶理论。把集群想象成是一个木桶，性能最差的节点，就是木桶里面最矮的板子。实际项目可以使用，比如 pcie、ssd 以及高速网卡等等机制，具体提升性能。最后，每个节点都会发生写操作，所以会存在写放大的问题。这也是为什么前面我们不建议 PXC 集群超过 8 个节点的主要原因。
