主节点启用 Binlog 之后，数据库的各种操作都会以事件的方式记录到 Binlog 里面去。从这里会通过 IO 线程和主节点通信，并监控主节点的 Binlog 变化。如果发现 blog 发生变化，就把变化复制到自己的 relaylog 里面去。之后再用 SQL 线程把相关的事件执行到自己的库里面去，从而和主库保持一致。


整个过程涉及三个线程一是 master 节点的 Binlog dump thread。它有了根据从节点 io thread 的请求，分批读取 Binlog 文件指定位置之后的信息，并返回给从接点。简单来说的话就是读取 Binlog 并返回给 io thread。

其中比较有意思的是 binlog format，它用来指定 blog 的格式有三种取值。一是statements，这种格式基于 SQL 语句，每条修改数据的 SQL 都会存储到 binlog 文件里面去。第二是row，这种模式基于行，每行数据的变化会记录到 binlog 文件里面去，但是并不记录原始的SQL。什么是数据的变化，我们暂时不知道。


第三是 mixed 混合模式。这个格式默认情况下会使用 statement 进行记录，如果 SQL 里面包含时间或者用户相关的函数的时候，会自动切换人row 。**不同的 MySQL 版本默认值是不一样的，比方 5. 6 默认使用的是statement，而 8. 0 默认使用的是row。这个配置比较重要。**同时我们也很难直观的从这段文字里面直观的看出具体的含义。所以我在这里准备了一个实验。

这种模式下，中间节点会充当主节点的从节点，此外，会同时充当这 3 个从节点的主节点，也就是中间节点，它又是从节点，又是主节点。这种模式的好处是它可以有效地降低主节点的压力。因为根据前面主从复制的原理，主节点下面挂的从节点数目越多，主节点的传输开销也会越大。比方你有一个主节点，后面拖了 6 个从节点，但是传输的开销已经比较大了。于是你可以在一个主节点下面挂两个中间节点，每个中间节点下面再挂 2- 3 个从节点，这样主节点的压力就会小很多。
