比方机房迁移，把数据从机房 a 迁移到机房b，从自有机房迁移到云平台。比方阿里云，从云平台 a 迁移到云平台b。比方从阿里云迁移到华为云。再比如系统重构，表结构发生了巨大的变化，这时候也需要做数据迁移。


还有分表调整，比方在项目初期没有去考虑分表，后续需要做分表了。这个时候需要把一个表里面的数据迁移到多个子表里面。又或者想要重新分表的情况。比方原先分了 100 个子表，后来想要分成 2048 个子表。还有更换数据库选型等等。原先用的是MySQL，后来发现 MySQL 不太合适，想要换成MongoDB，老的数据也需要迁移到 MongoDB 里面去。这些场景都会涉及到数据的迁移。在迁移的时候，

简单来说就是让新库作为原先主库的从库。当数据同步之后，再把新库升级成为主库，并将流量也切到新库上面去。步骤大概是这样的首先创建一个新库，并将新库设为老库的从库。从老库同步数据同步完成之后，为了保证数据的一致性，需要禁用老库的写入。也就是设置 read only 等于 on 以及 super read only 等于on，禁止往老库里面写入数据，并把新库升级成主库之后，再去让应用连接新库。



连接新库有几种情况如果应用是通过域名连接数据库的，就把域名解析到新库如果是通过 IP 连接的，需要修改连接的 IP 如果是通过虚拟 IP 连接的，则需要把 IP 票移到新库。这是挂重库的方案。在这种方案之下，在老库净写到应用连接到新库之间的这一段时间，业务是不可写入的。不可写入的时间一般会持续几秒到几十分钟，具体看你的应用能在多久之内连接到新库， 好。

简单来说，让应用同时写入老库以及新库，在迁移之后再把流量切到新库的一种玩法。这种方法它的局限性比较小，哪怕新旧库完全不同也能玩转。这里我举了个例子，假设我们的项目想要做重构，原先使用的是MySQL，后来想要换用MongoDB，新旧库的表结构，甚至用的数据库都是不一样的。


用双写去迁移步骤大概是这样的：

首先梳理出你想要迁移的表所涉及的写接口，也就是涉及增删改的接口。接着去实现双写数据迁移以及验证的功能。所谓的双写，就是修改涉及写的接口，让他同时写入新库以及老库。

**需要注意的是，涉及到增删改的接口都需要修改去实现双写。对于更新操作，如果新库里面没有相关的数据，需要从老库里面去查询，并把更新后的数据写到新库里面去。**


另外，**为了保证性能，写入新库可以异步执行**。你还可以为双写设置一个开关，让业务系统可以控制开启双写，或者继续只写老库业务，线上可能会业务双写之后性能不好，或者是写入老库成功，但是写入新库失败等等各种因素，需要去回退到原先的机制。所以留一个开关会比较的稳妥一些


OK，这是双写。数据迁移则是实现数据迁移的功能，一般是编写数据迁移的脚本。需要注意的是，你的数据迁移脚本需要记录非常详细的日志，便于一旦迁移失败，可以去方便的排查问题。至于验证，对迁移之后的数据进行校验，去验证数据是否正确，也就是编写一些脚本去验证新老库数据是否一致。


这其中数据迁移以及验证的工作不一定要自己实现。从 MySQL 迁移到 elasticsearch，就可以借助阿里的 Canal 中间件去实现迁移，但是实际项目里面更多的场景是自己写脚本。这些工作完成之后就可以正式去迁移数据了。


首先去上线双写的接口并开启双写。然后使用数据迁移脚本把开启双写之前的数据迁移到新库。你可以借助数据的一些字段去控制要迁移的数据范围，比方迁移 ID 小于等于某个值，或者是你的表有一个字段叫 create_time 创建时间，去迁移这个字段的值小于等于某一个时间的数据。最后使用验证脚本去验证数据是否正确，有没有数据遗漏等等问题了。你也可以先执行迁移脚本，再去开启双写。


这一步完成之后，**再去修改读的接口，让大部分的流量读取老库，小部分的流量读取新库，并观察业务是否正常，以及新老库的数据是不是一致。这其实是一个灰度验证的过程**。之后逐渐加大读取新库的流量比重，直到读的流量全部切到新库。最后关闭掉老库的写入，只写新库。这样迁移就完成了。后续可以去掉那些没有用的代码。
