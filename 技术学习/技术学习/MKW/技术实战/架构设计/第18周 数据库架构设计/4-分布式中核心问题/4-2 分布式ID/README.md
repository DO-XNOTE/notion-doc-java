第一种方案 UUID。这种方案很容易想到，它的好处在于使用起来非常的简单，也不需要额外引入组件。而缺点则是首先 UUID 是无序的，它没有办法实现范围查询。同时由于 UUID 它是无序的，而对于MySQL，它需要维护一个 b 加去的数据结构，而 b 加 tree 它是有序的。于是会导致插入操作是随机 io 使用 UUID 作为主键，相比于使用自增 id 作为主键，它的插入性能差不少。


比方这篇文章里面就有一个测评，用 UUID 作为主键，性能只有自增 id 的 1/4，性能差异还是非常大的。因此，实际项目里面往往使用自增 ID 加 UUID。自增 ID 依然作为表的主键，而业务上则使用 UUID 作为唯一标识拖拽的方式去提升插入的性能。

借助 Redis 之类的工具去生成 ID。 Redis 提供了 incr 指令，通过它可以实现原子性的自增，可以作为分布式 ID 的生成器。使用 Redis 这种方式，它的优点是使用起来非常简单，调一个 Redis API 就 OK 了，同时性能也挺好的。而缺点则是首先它需要引入Redis，也引入了额外的组件，某种程度上会增加系统的复杂度。第二，这种方案它强依赖Redis， Redis 不能挂，一旦挂掉 ID 就生成不了了。

比方假设 CacheSize 等于10， MaxID 现在是20， nextID 是11。执行这个方法的时候，就会返回12，一字6T。当 next ID 等于 20 的时候，就会把 next ID 设置成30，并把 next ID 设置成 30- 10 + 1，也就是21。也就是每返回 cache size 个 ID 之后，才会和数据库通信一次。

Snowflake 是业界比较著名的分布式 ID 算法，是由 Twitter 开源的一个 Snowflake ID 组成。大量是这样的。用它生成出来的 ID 是一个 long 类型的数字，long 占有 8 字节，也就是 64 bit。 snowflake 是这样分配每一 bit 的。第一个 bit 是标识部分。在java 里面，最高位是符号位，正数是0，负数是1。 snowflake 生成出来的 ID 是正数，所以这一位其实没有用，永远是0。


之后的 41 bit 是时间戳部分，会精确到毫秒。实际上 Snowflake 它一般不会直接存储当前时间的具体时间戳，而是一个时间戳差值，也就是用当前时间减去一个固定开始时间得到的毫秒数，这样可以让产生的 ID 值更小。理论上来说， 41 位的时间戳可以使用 69 年，你用 1 位移 41 位，再除以一年的毫秒数，就可以得到是 69 年了。第三部分是工作机器ID，占用 10 bit。这一部分比较灵活，比方你可以高位的 5 bit 存储数据中心ID，低位的 5 bit 存储工作节点ID。最后一部分是序列号，占用 12 bit。这个值在同一毫秒、同一节点之上，会从 0 开始不断累加，最多可以累加到4095。也就是在同一节点同一毫秒能够产生 4096 个ID。
