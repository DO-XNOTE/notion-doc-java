# 1-2 消息队列的基本概念（1517）

同学们大家好，这章节我们开始学习 Rocketmq 的架构设计解析的内容，我们也分三部分来介绍。首先是消息队列的基本概念，单纯介绍消息队列最基本的概念动作是消息的生产者和消息的消费者。现在市场上可选的这些消息队列，比如像 rock MQ， Kafka 或者 rapid MQ，他们在设计上都是基于这样一个概念模型，或多或少地做了一些个性化的扩展。


那么第二部分我们介绍一下 rock MQ 的技术架构。 rock MQ 的系统运行依赖四大组件，其中我们认为消息生产者和消息消费者它都作为客户端，那么作为客户端它都会依赖 rock MQ 的服务器。对于 rock MQ 的服务器，我们这里面分为 name Server 和 BROKER Server，其中的 name Server 它作为注册中心，同时也是我们的消息队列 topic 的路由选择器。那么 broker Server 它是作为一个代理服务器，负责消息存储、投递、查询以及服务的一些高可用的保障。


那么第三部分我们来介绍一下 rock MQ 的一个特性。我们介绍 rock MQ，它作为一个分布式的消息队列的实现，它除了实现基本的消息队列的生产、消费的功能，还扩展了哪些特性？这里面我们会介绍一下像消息的一些顺序消息如何过滤，消息的可靠性，比如说它的至少一次消费回溯、消费事务消息定词、消息重刺、消息重头以及流量控制死信队列等等这些特性的一些情况。


接下来我们来看一下消息队列的基本概念，那么这些基本概念里面，我们第一个先认识一下消息队列对应的生产者，同时也是producer，那么第二个就是消息队列的消费者consumer，我们我们知道生产者和消费者对于我们的 MQ 主题服务来说，它都是我们的客户端。

那么对于分布式消息的服务端，我们称之为消息队列的代理服务器，称之为 broker Server。 broker Server 我们可以理解为它是一个消息中转的角色，通常它并不能我们直接通过生产者把消息发送给消费者，它都是把生产者的，通过生产者把消息发给我们的代理服务器，也就是我们的 broker Server。那么 broker Server 它负责消息中转的一个决策，负责存储消息，转发消息。代理服务器在 Rocketmq 系统中负责接收从生产者发送过来的一些消息并存储，同时为消费者拉取请求做准备。那么代理服务器也存储消息相关的一些元数据信息，对于消息它的元数据信息它包含我们这个消息有哪些，消费者主也会包含我们的消费的进度偏移，以及我们主题和队列相关的一些信息。


那么现在我们知道了生产者和消费者通过代理服务器把消息传递过去，那么我们如何对于消息进行分类？这里面我们就需要了解一下消息队列的主题概念以及topic。那么每一个消息主题它表示一类消息的集合，每个主题它可以包含很多消息。我们这里面，比如说我们对于僧单这个业务来说，我每次创建僧单落库，我同时发出一个订单创建完成的消息发出去，那么这个消息队列的主题就是僧单完成，那么这个每一条消息主题它都属于我们生单完成这个主题。


那么如果说订单它状态流转到支付完成，我们再发出一个支付完成的消息，那么增单完成的消息和支付完成这个消息是两个不同类型的消息。那么对于我们做一个消费者订阅的过程中，我们必须基于主题来订阅，也就是说我们可以订阅生单消息，也可以订阅我们的支付消息，但是这两个是否可以同时订阅，这就需要我们去思考一下了。
这也跟不同的我们的 MQ 的实现机制有关系，有一些它可以基于一些前缀机子进行消费，比如说我们定义这个消息主题的时候，我们可以这样定义，对于生态消息，我们是 order 点creator，那么对于支付消息来说，我们可以定为 order 点pay。那么如果说它都以 order 作为我们的前缀，我们可以消费以 order 作为前缀的这些消息，那么我们可以同时消费生产消息和我们的支付消息。


那如果说对于我们的这个消息队列，它不支持我们支持前缀消费，那么就不行了，那么我们就需要支持我们通过我们的主题来进行我们的订阅，我们了解完消息主题，那么接下来看一下另一个重要的题。消费者主，我们知道消息队列通常我们可以做到对于生产者和消费者的解耦，也可以做到削峰填谷，同时也可以做到我们系统进行横向扩展的一种方式，那么这里面消费者主的话，它就可以用来具为我们的系统横向扩展的一种方式。同一个 consumer 的一个集合，这类 consumer 通常消费同一类消息，且消费逻辑是一致的，那么消费者主可以在消息消费方面实现负债均衡和一个容触的目的，这里面的负载均衡可以把压力分摊到我们整个消费者主，那么每个消费者主里面可以用多个消费者实力来进行我们功能的分摊，也就是通过扩展我们消费者实力来达到横向扩容我们消费能力的功能。


那么这里面要注意的是我们消费者主的消费者他它的实力必须订阅完全相同的 topic rock MQ，它支持多种消费模式，这里面涉及到集群消费或者是广播消费，那么集群消费我们可以理解为每一个实例，对于一个消费者，只要他消费完了，那么我们就认为这个已经消费完成了，那么广播模式是需要每个实例都去消费这个消息，所以说广播模式做不到我们的横向扩容的功能。


它对于广播模式通常用来一些信号的一些发送，比如说我们搜到这个广播信号以后，统一去更新一下我们的缓存信息，那么这个场景下我们可以用到广播消息。下面我们看一下消息队列的消费方式，这里面有拉取 pull 和推送push，这里面其实拉取和推送都是比较容易理解的，对于拉取通常是我们的业务系统，作为消费方的业务系统，它调用 consumer 客户端的方式，通过我们的 broker 或者是我们的消息服务器上去拉取信息。那么这个拉取的过程也是一个轮询探测的过程，它发现有消息存在的时候就进行拉取，那么这种方式我们知道它的实时性可能不是太高，同时也会有一些人群的一些消耗。


另一种方式是推送，那么推送是我们的消息服务器，当他接收到消息以后，他会去查询对应的订阅者，那么他找到对应的订阅者以后，就会把这消息用推送的方式推送到我们的对应的消费客户端。那么这里面有一点是这样，当我们的数据量非常大的时候，推送的这种方式有可能我们的消息队列把我们的业务应用程序给推垮，所以说推送这种方式，我对于一些大数据量的一个情况，采用的方式会更少一些，那么这里面拉取对于我们的消费方会有一些更好的一些自主性。


其实对于 rock MQ，它的发展可以分为三个阶段。第一代在数据传输过程中使用的是 pose 模式，在数据存储中使用关系型数据库，它展现了消息传递的一个低延迟，并且满足了具有分布式事务的一些典型的电子商务平台的一些用处。那么第二代是在数据传输中使用 Pro 模式，也就是拉取模式，这个拉取模式在数据存储中它可以使用文件系统，它更注重稳定性和可靠性，并显示出了第一代响应时间和 Kafka 日志收集相当的一些性能。
