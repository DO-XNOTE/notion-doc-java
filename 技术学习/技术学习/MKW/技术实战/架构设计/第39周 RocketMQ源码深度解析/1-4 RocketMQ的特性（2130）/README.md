# 1-4 RocketMQ的特性（2130）

同学们大家好，这章节我们开始学习 Rockemq 的功能特性，那么这里介绍的特性并不只是 Rocketmq 独有的功能特性，许多 MQ 的中间件的基础功能 rock MQ 它当然也实现支持了。这里面我们重点关注 rock MQ 它实现了哪些功能特性。


首先我们来看一下订阅发布，我们知道所有的 MQ 它都会实现订阅发布的功能，那么订阅发布也是 MQ 的基础功能。另外我们介绍消息的顺序、消费以及我们对于消息的一些过滤。那么首先我们来看一下订阅发布，消息的发布是指我们的生产者向某个 topic 发送消息，那么消息的订阅是指某个消费者关注了某个 topic 中带有某个 tag 的消息，那么有一些消息队列并没有 tag 的概念，我们基于 tag 进而从 topic 的消费数据，那么这里面我们的 topic 可以理解为一个队列，那么基于这个队列我们去消费对应的消息，这个是我们对应的定位发布的一个功能。那么这里面的消息顺序相对来说就比较独特，实现起来也会比较困难一些。


通常我们的消息有序是指的一类消息，消费者能够按照发送的顺序来消费。通常来说，我们对于电商的业务，一个订单它的生产过程，它包括订单的创建、订单付款以及订单的配送这样一个等等环节。如果我们消费的过程中并没有根据这个顺序进行消费，那么我们会得到一个错乱的一个业务图，所以说这种情况下我们应该要基于以一定的顺序来消费这个订单。


比如说对于相同的订单，我们在消费的过程中一定是订单创建在订单付款之前，订单付款在订单完成之前，所以说基于这个顺序我们才能获取到我们这个业务交互的意义。但是其实我们在处理一些高并发的请求的过程中，为了提高我们的性能，我们对于不同订单它的创建时间或者它的付款时间之间是并没有强烈要求的。所以这里面我们介绍这个消息的顺序消费，它可以分为全局的顺序消费与分区的顺序消费。


那么全局顺序消费是指某个 topic 下面所有的消息都要保证顺序，也就是我们刚才介绍的它不同订单的生产的顺序，那么我们消费的时候也要保持过来，那么这样的情况下对我们的性能是有一些部分影响的。那么另一种情况就是我们的分区顺序消息的时候，只要我们保证相同的订单，它的创建、付款和完成在相同的分区里面，那么它就能保证对于相同的订单，它的创建、付款完成是一个顺序消费的。那么这里面我们看对于全局顺序，我们通常是指定一个topic，它所有的消息是按照严格的先进先出的方式进行发送和消费，那么这种适用的场景我们知道，如果说我们做到全局的顺序这种事情上，它的性能要求肯定不高，那么所有的消息都按照先进先出的原则进行消息的发布和消息的消费的一种场景，那么通常我们只要满足分区的顺序就能满足大多的业务场景，那么分区的顺序在这里面我们可以理解。


我们基于订单号作为我们的分区k，那么对于相同的一个主题，一个topic，我们所有的消息只要根据我们的订单号进行分区，那么我们能保证相同的订单它可以分到相同的区块。对于同一个分区内的消息，我们可以做到严格的先进先出的顺序进行发布和消费，因为我们相同的订单它都会散列到我们相同的一个分区块，那么这样的话，我们保证在这个分区块里面我们对应的订单创建、付款和完成，只要保证这个顺序就能满足我们的业务要求，那么通常这是大多数从我们的业务的一个适用场景，通常是要求我们的性能会比较高一些。


这里面我们也要去选择合适的 starting k 作为我们的分区字段，在同一个区块中严格按照我们先进先出的原则进行消息的发布和消费的这样一种场景。另外对于消息过滤这一点可以理解为我们 rock MQ 的一个特性，有一大部分它的消息队列其实并没有实现消息过滤， rock MQ 的消费者可以根据 Tiger 进行消息过滤，这里面也支持自定义一些属性进行过滤。消息过滤目前是在 BROKER 实现的，我们可以理解为是在我们的服务端实现的那么优点，它减少了我们这些无效的消息跟我们 consumer 这样一个网络的传输。它的缺点也是它会增加 broker 的负担，并且实现的相对比较复杂。这里面我们简单说一下，对于基于 Tiger 进行过滤，它并不是基于 Tiger 这个制物串进行一口过滤的，它是对 Tiger 进行一层哈希运算，我们对哈希子进行一个过滤，所以这里面就会存在一个问题，如果说我们真的碰到哈希冲突的话，这个过滤是有问题的。


接下来我们了解一下消息的可靠性，以及我们对消息至少消费一次的个特性，以及我们的回溯消费这个功能。我们先说消息可靠性，对于 rock MQ，它是支持消息的高可靠性的，通常我们影响这个消息可靠性的情况分几种，一是我们的 BROKER Server 它非正常关闭，就是突然宕机，或者说是我们的 BROKER 通因为异常崩溃了，包括我们的操作系统崩溃，甚至我们的机器它突然断电。这些软硬件的一些故障，通常我们可以快速的恢复。比如说 broker 它非正常关闭，或者说崩溃了，那么我们的重启大法可以解决这个问题。我们如果说掉电了，那我们可以扩展我们的异言设备，保证这个断电它能快速恢复。这种情况下我们基本上能保证消息不丢失或者是少量丢失的消息。这里面主要是涉及到我们的消息落盘的方式，就是我们消息刷盘方落盘的方式，其实它分同步和异步，如果我们基于是同步的方式去刷盘的话，那么这个消息基本上可以理解为不丢，它都已经写到我们的磁盘里面了。


其实还有一种情况，就是比如我们的消息的一个单点的一个问题，比如说我们这个消息在发布的过程，它只会发到对应的一个机器上，那么这种情况下，如果说我们机器的磁盘坏了，或者说是真的它发生了一些硬件的一个故障，它不能进行启动，我们的重启大法不管用了。那么这种情况下一旦发生它其实很难恢复，那么在这个单点上的消息它就会丢失掉。


rock MQ，在这种情况下它如果说通过异步复制可以保证大多数的消息不丢，但仍然会有一些极早量的消息可能丢失。如果说我们通过同步双起的技术，基本上可以完全避免单点，这里面也是提到了对于我们是使用异步还是同步的方式，通常来说我们的异步它的性能会好一些。


当然这里面可能会丢消息，那我们采用同步双写的话，肯定会对我们的性能会有一些影响，这也是我们根据不同的业务场景去选择不同的部署方式。对于同步双写适合这种对消息的可靠性要求极高的一些场景，通常也跟我们的财务资金相关的，这些业务的话会选择这种方式，当然这对于我们的 rock MQ 部署成本要求也是比较高的。
