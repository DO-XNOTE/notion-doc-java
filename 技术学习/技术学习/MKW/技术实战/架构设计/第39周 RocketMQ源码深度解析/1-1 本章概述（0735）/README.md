# 1-1 本章概述（0735）

同学们大家好，这章节我们开始学习架构是源码解析的 rock MQ 部分。rockmq，它是一个分布式的消息和流数据的处理平台，它具有那些低延迟、高性能、高可靠性，同时支持万亿级的容量和灵活的可扩展性。Rocketmq，它是 2012 年阿里巴巴开源的第三代分布式消息中间件。那么他在 2016 年4月 21 日，阿里巴巴向阿帕奇软件基金会捐赠了 rock MQ，那么捐赠的 rock male 来注意开始四代阿帕奇的腐化项目。那么在第二年的2月 20 号，在短短的时间内，阿帕奇软件基金会宣布阿帕奇的 rock MQ 成为一个顶级项目，也就孵化成功。


早期阿里曾经基于 activity MQ 研发消息系统，随着业务消息的规模增大，瓶颈逐渐显现，后来也考虑过 Kafka 处理大数据，吞吐量处理得非常优秀，但是因为在低颜辞和高可靠性方面并没有好的选择，最后才自主研发了Rocketmq。通过各方面的性能都比目前已有的这些消息队列要好。 Rocketmq 和 Kafka 的概念和原理上都非常相似，数据侧也经常会拿来去进行比较。 Rocketmq 它默认会使用一个长轮询的拉取模式，单机支持千万级的消息队列，可以非常好的应用在海量的消息系统中。


那么我们学习 rock MQ 作为消息队列，它适应的一些场景和一些选型我们怎么去处理？对于哪些场合我们采用消息队列，比如说消息队列，它作为高并发系统的一个核心组件，它能够帮助业务系统解耦，并且提升开发效率和一些系统的稳定性。主要具有哪些优势？我们看，首先是削峰填谷，它主要是解决我们瞬时写入压力大于应用服务能力导致我们的消息丢失、系统崩溃等问题。另一方面是系统解耦，解决不同重要程度、不同能力级别系统之间依赖导致的一些系统雪崩。另外就是提升我们的性能，当存在一对多的调用，那么可以发一条消息给我们的消息系统，让消息系统通知相关的系统进行一些事件处理和一些消息的处理。


另一次我们的续流压测线上有些链路，它并不太好压测，我们可以堆积一定的消息进行一些重放来执行压测的效果。目前主流的 MQ 主要是像 rock MQ，Kafka， Rabbit MQ，那么 rock m q 相比 rapid MQ 卡不卡？它具有哪些优势？比如说它是支持树形消息，那么消息发送和 DB 抄路保持两方的最终一致性，那么 Rabbit MQ 和 Kafka 它明显是不支持的。另外 rock MQ 它支持结合多系统之间数据的最终一致性，也就是提供我们多方四五，那么它是基于二方四五的一个前提。


另外它支持一些言辞消息，这里面定义了 18 个级别，这样的话就可以延迟指定的一个时间段，并不是指定就是延定 Rocketmq 它设置的一个时间段进行一个延迟发送，那么像 Rabbit MQ 和 Kafka 它也是不支持的。另外 Rocketmq 它支持指定次数和时间间隔的一个失败消息的重复发，这在我们通常的业务场景很有用。像 Kafka 默认是不支持，那么 rap MQ 需要手动的去确认。另外 Kafka 的支持 consumer 端的 Tiger 过滤，减少一些不必要的一些网络传输。 rock m q 二和卡夫卡它并不具备这样的功能。


说了这么多， rock m q 的优点其实也是针对于 rock m q 它的一些特性来介绍。那么相比来说，像kafka， rock m q， rapid MQ， Kafka 它的系统定位通常是定位在系统间的一些数据的流管道，它可以支持一些实时性的数据处理，通常是对于消息规模比较大会用在哪些？像常规的一些消息系统，网站的一些性能跟踪监控数据，以及一些日志收集之类处理的一些情况。


那么像 rock MQ，通常它不会用来去处理一些日志，那么它更多的是用来去处理一些业务逻辑，比如说像订单交易、充值流计算、消息推送，像 binlog 的一些分发等等这种业务类型的这些消息队列。另外像 rock and MQ，它的一些特性其实跟 rock MQ 很类似，但为什么我们选择 rock MQ 更多的是它和开发语言相关。像Rockemq，它是基于 Java 的开发语言，也就是现在受众非常广的 Java 语言。那么像 Kafka 是Scala，我们可以理解为 Scala 也是一个基于 JIM 的一个虚拟机语言，但是 rapid MQ 它就相对来说比较个性一点，他用 Alarm 这种语言相对来说用户量会比较少一些。那么这是我们大概了解我们为什么去学习Rocketmq。来看一下 Rocketmq 我们会介绍哪些内容。
